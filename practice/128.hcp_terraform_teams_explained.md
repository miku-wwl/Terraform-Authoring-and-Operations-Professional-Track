> 来源：`128.md`  
> 主题：HCP Terraform team、用户邀请、owners team、最小权限原则  
> 类型：考试理论题 + 生产实践理解

---

## 1. 这一节到底在讲什么？

本节讲的是 **HCP Terraform / Terraform Cloud 里的 Teams 机制**。

一句话总结：

> **Team 是 organization 里面的用户分组；权限应该分给 team，而不是到处给单个人乱配权限。**

在真实公司里，HCP Terraform 不是一个人自己玩。通常会有：

- 平台团队：维护 workspace、变量、模块、运行策略
- 开发团队：提交 Terraform 代码、查看 plan、触发 run
- 安全团队：查看合规结果、审计权限、审查 policy
- 外包或临时成员：只允许查看部分 workspace
- 管理员：管理 organization、billing、全局设置

如果每个人都单独配置权限，很快就会乱成一锅粥。所以 HCP Terraform 用 **team** 来做权限边界。

---

## 2. 核心概念：Team 是什么？

在 HCP Terraform 里，**team 是 organization 内的用户组**。

它的作用不是运行 Terraform，也不是保存 `.tf` 文件，而是：

> **把一类人放在一起，然后统一授予 project 或 workspace 权限。**

例如：

```text
organization: acme-cloud
├── team: developers
│   └── 访问 app-dev / app-staging workspace
├── team: security
│   └── 只读访问 prod workspace + 查看 policy 结果
├── team: platform-admins
│   └── 管理 workspace、变量、运行设置
└── team: owners
    └── 组织最高权限
```

考试中看到这些词，要立刻联想到 team：

- organization
- users
- invite
- permissions
- workspace access
- project access
- least privilege
- owners team

---

## 3. Owners team 是什么？

`owners` team 是 HCP Terraform organization 里的高权限团队。

可以把它理解成：

> **HCP Terraform 组织里的“超级管理员组”。**

它通常拥有最高级别权限，能管理组织级设置、用户、团队、workspace、权限等关键资源。

所以生产环境中不要这样搞：

```text
新同事入职？
直接加 owners。
外包人员来了？
也加 owners。
开发同学只是想看一个 workspace？
还是加 owners。
```

这就是权限爆炸。

正确做法应该是：

```text
新开发者：加入 developers team
安全审计：加入 security-readonly team
平台管理员：加入 platform-admins team
真正组织管理员：才加入 owners team
```

考试里只要看到“普通用户”“新开发者”“只需要访问某个 workspace”，通常都不应该选 owners team。

---

## 4. Team 和 Permission 的关系

Team 本身只是“分组”，真正决定能干什么的是 **permissions**。

可以这样理解：

```text
Team = 人的集合
Permission = 这群人能做什么
Workspace / Project = 权限作用的目标
```

完整授权模型大概是：

```text
User
  ↓ 加入
Team
  ↓ 被授予
Workspace / Project Permission
  ↓ 获得
read / plan / write / apply / admin 等能力
```

所以设计权限时，不是问：

> “这个人是谁？”

而是问：

> “这个人属于哪类职责？这类职责应该访问哪些 workspace？应该能读、能 plan、能 apply，还是能管理？”

这就是 RBAC 的思路。

---

## 5. 最小权限原则

本节考试最重要的关键词是：

> **Least Privilege / 最小权限原则**

意思是：

> 用户只应该获得完成工作所需的最小权限，不要多给。

例如，一个开发者只需要查看 `app-dev` 的 plan 结果，那就不要给他 organization owner 权限。

一个安全审计人员只需要看 `prod` 的 run history 和 policy check，那也不要给他 apply 权限。

一个外包人员只需要临时查看某个 workspace，那最好给只读 team，并且项目结束后移除。

---

## 6. 常见权限设计方式

### 6.1 开发团队

适合权限：

- 查看 workspace
- 触发 plan
- 在非生产环境 apply
- 查看 run history

示例：

```text
team: developers
workspace: app-dev
permission: plan + apply
```

对于生产环境，可能只给：

```text
team: developers
workspace: app-prod
permission: read + plan
```

这样开发可以看到影响，但不能直接改生产。

---

### 6.2 平台团队

平台团队通常负责 workspace、变量、state、run settings、agent pool、policy 等配置。

适合权限：

- workspace admin
- 管理变量
- 管理执行模式
- 管理 workspace 设置
- 管理模块或共享基础设施

示例：

```text
team: platform-admins
project: shared-platform
permission: admin
```

---

### 6.3 安全团队

安全团队通常不需要改基础设施，但需要看结果。

适合权限：

- read-only
- 查看 policy check
- 查看 run history
- 查看 workspace 配置
- 审计变量是否配置合理

示例：

```text
team: security
workspace: network-prod
permission: read
```

---

### 6.4 外包 / 临时成员

外包人员最忌讳直接加 owners team。

适合做法：

```text
team: contractors-readonly
workspace: app-staging
permission: read
```

项目结束后：

```text
从 organization 或 team 中移除该用户
```

---

## 7. 考试高频陷阱

### 陷阱 1：把所有人都加 owners team

错误原因：

- owners 权限太大
- 违反最小权限原则
- 普通用户可能修改组织级设置
- 可能影响生产 workspace
- 审计风险很高

考试中如果选项里有：

```text
Add all users to owners team
```

大概率是错误选项。

---

### 陷阱 2：把 team 当成 workspace

Team 是用户组，不是 Terraform 的运行单元。

```text
team ≠ workspace
team ≠ state
team ≠ run
team ≠ Terraform 配置目录
```

Workspace 才是 HCP Terraform 里执行 plan/apply 和保存 state 的核心运行单元。

Team 只是控制谁能访问这些 workspace。

---

### 陷阱 3：以为邀请用户就自动有全部权限

用户被邀请进 organization，不代表他自动拥有所有 workspace 权限。

更合理的流程是：

```text
邀请用户进入 organization
↓
把用户加入合适 team
↓
team 获得对应 workspace/project 权限
↓
用户通过 team 间接获得权限
```

---

### 陷阱 4：共享 owner 账号

如果题目里出现：

```text
Share owner account credentials
```

这基本就是送分错误选项。

共享账号的问题：

- 无法审计是谁操作的
- 密码泄露影响巨大
- 离职人员可能仍然知道凭证
- 违反基本安全原则

正确答案应该围绕：

```text
invite user + team + permission + least privilege
```

---

## 8. 场景题讲解

### 场景 1：30 名开发人员都在 owners team

风险：

1. 权限过大，任何开发者都可能修改 organization 设置。
2. 普通用户可能修改关键 workspace、变量或运行策略。
3. 生产环境容易被误操作。
4. 审计不清晰，权限边界混乱。
5. 不符合最小权限原则。

改进方向：

1. 从 owners team 中移除普通开发人员。
2. 创建职责明确的 team，例如：
   - `developers`
   - `platform-admins`
   - `security-readonly`
   - `contractors-readonly`
3. 给不同 team 授予对应 project 或 workspace 权限。
4. owners team 只保留少数真正组织管理员。
5. 定期审查 team 成员和权限。

考试答案可以这样写：

> 应移除普通开发人员的 owners 权限，按职责创建 team，并基于 workspace 或 project 授予最小必要权限。owners team 只应保留真正需要组织级管理权限的管理员。

---

### 场景 2：外包人员只允许查看某个 workspace 的 run 结果，不允许 apply

正确思路：

```text
创建只读 team
↓
把外包人员加入该 team
↓
只在目标 workspace 授予 read-only 权限
↓
不要加入 owners team
↓
不要授予 apply/write/admin 权限
```

考试答案可以这样写：

> 应创建或使用一个只读 team，将外包人员加入该 team，并仅在对应 workspace 上授予查看 run 结果所需的只读权限，不应授予 apply 权限，也不应加入 owners team。

---

## 9. 跟真实项目怎么对应？

在真实平台工程里，HCP Terraform 的权限通常不是随便配的，而是配合组织结构设计。

例如你做一个 `cloud-governance-x` 或 AKS/EKS 平台项目，可以这样类比：

```text
organization: personal-platform-lab
project: aks-platform
workspace:
  - aks-dev
  - aks-prod
  - network-shared
  - observability
teams:
  - platform-admins
  - developers
  - security-readonly
  - auditors
```

权限设计：

```text
platform-admins:
  - aks-dev: admin
  - aks-prod: admin
  - network-shared: admin

developers:
  - aks-dev: write/apply
  - aks-prod: read/plan

security-readonly:
  - aks-prod: read
  - network-shared: read
  - observability: read

auditors:
  - all prod workspaces: read-only
```

这才像生产系统，而不是所有人全塞进 owners。

## 11. 考试判断模板

遇到 HCP Terraform team 题，可以按下面步骤判断：

```text
1. 题目有没有提 organization 内多个用户？
   有 → 想 team。

2. 有没有提不同职责？
   有 → 想按职责分 team。

3. 有没有提只访问某个 workspace？
   有 → 想 workspace/project permission。

4. 有没有提普通用户加入 owners？
   有 → 通常错误。

5. 有没有提最小权限？
   有 → 不要给 admin / owner，给刚好够用的权限。
```

## 12. 原题答案复盘

### 选择题

1. HCP Terraform team 的主要作用是什么？  
   答案：A  
   解析：team 用于在 organization 内按职责分组用户并管理权限。

2. 默认 owners team 通常具有什么特点？  
   答案：A  
   解析：owners team 通常拥有最高级别权限。

3. 新开发者只需要访问一个应用 workspace，最佳做法是什么？  
   答案：A  
   解析：应该加入合适的低权限 team，并授予对应 workspace 权限，而不是直接加入 owners。

---

### 判断题

1. Team 可以帮助按职责管理 organization 内的用户。  
   答案：对。

2. Owners team 适合作为所有普通用户的默认团队。  
   答案：错。owners 权限太高，不适合普通用户。

3. 用户邀请和 team 权限设计都与协作安全有关。  
   答案：对。

## 13. 面试表达

如果面试官问：

> HCP Terraform 里 team 有什么用？

可以这样答：

> Team is used to group users inside an HCP Terraform organization and assign permissions based on responsibilities. In production, I would avoid putting normal users into the owners team. Instead, I would create teams such as developers, security-readonly, and platform-admins, then grant project or workspace-level permissions according to the least privilege principle.

中文理解：

> Team 是组织内的用户分组，用来按职责分配权限。生产里不应该把普通用户放进 owners team，而应该根据开发、安全、平台等职责创建不同 team，再给 project 或 workspace 级别权限。

---

## 14. 本节最终结论

这一节不要死记 `team` 这个词，而要记住它背后的权限治理思路：

> **多人协作时，不能靠共享账号，也不能靠全员 owner。要靠 team 做用户分组，再按 workspace/project 授权。**

考试看到 team 题，大概率考的是：

- organization 内用户管理
- owners team 高权限
- 最小权限原则
- workspace/project 权限边界
- 不要给普通用户过高权限

能抓住这几个点，这类题基本不会丢分。
