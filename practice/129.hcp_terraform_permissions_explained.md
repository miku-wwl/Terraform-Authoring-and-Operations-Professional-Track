# Terraform 考试训练 129：Permissions 讲解版

> 来源：`doc/129.txt`  
> 主题：HCP Terraform 权限模型、team access、custom permissions、最小权限原则  
> 类型：理论题 / 考试高频概念题

---

## 1. 这一节到底在讲什么？

这一节讲的是 **HCP Terraform 里面怎么控制“谁能看、谁能改、谁能执行、谁能管理”**。

在真实团队里，Terraform 不只是一个人在本地 `apply`。一旦进入团队协作，就会出现很多权限问题：

- 开发人员能不能对生产环境运行 `plan`？
- 开发人员能不能直接 `apply` 生产环境？
- 安全团队能不能查看 policy check 结果？
- 审计团队能不能看 run history？
- 谁能读取 state？
- 谁能修改 workspace variables？
- 谁能管理 workspace？
- 谁能管理整个 organization？

HCP Terraform 的权限模型，就是用来解决这些问题的。

考试不会让你背 UI 菜单，而是会考你能不能判断：

> 在某个团队场景下，应该给什么权限，哪些权限不能乱给。

核心原则只有一句话：

> **按 team 和职责授予最小必要权限。不要一上来就给 owner/admin。**

---

## 2. HCP Terraform 权限的层级

HCP Terraform 的权限可以出现在多个层级：

```text
Organization
└── Project
    └── Workspace
        ├── Runs
        ├── State
        ├── Variables
        ├── Outputs
        ├── Policies
        └── Run tasks
```

可以粗暴理解成：

| 层级 | 作用 |
|---|---|
| Organization | 最大的团队/计费/管理边界 |
| Project | 用来组织一组 workspace |
| Workspace | 具体 Terraform 工作负载的运行单元 |
| Team access | 把某个 team 绑定到某些 workspace/project/org 权限 |
| Custom permissions | 对 state、variables、runs 等能力做细粒度控制 |

考试中最常见的是 **workspace 级别的 team access**。

---

## 3. Team access 是什么？

Team access 的意思是：

> 给一个 team 分配某个 workspace 的访问权限。

例如：

```text
team: developers
workspace: prod-network
permission: plan only
```

这代表开发团队可以对 `prod-network` 这个 workspace 运行 plan，但不能直接 apply。

再比如：

```text
team: platform-admins
workspace: prod-network
permission: admin
```

这代表平台管理员团队可以管理这个 workspace。

---

## 4. 常见权限等级怎么理解？

HCP Terraform 中常见权限等级包括：

- `read`
- `plan`
- `write`
- `admin`
- `custom`

不要死记字面，要理解它们的风险等级。

### 4.1 read

`read` 通常用于查看 workspace 信息、运行记录、部分输出等。

适合：

- 审计人员
- 只需要看结果的人
- 安全团队查看 policy 结果
- 旁观排障人员

但注意：**read 不等于可以读取完整 state**。state 权限经常是单独控制的。

---

### 4.2 plan

`plan` 允许用户发起计划阶段。

适合：

- 开发者验证变更影响
- PR 前检查变更结果
- 非生产环境预览

如果一个团队只需要“看看会改什么”，那通常给 `plan` 就够了。

---

### 4.3 write

`write` 权限通常意味着可以发起更强的运行能力，例如 apply 相关操作。

适合：

- 负责该环境交付的工程团队
- 非生产环境 owner
- 有授权的发布人员

生产环境不要随便给 write。

---

### 4.4 admin

`admin` 可以管理 workspace 级别设置。

适合：

- 平台团队
- 基础设施 owner
- 少数负责 workspace 生命周期的人

不要把 admin 当成普通权限。admin 是管理权，不只是使用权。

---

### 4.5 custom

`custom` 是考试很喜欢考的点。

它的价值是：

> 不再只能粗暴给 read/write/admin，而是可以对变量、state、run、Sentinel mocks、run tasks 等能力做更细粒度控制。

例如：

- 可以看 outputs，但不能看完整 state
- 可以运行 plan，但不能 apply
- 可以查看 run history，但不能修改 variables
- 可以管理 variables，但不能管理 workspace
- 可以读取 policy 结果，但不能覆盖 policy

考试看到 **fine-grained permissions / variables / state / run tasks / Sentinel mocks**，要想到 custom permissions。

---

## 5. 最小权限原则：考试第一反应

权限题的第一反应不是“给 admin 省事”。

考试中看到这类题：

> 某团队只需要查看运行记录。  
> 某团队只需要运行 plan。  
> 某团队只需要读取 outputs。  
> 某团队不能修改生产变量。  
> 某团队不能 apply 生产环境。

你应该立刻想到：

> 给最小必要权限。

这就是 Least Privilege。

错误思路：

```text
所有人加入 owners team
所有人都给 admin
所有人都能读完整 state
所有人都能改 variables
所有人都能 apply prod
```

正确思路：

```text
按 team 分组
按职责授权
按 workspace/project 限制范围
敏感权限单独控制
生产环境权限更严格
```

---

## 6. State 权限为什么特别敏感？

Terraform state 是非常敏感的东西。

很多初学者以为 state 只是 Terraform 内部记录，其实它可能包含：

- 资源 ID
- endpoint
- IP 地址
- ARN
- 数据库连接信息
- 部分 provider 返回的属性
- 某些敏感字段
- 输出值
- 基础设施拓扑信息

所以考试里经常强调：

> **读取 outputs only 和读取完整 state 不是一回事。**

这句话非常重要。

### 6.1 outputs only

只允许用户看 Terraform outputs。

适合：

- 应用团队只需要 endpoint
- 审计团队只需要部署结果
- 下游系统只需要有限输出

### 6.2 full state read

允许用户读取完整 state。

风险更高，因为 state 可能暴露大量资源细节，甚至敏感信息。

所以除非必要，不要给完整 state 权限。

---

## 7. Variables 权限为什么也敏感？

Workspace variables 可能保存：

- provider credentials
- token
- API key
- environment variables
- Terraform variables
- cloud authentication 信息

所以能不能读取、创建、修改 variables，也需要单独控制。

典型考试场景：

> 开发人员需要运行 plan，但不能查看生产凭证。

这时不能给他们变量读取权限，尤其不能让他们读取 sensitive variables。

---

## 8. 常见团队应该怎么授权？

### 8.1 开发团队

需求：

- 可以看 plan
- 可以验证变更
- 不能 apply 生产
- 不能改生产变量
- 不能读完整 state

建议方向：

```text
dev team:
  prod workspace:
    read/plan
    no apply
    no admin
    no full state read
    no variable write
```

### 8.2 平台团队

需求：

- 管理 workspace
- 管理变量
- 管理运行配置
- 维护 state
- 处理故障

建议方向：

```text
platform team:
  relevant workspace:
    admin or custom high privilege
```

但即使是平台团队，也最好按职责拆分，不要所有人都是 organization owner。

### 8.3 安全团队

需求：

- 查看 policy check
- 查看 run 结果
- 查看合规状态
- 不一定需要 apply
- 不一定需要改变量

建议方向：

```text
security team:
  read access
  policy visibility
  no apply unless explicitly required
  no state access unless justified
```

### 8.4 审计团队

需求：

- 查看 run history
- 查看审批记录
- 查看 outputs
- 不应看到完整 state 敏感细节

建议方向：

```text
audit team:
  read run history
  outputs only
  no full state read
  no variable access
  no apply
```

## 9. 场景题拆解

### 场景 1：开发团队需要运行 plan，但不能 apply 生产 workspace

题目关键词：

- developers
- production workspace
- can run plan
- cannot apply

正确方向：

```text
给开发 team 授予生产 workspace 的 read/plan 权限。
不要授予 write、apply、admin 权限。
```

解释：

开发人员只需要看变更影响，不需要真正改生产环境。生产 apply 应该由受控发布流程、平台团队或审批后的授权人员执行。

---

### 场景 2：审计团队需要查看运行记录和输出，但不应看到敏感 state 细节

题目关键词：

- audit team
- run history
- outputs
- should not see sensitive state details

正确方向：

```text
授予查看 run history 和 outputs 的权限。
避免授予 full state read/write 权限。
```

解释：

审计人员要的是“谁做了什么、什么时候做的、结果是什么”，不是完整基础设施内部状态。完整 state 可能包含敏感信息，所以不应该默认开放。

---

## 10. 和真实工程实践怎么对应？

在真实公司里，这一节不是纸面知识，而是生产环境事故预防。

如果所有人都有 admin，会发生什么？

- 新人误 apply 生产
- 测试变量覆盖生产变量
- state 被误读或误改
- 敏感输出泄露
- workspace 设置被误改
- 审计时查不清谁有权限做什么

更靠谱的做法是：

```text
开发团队：plan 权限
发布团队：apply 权限
平台团队：workspace admin
安全团队：policy/read 权限
审计团队：run history + outputs only
组织 owner：极少数人
```

这就是 HCP Terraform 权限治理的价值。

## 11. 考试高频判断口诀

### 口诀 1：权限题先套最小权限

看到权限题，先问：

> 这个人完成任务最少需要什么权限？

不要自动给 admin。

---

### 口诀 2：state 不是普通文件

看到 state，立刻提高警惕。

> state 可能包含敏感资源属性和基础设施细节。

---

### 口诀 3：outputs only 不等于 full state

这两个权限边界不一样。

> 能看 output，不代表应该能看完整 state。

---

### 口诀 4：custom permissions = 细粒度治理

看到 custom、variables、state、run tasks、Sentinel mocks，就想到细粒度权限控制。

## 12. 本节选择题解析

### 题 1

> 哪项最符合 HCP Terraform 权限管理的最佳实践？

正确答案：A

```text
A. 按 team 和职责授予最小必要权限
```

解析：

这是最小权限原则。不要把所有人加入 owners team，更不能把密钥写入 Git。

---

### 题 2

> Workspace 权限中，custom 权限的价值是什么？

正确答案：A

```text
A. 对变量、state、run 等能力做更细粒度控制
```

解析：

custom permissions 的关键就是细粒度控制。它不是用来创建云资源，也不是替代 module。

---

### 题 3

> 为什么 state 访问权限需要谨慎控制？

正确答案：A

```text
A. State 可能包含资源属性和敏感信息
```

解析：

state 是 Terraform 的核心记录文件，可能包含很多敏感或半敏感信息。考试很爱考这一点。

---

## 13. 判断题解析

### 判断题 1

> HCP Terraform 可以在 workspace 级别为 team 配置访问权限。

答案：对。

workspace 是 HCP Terraform 中很重要的运行和协作单元，可以对 team 配置访问权限。

---

### 判断题 2

> 最小权限原则意味着普通开发者默认应该拥有 organization owner 权限。

答案：错。

这正好反了。普通开发者不应该默认拥有 organization owner 权限。

---

### 判断题 3

> 读取 outputs only 和读取完整 state 是不同权限边界。

答案：对。

outputs only 更窄，full state read 更敏感，二者不是一回事。

---

## 14. 容易踩坑的点

### 坑 1：把 read 理解成能看所有东西

read 不一定等于 full state read。考试看到 state 要单独判断。

---

### 坑 2：觉得 admin 最方便，所以都给 admin

真实工程里这是事故入口。考试里这也是错误答案常见套路。

---

### 坑 3：忽略 production workspace 的特殊性

生产 workspace 权限要更严格。能 plan 不代表能 apply。

---

### 坑 4：把 organization owner 和 workspace admin 混为一谈

organization owner 是组织级最高权限，workspace admin 是某个 workspace 的管理权限。范围不同，风险也不同。

---

## 15. 最终复习版总结

本节只需要抓住四个点：

1. **HCP Terraform 支持 organization、project、workspace 等层级权限控制。**
2. **Team access 可以把团队和 workspace 权限绑定起来。**
3. **Custom permissions 用于变量、state、run 等细粒度控制。**
4. **权限题优先使用最小权限原则，尤其谨慎授予 state 和 admin 权限。**

一句话总结：

> HCP Terraform Permissions 考的不是“谁最大”，而是“谁只该拿到刚好够用的权限”。生产环境里，权限越大，锅越大。
