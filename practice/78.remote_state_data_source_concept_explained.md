# Terraform 实操训练 78 讲解：Remote State Data Source 概念

## 1. 这一节到底在学什么？

第 78 节学的是 Terraform 的 **Remote State Data Source**，核心对象是：

```hcl
data "terraform_remote_state" "xxx" {
  ...
}
```

一句话理解：

> `terraform_remote_state` 可以让一个 Terraform 配置读取另一个 Terraform state 里暴露出来的 output 值。

本节的典型场景是：

```text
network 团队创建网络资源，并把 state 写入 S3 backend
consumer 团队不手工复制值，而是从 network 的 remote state 读取 output
```

课程文件里的例子是：

```text
networking team 创建公网 IP
security team 从 remote state 自动读取 IP
然后用于防火墙规则
```

这就是 Terraform 里的跨配置、跨团队状态依赖。

---

## 2. 为什么需要 remote state？

真实公司里，基础设施通常不是一个 Terraform 项目全包。

常见拆分方式：

```text
network/
  管 VPC、subnet、公网 IP、路由

security/
  管 security group、防火墙、WAF、policy

app/
  管应用、容器、服务配置

monitoring/
  管告警、dashboard、日志采集
```

这些配置之间经常有依赖。

例如：

```text
security 配置需要知道 network 创建出来的 public IP
app 配置需要知道 network 创建出来的 subnet id
monitoring 配置需要知道 app 输出的 endpoint
```

如果靠人工复制：

```text
network 输出 IP -> 人手复制到 security 变量
```

问题很多：

- 容易复制错
- 容易过期
- 没有自动同步
- 没有依赖关系可追踪
- 跨团队沟通成本高
- 变更后容易漏改
- CI/CD 不好自动化

所以需要：

```text
consumer 自动读取 producer 的 output
```

这就是 `terraform_remote_state` 的价值。

---

## 3. `terraform_remote_state` 是什么？

`terraform_remote_state` 是 Terraform 内置数据源。

它不是某个云 provider 的资源，而是 Terraform 自己提供的 data source。

基本形式：

```hcl
data "terraform_remote_state" "network" {
  backend = "s3"

  config = {
    bucket = "terraform-state"
    key    = "network/terraform.tfstate"
    region = "us-east-1"
  }
}
```

然后可以读取另一个 state 中的 output：

```hcl
data.terraform_remote_state.network.outputs.public_ip
```

注意：

> 它只能读取对方 state 里声明为 `output` 的值。

不是 state 里所有资源属性都能随便读。

---

## 4. producer 和 consumer

这节可以用两个角色理解。

### Producer

生产者配置，也就是 network 团队。

它负责创建资源，并输出值：

```hcl
output "public_ip" {
  value = terraform_data.public_ip.output
}
```

它的 state 写入 S3 backend：

```text
s3://terraform-state/network/terraform.tfstate
```

---

### Consumer

消费者配置，也就是 security 团队或 app 团队。

它不直接创建 network 资源。  
它只读取 network 输出：

```hcl
data "terraform_remote_state" "network" {
  backend = "s3"

  config = {
    bucket = "terraform-state"
    key    = "network/terraform.tfstate"
    region = "us-east-1"
  }
}
```

然后使用：

```hcl
locals {
  allowed_ip = data.terraform_remote_state.network.outputs.public_ip
}
```

---

## 5. 本节的核心链路

本节最重要的链路是：

```text
network 配置 apply
-> network state 写入 S3
-> network 声明 output
-> consumer 配置读取 network remote state
-> consumer 使用 network output
```

可以画成：

```text
network Terraform
  └── output "public_ip"
        ↓
S3 backend state
        ↓
terraform_remote_state
        ↓
consumer Terraform
  └── 使用 public_ip
```

这就是跨团队状态依赖。

---

## 6. `terraform_remote_state` 只能读取 output

这是最重要的限制之一。

假设 network 配置里有资源：

```hcl
resource "terraform_data" "public_ip" {
  input = "203.0.113.10"
}
```

如果没有写：

```hcl
output "public_ip" {
  value = terraform_data.public_ip.output
}
```

consumer 就不能直接这样读：

```hcl
data.terraform_remote_state.network.resources...
```

正确方式是 producer 显式暴露 output：

```hcl
output "public_ip" {
  value = terraform_data.public_ip.output
}
```

consumer 再读：

```hcl
data.terraform_remote_state.network.outputs.public_ip
```

一句话：

> remote state 的契约是 output，不是内部资源细节。

这非常像 API 设计。  
producer 暴露什么 output，consumer 才能依赖什么。

---

## 7. 为什么只读取 output 是好事？

表面上看限制很多，但这是好事。

因为它强迫团队设计清晰边界。

network 团队可以决定暴露：

```hcl
output "public_ip" { ... }
output "vpc_id" { ... }
output "private_subnet_ids" { ... }
```

security/app 团队只依赖这些公开 output。

这样避免 consumer 直接依赖 producer 内部资源实现。

例如 network 内部从：

```hcl
aws_eip.public
```

改成：

```hcl
aws_lb.frontend
```

只要 output 名称和语义保持稳定，consumer 不一定需要改。

这就是模块边界和跨团队接口思维。

---

## 8. 本实验为什么用 LocalStack？

课程文件说明，本项目不直接操作真实 AWS，而是统一使用 Docker 启动 LocalStack 模拟：

- S3
- DynamoDB
- STS

本节用 LocalStack 的 S3 backend 保存 state。

好处：

1. 不需要真实 AWS 账号。
2. 不产生云费用。
3. 可以本地重复实验。
4. 可以模拟 S3 backend。
5. 可以训练跨 state 读取流程。
6. 适合 CI 自动验证。

本节备注也说明：

```text
本实验使用 terraform_data 模拟跨团队输出值，不创建真实 AWS 资源。
```

也就是说，重点不是创建真实公网 IP，而是练：

```text
network state output -> consumer remote state read
```

---

## 9. 为什么用 `terraform_data` 模拟？

`terraform_data` 是 Terraform 内置资源类型，适合做本地训练和数据模拟。

例如：

```hcl
resource "terraform_data" "public_ip" {
  input = "203.0.113.10"
}
```

它可以模拟：

```text
network 团队创建出来的公网 IP
```

然后 output：

```hcl
output "public_ip" {
  value = terraform_data.public_ip.output
}
```

consumer 就可以读取这个 output。

这样不用真实 AWS 资源，也能训练 remote state 概念。

## 10. 典型目录结构

本节通常会有两个 Terraform 配置目录。

类似：

```text
practice/labs/78/
├── network/
│   ├── main.tf
│   └── backend config
├── consumer/
│   ├── main.tf
│   └── backend config
└── README.md
```

或者类似结构。

重点是：

```text
network 和 consumer 是两个独立 Terraform root module
它们有各自的 state
consumer 通过 terraform_remote_state 读取 network state
```

不要把它理解成普通 module 调用。

## 11. remote state 和 module output 的区别

这点很容易混。

### Module output

同一个 root module 内调用子模块：

```hcl
module "network" {
  source = "./modules/network"
}

resource "xxx" "example" {
  vpc_id = module.network.vpc_id
}
```

这是同一次 Terraform run 里的模块依赖。

---

### Remote state output

两个独立 root module，state 分开：

```text
network/
consumer/
```

consumer 读取 network 的远端 state：

```hcl
data.terraform_remote_state.network.outputs.vpc_id
```

这是跨 Terraform 配置、跨 state 的依赖。

简单对比：

| 方式 | 关系 |
|---|---|
| `module.network.output` | 同一个 root module 内部 |
| `terraform_remote_state.network.outputs.xxx` | 另一个 Terraform state 的 output |

---

## 12. 为什么不是所有东西都放一个 Terraform 项目？

小项目可以放一起。  
但大项目经常要拆。

原因包括：

- 团队边界不同
- 生命周期不同
- 权限不同
- 风险不同
- apply 频率不同
- state 过大
- blast radius 太大
- CI/CD pipeline 不同

例如：

```text
network 很稳定，几个月改一次
app 经常发布，每天改
security policy 有单独审批
monitoring 由另一个团队维护
```

如果全放一个 state，任何小改动都可能影响巨大范围。

拆 state 后，就需要跨 state 读取 output。  
这就是 remote state 的典型场景。

## 13. 本节实验目标

课程文件给出四个目标。

### 目标 1：创建 network 配置，并把 state 写入 LocalStack S3 backend

也就是 producer 先 apply，并把 output 写进远端 state。

---

### 目标 2：consumer 使用 `terraform_remote_state` 读取 network 输出

也就是 consumer 读取 producer 的 output。

---

### 目标 3：验证 consumer plan/apply 能自动使用 network 输出

不是人工复制 IP，而是 consumer 自动从 remote state 得到值。

---

### 目标 4：按正确顺序销毁 consumer、network，并清理 state 对象

这点很重要。

因为 consumer 依赖 network 的 output，所以销毁顺序不能乱。

---

## 14. 正确 apply 顺序

因为 consumer 依赖 network 的 state output，所以正确顺序是：

```text
1. apply network
2. apply consumer
```

如果先 apply consumer，它会找不到 network state 或 output。

命令逻辑：

```sh
cd network
terraform init
terraform apply

cd ../consumer
terraform init
terraform apply
```

这就是跨 state 依赖的自然顺序。

---

## 15. 正确 destroy 顺序

销毁时顺序要反过来：

```text
1. destroy consumer
2. destroy network
```

原因：

consumer 依赖 network 的输出。  
如果你先 destroy network，network state/output 可能消失，consumer 后续可能无法正常读取 remote state。

所以：

```text
创建：先 producer，再 consumer
销毁：先 consumer，再 producer
```

这要记牢。

## 16. 典型 network 配置示例

network 配置可能类似：

```hcl
terraform {
  backend "s3" {
    bucket = "terraform-state"
    key    = "labs/78/network/terraform.tfstate"
    region = "us-east-1"

    endpoints = {
      s3 = "http://localhost:4566"
    }

    skip_credentials_validation = true
    skip_metadata_api_check     = true
    skip_region_validation      = true
    force_path_style            = true
  }

  input = "203.0.113.10"
}

output "public_ip" {
  value = terraform_data.public_ip.output
}
```

这里重点是：

```text
output "public_ip"
```

consumer 只能读取 output。

## 17. 典型 consumer 配置示例

consumer 配置可能类似：

```hcl
data "terraform_remote_state" "network" {
  backend = "s3"

  config = {
    bucket = "terraform-state"
    key    = "labs/78/network/terraform.tfstate"
    region = "us-east-1"

    endpoints = {
      s3 = "http://localhost:4566"
    }

    skip_credentials_validation = true
    skip_metadata_api_check     = true
    skip_region_validation      = true
    force_path_style            = true
  }
}

resource "terraform_data" "firewall_rule" {
  input = {
    allowed_ip = data.terraform_remote_state.network.outputs.public_ip
  }
}
```

这表示：

```text
consumer 从 network state 读取 public_ip
然后把它用于 firewall_rule 模拟资源
```

真实场景里可以换成：

- security group rule
- firewall rule
- WAF allowlist
- Kubernetes network policy
- monitoring target
- application config

## 18. consumer 是否需要自己的 backend？

通常需要。

network 和 consumer 是两个独立 root module。  
它们应该有各自 state。

例如：

```text
network state:
s3://terraform-state/labs/78/network/terraform.tfstate

consumer state:
s3://terraform-state/labs/78/consumer/terraform.tfstate
```

consumer 读取 network state，但 consumer 自己的资源状态仍然写入自己的 backend。

不要让 network 和 consumer 共用同一个 state key。

---

## 19. state key 设计

本节最容易犯的错误是 state key 混乱。

推荐：

```text
labs/78/network/terraform.tfstate
labs/78/consumer/terraform.tfstate
```

不推荐：

```text
labs/78/terraform.tfstate
```

如果两个 root module 共用同一个 key，会互相覆盖 state。

跨 state 依赖的前提是：

```text
producer 和 consumer 各自有独立 state
consumer 读取 producer 的 output
```

不是共用一份 state。

---

## 20. remote state 读取失败的常见原因

### 原因 1：network 还没 apply

consumer 读取不到 state 或 output。

解决：

```text
先 apply network
再 apply consumer
```

---

### 原因 2：network 没有 output

producer 没写：

```hcl
output "public_ip" { ... }
```

consumer 读：

```hcl
outputs.public_ip
```

会失败。

---

### 原因 3：output 名称写错

network 输出：

```hcl
output "public_ip_address" { ... }
```

consumer 却读：

```hcl
outputs.public_ip
```

会失败。

---

### 原因 4：S3 key 写错

consumer 的 remote state config 必须指向 network state key。

如果 key 指向 consumer 自己，或者写错路径，就读不到预期 output。

---

### 原因 5：LocalStack endpoint 配置错误

没有指向：

```text
http://localhost:4566
```

可能会尝试连接真实 AWS。

---

### 原因 6：state 对象被清理了

如果 network state object 被删除，consumer 就无法读取。

---

## 21. remote state 的敏感信息风险

`terraform_remote_state` 可以读取另一个 state 的 output。

如果 producer 输出了敏感信息，consumer 可能也能读到。

例如：

```hcl
output "db_password" {
  value     = var.db_password
  sensitive = true
}
```

即使标记 sensitive，也不代表所有风险消失。

state 本身仍然可能包含值。  
能访问 remote state 的人或 pipeline，要按敏感权限对待。

所以生产里不要随便把 secret 暴露成 remote state output。

更好的 secret 共享方式通常是：

- AWS Secrets Manager
- Azure Key Vault
- HashiCorp Vault
- Parameter Store
- Kubernetes Secret workflow
- 专门的 secret distribution mechanism

remote state 更适合共享基础设施 ID、ARN、IP、subnet 等非高敏值。

---

## 22. remote state 是强耦合吗？

是的，它会引入跨 state 依赖。

consumer 依赖 network 的：

- backend 类型
- bucket/key
- output 名称
- output 类型
- output 语义

如果 network 改 output 名称，consumer 会坏。

所以 remote state 要当成 API 来管理。

建议：

1. output 名称稳定。
2. output 语义清楚。
3. 不轻易删除 output。
4. 变更 output 要通知 consumer。
5. 重要 output 可以写文档。
6. 跨团队约定版本和变更流程。

不要把 remote state 当随便读内部变量的捷径。

---

## 23. remote state 和数据源直接查询的取舍

有些场景不一定要用 remote state。

例如 consumer 需要 VPC ID，可以选择：

### 方式 1：remote state

```hcl
data.terraform_remote_state.network.outputs.vpc_id
```

### 方式 2：云 provider data source

```hcl
data "aws_vpc" "selected" {
  tags = {
    Name = "main"
  }
}
```

对比：

| 方式 | 优点 | 风险 |
|---|---|---|
| remote state | 精确读取 producer 输出 | 跨 state 耦合，需 state 权限 |
| provider data source | 从真实云 API 查询 | 依赖标签/命名，查询可能不唯一 |
| 手工变量 | 简单直接 | 容易过期、人工错误 |

没有绝对最好。  
要看团队边界、权限、安全和稳定性。

---

## 24. remote state 不会自动 apply producer

这是一个常见误解。

consumer 读取 network remote state，不代表 Terraform 会自动帮你 apply network。

如果 network 配置改了，但还没 apply：

```text
consumer 读到的还是旧 output
```

remote state 读取的是 state 当前保存的 output，不是 network 配置文件里即将变化的值。

所以正确流程仍然是：

```text
network 变更 -> apply network -> state output 更新 -> apply consumer
```

这和普通 module 依赖不一样。

---

## 25. remote state 和依赖顺序

Terraform 只能在 consumer 当前配置内部建立依赖。

它不能跨 root module 自动调度：

```text
先 apply network，再 apply consumer
```

这个顺序要由你、CI/CD pipeline 或 orchestration 工具管理。

例如 pipeline 可以设计为：

```text
network pipeline apply 成功
-> 触发 consumer pipeline plan/apply
```

或者用文档规定人工顺序。

这就是跨 state 架构的运维成本。

## 26. 本节推荐实验流程

### Step 1：启动 LocalStack

确保本地 S3 backend 可用。

---

### Step 2：创建 S3 bucket

```sh
aws --endpoint-url=http://localhost:4566 s3 mb s3://terraform-state
```

---

### Step 3：apply network

```sh
cd practice/labs/78/network
terraform init -input=false
terraform plan -input=false -no-color -out=tfplan
terraform apply -auto-approve tfplan
terraform output
```

确认 network 输出了类似：

```text
public_ip = "203.0.113.10"
```

---

### Step 4：apply consumer

```sh
cd ../consumer
terraform init -input=false
terraform plan -input=false -no-color -out=tfplan
terraform apply -auto-approve tfplan
terraform output
```

确认 consumer 使用了 network 输出。

---

### Step 5：验证 S3 state

查看 network state：

```sh
aws --endpoint-url=http://localhost:4566 s3 ls s3://terraform-state/labs/78/network/
```

查看 consumer state：

```sh
aws --endpoint-url=http://localhost:4566 s3 ls s3://terraform-state/labs/78/consumer/
```

---

### Step 6：按反向顺序销毁

```sh
cd consumer
terraform destroy -auto-approve

cd ../network
terraform destroy -auto-approve
```

---

### Step 7：清理 state 对象和本地文件

```sh
rm -f tfplan
rm -rf .terraform .terraform.lock.hcl
```

根据实验脚本清理 LocalStack 对象。

## 27. 为什么销毁要 consumer 先？

因为 consumer 依赖 network output。

如果先 destroy network：

```text
network state/output 可能消失
consumer 再 plan/destroy 时读取 remote state 失败
```

所以销毁顺序是反向依赖：

```text
apply：network -> consumer
destroy：consumer -> network
```

这是跨 state 依赖中必须养成的习惯。

```text
LocalStack + S3 backend + terraform_remote_state
```

并且备注说明：

```text
使用 terraform_data 模拟跨团队输出值，不创建真实 AWS 资源。
```

测试/验证重点是：

- network state 是否写入 S3

- 销毁顺序是否正确

- 不依赖真实 AWS 资源

这类测试非常适合训练跨配置依赖。

## 29. 常见坑

### 坑 1：把 remote state 当 module

`terraform_remote_state` 读取的是另一个 state 的 output。  
它不是 module 调用。

---

### 坑 2：network 没 apply

consumer 读取不到 output。

---

### 坑 3：output 没声明

remote state 只能读取 output。  
没有 output 就读不到。

---

### 坑 4：state key 写错

consumer 必须指向 network 的 state key。

---

### 坑 5：销毁顺序错

应该先 destroy consumer，再 destroy network。

---

### 坑 6：暴露敏感 output

不要随便通过 remote state 共享 secret。

---

### 坑 7：以为 producer 配置变了 consumer 就自动读到

只有 producer apply 后 state output 更新，consumer 才能读到新值。

---

## 30. 生产建议

如果生产里使用 remote state，建议：

1. output 当 API 设计。
2. output 名称稳定。
3. output 类型稳定。
4. output 文档化。
5. 不暴露敏感 secret。
6. consumer 只读必要 output。
7. state backend 权限严格控制。
8. CI/CD 管理 apply 顺序。
9. 销毁时按反向依赖顺序。
10. 变更 output 要做影响评估。
11. 不要让太多 consumer 直接依赖同一个 state。
12. 大规模依赖可考虑服务目录或配置平台替代。

remote state 很方便，但不是没有代价。

## 32. 和 CI/CD 的关系

在 CI/CD 中，remote state 依赖需要明确 pipeline 顺序。

例如：

```text
network pipeline:
  terraform apply
  输出 public_ip 到 state

consumer pipeline:
  terraform plan/apply
  通过 terraform_remote_state 读取 public_ip
```

如果 network 变更了 output，但 consumer pipeline 没重新跑，consumer 仍然可能使用旧值。

成熟做法：

- producer apply 后触发 consumer plan
- consumer plan 显示 remote state 变化影响
- 变更需要 review
- output 破坏性变更要公告
- 多团队依赖要有版本/变更流程

Terraform 不会自动替你做跨 repo 编排。

## 33. SRE / Platform Engineering 视角

从平台工程角度看，remote state 是一种轻量级跨团队接口。

它适合共享：

- VPC ID
- subnet IDs
- security group IDs
- public IP
- load balancer DNS
- cluster endpoint
- account IDs
- region 配置
- 非敏感基础设施元数据

但它也会带来耦合：

```text
consumer 知道 producer state 的位置
consumer 依赖 producer output 名称
consumer 需要读 producer state 权限
```

所以不要滥用。

如果依赖越来越复杂，可以考虑：

- service catalog
- configuration registry
- parameter store
- API
- platform control plane
- Terraform Cloud workspace outputs
- HCP Terraform run triggers

remote state 是工具，不是万能架构。

## 34. 面试/考试记忆版

可以这样记：

- `terraform_remote_state` 是 Terraform 内置 data source。
- 它读取另一个 Terraform state 中声明的 output。
- 它不能随便读取对方 state 中所有内部资源属性。
- producer 必须先 apply，state 中才有 output。
- consumer 通过 `data.terraform_remote_state.xxx.outputs.name` 读取。
- apply 顺序：先 producer/network，再 consumer。
- destroy 顺序：先 consumer，再 producer/network。
- remote state 会引入跨 state 耦合。
- 不要通过 remote state 暴露敏感 secret。
- consumer 不会自动 apply producer。
- state key 要分开，不能多个 root module 共用同一个 key。
- LocalStack 可以模拟 S3 backend 做本地实验。

一句话总结：

> `terraform_remote_state` 的核心价值，是让一个 Terraform 配置读取另一个配置暴露的 output，从而实现跨团队、跨 state 的基础设施数据共享。

## 35. 本 Lab 的价值

第 78 节的价值在于，它让你从“单个 Terraform 项目”进入“多 Terraform 配置协作”。

真实平台工程里，network、security、app、monitoring 很可能不是同一个 state。  
它们之间需要共享有限的输出值。

这节训练的不是一个小语法，而是一个工程模式：

```text
producer 暴露 output
consumer 读取 remote state
CI/CD 管理顺序
团队维护 output 契约
```

记住一句话：

> Remote state 不是偷看别人 state 的后门，而是通过 output 建立跨配置契约的方式。
