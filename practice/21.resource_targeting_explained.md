# Terraform 实操训练 21：Resource Targeting 基础讲解

## 1. 这一课在学什么

本 Lab 讲的是 Terraform 的 **resource targeting**，也就是通过 `-target` 参数，只针对某一个资源地址执行 `plan`、`apply` 或 `destroy`。

课程里的核心命令是：

```sh
terraform plan -input=false -no-color -target=local_file.release_note -out=tfplan
terraform apply -auto-approve tfplan
```

这代表：

> 这次变更只关注 `local_file.release_note` 这个资源，不按默认方式计算整个目录里的所有资源。

Terraform 正常情况下会把当前目录下所有 `.tf` 文件合并起来，形成完整的 dependency graph，然后计算整体变更。`-target` 是一种“打断默认完整图计算”的方式，让 Terraform 只处理你指定的资源以及它必须依赖的上游资源。

所以这不是普通部署手段，而是一个 **局部修复工具**。

---

## 2. 为什么需要 `-target`

假设一个 Terraform 目录里有两个本地文件资源：

- `local_file.release_note`
- `local_file.runbook`

默认执行：

```sh
terraform plan
```

Terraform 会检查整个目录里的所有资源。如果两个资源都还没创建，它会计划创建两个文件。

但本实验的场景是：

> 运维只想先创建 release note，不想碰 runbook。

这时候可以用：

```sh
terraform plan -target=local_file.release_note -out=tfplan
```

这样 Terraform 会只计划 `local_file.release_note` 相关的变更。

这就像你本来要装修整套房子，但今天只想先修厨房水龙头。`-target` 就是告诉 Terraform：

> 今天别管全屋装修，先盯着这个水龙头。

---

## 3. `-target` 的资源地址是什么

`-target` 后面要写的是 **Terraform 资源地址**。

本实验里是：

```sh
-target=local_file.release_note
```

它由两部分组成：

```text
资源类型.资源名称
```

对应关系大概是：

```hcl
resource "local_file" "release_note" {
  # ...
}
```

所以：

```text
local_file.release_note
```

就是这个资源在 Terraform 里的地址。

如果资源在 module 里，地址会更长，例如：

```text
module.app.local_file.release_note
```

如果用了 `count` 或 `for_each`，地址还会带索引或 key，例如：

```text
local_file.release_note[0]
local_file.release_note["prod"]
```

在真实项目里，如果你不确定资源地址，可以用：

```sh
terraform state list
```

查看当前 state 里已经存在的资源地址。

## 4. 本 Lab 的完整执行流程

进入实验目录：

```sh
cd /workspace/practice/labs/21
```

初始化 provider 和 backend：

```sh
terraform init -input=false
```

检查格式：

```sh
terraform fmt -check
```

检查配置语法和 provider schema：

```sh
terraform validate
```

运行 Terraform 原生测试：

```sh
terraform test
```

生成 targeted plan：

```sh
terraform plan -input=false -no-color -target=local_file.release_note -out=tfplan
```

执行刚才保存下来的 plan：

```sh
terraform apply -auto-approve tfplan
```

查看输出：

```sh
terraform output
```

最后清理：

```sh
terraform destroy -auto-approve
```

## 5. 为什么这里还要配合 `-out=tfplan`

本 Lab 里不是直接执行：

```sh
terraform apply -target=local_file.release_note
```

而是先：

```sh
terraform plan -target=local_file.release_note -out=tfplan
```

再：

```sh
terraform apply tfplan
```

这和前面 Lab 16 的保存 plan 思路是一致的。

好处是：

1. 先生成计划，方便审查。
2. 审查过什么，就执行什么。
3. 避免审批后重新计算 plan，导致执行内容变化。

在 CI/CD 里，更推荐这种模式：

```text
plan 阶段生成 tfplan
审批阶段审查 tfplan
apply 阶段执行同一个 tfplan
```

也就是说，`-target` 本身已经是一个危险工具了，再配合保存 plan，至少能让操作更可追踪。

## 6. `-target` 到底会不会处理依赖

会，但只处理必要依赖。

如果目标资源依赖上游资源，Terraform 可能会把上游资源也纳入计划。比如：

```hcl
resource "local_file" "config" {
  filename = "config.txt"
  content  = "hello"
}

resource "local_file" "release_note" {
  filename = "release-note.txt"
  content  = local_file.config.content
}
```

如果执行：

```sh
terraform plan -target=local_file.release_note
```

Terraform 可能需要先考虑 `local_file.config`，因为 `release_note` 依赖它。

但是反过来不一定。

如果有一个下游资源依赖 `release_note`，例如：

```hcl
resource "local_file" "runbook" {
  content = local_file.release_note.content
}
```

你 target `release_note` 时，Terraform 不一定会自动创建 `runbook`。

这就是 `-target` 的关键风险：

> 它可能只照顾目标资源和必要上游依赖，不保证整个系统状态完整一致。

## 7. 为什么生产环境不能长期依赖 `-target`

`-target` 很像“手术刀”。

手术刀可以救命，但不能天天拿来吃饭。

生产里可以在这些场景使用：

- 某个关键资源坏了，需要单独修复。
- 某个资源 apply 失败，需要分阶段恢复。
- 大规模变更里，需要先处理一个阻塞资源。
- 紧急事故中，需要临时缩小 Terraform 操作范围。

但不应该长期这样用：

```sh
terraform apply -target=xxx
terraform apply -target=yyy
terraform apply -target=zzz
```

如果团队每次部署都靠 target，说明 Terraform 代码结构可能有问题。

常见根因包括：

1. 模块边界太大，一个目录管了太多东西。
2. 依赖关系写得不清楚。
3. state 拆分不合理。
4. 部署流程没有分层。
5. 团队害怕完整 plan，因为变更不可控。

这时候正确做法不是继续 target，而是重构：

- 拆 module。
- 拆 state。
- 明确依赖边界。
- 分环境管理。
- 建立 plan review 流程。

## 8. 看到 targeting warning 正常吗

正常。

Terraform 使用 `-target` 时通常会提醒你：

> Resource targeting is in effect.

大意是：

> 你现在做的不是完整 plan，结果可能不代表整个配置的最终状态。

这不是报错，而是警告。

它是在提醒你：

> 别把这次 targeted plan 当成完整系统审查。

完成 targeted 操作后，真实项目里建议再跑一次完整 plan：

```sh
terraform plan
```

确认整个系统没有遗留 drift 或未应用变更。

## 9. 本 Lab 的验证结果怎么理解

实验验证结果是：

```text
terraform test: 1 passed, 0 failed
```

并且额外验证 targeted workflow：

```text
terraform plan -target=local_file.release_note
```

只创建：

```text
release-note.txt
```

不会创建：

```text
runbook.txt
```

这正好证明了 `-target` 的行为：

> 它确实能缩小变更范围。

但也同时说明：

> 它不会帮你保证整个目录里的所有资源都进入目标状态。

所以这节课真正想让你记住的不是“target 很方便”，而是：

> target 可以救急，但它绕开了完整计划，所以要谨慎使用。

## 10. 常见错误和排障

### 10.1 target 地址写错

错误示例：

```sh
terraform plan -target=release_note
```

Terraform 不认识这个地址。

应该写完整资源地址：

```sh
terraform plan -target=local_file.release_note
```

如果资源已经在 state 中，可以用：

```sh
terraform state list
```

确认地址。

---

### 10.2 target 后发现有些资源没创建

这不是 bug。

`-target` 本来就不是完整部署，它只关注目标资源和必要依赖。

如果你想创建全部资源，就不要加 `-target`：

```sh
terraform plan
terraform apply
```

---

### 10.3 target 后出现 warning

这是预期行为。

warning 的意思是：

> 你这次只计算了部分图，不要把它当成完整系统状态。

处理方式：

1. 确认这次确实需要局部操作。
2. apply 后再执行完整 plan。
3. 如果长期需要 target，考虑重构代码结构。

---

### 10.4 destroy 时是否也可以 target

可以，例如：

```sh
terraform destroy -target=local_file.release_note
```

但风险更高。

因为局部 destroy 可能留下依赖它的资源，导致系统进入奇怪状态。

真实环境里，targeted destroy 要非常谨慎，尤其是网络、IAM、Kubernetes、数据库这类资源。

## 11. 清理命令

实验结束后执行：

```sh
terraform destroy -auto-approve
rm -f tfplan
rm -rf output .terraform .terraform.lock.hcl
```

说明：

- `terraform destroy -auto-approve`：删除 Terraform 创建的资源。
- `rm -f tfplan`：删除保存的 plan 文件。
- `rm -rf output`：删除本地输出目录。
- `rm -rf .terraform .terraform.lock.hcl`：清理初始化缓存和 lock file。

## 12. 面试/工作中怎么说

可以这样表达：

> Terraform `-target` allows partial planning or applying for specific resource addresses. It is useful for exceptional recovery scenarios, but it should not be used as a normal deployment workflow because it bypasses the full dependency graph evaluation. After a targeted apply, I would normally run a full `terraform plan` to verify that the overall state is still consistent.

中文理解：

> `-target` 可以局部操作某个资源，适合故障恢复或临时修复，但不能当成日常发布方式。target 后最好再跑完整 plan，确认整体状态没问题。

---

## 13. 本课一句话总结

`-target` 是 Terraform 的局部操作工具，适合救急，不适合长期部署；真正成熟的 IaC 流程应该依赖清晰模块边界、完整 plan review 和可预测的 state 管理。
