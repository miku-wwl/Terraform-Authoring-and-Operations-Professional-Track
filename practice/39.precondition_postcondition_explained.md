# Terraform 实操训练 39 讲解：Precondition 与 Postcondition

## 1. 这一节到底在学什么？

Lab 39 学的是 Terraform 里的**资源级约束**：`precondition` 和 `postcondition`。

前面 Lab 36、37、38 主要讲的是变量验证，也就是在变量入口处挡住错误输入。  
这一节更进一步：把检查写到资源或 data source 的 `lifecycle` 里，让 Terraform 在资源评估前后做更贴近资源语义的检查。

核心概念：

- `precondition`：资源评估前检查，不满足就阻止继续执行。
- `postcondition`：资源评估后检查，用来确认资源结果满足预期。
- 两者都写在 `lifecycle` 块里。

一句话理解：

> `precondition` 是“你这个请求能不能进门”；`postcondition` 是“执行后的结果是不是合格”。

---

## 2. 为什么有了变量 validation，还需要 precondition？

变量 `validation` 很适合检查输入值，比如：

- 环境名是否合法
- 密码长度够不够
- instance size 是否在白名单里
- 命名是否符合正则

但变量验证有一个限制：它主要站在“变量入口”的角度看问题。

有时候，约束并不只是某个变量本身，而是跟资源、组合逻辑、计算结果有关。

例如：

```hcl
var.environment == "dev" && var.instance_size == "xlarge"
```

单看 `environment = dev` 没问题。  
单看 `instance_size = xlarge` 可能也没问题。  
但组合起来就有问题：开发环境不允许用大规格。

这种情况下，用资源级 `precondition` 表达会更贴近业务含义。

---

## 3. Precondition 是什么？

`precondition` 在 Terraform 评估资源之前执行。  
它适合用来阻止不合规请求。

典型结构：

```hcl
resource "terraform_data" "compute_request" {
  input = {
    environment   = var.environment
    instance_size = var.instance_size
  }

  lifecycle {
    precondition {
      condition     = !(var.environment == "dev" && var.instance_size == "xlarge")
      error_message = "Development environment cannot use xlarge instance size."
    }
  }
}
```

这段逻辑的意思是：

> 如果环境是 dev，就不允许使用 xlarge。

如果条件失败，Terraform 会在执行前直接报错，不会继续处理这个资源。

---

## 4. Postcondition 是什么？

`postcondition` 在对象评估之后执行。  
它适合检查资源、data source 或计算结果是否符合预期。

典型结构：

```hcl
resource "terraform_data" "compute_request" {
  input = {
    environment   = var.environment
    instance_size = var.instance_size
  }

  lifecycle {
    postcondition {
      condition     = self.output.instance_size == var.instance_size
      error_message = "Computed instance size does not match requested size."
    }
  }
}
```

这里的重点是 `self`。

`self` 代表当前资源评估后的对象。  
所以 postcondition 可以拿资源结果做判断。

简单说：

> precondition 多看输入请求；postcondition 多看资源结果。

---

## 5. 本实验的场景

Lab 39 的背景是：开发环境只允许小规格计算请求。

问题是：如果有人在开发环境申请大规格，例如 `xlarge`，就会带来成本风险。

实验任务是：

1. 在资源中添加 `precondition`。
2. 在资源中添加 `postcondition`。
3. 默认 `small` 规格通过。
4. `xlarge` 规格被拒绝。

这不是为了创建真实云资源，而是为了训练 Terraform 的资源级约束写法。

---

## 6. Precondition 和 Postcondition 的核心区别

| 对比项 | Precondition | Postcondition |
|---|---|---|
| 执行时机 | 资源评估前 | 资源评估后 |
| 主要用途 | 阻止不合规请求 | 验证结果是否符合预期 |
| 常见检查对象 | 变量、local、输入组合 | `self`、data source 结果、资源输出 |
| 失败含义 | 请求本身不应该继续 | 执行结果不满足约束 |
| 类比 | 门口安检 | 出厂质检 |

简单记忆：

> Pre 是 before，Post 是 after。

---

## 7. 为什么二者都写在 lifecycle 里？

Terraform 把 `precondition` 和 `postcondition` 都放在 `lifecycle` 中，是因为它们和资源生命周期强相关。

资源生命周期不是只有创建和删除，还包括：

- 评估前能不能继续
- 创建或读取后结果是否合格
- 替换策略
- 防止销毁
- 忽略某些变更

所以 `lifecycle` 不只是 `prevent_destroy` 和 `ignore_changes`，也可以表达资源级质量门禁。

## 9. 如何手动验证 precondition 失败？

可以故意传入一个不允许的规格，例如：

```sh
terraform plan -var='instance_size=xlarge'
```

如果实验配置限制开发环境只能使用小规格，那么预期结果应该是 plan 失败，并显示自定义错误信息。

这说明 `precondition` 生效了。

---

## 10. 如何理解 postcondition 失败？

`postcondition` 通常用于这种场景：

- data source 查到的资源必须符合某个 tag
- 远端返回的对象必须处于某个状态
- 资源计算后的属性必须满足约束
- 输出结果不能为空
- 资源结果必须落在允许范围内

例如读取已有资源时：

```hcl
data "aws_instance" "selected" {
  filter {
    name   = "tag:Name"
    values = ["app-server"]
  }

  lifecycle {
    postcondition {
      condition     = self.instance_state == "running"
      error_message = "Selected EC2 instance must be running."
    }
  }
}
```

这就不是检查输入变量了，而是在检查“查出来的结果”。

## 11. 什么时候用 validation，什么时候用 precondition，什么时候用 postcondition？

可以按这个思路判断：

### 用变量 validation

当规则只跟单个变量强相关时，用 `validation`。

例如：

```hcl
variable "environment" {
  validation {
    condition     = contains(["dev", "test", "prod"], var.environment)
    error_message = "environment must be dev, test, or prod."
  }
}
```

适合：

- 命名规范
- 密码长度
- 环境枚举
- region 白名单
- instance type 白名单

---

### 用 precondition

当规则跟资源请求或多个输入组合有关时，用 `precondition`。

例如：

```hcl
precondition {
  condition     = !(var.environment == "dev" && var.instance_size == "xlarge")
  error_message = "Dev environment cannot use xlarge."
}
```

适合：

- dev 环境不能使用大规格
- prod 环境必须开启高可用
- 某种部署模式必须搭配某个配置
- 资源创建前必须满足某个策略

---

### 用 postcondition

当你要检查资源或 data source 评估后的结果时，用 `postcondition`。

例如：

```hcl
postcondition {
  condition     = self.output.environment == var.environment
  error_message = "Output environment does not match requested environment."
}
```

适合：

- 远端返回值校验
- data source 查询结果校验
- 资源属性结果校验
- 确认最终对象符合组织策略

---

## 12. 常见坑

### 坑 1：把所有规则都塞到变量 validation

变量 validation 很好用，但不是所有规则都适合放那里。

如果规则依赖资源结果、data source 结果、或多个变量组合，用 precondition / postcondition 更清楚。

---

### 坑 2：错误信息太抽象

不好的错误信息：

```hcl
error_message = "Invalid."
```

好的错误信息：

```hcl
error_message = "Development environment can only use small instance size."
```

错误信息要告诉用户：

- 哪里错了
- 为什么错了
- 应该怎么修

---

### 坑 3：postcondition 里忘记使用 `self`

postcondition 经常需要检查当前对象结果，所以 `self` 很关键。

比如：

```hcl
condition = self.output.instance_size == "small"
```

这比只看变量更贴近“资源结果是否正确”。

---

### 坑 4：条件写得太复杂

如果条件越来越难读，可以先拆成 `locals`：

```hcl
locals {
  dev_uses_allowed_size = var.environment != "dev" || var.instance_size == "small"
}
```

然后：

```hcl
precondition {
  condition     = local.dev_uses_allowed_size
  error_message = "Development environment can only use small instance size."
}
```

可读性会好很多。

## 13. 和前面变量验证 Lab 的关系

| Lab | 主题 | 主要机制 | 检查位置 |
|---|---|---|---|
| 36 | 输入变量验证基础 | `validation` + `regex` | 变量入口 |
| 37 | 密码长度验证 | `validation` + `length()` | 变量入口 |
| 38 | 多条件变量验证 | `validation` + `contains()` | 变量入口 |
| 39 | Precondition / Postcondition | `lifecycle` 条件 | 资源评估前后 |

这说明 Terraform 的校验能力是分层的：

1. 变量入口先挡一层。
2. 资源评估前再挡一层。
3. 资源评估后再确认一层。

生产模块就应该这样设计，不要全靠人肉 review。

## 14. SRE / Platform Engineering 视角

这个 Lab 看起来只是语法，但它背后的工程意义很强。

平台团队提供 Terraform 模块时，真正要解决的不是“用户能不能创建资源”，而是：

- 用户能不能只创建合规资源
- 成本风险能不能提前挡住
- 错误能不能在 plan 阶段暴露
- 模块能不能自带质量门禁
- 规则能不能沉淀到代码里，而不是靠口头约定

`precondition` 和 `postcondition` 就是把平台规则写进 Terraform 模块的一种方式。

这比 README 写一句“请不要在 dev 用 xlarge”靠谱多了。

README 是建议。  
condition 是拦截。

## 15. 面试/考试记忆版

可以这样记：

- `precondition`：资源评估前检查，用来阻止不合规请求。
- `postcondition`：资源评估后检查，用来验证结果是否合格。
- 两者都写在 `lifecycle` 里。
- `postcondition` 经常使用 `self` 引用当前对象结果。
- 变量 validation 适合检查输入，pre/postcondition 适合检查资源级规则。
- 条件失败应该提供清楚的 `error_message`。

一句话总结：

> `validation` 管输入，`precondition` 管执行前能不能做，`postcondition` 管做完之后结果对不对。

## 16. 本 Lab 的价值

Lab 39 是 Terraform 从“参数校验”走向“资源级治理”的关键一步。

你学会这节后，写模块就不只是堆资源，而是能把平台规则一起封进去：

- dev 只能用小规格
- prod 必须启用高可用
- 查询到的资源必须是 running
- 输出结果不能为空
- 模块行为必须符合组织策略

这类能力对 Terraform Professional、SRE、Platform Engineering 都很实用。它体现的是一种更成熟的 IaC 思维：不是让 Terraform 尽量跑成功，而是让 Terraform 在错误发生前主动拒绝坏配置。
