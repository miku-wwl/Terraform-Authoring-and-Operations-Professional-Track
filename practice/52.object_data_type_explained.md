# Terraform 实操训练 52 讲解：object 数据类型

## 1. 这一节到底在学什么？

Lab 52 学的是 Terraform 里的 `object` 数据类型。

前面两节已经讲了：

- Lab 50：`list`，有顺序的一组值
- Lab 51：`map`，按 key 管理的一组值
- Lab 52：`object`，把多个不同字段组合成一个结构化对象

一句话理解：

> `object` 是 Terraform 里的结构化数据类型，适合表达“一个东西有多个属性”。

例如一个服务不是只有一个字符串，它可能有：

- name
- port
- replicas
- image
- environment
- enabled

这时用 `object` 就比散落多个变量更清楚。

---

## 2. 什么是 object？

Terraform 中的 object 通常这样声明：

```hcl
variable "service" {
  type = object({
    name     = string
    port     = number
    replicas = number
    enabled  = bool
  })

  default = {
    name     = "api"
    port     = 8080
    replicas = 2
    enabled  = true
  }
}
```

这个变量表示一个结构化服务配置。

它不是单个 string，也不是简单 list，而是一个有明确字段的对象。

访问方式：

```hcl
var.service.name
var.service.port
var.service.replicas
var.service.enabled
```

---

## 3. object 的核心价值

`object` 的核心价值是：**把相关字段放在一起，并给每个字段明确类型。**

例如你可以用多个变量：

```hcl
variable "service_name" {
  type = string
}

variable "service_port" {
  type = number
}

variable "service_replicas" {
  type = number
}
```

这能跑，但字段是散的。

如果改成 object：

```hcl
variable "service" {
  type = object({
    name     = string
    port     = number
    replicas = number
  })
}
```

语义就更清楚：

> 这些字段共同描述一个 service。

这就是建模能力。

---

## 4. object 和 map 的区别

Lab 51 讲的 map 是 key-value 集合，例如：

```hcl
{
  dev  = "small"
  prod = "large"
}
```

它适合表达一组同类 key-value。

object 则适合表达一个结构化实体：

```hcl
{
  name     = "api"
  port     = 8080
  replicas = 2
}
```

对比：

| 类型 | 适合表达 | 访问方式 |
|---|---|---|
| map | 一组 key-value | `var.config["dev"]` |
| object | 一个有固定字段的结构 | `var.service.name` |
| map(object) | 多个有名字的结构化对象 | `var.services["api"].port` |
| list(object) | 多个按顺序排列的结构化对象 | `var.services[0].port` |

一句话：

> map 管一组键值，object 管一个结构。

---

## 5. object 的字段类型可以不同

这是 object 和 `map(string)` 的重要区别。

map 通常要求 value 类型一致：

```hcl
map(string)
```

但 object 每个字段可以有不同类型：

```hcl
object({
  name     = string
  port     = number
  enabled  = bool
  tags     = map(string)
  aliases  = list(string)
})
```

这非常适合生产配置。

例如：

```hcl
variable "app" {
  type = object({
    name        = string
    port        = number
    enabled     = bool
    environment = string
    tags        = map(string)
  })
}
```

这种结构比一堆散变量更容易维护。

---

## 6. object 的字段访问

假设：

```hcl
locals {
  service = {
    name     = "payments"
    port     = 8080
    replicas = 3
  }
}
```

访问字段：

```hcl
local.service.name
local.service.port
local.service.replicas
```

也可以用方括号：

```hcl
local.service["name"]
```

一般固定字段用点号更清楚：

```hcl
local.service.name
```

动态字段名才考虑方括号。

---

## 7. object 适合什么生产场景？

object 很适合描述一个完整配置单元。

例如：

### 应用配置

```hcl
object({
  name     = string
  image    = string
  port     = number
  replicas = number
})
```

### 数据库配置

```hcl
object({
  engine         = string
  version        = string
  instance_class = string
  storage_gb     = number
  multi_az       = bool
})
```

### 网络配置

```hcl
object({
  cidr_block = string
  az_count   = number
  nat_enabled = bool
})
```

### 告警配置

```hcl
object({
  threshold = number
  severity  = string
  enabled   = bool
})
```

这些都不是一个简单 string 能表达清楚的。

---

## 8. object 和模块输入

object 在 Terraform module 设计里非常常见。

例如模块需要一个服务配置：

```hcl
module "app" {
  source = "./modules/app"

  service = {
    name     = "api"
    image    = "api:v1"
    port     = 8080
    replicas = 2
  }
}
```

模块内声明：

```hcl
variable "service" {
  type = object({
    name     = string
    image    = string
    port     = number
    replicas = number
  })
}
```

这样模块接口就非常清楚。

调用方知道必须传哪些字段，字段类型也明确。

---

## 9. object 比多个变量更适合表达聚合配置

假设你有这些变量：

```hcl
variable "app_name" {}
variable "app_port" {}
variable "app_image" {}
variable "app_replicas" {}
```

短期看没问题。  
但变量越来越多后，模块接口会变得很散。

改成：

```hcl
variable "app" {
  type = object({
    name     = string
    port     = number
    image    = string
    replicas = number
  })
}
```

优势是：

1. 相关字段聚合在一起。
2. 模块输入更清楚。
3. 类型约束更强。
4. 测试更容易组织。
5. 后续扩展更自然。

这就是 object 的生产价值。

---

## 10. object 和 list(object)

如果你有多个服务，并且顺序有意义，可以用：

```hcl
variable "services" {
  type = list(object({
    name = string
    port = number
  }))

  default = [
    {
      name = "api"
      port = 8080
    },
    {
      name = "worker"
      port = 9090
    }
  ]
}
```

访问：

```hcl
var.services[0].name
var.services[0].port
```

适合：

- 按顺序生成配置
- 顺序有业务意义
- 只是输出或生成文件，不创建长期稳定资源

但如果要用这些服务创建长期资源，要小心 list 索引漂移。

---

## 11. object 和 map(object)

生产里更常见的是：

```hcl
variable "services" {
  type = map(object({
    port     = number
    replicas = number
    image    = string
  }))

  default = {
    api = {
      port     = 8080
      replicas = 2
      image    = "api:v1"
    }

    worker = {
      port     = 9090
      replicas = 1
      image    = "worker:v1"
    }
  }
}
```

访问：

```hcl
var.services["api"].port
var.services["worker"].image
```

配合 `for_each`：

```hcl
resource "local_file" "service" {
  for_each = var.services

  filename = "${path.module}/output/${each.key}.txt"
  content  = "image=${each.value.image}, port=${each.value.port}"
}
```

这就是非常典型的生产 Terraform 建模方式。

一句话：

> 多个有名字的复杂对象，优先考虑 `map(object)`。

---

## 12. object 的可选字段

Terraform 支持 object 里的 optional 字段。

示例：

```hcl
variable "service" {
  type = object({
    name     = string
    port     = number
    replicas = optional(number, 1)
  })
}
```

这里：

```hcl
replicas = optional(number, 1)
```

表示调用方可以不传 `replicas`，默认是 `1`。

这对模块设计很有用，因为并不是每个字段都应该强制用户传。

但是要注意：

> optional 不是随便省字段，而是模块作者明确设计了默认行为。

## 13. object 的校验

object 也可以配合变量 validation。

例如限制端口范围：

```hcl
variable "service" {
  type = object({
    name = string
    port = number
  })

  validation {
    condition     = var.service.port > 0 && var.service.port <= 65535
    error_message = "service.port must be between 1 and 65535."
  }
}
```

也可以限制名称格式：

```hcl
validation {
  condition     = can(regex("^[a-z][a-z0-9-]*$", var.service.name))
  error_message = "service.name must use lowercase letters, numbers, and hyphens."
}
```

这就是 object + validation 的组合价值：

> 不只约束字段类型，还约束字段内容。

## 18. 常见坑

### 坑 1：字段名写错

如果 object 定义里是：

```hcl
name = string
```

你访问：

```hcl
var.service.service_name
```

就会失败。

字段名必须一致。

---

### 坑 2：字段类型不匹配

如果定义：

```hcl
port = number
```

你传：

```hcl
port = "8080"
```

可能会触发类型转换或报错。  
生产里最好按声明类型传值，不要依赖隐式转换。

---

### 坑 3：object 太大

如果一个 object 里塞几十个字段，可能说明模块边界不清楚。

可以考虑拆成多个 object：

```hcl
network_config
runtime_config
monitoring_config
security_config
```

而不是一个巨型 `config` 包天下。

---

### 坑 4：该用 map(object)，却只用了 object

如果你只有一个服务，用 object 没问题。  
如果你要描述多个服务，应该考虑：

```hcl
map(object({...}))
```

否则扩展性会很差。

---

### 坑 5：object key 和业务身份混淆

比如：

```hcl
object({
  name = string
  port = number
})
```

如果这个对象被放进 map：

```hcl
services = {
  api = {
    name = "api"
    port = 8080
  }
}
```

这里 map key 和 `name` 字段可能重复。  
要不要保留两个，要看模块设计。

常见做法是：

- map key 作为稳定资源身份
- object 内部只放配置字段

例如：

```hcl
services = {
  api = {
    port = 8080
  }
}
```

然后资源里用：

```hcl
each.key
each.value.port
```

这样更简洁。

---

## 19. object 设计建议

设计 object 时可以遵循这些原则：

1. 字段名要清楚。
2. 字段类型要明确。
3. 相关字段放一起。
4. 不相关字段不要硬塞进同一个 object。
5. 可选字段要有合理默认值。
6. 关键字段要加 validation。
7. object 不要无限膨胀。
8. 多个同类对象优先考虑 `map(object)`。

好的 object 让模块接口像 API 一样清晰。

## 20. object 和输出

你可以直接输出整个 object：

```hcl
output "service" {
  value = var.service
}
```

也可以输出某个字段：

```hcl
output "service_name" {
  value = var.service.name
}
```

生产里要注意：

- 如果 object 里有 secret，整个 object 可能都需要 sensitive。
- 输出整个 object 可能暴露太多信息。
- 更推荐输出必要字段或非敏感派生值。

比如：

```hcl
output "service_summary" {
  value = "${var.service.name}:${var.service.port}"
}
```

---

## 21. object 和 sensitive

如果 object 里包含敏感字段，例如：

```hcl
variable "db" {
  type = object({
    username = string
    password = string
  })

  sensitive = true
}
```

那整个变量都可以标记为 sensitive。

但要注意：

> sensitive 只是减少 CLI 明文输出，不是加密 state。

这和 Lab 42、Lab 43、Lab 44 是连起来的。

如果 object 里有 password、token、key，就要特别小心 state 和 output。

## 22. object 和生产模块接口

Terraform 模块接口如果设计得好，调用方会很舒服。

差的接口：

```hcl
app_name = "api"
app_port = 8080
app_image = "api:v1"
app_replicas = 2
app_cpu = "500m"
app_memory = "512Mi"
```

好的接口：

```hcl
app = {
  name     = "api"
  image    = "api:v1"
  port     = 8080
  replicas = 2
  resources = {
    cpu    = "500m"
    memory = "512Mi"
  }
}
```

后者更像一个结构化 API。

这就是 object 的价值。

## 23. 和 list / map 的关系

| 类型 | 核心特点 | 典型用途 |
|---|---|---|
| list | 有序集合 | 多个按顺序处理的值 |
| map | key-value 集合 | 按名字查配置 |
| object | 固定字段结构 | 描述一个复杂对象 |
| list(object) | 有序对象列表 | 多个按顺序排列的对象 |
| map(object) | 命名对象集合 | 多个长期资源配置 |

你可以这样记：

- 一个复杂对象：`object`
- 多个有顺序复杂对象：`list(object)`
- 多个有名字复杂对象：`map(object)`

## 24. SRE / Platform Engineering 视角

从平台工程角度看，object 是模块接口设计的核心工具。

因为平台模块不是只接收简单字符串，而是接收结构化需求：

- 一个服务怎么运行
- 一个数据库怎么配置
- 一个环境有什么策略
- 一个告警规则有什么阈值
- 一个网络单元有什么 CIDR 和开关
- 一个应用有什么镜像、端口、副本数和资源限制

这些都适合 object。

如果你未来做 AKS/EKS、平台工程、SRE 项目，`object` 和 `map(object)` 会非常高频。

## 25. 面试/考试记忆版

可以这样记：

- `object` 是固定字段结构。
- 每个字段可以有不同类型。
- 字段用 `.field` 访问。
- object 适合描述一个复杂配置单元。
- 多个对象按顺序：`list(object)`。
- 多个对象按名字：`map(object)`。
- 生产里 `map(object)` 很适合配合 `for_each`。
- object 字段可以配合 validation 做内容约束。
- object 里有 secret 时，要注意 sensitive 和 state 风险。

一句话总结：

> Terraform object 的核心价值是把散乱变量变成结构化配置，让模块接口更像稳定 API。
