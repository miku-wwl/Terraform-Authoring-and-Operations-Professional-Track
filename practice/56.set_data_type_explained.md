# Terraform 实操训练 56 讲解：set 数据类型

## 1. 这一节到底在学什么？

Lab 56 学的是 Terraform 里的 `set` 数据类型。

前面几节已经讲过：

- `list`：有顺序的一组值
- `map`：按 key 管理的一组值
- `object`：有固定字段的结构化对象
- 嵌套类型：把 list、map、object 组合起来
- `count` 的索引挑战：list + count 容易因为索引变化引发资源地址漂移

这一节的 `set` 是另一种集合类型。

一句话理解：

> `set` 是 Terraform 里的无序去重集合，适合表达“这些成员存在即可，顺序不重要”。

---

## 2. 什么是 set？

Terraform 中的 set 通常这样声明：

```hcl
variable "allowed_regions" {
  type = set(string)

  default = [
    "us-east-1",
    "ap-southeast-2",
    "eu-west-1"
  ]
}
```

虽然 default 看起来像 list 语法，但类型声明是：

```hcl
set(string)
```

Terraform 会把它当成 set 处理。

set 的核心特点：

1. 无序。
2. 自动去重。
3. 只关心成员是否存在。
4. 不适合用数字索引访问。
5. 适合白名单、功能开关集合、去重输入。
6. 可以和 `for_each` 搭配使用。

---

## 3. set 和 list 的核心区别

`list` 是有序的：

```hcl
["api", "worker", "frontend"]
```

顺序有意义：

```hcl
var.services[0]
var.services[1]
```

`set` 是无序的：

```hcl
toset(["api", "worker", "frontend"])
```

它关心的是：

```text
api 是否存在
worker 是否存在
frontend 是否存在
```

而不是谁排第一个、谁排第二个。

对比：

| 类型 | 是否有序 | 是否允许重复 | 访问方式 | 适合场景 |
|---|---|---|---|---|
| list | 有序 | 允许重复 | `[index]` | 顺序有意义 |
| set | 无序 | 自动去重 | 成员判断 / 遍历 | 成员关系、去重 |
| map | key-value | key 唯一 | `["key"]` | 命名对象 |

一句话：

> list 管顺序，set 管成员，map 管名字。

---

## 4. set 会自动去重

这是 set 最重要的特点之一。

例如：

```hcl
locals {
  regions = toset([
    "us-east-1",
    "us-east-1",
    "ap-southeast-2"
  ])
}
```

Terraform 会把重复的 `"us-east-1"` 去掉。

最终逻辑上相当于：

```hcl
toset([
  "us-east-1",
  "ap-southeast-2"
])
```

这对输入去重很有用。

例如用户传了一堆允许 CIDR、region、feature flag，其中有重复项，set 可以自然去重。

---

## 5. set 是无序的

set 不保证顺序。

这意味着你不应该写：

```hcl
var.allowed_regions[0]
```

因为 set 不适合索引访问。

如果你确实需要索引，说明你其实需要的是 list，而不是 set。

如果你必须把 set 转成 list，可以用：

```hcl
tolist(var.allowed_regions)
```

但要注意：  
转成 list 后的顺序不应该被当成业务逻辑依赖。

如果顺序重要，一开始就用 list。

---

## 6. `toset()` 是什么？

`toset()` 可以把 list 转成 set。

示例：

```hcl
locals {
  unique_services = toset(["api", "api", "worker"])
}
```

结果是一个去重集合：

```hcl
toset([
  "api",
  "worker"
])
```

常见用途：

- 去重 list
- 把 list 转给 `for_each`
- 明确表示只关心成员，不关心顺序

例如：

```hcl
for_each = toset(var.enabled_features)
```

这表示每个 feature 都作为一个独立 key 来处理。

---

## 7. set 和 `contains()`

set 可以配合 `contains()` 判断成员是否存在。

```hcl
variable "allowed_environments" {
  type = set(string)

  default = ["dev", "test", "prod"]
}

output "is_prod_allowed" {
  value = contains(var.allowed_environments, "prod")
}
```

返回：

```text
true
```

这和 Lab 38 的白名单验证有关。  
白名单可以用 list，也可以用 set。

如果你不关心顺序，而且不允许重复，set 更贴近语义。

---

## 8. set 和 `for_each`

set 可以直接用于 `for_each`，尤其是 `set(string)`。

示例：

```hcl
variable "features" {
  type    = set(string)
  default = ["logging", "metrics", "tracing"]
}

resource "local_file" "feature" {
  for_each = var.features

  filename = "${path.module}/output/${each.key}.txt"
  content  = "feature=${each.value}"
}
```

对于 set(string)，`each.key` 和 `each.value` 通常相同。

资源地址会是：

```text
local_file.feature["logging"]
local_file.feature["metrics"]
local_file.feature["tracing"]
```

这比 `count` 的 `[0]`、`[1]` 更稳定。

---

## 9. set 为什么适合 for_each？

Lab 55 讲过 `count` 的索引挑战。  
`count` 用数字索引：

```text
resource.example[0]
resource.example[1]
```

set + for_each 用成员值作为 key：

```text
resource.example["logging"]
resource.example["metrics"]
```

如果删除 `"metrics"`，只影响：

```text
resource.example["metrics"]
```

不会因为 list 中间元素删除导致后续索引漂移。

所以对于一组唯一字符串成员，`set(string)` + `for_each` 很自然。

---

## 10. set 不适合表达复杂对象身份

虽然 Terraform 支持更复杂的 set 类型，比如 `set(object(...))`，但生产里要慎重。

如果对象有稳定名字，更常用的是：

```hcl
map(object({...}))
```

而不是：

```hcl
set(object({...}))
```

原因是 map 的 key 更明确、更稳定、更适合作为 Terraform 资源地址。

例如服务配置：

```hcl
services = {
  api = {
    port = 8080
  }

  worker = {
    port = 9090
  }
}
```

比：

```hcl
services = toset([
  {
    name = "api"
    port = 8080
  },
  {
    name = "worker"
    port = 9090
  }
])
```

更适合生产资源建模。

一句话：

> set 适合简单唯一成员；复杂命名对象优先用 map(object)。

## 15. set 的常见函数组合

### `toset()`

把 list 转成 set：

```hcl
toset(["a", "a", "b"])
```

---

### `tolist()`

把 set 转成 list：

```hcl
tolist(var.enabled_features)
```

注意不要依赖顺序。

---

### `contains()`

判断成员存在：

```hcl
contains(var.enabled_features, "metrics")
```

---

### `setunion()`

合并多个 set：

```hcl
setunion(var.base_features, var.extra_features)
```

---

### `setintersection()`

取交集：

```hcl
setintersection(var.enabled_features, var.supported_features)
```

---

### `setsubtract()`

取差集：

```hcl
setsubtract(var.requested_features, var.supported_features)
```

这类函数在治理和校验场景很有用。

## 16. set 和变量验证

set 很适合配合 validation。

例如用户传入 requested features，必须都是 supported features 的子集：

```hcl
variable "requested_features" {
  type = set(string)

  validation {
    condition = length(setsubtract(var.requested_features, toset([
      "logging",
      "metrics",
      "tracing"
    ]))) == 0

    error_message = "requested_features contains unsupported feature names."
  }
}
```

这段逻辑是：

1. 用 `setsubtract()` 找出 requested 里不被支持的成员。
2. 如果差集长度是 0，说明全部合法。
3. 如果差集不为空，说明有非法 feature。

这就是 set 在治理里的高级用法。

---

## 17. set 和 list 的转换要小心

有时你会看到：

```hcl
tolist(var.my_set)[0]
```

这通常是危险信号。

因为 set 无序。  
你把它转成 list 再取 `[0]`，得到的“第一个元素”不应该承载业务含义。

如果你真的需要第一个、第二个、第三个，应该用 list。

如果你只是想遍历所有成员，可以用：

```hcl
for item in var.my_set : item
```

或者直接：

```hcl
for_each = var.my_set
```

---

## 18. set 的输出顺序问题

Terraform 输出 set 时，显示顺序可能不是你写入的顺序。

例如你写：

```hcl
["prod", "dev", "test"]
```

输出可能显示为：

```hcl
[
  "dev",
  "prod",
  "test"
]
```

不要把这个当 bug。

set 本来就不表达顺序。

如果输出顺序重要，可以显式排序：

```hcl
sort(tolist(var.my_set))
```

但排序后的 list 是为了展示或稳定输出，不代表 set 本身有顺序。

---

## 19. set 和 state

set 的状态表示和 list 不一样。

list 关心顺序：

```hcl
["a", "b", "c"]
```

改成：

```hcl
["c", "b", "a"]
```

对 list 来说顺序变了。

set 不关心顺序：

```hcl
toset(["a", "b", "c"])
```

和：

```hcl
toset(["c", "b", "a"])
```

逻辑上是同一组成员。

这可以减少因为顺序变化引起的无意义 diff。

但前提是：你的业务确实不关心顺序。

---

## 20. set 和 count 的关系

一般不建议把 set 直接转 list 后配 count：

```hcl
count = length(var.my_set)
name  = tolist(var.my_set)[count.index]
```

因为 set 无序，转 list 后的顺序不应该被依赖。

如果要为 set 中每个成员创建资源，更推荐：

```hcl
for_each = var.my_set
```

这样资源地址使用成员值，而不是数字索引。

---

## 21. set 和 map 的关系

set 适合简单成员集合：

```hcl
set(string)
```

map 适合 key-value：

```hcl
map(string)
```

比如：

```hcl
enabled_features = toset(["logging", "metrics"])
```

只表示这些功能启用了。

如果你需要每个 feature 还有配置：

```hcl
features = {
  logging = {
    retention_days = 30
  }

  metrics = {
    scrape_interval = "30s"
  }
}
```

那就应该用：

```hcl
map(object({...}))
```

而不是 set。

---

## 22. 常见坑

### 坑 1：把 set 当 list 用

错误思路：

```hcl
var.my_set[0]
```

set 不适合索引访问。

---

### 坑 2：依赖 set 输出顺序

set 无序。  
输出顺序变化不代表业务变化。

如果只是为了展示稳定，用：

```hcl
sort(tolist(var.my_set))
```

---

### 坑 3：用 set(object) 管复杂资源

复杂对象如果有业务名字，更推荐 `map(object)`。

---

### 坑 4：把 `toset()` 当成排序工具

`toset()` 是去重和表达集合，不是排序。

排序要用：

```hcl
sort()
```

---

### 坑 5：set 转 list 后配 count

这会把无序集合强行变成索引资源，容易制造不稳定。

能用 `for_each = var.my_set` 就别绕远。

---

## 23. 什么时候选 list、set、map？

可以按这个判断：

### 用 list

当顺序重要时：

```hcl
["step-1", "step-2", "step-3"]
```

### 用 set

当只关心唯一成员，顺序不重要时：

```hcl
toset(["logging", "metrics", "tracing"])
```

### 用 map

当每个成员有名字和对应配置时：

```hcl
{
  api = {
    port = 8080
  }
}
```

一句话：

> 顺序重要用 list，成员去重用 set，命名配置用 map。

## 24. 和 Lab 55 的关系

Lab 55 讲 `count` 的索引挑战。  
Lab 56 讲 set 的无序去重集合。

它们可以这样连起来理解：

- list + count：容易被索引影响。
- set + for_each：适合唯一字符串成员，资源地址更稳定。
- map + for_each：适合复杂命名对象，生产最常用。

如果你只是有一组唯一 feature：

```hcl
toset(["logging", "metrics"])
```

可以用 set。

如果每个 feature 还有配置：

```hcl
features = {
  logging = {
    retention = 30
  }
}
```

用 map object。

Lab 56 的验证报告显示：

```text

```

对 set 来说，测试可以验证：

- for_each 是否按成员创建资源。
- 输出是否不依赖顺序。
- 排序后的展示是否稳定。

这能避免你误把 set 当成 list 用。

## 26. SRE / Platform Engineering 视角

从平台工程角度看，set 适合表达“策略集合”。

例如：

- 允许的环境集合
- 启用的 feature 集合
- 支持的 region 集合
- 告警接收人集合
- 安全扫描规则集合
- 允许的部署模式集合

这些东西的共同点是：

> 成员是否存在很重要，顺序通常不重要。

这就是 set 的位置。

但是平台资源配置通常更复杂。  
一旦每个成员有独立配置，就要升级成 map(object)。

这就是数据建模判断能力。

可以这样记：

- set 是无序集合。

- set 不适合索引访问。
- `toset()` 可以把 list 转成 set。

一句话总结：

> Terraform set 的核心价值是表达“唯一成员集合”，不是表达顺序，也不是表达复杂对象配置。

## 28. 本 Lab 的价值

Lab 56 的价值在于，它补齐了 Terraform 三大集合类型的最后一块：

- list：有序
- set：无序去重
- map：按 key 管理

很多 Terraform 代码写得不稳，不是资源不会写，而是数据结构选错了。

如果顺序不重要却用了 list，可能产生无意义 diff。  
如果资源有名字却用了 count，可能索引漂移。  
如果复杂对象用了 set，后续维护会痛苦。

学好 set，你就能更准确地判断：

> 这个输入到底是顺序列表、唯一集合，还是命名配置表？
