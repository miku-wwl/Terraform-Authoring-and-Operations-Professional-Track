# Terraform 实操训练 95：模块 `required_providers` 讲解版

## 1. 本节定位

本节主题是 **模块 `required_providers`**。

它看起来只是一个 Terraform 配置块，但在真实工程里非常重要：  
模块不是随便写几个 `.tf` 文件就完事了。一个模块如果依赖某个 provider，就应该明确声明它需要什么 provider、provider 的来源是什么、版本范围是什么。

否则调用方一多，环境一复杂，就容易出现：

- 本地能跑，CI 不能跑；
- A 项目能跑，B 项目不能跑；
- provider 版本悄悄升级后行为变化；
- 模块复用时依赖关系不清楚；
- 报错时不知道到底是 root module 的 provider 问题，还是 child module 的 provider 问题。

所以，本节的核心不是“背 `required_providers` 怎么写”，而是理解：  
**模块应该主动声明自己的 provider 依赖，调用方负责最终安装和配置 provider。**

---

## 2. 一句话讲清楚

`required_providers` 的作用是：

> 告诉 Terraform：这个模块需要哪些 provider，以及这些 provider 应该从哪里来、版本范围是什么。

它解决的是 **模块依赖声明问题**，不是 provider 运行时配置问题。

---

## 3. 先区分两个概念

Terraform 里经常容易把这两个东西混在一起：

```hcl
terraform {
  required_providers {
    local = {
      source  = "hashicorp/local"
      version = "~> 2.5"
    }
  }
}

provider "local" {}
```

这两个块不是一回事。

### 3.1 `required_providers`

它声明的是：

- 需要哪个 provider；
- provider 的 source 地址；
- provider 的版本约束。

例如：

```hcl
terraform {
  required_providers {
    local = {
      source  = "hashicorp/local"
      version = "~> 2.5"
    }
  }
}
```

这等于告诉 Terraform：

> 我这个模块用到了 `local` provider，它来自 `hashicorp/local`，版本最好是 `2.5.x` 这一类兼容版本。

### 3.2 `provider` block

它配置的是 provider 的运行参数。

例如 AWS：

```hcl
provider "aws" {
  region = "ap-southeast-2"
}
```

这等于告诉 Terraform：

> 使用 AWS provider 时，默认 region 是 `ap-southeast-2`。

所以你可以这样记：

| 配置 | 解决什么问题 |
|---|---|
| `required_providers` | 这个模块需要什么 provider |
| `provider` | 这个 provider 运行时怎么连接云平台或服务 |

---

## 4. 为什么模块里也要写 `required_providers`

很多初学者会觉得：

> root module 已经写了 provider，child module 还要写吗？

真实工程里，**推荐 child module 自己声明 `required_providers`**。

原因很简单：模块是一个可复用单元。  
既然它自己用了某个 provider，就应该自己把依赖说清楚。

比如一个模块内部用了 `local_file`：

```hcl
resource "local_file" "example" {
  filename = "${path.module}/output.txt"
  content  = "hello"
}
```

这个资源来自 `hashicorp/local` provider。

那么这个模块内部最好写：

```hcl
terraform {
  required_providers {
    local = {
      source  = "hashicorp/local"
      version = "~> 2.5"
    }
  }
}
```

这样别人调用这个模块时，不需要猜：

- 你到底用了哪个 provider？
- 这个 provider 是官方的还是第三方的？
- 版本范围有没有要求？
- provider 升级会不会炸？

模块把依赖写清楚，调用方就更安心。

---

## 5. root module 和 child module 的职责边界

这里是考试和工程实践都很容易考的点。

### 5.1 child module 应该做什么

child module 应该声明自己需要的 provider：

```hcl
terraform {
  required_providers {
    local = {
      source  = "hashicorp/local"
      version = "~> 2.5"
    }
  }
}
```

它像是在说：

> 我需要 `hashicorp/local` 这个 provider。

### 5.2 root module 应该做什么

root module 负责最终调用模块，并进行 provider 配置：

```hcl
terraform {
  required_providers {
    local = {
      source  = "hashicorp/local"
      version = "~> 2.5"
    }
  }
}

provider "local" {}

module "demo" {
  source = "./modules/demo"
}
```

root module 像是在说：

> 我来决定整个部署入口怎么初始化 provider，怎么安装 provider，怎么配置 provider。

## 6. 一个推荐的模块结构

本节实验路径是：

```text
practice/labs/95
```

一个更贴近真实项目的结构可以是：

```text
practice/labs/95/
├── main.tf
├── outputs.tf
├── variables.tf
├── versions.tf
├── modules/
│   └── file_writer/
│       ├── main.tf
│       ├── outputs.tf
│       ├── variables.tf
│       └── versions.tf
└── tests/
    └── module_required_providers.tftest.hcl
```

其中：

- root module 的 `versions.tf` 声明根模块依赖；
- child module 的 `versions.tf` 声明模块自己的依赖；
- `main.tf` 调用模块；
- `tests/` 放 Terraform test 断言。

---

## 7. 示例：模块内部声明 `required_providers`

假设我们有一个本地模块 `modules/file_writer`。

### 7.1 模块的 `versions.tf`

```hcl
terraform {
  required_version = ">= 1.11.0"

  required_providers {
    local = {
      source  = "hashicorp/local"
      version = "~> 2.5"
    }
  }
}
```

这里表达了两个依赖：

- 这个模块要求 Terraform CLI 至少是 `1.11.0`；
- 这个模块需要 `hashicorp/local` provider。

### 7.2 模块的 `main.tf`

```hcl
resource "local_file" "note" {
  filename = "${path.module}/generated-note.txt"
  content  = var.content
}
```

这个资源明显依赖 `local` provider。

### 7.3 模块的 `variables.tf`

```hcl
variable "content" {
  type        = string
  description = "Content written by the module."
}
```

### 7.4 模块的 `outputs.tf`

```hcl
output "file_path" {
  value       = local_file.note.filename
  description = "Generated file path."
}
```

---

## 8. root module 如何调用它

root module 里可以这样写：

```hcl
module "file_writer" {
  source  = "./modules/file_writer"
  content = "hello from root module"
}

output "generated_file_path" {
  value = module.file_writer.file_path
}
```

执行时，Terraform 会分析 root module 和 child module 里的 provider requirement，然后在 `terraform init` 时完成 provider 安装和锁定。

---

## 9. `terraform init` 在这里做了什么

本节运行手册第一步是：

```sh
terraform init -input=false
```

这一步不是“走流程”，它很关键。

它会做几件事：

1. 读取 root module 的配置；
2. 读取 child module 的配置；
3. 汇总所有模块里的 `required_providers`；
4. 根据版本约束选择合适 provider 版本；
5. 下载 provider plugin；
6. 生成或更新 `.terraform.lock.hcl`。

如果模块没有正确声明 provider，`init` 或后续 `validate/test/plan` 就可能暴露问题。

---

## 10. `.terraform.lock.hcl` 和 `required_providers` 的关系

两者也不能混。

| 文件/配置 | 作用 |
|---|---|
| `required_providers` | 声明允许使用的 provider 来源和版本范围 |
| `.terraform.lock.hcl` | 锁定本次实际选中的 provider 版本和校验信息 |

举个例子：

```hcl
version = "~> 2.5"
```

这个约束通常允许 `2.5.x` 范围内的兼容版本。

而 lock file 可能最终锁定：

```text
hashicorp/local 2.5.3
```

也就是说：

- `required_providers` 是规则；
- lock file 是实际结果。

真实团队里，这个 lock file 通常应该提交到 Git 仓库。  
否则 CI 和本地环境可能下载到不同 provider patch 版本，带来不可控差异。

---

## 11. 为什么不要在模块里乱写 provider 配置

在 child module 里声明 `required_providers` 是好事。  
但在 child module 里随便写完整 `provider` 配置，通常要谨慎。

例如，不建议模块内部硬编码：

```hcl
provider "aws" {
  region = "ap-southeast-2"
}
```

这会让模块变得不够通用。

如果模块内部强行写 region，那么调用方想部署到：

- `ap-southeast-2`
- `ap-southeast-1`
- `us-east-1`
- `eastus`

就会很尴尬。

更合理的方式是：

- child module 声明自己需要 `aws` provider；
- root module 配置具体 region；
- root module 调用模块时传入需要的变量。

---

## 12. 多 provider 场景下更重要

比如一个模块同时使用：

- `hashicorp/aws`
- `hashicorp/random`
- `hashicorp/local`

那模块的 `versions.tf` 应该类似：

```hcl
terraform {
  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 5.0"
    }

    random = {
      source  = "hashicorp/random"
      version = "~> 3.6"
    }

    local = {
      source  = "hashicorp/local"
      version = "~> 2.5"
    }
  }
}
```

这会让模块依赖非常清楚。  
后面有人接手代码，一看就知道这个模块用了什么 provider，不用靠猜。

---

## 13. 版本约束怎么写更稳

常见写法有几种：

```hcl
version = ">= 2.5.0"
```

表示至少使用 `2.5.0`。

```hcl
version = "~> 2.5"
```

表示允许 `2.5.x` 这类兼容版本。

```hcl
version = ">= 2.5.0, < 3.0.0"
```

表示允许 `2.x` 中符合条件的版本，但不要升级到 `3.x`。

真实工程里，比较稳的写法通常是：

```hcl
version = ">= 2.5.0, < 3.0.0"
```

或者：

```hcl
version = "~> 2.5"
```

核心思想是：

> 不要完全不限制版本，也不要过度锁死到一个 patch 版本。

patch 版本最终让 `.terraform.lock.hcl` 去锁。

```sh
cd /workspace/practice/labs/95

```

### 14.1 进入实验目录

```sh
cd /workspace/practice/labs/95
```

进入第 95 个 lab 的 Terraform root module。

### 14.2 初始化

```sh
terraform init -input=false
```

重点观察：

- 是否识别到本地模块；
- 是否下载 provider；
- 是否生成 `.terraform.lock.hcl`；
- 是否出现 provider source 或 version 相关报错。

### 14.3 格式检查

```sh
terraform fmt -check
```

这一步检查 HCL 格式是否标准。  
CI 里经常会这样卡格式，避免代码风格乱飞。

如果失败，可以本地运行：

```sh
terraform fmt -recursive
```

### 14.4 静态校验

```sh
terraform validate
```

这一步检查 Terraform 配置是否合法。

对于本节，重点关注：

- 模块路径是否正确；
- provider 是否能解析；
- output/variable 引用是否正确；
- 资源类型是否存在。

### 14.5 运行测试

```sh
terraform test
```

本节验证报告里提到：

```text
1 passed, 0 failed
```

这说明测试断言通过。

对于模块类实验，`terraform test` 的价值是：

- 验证模块调用是否正常；
- 验证 output 是否符合预期；
- 验证 provider requirement 没有破坏模块可用性。

### 14.6 生成计划

```sh
terraform plan -input=false -no-color -out=tfplan
```

这里把 plan 保存为 `tfplan`，而不是直接 apply。  
这更像 CI/CD 里的标准流程。

### 14.7 应用计划

```sh
terraform apply -auto-approve tfplan
```

注意这里 apply 的是已经保存的 `tfplan`。  
这能保证 apply 的内容就是刚才审查过的 plan。

### 14.8 查看输出

```sh
terraform output
```

模块实验通常会通过 output 暴露关键结果，比如生成文件路径、资源名称、模块输出值。

### 14.9 销毁资源

```sh
terraform destroy -auto-approve
```

实验结束后清理资源，避免状态和生成文件影响下一轮测试。

## 15. Terraform test 可以怎么断言

假设模块输出了文件路径：

```hcl
output "generated_file_path" {
  value = module.file_writer.file_path
}
```

测试可以写成：

```hcl
run "module_should_return_file_path" {
  command = apply

  assert {
    condition     = output.generated_file_path != ""
    error_message = "The module should return a generated file path."
  }
}
```

这个断言不是为了证明 Terraform 多厉害，而是为了证明：

> 模块调用后，调用方确实能拿到模块输出。

这就是模块测试的基本价值。

## 16. 常见错误 1：模块用了 provider，但没有声明

模块里有：

```hcl
resource "local_file" "note" {
  filename = "${path.module}/note.txt"
  content  = "hello"
}
```

但模块没有写：

```hcl
terraform {
  required_providers {
    local = {
      source = "hashicorp/local"
    }
  }
}
```

这会导致依赖不清晰。  
简单项目可能还能跑，但复用到其他项目或 CI 后，维护成本会上升。

正确做法是：

```hcl
terraform {
  required_providers {
    local = {
      source  = "hashicorp/local"
      version = "~> 2.5"
    }
  }
}
```

---

## 17. 常见错误 2：provider source 写错

错误示例：

```hcl
required_providers {
  local = {
    source = "local"
  }
}
```

更推荐写完整 source：

```hcl
required_providers {
  local = {
    source  = "hashicorp/local"
    version = "~> 2.5"
  }
}
```

完整 source 更清楚，也更适合团队协作。

---

## 18. 常见错误 3：模块和 root module 版本约束冲突

child module 写：

```hcl
version = "~> 2.5"
```

root module 写：

```hcl
version = "~> 1.4"
```

这就可能冲突。  
Terraform 需要找到一个同时满足所有约束的 provider 版本。

如果找不到，就会在 `terraform init` 阶段报错。

所以模块版本约束要合理，不要过度狭窄。  
尤其是公共模块，如果版本约束写得太死，调用方会很痛苦。

---

## 19. 常见错误 4：以为 child module 会单独下载一套 provider

Terraform 不是每个模块都下载一套 provider。  
它会汇总整个配置的 provider requirements，然后选择一组满足约束的 provider。

也就是说：

- root module 有 requirements；
- child module 也有 requirements；
- Terraform 会统一解析；
- 最后 provider plugin 下载在当前工作目录的 `.terraform` 里。

你可以理解成：

> 每个模块可以声明需求，但最终由 root module 这个入口统一完成初始化。

## 20. 生产环境中的建模价值

`required_providers` 在真实环境中的价值主要是四个字：

> 可控复用。

### 20.1 可读性

别人看到模块就知道依赖什么 provider。

### 20.2 可测试性

CI 能通过 `terraform init/validate/test` 直接验证模块依赖是否完整。

### 20.3 可升级性

provider 升级时，可以清楚看到版本约束在哪里。

### 20.4 可迁移性

模块从一个 repo 复制到另一个 repo 时，不会丢失关键依赖信息。

## 22. 你应该重点记住什么

本节真正要记住的不是某一段代码，而是下面几条判断：

1. 模块用了 provider，就应该声明 `required_providers`。
2. `required_providers` 声明依赖，不等于配置 provider。
3. provider 的具体运行配置通常由 root module 管理。
4. `.terraform.lock.hcl` 锁定实际 provider 版本，应该和版本约束配合使用。
5. 模块版本约束不能太松，也不能太死。
6. `terraform init` 是发现 provider 依赖问题的第一入口。
7. `terraform test` 可以验证模块调用和输出是否符合预期。

## 23. 面试表达方式

如果面试官问：

> Terraform 模块里为什么要写 `required_providers`？

可以这样回答：

> 因为模块是可复用单元，模块内部如果使用了某个 provider，就应该明确声明 provider 的 source 和版本约束。这样调用方在 `terraform init` 时可以统一解析 provider 依赖，避免版本漂移和隐式依赖。`required_providers` 负责声明依赖，具体 provider 配置通常由 root module 负责。这样模块更清晰、更容易测试，也更适合团队复用。

这个回答比只说“Terraform 要下载 provider”高级很多。

## 26. 本节结论

本节的核心结论是：

> Terraform 模块应该显式声明自己的 provider 依赖。  
> root module 负责组合模块、配置 provider，并通过 init/test/plan/apply 验证整个调用链。

你现在学到的不是一个小语法点，而是模块工程化里非常基础的一环。  
以后你写 AKS/EKS、网络、监控、FinOps、平台工程模块时，这个习惯都用得上。
