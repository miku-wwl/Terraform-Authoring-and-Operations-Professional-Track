# Terraform 实操训练 77 讲解：Terraform State 管理命令

## 1. 这一节到底在学什么？

第 77 节学的是 Terraform 的 **state 管理命令**，重点是：

```sh
terraform state list
terraform state show
terraform state pull
```

本节是在远端 S3 backend 的基础上练习这些命令。

一句话理解：

> 第 77 节不是教你手工改 state，而是教你如何安全地查看、检查、拉取远端 state。

这一节特别重要，因为 Terraform state 是基础设施的“账本”。  
你可以查它、读它、备份它，但不要随便手工编辑它。

---

## 2. 先复习：Terraform state 是什么？

Terraform state 是 Terraform 用来记录真实资源和配置之间映射关系的文件。

例如配置里有：

```hcl
resource "local_file" "example" {
  filename = "${path.module}/output/example.txt"
  content  = "hello"
}
```

Terraform state 会记录：

```text
local_file.example
```

对应真实创建出来的文件、属性、ID 等信息。

在云资源里也是一样：

```text
aws_instance.web -> i-xxxxxxxx
aws_s3_bucket.state -> bucket-name
azurerm_resource_group.rg -> /subscriptions/...
```

没有 state，Terraform 就不知道：

- 哪些资源是它管理的
- 配置里的 resource 对应哪个真实资源
- 下次 plan 应该新增、修改还是删除
- 哪些属性已经存在
- 哪些资源需要销毁

所以 state 是 Terraform 的核心控制文件。

- 第 74 节：S3 中央后端

## 4. 本节核心命令总览

| 命令 | 作用 |
|---|---|
| `terraform state list` | 列出 state 中 Terraform 正在管理的资源地址 |
| `terraform state show <address>` | 查看某个资源在 state 中的详细属性 |
| `terraform state pull` | 从 backend 拉取完整 state JSON 到标准输出 |

这三个命令的危险程度不同：

```text
state list  -> 安全查看资源列表
state show  -> 安全查看单个资源详情
state pull  -> 拉完整 state，可能包含敏感信息，要谨慎
```

---

## 5. `terraform state list` 是什么？

`terraform state list` 用来列出当前 state 中所有资源地址。

示例：

```sh
terraform state list
```

输出可能类似：

```text
local_file.example
aws_s3_bucket.state
aws_dynamodb_table.lock
module.network.aws_vpc.main
module.network.aws_subnet.public["az1"]
```

它回答的是：

> Terraform 当前 state 里管理了哪些资源？

这个命令非常常用，也相对安全。

---

## 6. `terraform state list` 什么时候用？

常见场景：

### 1. 确认资源是否被 Terraform 管理

```sh
terraform state list
```

如果能看到：

```text
aws_s3_bucket.example
```

说明这个资源地址在 state 里。

---

### 2. 查找资源地址

有些资源地址很长，特别是 module 和 `for_each`。

例如：

```text
module.eks.module.node_group.aws_iam_role.this["worker"]
```

你要执行 `state show` 或 `state mv` 前，通常先用：

```sh
terraform state list
```

查准确地址。

---

### 3. 验证远端 state 是否可读

如果使用 S3 backend，运行：

```sh
terraform state list
```

能正常输出，说明 Terraform 可以从远端 backend 读取 state。

---

### 4. 排查资源是否还在 state 里

如果真实资源还在云上，但 state list 里没有，说明 Terraform 可能已经不管理它了。

---

## 7. `terraform state show` 是什么？

`terraform state show` 用来查看 state 中某个资源的详细属性。

语法：

```sh
terraform state show <resource_address>
```

例如：

```sh
terraform state show local_file.example
```

或者：

```sh
terraform state show 'local_file.service["api"]'
```

注意：  
如果资源地址里有 `["key"]`，建议用引号包起来，避免 shell 解析出问题。

---

## 8. `terraform state show` 看什么？

它会展示 state 里记录的资源属性。

例如：

```text
# local_file.example:
resource "local_file" "example" {
    content              = "hello"
    filename             = "./output/example.txt"
    id                   = "..."
}
```

云资源可能会展示：

- id
- arn
- name
- tags
- region
- endpoint
- generated attributes
- provider 返回的属性

这个命令回答的是：

> Terraform state 里记录的这个资源具体长什么样？

---

## 9. `state show` 和 cloud console 的区别

`terraform state show` 看的不是实时云控制台，而是 Terraform state 记录。

如果真实云资源被别人手工改了，state 可能还没更新。

这时：

```sh
terraform state show
```

看到的是 state 里的旧值。

要让 Terraform 刷新真实资源状态，通常通过：

```sh
terraform plan
```

或者相关 refresh 行为。

所以你要分清：

```text
state show = 看 Terraform 当前 state 记录
cloud console = 看真实云上当前状态
plan = 对比配置、state、真实资源后给出变化
```

---

## 10. `terraform state pull` 是什么？

`terraform state pull` 会从当前 backend 拉取完整 state，并输出到标准输出。

命令：

```sh
terraform state pull
```

如果是 S3 backend，它会从 S3 下载当前 state 内容，然后打印出来。

你也可以保存到文件：

```sh
terraform state pull > pulled-state.json
```

这个命令回答的是：

> 当前 backend 里的完整 state JSON 是什么？

但这个命令要谨慎，因为 state 里可能有敏感信息。

---

## 11. `state pull` 什么时候用？

常见场景：

### 1. 备份 state

```sh
terraform state pull > backup-$(date +%Y%m%d-%H%M%S).tfstate
```

在做高风险操作前，可以先备份。

---

### 2. 调试 backend

如果你怀疑 Terraform 读的是哪个 backend，可以拉取 state 看 metadata。

---

### 3. 检查 state 结构

学习或排障时，可以查看完整 JSON 结构。

---

### 4. 迁移前确认

backend 迁移、state mv、state rm、import 之前，可以先 pull 一份备份。

---

## 12. `state pull` 的风险

`terraform state pull` 可能输出敏感信息。

state 中可能包含：

- password
- token
- connection string
- private endpoint
- generated secret
- cloud resource metadata
- user data
- provider 返回的敏感属性

所以不要随便：

```sh
terraform state pull
```

然后把结果：

- 发到群里
- 上传到 GitHub
- 放进 CI artifact
- 贴到工单
- 发给不该看的人

正确做法是：

```text
能不用 pull 就不用
必须 pull 就保护好文件
调试完及时删除
不要提交到 Git
```

---

## 13. 为什么不要手工编辑 state 文件？

本节核心提醒之一是：

> 避免手工编辑 state 文件。

原因很简单：state 是 Terraform 的控制账本，格式和内容都很敏感。

手工编辑可能导致：

- JSON 格式损坏
- resource address 错乱
- provider schema 不匹配
- 资源 ID 改错
- Terraform 误删资源
- Terraform 误以为资源不存在
- 下次 plan 出现大规模异常
- state 和真实资源彻底不一致

你可能只是改了一行，结果 Terraform 后面 plan 出来一堆灾难。

所以：

```text
不要用 vim 直接改 terraform.tfstate
不要手工编辑 S3 里的 state object
不要手动改 pulled-state.json 再 push 回去
```

---

## 14. 如果必须改 state，应该怎么做？

Terraform 提供了专门的 state 命令，而不是让你手改文件。

常见命令包括：

```sh
terraform state mv
terraform state rm
terraform state replace-provider
terraform import
```

例如移动资源地址：

```sh
terraform state mv old_address new_address
```

移除 state 中的资源记录：

```sh
terraform state rm resource_address
```

但这些都是高风险操作，需要：

1. 先备份 state。
2. 确认资源地址。
3. 确认没有其他人同时 apply。
4. 确认 backend lock 生效。
5. 在 PR / 变更窗口中操作。
6. 操作后马上 `terraform plan` 检查。

第 77 节重点是 list/show/pull，先把“安全查看”练熟。

## 15. 远端 S3 backend 下 state 命令怎么工作？

当 backend 是 S3 时，你运行：

```sh
terraform state list
```

Terraform 会：

```text
读取 backend 配置
连接 S3
下载或读取远端 state
解析 state
输出资源地址
```

你运行：

```sh
terraform state show <address>
```

Terraform 会：

```text
从远端 state 中找到该 address
展示它的属性
```

你运行：

```sh
terraform state pull
```

Terraform 会：

```text
把完整远端 state 拉出来输出
```

所以这些命令不是只看本地文件。

它们会和当前 backend 交互。

## 21. `state pull` 示例解释

执行：

```sh
terraform state pull > pulled-state.json
```

会得到一个 JSON 文件。

里面可能包含：

```json
{
  "version": 4,
  "terraform_version": "...",
  "serial": 3,
  "lineage": "...",
  "outputs": {},
  "resources": []
}
```

几个常见字段：

- `version`：state 文件格式版本
- `terraform_version`：写入该 state 的 Terraform 版本
- `serial`：state 变更序号
- `lineage`：state lineage，表示 state 族系
- `outputs`：Terraform outputs
- `resources`：资源记录

你不需要手改这些字段。  
知道它们用于 Terraform 内部追踪即可。

---

## 22. `serial` 是什么？

`serial` 可以理解为 state 的版本序号。

每次 state 更新，serial 通常会增加。

例如：

```text
apply 一次 -> serial +1
destroy 一次 -> serial +1
state 操作 -> serial 可能变化
```

它帮助 Terraform 和 backend 判断 state 是否更新。

不要手工改 serial。  
这不是你拿来随便修的计数器。

---

## 23. `lineage` 是什么？

`lineage` 可以理解为 state 的“家族 ID”。

它用于区分不同 state 的来源。

如果两个 state lineage 不同，Terraform 会认为它们不是同一条状态历史。

这在 state 迁移、复制、恢复时很重要。

同样，不要手工改 lineage。

## 26. 为什么不要用 AWS 控制台直接改 state？

即使 state 在 S3，你也不要去 S3 控制台下载、编辑、再上传覆盖。

这样风险非常高：

- 可能破坏 JSON
- 可能覆盖别人刚写入的 state
- 可能绕过 lock
- 可能丢失 serial
- 可能引入 state lineage 问题
- 可能泄露敏感数据
- 可能导致 Terraform 后续误删资源

正确方式是使用 Terraform 命令：

```sh
terraform state list
terraform state show
terraform state pull
terraform state mv
terraform state rm
terraform import
```

命令也要谨慎，但比手改文件安全得多。

## 27. `state pull` 后能不能改了再推回去？

Terraform 有一些高级场景可能涉及 state push，但这属于高风险操作，不是本节重点。

初学阶段应该记住：

```text
state pull 是查看和备份
不是鼓励你手改后再覆盖
```

如果真的遇到 state 损坏或迁移问题，应该：

1. 先备份。
2. 查看官方文档。
3. 在测试环境演练。
4. 使用 Terraform state 命令优先。
5. 避免手工 JSON 编辑。
6. 生产环境需要变更审批。

不要把 state 当普通 JSON 配置文件。

## 29. state 命令和 locking 的关系

有些 state 命令会修改 state，有些只是读取。

本节重点命令：

```text
state list -> 读
state show -> 读
state pull -> 读
```

它们相对安全。

但下面这些会改 state：

```text
state mv
state rm
state replace-provider
import
```

修改类操作要更谨慎，并且依赖 backend locking 防止并发写入。

所以 state 命令不能一概而论：

```text
读 state：用于观察和排障
改 state：属于高风险运维操作
```

## 33. SRE / Platform Engineering 视角

从 SRE / Platform Engineering 角度看，state 命令是 Terraform 排障基础工具。

例如：

### 场景 1：plan 出现异常替换

你可以：

```sh
terraform state show <address>
```

看 state 当前记录的属性。

### 场景 2：不知道资源地址

先：

```sh
terraform state list
```

找到准确地址。

### 场景 3：迁移前备份

```sh
terraform state pull > backup.tfstate
```

### 场景 4：怀疑 backend 配错

```sh
terraform state list
```

确认当前 backend 能读到预期资源。

这些是日常 Terraform 运维能力。

## 34. 面试/考试记忆版

可以这样记：

- `terraform state list` 列出 state 里的资源地址。
- `terraform state show <address>` 查看某个资源的 state 详情。
- `terraform state pull` 从 backend 拉取完整 state JSON。
- 使用 S3 backend 时，这些命令读取的是远端 S3 state。
- `state pull` 可能包含敏感信息，要谨慎保存和分享。
- 不要手工编辑 state 文件。
- 如果要改 state，优先使用 Terraform state 命令。
- `for_each` 资源地址常需要加引号，例如 `'resource.name["key"]'`。
- `.terraform/` 存在不代表本地 state。
- 远端 backend 下本地不应生成主 `terraform.tfstate`。

一句话总结：

> 第 77 节的核心，是学会用 Terraform 自带命令安全查看远端 state，而不是直接打开或手工编辑 state 文件。
