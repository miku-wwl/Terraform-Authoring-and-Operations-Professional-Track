# Terraform 实操训练 26：Heredoc 多行字符串讲解版

> 主题：`heredoc`、多行字符串、脚本生成、缩进与插值风险  
> 适合场景：Terraform 生成脚本、配置文件、说明文档、JSON/YAML 片段

---

## 1. 这节 Lab 到底在练什么？

这节 Lab 练的是 Terraform 里维护**多行字符串**的方式：`heredoc`。

你可以把 heredoc 理解成：

> 不再用一堆 `\n` 拼接字符串，而是直接在 Terraform 代码里写一整段多行文本。

比如生成一个 shell 脚本时，如果用普通 quoted string，会变成这样：

```hcl
content = "#!/bin/sh\necho start\necho deploy\necho done\n"
```

这能跑，但非常难读。

用 heredoc 后会变成：

```hcl
content = <<EOF
#!/bin/sh
echo start
echo deploy
echo done
EOF
```

这就像正常写脚本一样，可读性高很多。

---

## 2. Heredoc 的基本语法

Terraform heredoc 的基本格式是：

```hcl
content = <<EOF
第一行内容
第二行内容
第三行内容
EOF
```

关键点：

- `<<EOF` 表示多行字符串开始。
- 最后的 `EOF` 表示多行字符串结束。
- `EOF` 不是固定名字，也可以叫 `EOT`、`SCRIPT`、`CONFIG`。
- 开始和结束的标记必须一致。
- 结束标记必须单独占一行。

例如：

```hcl
content = <<SCRIPT
#!/bin/sh
echo "deploying application"
echo "done"
SCRIPT
```

这里 `SCRIPT` 只是 delimiter，也就是边界标记。

---

## 3. 为什么这节用 heredoc 生成 `deploy.sh`？

Lab 里的场景是：平台团队要用 Terraform 生成一个部署脚本。

这类内容天然是多行文本：

```sh
#!/bin/sh
echo "Starting deployment"
echo "Running migration"
echo "Deployment complete"
```

如果用普通字符串写，代码会很丑：

```hcl
content = "#!/bin/sh\necho \"Starting deployment\"\necho \"Running migration\"\necho \"Deployment complete\"\n"
```

这就是典型的“代码能跑，但人类不想维护”。

heredoc 更适合这种场景，因为它保留了脚本原本的结构：

```hcl
content = <<EOF
#!/bin/sh
echo "Starting deployment"
echo "Running migration"
echo "Deployment complete"
EOF
```

一句话：

> 普通字符串适合短文本，heredoc 适合长文本、多行文本、配置片段和脚本。

本实验要证明几件事：

1. Terraform 可以用 heredoc 维护多行内容。
2. heredoc 生成的文件可以保留换行。

## 7. 每条命令在干嘛？

### 7.1 `terraform init -input=false`

初始化当前目录，下载 provider，准备 `.terraform` 工作目录。

`-input=false` 的意思是：

> 不要在命令行里问我问题，适合自动化环境。

---

### 7.2 `terraform fmt -check`

检查 Terraform 代码格式是否符合标准。

注意：`fmt -check` 只检查，不自动修改。CI 里常用这个方式阻断格式不规范的提交。

---

### 7.3 `terraform validate`

检查 Terraform 配置语法和结构是否有效。

如果 heredoc 的开始和结束标记不匹配，通常会在这里暴露。

---

### 7.4 `terraform test`

运行 `.tftest.hcl` 测试文件。

本 Lab 里它主要验证：

- 生成脚本是否包含 `#!/bin/sh`
- 多行内容是否符合预期
- heredoc 生成结果是否稳定

---

### 7.5 `terraform plan -out=tfplan`

生成执行计划，并保存为二进制 plan 文件。

这一步不会真正创建文件，只是告诉你 Terraform 准备做什么。

---

### 7.6 `terraform apply tfplan`

执行刚才保存的 plan。

执行后会真正生成 `deploy.sh` 这类本地文件资源。

---

### 7.7 `terraform destroy`

销毁 Terraform 管理的资源。

因为本实验用的是本地文件资源，所以 destroy 会删除 Terraform 创建出来的文件。

## 8. Heredoc 的一个关键坑：缩进会被保留

普通 heredoc 会保留前导空白。

例如：

```hcl
content = <<EOF
    line one
    line two
EOF
```

生成内容里，`line one` 和 `line two` 前面会真的有 4 个空格。

这在脚本或 YAML 里可能会出问题。

如果你想让 Terraform 帮你去掉缩进，可以使用 indented heredoc：

```hcl
content = <<-EOF
    line one
    line two
EOF
```

`<<-EOF` 会让 Terraform 移除公共缩进，让代码写起来更整齐。

但不要乱用。尤其是 YAML、Python、shell here-doc 这种本身对缩进敏感的内容，要确认最终生成文件是否符合预期。

## 9. Heredoc 里的变量插值

heredoc 默认支持 Terraform 插值。

例如：

```hcl
variable "environment" {
  default = "dev"
}

locals {
  script = <<EOF
#!/bin/sh
echo "Deploying to ${var.environment}"
EOF
}
```

生成结果会是：

```sh
#!/bin/sh
echo "Deploying to dev"
```

这很方便，但也有风险。

如果你想生成的文件里保留字面量 `${...}`，而不是让 Terraform 插值，就需要转义。

例如：

```hcl
content = <<EOF
echo "Runtime variable is $${APP_NAME}"
EOF
```

这里 `$${APP_NAME}` 会在生成文件里保留成 `${APP_NAME}`。

这个点很重要。否则 Terraform 会以为 `${APP_NAME}` 是 Terraform 自己的表达式，然后报错。

## 10. 什么场景应该用 heredoc？

适合用 heredoc 的场景：

| 场景 | 示例 |
|---|---|
| Shell 脚本 | `deploy.sh`、`bootstrap.sh` |
| 配置文件 | Nginx、Prometheus、应用配置 |
| 多行说明 | README 片段、runbook |
| JSON/YAML 小片段 | 简单策略、manifest |
| cloud-init | VM 初始化脚本 |

不太适合 heredoc 的场景：

| 场景 | 更推荐 |
|---|---|
| 很复杂的模板 | `templatefile()` |
| 需要大量条件判断 | 拆成模板文件 |
| 需要复用的脚本 | 独立脚本文件 |
| 大型 JSON/YAML | 原生对象 + `jsonencode()` / `yamlencode()` |

---

## 11. heredoc、templatefile、jsonencode 怎么选？

可以按这个标准判断：

### 简单多行文本：用 heredoc

```hcl
content = <<EOF
hello
world
EOF
```

### 复杂脚本模板：用 templatefile

```hcl
content = templatefile("${path.module}/deploy.sh.tftpl", {
  environment = var.environment
})
```

### JSON 内容：优先用 jsonencode

不要手搓 JSON：

```hcl
content = <<EOF
{
  "name": "${var.name}",
  "enabled": true
}
EOF
```

更推荐：

```hcl
content = jsonencode({
  name    = var.name
  enabled = true
})
```

原因很简单：

> 手写 JSON 容易漏逗号、漏引号、转义错误；`jsonencode()` 更稳定。

## 12. 生产环境注意事项

### 12.1 检查生成文件权限

如果生成的是脚本，光有内容还不够，还要确认权限。

例如本地文件资源可能需要：

```hcl
file_permission = "0755"
```

否则脚本可能生成了，但不能直接执行。

---

### 12.2 检查换行符

Linux shell 脚本推荐使用 LF 换行。

如果你在 Windows 环境编辑文件，要小心 CRLF 导致脚本执行异常。

---

### 12.3 不要把大段业务逻辑塞进 Terraform

Terraform 是基础设施编排工具，不是脚本仓库。

少量 bootstrap 脚本可以放 heredoc。复杂部署逻辑建议独立成脚本文件或 CI/CD 步骤。

---

### 12.4 heredoc 不是配置管理工具

不要用 Terraform heredoc 硬写大量应用配置，然后假装自己在做配置管理。

配置复杂后，应该考虑：

- 模板文件
- Helm values
- Kustomize
- Ansible
- cloud-init
- 专门的配置管理系统

---

## 13. 常见错误与排障

### 错误 1：结束标记不匹配

错误示例：

```hcl
content = <<EOF
hello
EOT
```

开始是 `EOF`，结束是 `EOT`，Terraform 会报错。

修复：

```hcl
content = <<EOF
hello
EOF
```

---

### 错误 2：结束标记前面有多余字符

错误示例：

```hcl
content = <<EOF
hello
  EOF
```

普通 heredoc 的结束标记不能随便缩进。

修复：

```hcl
content = <<EOF
hello
EOF
```

或者使用 indented heredoc：

```hcl
content = <<-EOF
  hello
EOF
```

---

### 错误 3：`${}` 被 Terraform 提前解析

错误示例：

```hcl
content = <<EOF
echo ${APP_NAME}
EOF
```

如果 `APP_NAME` 不是 Terraform 变量，Terraform 会报错。

修复：

```hcl
content = <<EOF
echo $${APP_NAME}
EOF
```

---

### 错误 4：脚本生成了，但不能执行

可能原因：

- 没有执行权限
- shebang 写错
- 换行符不对
- 文件路径不对

排查：

```sh
ls -l output/deploy.sh
cat output/deploy.sh
sh output/deploy.sh
```

## 14. 面试/认证可以怎么理解？

这节可以记成一句话：

> Heredoc 是 Terraform 维护多行字符串的方式，适合脚本和配置片段；生产中要注意缩进、插值、权限和可维护性。

更工程化一点的表达：

> 在 IaC 项目中，heredoc 可以提升多行文本的可读性，但当模板逻辑复杂时，应迁移到 `templatefile()`、`jsonencode()` 或专门的配置管理工具，避免 Terraform 配置变成不可维护的脚本仓库。

---

## 15. 本 Lab 的核心结论

这节 Lab 的核心不是“会写 `<<EOF`”这么简单。

真正要掌握的是：

1. Terraform 可以用 heredoc 管理多行字符串。
2. heredoc 比 `\n` 拼接更适合脚本和配置文件。
3. 普通 heredoc 会保留缩进。
4. `<<-EOF` 可以处理缩进，但要确认最终内容。
5. heredoc 默认会执行 Terraform 插值。
6. 想保留 `${}` 字面量时，要写成 `$${}`。
7. 复杂模板不要硬塞 heredoc，应该用 `templatefile()` 或编码函数。
