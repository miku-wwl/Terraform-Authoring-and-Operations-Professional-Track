# Terraform AWS Sandbox 32：Provider 文档中的 Resource 与 Data Source 讲解版

## 0. 这一节到底在学什么？

这一节不是让你去创建 AWS 资源，而是让你学会一个非常关键的 Terraform 判断能力：

> 看到 Provider 文档时，要能区分：  
> **这个东西应该用 resource 创建，还是用 data source 读取？**

在 Terraform 里：

- `resource`：负责创建、修改、删除资源。
- `data source`：负责读取已经存在的信息，不负责创建资源。

本实验使用的是：

```hcl
data "aws_caller_identity" "current" {}
```

它只读取当前 AWS 调用者身份信息，不会创建任何 AWS 资源。

在 LocalStack 里，它会返回一个模拟账号：

```text
000000000000
```

所以这个实验的核心目的就是：

> 确认 Terraform 当前打到的是 LocalStack sandbox，而不是真实 AWS 账号。

这一步非常重要。因为如果 provider endpoint 配错了，你以为自己在 sandbox 里练习，实际上 Terraform 可能正在连接真实 AWS。那就不是实验了，那是“账单刺客”。

---

## 1. Resource 和 Data Source 的区别

### 1.1 Resource 是“我来创建和管理”

例如：

```hcl
resource "aws_s3_bucket" "demo" {
  bucket = "my-demo-bucket"
}
```

这代表 Terraform 要管理这个 S3 Bucket 的生命周期：

- `terraform apply` 会创建或修改它；
- `terraform destroy` 会删除它；
- state 里会记录它的真实 ID；
- 以后 Terraform 会持续追踪它有没有 drift。

所以 resource 的关键词是：

> **创建、管理、修改、删除。**

---

### 1.2 Data source 是“我只读取已有信息”

例如：

```hcl
data "aws_caller_identity" "current" {}
```

这代表 Terraform 调用 AWS provider 去读取当前身份信息。

它不会创建账号，不会创建 IAM 用户，也不会创建任何资源。

它只是读取：

- account ID
- caller ARN
- user ID

所以 data source 的关键词是：

> **查询、读取、引用、不创建。**

---

## 2. 为什么本节用 `aws_caller_identity`？

因为它是 AWS provider 中最小、最安全、最适合做连通性验证的数据源之一。

真实 AWS 中，它常用于：

- 确认当前 Terraform 连到哪个 AWS 账号；
- 拼接 ARN；
- 防止 pipeline 用错账号；
- 在多账号环境中输出当前 account ID；
- 做部署前安全检查。

例如：

```hcl
data "aws_caller_identity" "current" {}

output "account_id" {
  value = data.aws_caller_identity.current.account_id
}
```

如果你在 LocalStack 里运行，预期输出是：

```text
account_id = "000000000000"
```

如果你看到真实 AWS 账号 ID，例如 12 位真实账号数字，那就说明 provider 可能没有正确指向 LocalStack。

这时候应该立即停下来检查配置。

## 3. 本实验的架构逻辑

这一节的实验链路可以理解成：

```text
Terraform
   |
   | 通过 AWS provider
   v
LocalStack endpoint: http://localhost:4566
   |
   | 模拟 STS
   v
返回 caller identity
```

Terraform 本身并不知道你连接的是 LocalStack 还是真实 AWS。

决定它打到哪里的是 provider 配置里的 endpoint。

所以本实验重点不是 STS 本身，而是：

> Terraform provider endpoint 是否正确指向 sandbox。

## 4. 实验文件说明

本实验包含这些文件：

```text
practice/labs/32/
├── versions.tf
├── provider.tf
├── main.tf
├── outputs.tf
├── scripts/
│   ├── check-sandbox.sh
│   ├── verify.sh
│   └── clean.sh
└── README.md

work/32/
└── 用户做题目录
```

### 4.1 `versions.tf`

通常用于声明 Terraform 和 provider 版本。

作用是固定实验环境，避免 provider 版本漂移导致结果不同。

典型结构类似：

```hcl
terraform {
  required_version = ">= 1.11"

  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "..."
    }
  }
}
```

### 4.2 `provider.tf`

这是本实验最关键的文件之一。

它负责让 AWS provider 不要打真实 AWS，而是打 LocalStack：

```hcl
provider "aws" {
  region                      = var.aws_region
  access_key                  = "test"
  secret_key                  = "test"
  skip_credentials_validation = true
  skip_metadata_api_check     = true
  skip_requesting_account_id  = true

  endpoints {
    sts = var.localstack_endpoint
  }
}
```

重点是 `endpoints`。

如果 endpoint 没配置好，Terraform 就可能去真实 AWS 查 STS。

### 4.3 `main.tf`

这里一般会声明 data source：

```hcl
data "aws_caller_identity" "current" {}
```

也可能读取当前 region：

```hcl
data "aws_region" "current" {}
```

这两个 data source 都是读取信息，不创建资源。

### 4.4 `outputs.tf`

输出读取到的结果，例如：

```hcl
output "account_id" {
  value = data.aws_caller_identity.current.account_id
}

output "caller_arn" {
  value = data.aws_caller_identity.current.arn
}

output "current_region" {
  value = data.aws_region.current.name
}
```

这能帮助你确认 Terraform 读到了什么。

## 5. 运行步骤讲解

进入实验目录：

```sh
cd practice/labs/32
```

设置假 AWS 凭证：

```sh
export AWS_ACCESS_KEY_ID=test
export AWS_SECRET_ACCESS_KEY=test
export AWS_DEFAULT_REGION=us-east-1
export LOCALSTACK_ENDPOINT=http://localhost:4566
```

这里的 `test/test` 是 LocalStack 常用假凭证。

它不是为了登录真实 AWS，而是为了让 AWS SDK/provider 有凭证字段可用。

---

### 5.1 检查 sandbox

```sh
bash scripts/check-sandbox.sh
```

这一步通常会检查：

- LocalStack 是否可访问；
- STS 是否能返回身份；
- endpoint 是否正常。

这一步是安全闸门。

如果这里失败，不要继续跑 Terraform。

---

### 5.2 初始化 Terraform

```sh
terraform init -input=false
```

作用：

- 下载 AWS provider；
- 初始化 `.terraform/`；
- 生成 `.terraform.lock.hcl`；
- 准备后续 plan/apply。

`-input=false` 表示不允许交互输入，适合自动化脚本。

---

### 5.3 格式检查

```sh
terraform fmt -check
```

作用：

- 检查 `.tf` 文件格式；
- 不自动修改；
- 如果格式不符合要求会失败。

CI 里常用这个命令保证代码风格统一。

---

### 5.4 配置校验

```sh
terraform validate
```

作用：

- 检查 Terraform 配置语法；
- 检查资源/data source 引用是否合理；
- 不连接真实云资源创建东西。

---

### 5.5 生成 plan

```sh
terraform plan -input=false -no-color -out=tfplan
```

作用：

- 计算 Terraform 将要做什么；
- 保存 plan 到 `tfplan`；
- 因为本实验只有 data source，所以不会出现创建 AWS 资源的动作。

这里你应该关注 plan 里是不是没有 resource 创建。

---

### 5.6 执行 apply

```sh
terraform apply -auto-approve tfplan
```

这里虽然叫 `apply`，但并不代表一定会创建资源。

本实验里 Terraform 主要做的是：

- 读取 data source；
- 把读取结果写入 state；
- 生成 output。

也就是说：

> data source 在 apply 阶段会被读取，但不会创建云资源。

---

### 5.7 查看输出

```sh
terraform output
```

预期看到类似：

```text
account_id = "000000000000"
caller_arn = "arn:aws:iam::000000000000:root"
current_region = "us-east-1"
```

这里最关键的是：

```text
000000000000
```

这说明 Terraform 读取的是 LocalStack 模拟账号。

---

### 5.8 验证和清理

```sh
bash scripts/verify.sh
bash scripts/clean.sh
```

因为本实验没有创建真实 AWS 资源，所以清理主要是删除本地 Terraform 文件：

```sh
rm -rf .terraform .terraform.lock.hcl tfplan terraform.tfstate terraform.tfstate.backup
```

## 6. 关键知识点总结

### 6.1 Provider 文档里看到 Resource，要想到“生命周期管理”

如果文档标题是 resource，例如：

```text
aws_instance
aws_s3_bucket
aws_security_group
```

那它通常代表 Terraform 可以创建和管理这个对象。

你要思考：

- apply 会不会创建资源？
- destroy 会不会删除资源？
- state 里会不会记录它？
- 是否有计费风险？
- 是否需要真实云权限？

---

### 6.2 Provider 文档里看到 Data Source，要想到“只读查询”

如果文档标题是 data source，例如：

```text
aws_caller_identity
aws_region
aws_ami
aws_vpc
aws_subnets
```

那它通常代表 Terraform 读取已有信息。

你要思考：

- 查询条件是什么？
- 如果查询不到会不会失败？
- 如果查到多个结果会怎样？
- 读取结果会被哪些 resource 或 module 引用？

---

### 6.3 Data source 不是完全“无风险”

虽然 data source 不创建资源，但它仍然可能有风险：

1. **连错账号**  
   你以为查 sandbox，结果查了真实 AWS。

2. **查错对象**  
   比如用 tag 查 VPC，结果 tag 重复，查到了错误 VPC。

3. **结果变化导致部署变化**  
   比如 `data "aws_ami"` 每次查 latest AMI，可能导致 EC2 替换。

所以 data source 的风险不是“创建资源”，而是：

> 读到错误信息，然后被后续资源引用，造成错误部署。

---

## 7. 生产环境怎么用这个知识？

在真实项目里，`aws_caller_identity` 很常见。

### 7.1 输出当前账号

```hcl
data "aws_caller_identity" "current" {}

output "current_account_id" {
  value = data.aws_caller_identity.current.account_id
}
```

这样可以在 CI 日志里确认当前账号。

---

### 7.2 拼接 ARN

```hcl
locals {
  account_id = data.aws_caller_identity.current.account_id
  role_arn   = "arn:aws:iam::${local.account_id}:role/example-role"
}
```

这种写法比硬编码账号 ID 更灵活。

---

### 7.3 做账号安全检查

可以配合变量校验或 precondition，防止部署到错误账号。

思路类似：

```hcl
locals {
  expected_account_id = "123456789012"
}

check "expected_account" {
  assert {
    condition     = data.aws_caller_identity.current.account_id == local.expected_account_id
    error_message = "当前 AWS 账号不是预期账号，停止部署。"
  }
}
```

这个逻辑在生产 CI 里非常实用。

## 8. 常见错误与排障

### 8.1 返回了真实 AWS 账号 ID

现象：

```text
account_id = "123456789012"
```

这说明 Terraform 可能没有打到 LocalStack。

检查：

```sh
echo $LOCALSTACK_ENDPOINT
echo $AWS_ACCESS_KEY_ID
echo $AWS_DEFAULT_REGION
```

还要检查 `provider.tf` 里的 endpoint 配置。

这个问题不要硬跑。

因为这代表你的实验环境隔离失败了。

---

### 8.2 STS endpoint 失败

常见原因：

- LocalStack 没启动；
- 端口 `4566` 不通；
- endpoint 写错；
- Docker 网络不通。

先验证：

```sh
aws --endpoint-url=http://localhost:4566 sts get-caller-identity
```

如果 AWS CLI 都不通，Terraform 也大概率不通。

---

### 8.3 输出为空

检查是否声明了：

```hcl
data "aws_caller_identity" "current" {}
```

以及 output 是否引用了正确地址：

```hcl
data.aws_caller_identity.current.account_id
```

Terraform 地址写错时，通常 `validate` 阶段就会报错。

---

### 8.4 Region 不符合预期

检查：

```sh
echo $AWS_DEFAULT_REGION
```

也检查 Terraform 变量或 provider 里的 region 配置。

如果 provider region 和环境变量不一致，以 provider 配置为准。

## 9. 考试和面试怎么理解？

这节内容在 Terraform 认证和真实面试里都很基础。

可以这样记：

> resource 是 Terraform 管对象；  
> data source 是 Terraform 问别人要信息。

面试里可以这样说：

> In Terraform, resources manage infrastructure lifecycle, while data sources read existing information from providers. For AWS, `aws_caller_identity` is a read-only data source commonly used to confirm the active account ID and build ARNs dynamically. In a sandbox setup, returning LocalStack account `000000000000` confirms the provider endpoint is not hitting real AWS.

中文理解就是：

> resource 是“我来造”；data source 是“我来查”。  
> `aws_caller_identity` 不造东西，只是问当前账号是谁。

## 10. 本节你应该掌握的结论

完成 Lab 32 后，你应该能说清楚：

1. Terraform provider 文档中 resource 和 data source 的区别。
2. `data "aws_caller_identity"` 不创建 AWS 资源。
3. LocalStack 返回 `000000000000` 可以用于验证 sandbox endpoint。
4. provider endpoint 配错可能导致 Terraform 连接真实 AWS。
5. data source 虽然只读，但查错信息也会影响后续部署。
6. 生产环境中可以用 caller identity 做账号确认、ARN 拼接和部署保护。

## 11. 一句话总结

> Lab 32 的核心不是 STS，而是 Terraform 的一个基本判断：  
> **resource 负责管理资源，data source 负责读取信息；先确认自己连的是 sandbox，再谈后面的部署。**
