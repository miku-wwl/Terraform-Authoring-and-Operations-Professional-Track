# 第 164 节讲解：挑战四解法一：`csvdecode` 驱动 EC2 配置数据

## 1. 本节定位

这一节属于 Terraform 综合挑战系列，主题是：

> 使用 `csvdecode` 把 CSV 配置文件转换成 Terraform 可消费的数据结构，再用这些数据批量驱动 EC2 配置。

这类题的价值不在于“背某个 EC2 参数”，而在于训练你把外部配置数据接入 Terraform 的能力。真实平台工程里，很多资源规格、实例清单、环境配置、标签策略，并不一定直接硬编码在 `.tf` 文件里，而是来自 CSV、YAML、JSON、数据库、CMDB 或内部平台。

本节的核心思想可以概括为一句话：

> 配置数据外置，Terraform 负责读取、转换、建模、创建和验证。

---

## 2. 你要掌握的核心能力

### 2.1 `csvdecode` 是什么

`csvdecode` 是 Terraform 的内置函数，用来把 CSV 字符串转换成对象列表。

典型写法如下：

```hcl
locals {
  instances = csvdecode(file("${path.module}/instances.csv"))
}
```

如果 CSV 内容类似：

```csv
name,instance_type,ami
web-1,t3.micro,ami-123456
web-2,t3.small,ami-abcdef
```

Terraform 读出来后，大致会变成：

```hcl
[
  {
    name          = "web-1"
    instance_type = "t3.micro"
    ami           = "ami-123456"
  },
  {
    name          = "web-2"
    instance_type = "t3.small"
    ami           = "ami-abcdef"
  }
]
```

注意，CSV 读出来的值默认都是字符串。哪怕 CSV 里写的是数字或布尔值，也要根据资源参数需要自行转换。

---

## 3. 为什么这个能力重要

在初学 Terraform 时，你可能会这样写资源：

```hcl
resource "aws_instance" "web" {
  ami           = "ami-123456"
  instance_type = "t3.micro"

  tags = {
    Name = "web-1"
  }
}
```

这能跑，但扩展性很差。

如果有 10 台、50 台、100 台实例，并且每台实例的名称、类型、AMI、标签、子网都不同，全部手写 resource 会变成灾难。更合理的做法是：

1. 把实例清单放到 CSV。
2. 用 `file()` 读取 CSV。
3. 用 `csvdecode()` 转换成 Terraform 对象。
4. 用 `for_each` 批量创建资源。
5. 用输出和验证脚本确认资源符合预期。

这就是本节要训练的工程化思维。

---

## 4. 标准建模路径

### 4.1 第一步：准备 CSV 数据

例如：

```csv
name,instance_type,ami
app-1,t3.micro,ami-000001
app-2,t3.micro,ami-000002
worker-1,t3.small,ami-000003
```

CSV 的第一行是表头，Terraform 会把表头作为对象字段名。

字段命名建议：

- 使用小写。
- 使用下划线。
- 避免空格。
- 避免中文字段名。
- 避免和 Terraform 关键字混淆。

推荐：

```csv
name,instance_type,ami,env,owner
```

不推荐：

```csv
Instance Name,实例类型,AMI ID
```

---

### 4.2 第二步：读取 CSV

```hcl
locals {
  raw_instances = csvdecode(file("${path.module}/instances.csv"))
}
```

这里有两个关键函数：

- `file()`：读取本地文件内容。
- `csvdecode()`：把 CSV 文本解析成对象列表。

---

### 4.3 第三步：转换成适合 `for_each` 的 map

`for_each` 更适合接收 map，而不是 list。因为 map 的 key 可以稳定标识每个资源。

```hcl
locals {
  instances = {
    for instance in local.raw_instances :
    instance.name => instance
  }
}
```

这样就可以用实例名称作为资源地址的一部分。

例如：

```hcl
aws_instance.this["app-1"]
aws_instance.this["app-2"]
aws_instance.this["worker-1"]
```

这个设计比用 `count.index` 更稳定。

---

### 4.4 第四步：批量创建资源

真实 AWS 中可能类似：

```hcl
resource "aws_instance" "this" {
  for_each = local.instances

  ami           = each.value.ami
  instance_type = each.value.instance_type

  tags = {
    Name = each.value.name
    Env  = each.value.env
  }
}
```

本项目使用 LocalStack，所以重点不是验证真实 EC2 能力，而是验证 Terraform 的数据读取、资源建模、依赖关系和生命周期闭环。

---

## 5. `for_each` 为什么比 `count` 更适合这里

如果你用 `count`：

```hcl
resource "aws_instance" "this" {
  count = length(local.raw_instances)

  ami           = local.raw_instances[count.index].ami
  instance_type = local.raw_instances[count.index].instance_type
}
```

它能工作，但有一个明显问题：

> 资源身份依赖列表顺序。

如果 CSV 中间插入一行，后面的索引可能整体变化，Terraform 可能误判资源需要替换。

而 `for_each` 使用名称作为 key：

```hcl
for_each = local.instances
```

资源身份更稳定。只要 `name` 不变，资源地址就不变。

这就是 Terraform 工程实践里经常强调的原则：

> 能用稳定业务 key，就不要依赖数组下标。

## 9. 常见坑

### 9.1 CSV 字段名和 Terraform 引用不一致

CSV 中是：

```csv
instanceType
```

Terraform 中却写：

```hcl
each.value.instance_type
```

这样会报错，因为字段名不匹配。

建议从一开始就统一字段命名，例如：

```csv
name,instance_type,ami
```

---

### 9.2 CSV 值都是字符串

例如 CSV：

```csv
name,enabled
app-1,true
```

`enabled` 被读出来通常是字符串 `"true"`，不是布尔值 `true`。

如果资源参数需要布尔值，需要转换：

```hcl
enabled = each.value.enabled == "true"
```

数字也类似：

```hcl
volume_size = tonumber(each.value.volume_size)
```

---

### 9.3 `for_each` 的 key 不唯一

如果 CSV 里有重复名称：

```csv
name,instance_type
app-1,t3.micro
app-1,t3.small
```

转换成 map 时会冲突。

所以作为 key 的字段必须唯一。通常可以使用：

- name
- env + name
- account + region + name
- workload + environment

例如：

```hcl
locals {
  instances = {
    for instance in local.raw_instances :
    "${instance.env}-${instance.name}" => instance
  }
}
```

---

### 9.4 CSV 中有多余空格

例如：

```csv
name,instance_type
app-1, t3.micro
```

`instance_type` 可能会带前导空格。可以用 `trimspace()` 清理：

```hcl
instance_type = trimspace(each.value.instance_type)
```

更好的做法是在数据源头保持 CSV 干净。

---

### 9.5 把 LocalStack 结果当成真实 AWS 结果

LocalStack 能让你练 Terraform 闭环，但不代表真实 AWS 一定完全一样。

尤其是 EC2、IAM、STS、网络类资源，在真实 AWS 中会受到更多限制：

- 区域限制。
- AMI 是否存在。
- IAM 权限是否足够。
- Service quota 是否够。
- VPC 网络配置是否正确。
- 安全组是否允许访问。
- 资源创建耗时和状态转换。

考试训练阶段可以用 LocalStack 快速练手，但真正迁移到项目时，必须重新做真实环境验证。

## 10. 考试和面试怎么理解这一节

这节背后的考点不是“CSV 语法”，而是：

1. Terraform 可以读取外部文件。
2. `csvdecode()` 可以把 CSV 转换成对象列表。
3. `for_each` 可以基于外部配置批量创建资源。
4. map key 的稳定性影响 Terraform 资源地址。
5. 外部配置数据需要校验和规范化。
6. LocalStack 是模拟环境，不等于真实 AWS。

面试里可以这样表达：

> 我会把实例配置从 Terraform 代码中抽离出来，放到 CSV 或其他结构化数据源中。Terraform 通过 `file()` 和 `csvdecode()` 读取数据，再转换成以实例名称为 key 的 map，最后用 `for_each` 创建资源。这样资源身份更稳定，也方便平台团队维护批量配置。

这句话非常工程化，比单纯说“我会用 csvdecode”强很多。

---

## 11. 本节一句话总结

`csvdecode` 的核心价值是把外部表格化配置变成 Terraform 数据结构，再配合 `for_each` 实现可维护、可扩展、资源地址稳定的批量基础设施建模。
