# Terraform 实操训练 38 讲解：多条件变量验证

## 1. 这一节到底在学什么？

Lab 38 学的是 Terraform 输入变量验证里的一个非常实用场景：**用白名单限制用户能传什么值**。

前面 Lab 36 和 Lab 37 分别讲了：

- 用 `regex` 校验命名规则
- 用 `length()` 校验密码长度

这一节进一步升级，使用 `contains()` 来判断某个输入值是否在允许列表里。

核心思想很简单：

> 用户不是想传什么就传什么，而是只能从平台团队批准过的选项里选。

这就是生产环境里非常常见的治理套路。

---

## 2. 为什么需要允许列表？

在 Terraform 项目里，很多变量表面上只是一个字符串，但背后其实代表成本、权限、容量和风险。

比如：

```hcl
variable "instance_size" {
  type = string
}
```

如果没有任何限制，用户可能传：

```hcl
instance_size = "small"
```

也可能传：

```hcl
instance_size = "super-expensive-ultra-large"
```

在真实云平台里，这可能对应：

- 过大的虚拟机规格
- 未批准的数据库 SKU
- 不符合团队标准的环境名
- 错误的 region
- 不允许使用的部署模式

所以变量验证不是“语法洁癖”，而是治理防线。

---

## 3. `contains()` 是什么？

`contains()` 用来判断一个 list、tuple 或 set 里是否包含某个值。

典型写法：

```hcl
contains(["dev", "test", "prod"], var.environment)
```

如果 `var.environment` 是 `"dev"`，结果就是 `true`。  
如果是 `"production"`，结果就是 `false`。

在变量验证中，它通常放在 `condition` 里：

```hcl
variable "environment" {
  type    = string
  default = "dev"

  validation {
    condition     = contains(["dev", "test", "prod"], var.environment)
    error_message = "environment must be one of: dev, test, prod."
  }
}
```

这段配置的意思是：

> `environment` 只能是 `dev`、`test`、`prod` 之一。

---

## 4. 本实验的场景

本 Lab 的背景是：开发环境只允许使用批准过的规格和环境名。

也就是说，平台团队不希望用户随便传：

- 未批准的实例规格
- 拼错的环境名称
- 大小写不一致的值
- 可能造成成本失控的配置

所以实验任务是给两个变量加允许列表：

- `instance_size`
- `environment`

预期结果是：

- 默认值可以通过
- 非法规格会在 plan 阶段被拒绝

---

## 5. 多条件变量验证怎么理解？

所谓“多条件”，这里不是指一个 `validation` 里一定要写复杂逻辑，而是指多个变量都可以有自己的约束。

比如：

```hcl
variable "instance_size" {
  type    = string
  default = "small"

  validation {
    condition     = contains(["small", "medium", "large"], var.instance_size)
    error_message = "instance_size must be one of: small, medium, large."
  }
}

variable "environment" {
  type    = string
  default = "dev"

  validation {
    condition     = contains(["dev", "test", "prod"], var.environment)
    error_message = "environment must be one of: dev, test, prod."
  }
}
```

这就形成了两条独立防线：

1. `instance_size` 必须来自规格白名单。
2. `environment` 必须来自环境白名单。

这比只写 README 可靠得多。README 是提醒，`validation` 是强制执行。

---

## 6. 为什么要在 plan 阶段失败？

Terraform 的工作流通常是：

```text
init -> validate -> plan -> apply
```

如果变量值不合法，最好在 `plan` 阶段就失败，而不是等到 `apply` 之后让云平台 API 报错。

原因很现实：

- CI/CD 能更早失败，节省时间。
- 错误信息由你控制，用户更容易修。
- 不会创建半成品资源。
- 不会让远端 API 拒绝请求后才发现问题。
- 可以把组织规范前置到代码里。

这就是 IaC 的价值：不是把错误推迟到线上，而是尽量提前发现。

## 7. `contains()` 的典型生产用途

这个函数在真实 Terraform 项目里非常常见。

### 限制环境名称

```hcl
contains(["dev", "test", "staging", "prod"], var.environment)
```

防止用户传入：

```text
production
prd
Prod
uat-prod
```

除非这些值本来就在团队标准里。

---

### 限制实例规格

```hcl
contains(["t3.micro", "t3.small", "t3.medium"], var.instance_type)
```

防止开发环境误用昂贵规格。

---

### 限制 region

```hcl
contains(["us-east-1", "ap-southeast-2"], var.region)
```

防止资源被部署到未批准区域。

---

### 限制模块模式

```hcl
contains(["minimal", "standard", "ha"], var.deployment_mode)
```

让用户只能选择平台团队定义好的部署模式。

---

## 8. 大小写问题要特别注意

`contains()` 是精确匹配。

也就是说：

```hcl
contains(["dev", "test", "prod"], "Prod")
```

结果是 `false`。

所以如果你的团队约定环境名都用小写，那就应该坚持小写。

也可以配合 `lower()` 做兼容：

```hcl
contains(["dev", "test", "prod"], lower(var.environment))
```

但这要看团队风格。严格一点的团队会直接拒绝大小写错误，因为这样能逼大家统一规范。

---

## 9. 错误信息要写得像给人看的

不好的错误信息：

```hcl
error_message = "Invalid value."
```

用户看到这个会很烦，因为他不知道怎么修。

更好的错误信息：

```hcl
error_message = "instance_size must be one of: small, medium, large."
```

好的错误信息应该包含：

- 哪个变量错了
- 为什么错
- 允许传哪些值
- 用户应该如何修改

这在 CI 里特别重要。因为出错的人不一定是写模块的人。

## 10. 标准运行流程

进入实验目录：

```sh
cd /workspace/practice/labs/38
```

初始化：

```sh
terraform init -input=false
```

格式检查：

```sh
terraform fmt -check
```

配置校验：

```sh
terraform validate
```

运行测试：

```sh
terraform test
```

生成 plan：

```sh
terraform plan -input=false -no-color -out=tfplan
```

执行 apply：

```sh
terraform apply -auto-approve tfplan
```

查看输出：

```sh
terraform output
```

清理资源：

```sh
terraform destroy -auto-approve
```

清理本地文件：

```sh
rm -f tfplan
rm -rf .terraform .terraform.lock.hcl
```

## 11. 如何手动验证非法值？

你可以故意传一个不在白名单里的规格：

```sh
terraform plan -var='instance_size=mega-large'
```

预期结果应该是失败，并提示类似：

```text
instance_size must be one of: ...
```

这说明变量验证生效了。

如果非法值居然通过了，说明：

- `validation` 没写对
- `condition` 条件太宽
- 白名单包含了不该包含的值
- 测试没有覆盖非法输入

## 13. 常见坑

### 坑 1：新增规格后忘记更新白名单

如果团队批准了新规格，但变量验证没更新，用户就会被挡住。

解决方式：

- 把允许值集中写在 `locals`
- README 同步更新
- 测试覆盖新规格

例如：

```hcl
locals {
  allowed_instance_sizes = ["small", "medium", "large"]
}
```

然后：

```hcl
condition = contains(local.allowed_instance_sizes, var.instance_size)
```

这样后续维护更清楚。

---

### 坑 2：错误信息没写允许值

不要只写“invalid instance size”。

应该直接告诉用户可选项：

```hcl
error_message = "instance_size must be one of: small, medium, large."
```

这能减少很多无意义沟通。

---

### 坑 3：大小写不统一

如果白名单是：

```hcl
["dev", "test", "prod"]
```

那 `"DEV"`、`"Prod"` 都不会通过。

这不是 Terraform 坑，这是精确匹配的结果。

---

### 坑 4：把所有治理都塞进变量验证

变量验证适合做输入边界控制，但它不是万能的。

更复杂的治理可能还需要：

- Policy as Code
- Sentinel
- OPA / Conftest
- Checkov
- Azure Policy / AWS Config
- CI 审批流程

变量验证是第一道门，不是唯一一道门。

## 14. 面试/考试记忆版

可以这样记：

- `validation`：给变量加约束。
- `contains()`：检查值是否在允许列表里。
- 允许列表适合限制环境名、规格、region、部署模式。
- 错误应该在 plan 阶段暴露。
- `contains()` 是精确匹配，大小写不同就是不同值。
- 错误信息要写清楚可选值。

一句话总结：

> `contains()` 适合做白名单校验，把 Terraform 变量从“随便传字符串”变成“只能选择平台批准的值”。

## 15. 本 Lab 的价值

Lab 38 的价值在于，它把 Terraform 从“能跑”提升到“可治理”。

很多初学者写 Terraform 时只关心资源能不能创建，但生产环境更关心：

- 谁可以传什么值
- 哪些规格被批准
- 哪些环境名是合法的
- 错误能不能提前失败
- 模块能不能被别人安全复用

这就是 Platform Engineering 的味道。  
不是每次靠人审，而是把规范写进模块，让错误在进入生产前就被挡住。
