# Terraform 实操训练 87 讲解：选择合适模块

> 原始实验：`Terraform 实操训练 87：选择合适模块`  
> 主题：Terraform modules / 模块选型 / 平台工程复用  
> 环境：Docker + `hashicorp/terraform:1.11`  
> 实验目录：`practice/labs/87`

---

## 1. 这一节到底在讲什么？

这一节的核心不是“怎么调用 module”，而是更靠近真实工程里的一个问题：

> **面对多个 Terraform module，平台团队应该如何判断哪个模块适合引入？**

在生产环境里，Terraform module 不是随便拿来就用的。一个 module 一旦进入团队基础设施代码库，就会影响：

- 资源命名规范
- 输入变量设计
- 输出值结构
- provider 版本约束
- state 地址
- 后续升级路径
- 安全默认值
- CI/CD 流程
- 团队排障成本

所以“选择合适模块”本质上是一个工程判断题，而不是语法题。

---

## 2. 为什么不能只看模块能不能跑？

很多初学者选择 module 的标准很简单：

> `terraform init` 能过，`terraform plan` 能出结果，就觉得这个模块可以用了。

这在练习环境里勉强可以，但在生产环境里远远不够。

一个 module 能跑，只能说明它暂时没有明显语法错误。它不代表：

- 它的默认配置安全
- 它的变量接口清晰
- 它的版本维护稳定
- 它适合你团队的命名规范
- 它方便接入监控、日志、标签、成本中心
- 它未来升级不会破坏 state
- 它适合放进 GitHub Actions / Azure DevOps / Terraform Cloud 这类流程

真实工程里，module 选择错误，后面会变成“基础设施技术债”。

---

## 3. 模块选型要看哪些维度？

### 3.1 维护活跃度

优先选择仍然有人维护的模块。判断方式包括：

- 最近是否有提交
- issue 是否长期无人处理
- 是否支持当前 Terraform 版本
- 是否支持当前 provider 大版本
- release 是否有清晰 changelog

如果一个模块两三年没人维护，哪怕现在能跑，也要谨慎。

---

### 3.2 Provider 版本约束是否合理

模块通常会声明类似：

```hcl
terraform {
  required_providers {
    azurerm = {
      source  = "hashicorp/azurerm"
      version = "~> 4.0"
    }
  }
}
```

需要重点看：

- 是否锁死了很旧的 provider 版本
- 是否和你根模块的 provider 版本冲突
- 是否使用了即将废弃的 resource 或 argument
- 是否有 `.terraform.lock.hcl` 配合版本锁定

模块选型不是只看 module 自己，还要看它能不能和你现有项目共存。

---

### 3.3 输入变量是否清晰

好的 module 通常会把关键配置暴露成变量，例如：

```hcl
variable "name" {
  type        = string
  description = "Resource name prefix"
}

variable "tags" {
  type        = map(string)
  description = "Common tags applied to all resources"
  default     = {}
}
```

需要关注：

- 变量是否有类型约束
- 变量是否有 description
- 默认值是否安全
- 是否暴露了团队真正需要控制的配置
- 是否隐藏了太多关键细节

差的 module 经常有两个极端：

1. 什么都暴露，变量爆炸，调用者很痛苦。
2. 什么都不暴露，默认值写死，生产不可控。

---

### 3.4 输出值是否可组合

Terraform module 的 `output` 很关键，因为它决定了这个模块能否被下游模块继续组合使用。

例如：

```hcl
output "resource_group_name" {
  value = azurerm_resource_group.this.name
}

output "subnet_ids" {
  value = {
    for k, subnet in azurerm_subnet.this : k => subnet.id
  }
}
```

好的 output 应该：

- 命名清晰
- 类型稳定
- 不泄露敏感值
- 能被下游模块直接引用
- 不因为内部重构轻易改变结构

如果一个 module 没有输出关键 ID、name、endpoint，那它就很难参与真实系统组合。

---

### 3.5 是否符合团队命名和标签规范

平台工程里，命名和标签不是装饰品，而是治理入口。

常见标签包括：

```hcl
tags = {
  environment = "dev"
  owner       = "platform"
  cost_center = "training"
  managed_by  = "terraform"
}
```

你要看 module 是否支持：

- 统一 tags
- 命名 prefix / suffix
- environment 区分
- cost center
- owner
- workload / application 标识

如果 module 不支持统一标签，后面做 FinOps、资源盘点、合规扫描都会很痛苦。

---

### 3.6 是否有测试

这一组 lab 前面已经讲到 `terraform test`，所以这里要把知识串起来。

一个可靠 module 最好至少有：

- `terraform validate`
- `terraform fmt`
- `terraform test`
- example 目录
- CI 自动检查
- 关键 output / variable 的断言

这也是为什么本实验要求运行：

```sh
terraform test
```

不是为了凑流程，而是训练你形成习惯：**模块不是看着顺眼就用，要能被验证。**

## 4. 本实验在训练什么能力？

原始实验要求进入：

```sh
cd /workspace/practice/labs/87
```

然后执行完整 Terraform 生命周期：

```sh
terraform init -input=false
terraform fmt -check
terraform validate
terraform test
terraform plan -input=false -no-color -out=tfplan
terraform apply -auto-approve tfplan
terraform output
terraform destroy -auto-approve
```

这套流程训练的是：

| 步骤 | 作用 |
|---|---|
| `terraform init` | 检查 provider 和 module 能否初始化 |
| `terraform fmt -check` | 检查代码格式是否符合 Terraform 标准 |
| `terraform validate` | 检查配置语法和基本引用是否正确 |
| `terraform test` | 检查模块行为是否符合断言 |
| `terraform plan` | 预览资源变化 |
| `terraform apply` | 创建或更新资源 |
| `terraform output` | 查看模块暴露的结果 |
| `terraform destroy` | 清理实验资源 |

对于“选择合适模块”这个主题来说，这些命令对应的是一个判断闭环：

> 能初始化、能格式化、能校验、能测试、能计划、能部署、能输出、能销毁，才算完成最低限度的模块验证。

## 5. 选择模块时的实战判断清单

以后你看到一个公共 module，可以按下面顺序判断。

### 第一步：看来源

优先级大概是：

1. 官方或云厂商维护模块
2. 大型社区长期维护模块
3. 公司内部平台团队维护模块
4. 个人维护模块
5. 不明来源、无版本、无文档模块

不是说个人模块一定不能用，而是要提高审查强度。

---

### 第二步：看版本

重点看：

- 是否有 release
- 是否支持语义化版本，例如 `v1.2.3`
- 是否有 breaking changes 说明
- 是否能固定版本引用

不要在生产中随便这样写：

```hcl
module "network" {
  source = "github.com/example/network-module"
}
```

更好的方式是固定版本：

```hcl
module "network" {
  source = "github.com/example/network-module?ref=v1.2.3"
}
```

否则上游一改，你的生产环境可能跟着炸。

---

### 第三步：看变量设计

变量设计要回答三个问题：

1. 调用者是否容易理解？
2. 默认值是否安全？
3. 是否支持团队必须控制的参数？

例如，下面这种变量就比裸 string 更安全：

```hcl
variable "environment" {
  type        = string
  description = "Deployment environment."

  validation {
    condition     = contains(["dev", "test", "stage", "prod"], var.environment)
    error_message = "environment must be one of dev, test, stage, prod."
  }
}
```

这类 validation 可以把错误挡在 plan 阶段，而不是等云资源创建失败才发现。

---

### 第四步：看输出是否稳定

如果一个 module 的 output 经常变，调用它的下游模块也会被连带破坏。

比如以前输出：

```hcl
output "subnet_id" {
  value = azurerm_subnet.this.id
}
```

后来改成：

```hcl
output "subnets" {
  value = {
    app = azurerm_subnet.app.id
    db  = azurerm_subnet.db.id
  }
}
```

这就是破坏性变化。下游代码要跟着改。

所以选 module 时要看它是否有稳定的 output contract。

---

### 第五步：看测试和示例

一个好 module 至少应该告诉你：

- 最小调用方式
- 常见调用方式
- 生产推荐调用方式
- 如何运行测试
- 测试覆盖了什么行为

如果 examples 很多、测试清晰，说明维护者知道这个 module 是要被别人长期使用的。

## 6. 和平台工程/SRE的关系

你以后做 AKS / EKS / SRE / Platform Engineering，这节非常重要。

平台团队经常会做一件事：

> 把云资源封装成内部标准 module，然后让业务团队复用。

例如：

- `aks-cluster`
- `eks-cluster`
- `vnet`
- `subnet`
- `storage-account`
- `rds-postgres`
- `monitoring-stack`
- `workload-identity`
- `github-oidc`
- `private-endpoint`

这时候 module 选型和设计就直接影响平台质量。

一个好 module 能让业务团队：

- 少写重复代码
- 少踩安全坑
- 自动获得标准标签
- 自动接入监控
- 自动符合网络规范
- 自动符合合规要求

一个烂 module 会让平台团队变成“Terraform 修锅队”。

## 7. 常见错误理解

### 错误 1：公共模块一定比自己写好

不一定。

公共模块适合通用场景，但你的公司可能有自己的：

- 命名规范
- 网络规范
- 安全基线
- 日志标准
- 成本标签
- 账号结构
- landing zone 设计

如果公共模块不适配这些要求，硬用反而会更麻烦。

---

### 错误 2：module 越复杂越专业

不对。

很多成熟平台 module 反而追求边界清晰、接口稳定。

复杂 module 的风险是：

- 变量太多
- 默认行为难预测
- state 结构复杂
- 升级困难
- 调试困难
- 业务团队不敢用

好 module 不是“功能堆满”，而是“抽象刚刚好”。

---

### 错误 3：只要能 plan 就算模块可用

不对。

最少要看：

- `fmt`
- `validate`
- `test`
- `plan`
- `apply`
- `output`
- `destroy`

这也是本 lab 要你跑完整流程的原因。

## 8. 本实验的排障重点

### 8.1 `terraform init` 失败

常见原因：

- provider 版本冲突
- module source 路径错误
- 网络无法下载 provider
- 本地模块目录不存在
- lock file 和当前平台不匹配

处理方式：

```sh
terraform init -upgrade
```

或者删除本地缓存后重试：

```sh
rm -rf .terraform .terraform.lock.hcl
terraform init -input=false
```

---

### 8.2 `terraform fmt -check` 失败

说明代码格式不符合 Terraform 标准。

修复：

```sh
terraform fmt -recursive
```

再检查：

```sh
terraform fmt -check
```

---

### 8.3 `terraform test` 失败

优先看：

- output 名称是否写错
- assert 条件是否和实际输出一致
- 测试文件是否引用了不存在的 local / variable / output
- run block 的 command 是 `plan` 还是 `apply`
- 是否把 apply 后才知道的值放进 plan 阶段断言

---

### 8.4 `plan` 和 `apply` 表现不一致

常见原因：

- 使用了 apply 后才生成的属性
- 资源依赖没有建模清楚
- provider 行为包含默认值补全
- 模块 output 暴露了 computed value

这类问题在真实云资源里更明显，比如公网 IP、自动生成的 hostname、托管身份 principal id 等。

## 9. 生产环境里的推荐做法

### 9.1 固定 module 版本

```hcl
module "network" {
  source = "github.com/org/terraform-azurerm-network?ref=v1.4.2"
}
```

不要让生产环境自动追踪上游最新代码。

---

### 9.2 写 wrapper module

如果公共 module 很强，但不完全符合团队规范，可以包一层内部 wrapper：

```hcl
module "base_network" {
  source = "github.com/community/network-module?ref=v2.0.0"

  name = local.name
  tags = local.standard_tags
}
```

业务团队只调用你的内部 module，不直接接触外部 module。

这样平台团队可以统一：

- 命名
- 标签
- 默认安全配置
- provider 版本
- 输出结构
- 升级节奏

---

### 9.3 把 module 放进 CI

至少包括：

```sh
terraform fmt -check
terraform init -input=false
terraform validate
terraform test
```

对于真实云资源，再加：

- plan artifact 保存
- 人工审批
- policy check
- security scan
- cost estimate
- drift detection

---

## 10. 你应该掌握的判断句

学完这节，你应该能说出：

> 我选择 Terraform module 时，不只看它能不能运行，还会看它的维护状态、版本约束、变量接口、输出契约、测试覆盖、标签治理、安全默认值和升级风险。生产环境里最好固定版本，并通过 wrapper module 把公共模块纳入团队自己的平台规范。

这句话就是这节课的工程含义。

## 11. 复盘问题

### 1. 为什么不能直接在生产里引用没有版本号的公共 module？

因为上游代码变化会直接影响你的基础设施代码，可能导致 plan 结果变化、资源重建、接口变更或部署失败。生产环境应该固定版本，确保可重复部署。

---

### 2. module 的 output 为什么重要？

因为 output 是模块和下游代码之间的契约。下游模块、根模块、CI/CD、监控配置都可能依赖这些 output。如果 output 命名或结构不稳定，整个调用链都会被影响。

---

### 3. 为什么选择 module 要看 tags 支持？

因为 tags 是成本治理、资源归属、环境区分、合规扫描和自动化清理的重要入口。如果 module 不支持统一 tags，后续治理成本会明显增加。

---

### 4. `terraform test` 在 module 选择里有什么价值？

它可以验证 module 的关键行为是否符合预期，例如输出值、命名规则、变量组合、断言条件等。它让 module 从“看起来能用”变成“可以被自动验证”。

---

### 5. 公共 module 和内部 module 应该怎么配合？

公共 module 可以作为底层能力来源，内部 module 或 wrapper module 负责统一公司规范。这样既能复用社区能力，又能保留平台团队对安全、命名、标签、版本和输出契约的控制。

---

## 12. 本节结论

Terraform module 选型的重点不是“找一个能跑的模块”，而是判断它是否适合长期进入你的基础设施体系。

这一节真正训练的是平台工程判断力：

- 能不能复用
- 能不能治理
- 能不能测试
- 能不能升级
- 能不能被团队稳定使用

能跑只是入门。能长期维护，才是工程。
