# Terraform 实操训练 110：多 Provider Alias 配置（讲解版）

> 来源：`doc/110.txt`  
> 主题：在同一个 root module 中定义多个 AWS provider alias，并让不同资源使用不同 provider。  
> 实验环境：LocalStack，本实验不访问真实 AWS。

---

## 1. 这一节到底在学什么？

这一节的核心不是“多写几个 provider 块”这么简单，而是理解 Terraform 在一个 root module 里面如何区分不同的 provider 配置。

在真实项目里，你经常会遇到这种情况：

- 同一个 AWS 账号里，要同时操作多个 region；
- 同一套 Terraform 配置里，要管理不同环境；
- 某些资源必须走默认 provider，某些资源必须走 alias provider；
- module 或 resource 需要显式指定使用哪个 provider。

所以 provider alias 的本质是：

> **同一个 provider 类型，可以有多份不同配置；Terraform 通过 alias 区分这些配置，并通过 `provider = aws.xxx` 指定资源到底使用哪一份配置。**

这一节训练的是 Terraform Provider 绑定关系，不是 AWS 服务本身。

---

## 2. 为什么需要 Provider Alias？

默认情况下，Terraform 只会使用无 alias 的 provider 配置，例如：

```hcl
provider "aws" {
  region = "us-east-1"
}
```

这份配置就是默认 AWS provider。资源如果没有显式指定 provider，就会默认使用它：

```hcl
resource "aws_s3_bucket" "default" {
  bucket = "example-default-bucket"
}
```

但是如果你还想在同一个 root module 里操作另一个 region，比如 `us-west-2`，就不能再写另一个普通的 `provider "aws"` 了。你需要给第二份配置加 alias：

```hcl
provider "aws" {
  alias  = "west"
  region = "us-west-2"
}
```

然后资源通过 `provider` 参数显式绑定：

```hcl
resource "aws_s3_bucket" "west" {
  provider = aws.west
  bucket   = "example-west-bucket"
}
```

这里的关键是：

- `aws` 是 provider 类型；
- `west` 是这份 provider 配置的别名；
- `aws.west` 是资源引用 alias provider 的写法；
- 没写 `provider = ...` 的资源会走默认 provider。

---

## 3. 可以把它理解成什么？

可以把 provider alias 理解成“多把 AWS 遥控器”。

默认 provider 是第一把遥控器：

```hcl
provider "aws" {
  region = "us-east-1"
}
```

alias provider 是第二把遥控器：

```hcl
provider "aws" {
  alias  = "west"
  region = "us-west-2"
}
```

资源如果不说用哪把遥控器，Terraform 就默认拿第一把。  
资源如果写了：

```hcl
provider = aws.west
```

那就明确告诉 Terraform：

> 这个资源别用默认遥控器，用 `west` 这把。

这就是 provider alias 的价值：**把资源和 provider 配置的关系写清楚，避免 Terraform 自己猜。**

## 4. 本实验和真实 AWS 的边界

本节实验使用 LocalStack 模拟 AWS API：

```text
http://localhost:4566
```

也就是说：

- 不访问真实 AWS；
- 不会真正创建云上资源；
- 使用的是假凭证，例如 `test/test`；
- 重点是验证 Terraform Provider 行为；
- LocalStack 只是帮你跑通 provider 初始化、plan、apply、destroy 的闭环。

所以你不要把本实验理解成“我真的在 AWS 多 region 创建资源了”。  
更准确地说，它是在本地验证：

> Terraform 能不能正确识别多个 AWS provider alias，并把资源绑定到指定 provider 配置上。

## 5. Provider Alias 的典型写法

一个 root module 中常见结构如下：

```hcl
terraform {
  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 5.0"
    }
  }
}

provider "aws" {
  region                      = "us-east-1"
  access_key                  = "test"
  secret_key                  = "test"
  s3_use_path_style           = true
  skip_credentials_validation = true
  skip_metadata_api_check     = true
  skip_requesting_account_id  = true

  endpoints {
    s3 = "http://localhost:4566"
  }
}

provider "aws" {
  alias                       = "west"
  region                      = "us-west-2"
  access_key                  = "test"
  secret_key                  = "test"
  s3_use_path_style           = true
  skip_credentials_validation = true
  skip_metadata_api_check     = true
  skip_requesting_account_id  = true

  endpoints {
    s3 = "http://localhost:4566"
  }
}
```

资源绑定方式：

```hcl
resource "aws_s3_bucket" "east_bucket" {
  bucket = "tf-lab-110-east"
}

resource "aws_s3_bucket" "west_bucket" {
  provider = aws.west
  bucket   = "tf-lab-110-west"
}
```

这里非常适合观察 provider 行为：

- `east_bucket` 没写 provider，走默认 AWS provider；
- `west_bucket` 写了 `provider = aws.west`，走 alias provider；
- 两个 provider 都连向 LocalStack，但 region 配置不同；
- 在真实 AWS 中，这种写法通常用于多 region、多账号、多角色场景。

## 6. root module 中的 Provider Alias 关系

这一节尤其要注意：provider alias 是在 root module 中定义的。

root module 是你运行 `terraform init/plan/apply` 的目录。  
也就是本节的：

```sh
practice/labs/110/
```

Terraform 会从这个目录读取配置，解析 provider，并构建资源图。

Provider alias 的生命周期由当前 root module 管理。  
如果资源或 module 想用 alias provider，就必须能在当前调用链里拿到这个 provider 配置。

简单说：

> Provider alias 不是一个全局魔法变量，它必须在 Terraform 配置关系里被明确声明、明确引用。

---

## 7. 资源如何选择 Provider？

Terraform 资源选择 provider 的规则可以这样记：

### 情况 1：资源没有写 `provider`

```hcl
resource "aws_s3_bucket" "example" {
  bucket = "example"
}
```

Terraform 会使用默认 provider：

```hcl
provider "aws" {
  region = "us-east-1"
}
```

### 情况 2：资源写了 alias provider

```hcl
resource "aws_s3_bucket" "example" {
  provider = aws.west
  bucket   = "example"
}
```

Terraform 会使用：

```hcl
provider "aws" {
  alias  = "west"
  region = "us-west-2"
}
```

### 情况 3：资源引用了不存在的 alias

```hcl
resource "aws_s3_bucket" "bad" {
  provider = aws.eu
  bucket   = "bad"
}
```

但是配置里没有：

```hcl
provider "aws" {
  alias = "eu"
}
```

那就会报错。Terraform 不会凭空帮你创建 provider alias。

## 9. 观察重点

跑实验时不要只看“成功了没有”。这一节你要盯住这些点：

### 1. `terraform init` 是否能识别 AWS provider

如果 provider 下载失败，先看网络、版本约束、`.terraform.lock.hcl`。

### 2. `terraform validate` 是否能识别 alias provider

如果写错了：

```hcl
provider = aws.wset
```

这种拼写错误通常会在 validate 或 plan 阶段暴露。

### 3. `terraform plan` 中资源是否绑定到预期 provider

重点看资源是否按你期望的 provider 配置执行。

### 4. `terraform test` 是否验证了核心行为

这类 lab 的测试不一定是为了测 AWS 本身，而是为了测 Terraform 配置结构是否符合预期。

## 10. 常见错误

### 错误 1：只定义 alias provider，没有默认 provider

比如只写：

```hcl
provider "aws" {
  alias  = "west"
  region = "us-west-2"
}
```

然后资源没有写 provider：

```hcl
resource "aws_s3_bucket" "example" {
  bucket = "example"
}
```

Terraform 可能会尝试使用默认 `aws` provider，但你没有定义默认 provider，容易引发混乱。

更稳的做法是：

- 要么保留默认 provider；
- 要么所有资源都显式写 `provider = aws.xxx`；
- 项目规范里统一一种风格。

### 错误 2：alias 写了，但资源没绑定

你写了：

```hcl
provider "aws" {
  alias  = "west"
  region = "us-west-2"
}
```

但是资源还是：

```hcl
resource "aws_s3_bucket" "west_bucket" {
  bucket = "west-bucket"
}
```

这并不会自动使用 `aws.west`。  
Terraform 不会根据资源名字里的 `west` 来猜 provider。

必须写：

```hcl
resource "aws_s3_bucket" "west_bucket" {
  provider = aws.west
  bucket   = "west-bucket"
}
```

### 错误 3：把 alias 当成 provider 名字

错误理解：

```hcl
provider "aws.west" {
  region = "us-west-2"
}
```

这是错的。

正确写法：

```hcl
provider "aws" {
  alias  = "west"
  region = "us-west-2"
}
```

引用时才写：

```hcl
provider = aws.west
```

### 错误 4：LocalStack endpoint 配置不完整

AWS provider 默认会访问真实 AWS endpoint。  
在 LocalStack 实验里，如果 endpoint、假凭证、跳过校验参数没配好，就可能出现奇怪的认证或网络错误。

## 11. 真实项目中的迁移方式

真实 AWS 项目里，多 provider alias 常见于这几类场景：

### 多 Region

```hcl
provider "aws" {
  region = "ap-southeast-2"
}

provider "aws" {
  alias  = "use1"
  region = "us-east-1"
}
```

典型用途：

- 主 region 部署应用；
- `us-east-1` 管理 CloudFront 相关证书；
- 跨 region 灾备资源；
- 多 region 监控和日志。

### 多账号

```hcl
provider "aws" {
  region = "ap-southeast-2"
}

provider "aws" {
  alias  = "security"
  region = "ap-southeast-2"

  assume_role {
    role_arn = "arn:aws:iam::123456789012:role/security-admin"
  }
}
```

典型用途：

- workload account 创建业务资源；
- security account 管理安全审计；
- network account 管理共享 VPC；
- logging account 管理集中日志。

### 多角色

同一个账号内也可能使用多个 role：

```hcl
provider "aws" {
  alias  = "readonly"
  region = "ap-southeast-2"

  assume_role {
    role_arn = "arn:aws:iam::123456789012:role/readonly"
  }
}
```

这种方式可以把权限边界写进 Terraform 配置结构里。

## 12. 考试容易怎么考？

Terraform 考试中，这一类知识点通常不会考你 AWS 细节，而是考 Terraform provider 机制：

### 考点 1：多个同类型 provider 怎么区分？

答案：使用 `alias`。

### 考点 2：资源怎么指定使用 alias provider？

答案：

```hcl
provider = aws.alias_name
```

### 考点 3：没写 provider 的资源用哪个？

答案：默认 provider，也就是没有 alias 的那份 provider 配置。

### 考点 4：provider alias 能不能自动传给 module？

不能乱想。module 场景需要通过 `providers` map 显式传递，尤其是 child module 需要 alias provider 时。

### 考点 5：alias 是不是 provider source？

不是。  
`source = "hashicorp/aws"` 是 provider 插件来源。  
`alias = "west"` 是同一个 provider 类型下的一份配置别名。

## 14. 排障 checklist

遇到问题时按这个顺序查：

1. `required_providers` 是否声明了 `hashicorp/aws`；
2. 是否至少有一份默认 provider，或者资源是否全部显式指定 alias provider；
3. alias 名字是否一致；
4. 资源里 `provider = aws.xxx` 是否拼写正确；
5. LocalStack 是否启动；
6. endpoint 是否指向 `http://localhost:4566`；
7. 是否使用了假凭证 `test/test`；
8. 是否配置了跳过真实 AWS 校验的参数；
9. `.terraform.lock.hcl` 是否因为 provider 版本变化导致冲突；
10. `terraform fmt` 和 `terraform validate` 是否先通过。

## 15. 本节一句话总结

Provider alias 解决的是：

> **同一个 Terraform 配置中，如何让不同资源使用同一个 provider 的不同配置。**

这不是花活，这是生产 Terraform 里非常常见的基础能力。  
多 region、多账号、多角色，只要你想写得清楚、可维护，provider alias 基本绕不开。

---

## 16. 复习卡片

- `provider "aws"`：默认 AWS provider；
- `provider "aws" { alias = "west" }`：AWS alias provider；
- `provider = aws.west`：资源使用 alias provider；
- 没写 `provider`：资源使用默认 provider；
- `source = "hashicorp/aws"`：provider 插件来源，不是 alias；
- LocalStack：本地模拟 AWS API，不是真实 AWS；
- 真实项目：常用于多 region、多账号、多 role；
- module 场景：需要关注 `providers` map 显式传递。
