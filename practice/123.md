## 2. 知识点讲解

HCP Terraform 创建 workspace 时常见 workflow 包括 Version Control Workflow、CLI-driven Workflow 和 API-driven Workflow。CLI-driven Workflow 适合已经熟悉 Terraform CLI 的用户：代码可以保留在本地目录，用户仍然运行 `terraform plan`、`terraform apply` 等命令，但运行会被 HCP Terraform 接管，state 和 run 记录也进入平台。

与 Version Control Workflow 相比，CLI-driven Workflow 不以 Git 仓库变更作为主要触发源。它更像是把本地 CLI 操作连接到远端 workspace。要做到这一点，Terraform 配置里通常需要 `cloud` block，指定 organization 和 workspace。

考试中常见的问题是让你根据场景选择 workflow。如果团队强调 Git PR、审计和自动触发，优先想到 Version Control Workflow。如果用户强调本地 CLI 习惯、代码暂不接入 VCS，但希望远端执行和 remote state，CLI-driven Workflow 更合适。

## 3. 考试选择题

1. CLI-driven Workflow 的典型特点是什么？
   - A. 本地运行 Terraform CLI 命令，但 run 与 state 由 HCP Terraform 管理
   - B. 完全不需要 Terraform CLI
   - C. 必须从 GitHub webhook 触发
   - D. 只能用于 Terraform Enterprise 离线安装

2. CLI-driven Workflow 通常需要在 Terraform 配置中添加什么？
   - A. `cloud` block，指定 organization 和 workspace
   - B. `docker compose` block
   - C. `provider "git"` block
   - D. `local-exec` 强制打开浏览器

3. 哪个场景更适合 Version Control Workflow？
   - A. 所有变更都来自 Git PR，并希望提交后自动 plan
   - B. 临时本地目录中手动运行 CLI
   - C. 只想测试 `terraform console`
   - D. 只做 Markdown 格式检查

## 4. 判断题

1. CLI-driven Workflow 仍然要求用户理解 Terraform CLI。
2. CLI-driven Workflow 的代码一定必须托管在 GitHub。
3. `cloud` block 可以把本地配置与 HCP Terraform organization 和 workspace 关联起来。

## 5. 场景题

1. 平台团队希望工程师继续使用本地 `terraform plan`，但统一保存 state 和 run 历史。应该选择哪种 workflow？

2. 公司要求任何 Terraform 变更必须经过 Git 分支策略和 PR 审批。CLI-driven Workflow 是否是首选？请说明。

## 6. 参考答案与解析

选择题：1-A，2-A，3-A。

判断题：1-对，2-错，3-对。

场景题参考：

1. 适合 CLI-driven Workflow，因为它保留本地 CLI 入口，同时把运行和状态交给 HCP Terraform。
2. 通常不是首选。这个场景更适合 Version Control Workflow，因为代码变更和运行触发都围绕 Git 审批链路。

## 7. 复习提醒

- 三种 workflow 的选择题很常见：VCS、CLI、API。
- CLI-driven 不等于本地 backend，它仍然可以使用 HCP Terraform 保存 state 和 run。
- `cloud` block 是本节关键词。
