# 第 106 节：AWS CLI `config` 与 `credentials` 文件讲解

> 原始主题：理解 AWS CLI 的 `config` 文件保存 `region/output`，`credentials` 文件保存 `access key/secret key`。  
> 实验环境：LocalStack，本实验不访问真实 AWS。  
> 课程来源：`doc/106.txt`  
> 参考实现：`practice/labs/106/`

---

## 1. 本节你要真正学会什么？

这一节不是让你背 AWS CLI 文件名，而是让你理解：

1. AWS CLI 为什么要把配置拆成两个文件。
2. Terraform AWS Provider 如何读取这些配置。
3. LocalStack 实验里为什么可以使用假的 AK/SK。
4. 真实 AWS 环境中，哪些东西绝对不能乱写、不能提交到 Git。
5. 当 Terraform 报 “找不到凭证”“region 不存在”“provider 初始化失败” 时，应该往哪里排查。

一句话总结：

> `~/.aws/config` 主要放“怎么连 AWS”，比如 region、output、profile 配置。  
> `~/.aws/credentials` 主要放“用什么身份连 AWS”，比如 access key 和 secret key。

---

## 2. AWS CLI 的两个核心文件

AWS CLI 默认会读取用户家目录下的 `.aws` 目录：

```bash
~/.aws/config
~/.aws/credentials
```

在 Linux/macOS 中通常是：

```bash
/home/<user>/.aws/config
/home/<user>/.aws/credentials
```

在 Windows 中通常是：

```powershell
C:\Users\<user>\.aws\config
C:\Users\<user>\.aws\credentials
```

---

## 3. `config` 文件：保存 region / output / profile 行为

`~/.aws/config` 更偏向“客户端配置”。

典型内容如下：

```ini
[default]
region = us-east-1
output = json
```

含义：

| 字段 | 作用 |
|---|---|
| `region` | 默认 AWS 区域，例如 `us-east-1`、`ap-southeast-2` |
| `output` | AWS CLI 输出格式，例如 `json`、`table`、`text` |
| `[default]` | 默认 profile，未显式指定 profile 时使用它 |

例如：

```bash
aws s3 ls
```

如果你没有额外指定 region，AWS CLI 会尝试使用 `~/.aws/config` 里的默认 region。

---

## 4. `credentials` 文件：保存 AK / SK

`~/.aws/credentials` 更偏向“身份凭证”。

典型内容如下：

```ini
[default]
aws_access_key_id = test
aws_secret_access_key = test
```

含义：

| 字段 | 作用 |
|---|---|
| `aws_access_key_id` | Access Key ID，类似用户名 |
| `aws_secret_access_key` | Secret Access Key，类似密码 |
| `[default]` | 默认 profile 的凭证 |

在真实 AWS 中，这两个值是敏感信息，不能提交到 GitHub，不能写进公开文档，也不能贴到聊天窗口里。

但本节实验用的是 LocalStack，所以可以使用假凭证：

```ini
aws_access_key_id = test
aws_secret_access_key = test
```

原因是 LocalStack 只是本地模拟 AWS API，不会拿这组凭证去访问真实 AWS。

---

## 5. 为什么 AWS 要拆成两个文件？

你可以这样理解：

```text
config       = 我想怎么连接 AWS
credentials = 我是谁，我凭什么连接 AWS
```

也就是：

| 文件 | 主要内容 | 是否敏感 |
|---|---|---|
| `~/.aws/config` | region、output、profile 行为 | 一般不敏感 |
| `~/.aws/credentials` | access key、secret key | 高度敏感 |
| Terraform 代码 | 声明资源与 provider 行为 | 取决于是否写死敏感信息 |

拆开之后，好处是：

1. 配置和凭证职责清晰。
2. 可以让多个 profile 复用不同区域和身份。
3. 更容易避免把真实凭证写进 Terraform 代码。
4. CI/CD 中也更方便用环境变量或 secret 注入凭证。

---

## 6. Profile 是什么？

Profile 可以理解为“多套 AWS 身份和配置”。

比如你有三套环境：

- 个人测试账号
- 公司 dev 账号
- 公司 prod 账号

就可以配置多个 profile。

### 6.1 `credentials` 文件示例

```ini
[default]
aws_access_key_id = test
aws_secret_access_key = test

[dev]
aws_access_key_id = DEV_ACCESS_KEY
aws_secret_access_key = DEV_SECRET_KEY

[prod]
aws_access_key_id = PROD_ACCESS_KEY
aws_secret_access_key = PROD_SECRET_KEY
```

### 6.2 `config` 文件示例

注意：在 `config` 文件里，非 default profile 要写成 `[profile name]`。

```ini
[default]
region = us-east-1
output = json

[profile dev]
region = ap-southeast-2
output = json

[profile prod]
region = us-east-1
output = json
```

### 6.3 使用指定 profile

```bash
aws sts get-caller-identity --profile dev
```

或者设置环境变量：

```bash
export AWS_PROFILE=dev
aws sts get-caller-identity
```

Terraform 里也可以指定：

```hcl
provider "aws" {
  profile = "dev"
  region  = "ap-southeast-2"
}
```

## 7. 本节为什么用 LocalStack？

本实验不访问真实 AWS，而是通过 LocalStack 模拟 AWS API。

LocalStack 默认 endpoint：

```text
http://localhost:4566
```

你可以把它理解为：

```text
Terraform / AWS CLI
        |
        v
http://localhost:4566
        |
        v
LocalStack 模拟出来的 AWS 服务
```

所以本实验中的凭证：

```text
test/test
```

只是为了满足 AWS SDK / Provider 的格式要求，并不代表真实 AWS 账号。

这点非常重要：

> LocalStack 需要“看起来像 AWS 凭证”的值，但不会用这组假凭证登录真实 AWS。

---

## 8. Terraform AWS Provider 如何读取这些信息？

Terraform AWS Provider 可以从多种地方读取 region 和 credentials，常见优先级包括：

1. provider block 里显式写的参数。
2. 环境变量。
3. AWS shared config / credentials 文件。
4. EC2/ECS/EKS 等运行环境里的 IAM Role。

在本实验里，常见写法可能类似：

```hcl
provider "aws" {
  region                      = "us-east-1"
  access_key                  = "test"
  secret_key                  = "test"
  s3_use_path_style           = true
  skip_credentials_validation = true
  skip_metadata_api_check     = true
  skip_requesting_account_id  = true

  endpoints {
    s3 = "http://localhost:4566"
  }
}
```

或者让 provider 从本地 AWS 配置文件中读取：

```hcl
provider "aws" {
  region  = "us-east-1"
  profile = "default"

  endpoints {
    s3 = "http://localhost:4566"
  }

  s3_use_path_style           = true
  skip_credentials_validation = true
  skip_metadata_api_check     = true
  skip_requesting_account_id  = true
}
```

这两种方式都可以用于本地实验。

但真实项目里更推荐：

- 不把 AK/SK 写死在 `.tf` 文件里。
- 使用环境变量、SSO、IAM Role、CI/CD secret。
- 把 Terraform 状态和凭证安全分开管理。

## 9. LocalStack 实验中的典型命令

### 9.1 启动 LocalStack

如果课程目录里提供了 `docker-compose.yml`，通常可以执行：

```bash
docker compose up -d
```

确认容器运行：

```bash
docker ps
```

检查 LocalStack 是否可访问：

```bash
curl http://localhost:4566/_localstack/health
```

---

### 9.2 准备 AWS CLI 配置

创建目录：

```bash
mkdir -p ~/.aws
```

写入 `credentials`：

```bash
cat > ~/.aws/credentials <<'EOF'
[default]
aws_access_key_id = test
aws_secret_access_key = test
EOF
```

写入 `config`：

```bash
cat > ~/.aws/config <<'EOF'
[default]
region = us-east-1
output = json
EOF
```

验证文件：

```bash
cat ~/.aws/config
cat ~/.aws/credentials
```

---

### 9.3 使用 AWS CLI 访问 LocalStack

访问真实 AWS 时，AWS CLI 默认会打到 AWS 官方 endpoint。

访问 LocalStack 时，需要显式指定 endpoint：

```bash
aws --endpoint-url=http://localhost:4566 s3 ls
```

如果要创建一个测试 bucket：

```bash
aws --endpoint-url=http://localhost:4566 s3 mb s3://tf-lab-106-demo
```

查看 bucket：

```bash
aws --endpoint-url=http://localhost:4566 s3 ls
```

删除 bucket：

```bash
aws --endpoint-url=http://localhost:4566 s3 rb s3://tf-lab-106-demo
```

## 11. 真实 AWS 与 LocalStack 的边界

这个实验的价值是理解配置机制，但你要清楚它和真实 AWS 的区别。

| 对比项 | LocalStack | 真实 AWS |
|---|---|---|
| endpoint | `http://localhost:4566` | AWS 官方 endpoint |
| 凭证 | 可以用 `test/test` | 必须是真实凭证、SSO 或 IAM Role |
| 费用 | 本地模拟，通常无 AWS 费用 | 可能产生真实费用 |
| 权限模型 | 模拟为主 | IAM 权限严格生效 |
| 服务覆盖 | 只覆盖部分 AWS 行为 | 完整 AWS 服务 |
| 适合场景 | 学习、单元测试、本地验证 | 生产、预发、真实集成 |

不要把 LocalStack 实验误认为完整生产验证。它适合练 Terraform 基础路径，但不能替代真实云环境里的权限、网络、计费、配额和服务限制验证。

## 12. 常见错误与排查

### 12.1 报错：找不到 credentials

典型现象：

```text
NoCredentialProviders
failed to refresh cached credentials
```

排查：

```bash
ls -la ~/.aws
cat ~/.aws/credentials
```

确认有：

```ini
[default]
aws_access_key_id = test
aws_secret_access_key = test
```

如果使用 profile，也要确认 profile 名称一致。

---

### 12.2 报错：找不到 region

典型现象：

```text
MissingRegion
could not find region configuration
```

排查：

```bash
cat ~/.aws/config
```

确认有：

```ini
[default]
region = us-east-1
```

或者在 Terraform provider 中显式写：

```hcl
provider "aws" {
  region = "us-east-1"
}
```

---

### 12.3 AWS CLI 不小心访问真实 AWS

如果你忘了加 endpoint：

```bash
aws s3 ls
```

这条命令默认会访问真实 AWS，而不是 LocalStack。

本实验中访问 LocalStack 应该写：

```bash
aws --endpoint-url=http://localhost:4566 s3 ls
```

Terraform 中则要确认 provider 配置了 LocalStack endpoint。

---

### 12.4 Terraform 访问 LocalStack 失败

排查顺序：

```bash
docker ps
curl http://localhost:4566/_localstack/health
terraform init
terraform plan
```

重点看：

1. LocalStack 容器是否启动。
2. endpoint 是否是 `http://localhost:4566`。
3. provider 是否跳过了真实 AWS 账号校验。
4. credentials 是否存在。
5. region 是否存在。

## 13. 面试怎么讲这一节？

可以这样回答：

> AWS CLI 默认使用 `~/.aws/config` 和 `~/.aws/credentials`。  
> `config` 主要保存 region、output 和 profile 配置；`credentials` 保存 access key 和 secret key。  
> Terraform AWS Provider 可以读取这些 shared config，也可以通过环境变量、provider 参数或 IAM Role 获取配置。  
> 在 LocalStack 实验中，我们会使用 `test/test` 作为假凭证，并通过 `http://localhost:4566` endpoint 访问本地模拟 AWS。  
> 真实项目中不能把 AK/SK 写死进 Terraform，也不能提交到 Git，应该使用 SSO、IAM Role 或 CI/CD Secret 管理凭证。

这个回答就很稳，不浮夸，也不只是背命令。

## 14. 迁移到真实项目时怎么做？

### 14.1 本地开发

推荐优先使用 AWS SSO：

```bash
aws configure sso
aws sso login --profile dev
```

Terraform：

```hcl
provider "aws" {
  profile = "dev"
  region  = "ap-southeast-2"
}
```

执行：

```bash
terraform init
terraform plan
```

---

### 14.2 CI/CD 环境

不要在代码里写：

```hcl
access_key = "xxx"
secret_key = "xxx"
```

更好的方式：

- GitHub Actions 使用 OIDC Assume Role。
- GitLab CI 使用受保护变量或 OIDC。
- Jenkins 使用凭证管理系统。
- AWS 内部运行环境使用 IAM Role。

核心原则：

> 凭证应该由运行环境安全注入，而不是写死在 Terraform 代码里。

## 17. 最终结论

第 106 节的核心不是 S3，也不是 LocalStack 本身，而是 AWS 工具链的配置模型。

你要记住：

> AWS CLI 和 Terraform 都需要两类信息：  
> 第一类是“我连哪里”，也就是 region、endpoint、profile。  
> 第二类是“我是谁”，也就是 credentials 或 IAM Role。

LocalStack 让你可以在不花钱、不访问真实 AWS 的情况下，把这套机制练熟。  
等你进入真实 AWS 项目时，只需要把 endpoint、凭证来源、权限模型替换成生产级方案，底层思路是一样的。
