# Terraform 考试训练 170 讲解版：CSV 数据转换与多种输出形态

## 1. 本节定位

第 170 节属于综合挑战系列：**挑战七解法：CSV 数据转换与多种输出形态**。

这类题的重点不是死记某一个 AWS 资源参数，而是训练你在限定时间内完成一套 Terraform 闭环：

1. 读懂题目输入是什么；
2. 判断输入数据要转换成什么结构；
3. 用 Terraform 表达式完成转换；
4. 用 `output` 暴露不同形态的结果；
5. 通过 `plan/apply/output/verify/destroy` 完成验证。

本节的核心关键词可以记成一句话：

> CSV 是原始输入，Terraform 需要把它转换成可计算的数据结构，再输出成题目要求的多种形态。

---

## 2. 为什么 CSV 转换会出现在 Terraform 题里

Terraform 并不只是“写几个 resource”。在真实工程里，经常会遇到这种情况：

- 网络规则来自表格；
- 用户清单来自 CSV；
- bucket、tag、环境配置来自外部文件；
- 不同环境需要复用同一份数据源；
- 输出结果需要给其他模块、CI/CD 或人工检查使用。

所以题目考 CSV，不是在考 Excel，而是在考你是否理解 Terraform 的数据流：

```text
外部文件
  ↓
file()
  ↓
csvdecode()
  ↓
list(object)
  ↓
for 表达式 / map 转换 / group 转换
  ↓
resource for_each 或 output
```

本节对应的练习是在 LocalStack 或本地 provider 中完成 Terraform 路径闭环，不访问真实 AWS 账号。

---

## 3. 最核心函数：`file()` + `csvdecode()`

典型写法如下：

```hcl
locals {
  raw_csv = file("${path.module}/data/items.csv")
  rows    = csvdecode(local.raw_csv)
}
```

这里分两步：

- `file()`：读取本地文件内容，结果是字符串；
- `csvdecode()`：把 CSV 字符串解析成 Terraform 可处理的数据结构。

假设 CSV 内容是：

```csv
name,env,team
app-dev,dev,platform
app-prod,prod,platform
```

那么 `csvdecode()` 后，大概会变成：

```hcl
[
  {
    name = "app-dev"
    env  = "dev"
    team = "platform"
  },
  {
    name = "app-prod"
    env  = "prod"
    team = "platform"
  }
]
```

考试里你要抓住一点：

> `csvdecode()` 的结果通常是 `list(object)`，不是 map。

这很重要。因为很多资源的 `for_each` 更适合 map，而不是 list。

---

## 4. list(object) 转 map：为 `for_each` 做准备

如果题目要求根据 CSV 创建多个资源，通常要把 list 转成 map：

```hcl
locals {
  items_by_name = {
    for item in local.rows :
    item.name => item
  }
}
```

转换后的结构类似：

```hcl
{
  "app-dev" = {
    name = "app-dev"
    env  = "dev"
    team = "platform"
  }

  "app-prod" = {
    name = "app-prod"
    env  = "prod"
    team = "platform"
  }
}
```

这样就可以用于：

```hcl
resource "local_file" "item" {
  for_each = local.items_by_name

  filename = "${path.module}/generated/${each.key}.txt"
  content  = "name=${each.value.name}, env=${each.value.env}, team=${each.value.team}"
}
```

记忆方式：

> CSV 解码后是 list；资源批量创建更喜欢 map；中间靠 `for` 表达式转换。

## 5. 多种 output 形态是什么意思

题目说“多种输出形态”，一般不是让你随便输出，而是让你从同一份数据中整理出不同角度的结果。

常见 output 类型包括：

### 5.1 输出原始行

```hcl
output "rows" {
  value = local.rows
}
```

适合验证 CSV 是否被正确读取和解析。

---

### 5.2 输出名称列表

```hcl
output "names" {
  value = [
    for item in local.rows : item.name
  ]
}
```

结果类似：

```hcl
[
  "app-dev",
  "app-prod"
]
```

这类输出适合给人快速检查“读到了哪些对象”。

---

### 5.3 输出 map 结构

```hcl
output "items_by_name" {
  value = local.items_by_name
}
```

这类输出适合给其他模块或 `for_each` 消费。

---

### 5.4 输出筛选后的数据

比如只输出 `prod` 环境：

```hcl
output "prod_items" {
  value = [
    for item in local.rows : item
    if item.env == "prod"
  ]
}
```

考试看到 `if` 放在 `for` 表达式末尾，不要慌，它表示过滤。

---

### 5.5 输出重组后的对象

```hcl
output "summary" {
  value = {
    total = length(local.rows)
    names = [
      for item in local.rows : item.name
    ]
  }
}
```

这类输出通常用于“汇总型答案”。

---

## 6. 本节最容易错的地方

### 易错点 1：以为 `csvdecode()` 直接返回 map

错。它通常返回的是对象列表。

所以你不能默认这样写：

```hcl
for_each = local.rows
```

更稳的写法是先转 map：

```hcl
for_each = {
  for item in local.rows :
  item.name => item
}
```

---

### 易错点 2：CSV 表头就是字段名

CSV 第一行会变成对象的属性名。

例如：

```csv
bucket_name,environment
demo-bucket,dev
```

解析后字段是：

```hcl
item.bucket_name
item.environment
```

如果你写成 `item.name` 或 `item.env`，就会报错。

---

### 易错点 3：CSV 的值默认都是字符串

即使 CSV 里写的是：

```csv
name,enabled,count
app,true,3
```

`csvdecode()` 读出来通常也是字符串形式。

如果你需要布尔值或数字，需要显式转换，例如：

```hcl
enabled = item.enabled == "true"
count   = tonumber(item.count)
```

考试如果考类型转换，重点看 `tonumber()`、字符串比较、条件表达式。

---

### 易错点 4：`for_each` 的 key 必须稳定且唯一

如果你用 CSV 某一列作为 key，这一列必须唯一。

```hcl
locals {
  items_by_name = {
    for item in local.rows :
    item.name => item
  }
}
```

如果 CSV 里有两个相同 `name`，这个转换会出问题。

工程上，适合做 key 的字段通常是：

- name；
- id；
- bucket name；
- environment + name 组合；
- region + name 组合。

例如：

```hcl
locals {
  items_by_env_name = {
    for item in local.rows :
    "${item.env}-${item.name}" => item
  }
}
```

## 7. 和 LocalStack 的关系

本项目把真实 AWS 操作改造成 LocalStack 或本地 provider 可验证的练习。

你需要记住边界：

- LocalStack endpoint 通常是 `http://localhost:4566`；
- 凭证固定使用 `test/test`；
- 不访问真实 AWS；
- 不写真实 access key、secret key、root 账号信息；
- LocalStack 可以训练 Terraform 写法、状态变化和验证闭环；
- 但它不能完全替代真实 AWS 的 IAM 权限评估、网络行为和云服务细节。

对于本节来说，CSV 转换本身大多属于 Terraform 表达式能力，本地就能很好练；如果 CSV 转换结果被用于 AWS 资源，那么 LocalStack 用来验证资源创建和销毁闭环。

## 8. 推荐解题步骤

遇到这类题，按下面流程做，不要一上来乱改 resource。

### 第一步：先看输入文件

确认 CSV 的表头是什么：

```csv
name,env,team
...
```

你后面所有字段访问都依赖表头。

---

### 第二步：先做最小 locals

```hcl
locals {
  rows = csvdecode(file("${path.module}/data/items.csv"))
}
```

---

### 第三步：先 output 原始解析结果

```hcl
output "debug_rows" {
  value = local.rows
}
```

先确认数据读对了，再做资源。

---

### 第四步：按题目要求转换结构

常见转换：

```hcl
locals {
  items_by_name = {
    for item in local.rows :
    item.name => item
  }

  names = [
    for item in local.rows : item.name
  ]
}
```

---

### 第五步：再绑定 resource 或输出结果

```hcl
resource "local_file" "item" {
  for_each = local.items_by_name

  filename = "${path.module}/generated/${each.key}.txt"
  content  = jsonencode(each.value)
}
```

---

### 第六步：验证和清理

综合挑战里不要忘记完整闭环：

```bash
terraform fmt
terraform init
terraform validate
terraform plan
terraform apply
terraform output
terraform destroy
```

## 9. 考试判断点总结

### 9.1 `csvdecode(file(...))`

看到 CSV 文件输入，优先想到：

```hcl
csvdecode(file("xxx.csv"))
```

---

### 9.2 `for` 表达式

看到“从 CSV 生成列表、map、筛选结果、汇总结果”，优先想到：

```hcl
[for x in xs : x.name]
```

或：

```hcl
{
  for x in xs :
  x.name => x
}
```

---

### 9.3 `for_each`

看到“根据 CSV 创建多个资源”，优先考虑：

```hcl
for_each = local.items_by_name
```

不要直接把不稳定的 list 塞进去。

---

### 9.4 output

看到“多种输出形态”，不是多个乱写的 output，而是从同一份数据整理出：

- 原始 rows；
- name 列表；
- by_name map；
- 筛选后的列表；
- 汇总对象；
- JSON 字符串；
- 资源 ID 列表。

---

## 10. 典型题目与参考答案

### 题目 1：`csvdecode()` 常用于什么？

A. 把 CSV 字符串转换成 Terraform 可处理的数据结构  
B. 删除 Terraform state  
C. 初始化 provider plugin cache  
D. 强制执行远端 run

答案：A。

解析：`csvdecode()` 是数据解码函数，核心作用是把 CSV 文本转为对象列表。

---

### 题目 2：CSV 解码后要用于 `for_each`，通常需要做什么？

A. 把 list 转成 key 稳定且唯一的 map  
B. 删除 `.terraform.lock.hcl`  
C. 把 provider 改成 random  
D. 禁用 output

答案：A。

解析：`for_each` 最适合 map 或 set(string)。CSV 解码结果通常是 list(object)，所以常见做法是用 `for` 表达式转 map。

---

### 题目 3：CSV 中的 `count` 字段读出来后一定是数字吗？

A. 一定是  
B. 通常不是，CSV 字段更偏字符串，需要时应显式转换  
C. 只有 AWS Provider 会自动转换  
D. 只有 LocalStack 会自动转换

答案：B。

解析：CSV 是文本格式，字段值不要默认当成数字或布尔值。需要数字时使用 `tonumber()`。

## 11. 场景题讲解

### 场景 1：题目要求读取 CSV 后输出所有 bucket 名称

思路：

```hcl
locals {
  rows = csvdecode(file("${path.module}/buckets.csv"))
}

output "bucket_names" {
  value = [
    for row in local.rows : row.bucket_name
  ]
}
```

关键点是确认 CSV 表头是不是 `bucket_name`。

---

### 场景 2：题目要求按 CSV 创建多个对象

思路：

```hcl
locals {
  rows = csvdecode(file("${path.module}/objects.csv"))

  objects_by_key = {
    for row in local.rows :
    row.key => row
  }
}
```

然后：

```hcl
resource "aws_s3_object" "this" {
  for_each = local.objects_by_key

  bucket  = each.value.bucket
  key     = each.value.key
  content = each.value.content
}
```

关键点：

- `row.key` 必须唯一；
- `each.key` 是 map 的 key；
- `each.value` 是对应 CSV 行对象。

---

### 场景 3：题目要求输出 JSON 字符串

思路：

```hcl
output "items_json" {
  value = jsonencode(local.rows)
}
```

注意：

- `jsonencode()` 输出 JSON 字符串；
- 不是所有场景都需要 JSON；
- 如果 output 是给人看，直接输出对象结构可能更清楚；
- 如果 output 是给脚本或外部系统消费，JSON 字符串可能更合适。

---

## 12. 一句话记忆

> CSV 题的主线是：`file()` 读文本，`csvdecode()` 变对象列表，`for` 表达式重组结构，`for_each` 批量创建资源，`output` 展示不同视角。
