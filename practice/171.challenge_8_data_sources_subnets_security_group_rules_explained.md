# Terraform 考试训练 171：挑战八解法——数据源读取子网并生成安全组规则（讲解版）

## 1. 本节定位

本节属于 Terraform 综合挑战系列，主题是：

> **使用 data source 读取已有子网信息，然后根据这些子网动态生成安全组规则。**

它不是单纯考一个 AWS 安全组参数，而是在考你能不能把下面几件事串起来：

1. 用 **data source** 读取已有基础设施对象。
2. 把读取到的数据转换成 Terraform 可循环处理的数据结构。
3. 用 `for` 表达式、`for_each` 或动态规则生成多条安全组规则。
4. 输出结果，验证规则是否符合题目要求。
5. 完成 `plan → apply → verify → destroy` 的闭环。

这类题很像真实工作中的场景：  
网络、VPC、Subnet 可能已经由平台团队提前创建好，应用团队不能随便重建，只能读取已有信息，然后基于这些信息创建自己的安全边界。

---

## 2. 核心知识点

### 2.1 Data source 是“读已有资源”，不是“创建资源”

Terraform 里有两类常见块：

```hcl
resource "aws_subnet" "example" {
  # 创建一个 subnet
}
```

和：

```hcl
data "aws_subnet" "example" {
  # 查询一个已经存在的 subnet
}
```

区别非常关键：

| 类型 | 作用 | 是否创建资源 | 是否进入 state |
|---|---|---:|---:|
| `resource` | 声明并管理资源 | 是 | 是 |
| `data` | 查询已有资源 | 否 | 会记录读取结果 |

考试题里如果出现这些关键词，通常应该想到 data source：

- existing subnet
- already created VPC
- read from remote cloud
- lookup
- query
- use existing infrastructure
- do not recreate

这类题不要上来就写 `resource "aws_subnet"`。  
题目要你读已有子网，你却创建子网，就是方向错了。

---

### 2.2 子网信息通常会被转换成 CIDR 列表

安全组规则一般关心的是：

```hcl
cidr_blocks = ["10.0.1.0/24"]
```

而 data source 读到的 subnet 对象里，常用字段可能包括：

```hcl
data.aws_subnet.app.cidr_block
data.aws_subnet.app.id
data.aws_subnet.app.availability_zone
```

如果读取多个 subnet，最终常见目标是得到类似：

```hcl
local.subnet_cidrs = [
  "10.0.1.0/24",
  "10.0.2.0/24",
  "10.0.3.0/24",
]
```

然后安全组规则可以基于这些 CIDR 生成。

---

### 2.3 多个子网通常配合 `for_each` 或 `for` 表达式

如果题目给了多个 subnet id，例如：

```hcl
variable "subnet_ids" {
  type = set(string)
}
```

可以用 `for_each` 读取多个 subnet：

```hcl
data "aws_subnet" "selected" {
  for_each = var.subnet_ids
  id       = each.value
}
```

这样会得到一个 map 结构：

```hcl
data.aws_subnet.selected
```

里面每个元素都是一个 subnet 查询结果。

之后可以用 `for` 表达式提取 CIDR：

```hcl
locals {
  subnet_cidrs = [
    for subnet in data.aws_subnet.selected : subnet.cidr_block
  ]
}
```

这一步是考试高频点。  
它考的是你能不能把 provider 返回的对象集合，整理成后续资源需要的输入。

---

## 3. 安全组规则生成方式

### 3.1 简单方式：一条规则里放多个 CIDR

如果题目只要求允许多个子网访问同一个端口，可以写成：

```hcl
resource "aws_security_group_rule" "allow_from_subnets" {
  type              = "ingress"
  security_group_id = aws_security_group.app.id

  from_port   = 443
  to_port     = 443
  protocol    = "tcp"
  cidr_blocks = local.subnet_cidrs
}
```

这种方式创建的是一条规则，但 `cidr_blocks` 里包含多个 CIDR。

优点是代码短。  
缺点是每个 CIDR 不是单独一条 Terraform resource，后续 diff 粒度没有那么细。

---

### 3.2 更考试化方式：每个 CIDR 单独生成一条规则

如果题目要求“根据每个 subnet 生成对应规则”，更稳的是：

```hcl
resource "aws_security_group_rule" "allow_from_subnet" {
  for_each = toset(local.subnet_cidrs)

  type              = "ingress"
  security_group_id = aws_security_group.app.id

  from_port   = 443
  to_port     = 443
  protocol    = "tcp"
  cidr_blocks = [each.value]
}
```

这里的重点是：

```hcl
for_each = toset(local.subnet_cidrs)
```

然后：

```hcl
cidr_blocks = [each.value]
```

注意 `cidr_blocks` 需要的是 list，所以即使 `each.value` 是单个 CIDR，也要包成：

```hcl
[each.value]
```

这是很容易写错的小坑。

---

### 3.3 `for_each` 的 key 要稳定

真实项目里，`for_each` 的 key 很重要。

如果直接用 list 的 index，很容易因为顺序变化导致资源重建。  
更稳的方式是用 subnet id 或 CIDR 作为 key：

```hcl
locals {
  subnet_cidr_map = {
    for id, subnet in data.aws_subnet.selected :
    id => subnet.cidr_block
  }
}
```

然后：

```hcl
resource "aws_security_group_rule" "allow_from_subnet" {
  for_each = local.subnet_cidr_map

  type              = "ingress"
  security_group_id = aws_security_group.app.id

  from_port   = 443
  to_port     = 443
  protocol    = "tcp"
  cidr_blocks = [each.value]
}
```

这种写法更接近工程实践，因为 `each.key` 是 subnet id，语义更清楚。

## 4. 推荐解题流程

### 第一步：先读题，确认是“查询已有资源”还是“创建资源”

如果题目说已有 subnet，就不要创建 subnet。  
你应该先找 data source：

```hcl
data "aws_subnet" "selected" {
  id = var.subnet_id
}
```

或多个：

```hcl
data "aws_subnet" "selected" {
  for_each = var.subnet_ids
  id       = each.value
}
```

---

### 第二步：把 data source 输出整理成 locals

不要把复杂表达式直接塞进资源里。  
考试时越紧张，越应该把中间结果拆出来：

```hcl
locals {
  subnet_cidrs = [
    for subnet in data.aws_subnet.selected : subnet.cidr_block
  ]
}
```

这样后面排错更容易。

---

### 第三步：用安全组规则消费 locals

例如：

```hcl
resource "aws_security_group_rule" "allow_https_from_subnets" {
  for_each = toset(local.subnet_cidrs)

  type              = "ingress"
  security_group_id = aws_security_group.app.id

  from_port   = 443
  to_port     = 443
  protocol    = "tcp"
  cidr_blocks = [each.value]
}
```

---

### 第四步：输出关键结果

挑战题一定要输出关键结果，方便验证脚本或人工检查：

```hcl
output "subnet_cidrs" {
  value = local.subnet_cidrs
}

output "security_group_id" {
  value = aws_security_group.app.id
}
```

输出不是装饰品。  
在综合挑战里，output 经常就是验收点。

---

### 第五步：按闭环命令验证

建议固定节奏：

```bash
terraform fmt -recursive
terraform init
terraform validate
terraform plan
terraform apply
terraform output
terraform destroy
```

如果是本项目练习，还要跑对应验证脚本。

## 5. LocalStack 练习边界

本节可以在 LocalStack 中完成核心 Terraform 路径闭环。  
也就是说，你可以练：

- AWS provider 配置；
- data source 写法；
- 安全组规则资源写法；
- `for_each` / `for` 表达式；
- state 变化；
- output 验证；
- destroy 清理。

但 LocalStack 不能完全替代真实 AWS 的所有行为。

特别是安全组这种网络控制类资源，在真实 AWS 中还涉及：

- ENI 绑定；
- VPC 实际流量；
- 安全组入站/出站生效逻辑；
- NACL 与路由表共同影响；
- 跨安全组引用；
- 真实服务连通性测试。

LocalStack 更适合验证 Terraform 配置闭环，不适合证明真实云上网络策略一定生效。

一句话：

> LocalStack 练 Terraform 写法和状态闭环；真实 AWS 才能验证完整网络语义。

## 6. 常见错误

### 错误 1：把已有 subnet 写成 resource

错误思路：

```hcl
resource "aws_subnet" "selected" {
  # ...
}
```

如果题目要求读取已有 subnet，这就是错的。  
应该使用：

```hcl
data "aws_subnet" "selected" {
  id = var.subnet_id
}
```

---

### 错误 2：忘记 `cidr_blocks` 是 list

错误写法：

```hcl
cidr_blocks = each.value
```

正确写法：

```hcl
cidr_blocks = [each.value]
```

---

### 错误 3：把对象集合直接塞进 `cidr_blocks`

错误写法：

```hcl
cidr_blocks = data.aws_subnet.selected
```

这不行。  
`data.aws_subnet.selected` 是对象集合，不是 CIDR 字符串列表。

正确做法是先提取字段：

```hcl
locals {
  subnet_cidrs = [
    for subnet in data.aws_subnet.selected : subnet.cidr_block
  ]
}
```

---

### 错误 4：`for_each` 输入类型不合适

`for_each` 需要 map 或 set of strings。  
如果你手里是 list，可以转成 set：

```hcl
for_each = toset(local.subnet_cidrs)
```

如果你想保留更清晰的 key，用 map：

```hcl
for_each = local.subnet_cidr_map
```

---

### 错误 5：只 apply，不 destroy

综合挑战训练必须养成销毁习惯：

```bash
terraform destroy
```

这不只是节省资源，也是在训练你理解 Terraform 对完整生命周期的管理。

## 7. 考试记忆点

### 7.1 看到“existing”，先想 data source

题目出现：

- existing VPC
- existing subnet
- existing security group
- lookup
- query

优先考虑：

```hcl
data "..." "..." {}
```

---

### 7.2 看到“multiple”，先想循环

多个子网、多个端口、多个规则，通常离不开：

- `for_each`
- `count`
- `for` expression
- `dynamic` block

本节更偏向：

```hcl
for_each
```

和：

```hcl
for ... in ...
```

---

### 7.3 看到“generate rules”，注意输入结构

安全组规则不是凭空生成。  
它通常需要这些输入：

- 目标安全组 ID；
- 端口；
- 协议；
- CIDR blocks；
- 规则方向：ingress / egress。

## 8. 场景题强化

### 场景 1：平台团队已经创建了 3 个私有子网，应用团队需要允许这些子网访问应用安全组的 443 端口。你会怎么做？

推荐思路：

1. 把 3 个 subnet id 作为变量输入。
2. 使用 `data "aws_subnet"` 和 `for_each` 查询这些子网。
3. 用 `for` 表达式提取每个子网的 `cidr_block`。
4. 用 `aws_security_group_rule` 为这些 CIDR 生成 ingress 规则。
5. 输出 security group id 和 subnet CIDR 列表。
6. 运行 `fmt/init/validate/plan/apply/output/destroy` 完成闭环。

---

### 场景 2：为什么不直接手写 CIDR？

可以手写，但不推荐作为长期方案。

手写 CIDR 的问题：

- 子网变更后容易忘记同步；
- 容易写错；
- 与真实云资源脱节；
- 不利于复用；
- 不利于审计。

data source 的好处是 Terraform 可以从云平台读取当前真实对象信息，然后基于这些信息生成配置。

---

### 场景 3：LocalStack apply 成功，是否代表真实 AWS 网络一定通？

不代表。

LocalStack apply 成功只能说明：

- Terraform 配置语法大体正确；
- provider 调用路径可以完成；
- state 和 output 可以闭环；
- 模拟资源创建成功。

但真实 AWS 网络连通性还需要在真实环境中验证：

- EC2 / ENI 是否绑定对应安全组；
- subnet route table 是否正确；
- NACL 是否允许；
- 目标服务是否监听端口；
- 源和目标是否在预期网络路径中。

---

## 9. 一句话总结

本节真正要掌握的是：

> 用 data source 读取已有 subnet，把 subnet 的 CIDR 提取成结构化数据，再用 `for_each` 动态生成安全组规则，并完成 Terraform 生命周期闭环。

这类题非常贴近实际工程。  
平台团队创建网络，应用团队读取网络，然后生成应用侧安全策略。  
你要练的不是背 API，而是把“查询 → 转换 → 生成 → 验证 → 清理”这条链路打通。
