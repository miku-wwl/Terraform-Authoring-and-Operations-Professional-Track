# Terraform 实操训练 45 讲解：资源行为与 Meta-Argument

## 1. 这一节到底在学什么？

Lab 45 学的是 Terraform 的一个底层核心能力：

> Terraform 到底是怎么判断一个资源应该创建、更新、替换，还是销毁的？

很多初学者看到 `terraform plan` 会慌，因为 plan 里可能出现：

- create
- update in-place
- destroy
- replace
- forces replacement

如果你不理解这些行为，很容易误判风险。

这一节的重点就是建立 Terraform 资源行为的基本模型，并理解 meta-argument 是如何影响这些默认行为的。

一句话总结：

> Terraform 会根据配置、state 和远端对象推导动作；meta-argument 用来调整这些默认行为。

---

## 2. Terraform 判断资源动作的三个依据

Terraform 决定资源行为时，主要看三样东西：

1. 当前 `.tf` 配置
2. Terraform state
3. 远端真实资源状态

可以这样理解：

```text
配置：我希望世界长什么样
state：Terraform 记得世界之前长什么样
远端：真实世界现在长什么样
```

Terraform plan 的本质就是比较这三者，然后推导下一步要做什么。

---

## 3. 四类核心资源行为

Terraform 最常见的资源行为有四类：

| 行为 | 含义 | 常见原因 |
|---|---|---|
| Create | 创建资源 | 配置里有，state 里没有 |
| Update | 原地更新资源 | 属性变化，且 provider 支持原地修改 |
| Replace | 替换资源 | 属性变化，但不能原地修改 |
| Destroy | 销毁资源 | state 里有，但配置里删除了 |

这四个动作是读懂 plan 的基础。

---

## 4. Create：什么时候会创建资源？

当 Terraform 发现：

```text
配置里声明了资源，但 state 里没有这个资源
```

它就会计划创建。

例如你新增：

```hcl
resource "local_file" "example" {
  filename = "${path.module}/output/example.txt"
  content  = "hello"
}
```

第一次执行 plan 时，Terraform 会显示要创建这个资源。

典型信号：

```text
+ create
```

这说明 Terraform 认为这个资源当前不存在，需要新建。

---

## 5. Destroy：什么时候会销毁资源？

当 Terraform 发现：

```text
state 里记录了资源，但配置里已经没有这个资源
```

它就会计划销毁。

比如你之前创建了：

```hcl
resource "local_file" "example" {
  filename = "${path.module}/output/example.txt"
  content  = "hello"
}
```

后来你把这个资源块从 `.tf` 文件里删除了。

Terraform 会认为：

> 你不再希望我管理这个资源了，所以我要把它删掉。

典型信号：

```text
- destroy
```

这个动作要谨慎，尤其是数据库、存储桶、生产网络这类长期资源。

---

## 6. Update：什么时候会原地更新？

当 Terraform 发现：

```text
资源还在，但某些可变属性变了
```

而 provider 支持原地修改时，就会执行 update in-place。

例如：

```hcl
resource "local_file" "example" {
  filename = "${path.module}/output/example.txt"
  content  = "version 1"
}
```

改成：

```hcl
resource "local_file" "example" {
  filename = "${path.module}/output/example.txt"
  content  = "version 2"
}
```

如果这个属性可以原地更新，Terraform 会显示类似：

```text
~ update in-place
```

含义是：

> 不需要删除资源，只要修改属性即可。

---

## 7. Replace：什么时候会替换资源？

Replace 是最需要警惕的行为。

当某些属性变化后，provider 认为这个属性不能原地修改，就会触发替换。

替换通常等价于：

```text
先销毁旧资源，再创建新资源
```

或者在某些策略下：

```text
先创建新资源，再销毁旧资源
```

典型信号：

```text
-/+ destroy and then create replacement
```

或者：

```text
+/- create replacement and then destroy
```

plan 里经常会看到：

```text
forces replacement
```

这说明某个属性变化会强制替换资源。

---

## 8. 什么是 ForceNew？

在 provider 的资源 schema 里，有些属性被标记为类似 ForceNew 的语义。

意思是：

> 这个属性一旦变化，不能原地更新，只能重新创建资源。

常见例子包括：

- 某些资源名称
- 某些区域
- 某些不可变 ID
- 某些网络绑定关系
- 某些云服务创建后不能修改的基础属性

所以看到 replace 时，不要只说“Terraform 抽风了”。

你要追问：

> 是哪个属性 forces replacement？

这是读 plan 的关键能力。

---

## 9. Meta-Argument 是什么？

Meta-argument 是 Terraform 语言提供的一类特殊参数，用来调整资源行为或资源实例数量。

常见 meta-argument 包括：

- `count`
- `for_each`
- `depends_on`
- `provider`
- `lifecycle`

Lab 45 特别强调的是：meta-argument 可以影响 Terraform 的默认行为，尤其是 `lifecycle`。

---

## 10. `lifecycle` 是什么？

`lifecycle` 是资源里最常见、最重要的 meta-argument 之一。

它可以控制资源生命周期行为。

常见配置包括：

```hcl
lifecycle {
  prevent_destroy = true
}
```

```hcl
lifecycle {
  create_before_destroy = true
}
```

```hcl
lifecycle {
  ignore_changes = [
    tags
  ]
}
```

以及前面 Lab 39 / 40 学过的：

```hcl
lifecycle {
  precondition {
    condition     = ...
    error_message = ...
  }

  postcondition {
    condition     = ...
    error_message = ...
  }
}
```

---

## 11. `prevent_destroy`：防止误删

`prevent_destroy` 可以阻止 Terraform 销毁资源。

示例：

```hcl
resource "local_file" "important" {
  filename = "${path.module}/output/important.txt"
  content  = "do not delete"

  lifecycle {
    prevent_destroy = true
  }
}
```

如果某次 plan 要销毁这个资源，Terraform 会直接报错。

适合保护：

- 数据库
- 存储桶
- 关键密钥
- 生产网络
- 关键 DNS
- 长期存在的共享资源

但它不是万能护身符。  
如果你真的要删除资源，需要先移除或调整这个 lifecycle 设置。

---

## 12. `create_before_destroy`：先建新，再删旧

默认替换资源时，很多情况下 Terraform 可能先销毁旧资源，再创建新资源。

这对某些资源会造成短暂中断。

`create_before_destroy` 可以改变顺序：

```hcl
lifecycle {
  create_before_destroy = true
}
```

含义是：

> 替换时尽量先创建新资源，再销毁旧资源。

这对降低停机风险有帮助。

但要注意，它不是永远可用。  
如果资源名称必须唯一，旧资源不删，新资源可能创建不了。

比如两个资源不能同名，那么先创建新资源就会冲突。

---

## 13. `ignore_changes`：忽略某些属性变化

`ignore_changes` 用来告诉 Terraform：

> 某些属性即使和配置不一致，也不要因为它们产生变更。

示例：

```hcl
lifecycle {
  ignore_changes = [
    tags["LastModifiedBy"]
  ]
}
```

常见场景：

- 外部系统会自动改 tag
- 云平台会自动填默认值
- 某些字段由另一个控制面管理
- 避免 Terraform 和平台控制器互相打架

但是要小心：

> 不要用 `ignore_changes` 掩盖建模错误。

如果你把太多关键字段 ignore 掉，Terraform 就不再真正管理这些属性了。长期看会制造漂移。

---

## 14. `depends_on`：显式依赖

Terraform 默认会根据引用自动推导依赖。

例如：

```hcl
resource "local_file" "a" {
  filename = "${path.module}/output/a.txt"
  content  = "hello"
}

resource "local_file" "b" {
  filename = "${path.module}/output/b.txt"
  content  = local_file.a.content
}
```

因为 `b` 引用了 `a`，Terraform 自动知道 `b` 依赖 `a`。

但有些依赖不是通过表达式体现的，就可能需要：

```hcl
depends_on = [
  local_file.a
]
```

`depends_on` 的含义是：

> 即使配置表达式里看不出来，也要强制 Terraform 先处理这个依赖。

不要滥用。  
能通过真实引用表达依赖，就优先用真实引用。

---

## 15. `count`：创建多个同类实例

`count` 用来创建多个资源实例。

示例：

```hcl
resource "local_file" "example" {
  count = 3

  filename = "${path.module}/output/file-${count.index}.txt"
  content  = "file ${count.index}"
}
```

这会创建：

```text
local_file.example[0]
local_file.example[1]
local_file.example[2]
```

`count` 适合数量简单、顺序稳定的场景。

但如果列表中间删除一个元素，索引变化可能导致资源地址变化，引发不必要替换。

---

## 16. `for_each`：按 key 创建多个实例

`for_each` 更适合按 key 管理多个资源。

示例：

```hcl
resource "local_file" "service" {
  for_each = {
    api = "api service"
    web = "web service"
  }

  filename = "${path.module}/output/${each.key}.txt"
  content  = each.value
}
```

资源地址会是：

```text
local_file.service["api"]
local_file.service["web"]
```

相比 `count`，`for_each` 的 key 更稳定。  
生产里处理多个命名对象时，通常更推荐 `for_each`。

---

## 17. `provider` meta-argument

`provider` 可以指定某个资源使用哪个 provider 配置。

例如多 region、多账号场景：

```hcl
provider "aws" {
  alias  = "use1"
  region = "us-east-1"
}

provider "aws" {
  alias  = "apse2"
  region = "ap-southeast-2"
}

resource "aws_s3_bucket" "example" {
  provider = aws.use1

  bucket = "example-bucket"
}
```

这在多账号、多区域、多集群管理中很常见。

## 19. 标准运行流程

进入实验目录：

```sh
cd /workspace/practice/labs/45
```

初始化：

```sh
terraform init -input=false
```

格式检查：

```sh
terraform fmt -check
```

配置校验：

```sh
terraform validate
```

运行测试：

```sh
terraform test
```

生成 plan：

```sh
terraform plan -input=false -no-color -out=tfplan
```

执行 apply：

```sh
terraform apply -auto-approve tfplan
```

查看输出：

```sh
terraform output
```

销毁：

```sh
terraform destroy -auto-approve
```

清理本地文件：

```sh
rm -f tfplan
rm -rf .terraform .terraform.lock.hcl
```

## 20. 如何读懂 plan？

看到 plan 时，先不要慌，按这个顺序看：

### 第一步：看动作类型

是：

- create
- update
- replace
- destroy

先分类。

### 第二步：看资源地址

例如：

```text
local_file.example
aws_instance.app
module.network.aws_vpc.main
```

确认是哪一个资源。

### 第三步：看触发属性

重点找：

```text
forces replacement
```

或者某个属性从旧值变成新值。

### 第四步：判断风险

问自己：

- 这是生产资源吗？
- 会不会删数据？
- 会不会中断服务？
- 是不是替换而不是原地更新？
- 有没有 `prevent_destroy`？
- 有没有 `create_before_destroy`？
- 有没有被 `ignore_changes` 掩盖的问题？

这才是工程师读 plan 的方式。

## 21. 常见坑

### 坑 1：把 update 和 replace 混为一谈

Update 是原地更新。  
Replace 是重建。

这两个风险完全不同。

看到 `-/+` 或 `forces replacement` 要特别警惕。

---

### 坑 2：看到 destroy 没反应

Destroy 不是小事。

如果是数据库、存储桶、生产网络，destroy 可能就是事故。

生产中应该配合：

- 审批
- saved plan
- `prevent_destroy`
- 备份
- 变更窗口
- 权限控制

---

### 坑 3：用 `ignore_changes` 掩盖漂移

`ignore_changes` 可以解决某些外部系统自动修改字段的问题。

但如果你用它忽略关键配置，Terraform 就失去了管理能力。

不要为了让 plan “干净”而随便 ignore。

---

### 坑 4：滥用 `depends_on`

很多依赖 Terraform 可以自动推导。  
如果你到处写 `depends_on`，说明模型可能不清晰。

优先通过真实引用建立依赖。

---

### 坑 5：`count` 索引漂移

如果用 list + count 管资源，中间删除元素可能导致后续索引变化。

这会让 Terraform 以为多个资源都要变。

生产里管理一组有名字的对象时，优先考虑 `for_each`。

## 22. SRE / Platform Engineering 视角

Lab 45 很基础，但非常重要。

因为 SRE / Platform Engineer 不是只会写 Terraform 资源块，还要能评估变更风险。

你需要能回答：

- 为什么这次是 replace？
- 哪个字段触发了替换？
- 有没有可能造成停机？
- 是否应该加 `create_before_destroy`？
- 是否应该加 `prevent_destroy`？
- 是否存在建模问题？
- 是否有人用 `ignore_changes` 把漂移藏起来了？
- 这个资源适合 `count` 还是 `for_each`？

这就是从“会写 Terraform”到“能审 Terraform plan”的区别。

## 24. 面试/考试记忆版

可以这样记：

- Terraform 根据配置、state、远端对象决定动作。
- 资源行为主要有 create、update、replace、destroy。
- update 是原地修改，replace 是重建。
- `forces replacement` 表示属性变化触发替换。
- meta-argument 可以调整默认行为。
- 常见 meta-argument：`count`、`for_each`、`depends_on`、`provider`、`lifecycle`。
- `lifecycle` 常见配置：`prevent_destroy`、`create_before_destroy`、`ignore_changes`、`precondition`、`postcondition`。
- 不要用 lifecycle 掩盖建模问题。

一句话总结：

> 读 Terraform plan 的第一步，是先判断资源动作；治理 Terraform 风险的第一步，是理解 meta-argument 如何改变这些动作。

## 25. 本 Lab 的价值

Lab 45 的价值在于，它训练的是 Terraform 的“行为判断能力”。

这比记几个命令更重要。

真正做生产 Terraform 时，最怕的不是 `terraform apply` 不会敲，而是看不懂 plan：

- 明明是 replace，却以为只是 update
- 明明会 destroy，却没意识到风险
- 明明是 ForceNew 属性，却不知道为什么重建
- 明明是建模问题，却用 `ignore_changes` 糊住
- 明明应该用 `for_each`，却用 `count` 制造地址漂移

所以这一节要好好掌握。  
它是 Terraform 工程化能力的地基。
