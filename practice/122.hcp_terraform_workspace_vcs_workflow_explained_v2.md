# Terraform 考试训练 122 讲解版：Workspace 与 Version Control Workflow

## 1. 这一节到底在讲什么

这一节讲的是 **HCP Terraform / Terraform Cloud 里的 workspace 如何和 Git 仓库配合工作**。

你可以把 workspace 理解成 HCP Terraform 里的一个“基础设施运行单元”。它不是简单的文件夹，也不是本地 CLI 里的 `terraform workspace` 那个概念。它更像是一个集中管理 Terraform 执行、变量、状态文件、审批、运行历史和版本控制触发规则的平台对象。

本节的核心关键词是：

- HCP Terraform workspace
- Version Control Workflow
- VCS 集成
- working directory
- run trigger
- plan / policy check / cost estimation / apply
- state 与代码分离管理

考试里，这类题通常不是考你背 UI 按钮，而是考你能不能判断：**代码在哪里、state 在哪里、run 谁触发、workspace 边界怎么切。**

---

## 2. 先把 workspace 这个词讲明白

在 HCP Terraform 里，workspace 通常管理一套 Terraform 配置的运行上下文。

一个 workspace 可能包含：

- Terraform state
- Terraform 版本
- workspace variables
- environment variables
- sensitive variables
- run 历史
- plan / apply 记录
- VCS 仓库连接
- working directory 设置
- 执行模式
- 触发规则
- policy check
- cost estimation
- apply 审批流程

所以 workspace 不只是“放代码的地方”。

代码一般还在 GitHub、GitLab、Bitbucket 等 VCS 仓库里。HCP Terraform workspace 是把代码拉过来执行，并且集中保存 state、变量、运行结果和审批记录。

一句话总结：

> Git 管代码，workspace 管运行、变量、state、审批和历史。

---

## 3. Version Control Workflow 是什么

Version Control Workflow 指的是：

1. Terraform 配置代码存放在 Git 仓库中。
2. HCP Terraform workspace 连接这个 Git 仓库。
3. 开发者提交代码或创建 Pull Request。
4. HCP Terraform 自动触发 plan。
5. 根据配置执行 policy check、cost estimation。
6. 如果 plan 通过，进入 apply 阶段。
7. apply 可以自动执行，也可以等待人工审批。
8. 运行结果和 state 由 HCP Terraform 统一保存。

这个流程的价值是：**Terraform 执行从个人电脑迁移到集中平台**。

本地 CLI 模式下，常见风险是：

- 谁本地跑了 apply 不清楚。
- 变量散落在个人电脑。
- state 后端配置不统一。
- 审批流程靠口头沟通。
- plan 结果难以审计。
- 生产变更缺少历史记录。

Version Control Workflow 解决的是工程化协作问题，不只是“自动跑 Terraform”。

---

## 4. workspace 和 Git 仓库目录的关系

考试里很容易考这个点。

一个 Git 仓库可以有多个 Terraform 子目录，例如：

```text
infra-repo/
├── network/
│   ├── main.tf
│   └── variables.tf
├── app/
│   ├── main.tf
│   └── variables.tf
└── security/
    ├── main.tf
    └── variables.tf
```

如果团队希望 `network/`、`app/`、`security/` 分别有独立 state、变量、审批和运行历史，那么通常会创建多个 workspace：

| Workspace | Git 仓库 | Working Directory | State |
|---|---|---|---|
| network-prod | infra-repo | `network/` | 独立 |
| app-prod | infra-repo | `app/` | 独立 |
| security-prod | infra-repo | `security/` | 独立 |

这里的重点是：**workspace 可以指向同一个仓库里的不同 working directory。**

所以看到题目说：

> 一个 repo 里有多个 Terraform 配置目录，想分别管理 state 和审批。

答案基本就是：

> 创建多个 workspace，并为每个 workspace 配置不同的 working directory。

---

## 5. workspace 不是本地 terraform workspace

这个坑必须单独拎出来。

Terraform 里有两个容易混淆的概念：

### 5.1 本地 CLI 的 `terraform workspace`

这是 Terraform CLI 的功能，常用于在同一套配置下切换多个 state，例如：

```sh
terraform workspace new dev
terraform workspace select prod
```

它更多是本地/后端 state 命名空间的概念。

### 5.2 HCP Terraform workspace

这是 HCP Terraform 平台里的执行单元，包含：

- 配置来源
- state
- variables
- run history
- VCS trigger
- 审批
- policy
- cost estimation

考试里如果出现 HCP Terraform、Terraform Cloud、VCS workflow，通常说的是平台 workspace，而不是本地 CLI workspace。

记忆方式：

> 本地 `terraform workspace` 偏 state 命名空间。  
> HCP Terraform workspace 偏平台运行单元。

---

## 6. 生产环境里为什么要用多个 workspace

生产环境不会把所有东西塞进一个 workspace。

常见拆分方式包括：

### 6.1 按环境拆

```text
app-dev
app-staging
app-prod
```

优点：

- dev 和 prod state 隔离
- 变量隔离
- apply 权限可以不同
- prod 可以要求人工审批

### 6.2 按系统拆

```text
network-prod
database-prod
application-prod
```

优点：

- blast radius 小
- state 不会过大
- 团队职责更清晰
- 一个系统出问题不影响全部基础设施

### 6.3 按团队拆

```text
platform-shared
security-shared
data-platform-prod
```

优点：

- 每个团队有自己的权限边界
- 变量和密钥分开管理
- 审计和 run history 更清晰

## 7. VCS 触发 run 的实际流程

一个典型流程是：

```text
Developer push code
        ↓
GitHub / GitLab webhook
        ↓
HCP Terraform workspace detects change
        ↓
Run is queued
        ↓
Terraform plan
        ↓
Policy check / cost estimation
        ↓
Manual approval or auto apply
        ↓
Terraform apply
        ↓
State updated in HCP Terraform
```

这里要注意两点：

第一，触发来自 Git 变更，但真正执行发生在 HCP Terraform。

第二，apply 是否自动执行取决于 workspace 的配置和团队治理要求。生产环境常见做法是自动 plan，人工 approve apply。

---

## 8. policy check 和 cost estimation 的意义

很多题会把 workspace、policy、cost estimation 放在一起考。

### 8.1 Policy Check

Policy Check 通常用于阻止不符合规范的变更，例如：

- 禁止创建公网暴露的安全组
- 禁止使用不合规 region
- 要求资源必须有 tag
- 限制高风险实例规格
- 要求生产环境开启加密

它的作用不是创建资源，而是在 apply 之前做治理检查。

### 8.2 Cost Estimation

Cost Estimation 用于在 apply 前估算成本变化，例如：

- 新增 VM 会增加多少月成本
- 扩容数据库会增加多少费用
- 删除资源会减少多少费用

考试里看到 cost estimation，要想到：

> 它发生在 run 流程中，用于帮助团队在 apply 前理解成本影响。

## 9. 常见考试题型拆解

### 题型 1：workspace 从哪里拿 Terraform 代码？

如果是 Version Control Workflow，答案是：

> 已连接的 Git / VCS 仓库。

不是本地缓存，不是 Docker Hub，也不是 provider registry。

Provider registry 是下载 provider 的地方，不是 Terraform 配置代码来源。

---

### 题型 2：workspace 管理什么？

正确理解：

> workspace 管理变量、环境变量、state、run history、Terraform 版本、代码来源、触发规则和执行设置。

错误说法：

> workspace 只保存 Terraform 代码。

这句话是错的。代码通常保存在 VCS 仓库里，workspace 连接代码并执行。

---

### 题型 3：一个 repo 多个 Terraform 子目录怎么办？

答案：

> 配置 working directory，必要时创建多个 workspace。

例如：

```text
repo/network → network-prod workspace
repo/app     → app-prod workspace
```

这样可以分别管理 state 和审批。

---

### 题型 4：提交代码后自动 plan，但 apply 要人工确认，体现了什么能力？

体现了：

- VCS 集成
- 自动触发 run
- 自动 plan
- run history
- 集中执行
- 人工审批控制
- state 集中管理

注意：这里不是“本地 CLI 自动执行”，而是 HCP Terraform 平台触发运行。

---

## 10. 和真实项目结合怎么理解

假设你有一个平台工程仓库：

```text
platform-infra/
├── envs/
│   ├── dev/
│   ├── staging/
│   └── prod/
├── modules/
│   ├── network/
│   ├── aks/
│   └── monitor/
```

你可能会设计这些 workspace：

| Workspace | Working Directory | 用途 |
|---|---|---|
| platform-dev | `envs/dev` | 开发环境 |
| platform-staging | `envs/staging` | 预发布环境 |
| platform-prod | `envs/prod` | 生产环境 |

`modules/` 目录一般不会直接对应 workspace，因为 module 是复用组件。真正运行的是 `envs/dev`、`envs/staging`、`envs/prod` 这些 root module 目录。

这也是考试常见隐含点：

> workspace 指向的是可运行的 Terraform root module，而不是随便一个 module 目录。

## 11. 易错点总结

### 易错点 1：把 workspace 当成代码仓库

错误理解：

> workspace 保存 Terraform 代码。

更准确：

> 代码保存在 Git 仓库，workspace 连接 Git 仓库并执行配置，同时管理 state、变量和运行历史。

### 易错点 2：忽略 working directory

如果 repo 里有多个 Terraform 配置目录，不配置 working directory，workspace 可能会在错误目录执行，导致找不到配置或跑错环境。

### 易错点 3：把 provider registry 当代码来源

Provider registry 是 provider 的来源，不是 `.tf` 配置文件的来源。

### 易错点 4：认为 plan 后一定自动 apply

不一定。

生产环境通常会自动 plan，但 apply 可能需要审批。

### 易错点 5：把 HCP workspace 和 CLI workspace 混为一谈

考试看到 HCP Terraform、Terraform Cloud、VCS workflow，优先按平台 workspace 理解。

## 13. 本节参考答案复盘

### 选择题

1. Version Control Workflow 中，workspace 通常从哪里获取 Terraform 代码？  
   答案：A. 已连接的 Git 仓库

2. Workspace 可以管理哪些内容？  
   答案：A. 变量、state、run 历史、Terraform 版本和代码来源

3. 如果一个仓库中有多个 Terraform 子目录，workspace 通常需要配置什么？  
   答案：A. working directory

### 判断题

1. Workspace 可以与 VCS 仓库关联，并基于代码变更触发运行。  
   答案：对

2. Workspace 只保存 Terraform 代码，不保存任何 run 或 state 信息。  
   答案：错

3. 生产中常用多个 workspace 区分环境、系统或团队边界。  
   答案：对

## 14. 面试级表达

如果面试官问你：

> HCP Terraform workspace 在 VCS workflow 中有什么作用？

可以这样答：

> In HCP Terraform, a workspace is the execution and state boundary for Terraform runs. In a VCS-driven workflow, the workspace connects to a Git repository, optionally selects a working directory, and automatically triggers runs when code changes. It manages variables, sensitive values, Terraform version, state, run history, policy checks, cost estimation, and apply approvals. In production, we usually split workspaces by environment, component, or team boundary to keep state and change control isolated.

中文理解：

> HCP Terraform workspace 是运行边界和 state 边界。VCS workflow 中，它连接 Git 仓库，根据代码变更触发 plan，并集中管理变量、敏感值、state、运行历史、策略检查、成本估算和 apply 审批。生产中通常按环境、系统或团队拆多个 workspace，避免所有资源混在一个 state 里。

---

## 15. 最后复习重点

本节你要抓住四句话：

1. Workspace 是 HCP Terraform 的运行单元，不只是代码目录。
2. Version Control Workflow 的代码来源是 Git / VCS 仓库。
3. 一个 Git 仓库多个 Terraform 子目录时，用 working directory 指定执行目录。
4. 多个 workspace 可以隔离 state、变量、审批和运行历史。
