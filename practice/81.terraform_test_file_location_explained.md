# Terraform 实操训练 81 讲解：Terraform test 文件位置

## 1. 这一节到底在学什么？

Lab 81 学的是 Terraform test 的**文件位置规则**。

上一节 Lab 80 已经讲了 Terraform 原生测试框架：

```sh
terraform test
```

这一节继续讲一个更基础但很容易踩坑的问题：

> Terraform test 文件到底应该放在哪里？

一句话理解：

> Terraform 测试文件通常放在当前 root module 下的 `tests/` 目录中，并使用 `.tftest.hcl` 后缀，这样 `terraform test` 才能自动发现并运行。

这节看起来小，但非常实用。  
因为很多测试失败不是断言写错，而是 Terraform 根本没找到你的测试文件。

---

## 2. Terraform test 文件的标准位置

常见推荐结构：

```text
practice/labs/81/
├── main.tf
├── variables.tf
├── outputs.tf
├── tests/
│   └── basic.tftest.hcl
└── README.md
```

重点是：

```text
tests/
```

目录在 root module 下面。

如果你在：

```text
practice/labs/81
```

运行：

```sh
terraform test
```

Terraform 会扫描当前配置对应的测试文件，并运行其中的 `run` block。

---

## 3. 测试文件后缀是什么？

Terraform test 文件通常使用：

```text
.tftest.hcl
```

例如：

```text
basic.tftest.hcl
variables.tftest.hcl
outputs.tftest.hcl
template.tftest.hcl
```

不要写成：

```text
basic.test.hcl
basic.tf
basic.hcl
test_basic.hcl
```

这些文件可能不会被 Terraform 当作 test 文件处理。

简单记忆：

```text
文件名可以自定义
后缀要用 .tftest.hcl
```

---

## 4. 为什么文件位置很重要？

Terraform test 不是“随便哪个目录放一个 HCL 文件都能跑”。

如果测试文件位置错了，可能出现：

- `terraform test` 没有运行你的测试
- 你以为测试通过，其实没有测试被执行
- CI 显示通过，但关键断言根本没跑
- 后续 apply 才发现问题

这比测试失败更危险。

测试失败至少告诉你有问题。  
测试没被发现，才是悄悄埋雷。

---

## 5. 本节的核心目录结构

Lab 81 的实验文件包括：

```text
practice/labs/81/main.tf
practice/labs/81/tests/
practice/labs/81/README.md
```

这说明本节重点是：

```text
main.tf 是被测 Terraform 配置
tests/ 是测试目录
README.md 是实验说明
```

也就是说，你要在：

```sh
cd /workspace/practice/labs/81
```

之后运行：

```sh
terraform test
```

而不是跑到别的目录。

## 6. `terraform test` 从哪个目录运行？

你应该在 root module 目录运行：

```sh
cd /workspace/practice/labs/81
terraform test
```

如果你跑到父目录：

```sh
cd /workspace/practice/labs
terraform test
```

Terraform 看到的是另一个目录，不一定能找到 Lab 81 的配置和测试。

如果你跑到 tests 目录：

```sh
cd /workspace/practice/labs/81/tests
terraform test
```

也不对。  
因为 tests 目录通常不是 root module 本身。

正确心法：

> 在被测 Terraform 配置所在目录运行 `terraform test`。

---

## 7. 什么是 root module？

root module 就是你当前执行 Terraform 命令的目录。

例如：

```sh
cd /workspace/practice/labs/81
terraform plan
```

那么：

```text
/workspace/practice/labs/81
```

就是当前 root module。

它里面的：

```text
main.tf
variables.tf
outputs.tf
```

就是当前被测试对象。

`tests/` 目录应该放在这个 root module 下面。

---

## 8. 测试文件里的基本结构

一个最简单的测试文件可能是：

```hcl
run "basic" {
  command = plan

  assert {
    condition     = output.example == "hello"
    error_message = "Expected example output to be hello."
  }
}
```

文件位置：

```text
tests/basic.tftest.hcl
```

完整结构：

```text
practice/labs/81/
├── main.tf
└── tests/
    └── basic.tftest.hcl
```

运行：

```sh
terraform test
```

Terraform 会执行 `basic` 这个 run block。

---

## 9. 测试文件可以有多个吗？

可以。

例如：

```text
tests/
├── basic.tftest.hcl
├── variables.tftest.hcl
├── outputs.tftest.hcl
└── invalid-input.tftest.hcl
```

这样可以按主题拆分测试。

例如：

- `basic.tftest.hcl`：基础 happy path
- `variables.tftest.hcl`：变量输入测试
- `outputs.tftest.hcl`：输出断言测试
- `invalid-input.tftest.hcl`：错误输入测试

这样比把所有测试塞进一个超大文件更清楚。

---

## 10. 测试文件里可以有多个 run 吗？

可以。

例如：

```hcl
run "dev_environment" {
  command = plan

  variables {
    environment = "dev"
  }

  assert {
    condition     = output.replica_count == 1
    error_message = "dev should use 1 replica."
  }
}

run "prod_environment" {
  command = plan

  variables {
    environment = "prod"
  }

  assert {
    condition     = output.replica_count == 3
    error_message = "prod should use 3 replicas."
  }
}
```

一个 `.tftest.hcl` 文件可以包含多个测试场景。

建议：

```text
小模块：一个文件多个 run 就够
大模块：按主题拆多个 .tftest.hcl 文件
```

---

## 11. 常见错误：测试文件放错目录

错误结构：

```text
practice/labs/81/
├── main.tf
└── test/
    └── basic.tftest.hcl
```

注意这里是：

```text
test/
```

不是：

```text
tests/
```

如果 Terraform 没自动发现，测试就不会按预期运行。

推荐统一用：

```text
tests/
```

目录名不要随便改。

---

## 12. 常见错误：测试文件后缀写错

错误：

```text
tests/basic.test.hcl
```

正确：

```text
tests/basic.tftest.hcl
```

错误：

```text
tests/basic.tf
```

正确：

```text
tests/basic.tftest.hcl
```

这个坑很低级，但很常见。  
CI 里如果没有认真看 test 输出，你可能以为测试跑了，其实没有。

---

## 13. 常见错误：在错误目录运行命令

错误：

```sh
cd /workspace/practice/labs

```

正确：

```sh
cd /workspace/practice/labs/81

```

原因是：

```text

```

你在哪个目录运行，它就测试哪个目录的配置。

所以执行命令前一定先确认：

```sh

```

应该能看到：

```text

```

## 15. 标准运行流程

进入实验目录：

```sh
cd /workspace/practice/labs/81
```

初始化：

```sh
terraform init -input=false
```

格式检查：

```sh
terraform fmt -check
```

配置校验：

```sh
terraform validate
```

运行测试：

```sh
terraform test
```

生成 plan：

```sh
terraform plan -input=false -no-color -out=tfplan
```

执行 apply：

```sh
terraform apply -auto-approve tfplan
```

查看输出：

```sh
terraform output
```

销毁：

```sh
terraform destroy -auto-approve
```

清理：

```sh
rm -f tfplan
rm -rf output logs .terraform .terraform.lock.hcl terraform.tfstate terraform.tfstate.backup
```

## 16. 为什么 `terraform fmt -check` 也重要？

测试文件也是 HCL。

所以格式检查不只检查 `main.tf`，也应该覆盖测试文件。

常见做法：

```sh
terraform fmt -check
```

如果要递归检查：

```sh
terraform fmt -recursive -check
```

如果格式不通过，可以运行：

```sh
terraform fmt -recursive
```

课程验证报告里也提到：

```text
部分生成文件需要 terraform fmt 标准化
```

这说明测试文件和主配置一样，都要保持格式规范。

---

## 17. `terraform validate` 会验证测试文件吗？

`terraform validate` 主要验证 Terraform 配置本身。  
测试文件的逻辑通常通过：

```sh
terraform test
```

来验证。

所以不要以为：

```sh
terraform validate
```

通过了，就代表测试文件也没问题。

正确流程是：

```text
fmt
validate
test
```

三个都跑。

---

## 18. `terraform test` 输出怎么看？

成功时可能类似：

```text
tests/basic.tftest.hcl... pass
Success! 1 passed, 0 failed.
```

Lab 81 的验证报告里写：

```text
terraform test 返回 1 passed, 0 failed
```

这说明至少有一个测试用例被 Terraform 发现并通过。

你要重点看：

```text
1 passed
0 failed
```

如果是：

```text
0 passed
0 failed
```

就要警惕：是不是根本没有测试被发现？

---

## 19. 怎么确认测试真的被执行了？

可以看 `terraform test` 输出。

你应该看到测试文件名或 run 名称。

例如：

```text
tests/basic.tftest.hcl... pass
```

或者类似 run 的执行信息。

如果输出很奇怪，或者没有任何测试数量，就要检查：

1. 当前目录是不是 root module。
2. 是否存在 `tests/` 目录。
3. 文件是否以 `.tftest.hcl` 结尾。
4. 测试文件里是否有 `run` block。
5. Terraform 版本是否支持 test 框架。

不要盲信 CI 的绿色勾。

---

## 20. 测试文件位置和 CI/CD

CI/CD 里最容易犯的错误是 working directory 配错。

例如 GitHub Actions：

```yaml
- name: Terraform test
  run: terraform test
```

如果 job 默认目录是仓库根目录，但 Terraform 配置在：

```text
practice/labs/81
```

那就要设置：

```yaml
working-directory: practice/labs/81
```

否则 `terraform test` 会在错误目录执行。

这就是为什么 Lab 81 很重要。  
真实 CI 里，目录位置错误非常常见。

---

## 21. 多模块仓库怎么组织 tests？

如果一个仓库有多个 Terraform root module，推荐每个 root module 自己带 tests：

```text
envs/dev/
├── main.tf
└── tests/
    └── basic.tftest.hcl

envs/prod/
├── main.tf
└── tests/
    └── basic.tftest.hcl

modules/network/
├── main.tf
└── tests/
    └── basic.tftest.hcl
```

CI 应该分别进入对应目录运行：

```sh
cd modules/network
terraform test
```

或者矩阵运行多个目录。

不要把所有测试乱堆在仓库根目录，后面很难维护。

---

## 22. root module 测试和 reusable module 测试

Terraform 项目里有两类常见目录：

### root module

例如：

```text
envs/dev/
```

里面是真正部署环境的配置。

测试重点：

- backend 配置
- provider 参数
- environment-specific 变量
- remote state 读取
- 资源组合是否正确

---

### reusable module

例如：

```text
modules/network/
```

这是可复用模块。

测试重点：

- 输入变量
- output
- 条件分支
- default 值
- 错误输入
- 不同场景的模块行为

无论是哪种，测试文件都应该跟着被测对象放。

---

## 23. 测试文件不要和生产配置混在一起

不推荐这样：

```text
practice/labs/81/
├── main.tf
├── test_basic.tftest.hcl
├── variables.tftest.hcl
└── outputs.tf
```

虽然某些情况下 Terraform 可能能处理，但工程上更清晰的是：

```text
practice/labs/81/
├── main.tf
├── outputs.tf
└── tests/
    ├── basic.tftest.hcl
    └── variables.tftest.hcl
```

把测试文件放到 `tests/` 目录，维护更清楚。

## 24. 测试文件命名建议

好的命名：

```text
basic.tftest.hcl
variables.tftest.hcl
outputs.tftest.hcl
validation.tftest.hcl
template.tftest.hcl
remote-state.tftest.hcl
```

不好的命名：

```text
test1.tftest.hcl
aaa.tftest.hcl
new.tftest.hcl
tmp.tftest.hcl
```

命名要表达测试目标。

以后你回头看，不需要打开文件就大概知道它测什么。

## 25. 测试位置和相对路径

如果测试涉及文件路径，比如：

```hcl
file("${path.module}/mock/services.json")
```

或者：

```hcl
templatefile("${path.module}/templates/config.tftpl", ...)
```

要注意当前 root module 路径。

`path.module` 指向模块目录，不是 tests 目录。

这通常是好事，因为测试跑的是模块本身。

如果你在测试文件里引用 fixture 文件，也要明确路径。

常见结构：

```text
tests/
├── basic.tftest.hcl
└── fixtures/
    └── services.json
```

复杂项目里要小心相对路径，不要让测试在本地能跑、CI 里失败。

---

## 26. Terraform test 和测试夹具 fixtures

有些测试需要输入文件或模拟数据。

可以组织成：

```text
tests/
├── basic.tftest.hcl
└── fixtures/
    └── services.json
```

或者：

```text
fixtures/
└── services.json
tests/
└── json.tftest.hcl
```

关键是保持约定一致。

如果是模块本身需要读取 mock 数据，也可以放在：

```text
mock/
```

例如前面 JSON mock lab：

```text
mock/services.json
tests/json_mock.tftest.hcl
```

原则：

> 测试数据和测试文件要跟被测模块保持近距离，方便迁移和复现。

## 28. 常见排障入口

课程文件给出的排障指南包括：

### `terraform init` 失败

检查：

- provider 约束
- 模块路径
- 网络
- Docker 环境
- backend 配置

---

### `terraform fmt -check` 失败

运行：

```sh
terraform fmt -recursive
```

再验证。

---

### `terraform test` 失败

优先检查：

- output 名称
- assert 条件
- 测试文件位置
- `.tftest.hcl` 后缀
- 是否引用了未声明的 local/output
- plan 阶段是否能得到断言值

---

### plan/apply 不一致

检查：

- 是否有 apply 后才确定的值
- 测试用了 `command = plan` 还是 `command = apply`
- 是否依赖真实 provider 返回结果

## 29. 最容易忽略的坑：引用未声明的 local

课程验证报告里提到：

```text
测试框架实验需避免引用未声明的 local
```

这点要注意。

测试里不要想当然写：

```hcl
condition = local.name == "demo"
```

如果测试上下文不能直接访问，或者 local 没暴露，容易出问题。

更稳的做法是：

```hcl
output "name" {
  value = local.name
}
```

然后测试：

```hcl
condition = output.name == "demo"
```

也就是说：

> 测试尽量断言 output 或明确可访问的对象。

## 30. 测试文件位置的生产建议

生产项目里建议：

1. 每个 root module 或 reusable module 自带 `tests/`。
2. 测试文件使用 `.tftest.hcl`。
3. 测试命名表达业务含义。
4. CI 设置正确 working directory。
5. 不要把测试文件散落在仓库各处。
6. 测试数据和 fixture 放在模块附近。
7. PR 必跑 `terraform test`。
8. 看 test 输出，确认不是 0 个测试。
9. 测试失败信息要清晰。
10. 不同模块测试不要互相污染 state。

这些都是工程化习惯。

## 33. SRE / Platform Engineering 视角

从平台工程角度看，测试文件位置是团队规范的一部分。

一个平台模块如果结构混乱：

```text

CI 不知道跑哪

```

好的结构应该让人一眼看懂：

```text

```

这就是模块工程化的基础。

不是所有高级能力都靠复杂工具。  
有时候最重要的是目录结构稳定、约定清楚、CI 能复现。

## 34. 面试/考试记忆版

可以这样记：

- Terraform test 文件通常放在 root module 下的 `tests/` 目录。
- 测试文件后缀通常是 `.tftest.hcl`。
- 在被测 Terraform 配置所在目录运行 `terraform test`。
- `run` block 定义测试用例。
- `assert` block 定义断言。
- 文件位置错或后缀错，测试可能不会被发现。
- CI/CD 中要设置正确 working directory。
- 多模块仓库中，每个模块可以有自己的 `tests/`。
- 测试尽量断言 output，不要乱引用未暴露的内部实现。
- `terraform test` 通过时要确认不是 0 个测试。

一句话总结：

> Lab 81 的核心，是把 Terraform test 文件放在正确位置，让测试能被稳定发现、稳定执行、稳定进入 CI/CD。

Lab 81 的价值在于，它把 Terraform test 从“会写断言”推进到“能组织测试文件”。
