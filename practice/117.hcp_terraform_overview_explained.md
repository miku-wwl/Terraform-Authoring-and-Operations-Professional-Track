## 1. 这一节到底在讲什么？

这一节讲的是 **HCP Terraform 是什么，以及它为什么会出现在 Terraform 考试里**。

你可以先把它理解成一句话：

> **Terraform CLI 负责执行 Terraform，HCP Terraform 负责把 Terraform 的执行过程平台化、协作化、可审计化。**

也就是说，HCP Terraform 不是让你不用写 `.tf` 文件，也不是替代 Terraform 语言本身。它更像一个团队版的 Terraform 运行平台，把原来散落在每个人电脑上的 `plan`、`apply`、变量、state、审批、日志、策略检查集中管理起来。

考试不会重点考你点哪个 UI 按钮，而是考你能不能判断：

- 什么场景适合用 HCP Terraform；
- 它解决了本地 Terraform CLI 工作流的哪些问题；
- run、workspace、state、variable、VCS、policy、cost estimation 分别处在什么位置；
- HCP Terraform 和 Terraform Cloud 是什么关系；
- 它和 Terraform CLI 是替代关系，还是协作关系。

---

## 2. 先理解传统 Terraform CLI 工作流的问题

在最基础的 Terraform 用法里，工程师通常会在自己的电脑上执行：

```bash
terraform init
terraform plan
terraform apply
terraform destroy
```

这种方式在学习阶段很直观，但是进入团队协作以后，会暴露几个问题。

### 2.1 谁执行了变更，不一定清楚

如果每个人都在本地执行 `terraform apply`，几天之后团队可能会很难回答：

- 这次 VPC 改动是谁执行的？
- 当时 plan 输出是什么？
- 谁批准了这次变更？
- apply 成功了吗？
- 失败日志在哪里？
- state 在谁手上？
- 有没有人绕过流程直接改了生产环境？

这就是考试里常见的 **audit / governance / collaboration** 场景。

HCP Terraform 的价值，就是把这些操作记录到平台里，让团队有统一的 run 历史、审批记录、执行日志和 state 版本。

---

### 2.2 State 管理容易混乱

Terraform 的 state 非常关键。它记录了 Terraform 认为自己管理了哪些资源，以及这些资源的当前属性。

如果 state 放在本地，团队协作会很危险：

- A 同学 apply 了资源；
- B 同学本地 state 没更新；
- C 同学误删了本地 state；
- 多个人同时 apply 可能造成冲突；
- 生产 state 没有版本记录，出了问题不好回滚和审计。

HCP Terraform 可以把 state 放到远程平台中，并且关联到 workspace。这样 state 不再依赖某个人的笔记本，而是成为团队共享、可追踪的基础设施记录。

---

### 2.3 变量和敏感信息不好统一管理

本地 CLI 工作流里，变量可能散落在：

- `terraform.tfvars`
- shell 环境变量
- CI/CD secret
- 本地命令参数
- 团队私下传递的配置文件

这会导致两个问题：

1. 团队成员之间配置不一致；
2. 敏感信息容易泄露。

HCP Terraform 可以在 workspace 里集中管理 variables，尤其是敏感变量，例如 cloud credential、API token、数据库密码等。

考试中看到 **sensitive variables / workspace variables / centralized variable management**，通常就是在说 HCP Terraform 的协作价值。

## 3. HCP Terraform 的核心定位

HCP Terraform 是 HashiCorp 提供的 Terraform 托管平台。它的旧名称是 Terraform Cloud。考试中如果看到 Terraform Cloud 和 HCP Terraform，不要慌，它们在很多语境下指向同一类托管协作能力。

它主要提供这些能力：

| 能力 | 解决的问题 |
|---|---|
| Remote runs | 把 plan / apply 放到平台统一执行 |
| Remote state | 集中保存 state，支持版本记录 |
| Workspace | 承载某一套 Terraform 配置、变量、state 和运行历史 |
| Run history | 记录每次 plan / apply 的过程 |
| VCS integration | GitHub / GitLab 等代码变更触发运行 |
| Variables | 统一管理普通变量和敏感变量 |
| Policy as Code | 在 apply 前进行治理检查 |
| Cost estimation | 在变更前预估成本影响 |
| Approval workflow | apply 前加入人工审批 |
| Auditability | 方便审计、排障和追责 |

一句话总结：

> **HCP Terraform 把 Terraform 从“个人电脑上的命令行工具”升级成“团队基础设施变更平台”。**

---

## 4. 重点概念一：Run

在 HCP Terraform 里，run 可以理解为一次 Terraform 运行流程。

典型 run 流程大致是：

```text
代码变更 / 手动触发
        ↓
Terraform plan
        ↓
成本估算 cost estimation（如果启用）
        ↓
策略检查 policy check（如果启用）
        ↓
人工审批 approval（如果需要）
        ↓
Terraform apply
        ↓
保存结果和 state
```

考试里非常喜欢问：

> cost estimation 和 policy check 在哪个阶段？

你要记住：

> **通常在 plan 之后、apply 之前。**

为什么？

因为只有 plan 出来以后，平台才知道“你准备创建、修改、删除什么资源”。有了这个计划，才能判断：

- 成本是否变高；
- 是否违反标签规范；
- 是否用了不允许的实例类型；
- 是否要阻断生产 apply。

如果还没有 plan，平台根本不知道要检查什么。  
如果已经 apply 完，再检查就晚了。

---

## 5. 重点概念二：Workspace

Workspace 是 HCP Terraform 的高频考点。

你可以把 workspace 理解为：

> **一套 Terraform 工作负载的运行容器。**

一个 HCP Terraform workspace 通常包含：

- Terraform 配置来源；
- 工作目录；
- variables；
- state；
- run history；
- execution settings；
- VCS 触发配置；
- policy 绑定；
- team permissions。

比如一个团队可能这样组织：

```text
workspace: network-dev
workspace: network-prod
workspace: app-dev
workspace: app-prod
workspace: security-prod
```

每个 workspace 都有自己的 state 和变量，所以 `network-dev` 和 `network-prod` 不会混在一起。

---

## 6. HCP Terraform workspace 不等于 Terraform CLI workspace

这是考试陷阱。

Terraform CLI 里也有 workspace，例如：

```bash
terraform workspace new dev
terraform workspace select prod
```

CLI workspace 更像是：

> **同一套配置下的多个 state 命名空间。**

而 HCP Terraform workspace 是平台上的协作和运行单元，包含 state、变量、run 历史、VCS 配置、权限、策略等。

简单对比：

| 对比项 | Terraform CLI workspace | HCP Terraform workspace |
|---|---|---|
| 所属环境 | 本地 CLI 概念 | HCP Terraform 平台概念 |
| 主要作用 | 区分多个 state 命名空间 | 管理一套 Terraform 工作负载 |
| 是否包含 run history | 不包含 | 包含 |
| 是否包含变量管理 | 不直接包含 | 包含 |
| 是否支持审批和策略 | 不直接支持 | 支持 |
| 考试关键词 | local workspace / state namespace | run / state / variables / VCS / policy |

记忆方式：

> **CLI workspace 偏 state 命名空间；HCP workspace 偏团队运行单元。**

---

## 7. 重点概念三：VCS Integration

VCS integration 指 HCP Terraform 可以连接 GitHub、GitLab、Bitbucket 等代码仓库。

常见流程是：

```text
开发者提交 Terraform 代码
        ↓
Pull Request / Merge
        ↓
HCP Terraform 自动触发 plan
        ↓
团队查看 plan 输出
        ↓
审批后 apply
```

这样做的好处是：

- 代码变更和基础设施变更绑定；
- plan 输出可见；
- apply 可以受控；
- 每次变更都有 commit、run、审批记录；
- 比每个人手动在本地执行更适合团队协作。

考试里看到这些关键词，要联想到 HCP Terraform：

- VCS-driven workflow
- GitHub integration
- pull request plan
- remote run
- run history
- approval before apply

---

## 8. 重点概念四：Policy Check

HCP Terraform 可以结合 Sentinel 等 Policy as Code 能力，在 apply 前检查 Terraform plan 是否符合组织要求。

常见策略包括：

- 所有资源必须有 `Owner` 标签；
- 所有资源必须有 `CostCenter` 标签；
- 不允许使用太贵的实例类型；
- 不允许公网开放数据库；
- 生产环境必须加密存储；
- 禁止创建不符合规范的网络规则。

典型流程：

```text
plan 生成资源变更计划
        ↓
policy 根据 plan 内容做检查
        ↓
通过：允许继续 apply
失败：告警、等待覆盖，或直接阻断
```

考试要点：

> **Policy check 是 plan 之后、apply 之前的治理关卡。**

---

## 9. 重点概念五：Cost Estimation

Cost estimation 是成本估算能力。

它不是“自动省钱神器”，也不是“账单系统替代品”。它更像是在 apply 之前提醒团队：

> 这次 Terraform 变更可能会让成本增加或减少。

典型场景：

- 新增高规格虚拟机；
- 创建数据库实例；
- 增加负载均衡器；
- 扩容节点池；
- 新增大量存储资源。

考试关键词：

- cost estimation
- before apply
- after plan
- cost impact
- review infrastructure cost before provisioning

要点仍然是：

> **先有 plan，才知道成本可能怎么变。**

---

## 10. CLI 还有没有价值？

有，而且考试会考这个判断。

使用 HCP Terraform 不代表 Terraform CLI 没用了。

CLI 仍然常用于：

- 本地格式化：`terraform fmt`
- 本地验证：`terraform validate`
- 初始化和调试；
- CLI-driven workflow；
- 触发远端运行；
- 查看输出；
- 开发和排错。

更准确的关系是：

> **Terraform CLI 是工具入口，HCP Terraform 是协作平台。两者不是互相消灭，而是互相配合。**

所以判断题：

> 使用 HCP Terraform 后，Terraform CLI 就完全没有学习价值了。

答案是：错。

## 11. 考试怎么出题？

### 11.1 问“主要作用”

题目可能问：

> HCP Terraform 的主要作用是什么？

正确方向：

> 提供托管的运行、协作、state、审批、变量、治理和审计能力。

错误方向通常是：

- 替代 Terraform 语言；
- 只支持 AWS；
- 只用于本地调试；
- 不需要 `.tf` 文件；
- 只用于下载 provider。

---

### 11.2 问“run history 的价值”

正确方向：

> 记录 plan、apply、审批、输出和执行结果，方便审计与排障。

错误方向通常是：

- 自动删除旧 state；
- 禁止 Git；
- 强制所有人共享本地目录；
- 替代云平台 IAM。

---

### 11.3 问“policy / cost 在哪里”

正确方向：

> plan 之后，apply 之前。

错误方向通常是：

- init 之前；
- fmt 之前；
- destroy 之后；
- apply 之后才第一次检查。

---

## 12. 场景题拆解

### 场景 1：多人本地 apply 导致无法审计

题目：

> 一个团队中多名工程师各自在笔记本运行 `terraform apply`，几天后无法确认某次基础设施变更是谁审批和执行的。HCP Terraform 能缓解哪些问题？

答题思路：

HCP Terraform 可以集中保存：

- run history；
- plan 输出；
- apply 结果；
- 审批记录；
- state 版本；
- VCS 触发来源；
- 执行日志。

所以它能缓解：

- 谁执行变更不清楚；
- 谁批准变更不清楚；
- plan 输出无法追踪；
- state 分散在本地；
- 生产变更缺少审计记录；
- 多人协作标准不统一。

可以这样回答：

> HCP Terraform 把 Terraform 运行集中到平台中，保存每次 run 的 plan、apply、审批、日志和 state 版本。这样团队可以追踪某次基础设施变更的来源、执行人和结果，减少多人本地 apply 带来的审计缺口和协作风险。

---

### 场景 2：安全团队要求所有 EC2 必须打标签

题目：

> 安全团队要求所有 EC2 都必须打标签，否则不能进入 apply 阶段。HCP Terraform 中哪个能力适合承载这类控制？

答题思路：

关键词是：

- 必须打标签；
- 不能进入 apply；
- 组织治理；
- 检查 Terraform plan。

对应能力：

> Policy as Code，例如 Sentinel policy。

可以这样回答：

> 适合使用 Policy as Code，例如 Sentinel。Terraform plan 生成后，Sentinel policy 可以检查计划中的 EC2 是否包含必要标签。如果策略失败，可以根据 enforcement mode 在 apply 前阻断运行。

## 13. 和 CI/CD 的关系

HCP Terraform 不是传统意义上的“任意 CI/CD 平台”，但它承担 Terraform 专用的运行工作流。

GitHub Actions 也能跑 Terraform，但你需要自己设计：

- state backend；
- secret 管理；
- plan 输出保存；
- apply 审批；
- policy 检查；
- 并发锁；
- 审计记录；
- drift 处理；
- 权限隔离。

HCP Terraform 则把这些 Terraform 常见需求做成托管平台能力。

考试里如果问：

> 为什么团队可能选择 HCP Terraform，而不是每个人本地运行？

核心答案就是：

> 因为它提供统一远程运行、state 管理、变量管理、审批、审计和治理。

## 14. 真实项目中的使用方式

真实团队常见做法：

```text
GitHub Repo
   └── envs/
       ├── dev/
       ├── staging/
       └── prod/
```

然后在 HCP Terraform 中建立多个 workspace：

```text
app-dev       -> repo/envs/dev
app-staging   -> repo/envs/staging
app-prod      -> repo/envs/prod
```

每个 workspace 设置不同变量：

| Workspace | Region | Instance Size | Approval |
|---|---|---|---|
| app-dev | ap-southeast-2 | small | 可自动 apply |
| app-staging | ap-southeast-2 | medium | 可手动审批 |
| app-prod | ap-southeast-2 | large | 必须审批 + policy |

这样，开发、测试、生产之间既共享代码结构，又能隔离 state、变量和审批规则。

| 易错说法 | 正确理解 |
|---|---|
| HCP Terraform 替代 Terraform CLI | 错，它是托管运行和协作平台 |

| Policy check 在 apply 后才检查 | 错，通常在 plan 后、apply 前 |
| Cost estimation 在 init 前发生 | 错，通常依赖 plan 结果 |
| HCP workspace 等于 CLI workspace | 错，语义不同 |
| run history 只是日志 | 不只是日志，还包括 plan、apply、审批、结果等记录 |
| 使用 HCP 后 CLI 没价值 | 错，CLI 仍用于开发、验证和触发工作流 |
