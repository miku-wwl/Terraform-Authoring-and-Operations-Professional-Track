# Terraform AWS Sandbox 34：AMI 动态查询的必要性

## 1. 课程元数据

- 课程编号：34
- 来源文件：`doc/34.txt`
- 环境分类：`aws-localstack-limited`
- 是否需要真实 AWS：`no`
- 是否已在 LocalStack 验证：`no`
- 涉及 AWS 服务：EC2、AMI
- 核心 Terraform 能力：避免硬编码 AMI、理解 region-specific ID、data source 设计
- 前置条件：HashiCorp Terraform Sandbox、LocalStack、Terraform CLI、AWS CLI

## 2. 知识点讲解

AMI ID 是 region-specific 的。真实 AWS 中，同一个操作系统在 `us-east-1`、`ap-southeast-2`、`ap-south-1` 等区域会有不同 AMI ID。如果把某个 region 的 AMI ID 硬编码到 Terraform 里，切换 region 后就可能失败。

正确思路是使用 `data "aws_ami"` 通过 owner、name、architecture、virtualization type 等条件动态查询镜像，再把查询结果传给 `aws_instance.ami`。

LocalStack 可以帮助练习 Terraform data source 语法、filter 写法和 `most_recent` 的表达方式，但它不提供真实 AWS 公共 AMI 目录。因此本实验是 limited：验证的是 Terraform 写法和模拟查询路径，不验证真实 Amazon Linux/Ubuntu AMI 更新逻辑。

## 3. 实验设计

背景：团队希望避免硬编码 AMI ID，为后续跨 region 部署做准备。

任务：用 bootstrap 在 LocalStack 中注册两个模拟 AMI，然后用 `data "aws_ami"` 按名称筛选最新镜像。

预期结果：Terraform 输出最新模拟 AMI ID，并说明真实 AWS 中需要替换为官方 owner 和真实 name pattern。

## 7. 排障指南

1. `aws_ami` 找不到结果：检查 bootstrap 是否成功注册模拟 AMI。
2. 找到多个结果：使用 `most_recent = true` 或收紧 filter。
3. 真实 AWS 中 AMI 不匹配实例类型：检查 architecture，例如 `x86_64` 与 `arm64`。
4. 切换 region 后失败：AMI ID 不能跨 region 复用。
