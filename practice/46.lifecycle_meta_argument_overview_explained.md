# Terraform 实操训练 46：Lifecycle meta-argument 总览（讲解版）

## 1. 本节核心：Lifecycle 用来改变资源默认生命周期行为

Terraform 默认会根据配置和 state 的差异来决定资源动作：

```text
create
update
replace
destroy
```

大多数时候，默认行为就够用。

但有些生产场景下，默认行为可能不安全、不符合业务要求，或者不能表达资源之间的特殊关系。

这时候就可以使用 `lifecycle` meta-argument。

本节重点是四个常见参数：

```hcl
lifecycle {
  create_before_destroy = true
  prevent_destroy       = true
  ignore_changes        = [...]
  replace_triggered_by  = [...]
}
```

它们分别解决四类问题：

| 参数 | 主要作用 |
|---|---|
| `create_before_destroy` | 替换资源时先创建新资源，再销毁旧资源 |
| `prevent_destroy` | 防止 Terraform 销毁关键资源 |
| `ignore_changes` | 忽略某些属性的外部变化 |
| `replace_triggered_by` | 某个依赖对象变化时，强制当前资源替换 |

一句话记忆：

```text
先建后删：create_before_destroy
禁止误删：prevent_destroy
忽略漂移：ignore_changes
联动替换：replace_triggered_by
```

---

## 2. lifecycle 是 meta-argument，不是普通资源参数

`lifecycle` 不是某个 provider 独有的参数。  
它是 Terraform 语言层面的 meta-argument。

也就是说，很多资源都可以写：

```hcl
resource "xxx" "example" {
  # resource arguments

  lifecycle {
    # lifecycle rules
  }
}
```

它控制的是 Terraform 如何管理这个资源的生命周期，而不是云服务本身的业务配置。

例如：

```hcl
resource "aws_instance" "app" {
  ami           = var.ami_id
  instance_type = "t3.micro"

  lifecycle {
    create_before_destroy = true
  }
}
```

这里 `create_before_destroy` 不是 AWS EC2 的参数，而是 Terraform 的生命周期控制规则。

---

## 3. create_before_destroy：先创建新资源，再删除旧资源

## 3.1 默认替换行为

有些资源属性一旦变化，Terraform 不能原地更新，只能替换。

默认替换顺序通常是：

```text
destroy old
create new
```

也就是先删旧资源，再建新资源。

这可能带来中断。

例如某些资源被替换时，如果旧资源先被删除，服务就可能短暂不可用。

---

## 3.2 create_before_destroy 的作用

`create_before_destroy` 可以把替换顺序改成：

```text
create new
destroy old
```

写法：

```hcl
resource "aws_instance" "app" {
  ami           = var.ami_id
  instance_type = "t3.micro"

  lifecycle {
    create_before_destroy = true
  }
}
```

含义：

> 如果这个资源必须替换，Terraform 会先尝试创建新资源，成功后再销毁旧资源。

这可以减少服务中断。

---

## 3.3 适合什么场景？

适合：

```text
需要减少替换中断
新旧资源可以短暂共存
资源名称不冲突
有足够配额创建新资源
替换期间允许双实例存在
```

常见例子：

```text
EC2 实例
Launch Template 相关资源
某些 DNS / LB 后端资源
可并行存在的应用节点
```

---

## 3.4 风险和限制

`create_before_destroy` 不是万能的。

它要求新旧资源能同时存在。

如果资源名称必须全局唯一，或者云服务不允许重复，就可能失败。

例如：

```text
固定名称的 S3 bucket
固定名称的 IAM role
固定名称的某些 DNS record
固定名称的资源组
```

如果新资源创建时名字冲突，Terraform 仍然无法先建。

所以它适合“可并存资源”，不适合所有资源。

---

## 4. prevent_destroy：防止误删关键资源

## 4.1 它解决什么问题？

有些资源一旦被删，后果很严重。

例如：

```text
生产数据库
核心 VPC
重要 S3 bucket
状态存储 bucket
关键 KMS key
生产 Kubernetes 集群
```

如果误执行：

```sh
terraform destroy
```

或者某次配置变更导致 Terraform 计划销毁这些资源，可能就是生产事故。

`prevent_destroy` 用来阻止 Terraform 销毁该资源。

---

## 4.2 基本写法

```hcl
resource "aws_s3_bucket" "critical" {
  bucket = "my-critical-state-bucket"

  lifecycle {
    prevent_destroy = true
  }
}
```

如果 Terraform 计划销毁这个资源，会报错并中止。

---

## 4.3 重要细节：prevent_destroy 必须仍在配置中

这是考试和实战都容易踩坑的点。

`prevent_destroy` 只有在这个 resource block 仍然存在于配置里时才会生效。

如果你直接把整个 resource block 从代码里删掉：

```hcl
# resource "aws_s3_bucket" "critical" 被删掉了
```

Terraform 配置里已经没有 `prevent_destroy` 规则了。  
这时 Terraform 可能会计划销毁 state 里残留的资源。

所以不要误以为：

```text
曾经写过 prevent_destroy，就永远不能删。
```

不对。

更准确地说：

```text
prevent_destroy 是当前配置中资源的生命周期规则。
```

---

## 4.4 适合什么场景？

适合保护：

```text
生产数据库
生产存储
共享网络
关键密钥
远端 state 存储
不可轻易恢复的资源
```

例如：

```hcl
resource "aws_db_instance" "prod" {
  identifier = "prod-db"

  lifecycle {
    prevent_destroy = true
  }
}
```

如果有人不小心 destroy，Terraform 会阻止。

---

## 4.5 风险和使用建议

`prevent_destroy` 很有用，但也不能乱用。

如果所有资源都加：

```hcl
prevent_destroy = true
```

后果是：

```text
正常销毁困难
测试环境清理麻烦
CI/CD destroy 阶段失败
重构资源时阻塞计划
```

推荐只给真正关键、不可轻易删除的资源加，并在注释里写清原因：

```hcl
lifecycle {
  prevent_destroy = true
  # Protect production database from accidental destroy.
}
```

---

## 5. ignore_changes：忽略某些属性变化

## 5.1 它解决什么问题？

Terraform 会比较配置和真实资源状态。

如果发现真实资源某些属性和配置不一致，就会计划修正。

但有些属性可能由外部系统管理，例如：

```text
自动扩缩容修改 desired capacity
云平台自动添加 tag
安全工具修改某些配置
Kubernetes controller 修改字段
运维系统临时调整某些值
```

如果 Terraform 每次都想改回去，就会产生 plan 噪音，甚至和外部系统打架。

这时候可以用 `ignore_changes`。

---

## 5.2 基本写法

```hcl
resource "aws_instance" "app" {
  ami           = var.ami_id
  instance_type = "t3.micro"

  tags = {
    Name = "app"
  }

  lifecycle {
    ignore_changes = [
      tags["LastPatchedAt"]
    ]
  }
}
```

含义：

```text
如果 LastPatchedAt 这个 tag 被外部系统修改，Terraform 不要试图改回去。
```

---

## 5.3 忽略整个属性

也可以忽略整个字段：

```hcl
lifecycle {
  ignore_changes = [
    tags
  ]
}
```

但这要小心。

因为忽略整个 `tags` 后，Terraform 可能不再帮你纠正任何 tag 漂移。

这会掩盖问题。

---

## 5.4 适合什么场景？

适合：

```text
某些字段由外部系统管理
某些动态字段每次都会变化
某些平台自动注入标签
某些属性不希望 Terraform 强制拉回
```

例如 Auto Scaling Group 的 desired capacity：

```hcl
resource "aws_autoscaling_group" "app" {
  desired_capacity = 3

  lifecycle {
    ignore_changes = [
      desired_capacity
    ]
  }
}
```

如果实际容量由 autoscaling policy 调整，Terraform 不应该每次都把它改回 3。

---

## 5.5 最大风险：掩盖漂移

`ignore_changes` 最危险的地方是：

```text
你可能忽略了真正应该修复的 drift。
```

例如你忽略了整个 `security_group_rules` 或 `tags`，结果别人手动改了关键安全配置，Terraform 也不提醒你。

所以使用原则是：

```text
只忽略明确需要外部系统管理的字段
不要为了让 plan 变干净就乱 ignore
```

Terraform plan 里有变化，不一定是坏事。  
它是在提醒你基础设施发生了变化。

如果你乱 ignore，就等于把报警器静音。

---

## 6. replace_triggered_by：依赖变化时强制替换

## 6.1 它解决什么问题？

有时候某个资源本身的参数没有变化，但你希望当另一个对象变化时，它也被替换。

例如：

```text
AMI 变化时，强制替换实例
启动模板变化时，替换相关资源
配置文件 hash 变化时，替换实例
证书变化时，重建依赖资源
```

这时候可以用 `replace_triggered_by`。

---

## 6.2 基本写法

```hcl
resource "aws_instance" "app" {
  ami           = var.ami_id
  instance_type = "t3.micro"

  lifecycle {
    replace_triggered_by = [
      terraform_data.ami_version
    ]
  }
}
```

如果 `terraform_data.ami_version` 发生变化，Terraform 会替换 `aws_instance.app`。

---

## 6.3 常见配合：terraform_data

在新版本 Terraform 中，可以用 `terraform_data` 作为触发器。

例如：

```hcl
resource "terraform_data" "config_hash" {
  input = filesha256("${path.module}/user_data.sh")
}

resource "aws_instance" "app" {
  ami           = var.ami_id
  instance_type = "t3.micro"

  user_data = file("${path.module}/user_data.sh")

  lifecycle {
    replace_triggered_by = [
      terraform_data.config_hash
    ]
  }
}
```

含义：

```text
user_data.sh 内容变化
-> terraform_data.config_hash 变化
-> aws_instance.app 被强制替换
```

这能表达一些 Terraform 默认推断不到的替换关系。

---

## 6.4 适合什么场景？

适合：

```text
某个外部配置变化后必须重建资源
某个依赖对象变化后当前资源不能原地更新
某些变更需要强制滚动替换
需要显式表达替换触发关系
```

但要注意：

```text
replace_triggered_by 会增加替换概率
```

生产里要确保替换是你真的想要的。

---

## 7. 四个参数对比总表

| 参数 | 解决的问题 | 典型场景 | 主要风险 |
|---|---|---|---|
| `create_before_destroy` | 替换顺序 | 减少资源替换中断 | 新旧资源不能共存时失败 |
| `prevent_destroy` | 防误删 | 保护生产数据库、关键存储 | 正常销毁和重构会被阻止 |
| `ignore_changes` | 忽略外部漂移 | 外部系统管理某些字段 | 掩盖真正的 drift |
| `replace_triggered_by` | 联动替换 | 配置变化后强制重建 | 非预期替换变多 |

一句话：

```text
create_before_destroy 管顺序
prevent_destroy 管删除
ignore_changes 管漂移
replace_triggered_by 管触发
```

---

## 8. lifecycle 的使用原则

`lifecycle` 很强，但不要滥用。

推荐原则：

```text
1. 默认不用，确实需要再用。
2. 每个 lifecycle 规则都应该有明确理由。
3. 不要为了让 plan 干净而滥用 ignore_changes。
4. 不要给所有资源无脑加 prevent_destroy。
5. create_before_destroy 前确认资源能共存。
6. replace_triggered_by 前确认替换是预期行为。
7. 重要 lifecycle 规则写注释说明原因。
```

Terraform 默认生命周期行为是有设计逻辑的。  
你改它，就要知道自己为什么改。

## 12. 常见错误和排障

## 12.1 参数混用

比如把 `create_before_destroy` 当成防误删。  
这是错的。

防误删是：

```hcl
prevent_destroy = true
```

先建后删才是：

```hcl
create_before_destroy = true
```

---

## 12.2 以为 prevent_destroy 永久保护资源

不对。

`prevent_destroy` 必须仍在配置中才有效。

如果 resource block 被删掉，它也就不在配置里了。

---

## 12.3 ignore_changes 写太宽

比如：

```hcl
ignore_changes = all
```

这几乎等于让 Terraform 对这个资源的大部分漂移闭眼。

除非非常明确，否则不建议这么做。

---

## 12.4 replace_triggered_by 导致频繁替换

如果触发对象经常变化，资源就可能经常被替换。

生产环境里要特别小心。

---

## 12.5 create_before_destroy 遇到唯一名称冲突

如果资源名称固定且不能重复，Terraform 可能无法先创建新资源。

解决方向：

```text
使用 name_prefix
让新旧资源可并存
调整替换策略
接受短暂中断
拆分资源设计
```

```hcl

    ignore_changes = [
      desired_capacity
    ]
  }

```

适合避免 Terraform 和 autoscaling policy 打架。

## 13.3 配置文件变化触发实例替换

```hcl
resource "terraform_data" "user_data_hash" {

resource "aws_instance" "app" {
  ami           = var.ami_id

    replace_triggered_by = [
      terraform_data.user_data_hash
    ]
  }
}
```

适合强制实例使用新启动配置。

---

## 13.4 替换时尽量减少中断

```hcl
resource "aws_instance" "app" {
  ami           = var.ami_id
  instance_type = "t3.micro"

  lifecycle {
    create_before_destroy = true
  }
}
```

适合新旧实例可以短暂共存的场景。

## 17. 最终总结

本节最重要的是掌握 lifecycle 四个参数的用途和风险：

```text
create_before_destroy：改变替换顺序，减少中断。
prevent_destroy：防止关键资源被误删。
ignore_changes：忽略特定属性漂移，避免和外部系统冲突。
replace_triggered_by：在依赖对象变化时强制替换当前资源。
```

一句话收尾：

```text
Lifecycle 不是让你逃避 Terraform 默认行为，而是在你明确知道风险和目标时，精确调整资源生命周期。
```
