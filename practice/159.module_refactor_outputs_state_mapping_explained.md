# Terraform 实操训练 159：挑战二解法二：模块重构后的输出与状态映射（讲解版）

## 1. 本节核心：模块重构后，代码地址和输出路径都会变

这一节是综合挑战里的“模块重构”场景。

核心不是某个 AWS 参数，而是这个问题：

> 原来资源写在 root module 里，现在重构进 child module 以后，Terraform state 地址、output 引用路径、验证脚本期望值都要一起对齐。

这类题非常贴近真实项目。

真实工作里经常会遇到：

```text
一开始所有资源都写在 main.tf
后来代码越来越大
于是把资源抽成 module
但是资源不能重建
output 还要保持给下游使用
state 地址也要正确迁移
```

这就是本节重点。

---

## 2. 什么叫模块重构？

原来代码可能长这样：

```hcl
resource "aws_s3_bucket" "app" {
  bucket = "demo-app-bucket"
}

output "bucket_name" {
  value = aws_s3_bucket.app.bucket
}
```

资源直接写在 root module。

后来你想把 bucket 抽到模块里：

```text
modules/
└── storage/
    ├── main.tf
    ├── variables.tf
    └── outputs.tf
```

root module 改成：

```hcl
module "storage" {
  source = "./modules/storage"

  bucket_name = "demo-app-bucket"
}
```

模块内部写：

```hcl
resource "aws_s3_bucket" "app" {
  bucket = var.bucket_name
}
```

这就是模块重构。

---

## 3. 最大风险：Terraform 资源地址变了

Terraform 不是只看真实 AWS 资源名来判断资源是不是同一个。

它还看 state 里的资源地址。

重构前，资源地址是：

```text
aws_s3_bucket.app
```

重构后，资源进入 module，地址变成：

```text
module.storage.aws_s3_bucket.app
```

从 Terraform 视角看：

```text
旧地址 aws_s3_bucket.app 消失了
新地址 module.storage.aws_s3_bucket.app 出现了
```

如果你不处理 state 映射，Terraform 可能会计划：

```text
destroy aws_s3_bucket.app
create  module.storage.aws_s3_bucket.app
```

这很危险。

你本来只是想“整理代码结构”，结果 Terraform 可能理解成“删除旧资源，创建新资源”。

生产环境里这就是事故入口。

---

## 4. 状态映射是什么？

状态映射就是告诉 Terraform：

> 这个资源没有换，只是从 root module 搬进了 child module。

也就是：

```text
aws_s3_bucket.app
-> module.storage.aws_s3_bucket.app
```

可以用两种方式处理。

第一种是命令式：

```sh
terraform state mv aws_s3_bucket.app module.storage.aws_s3_bucket.app
```

第二种是声明式：

```hcl
moved {
  from = aws_s3_bucket.app
  to   = module.storage.aws_s3_bucket.app
}
```

现代 Terraform 更推荐 `moved block`，因为迁移关系写在代码里，可以 code review，也更适合团队协作。

---

## 5. moved block 在模块重构里的作用

模块重构时，`moved block` 的作用就是保留资源身份。

示例：

```hcl
moved {
  from = aws_s3_bucket.app
  to   = module.storage.aws_s3_bucket.app
}
```

这表示：

```text
原来 root 里的 aws_s3_bucket.app
现在搬到了 module.storage 里的 aws_s3_bucket.app
```

Terraform 看到后，会更新 state 地址，而不是销毁真实资源再创建。

你要记住：

```text
moved block 改的是 Terraform state 地址映射
不是去 AWS 上重建资源
```

---

## 6. 模块重构后的 output 怎么处理？

模块重构以后，output 也要重新设计。

原来 root module 直接输出资源属性：

```hcl
output "bucket_name" {
  value = aws_s3_bucket.app.bucket
}
```

资源进 module 后，root module 不能再直接这样引用：

```hcl
aws_s3_bucket.app.bucket
```

因为 root module 里已经没有这个资源了。

正确做法是：

```text
child module 先输出
root module 再转接输出
```

---

## 7. child module 的 output

在 `modules/storage/outputs.tf` 中写：

```hcl
output "bucket_name" {
  value = aws_s3_bucket.app.bucket
}
```

这表示模块内部暴露一个输出：

```text
module.storage.bucket_name
```

---

## 8. root module 的 output

在 root module 里写：

```hcl
output "bucket_name" {
  value = module.storage.bucket_name
}
```

这叫 output 转接。

完整路径是：

```text
aws_s3_bucket.app.bucket
-> module 内 output "bucket_name"
-> root output "bucket_name"
-> terraform output bucket_name
```

这样做的好处是：

```text
模块内部资源地址可以变化
外部调用者仍然使用稳定 output 名称
```

这就是模块封装的关键。

---

## 9. 为什么 root output 还要保留原名字？

挑战题里验证脚本通常会检查固定 output 名称。

例如它可能执行：

```sh
terraform output -raw bucket_name
```

如果题目要求输出 `bucket_name`，你改成：

```hcl
output "storage_bucket_name" {
  value = module.storage.bucket_name
}
```

从工程上也许能解释，但验证脚本可能直接失败。

所以挑战题里要特别注意：

```text
output 名称必须和题目要求一致
```

模块内部怎么组织可以调整，但对外契约不要乱改。

这和真实项目也一样：  
output 是模块或 workspace 对外暴露的 API，不能随便破坏。

---

## 10. 模块输出和 root 输出的关系

可以这样理解：

```text
child module output = 模块内部对外暴露的接口
root module output  = 当前 Terraform 配置最终暴露给用户 / CI / 下游的接口
```

示例结构：

```text
root
├── main.tf
├── outputs.tf
└── modules
    └── storage
        ├── main.tf
        ├── variables.tf
        └── outputs.tf
```

模块内部：

```hcl
output "bucket_id" {
  value = aws_s3_bucket.app.id
}
```

root 外部：

```hcl
output "bucket_id" {
  value = module.storage.bucket_id
}
```

这就是常见转接。

---

## 11. 模块重构的安全流程

比较稳的流程是：

```text
1. 先查看当前 state 地址
2. 抽出 module 代码
3. root module 调用 child module
4. child module 补 outputs
5. root module 转接 outputs
6. 添加 moved block 或执行 state mv
7. terraform fmt
8. terraform validate
9. terraform plan
10. 确认没有非预期 destroy/create
11. terraform apply
12. terraform output
13. 运行验证脚本
14. terraform destroy
```

关键是第 10 步：

```text
模块重构不应该导致真实资源重建
```

如果 plan 里出现 destroy/create，要先停下来。

---

## 12. 本节为什么强调“输出与状态映射”？

因为模块重构会同时影响两条线。

第一条线：state 地址线

```text
aws_xxx.resource
-> module.name.aws_xxx.resource
```

第二条线：output 引用线

```text
aws_xxx.resource.attr
-> module.name.output_name
-> root output
```

很多人只处理了一条线。

例如：

```text
只改了 module，但没迁 state
```

结果 Terraform 要重建资源。

或者：

```text
state 迁了，但 root output 还引用旧资源地址
```

结果 validate / output 失败。

所以这节的标题说“输出与状态映射”，这两个都不能漏。

## 13. 一个完整示例：S3 Bucket 模块重构

## 13.1 重构前

```hcl
resource "aws_s3_bucket" "app" {
  bucket = "demo-app-bucket"
}

output "bucket_name" {
  value = aws_s3_bucket.app.bucket
}
```

state 地址：

```text
aws_s3_bucket.app
```

---

## 13.2 重构后：模块内部

`modules/storage/main.tf`：

```hcl
resource "aws_s3_bucket" "app" {
  bucket = var.bucket_name
}
```

`modules/storage/variables.tf`：

```hcl
variable "bucket_name" {
  type = string
}
```

`modules/storage/outputs.tf`：

```hcl
output "bucket_name" {
  value = aws_s3_bucket.app.bucket
}
```

---

## 13.3 重构后：root module

`main.tf`：

```hcl
module "storage" {
  source = "./modules/storage"

  bucket_name = "demo-app-bucket"
}
```

`outputs.tf`：

```hcl
output "bucket_name" {
  value = module.storage.bucket_name
}
```

`moved.tf`：

```hcl
moved {
  from = aws_s3_bucket.app
  to   = module.storage.aws_s3_bucket.app
}
```

这样 Terraform 就知道：

```text
资源搬家了，不是新建。
output 换路径了，但对外名字没变。
```

## 14. 挑战题里常见的验证点

本节这种挑战题，验证脚本可能会检查：

```text
terraform fmt 是否通过
terraform init 是否通过
terraform validate 是否通过
terraform plan 是否合理
terraform apply 是否成功
terraform output 是否存在
output 值是否符合预期
state 中资源地址是否进入 module
destroy 是否能清理资源
```

所以你不能只让 `apply` 成功。

你还要确保：

```text
output 对
state 对
验证脚本对
清理也对
```

综合题就是这样，不是跑通一步就完事。

## 15. LocalStack 在本节里的作用

本项目把真实 AWS 操作改造成 LocalStack 可验证练习。

默认：

```text
AWS API endpoint = http://localhost:4566
credentials = test/test
```

好处：

```text
不访问真实 AWS
不会产生费用
不会误删真实资源
可以反复练习
适合训练 Terraform 行为
```

但边界要记住：

```text
LocalStack 验证的是 Terraform 配置结构、Provider 调用路径、state 变化和输出设计
不能替代真实 AWS 的 IAM 权限、真实网络、真实服务限制和生产风险评估
```

也就是说，本节练的是 Terraform 工程能力，不是 AWS 生产资源本身。

## 16. 实验运行手册

进入实验目录：

```sh
cd /workspace/practice/labs/159
```


```sh
terraform init -input=false
```


查看计划：

```sh
terraform plan -input=false -no-color -out=tfplan
```

```sh
terraform apply -auto-approve tfplan
```

```sh
terraform output
```

```sh
terraform destroy -auto-approve
```

```sh
rm -f tfplan
rm -rf .terraform .terraform.lock.hcl terraform.tfstate terraform.tfstate.backup
```

## 17. 运行时应该观察什么？

这一节不要只看“成功”。

重点观察：

```text
1. module 是否被正确调用
2. module 内部 output 是否存在
3. root output 是否转接 module output
4. state 地址是否从 root 资源迁到 module 资源
5. plan 中是否出现非预期 destroy/create
6. terraform output 是否仍然返回题目要求的名称和值
7. destroy 是否能清理模块中的资源
```

你可以用：

```sh
terraform state list
```

观察状态地址。

重构后你可能会看到类似：

```text
module.storage.aws_s3_bucket.app
```

这说明资源已经进入 module 地址空间。

## 18. 常见错误和排障

## 18.1 root output 还引用旧资源

错误：

```hcl
output "bucket_name" {
  value = aws_s3_bucket.app.bucket
}
```

但资源已经移到 module 里了。

应该改成：

```hcl
output "bucket_name" {
  value = module.storage.bucket_name
}
```

---

## 18.2 child module 忘记写 output

root 写了：

```hcl
output "bucket_name" {
  value = module.storage.bucket_name
}
```

但 `modules/storage/outputs.tf` 没有：

```hcl
output "bucket_name" {
  value = aws_s3_bucket.app.bucket
}
```

那 Terraform 会报错：

```text
module.storage has no output named bucket_name
```

解决：在 child module 中补 output。

---

## 18.3 moved block 地址写错

错误示例：

```hcl
moved {
  from = aws_s3_bucket.app
  to   = module.storage.aws_s3_bucket.main
}
```

但模块里真实资源是：

```hcl
resource "aws_s3_bucket" "app" {}
```

正确地址应该是：

```hcl
moved {
  from = aws_s3_bucket.app
  to   = module.storage.aws_s3_bucket.app
}
```

排查：

```sh
terraform state list
```

以及查看 module 内资源名称。

---

## 18.4 模块名改了，state 地址也要跟着变

如果 root 里写：

```hcl
module "bucket" {
  source = "./modules/storage"
}
```

那资源地址是：

```text
module.bucket.aws_s3_bucket.app
```

不是：

```text
module.storage.aws_s3_bucket.app
```

模块名是地址的一部分。

所以 moved block 必须和实际 module block 名称一致。

---

## 18.5 plan 出现 destroy/create

如果只是模块重构，看到这种计划就要停：

```text
Plan: 1 to add, 0 to change, 1 to destroy
```

先检查：

```text
moved block 是否写了
from 地址是否等于旧 state 地址
to 地址是否等于新 module 资源地址
module 名称是否正确
resource 名称是否正确
是否同时改了不可变参数
```

不要硬 apply。

模块重构时硬 apply，很可能把“代码整理”变成“资源重建”。

## 19. import、state mv、moved block 的关系

综合挑战里经常会混到这些概念。

| 能力 | 作用 |
|---|---|
| `terraform import` | 把已有真实资源纳入 state |
| `terraform state mv` | 手动移动 state 地址 |
| `moved block` | 用代码声明 state 地址迁移 |
| `output` | 暴露资源或模块结果 |

本节更偏向：

```text
模块重构后的 state 地址映射 + output 转接
```

如果真实资源还没进 state，可能需要 import。  
如果资源已经在 state，只是从 root 搬进 module，通常是 state mv 或 moved block。

---

## 20. 生产环境推荐做法

真实项目做模块重构时，建议：

```text
1. 不要一次重构太多资源
2. 先备份 state
3. 先在 dev 环境验证
4. 使用 moved block 表达迁移关系
5. PR 中清楚说明旧地址和新地址
6. plan 必须人工 review
7. 不能出现非预期 destroy/create
8. output 名称尽量保持兼容
9. 下游依赖 output 的 workspace 要评估影响
10. apply 后检查 state list 和 output
```

尤其是 output，如果被下游使用：

```text
remote state
CI/CD
其他 module
脚本
监控系统
```

就不能随便改名。

output 是对外契约，不是内部变量名。

## 21. 模块重构的正确心态

模块化不是为了显得高级。

模块化的目的应该是：

```text
减少重复
明确边界
提高复用
降低维护成本
隐藏内部实现
暴露稳定接口
```

如果你重构后：

```text
state 乱了
output 断了
plan 要重建
验证失败
下游全挂
```

那不是模块化，那是把问题包装成了更难排查的形式。

好的模块重构应该做到：

```text
外部接口尽量稳定
内部结构可以优化
真实资源不被无故重建
state 地址迁移可审计
plan 结果可解释
```

## 24. 最终总结

本节的核心是：

> 模块重构不只是移动 `.tf` 文件，还要同步处理 output 转接和 state 地址映射。

完整主线：

```text
原资源在 root
-> 抽到 child module
-> module 暴露 output
-> root 转接 output
-> moved block 映射旧地址到新地址
-> plan 确认不重建
-> apply 更新 state
-> output / verify / destroy 全流程通过
```

一句话收尾：

```text
Terraform 模块重构的底线是：代码可以搬家，资源别被误拆；模块可以封装，输出别断契约。
```
