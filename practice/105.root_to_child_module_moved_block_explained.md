# 第 105 节讲解：把 Root Module 拆分为 Child Modules，并用 moved block 避免重建

## 1. 这一节到底在练什么？

这一节练的是 Terraform 工程化里非常重要的一件事：

> 原来所有资源都堆在 root module 里，后面代码变大了，需要拆成 child modules；但是拆代码时不能把真实云资源删了重建。

也就是说，这不是单纯的“把文件移动一下”。  
Terraform 关心的不是你文件怎么放，而是 **state 里记录的资源地址**。

例如原来资源地址是：

```hcl
aws_s3_bucket.demo
```

你把它挪进子模块后，新地址会变成：

```hcl
module.storage.aws_s3_bucket.demo
```

Terraform 一看地址变了，如果你不告诉它“这是同一个资源，只是换了地址”，它就可能认为：

- 旧资源 `aws_s3_bucket.demo` 不要了，要 destroy
- 新资源 `module.storage.aws_s3_bucket.demo` 没有，要 create

这就很危险。  
真实 AWS 里如果是 S3、IAM、VPC、RDS 这种资源，被误删重建就不是小事了。

本节核心就是：

> 用 `moved` block 告诉 Terraform：这个资源只是从旧地址搬到了新地址，不要删了重建。

---

## 2. Root Module 和 Child Module 是什么？

### Root Module

Root module 就是你执行 `terraform init / plan / apply` 的那个目录。

例如：

```text
practice/labs/105/work/
├── main.tf
├── variables.tf
├── outputs.tf
└── providers.tf
```

这个目录就是 root module。

### Child Module

Child module 是被 root module 调用的模块。

例如：

```text
practice/labs/105/work/
├── main.tf
├── providers.tf
└── modules/
    └── storage/
        ├── main.tf
        ├── variables.tf
        └── outputs.tf
```

root module 通过 `module` 块调用它：

```hcl
module "storage" {
  source = "./modules/storage"

  bucket_name = var.bucket_name
}
```

考试里要记住：

- root module：当前执行 Terraform 的入口目录
- child module：被 `module` 块调用的复用单元
- module 的目的：复用、封装、拆分复杂配置
- module 本身不会自动让资源变安全，state 地址迁移才是关键

---

## 3. 为什么拆模块会导致资源重建？

Terraform 通过 **资源地址** 来追踪 state 里的对象。

原来你在 root module 里写：

```hcl
resource "aws_s3_bucket" "demo" {
  bucket = var.bucket_name
}
```

资源地址是：

```text
aws_s3_bucket.demo
```

后来你把它放到 `modules/storage/main.tf` 里：

```hcl
resource "aws_s3_bucket" "demo" {
  bucket = var.bucket_name
}
```

然后 root module 调用：

```hcl
module "storage" {
  source = "./modules/storage"

  bucket_name = var.bucket_name
}
```

资源地址变成：

```text
module.storage.aws_s3_bucket.demo
```

注意：资源名字还是 `demo`，但是完整地址已经变了。

Terraform 不会自动猜测“这两个其实是同一个资源”。  
所以它可能计划：

```text
- destroy aws_s3_bucket.demo
+ create  module.storage.aws_s3_bucket.demo
```

这就是拆模块时最容易翻车的地方。

---

## 4. moved block 是干嘛的？

`moved` block 用来声明资源地址迁移。

典型写法：

```hcl
moved {
  from = aws_s3_bucket.demo
  to   = module.storage.aws_s3_bucket.demo
}
```

意思是：

> state 里原来叫 `aws_s3_bucket.demo` 的对象，现在应该改名为 `module.storage.aws_s3_bucket.demo`。

它解决的是 **Terraform state 地址迁移问题**，不是资源本身的配置问题。

正确使用后，`terraform plan` 应该看到类似含义：

```text
resource has moved
from aws_s3_bucket.demo
to   module.storage.aws_s3_bucket.demo
```

而不是 destroy/create。

---

## 5. moved block 和 terraform state mv 的区别

这两个都能做 state 地址迁移，但适用场景不一样。

### moved block

写在 `.tf` 文件里：

```hcl
moved {
  from = aws_s3_bucket.demo
  to   = module.storage.aws_s3_bucket.demo
}
```

优点：

- 可代码审查
- 可提交到 Git
- 团队成员都能看到迁移意图
- 适合长期维护
- 更符合现代 Terraform 推荐方式

### terraform state mv

命令式操作：

```bash
terraform state mv aws_s3_bucket.demo module.storage.aws_s3_bucket.demo
```

优点：

- 适合一次性紧急修复
- 老版本 Terraform 也能用

缺点：

- 操作历史不一定体现在代码中
- 多人协作容易漏
- CI/CD 环境里不如 moved block 清晰

考试里看到“重构配置、资源地址变更、避免重建”，优先想到 `moved` block。

## 6. 本实验为什么用 LocalStack？

本节使用 LocalStack 的目的，是让你不用真实 AWS 账号，也能练 AWS Provider 的基本行为。

实验中的 AWS API endpoint 是：

```text
http://localhost:4566
```

凭证是假凭证：

```text
test/test
```

这说明：

- 不会访问真实 AWS
- 不会产生真实费用
- 可以验证 Terraform 配置、provider 调用、state 迁移路径
- 但不能完全等价于真实 AWS 的所有边界条件

真实 AWS 里还要考虑：

- IAM 权限
- S3 bucket 全局唯一命名
- 删除保护
- 数据保留
- 生产变更审批
- remote backend 锁
- 多人协作状态一致性

## 7. 推荐的实验操作流程

### 第一步：进入 work 目录

```bash
cd practice/labs/105/work
```

### 第二步：启动 LocalStack

如果实验提供了 Docker Compose：

```bash
docker compose up -d
```

确认 LocalStack 正常：

```bash
curl http://localhost:4566/_localstack/health
```

### 第三步：初始化 Terraform

```bash
terraform init
```

### 第四步：先 apply 初始 root module 版本

```bash
terraform plan
terraform apply -auto-approve
```

这一步的目标是先创建一份 state，让 Terraform 记录旧资源地址。

### 第五步：拆分资源到 child module

把原来 root module 里的资源移动到类似目录：

```text
modules/storage/
```

然后在 root module 中增加：

```hcl
module "storage" {
  source = "./modules/storage"

  bucket_name = var.bucket_name
}
```

### 第六步：添加 moved block

在 root module 里增加：

```hcl
moved {
  from = aws_s3_bucket.demo
  to   = module.storage.aws_s3_bucket.demo
}
```

具体资源类型和名字要以你的实验文件为准。

### 第七步：重新 plan

```bash
terraform plan
```

你要重点看 plan 结果：

理想情况：

```text
Plan: 0 to add, 0 to change, 0 to destroy
```

或者出现 moved 说明，但没有 destroy/create。

危险情况：

```text
Plan: 1 to add, 0 to change, 1 to destroy
```

这说明 Terraform 没有识别为迁移，而是准备删旧建新。

### 第八步：apply 迁移

```bash
terraform apply -auto-approve
```

### 第九步：验证 state 地址

```bash
terraform state list
```

你应该看到类似：

```text
module.storage.aws_s3_bucket.demo
```

旧地址：

```text
aws_s3_bucket.demo
```

应该不再出现。

### 第十步：销毁实验资源

```bash
terraform destroy -auto-approve
```

如果有 LocalStack 容器：

```bash
docker compose down
```

## 8. 典型目录结构参考

拆分前：

```text
work/
├── main.tf
├── variables.tf
├── outputs.tf
└── providers.tf
```

拆分后：

```text
work/
├── main.tf
├── variables.tf
├── outputs.tf
├── providers.tf
└── modules/
    └── storage/
        ├── main.tf
        ├── variables.tf
        └── outputs.tf
```

root module 的 `main.tf` 可能只保留模块调用：

```hcl
module "storage" {
  source = "./modules/storage"

  bucket_name = var.bucket_name
}
```

child module 的 `main.tf` 放真实资源：

```hcl
resource "aws_s3_bucket" "demo" {
  bucket = var.bucket_name
}
```

child module 的 `variables.tf`：

```hcl
variable "bucket_name" {
  type        = string
  description = "S3 bucket name used by the storage module."
}
```

## 9. 考试重点

### 重点 1：module 是封装，不是复制粘贴

module 的价值是：

- 减少重复代码
- 统一资源标准
- 形成工程边界
- 方便团队协作
- 提升可维护性

但 module 不是魔法。  
如果 state 地址没处理好，照样会出现误删重建。

### 重点 2：moved block 只处理地址迁移

它不负责：

- 修复错误配置
- 自动导入云上已有资源
- 自动创建 provider
- 自动解决权限问题

它只是在告诉 Terraform：

> 旧地址和新地址是同一个对象。

### 重点 3：看 plan 是保命动作

拆模块后一定要看：

```text
Plan: x to add, x to change, x to destroy
```

生产环境里，看到 destroy 就要停下来确认。  
不要闭眼 `apply`。

## 10. 常见错误

### 错误 1：只移动文件，不写 moved block

结果 Terraform 可能认为旧资源要删，新资源要建。

### 错误 2：moved block 地址写错

例如模块名叫 `module.storage`，你却写成：

```hcl
module.s3.aws_s3_bucket.demo
```

Terraform 就无法正确迁移。

### 错误 3：误以为资源名称一样就不会重建

资源短名一样不代表完整地址一样。

```text
aws_s3_bucket.demo
module.storage.aws_s3_bucket.demo
```

这两个在 Terraform 看来是两个不同地址。

### 错误 4：把 child module 当成独立环境

child module 通常不是你直接进去 `terraform apply` 的地方。  
真正执行 Terraform 的入口还是 root module。

### 错误 5：生产环境不检查 plan

这属于 Terraform 里的“闭眼开挖掘机”。  
拆模块是高风险重构，必须先看 plan。

## 11. 面试解释模板

如果面试官问：

> 你如何把 Terraform root module 中的资源迁移到 child module，并避免资源重建？

可以这样回答：

```text
我会先确认当前 state 中的资源地址，然后把资源配置移动到 child module。
root module 通过 module block 调用新的 child module。
因为资源从 root module 移到 child module 后，Terraform 地址会变化，
例如从 aws_s3_bucket.demo 变成 module.storage.aws_s3_bucket.demo，
所以我会在 root module 中添加 moved block，把旧地址映射到新地址。
之后运行 terraform plan，确认 plan 中只显示资源 moved，而不是 destroy/create。
确认无误后再 apply，并用 terraform state list 验证 state 地址已经迁移完成。
```

这段回答很适合 Terraform Professional / DevOps / SRE 面试。

---

## 12. 本节一句话总结

> 拆 Terraform 模块时，真正危险的不是移动代码，而是资源地址变了；`moved` block 的作用就是把 state 里的旧地址安全迁移到新地址，避免真实资源被误删重建。
