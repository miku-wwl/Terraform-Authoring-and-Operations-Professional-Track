# Terraform 实操训练 100：模块中的多 Provider 配置讲解

## 1. 本节主题

本节讲的是 Terraform 中一个非常容易在真实项目里踩坑的点：**模块中的多 Provider 配置**。

核心场景是：

> 在 root module 中定义多个 AWS provider 配置，例如默认 provider 和带 `alias` 的 provider，然后在调用 child module 时，通过 `providers` map 明确告诉模块：你应该使用哪一个 provider 配置。

这节不是单纯背 `alias` 语法，而是要理解 Terraform 的 Provider 配置边界：

- Provider 配置通常应该放在 root module。
- Child module 不应该自己随便配置真实环境 provider。
- Root module 调用 child module 时，可以通过 `providers = { ... }` 显式传入 provider。
- 多区域、多账号、多环境部署时，这个机制非常关键。

本实验使用 LocalStack 模拟 AWS API，不访问真实 AWS。

---

## 2. 为什么需要多 Provider 配置

在真实 AWS 项目中，经常会遇到这些场景：

1. **多 Region 部署**
   - 例如主资源在 `ap-southeast-2`，灾备资源在 `us-east-1`。

2. **多账号部署**
   - 例如一个账号放网络基础设施，另一个账号放应用资源。

3. **共享模块复用**
   - 同一个模块可以被不同环境调用，例如 dev、staging、prod。

4. **跨区域资源依赖**
   - 例如 ACM 证书必须在 `us-east-1`，但应用资源在另一个 region。

如果没有明确的 provider 传递机制，模块很容易出现：

- 用错 region；
- 用错 AWS account；
- 在 child module 里硬编码 provider；
- 模块复用性变差；
- CI/CD 环境行为不确定。

Terraform 的解决方式就是：

```hcl
provider "aws" {
  region = "ap-southeast-2"
}

provider "aws" {
  alias  = "dr"
  region = "us-east-1"
}

module "example" {
  source = "./modules/example"

  providers = {
    aws = aws.dr
  }
}
```

这段的意思是：

> root module 中虽然有多个 AWS provider，但调用 `example` 模块时，明确让它使用 `aws.dr` 这个 alias provider。

---

## 3. root module 和 child module 的职责边界

### root module 负责什么

Root module 是整个 Terraform 执行入口，通常负责：

- 定义 provider 配置；
- 配置 backend；
- 组织不同 child module；
- 传入环境变量、region、命名规则等参数；
- 决定哪个模块使用哪个 provider。

也就是说，root module 更像是“总控台”。

### child module 负责什么

Child module 应该负责封装资源逻辑，例如：

- 创建 S3 bucket；
- 创建 VPC；
- 创建 IAM role；
- 创建一组标准化应用资源。

Child module 最好不要把 provider 配置写死，比如不要在模块内部硬编码：

```hcl
provider "aws" {
  region = "us-east-1"
}
```

这会导致模块强绑定某个环境，复用性很差。

更好的方式是：

```hcl
resource "aws_s3_bucket" "this" {
  bucket = var.bucket_name
}
```

然后由 root module 决定这个资源到底使用哪个 provider。

---

## 4. providers map 是什么

`providers` map 是 module block 里的一个特殊参数，用来显式传递 provider 配置。

典型写法：

```hcl
module "storage" {
  source = "./modules/storage"

  providers = {
    aws = aws.secondary
  }

  bucket_name = "demo-bucket"
}
```

含义是：

- 左边的 `aws`：child module 里看到的 provider 名称；
- 右边的 `aws.secondary`：root module 中已经配置好的 alias provider。

可以把它理解成：

> “模块里面叫 aws 的 provider，实际使用 root module 里的 aws.secondary。”

这个机制很像依赖注入。模块不关心 provider 具体怎么配置，只关心自己要创建什么资源。

---

## 5. alias provider 的作用

Provider alias 用于在同一个 root module 中创建多个同类型 provider 配置。

例如：

```hcl
provider "aws" {
  region                      = "ap-southeast-2"
  access_key                  = "test"
  secret_key                  = "test"
  s3_use_path_style           = true
  skip_credentials_validation = true
  skip_metadata_api_check     = true
  skip_requesting_account_id  = true

  endpoints {
    s3 = "http://localhost:4566"
  }
}

provider "aws" {
  alias                       = "use1"
  region                      = "us-east-1"
  access_key                  = "test"
  secret_key                  = "test"
  s3_use_path_style           = true
  skip_credentials_validation = true
  skip_metadata_api_check     = true
  skip_requesting_account_id  = true

  endpoints {
    s3 = "http://localhost:4566"
  }
}
```

这里有两个 AWS provider：

- `aws`：默认 provider；
- `aws.use1`：带 alias 的 provider。

在真实 AWS 中，它们可能对应不同 region 或不同 account。

在本实验里，它们都指向 LocalStack 的 `http://localhost:4566`，只是用来模拟多 provider 场景。

## 6. Child module 里需要不要写 alias？

这要看模块内部怎么使用 provider。

### 情况一：模块只需要一个 AWS provider

如果 child module 里所有资源都使用默认的 `aws` provider，那么模块内部不需要写 alias。

例如 child module：

```hcl
resource "aws_s3_bucket" "this" {
  bucket = var.bucket_name
}
```

Root module 调用时这样传：

```hcl
module "storage" {
  source = "./modules/storage"

  providers = {
    aws = aws.use1
  }

  bucket_name = "demo-bucket"
}
```

这样 child module 里面的 `aws_s3_bucket.this` 就会使用 root module 传入的 `aws.use1`。

### 情况二：模块内部同时使用多个 AWS provider

如果 child module 内部要显式区分多个 provider，例如：

```hcl
resource "aws_s3_bucket" "primary" {
  provider = aws.primary
  bucket   = var.primary_bucket_name
}

resource "aws_s3_bucket" "replica" {
  provider = aws.replica
  bucket   = var.replica_bucket_name
}
```

那 child module 需要在 `terraform.required_providers` 里声明 `configuration_aliases`：

```hcl
terraform {
  required_providers {
    aws = {
      source = "hashicorp/aws"
      configuration_aliases = [
        aws.primary,
        aws.replica
      ]
    }
  }
}
```

然后 root module 调用时传入：

```hcl
module "multi_region_storage" {
  source = "./modules/multi-region-storage"

  providers = {
    aws.primary = aws
    aws.replica = aws.use1
  }

  primary_bucket_name = "primary-demo-bucket"
  replica_bucket_name = "replica-demo-bucket"
}
```

这个写法更适合真正的多区域、多账号模块。

## 7. 本实验为什么使用 LocalStack

上传文件中说明，本节实验不访问真实 AWS，所有 AWS API 都通过 LocalStack 的 `http://localhost:4566` endpoint 模拟，并使用假凭证 `test/test`。

这很适合训练 Terraform provider 行为，因为：

- 不需要真实 AWS 账号；
- 不会产生云账单；
- 可以在本地快速验证 `init / plan / apply / destroy`；
- 可以把重点放在 Terraform provider 传递机制，而不是 AWS 权限和网络问题上。

但是要注意：LocalStack 是模拟环境，不等于真实 AWS。

它适合验证 Terraform 配置结构和基础 API 调用路径，不适合完全证明真实生产环境行为。

## 8. 实验目标拆解

原始文件中的实验目标可以拆成四层。

### 第一层：理解 Terraform / AWS Provider 行为

你要看懂：

- provider 是在 root module 配置的；
- alias provider 是同类型 provider 的另一个配置实例；
- child module 可以通过 providers map 接收 provider；
- provider 的选择会影响资源创建位置。

### 第二层：完成 LocalStack 闭环

实验应能完成：

```sh
terraform init
terraform plan
terraform apply
terraform destroy
```

如果是课程标准化 lab，还应包含：

```sh
terraform fmt -check
terraform validate
terraform test
```

### 第三层：区分模拟环境和真实 AWS

LocalStack 中的 endpoint、credentials、account id 检查都可能被跳过。

真实 AWS 中你还需要关注：

- AWS credentials 来源；
- IAM 权限；
- region 是否正确；
- provider version lock；
- backend 和 state isolation；
- 跨账号 assume role；
- CI/CD 中的密钥注入方式。

### 第四层：迁移到真实项目

真实项目里，这个能力可以迁移到：

- 多 region S3 / DynamoDB / KMS；
- 多账号 IAM role 管理；
- 网络模块复用；
- shared service account 和 workload account 分离；
- 灾备架构；
- 平台工程中的标准模块治理。

## 11. 常见错误与排障

### 错误一：child module 找不到 provider alias

常见报错类似：

```text
Provider configuration not present
```

或者：

```text
Missing required provider configuration
```

排查方向：

- root module 是否定义了 alias provider；
- module block 是否写了 `providers = { ... }`；
- child module 是否声明了 `configuration_aliases`；
- 左右两边 provider 名称是否匹配。

### 错误二：providers map 写反了

错误理解：

```hcl
providers = {
  aws.use1 = aws
}
```

如果 child module 只是使用默认 `aws` provider，这通常不是你想要的写法。

更常见的是：

```hcl
providers = {
  aws = aws.use1
}
```

意思是模块里的默认 `aws` 使用 root module 里的 `aws.use1`。

### 错误三：在 child module 里硬编码 provider

不推荐：

```hcl
provider "aws" {
  region = "us-east-1"
}
```

原因：

- 模块被环境绑定；
- 调用方难以覆盖；
- 多账号、多区域复用困难；
- 生产环境可维护性差。

### 错误四：LocalStack endpoint 没配置全

如果 provider 没有正确指向 LocalStack，就可能尝试访问真实 AWS。

排查方向：

- 是否配置了 `endpoints`；
- 是否使用了假 credentials；
- 是否设置了跳过 AWS 校验的参数；
- LocalStack 容器是否启动；
- 端口 `4566` 是否可访问。

## 12. 生产环境中的最佳实践

### 1. Provider 配置放在 root module

Child module 只声明 provider requirement，不负责真实 provider 连接细节。

### 2. 多 provider 显式传递

不要依赖隐式继承来处理复杂场景。

多 region、多 account 时，推荐明确写：

```hcl
providers = {
  aws = aws.dr
}
```

或者：

```hcl
providers = {
  aws.primary = aws.primary
  aws.replica = aws.replica
}
```

### 3. 模块不要偷偷决定部署位置

模块应该通过变量和 provider 注入来复用，而不是在内部写死 region、account、endpoint。

### 4. 真实环境要锁 provider 版本

使用 `.terraform.lock.hcl` 固定 provider 版本，避免 CI/CD 中 provider 行为突然变化。

### 5. 多账号建议配合 assume role

真实生产中，不建议把长期 AK/SK 写入 provider。

更常见做法是：

```hcl
provider "aws" {
  alias  = "prod"
  region = "ap-southeast-2"

  assume_role {
    role_arn = "arn:aws:iam::123456789012:role/TerraformExecutionRole"
  }
}
```

## 13. 本节你真正要记住的东西

这节的核心不是：

> 我会写 provider alias。

而是：

> 我知道 provider 配置是 Terraform 模块复用边界的一部分。Root module 负责注入 provider，child module 负责声明资源逻辑。多 region、多账号、多环境时，必须显式管理 provider 传递关系。

一句话总结：

> **Provider alias 是多个环境入口，module providers map 是把入口接到模块上的插线板。线接错了，资源就可能部署到错误账号或错误 region。**
