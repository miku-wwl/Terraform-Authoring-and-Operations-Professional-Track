# Terraform 实操训练 24：`moved` block 与资源重命名

## 1. 课程元数据

- 课程编号：24
- 来源文件：`doc/24.txt`
- 环境分类：`docker`
- 是否可用 Docker 运行：`yes`
- 核心主题：`moved` block、资源地址重命名、避免重建
- 前置条件：Docker、`hashicorp/terraform:1.11`

## 2. 知识点讲解

Terraform 资源地址是 state 中的身份标识。直接把资源名从 `old` 改成 `new`，Terraform 往往会理解成删除旧地址、创建新地址。`moved` block 用声明式方式告诉 Terraform：这是一次地址迁移，不是资源替换。

生产中重命名资源、移动到模块、拆分模块时都应优先考虑 `moved` block，避免对长期资源造成误删风险。

## 3. 实验设计

背景：团队要把旧命名 `database_firewall` 改为 `payment_database_firewall`。

问题：直接重命名会被解释为重建。

任务：使用 `moved` block 记录旧地址到新地址的迁移。

预期结果：输出旧地址、新地址和新资源名称。

## 5. 运行手册

```sh
cd /workspace/practice/labs/24

terraform apply -auto-approve tfplan

terraform destroy -auto-approve
```

状态：通过。

已执行 `init/fmt/validate/test/plan/apply/output/destroy`。`terraform test` 返回 `1 passed, 0 failed`，输出记录旧地址 `terraform_data.database_firewall` 和新地址 `terraform_data.payment_database_firewall`。

## 7. 排障指南

1. moved 源地址不存在：确认旧资源曾经在 state 中存在。
2. 地址写错：使用完整资源地址，包括模块路径。
3. 仍显示 destroy/create：检查是否同时改变了不可更新属性。
4. 多次重构：每次迁移都保留清晰的 moved 记录。

## 8. 清理方式

```sh
terraform destroy -auto-approve
rm -f tfplan
rm -rf .terraform .terraform.lock.hcl
```
