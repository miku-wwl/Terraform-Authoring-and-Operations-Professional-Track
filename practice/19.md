# Terraform 实操训练 19：Terraform import 工作流

## 1. 课程元数据

- 课程编号：19
- 来源文件：`doc/19.txt`
- 环境分类：`docker`
- 是否可用 Docker 运行：`yes`
- 核心主题：Terraform import、import block、生成配置、状态接管
- 前置条件：Docker、`hashicorp/terraform:1.11`

## 2. 知识点讲解

Terraform import 用来把已经存在的资源纳入 Terraform state。老方式只能导入 state，配置仍要人工补齐；Terraform 1.5 之后可以配合 import block 和 `-generate-config-out` 生成候选配置，显著降低接管遗留资源的成本。

生产环境中，import 的核心风险不是命令本身，而是资源地址、真实 ID、生成配置和现有运行状态是否一致。导入前要先盘点资源，导入后要审查生成配置，不要把生成文件直接当成最终生产代码。

## 3. 实验设计

背景：平台团队要把手工维护的资源纳入 IaC 管理。

问题：资源已经存在，但没有 Terraform 配置和 state。

任务：用本地 `terraform_data` 模拟遗留资源，固化 import 工作流说明。

预期结果：实验能输出 import 的关键步骤，并保留被接管资源的业务 ID。

## 5. 运行手册

```sh
cd /workspace/practice/labs/19
terraform init -input=false
terraform fmt -check
terraform validate
terraform test
terraform plan -input=false -no-color -out=tfplan
terraform apply -auto-approve tfplan
terraform output
terraform destroy -auto-approve
```

课程中的安全组导入需要 AWS，本实验是本地替代实验，不等价于真实 AWS 导入。

## 7. 排障指南

1. 生成配置命令不可用：确认 Terraform 版本不低于 1.5。
2. 导入地址写错：检查 `to` 是否是配置中的完整资源地址。
3. 真实 ID 写错：先在目标平台确认资源 ID。
4. 生成配置过于冗长：审查后再重构为可维护代码。

## 8. 清理方式

```sh
terraform destroy -auto-approve
rm -f tfplan
rm -rf .terraform .terraform.lock.hcl
```
