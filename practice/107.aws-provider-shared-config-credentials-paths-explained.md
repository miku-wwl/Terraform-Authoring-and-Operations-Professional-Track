# Terraform 实操训练 107：AWS Provider 指定 shared config / credentials 路径

## 1. 本节定位

本节主题是 **AWS Provider 如何读取非默认位置的 shared config 和 shared credentials 文件**。

在真实 AWS 项目里，AWS Provider 默认会从标准路径读取配置，例如：

- `~/.aws/config`
- `~/.aws/credentials`

但是在团队工程、CI/CD、容器实验、临时沙箱、LocalStack 模拟环境里，我们经常不希望 Terraform 依赖默认家目录。  
这时就可以用：

- `shared_config_files`
- `shared_credentials_files`

显式告诉 AWS Provider：  
**不要去默认位置找配置，去我指定的文件路径找。**

一句话理解：

> 这是把 AWS Provider 的凭证来源从“隐式默认路径”改成“显式工程路径”。

---

## 2. 为什么这个知识点重要

很多初学 Terraform 的人会直接在 provider 里写：

```hcl
provider "aws" {
  region     = "ap-southeast-2"
  access_key = "test"
  secret_key = "test"
}
```

这在本地练习能跑，但生产环境不推荐。原因很简单：

1. 凭证容易写进 Git 仓库。
2. 多环境切换不清晰。
3. CI/CD 里不好统一管理。
4. 不利于模拟真实 AWS CLI / SDK 的凭证加载逻辑。

更合理的做法是让 provider 读取外部 credentials/config 文件。  
这样 Terraform、AWS CLI、SDK 的行为会更接近真实工程。

本节训练的就是这个能力。

---

## 3. shared_config_files 和 shared_credentials_files 是什么

### 3.1 shared_credentials_files

`shared_credentials_files` 指定 AWS credentials 文件路径。

典型内容类似：

```ini
[default]
aws_access_key_id = test
aws_secret_access_key = test
```

或者：

```ini
[localstack]
aws_access_key_id = test
aws_secret_access_key = test
```

它主要负责身份凭证。

---

### 3.2 shared_config_files

`shared_config_files` 指定 AWS config 文件路径。

典型内容类似：

```ini
[profile localstack]
region = us-east-1
output = json
```

它主要负责 profile、region、output 等配置。

注意一个小坑：

- credentials 文件里 profile 名通常写 `[localstack]`
- config 文件里 profile 名通常写 `[profile localstack]`

这俩格式不完全一样。别写错，不然 provider 会像没吃饭一样找不到配置。

---

## 4. 本节实验场景

本节实验不访问真实 AWS。

所有 AWS API 都通过 LocalStack 模拟：

```text
http://localhost:4566
```

凭证使用假值：

```text
test / test
```

这说明：

- 实验重点不是验证真实 AWS IAM 权限。
- 实验重点是验证 AWS Provider 能不能从指定路径读取配置。
- LocalStack 只是本地模拟 AWS API 的后端。
- 真实 AWS 项目里，需要替换成真实凭证来源和真实 endpoint。

---

## 5. 典型 provider 写法

一个常见写法如下：

```hcl
provider "aws" {
  region  = "us-east-1"
  profile = "localstack"

  shared_config_files      = ["${path.module}/aws/config"]
  shared_credentials_files = ["${path.module}/aws/credentials"]

  access_key                  = "test"
  secret_key                  = "test"
  skip_credentials_validation = true
  skip_metadata_api_check     = true
  skip_requesting_account_id  = true
  s3_use_path_style           = true

  endpoints {
    s3 = "http://localhost:4566"
  }
}
```

不同实验的 provider 内容可能略有差异，但核心就是：

```hcl
shared_config_files      = [...]
shared_credentials_files = [...]
```

这两个参数把 AWS Provider 的配置读取路径显式固定下来。

---

## 6. 这和环境变量有什么区别

AWS Provider 可以从很多地方读取凭证，例如：

1. provider block 里直接写 `access_key` / `secret_key`
2. 环境变量 `AWS_ACCESS_KEY_ID` / `AWS_SECRET_ACCESS_KEY`
3. 默认文件 `~/.aws/config` 和 `~/.aws/credentials`
4. 显式指定的 `shared_config_files` 和 `shared_credentials_files`
5. EC2/ECS/EKS workload identity / metadata
6. AssumeRole / SSO / OIDC

本节关注第 4 种。

它的好处是：

- 路径清楚。
- 适合实验目录自包含。
- 适合 Docker / CI/CD。
- 不依赖操作者本机的 AWS 配置。
- 不会因为用户电脑上已经登录过 AWS 而“误打误撞成功”。

这点很关键。  
实验环境最怕的就是：你以为代码写对了，其实是你本机老凭证帮你兜底了。

## 11. 应该观察什么

运行时重点观察几类现象。

### 11.1 init 是否能正常下载 provider

如果 `terraform init` 失败，通常不是 shared config 的问题，而是 provider 下载、版本约束、网络或 registry 访问问题。

---

### 11.2 provider 是否使用了指定 profile

如果 profile 名写错，常见报错会类似：

```text
failed to get shared config profile
```

或者：

```text
no valid credential sources found
```

这说明 Terraform 没找到你指定的 profile。

---

### 11.3 LocalStack endpoint 是否正确

如果 endpoint 没指向 LocalStack，Terraform 可能会尝试访问真实 AWS。

这在实验里是不希望发生的。

所以本地模拟实验里，provider 通常还会配：

```hcl
skip_credentials_validation = true
skip_metadata_api_check     = true
skip_requesting_account_id  = true
```

这些配置是为了避免 provider 去真实 AWS 做身份校验。

## 12. 常见错误与排查

### 错误 1：config 文件 profile 写错

错误写法：

```ini
[localstack]
region = us-east-1
```

在 config 文件里，如果你用的是具名 profile，更常见写法应该是：

```ini
[profile localstack]
region = us-east-1
```

---

### 错误 2：credentials 文件 profile 写错

错误写法：

```ini
[profile localstack]
aws_access_key_id = test
aws_secret_access_key = test
```

credentials 文件里通常不加 `profile` 前缀，应写成：

```ini
[localstack]
aws_access_key_id = test
aws_secret_access_key = test
```

---

### 错误 3：路径写成相对当前 shell 的路径

不推荐：

```hcl
shared_config_files = ["./aws/config"]
```

更推荐：

```hcl
shared_config_files = ["${path.module}/aws/config"]
```

原因是 `path.module` 更稳定，表示当前 Terraform module 所在目录。

---

### 错误 4：忘记启动 LocalStack

如果 LocalStack 没启动，Terraform 可能会报连接失败，例如：

```text
connection refused
```

这不是 Terraform 配置语法问题，而是本地模拟服务没起来。

---

### 错误 5：误连真实 AWS

如果没有配置 endpoint，或者 skip 参数不完整，provider 可能尝试访问真实 AWS。

本节实验目标是 LocalStack，所以要确认 endpoint 指向：

```text
http://localhost:4566
```

## 13. 和生产环境的迁移关系

本节虽然用的是 LocalStack，但真实项目里有很强的迁移价值。

### 本地实验

```hcl
shared_config_files      = ["${path.module}/aws/config"]
shared_credentials_files = ["${path.module}/aws/credentials"]
profile                  = "localstack"
```

### 生产环境

可能改成：

```hcl
profile = "prod-platform"
```

或者完全不使用静态 credentials 文件，而是使用：

- GitHub Actions OIDC AssumeRole
- AWS IAM Identity Center
- EKS IRSA
- ECS Task Role
- EC2 Instance Profile

但核心思想不变：

> Terraform Provider 的认证来源必须清晰、可控、可审计。

## 14. 工程化理解

这节课不是让你背两个参数，而是让你理解一个工程原则：

> Provider 认证来源不要靠玄学，应该显式、稳定、可复现。

在个人电脑上，默认 `~/.aws/credentials` 可能没问题。  
但到了团队项目、CI/CD、Docker、教学实验、开源仓库，就会出现大量不确定性。

比如：

- A 同学本机有默认 profile，跑通了。
- B 同学没有默认 profile，失败了。
- CI runner 没有家目录配置，失败了。
- Docker 容器里没有 `~/.aws`，失败了。
- 某人默认 profile 指向真实 AWS，实验误创建资源。

这些问题都可以通过显式路径减少。
