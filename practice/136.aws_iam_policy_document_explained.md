# Terraform 考试训练 136 讲解版：`aws_iam_policy_document` 策略文档数据源

## 1. 本节定位

这一节的核心不是“真的去 AWS 控制台里配 IAM”，而是通过 Terraform AWS Provider + LocalStack，理解一个非常常见的写法：

> 用 `aws_iam_policy_document` 数据源生成标准 IAM policy JSON，然后把这个 JSON 交给 `aws_iam_policy`、角色策略、用户策略或其他需要 policy document 的资源使用。

你要抓住一句话：

> `aws_iam_policy_document` 本身不创建 AWS 资源，它只是帮 Terraform 生成一段合法的 IAM 策略 JSON。

这就是考试和实操里最容易混淆的点。

## 3. 为什么需要 `aws_iam_policy_document`

IAM policy 本质上是 JSON，例如：

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": ["s3:GetObject"],
      "Resource": ["arn:aws:s3:::example-bucket/*"]
    }
  ]
}
```

当然，你可以直接在 Terraform 里用 `jsonencode()` 或 heredoc 写 JSON。但这会带来几个问题：

1. 手写 JSON 容易漏逗号、引号、括号。
2. 策略越来越复杂后，可读性变差。
3. Terraform 无法很好地帮你结构化检查 statement、actions、resources。
4. 多个策略片段组合时，维护成本会变高。

`aws_iam_policy_document` 的价值就是：

> 用 HCL 的结构化写法描述 IAM policy，然后由 provider 帮你生成 JSON。

---

## 4. Data Source 和 Resource 的区别

这一节必须先把 Terraform 的两个概念分清楚。

### 4.1 `data` 是查询或生成信息

例如：

```hcl
data "aws_iam_policy_document" "read_s3" {
  statement {
    effect = "Allow"

    actions = [
      "s3:GetObject"
    ]

    resources = [
      "arn:aws:s3:::example-bucket/*"
    ]
  }
}
```

这里的 `data.aws_iam_policy_document.read_s3.json` 会生成一段 JSON。

它不是 IAM policy 资源。

它不会真的在 AWS / LocalStack 里创建一个 IAM policy。

它只是 Terraform 配置里的“策略文档生成器”。

### 4.2 `resource` 才会创建资源

例如：

```hcl
resource "aws_iam_policy" "read_s3" {
  name   = "read-s3-policy"
  policy = data.aws_iam_policy_document.read_s3.json
}
```

这里才是把生成出来的 JSON 用于创建 IAM policy。

所以完整关系是：

```text
aws_iam_policy_document data source
        ↓ 生成 .json
aws_iam_policy resource
        ↓ 创建 IAM policy 资源
AWS / LocalStack
```

考试里看到 `data "aws_iam_policy_document"`，第一反应应该是：

> 这是生成策略 JSON，不是创建策略资源。

## 5. 最小可理解示例

下面是一个最小的策略文档例子：

```hcl
data "aws_iam_policy_document" "example" {
  statement {
    sid    = "AllowReadObject"
    effect = "Allow"

    actions = [
      "s3:GetObject"
    ]

    resources = [
      "arn:aws:s3:::my-demo-bucket/*"
    ]
  }
}

resource "aws_iam_policy" "example" {
  name   = "demo-read-object-policy"
  policy = data.aws_iam_policy_document.example.json
}
```

重点看这一行：

```hcl
policy = data.aws_iam_policy_document.example.json
```

这里的 `.json` 是 `aws_iam_policy_document` 最常用的输出属性。

它把 HCL 里的 statement 转换成 IAM policy JSON。

## 6. `statement`、`actions`、`resources` 怎么理解

### 6.1 `statement`

`statement` 是一条权限规则。

一个 policy document 可以有多个 statement。

```hcl
data "aws_iam_policy_document" "example" {
  statement {
    sid    = "AllowListBucket"
    effect = "Allow"

    actions = [
      "s3:ListBucket"
    ]

    resources = [
      "arn:aws:s3:::my-demo-bucket"
    ]
  }

  statement {
    sid    = "AllowReadObjects"
    effect = "Allow"

    actions = [
      "s3:GetObject"
    ]

    resources = [
      "arn:aws:s3:::my-demo-bucket/*"
    ]
  }
}
```

### 6.2 `actions`

`actions` 表示允许或拒绝哪些 API 动作。

例如：

```hcl
actions = [
  "s3:GetObject",
  "s3:PutObject"
]
```

这表示对对象执行读取和写入操作。

### 6.3 `resources`

`resources` 表示这些动作作用在哪些资源 ARN 上。

S3 这里特别容易错：

```hcl
# bucket 本身
"arn:aws:s3:::my-demo-bucket"

# bucket 里面的对象
"arn:aws:s3:::my-demo-bucket/*"
```

`ListBucket` 通常对应 bucket ARN。

`GetObject` / `PutObject` 通常对应 object ARN，也就是带 `/*` 的形式。

考试或者实操中如果把这两个 ARN 混了，就会出现“语法看起来没错，但真实 IAM 权限不符合预期”的问题。

## 7. LocalStack 在本节里的作用

本节使用 LocalStack，不是为了模拟完整 AWS 安全体系，而是为了在本地完成 Terraform 的核心路径闭环。

LocalStack 的典型 endpoint 是：

```text
http://localhost:4566
```

本地测试常用假凭证：

```text
access_key = test
secret_key = test
```

这类配置的意义是：

> 让 Terraform AWS Provider 把请求发到本地 LocalStack，而不是发到真实 AWS。

你可以训练：

- provider 配置
- data source 写法
- IAM policy JSON 生成
- resource 引用 data source 输出
- `terraform plan`
- `terraform apply`
- `terraform destroy`
- state 变化观察

但它不能完全替代真实 AWS 的 IAM 权限评估。

---

## 8. LocalStack Provider 配置示例

一个典型的本地练习 provider 可能长这样：

```hcl
provider "aws" {
  region                      = "us-east-1"
  access_key                  = "test"
  secret_key                  = "test"
  s3_use_path_style           = true
  skip_credentials_validation = true
  skip_metadata_api_check     = true
  skip_requesting_account_id  = true

  endpoints {
    iam = "http://localhost:4566"
    s3  = "http://localhost:4566"
  }
}
```

注意：不同 AWS Provider 版本对 LocalStack 相关参数支持可能略有差异。考试不太会让你背这个完整配置，核心是知道：

> 本地实验通过 provider endpoint 指向 LocalStack，而不是访问真实 AWS。

## 9. 推荐实验流程

本节的实践路径可以按下面顺序理解：

```bash
terraform fmt
terraform init
terraform validate
terraform plan
terraform apply
terraform destroy
```

每一步的意义：

| 命令 | 作用 |
|---|---|
| `terraform fmt` | 格式化 HCL 文件 |
| `terraform init` | 初始化 provider 和工作目录 |
| `terraform validate` | 检查 Terraform 配置语法和结构 |
| `terraform plan` | 预览将要创建的资源和 data source 计算结果 |
| `terraform apply` | 创建 IAM policy 等资源 |
| `terraform destroy` | 销毁实验创建的资源 |

考试里不会只考命令顺序，更常考的是：

> 哪一步负责初始化？哪一步负责验证？哪一步真正创建资源？data source 到底有没有创建资源？

## 10. `aws_iam_policy_document` vs `jsonencode`

两种方式都能生成 IAM policy JSON。

### 10.1 使用 `jsonencode`

```hcl
resource "aws_iam_policy" "example" {
  name = "example-policy"

  policy = jsonencode({
    Version = "2012-10-17"
    Statement = [
      {
        Effect = "Allow"
        Action = [
          "s3:GetObject"
        ]
        Resource = [
          "arn:aws:s3:::my-demo-bucket/*"
        ]
      }
    ]
  })
}
```

### 10.2 使用 `aws_iam_policy_document`

```hcl
data "aws_iam_policy_document" "example" {
  statement {
    effect = "Allow"

    actions = [
      "s3:GetObject"
    ]

    resources = [
      "arn:aws:s3:::my-demo-bucket/*"
    ]
  }
}

resource "aws_iam_policy" "example" {
  name   = "example-policy"
  policy = data.aws_iam_policy_document.example.json
}
```

### 10.3 怎么选

| 写法 | 特点 |
|---|---|
| `jsonencode` | 通用、直接、适合简单 JSON |
| `aws_iam_policy_document` | 更结构化、更贴近 IAM policy 语义、适合复杂策略 |
| heredoc JSON | 可用，但容易出格式错误，不太推荐作为主要方式 |

实际工作中，两种都能见到。考试里更希望你知道：

> `aws_iam_policy_document` 是 AWS Provider 提供的 data source，用于生成 IAM policy JSON。

## 11. 常见易错点

### 易错点 1：以为 data source 会创建 IAM policy

错误理解：

> 我写了 `data "aws_iam_policy_document"`，所以 AWS 里就有了一个 policy。

正确理解：

> data source 只生成 JSON。要创建 policy，还需要 `aws_iam_policy` 这类 resource。

---

### 易错点 2：忘记 `.json`

错误写法：

```hcl
policy = data.aws_iam_policy_document.example
```

正确写法：

```hcl
policy = data.aws_iam_policy_document.example.json
```

`aws_iam_policy` 的 `policy` 参数需要的是 JSON 字符串，而不是整个 data source 对象。

---

### 易错点 3：把 LocalStack 当成真实 AWS

LocalStack 很适合训练 Terraform 闭环，但不要以为它能 100% 复刻真实 AWS 行为。

尤其是 IAM 权限评估、跨服务权限边界、真实网络连通性、真实安全审计，这些都不能完全依赖 LocalStack 判断。

---

### 易错点 4：S3 ARN 写错

常见区别：

```hcl
# bucket ARN
arn:aws:s3:::my-demo-bucket

# object ARN
arn:aws:s3:::my-demo-bucket/*
```

`ListBucket` 和 `GetObject` 经常需要不同级别的 ARN。

---

### 易错点 5：把 `actions` 和 `resources` 写成单个字符串

在 `aws_iam_policy_document` 中，`actions` 和 `resources` 通常是 list：

```hcl
actions = [
  "s3:GetObject"
]

resources = [
  "arn:aws:s3:::my-demo-bucket/*"
]
```

不要写成随意字符串结构。

## 12. 考试判断框架

看到题目提到 `aws_iam_policy_document`，按下面三步判断：

### 第一步：它是不是 data source？

如果是：

```hcl
data "aws_iam_policy_document" "xxx" {}
```

那它就是生成或组合策略文档，不是创建 AWS 资源。

### 第二步：它输出什么？

最常见输出：

```hcl
data.aws_iam_policy_document.xxx.json
```

也就是 JSON 字符串。

### 第三步：谁消费这个 JSON？

常见消费者包括：

```hcl
resource "aws_iam_policy" ...
resource "aws_iam_role_policy" ...
resource "aws_iam_role" assume_role_policy = ...
```

所以题目问“策略在哪里被创建”，不要答 data source，要看哪个 resource 使用了它的 `.json`。

## 13. 典型场景题讲解

### 场景 1：要求用 Terraform 生成 IAM policy JSON

推荐方式：

```hcl
data "aws_iam_policy_document" "policy" {
  statement {
    effect = "Allow"

    actions = [
      "s3:GetObject"
    ]

    resources = [
      "arn:aws:s3:::example-bucket/*"
    ]
  }
}
```

解释：

> 这是结构化生成 IAM policy JSON，比手写 JSON 更清晰。

---

### 场景 2：要求创建一个 IAM managed policy

需要加 resource：

```hcl
resource "aws_iam_policy" "policy" {
  name   = "example-policy"
  policy = data.aws_iam_policy_document.policy.json
}
```

解释：

> data source 负责生成 JSON，resource 负责创建 AWS IAM policy。

---

### 场景 3：要求本地练习但不能访问真实 AWS

使用 LocalStack：

```hcl
provider "aws" {
  region     = "us-east-1"
  access_key = "test"
  secret_key = "test"

  endpoints {
    iam = "http://localhost:4566"
  }

  skip_credentials_validation = true
  skip_metadata_api_check     = true
  skip_requesting_account_id  = true
}
```

解释：

> endpoint 指向本地 LocalStack，避免访问真实 AWS。

## 14. 选择题练习

### 题 1

`aws_iam_policy_document` 的主要作用是什么？

A. 创建真实 EC2 实例  
B. 创建 Terraform workspace  
C. 生成 IAM policy JSON  
D. 删除 AWS 账号

答案：C

解析：  
`aws_iam_policy_document` 是 data source，用于生成策略 JSON，不是创建 EC2 或 workspace。

下面哪一行最常用于把 policy document 传给 `aws_iam_policy`？

A. `policy = aws_iam_policy_document.example`

C. `policy = terraform.workspace`  
D. `policy = provider.aws.json`

答案：B

解析：  
`aws_iam_policy_document` 常用 `.json` 属性输出 JSON 字符串。

### 题 3

使用 LocalStack 练习 AWS Provider 时，固定的假凭证 `test/test` 主要用于什么？

A. 登录真实 AWS root 账号

C. 获取生产环境管理员权限

如果只写了下面配置，AWS / LocalStack 中是否已经创建 IAM policy？

## 15. 判断题练习

1. `aws_iam_policy_document` 是 data source，不是 resource。  
   答案：对。

2. `data.aws_iam_policy_document.example.json` 可以作为 `aws_iam_policy.policy` 的输入。  
   答案：对。

3. LocalStack 可以完全替代真实 AWS 的 IAM 权限评估。  
   答案：错。LocalStack 适合本地练习 Terraform 闭环，但不能完全替代真实 AWS 安全行为。

4. `statement` 中的 `actions` 用来声明允许或拒绝哪些 API 操作。  
   答案：对。

5. 只要写了 `aws_iam_policy_document`，就一定会在云上创建 IAM managed policy。  
   答案：错。data source 只生成文档，resource 才创建资源。

## 17. 复习重点

- `aws_iam_policy_document` 是 AWS Provider 的 data source。
- 它主要用于生成 IAM policy JSON。
- 常用输出是 `.json`。
- 真正创建 IAM policy 的是 `aws_iam_policy` 等 resource。
- `statement` 表示权限规则。
- `actions` 表示 API 动作。
- `resources` 表示作用资源 ARN。
- LocalStack 可以完成 Terraform 本地练习闭环，但不能完全替代真实 AWS IAM 行为。
- 本节实验边界是不访问真实 AWS，不写真实 key，不使用 root 凭证。
- 考试遇到这类题，优先区分 data source 和 resource。
