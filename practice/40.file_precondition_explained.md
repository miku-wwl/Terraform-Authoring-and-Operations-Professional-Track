# Terraform 实操训练 40 讲解：文件存在性 Precondition

## 1. 这一节到底在学什么？

Lab 40 学的是 Terraform 里一个很贴近生产的检查场景：**在资源创建前，确认某个依赖文件已经存在并且不是空文件**。

这一节继续使用 `precondition`，但不再只是检查变量值，而是检查本地文件状态。

核心函数是：

```hcl
fileexists()
```

它可以判断指定路径的文件是否存在。

本实验的核心问题是：

> 如果应用配置依赖数据库凭据文件，那么在凭据文件缺失或为空时，Terraform 不应该继续生成应用配置。

这就是典型的部署前置条件检查。

---

## 2. 为什么要检查文件是否存在？

很多 Terraform 项目并不是孤立运行的。它经常依赖一些外部文件，例如：

- 数据库连接信息
- 私有证书
- 初始化脚本
- 配置模板
- 密钥引用文件
- 外部系统生成的参数文件
- 环境专属配置文件

如果这些文件不存在，Terraform 继续跑下去，可能会出现两种坏结果：

1. Terraform 直接报错，但错误信息不友好。
2. Terraform 生成了错误配置，后续服务启动才失败。

`precondition` 的价值就在这里：

> 在资源创建前主动拦截，并给出清楚的错误信息。

---

## 3. `fileexists()` 是什么？

`fileexists(path)` 用来判断某个路径是否存在文件。

例如：

```hcl
fileexists("${path.module}/input/db.txt")
```

如果文件存在，返回：

```hcl
true
```

如果文件不存在，返回：

```hcl
false
```

它特别适合放进 `precondition` 里：

```hcl
lifecycle {
  precondition {
    condition     = fileexists("${path.module}/input/db.txt")
    error_message = "input/db.txt must exist before generating app config."
  }
}
```

这段逻辑的意思是：

> 只有当 `input/db.txt` 已经存在时，才允许继续创建这个资源。

---

## 4. 只检查存在还不够：还要检查非空

一个文件存在，不代表它有用。

比如：

```text
input/db.txt
```

文件确实存在，但内容是空的。  
这对应用配置来说通常仍然是错误状态。

所以本实验强调的是两个条件：

1. `input/db.txt` 必须存在。
2. `input/db.txt` 必须非空。

可以用类似这样的思路：

```hcl
fileexists("${path.module}/input/db.txt") &&
length(trimspace(file("${path.module}/input/db.txt"))) > 0
```

这里拆开看：

```hcl
fileexists("${path.module}/input/db.txt")
```

检查文件是否存在。

```hcl
file("${path.module}/input/db.txt")
```

读取文件内容。

```hcl
trimspace(...)
```

去掉前后空白。

```hcl
length(...) > 0
```

确认剩余内容长度大于 0。

这就比单纯检查文件存在更稳。

---

## 5. 本实验的关键坑：Precondition 不能依赖同一次 apply 创建的文件

Lab 40 的验证报告里提到一个非常关键的修正：

> 首次实现把 `db.txt` 设计成同一次 apply 创建，precondition 因创建前检查而失败；已修正为实验前置文件并复测通过。

这个点非常重要。

`precondition` 是在资源创建前执行的。  
所以如果你写了这样的逻辑：

1. Terraform 资源 A 创建 `db.txt`
2. Terraform 资源 B 的 precondition 检查 `db.txt` 是否存在
3. 同一次 apply 里希望 B 等 A 创建完再检查

这种设计很容易失败，因为 `precondition` 的语义不是“等依赖创建完以后再检查文件”，而是“资源创建前先判断条件是否满足”。

所以本实验最后把 `input/db.txt` 改成**实验前置文件**，也就是 Terraform 执行前它就应该存在。

一句话记住：

> precondition 检查的是前置条件，不适合检查同一次 apply 才会生成的东西。

---

## 6. 本实验的场景

背景：

> 应用配置必须依赖已有数据库凭据文件。

问题：

> 如果凭据文件缺失时还继续生成应用配置，就会生成无效配置。

任务：

> 检查 `input/db.txt` 存在且非空。

预期结果：

> 前置文件存在时，Terraform 创建 `output/app.txt`。

这就是一个非常典型的平台工程场景：  
Terraform 不只是创建文件，而是确保创建前的依赖条件是合理的。

---

## 7. 可能的 Terraform 写法

示意写法如下：

```hcl
resource "local_file" "app_config" {
  filename = "${path.module}/output/app.txt"

  content = <<-EOT
  app = demo
  db  = ${trimspace(file("${path.module}/input/db.txt"))}
  EOT

  lifecycle {
    precondition {
      condition = (
        fileexists("${path.module}/input/db.txt") &&
        length(trimspace(file("${path.module}/input/db.txt"))) > 0
      )

      error_message = "input/db.txt must exist and must not be empty before generating app config."
    }
  }
}
```

这段配置体现了三个点：

1. `fileexists()` 检查文件存在。
2. `file()` 读取文件内容。
3. `trimspace()` + `length()` 检查文件不是空白内容。

---

## 8. 为什么错误应该提前失败？

如果没有 precondition，可能会发生这种情况：

- `file()` 读取不到文件，Terraform 报一个偏底层的错误。
- 生成的配置缺少数据库信息。
- 应用部署后才发现连接不上数据库。
- CI/CD 到后面步骤才失败，浪费时间。
- 排障人员需要倒查是哪个输入文件缺失。

有了 precondition，错误可以变成：

```text
input/db.txt must exist and must not be empty before generating app config.
```

这就清楚多了。  
用户一看就知道该修什么。

## 9. 标准运行流程

进入实验目录：

```sh
cd /workspace/practice/labs/40
```

初始化：

```sh
terraform init -input=false
```

格式检查：

```sh
terraform fmt -check
```

配置校验：

```sh
terraform validate
```

运行测试：

```sh
terraform test
```

生成 plan：

```sh
terraform plan -input=false -no-color -out=tfplan
```

执行 apply：

```sh
terraform apply -auto-approve tfplan
```

查看输出：

```sh
terraform output
```

清理：

```sh
terraform destroy -auto-approve
```

清理本地文件：

```sh
rm -f tfplan
rm -rf output .terraform .terraform.lock.hcl
```

## 11. 常见坑

### 坑 1：precondition 检查同一次 apply 创建的文件

这是本实验最重要的坑。

错误思路：

> Terraform 先创建 `db.txt`，然后 precondition 检查它。

问题是 precondition 的定位就是“创建前检查”。  
如果文件要在同一次 apply 才出现，就不适合作为 precondition 的检查对象。

正确思路：

> `input/db.txt` 应该是 Terraform 执行前就存在的前置文件。

---

### 坑 2：只用 `fileexists()`，但不检查内容

文件存在但为空，通常仍然是坏输入。

更稳的是：

```hcl
fileexists(path) && length(trimspace(file(path))) > 0
```

---

### 坑 3：路径写得不稳定

不要依赖当前 shell 目录乱猜路径。  
Terraform 模块里更推荐使用：

```hcl
${path.module}/input/db.txt
```

这样路径相对于当前模块，更稳定。

---

### 坑 4：错误信息太模糊

不要写：

```hcl
error_message = "Invalid file."
```

应该写：

```hcl
error_message = "input/db.txt must exist and must not be empty before generating app config."
```

错误信息要直接告诉用户修复动作。

---

## 12. `fileexists()`、`file()`、`trimspace()` 的组合

这三个函数经常一起出现：

| 函数 | 作用 |
|---|---|
| `fileexists(path)` | 判断文件是否存在 |
| `file(path)` | 读取文件内容 |
| `trimspace(text)` | 去掉前后空白字符 |
| `length(text)` | 判断内容长度 |

组合起来就是：

```hcl
fileexists(path) && length(trimspace(file(path))) > 0
```

这个表达式可以读成：

> 文件存在，并且去掉空白后仍然有内容。

这是非常实用的 Terraform 防御性写法。

## 14. 生产环境怎么用？

真实项目中，这种检查可以用于：

- TLS 证书文件是否存在
- kubeconfig 是否存在
- 数据库连接配置是否存在
- 应用模板变量文件是否存在
- 环境配置 JSON/YAML 是否存在
- CI 注入的密钥引用文件是否存在

例如：

```hcl
precondition {
  condition     = fileexists("${path.module}/config/${var.environment}.json")
  error_message = "Environment config file must exist before deployment."
}
```

这能把“缺配置文件”的问题提前挡住，而不是等应用部署失败。

---

## 15. 什么时候不要用这种方式？

虽然 `fileexists()` 很方便，但它不适合所有场景。

不建议用它处理：

- 真正的 secret 原文
- 会频繁变化的大文件
- 同一次 Terraform apply 才生成的文件
- 应该由远端 secret manager 管理的内容
- 需要跨团队共享的敏感凭据

生产里，数据库密码这类东西更推荐放在：

- AWS Secrets Manager
- Azure Key Vault
- HashiCorp Vault
- Kubernetes Secret 管理流程
- CI/CD secret store

本实验用 `input/db.txt` 是为了学习 precondition 机制，不是鼓励生产把密码明文放仓库里。

## 16. 面试/考试记忆版

可以这样记：

- `fileexists(path)`：判断文件是否存在。
- `file(path)`：读取文件内容。
- `trimspace()`：去掉空白，适合判断非空。
- `precondition`：资源创建前检查条件。
- precondition 不能依赖同一次 apply 才创建的文件。
- 文件存在不代表有效，最好同时检查非空。
- 路径建议用 `${path.module}`，不要靠当前目录猜。

一句话总结：

> `fileexists()` + `precondition` 适合检查 Terraform 执行前就必须存在的依赖文件；它是前置检查，不是同一次 apply 的依赖编排工具。

## 17. 本 Lab 的价值

Lab 40 的价值在于，它把 precondition 从“检查变量”推进到“检查执行环境”。

这对真实 Terraform 工程很重要。因为生产流水线失败，很多时候不是 HCL 写错，而是前置文件、配置、凭据、模板、环境变量缺失。

成熟的模块应该主动检查这些前置条件，而不是等下游步骤爆炸。

这就是 IaC 工程化的思路：

> 不要假设环境是对的；把关键假设写成可执行检查。
