# Terraform 实操训练 18：`TF_IN_AUTOMATION` 与自动化日志讲解

## 1. 这一讲要解决什么问题？

这一讲不是在教你创建复杂资源，而是在教你一个很容易被忽略、但在 CI/CD 里很实用的 Terraform 环境变量：

```sh
TF_IN_AUTOMATION=true
```

它的核心作用是：

> 告诉 Terraform：现在不是人在终端里一步一步操作，而是在自动化流水线里执行，请减少面向人工交互的提示，让日志更适合 CI/CD 阅读和归档。

也就是说，这个实验重点不是“功能变化”，而是“日志行为变化”。

---

## 2. 为什么 CI 里要设置 `TF_IN_AUTOMATION`？

Terraform 默认假设你是在本地终端里手动执行命令。

所以它会输出一些偏“教学式”或“下一步提醒式”的内容，例如：

- 初始化完成后提示你可以运行 `terraform plan`
- 输出一些适合人工继续操作的说明
- 日志更偏向交互式终端阅读

这些内容对新人本地练习有帮助，但在 CI/CD 里会变成噪音。

在真实团队里，Terraform 日志通常会被：

- 存进 CI 系统
- 用于审计
- 用于排障
- 被 PR reviewer 查看
- 被平台团队长期归档

所以日志越清楚越好。  
无关提示越少，越容易定位真正的问题。

---

## 3. `TF_IN_AUTOMATION=true` 到底改变了什么？

设置之后，Terraform 会知道自己运行在自动化环境中。

它不会改变 Terraform 的核心执行逻辑，例如：

- 不会改变 `plan` 的计算结果
- 不会改变 `apply` 的资源创建逻辑
- 不会改变 state 行为
- 不会让 Terraform 自动跳过确认

它主要影响的是 CLI 输出风格。

简单说：

```text
不设置 TF_IN_AUTOMATION：
Terraform 更像是在对一个人说话。

设置 TF_IN_AUTOMATION=true：
Terraform 更像是在给流水线输出日志。
```

所以它是一个“日志整洁度”和“自动化语义”的设置，不是一个权限设置，也不是安全开关。

---

## 4. 本实验的核心场景

### 背景

平台团队已经把 Terraform 放进 CI/CD 了，但是日志里有很多 Terraform 给人工操作员看的提示。

### 问题

这些提示在 CI 中没什么价值，反而让排障时更难读。

### 任务

设置：

```sh
export TF_IN_AUTOMATION=true
```

然后跑完整的 Terraform 流程：

```sh
terraform init
terraform fmt
terraform validate
terraform test
terraform plan
terraform apply
terraform output
terraform destroy
```

### 预期结果

所有命令正常通过，并且 `terraform init` 的输出会比默认模式更简洁。

## 6. 推荐执行命令：Linux / macOS / Docker Shell

进入实验目录：

```sh
cd /workspace/practice/labs/18
```

设置自动化环境变量：

```sh
export TF_IN_AUTOMATION=true
```

初始化 Terraform：

```sh
terraform init -input=false
```

检查格式：

```sh
terraform fmt -check
```

校验配置：

```sh
terraform validate
```

运行 Terraform test：

```sh
terraform test
```

生成保存版 plan：

```sh
terraform plan -input=false -no-color -out=tfplan
```

执行 plan：

```sh
terraform apply -auto-approve tfplan
```

查看输出：

```sh
terraform output
```

销毁资源：

```sh
terraform destroy -auto-approve
```

## 7. PowerShell 写法

如果你在 Windows PowerShell 里执行，不要用 `export`，而是用：

```powershell
$env:TF_IN_AUTOMATION = "true"
```

然后继续执行 Terraform 命令：

```powershell
terraform init -input=false
terraform fmt -check
terraform validate
terraform test
terraform plan -input=false -no-color -out=tfplan
terraform apply -auto-approve tfplan
terraform output
terraform destroy -auto-approve
```

清理环境变量：

```powershell
Remove-Item Env:\TF_IN_AUTOMATION
```

## 8. 这里为什么还要配合其他自动化参数？

`TF_IN_AUTOMATION=true` 不是万能的。  
在 CI/CD 里，它通常要和其他参数一起使用。

### `-input=false`

```sh
terraform init -input=false
terraform plan -input=false
```

含义是：不要在命令行里等待用户输入。

CI 环境没人帮你敲 yes/no，所以这个参数很重要。

---

### `-no-color`

```sh
terraform plan -no-color
```

含义是：关闭彩色输出。

CI 日志不一定能正确显示终端颜色码，关闭颜色可以让日志更干净。

---

### `-out=tfplan`

```sh
terraform plan -out=tfplan
```

含义是：保存 plan 文件。

这可以保证审批看到的 plan 和最终 apply 的 plan 是同一个东西。

---

### `terraform apply -auto-approve tfplan`

```sh
terraform apply -auto-approve tfplan
```

这里有两个点：

1. 使用保存下来的 `tfplan`
2. 不再要求人工确认

在生产环境里，这一步通常放在审批 gate 之后。

---

## 9. 推荐的 CI 写法

在 GitHub Actions 或其他 CI 中，可以在 job 级别设置：

```yaml
env:
  TF_IN_AUTOMATION: "true"
```

然后 Terraform 命令尽量写成非交互模式：

```yaml
- name: Terraform Init
  run: terraform init -input=false

- name: Terraform Format
  run: terraform fmt -check

- name: Terraform Validate
  run: terraform validate

- name: Terraform Plan
  run: terraform plan -input=false -no-color -out=tfplan
```

如果是生产 apply，建议不要直接裸 apply，而是走审批后的保存 plan：

```yaml
- name: Terraform Apply
  run: terraform apply -auto-approve tfplan
```

## 10. 学习重点：这不是“让 Terraform 自动化”的唯一开关

很多人容易误解：

> 设置了 `TF_IN_AUTOMATION=true`，是不是 Terraform 就变成 CI 模式了？

不完全是。

它只是告诉 Terraform：  
“现在运行在自动化上下文里，请调整输出行为。”

真正让 Terraform 适合 CI/CD 的，是一整套组合：

```text
TF_IN_AUTOMATION=true
-input=false
-no-color
plan -out=tfplan
apply tfplan
CI approval gate
remote backend
state lock
secret management
policy scan
```

本实验只覆盖其中的一个小点：自动化日志。

但这个小点很实用，因为它体现的是工程习惯。

## 11. 常见问题

### 问题 1：我设置了 `TF_IN_AUTOMATION=true`，为什么日志还是很多？

因为它只减少部分面向人工的提示，不会隐藏 Terraform 的核心日志。

如果你想进一步减少日志噪音，可以继续配合：

```sh
-no-color
-compact-warnings
```

但不要为了“安静”而隐藏真正的错误。

---

### 问题 2：它会不会自动帮我 approve？

不会。

`TF_IN_AUTOMATION=true` 不等于 `-auto-approve`。

如果你执行裸的：

```sh
terraform apply
```

Terraform 仍然可能要求确认。

---

### 问题 3：生产 CI 里一定要设置吗？

建议设置。

原因不是它能提高安全性，而是它能让日志更符合自动化环境，减少无效提示。

---

### 问题 4：本地练习要不要设置？

本地练习可以不设置。

因为本地学习时，Terraform 的下一步提示反而有帮助。

但是如果你在模拟 CI/CD 流程，建议设置。

## 13. 清理命令

销毁 Terraform 创建的资源：

```sh
terraform destroy -auto-approve
```

删除保存的 plan 文件：

```sh
rm -f tfplan
```

取消环境变量：

```sh
unset TF_IN_AUTOMATION
```

PowerShell：

```powershell
Remove-Item Env:\TF_IN_AUTOMATION
```

## 14. 这一讲应该怎么记？

可以这样记：

```text
TF_IN_AUTOMATION=true
不是让 Terraform 变“自动执行”，
而是让 Terraform 知道：
“别像教人一样啰嗦了，现在你在 CI 里，日志干净点。”
```

一句话总结：

> `TF_IN_AUTOMATION=true` 是 Terraform CI/CD 工程化里的日志语义开关，配合 `-input=false`、`-no-color`、保存 plan 和审批 gate，才能组成真正可靠的自动化流程。
