# Terraform 实操训练 28：Indented heredoc 的前导空白规则（讲解版）

## 1. 这一节到底在学什么？

这一节学习 Terraform 里的 **indented heredoc**，也就是 `<<-EOT` 这种多行字符串写法。

它和普通 heredoc `<<EOT` 最大的区别是：

- `<<EOT`：原样保留文本里的前导空白。
- `<<-EOT`：会自动移除所有行共有的那一部分前导空白。

这看起来是一个很小的语法点，但在真实工程里很重要。因为 Terraform 经常会生成：

- YAML 配置片段；
- shell 脚本；
- cloud-init；
- JSON / policy 文本；
- Kubernetes manifest；
- CI/CD 配置文件。

这些内容很多都对缩进敏感。尤其 YAML，缩进错一点，配置就可能直接废掉。

## 2. 核心概念：什么叫“共同前导空白”？

Indented heredoc 的规则可以简单理解成一句话：

> Terraform 会找出所有非空行共有的最短缩进，然后从每一行前面都删掉这么多空白。

举个例子，假设 heredoc 内容在 Terraform 代码里长这样：

```hcl
content = <<-EOT
  app:
    name: payments
    tier: backend
EOT
```

这里三行实际内容分别是：

```text
  app:
    name: payments
    tier: backend
```

三行共同拥有的最短前导空白是 **两个空格**。

所以 Terraform 会从每一行前面删除两个空格，最终生成：

```yaml
app:
  name: payments
  tier: backend
```

注意：`name` 和 `tier` 前面的两个空格没有被删光，因为它们不是“外层共同缩进”，而是 YAML 自己需要的相对缩进。

这就是本 Lab 的核心。

## 3. 为什么这个规则很重要？

如果你不用 indented heredoc，Terraform 代码为了可读性通常会缩进，但生成出来的文件也会带上这层缩进。

例如普通 heredoc：

```hcl
content = <<EOT
  app:
    name: payments
EOT
```

可能生成：

```yaml
  app:
    name: payments
```

第一行前面多了两个空格。对某些格式来说，这可能只是难看；但对 YAML、shell、配置模板来说，这可能是直接出错。

而 `<<-EOT` 的好处是：

```hcl
content = <<-EOT
  app:
    name: payments
EOT
```

Terraform 代码本身可以保持缩进，生成文件又能保持干净：

```yaml
app:
  name: payments
```

一句话：

> `<<-EOT` 让 Terraform 代码好看，也让生成文件不乱缩进。

## 4. 本实验的场景

本 Lab 的背景是：平台团队需要用 Terraform 生成一段 YAML 配置。

问题是：

- Terraform 代码中 heredoc 为了可读性会缩进；
- 但生成的 YAML 第一行不能带外层缩进；
- 同时，YAML 内部层级缩进还必须保留。

所以实验任务是：

1. 使用 `<<-EOT` 写一段 YAML；
2. 生成输出文件；
3. 验证第一行 `app:` 没有外层缩进；
4. 验证第二行 `name: payments` 仍然保留两个空格缩进。

## 7. 每条命令在干嘛？

### `terraform init -input=false`

初始化 Terraform 工作目录，下载 provider，准备 `.terraform` 目录。

`-input=false` 表示不要在命令行里交互提问，这在 CI/CD 中是好习惯。

### `terraform fmt -check`

检查 Terraform 代码格式是否符合标准。

这一步不会自动改文件，只会检查。如果格式不对，它会失败。

### `terraform validate`

检查配置语法和基本结构是否正确。

比如 heredoc 结束标记写错、字符串语法错误、资源参数写错，都可能在这一步暴露。

### `terraform test`

执行 `.tftest.hcl` 测试文件。

本 Lab 的测试重点是：

- 第一行是不是 `app:`；
- 第二行是不是保留了两个空格的 `name: payments`；
- `<<-EOT` 是否正确移除了共同缩进。

### `terraform plan -out=tfplan`

生成执行计划，并保存成二进制 plan 文件。

保存 plan 的好处是：后面 apply 的内容和这次审批/检查的内容一致。

### `terraform apply -auto-approve tfplan`

执行刚才保存的 plan。

这里不是重新计算计划，而是执行 `tfplan` 里已经保存好的动作。

### `terraform destroy -auto-approve`

清理本实验创建的本地资源和文件。

## 8. 关键观察点

这个 Lab 不只是为了“生成一个文件”，而是为了观察生成文件中的缩进。

你重点看这两行：

```yaml
app:
  name: payments
```

正确结果应该是：

- `app:` 前面没有空格；
- `name: payments` 前面有两个空格；
- 第二行的两个空格是 YAML 层级缩进，不是 Terraform 代码外层缩进。

这就证明：

> indented heredoc 移除的是共同外层缩进，不会破坏行与行之间的相对缩进。

## 9. 常见错误

### 错误 1：以为 `<<-EOT` 会删除所有缩进

不会。

它只删除“所有行共同拥有的最短前导空白”。

如果某一行本来就比其他行多缩进，多出来的部分会保留。

这正是 YAML 能保持层级的原因。

### 错误 2：第一行仍然有空格

如果生成文件第一行仍然有空格，通常说明不是所有内容行都有相同的共同缩进。

检查点：

- 是否有某一行缩进比其他行更少；
- 是否有空行影响阅读；
- heredoc 起始和结束标记是否写对；
- 内容是否混用了 tab 和空格。

### 错误 3：YAML 层级丢失

如果生成结果变成：

```yaml
app:
name: payments
```

那说明你把所有行写成了相同缩进，Terraform 删除共同缩进后，YAML 的相对层级就没了。

正确写法应该让子级比父级多缩进：

```hcl
content = <<-EOT
  app:
    name: payments
EOT
```

而不是：

```hcl
content = <<-EOT
  app:
  name: payments
EOT
```

## 10. 生产环境建议

在真实项目里，heredoc 很常见，但不要滥用。

建议如下：

1. 简单多行文本可以直接用 heredoc。
2. 对缩进敏感的 YAML，优先用 `<<-EOT`。
3. 复杂 YAML/JSON 配置，考虑用 `templatefile()` 或 `yamlencode()` / `jsonencode()`。
4. 生成后最好用目标工具校验，比如 `yamllint`、`kubectl apply --dry-run`、`jq` 等。
5. 不要只相信 Terraform validate，因为它只能验证 Terraform 配置，不一定能验证生成文件的业务语义。

## 11. 和上一节 Lab 27 的关系

Lab 27 讲的是：

- 普通 heredoc `<<EOT`；
- indented heredoc `<<-EOT`；
- 两者整体差异。

Lab 28 更进一步，专门讲 `<<-EOT` 的细节：

> 它不是简单粗暴地删除所有缩进，而是删除共同缩进，并保留相对缩进。

所以 Lab 27 是“知道有区别”，Lab 28 是“知道它到底怎么删缩进”。

## 12. 一句话总结

`<<-EOT` 的价值不是“把缩进全部清空”，而是：

> 删除 Terraform 代码里的外层共同缩进，同时保留生成内容自己的结构缩进。

对 YAML、脚本、配置文件来说，这个细节很关键。别小看它，很多奇怪的配置错误，就是缩进这种小妖怪搞出来的。
