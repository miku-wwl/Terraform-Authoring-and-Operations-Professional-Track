# Terraform 实操训练 115：Terraform 调试概览（讲解版）

## 1. 这一节到底在学什么？

这一节的主题是 **Terraform 调试概览**。

它不是让你背某一个单独命令，而是让你建立一套排查 Terraform 问题的思路：

> Terraform 出问题时，不要上来就乱改代码。  
> 先判断问题发生在哪一层：语法、初始化、Provider、状态、计划、执行、测试，还是云平台 API。

Terraform 的调试，本质上就是把一个“看起来很玄学”的失败，拆成几个可验证的阶段。

---

## 2. Terraform 常见问题发生在哪些阶段？

一个标准 Terraform 工作流通常是：

```sh
terraform init
terraform fmt
terraform validate
terraform test
terraform plan
terraform apply
terraform output
terraform destroy
```

每一步都有自己的排障含义。

| 阶段 | 主要检查什么 | 常见问题 |
|---|---|---|
| `terraform init` | Provider、backend、module 下载 | provider 版本冲突、网络问题、模块路径错误 |
| `terraform fmt` | HCL 格式 | 格式不规范 |
| `terraform validate` | 静态语法与引用关系 | 变量、资源、local、output 引用错误 |
| `terraform test` | 测试断言 | 输出不符合预期、断言条件写错 |
| `terraform plan` | 生成变更计划 | 状态漂移、未知值、依赖关系问题 |
| `terraform apply` | 真正执行变更 | 云 API 报错、权限不足、配额不足 |
| `terraform output` | 查看结果 | output 名称错误、敏感值显示受限 |
| `terraform destroy` | 清理资源 | 依赖阻塞、资源保护、远端对象已变化 |

你排查的时候，不要把所有问题都混在一起看。  
哪一步失败，就优先看哪一层。

---

## 3. 最重要的调试习惯：先定位失败阶段

很多初学 Terraform 的人遇到错误会直接开始改 `main.tf`，这很危险。

更靠谱的做法是：

```sh
terraform init -input=false
terraform fmt -check
terraform validate
terraform test
terraform plan -input=false -no-color -out=tfplan
```

如果 `validate` 都没过，就不要急着看 `plan`。

如果 `plan` 都还没生成，就不要猜 `apply` 会怎样。

如果 `test` 失败，先看测试断言，不要先怀疑 Provider。

---

## 4. `terraform init` 调试：先看依赖是不是装好了

`terraform init` 主要做三类事情：

1. 初始化 backend
2. 下载 provider
3. 下载 module

常见错误包括：

```text
Failed to query available provider packages
```

可能原因：

- provider source 写错
- provider 版本约束冲突
- 网络无法访问 registry
- lock file 和当前平台不匹配

还有一种常见错误：

```text
Unreadable module directory
```

一般是模块路径写错，比如：

```hcl
module "example" {
  source = "../modules/example"
}
```

但实际目录不存在。

### 排查方式

```sh
terraform init -input=false
```

如果怀疑锁文件或 provider 缓存问题，可以清理后重新初始化：

```sh
rm -rf .terraform .terraform.lock.hcl
terraform init -input=false
```

注意：生产项目里不要随便删 lock file。  
实验环境可以删，生产环境要理解后果。

---

## 5. `terraform fmt` 调试：格式问题不要靠肉眼看

`terraform fmt -check` 用来检查 HCL 格式是否规范。

如果失败，通常不是逻辑错误，只是格式不符合 Terraform 标准。

实验中可以直接运行：

```sh
terraform fmt -recursive
```

然后再检查：

```sh
terraform fmt -check
```

这一步很适合放到 CI 里。  
它的作用类似代码里的 `go fmt` 或 `prettier`。

---

## 6. `terraform validate` 调试：静态语法和引用检查

`terraform validate` 不会真的创建资源，它主要检查配置是否合法。

常见问题：

### 1. 引用了不存在的变量

```hcl
name = var.project_name
```

但没有定义：

```hcl
variable "project_name" {}
```

### 2. 引用了不存在的 local

```hcl
name = local.prefix
```

但没有：

```hcl
locals {
  prefix = "demo"
}
```

### 3. output 指向不存在的资源

```hcl
output "id" {
  value = null_resource.demo.id
}
```

但资源名不是 `demo`。

### 排查方式

```sh
terraform validate
```

然后根据错误提示定位文件和行号。

Terraform 的报错通常会告诉你：

- 哪个文件
- 哪一行
- 哪个表达式
- 哪个引用不存在

不要只看最后一行。  
Terraform 错误信息的中间部分，经常才是重点。

---

## 7. `terraform test` 调试：先看断言，不要乱怀疑代码

这一节的实验要求 `terraform test` 返回：

```text
1 passed, 0 failed
```

Terraform test 常见失败原因：

- output 名称写错
- assert 条件不成立
- 测试文件引用了不存在的变量或 local
- plan 阶段检查了 apply 后才知道的值

例如：

```hcl
assert {
  condition     = output.resource_name == "demo"
  error_message = "resource_name should be demo"
}
```

如果真实 output 是 `"terraform-debug-demo"`，测试就会失败。

这不一定是 Terraform 代码错了，也可能是测试期望错了。

### 调试重点

看测试失败时的这几类信息：

1. 哪个 test file 失败
2. 哪个 run block 失败
3. 哪个 assert 失败
4. 真实值和期望值是否一致

---

## 8. `terraform plan` 调试：这是最关键的预演阶段

`terraform plan` 会告诉你 Terraform 打算做什么。

常见输出包括：

```text
Plan: 1 to add, 0 to change, 0 to destroy.
```

或者：

```text
Plan: 0 to add, 1 to change, 0 to destroy.
```

或者比较危险的：

```text
Plan: 0 to add, 0 to change, 1 to destroy.
```

看到 destroy 时一定要谨慎。  
在生产环境里，`plan` 里出现 destroy，不要随手 apply。

### 保存 plan 文件

实验中使用：

```sh
terraform plan -input=false -no-color -out=tfplan
```

这会把 plan 保存为 `tfplan` 文件。

然后执行：

```sh
terraform apply -auto-approve tfplan
```

这样可以保证 apply 执行的是刚才生成的那份计划，而不是重新计算出来的一份新计划。

这在 CI/CD 里非常重要。

---

## 9. `terraform apply` 调试：真实执行阶段的问题通常来自外部系统

如果 `plan` 成功，但 `apply` 失败，问题往往不在 HCL 语法，而在执行层。

常见原因：

- 云平台权限不足
- API endpoint 错误
- 资源配额不足
- 资源名冲突
- 依赖资源还没准备好
- provider 调用远端 API 返回错误

例如 AWS 场景中可能遇到：

```text
AccessDenied
```

这通常是 IAM 权限问题。

Azure 场景中可能遇到：

```text
QuotaExceeded
```

这通常是订阅或区域配额问题。

LocalStack 场景中可能遇到：

```text
connection refused
```

这通常是 LocalStack 没启动，或者 endpoint 配错。

## 10. Terraform 日志：`TF_LOG` 是最后的大招，不是第一招

Terraform 支持通过环境变量打开详细日志。

常见级别：

```sh
export TF_LOG=INFO
export TF_LOG=DEBUG
export TF_LOG=TRACE
```

其中：

- `INFO`：一般信息
- `DEBUG`：详细调试信息
- `TRACE`：非常详细，信息量巨大

如果想把日志写入文件：

```sh
export TF_LOG=TRACE
export TF_LOG_PATH=terraform-debug.log
terraform plan
```

查看完之后，记得关闭：

```sh
unset TF_LOG
unset TF_LOG_PATH
```

### 注意

`TRACE` 日志可能包含敏感信息。  
不要把完整日志直接贴到 GitHub issue、公开论坛或公司群里。

生产环境里排查问题，要先脱敏。

---

## 11. `terraform output` 调试：确认 apply 后的真实结果

`terraform output` 用来查看 Terraform 暴露出来的结果。

例如：

```sh
terraform output
```

如果 output 没有出现，可能是：

- 没有定义 output
- output 名称写错
- apply 没成功
- output 被标记为 sensitive

示例：

```hcl
output "resource_name" {
  value = local.name
}
```

如果是敏感输出：

```hcl
output "password" {
  value     = random_password.demo.result
  sensitive = true
}
```

Terraform 默认不会直接展示明文。

---

## 12. 状态文件调试：谨慎使用，不要乱改

Terraform 的状态文件记录了：

- 当前 Terraform 管理了哪些资源
- 资源地址是什么
- 远端资源 ID 是什么
- output 是什么
- 依赖关系是什么

常用查看命令：

```sh
terraform state list
terraform state show <resource_address>
```

例如：

```sh
terraform state list
terraform state show null_resource.example
```

状态相关问题通常包括：

- 资源已经在云上存在，但状态里没有
- 状态里有资源，但云上已经被手动删了
- resource address 改名导致 Terraform 认为要重建
- count / for_each key 改了导致实例地址变化

### 关键提醒

不要手动编辑 `terraform.tfstate`。  
能用 `moved block`、`terraform import`、`terraform state mv` 解决的，就不要直接改 JSON。

## 13. 生产环境的调试顺序

建议你以后按这个顺序来：

```text
1. 看错误发生在哪个命令
2. 看 Terraform 报错中的文件、行号、资源地址
3. 运行 terraform fmt / validate 排除静态问题
4. 运行 terraform plan 看真实变更
5. 检查 provider、变量、backend、状态
6. 必要时打开 TF_LOG
7. 如果是云 API 报错，再查 IAM、配额、endpoint、资源限制
```

不要一上来就开 `TF_LOG=TRACE`。

这就像看病：  
先量体温、看症状，不是上来直接进 ICU。

## 14. 本实验的运行流程

进入实验目录：

```sh
cd /workspace/practice/labs/115
```

初始化：

```sh
terraform init -input=false
```

格式检查：

```sh
terraform fmt -check
```

语法验证：

```sh
terraform validate
```

运行测试：

```sh
terraform test
```

生成计划：

```sh
terraform plan -input=false -no-color -out=tfplan
```

执行计划：

```sh
terraform apply -auto-approve tfplan
```

查看输出：

```sh
terraform output
```

销毁资源：

```sh
terraform destroy -auto-approve
```

## 15. 清理方式

实验结束后可以清理：

```sh
terraform destroy -auto-approve
rm -f tfplan
rm -rf output logs .terraform .terraform.lock.hcl terraform.tfstate terraform.tfstate.backup
```

这些文件分别代表：

| 文件 / 目录 | 含义 |
|---|---|
| `tfplan` | 保存下来的 plan 文件 |
| `.terraform/` | provider 和 module 初始化目录 |
| `.terraform.lock.hcl` | provider 版本锁定文件 |
| `terraform.tfstate` | Terraform 状态文件 |
| `terraform.tfstate.backup` | 状态文件备份 |
| `logs/` | 可能产生的日志目录 |
| `output/` | 实验可能产生的输出目录 |

## 16. 迁移到真实项目时怎么用？

真实项目里，Terraform 调试不只是“让命令跑通”。

你要关注：

### 1. CI/CD 里如何暴露错误

建议至少包含：

```sh
terraform fmt -check
terraform validate
terraform test
terraform plan -out=tfplan
```

不要只在本地跑。

### 2. plan 是否需要人工审批

生产环境尤其要关注：

```text
Plan: x to add, y to change, z to destroy.
```

如果 `z to destroy` 大于 0，最好进入人工审批。

### 3. 日志是否会泄露敏感信息

`TF_LOG=TRACE` 不要默认开在 CI 里。  
它可能把请求细节、provider 调用细节、变量内容写出来。

### 4. 状态文件必须保护好

远程 backend 要配好：

- 加密
- 锁
- 权限控制
- 备份策略

状态文件不是普通缓存，它是 Terraform 的“账本”。

## 17. 面试怎么说？

如果面试官问：

> Terraform 出问题你怎么 debug？

可以这样回答：

```text
我会先根据失败命令定位问题层级。
如果 init 失败，我会检查 provider、backend、module source 和网络。
如果 validate 失败，我会根据文件和行号检查 HCL 引用关系。
如果 plan 异常，我会重点看资源地址、状态漂移、count/for_each key 变化以及 destroy 风险。
如果 apply 失败，我会检查云平台 API 返回、权限、配额和 provider 配置。
必要时我会打开 TF_LOG 和 TF_LOG_PATH 保存日志，但会注意脱敏。
生产环境中我会用 saved plan、审批门禁和远程状态锁来降低误操作风险。
```

这就是比较工程化的回答。  
不是“我会看报错”，而是知道怎么分层定位。

---

## 18. 本节一句话总结

Terraform 调试的核心不是记住某个神奇命令，而是建立一套分层排查路径：

```text
格式 → 语法 → 测试 → 计划 → 状态 → Provider → 云 API → 日志
```

先定位层级，再精准下手。  
别一出错就乱改。Terraform 很怕“手比脑子快”。
