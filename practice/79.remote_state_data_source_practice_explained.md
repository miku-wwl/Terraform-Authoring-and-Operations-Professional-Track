# Terraform 实操训练 79 讲解：Remote State Data Source 实操

## 1. 这一节到底在学什么？

第 79 节学的是 `terraform_remote_state` 的完整实操。

第 78 节讲的是概念：

```text
consumer 读取 producer 的 remote state output
```

第 79 节则把这个概念落地成两个目录：

```text
network/
security/
```

典型场景是：

```text
networking team 创建公网 IP，并把 state 写入 S3 backend
security team 不手工复制 IP
security 通过 terraform_remote_state 自动读取 network output
然后生成防火墙规则说明
```

一句话理解：

> 第 79 节是完整实现跨 Terraform state 的 producer / consumer 依赖：network 输出值，security 自动读取值。

这已经很接近真实团队协作场景。

---

## 2. 第 78 节和第 79 节的区别

可以这样理解：

| 章节 | 重点 | 你要掌握什么 |
|---|---|---|
| 第 78 节 | 概念 | 什么是 `terraform_remote_state`，为什么要读取 remote output |
| 第 79 节 | 实操 | network/security 两个目录如何真正跑通 |

第 78 节是“听懂”。  
第 79 节是“做出来”。

本节最重要的是完整闭环：

```text
apply network
-> network state 写入 LocalStack S3
-> network 暴露 output
-> apply security
-> security 通过 terraform_remote_state 读取 network output
-> security 生成规则说明
-> destroy security
-> destroy network
-> 清理 state 对象
```

---

## 3. 本节为什么拆成 network 和 security 两个目录？

因为真实公司里，基础设施通常不是一个 Terraform 项目全包。

可能是：

```text
network 团队负责网络
security 团队负责安全规则
application 团队负责应用
platform 团队负责平台组件
```

network 创建出来的东西，security 可能要用。

例如：

```text
network 创建 public IP
security 需要把这个 IP 放进防火墙规则
```

如果 security 人工复制 IP：

```text
203.0.113.10
```

就容易出问题：

- 复制错
- 忘记更新
- IP 改了但安全规则没改
- CI/CD 不能自动化
- 依赖关系没有记录
- 跨团队沟通成本高

所以更好的做法是：

```text
network output public_ip
security 从 network remote state 读取 public_ip
```

这就是本节的核心工程模式。

---

## 4. Producer / Consumer 模型

本节可以用 producer / consumer 来理解。

### Producer：network

network 是 producer。

它负责：

1. 生成一个模拟公网 IP。
2. 把自己的 state 写入 S3 backend。
3. 通过 output 暴露 `public_ip`。

示意：

```hcl
resource "terraform_data" "public_ip" {
  input = "203.0.113.10"
}

output "public_ip" {
  value = terraform_data.public_ip.output
}
```

---

### Consumer：security

security 是 consumer。

它负责：

1. 使用 `data "terraform_remote_state"` 读取 network 的 state。
2. 获取 network 输出的 `public_ip`。
3. 用这个 IP 生成规则说明。

示意：

```hcl
data "terraform_remote_state" "network" {
  backend = "s3"

  config = {
    bucket = "terraform-state"
    key    = "labs/79/network/terraform.tfstate"
    region = "us-east-1"
  }
}

locals {
  allowed_ip = data.terraform_remote_state.network.outputs.public_ip
}
```

---

## 5. `terraform_remote_state` 到底读取什么？

`terraform_remote_state` 读取的是另一个 Terraform state 里的 **outputs**。

关键点：

> 它不是读取对方所有 resource 内部属性，而是读取对方显式声明的 output。

network 必须写：

```hcl
output "public_ip" {
  value = terraform_data.public_ip.output
}
```

security 才能读：

```hcl
data.terraform_remote_state.network.outputs.public_ip
```

如果 network 没有 output，security 就读不到。

这就是跨团队接口思维：

```text
network 暴露 output
security 消费 output
```

output 就像 API contract。

## 6. 本节使用 LocalStack 的原因

课程文件说明，本项目不直接操作真实 AWS，而是用 Docker 启动 LocalStack 模拟：

- S3
- DynamoDB
- STS

本节使用 LocalStack 的目的：

1. 不需要真实 AWS 账号。
2. 不产生 AWS 费用。
3. 可以本地反复练习。
4. 可以模拟 S3 backend。
5. 可以验证 remote state 读取。
6. 适合自动化测试和教学。

本节备注也说明：

```text
本实验使用 terraform_data 模拟跨团队输出值，不创建真实 AWS 资源。
```

所以你不用纠结为什么没有真的创建公网 IP。  
重点不是云资源本身，而是：

```text
remote state output 读取链路
```

## 7. 为什么用 `terraform_data` 模拟公网 IP？

`terraform_data` 是 Terraform 内置资源，适合训练和模拟。

network 可以写：

```hcl
resource "terraform_data" "public_ip" {
  input = "203.0.113.10"
}
```

这相当于模拟：

```text
network 团队创建出来了一个 public IP
```

然后 output：

```hcl
output "public_ip" {
  value = terraform_data.public_ip.output
}
```

security 读取：

```hcl
data.terraform_remote_state.network.outputs.public_ip
```

这样不用真实 AWS EIP，也能练完整依赖链路。

## 8. 推荐目录结构

本节通常会有类似结构：

```text
practice/labs/79/
├── network/
│   ├── main.tf
│   └── README.md
├── security/
│   ├── main.tf
│   └── README.md
└── README.md
```

重点是：

```text
network 和 security 是两个独立 root module
network 有自己的 state
security 有自己的 state
security 通过 remote state 读取 network 的 output
```

不要把它当作普通 module 调用。

这不是：

```hcl
module "network" {
  source = "../network"
}
```

而是：

```hcl
data "terraform_remote_state" "network" {
  ...
}
```

## 9. network 的职责

network 目录负责：

1. 初始化自己的 S3 backend。
2. 创建模拟 public IP。
3. 输出 `public_ip`。
4. 把 state 写入指定 S3 key。

典型 state key：

```text
labs/79/network/terraform.tfstate
```

这代表：

```text
network 的 state 在 S3 bucket 的 labs/79/network/terraform.tfstate
```

security 后面要读取的就是这个 key。

---

## 10. security 的职责

security 目录负责：

1. 初始化自己的 backend。
2. 配置 `terraform_remote_state` 指向 network state。
3. 读取 network output。
4. 生成规则说明。
5. 把自己的 state 写入自己的 S3 key。

典型 state key：

```text
labs/79/security/terraform.tfstate
```

注意：

> security 自己的 state 和 network 的 state 应该分开。

security 读取 network state，但不能和 network 共用同一个 state key。

---

## 11. 为什么 network 和 security 不能共用同一个 state key？

如果两个 root module 都写：

```text
labs/79/terraform.tfstate
```

那它们会互相覆盖 state。

正确方式：

```text
network state:
labs/79/network/terraform.tfstate

security state:
labs/79/security/terraform.tfstate
```

这才是两个独立 root module 的正确边界。

跨 state 依赖不是共用 state。

跨 state 依赖是：

```text
security 读取 network 的 output
但 security 自己的资源状态写到自己的 state
```

这个概念要分清楚。

## 12. network 配置示例

network 可以类似这样：

```hcl
terraform {
  backend "s3" {
    bucket = "terraform-state"
    key    = "labs/79/network/terraform.tfstate"
    region = "us-east-1"

    endpoints = {
      s3 = "http://localhost:4566"
    }

    skip_credentials_validation = true
    skip_metadata_api_check     = true
    skip_region_validation      = true
    force_path_style            = true
  }
}

resource "terraform_data" "public_ip" {
  input = "203.0.113.10"
}

output "public_ip" {
  value = terraform_data.public_ip.output
}
```

这段配置的重点不是 `terraform_data` 本身，而是：

```text
output "public_ip"
```

因为 consumer 只能读取 output。

## 13. security 配置示例

security 可以类似这样：

```hcl
terraform {
  backend "s3" {
    bucket = "terraform-state"
    key    = "labs/79/security/terraform.tfstate"
    region = "us-east-1"

    endpoints = {
      s3 = "http://localhost:4566"
    }

    skip_credentials_validation = true
    skip_metadata_api_check     = true
    skip_region_validation      = true
    force_path_style            = true
  }
}

data "terraform_remote_state" "network" {
  backend = "s3"

  config = {
    bucket = "terraform-state"
    key    = "labs/79/network/terraform.tfstate"
    region = "us-east-1"

    endpoints = {
      s3 = "http://localhost:4566"
    }

    skip_credentials_validation = true
    skip_metadata_api_check     = true
    skip_region_validation      = true
    force_path_style            = true
  }
}

resource "terraform_data" "security_rule" {
  input = "Allow traffic from ${data.terraform_remote_state.network.outputs.public_ip}"
}
```

真实场景里 `terraform_data.security_rule` 可以替换成：

- AWS security group rule
- Azure NSG rule
- firewall rule
- WAF rule
- Kubernetes NetworkPolicy
- 配置文件生成逻辑

---

## 14. 为什么 security 也要配置自己的 backend？

因为 security 自己也会创建资源或保存状态。

即使它只是读取 network output，它自己的 `terraform_data.security_rule` 也会进入 state。

所以 security 应该有自己的 backend：

```hcl
terraform {
  backend "s3" {
    key = "labs/79/security/terraform.tfstate"
  }
}
```

同时它还要读取 network 的 backend：

```hcl
data "terraform_remote_state" "network" {
  config = {
    key = "labs/79/network/terraform.tfstate"
  }
}
```

这两个 key 不同，职责不同。

## 15. 正确 apply 顺序

本节必须按顺序执行。

创建顺序：

```text
1. apply network
2. apply security
```

原因：

security 需要读取 network 的 remote state output。

如果 network 还没 apply：

```text
network state 不存在
network output 不存在
security 读取失败
```

所以先 network，再 security。

## 20. 正确 destroy 顺序

销毁顺序必须反过来：

```text
1. destroy security
2. destroy network
```

原因：

security 依赖 network output。

如果你先 destroy network：

```text
network state/output 可能消失
security 再 destroy 时读取 remote state 失败
```

所以正确命令顺序：

```sh
cd practice/labs/79/security
terraform destroy -auto-approve
```

然后：

```sh
cd ../network
terraform destroy -auto-approve
```

创建和销毁顺序可以记成：

```text
创建：先 producer，再 consumer
销毁：先 consumer，再 producer
```

## 23. `terraform_remote_state` 读取失败时怎么排查？

### 1. network 是否已经 apply？

先检查：

```sh
cd network
terraform output
```

如果 network output 都没有，security 肯定读不到。

---

### 2. network state 是否在 S3？

```sh
aws --endpoint-url=http://localhost:4566 s3 ls s3://terraform-state/labs/79/network/
```

如果没有 `terraform.tfstate`，说明 network state 没写进去或 key 写错。

---

### 3. security 的 remote state key 是否正确？

security 里应该读取：

```text
labs/79/network/terraform.tfstate
```

不是：

```text
labs/79/security/terraform.tfstate
```

---

### 4. output 名称是否一致？

network：

```hcl
output "public_ip" { ... }
```

security：

```hcl
data.terraform_remote_state.network.outputs.public_ip
```

名称必须一致。

---

### 5. LocalStack endpoint 是否一致？

network 和 security 都要指向同一个 LocalStack endpoint：

```text
http://localhost:4566
```

否则可能一个写本地 LocalStack，一个尝试读真实 AWS。

---

### 6. bucket 是否一致？

network 写入的 bucket 和 security 读取的 bucket 必须一致。

---

## 24. remote state 的输出值什么时候更新？

只有 network apply 后，network state 里的 output 才会更新。

如果你修改 network 配置：

```hcl
input = "203.0.113.20"
```

但没有重新 apply network，security 读到的仍然是旧 state output。

正确流程：

```text
修改 network
-> apply network
-> network state output 更新
-> plan/apply security
-> security 读取新 output
```

`terraform_remote_state` 不会自动 apply producer。

---

## 25. output 契约要稳定

network 暴露的 output 相当于跨团队 API。

例如：

```hcl
output "public_ip" {
  value = terraform_data.public_ip.output
}
```

security 依赖：

```hcl
outputs.public_ip
```

如果 network 把 output 改名：

```hcl
output "public_ip_address" {
  ...
}
```

security 就会坏。

所以 output 要稳定：

- 名称稳定
- 类型稳定
- 语义稳定
- 不随便删除
- 变更前通知 consumer
- 重要 output 写 README

这就是跨团队协作的基本纪律。

---

## 26. remote state 和敏感数据

不要随便通过 remote state 共享 secret。

例如 network 输出：

```hcl
output "db_password" {
  value     = var.db_password
  sensitive = true
}
```

即使标记 sensitive，state 里仍然可能保存敏感值。  
consumer 如果能读 state，也可能间接接触这些数据。

remote state 更适合共享：

- IP
- subnet IDs
- VPC ID
- security group IDs
- LB DNS
- cluster endpoint
- account ID
- resource ARN
- 非敏感平台元数据

敏感数据应该通过：

- AWS Secrets Manager
- Azure Key Vault
- HashiCorp Vault
- SSM Parameter Store
- Kubernetes Secret 管理流程
- 专门的 secret distribution workflow

---

## 27. remote state 的优点

优点：

1. 减少人工复制。
2. 自动读取最新已 apply output。
3. 适合跨团队基础设施依赖。
4. 比手工变量更可靠。
5. output 契约清晰。
6. 可以配合 CI/CD 自动化。
7. 不需要把所有资源放一个 state。
8. 降低单个 state 的 blast radius。

对于 network/security 这种团队拆分，非常实用。

---

## 28. remote state 的缺点

缺点也要知道：

1. 引入跨 state 耦合。
2. consumer 需要知道 producer state 位置。
3. consumer 需要读取 producer state 权限。
4. output 变更会影响 consumer。
5. apply 顺序要人为或 CI 管理。
6. destroy 顺序要反过来。
7. 不适合共享敏感 secret。
8. 大规模依赖会变复杂。

所以 remote state 不是万能方案，而是适合有限、明确、稳定的跨配置输出共享。

---

## 29. remote state vs provider data source

有些值也可以通过 provider data source 查询。

例如 security 需要 public IP，有时可以：

```hcl
data "aws_eip" "network" {
  tags = {
    Name = "network-public-ip"
  }
}
```

也可以：

```hcl
data.terraform_remote_state.network.outputs.public_ip
```

对比：

| 方式 | 优点 | 风险 |
|---|---|---|
| remote state | 精确读取 producer output | 跨 state 耦合，需要 state 权限 |
| provider data source | 从真实云 API 查询 | 依赖 tag/name，可能查错 |
| 手工变量 | 最简单 | 容易过期和复制错 |

工程上没有绝对答案。  
network/security 这种明确 producer-consumer 关系，用 remote state 很合理。

## 30. 和普通 module 的区别

如果 network 和 security 在同一个 root module 里，可以直接：

```hcl
module "network" {
  source = "./modules/network"
}

module "security" {
  source    = "./modules/security"
  public_ip = module.network.public_ip
}
```

但本节不是这种。

本节是两个独立 root module：

```text
network/
security/
```

security 用：

```hcl
data.terraform_remote_state.network.outputs.public_ip
```

区别：

| 方式 | state |
|---|---|
| module output | 同一个 root state |
| remote state | 不同 root state |

第 79 节练的是后者。

## 34. SRE / Platform Engineering 视角

从 SRE / Platform Engineering 角度看，本节非常像真实平台工程。

常见场景：

```text
network 团队输出 subnet IDs
platform 团队读取 subnet IDs 创建 EKS/AKS node pool

network 团队输出 public IP
security 团队读取 IP 创建 firewall allow rule

platform 团队输出 cluster endpoint
monitoring 团队读取 endpoint 配置监控

identity 团队输出 role ARN
app 团队读取 role ARN 配置 workload identity
```

这就是跨团队基础设施数据流。

但你必须管住边界：

- output 是 API
- state 是敏感对象
- consumer 不能乱读内部细节
- CI/CD 要控制顺序
- destroy 要反向依赖
- output 变更要审慎

这就是从“会写 Terraform”走向“会组织 Terraform 工程”的区别。

## 35. 面试/考试记忆版

可以这样记：

- 第 79 节是 `terraform_remote_state` 的完整实操。
- network 是 producer，security 是 consumer。
- network 先 apply，把 state 写入 S3 backend。
- network 必须声明 output，例如 `public_ip`。
- security 用 `data "terraform_remote_state"` 读取 network output。
- security 不能读取 network 内部资源，只能读取 output。
- apply 顺序：先 network，再 security。
- destroy 顺序：先 security，再 network。
- network 和 security 应该使用不同 state key。
- remote state 会引入跨 state 耦合。
- 不要通过 remote state 暴露敏感 secret。
- LocalStack 用来模拟 S3 backend，不创建真实 AWS 资源。

一句话总结：

> 第 79 节的核心，是把 remote state 从概念落到实操：network 暴露 output，security 自动读取 output，并按正确顺序完成 apply、destroy 和清理闭环。

## 36. 本 Lab 的价值

第 79 节的价值在于，它让你真正做了一次跨 Terraform root module 的协作。

这比单目录 Terraform 更接近真实生产。

你会开始理解：

```text
为什么 state 要拆
为什么 output 是契约
为什么 consumer 不能乱读 producer 内部实现
为什么 apply/destroy 顺序重要
为什么 remote state 方便但有耦合
```

如果你未来做 AKS/EKS、平台工程、SRE、FinOps、权限治理，这类 producer-consumer 结构会经常出现。

记住一句话：

> remote state 实操的关键不是“能读到值”，而是能把跨团队依赖设计成稳定、清晰、可销毁、可清理、不会乱动 state 的工程闭环。
