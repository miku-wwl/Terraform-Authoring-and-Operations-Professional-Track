## 1. 这一节到底在讲什么？

第 160 节不是让你马上创建 AWS 资源，而是挑战三的**赛前热身**。

你可以把它理解成：

> 前面很多 lab 是单点技能训练；从挑战三开始，题目会把多个 AWS Provider 能力揉在一起考。你需要知道每个配置项在 Terraform 执行链路里的位置。

这一节的重点不是背某个资源参数，而是建立一个清晰的心智模型：

```text
Terraform 配置
   ↓
AWS Provider 配置
   ↓
凭证来源 / profile / assume role / alias / default tags
   ↓
AWS API 或 LocalStack endpoint
   ↓
创建、读取、更新、销毁资源
```

在挑战题里，你很可能会同时看到这些东西：

- `provider "aws"`
- `profile`
- `alias`
- `default_tags`
- `data "aws_caller_identity"`
- LocalStack endpoint
- 假凭证 `test/test`
- 多 provider 实例
- 模块中显式传 provider

这一节就是提前告诉你：后面不是单独考一个关键字，而是考你能不能把这些东西串起来。

## 2. 为什么 AWS Provider 是挑战三的核心？

Terraform 自己并不会直接知道怎么创建 AWS 资源。

Terraform 的核心职责更像是：

```text
我负责读取 .tf 配置、维护 state、计算 diff、执行 plan/apply。
```

但真正调用 AWS API 的是 AWS Provider。

也就是说：

```text
Terraform Core：调度器 / 状态机
AWS Provider：AWS API 适配器
AWS API / LocalStack：最终被调用的一方
```

所以，当题目说“AWS Provider 集成”的时候，真正考的是：

1. Terraform 怎么拿到 AWS 凭证。
2. Provider 怎么知道请求发到哪里。
3. 多套 Provider 配置怎么区分。
4. 资源应该绑定哪个 Provider。
5. 默认标签怎么统一注入。
6. 当前身份怎么验证。

这几个点混在一起，就是生产环境里很常见的 Provider 集成问题。

## 3. 本节的关键词总览

| 关键词 | 作用 | 你应该记住的点 |
|---|---|---|
| `profile` | 指定 AWS named profile | 从本机 AWS config/credentials 中读取某个 profile |
| `alias` | 创建多个 provider 实例 | 用于多区域、多账号、多角色、多环境场景 |
| `default_tags` | 给 provider 管理的资源统一加标签 | 常用于成本、环境、owner、项目归属治理 |
| `aws_caller_identity` | 查询当前 AWS 身份 | 用来确认当前 credentials 对应哪个账号 / 用户 / role |
| endpoint override | 改写 AWS API 地址 | LocalStack 练习中指向 `http://localhost:4566` |
| fake credentials | 本地模拟凭证 | LocalStack 用 `test/test`，不访问真实 AWS |

这几个点是第 161~162 节继续展开的基础。

## 4. profile：Provider 怎么找到凭证？

### 4.1 profile 是什么？

`profile` 指的是 AWS CLI / SDK 里的 named profile。

常见配置文件是：

```text
~/.aws/credentials
~/.aws/config
```

比如：

```ini
[default]
aws_access_key_id = xxx
aws_secret_access_key = xxx

[dev]
aws_access_key_id = xxx
aws_secret_access_key = xxx

[prod]
aws_access_key_id = xxx
aws_secret_access_key = xxx
```

Terraform AWS Provider 可以这样指定：

```hcl
provider "aws" {
  region  = "ap-southeast-2"
  profile = "dev"
}
```

意思是：

> 这套 Terraform 配置执行时，不使用 default profile，而是使用 `dev` 这个 named profile。

### 4.2 profile 的生产价值

在真实项目里，profile 可以帮助你区分：

- 开发账号
- 测试账号
- 生产账号
- sandbox 账号
- 安全账号
- shared services 账号

比如：

```hcl
provider "aws" {
  region  = "ap-southeast-2"
  profile = "platform-dev"
}
```

这比把 access key 写死在 `.tf` 文件里安全得多。

### 4.3 考试/挑战题常见考点

看到 `profile`，你要立刻想到：

```text
这是 Provider 的凭证选择机制，不是资源属性。
```

它不是给 EC2、S3、IAM 资源加的参数，而是告诉 AWS Provider：

> 你用哪套身份去调用 AWS API。

常见错误是把 `profile` 当成资源标签、环境名或者 workspace 名。

这不对。

`profile` 是 Provider 层的凭证配置。

## 5. alias：为什么需要多个 Provider 实例？

### 5.1 默认 Provider

最简单的 AWS Provider 是这样的：

```hcl
provider "aws" {
  region = "ap-southeast-2"
}
```

然后资源默认使用它：

```hcl
resource "aws_s3_bucket" "example" {
  bucket = "example-bucket"
}
```

此时资源没有写 `provider = ...`，所以它会使用默认的 `aws` provider。

### 5.2 alias Provider

如果你需要同一个配置里使用多套 AWS Provider，就要用 `alias`。

比如：

```hcl
provider "aws" {
  region  = "ap-southeast-2"
  profile = "dev"
}

provider "aws" {
  alias   = "prod"
  region  = "ap-southeast-2"
  profile = "prod"
}
```

这里有两个 provider 实例：

```text
aws        -> 默认 provider，使用 dev profile
aws.prod   -> 带 alias 的 provider，使用 prod profile
```

资源可以显式绑定：

```hcl
resource "aws_s3_bucket" "prod_logs" {
  provider = aws.prod
  bucket   = "prod-logs-example"
}
```

这表示：

> 这个 bucket 用 `aws.prod` 这个 Provider 实例创建，而不是默认的 `aws`。

### 5.3 alias 常见使用场景

`alias` 在真实生产中很常见，尤其是：

1. **多区域**

```hcl
provider "aws" {
  region = "ap-southeast-2"
}

provider "aws" {
  alias  = "use1"
  region = "us-east-1"
}
```

比如 CloudFront 证书常常要在 `us-east-1`。

2. **多账号**

```hcl
provider "aws" {
  alias   = "network"
  profile = "network-account"
  region  = "ap-southeast-2"
}

provider "aws" {
  alias   = "app"
  profile = "app-account"
  region  = "ap-southeast-2"
}
```

3. **跨账号共享资源**

比如一个账号创建 VPC，另一个账号部署应用。

4. **模块显式传 Provider**

```hcl
module "network" {
  source = "./modules/network"

  providers = {
    aws = aws.network
  }
}
```

这个非常像真实平台工程里的写法。

## 6. alias 的一个关键坑：资源不一定用你以为的 Provider

如果资源没有显式写：

```hcl
provider = aws.xxx
```

那它默认使用默认 Provider。

比如：

```hcl
provider "aws" {
  region  = "ap-southeast-2"
  profile = "dev"
}

provider "aws" {
  alias   = "prod"
  region  = "ap-southeast-2"
  profile = "prod"
}

resource "aws_s3_bucket" "important" {
  bucket = "very-important-bucket"
}
```

这个 `aws_s3_bucket.important` 用的是谁？

答案是：

```text
默认 aws provider，也就是 dev。
```

不是 `aws.prod`。

想用 prod，就要写：

```hcl
resource "aws_s3_bucket" "important" {
  provider = aws.prod
  bucket   = "very-important-bucket"
}
```

考试题里很喜欢考这种“你以为用了 alias，其实没有绑定”的陷阱。

## 7. default_tags：统一标签治理

### 7.1 default_tags 是什么？

`default_tags` 是 AWS Provider 的一个配置块，用于给 Provider 管理的资源统一加标签。

示例：

```hcl
provider "aws" {
  region = "ap-southeast-2"

  default_tags {
    tags = {
      Environment = "dev"
      ManagedBy   = "Terraform"
      Owner       = "platform-team"
      Project     = "terraform-training"
    }
  }
}
```

之后通过这个 Provider 创建的资源，会自动带上这些默认标签。

### 7.2 生产价值

标签不是装饰品。

在真实云平台里，标签通常用于：

- 成本归属
- 资源所有权
- 环境区分
- 自动化清理
- 合规审计
- 安全扫描分组
- FinOps 报表

比如：

```text
Environment = prod
Owner       = sre-team
CostCenter  = cc-1234
ManagedBy   = Terraform
```

这些标签可以帮助团队回答：

```text
这个资源谁负责？
这个资源属于哪个环境？
这个资源是否可以删除？
这个资源成本应该算到哪个团队？
```

### 7.3 default_tags 和资源 tags 的关系

Provider 默认标签：

```hcl
provider "aws" {
  region = "ap-southeast-2"

  default_tags {
    tags = {
      ManagedBy   = "Terraform"
      Environment = "dev"
    }
  }
}
```

资源自身标签：

```hcl
resource "aws_s3_bucket" "example" {
  bucket = "example-bucket"

  tags = {
    Name = "example-bucket"
  }
}
```

最终资源通常会同时拥有：

```text
ManagedBy   = Terraform
Environment = dev
Name        = example-bucket
```

注意：不同资源、不同 Provider 版本在标签合并细节上可能有差异，但考试层面重点是：

> `default_tags` 是 Provider 级别的统一标签机制。

## 8. caller identity：确认当前是谁在操作 AWS

### 8.1 aws_caller_identity 是什么？

`aws_caller_identity` 是一个 data source：

```hcl
data "aws_caller_identity" "current" {}
```

它会调用 AWS STS 的 `GetCallerIdentity` API，返回当前身份信息。

常见输出包括：

```hcl
output "account_id" {
  value = data.aws_caller_identity.current.account_id
}

output "user_id" {
  value = data.aws_caller_identity.current.user_id
}

output "arn" {
  value = data.aws_caller_identity.current.arn
}
```

### 8.2 它的作用是什么？

它不是创建资源。

它的作用是确认：

```text
当前 Terraform 使用的是哪个 AWS 身份？
```

在真实生产中，这非常重要。

因为你可能以为自己正在 dev 账号操作，结果凭证指向 prod 账号。

这就是云上经典恐怖故事：

```text
我只是想 apply 一下 dev，怎么 prod 没了？
```

所以很多项目会在输出或验证脚本里加入 caller identity 检查。

### 8.3 caller identity 和 profile 的关系

`profile` 是你指定的凭证来源。

`aws_caller_identity` 是你验证最终身份。

可以理解为：

```text
profile：我打算用谁
caller identity：我实际用的是谁
```

这个区分非常关键。

## 9. LocalStack 场景下怎么理解这些概念？

本课程把真实 AWS 操作改造成 LocalStack 或本地 provider 可验证的练习。

也就是说，默认 AWS API 会指向：

```text
http://localhost:4566
```

凭证固定是：

```text
AWS_ACCESS_KEY_ID=test
AWS_SECRET_ACCESS_KEY=test
```

### 9.1 LocalStack 能验证什么？

LocalStack 可以帮你练：

- Terraform AWS Provider 基础配置
- endpoint override 写法
- 资源依赖关系
- `terraform init`
- `terraform validate`
- `terraform plan`
- `terraform apply`
- `terraform destroy`
- 部分 AWS API 的模拟行为
- data source 调用路径
- output 设计
- 标签建模

这些对 Terraform 学习很有价值。

### 9.2 LocalStack 不能完全替代什么？

LocalStack 不能完全替代真实 AWS 的：

- 真实 IAM 权限评估
- 真实 STS 跨账号 assume role
- 真实服务配额
- 真实网络连通性
- 真实区域差异
- 真实资源生命周期
- 真实 AWS 账单
- 某些高级 API 行为

所以你要有边界感。

本节是 `aws-theory`，意思就是：

> 重点在理解 Provider 集成方式，不是创建真实 AWS 资源。

## 10. 一个完整 Provider 配置示例

下面是一个 LocalStack 风格的示例，帮助你把概念串起来。

```hcl
terraform {
  required_version = ">= 1.11.0"

  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 5.0"
    }
  }
}

provider "aws" {
  region                      = "ap-southeast-2"
  access_key                  = "test"
  secret_key                  = "test"
  skip_credentials_validation = true
  skip_metadata_api_check     = true
  skip_requesting_account_id  = true
  s3_use_path_style           = true

  endpoints {
    s3  = "http://localhost:4566"
    sts = "http://localhost:4566"
  }

  default_tags {
    tags = {
      Environment = "local"
      ManagedBy   = "Terraform"
      Lab         = "160"
    }
  }
}

data "aws_caller_identity" "current" {}

output "caller_identity" {
  value = {
    account_id = data.aws_caller_identity.current.account_id
    arn        = data.aws_caller_identity.current.arn
    user_id    = data.aws_caller_identity.current.user_id
  }
}
```

这个配置展示了：

- AWS Provider 配置
- LocalStack endpoint
- 假凭证
- STS caller identity
- default tags
- Terraform output

真实 AWS 中通常不会写死 `access_key` 和 `secret_key`，而是用：

- environment variables
- named profile
- SSO profile
- assume role
- CI/CD OIDC
- HCP Terraform variable set

## 11. 多 Provider alias 示例

如果挑战题出现多个 Provider，你可以这样理解：

```hcl
provider "aws" {
  region  = "ap-southeast-2"
  profile = "dev"

  default_tags {
    tags = {
      Environment = "dev"
      ManagedBy   = "Terraform"
    }
  }
}

provider "aws" {
  alias   = "prod"
  region  = "ap-southeast-2"
  profile = "prod"

  default_tags {
    tags = {
      Environment = "prod"
      ManagedBy   = "Terraform"
    }
  }
}
```

资源绑定默认 Provider：

```hcl
resource "aws_s3_bucket" "dev_bucket" {
  bucket = "example-dev-bucket"
}
```

资源绑定 alias Provider：

```hcl
resource "aws_s3_bucket" "prod_bucket" {
  provider = aws.prod
  bucket   = "example-prod-bucket"
}
```

考点就是：

```text
alias 只是定义了另一个 provider 实例。
资源是否使用它，要看资源或模块有没有显式绑定。
```

## 12. 模块中的 Provider 传递

挑战题如果进入模块，就更容易绕。

比如 root module：

```hcl
provider "aws" {
  region = "ap-southeast-2"
}

provider "aws" {
  alias  = "security"
  region = "ap-southeast-2"
}

module "audit" {
  source = "./modules/audit"

  providers = {
    aws = aws.security
  }
}
```

这表示：

> `modules/audit` 里面默认叫 `aws` 的 Provider，实际绑定的是 root module 里的 `aws.security`。

这就是 Terraform 模块化里非常重要的 Provider 传递机制。

考试或挑战中，如果看到：

```hcl
providers = {
  aws = aws.xxx
}
```

你要知道它是在告诉 child module：

```text
你里面的 aws provider，用外面传进来的这个。
```

## 13. default_tags + alias 的组合坑

假设你有两个 Provider：

```hcl
provider "aws" {
  region = "ap-southeast-2"

  default_tags {
    tags = {
      Environment = "dev"
    }
  }
}

provider "aws" {
  alias  = "prod"
  region = "ap-southeast-2"

  default_tags {
    tags = {
      Environment = "prod"
    }
  }
}
```

资源 A：

```hcl
resource "aws_s3_bucket" "a" {
  bucket = "bucket-a"
}
```

资源 B：

```hcl
resource "aws_s3_bucket" "b" {
  provider = aws.prod
  bucket   = "bucket-b"
}
```

那么：

```text
bucket-a 默认会带 Environment=dev
bucket-b 默认会带 Environment=prod
```

所以 default tags 也是跟 Provider 实例绑定的。

不是全局魔法。

## 14. caller identity + alias 的组合

如果有多个 Provider，你可能要分别查询身份：

```hcl
data "aws_caller_identity" "default" {}

data "aws_caller_identity" "prod" {
  provider = aws.prod
}
```

输出：

```hcl
output "default_account_id" {
  value = data.aws_caller_identity.default.account_id
}

output "prod_account_id" {
  value = data.aws_caller_identity.prod.account_id
}
```

这可以验证：

```text
默认 provider 当前是谁？
prod alias provider 当前是谁？
```

这在多账号 Terraform 项目里非常有用。

## 16. 你做挑战题时应该怎么拆题？

遇到 AWS Provider 集成题，可以按这个顺序拆：

### 第一步：看 Provider 有没有配置完整

检查：

```text
region 有没有？
profile 有没有？
LocalStack endpoint 有没有？
skip 参数有没有？
default_tags 有没有？
alias 有没有？
```

### 第二步：看凭证来源

判断它使用的是：

```text
环境变量？
profile？
写死 test/test？
assume_role？
CI/CD 注入？
```

LocalStack 练习一般是 `test/test`。

真实 AWS 项目里不要把真实密钥写进 `.tf`。

### 第三步：看资源绑定哪个 Provider

重点看有没有：

```hcl
provider = aws.xxx
```

没有的话，就是默认 Provider。

### 第四步：看模块有没有 provider 显式传递

重点看：

```hcl
providers = {
  aws = aws.xxx
}
```

### 第五步：看 default_tags 是否符合要求

题目如果要求统一标签，就检查 Provider 里有没有：

```hcl
default_tags {
  tags = {
    ...
  }
}
```

### 第六步：看 caller identity 输出是否合理

如果题目要求验证当前身份，就检查：

```hcl
data "aws_caller_identity" "current" {}
```

以及 output 是否输出了 account id / arn。

## 17. 常见错误与排障方向

### 错误 1：LocalStack endpoint 没配

现象：Terraform 试图访问真实 AWS。

排查：检查 Provider 中是否配置了：

```hcl
endpoints {
  ... = "http://localhost:4566"
}
```

以及是否设置了必要的 skip 参数。

---

### 错误 2：profile 写了，但本地不存在

现象：报错找不到 profile。

排查：检查：

```sh
cat ~/.aws/credentials
cat ~/.aws/config
aws configure list-profiles
```

LocalStack lab 通常不应该依赖你真实机器里的 profile，除非题目就是考 named profile。

---

### 错误 3：定义了 alias，但资源没有绑定

现象：资源跑到了默认 Provider 上。

排查：检查资源是否写了：

```hcl
provider = aws.alias_name
```

---

### 错误 4：模块没有接收正确 Provider

现象：child module 里资源仍然使用默认 Provider。

排查 root module 的 module block 是否有：

```hcl
providers = {
  aws = aws.alias_name
}
```

---

### 错误 5：caller identity 和预期不一致

现象：输出的账号 ID / ARN 不对。

排查：

- profile 是否选错
- 环境变量是否覆盖 profile
- assume role 是否成功
- alias provider 是否绑定正确
- LocalStack 是否返回固定模拟身份

---

## 18. 真实 AWS 中更推荐的凭证方式

虽然课程为了练习会用 LocalStack 的 `test/test`，但真实项目里更推荐：

### 本地开发

- AWS SSO profile
- named profile
- short-lived credentials

### CI/CD

- OIDC federation
- AssumeRoleWithWebIdentity
- 临时凭证
- 不长期保存 access key

### HCP Terraform

- Workspace variables
- Variable sets
- Dynamic credentials
- OIDC workload identity federation

简单说：

```text
真实生产里，尽量用临时凭证，不要长期 access key。
```

这是安全底线。

## 19. 考试记忆版

记住这几句话就够用了：

1. **AWS Provider 是 Terraform 调 AWS API 的桥。**
2. **profile 决定用哪套 named credentials。**
3. **alias 用来创建多个 Provider 实例。**
4. **资源不写 `provider = aws.xxx`，就默认用 default provider。**
5. **default_tags 是 Provider 级统一标签机制。**
6. **aws_caller_identity 用来确认当前实际身份。**
7. **LocalStack 能验证 Terraform 配置链路，但不能完全替代真实 AWS 行为。**
8. **挑战题先看 Provider，再看资源，再看输出和验证。**

## 20. 场景题演练

### 场景 1：团队要求所有资源带上 `ManagedBy=Terraform`

应该在哪里配置？

答案：

```hcl
provider "aws" {
  region = "ap-southeast-2"

  default_tags {
    tags = {
      ManagedBy = "Terraform"
    }
  }
}
```

解析：

这是统一标签治理，应该放在 Provider 的 `default_tags` 中，而不是每个资源手工重复写。

---

### 场景 2：一个配置同时管理 dev 和 prod

应该怎么做？

答案：

使用多个 AWS Provider，其中一个使用默认 Provider，另一个使用 `alias`。

```hcl
provider "aws" {
  profile = "dev"
  region  = "ap-southeast-2"
}

provider "aws" {
  alias   = "prod"
  profile = "prod"
  region  = "ap-southeast-2"
}
```

prod 资源要显式绑定：

```hcl
provider = aws.prod
```

---

### 场景 3：你想确认当前 Terraform 到底用了哪个 AWS 身份

应该用什么？

答案：

```hcl
data "aws_caller_identity" "current" {}
```

然后输出：

```hcl
output "current_arn" {
  value = data.aws_caller_identity.current.arn
}
```

解析：

`profile` 是配置意图，`caller identity` 是实际身份验证。

---

### 场景 4：你定义了 `aws.prod`，但资源没有写 `provider = aws.prod`

这个资源会用哪个 Provider？

答案：

默认 Provider。

解析：

alias Provider 不会自动接管资源。资源或模块必须显式绑定。

## 21. 判断题

1. `profile` 是 AWS Provider 的凭证选择配置。  
   答案：对。

2. 只要定义了 `provider "aws" { alias = "prod" }`，所有资源都会自动使用 prod。  
   答案：错。资源必须显式绑定 `provider = aws.prod`。

3. `default_tags` 可以用于统一资源标签治理。  
   答案：对。

4. `aws_caller_identity` 会创建一个 IAM 用户。  
   答案：错。它是 data source，用于查询当前身份。

5. LocalStack 的 `test/test` 凭证可以直接用于真实 AWS。  
   答案：错。那只是本地模拟环境用的假凭证。
