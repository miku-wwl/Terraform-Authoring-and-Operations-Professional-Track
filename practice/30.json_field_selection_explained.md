# Terraform 实操训练 30 讲解版：从 JSON 中提取指定字段

## 1. 本 Lab 要学什么

这一节练的是 Terraform 里非常实用的一类能力：**把外部 JSON 读进来，然后只提取自己需要的字段**。

上一节 Lab 29 已经讲过：

- `jsonencode`：Terraform 数据结构 → JSON 字符串
- `jsondecode`：JSON 字符串 → Terraform 原生数据结构

Lab 30 往前走一步：  
不是只把 JSON 解出来，而是要从一个服务目录 `catalog.json` 里筛选出 `tier == "backend"` 的服务，然后读取它的服务名、端口和端口数量。

一句话理解：

> `jsondecode` 之后，JSON 就不再是“文本”，而是 Terraform 可以直接访问的 object / list。

---

## 2. 为什么这个能力重要

真实工作中，Terraform 很少只处理手写变量。

更常见的是：

- 一个平台系统给你一份服务目录 JSON
- 一个 CMDB 导出资源清单 JSON
- 一个安全平台导出 IP range JSON
- 一个应用团队维护服务配置 JSON
- CI/CD 里生成一份环境参数 JSON

Terraform 要做的事情通常不是“照单全收”，而是：

> 只从里面拿出需要的字段，然后传给资源、模块、输出或本地文件。

比如本实验里，外部 JSON 有多个服务，但 Terraform 只关心 backend 服务：

```hcl
tier == "backend"
```

然后继续读取：

```hcl
name
ports[0]
length(ports)
```

这就是生产环境里常见的“外部数据接入 + 字段筛选”。

---

## 3. 核心知识点

### 3.1 `jsondecode`：把 JSON 文本变成 Terraform 数据

通常会搭配 `file()` 使用：

```hcl
locals {
  catalog = jsondecode(file("${path.module}/data/catalog.json"))
}
```

这里可以拆成两步理解：

```hcl
file(...)
```

读取 JSON 文件内容，得到的是字符串。

```hcl
jsondecode(...)
```

把这个字符串解析成 Terraform 的 object / list / number / string / bool 等原生类型。

所以原本的 JSON：

```json
{
  "services": [
    {
      "name": "payments",
      "tier": "backend",
      "ports": [8080, 9090]
    }
  ]
}
```

解码后就可以这样访问：

```hcl
local.catalog.services
local.catalog.services[0].name
local.catalog.services[0].ports[0]
```

---

### 3.2 对象字段访问：`.key`

JSON object 解码后，可以像 Terraform object 一样用 `.key` 访问字段。

例如：

```hcl
local.catalog.services
```

表示访问 catalog 对象里的 `services` 字段。

如果某个服务对象长这样：

```json
{
  "name": "payments",
  "tier": "backend"
}
```

那么 Terraform 里可以写：

```hcl
service.name
service.tier
```

注意：JSON key 是区分大小写的。

`name` 和 `Name` 不是一回事。

---

### 3.3 列表索引访问：`[index]`

JSON array 解码后，就是 Terraform list / tuple，可以用索引访问。

例如：

```hcl
local.backend_services[0]
```

表示取第一个 backend 服务。

```hcl
local.backend_service.ports[0]
```

表示取这个服务的第一个端口。

索引从 0 开始，所以：

- `[0]` 是第一个
- `[1]` 是第二个
- `[2]` 是第三个

本实验预期第一个端口是：

```text
8080
```

---

### 3.4 用 `for` 表达式筛选数组

如果服务目录里有多个服务，我们可以用 `for` 表达式筛选。

典型写法：

```hcl
locals {
  backend_services = [
    for service in local.catalog.services : service
    if service.tier == "backend"
  ]
}
```

这段逻辑可以翻译成人话：

> 遍历 `local.catalog.services` 里的每个 service，只保留 `tier` 等于 `backend` 的服务。

然后如果我们确定只会有一个 backend 服务，可以继续取第一个：

```hcl
locals {
  backend_service = local.backend_services[0]
}
```

本实验中筛选出来的 backend 服务是：

```text
payments
```

## 4. 本实验在做什么

实验背景是：外部系统提供了一个服务目录 JSON。

Terraform 不需要整个目录，只需要：

- backend 服务名
- backend 服务的第一个端口
- backend 服务的端口数量

实验目标：

1. 读取 `data/catalog.json`
2. 使用 `jsondecode` 解码 JSON
3. 使用 `for` 表达式筛选 `tier == "backend"` 的服务
4. 读取 backend 服务的：
   - `name`
   - `ports[0]`
   - `length(ports)`
5. 输出摘要信息

预期结果：

```text
服务名：payments
第一个端口：8080
端口数量：2
```

---

## 5. 关键 Terraform 思路

可以把整个逻辑拆成 4 层。

### 第 1 层：读取 JSON 文件

```hcl
locals {
  catalog = jsondecode(file("${path.module}/data/catalog.json"))
}
```

这一层做“外部文本 → Terraform 数据结构”。

---

### 第 2 层：拿到服务列表

```hcl
locals {
  services = local.catalog.services
}
```

这一步是从 catalog 对象里取出 `services` 数组。

---

### 第 3 层：筛选 backend 服务

```hcl
locals {
  backend_services = [
    for service in local.services : service
    if service.tier == "backend"
  ]
}
```

这一步做筛选。

你可以把它理解成 Terraform 版的：

```text
services.filter(service => service.tier == "backend")
```

---

### 第 4 层：提取字段

```hcl
locals {
  backend_service = local.backend_services[0]
  backend_name    = local.backend_service.name
  first_port      = local.backend_service.ports[0]
  port_count      = length(local.backend_service.ports)
}
```

这里分别取：

- 服务名：`.name`
- 第一个端口：`.ports[0]`
- 端口数量：`length(...)`

## 8. 常见坑

### 坑 1：索引越界

如果筛选结果为空：

```hcl
local.backend_services[0]
```

就会报错。

因为根本没有第一个元素。

更稳的写法可以先确认长度：

```hcl
length(local.backend_services) > 0
```

在生产环境里，不要盲目假设筛选结果一定存在。

---

### 坑 2：字段名写错

JSON key 区分大小写。

如果 JSON 里是：

```json
"name": "payments"
```

你写成：

```hcl
service.Name
```

就会失败。

Terraform 不会帮你自动猜字段名。

---

### 坑 3：数字列表不能直接 `join`

如果端口是数字列表：

```hcl
[8080, 9090]
```

直接 `join(",", ports)` 可能会因为类型不匹配出问题。

更稳的写法是先转字符串：

```hcl
join(",", [for p in local.backend_service.ports : tostring(p)])
```

---

### 坑 4：外部 JSON 结构变了

如果 `catalog.json` 原来是：

```json
{
  "services": []
}
```

后来变成：

```json
{
  "items": []
}
```

那么：

```hcl
local.catalog.services
```

就会失败。

所以外部 JSON 最好配合：

- `terraform test`
- 变量校验
- CI 检查
- schema 文档

否则配置一变，Terraform 才发现已经晚了。

## 9. 生产环境建议

### 9.1 外部 JSON 要有版本

服务目录最好带版本字段，例如：

```json
{
  "schema_version": "1.0",
  "services": []
}
```

这样 Terraform 可以基于版本做兼容判断。

---

### 9.2 不要让 JSON 结构太随意

Terraform 很擅长处理结构稳定的数据。

如果 JSON 字段经常变化，IaC 代码会变得很脆弱。

---

### 9.3 对关键筛选逻辑写测试

比如本实验里应该测试：

- backend 服务必须存在
- backend 服务名符合预期
- ports 至少有一个端口
- 端口类型是 number
- 端口数量符合预期

这比只跑 `terraform validate` 更有价值。

---

### 9.4 对筛选结果做防御式处理

生产中可以用 `try()`、`can()`、`length()` 等函数降低爆炸风险。

例如：

```hcl
locals {
  backend_service_name = try(local.backend_services[0].name, null)
}
```

不过注意：  
`try()` 不是用来掩盖脏数据的。关键配置缺失时，最好让 CI 明确失败。

## 11. 一句话总结

`jsondecode` 让 Terraform 可以把外部 JSON 当作原生数据结构处理；对象用 `.key`，列表用 `[index]`，数组筛选用 `for ... if ...`。真正生产里，关键不是“能不能读 JSON”，而是筛选逻辑是否稳定、字段是否可靠、结构变化是否有测试兜底。
