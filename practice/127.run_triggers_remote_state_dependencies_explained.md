# Terraform 考试训练 127：Run Triggers 与 Remote State 依赖（讲解版）

## 1. 本节核心：读数据和触发运行是两回事

这一节非常重要，因为它考的是 HCP Terraform 里 workspace 之间的依赖关系。

核心一句话：

> Remote state 解决“下游怎么读取上游输出”；Run trigger 解决“上游变化后，下游什么时候重新运行”。

这两个能力经常一起出现，但它们不是一个东西。

可以这样理解：

```text
remote state data source = 读取别人的 output
run trigger              = 别人 apply 成功后，触发我重新 run
```

如果只用 remote state，没有 run trigger，下游确实能读上游输出。  
但问题是：上游变了以后，下游不会天然自动重新 plan。

这就是本节的考点。

---

## 2. 先看一个真实场景

假设你有两个 workspace：

```text
Workspace A: networking
Workspace B: security
```

`networking` 负责创建网络资源，比如：

- VPC / VNet
- Subnet
- Public IP
- NAT Gateway
- Route table

它输出一些值：

```hcl
output "public_ip_allowlist" {
  value = [
    "1.2.3.4/32",
    "5.6.7.8/32"
  ]
}
```

`security` workspace 读取这些输出，用来生成安全规则：

```hcl
data "terraform_remote_state" "networking" {
  backend = "remote"

  config = {
    organization = "example-org"

    workspaces = {
      name = "networking"
    }
  }
}
```

然后：

```hcl
resource "aws_security_group_rule" "allow_public_ips" {
  for_each = toset(data.terraform_remote_state.networking.outputs.public_ip_allowlist)

  type        = "ingress"
  cidr_blocks = [each.value]
}
```

看起来没问题，对吧？

但是坑来了。

---

## 3. Remote state 只负责读取，不负责触发

假设网络团队更新了 `networking` workspace：

```text
旧 public_ip_allowlist:
["1.2.3.4/32", "5.6.7.8/32"]

新 public_ip_allowlist:
["1.2.3.4/32", "5.6.7.8/32", "9.9.9.9/32"]
```

`networking` apply 成功后，state 里的 output 确实变了。

但 `security` workspace 会不会自动重新 plan？

答案是：

```text
不会天然自动重新 plan。
```

因为 remote state 的能力是：

```text
当 security workspace 运行时，它可以读取 networking workspace 的 output。
```

但它不负责：

```text
networking 一变，就自动让 security 运行。
```

所以你要区分：

```text
读取依赖 ≠ 运行触发
```

这句话就是本节灵魂。

---

## 4. Run trigger 是干什么的？

Run trigger 用来解决 workspace 之间的自动触发问题。

典型关系：

```text
上游 workspace apply 成功
-> 触发下游 workspace run
-> 下游 workspace 重新 plan
-> 下游读取最新 remote state output
-> 下游根据新数据更新资源
```

对应刚才的例子：

```text
networking apply 成功
-> 自动触发 security run
-> security 重新读取 networking outputs
-> security 更新安全规则
```

这就是 run trigger 的价值。

---

## 5. 上游和下游怎么判断？

你可以用数据流来判断。

如果 A 的 output 被 B 读取：

```text
A output -> B input
```

那 A 是上游，B 是下游。

例如：

```text
networking outputs subnet_id
application reads subnet_id
```

那么：

```text
networking = upstream workspace
application = downstream workspace
```

再比如：

```text
identity outputs role_arn
eks reads role_arn
```

那么：

```text
identity = upstream workspace
eks = downstream workspace
```

run trigger 通常配置成：

```text
上游 apply 成功后，触发下游 run。
```

---

## 6. Remote state 和 Run trigger 的职责分工

| 能力 | 解决的问题 | 不解决的问题 |
|---|---|---|
| Remote state data source | 读取另一个 Terraform state 暴露的 output | 不保证上游变化后下游自动运行 |
| Run trigger | 上游 workspace apply 成功后触发下游 run | 不负责传递具体数据内容 |
| 两者一起用 | 下游能被触发，并读取最新上游 output | 仍然需要合理设计 workspace 边界 |

一句话：

```text
Remote state 负责“拿数据”。
Run trigger 负责“叫醒下游”。
```

---

## 7. 为什么不能只靠 remote state？

因为 Terraform 不是实时监听系统。

它不会因为某个 state output 变化，就自动知道所有依赖这个 output 的 workspace 都该重新执行。

如果没有 run trigger，就可能出现：

```text
上游已经更新
下游还没运行
下游资源仍然基于旧输出
```

例如：

```text
networking 新增了公网 IP
security 没有重新 plan
安全组规则没有加入新 IP
应用访问失败
```

这时候不是 remote state 读不到数据，而是下游根本没被触发运行。

等你手动运行下游 workspace 时，它才会读到最新 output。

---

## 8. Run trigger 不是替代 remote state

这是考试很容易挖坑的地方。

Run trigger 不是用来传数据的。

它只是触发下游 workspace 运行。

数据仍然需要通过：

- remote state
- variables
- HCP Terraform workspace outputs
- 其他数据源
- provider data source

来读取。

所以正确理解是：

```text
remote state data source + run trigger
```

而不是：

```text
run trigger 替代 remote state
```

如果题目问：

> 下游 workspace 如何读取上游 workspace 的 output？

答案是：

```text
remote state data source
```

如果题目问：

> 上游 workspace apply 后，如何自动触发下游 workspace 重新 plan？

答案是：

```text
run triggers
```

这两个题眼不一样。

---

## 9. 为什么不要把所有配置合并成一个巨大 workspace？

有人可能会说：

> 既然 workspace 之间有依赖，那我把网络、安全、应用全部放一个 workspace，不就没有触发顺序问题了吗？

这思路看似省事，实际很容易变成灾难。

巨大 workspace 的问题：

```text
1. blast radius 变大
2. plan 噪音变多
3. apply 风险变高
4. 权限边界不清晰
5. 团队职责混在一起
6. state 文件越来越大
7. 排障困难
8. 变更审批粒度太粗
9. 小改动也可能影响一大片资源
10. 不利于平台团队和应用团队分工
```

比如你只是想改一个安全组规则，但 plan 里出现网络、数据库、Kubernetes、IAM、应用几十类资源。  
这时候 review 质量一定下降。

正确方向通常是：

```text
合理拆分 workspace
显式暴露 output
下游通过 remote state 读取
通过 run trigger 管理触发依赖
```

不是把所有东西塞进一个超级 workspace。

---

## 10. Blast radius 是什么？

Blast radius 可以理解为“爆炸半径”。

在 Terraform 里，就是一次错误变更可能影响多大范围。

如果你把所有资源放进一个 workspace：

```text
network
security
database
kubernetes
application
monitoring
iam
```

一次错误 apply 可能影响所有层。

如果你拆分 workspace：

```text
networking workspace
security workspace
database workspace
kubernetes workspace
application workspace
```

每个 workspace 的影响范围更清楚。

这就是平台工程里的基本思想：

```text
拆分边界不是为了麻烦，而是为了控制风险。
```

## 11. 典型 workspace 依赖链

一个真实项目可能长这样：

```text
networking
    ↓
security
    ↓
eks / aks
    ↓
application
    ↓
monitoring
```

或者：

```text
identity
    ↓
kubernetes
    ↓
addons
    ↓
apps
```

再比如：

```text
dns
    ↓
load-balancer
    ↓
application
```

每条箭头表示：

```text
下游读取上游 outputs
上游 apply 后可能需要触发下游 run
```

run trigger 就是在管理这些箭头背后的执行顺序。

## 12. 考试选择题解析

## 题目 1

Run trigger 主要解决什么问题？

```text
A. 上游 workspace 变更后，自动触发依赖 workspace 的 run
B. 格式化 Terraform 文件
C. 下载 Docker 镜像
D. 创建本地 SSH key
```

答案：

```text
A
```

解析：

Run trigger 的核心用途就是在 workspace 之间建立运行触发关系。  
上游 workspace apply 成功后，可以触发下游 workspace 的 run。

---

## 题目 2

Remote state data source 的作用是什么？

```text
A. 读取另一个 Terraform state 暴露的输出
B. 自动阻断所有 apply
C. 替代所有变量
D. 管理浏览器缓存
```

答案：

```text
A
```

解析：

Remote state data source 用来读取另一个 Terraform state 中暴露的 output。  
它不负责审批，也不负责自动触发下游 apply。

## 题目 3

如果下游配置依赖上游输出，但上游变更后下游没有自动运行，应该考虑什么 HCP Terraform 功能？

```text
A. run triggers
B. terraform fmt
C. provider lock file
D. terraform graph
```

答案：

```text
A
```

解析：

这里的问题不是格式化，不是 provider 版本锁定，也不是画依赖图。  
问题是上游变化后下游没有自动 run，所以应该考虑 run triggers。

## 14. 场景题 1：网络输出 IP，安全读取 IP

题目：

> 网络 workspace 输出公网 IP 列表，安全 workspace 读取这个列表生成安全规则。网络团队新增 IP 后，安全 workspace 应该自动重新 plan。你会使用什么能力？

推荐答案：

```text
使用 run triggers。
```

完整解释：

```text
网络 workspace 是上游，安全 workspace 是下游。安全 workspace 可以通过 remote state data source 读取网络 workspace 输出的公网 IP 列表。但 remote state 只负责读取数据，不会自动触发下游运行。因此应配置 run trigger，让网络 workspace apply 成功后触发安全 workspace 的 run，使安全 workspace 重新读取最新输出并重新 plan。
```

一句话版：

```text
remote state 负责读 IP，run trigger 负责网络变更后叫醒安全 workspace。
```

---

## 15. 场景题 2：为了避免顺序问题，把所有配置放一个 workspace？

题目：

> 有人建议把网络、安全、应用所有 Terraform 配置都放进一个 workspace，只为避免触发顺序问题。请说明这种方案可能带来的问题。

推荐答案：

```text
不推荐。把所有配置合并进一个巨大 workspace 会扩大 blast radius，降低职责边界清晰度，增加 plan 噪音和权限管理难度。网络、安全、应用通常属于不同职责边界，应该合理拆分 workspace，并通过 remote state 暴露必要输出，通过 run triggers 显式管理上游到下游的运行依赖。
```

可以展开成：

```text
巨大 workspace 的问题包括：
1. 一次错误 apply 影响范围更大。
2. plan 输出更复杂，review 成本更高。
3. 不同团队权限难以隔离。
4. state 文件更大，排障更困难。
5. 小变更也可能牵连大量无关资源。
```

## 16. run trigger 的典型使用条件

看到下面条件，可以考虑 run trigger：

```text
1. 有多个 HCP Terraform workspace。
2. 下游 workspace 依赖上游 workspace 的 output。
3. 上游 output 变化后，下游需要重新 plan。
4. 希望自动触发，而不是靠人记得手动运行。
```

典型关键词：

```text
upstream workspace
downstream workspace
remote state output
dependency
automatically trigger
after successful apply
re-run downstream
```

考试看到这些，基本就是 run trigger。

---

## 17. 不适合 run trigger 的场景

不是所有问题都要用 run trigger。

下面这些不是 run trigger 解决的问题：

```text
terraform fmt 格式化失败
provider 下载失败
变量写错
backend 初始化失败
本地状态损坏
资源之间同一个 workspace 内部依赖
```

同一个 workspace 里的资源依赖，Terraform 自己会通过 dependency graph 处理：

```hcl
resource "aws_subnet" "example" {
  vpc_id = aws_vpc.main.id
}
```

这种不需要 run trigger。

Run trigger 是 workspace 和 workspace 之间的运行触发，不是同一个配置内部的资源依赖。

---

## 18. run trigger 和 terraform graph 的区别

`terraform graph` 可以展示当前配置内部的资源依赖图。

但它不解决 HCP Terraform workspace 之间的自动触发问题。

区别：

| 工具 / 能力 | 作用 |
|---|---|
| `terraform graph` | 查看当前配置内部资源依赖 |
| `remote state` | 读取另一个 state 的 outputs |
| `run trigger` | 上游 workspace apply 后触发下游 workspace run |

所以如果题目是：

> 想知道当前配置内部资源依赖关系。

可以考虑：

```text
terraform graph
```

如果题目是：

> 上游 workspace apply 后自动触发下游 workspace。

答案是：

```text
run trigger
```

---

## 19. 生产环境设计建议

使用 run triggers 时，不要乱拉依赖链。

建议：

```text
1. 只暴露必要 output
2. 不要把整个对象都输出
3. 避免循环依赖
4. 保持上游 / 下游关系清晰
5. 不要让太多 workspace 链式触发
6. 对生产 apply 保留审批
7. 对关键 workspace 做权限隔离
8. run trigger 配置也要经过 review
```

尤其要避免这种依赖：

```text
A triggers B
B triggers C
C triggers A
```

这会造成循环依赖和排障地狱。

更好的结构是有清晰方向：

```text
foundation -> platform -> application
```

例如：

```text
networking -> security -> cluster -> addons -> apps
```

方向清楚，团队才好维护。

---

## 20. remote state 输出设计建议

上游 workspace 不要乱输出。

推荐输出稳定、必要、低敏感的数据：

```hcl
output "vpc_id" {
  value = aws_vpc.main.id
}

output "private_subnet_ids" {
  value = aws_subnet.private[*].id
}

output "public_ip_allowlist" {
  value = var.public_ip_allowlist
}
```

不推荐输出：

```text
secret
password
token
private key
过大的复杂对象
内部实现细节
```

Remote state output 会成为跨 workspace 合同。  
一旦下游依赖它，上游随便改 output 名称或结构，就会影响下游。

所以 output 也要像 API 一样设计，不能乱改。

---

## 21. run trigger 的风险

Run trigger 很方便，但也有风险：

```text
1. 上游频繁变更导致下游频繁 plan
2. 依赖链太长导致 run 风暴
3. 触发关系不清楚导致排障困难
4. 下游 plan 自动触发后没人 review
5. 错误 output 传播到多个 workspace
```

所以生产里要配合：

- 明确依赖图
- PR review
- run approval
- workspace 权限
- policy check
- 通知机制
- 变更窗口

不要把 run trigger 当成“自动化越多越好”。  
自动化的前提是边界清楚，不然只是自动制造事故。

## 24. 和真实平台工程的关系

这节非常接近真实平台工程。

平台团队经常会把基础设施拆成多个层：

```text
foundation
networking
security
identity
kubernetes
observability
applications
```

每一层都有不同团队负责，也有不同权限和变更频率。

如果全部放一个 workspace，风险太大。  
如果拆分 workspace 但不管理依赖，下游又可能不及时更新。

所以真实做法是：

```text
合理拆 workspace
稳定输出 output
remote state 读取依赖数据
run trigger 管理重新运行
审批策略控制生产 apply
```

这就是 HCP Terraform 里比较成熟的 workspace 依赖管理方式。

## 25. 最终总结

本节最重要的三句话：

```text
1. Remote state 解决“下游如何读取上游输出”。
2. Run trigger 解决“上游 apply 成功后，如何自动触发下游 run”。
3. 不要为了避免触发顺序问题，把所有配置合并成一个巨大 workspace。
```

考试遇到这类题，先看题眼：

```text
读另一个 state 的 output？选 remote state。
上游变化后触发下游运行？选 run triggers。
避免依赖就全塞一个 workspace？通常不推荐。
```

一句话收尾：

```text
Remote state 是数据线，run trigger 是闹钟；数据线负责传值，闹钟负责叫下游起来重新算。
```
