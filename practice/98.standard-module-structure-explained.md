# Terraform 实操训练 98：标准模块结构（讲解版）

> 来源：`98.md`  
> 主题：标准模块结构  
> 实验路径：`practice/labs/98`

---

## 1. 这一节到底在学什么？

这一节不是单纯背 Terraform 的文件名，而是让你理解：**一个 Terraform module 应该如何组织，才能被别人稳定复用、测试、维护和升级。**

在生产环境里，Terraform 不只是写几个 `.tf` 文件跑通就完事了。真正有价值的是：

- 目录结构清楚，别人一眼知道入口在哪里；
- 输入变量、输出值、provider 约束、版本约束分工明确；
- module 可以被 root module 调用，也可以被测试；
- 后续重构时，结构不会把状态管理搞乱；
- CI/CD 里可以统一跑 `fmt / validate / test / plan`。

所以，“标准模块结构”本质上是 Terraform 工程化的基本功。

---

## 2. 先建立一个核心概念：module 不是高级功能，而是 Terraform 的基本组织单位

Terraform 里每一个包含 `.tf` 文件的目录，本质上都可以看成一个 module。

常见有两类：

### 2.1 Root module

你直接执行命令的目录，就是 root module。

例如本实验里：

```sh
cd /workspace/practice/labs/98
terraform init
terraform plan
terraform apply
```

这里的 `practice/labs/98` 就是当前执行入口，也就是 root module。

### 2.2 Child module

被另一个 module 调用的 module，就是 child module。

典型写法：

```hcl
module "network" {
  source = "./modules/network"

  name = "demo"
}
```

这里 `./modules/network` 就是 child module。

root module 更像“总装车间”，负责把不同模块拼起来；child module 更像“标准零件”，负责封装可复用能力。

---

## 3. 标准模块结构长什么样？

Terraform 官方和社区里常见的模块结构，大概会拆成这些文件：

```text
module-name/
├── main.tf
├── variables.tf
├── outputs.tf
├── versions.tf
├── README.md
├── tests/
│   └── *.tftest.hcl
└── examples/
    └── basic/
        └── main.tf
```

不是每个实验都必须完整包含这些目录，但生产环境里建议尽量保持这种习惯。

---

## 4. 每个文件应该放什么？

### 4.1 `main.tf`：主要资源和核心逻辑

`main.tf` 通常放 module 的主要资源声明。

例如：

```hcl
resource "local_file" "example" {
  filename = "${path.module}/output/example.txt"
  content  = var.content
}
```

你可以把它理解成：**这个模块真正要创建什么东西。**

不要把所有变量、输出、版本约束都塞进 `main.tf`。能跑是最低要求，能维护才是工程要求。

---

### 4.2 `variables.tf`：模块对外暴露的输入参数

`variables.tf` 放变量定义。

例如：

```hcl
variable "content" {
  type        = string
  description = "Content written into the generated file."
}
```

它回答的问题是：

> 使用这个 module 的人，需要传什么进来？

变量定义要尽量写清楚：

- 类型是什么；
- 是否有默认值；
- 这个变量代表什么；
- 有没有校验规则；
- 是否 sensitive。

一个模块的 `variables.tf` 写得好不好，基本决定了别人愿不愿意复用它。

---

### 4.3 `outputs.tf`：模块对外暴露的结果

`outputs.tf` 放输出值。

例如：

```hcl
output "file_path" {
  value       = local_file.example.filename
  description = "Path of the generated file."
}
```

它回答的问题是：

> 这个 module 执行完以后，外部还能拿到什么信息？

比如网络模块可能输出：

- VPC ID；
- subnet IDs；
- route table IDs；
- security group ID。

AKS/EKS 模块可能输出：

- cluster name；
- kubeconfig 相关信息；
- node resource group；
- OIDC issuer URL。

输出值不是越多越好。输出太少，调用方不好用；输出太多，模块内部实现会被外部过度依赖。

---

### 4.4 `versions.tf`：Terraform 和 provider 版本约束

`versions.tf` 一般放：

```hcl
terraform {
  required_version = ">= 1.11"

  required_providers {
    local = {
      source  = "hashicorp/local"
      version = "~> 2.5"
    }
  }
}
```

它解决的是可重复性问题。

没有版本约束，今天能跑，明天 provider 升级后就可能炸。生产环境里这类问题很恶心，因为它不是你的代码逻辑变了，而是依赖变化导致结果变化。

所以版本约束和 `.terraform.lock.hcl` 是 Terraform 工程稳定性的基础。

---

### 4.5 `README.md`：给人看的使用说明

`README.md` 不只是摆设，它应该说明：

- 这个 module 用来做什么；
- 最小使用示例；
- 输入变量；
- 输出值；
- 运行方式；
- 注意事项；
- 限制条件。

一个没有 README 的 module，就像一个没有说明书的工具。作者自己能用，别人不一定敢用。

---

### 4.6 `tests/`：Terraform 原生测试

Terraform 的测试文件一般放在 `tests/` 下，扩展名通常是：

```text
*.tftest.hcl
```

测试的价值是让 module 不只是“能 apply”，而是可以自动证明某些行为是对的。

例如你可以测试：

- 输出值是否符合预期；
- 变量传入后资源命名是否正确；
- 错误输入是否会被拒绝；
- module 的基本 contract 是否稳定。

本实验验证报告里提到：`terraform test` 返回 `1 passed, 0 failed`，说明至少有一个测试断言通过。

---

### 4.7 `examples/`：给使用者看的最小可运行例子

成熟模块通常会有 `examples/` 目录，例如：

```text
examples/
└── basic/
    └── main.tf
```

它的作用不是测试，而是给人复制参考。

README 里写得再清楚，也不如一个能直接运行的例子有说服力。

---

## 5. 这一节为什么重要？

因为 Terraform 项目很容易从“小脚本”变成“祖传大泥球”。

一开始你可能只有一个 `main.tf`：

```text
main.tf
```

后来你加了变量：

```text
main.tf
variables.tf
```

再后来你加了输出：

```text
main.tf
variables.tf
outputs.tf
```

再后来你加了 provider 约束、测试、模块、环境目录、CI/CD：

```text
main.tf
variables.tf
outputs.tf
versions.tf
tests/
modules/
envs/
.github/workflows/
```

如果一开始没有结构意识，后面很快就会变成：

```text
main.tf
main2.tf
new-main.tf
final-main.tf
fix-final-main.tf
do-not-touch.tf
```

这就很像真实公司的基础设施仓库：不是不会写 Terraform，而是没人敢改。

标准模块结构的意义就是：**让基础设施代码可以长期演进，而不是只能靠作者本人记忆维护。**

## 6. 本实验应该怎么跑？

根据原始实验，进入目录后依次执行：

```sh
cd /workspace/practice/labs/98
terraform init -input=false
terraform fmt -check
terraform validate
terraform test
terraform plan -input=false -no-color -out=tfplan
terraform apply -auto-approve tfplan
terraform output
terraform destroy -auto-approve
```

下面逐条解释。

---

## 7. 每条命令在检查什么？

### 7.1 `terraform init -input=false`

初始化工作目录。

它会做几件事：

- 下载 provider；
- 初始化 backend；
- 初始化 module；
- 生成或更新 `.terraform.lock.hcl`。

`-input=false` 表示不要在命令行里交互式提问。这个参数在 CI/CD 里很重要，因为流水线不能像人一样回答问题。

---

### 7.2 `terraform fmt -check`

检查格式是否符合 Terraform 标准风格。

注意，它只是检查，不会自动修复。

如果失败，可以本地执行：

```sh
terraform fmt -recursive
```

然后再重新检查。

生产环境里 `fmt -check` 很常见，因为团队不应该在代码格式上浪费 review 时间。

---

### 7.3 `terraform validate`

检查配置语法和基本结构是否合法。

它能发现：

- HCL 语法错误；
- provider 配置问题；
- 变量引用错误；
- resource block 写法错误。

但是它不等于真正部署。`validate` 通过，只能说明代码基本合法，不代表云资源一定能创建成功。

---

### 7.4 `terraform test`

运行 Terraform 原生测试。

这一节里测试结果是：

```text
1 passed, 0 failed
```

这说明测试文件中的断言通过了。

你可以把 `terraform test` 理解成 Terraform module 的单元测试入口。以后做模块化工程，尤其是准备 Terraform Professional，这个命令很值得熟悉。

---

### 7.5 `terraform plan -input=false -no-color -out=tfplan`

生成执行计划，并保存到 `tfplan` 文件。

重点是 `-out=tfplan`。

这表示后面的 `apply` 会使用同一个已保存计划，而不是重新生成一份 plan。

在生产环境里，这是非常重要的流程：

```text
plan → review / approval → apply the saved plan
```

这样可以避免“review 的 plan”和“实际 apply 的 plan”不是同一份。

---

### 7.6 `terraform apply -auto-approve tfplan`

应用刚才保存的计划。

这里传入的是 `tfplan` 文件，所以它执行的是前一步已经生成好的计划。

`-auto-approve` 表示跳过人工确认。实验环境里可以这样用，但生产环境通常会配合审批流程，而不是随便自动 apply。

---

### 7.7 `terraform output`

查看输出值。

它可以帮你确认 module 最终暴露了哪些结果。

如果这是一个网络模块，你可能会看到 subnet id；如果是文件实验，你可能会看到生成文件路径或内容摘要。

---

### 7.8 `terraform destroy -auto-approve`

销毁资源。

实验结束后必须清理，避免状态和资源残留影响下一次运行。

---

## 8. 这节课的 Terraform Professional 考点味道

这一节非常像 Terraform Professional 考试里的工程化基础题。

你需要掌握的不只是“文件叫什么”，而是背后的判断：

### 8.1 为什么要拆 `variables.tf` 和 `outputs.tf`？

因为 module 的输入输出就是 contract。

- `variables.tf` 是调用者要给模块什么；
- `outputs.tf` 是模块会返还什么。

这和写后端 API 很像：

- request 是输入；
- response 是输出；
- 内部实现不要随便暴露。

---

### 8.2 为什么要有 `versions.tf`？

因为 Terraform 是声明式工具，但 provider 是会升级的。

没有版本约束，执行结果可能不稳定。

在团队协作里，版本约束就是防止“你电脑能跑，我电脑炸了”的第一道防线。

---

### 8.3 为什么要有 `tests/`？

因为 module 复用之后，影响范围会变大。

一个网络 module 被 10 个环境调用，如果你改坏了，炸的不是一个实验，而是一串环境。

测试就是提前拦截这种问题。

---

### 8.4 为什么要保存 plan？

因为生产环境需要可审计流程。

如果只是：

```sh
terraform apply
```

Terraform 会现场生成计划并执行。

如果是：

```sh
terraform plan -out=tfplan
terraform apply tfplan
```

那就可以做到：

```text
生成计划 → 人工/系统审批 → 执行同一份计划
```

这就是 IaC 上生产的基本姿势。

---

## 9. 常见错误和排障思路

### 9.1 `terraform fmt -check` 失败

说明格式不标准。

处理方式：

```sh
terraform fmt -recursive
```

然后重新执行：

```sh
terraform fmt -check
```

---

### 9.2 `terraform test` 失败

优先看测试断言。

常见原因：

- output 名称写错；
- assert 条件不对；
- 测试里引用了不存在的 local；
- 期望值和真实值不一致；
- plan 阶段拿不到 apply 后才知道的值。

原始文件里也提到了一个典型坑：测试框架实验要避免引用未声明的 local。

---

### 9.3 `plan/apply` 行为不一致

常见原因是你在 plan 阶段判断了 apply 后才知道的值。

比如某些云资源 ID、随机值、动态生成的属性，在 plan 阶段可能是 unknown。

解决思路：

- 区分 plan-time 和 apply-time；
- 不要在 plan 测试里断言 apply 后才确定的值；
- 必要时使用 apply 类型的测试运行。

---

### 9.4 `moved` block 写错

原始验证报告里提到：`moved` block 需要写成真实 HCL block。

正确形式类似：

```hcl
moved {
  from = local_file.old
  to   = local_file.new
}
```

不要把它写成字符串、注释或伪代码。Terraform 只认真实 HCL block。

---

## 10. 如何把这节应用到你的 AKS/EKS 项目？

你之后做 AKS/EKS、平台工程、SRE 项目时，模块结构可以这样设计：

```text
infra/
├── envs/
│   ├── dev/
│   │   └── main.tf
│   ├── staging/
│   │   └── main.tf
│   └── prod/
│       └── main.tf
├── modules/
│   ├── aks-cluster/
│   │   ├── main.tf
│   │   ├── variables.tf
│   │   ├── outputs.tf
│   │   ├── versions.tf
│   │   ├── README.md
│   │   └── tests/
│   ├── eks-cluster/
│   ├── node-pool/
│   └── observability/
└── README.md
```

这里有几个关键点：

- `envs/` 放不同环境的 root module；
- `modules/` 放可复用 child module；
- 每个 module 都保持标准结构；
- 每个 module 都有 README 和 tests；
- CI/CD 对每个目录执行 `fmt / validate / test / plan`。

这样你的项目看起来就不是“我会 Terraform 命令”，而是“我能把 Terraform 当工程系统来管理”。

这对 Platform Engineering / SRE 方向非常加分。

## 11. 一句话记忆

标准模块结构可以这样记：

> `main.tf` 写资源，`variables.tf` 写输入，`outputs.tf` 写输出，`versions.tf` 锁版本，`README.md` 教别人怎么用，`tests/` 证明它没坏。

再压缩一点：

> **资源、输入、输出、版本、说明、测试。**

## 12. 本节复盘问题

### 问题 1：为什么不能所有内容都写进 `main.tf`？

因为短期能跑，长期难维护。变量、输出、版本约束、测试和说明混在一起，会让模块边界变得混乱。

---

### 问题 2：`variables.tf` 和 `outputs.tf` 的本质是什么？

它们定义 module 的 contract。

- `variables.tf` 是输入 contract；
- `outputs.tf` 是输出 contract。

---

### 问题 3：`terraform validate` 和 `terraform test` 有什么区别？

`validate` 检查配置是否合法；`test` 检查模块行为是否符合预期。

一个偏语法和结构，一个偏行为和断言。

---

### 问题 4：为什么生产环境常用 saved plan？

因为可以先生成计划，再审查，再执行同一份计划，避免 apply 时偷偷变成另一份计划。

---

### 问题 5：这节和真实工作有什么关系？

真实工作里，Terraform 最大的问题往往不是“不会写 resource”，而是：

- 模块不可复用；
- 状态迁移混乱；
- 输入输出不清楚；
- 没测试；
- 没 README；
- CI/CD 流程不稳定。

标准模块结构就是解决这些问题的第一步。

---

## 13. 小结

这一节的核心不是文件名，而是 Terraform 工程化思维。

你要形成的判断是：

- module 是 Terraform 的复用单元；
- 标准结构让 module 可读、可测、可维护；
- 输入输出就是模块 contract；
- 版本约束保证稳定性；
- 测试保证模块行为不被改坏；
- saved plan 是生产环境审批和审计的基础。

学完这一节，你应该能看懂一个 Terraform module 的基本骨架，也能开始设计自己的 AKS/EKS、网络、监控、FinOps 等模块结构。
