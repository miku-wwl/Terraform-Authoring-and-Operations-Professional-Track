# 第 169 节讲解版：AWS CLI profile、source_profile 与 provider alias

> 来源：`doc/169.txt` / 原始训练文件 `169.md`  
> 主题：AWS CLI profile、`source_profile`、AssumeRole 链路、Terraform provider alias  
> 环境：LocalStack / 本地可验证练习  
> 分类：`aws-localstack-limited`

---

## 1. 这一节到底在练什么？

这一节是综合挑战六的解法讲解，核心不是“记住某个 AWS 参数”，而是训练你在 Terraform 里处理 **多身份、多账号、多 provider 配置** 的能力。

真实 AWS 项目里，经常会出现这种情况：

- 本地 AWS CLI 有多个 profile；
- 一个 profile 使用静态凭证；
- 另一个 profile 通过 `source_profile` 去 AssumeRole；
- Terraform 需要用不同身份操作不同资源；
- 同一个 AWS provider 需要配置多个别名，也就是 `provider alias`。

考试和实战里，最容易混的是：

> AWS CLI profile 负责“我是谁、怎么拿凭证”；  
> Terraform provider 负责“Terraform 用哪个身份去调用 AWS API”；  
> provider alias 负责“同一个 provider 配多份，用于不同区域、账号或角色”。

---

## 2. 关键概念一：AWS CLI profile

AWS CLI profile 是本地 AWS CLI 配置里的一个命名身份。

常见位置：

```text
~/.aws/credentials
~/.aws/config
```

例如：

```ini
# ~/.aws/credentials
[default]
aws_access_key_id = test
aws_secret_access_key = test

[dev]
aws_access_key_id = test
aws_secret_access_key = test
```

在真实 AWS 里，profile 可能对应真实 IAM User、SSO 登录、AssumeRole 链路等。

在本项目里，为了避免误操作真实云账号，所有练习都指向 LocalStack：

```text
endpoint: http://localhost:4566
access key: test
secret key: test
```

所以这里的重点不是验证 AWS 真权限，而是理解 Terraform 配置路径。

---

## 3. 关键概念二：source_profile

`source_profile` 常用于 AssumeRole 场景。

它的意思是：

> 当前 profile 不直接持有长期凭证，而是先使用另一个 profile 的凭证，再去 AssumeRole 获得临时凭证。

典型配置长这样：

```ini
# ~/.aws/config
[profile admin]
region = ap-southeast-2

[profile audit]
role_arn = arn:aws:iam::123456789012:role/AuditRole
source_profile = admin
region = ap-southeast-2
```

含义是：

1. `audit` profile 自己没有 access key；
2. 它先读取 `admin` profile 的凭证；
3. 再用 `admin` 的身份调用 STS AssumeRole；
4. 最后拿到 `AuditRole` 的临时凭证；
5. Terraform 或 AWS CLI 再用这个临时凭证访问资源。

一句话：

> `source_profile` 是“借谁的初始身份去换角色”。

---

## 4. 关键概念三：AssumeRole

AssumeRole 是 AWS STS 的能力。

它解决的问题是：

> 不要到处发长期 access key，而是让一个身份临时扮演另一个 role。

真实 AWS 中，AssumeRole 是否成功取决于两边：

### 4.1 调用方权限 policy

调用方必须被允许执行：

```text
sts:AssumeRole
```

### 4.2 目标 Role 的 trust policy

目标 Role 必须信任调用方。

例如：

```json
{
  "Effect": "Allow",
  "Principal": {
    "AWS": "arn:aws:iam::111111111111:user/admin"
  },
  "Action": "sts:AssumeRole"
}
```

这两个条件缺一不可。

考试里经常会出现这种坑：

> A 账号用户有 `sts:AssumeRole` 权限，但 B 账号 Role 的 trust policy 没信任它，所以还是不能 AssumeRole。

你可以理解为：

- permission policy：你有没有资格敲门；
- trust policy：对方愿不愿意给你开门。

## 5. 关键概念四：Terraform provider alias

默认情况下，Terraform 里的 provider 是一份配置：

```hcl
provider "aws" {
  region = "ap-southeast-2"
}
```

所有没有显式指定 provider 的 AWS resource，都会使用这份默认 provider。

但如果你要同时操作多个区域、多个账号、多个身份，就需要 alias。

例如：

```hcl
provider "aws" {
  region = "ap-southeast-2"
}

provider "aws" {
  alias  = "audit"
  region = "ap-southeast-2"

  profile = "audit"
}
```

然后资源里这样指定：

```hcl
resource "aws_s3_bucket" "audit_logs" {
  provider = aws.audit

  bucket = "example-audit-logs"
}
```

这里的重点是：

> `provider = aws.audit` 表示这个资源不用默认 AWS provider，而是使用 alias 为 `audit` 的那份 provider 配置。

---

## 6. profile 和 provider alias 的关系

很多人会把它们混在一起。

它们其实是两层东西：

| 层级 | 负责什么 | 例子 |
|---|---|---|
| AWS CLI profile | 凭证来源、AssumeRole 链路、region 默认值 | `profile = "audit"` |
| Terraform provider | Terraform 调用 AWS API 的配置入口 | `provider "aws" { ... }` |
| provider alias | 同一个 provider 的多份配置 | `alias = "audit"` |
| resource provider meta-argument | 指定某个资源使用哪份 provider | `provider = aws.audit` |

也就是说：

```hcl
provider "aws" {
  alias   = "audit"
  profile = "audit"
  region  = "ap-southeast-2"
}
```

这段里面有两个 `audit`，但含义不同：

- `alias = "audit"`：Terraform 内部给这份 provider 配置起的名字；
- `profile = "audit"`：AWS SDK 去本地 AWS 配置里找的 profile 名字。

它们可以同名，也可以不同名。

例如下面也合法：

```hcl
provider "aws" {
  alias   = "security_account"
  profile = "audit"
  region  = "ap-southeast-2"
}
```

这时 Terraform 里引用它要写：

```hcl
provider = aws.security_account
```

而不是 `aws.audit`。

## 7. LocalStack 里的练习边界

本节分类是：

```text
aws-localstack-limited
```

意思是：

> 可以用 LocalStack 完成 Terraform 配置、plan/apply/state/output/destroy 的闭环，但不能完整复刻真实 AWS 的 IAM 权限评估和 STS 行为。

LocalStack 适合练：

- provider 多实例配置；
- alias 写法；
- resource 指定 provider；
- Terraform 状态变化；
- output 设计；
- validate / plan / apply / destroy 流程。

LocalStack 不适合完全验证：

- 真实 AWS IAM 权限边界；
- 真实 AssumeRole trust policy 是否精确生效；
- AWS Organizations 跨账号场景；
- SSO 登录链路；
- 真实 STS 临时凭证行为。

所以这节的正确学习姿势是：

> 用 LocalStack 练 Terraform 写法，用真实 AWS 文档理解 IAM 行为边界。

## 8. 典型 Terraform 写法示例

下面是一个教学版结构，不代表你的项目必须完全照抄。

### 8.1 provider 配置

```hcl
terraform {
  required_version = ">= 1.6.0"

  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 5.0"
    }
  }
}

provider "aws" {
  region                      = "ap-southeast-2"
  access_key                  = "test"
  secret_key                  = "test"
  s3_use_path_style           = true
  skip_credentials_validation = true
  skip_metadata_api_check     = true
  skip_requesting_account_id  = true

  endpoints {
    s3  = "http://localhost:4566"
    iam = "http://localhost:4566"
    sts = "http://localhost:4566"
  }
}

provider "aws" {
  alias                       = "audit"
  region                      = "ap-southeast-2"
  access_key                  = "test"
  secret_key                  = "test"
  s3_use_path_style           = true
  skip_credentials_validation = true
  skip_metadata_api_check     = true
  skip_requesting_account_id  = true

  endpoints {
    s3  = "http://localhost:4566"
    iam = "http://localhost:4566"
    sts = "http://localhost:4566"
  }
}
```

这里有两份 AWS provider：

- 默认 provider：`aws`
- alias provider：`aws.audit`

---

### 8.2 使用默认 provider 创建资源

```hcl
resource "aws_s3_bucket" "main" {
  bucket = "main-bucket"
}
```

没有写 `provider = ...`，所以默认使用：

```hcl
provider "aws" { ... }
```

---

### 8.3 使用 alias provider 创建资源

```hcl
resource "aws_s3_bucket" "audit" {
  provider = aws.audit

  bucket = "audit-bucket"
}
```

这里显式写了：

```hcl
provider = aws.audit
```

所以它使用 alias provider。

---

## 9. module 中使用 provider alias

综合挑战里很可能会把 provider alias 和 module 结合起来考。

例如 root module 里：

```hcl
provider "aws" {
  region = "ap-southeast-2"
}

provider "aws" {
  alias  = "audit"
  region = "ap-southeast-2"
}

module "audit_bucket" {
  source = "./modules/bucket"

  providers = {
    aws = aws.audit
  }

  bucket_name = "audit-bucket"
}
```

这里的重点是：

```hcl
providers = {
  aws = aws.audit
}
```

含义是：

> 子模块里的默认 `aws` provider，实际由 root module 的 `aws.audit` 注入。

子模块里通常不用写 alias：

```hcl
resource "aws_s3_bucket" "this" {
  bucket = var.bucket_name
}
```

它看到的是普通 `aws` provider，但这个 provider 是 root module 传进来的 `aws.audit`。

---

## 10. 常见错误一：定义了 alias，但资源没用它

错误示例：

```hcl
provider "aws" {
  alias  = "audit"
  region = "ap-southeast-2"
}

resource "aws_s3_bucket" "audit" {
  bucket = "audit-bucket"
}
```

问题：

```hcl
resource "aws_s3_bucket" "audit"
```

没有写：

```hcl
provider = aws.audit
```

所以它仍然使用默认 provider，而不是 alias provider。

修正：

```hcl
resource "aws_s3_bucket" "audit" {
  provider = aws.audit

  bucket = "audit-bucket"
}
```

---

## 11. 常见错误二：把 profile 名和 alias 名混淆

错误理解：

> `profile = "audit"` 之后，资源就可以写 `provider = aws.audit`。

不一定。

只有当 provider 里明确写了：

```hcl
alias = "audit"
```

资源才可以写：

```hcl
provider = aws.audit
```

下面这个配置不能用 `aws.audit`：

```hcl
provider "aws" {
  profile = "audit"
  region  = "ap-southeast-2"
}
```

因为它没有 alias，它只是默认 provider。

---

## 12. 常见错误三：以为 LocalStack 能完整验证 AssumeRole

LocalStack 可以帮助你完成 Terraform 练习闭环，但不要把 LocalStack 的通过理解成真实 AWS 权限也一定通过。

真实 AWS 里，AssumeRole 会受到这些影响：

- IAM identity policy；
- IAM role trust policy；
- permission boundary；
- SCP；
- session policy；
- MFA 条件；
- external ID；
- region 和 endpoint；
- SSO / OIDC / web identity 登录链路。

考试题如果问真实 AWS 权限问题，要按 AWS IAM 规则判断，不能按 LocalStack 简化行为判断。

## 13. 解题流程：遇到 provider alias 题怎么拆？

综合挑战里，建议按这个顺序做。

### 第一步：读题，确认身份数量

先问自己：

- 题目里有几个账号？
- 有几个角色？
- 有几个 region？
- 有几个 profile？
- 哪些资源要用默认身份？
- 哪些资源要用特殊身份？

---

### 第二步：画出身份链路

例如：

```text
default profile
   |
   | source_profile
   v
audit profile
   |
   | assume role
   v
AuditRole
```

或者：

```text
Terraform resource
   |
   | provider = aws.audit
   v
Terraform provider alias audit
   |
   | profile = audit
   v
AWS CLI profile audit
   |
   | source_profile = default
   v
base credentials
```

这种图能迅速防止你把 alias、profile、role 混在一起。

---

### 第三步：配置 provider

先把 provider 写清楚：

```hcl
provider "aws" {
  region = var.region
}

provider "aws" {
  alias   = "audit"
  region  = var.region
  profile = "audit"
}
```

LocalStack 场景则加 endpoint 和 skip 参数。

---

### 第四步：资源绑定 provider

资源需要特殊身份时，显式写：

```hcl
provider = aws.audit
```

module 需要特殊身份时，显式传：

```hcl
providers = {
  aws = aws.audit
}
```

---

### 第五步：输出验证

输出要能证明资源确实创建出来，例如：

```hcl
output "audit_bucket_name" {
  value = aws_s3_bucket.audit.bucket
}
```

如果是 IAM Role：

```hcl
output "role_arn" {
  value = aws_iam_role.example.arn
}
```

综合挑战不是写完就算完，必须能通过输出和验证脚本证明结果。

---

## 14. 考试高频判断点

### 14.1 `source_profile` 是谁依赖谁？

```ini
[profile audit]
role_arn = arn:aws:iam::123456789012:role/AuditRole
source_profile = default
```

意思是：

> `audit` 依赖 `default` 的基础凭证。

不是 default 去依赖 audit。

---

### 14.2 `provider alias` 是不是 AWS 里的东西？

不是。

`provider alias` 是 Terraform 配置概念，不是 AWS IAM 概念。

AWS 不知道你 Terraform 里叫它 `aws.audit`。

---

### 14.3 `profile` 是不是 Terraform alias？

不是。

`profile` 是 AWS SDK 读取本地 AWS 配置用的名字。

Terraform alias 是：

```hcl
alias = "xxx"
```

---

### 14.4 module 里怎么用 alias provider？

root module 负责传：

```hcl
module "x" {
  source = "./modules/x"

  providers = {
    aws = aws.audit
  }
}
```

子模块里正常使用 provider 即可。

---

### 14.5 没写 `provider = aws.audit` 会怎样？

资源默认使用无 alias 的 provider。

也就是：

```hcl
provider "aws" { ... }
```

---

## 15. 选择题练习

### 题 1

Terraform 中 `provider "aws" { alias = "audit" }` 的作用是什么？

A. 在 AWS IAM 中创建一个名为 audit 的 Role  
B. 创建一份 Terraform AWS provider 的别名配置  
C. 自动创建 AWS CLI profile  
D. 替代所有 resource 的 provider 配置  

答案：B

解析：`alias` 是 Terraform provider 多实例机制。它不会创建 AWS IAM Role，也不会修改本地 AWS CLI profile。

---

### 题 2

下面配置中，`source_profile = default` 表示什么？

```ini
[profile audit]
role_arn = arn:aws:iam::123456789012:role/AuditRole
source_profile = default
```

A. audit profile 会作为 default 的来源  
B. audit profile 会先使用 default 的基础凭证去 AssumeRole  
C. default profile 会自动删除  
D. Terraform 会自动创建 AuditRole  

答案：B

解析：`source_profile` 表示 AssumeRole 前使用哪个 profile 作为基础凭证来源。

---

### 题 3

如果资源必须使用 alias 为 `audit` 的 AWS provider，应该怎么写？

A.

```hcl
profile = "audit"
```

B.

```hcl
provider = aws.audit
```

C.

```hcl
alias = "audit"
```

D.

```hcl
source_profile = "audit"
```

答案：B

解析：资源级别使用 provider meta-argument 绑定 provider alias。

---

### 题 4

下面配置中资源会使用哪个 provider？

```hcl
provider "aws" {
  region = "ap-southeast-2"
}

provider "aws" {
  alias  = "audit"
  region = "ap-southeast-2"
}

resource "aws_s3_bucket" "logs" {
  bucket = "logs-bucket"
}
```

A. 默认 `aws` provider  
B. `aws.audit` provider  
C. 两个 provider 同时使用  
D. 无法判断，一定报错  

答案：A

解析：资源没有显式写 `provider = aws.audit`，因此使用默认 provider。

## 17. 场景题

### 场景 1：资源用了错误账号

你有两个 provider：

```hcl
provider "aws" {
  region = "ap-southeast-2"
}

provider "aws" {
  alias  = "audit"
  region = "ap-southeast-2"
}
```

题目要求 audit bucket 必须由 audit provider 创建，但当前配置是：

```hcl
resource "aws_s3_bucket" "audit" {
  bucket = "audit-bucket"
}
```

请指出问题并修正。

参考答案：

问题是资源没有绑定 alias provider，因此会使用默认 provider。应修改为：

```hcl
resource "aws_s3_bucket" "audit" {
  provider = aws.audit

  bucket = "audit-bucket"
}
```

---

### 场景 2：module 要使用 audit provider

root module 里定义了：

```hcl
provider "aws" {
  alias  = "audit"
  region = "ap-southeast-2"
}
```

现在 `./modules/bucket` 子模块里的资源要使用这个 audit provider。root module 应该如何调用？

参考答案：

```hcl
module "audit_bucket" {
  source = "./modules/bucket"

  providers = {
    aws = aws.audit
  }

  bucket_name = "audit-bucket"
}
```

解析：通过 `providers` map，把 root module 的 `aws.audit` 传给子模块里的默认 `aws` provider。

---

### 场景 3：profile 和 alias 不同名

provider 配置如下：

```hcl
provider "aws" {
  alias   = "security"
  profile = "audit"
  region  = "ap-southeast-2"
}
```

资源里应该写 `provider = aws.audit` 还是 `provider = aws.security`？

参考答案：

应该写：

```hcl
provider = aws.security
```

解析：Terraform 资源引用的是 provider alias，不是 AWS CLI profile 名。

## 20. 最后总结

第 169 节的核心不是 AWS CLI 配置文件本身，而是把多身份调用链路拆清楚。

你需要牢牢记住三层关系：

```text
AWS CLI profile / source_profile
        ↓
Terraform provider / provider alias
        ↓
resource 或 module 绑定 provider
```

考试遇到这类题，不要慌，也不要上来就改资源。先画清楚：

1. 资源应该用哪个身份；
2. 这个身份对应哪个 Terraform provider alias；
3. 这个 provider alias 里面使用哪个 AWS CLI profile；
4. 这个 profile 是否通过 `source_profile` 去 AssumeRole。

能把这条链路讲明白，这节就算真正吃透了。
