# Terraform 实操训练 80 讲解：Terraform test 框架概览

## 1. 这一节到底在学什么？

Lab 80 学的是 Terraform 的原生测试框架：

```sh
terraform test
```

一句话理解：

> `terraform test` 是 Terraform 自带的测试机制，用来验证模块行为、输出结果、断言条件，而不只是检查语法是否正确。

前面很多 Lab 已经反复出现：

```sh
terraform test
```

并且验证报告里经常写：

```text
terraform test 返回 1 passed, 0 failed
```

Lab 80 开始正式把这个测试框架作为主题讲清楚。

---

## 2. 为什么 Terraform 需要测试？

很多人初学 Terraform 时，只会跑：

```sh
terraform init
terraform validate
terraform plan
terraform apply
```

这些当然重要，但它们解决的问题不同。

例如：

```sh
terraform validate
```

只能告诉你：

```text
语法和基本配置是否合法
```

但它不一定能告诉你：

```text
输出值是不是符合预期
资源数量是不是正确
for 表达式转换结果是不是正确
条件表达式是否走了正确分支
模块默认值是否符合设计
错误输入是否被拒绝
```

这些就需要测试。

Terraform test 的价值是：

> 用可重复的断言，把 Terraform 模块的预期行为固定下来。

---

## 3. `terraform test` 和 `terraform validate` 的区别

### `terraform validate`

检查配置是否合法：

```sh
terraform validate
```

它关注：

- HCL 语法
- provider 配置引用
- resource block 结构
- input/output 基本合法性
- 表达式是否能被解析

但它不会深入验证业务预期。

---

### `terraform test`

运行测试文件并执行断言：

```sh
terraform test
```

它关注：

- output 是否符合预期
- plan/apply 行为是否符合预期
- 变量输入是否触发正确结果
- 断言条件是否成立
- 模块在不同测试场景下是否工作

简单说：

```text
validate 检查“能不能解析”
test 检查“是不是按预期工作”
```

这两个都要跑。

---

## 4. Terraform test 文件放在哪里？

Terraform 测试文件通常放在：

```text
tests/
```

目录下。

文件后缀通常是：

```text
.tftest.hcl
```

例如：

```text
practice/labs/80/
├── main.tf
├── tests/
│   └── basic.tftest.hcl
└── README.md
```

测试文件示例：

```hcl
run "basic" {
  command = plan

  assert {
    condition     = output.name == "demo"
    error_message = "Expected name output to be demo."
  }
}
```

---

## 5. `run` block 是什么？

Terraform test 里用 `run` block 定义一个测试用例。

基本结构：

```hcl
run "basic" {
  command = plan

  assert {
    condition     = output.example == "expected"
    error_message = "Output example should match expected value."
  }
}
```

其中：

```hcl
run "basic"
```

表示测试用例名称是 `basic`。

```hcl
command = plan
```

表示这个测试运行 `terraform plan` 级别的检查。

`assert` 表示断言。

如果断言成立，测试通过。  
如果断言失败，测试失败。

---

## 6. `command = plan` 和 `command = apply`

Terraform test 里的 `run` 可以使用不同 command。

常见有：

```hcl
command = plan
```

和：

```hcl
command = apply
```

### `command = plan`

只运行到 plan 阶段。

适合验证：

- 计划阶段可确定的 output
- 变量逻辑
- locals 计算
- 条件表达式
- for 表达式
- 不需要真实创建资源的逻辑

优点：

```text
快
风险低
不创建真实资源
适合 CI
```

---

### `command = apply`

会实际 apply。

适合验证：

- apply 后才知道的值
- 真实资源创建结果
- provider 返回属性
- resource 行为
- output 依赖 apply 结果的情况

缺点：

```text
慢
可能产生费用
需要清理
有更高风险
```

本地训练中如果用 `local_file`、`terraform_data`，apply 风险较低。  
真实云资源中要更谨慎。

---

## 7. `assert` 是什么？

`assert` 是测试断言。

基本结构：

```hcl
assert {
  condition     = output.example == "expected"
  error_message = "example output did not match expected value."
}
```

含义：

- `condition`：必须返回 true。
- `error_message`：失败时显示的错误信息。

例如：

```hcl
assert {
  condition     = output.service_count == 3
  error_message = "Expected 3 services."
}
```

如果：

```hcl
output.service_count
```

不是 3，测试就失败。

---

## 8. 测试 output

Terraform test 经常断言 output。

假设 main.tf 有：

```hcl
output "service_name" {
  value = "api"
}
```

测试：

```hcl
run "service_name" {
  command = plan

  assert {
    condition     = output.service_name == "api"
    error_message = "service_name should be api."
  }
}
```

这类测试适合验证：

- locals 计算结果
- 变量默认值
- for 表达式转换结果
- jsondecode/csvdecode 结果
- templatefile 渲染结果
- remote state 读取结果

---

## 9. 测试变量输入

测试可以给模块传入变量。

示例：

```hcl
run "prod_environment" {
  command = plan

  variables {
    environment = "prod"
  }

  assert {
    condition     = output.replica_count == 3
    error_message = "prod should use 3 replicas."
  }
}
```

再写一个 dev 场景：

```hcl
run "dev_environment" {
  command = plan

  variables {
    environment = "dev"
  }

  assert {
    condition     = output.replica_count == 1
    error_message = "dev should use 1 replica."
  }
}
```

这样可以验证不同输入下模块行为是否正确。

这比人工改变量再 plan 更可靠。

---

## 10. 测试错误输入

Terraform test 也可以测试错误输入是否被拒绝。

例如变量 validation：

```hcl
variable "environment" {
  type = string

  validation {
    condition     = contains(["dev", "test", "prod"], var.environment)
    error_message = "environment must be dev, test, or prod."
  }
}
```

测试可以传入非法值：

```hcl
run "invalid_environment" {
  command = plan

  variables {
    environment = "invalid"
  }

  expect_failures = [
    var.environment
  ]
}
```

这类测试可以验证：

> 模块不仅能接受正确输入，还能拒绝错误输入。

这对生产模块很重要。

---

## 11. Terraform test 能测什么？

Terraform test 很适合测这些内容：

- variable validation
- locals 计算
- output 值
- conditional expression
- for 表达式转换
- flatten/distinct 结果
- csvdecode/jsondecode 解析结果
- templatefile 渲染结果
- for_each map key
- resource count
- check block
- precondition/postcondition
- remote state output
- 模块默认行为
- 错误输入是否失败

简单说：

> 只要 Terraform 能在 plan/apply 阶段算出来，就可以考虑用 test 固定它。

## 13. 标准运行流程

进入实验目录：

```sh
cd /workspace/practice/labs/80
```

初始化：

```sh
terraform init -input=false
```

格式检查：

```sh
terraform fmt -check
```

配置校验：

```sh
terraform validate
```

运行测试：

```sh
terraform test
```

生成 plan：

```sh
terraform plan -input=false -no-color -out=tfplan
```

执行 apply：

```sh
terraform apply -auto-approve tfplan
```

查看输出：

```sh
terraform output
```

销毁：

```sh
terraform destroy -auto-approve
```

清理：

```sh
rm -f tfplan
rm -rf output logs .terraform .terraform.lock.hcl terraform.tfstate terraform.tfstate.backup
```

## 14. 为什么 test 要放在 plan/apply 前面？

标准流程里通常先跑：

```sh
terraform test
```

再跑：

```sh
terraform plan
terraform apply
```

原因很直接：

> 如果测试都不通过，就不应该继续 apply。

在 CI/CD 中也是一样。

推荐顺序：

```text
fmt
validate
test
plan
approval
apply
```

这能尽早发现问题，减少错误 apply 的机会。

---

## 15. `terraform test` 输出怎么看？

运行：

```sh
terraform test
```

如果成功，可能看到类似：

```text
tests/basic.tftest.hcl... pass
Success! 1 passed, 0 failed.
```

这说明测试通过。

如果失败，可能看到：

```text
Error: Test assertion failed
```

并显示你写的：

```hcl
error_message
```

所以 error_message 要写得清楚，不要写成：

```text
failed
```

更好的错误信息：

```text
Expected output service_count to be 3.
```

这样排障时能快速知道哪里错了。

---

## 16. 好的 error_message 怎么写？

不好的：

```hcl
error_message = "wrong"
```

好的：

```hcl
error_message = "Expected service_names to contain api and worker."
```

更好的：

```hcl
error_message = "Expected prod environment to use 3 replicas."
```

原则：

1. 说明期望是什么。
2. 说明失败代表什么。
3. 不要太模糊。
4. 尽量指向业务规则。

测试失败时，错误信息就是未来的救命绳。

---

## 17. Terraform test 和 CI/CD

Terraform test 很适合放进 CI。

典型 GitHub Actions 步骤：

```sh
terraform fmt -check
terraform init -input=false
terraform validate
terraform test
terraform plan -input=false -no-color -out=tfplan
```

它的作用是：

- PR 阶段提前发现模块逻辑错误
- 防止错误输入通过
- 防止 output 变化不受控
- 防止 for 表达式转换结果被改坏
- 防止模板渲染结果被改坏
- 提升 Terraform 模块质量

这就是从“我本地能跑”升级到“团队可验证”的关键。

---

## 18. Terraform test 和 saved plan 的关系

`terraform test` 不能替代 saved plan。

它们解决的问题不同。

### terraform test

验证模块逻辑：

```text
输入 A 时 output 是否是 B
错误输入是否失败
模板是否生成正确内容
```

### saved plan

锁定即将 apply 的变更：

```sh
terraform plan -out=tfplan
terraform apply tfplan
```

它保证 apply 的就是 review 过的 plan。

成熟流程里两者都要有：

```text
test 负责模块行为
plan 负责基础设施变更审查
```

## 19. Terraform test 和真实云资源

如果测试用真实云资源，要小心：

- 可能产生费用
- 可能失败后残留资源
- 需要账号权限
- 需要并发隔离
- 需要清理机制
- 可能受云端配额影响
- CI 速度慢

所以训练阶段经常用：

- `terraform_data`
- `local_file`
- mock JSON/CSV
- LocalStack
- 小型可销毁资源

这也是你前面很多 lab 使用 Docker/LocalStack/local provider 的原因。

## 21. 常见坑

### 坑 1：测试文件后缀写错

测试文件应该类似：

```text
basic.tftest.hcl
```

如果写成：

```text
basic.test.hcl
```

Terraform 可能不会按测试文件识别。

---

### 坑 2：断言引用了不存在的 output

例如：

```hcl
condition = output.name == "api"
```

但 main.tf 里没有：

```hcl
output "name"
```

测试会失败。

---

### 坑 3：把 local 当 output 直接测

测试里通常更容易断言 output。  
如果你想测 local，可以把关键结果暴露成 output，或者通过资源/输出间接验证。

课程验证报告里也提到：

```text
测试框架实验需避免引用未声明的 local
```

这就是常见坑。

---

### 坑 4：plan 阶段测 apply 后才确定的值

有些值 plan 阶段未知。

如果用：

```hcl
command = plan
```

断言 apply 后才知道的值，可能失败。

这种场景要考虑：

```hcl
command = apply
```

或者调整测试目标。

---

### 坑 5：测试依赖真实环境但没准备好

例如测试需要 S3、LocalStack、provider 认证。  
如果环境没启动，测试会失败。

---

### 坑 6：测试通过但 apply 失败

测试不是万能的。  
尤其是只用 `command = plan` 的测试，不代表真实 apply 一定成功。

所以标准流程仍然需要：

```text
test + plan + apply
```

---

## 22. `terraform test` 适合测模块，不是替代所有测试

Terraform test 很有用，但不要神化。

它适合测 Terraform 模块行为。

但不适合替代：

- 应用单元测试
- 集成测试
- 安全扫描
- Policy as Code
- 云资源运行时探测
- 性能测试
- 灾备演练
- Chaos Engineering

例如：

```text
terraform test 可以验证 Security Group rule 计划创建
但不能证明真实业务流量一定可达
```

Terraform test 是 IaC 测试体系的一部分，不是全部。

## 23. 和 check / precondition 的关系

前面学过：

- variable validation
- precondition
- postcondition
- check block

它们和 test 不是互斥关系。

### validation/precondition/check

写在 Terraform 配置里，运行 plan/apply 时生效。

### terraform test

写在测试文件里，用于测试模块行为。

可以这样理解：

```text
配置内约束：保护模块运行时行为
测试文件：验证模块在不同场景下符合预期
```

成熟模块通常两者都用。

---

## 24. 测试 for 表达式结果

例如前面 Lab 59-64 学了很多数据转换。

如果 main.tf 有：

```hcl
output "unique_teams" {
  value = sort(distinct(flatten([
    for svc in var.services : svc.teams
  ])))
}
```

测试可以写：

```hcl
run "unique_teams" {
  command = plan

  assert {
    condition     = output.unique_teams == ["backend", "frontend", "platform"]
    error_message = "unique_teams should be sorted and deduplicated."
  }
}
```

这能防止别人改坏 for/flatten/distinct 逻辑。

---

## 25. 测试 templatefile 结果

例如 Lab 65-66 的模板。

如果 output 暴露了渲染结果：

```hcl
output "rendered_config" {
  value = local.rendered_config
}
```

测试可以写：

```hcl
run "rendered_config" {
  command = plan

  assert {
    condition     = strcontains(output.rendered_config, "APP_ENV=dev")
    error_message = "Rendered config should contain APP_ENV=dev."
  }
}
```

这可以抓住模板变量漏传、循环错误、多行内容缺失等问题。

---

## 26. 测试 remote state 读取

例如 Lab 78-79 的 remote state。

consumer 可以测试：

```hcl
assert {
  condition     = output.allowed_ip == "203.0.113.10"
  error_message = "consumer should read public_ip from network remote state."
}
```

不过 remote state 测试需要先准备 producer state。

所以这种测试要注意前置流程：

```text
先 apply producer/network
再 test consumer
```

这就是跨 state 测试的复杂点。

---

## 27. 测试 backend 行为

像 Lab 74-76 的 backend 实验，测试重点不是普通 output，而是验证流程：

- backend init 成功
- state 写入 S3
- 本地没有主 state
- state list 能读取远端 state
- lock 配置正确
- destroy 和 clean 完整

有些验证可能通过 shell 脚本完成，不一定全部写进 `.tftest.hcl`。

但核心思路一样：

> 不要只靠肉眼看，要有可重复验证。

---

## 28. 生产模块测试建议

如果你写生产 Terraform 模块，建议至少测试：

1. 默认输入是否能 plan。
2. 关键 output 是否正确。
3. 错误变量是否失败。
4. 可选功能开关是否生效。
5. for_each 创建数量是否符合预期。
6. 重要 tags 是否生成。
7. templatefile 渲染是否包含关键内容。
8. 不同环境 dev/prod 配置是否不同。
9. 敏感 output 是否标记 sensitive。
10. 资源命名是否符合规范。

Terraform test 不一定覆盖一切，但至少能避免很多低级事故。

## 31. SRE / Platform Engineering 视角

从 SRE / Platform Engineering 角度看，Terraform test 是工程质量门槛。

没有测试的 Terraform 模块，风险很高：

```text
改一个变量默认值，没人知道影响
改一个 for 表达式，输出悄悄变了
改一个模板，生成配置少一行
改一个 validation，错误输入进来了
```

有测试后，CI 可以提前拦住这些问题。

平台工程不是靠“我觉得没问题”。  
平台工程要靠：

```text
可重复验证
可审查 plan
可回滚设计
可观察输出
可控 state
```

Terraform test 就是其中一环。

- 测试文件通常放在 `tests/` 目录。
- 测试文件后缀通常是 `.tftest.hcl`。
- `run` block 定义测试用例。
- `assert` block 定义断言。
- `command = plan` 适合验证 plan 阶段可确定的值。
- `command = apply` 适合验证 apply 后才知道的值。
- `terraform validate` 检查配置合法性，`terraform test` 检查行为是否符合预期。
- test 适合验证 output、变量输入、for 表达式、模板、错误输入。
- 测试失败时 `error_message` 要写清楚。
- CI/CD 中推荐顺序：fmt -> init -> validate -> test -> plan -> apply。
- test 不能替代 saved plan，也不能替代所有运行时测试。

一句话总结：

> Terraform test 的核心价值，是把模块的预期行为写成可重复执行的断言，让 Terraform 配置不再只靠人工看 plan 和经验判断。
