# Terraform AWS Sandbox 34 讲解：AMI 动态查询的必要性

## 1. 这一节到底在学什么？

Lab 34 的核心不是“怎么创建 EC2”，而是学习一个非常重要的 AWS Terraform 习惯：

> **不要把 AMI ID 写死在 Terraform 代码里。**

AMI 是 Amazon Machine Image，也就是 EC2 启动时使用的系统镜像。它决定了实例启动后是什么操作系统、什么基础环境、什么架构。

很多初学 Terraform 的人会这样写：

```hcl
resource "aws_instance" "demo" {
  ami           = "ami-xxxxxxxxxxxxxxxxx"
  instance_type = "t3.micro"
}
```

这在单个 region、短期 demo 里看起来没问题，但放到真实环境就很容易炸。

原因是：**AMI ID 是 region-specific 的。**

同一个 Ubuntu、Amazon Linux 或 Windows 镜像，在不同 AWS region 里的 AMI ID 通常不同。你在 `us-east-1` 找到的 AMI ID，不能直接拿去 `ap-southeast-2` 或 `ap-south-1` 用。

所以这节课要训练的是：

> 用 `data "aws_ami"` 按条件动态查询 AMI，而不是硬编码 AMI ID。

---

## 2. 为什么硬编码 AMI ID 是坏味道？

硬编码 AMI ID 的问题主要有三个。

### 问题 1：跨 region 不可复用

假设你在 `us-east-1` 写死了一个 AMI：

```hcl
ami = "ami-0123456789abcdef0"
```

切换到 `ap-southeast-2` 后，这个 AMI ID 很可能不存在。

结果就是：

- plan 可能失败
- apply 可能失败
- EC2 无法创建
- pipeline 在换 region 后直接挂掉

这就是典型的“本地能跑，换环境就死”。

---

### 问题 2：镜像会过期

公共 AMI 会持续更新。Ubuntu、Amazon Linux、Windows Server 都会发布新版本。

如果长期写死旧 AMI，你可能会遇到：

- 旧镜像缺安全补丁
- 旧镜像被官方弃用
- 新实例启动后基础包过旧
- 安全扫描出现大量 CVE
- 合规检查不通过

对于生产环境来说，镜像不是一次性配置，而是生命周期管理对象。

---

### 问题 3：架构可能不匹配

现在 AWS 里常见两类实例架构：

- `x86_64`
- `arm64`

如果你用的是 Graviton 实例，例如 `t4g.micro`、`m7g.large`，就需要 `arm64` AMI。

如果 AMI 是 `x86_64`，实例类型是 `arm64`，就会不匹配。

所以生产中查询 AMI 时，不能只看名字，还要关注：

- architecture
- virtualization type
- owner
- root device type
- name pattern

---

## 3. 正确姿势：用 `data "aws_ami"` 动态查询

Terraform 的 data source 可以读取外部已经存在的信息。`aws_ami` 就是用来查询 AMI 的 data source。

典型写法是：

```hcl
data "aws_ami" "latest" {
  most_recent = true

  owners = ["099720109477"]

  filter {
    name   = "name"
    values = ["ubuntu/images/hvm-ssd/ubuntu-jammy-22.04-amd64-server-*"]
  }

  filter {
    name   = "architecture"
    values = ["x86_64"]
  }

  filter {
    name   = "virtualization-type"
    values = ["hvm"]
  }
}
```

然后 EC2 里引用：

```hcl
resource "aws_instance" "demo" {
  ami           = data.aws_ami.latest.id
  instance_type = "t3.micro"
}
```

这样 Terraform 会在当前 region 里，根据条件查询最新的 AMI。

这就比硬编码强很多：

- region 切换后，会在新 region 重新查询
- AMI 可以随着官方发布更新
- 条件可以约束架构和虚拟化类型
- 代码更容易复用

---

## 4. `most_recent = true` 是什么？

`most_recent = true` 的意思是：

> 如果过滤条件匹配到多个 AMI，选择最新的那个。

这很常用，但也要小心。

它的优点是能自动选新版本镜像。  
它的风险是镜像升级可能带来行为变化。

比如今天查到的是：

```text
ubuntu-2026-01-01
```

过几天官方发布新镜像，查询结果可能变成：

```text
ubuntu-2026-01-15
```

如果你的应用对系统包版本、内核版本、启动脚本很敏感，就可能出现“同样的 Terraform 代码，过几天创建出来的实例不完全一样”。

所以生产中一般有两种策略：

### 策略 A：开发/测试环境使用 latest

适合快速拿到新镜像，减少手工维护。

### 策略 B：生产环境固定经过验证的镜像版本

例如通过 Packer 构建 golden image，然后用明确的 name pattern 或版本 tag 查询。

生产里更稳的做法是：

> 不是永远追最新，而是查询“组织批准过的最新镜像”。

---

## 5. 本实验为什么是 LocalStack limited？

Lab 34 的环境分类是：

```text
aws-localstack-limited
```

这个分类很重要。

LocalStack 可以模拟部分 AWS API，用来练 Terraform provider endpoint、data source 查询、filter 写法和 output 引用。

但是 LocalStack **不提供真实 AWS 公共 AMI 目录**。

也就是说，它不能完整验证这些真实行为：

- Amazon Linux 官方 AMI 是否存在
- Ubuntu owner ID 是否正确
- 不同 region 的官方 AMI ID 是否不同
- AMI 和真实 EC2 instance type 是否兼容
- AMI 发布时间和安全更新逻辑

所以本实验用 bootstrap 先在 LocalStack 注册两个模拟 AMI，然后 Terraform 再用 `data "aws_ami"` 查询它们。

这验证的是 Terraform 查询路径，不是 AWS 官方 AMI 生态本身。

一句话：

> LocalStack 在这里适合练写法，不适合证明真实 AMI 选择一定正确。

---

## 6. 本实验的设计思路

实验流程是：

1. `bootstrap.sh` 在 LocalStack 中注册两个模拟 AMI。
2. 两个 AMI 都符合 `tf-lab-ubuntu-*` 这类命名模式。
3. Terraform 使用 `data "aws_ami"` 查询最新模拟 AMI。
4. 输出最新 AMI 的名称和 ID。
5. `verify.sh` 验证查询结果。
6. `clean.sh` deregister 模拟 AMI。

预期输出里应该看到：

```text
latest_ami_name = "tf-lab-ubuntu-2026-01-01-x86_64"
```

以及一个模拟 AMI ID。

这说明 Terraform 成功通过 data source 找到了最新镜像。

## 7. 标准运行流程

进入实验目录：

```sh
cd practice/labs/34
```

设置 sandbox 环境变量：

```sh
export AWS_ACCESS_KEY_ID=test
export AWS_SECRET_ACCESS_KEY=test
export AWS_DEFAULT_REGION=us-east-1
export LOCALSTACK_ENDPOINT=http://localhost:4566
```

检查 LocalStack：

```sh
bash scripts/check-sandbox.sh
```

注册模拟 AMI：

```sh
bash scripts/bootstrap.sh
```

执行 Terraform：

```sh
terraform init -input=false
terraform fmt -check
terraform validate
terraform plan -input=false -no-color -out=tfplan
terraform apply -auto-approve tfplan
terraform output
```

验证：

```sh
bash scripts/verify.sh
```

清理：

```sh
bash scripts/clean.sh
```

清理本地 Terraform 文件：

```sh
rm -rf .terraform .terraform.lock.hcl tfplan terraform.tfstate terraform.tfstate.backup
```

## 8. 运行结果应该怎么看？

关键输出应该类似：

```text
latest_ami_name = "tf-lab-ubuntu-2026-01-01-x86_64"
latest_ami_id   = "ami-..."
```

你要看的不是 AMI ID 具体是多少，而是：

1. Terraform 没有写死 AMI ID。
2. Terraform 通过 filter 查询到了 AMI。
3. `most_recent = true` 选择了最新的模拟镜像。
4. 输出结果来自 data source。

这说明你掌握了动态查询 AMI 的基本工作流。

## 9. 真实 AWS 中应该怎么改？

LocalStack 里的模拟 AMI name 不能直接搬到真实 AWS。

真实 AWS 中，你需要替换这些内容：

### 1. owner

Ubuntu 常见 owner 是 Canonical 的 AWS account ID，例如：

```hcl
owners = ["099720109477"]
```

Amazon Linux 则通常使用 Amazon owner。

### 2. name pattern

例如 Ubuntu 22.04 x86_64 可能使用类似：

```hcl
values = ["ubuntu/images/hvm-ssd/ubuntu-jammy-22.04-amd64-server-*"]
```

Amazon Linux 2023 可能使用另一套 name pattern。

### 3. architecture

根据实例类型选择：

```hcl
values = ["x86_64"]
```

或者：

```hcl
values = ["arm64"]
```

### 4. virtualization type

现代 EC2 通常使用：

```hcl
values = ["hvm"]
```

### 5. region

切换 region 后，data source 会在当前 region 查询对应 AMI。  
这正是动态查询的价值。

## 10. 常见坑

### 坑 1：`aws_ami` 找不到结果

通常是 filter 太严格或 bootstrap 没成功。

在 LocalStack 里，先确认：

```sh
bash scripts/bootstrap.sh
```

在真实 AWS 里，检查：

- owner 是否正确
- name pattern 是否正确
- region 是否正确
- architecture 是否正确

---

### 坑 2：找到多个结果

如果没有设置：

```hcl
most_recent = true
```

或者过滤条件太宽，就可能匹配多个 AMI。

解决方式：

- 加 `most_recent = true`
- 收紧 name pattern
- 增加 architecture filter
- 增加 virtualization type filter
- 增加 owner 限制

---

### 坑 3：AMI 和实例类型架构不匹配

例如：

- AMI 是 `x86_64`
- 实例类型是 `t4g.micro`

`t4g` 是 Graviton，也就是 arm64 架构。这样会不匹配。

解决方式：让 AMI architecture 和 instance type 对齐。

---

### 坑 4：过度依赖 latest

`most_recent = true` 看起来很方便，但生产中不能无脑追最新。

如果镜像变更没有测试流程，自动选最新可能带来不可控风险。

更好的工程做法是：

1. 用 Packer 构建组织内部 golden image。
2. 给镜像打版本 tag。
3. 测试通过后才发布。
4. Terraform 查询已批准版本。
5. 生产发布走明确审批。

---

## 11. 生产环境推荐模式

### 简单模式：官方 AMI 动态查询

适合学习、demo、小型环境：

```hcl
data "aws_ami" "ubuntu" {
  most_recent = true
  owners      = ["099720109477"]

  filter {
    name   = "name"
    values = ["ubuntu/images/hvm-ssd/ubuntu-jammy-22.04-amd64-server-*"]
  }

  filter {
    name   = "architecture"
    values = ["x86_64"]
  }

  filter {
    name   = "virtualization-type"
    values = ["hvm"]
  }
}
```

### 进阶模式：组织 golden image

适合生产平台团队：

```hcl
data "aws_ami" "golden" {
  most_recent = true
  owners      = [var.platform_account_id]

  filter {
    name   = "tag:ImageFamily"
    values = ["platform-ubuntu-2204"]
  }

  filter {
    name   = "tag:Approved"
    values = ["true"]
  }

  filter {
    name   = "architecture"
    values = [var.architecture]
  }
}
```

这个模式更符合真实公司里的平台工程实践。

## 12. 面试/考试记忆版

可以这样记：

- AMI ID 是 region-specific，不能跨 region 复用。
- 硬编码 AMI ID 会降低 Terraform 的可移植性。
- `data "aws_ami"` 可以按 owner、name、architecture、virtualization type 查询镜像。
- `most_recent = true` 会在多个匹配结果中选择最新 AMI。
- LocalStack 可以练 data source 写法，但不能代表真实 AWS 公共 AMI 目录。
- 生产环境最好使用经过验证的 golden image，而不是无脑追最新公共 AMI。

一句话答案：

> AMI ID 不应该硬编码，因为它和 region、架构、版本强相关；更好的方式是用 `aws_ami` data source 按条件动态查询，并在生产中配合 golden image 和审批流程。

---

## 13. 本 Lab 的价值

Lab 34 的价值在于，它把 Terraform 从“写死参数创建资源”推进到了更真实的工程模式：

> 根据当前环境动态查询正确资源，再把结果传给后续配置。

这是做 AWS Terraform、Platform Engineering、SRE 自动化时非常基础但非常关键的能力。

你以后写 EKS、EC2、ASG、Launch Template、Packer golden image 流水线时，都会反复遇到这个问题：  
镜像到底是写死，还是动态查询？

工程上成熟的答案是：

> 开发环境可以动态追新，生产环境查询组织批准过的镜像版本。
