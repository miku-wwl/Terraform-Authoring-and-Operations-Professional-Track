# Terraform 实操训练 60 讲解：`csvdecode` 基础

## 1. 这一节到底在学什么？

Lab 60 学的是 Terraform 里的 `csvdecode()` 函数。

前面几节你已经学过：

- `list` / `map` / `object`
- 嵌套类型
- 读取嵌套值
- `for_each`
- 条件表达式
- `for` 表达式

这一节开始把外部结构化文本引入 Terraform：

```hcl
csvdecode(...)
```

一句话理解：

> `csvdecode()` 可以把 CSV 文本解析成 Terraform 能处理的对象列表。

也就是说，它能把这种文本：

```csv
name,port,environment
api,8080,dev
worker,9090,dev
web,3000,prod
```

变成 Terraform 里的结构化数据：

```hcl
[
  {
    name        = "api"
    port        = "8080"
    environment = "dev"
  },
  {
    name        = "worker"
    port        = "9090"
    environment = "dev"
  },
  {
    name        = "web"
    port        = "3000"
    environment = "prod"
  }
]
```

注意：CSV 解析出来的字段通常都是字符串。

---

## 2. `csvdecode()` 是什么？

`csvdecode()` 是 Terraform 的内置函数，用来解析 CSV 格式字符串。

基本语法：

```hcl
csvdecode(string)
```

例如：

```hcl
locals {
  raw_csv = <<EOT
name,port
api,8080
worker,9090
EOT

  services = csvdecode(local.raw_csv)
}
```

`local.services` 会变成类似：

```hcl
[
  {
    name = "api"
    port = "8080"
  },
  {
    name = "worker"
    port = "9090"
  }
]
```

它的返回结果是：

```hcl
list(object(...))
```

或者更直观地说：

> 每一行 CSV 数据变成一个 object；所有行组成一个 list。

---

## 3. CSV 第一行为什么重要？

CSV 的第一行通常是表头。

例如：

```csv
name,port,environment
api,8080,dev
worker,9090,dev
```

第一行：

```csv
name,port,environment
```

会变成 object 的字段名。

所以 Terraform 解析后，每一行会有这些字段：

```hcl
row.name
row.port
row.environment
```

如果表头写错了，后面访问字段也会错。

例如表头写成：

```csv
service_name,port
api,8080
```

那你就不能访问：

```hcl
row.name
```

你要访问：

```hcl
row.service_name
```

这点很关键。

---

## 4. `csvdecode()` 的返回值长什么样？

假设 CSV 是：

```csv
name,port
api,8080
worker,9090
```

Terraform：

```hcl
locals {
  services = csvdecode(local.raw_csv)
}
```

返回值逻辑上是：

```hcl
[
  {
    name = "api"
    port = "8080"
  },
  {
    name = "worker"
    port = "9090"
  }
]
```

注意 `port` 是：

```hcl
"8080"
```

不是：

```hcl
8080
```

因为 CSV 本质是文本格式。

如果你需要 number，要手动转换：

```hcl
tonumber(row.port)
```

## 5. 为什么 Terraform 需要读取 CSV？

很多团队的配置数据不是一开始就写在 HCL 里，而是来自表格或清单。

例如：

- 用户清单
- 服务清单
- 端口清单
- 成本中心清单
- 环境矩阵
- DNS 记录清单
- 告警规则清单
- 团队 ownership 表
- 资源 tag 标准表

这些数据很容易用 CSV 表达。

例如服务清单：

```csv
name,port,replicas,environment
api,8080,2,dev
worker,9090,1,dev
web,3000,3,prod
```

Terraform 用 `csvdecode()` 读取后，就可以把它变成 module input 或 `for_each` 数据源。

## 6. `csvdecode()` 和 `file()` 的组合

真实项目里，CSV 通常在文件里。

例如：

```text
services.csv
```

内容：

```csv
name,port,environment
api,8080,dev
worker,9090,dev
```

Terraform 可以这样读取：

```hcl
locals {
  services = csvdecode(file("${path.module}/services.csv"))
}
```

这里有两个函数：

```hcl
file(...)
```

读取文件内容，返回字符串。

```hcl
csvdecode(...)
```

把 CSV 字符串解析成对象列表。

组合起来就是：

> 读取 CSV 文件，并解析成 Terraform 数据结构。

## 7. `csvdecode()` 和 heredoc

在实验里，也可能直接把 CSV 写在 Terraform 文件里。

例如：

```hcl
locals {
  raw_csv = <<EOT
name,port,environment
api,8080,dev
worker,9090,dev
EOT

  services = csvdecode(local.raw_csv)
}
```

这里的 `<<EOT ... EOT` 是 heredoc，多行字符串。

Lab 26、27、28 已经讲过 heredoc。  
这一节就把 heredoc 和 `csvdecode()` 串起来了。

## 8. `csvdecode()` 和 `for` 表达式

`csvdecode()` 返回的是 list(object)，所以经常会配合 `for` 表达式做数据转换。

例如：

```hcl
locals {
  services = csvdecode(local.raw_csv)

  service_names = [
    for svc in local.services : svc.name
  ]
}
```

结果：

```hcl
["api", "worker"]
```

把 port 转成 number：

```hcl
locals {
  service_ports = [
    for svc in local.services : tonumber(svc.port)
  ]
}
```

生成摘要：

```hcl
locals {
  service_summaries = [
    for svc in local.services : "${svc.name}:${svc.port}"
  ]
}
```

这就是 Lab 59 的 `for` 表达式在这里的应用。

---

## 9. `csvdecode()` 和 `for_each`

如果要根据 CSV 创建多个资源，通常不能直接把 `csvdecode()` 的结果交给 `for_each`。

因为 `csvdecode()` 返回的是 list(object)。

更好的做法是先转成 map：

```hcl
locals {
  services = csvdecode(local.raw_csv)

  services_by_name = {
    for svc in local.services : svc.name => svc
  }
}
```

然后：

```hcl
resource "local_file" "service" {
  for_each = local.services_by_name

  filename = "${path.module}/output/${each.key}.txt"

  content = jsonencode({
    name        = each.key
    port        = tonumber(each.value.port)
    environment = each.value.environment
  })
}
```

这样资源地址就是：

```text
local_file.service["api"]
local_file.service["worker"]
```

这比 `count.index` 稳定。

---

## 10. 为什么要把 CSV list 转成 map？

因为 `csvdecode()` 返回的是 list：

```hcl
[
  { name = "api", port = "8080" },
  { name = "worker", port = "9090" }
]
```

如果直接用 list + count，会回到 Lab 55 的索引挑战：

```text
[0]
[1]
```

如果把它转成 map：

```hcl
{
  api = {
    name = "api"
    port = "8080"
  }

  worker = {
    name = "worker"
    port = "9090"
  }
}
```

资源地址就更稳定：

```text
["api"]
["worker"]
```

所以常见套路是：

```hcl
csvdecode -> list(object) -> for 表达式 -> map(object) -> for_each
```

这条链路很重要。

---

## 11. CSV 字段类型都要注意

CSV 是文本格式。  
所以解析出来的字段通常是 string。

例如：

```csv
name,port,enabled
api,8080,true
```

解析后：

```hcl
{
  name    = "api"
  port    = "8080"
  enabled = "true"
}
```

如果你需要 number：

```hcl
tonumber(svc.port)
```

如果你需要 bool，可以做判断：

```hcl
svc.enabled == "true"
```

或者更严格地设计输入和 validation。

不要假设 CSV 里的 `8080` 自动就是 number。

---

## 12. CSV 表头命名建议

CSV 表头会变成 object 字段名，所以建议：

- 使用小写
- 使用下划线或简单英文
- 避免空格
- 避免特殊字符
- 避免中文字段名
- 避免重复字段名
- 字段名要和 Terraform 访问路径一致

推荐：

```csv
name,port,environment,enabled
```

不推荐：

```csv
Service Name,Port Number,环境,是否启用
```

后者人类看着舒服，但 Terraform 访问会很难受。

---

## 13. CSV 数据质量很重要

`csvdecode()` 不是数据治理工具。  
如果 CSV 里数据乱，Terraform 后面也会乱。

常见问题包括：

- 表头拼错
- 字段缺失
- 多余逗号
- 端口不是数字
- bool 写成 yes/no/TRUE/true 混用
- name 重复
- 空行
- 空字段
- 字段里有逗号但没正确引号包裹

所以生产里如果用 CSV 驱动 Terraform，要加测试和校验。

---

## 14. 处理重复 key

如果你把 CSV 转成 map：

```hcl
locals {
  services_by_name = {
    for svc in local.services : svc.name => svc
  }
}
```

那么 `svc.name` 必须唯一。

如果 CSV 里有两行同名：

```csv
name,port
api,8080
api,9090
```

就会出现 key 冲突。

这不是 Terraform 的问题，是数据模型问题。

如果 name 是资源身份，就必须唯一。  
如果允许重复，那就不能直接用 name 做 map key，要重新设计 key。

---

## 15. `csvdecode()` 和 validation

Terraform 变量 validation 通常用于变量。  
如果 CSV 是 local 读入的数据，不能像 variable 那样直接写 validation。

但你可以通过 `terraform test`、`check` block、precondition，或者 locals 中的表达式来验证。

例如检查所有端口是否为数字，可以尝试：

```hcl
locals {
  ports = [
    for svc in local.services : tonumber(svc.port)
  ]
}
```

如果某个 port 不是数字，`terraform plan` 就会失败。

更成熟的方式是写测试断言，例如：

```hcl
assert {
  condition = alltrue([
    for svc in local.services : can(tonumber(svc.port))
  ])
  error_message = "All service ports must be numeric."
}
```

## 20. 常见坑

### 坑 1：忘记 CSV 字段都是字符串

CSV 里的：

```csv
8080
```

解析后通常是：

```hcl
"8080"
```

需要 number 时要：

```hcl
tonumber(svc.port)
```

---

### 坑 2：表头写错

如果表头是：

```csv
service_name,port
```

就不能访问：

```hcl
svc.name
```

要访问：

```hcl
svc.service_name
```

字段名要对齐。

---

### 坑 3：重复 key

把 CSV 转成 map 时，key 必须唯一：

```hcl
svc.name => svc
```

如果 name 重复，就会出问题。

---

### 坑 4：直接用 list + count 创建长期资源

`csvdecode()` 返回 list(object)。  
如果你直接配 `count.index`，会有索引漂移风险。

更推荐转成 map 后用 `for_each`。

---

### 坑 5：CSV 格式不规范

字段里有逗号时要正确引用，例如：

```csv
name,description
api,"service, public endpoint"
```

否则 CSV 解析可能不符合预期。

## 21. `csvdecode()` 和 `jsondecode()` 的区别

Lab 29 学过 `jsonencode` / `jsondecode`。  
现在学 `csvdecode()`，可以对比一下。

| 函数 | 输入格式 | 适合场景 |
|---|---|---|
| `jsondecode()` | JSON | 嵌套结构、复杂对象 |
| `csvdecode()` | CSV | 表格数据、简单清单 |

CSV 适合扁平表格：

```csv
name,port,env
api,8080,dev
```

JSON 适合复杂嵌套：

```json
{
  "services": {
    "api": {
      "ports": [8080, 9090]
    }
  }
}
```

如果数据有多层嵌套，JSON 通常比 CSV 更合适。

## 22. 什么时候适合用 CSV 驱动 Terraform？

适合：

- 简单表格数据
- 成本中心清单
- 用户或团队清单
- DNS 记录清单
- 服务端口清单
- 规则矩阵
- 由非工程人员维护的配置表
- 从 Excel / Google Sheets 导出的数据

不适合：

- 深度嵌套配置
- 有复杂条件逻辑的数据
- 高敏感信息
- 需要强 schema 校验的数据
- 需要频繁变化且影响生产资源身份的数据

CSV 很方便，但也容易变成“表格驱动事故”。

## 23. 生产建议

如果你要在生产 Terraform 中使用 CSV，建议：

1. CSV 表头规范化。
2. key 字段必须唯一。
3. 数字字段显式 `tonumber()`。
4. bool 字段统一写法，例如只允许 `true/false`。
5. 用 `terraform test` 验证关键字段。
6. 用 `for` 表达式转成 map。
7. 用 `for_each` 管理资源。
8. 不要把敏感数据放 CSV。
9. CSV 改动要走 PR review。
10. 对 CSV 数据结构写 README。

这不是小题大做。  
生产里“一个 CSV 表格改错一行”就可能导致 Terraform plan 大变。
