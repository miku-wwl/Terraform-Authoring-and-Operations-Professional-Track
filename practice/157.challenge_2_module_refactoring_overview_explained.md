## 3. 什么是模块重构？

Terraform module 本质上就是一组可复用的 Terraform 配置。你可以把一堆资源、变量、输出封装到一个目录里，然后在 root module 中通过 `module` block 调用它。

一个典型的模块结构可能长这样：

```text
practice/labs/158/
├── main.tf
├── variables.tf
├── outputs.tf
└── modules/
    └── app_bucket/
        ├── main.tf
        ├── variables.tf
        └── outputs.tf
```

其中：

- 根目录是 **root module**
- `modules/app_bucket/` 是 **child module**
- root module 负责调用模块
- child module 负责封装具体资源
- `variables.tf` 定义输入
- `outputs.tf` 暴露输出

考试里看到“refactor into module”“move resource into module”“reuse module”“module source”这些关键词，就要立刻想到模块重构题型。

---

## 4. 模块重构类挑战在考什么？

这类挑战通常不是考你背参数，而是考你能不能完成几个工程动作。

### 4.1 识别哪些代码应该被封装

比如原来 root module 里有一堆重复资源：

```hcl
resource "aws_s3_bucket" "app" {
  bucket = "demo-app-bucket"
}

resource "aws_s3_bucket" "logs" {
  bucket = "demo-logs-bucket"
}
```

如果多个环境都要创建类似结构，就可以考虑把 bucket 创建逻辑封装成 module。

### 4.2 设计 module 输入变量

模块不能把所有值写死，否则就失去复用意义。应该把可变化的内容变成变量，例如：

```hcl
variable "bucket_name" {
  type        = string
  description = "Name of the S3 bucket."
}
```

然后模块内部使用：

```hcl
resource "aws_s3_bucket" "this" {
  bucket = var.bucket_name
}
```

### 4.3 设计 module 输出

模块创建资源后，root module 往往需要使用资源 ID、ARN、名称等信息，所以需要输出：

```hcl
output "bucket_id" {
  value = aws_s3_bucket.this.id
}
```

root module 可以通过：

```hcl
module.app_bucket.bucket_id
```

来引用。

### 4.4 正确配置 module source

本地模块通常这样调用：

```hcl
module "app_bucket" {
  source      = "./modules/app_bucket"
  bucket_name = "demo-app-bucket"
}
```

考试重点是你要知道：

- `source` 指向模块位置
- 本地路径常见写法是 `./modules/xxx`
- 修改 module source 或新增 module 后，通常需要重新执行 `terraform init`
- module block 的参数要和模块的 input variable 对得上

## 5. module source 是什么？

`source` 是 Terraform 找到模块代码的地址。

常见来源包括：

```hcl
source = "./modules/network"
```

表示本地模块。

```hcl
source = "terraform-aws-modules/vpc/aws"
```

表示 Terraform Registry 模块。

```hcl
source = "git::https://github.com/example/terraform-modules.git//vpc"
```

表示 Git 仓库中的模块。

本训练项目更偏向本地可验证练习，因此你更常见到的是：

```hcl
source = "./modules/xxx"
```

考试里不要把 `source` 和 provider source 混淆：

```hcl
required_providers {
  aws = {
    source  = "hashicorp/aws"
    version = "~> 5.0"
  }
}
```

这里的 `source = "hashicorp/aws"` 是 provider source，不是 module source。

## 6. 输入、输出和依赖关系

模块重构最容易错在“搬了代码，但没接线”。

所谓接线，就是 root module 和 child module 之间的输入输出关系要连上。

### 6.1 输入变量方向

输入变量是 root module 传给 child module：

```hcl
module "app_bucket" {
  source      = "./modules/app_bucket"
  bucket_name = var.bucket_name
}
```

方向是：

```text
root module -> child module
```

### 6.2 输出值方向

输出值是 child module 暴露给 root module：

```hcl
output "bucket_arn" {
  value = aws_s3_bucket.this.arn
}
```

root module 使用：

```hcl
module.app_bucket.bucket_arn
```

方向是：

```text
child module -> root module
```

### 6.3 依赖关系

如果一个模块的输出要传给另一个模块，就会形成隐式依赖：

```hcl
module "app" {
  source     = "./modules/app"
  bucket_arn = module.bucket.bucket_arn
}
```

Terraform 会根据引用关系判断执行顺序。

## 7. 模块重构和 state 的关系

这是综合挑战中非常重要的点。

如果你只是把资源代码从 root module 搬到 child module，Terraform 的资源地址会发生变化。

原来可能是：

```text
aws_s3_bucket.app
```

搬进模块后会变成：

```text
module.app_bucket.aws_s3_bucket.this
```

Terraform state 里记录的是资源地址。如果地址变了，Terraform 可能会认为：

- 老资源不存在了，要销毁
- 新资源出现了，要创建

这就是模块重构里最危险的地方。

---

## 8. 为什么 state 迁移重要？

假设真实环境中已经有一个 bucket，由 Terraform 管理：

```text
aws_s3_bucket.app
```

你把它重构到模块后，如果不迁移 state，Terraform plan 可能显示：

```text
- destroy aws_s3_bucket.app
+ create module.app_bucket.aws_s3_bucket.this
```

这在生产环境非常危险。

正确思路通常是使用 `moved` block 或 `terraform state mv` 来告诉 Terraform：

> 这不是要删了重建，只是资源地址从旧位置移动到了新位置。

例如：

```hcl
moved {
  from = aws_s3_bucket.app
  to   = module.app_bucket.aws_s3_bucket.this
}
```

这类知识点会在模块重构挑战中非常关键。

---

## 9. `moved` block 和 `terraform state mv` 的区别

### 9.1 `moved` block

`moved` block 写在 Terraform 配置里，适合把 state 迁移关系记录进代码。

优点：

- 可审计
- 可 review
- 对团队协作更友好
- 迁移意图写在代码里

示例：

```hcl
moved {
  from = aws_s3_bucket.app
  to   = module.app_bucket.aws_s3_bucket.this
}
```

### 9.2 `terraform state mv`

`terraform state mv` 是命令式操作，直接修改 state 中的资源地址。

示例：

```bash
terraform state mv aws_s3_bucket.app module.app_bucket.aws_s3_bucket.this
```

它更像一次性手工操作。生产中使用前要备份 state，并且要确保团队知道这次变更。

考试里如果看到“配置中声明资源移动关系”，优先想到 `moved` block。  
如果看到“直接移动 state 地址”，才想到 `terraform state mv`。

## 10. 综合挑战的解题流程

做模块重构题时，不要一上来就乱改文件。建议按下面流程走。

### 第一步：读题

先确认题目到底要求什么：

- 是把资源移动到 module？
- 是修复 module source？
- 是补 input variable？
- 是补 output？
- 是处理 state 地址变化？
- 是让 plan 不出现 destroy/create？

很多题错，不是 Terraform 不会，而是题没读完就开始改。

### 第二步：定位资源

找到原来资源在哪里：

```bash
grep -R "resource" .
```

或者看目录结构：

```bash
find . -maxdepth 3 -type f
```

目标是确认：

- root module 有什么资源
- child module 有什么资源
- 哪些变量已经存在
- 哪些输出已经存在

### 第三步：设计模块边界

判断哪些东西放进 child module，哪些保留在 root module。

一般原则：

- 通用资源逻辑放进 module
- 环境差异通过 variable 传入
- root module 保留编排逻辑
- 输出只暴露调用方需要的值

### 第四步：修改配置

常见动作包括：

- 创建 `modules/xxx/`
- 移动 resource block
- 添加 module `variables.tf`
- 添加 module `outputs.tf`
- 在 root module 添加 `module` block
- 调整引用路径
- 添加 `moved` block 或准备 state 迁移

### 第五步：重新初始化

新增或修改模块后，通常需要执行：

```bash
terraform init
```

如果题目使用 LocalStack，也要确认 provider endpoint 仍然指向：

```text
http://localhost:4566
```

不要误连真实 AWS。

### 第六步：验证 plan

执行：

```bash
terraform plan
```

重点看：

- 是否有语法错误
- 是否变量缺失
- 是否输出引用错误
- 是否出现不该有的 destroy/create
- 是否符合题目期望

### 第七步：apply、验证、destroy

综合挑战通常要求闭环：

```bash
terraform apply
terraform output
terraform destroy
```

理论型 lab 则可能通过脚本验证 checklist。

本项目里的 AWS 练习默认使用 LocalStack：


- provider 配置
- module 调用
- 输入输出关系
- state 地址变化
- plan/apply/destroy 闭环

但不能完全代表真实 AWS 行为，例如：

- 真实 IAM 权限评估
- 真实网络连通性
- 真实服务配额
- 真实云费用

所以考试训练可以用 LocalStack，生产经验不能只靠 LocalStack 吹牛。这个边界要守住。

## 12. 高频考试陷阱

### 陷阱 1：以为 module 只是复制文件

错。模块化不是复制粘贴，而是抽象输入、输出和复用边界。

### 陷阱 2：忘记 `terraform init`

新增 module 或修改 module source 后，通常要重新 init。

### 陷阱 3：变量没有传进去

child module 定义了变量：

```hcl
variable "bucket_name" {
  type = string
}
```

root module 调用时却没传：

```hcl
module "app_bucket" {
  source = "./modules/app_bucket"
}
```

这会导致变量缺失错误。

### 陷阱 4：输出没有暴露

模块内部资源创建成功，不代表 root module 能直接访问内部资源。

应该通过 output 暴露：

```hcl
output "bucket_arn" {
  value = aws_s3_bucket.this.arn
}
```

然后 root module 使用：

```hcl
module.app_bucket.bucket_arn
```

### 陷阱 5：搬资源导致重建

这是模块重构中最危险的坑。

资源从：

```text
aws_s3_bucket.app
```

变成：

```text
module.app_bucket.aws_s3_bucket.this
```

Terraform 可能认为这是两个不同资源。

所以要关注 state 迁移，不能只看代码“长得对”。

### 陷阱 6：把 provider source 和 module source 混淆

Provider source：

```hcl
source = "hashicorp/aws"
```

Module source：

```hcl
source = "./modules/app_bucket"
```

这两个不是一回事。

## 16. 考试题型预测

### 选择题方向

1. `module` block 中 `source` 的作用是什么？
2. child module 需要从 root module 接收值时，应使用什么？
3. child module 创建的资源信息要暴露给 root module，应使用什么？
4. 将资源移动进 module 后，为避免 Terraform 误判销毁重建，应关注什么？
5. 新增本地 module 后，通常需要重新运行哪个命令？

### 判断题方向

1. Module 可以帮助复用 Terraform 配置。
2. Module source 和 provider source 是同一个概念。
3. 把资源代码移动到 module 后，state 地址可能发生变化。
4. Output 可以让 child module 向调用方暴露信息。
5. LocalStack 可以完全模拟真实 AWS 的 IAM、安全和网络行为。

### 场景题方向

1. 原来 root module 里有重复的 S3 bucket 资源，现在要重构为可复用模块，你会怎么做？
2. 重构后 plan 显示要 destroy 旧资源并 create 新资源，你会检查什么？
3. 模块内部创建了资源，但 root module 需要 ARN，你会怎么暴露？
4. 新同事把 provider source 和 module source 混淆了，你怎么解释？

## 17. 本节结论

第 157 节是综合挑战二的入口。你要记住的不是某个 AWS 参数，而是模块重构的完整工程思路：

```text
读题
 -> 定位资源
 -> 设计 module 边界
 -> 配置 input/output
 -> 调整 module source
 -> 关注 state 地址变化
 -> init/plan/apply/output/destroy 验证
```

真正能在考试和工作里拉开差距的，不是“知道 module 这个词”，而是你能不能在不破坏现有 state 的前提下，把 Terraform 配置重构成清晰、可复用、可维护的模块结构。
