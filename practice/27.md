# Terraform 实操训练 27：基础 heredoc 与 indented heredoc

## 1. 课程元数据

- 课程编号：27
- 来源文件：`doc/27.txt`
- 环境分类：`docker`
- 是否可用 Docker 运行：`yes`
- 核心主题：`<<EOT`、`<<-EOT`、前导空白
- 前置条件：Docker、`hashicorp/terraform:1.11`

## 2. 知识点讲解

基础 heredoc 会保留所有内容，包括前导空白。indented heredoc 使用 `<<-`，会移除所有行共同的最短前导空白，使 Terraform 代码本身可以保持缩进，同时生成文件更干净。

这个差异在生成 YAML、脚本和配置片段时非常重要。

## 3. 实验设计

背景：同一段文本分别用基础 heredoc 和 indented heredoc 输出。

问题：团队需要理解为什么两个文件的缩进不同。

任务：生成 `basic.txt` 和 `indented.txt` 并比较输出。

预期结果：基础 heredoc 保留外层缩进，indented heredoc 去掉共同缩进。

## 5. 运行手册

```sh
cd /workspace/practice/labs/27
terraform init -input=false
terraform fmt -check
terraform validate
terraform test
terraform plan -input=false -no-color -out=tfplan
terraform apply -auto-approve tfplan
terraform output
terraform destroy -auto-approve
```

## 6. 验证报告

状态：通过。

已执行 `init/fmt/validate/test/plan/apply/output/destroy`。`terraform test` 返回 `1 passed, 0 failed`，基础 heredoc 保留前导空白，indented heredoc 移除共同前导空白。

## 7. 排障指南

1. indented 没生效：确认使用 `<<-EOT`。
2. 输出仍有缩进：说明该缩进不是所有行共同拥有的部分。
3. 结束标记缩进异常：结束标记必须符合 heredoc 语法。
4. YAML 解析失败：检查相对缩进是否被破坏。

## 8. 清理方式

```sh
terraform destroy -auto-approve
rm -f tfplan
rm -rf output .terraform .terraform.lock.hcl
```
