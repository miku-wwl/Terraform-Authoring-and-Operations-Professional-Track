# Terraform 实操训练 58 讲解：条件表达式

## 1. 这一节到底在学什么？

Lab 58 学的是 Terraform 里的**条件表达式**。

条件表达式的基本形式是：

```hcl
condition ? true_value : false_value
```

它的意思是：

> 如果 condition 为 true，就使用 true_value；否则使用 false_value。

一句话理解：

> Terraform 条件表达式就是 HCL 里的“三元判断”，用来根据条件选择不同的值。

例如：

```hcl
var.environment == "prod" ? "large" : "small"
```

意思是：

```text
如果环境是 prod，就用 large；否则用 small。
```

---

## 2. 条件表达式的基本语法

Terraform 条件表达式格式：

```hcl
条件 ? 条件为真时的值 : 条件为假时的值
```

示例：

```hcl
locals {
  instance_size = var.environment == "prod" ? "large" : "small"
}
```

如果：

```hcl
var.environment = "prod"
```

那么：

```hcl
local.instance_size = "large"
```

如果：

```hcl
var.environment = "dev"
```

那么：

```hcl
local.instance_size = "small"
```

这就是最基础的条件选择。

---

## 3. 为什么 Terraform 需要条件表达式？

Terraform 是声明式语言。  
你通常不是写传统代码里的：

```text
if (...) {
  ...
} else {
  ...
}
```

而是在资源参数、locals、outputs、count、for_each 里根据条件选择值。

常见用途包括：

- prod 和 dev 使用不同规格
- 是否启用某个功能
- 是否创建某个资源
- 是否输出某个值
- 是否使用默认配置
- 是否启用监控
- 是否加额外 tag
- 是否打开 backup
- 是否启用高可用

条件表达式是 Terraform 配置动态化的基础工具。

---

## 4. 最简单的例子：根据环境选择规格

示例：

```hcl
variable "environment" {
  type    = string
  default = "dev"
}

locals {
  size = var.environment == "prod" ? "large" : "small"
}
```

这表达的是：

```text
prod 环境用 large
非 prod 环境用 small
```

这类写法在生产里很常见。

例如：

```hcl
instance_type = var.environment == "prod" ? "m6i.large" : "t3.micro"
```

或者：

```hcl
replicas = var.environment == "prod" ? 3 : 1
```

---

## 5. 条件必须返回同兼容类型的值

Terraform 条件表达式两边的返回值类型应该兼容。

推荐：

```hcl
var.enabled ? "enabled" : "disabled"
```

两边都是 string。

推荐：

```hcl
var.environment == "prod" ? 3 : 1
```

两边都是 number。

不推荐：

```hcl
var.enabled ? "enabled" : 0
```

一边是 string，一边是 number，容易造成类型问题。

Terraform 有时会尝试自动转换，但生产里别依赖隐式转换。  
你应该让 true 分支和 false 分支类型清楚一致。

---

## 6. 条件表达式和 bool 变量

很多开关可以直接用 bool。

示例：

```hcl
variable "enable_monitoring" {
  type    = bool
  default = true
}

locals {
  monitoring_mode = var.enable_monitoring ? "enabled" : "disabled"
}
```

如果：

```hcl
enable_monitoring = true
```

结果：

```text
enabled
```

如果：

```hcl
enable_monitoring = false
```

结果：

```text
disabled
```

bool 变量 + 条件表达式，是 Terraform 模块里非常常见的组合。

---

## 7. 条件表达式和 `count`

条件表达式经常用于控制资源是否创建。

示例：

```hcl
resource "local_file" "debug" {
  count = var.enable_debug ? 1 : 0

  filename = "${path.module}/output/debug.txt"
  content  = "debug enabled"
}
```

这里：

```hcl
var.enable_debug ? 1 : 0
```

意思是：

- `enable_debug = true`：创建 1 个资源
- `enable_debug = false`：创建 0 个资源

这就是 Terraform 里常见的“条件创建资源”。

---

## 8. `count = 0/1` 的引用坑

如果你用：

```hcl
count = var.enable_debug ? 1 : 0
```

那么资源可能不存在。

当 `count = 0` 时，下面这种引用会失败：

```hcl
local_file.debug[0].filename
```

因为根本没有第 0 个实例。

正确做法是也用条件保护：

```hcl
output "debug_file" {
  value = var.enable_debug ? local_file.debug[0].filename : null
}
```

这点和 Lab 55 的 `count` 索引挑战可以串起来。

---

## 9. 条件表达式和 `for_each`

条件表达式也可以用于 `for_each`。

例如，如果启用功能，就创建多个文件；否则一个都不创建：

```hcl
resource "local_file" "feature" {
  for_each = var.enabled ? var.features : {}

  filename = "${path.module}/output/${each.key}.txt"
  content  = each.value
}
```

这里要求：

```hcl
var.features
```

是 map。

如果 `enabled = true`，for_each 使用 `var.features`。  
如果 `enabled = false`，for_each 使用空 map `{}`，也就是创建 0 个实例。

注意：true 分支和 false 分支类型要兼容。

如果 true 分支是 map，false 分支也应该是 map。

---

## 10. 条件表达式和 null

Terraform 里经常用 `null` 表示“不设置这个参数”。

示例：

```hcl
locals {
  description = var.enable_description ? "managed by terraform" : null
}
```

如果 `enable_description = false`，结果是：

```hcl
null
```

对于很多 resource 参数来说，`null` 表示让 provider 使用默认行为，或者不传这个参数。

这在模块设计里很常见。

---

## 11. 条件表达式和 tags

根据环境加不同 tag：

```hcl
locals {
  base_tags = {
    ManagedBy = "Terraform"
    Project   = "training"
  }

  prod_tags = var.environment == "prod" ? {
    Criticality = "high"
    Backup      = "required"
  } : {}

  tags = merge(local.base_tags, local.prod_tags)
}
```

这里：

```hcl
var.environment == "prod" ? {...} : {}
```

表示：

- prod 环境加额外 tag
- 非 prod 环境不加额外 tag

这种写法非常实战。

---

## 12. 条件表达式和环境差异

条件表达式最常见的用途之一就是处理环境差异。

例如：

```hcl
locals {
  replica_count = var.environment == "prod" ? 3 : 1
  backup_enabled = var.environment == "prod" ? true : false
  log_retention_days = var.environment == "prod" ? 30 : 7
}
```

这表达了：

- prod 副本更多
- prod 开启备份
- prod 日志保留更久

这种写法清楚，但不要过度堆条件。  
如果环境差异很多，更适合用 map 配置表。

---

## 13. 条件表达式过多时，用 map 更清楚

如果你写很多这样的条件：

```hcl
instance_type = var.environment == "prod" ? "m6i.large" : "t3.micro"
replicas      = var.environment == "prod" ? 3 : 1
retention     = var.environment == "prod" ? 30 : 7
```

还可以接受。

但如果环境很多：

```text
dev
test
staging
prod
```

条件表达式会变得很乱。

这时更推荐 map：

```hcl
locals {
  env_config = {
    dev = {
      instance_type = "t3.micro"
      replicas      = 1
      retention     = 7
    }

    prod = {
      instance_type = "m6i.large"
      replicas      = 3
      retention     = 30
    }
  }

  current = local.env_config[var.environment]
}
```

然后使用：

```hcl
local.current.instance_type
local.current.replicas
local.current.retention
```

一句话：

> 少量二选一用条件表达式，多环境配置表用 map(object)。

---

## 14. 条件表达式嵌套：能用，但别滥用

Terraform 支持嵌套条件：

```hcl
locals {
  size = var.environment == "prod" ? "large" : var.environment == "test" ? "medium" : "small"
}
```

这表示：

- prod -> large
- test -> medium
- 其他 -> small

但是这种写法可读性很差。

更推荐 map：

```hcl
locals {
  sizes = {
    dev  = "small"
    test = "medium"
    prod = "large"
  }

  size = local.sizes[var.environment]
}
```

嵌套条件不是不能用，而是要控制复杂度。

## 19. 常见坑

### 坑 1：两个分支类型不兼容

不推荐：

```hcl
var.enabled ? "yes" : 0
```

推荐：

```hcl
var.enabled ? "yes" : "no"
```

或者：

```hcl
var.enabled ? 1 : 0
```

---

### 坑 2：嵌套条件太复杂

不推荐写很长：

```hcl
var.env == "prod" ? "large" : var.env == "test" ? "medium" : var.env == "dev" ? "small" : "tiny"
```

更推荐 map。

---

### 坑 3：条件创建资源后仍然直接引用

如果：

```hcl
count = var.enabled ? 1 : 0
```

那引用时要保护：

```hcl
var.enabled ? resource.example[0].id : null
```

---

### 坑 4：用条件表达式掩盖配置设计问题

如果到处都是条件，说明模块可能需要重新建模。

比如环境差异很多时，map(object) 往往更清楚。

---

### 坑 5：把条件写反

这很常见。

例如本来想 prod 用大规格，却写成：

```hcl
var.environment != "prod" ? "large" : "small"
```

所以测试很重要。

---

## 20. 条件表达式和变量验证的区别

条件表达式用于**选择值**：

```hcl
var.environment == "prod" ? 3 : 1
```

变量 validation 用于**拒绝非法输入**：

```hcl
validation {
  condition     = contains(["dev", "test", "prod"], var.environment)
  error_message = "environment must be dev, test, or prod."
}
```

不要混淆。

一个负责选择，一个负责校验。

成熟模块通常两个都会有：

1. validation 保证输入合法。
2. 条件表达式根据合法输入选择配置。

---

## 21. 条件表达式和 `try()` / `can()`

复杂对象里有些字段可能不存在，这时你可能会看到：

```hcl
try(var.config.replicas, 1)
```

或者：

```hcl
can(var.config.replicas) ? var.config.replicas : 1
```

`try()` 更适合处理“如果访问失败，就给默认值”的场景。

条件表达式更适合处理明确 bool 判断：

```hcl
var.environment == "prod" ? 3 : 1
```

不要把所有问题都塞进条件表达式。

---

## 22. 条件表达式和 `null` 的生产用法

很多资源参数可以根据条件传 `null`。

示例：

```hcl
locals {
  kms_key_id = var.enable_encryption ? var.kms_key_id : null
}
```

如果启用加密，就传 key。  
如果不启用，就传 null，让 provider 按默认逻辑处理。

但是要注意：

- 不是所有参数都接受 null。
- null 和空字符串 `""` 不是一回事。
- null 通常表示“不设置”。
- 空字符串通常表示“设置为空字符串”。

这在排障时很重要。

---

## 23. 条件表达式的可读性建议

好的写法：

```hcl
locals {
  is_prod = var.environment == "prod"

  replica_count = local.is_prod ? 3 : 1
  retention_days = local.is_prod ? 30 : 7
}
```

比到处重复写：

```hcl
var.environment == "prod" ? ...
```

更清楚。

如果条件会重复使用，先放进 local：

```hcl
locals {
  is_prod = var.environment == "prod"
}
```

然后复用：

```hcl
local.is_prod ? true_value : false_value
```

这就是可维护写法。

---

## 24. 生产环境里的典型用法

### 按环境决定副本数

```hcl
replicas = var.environment == "prod" ? 3 : 1
```

### 按开关决定是否创建

```hcl
count = var.enabled ? 1 : 0
```

### 按环境决定日志保留

```hcl

```

### 按开关决定是否传参数

```hcl
kms_key_id = var.enable_encryption ? var.kms_key_id : null
```

### 按环境合并 tag

```hcl
tags = merge(
  local.common_tags,
  var.environment == "prod" ? local.prod_tags : {}
)
```

这些都是很实际的 Terraform 写法。

## 26. SRE / Platform Engineering 视角

从平台工程角度看，条件表达式用于表达策略差异。

例如：

- prod 比 dev 更高可用
- prod 备份必须开启
- prod 日志保留更久
- dev 可以低成本运行
- 某些功能按开关启用
- 某些参数只有在特定 provider 或环境下传入

但是平台模块不能被条件表达式堆成意大利面。

好的设计是：

- 简单二选一：用条件表达式
- 多环境配置：用 map(object)
- 多个资源实例：用 for_each
- 输入合法性：用 validation
- 可选字段：用 optional / null
- 复杂异常处理：考虑 try/can

这才是成熟 Terraform 模块设计。

## 27. 面试/考试记忆版

可以这样记：

- 条件表达式格式是 `condition ? true_value : false_value`。
- condition 必须能判断为 bool。
- true 分支和 false 分支类型应该兼容。
- 常用于环境差异、开关逻辑、资源数量、参数选择。
- `count = var.enabled ? 1 : 0` 是常见写法。
- `for_each = var.enabled ? var.map : {}` 也是常见写法。
- 复杂多分支不要硬嵌套，优先考虑 map。
- 条件创建资源后，引用资源时要处理不存在的情况。
- `null` 可以表达“不设置参数”，但要看 provider 是否支持。
- validation 是校验输入，条件表达式是选择值。

一句话总结：

> Terraform 条件表达式适合做简单分支选择，但如果条件越来越多，就该升级成更清楚的数据建模。

## 28. 本 Lab 的价值

Lab 58 的价值在于，它让 Terraform 配置从“静态写死”变成“根据输入选择”。

没有条件表达式，你只能写固定配置。  
有了条件表达式，你可以表达：

- dev/prod 差异
- 是否启用功能
- 是否创建资源
- 是否传入参数
- 是否合并额外配置

但你也要记住：

> 条件表达式是工具，不是万能补丁。用少了配置死板，用多了代码混乱。

真正成熟的 Terraform 写法，是在条件表达式、map/object 建模、for_each、validation 之间做取舍。
