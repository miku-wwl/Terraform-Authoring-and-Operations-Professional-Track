# Terraform 实操训练 65 讲解：`templatefile` 基础

## 1. 这一节到底在学什么？

Lab 65 学的是 Terraform 里的 `templatefile()` 函数。

前面你已经学过：

- heredoc：在 `.tf` 文件里写多行字符串
- `jsonencode()`：用 Terraform 数据结构生成 JSON
- `csvdecode()`：读取 CSV 数据
- `for` 表达式：转换数据
- `flatten()` / `distinct()`：处理复杂集合

这一节的 `templatefile()` 解决的是另一个常见问题：

> 当你需要生成一份比较长、比较像“配置文件”的文本时，不要把所有内容都硬塞在 `.tf` 里，而是把模板单独放到文件里，再用 Terraform 传变量进去渲染。

一句话理解：

> `templatefile()` 是 Terraform 用外部模板文件生成文本内容的工具。

---

## 2. `templatefile()` 是什么？

`templatefile()` 是 Terraform 内置函数，用来读取一个模板文件，并把变量填进去。

基本语法：

```hcl
templatefile(path, vars)
```

其中：

- `path`：模板文件路径
- `vars`：传给模板的变量 map

示例：

```hcl
locals {
  rendered = templatefile("${path.module}/templates/config.tftpl", {
    app_name = "api"
    port     = 8080
  })
}
```

模板文件 `templates/config.tftpl` 可以写：

```text
app_name=${app_name}
port=${port}
```

渲染后结果是：

```text
app_name=api
port=8080
```

---

## 3. 为什么需要 `templatefile()`？

如果只是一两行字符串，用普通字符串或 heredoc 就够了。

例如：

```hcl
content = "hello ${var.name}"
```

或者：

```hcl
content = <<EOT
app=${var.app}
port=${var.port}
EOT
```

但如果内容变长，例如：

- Nginx 配置
- cloud-init user data
- systemd unit file
- application config
- shell script
- Kubernetes YAML
- Prometheus scrape config
- JSON / YAML 配置文件
- README 或报告片段

继续塞在 `.tf` 文件里会很乱。

这时更好的方式是：

```text
main.tf                放 Terraform 资源逻辑
templates/config.tftpl 放文本模板
```

这样结构更清楚。

---

## 4. 基本示例：生成配置文件

Terraform：

```hcl
resource "local_file" "app_config" {
  filename = "${path.module}/output/app.conf"

  content = templatefile("${path.module}/templates/app.conf.tftpl", {
    app_name    = "payments"
    environment = "dev"
    port        = 8080
  })
}
```

模板文件 `templates/app.conf.tftpl`：

```text
app_name=${app_name}
environment=${environment}
port=${port}
```

生成文件：

```text
app_name=payments
environment=dev
port=8080
```

这个模式非常常见：

```text
Terraform 数据 -> templatefile 渲染 -> local_file / cloud-init / config
```

---

## 5. 模板变量从哪里来？

`templatefile()` 的第二个参数是一个 map：

```hcl
{
  app_name = var.app_name
  port     = var.port
}
```

模板里可以用同名变量：

```text
${app_name}
${port}
```

注意：

> 模板里只能访问你显式传进去的变量。

如果你没有传：

```hcl
environment = var.environment
```

模板里却写：

```text
${environment}
```

Terraform 会报错。

这和普通 Terraform 表达式不一样。  
模板文件不是自动拥有所有 `.tf` 里的变量作用域。

---

## 6. 模板文件扩展名

Terraform 不强制模板文件扩展名。

常见写法包括：

```text
config.tftpl
userdata.sh.tftpl
app.yaml.tftpl
nginx.conf.tftpl
```

推荐使用：

```text
.tftpl
```

因为它能清楚表达：

> 这是 Terraform template 文件，不是最终配置文件。

例如：

```text
templates/user-data.sh.tftpl
```

比：

```text
templates/user-data.sh
```

更不容易误解。

---

## 7. `path.module` 为什么重要？

常见写法：

```hcl
templatefile("${path.module}/templates/app.conf.tftpl", {
  app_name = var.app_name
})
```

`path.module` 表示当前模块目录。

这样无论你从哪个工作目录执行 Terraform，都能正确找到模板文件。

不推荐写相对路径：

```hcl
templatefile("templates/app.conf.tftpl", {...})
```

因为在模块、CI/CD、不同执行目录里可能出现路径问题。

一句话：

> 模板文件路径优先用 `${path.module}` 拼接。

## 11. `templatefile()` 和 heredoc 的区别

heredoc 是把多行字符串直接写在 `.tf` 文件里：

```hcl
content = <<EOT
app=${var.app_name}
port=${var.port}
EOT
```

`templatefile()` 是把多行模板放到外部文件：

```hcl
content = templatefile("${path.module}/templates/app.conf.tftpl", {
  app_name = var.app_name
  port     = var.port
})
```

对比：

| 方式 | 适合场景 |
|---|---|
| heredoc | 内容短、只在一个地方用 |
| templatefile | 内容长、需要单独维护、像配置文件或脚本 |

简单记忆：

> 小段文本用 heredoc，大段配置用 templatefile。

---

## 12. `templatefile()` 和 `jsonencode()` 的区别

如果你要生成 JSON，很多时候更推荐 `jsonencode()`，而不是手写 JSON 模板。

推荐：

```hcl
content = jsonencode({
  app_name = var.app_name
  port     = var.port
})
```

不太推荐：

```hcl
content = templatefile("${path.module}/templates/config.json.tftpl", {
  app_name = var.app_name
  port     = var.port
})
```

原因是手写 JSON 模板容易出现：

- 逗号错误
- 引号错误
- 转义错误
- 数字和字符串混淆
- bool 类型写错

如果目标格式是 JSON，`jsonencode()` 通常更安全。

但如果目标是：

- shell script
- nginx.conf
- systemd unit
- cloud-init
- YAML
- 普通文本配置

`templatefile()` 就很有价值。

---

## 13. 模板里可以写条件吗？

Terraform template 支持模板指令，比如条件和循环。

简单条件示例：

```text
%{ if enable_metrics }
metrics_enabled=true
%{ endif }
```

Terraform：

```hcl
templatefile("${path.module}/templates/app.conf.tftpl", {
  enable_metrics = true
})
```

如果 `enable_metrics = true`，会输出：

```text
metrics_enabled=true
```

如果是 false，这段不会输出。

不过 Lab 65 是基础，重点先掌握变量替换。  
条件和循环可以后面再深入。

---

## 14. 模板里可以写循环吗？

可以。

例如传入 ports：

```hcl
templatefile("${path.module}/templates/ports.tftpl", {
  ports = [80, 443, 8080]
})
```

模板：

```text
%{ for port in ports ~}
port=${port}
%{ endfor ~}
```

输出：

```text
port=80
port=443
port=8080
```

注意 `~` 可以控制空白和换行。  
模板循环很有用，但也容易写乱。

如果可以用 `jsonencode()` 或 `yamlencode()` 生成结构化格式，通常更稳。

---

## 15. 空白和换行问题

模板渲染最容易遇到的问题之一是空白和换行。

例如：

```text
%{ for item in items }
${item}
%{ endfor }
```

可能产生多余空行。

Terraform 模板语法支持 `~` 修剪空白：

```text
%{ for item in items ~}
${item}
%{ endfor ~}
```

这能减少多余换行。

生产里生成脚本或 YAML 时，空白很关键。  
测试应该覆盖关键输出片段，避免模板改坏。

---

## 16. 模板变量类型

传给 `templatefile()` 的值可以是：

- string
- number
- bool
- list
- map
- object

例如：

```hcl
templatefile("${path.module}/templates/config.tftpl", {
  app = {
    name = "api"
    port = 8080
  }

  tags = {
    Environment = "dev"
    Owner       = "platform"
  }

  ports = [80, 443]
})
```

模板里可以访问：

```text
${app.name}
${app.port}
${tags.Environment}
```

也可以配合 `%{ for ... }` 遍历 list/map。

---

## 17. 传入变量建议：先在 locals 整形

生产里不建议在 `templatefile()` 第二个参数里写一堆复杂表达式。

不太好：

```hcl
content = templatefile("${path.module}/templates/app.tftpl", {
  name = lower(trimspace(var.service.name))
  port = tonumber(var.service.port)
  tags = merge(local.common_tags, var.service.tags)
})
```

更好：

```hcl
locals {
  template_vars = {
    name = lower(trimspace(var.service.name))
    port = tonumber(var.service.port)
    tags = merge(local.common_tags, var.service.tags)
  }
}

resource "local_file" "app" {
  content = templatefile("${path.module}/templates/app.tftpl", local.template_vars)
}
```

这样：

- 可读性更好
- 测试更容易
- output 调试更方便
- 模板输入结构更清晰

## 18. 模板文件不存在怎么办？

如果路径写错，比如：

```hcl
templatefile("${path.module}/template/app.tftpl", {...})
```

但实际目录是：

```text
templates/app.tftpl
```

Terraform 会报错，提示找不到文件。

排查：

1. 检查目录名。
2. 检查文件名。
3. 检查扩展名。
4. 检查 `${path.module}` 是否正确。
5. 检查文件是否被复制到模块目录。
6. 检查 CI 环境是否包含模板文件。

模板文件是模块的一部分，不能忘记提交到 Git。

---

## 19. 模板变量不存在怎么办？

如果模板里写：

```text
app=${app_name}
env=${environment}
```

但 Terraform 只传了：

```hcl
{
  app_name = "api"
}
```

没有传 `environment`，就会报错。

解决：

```hcl
templatefile("${path.module}/templates/app.tftpl", {
  app_name    = "api"
  environment = "dev"
})
```

模板变量必须显式传入。

这是最常见的 `templatefile()` 初学者错误之一。

## 20. `templatefile()` 和 sensitive

如果模板内容里包含敏感信息，比如 password、token、private key，要非常小心。

例如：

```hcl
content = templatefile("${path.module}/templates/app.tftpl", {
  password = var.password
})
```

生成内容可能进入：

- Terraform state
- plan
- output
- 本地文件
- 日志
- CI artifact

`sensitive = true` 只能减少 CLI 输出，不等于 state 加密。

所以不要随便用 `templatefile()` 生成明文 secret 文件，除非你很清楚风险和保护措施。

---

## 21. 适合使用 `templatefile()` 的场景

适合：

- cloud-init user data
- shell script
- nginx config
- systemd unit
- application config
- Prometheus scrape config
- README / 报告片段
- 非 JSON 的文本配置
- 需要人类可读模板的长文本

不太适合：

- 简单一行字符串
- 可以用 `jsonencode()` 安全生成的 JSON
- 高敏感 secret
- 极复杂逻辑模板
- 应该由应用自己管理的运行时配置

一句话：

> templatefile 适合生成文本配置，不适合把 Terraform 变成通用模板引擎乱写业务逻辑。

---

## 22. 常见坑

### 坑 1：模板路径写错

优先使用：

```hcl
"${path.module}/templates/xxx.tftpl"
```

不要依赖执行目录。

---

### 坑 2：模板变量没传

模板里用到的变量必须出现在第二个参数 map 中。

---

### 坑 3：JSON 用模板手写

JSON 更推荐 `jsonencode()`，少踩转义和逗号坑。

---

### 坑 4：模板逻辑太复杂

如果模板里充满条件、循环、嵌套判断，说明你可能该先在 locals 里把数据整理好。

---

### 坑 5：输出敏感信息

模板渲染结果可能进入 state 或文件。  
不要轻易把 secret 放进去。

---

### 坑 6：多余空行导致格式错误

生成 YAML、shell script、nginx config 时，空白和缩进可能影响结果。

必要时使用 `~` 控制模板空白。

## 23. Terraform test 在这里有什么价值？

Lab 65 的验证报告显示：

```text
terraform test 返回 1 passed, 0 failed
```

对 `templatefile()` 来说，测试可以验证：

- 模板是否能成功渲染
- 变量是否传全
- 输出是否包含预期内容
- 文件是否生成
- 渲染内容是否符合格式
- 条件或循环是否按预期工作
- 没有因为路径错误导致失败

模板最怕“看着像文本，实际少一个变量或多一个空行”。  
测试能把这类问题提前抓出来。

## 26. SRE / Platform Engineering 视角

从平台工程角度看，`templatefile()` 常用于生成启动配置和平台配置。

例如：

- VM user data
- bootstrap script
- agent config
- observability config
- sidecar 配置
- deployment 说明文件
- 自动生成的配置片段

但平台工程师要控制边界：

- Terraform 负责生成基础配置
- 应用运行时配置不要全塞 Terraform
- secret 不要随便落文件
- JSON 优先用 `jsonencode()`
- 模板逻辑不要过度复杂
- 模板输出要可测试

这才是生产级用法。

## 27. 面试/考试记忆版

可以这样记：

- `templatefile(path, vars)` 读取模板文件并渲染变量。
- 模板变量必须通过第二个参数显式传入。
- 模板里用 `${name}` 引用变量。
- 模板文件路径建议用 `${path.module}`。
- 长文本配置适合 `templatefile()`。
- 简单多行文本可用 heredoc。
- JSON 通常更推荐 `jsonencode()`。
- 模板文件建议使用 `.tftpl` 后缀。
- 模板内容可能进入 state，敏感信息要谨慎。
- 复杂数据先在 locals 整形，再传给模板。

一句话总结：

> `templatefile()` 的核心价值，是把长文本模板从 Terraform 资源逻辑中拆出来，让配置生成更清晰、更可维护、更容易测试。
