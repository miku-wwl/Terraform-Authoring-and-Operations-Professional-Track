# Terraform 实操训练 43 讲解：HashiCorp Vault 基础

## 1. 这一节到底在学什么？

Lab 43 学的是 Terraform 和 HashiCorp Vault 的基础概念。

这节不是让你把 Vault 生产化部署起来，而是先建立一个非常关键的安全认知：

> Vault 可以集中管理 secret，但 Terraform 读取到 secret 后，secret 仍然可能进入 Terraform state。

所以这节的重点不是“Vault 很安全，所以万事大吉”，而是：

- 为什么需要 Vault
- Vault dev server 适合做什么
- Terraform 和 Vault 集成时有什么风险
- 为什么 state 依然要按敏感资产管理
- 为什么生产环境不能使用 dev token

一句话总结：

> Vault 解决 secret 集中管理问题，但不自动解决 Terraform state 泄密问题。

---

## 2. Vault 是什么？

HashiCorp Vault 是一个 secret 管理工具，用来集中存储、读取、审计和控制敏感数据。

常见 secret 包括：

- 数据库用户名和密码
- API token
- TLS 证书
- SSH key
- Cloud credential
- 动态数据库凭据
- 短期访问 token

如果没有 Vault，团队很容易把 secret 散落在：

- Git 仓库
- `.tfvars` 文件
- CI/CD 环境变量
- 本地脚本
- README
- 运维人员电脑
- Slack / 邮件 / 文档截图

这就是典型的 secret 管理失控。

Vault 的目标是把这些敏感数据集中起来管理，并提供访问控制、审计和生命周期管理。

---

## 3. 为什么 Terraform 项目会需要 Vault？

Terraform 经常需要连接云平台、数据库、Kubernetes、第三方服务。  
这些连接通常需要凭据。

例如：

```hcl
variable "db_password" {
  type      = string
  sensitive = true
}
```

这种方式可以跑，但生产里有明显问题：

- 密码可能被写进 `.tfvars`
- `.tfvars` 可能被误提交
- CI 日志可能泄露变量
- 密码轮换困难
- 谁访问过 secret 不好审计
- secret 分散在多个地方

Vault 的思路是：

> Terraform 不直接硬编码 secret，而是从 Vault 读取需要的 secret。

这样至少能把 secret 来源集中起来。

---

## 4. 本实验的场景

Lab 43 的背景是：

> 团队准备把数据库凭据从代码仓库迁移到 Vault。

问题是：

> 密钥直接进入 Terraform 代码和日志会造成泄露。

任务是：

1. 记录 Vault 治理原则。
2. 使用 Vault dev server 做 smoke test。
3. 写入并读取 `secret/db_creds`。
4. Terraform 测试通过。

预期结果：

- Terraform 完整流程通过。
- Vault CLI 可以写入并读取 secret。
- 你理解 Vault 不等于 state 自动安全。

---

## 5. Vault dev server 是什么？

Vault dev server 是 Vault 的开发模式。

它非常适合本地实验、教学、临时 smoke test。

典型特点：

- 启动快
- 配置简单
- 默认已经 unseal
- 使用固定 root token
- 数据通常是临时的
- 不适合生产

本实验使用的是：

```text
hashicorp/vault:1.18
```

然后用脚本启动 dev server，写入并读取一个测试 secret。

这适合验证：

- Vault CLI 能不能启动
- dev server 能不能读写 secret
- smoke test 脚本是否可用
- secret 路径是否正确

但它不验证生产级 Vault 的高可用、持久化、审计、认证、授权和密钥轮换。

---

## 6. 为什么不能在生产使用 dev token？

Vault dev server 通常会使用 root token。  
root token 权限极高，类似“上帝模式”。

生产里如果使用 dev token，会有严重风险：

- 权限过大
- 无法体现最小权限
- 容易被泄露
- 审计意义弱
- 不适合长期使用
- 不符合生产安全要求

所以要记住：

> dev server 是实验玩具，不是生产架构。

生产 Vault 应该使用正式的认证方式，例如：

- AppRole
- Kubernetes Auth
- OIDC / JWT
- Cloud IAM Auth
- LDAP / SSO
- Token with least privilege

并且要配置 policy，限制 Terraform 只能读取它需要的路径。

---

## 7. Terraform 读取 Vault secret 的真正风险

这是本节最重要的点。

很多人会以为：

> Secret 放进 Vault 之后，Terraform 就安全了。

不完全对。

如果 Terraform 通过 provider 或 data source 读取 Vault secret，这个 secret 可能会进入：

- Terraform state
- Terraform plan
- Terraform output
- provider debug log
- CI/CD 缓存
- 本地 `.terraform` 相关文件

所以 Vault 只是 secret 来源更安全，不代表 Terraform 处理 secret 的整个链路都自动安全。

关键结论：

> 只要 Terraform 读到了 secret，就必须假设 state 可能包含 secret。

---

## 8. Terraform state 为什么仍然敏感？

Terraform state 是 Terraform 的事实来源。  
它记录资源属性、data source 读取结果、输出值等信息。

如果 Terraform 读取了 Vault 里的数据库密码，那么这个密码可能进入 state。

即使 CLI 显示：

```text
(sensitive value)
```

state 里仍可能保存真实值。

所以 state 后端要按 secret 级别保护：

- 启用远程 backend
- 启用加密
- 限制读取权限
- 开启审计
- 不要提交到 Git
- 不要随便下载分发
- CI/CD 中限制 artifact 暴露
- plan 文件也按敏感文件处理

这和 Lab 42 的 `sensitive` 是连起来的：  
`sensitive = true` 只是减少 CLI 明文输出，不是 state 加密。

## 9. Vault smoke test 在做什么？

本实验里的脚本：

```text
practice/labs/43/scripts/vault-dev-smoke.sh
```

作用是用 Vault CLI 做最小可用性验证。

运行命令：

```sh
docker run --rm --cap-add IPC_LOCK -v "${PWD}:/workspace" -w /workspace hashicorp/vault:1.18 sh practice/labs/43/scripts/vault-dev-smoke.sh
```

这条命令大概做了几件事：

1. 用 Docker 启动 Vault 工具环境。
2. 在容器里运行 smoke test 脚本。
3. 启动 Vault dev server。
4. 写入测试 secret。
5. 读取 `secret/db_creds`。
6. 脚本结束后容器自动删除。

`--rm` 表示容器退出后自动清理。  
`--cap-add IPC_LOCK` 是 Vault 常见运行参数，用于允许内存锁定相关能力，减少敏感数据被 swap 到磁盘的风险。dev 实验里主要是保持 Vault 运行要求一致。

## 10. 标准 Terraform 运行流程

进入实验目录：

```sh
cd /workspace/practice/labs/43
```

初始化：

```sh
terraform init -input=false
```

格式检查：

```sh
terraform fmt -check
```

配置校验：

```sh
terraform validate
```

运行测试：

```sh
terraform test
```

生成 plan：

```sh
terraform plan -input=false -no-color -out=tfplan
```

执行 apply：

```sh
terraform apply -auto-approve tfplan
```

查看输出：

```sh
terraform output
```

清理：

```sh
terraform destroy -auto-approve
```

清理本地文件：

```sh
rm -f tfplan
rm -rf .terraform .terraform.lock.hcl
```

## 13. 常见坑

### 坑 1：以为 Vault 解决所有 secret 风险

Vault 只是 secret 管理系统。  
Terraform 读取 secret 后，secret 仍可能进入 Terraform state。

所以你必须同时保护：

- Vault
- Terraform state
- plan 文件
- CI/CD 日志
- runner 文件系统
- output
- provider debug log

---

### 坑 2：生产使用 dev server

dev server 只能本地实验用。  
生产 Vault 需要正式部署和治理，包括：

- storage backend
- seal / unseal 流程
- auth method
- policy
- audit device
- backup
- HA
- monitoring
- token TTL
- rotation

---

### 坑 3：root token 随便用

root token 权限太大。  
生产中应该按最小权限创建 policy 和 token。

Terraform 只需要读某个路径，就不要给它全局权限。

---

### 坑 4：把 secret 输出出来

即使来自 Vault，也不要随便：

```hcl
output "db_password" {
  value = data.vault_generic_secret.db.data["password"]
}
```

如果确实需要输出，也要：

```hcl
sensitive = true
```

更好的做法是不要输出 secret 原文。

---

### 坑 5：把 plan 文件当普通文件

`tfplan` 可能包含敏感变更和 secret 相关信息。  
不要提交、上传或随便转发。

清理：

```sh
rm -f tfplan
```

---

## 14. 生产环境更推荐的模式

在生产里，Terraform 和 secret 的关系要谨慎设计。

### 模式 1：Terraform 只创建 secret 容器，不读取 secret 原文

例如 Terraform 创建 Vault path、Azure Key Vault secret name、AWS Secrets Manager secret metadata，但不把 secret value 放入 Terraform。

优点：

- state 里不保存 secret 原文
- secret 由 CI/CD 或运维流程写入
- Terraform 管结构，不管秘密内容

---

### 模式 2：应用运行时读取 secret

应用通过 runtime identity 读取 secret，例如：

- Kubernetes Service Account + Vault Agent
- AWS IAM Role + Secrets Manager
- Azure Managed Identity + Key Vault
- Vault Agent Injector

Terraform 只配置访问关系，不直接读取 secret value。

这是更成熟的生产模式。

---

### 模式 3：Terraform 读取 secret，但严格保护 state

如果确实需要 Terraform 读取 secret，那就必须接受 state 风险，并加强保护：

- remote backend 加密
- 严格 IAM 权限
- 最小化 output
- 禁止 debug log 泄露
- CI runner 权限控制
- 定期轮换 secret
- 审计 state 访问

这不是不能做，但要知道代价。

## 16. 面试/考试记忆版

可以这样记：

- Vault 是 secret 管理系统。
- Vault dev server 只适合实验，不适合生产。
- Terraform 可以读取 Vault secret。
- Terraform 读取到的 secret 可能进入 state。
- `sensitive = true` 不等于加密。
- state backend 必须按敏感资产保护。
- 生产不要使用 dev token 或 root token。
- 更成熟的模式是应用运行时读取 secret，Terraform 尽量不接触 secret 原文。

一句话总结：

> Vault 让 secret 来源更集中，但 Terraform 一旦读取 secret，state 就可能成为新的泄露点。

---

## 17. 本 Lab 的价值

Lab 43 的价值不是让你“会跑 Vault dev server”这么简单。

真正价值是建立生产级安全边界意识：

- secret 不应该写死在代码里
- Vault 可以集中管理 secret
- dev server 只能做实验
- Terraform state 仍然危险
- secret 管理要覆盖完整链路
- 最好让 Terraform 少碰 secret 原文

这对 SRE / Platform Engineering 很关键。  
成熟工程师不会只说“用了 Vault 就安全了”，而是会追问：

> Terraform state 里有没有？谁能读 state？plan 文件会不会泄露？CI 日志有没有打印？token 有没有最小权限？

这才是生产级 secret 管理思维。
