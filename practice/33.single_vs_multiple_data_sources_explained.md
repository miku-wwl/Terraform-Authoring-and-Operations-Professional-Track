# Terraform AWS Sandbox 33 讲解：单实例与多实例 Data Source

## 1. 这一节到底在学什么？

这一节的核心不是“怎么创建 EC2”，而是学习 Terraform 如何**读取已经存在的 EC2 实例**。

在 Terraform 里，`resource` 是创建和管理资源，`data source` 是读取外部已经存在的信息。本 Lab 33 专门对比两个很容易混淆的 AWS EC2 data source：

- `data "aws_instance"`：读取**单个** EC2 实例
- `data "aws_instances"`：读取**多个** EC2 实例

这两个名字只差一个 `s`，但语义完全不一样。真实生产里，如果用错，轻则查不到资源，重则把错误资源传给后续模块，导致部署打到错误实例、错误 VPC、错误环境。

---

## 2. 为什么要区分 `aws_instance` 和 `aws_instances`？

### `aws_instance`：我只想要一个明确对象

`aws_instance` 适合这种场景：

> 我明确知道我要找的是哪一台 EC2，例如生产环境的一台 bastion、某个固定的跳板机、某个被 tag 唯一标识的实例。

它的关键特征是：**过滤条件应该只匹配一个实例**。

如果真实 AWS provider 发现过滤结果匹配了多个实例，通常会报错。这个设计是合理的，因为 Terraform 不应该帮你“猜”哪一台才是你真正想要的。

典型用法：

```hcl
data "aws_instance" "production" {
  filter {
    name   = "tag:Team"
    values = ["production"]
  }

  filter {
    name   = "instance-state-name"
    values = ["running"]
  }
}
```

这里表达的是：
“请帮我找一台 `Team=production`，并且状态是 running 的实例。”

如果这个条件能匹配多台，那说明你的过滤条件不够精确。

---

### `aws_instances`：我就是想要一组对象

`aws_instances` 适合这种场景：

> 我需要获取一组实例 ID，例如某个 lab、某个环境、某个 Auto Scaling 之外的实例集合。

它返回的是列表，例如：

```hcl
data "aws_instances" "lab" {
  filter {
    name   = "tag:Project"
    values = ["tf-lab-33"]
  }
}
```

后续可以输出数量：

```hcl
output "all_lab_instance_count" {
  value = length(data.aws_instances.lab.ids)
}
```

也可以把实例 ID 列表传给模块、脚本或其他资源。

## 3. 本实验的设计思路

课程原本关注 AWS EC2 查询，但为了不消耗真实 AWS 资源，本实验改成 LocalStack sandbox。

实验流程是：

1. `bootstrap.sh` 先在 LocalStack 中创建两台模拟 EC2。
2. 其中一台实例打上 `Team=production` 标签。
3. Terraform 用 `aws_instance` 精确读取 production 实例。
4. Terraform 用 `aws_instances` 读取所有 lab 实例。
5. 输出一个 production 实例 ID，以及 lab 实例总数。

这个设计非常好，因为它把“单个对象查询”和“多个对象查询”放在同一个 lab 里对比。

---

## 4. 你应该记住的核心区别

| 对比项 | `aws_instance` | `aws_instances` |
|---|---|---|
| 查询目标 | 单个 EC2 实例 | 多个 EC2 实例 |
| 返回结果 | 一个对象 | 一组列表 |
| 常用输出 | `.id`、`.private_ip`、`.arn` 等单值属性 | `.ids`、`.private_ips` 等列表属性 |
| 过滤条件要求 | 必须足够精确，最好只匹配一个 | 可以匹配多个 |
| 典型用途 | 读取唯一 bastion、唯一生产实例、唯一固定节点 | 读取某个环境的一组实例 |
| 常见错误 | 过滤太宽，匹配多个实例 | 忘记它返回的是 list |

一句话总结：

> `aws_instance` 是“给我那一台”；`aws_instances` 是“给我这一批”。

---

## 5. 为什么 tag filter 很重要？

生产环境里，data source 查询最怕两件事：

1. **过滤太宽**：读到了不该读的资源。
2. **过滤太窄**：什么都读不到。

例如只按 `Team=production` 过滤，可能不够安全。因为一个账号里可能有多个 production 实例。

更稳的做法是组合多个条件：

```hcl
filter {
  name   = "tag:Team"
  values = ["production"]
}

filter {
  name   = "tag:Project"
  values = ["payment-platform"]
}

filter {
  name   = "instance-state-name"
  values = ["running"]
}
```

这样能把查询范围收得更窄，减少误读资源的风险。

在 SRE / Platform Engineering 视角里，这其实是一个治理问题：资源 tag 不规范，Terraform data source 就很难可靠查询。所以 tag 不是装饰品，是自动化系统的索引。

## 6. LocalStack 在这里验证了什么？

这个 lab 里 LocalStack 主要验证这些 Terraform 能力：

- AWS provider endpoint 是否正确指向 sandbox
- Terraform 是否能通过 data source 查询 EC2
- `aws_instance` 和 `aws_instances` 的返回结构差异
- output 是否能正确引用 data source 结果
- bootstrap / verify / clean 工作流是否完整

但是要注意：

> LocalStack 不等价于真实 AWS。

它可以帮助你练 Terraform 语法、provider endpoint、data source 查询模式，但不能完整模拟真实 AWS 的 EC2 网络、AMI、metadata、计费和 IAM 行为。

所以这个 lab 的定位应该是：

> 用 LocalStack 练 Terraform data source 的工程套路，而不是验证真实 EC2 生产行为。

## 7. 标准运行流程

进入实验目录：

```sh
cd practice/labs/33
```

设置假 AWS 凭据和 LocalStack endpoint：

```sh
export AWS_ACCESS_KEY_ID=test
export AWS_SECRET_ACCESS_KEY=test
export AWS_DEFAULT_REGION=us-east-1
export LOCALSTACK_ENDPOINT=http://localhost:4566
```

检查 sandbox：

```sh
bash scripts/check-sandbox.sh
```

创建前置 EC2：

```sh
bash scripts/bootstrap.sh
```

执行 Terraform：

```sh
terraform init -input=false
terraform fmt -check
terraform validate
terraform plan -input=false -no-color -out=tfplan
terraform apply -auto-approve tfplan
terraform output
```

验证结果：

```sh
bash scripts/verify.sh
```

清理：

```sh
terraform destroy -auto-approve
bash scripts/clean.sh
```

最后清理本地 Terraform 文件：

```sh
rm -rf .terraform .terraform.lock.hcl tfplan terraform.tfstate terraform.tfstate.backup
```

## 8. 运行结果应该怎么看？

关键结果应该包括：

```text
all_lab_instance_count = 2
```

这说明 `aws_instances` 成功读到了两台 lab 实例。

同时还应该有一个 production 实例 ID：

```text
production_instance_id = "i-..."
```

这说明 `aws_instance` 成功通过 `Team=production` 精确匹配到单个实例。

你真正要看的不是实例 ID 长什么样，而是这两个事实：

1. 单实例 data source 能通过 tag 精确匹配。
2. 多实例 data source 能返回实例集合。

## 9. 常见坑

### 坑 1：`aws_instance` 匹配了多个实例

这通常说明 filter 太宽。

解决方式：增加更多条件，例如：

- `tag:Project`
- `tag:Environment`
- `tag:Name`
- `instance-state-name`
- `vpc-id`
- `subnet-id`

---

### 坑 2：`aws_instances` 返回列表，但你当成单值用

错误思路：

```hcl
data.aws_instances.lab.id
```

`aws_instances` 返回的是多个实例，所以通常要用：

```hcl
data.aws_instances.lab.ids
```

如果只想拿第一个：

```hcl
data.aws_instances.lab.ids[0]
```

但生产里要谨慎使用 `[0]`，因为列表顺序不应该被当成业务逻辑。

---

### 坑 3：查询结果为空

先检查 bootstrap 是否执行：

```sh
bash scripts/bootstrap.sh
```

再检查 LocalStack endpoint 和 region 是否一致。

如果 Terraform provider 指到了真实 AWS，而 AWS CLI 指到了 LocalStack，就会出现“CLI 看得到，Terraform 看不到”的错觉。

---

### 坑 4：误打真实 AWS

这个是最危险的。

如果输出里出现真实账号、真实区域、真实资源 ID，就要立刻停下来检查 provider endpoint。

sandbox 实验里应确认 provider 明确配置了 LocalStack endpoint，并且凭据是：

```sh
AWS_ACCESS_KEY_ID=test
AWS_SECRET_ACCESS_KEY=test
```

## 10. 生产环境中的真实用法

这个知识点在生产里很常见，例如：

- 读取已有 bastion host
- 读取已有 VPC 中的实例
- 读取蓝绿部署中的某一组节点
- 根据 tag 找到某个环境的机器
- 把实例 ID 传给监控、告警、SSM、负载均衡或自动化脚本

但是生产环境要加一条铁律：

> data source 的 filter 必须足够明确，不能靠“应该只有一个”这种玄学假设。

工程上更稳的做法是：

1. 统一 tag 规范。
2. 给环境、项目、owner、用途都打 tag。
3. 查询时组合多个 filter。
4. 对输出数量做验证。
5. 避免对列表顺序产生依赖。

## 11. 面试/考试记忆版

可以这样记：

- `data source`：读取已有资源，不创建资源。
- `aws_instance`：读取一台 EC2，filter 必须唯一。
- `aws_instances`：读取多台 EC2，返回列表。
- `filter`：生产环境必须写清楚，避免误读资源。
- `tag`：不是装饰，是自动化查询的索引。
- LocalStack：适合练 provider endpoint 和 data source 查询，但不等于真实 AWS。

一句话答案：

> `aws_instance` 用来精确读取一个实例，`aws_instances` 用来读取一组实例；前者怕匹配多个，后者要按列表处理。

## 12. 本 Lab 的价值

Lab 33 是 AWS Terraform sandbox 里很实用的一节。它不追求复杂资源创建，而是训练你一个生产里高频出现的动作：

> Terraform 如何安全、准确地读取已经存在的云资源。

这类能力对 Platform Engineer / SRE 很重要。因为真实公司里不可能所有资源都从零开始由你创建，更多时候你需要把 Terraform 接入已有环境，然后可靠地读取、引用和治理这些资源。
