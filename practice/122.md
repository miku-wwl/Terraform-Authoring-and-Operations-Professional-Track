## 2. 知识点讲解

HCP Terraform workspace 是平台中的核心执行单元。一个 workspace 通常关联某个 Terraform 配置来源，可以是 GitHub 等 VCS 仓库，也可以是 CLI-driven 或 API-driven 的来源。课程重点演示了 Version Control Workflow：workspace 连接到代码仓库，从仓库中拉取 Terraform 配置，然后执行 plan、policy check、cost estimation 和 apply 等步骤。

Workspace 不只是代码链接。它还维护变量、环境变量、敏感值、state、Terraform 版本、工作目录、执行模式、触发规则和 run 历史。生产环境里，一个 workspace 往往代表一套具体基础设施边界，例如 `network-prod`、`app-dev` 或 `security-shared`。

考试容易混淆的地方是“workspace”和“目录”的关系。Git 仓库可以有多个目录，一个 workspace 可以指向仓库中的特定工作目录。代码在 Git 中保存，运行和 state 在 HCP Terraform 中管理。修改代码后，workspace 可以根据 VCS 配置触发新的 run。

## 3. 考试选择题

1. Version Control Workflow 中，HCP Terraform workspace 通常从哪里获取 Terraform 代码？
   - A. 已连接的 Git 仓库
   - B. 本地浏览器缓存
   - C. Docker Hub 镜像标签
   - D. Terraform Registry 的 provider 二进制文件

2. Workspace 可以管理哪些内容？
   - A. 变量、state、run 历史、Terraform 版本和代码来源
   - B. 只管理本地 `.ssh` 目录
   - C. 只管理 Markdown 文档
   - D. 只管理操作系统用户

3. 如果一个仓库中有多个 Terraform 子目录，workspace 通常需要配置什么？
   - A. working directory
   - B. CPU 核心数
   - C. 浏览器语言
   - D. Git 用户头像

## 4. 判断题

1. Workspace 可以与 VCS 仓库关联，并基于代码变更触发运行。
2. Workspace 只保存 Terraform 代码，不保存任何 run 或 state 信息。
3. 生产中常用多个 workspace 区分环境、系统或团队边界。

## 5. 场景题

1. 一个 Git 仓库下有 `network/` 和 `app/` 两套 Terraform 配置。团队希望分别管理它们的 state 和审批。应该如何设计 workspace？

2. 开发者提交代码后，HCP Terraform 自动生成 plan，但 apply 需要人工确认。这个流程体现了哪些 workspace 能力？

## 6. 参考答案与解析

选择题：1-A，2-A，3-A。

判断题：1-对，2-错，3-对。

场景题参考：

1. 可以创建两个 workspace，分别指向同一仓库中的不同 working directory，让 `network` 和 `app` 拥有独立 state、变量和 run 历史。
2. 体现了 VCS 集成、自动 plan、run 历史、审批控制和集中执行能力。

## 7. 复习提醒

- Workspace 是 HCP Terraform 的运行单元，不只是代码目录。
- Version Control Workflow 的关键词是 Git、透明追踪、自动触发和审批。
- 看到 working directory，要想到单仓库多 Terraform 配置的场景。
