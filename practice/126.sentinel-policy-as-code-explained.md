# 第 126 节讲解：Sentinel Policy as Code

> 来源：`126.md`  
> 主题：Sentinel、Policy Set、Enforcement Mode、Policy Check、Apply 阻断  
> 适用场景：Terraform Professional / HCP Terraform 治理类考点复习

---

## 1. 一句话讲明白

**Sentinel 是 HCP Terraform 里的策略守门员。**

Terraform 先生成 plan，Sentinel 再检查这个 plan 是否符合组织规则。  
如果规则不通过，就根据 enforcement mode 决定：

- 只是提醒你；
- 允许有权限的人强行通过；
- 还是直接硬性阻断 apply。

所以它的核心位置是：

```text
代码提交 / Run 触发
        ↓
terraform plan
        ↓
Sentinel policy check
        ↓
通过后才允许 terraform apply
```

考试里只要看到：

```text
policy as code
compliance
tag enforcement
allowed instance types
mandatory policy
soft mandatory
hard mandatory
advisory
apply blocked
```

大概率就是在考 Sentinel 或 HCP Terraform policy check。

---

## 2. Sentinel 到底解决什么问题？

Terraform 本身负责“创建资源”。  
但是企业环境里，问题不是“能不能创建”，而是：

- 你创建的资源有没有标签？
- 你是不是用了太贵的实例？
- 生产环境是否禁止公开 S3 bucket？
- 是否只能在指定 region 部署？
- 是否必须使用 approved module？
- 是否必须满足安全、成本、合规要求？

这些都属于**治理规则**。

如果只靠人工 code review，很容易漏。  
Sentinel 的思路是：**把治理规则写成代码，让每次 Terraform run 自动检查。**

可以理解为：

```text
Terraform plan：我要创建什么
Sentinel policy：你这样创建合不合规
```

---

## 3. Sentinel 在 Terraform 工作流中的位置

Sentinel 不是在资源创建完成之后才检查。  
它是在 **plan 之后，apply 之前** 检查。

完整流程可以理解为：

```text
1. 开发者提交 Terraform 代码
2. HCP Terraform 触发 run
3. 执行 terraform plan
4. 生成计划：将新增、修改、删除哪些资源
5. Sentinel 读取 plan 结果
6. 根据 policy 判断是否合规
7. 如果通过，才允许 apply
8. 如果失败，根据 enforcement mode 决定是否阻断
```

关键点：

```text
Sentinel 检查的是 plan，不是 apply 后的真实资源。
```

这很重要，因为治理应该发生在变更落地之前。  
不然等资源已经创建出来，再发现不合规，就变成事故补救了。

## 4. Policy、Policy Set、Workspace 的关系

这三个概念考试很容易连着考。

### 4.1 Policy

Policy 就是一条策略规则。

例如：

```text
所有 EC2 必须有 Owner 标签
所有资源必须有 CostCenter 标签
不允许使用 t3.2xlarge 以上的实例
生产环境不允许 public subnet 自动分配公网 IP
```

它回答的问题是：

```text
这条规则具体检查什么？
```

---

### 4.2 Policy Set

Policy set 是一组 policy 的集合。

比如安全团队可以建一个 policy set：

```text
security-baseline-policy-set
├── require-owner-tag
├── require-costcenter-tag
├── deny-public-s3
└── restrict-open-security-group
```

它回答的问题是：

```text
把哪些 policy 打包成一组，并应用到哪些 workspace？
```

所以 policy set 更像“策略包”。

---

### 4.3 Workspace

Workspace 是 HCP Terraform 里实际承载 run、state、variables、VCS 配置的执行单元。

比如：

```text
project: platform
├── workspace: network-dev
├── workspace: network-prod
├── workspace: aks-dev
└── workspace: aks-prod
```

Policy set 可以关联到一个或多个 workspace。  
这样每个 workspace 在运行 plan 后，都会自动执行对应的 Sentinel policy check。

```text
Organization

    │   └── Run → Plan → Sentinel Check → Apply
    │
    ├── Workspace: app-prod

    │

```

一句话记忆：

```text
Policy 是规则。
Policy set 是规则包。
Workspace 是规则生效的运行环境。
```

## 6. Enforcement Mode：考试最高频混淆点

Enforcement mode 决定 policy 失败后怎么办。

常见理解如下：

| Enforcement mode | 中文理解 | 失败后效果 | 考试关键词 |
|---|---|---|---|
| Advisory | 只提醒 | 不阻断 apply | warning、informational、不强制 |
| Soft mandatory | 软强制 | 默认阻断，但有权限的人可以 override | override、authorized user |
| Hard mandatory | 硬强制 | 绝对阻断，不能 override | apply blocked、cannot override、must comply |

---

## 7. Advisory、Soft Mandatory、Hard Mandatory 怎么区分？

### 7.1 Advisory：只是提醒

场景：

```text
公司建议所有资源都加 Environment 标签，但目前只是试运行治理规则。
```

如果 policy 失败：

```text
提示：你没有 Environment 标签
但 apply 仍然可以继续
```

这就是 advisory。

记忆方式：

```text
advisory = 顾问模式 = 我提醒你，但不拦你
```

---

### 7.2 Soft Mandatory：可以被授权人员覆盖

场景：

```text
生产资源必须带 CostCenter 标签。
但紧急修复时，平台管理员可以临时 override。
```

如果 policy 失败：

```text
普通用户不能继续 apply
有权限的管理员可以 override 后继续
```

这就是 soft mandatory。

记忆方式：

```text
soft mandatory = 默认强制，但能被授权例外处理
```

---

### 7.3 Hard Mandatory：绝对阻断

场景：

```text
生产环境绝不允许 0.0.0.0/0 暴露 22 端口。
```

如果 policy 失败：

```text
任何人都不能 apply
必须改代码让 policy 通过
```

这就是 hard mandatory。

记忆方式：

```text
hard mandatory = 铁门，谁来都不好使
```

---

## 8. 典型场景：所有生产资源必须有 Owner 和 CostCenter 标签

安全团队要求：

```text
所有生产资源必须包含：
- Owner
- CostCenter
```

否则不能部署。

用 Sentinel 可以这样做：

```text
1. 编写 Sentinel policy
   检查 Terraform plan 中即将创建或修改的资源是否包含 Owner 和 CostCenter 标签

2. 把 policy 放进 policy set
   例如 production-tagging-policy-set

3. 把 policy set 关联到生产 workspace
   例如 app-prod、network-prod、database-prod

4. 当生产 workspace 执行 run 时
   terraform plan 先生成变更计划

5. Sentinel 检查 plan
   如果发现资源缺少标签，则 policy check 失败

6. 根据 enforcement mode 决定结果
   - advisory：只提醒
   - soft mandatory：可由授权人员 override
   - hard mandatory：直接阻断 apply
```

如果题目说：

```text
否则不能部署
```

那通常倾向于：

```text
hard mandatory
```

如果题目说：

```text
可以由管理员例外批准
```

那通常倾向于：

```text
soft mandatory
```

如果题目说：

```text
只提醒，不阻断
```

那就是：

```text
advisory
```

---

## 9. Sentinel 和 Terraform plan 的关系

Sentinel 的强大之处在于它可以基于 plan 做判断。

Terraform plan 里会告诉你：

```text
哪些资源会被创建
哪些资源会被修改
哪些资源会被删除
资源的属性是什么
资源的标签是什么
实例类型是什么
region 是什么
```

Sentinel policy 可以读取这些信息，然后判断：

```text
这个变更能不能进入 apply 阶段？
```

例如：

```text
plan 显示要创建 aws_instance
instance_type = m5.24xlarge
```

如果公司策略禁止高成本实例，Sentinel 可以失败这个 run。

## 10. 一个伪代码例子：禁止缺少标签的资源

真实 Sentinel 语法考试一般不会要求你写。  
但理解逻辑很有帮助。

伪代码大概是：

```text
for each resource in terraform_plan:
    if resource is being created or updated:
        if resource.tags does not contain "Owner":
            fail
        if resource.tags does not contain "CostCenter":
            fail

pass only if all checked resources are compliant
```

真正考试重点不是语法，而是你要知道：

```text
这是 policy as code。
它检查 plan。
失败后是否阻断，由 enforcement mode 决定。
```

## 11. Sentinel vs 普通 Terraform 校验

很多人会把 Sentinel 和 Terraform 自带功能混在一起。

| 能力 | 主要解决什么 | 运行位置 |
|---|---|---|
| `terraform fmt` | 格式化代码 | 本地或 CI |
| `terraform validate` | 检查配置语法和基本结构 | plan 之前 |
| variable validation | 检查变量输入是否合法 | plan 阶段 |
| precondition / postcondition | 检查资源或输出条件 | plan/apply 相关 |
| Sentinel | 组织级治理和合规策略 | HCP Terraform plan 之后、apply 之前 |

考试看到：

```text
organization-wide policy
compliance
governance
policy set
enforcement
block apply
```

不要选 `terraform fmt`、`terraform validate`、provider cache 这种本地能力。  
这些不是组织级 policy as code 治理。

## 12. Sentinel vs OPA / Conftest

实际工程里你可能听过 OPA、Conftest、Checkov、tfsec。  
考试里如果题目明确说 HCP Terraform / HashiCorp policy as code，那么优先想到 Sentinel。

简单区分：

| 工具 | 常见定位 |
|---|---|
| Sentinel | HashiCorp/HCP Terraform 的 policy as code |
| OPA / Rego | 通用 policy engine |
| Conftest | 用 OPA/Rego 检查配置文件 |
| Checkov / tfsec | IaC 静态扫描、安全规则扫描 |

Terraform Professional 考试里，Sentinel 的高频点通常不是“和 OPA 谁更好”，而是：

```text
policy set
workspace 绑定
enforcement mode
plan 之后 apply 之前
```

## 14. 常见错误理解

### 错误 1：Sentinel 在 apply 后检查

不对。

正确是：

```text
plan 之后，apply 之前
```

它是为了在资源创建前阻止不合规变更。

---

### 错误 2：Policy set 是 workspace

不对。

正确是：

```text
Policy set 是一组 policy。
Workspace 是运行 Terraform 的执行单元。
Policy set 可以关联到 workspace。
```

---

### 错误 3：Advisory 会阻断 apply

不对。

Advisory 只是提醒，不阻断。

---

### 错误 4：Hard mandatory 可以被管理员 override

不对。

Hard mandatory 不能 override。  
如果能被授权人员覆盖，那是 soft mandatory。

---

### 错误 5：Sentinel 负责下载 provider

不对。

下载 provider 是 `terraform init` 和 provider installation 的事情。  
Sentinel 负责 policy check。

## 15. 用一个门禁系统来记

可以把 HCP Terraform 的 run 想成进公司大楼：

```text
terraform plan = 你要带什么东西进门的清单
Sentinel = 门口安检
policy = 安检规则
policy set = 一整套安检规则
workspace = 这个门禁规则生效的大楼入口
enforcement mode = 安检拦人的力度
```

三种力度：

```text
advisory：保安说“兄弟注意点”，但放你进去
soft mandatory：保安拦住你，经理签字才能进
hard mandatory：谁签字都没用，不能进
```

这个比喻非常适合记考试题。

## 16. 和真实项目怎么对应？

在真实公司里，Sentinel 常用于平台工程、SRE、Cloud Governance 场景。

比如平台团队可以定义：

```text
生产 workspace 必须满足：
- 所有资源有 Owner 标签
- 所有资源有 CostCenter 标签
- 禁止 public S3 bucket
- 禁止过大的 instance type
- 只能部署到批准 region
- 只能使用公司批准的 module source
```

这样开发团队仍然可以自助使用 Terraform，但不会随便创建不合规资源。

这就是平台工程经常说的：

```text
Self-service with guardrails
```

也就是：

```text
让团队自己部署，但边界由平台提前设好。
```

## 17. 本节选择题解析

### 题 1

> Sentinel 在 HCP Terraform 中主要用于什么？

正确答案：

```text
A. 对 Terraform plan 执行策略检查
```

解析：

Sentinel 是 policy as code。  
它不是 provider 下载器，不是 state 压缩工具，也不是 Git commit 生成器。

---

### 题 2

> Policy set 的作用更接近什么？

正确答案：

```text
A. 将一组 policy 组织起来并关联到 workspace 范围
```

解析：

Policy set 的重点是“成组管理策略 + 分发到 workspace”。  
这就是企业治理里的批量绑定机制。

---

### 题 3

> 如果策略失败后绝对不允许 apply，最接近哪类 enforcement？

正确答案：

```text
A. hard mandatory
```

解析：

绝对不允许 apply，不能 override，就是 hard mandatory。

## 19. 场景题参考答案

### 场景题 1

> 安全团队要求所有生产资源必须包含 `Owner` 和 `CostCenter` 标签，否则不能部署。请说明 Sentinel 可以如何参与这个流程。

参考答案：

```text
可以编写 Sentinel policy 检查 Terraform plan 中的资源标签，确认生产资源是否包含 Owner 和 CostCenter。
然后把这些 policy 放入 policy set，并关联到生产 workspace。
当 workspace 执行 run 时，HCP Terraform 会先生成 plan，再执行 Sentinel policy check。
如果资源缺少必需标签，则 policy check 失败。
由于题目要求“否则不能部署”，应该使用 hard mandatory 这类强制 enforcement mode 来阻断 apply。
```

---

### 场景题 2

> 某策略失败后只提醒用户，但不阻断 apply。这个行为更接近强制阻断，还是 advisory 类型治理？

参考答案：

```text
更接近 advisory。
Advisory policy 失败后只提供提醒或警告，不会阻断 apply。
如果要强制阻断，则应使用 hard mandatory。
如果允许授权人员覆盖，则是 soft mandatory。
```


## 21. 最容易出的考点

1. **Sentinel 检查 Terraform plan。**
2. **Policy set 可以关联多个 workspace。**
3. **Enforcement mode 决定失败后是否阻断。**
4. **Advisory 不阻断。**
5. **Soft mandatory 可以被授权 override。**
6. **Hard mandatory 不能 override。**
7. **Sentinel 位于 plan 之后、apply 之前。**
8. **它是组织治理能力，不是本地格式化或 provider 下载能力。**
