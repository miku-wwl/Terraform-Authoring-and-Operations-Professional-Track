# Terraform 实操训练 44 讲解：Vault Provider

## 1. 这一节到底在学什么？

Lab 44 学的是 Terraform 如何通过 **Vault provider** 和 HashiCorp Vault 交互。

上一节 Lab 43 主要是 Vault 基础：用 Vault dev server 做 smoke test，理解 secret 管理和 state 风险。  
这一节更进一步：Terraform 不只是旁边跑一个 Vault CLI，而是真正通过 provider 去管理 Vault 资源。

本节核心点：

- Terraform 可以使用 Vault provider 管理 Vault。
- 可以创建 KV v2 secret mount。
- 可以写入 secret。
- 可以读取 secret。
- Vault secret 被 Terraform 读取后，仍然要警惕 state、输出、日志泄露。
- sensitive 输出要正确处理。
- Vault dev server 默认已有 `secret/` mount，所以本实验改用 `training/`。

一句话总结：

> Vault provider 让 Terraform 能操作 Vault，但 secret 一旦进入 Terraform，就必须继续按敏感数据处理。

---

## 2. Vault Provider 是什么？

Terraform provider 是 Terraform 和外部系统交互的插件。

AWS provider 让 Terraform 操作 AWS。  
AzureRM provider 让 Terraform 操作 Azure。  
Vault provider 就是让 Terraform 操作 Vault。

它可以做很多事情，例如：

- 创建 secret engine mount
- 配置 KV v2
- 写入 secret
- 读取 secret
- 配置 policy
- 配置 auth backend
- 管理 token、identity、namespace 等 Vault 对象

本实验只练最基础的一条路径：

> Terraform 启动后连接 Vault dev server，创建 `training/` KV v2 mount，写入并读取 `db_creds`。

---

## 3. 为什么这节要用 `training/`，不用 `secret/`？

Vault dev server 默认会启用一个 `secret/` KV mount。

如果你在 Terraform 里再尝试创建一个同名的 `secret/` mount，就可能发生冲突。

所以本实验最后改成：

```text
training/
```

也就是 Terraform 自己创建并管理一个新的 KV v2 mount。

这点很重要，因为生产环境里也经常遇到类似问题：

- 目标 mount 已经存在
- Terraform 想创建同名资源
- state 里没有这个资源
- apply 时报冲突
- 需要 import 或换路径

本实验用 `training/` 是为了避免和 dev server 默认配置打架，让训练流程更干净。

---

## 4. KV v2 是什么？

Vault 的 KV secret engine 用来存储键值对 secret。

KV 有两个常见版本：

- KV v1：简单 key-value，没有版本历史。
- KV v2：支持版本化、软删除、metadata。

本实验使用 KV v2。

典型路径类似：

```text
training/data/db_creds
```

但在 Terraform Vault provider 里，具体资源和 data source 会帮你处理一部分路径细节。你要理解的重点是：

> KV v2 不只是普通路径，它有版本化语义。

这也是为什么 Vault provider 里通常会看到 `kv_v2` 相关资源或 data source。

## 5. 本实验的场景

Lab 44 的背景是：

> Terraform 需要从 Vault 读取数据库凭据。

问题是：

> 直接把密码写进 `.tf` 文件不可接受。

任务是：

1. 启动本地 Vault dev server。
2. Terraform 连接 Vault。
3. 使用 Vault provider 创建 `training` KV v2 mount。
4. 写入 `db_creds`。
5. 读取 `db_creds`。
6. 输出 username 为 `app`。
7. 确认 password 字段存在。

预期结果：

- Vault provider 流程通过。
- username 输出为 `app`。
- password 不直接泄露，只输出训练用派生值。
- Terraform destroy 能清理资源。

## 6. 为什么输出 sensitive 会出问题？

Terraform 对 sensitive 值比较严格。

如果你读取 Vault secret，然后直接 output：

```hcl
output "password" {
  value = data.vault_kv_secret_v2.db_creds.data["password"]
}
```

Terraform 可能会阻止或提示你，因为这个值是敏感的。

正确做法通常有两类。

### 做法 1：输出也标记 sensitive

```hcl
output "password" {
  value     = data.vault_kv_secret_v2.db_creds.data["password"]
  sensitive = true
}
```

这适合你确实需要输出 secret，但不想在普通 CLI 输出中明文展示。

### 做法 2：输出非敏感派生值

例如只输出 password 是否存在：

```hcl
output "password_is_configured" {
  value = nonsensitive(contains(keys(data.vault_kv_secret_v2.db_creds.data), "password"))
}
```

这个实验采用的是训练友好的方式：  
不输出 password 原文，只输出派生状态。

这比把密码打印出来安全得多。

---

## 7. `nonsensitive()` 是什么？

`nonsensitive()` 用来告诉 Terraform：

> 我确认这个表达式结果不是敏感值，可以当普通值输出。

例如：

```hcl
nonsensitive(true)
```

或者：

```hcl
nonsensitive(data.vault_kv_secret_v2.db_creds.data["username"])
```

但你要小心。  
不要为了绕过 Terraform 的保护，随便对真正的 secret 原文使用 `nonsensitive()`。

错误示范：

```hcl
output "password" {
  value = nonsensitive(data.vault_kv_secret_v2.db_creds.data["password"])
}
```

这会把敏感信息强行变成普通输出，容易泄露。

本实验里更合理的是输出：

- username：训练值，可以作为非敏感演示
- password 是否存在：布尔值，不泄露密码原文

一句话：

> `nonsensitive()` 不是脱敏工具，而是“我确认这个值不是 secret”的声明。

## 8. 本实验为什么使用 Docker 网络？

运行手册里有这几步：

```powershell
docker network create tf-vault-lab
```

这会创建一个 Docker 网络。

然后启动 Vault：

```powershell
docker run -d --rm --name vault-lab-44 --network tf-vault-lab --network-alias vault --cap-add IPC_LOCK -e VAULT_DEV_ROOT_TOKEN_ID=root hashicorp/vault:1.18
```

注意这里：

```powershell
--network-alias vault
```

它让同一 Docker 网络里的其他容器可以通过主机名：

```text
vault
```

访问 Vault 容器。

然后 Terraform 容器也加入同一个网络：

```powershell
docker run --rm --name tf-practice-44 --network tf-vault-lab -v "${PWD}:/workspace" -w /workspace --entrypoint sh hashicorp/terraform:1.11 practice/labs/44/scripts/run-with-vault.sh
```

这样 Terraform 容器里可以访问：

```text
http://vault:8200
```

这就是为什么排障指南里说：

> 无法解析 `vault`：确认 Vault 容器有 `--network-alias vault`。

## 9. 为什么设置 `VAULT_DEV_ROOT_TOKEN_ID=root`？

运行 Vault dev server 时设置了：

```powershell
-e VAULT_DEV_ROOT_TOKEN_ID=root
```

这表示 dev server 的 root token 是：

```text
root
```

这样实验脚本可以稳定连接 Vault，不用每次解析随机生成的 token。

但这只适合实验。

生产中不要这么干。  
生产环境必须使用更安全的认证方式和最小权限 policy。

## 10. 标准运行流程

本实验不是普通 `terraform init && terraform apply` 就完事，因为它需要先启动 Vault。

### 第一步：创建 Docker 网络

```powershell
docker network create tf-vault-lab
```

### 第二步：启动 Vault dev server

```powershell
docker run -d --rm --name vault-lab-44 --network tf-vault-lab --network-alias vault --cap-add IPC_LOCK -e VAULT_DEV_ROOT_TOKEN_ID=root hashicorp/vault:1.18
```

### 第三步：运行 Terraform 实验容器

```powershell
docker run --rm --name tf-practice-44 --network tf-vault-lab -v "${PWD}:/workspace" -w /workspace --entrypoint sh hashicorp/terraform:1.11 practice/labs/44/scripts/run-with-vault.sh
```

### 第四步：停止 Vault 容器

```powershell
docker stop vault-lab-44
```

### 第五步：删除 Docker 网络

```powershell
docker network rm tf-vault-lab
```

这个流程的关键是：

> Vault 容器和 Terraform 容器必须在同一个 Docker 网络中。

## 11. `run-with-vault.sh` 大概做了什么？

脚本通常会做这些事情：

1. 设置 Vault 地址，例如 `VAULT_ADDR=http://vault:8200`。
2. 设置 Vault token，例如 `VAULT_TOKEN=root`。
3. 等待 Vault dev server 可用。
4. 进入 Lab 44 目录。
5. 执行 Terraform init / fmt / validate / test / plan / apply / output。
6. 执行 Terraform destroy。
7. 清理本地文件。

这类脚本的价值是把实验流程封装起来，让你不需要手动在容器里一步步操作。

## 12. 实验输出应该怎么看？

预期输出重点不是 password 原文，而是：

- username 为 `app`
- password 字段存在
- Terraform 流程通过

可以理解为：

```text
username = app

```

这说明 Terraform 成功通过 Vault provider 写入并读取了 secret，同时没有把 password 原文作为普通输出暴露。

## 13. 为什么 Terraform 读取 Vault secret 后仍然危险？

这是 Lab 43 和 Lab 44 连续强调的核心坑。

Vault 本身是 secret 管理系统，但 Terraform 读取 secret 后，secret 可能进入：

- Terraform state
- Terraform plan
- Terraform output
- provider debug log
- CI/CD 日志
- runner 工作目录
- 缓存文件

所以不要以为：

> 密码来自 Vault，所以 Terraform 这里就一定安全。

正确理解是：

> Vault 保护 secret 的来源；Terraform state 仍然必须保护。

尤其是本实验中，Terraform 通过 Vault provider 写入和读取 secret，这就更要注意 state 风险。

---

## 14. 真实生产中应该怎么限制 token 权限？

实验里用的是 root token：

```text
root
```

生产里不能这样。

生产 Terraform 应该使用最小权限 token，只允许它访问必要路径。

例如只允许：

```text
read training/data/db_creds
```

或者只允许某个环境的路径：

```text
read kv/data/prod/app/*
```

不应该给 Terraform：

- 全局 root 权限
- 所有 secret 读写权限
- policy 管理权限
- auth backend 管理权限
- audit 配置权限

最小权限原则很重要：

> Terraform 需要什么，就给什么；不需要的，一律不给。

---

## 15. 生产里更成熟的模式

虽然本实验演示 Terraform 写入和读取 secret，但生产里要谨慎。

更推荐的模式有三类。

### 模式 1：Terraform 只管理 Vault 结构，不管理 secret 原文

Terraform 创建：

- mount
- policy
- auth method
- role
- metadata

但 secret value 由单独流程写入。

优点是 Terraform state 不保存 secret 原文。

---

### 模式 2：应用运行时读取 secret

应用通过 Vault Agent、Kubernetes Auth、AppRole 或云身份运行时读取 secret。

Terraform 只配置访问关系，不读取 secret value。

这是更适合生产应用的方式。

---

### 模式 3：Terraform 读取 secret，但严格保护 state

如果必须读取，那就接受 state 风险，并做好：

- remote backend 加密
- IAM 最小权限
- state 访问审计
- plan 文件保护
- CI 日志清理
- output 最小化
- secret 定期轮换

这不是不能做，但要知道代价。

---

## 16. 常见坑

### 坑 1：无法解析 `vault`

错误现象：

```text
lookup vault: no such host
```

原因通常是 Terraform 容器和 Vault 容器不在同一个 Docker 网络，或者 Vault 容器没有设置：

```powershell
--network-alias vault
```

解决：

```powershell
docker network create tf-vault-lab
```

并确保两个容器都使用：

```powershell
--network tf-vault-lab
```

---

### 坑 2：mount 冲突

Vault dev server 默认已有：

```text
secret/
```

如果 Terraform 再创建 `secret/`，可能冲突。

本实验使用：

```text
training/
```

就是为了避开这个问题。

---

### 坑 3：输出 sensitive 报错

如果直接输出 Vault secret，Terraform 会提醒 sensitive 问题。

解决方案：

- 输出标记 `sensitive = true`
- 或者只输出非敏感派生值
- 不要随便对密码原文 `nonsensitive()`

---

### 坑 4：token 失败

实验里要确认：

```text
VAULT_DEV_ROOT_TOKEN_ID=root
```

同时 Terraform 脚本里要使用匹配的 token。

如果 token 不一致，会认证失败。

---

### 坑 5：忘记清理 Docker 网络

实验结束后要清理：

```powershell
docker stop vault-lab-44
docker network rm tf-vault-lab
```

否则后续重复实验可能遇到容器名或网络名冲突。

## 18. 面试/考试记忆版

可以这样记：

- Vault provider 让 Terraform 操作 Vault。
- KV v2 支持 secret 版本化。
- Vault dev server 默认有 `secret/`，实验中可用 `training/` 避免冲突。

- sensitive output 要显式处理。

- 生产不要使用 root token，要用最小权限 policy。

一句话总结：

> Vault provider 能把 Terraform 接到 Vault，但它不是 secret 风险的终点；Terraform state 和输出仍然要按敏感资产管理。
