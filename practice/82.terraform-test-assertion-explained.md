# Terraform 实操训练 82 讲解：Terraform test assertion

## 1. 这节课到底在练什么？

这一节的核心不是 `terraform test` 这个命令本身，而是 **Terraform test assertion**：也就是用测试断言把 Terraform 模块的预期行为写清楚，让它可以被自动验证。

你可以把它理解成：

> `terraform validate` 只检查语法和配置是否合法；  
> `terraform plan` 告诉你 Terraform 准备做什么；  
> `terraform test` 则进一步检查：这个模块产出的结果，是否符合你提前写好的工程规则。

这就很接近生产环境里的平台工程实践：不是只要能 apply 就算成功，而是要证明它符合约定。

---

## 2. 为什么 Terraform 需要 test assertion？

在真实团队里，Terraform 模块通常会被很多项目复用。问题在于：

- 模块能不能创建资源，只是最低要求；
- 模块输出的值是否正确，才是复用时更关心的点；
- 模块的命名规则、标签规则、安全默认值、变量组合，也都需要被自动验证；
- 不能每次都靠人工看 `plan` 输出，否则很容易漏。

所以 Terraform test assertion 的价值是：

> 把“我觉得应该这样”变成“代码自动检查必须这样”。

比如你希望模块输出的名称必须包含某个前缀，或者某个 output 不能是空值，就可以通过 assertion 固化下来。

## 3. `terraform test` 和普通命令的区别

这节 lab 的运行流程是：

```sh
cd /workspace/practice/labs/82
terraform init -input=false
terraform fmt -check
terraform validate
terraform test
terraform plan -input=false -no-color -out=tfplan
terraform apply -auto-approve tfplan
terraform output
terraform destroy -auto-approve
```

可以这样理解每一步：

| 命令 | 作用 | 重点 |
|---|---|---|
| `terraform init` | 初始化工作目录 | 下载 provider，准备 backend/module |
| `terraform fmt -check` | 检查格式 | CI 里常用，不自动修改，只负责拦截 |
| `terraform validate` | 检查语法和配置合法性 | 不等于业务逻辑正确 |
| `terraform test` | 执行测试文件里的 run/assert | 本节重点 |
| `terraform plan` | 生成执行计划 | 看 Terraform 准备变更什么 |
| `terraform apply` | 应用计划 | 真正创建或修改资源 |
| `terraform output` | 查看输出 | 验证模块对外暴露的信息 |
| `terraform destroy` | 清理资源 | 避免残留资源和状态 |

关键点：

> `terraform test` 是为了验证模块行为，不是替代 `plan/apply`，而是提前把模块规则检查出来。

## 4. Terraform test assertion 的基本结构

Terraform test 通常放在 `tests/` 目录中，例如：

```text
practice/labs/82/
├── main.tf
├── README.md
└── tests/
    └── *.tftest.hcl
```

测试文件里一般会有 `run` 块和 `assert` 块，结构大概像这样：

```hcl
run "check_output" {
  command = plan

  assert {
    condition     = output.example_value != ""
    error_message = "example_value should not be empty."
  }
}
```

这里有三个重点：

1. `run` 表示一次测试运行；
2. `command = plan` 表示这个测试基于 plan 阶段执行；
3. `assert` 表示断言条件，条件为 false 时测试失败。

断言本质上就是：

> 给 Terraform 模块加自动验收标准。

---

## 5. plan 阶段断言和 apply 阶段断言的区别

Terraform test 里有些断言可以在 `plan` 阶段完成，有些必须到 `apply` 阶段才能完成。

### plan 阶段适合检查什么？

适合检查那些 Terraform 在规划阶段就已经知道的值，例如：

- 输入变量组合是否合法；
- local 计算结果是否符合预期；
- 资源名称是否符合命名规则；
- tag/map/list 这类静态结构是否符合约定；
- output 是否由已知表达式计算出来。

### apply 阶段适合检查什么？

适合检查那些只有资源创建之后才确定的值，例如：

- 云厂商返回的资源 ID；
- 真实 endpoint；
- 动态生成的属性；
- apply 后才 known 的 output。

所以排障时要特别注意：

> 如果一个值在 plan 阶段还是 unknown，就不要强行在 plan 阶段 assert 它。

否则你会遇到“不是逻辑错，而是测试阶段选错了”的问题。

## 6. 这节 lab 应该观察什么？

运行 `terraform test` 时，重点看三类信息。

### 第一，看测试是否被发现

如果测试文件放错目录、文件名不符合预期，可能导致测试没有正常执行。

你应该确认 Terraform 能识别 `tests/` 目录中的测试文件。

### 第二，看 assert 是否真的执行

不要只看到命令成功就结束。要确认输出里类似：

```text
1 passed, 0 failed
```

这说明测试断言确实通过了。

### 第三，看失败信息是否有意义

好的 assertion 不只是失败，而是要告诉你为什么失败。

例如：

```hcl
error_message = "name output should contain the expected prefix."
```

比下面这种有用得多：

```hcl
error_message = "failed"
```

生产环境里，测试失败信息要服务于排障，而不是制造谜语。

## 7. 常见错误和排障思路

### 错误 1：`terraform fmt -check` 失败

这通常不是逻辑错误，而是 HCL 格式不符合 Terraform 标准。

处理方式：

```sh
terraform fmt -recursive
terraform fmt -check
```

在 CI 里推荐使用 `fmt -check`，因为它不会悄悄改文件，只会告诉你格式不合格。

---

### 错误 2：测试引用了不存在的 output/local

比如测试里写了：

```hcl
condition = output.name != ""
```

但是模块里根本没有：

```hcl
output "name" {
  value = ...
}
```

这时 `terraform test` 会失败。

处理方式：

- 检查 output 名称；
- 检查测试文件里的引用路径；
- 不要引用未声明的 local、variable、output。

---

### 错误 3：在 plan 阶段检查 apply 后才知道的值

有些属性在 plan 阶段是 unknown。你如果在 plan 阶段断言它，测试可能不稳定或失败。

处理方式：

- 静态规则放在 `command = plan`；
- 真实创建后才能知道的值放在 `command = apply`；
- 不要把所有测试都塞进 plan 阶段。

---

### 错误 4：断言条件写得太宽松

例如：

```hcl
condition = output.name != null
```

这可能没什么验证价值。因为它只证明有个值，不证明值是对的。

更好的写法是：

```hcl
condition = startswith(output.name, "lab82-")
```

这样才是在验证业务规则。

## 8. 生产环境里的使用场景

Terraform test assertion 在平台工程里很适合用于下面这些场景。

### 1. 模块输出契约测试

例如平台团队提供一个 VNet/VPC 模块，业务团队依赖它的 output：

- subnet id；
- resource group name；
- security group id；
- endpoint；
- tags。

这些 output 一旦改坏，下游项目就会炸。

所以可以用 `terraform test` 检查 output 是否存在、格式是否正确、是否符合约定。

---

### 2. 命名规范测试

例如公司要求资源名称必须包含：

- 环境名；
- 系统名；
- 区域；
- 序号。

这种规则很适合用 assert 固化。

否则靠人工 review，很容易出现“这次先这样吧”的技术债。

---

### 3. tag / label 规范测试

FinOps、审计、成本归属都依赖 tag。

可以用 assertion 检查：

- 是否包含 `owner`；
- 是否包含 `environment`；
- 是否包含 `cost_center`；
- tag value 是否为空。

这类测试非常实际，因为云账单和治理平台都靠这些元数据工作。

---

### 4. 安全默认值测试

例如：

- storage 是否默认禁用 public access；
- security group 是否避免 `0.0.0.0/0` 暴露敏感端口；
- database 是否默认启用加密；
- Kubernetes 相关配置是否默认关闭危险选项。

这些不是“能不能创建资源”的问题，而是“创建出来是否安全”的问题。

Terraform test assertion 就是把这些规则前置拦截。

## 9. 和 CI/CD 的关系

这节 lab 虽然是在本地 Docker 里跑，但它很适合迁移到 GitHub Actions 或其他 CI。

一个典型顺序是：

```sh
terraform init -input=false
terraform fmt -check
terraform validate
terraform test
terraform plan -input=false -no-color
```

这条链路的意义是：

1. 格式不对，先拦；
2. 语法不对，继续拦；
3. 模块行为不符合 assertion，再拦；
4. 最后才看 plan。

这就是平台团队常说的 shift-left：

> 越早发现问题，修复成本越低。

## 10. 本节实验的验收标准

完成这节 lab 后，你应该能做到：

- 知道 `terraform test` 是测试模块行为，不只是检查语法；
- 知道 `assert` 是把预期结果写成自动化规则；
- 能区分 plan 阶段断言和 apply 阶段断言；
- 知道 output/local/变量引用错误会导致测试失败；
- 能把测试结果 `1 passed, 0 failed` 作为实验通过证据；
- 能解释为什么 Terraform test assertion 适合放进 CI/CD。

## 11. 复盘问题

### 问题 1：`terraform validate` 通过了，为什么还需要 `terraform test`？

因为 `validate` 主要检查 Terraform 配置是否合法，不负责证明模块行为符合业务规则。  
`terraform test` 可以检查 output、命名、tag、安全默认值等工程约定。

---

### 问题 2：assertion 的核心价值是什么？

核心价值是把人工 review 里的隐性规则变成自动化测试规则。  
这样模块改动之后，可以快速知道有没有破坏既有约定。

---

### 问题 3：为什么有些测试不能放在 plan 阶段？

因为有些值只有 apply 之后才会出现，例如云厂商返回的真实 ID、endpoint 或动态属性。  
如果在 plan 阶段检查这些值，可能会因为 unknown value 导致测试失败或没有意义。

---

### 问题 4：好的 `error_message` 应该怎么写？

好的错误信息应该告诉维护者：

- 哪个规则失败了；
- 期望是什么；
- 当前问题大概在哪里。

不要只写 `failed`、`wrong` 这种没有排障价值的信息。

---

## 12. 一句话总结

Terraform test assertion 的本质是：

> 不只验证 Terraform 代码能不能跑，而是验证模块产物是否符合平台团队定义的工程契约。

这也是从“会写 Terraform”走向“会做可复用基础设施模块”的关键一步。
