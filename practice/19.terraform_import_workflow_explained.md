# Terraform 实操训练 19：Terraform import 工作流讲解版

## 1. 这一课到底在学什么

这一课讲的是 **Terraform import 工作流**。

一句话概括：

> Terraform import 是把“已经存在的资源”接进 Terraform state，让 Terraform 从此开始管理它。

这类场景在真实公司里非常常见：

- 资源以前是手工点控制台创建的；
- 资源以前是脚本创建的；
- 老系统已经跑了很多年，但现在团队想把它纳入 IaC；
- 新团队接手旧环境，需要先把现有资源“接管”下来。

所以 import 不是为了创建新资源，而是为了把旧资源纳入 Terraform 管理。

---

## 2. 为什么 import 很重要

正常 Terraform 流程是：

```text
写 Terraform 配置
        ↓
terraform plan
        ↓
terraform apply
        ↓
资源被 Terraform 创建
        ↓
资源记录进入 state
```

但是现实里经常反过来：

```text
资源已经存在
        ↓
没有 Terraform 配置
        ↓
没有 Terraform state
        ↓
Terraform 不知道它是谁
```

这时候如果直接写一个新资源再 `apply`，Terraform 可能会尝试创建一个重复资源。

所以正确做法是：

```text
先确认真实资源
        ↓
写 Terraform 资源地址
        ↓
用 import 把真实资源绑定到 Terraform state
        ↓
审查生成配置
        ↓
再进入正常 plan / apply 管理流程
```

---

## 3. Terraform import 解决的核心问题

Terraform 管理资源依赖两件事：

### 3.1 配置文件

也就是 `.tf` 文件，例如：

```hcl
resource "aws_security_group" "web" {
  name = "web-sg"
}
```

配置文件告诉 Terraform：

> 我希望管理一个资源，它在 Terraform 里的地址叫 `aws_security_group.web`。

---

### 3.2 State 文件

state 文件告诉 Terraform：

> `aws_security_group.web` 这个地址，对应云上真实的哪个资源。

例如真实 AWS 安全组 ID 可能是：

```text
sg-0123456789abcdef0
```

import 的本质就是建立这个映射关系：

```text
Terraform 地址                 真实资源 ID
aws_security_group.web   →      sg-0123456789abcdef0
```

没有这个映射，Terraform 就不知道源码里的资源和云上已有资源是不是同一个东西。

---

## 4. 老 import 方式的问题

传统 import 命令类似这样：

```sh
terraform import aws_security_group.web sg-0123456789abcdef0
```

这个命令会把真实资源导入 state。

但是老方式有一个麻烦点：

> 它只导入 state，不会自动帮你写完整 Terraform 配置。

也就是说，导入后你还得自己补 `.tf` 配置。

如果配置补得不对，下一次 `terraform plan` 可能会出现大量 drift，例如：

- Terraform 以为要修改资源；
- Terraform 以为要删除某些字段；
- Terraform 以为缺少必要参数；
- 真实环境和代码表达不一致。

所以老 import 的难点不是命令，而是导入后的配置还原。

---

## 5. Terraform 1.5 之后的新方式

Terraform 1.5 引入了更现代的 import block 工作流。

典型结构类似：

```hcl
import {
  to = aws_security_group.web
  id = "sg-0123456789abcdef0"
}
```

含义是：

```text
把真实 ID 为 sg-0123456789abcdef0 的资源
导入到 Terraform 地址 aws_security_group.web
```

再配合：

```sh
terraform plan -generate-config-out=generated.tf
```

Terraform 可以生成候选配置。

这一步的价值很大：

> 它能帮你从真实资源反推出一份 Terraform 配置草稿。

注意，是草稿，不是最终生产代码。

---

## 6. `-generate-config-out` 是什么

`-generate-config-out` 用来把导入资源对应的配置输出到一个文件里。

例如：

```sh
terraform plan -generate-config-out=generated.tf
```

这会生成一个候选配置文件：

```text
generated.tf
```

你可以把它理解成：

> Terraform 根据远端真实资源，帮你生成一份“能描述当前资源状态”的初始配置。

它适合用来启动接管流程，但不能直接无脑提交生产。

---

## 7. 为什么不能把生成配置直接当最终代码

因为自动生成的配置通常有几个问题。

### 7.1 字段可能太多

Terraform 可能会把很多默认值、计算值、冗余字段都写出来。

结果就是配置很长，不好维护。

---

### 7.2 结构不一定符合团队规范

例如团队希望：

```text
modules/
environments/
variables/
locals/
outputs/
```

但是生成文件通常只是平铺资源。

它能跑，不代表它工程化。

---

### 7.3 可能暴露不应该写死的值

有些值应该抽成变量，有些值应该放 locals，有些值应该由 data source 获取。

自动生成文件不会理解你的组织设计。

---

### 7.4 可能包含历史包袱

真实资源可能已经有很多不标准配置。

生成配置只是反映现状，不代表现状就是最佳实践。

所以正确心态是：

```text
generated.tf 是接管草稿，不是最终代码。
```

---

## 8. 本实验为什么用本地资源模拟

原课程里的导入对象可能是 AWS 安全组之类的真实云资源。

但是这个实验为了能在 Docker、本地环境里稳定运行，使用了本地 Terraform 资源模拟 import 工作流。

这样做的好处是：

- 不需要 AWS 账号；
- 不需要真实云资源；
- 不产生云费用；
- 可以在本地反复验证流程；
- 适合学习 import 的概念和步骤。

但是也要记住：

> 本地模拟实验不等价于真实 AWS import。真实生产 import 还需要查真实资源 ID、权限、云平台状态、网络策略和变更审批。

## 11. 运行命令逐行讲解

进入实验目录：

```sh
cd /workspace/practice/labs/19
```

意思是切到 Lab 19 的 Terraform 工作目录。

---

初始化：

```sh
terraform init -input=false
```

作用：

- 初始化 Terraform 工作目录；
- 下载 provider；
- 生成 `.terraform` 目录；
- `-input=false` 表示不要交互式提问，适合 CI。

---

格式检查：

```sh
terraform fmt -check
```

作用：

- 检查 `.tf` 文件格式是否符合 Terraform 标准；
- 不自动修改，只检查；
- CI 中常用来阻止格式混乱的代码合并。

---

配置校验：

```sh
terraform validate
```

作用：

- 检查 Terraform 配置语法；
- 检查参数和资源结构是否基本合理；
- 不访问真实云资源进行完整变更。

---

运行测试：

```sh
terraform test
```

作用：

- 执行 `.tftest.hcl` 测试；
- 验证实验里定义的行为是否符合预期。

这次实验中，测试返回：

```text
1 passed, 0 failed
```

说明 import 工作流说明已经被测试覆盖。

---

生成 plan：

```sh
terraform plan -input=false -no-color -out=tfplan
```

作用：

- 生成变更计划；
- `-out=tfplan` 把计划保存成文件；
- `-no-color` 让日志更适合 CI 和文本记录；
- `-input=false` 避免交互。

---

执行 apply：

```sh
terraform apply -auto-approve tfplan
```

作用：

- 执行刚才保存的 `tfplan`；
- `-auto-approve` 跳过人工确认；
- 在实验环境中方便自动验证。

生产环境通常不会直接这样无审批执行，而是会先审批 plan。

---

查看输出：

```sh
terraform output
```

作用：

- 查看 Terraform 输出值；
- 验证模拟资源 ID 和工作流信息是否正确。

---

销毁资源：

```sh
terraform destroy -auto-approve
```

作用：

- 清理本实验创建的本地状态资源；
- 防止后续实验被旧状态影响。

---

## 12. 为什么测试后来改成 `command = apply`

验证报告里提到：

> 首次验证发现 plan 阶段无法断言 apply 后才知道的输出，已将测试改为 `command = apply`。

这句话非常关键。

Terraform test 里有些值在 `plan` 阶段是未知的。

例如资源的某些输出，可能只有 apply 后才确定。

所以如果测试要断言这些 apply 后才知道的值，就不能只用 plan 阶段测试。

正确做法是使用：

```hcl
run "xxx" {
  command = apply
}
```

这表示：

> 先实际执行 apply，再检查输出或断言结果。

这就是为什么这个实验最终用 `command = apply` 更合理。

## 13. import 在生产环境中的标准流程

真实团队里，import 通常应该这样做。

### 第一步：盘点现有资源

先确认：

- 资源属于哪个环境；
- 资源真实 ID 是什么；
- 当前资源是否还在被业务使用；
- 是否有其他系统也在管理它；
- 是否有删除保护或关键依赖。

---

### 第二步：写资源地址

例如：

```hcl
resource "aws_security_group" "web" {
}
```

这一步先定义 Terraform 里的资源地址。

地址要慎重，因为它会进入 state。

---

### 第三步：写 import block

例如：

```hcl
import {
  to = aws_security_group.web
  id = "sg-0123456789abcdef0"
}
```

核心是：

- `to` 写 Terraform 资源地址；
- `id` 写真实平台里的资源 ID。

---

### 第四步：生成候选配置

```sh
terraform plan -generate-config-out=generated.tf
```

Terraform 会根据真实资源生成候选配置。

---

### 第五步：审查生成配置

重点看：

- 有没有敏感值；
- 有没有过度冗余字段；
- 有没有需要改成变量的字段；
- 有没有组织不允许的配置；
- 有没有会导致下一次 plan 变更的 drift。

---

### 第六步：重构为正式 Terraform 代码

把 `generated.tf` 里的内容整理进团队标准结构：

```text
modules/
envs/dev/
envs/prod/
variables.tf
outputs.tf
locals.tf
```

---

### 第七步：确认 plan 为 no-op 或可解释变更

最终目标是：

```text
No changes. Your infrastructure matches the configuration.
```

或者至少所有变更都能解释清楚。

---

### 第八步：提交代码并纳入 CI

最后再让它进入正常 IaC 生命周期：

```text
PR
  ↓
terraform fmt
  ↓
terraform validate
  ↓
terraform plan
  ↓
approval
  ↓
terraform apply
```

## 14. import 最容易踩的坑

### 坑 1：资源地址写错

例如你本来想导入：

```text
aws_security_group.web
```

结果写成：

```text
aws_security_group.app
```

那 state 里的映射就错了。

后面代码再改，会非常恶心。

---

### 坑 2：真实 ID 写错

例如安全组 ID 写错了，可能把别的资源导进来。

生产环境这很危险。

所以导入前必须确认真实资源 ID。

---

### 坑 3：导入后不审查配置

自动生成的配置只是草稿。

如果直接提交，后面维护成本可能很高。

---

### 坑 4：以为 import 会创建资源

import 不创建资源。

它只是把已经存在的资源接入 Terraform state。

记住：

```text
apply 是创建或修改资源
import 是接管已有资源
```

---

### 坑 5：多人同时操作 state

导入本质上会改 state。

如果多人同时操作，容易产生 state 冲突。

生产环境建议使用 remote backend 和 state lock。

---

## 15. import 和 drift detection 的关系

import 常常和 drift detection 配合使用。

导入之前：

```text
Terraform 不知道这个资源
```

导入之后：

```text
Terraform 可以对比配置和真实资源
```

这时候 `terraform plan` 才能告诉你：

- 配置和真实资源是否一致；
- 有哪些字段发生 drift；
- 是否需要修正 Terraform 代码；
- 是否需要修正真实资源。

所以 import 是把遗留资源带入治理体系的第一步。

## 19. 你应该真正掌握什么

完成这课以后，你应该能说清楚：

- Terraform import 是接管已有资源，不是创建资源；
- import 建立的是 Terraform 地址和真实资源 ID 的映射；
- 老 import 只导入 state，不自动生成完整配置；
- Terraform 1.5+ 可以用 import block 和 `-generate-config-out`；
- 生成配置只是候选草稿，需要审查和重构；
- 生产环境 import 的核心风险是地址、ID、state、drift 和审批；
- 本地实验只是模拟，不等于真实 AWS 导入。
