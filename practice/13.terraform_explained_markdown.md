# Terraform 实操训练 13：静态分析工具基础讲解版

## 1. 本节定位

本节开始进入 Terraform 工程化中的 **静态分析 / IaC 安全扫描**。

前面几节更多是在讲 Terraform CLI 的执行参数、provider cache、provider mirror、provider installation 等执行环境问题。本节开始关注另一个生产级问题：

> Terraform 代码还没有部署之前，能不能提前发现明显的安全风险？

答案是可以，典型工具就是 **Checkov**。

本节的核心不是让你记住一个 Docker 命令，而是理解：

```text
Terraform 代码进入生产之前，应该先经过自动化安全扫描。
```

---

## 2. 什么是静态分析？

静态分析的意思是：

```text
不真正创建资源
不连接真实云平台 API
只读取代码本身
然后判断代码里是否有风险、错误或不符合最佳实践的配置
```

对于 Terraform 来说，就是扫描 `.tf` 文件。

比如 Checkov 可以扫描：

```text
main.tf
variables.tf
outputs.tf
modules/
```

然后检查里面是否存在安全问题，例如：

```text
Kubernetes Pod 开启 hostNetwork
容器开启 privileged
镜像使用 latest 标签
没有配置资源 requests / limits
云资源未开启加密
安全组开放 0.0.0.0/0
存储桶公开访问
```

本节主要关注 Kubernetes Pod 相关风险。

---

## 3. Checkov 是什么？

Checkov 是一个 Infrastructure as Code 静态扫描工具。

它可以扫描多种 IaC 文件：

```text
Terraform
Kubernetes YAML
CloudFormation
ARM / Bicep
Dockerfile
Helm
GitHub Actions
```

本节只使用它扫描 Terraform：

```sh
-d /tf --framework terraform
```

含义是：

```text
-d /tf：扫描 /tf 目录
--framework terraform：只按 Terraform 规则扫描
```

---

## 4. 本节实验背景

实验场景是：

```text
平台团队要检查 Terraform 代码里是否存在明显的安全风险。
```

示例代码故意写了一些不安全的 Kubernetes Pod 配置，例如：

```text
hostNetwork
privileged container
latest image tag
缺少 resource requests / limits
```

这些配置在真实生产环境里都比较危险。

---

## 5. 这些 failed checks 为什么危险？

### 5.1 宿主机网络 hostNetwork

如果 Pod 使用宿主机网络，意味着 Pod 会共享 Node 的网络命名空间。

风险包括：

```text
Pod 网络隔离变弱
可能绕过部分 Kubernetes NetworkPolicy
攻击面扩大
容器更接近宿主机网络层
```

生产里不是绝对不能用，但必须有非常明确的理由。

---

### 5.2 privileged 容器

如果容器开启 privileged，相当于给容器非常高的宿主机权限。

风险包括：

```text
容器逃逸风险增加
容器可以访问更多宿主机能力
隔离边界被削弱
攻击者拿到容器后影响面更大
```

生产环境里，这通常是高危配置。

---

### 5.3 latest 镜像标签

例如：

```text
image = "nginx:latest"
```

问题是：

```text
latest 不是固定版本
今天拉到的镜像和明天可能不同
部署不可复现
回滚困难
审计困难
供应链风险更高
```

生产里更推荐：

```text
nginx:1.27.0
```

或者更严格地使用 digest：

```text
nginx@sha256:...
```

---

### 5.4 缺少资源 requests / limits

如果没有配置 CPU / Memory 的 requests 和 limits，Kubernetes 调度和资源隔离都会变弱。

风险包括：

```text
Pod 可能抢占过多资源
调度器无法准确判断资源需求
节点容易被打爆
SRE 排障困难
多租户环境下影响其他服务
```

生产环境里，至少应该设置合理的 requests。

---

## 6. 实验文件

本节文件包括：

```text
practice/labs/13/main.tf
practice/labs/13/README.md
```

这里没有 `outputs.tf`、`tests` 或 Terraform destroy 逻辑，因为本节不是部署 Terraform 资源，而是扫描代码。

也就是说，本节不需要：

```sh
terraform init
terraform plan
terraform apply
terraform destroy
```

Checkov 只读取 `.tf` 文件，不创建资源。

## 7. 运行命令讲解

原始命令是 PowerShell 风格：

```powershell
docker run --rm `
  -v "${PWD}/practice/labs/13:/tf" `
  bridgecrew/checkov:latest `
  -d /tf --framework terraform --soft-fail --compact
```

---

### 7.1 `docker run --rm`

```powershell
docker run --rm
```

表示启动一个临时容器，运行结束后自动删除。

好处是：

```text
不用在本机安装 Checkov
不污染本机环境
CI 里更容易保持一致
```

---

### 7.2 `-v "${PWD}/practice/labs/13:/tf"`

这一步是 volume 挂载。

含义是：

```text
宿主机目录：${PWD}/practice/labs/13
容器内目录：/tf
```

Checkov 在容器里扫描 `/tf`，实际上扫描的是你本机的 Lab 13 目录。

如果扫描不到结果，第一反应就是检查这个 volume 路径是否挂对。

---

### 7.3 `bridgecrew/checkov:latest`

表示使用 Checkov 的 Docker 镜像。

如果本机没有镜像，可以先执行：

```sh
docker pull bridgecrew/checkov:latest
```

---

### 7.4 `-d /tf`

表示扫描容器内的 `/tf` 目录。

这里的 `/tf` 是容器里的路径，不是宿主机真实路径。

---

### 7.5 `--framework terraform`

表示只启用 Terraform 扫描规则。

这样输出更聚焦，不会把其他类型文件一起扫描出来。

---

### 7.6 `--soft-fail`

这是本节最关键的参数之一。

`--soft-fail` 的意思是：

```text
即使发现 failed checks，也让命令返回成功退出码。
```

所以它适合：

```text
教学实验
安全扫描试运行
团队刚开始引入 Checkov
先观察问题数量，不马上阻断 PR
```

但是生产 CI 里，如果你希望安全问题阻断合并或部署，就应该移除 `--soft-fail`。

也就是从：

```sh
checkov -d /tf --framework terraform --soft-fail --compact
```

改成：

```sh
checkov -d /tf --framework terraform --compact
```

---

### 7.7 `--compact`

表示输出更紧凑，减少日志噪音。

CI 日志里通常很有用。

---

## 8. 验证报告怎么读？

本节验证结果是：

```text
Checkov 版本：3.3.5
Passed checks: 14
Failed checks: 13
```

这说明：

```text
Checkov 成功扫描到了 Terraform 文件
也成功识别出了预期中的风险配置
```

注意，这里的 failed checks 不是实验失败。

因为本节实验目的就是：

```text
观察 Checkov 能否发现这些故意写出来的不安全配置。
```

所以教学实验里：

```text
发现问题 = 实验成功
```

生产 CI 里：

```text
发现高危问题 = PR 失败 / 部署阻断
```

这两个语境要分开。

---

## 9. Checkov 在 Terraform CI 里的位置

比较合理的 Terraform CI 顺序是：

```text
terraform fmt -check
        ↓
terraform validate
        ↓
Checkov / TFLint / tfsec
        ↓
terraform test
        ↓
terraform plan
```

Checkov 应该尽早跑。

原因是：

```text
它不需要真实云权限
它不需要创建资源
它运行成本低
越早发现问题越便宜
```

---

## 10. Checkov 和 terraform validate 的区别

很多初学者容易混淆这两个。

| 工具 | 主要检查什么 |
|---|---|
| `terraform validate` | Terraform 语法、引用、配置结构是否合法 |
| Checkov | 安全、合规、最佳实践问题 |
| `terraform plan` | 真实变更计划 |
| `terraform test` | Terraform 原生测试逻辑 |

例如：

```text
privileged = true
```

这在 Terraform 语法上可能是合法的。

所以：

```text
terraform validate 可能通过
Checkov 会认为它是安全风险
```

这就是静态安全扫描的价值。

## 11. CI 中如何使用 soft-fail？

### 初期引入阶段

可以先使用：

```sh
checkov -d . --framework terraform --soft-fail --compact
```

作用是：

```text
报告问题
但不阻断 PR
```

适合团队刚开始引入安全扫描。

---

### 正式治理阶段

移除 `--soft-fail`：

```sh
checkov -d . --framework terraform --compact
```

作用是：

```text
发现 failed checks
命令返回非 0
CI 失败
PR 被阻断
```

---

### 成熟治理阶段

可以进一步配合：

```text
severity 阈值
skip-check 白名单
baseline
SARIF 报告
PR 注释
安全例外审批
```

例如：

```text
低风险只警告
高风险阻断
例外项必须写原因和过期时间
```

---

## 12. 排障指南

### 12.1 镜像不存在

如果 Docker 找不到镜像：

```sh
docker pull bridgecrew/checkov:latest
```

---

### 12.2 扫描不到结果

检查挂载目录：

```powershell
-v "${PWD}/practice/labs/13:/tf"
```

确认宿主机路径里真的有：

```text
practice/labs/13/main.tf
```

---

### 12.3 CI 需要阻断部署

移除：

```text
--soft-fail
```

这样 Checkov 发现 failed checks 时，CI 就会失败。

---

## 13. 本节不需要 destroy

因为 Checkov 不创建 Terraform 资源。

所以本节清理方式是：

```text
无需 terraform destroy
```

最多只是清理 Docker 镜像或本地缓存，但一般没必要。

---

## 14. 生产级理解

本节在真实生产里的意义是：

```text
不要等 terraform apply 之前才发现安全风险。
```

更好的做法是：

```text
开发者提交 PR
        ↓
CI 自动运行 Checkov
        ↓
发现不安全 Terraform 配置
        ↓
PR 阶段修复
        ↓
合并后再进入 plan / apply
```

这就是 shift-left：安全检查左移。

## 15. 面试级总结

英文表达：

```text
Checkov is a static analysis tool for Infrastructure as Code. It can scan Terraform files before deployment and detect security or compliance issues without connecting to the real cloud API. In CI, it is commonly used before terraform plan so risky configurations can be caught early. The --soft-fail option is useful during training or initial adoption because it reports findings without failing the pipeline. In production, teams usually remove soft-fail or enforce severity-based gates to block risky changes.
```

中文表达：

```text
Checkov 这类静态分析工具可以在 Terraform 部署前扫描 IaC 安全风险，不需要连接真实云平台，也不会创建资源。教学或初期引入阶段可以使用 --soft-fail 只报告不阻断；生产 CI 中通常移除 --soft-fail，或者按严重级别设置阻断策略。
```
