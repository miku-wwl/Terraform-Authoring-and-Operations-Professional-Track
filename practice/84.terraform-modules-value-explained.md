# Terraform 实操训练 84：Terraform Modules 价值 — 讲解版

> 来源：`84.md`  
> 主题：Terraform modules 价值  
> 适合阶段：Terraform 从单文件实验，过渡到可复用工程结构

---

## 1. 这节课到底在讲什么？

本节 Lab 的核心不是让你“背 module 语法”，而是理解：**Terraform module 是把基础设施代码从一次性脚本，升级成可复用、可测试、可治理工程单元的关键机制。**

在 Terraform 里，如果所有资源都堆在一个 `main.tf` 里，刚开始很爽：写得快、跑得快、看起来也简单。

但一旦进入真实团队环境，问题会马上出现：

- 同样的 VPC / 网络 / 存储 / IAM 配置在多个项目里重复写；
- 不同人复制粘贴后改坏了参数；
- 环境之间 dev / test / prod 结构不一致；
- 安全基线、命名规范、tag 规范无法统一；
- 代码越来越大，plan 输出越来越难读；
- 后期重构容易影响 state，风险变高。

**Module 的价值，就是把这些重复、混乱、不可控的基础设施逻辑收敛起来。**

---

## 2. Module 是什么？

Terraform module 可以理解成：

> 一组 Terraform 配置文件的封装单元。

最简单地说：

- 一个目录就是一个 module；
- 当前执行 `terraform plan/apply` 的目录叫 **root module**；
- 被 `module` block 调用的目录叫 **child module**。

例如：

```hcl
module "app_config" {
  source = "./modules/app_config"

  app_name    = "demo"
  environment = "dev"
}
```

这里的意思是：

- root module 不直接写所有细节；
- 它调用 `./modules/app_config` 这个子模块；
- 通过变量把差异传进去；
- 子模块负责生成统一结构的资源或输出。

---

## 3. 为什么生产环境一定会走向 Module？

### 3.1 复用：别让团队复制粘贴基础设施代码

没有 module 的时候，常见写法是：

```text
project-a/main.tf
project-b/main.tf
project-c/main.tf
```

每个项目都写一遍相似逻辑。

短期看没问题，长期就是技术债制造机。

Module 的做法是：

```text
modules/network
modules/storage
modules/iam
modules/app
```

不同项目只传参数：

```hcl
module "network" {
  source      = "../../modules/network"
  environment = "prod"
  cidr_block  = "10.0.0.0/16"
}
```

这样团队维护的是**一套标准实现**，而不是十几个复制版本。

---

### 3.2 标准化：把平台团队的经验固化成代码

平台团队最怕的不是“没人会写 Terraform”，而是“每个人都会写一点 Terraform”。

因为每个人都写一点，就会出现：

- 命名风格不一致；
- tag 缺失；
- 安全配置漏掉；
- 默认参数不合理；
- 资源层级混乱；
- 网络、安全、日志配置各玩各的。

Module 可以把这些规范封装进去。

调用方只需要提供必要参数：

```hcl
module "service_bucket" {
  source      = "./modules/secure_bucket"
  name        = "orders"
  environment = "prod"
}
```

而 module 内部可以统一处理：

- bucket 命名规则；
- 加密；
- versioning；
- public access block；
- lifecycle；
- tags；
- outputs。

这就是平台工程里的一个核心思想：

> 不要只写文档告诉别人怎么做，而是把正确做法封装成默认路径。

---

### 3.3 降低认知负担：调用者不用理解所有底层细节

一个好的 module 应该像一个干净的 API。

调用者不用知道里面创建了多少资源，只需要关心输入和输出。

例如：

```hcl
module "app" {
  source = "./modules/app"

  name        = "payment"
  environment = "dev"
  replicas    = 2
}
```

调用者关心：

- 应用叫什么；
- 部署到哪个环境；
- 副本数是多少。

至于内部怎么组合资源、怎么命名、怎么输出 endpoint，那是 module 负责的。

这就像写后端服务时，你不会要求每个业务开发都手写 TCP 连接池。基础设施也一样，应该有抽象层。

---

## 4. Module 的基本结构

常见 module 结构如下：

```text
modules/
  app_config/
    main.tf
    variables.tf
    outputs.tf
    README.md
```

各文件职责：

| 文件 | 作用 |
|---|---|
| `main.tf` | 定义主要资源或 locals 逻辑 |
| `variables.tf` | 定义模块输入参数 |
| `outputs.tf` | 定义模块输出结果 |
| `README.md` | 说明模块用途、输入、输出、示例 |

一个训练 lab 里可能不会创建真实云资源，而是用 `locals`、`outputs`、`terraform test` 来模拟 module 行为。

这没问题。因为本节重点是理解 module 的工程价值，而不是烧云资源。

---

## 5. Root Module 和 Child Module 的区别

### Root Module

你执行命令的目录就是 root module：

```sh
cd /workspace/practice/labs/84
terraform init
terraform plan
```

这个目录里的 `.tf` 文件就是 root module。

它通常负责：

- 组织整体环境；
- 调用多个 child module；
- 传入环境参数；
- 暴露关键 outputs。

---

### Child Module

被调用的 module 是 child module：

```hcl
module "example" {
  source = "./modules/example"
}
```

Child module 通常负责：

- 封装一类资源；
- 提供稳定输入输出；
- 隐藏内部实现细节；
- 被多个环境或项目复用。

---

## 6. Module 的输入和输出

### 输入：variable

Module 的输入由 `variable` 定义：

```hcl
variable "environment" {
  type        = string
  description = "Deployment environment"
}
```

调用时传入：

```hcl
module "app" {
  source      = "./modules/app"
  environment = "dev"
}
```

---

### 输出：output

Module 的输出由 `output` 定义：

```hcl
output "app_name" {
  value = local.app_name
}
```

Root module 可以这样引用：

```hcl
output "app_name" {
  value = module.app.app_name
}
```

这体现了 Terraform module 的接口思想：

> variable 是输入接口，output 是输出接口。

## 7. 本 Lab 应该重点观察什么？

你进入目录后运行：

```sh
cd /workspace/practice/labs/84
terraform init -input=false
terraform fmt -check
terraform validate
terraform test
terraform plan -input=false -no-color -out=tfplan
terraform apply -auto-approve tfplan
terraform output
terraform destroy -auto-approve
```

重点不是机械跑完，而是观察下面几个点。

---

### 7.1 `terraform init` 会初始化 module

当 Terraform 配置里出现 module 调用时，`terraform init` 会处理 module source。

本地 module 可能很快完成；远程 module 则可能需要下载。

这也是为什么 module 变更后，经常需要重新执行：

```sh
terraform init
```

尤其是修改了 module source、版本、registry 地址时。

---

### 7.2 `terraform validate` 会检查 module 结构

`terraform validate` 不只是检查 root module，也会检查被引用 module 的配置是否基本合法。

常见错误包括：

- module source 路径错误；
- variable 没有传必填值；
- output 引用了不存在的资源；
- child module 内部 HCL 语法错误；
- provider 配置关系不清晰。

---

### 7.3 `terraform test` 会验证 module 行为

这节课的训练里包含 `tests/` 目录。

`terraform test` 的意义是：

> 用断言确认 module 的输出或行为符合预期。

比如一个 module 负责生成标准命名，那么测试可以检查：

```hcl
assert {
  condition     = output.name == "dev-demo-app"
  error_message = "Module should generate expected app name."
}
```

这类测试在平台团队里很实用。

因为 module 一旦被多个项目复用，改 module 就不是小事。测试可以帮你提前发现：

- 输出结构变了；
- 命名规则变了；
- 默认值变了；
- 参数校验失效了；
- 某个环境组合不再工作。

---

## 8. Module 和 State 的关系

这点非常重要。

Module 不是单纯的代码组织方式，它也会影响 Terraform state 里的资源地址。

没有 module 时，资源地址可能是：

```text
terraform_data.example
```

放进 module 后，资源地址会变成：

```text
module.app.terraform_data.example
```

这意味着：

- module 重构可能改变 state address；
- address 变化后 Terraform 可能认为旧资源要删除、新资源要创建；
- 生产环境重构 module 时，需要非常小心；
- 必要时要使用 `moved` block 或 `terraform state mv`。

所以 module 的价值很大，但也不能乱拆乱改。

**一句话：module 是工程抽象，不是随便建文件夹。**

---

## 9. Module 设计的好坏标准

一个好的 Terraform module 应该满足这些条件：

### 9.1 输入少而清晰

不要让调用者传一堆内部细节。

差的设计：

```hcl
module "network" {
  source = "./modules/network"

  subnet_1_name = "..."
  subnet_2_name = "..."
  route_table_name = "..."
  nsg_name = "..."
  private_endpoint_name = "..."
}
```

好的设计：

```hcl
module "network" {
  source = "./modules/network"

  environment = "prod"
  cidr_block  = "10.0.0.0/16"
  subnet_plan = "standard"
}
```

调用者应该描述意图，而不是操作所有零件。

---

### 9.2 输出稳定

Module output 是其他模块或环境依赖的接口。

不要今天输出 `id`，明天改成 `resource_id`，后天又改成对象结构。

如果必须改，要考虑版本化和兼容期。

---

### 9.3 默认值安全

Module 的默认值应该偏向安全，而不是偏向偷懒。

例如：

- 默认开启加密；
- 默认禁用 public access；
- 默认打上 tags；
- 默认启用日志；
- 默认最小权限。

别写那种“为了 demo 方便，生产自己改”的 module。那种东西最后一定会混进生产。

---

### 9.4 可测试

如果一个 module 的行为很重要，就应该能被 `terraform test` 验证。

常见测试目标：

- 命名规则；
- tag 合并逻辑；
- 默认值；
- variable validation；
- output 结构；
- 不同环境参数组合。

---

## 10. 常见坑

### 坑 1：把 module 当成文件夹分类

有些人看到配置多了，就开始拆：

```text
network.tf
compute.tf
storage.tf
```

这只是文件拆分，不是 module。

Module 是有输入输出接口的封装单元，不是单纯把文件搬到另一个目录。

---

### 坑 2：module 层级套太深

Module 可以嵌套 module，但不要过度抽象。

如果你看到这种结构：

```text
root -> platform -> landing_zone -> network -> subnet -> route -> rule
```

调试的时候会很痛苦。

Terraform 的抽象不是越深越高级。能清楚表达边界就够了。

---

### 坑 3：输出太多内部细节

如果 module 把所有内部资源细节都 output 出去，等于没有封装。

输出应该服务调用方，而不是暴露全部内部实现。

---

### 坑 4：重构 module 不考虑 state

这是生产环境大坑。

把资源移进 module 后，Terraform 地址会变。

如果没有 `moved` block 或 state 迁移，Terraform 可能计划 destroy/create。

看到 plan 里出现大量 replacement，不要直接 apply。

---

### 坑 5：module 没有版本策略

本地实验可以用：

```hcl
source = "./modules/app"
```

生产环境常见做法是使用 Git tag 或 registry version。

例如：

```hcl
module "network" {
  source = "git::https://example.com/iac-modules.git//network?ref=v1.2.0"
}
```

这样可以避免所有环境被 main 分支上的改动直接影响。

---

## 11. 和真实工作的关系

在真实团队里，module 常常对应平台团队交付的“基础设施产品”。

例如：

- AKS module；
- EKS module；
- VPC / VNet module；
- IAM baseline module；
- logging module；
- monitoring module；
- app service module；
- secure storage module。

这些 module 不是为了让代码看起来高级，而是为了让团队做到：

- 快速交付；
- 行为一致；
- 安全默认；
- 可测试；
- 可审计；
- 可升级；
- 可回滚。

这就是 Terraform 从“个人脚本”进入“平台工程”的分水岭。

## 12. 本 Lab 的复盘问题

跑完实验后，可以用下面几个问题检查自己是否真的理解了。

### Q1：为什么 module 不是单纯的目录拆分？

因为 module 有明确的输入和输出接口。它封装一组基础设施逻辑，并通过变量接收参数、通过 output 暴露结果。

---

### Q2：为什么 module 可以提升团队一致性？

因为命名、tags、安全默认值、日志、加密、网络规则等通用规范可以写进 module。调用方只需要传业务参数，不需要每次重新实现标准。

---

### Q3：为什么 module 重构有 state 风险？

因为资源进入 module 后，Terraform state 里的资源地址会变化。例如从 `terraform_data.example` 变成 `module.app.terraform_data.example`。如果不处理地址迁移，Terraform 可能把它看成旧资源删除、新资源创建。

---

### Q4：为什么生产环境 module 需要版本管理？

因为 module 被多个环境复用。如果所有环境都直接跟随 main 分支，module 一次改动可能影响大量环境。版本化可以让升级变成可控动作。

---

### Q5：`terraform test` 对 module 有什么价值？

它可以用断言验证 module 的行为，例如输出是否正确、命名是否符合规范、默认值是否安全、不同变量组合是否符合预期。这样 module 改动时可以更早发现破坏性影响。

---

## 13. 建议你怎么学这一节

这一节不要只停留在“会写 module block”。建议你按这个顺序理解：

1. 先看 root module 如何调用 child module；
2. 再看 child module 暴露了哪些 variables；
3. 然后看 outputs 如何从 child module 返回 root module；
4. 再运行 `terraform test` 看断言验证了什么；
5. 最后思考：如果这个 module 被 10 个项目复用，哪些改动会变危险。

这样你才是在按工程方式学 Terraform，而不是只背命令。

---

## 14. 一句话总结

**Terraform module 的核心价值，不是把代码藏进目录里，而是把可复用的基础设施能力封装成稳定接口，让团队可以用统一、安全、可测试的方式交付环境。**
