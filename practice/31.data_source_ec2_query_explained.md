# Terraform AWS Sandbox 31 讲解：数据源基础与 EC2 查询

> 对应课程：Terraform AWS Sandbox 31  
> 主题：Terraform Data Source、AWS Provider Endpoint、LocalStack、EC2 查询  
> 实验类型：AWS LocalStack 模拟实验，不需要真实 AWS 账号

---

## 1. 这一课到底在学什么？

这一课的核心不是“创建 EC2”，而是学习 Terraform 如何**读取已经存在的资源**。

Terraform 里有两类常见对象：

| 类型 | 作用 | 会不会创建资源 |
|---|---|---|
| `resource` | 创建、更新、删除基础设施 | 会 |
| `data` source | 查询外部已经存在的信息 | 不会 |

本实验用的是：

```hcl
data "aws_instances" "lab" {
  filter {
    name   = "tag:Name"
    values = ["tf-lab-31-*"]
  }
}
```

它的意思是：  
Terraform 去 AWS Provider 连接的目标环境里查询 EC2 实例，筛选出 `Name` 标签符合 `tf-lab-31-*` 的实例，然后把实例 ID 读出来。

这里的重点是：

> Data source 是 Terraform 的“查询动作”，不是“创建动作”。

---

## 2. 为什么要学 data source？

真实生产环境里，Terraform 项目经常不是从零创建所有东西。

比如：

- VPC 已经存在，只想在里面创建 subnet 或 EC2。
- 安全组已经存在，只想引用它的 ID。
- AMI 由镜像团队维护，业务 Terraform 只需要查询最新 AMI。
- 当前账号 ID、region、caller identity 需要动态读取。
- 老系统资源已经存在，新项目只做增量 IaC 管理。

这时候就需要 data source。

如果硬编码资源 ID，例如：

```hcl
vpc_id = "vpc-0123456789abcdef0"
```

短期能跑，长期会变成维护灾难。  
更好的方式是通过标签、名称、条件查询：

```hcl
data "aws_vpc" "main" {
  tags = {
    Environment = "prod"
  }
}
```

这样 Terraform 配置更像“声明我要找哪个资源”，而不是死记一个 ID。

---

## 3. 本实验为什么要用 LocalStack？

课程原始思路是 AWS EC2 查询，但真实 AWS 会涉及：

- AWS 凭证
- region
- VPC/subnet
- AMI
- EC2 费用
- 账号权限

为了让实验更轻量，本 lab 使用 LocalStack 模拟 AWS API。

也就是说：

```text
Terraform AWS Provider
        |
        v
LocalStack endpoint: http://localhost:4566
        |
        v
模拟 STS / EC2 API
```

这能练到 Terraform 的关键能力：

- AWS Provider endpoint 如何改到本地
- data source 如何读取外部资源
- output 如何引用查询结果
- Terraform 如何和 AWS API 交互

但也要清楚：

> LocalStack 能练 Terraform 工作流，不等价于真实 AWS 的完整 EC2 行为。

它不能完整模拟真实 EC2 的网络、计费、metadata、AMI 生态和 IAM 复杂权限。

## 4. 实验整体流程

这一课的流程可以理解成 4 步：

```text
Step 1：启动/检查 LocalStack
Step 2：bootstrap.sh 先创建两台模拟 EC2
Step 3：Terraform 用 data.aws_instances 查询这些 EC2
Step 4：verify.sh 用 AWS CLI 再确认查询结果
```

关键点是：

```text
EC2 不是 Terraform 创建的
EC2 是 bootstrap 脚本提前创建的
Terraform 只是读取它们
```

这正好模拟生产环境：

```text
已有资源 -> Terraform data source 查询 -> 输出/引用资源属性
```

## 6. Provider endpoint 是本实验的关键

在真实 AWS 中，AWS Provider 默认会访问 AWS 官方 API。

但本实验不想访问真实 AWS，所以需要把 endpoint 改到 LocalStack。

概念上类似这样：

```hcl
provider "aws" {
  region                      = "us-east-1"
  access_key                  = "test"
  secret_key                  = "test"
  skip_credentials_validation = true
  skip_metadata_api_check     = true
  skip_requesting_account_id  = true

  endpoints {
    ec2 = "http://localhost:4566"
    sts = "http://localhost:4566"
  }
}
```

这里的重点：

- `access_key = "test"` 和 `secret_key = "test"` 是 LocalStack 假凭据。
- `endpoints` 把 AWS API 请求导向本机 LocalStack。
- `skip_*` 是为了避免 AWS Provider 做真实 AWS 校验。

这个思路在真实企业里也有类似场景，例如私有云兼容 S3 API、测试环境 mock API、本地集成测试等。

## 7. Data source 的执行时机

Data source 会在 `plan` 或 `apply` 阶段读取外部系统。

也就是说，当你执行：

```sh
terraform plan
```

Terraform 会尝试访问 LocalStack，查询 EC2 实例。

如果 LocalStack 没启动，或者 bootstrap 没创建实例，data source 就可能读不到东西。

所以本实验顺序不能乱：

```sh
bash scripts/check-sandbox.sh
bash scripts/bootstrap.sh
terraform plan
```

不要一上来就 `terraform plan`，否则 Terraform 会去查询一个还没准备好的外部环境。

## 8. 标准运行命令

进入实验目录：

```sh
cd practice/labs/31
```

设置假 AWS 凭据和 LocalStack endpoint：

```sh
export AWS_ACCESS_KEY_ID=test
export AWS_SECRET_ACCESS_KEY=test
export AWS_DEFAULT_REGION=us-east-1
export LOCALSTACK_ENDPOINT=http://localhost:4566
```

检查 LocalStack：

```sh
bash scripts/check-sandbox.sh
```

创建模拟 EC2：

```sh
bash scripts/bootstrap.sh
```

执行 Terraform：

```sh
terraform init -input=false
terraform fmt -check
terraform validate
terraform plan -input=false -no-color -out=tfplan
terraform apply -auto-approve tfplan
terraform output
```

验证结果：

```sh
bash scripts/verify.sh
```

清理：

```sh
terraform destroy -auto-approve
bash scripts/clean.sh
rm -rf .terraform .terraform.lock.hcl tfplan terraform.tfstate terraform.tfstate.backup
```

## 9. 预期输出怎么看？

成功后应该能看到类似输出：

```text
lab_instance_count = 2
lab_instance_ids = [
  "i-xxxxxxxxxxxxxxxxx",
  "i-yyyyyyyyyyyyyyyyy",
]
```

这里说明：

- Terraform 成功连到了 LocalStack。
- `bootstrap.sh` 创建的模拟 EC2 被读到了。
- `data.aws_instances.lab` 查询结果不为空。
- output 成功引用了 data source 的属性。

注意：这些实例不是 Terraform 创建的，所以 `terraform destroy` 不一定会删除它们。  
真正清理 bootstrap 资源的是：

```sh
bash scripts/clean.sh
```

这也是一个很真实的工程点：

> Terraform 只管理自己 state 里的 resource，不会自动销毁 data source 读到的外部资源。

---

## 10. `terraform destroy` 为什么不负责清理 data source？

因为 data source 只是读取。

例如：

```hcl
data "aws_instances" "lab" {
  ...
}
```

它没有创建 EC2，也没有把 EC2 纳入 Terraform resource 管理。

所以：

```sh
terraform destroy
```

只会销毁 Terraform 管理的 resource。  
如果当前配置里只有 data source，那么 destroy 基本没有真实资源可删。

这就是为什么实验还需要：

```sh
bash scripts/clean.sh
```

因为 EC2 是 bootstrap 脚本创建的，应该由 clean 脚本清理。

一句话总结：

> 谁创建，谁负责清理。Terraform data source 只读不删。

## 11. 常见错误与排查

### 问题 1：LocalStack endpoint 不通

现象：

```text
connection refused
could not connect to endpoint
```

处理：

```sh
bash scripts/check-sandbox.sh
```

确认 LocalStack 是否运行，endpoint 是否是：

```text
http://localhost:4566
```

---

### 问题 2：Provider 打到了真实 AWS

现象：

- 要求真实 AWS 凭证
- 报 IAM 权限错误
- 访问 AWS 官方 endpoint

处理：

确认环境变量：

```sh
export AWS_ACCESS_KEY_ID=test
export AWS_SECRET_ACCESS_KEY=test
export AWS_DEFAULT_REGION=us-east-1
export LOCALSTACK_ENDPOINT=http://localhost:4566
```

同时检查 `provider.tf` 里是否配置了 LocalStack endpoints。

---

### 问题 3：查询结果为空

现象：

```text
lab_instance_count = 0
```

处理：

先运行：

```sh
bash scripts/bootstrap.sh
```

然后再执行：

```sh
terraform plan -input=false -no-color -out=tfplan
terraform apply -auto-approve tfplan
```

原因通常是 Terraform 查询时，外部 EC2 还没有被 bootstrap 创建。

---

### 问题 4：`terraform init` 失败

现象：

```text
Failed to query available provider packages
```

处理：

检查 Terraform Sandbox 是否能访问 Terraform Registry。  
如果网络不稳定，可以考虑 provider plugin cache，但这是前面 Lab 9 的主题。

## 12. 面试/工作中怎么讲这个知识点？

可以这样讲：

> Terraform data source 用来读取外部已存在资源，不负责创建和销毁。比如在生产环境中，我们可能不会在每个模块里创建 VPC，而是通过 `data "aws_vpc"` 按标签查询已有 VPC，再把它的 ID 传给 subnet、security group 或 EKS 模块。本实验用 LocalStack 模拟 AWS EC2，先用 bootstrap 创建实例，再用 `data "aws_instances"` 查询实例 ID，验证了 provider endpoint、data source 读取和 output 引用的完整流程。

这个表达比只说“data source 是查询资源”更像工程实践。

## 13. 本 Lab 的核心记忆点

记住这 5 句话就够了：

1. `resource` 是创建和管理资源。
2. `data` source 是读取外部已有信息。
3. Data source 会在 plan/apply 时访问 provider。
4. Data source 读到的资源不归 Terraform destroy 管。
5. LocalStack 能验证 Terraform 调用 AWS API 的流程，但不是完整真实 AWS。

---

## 14. 和真实 AWS 的差异

| 维度 | LocalStack 实验 | 真实 AWS |
|---|---|---|
| 凭据 | `test/test` 假凭据 | IAM 用户、角色、SSO、OIDC |
| Endpoint | `localhost:4566` | AWS 官方区域 endpoint |
| EC2 行为 | API 模拟 | 完整计算、网络、计费、metadata |
| 成本 | 基本无成本 | 可能产生费用 |
| 适合用途 | Terraform 工作流练习 | 生产部署与真实验证 |

所以这个 Lab 的价值是：

> 低成本练 Terraform AWS Provider 和 data source 模式。

但到了真实项目，仍然需要补上 IAM、VPC、AMI、security group、region、tagging strategy 等真实 AWS 细节。

## 15. 最终总结

Lab 31 是从 Docker-only Terraform 练习，过渡到 AWS Provider / LocalStack 实验的关键一步。

它教你的不是 EC2 本身，而是这个工程模式：

```text
已有基础设施
    ↓
Terraform data source 查询
    ↓
把查询结果传给 output、resource、module
    ↓
实现跨团队、跨模块、跨环境复用
```

这是生产 Terraform 里非常常见的能力。尤其在平台工程、SRE、云迁移项目中，你很少能完全从零创建所有资源。更多时候，你要学会读取已有资源、组合已有资源，并逐步把它们纳入 IaC 工作流。
