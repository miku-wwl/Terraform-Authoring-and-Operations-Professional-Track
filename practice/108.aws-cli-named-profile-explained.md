# Terraform 实操训练 108：AWS CLI Named Profile 讲解版

> 来源：`108.md`  
> 主题：AWS CLI Named Profile  
> 实验环境：LocalStack 模拟 AWS API，不访问真实 AWS

---

## 1. 这一节到底在学什么？

这一节的核心不是“怎么写几行 AWS CLI 配置”，而是学会：

> **同一台机器上如何管理多套 AWS 身份，并让 AWS CLI / Terraform 明确使用其中某一套。**

在真实项目里，你很少只有一个 AWS 身份。常见情况包括：

- 一个账号用于开发环境；
- 一个账号用于测试环境；
- 一个账号用于生产环境；
- 一个账号用于公司项目；
- 一个账号用于个人实验；
- 一个账号用于只读审计；
- 一个账号用于 CI/CD 自动化。

如果所有命令都默认使用同一套凭证，就很容易出现事故：

```text
你以为在 dev 环境 apply，
结果 Terraform 用的是 prod 凭证。
```

Named profile 的价值就是：  
**把不同身份起名字，然后在运行命令时明确选择。**

---

## 2. Named Profile 是什么？

AWS CLI 默认会读取本机的 AWS 配置文件。

常见路径是：

```text
~/.aws/credentials
~/.aws/config
```

其中：

- `credentials` 通常放访问密钥；
- `config` 通常放区域、输出格式、SSO、role assumption 等配置。

默认配置一般长这样：

```ini
[default]
aws_access_key_id = test
aws_secret_access_key = test
```

如果只有 `[default]`，那 AWS CLI 默认就用它。

Named profile 就是给多套配置起名字，例如：

```ini
[default]
aws_access_key_id = test
aws_secret_access_key = test

[dev]
aws_access_key_id = dev-access-key
aws_secret_access_key = dev-secret-key

[prod]
aws_access_key_id = prod-access-key
aws_secret_access_key = prod-secret-key
```

这里的 `dev` 和 `prod` 就是 profile 名。

你可以理解成：

```text
default = 默认身份
dev     = 开发环境身份
prod    = 生产环境身份
```

---

## 3. 为什么 Terraform 也要关心 AWS Profile？

Terraform 的 AWS Provider 需要知道“用哪个身份去调用 AWS API”。

它可以从很多地方拿身份，例如：

1. 环境变量；
2. AWS CLI profile；
3. provider block 里的 `profile`；
4. EC2 / ECS / IRSA / OIDC 等运行环境身份；
5. CI/CD 注入的凭证。

在本节实验里，我们重点看：

```text
AWS CLI named profile -> Terraform AWS Provider
```

也就是：  
**你本地选了哪个 AWS profile，Terraform 就用哪个身份去操作资源。**

---

## 4. `AWS_PROFILE` 和 `--profile` 的区别

### 4.1 `--profile`

`--profile` 是 AWS CLI 命令参数，只对当前这条 AWS CLI 命令生效。

例如：

```sh
aws s3 ls --profile dev
```

意思是：

```text
这一次 aws s3 ls 使用 dev 这个 profile。
```

它不会永久改变当前 shell 的身份。

---

### 4.2 `AWS_PROFILE`

`AWS_PROFILE` 是环境变量，会影响当前 shell 会话里后续读取 AWS SDK 配置的程序。

例如：

```sh
export AWS_PROFILE=dev
aws s3 ls
terraform plan
```

这时：

```text
aws s3 ls 可能使用 dev profile
terraform plan 也可能通过 AWS Provider 使用 dev profile
```

这就是它和 `--profile` 最大的区别：

| 方式 | 影响范围 | 常见用途 |
|---|---|---|
| `--profile dev` | 只影响当前 AWS CLI 命令 | 临时执行一条命令 |
| `export AWS_PROFILE=dev` | 影响当前 shell 后续命令 | 让 Terraform / SDK / CLI 都走同一身份 |

---

## 5. Terraform 里怎么指定 profile？

常见有两种方式。

---

### 5.1 方式一：使用环境变量

```sh
export AWS_PROFILE=dev
terraform init
terraform plan
terraform apply
```

这种方式简单，适合本地实验。

优点：

```text
不用在 Terraform 代码里写死 profile。
```

缺点：

```text
如果当前 shell 的 AWS_PROFILE 设错了，Terraform 也会跟着错。
```

所以生产环境里执行前最好确认一下：

```sh
echo $AWS_PROFILE
aws sts get-caller-identity
```

---

### 5.2 方式二：在 provider 里指定

```hcl
provider "aws" {
  region  = "ap-southeast-2"
  profile = "dev"
}
```

这表示 AWS Provider 明确使用 `dev` profile。

优点：

```text
身份选择写在代码里，比较明确。
```

缺点：

```text
如果代码被多个环境复用，写死 profile 可能不灵活。
```

更常见的写法是把 profile 做成变量：

```hcl
variable "aws_profile" {
  type    = string
  default = "dev"
}

provider "aws" {
  region  = "ap-southeast-2"
  profile = var.aws_profile
}
```

然后运行时指定：

```sh
terraform plan -var="aws_profile=dev"
terraform plan -var="aws_profile=prod"
```


这非常适合学习 named profile，因为你可以练完整流程，但不会真的创建 AWS 资源，也不会产生云账单。

你可以把 LocalStack 理解成：

```text
本地模拟版 AWS API。
```

Terraform 以为自己在调用 AWS，实际上打到的是：

```text
http://localhost:4566
```

这对训练非常友好。

## 7. LocalStack 环境下的 provider 典型形态

在 LocalStack 实验里，AWS Provider 通常会有类似配置：

```hcl
provider "aws" {
  region                      = "us-east-1"
  profile                     = "localstack"
  access_key                  = "test"
  secret_key                  = "test"
  s3_use_path_style           = true
  skip_credentials_validation = true
  skip_metadata_api_check     = true
  skip_requesting_account_id  = true

  endpoints {
    s3 = "http://localhost:4566"
  }
}
```

不同实验文件可能略有不同，但思想一样：

```text
告诉 Terraform：
不要去真实 AWS；
去 LocalStack endpoint；
凭证只是占位用的 test/test。
```

## 8. 运行时应该怎么理解身份链路？

这一节最重要的是建立这条链路：

```text
~/.aws/credentials
        ↓
named profile
        ↓
AWS_PROFILE 或 --profile
        ↓
AWS CLI / Terraform AWS Provider
        ↓
LocalStack 或真实 AWS API
```

如果你以后在真实项目里排障 AWS 身份问题，也基本沿着这条链路查。

## 11. `aws sts get-caller-identity` 为什么很重要？

真实 AWS 环境里，执行 Terraform 前最好先跑：

```sh
aws sts get-caller-identity
```

它会告诉你当前命令到底是谁。

你可以把它理解成 AWS 版的：

```sh
whoami
```

在生产环境里，这一步非常重要。

因为 Terraform 事故很多时候不是 HCL 写错，而是身份选错：

```text
代码是对的。
命令也是对的。
但是 profile 错了。
结果资源被创建到了错误账号。
```

所以实际工作里，尤其是 apply 前，应该养成习惯：

```sh
echo $AWS_PROFILE
aws sts get-caller-identity
terraform plan
terraform apply
```

## 12. 常见坑 1：以为 `--profile` 会影响 Terraform

很多人会这样想：

```sh
aws s3 ls --profile dev
terraform plan
```

然后以为 Terraform 也会使用 `dev`。

这不一定。

因为 `--profile dev` 只影响前面那条 `aws s3 ls` 命令。

Terraform 并不会自动继承上一条 AWS CLI 命令里的 `--profile` 参数。

如果你希望 Terraform 也使用 dev，应该：

```sh
export AWS_PROFILE=dev
terraform plan
```

或者在 provider 中写：

```hcl
provider "aws" {
  profile = "dev"
}
```

---

## 13. 常见坑 2：profile 写在错误文件格式里

AWS 的两个文件格式容易混。

`~/.aws/credentials` 中通常写：

```ini
[dev]
aws_access_key_id = xxx
aws_secret_access_key = yyy
```

而 `~/.aws/config` 中 named profile 通常写：

```ini
[profile dev]
region = ap-southeast-2
output = json
```

注意：

```text
credentials 里是 [dev]
config 里是 [profile dev]
```

这个细节很容易错。

---

## 14. 常见坑 3：LocalStack 和真实 AWS 混用

本节实验走的是 LocalStack：

```text
http://localhost:4566
```

真实 AWS 不需要这个 endpoint。

所以从本地模拟迁移到真实 AWS 时，要特别注意：

- 删除或调整 LocalStack endpoints；
- 不再使用 `test/test` 假凭证；
- 不再随便跳过 credentials/account 校验；
- 确认 region；
- 确认 profile；
- 先跑 `plan`，不要直接 `apply`。

本地实验的目的是练流程，不是让你把 LocalStack provider 原样搬到生产。

---

## 15. 常见坑 4：多个终端的环境变量不一样

你在一个 PowerShell / Bash 里执行：

```sh
export AWS_PROFILE=dev
```

只影响当前这个终端。

你打开另一个终端，可能没有这个变量。

所以排障时一定要在当前终端里确认：

```sh
echo $AWS_PROFILE
```

Windows PowerShell 对应写法是：

```powershell
$env:AWS_PROFILE="dev"
echo $env:AWS_PROFILE
```

CMD 对应写法是：

```cmd
set AWS_PROFILE=dev
echo %AWS_PROFILE%
```

## 17. 和 Terraform 状态有什么关系？

Named profile 本身不直接改变 Terraform state。

但是它会影响 Terraform 连接到哪个 AWS 账号。

所以间接影响很大：

```text
同一份 Terraform 代码，
如果 profile 指向不同 AWS 账号，
就会操作不同账号里的资源。
```

这就是为什么 profile 管理和 backend 管理经常要一起设计。

例如：

```text
dev profile  -> dev state backend  -> dev AWS account
prod profile -> prod state backend -> prod AWS account
```

如果 profile 和 backend 搭错，就会非常危险。

比如：

```text
使用 prod profile，
却读到了 dev state。
```

或者：

```text
使用 dev profile，
却操作 prod backend。
```

这类错误在真实团队里属于高危问题。

- 知道 profile 的作用；

- 知道 apply 前要确认身份；
- 知道 state/backend/account 要匹配；
- 知道本地实验和生产 CI/CD 的区别。

## 20. 最终总结

这一节的关键收获是：

1. Named profile 用来管理多套 AWS 身份。
2. `--profile` 只影响当前 AWS CLI 命令。
3. `AWS_PROFILE` 会影响当前 shell 里的 AWS CLI、SDK、Terraform。
4. Terraform AWS Provider 可以从环境变量或 provider 配置中读取 profile。
5. LocalStack 里使用 `test/test` 假凭证和本地 endpoint，不会访问真实 AWS。
6. 真实项目里，profile、账号、region、backend、state 必须严格对应。
7. `aws sts get-caller-identity` 是 apply 前确认身份的好习惯。

一句话收尾：

> **Named profile 不是小技巧，而是多账号 AWS 工程实践的身份安全基础。**
