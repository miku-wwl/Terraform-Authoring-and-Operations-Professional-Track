# Terraform 实操训练 48 讲解：`prevent_destroy`

## 1. 这一节到底在学什么？

Lab 48 学的是 Terraform `lifecycle` 里的另一个重要参数：

```hcl
prevent_destroy = true
```

它的作用非常直接：

> 阻止 Terraform 销毁这个资源。

这节和 Lab 47 的 `create_before_destroy` 不一样。  
Lab 47 关注的是“资源替换时先建新还是先删旧”。  
Lab 48 关注的是“这个资源根本不应该被 Terraform 轻易删掉”。

一句话理解：

> `prevent_destroy` 是 Terraform 里的防误删保险丝。

---

## 2. 为什么需要 `prevent_destroy`？

真实生产里，有些资源一旦误删，后果很严重。

例如：

- 数据库
- 对象存储桶
- 关键配置文件
- DNS zone
- KMS key
- Vault mount
- 生产 VPC
- 状态型磁盘
- 长期保留的审计日志资源

如果有人误执行：

```sh
terraform destroy
```

或者改配置导致 Terraform 计划销毁关键资源，那就可能变成事故。

`prevent_destroy` 的价值就是在这种时候挡一下：

```text
不行，这个资源配置了 prevent_destroy，我拒绝销毁。
```

---

## 3. 基本写法

典型写法如下：

```hcl
resource "local_file" "critical_config" {
  filename = "${path.module}/output/critical-config.txt"
  content  = "critical configuration"

  lifecycle {
    prevent_destroy = true
  }
}
```

这表示：

> `local_file.critical_config` 这个资源不能被 Terraform destroy 直接销毁。

如果执行：

```sh
terraform destroy -auto-approve
```

Terraform 会失败，并提示这个资源有 `prevent_destroy` 保护。

---

## 4. 本实验的场景

Lab 48 的背景是：

> 关键配置文件不允许被自动销毁。

问题是：

> 误执行 destroy 应该被阻止。

任务是：

1. 创建受保护文件。
2. 验证 `terraform destroy` 会失败。
3. 使用 `terraform state rm` 把资源从 state 中移除。
4. 手动删除本地文件完成实验清理。

预期结果：

- destroy 被 `prevent_destroy` 阻止。
- state 清理后，本地文件可以手动删除。

这节非常适合训练你对 Terraform state 和真实资源关系的理解。

---

## 5. `prevent_destroy` 到底保护了什么？

它保护的是：

> Terraform 通过正常 plan/apply/destroy 流程销毁这个资源。

也就是说，只要资源块还在配置中，并且里面写着：

```hcl
prevent_destroy = true
```

Terraform 就会拒绝销毁它。

但是注意，它不是云平台级别的锁。  
它不是 AWS 删除保护，不是 Azure resource lock，也不是 Kubernetes finalizer。

它只是 Terraform 层面的保护。

如果有人绕过 Terraform，直接去云控制台删资源，`prevent_destroy` 管不了。

---

## 6. 关键边界：资源块还在配置中时才生效

这个点非常容易踩坑。

Lab 48 原文特别强调：

> `prevent_destroy` 只有在资源块仍存在于配置中时才生效，移除整个资源块并不会继续提供保护。

这是什么意思？

如果你有：

```hcl
resource "local_file" "critical_config" {
  filename = "${path.module}/output/critical-config.txt"
  content  = "critical configuration"

  lifecycle {
    prevent_destroy = true
  }
}
```

这时 destroy 会被挡住。

但是如果你把整个资源块从代码里删掉，那么 Terraform 配置里也就没有这段 lifecycle 了。  
这时 Terraform 不能再从配置中读取 `prevent_destroy` 规则。

所以不要误以为：

> 只要这个资源曾经设置过 `prevent_destroy`，它就永久安全。

不是。

`prevent_destroy` 是配置里的规则。配置没了，规则也没了。

---

## 7. 为什么实验清理要用 `terraform state rm`？

本实验里，正常 destroy 会失败，这是预期。

因为资源被保护了。

如果只是实验环境，要清理它，有一种方式是：

```sh
terraform state rm local_file.critical_config
```

这条命令的意思是：

> 把这个资源从 Terraform state 里移除，让 Terraform 不再管理它。

注意：`state rm` 不会删除真实资源。

所以执行完：

```sh
terraform state rm local_file.critical_config
```

之后，还需要手动删除本地文件：

```sh
rm -f output/critical-config.txt
```

这个流程非常适合训练一个重要概念：

> state rm 是“解除 Terraform 管理关系”，不是“删除资源”。

---

## 8. `terraform state rm` 和 `terraform destroy` 的区别

| 命令 | 是否删除真实资源 | 是否修改 state | 典型用途 |
|---|---|---|---|
| `terraform destroy` | 会 | 会 | 销毁 Terraform 管理的资源 |
| `terraform state rm` | 不会 | 会 | 让 Terraform 忘记这个资源 |
| 手动 `rm` / 控制台删除 | 会 | 不会 | 绕过 Terraform 删除真实资源 |

本实验清理流程是：

1. `terraform state rm`：先让 Terraform 忘掉这个受保护资源。
2. `rm -f output/critical-config.txt`：再手动删除本地文件。

这样就不会被 `prevent_destroy` 卡住。

---

## 9. 为什么生产不要随便 `state rm`？

实验里可以这么清理，是因为这是本地训练。

生产环境里，`terraform state rm` 是高风险操作。

因为它会让 Terraform 不再管理这个资源。后果可能包括：

- 资源还在，但 Terraform 不知道它存在。
- 后续可能重复创建同类资源。
- 配置和真实环境开始漂移。
- 审计链路断掉。
- 团队以为 Terraform 管着，实际上已经不管了。
- 未来 import / cleanup 变复杂。

所以真实生产里如果要 `state rm`，必须走变更审批，并且写清楚：

- 为什么要移除 state
- 是否保留真实资源
- 谁接管这个资源
- 是否后续要 import 回来
- 是否有备份和回滚方案

一句话：

> 实验里 state rm 是清理手段；生产里 state rm 是变更操作。

```sh

```

## 11. 如何判断 `prevent_destroy` 生效了？

最直观的方式就是执行：

```sh
terraform destroy -auto-approve
```

如果看到 Terraform 报错，提示资源受 `prevent_destroy` 保护，说明它生效了。

本实验的验证报告也明确说明：

> `terraform destroy` 按预期失败并提示 `prevent_destroy`。

所以不要看到 destroy 失败就慌。  
在这个 lab 里，destroy 失败就是成功信号。

---

## 12. `prevent_destroy` 适合保护什么？

适合保护高风险资源：

### 数据类资源

- 数据库
- 存储桶
- 磁盘
- 备份库
- 日志归档

### 安全类资源

- KMS key
- Vault mount
- 密钥容器
- 审计配置

### 网络类关键资源

- 生产 VPC
- 核心 subnet
- DNS zone
- 公网入口

### 平台类关键资源

- Kubernetes 集群
- Terraform backend 相关资源
- 共享镜像库
- 关键配置文件

核心判断标准是：

> 这个资源如果被误删，会不会造成明显事故、数据丢失或恢复成本很高？

如果答案是会，就值得考虑 `prevent_destroy`。

---

## 13. `prevent_destroy` 不适合什么？

不适合无脑到处加。

比如：

- 临时文件
- 测试资源
- 随时可重建的 demo 资源
- CI 临时环境
- ephemeral preview environment

如果所有资源都加 `prevent_destroy`，结果就是：

- destroy 很难执行
- 测试环境清理困难
- CI/CD 堆积垃圾资源
- 运维成本增加
- 真正需要保护的资源反而不突出

所以它应该用在关键资源上，而不是当成默认模板乱贴。

---

## 14. 和云平台删除保护的区别

很多云平台也有自己的删除保护机制。

例如：

- AWS RDS deletion protection
- Azure resource lock
- GCP deletion protection
- Kubernetes finalizer

这些是在远端平台层面保护资源。  
Terraform 的 `prevent_destroy` 是 Terraform 配置层面的保护。

两者可以配合，但不是同一个东西。

| 保护机制 | 生效位置 | 能否防止控制台删除 |
|---|---|---|
| Terraform `prevent_destroy` | Terraform plan/apply/destroy | 不能 |
| 云平台删除保护 | 云平台 API | 通常可以 |
| IAM 权限限制 | 身份权限层 | 可以 |
| 审批流程 | 组织流程层 | 间接防止 |

生产关键资源通常应该多层保护，而不是只靠一个 `prevent_destroy`。

---

## 15. 和 `create_before_destroy` 的区别

| 参数 | 作用 |
|---|---|
| `create_before_destroy` | 替换时尽量先创建新资源，再销毁旧资源 |
| `prevent_destroy` | 阻止 Terraform 销毁资源 |

简单说：

- `create_before_destroy` 关注替换顺序。
- `prevent_destroy` 关注禁止销毁。

如果一个资源配置了 `prevent_destroy`，任何会导致销毁的动作都可能被阻止，包括替换。

所以要小心：  
某些属性变化如果会触发 replace，而 replace 本质包含 destroy，就可能被 `prevent_destroy` 拦下来。

---

## 16. 和 `state rm` 的关系

本实验专门让你练了这个组合：

```sh
terraform destroy -auto-approve
terraform state rm local_file.critical_config
rm -f output/critical-config.txt
```

理解这个组合很重要：

1. destroy 失败：证明保护生效。
2. state rm：解除 Terraform 对资源的管理。
3. 手动删除文件：清理真实对象。

但生产里不应该把这个当常规删除流程。

如果真的要删除受保护资源，通常应该：

1. 提交变更申请。
2. 评估影响和备份。
3. 明确删除窗口。
4. 修改配置，移除或调整 `prevent_destroy`。
5. 执行计划并审批。
6. 删除后验证。

---

## 17. 常见坑

### 坑 1：以为 `prevent_destroy` 是永久保护

不是。

它只在配置里还存在时生效。  
删除整个资源块后，这条规则也没了。

---

### 坑 2：以为它能防控制台删除

不能。

如果有人有云平台权限，直接在控制台删除资源，Terraform 拦不住。

---

### 坑 3：以为 destroy 失败是异常

在本实验里 destroy 失败是预期结果。  
这说明 `prevent_destroy` 生效。

---

### 坑 4：生产随便 `state rm`

`state rm` 不删资源，只是让 Terraform 忘记资源。  
生产中随便用会制造漂移和治理风险。

---

### 坑 5：替换时被 prevent_destroy 卡住

如果某个属性变化触发 replace，而资源配置了 `prevent_destroy`，Terraform 会拒绝，因为 replace 包含 destroy。

这时要认真判断：

- 这个替换是否真的必要？
- 是否需要变更窗口？
- 是否需要备份？
- 是否应该移除保护？
- 是否有更安全的迁移方案？

---

## 18. SRE / Platform Engineering 视角

从 SRE 视角看，`prevent_destroy` 是防事故工具，不是偷懒工具。

它适合保护 blast radius 大的资源。

平台团队在模块里使用它时，应该写清楚原因，例如：

```hcl
lifecycle {
  prevent_destroy = true
}
```

同时在 README 里说明：

```text
This resource stores production data. Destruction requires explicit change approval.
```

成熟做法不是悄悄加一个参数，让别人 destroy 时被吓一跳，而是把保护策略写清楚。

---

## 19. 生产建议

对关键资源，建议组合使用多层保护：

1. Terraform `prevent_destroy`
2. 云平台 deletion protection / resource lock
3. IAM 最小权限
4. 远程 state 权限控制
5. 变更审批
6. 备份和恢复演练
7. plan 审查
8. saved plan apply

这样才是真正的防误删体系。

只靠一个 Terraform 参数，不够。

## 21. 面试/考试记忆版

可以这样记：

- `prevent_destroy` 写在 `lifecycle` 里。
- 它会阻止 Terraform 销毁资源。
- 适合保护数据库、关键配置、存储等高风险对象。
- 它只有在资源块仍存在于配置中时才生效。
- 它不能防止云控制台或 API 直接删除。
- `terraform state rm` 只是让 Terraform 忘记资源，不会删除真实资源。
- 生产中不要随便 `state rm`。
- 如果资源替换包含 destroy，也可能被 `prevent_destroy` 拦住。

一句话总结：

> `prevent_destroy` 是 Terraform 层面的防误删保险丝，但不是云平台删除保护，也不是 state 管理的免死金牌。

## 22. 本 Lab 的价值

Lab 48 的价值在于，它训练你对 Terraform 删除风险的敬畏感。

很多 Terraform 事故不是因为不会创建资源，而是因为：

- 看错 plan
- 误执行 destroy
- 替换导致删除
- 关键资源没有保护
- state 操作太随意
- 以为 Terraform 会自动帮你兜底

`prevent_destroy` 是一个基础但非常有用的防线。  
不过你要记住：真正的生产安全不是靠一个参数，而是靠 Terraform 配置、云平台保护、权限控制、审批流程、备份和审计一起兜住。
