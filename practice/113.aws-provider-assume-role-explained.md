# 第 113 节：AWS Provider Assume Role 实操讲解

## 1. 本节主题

本节讲的是 **Terraform AWS Provider 的 `assume_role` 用法**。

核心一句话：

> Terraform 启动时先使用一组基础凭证调用 STS，临时扮演另一个 IAM Role，然后再使用这个临时身份创建 AWS 资源。

这就是 AWS 里非常常见的 **跨账号部署**、**最小权限部署**、**CI/CD 临时授权** 的基础模式。

本实验使用 **LocalStack** 模拟 AWS API，不访问真实 AWS。LocalStack endpoint 为：

```text
http://localhost:4566
```

实验凭证是假的：

```text
access_key = test
secret_key = test
```

---

## 2. 为什么需要 Assume Role？

在真实 AWS 项目里，不建议把长期 Access Key 直接赋予很大的权限。

更常见的做法是：

1. 用户、CI/CD、GitHub Actions、GitLab Runner 或 Jenkins 拥有一个基础身份；
2. 这个基础身份权限很小，只允许调用 `sts:AssumeRole`；
3. Terraform 通过 STS 获取目标 Role 的临时凭证；
4. Terraform 使用临时凭证创建资源；
5. 临时凭证过期后自动失效。

这套模式的价值是：

- 不需要把高权限长期密钥到处散发；
- 可以把部署权限集中放在 IAM Role 上；
- 可以做跨账号部署；
- 可以给不同环境配置不同 Role；
- 方便审计 CloudTrail 里的 AssumeRole 行为；
- 符合企业级 Landing Zone / 多账号架构的权限模型。

---

## 3. 不用 Assume Role vs 使用 Assume Role

### 3.1 不用 Assume Role

Terraform 直接使用当前凭证创建资源。

```hcl
provider "aws" {
  region     = "us-east-1"
  access_key = "test"
  secret_key = "test"
}
```

这种模式简单，但在真实生产环境中风险较高。

因为谁拿到这组凭证，谁就可以直接用它的权限操作 AWS。

---

### 3.2 使用 Assume Role

Terraform 先用基础凭证认证，然后通过 `assume_role` 切换到目标角色。

```hcl
provider "aws" {
  region     = "us-east-1"
  access_key = "test"
  secret_key = "test"

  assume_role {
    role_arn     = "arn:aws:iam::000000000000:role/example-role"
    session_name = "terraform-lab-113"
  }
}
```

这里的执行逻辑是：

```text
Terraform
  -> AWS Provider
    -> 使用基础凭证调用 STS AssumeRole
      -> 拿到临时 AccessKey / SecretKey / SessionToken
        -> 使用临时身份创建资源
```

注意：

> `assume_role` 不是给 Terraform 起一个名字，而是让 Terraform Provider 换一个 AWS 身份继续执行。

---

## 4. STS 是什么？

STS 是 **Security Token Service**。

它的作用是签发临时安全凭证。

Assume Role 的本质就是调用 STS：

```text
sts:AssumeRole
```

调用成功后，AWS 返回三样东西：

```text
AccessKeyId
SecretAccessKey
SessionToken
```

这三样合起来就是一组临时凭证。

和普通长期 Access Key 不同，临时凭证有有效期，到期自动失效。

---

## 5. Terraform AWS Provider 中的 assume_role 块

常见配置如下：

```hcl
provider "aws" {
  region = "us-east-1"

  assume_role {
    role_arn     = "arn:aws:iam::123456789012:role/terraform-deploy-role"
    session_name = "terraform-session"
  }
}
```

关键字段：

| 字段 | 作用 |
|---|---|
| `role_arn` | 要扮演的 IAM Role ARN |
| `session_name` | STS 会话名称，方便审计和排查 |
| `external_id` | 第三方跨账号访问时常用的额外安全字段 |
| `duration` | 临时凭证有效期 |
| `policy` | 给这次会话额外收窄权限的 session policy |

最核心的是 `role_arn`。

`session_name` 虽然不是业务权限本身，但真实环境里建议认真命名，因为它会出现在审计日志中。

---

## 6. 真实 AWS 中需要满足什么条件？

在真实 AWS 中，Assume Role 能成功，需要两边都配置正确。

### 6.1 调用方需要有 sts:AssumeRole 权限

例如基础身份需要允许：

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": "sts:AssumeRole",
      "Resource": "arn:aws:iam::123456789012:role/terraform-deploy-role"
    }
  ]
}
```

意思是：

> 当前身份有资格请求扮演这个 Role。

---

### 6.2 目标 Role 需要信任调用方

目标 Role 的 trust policy 需要允许调用方扮演它。

例如：

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        "AWS": "arn:aws:iam::111111111111:user/terraform-user"
      },
      "Action": "sts:AssumeRole"
    }
  ]
}
```

意思是：

> 这个 Role 愿意被指定的用户、Role 或账号扮演。

---

## 7. 最容易搞混的点

### 7.1 有权限调用 sts:AssumeRole，不等于一定能成功

Assume Role 是双向校验：

```text
调用方 IAM Policy 允许 sts:AssumeRole
并且
目标 Role Trust Policy 信任调用方
```

两个条件缺一个都不行。

这就像进公司大楼：

- 你手里有门禁卡；
- 大楼门禁系统也得认识你。

只有你自己说“我有权限”没用，大楼也得放你进去。

---

### 7.2 assume_role 之后，真正创建资源的是目标 Role

Terraform 创建资源时用的是 Assume Role 后拿到的临时身份。

所以资源创建失败时，不要只看基础用户的权限，还要看目标 Role 的权限。

真实排查时要分两层：

1. 能不能 AssumeRole？
2. AssumeRole 后的身份有没有权限创建目标资源？

---

### 7.3 session_name 不是权限

`session_name` 只是这次临时会话的名字。

它方便审计，但不会自动赋予权限。

不要以为写了：

```hcl
session_name = "admin"
```

就变成管理员了。

这只是名字，不是魔法咒语。

---

## 8. LocalStack 中能验证什么？

本节实验使用 LocalStack，所以它适合验证：

- Terraform AWS Provider 能否正确读取 `assume_role` 配置；
- Provider 是否会走 STS AssumeRole 调用路径；
- Terraform init / plan / apply / destroy 的基本闭环；
- 本地 endpoint、假凭证、provider 配置是否正确。

也就是说，LocalStack 适合验证：

```text
配置链路是否打通
```

---

## 9. LocalStack 中不能完全验证什么？

LocalStack 不能完全替代真实 AWS IAM。

尤其是：

- 真实 IAM Role trust policy；
- 真实跨账号 AssumeRole；
- 真实 CloudTrail 审计；
- 真实 AWS Organizations / Control Tower / Landing Zone 账号结构；
- 企业级 Permission Boundary / SCP / Session Policy 组合效果。

所以本节验证结论是：

```text
aws-localstack-limited
```

意思是：

> 本地可以验证 provider assume_role 配置和 STS 调用路径，但真实 AWS 的跨账号信任策略仍然要在真实账号里验证。

## 11. Terraform 配置示例：LocalStack 版本

LocalStack 版本通常需要加 endpoint 和跳过真实 AWS 校验的参数。

示例：

```hcl
terraform {
  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 5.0"
    }
  }
}

provider "aws" {
  region                      = "us-east-1"
  access_key                  = "test"
  secret_key                  = "test"
  skip_credentials_validation = true
  skip_metadata_api_check     = true
  skip_requesting_account_id  = true
  s3_use_path_style           = true

  endpoints {
    sts = "http://localhost:4566"
    s3  = "http://localhost:4566"
  }

  assume_role {
    role_arn     = "arn:aws:iam::000000000000:role/localstack-terraform-role"
    session_name = "terraform-lab-113"
  }
}
```

这个配置表达的意思是：

1. Terraform 使用 AWS Provider；
2. 不访问真实 AWS；
3. 所有相关 API 指向 LocalStack；
4. 先走 STS AssumeRole；
5. 再用临时身份继续执行资源操作。

```bash
docker compose up -d
```

```bash
docker ps
```

```bash
terraform init
```

这一步会：

- 初始化工作目录；
- 下载 AWS Provider；
- 读取 provider 配置。

### 12.3 查看执行计划

```bash
terraform plan
```

这一步会触发 Provider 准备阶段。

如果 `assume_role` 配置、endpoint 或凭证有问题，通常会在这一步暴露。

### 12.4 应用资源

```

AWS_ACCESS_KEY_ID=test \

```

### 12.6 销毁资源

```bash

```

## 13. 真实 AWS 迁移时怎么改？

从 LocalStack 迁移到真实 AWS 时，主要改这些地方。

### 13.1 删除 LocalStack endpoint

真实 AWS 不需要：

```hcl
endpoints {
  sts = "http://localhost:4566"
  s3  = "http://localhost:4566"
}
```

### 13.2 删除跳过校验参数

真实 AWS 里一般不应该保留这些：

```hcl
skip_credentials_validation = true
skip_metadata_api_check     = true
skip_requesting_account_id  = true
```

### 13.3 使用真实 Role ARN

LocalStack 示例：

```hcl
role_arn = "arn:aws:iam::000000000000:role/localstack-terraform-role"
```

真实 AWS 示例：

```hcl
role_arn = "arn:aws:iam::123456789012:role/terraform-deploy-role"
```

### 13.4 使用真实基础凭证来源

真实项目里不建议把密钥写死在 `.tf` 文件中。

更推荐：

- AWS profile；
- 环境变量；
- GitHub Actions OIDC；
- AWS SSO；
- EC2/ECS/EKS workload identity；
- CI/CD secret manager。

例如使用 profile：

```hcl
provider "aws" {
  region  = "us-east-1"
  profile = "dev-admin"

  assume_role {
    role_arn     = "arn:aws:iam::123456789012:role/terraform-deploy-role"
    session_name = "terraform-dev"
  }
}
```

## 14. 常见报错与排查

### 14.1 AccessDenied: not authorized to perform sts:AssumeRole

含义：当前基础身份没有权限调用 `sts:AssumeRole`。

排查：

- 检查调用方 IAM policy；
- 检查 `Resource` 是否写对目标 Role ARN；
- 检查是否被 SCP、Permission Boundary 或 Session Policy 限制。

---

### 14.2 AccessDenied: role cannot be assumed

含义：目标 Role 不信任当前调用方。

排查：

- 检查目标 Role 的 trust policy；
- 检查 Principal 是否写对；
- 跨账号时检查账号 ID 是否正确；
- 如果用了 external_id，检查双方是否一致。

---

### 14.3 InvalidClientTokenId

含义：基础凭证无效，或者请求打到了错误 endpoint。

在 LocalStack 实验中重点检查：

- `AWS_ACCESS_KEY_ID` 是否是 `test`；
- `AWS_SECRET_ACCESS_KEY` 是否是 `test`；
- `endpoint` 是否指向 `http://localhost:4566`；
- LocalStack 是否启动。

---

### 14.4 RequestError: connection refused

含义：Terraform 访问不到 LocalStack。

排查：

```bash
docker ps
```

确认容器是否运行。

也可以检查：

```bash
curl http://localhost:4566/_localstack/health
```

---

## 15. 本节你真正应该掌握什么？

本节不是为了背一个 `assume_role` 代码块。

你真正要掌握的是这个身份链路：

```text
基础身份
  -> sts:AssumeRole
    -> 临时凭证
      -> 目标 Role 权限
        -> Terraform 创建资源
```

以及真实 AWS 的双重条件：

```text
调用方允许 AssumeRole
目标 Role 信任调用方
```

这个点非常重要。

很多 Terraform AWS 多账号部署问题，本质都是 AssumeRole 链路没打通。

---

## 16. 面试表达方式

可以这样说：

> In Terraform AWS Provider, `assume_role` allows Terraform to first authenticate with a base identity, then call AWS STS to assume a target IAM Role and use temporary credentials for resource provisioning. In real AWS, both the caller permission and the target role trust policy must be configured correctly. In this lab, LocalStack validates the provider configuration and STS call path, but real cross-account IAM trust still needs to be tested in actual AWS accounts.

中文解释：

> Terraform AWS Provider 的 `assume_role` 让 Terraform 先使用基础身份认证，然后调用 STS 扮演目标 IAM Role，再使用临时凭证创建资源。真实 AWS 中，调用方权限和目标 Role 的信任策略都必须正确。本实验用 LocalStack 验证 provider 配置和 STS 调用路径，但真实跨账号 IAM 信任关系仍需要在真实 AWS 账号中验证。

---

## 17. 本节小结

- `assume_role` 是 AWS Provider 的身份切换配置；
- 它底层调用 STS `AssumeRole`；
- 成功后 Terraform 使用临时凭证创建资源；
- 真实 AWS 中要同时配置调用方权限和目标 Role trust policy；
- LocalStack 可以验证配置路径，但不能完全替代真实 IAM 跨账号验证；
- 生产环境中不要把高权限长期密钥直接写进 Terraform；
- CI/CD、跨账号部署、多环境隔离都会大量使用 Assume Role。
