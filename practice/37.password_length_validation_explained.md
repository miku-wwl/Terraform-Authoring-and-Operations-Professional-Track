# Terraform 实操训练 37 讲解：密码长度验证

## 1. 这一节到底在学什么？

这一节学习 Terraform 输入变量验证中的一个非常典型场景：**密码长度校验**。

在真实项目里，很多错误不应该等到云平台 API、数据库 API 或应用启动时才暴露。像密码长度、环境名称、实例规格、端口范围这类规则，最好在 Terraform 的 plan 阶段就提前拦住。

本实验的核心是：

```hcl
validation {
  condition     = length(var.db_password) >= 12
  error_message = "Database password must be at least 12 characters long."
}
```

这段逻辑表达的是：

> `db_password` 的长度必须至少 12 位，否则 Terraform plan 直接失败。

---

## 2. 为什么密码长度要在 Terraform 里验证？

很多人会觉得密码规则应该由数据库、云平台或应用来验证。这个想法不算错，但不够工程化。

如果密码太短，等到 apply 时才被远端拒绝，会带来几个问题：

1. CI/CD 流水线已经跑了一大半，浪费时间。
2. 错误信息可能来自远端 API，不一定容易理解。
3. 部分资源可能已经创建，导致环境进入半完成状态。
4. 安全规则分散在多个系统里，后续维护困难。

Terraform 的变量验证可以把这些问题前置到 plan 阶段：

```text
输入不合格 → plan 失败 → 不进入 apply → 不触碰远端资源
```

这就是 IaC 里的“早失败”思想。

---

## 3. `validation` 由哪几部分组成？

Terraform 变量验证通常由三部分组成：

```hcl
variable "db_password" {
  type      = string
  sensitive = true

  validation {
    condition     = length(var.db_password) >= 12
    error_message = "Database password must be at least 12 characters long."
  }
}
```

### 3.1 `condition`

`condition` 是真正的判断条件。

在本实验里：

```hcl
length(var.db_password) >= 12
```

意思是：

> 取 `db_password` 的字符串长度，并判断是否大于等于 12。

如果结果是 `true`，变量通过验证。

如果结果是 `false`，Terraform plan 失败。

---

### 3.2 `error_message`

`error_message` 是失败时给用户看的提示。

它不应该写得太玄学，比如：

```text
Invalid value
```

这种信息没什么帮助。更好的写法是明确告诉用户怎么修：

```text
Database password must be at least 12 characters long.
```

好的错误信息应该包含两个点：

1. 什么错了。
2. 怎么改。

---

### 3.3 `sensitive = true`

密码变量必须标记为 sensitive：

```hcl
sensitive = true
```

这会让 Terraform 在 CLI 输出中隐藏敏感值，避免密码直接出现在日志里。

例如输出里可能看到：

```text
db_password = <sensitive>
```

这是预期行为，不是错误。

---

## 4. `length()` 函数怎么理解？

`length()` 用来计算字符串、列表、map 等对象的长度。

在密码场景里：

```hcl
length(var.db_password)
```

计算的是密码字符串的字符数量。

例如：

```text
"short"              → length = 5
"secure-pass-123"    → length = 15
```

所以本实验要求：

```hcl
length(var.db_password) >= 12
```

也就是密码至少 12 位。

---

## 5. 本实验的设计思路

Lab 37 的实验设计很直接：

- 默认密码满足长度要求。
- 短密码应该在 plan 阶段失败。
- 密码变量被标记为 sensitive。
- `terraform test` 固化这个规则。

它训练的是一个非常常见的生产习惯：

> 不要相信用户输入；能在 Terraform 层提前验证的，就提前验证。

## 6. 标准运行流程

进入实验目录：

```sh
cd /workspace/practice/labs/37
```

初始化：

```sh
terraform init -input=false
```

格式检查：

```sh
terraform fmt -check
```

语法验证：

```sh
terraform validate
```

运行 Terraform 测试：

```sh
terraform test
```

生成 plan：

```sh
terraform plan -input=false -no-color -out=tfplan
```

执行 apply：

```sh
terraform apply -auto-approve tfplan
```

查看输出：

```sh
terraform output
```

清理：

```sh
terraform destroy -auto-approve
```

## 7. 怎么验证短密码会失败？

可以手动传一个短密码测试：

```sh
terraform plan -var='db_password=short'
```

如果变量验证写对了，Terraform 应该直接失败，并提示密码至少需要 12 位。

这说明问题被挡在 plan 阶段，没有进入 apply。

---

## 8. 为什么 `sensitive` 不是万能保护？

`sensitive = true` 很有用，但它不是完整的秘密管理方案。

它主要解决的是：

> 不要把敏感值直接打印到 Terraform CLI 输出里。

但它不能解决所有问题，例如：

- 密码仍可能进入 Terraform state。
- state 后端如果不加密，仍有泄露风险。
- CI/CD 环境变量如果配置不当，也可能泄露。
- 日志、脚本、变量文件仍可能人为泄露。

所以生产环境里，密码处理要组合这些手段：

1. `sensitive = true`
2. 加密 remote state
3. 限制 state backend 访问权限
4. 使用 Secret Manager / Key Vault / Parameter Store
5. 避免把密码硬编码进 `.tf` 文件
6. 避免把 `*.tfvars` 秘密文件提交到 Git

## 9. 自动化里应该怎么传密码？

不要在命令行里长期明文传密码：

```sh
terraform plan -var='db_password=some-password'
```

这在本地演示可以，但在生产里容易进入 shell history 或 CI 日志。

更常见的方式是使用环境变量：

```sh
export TF_VAR_db_password='secure-password-123'
terraform plan
```

或者由 CI/CD Secret 注入：

```text
GitHub Actions Secrets
Azure DevOps Variable Group
GitLab CI/CD Variables
AWS Secrets Manager
Azure Key Vault
```

更成熟的做法是让 Terraform 读取 secret manager 中已经存在的 secret，而不是直接在 Terraform 变量里传明文密码。

## 10. 常见坑

### 坑 1：短密码 plan 失败

这是预期行为。

解决方式：把密码改成至少 12 位。

---

### 坑 2：输出显示 `<sensitive>`

这也是预期行为。

说明 `sensitive = true` 生效了。

不要为了“看清楚输出”就把 sensitive 去掉。工程上这属于自己拆安全带。

---

### 坑 3：以为 `sensitive` 会让 state 里也完全看不到密码

不要这样想。

`sensitive` 主要影响 CLI 展示，不代表 state 里没有保存敏感值。

生产环境必须保护 state backend。

---

### 坑 4：只校验长度，不校验复杂度

本实验只演示最基础的长度校验。真实项目可能还需要：

- 至少一个大写字母
- 至少一个小写字母
- 至少一个数字
- 至少一个特殊字符
- 禁止空白字符
- 禁止常见弱密码

但注意，不要把规则写得过度复杂，否则会降低可维护性。安全策略最好结合组织规范和 secret manager 能力统一设计。

---

## 11. 可以扩展成复杂度校验吗？

可以。例如要求密码至少 12 位，并且包含数字：

```hcl
validation {
  condition = length(var.db_password) >= 12 && can(regex("[0-9]", var.db_password))
  error_message = "Database password must be at least 12 characters long and contain at least one number."
}
```

这里的 `can(regex(...))` 用来判断正则是否能匹配成功。

不过这已经比本实验更进一步。Lab 37 的重点还是先掌握：

```hcl
length(var.db_password) >= 12
```

---

## 12. 面试/考试记忆版

可以这样记：

- `validation`：给输入变量加规则。
- `condition`：规则本体，必须返回 true 或 false。
- `error_message`：失败时告诉用户怎么修。
- `length()`：计算字符串、列表或 map 长度。
- `sensitive = true`：隐藏 CLI 输出中的敏感值。
- 密码校验应该尽量在 plan 阶段失败，不要等 apply。

一句话总结：

> 密码变量要用 `validation` 提前检查长度，用 `sensitive` 避免日志泄露；短密码应该在 plan 阶段失败，而不是等远端系统拒绝。

Lab 37 看起来很小，但它是 Terraform 工程化里非常重要的一类能力：**把输入质量控制前移**。
