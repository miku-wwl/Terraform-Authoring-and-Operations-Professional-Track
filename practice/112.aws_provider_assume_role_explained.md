# Terraform 实操训练 112：AWS Provider Assume Role 概念讲解

## 1. 本节主题

本节对应原始课程文件 `doc/112.txt`，主题是：**在 Terraform AWS Provider 中使用 `assume_role` 块，通过 AWS STS 获取临时凭证，再用临时凭证创建资源**。

一句话理解：

> Terraform 本身不一定直接拿最终账号的长期 AK/SK 操作 AWS，而是先用一组基础凭证去调用 STS，临时“扮演”目标 IAM Role，然后再用这个 Role 的临时权限去创建资源。

这套机制在真实企业环境中非常常见，尤其适合：

- 多账号 AWS Landing Zone；
- CI/CD 部署到不同环境账号；
- 平台团队集中管理部署权限；
- 避免把高权限长期密钥直接放在流水线里；
- 按环境、项目、团队隔离 Terraform 执行权限。

## 3. 为什么需要 Assume Role

在简单实验里，我们经常这样配置 AWS Provider：

```hcl
provider "aws" {
  region     = "ap-southeast-2"
  access_key = "test"
  secret_key = "test"
}
```

这表示 Terraform 直接用这组凭证操作 AWS。

但生产环境里，这种方式有几个明显问题：

1. **长期凭证风险高**  
   如果 AK/SK 泄露，攻击者可以长期使用，直到你手动禁用。

2. **权限不容易隔离**  
   不同环境、不同账号、不同团队如果都复用一组密钥，权限边界会很混乱。

3. **审计不清楚**  
   CloudTrail 里很难清晰区分到底是哪条流水线、哪个角色、哪个环境在执行操作。

4. **多账号部署不方便**  
   企业通常会有 `dev`、`test`、`prod`、`security`、`network` 等多个 AWS 账号。Terraform 如果要部署到目标账号，更合理的方式是 Assume Role。

所以生产中更常见的思路是：

```text
Terraform 基础身份
        │
        ▼
调用 AWS STS AssumeRole
        │
        ▼
获得目标 Role 的临时凭证
        │
        ▼
用临时凭证创建 / 修改 / 删除 AWS 资源
```

---

## 4. `assume_role` 在 Terraform AWS Provider 中的位置

典型写法如下：

```hcl
provider "aws" {
  region = "ap-southeast-2"

  assume_role {
    role_arn     = "arn:aws:iam::123456789012:role/TerraformExecutionRole"
    session_name = "terraform-lab-112"
  }
}
```

这里需要注意：

- `provider "aws"` 是 Terraform 调用 AWS 的入口；
- `assume_role` 是 AWS Provider 的配置块；
- `role_arn` 是 Terraform 要扮演的目标 IAM Role；
- `session_name` 是本次临时会话的名称，通常会出现在审计日志里。

也就是说，Terraform 并不是“变成了某个用户”，而是通过 STS 临时获得了某个 IAM Role 的权限。

---

## 5. STS 是什么

STS 全称是 **Security Token Service**。

你可以把它理解成 AWS 的“临时通行证签发处”。

当 Terraform 调用 `AssumeRole` 时，STS 会返回一组三件套：

```text
临时 Access Key ID
临时 Secret Access Key
临时 Session Token
```

这组三件套不是永久凭证，通常有有效期。

Terraform 后续创建 S3、VPC、IAM、Lambda、EKS 等资源时，实际使用的是这组临时凭证。

这就是 Assume Role 最大的价值：

> 不把长期高权限密钥直接交给 Terraform，而是通过受控的 Role 和临时凭证完成部署。

---

## 6. 真实 AWS 中还需要什么

在真实 AWS 中，只写 Terraform Provider 的 `assume_role` 不够。

目标账号中的 IAM Role 还必须配置 **Trust Policy**，允许 Terraform 当前使用的基础身份来扮演它。

例如目标账号里有一个 Role：

```text
arn:aws:iam::222222222222:role/TerraformExecutionRole
```

它的信任策略大概会表达这个意思：

```json
{
  "Effect": "Allow",
  "Principal": {
    "AWS": "arn:aws:iam::111111111111:role/CICDBaseRole"
  },
  "Action": "sts:AssumeRole"
}
```

含义是：

> 账号 `111111111111` 里的 `CICDBaseRole` 可以扮演账号 `222222222222` 里的 `TerraformExecutionRole`。

如果 Trust Policy 没写对，Terraform 会报类似错误：

```text
AccessDenied: User is not authorized to perform: sts:AssumeRole
```

这个错误不是 Terraform 语法问题，而是 AWS IAM 信任关系没打通。

## 7. 权限模型：谁允许谁做什么

Assume Role 里有两层权限，考试和生产排障都很容易混：

### 7.1 调用方权限

Terraform 当前使用的基础身份，必须有权限调用：

```text
sts:AssumeRole
```

也就是：

> 我有没有资格去申请扮演那个 Role？

### 7.2 目标 Role 的信任策略

目标 Role 的 Trust Policy 必须允许这个调用方来扮演。

也就是：

> 目标 Role 是否信任我？

### 7.3 目标 Role 的资源权限

扮演成功后，Terraform 能做什么，取决于目标 Role 自己绑定的 IAM Policy。

也就是：

> 我扮演成功之后，能不能创建 S3、VPC、EKS、IAM 等资源？

所以完整链路是：

```text
调用方有 sts:AssumeRole 权限
        +
目标 Role Trust Policy 信任调用方
        +
目标 Role 自身有资源操作权限
        =
Terraform 才能真正部署资源
```

少任何一个环节，都会失败。

## 8. LocalStack 实验能验证什么

本节实验使用 LocalStack，不访问真实 AWS。

它可以帮助你验证：

1. AWS Provider 的 `assume_role` 块写法是否正确；
2. Terraform 是否会尝试调用 STS；
3. LocalStack endpoint 是否配置正确；
4. Terraform init / plan / apply / destroy 流程是否能跑通；
5. 实验代码是否具备可重复执行能力。

这对于学习 Terraform Provider 行为已经很有价值。

但它不能完整替代真实 AWS。

---

## 9. LocalStack 不能完全验证什么

LocalStack 模拟的是 AWS API 行为，不等于真实 IAM 安全边界。

真实 AWS 中还必须额外验证：

- 跨账号 Trust Policy 是否正确；
- `sts:AssumeRole` 权限是否配置正确；
- 目标 Role 的 IAM Policy 是否足够但不过度；
- 是否需要 `external_id`；
- 是否受 AWS Organizations SCP 限制；
- CloudTrail 审计记录是否符合预期；
- CI/CD 平台的 OIDC 联邦身份是否配置正确。

所以本节的结论应该这样理解：

> LocalStack 验证 Terraform 配置路径，真实 AWS 验证 IAM 信任边界和安全治理。

这句话很重要，别把本地模拟当成生产安全验证。

## 11. Provider endpoint 在 LocalStack 中的意义

本节提到所有 AWS API 都通过：

```text
http://localhost:4566
```

这是 LocalStack 的统一入口。

Terraform AWS Provider 默认会访问真实 AWS endpoint。为了让它访问 LocalStack，通常需要配置 endpoint override。

示意写法类似：

```hcl
provider "aws" {
  region                      = "ap-southeast-2"
  access_key                  = "test"
  secret_key                  = "test"
  skip_credentials_validation = true
  skip_metadata_api_check     = true
  skip_requesting_account_id  = true

  endpoints {
    sts = "http://localhost:4566"
    s3  = "http://localhost:4566"
  }

  assume_role {
    role_arn     = "arn:aws:iam::000000000000:role/TerraformExecutionRole"
    session_name = "terraform-lab-112"
  }
}
```

不同实验可能配置的服务 endpoint 不完全一样，取决于本节创建了哪些资源。

注意：

> `test/test` 是 LocalStack 假凭证，不是真实 AWS 密钥。

## 12. 常见错误与排障

### 12.1 `NoCredentialProviders`

常见原因：

- 没有设置 `access_key` / `secret_key`；
- 环境变量没传进去；
- named profile 不存在；
- CI/CD 环境中没有注入凭证。

排查思路：

```sh
env | grep AWS
aws sts get-caller-identity
```

LocalStack 场景下则检查 provider 是否配置了假凭证。

---

### 12.2 `AccessDenied: sts:AssumeRole`

真实 AWS 中最常见。

可能原因：

- 调用方没有 `sts:AssumeRole` 权限；
- 目标 Role 的 Trust Policy 没有信任调用方；
- Role ARN 写错；
- 需要 External ID 但没传；
- 被 SCP 或 permission boundary 限制。

不要一看到 AccessDenied 就改 Terraform 资源代码。先查 IAM 链路。

---

### 12.3 `InvalidClientTokenId`

可能原因：

- 真实 AWS 凭证无效；
- 环境变量里混入了错误 AK/SK；
- LocalStack endpoint 没配好，Terraform 误打到真实 AWS。

如果你本来只想跑 LocalStack，却看到真实 AWS 的凭证错误，十有八九是 endpoint override 没写全。

---

### 12.4 `role_arn` 写错

Role ARN 格式通常是：

```text
arn:aws:iam::<account-id>:role/<role-name>
```

例如：

```text
arn:aws:iam::123456789012:role/TerraformExecutionRole
```

常见错误：

- account id 少位数；
- role name 写错；
- 把 user ARN 当成 role ARN；
- 跨账号时写成了源账号的 Role，而不是目标账号的 Role。

## 13. 生产环境中的推荐实践

### 13.1 每个环境单独 Role

不要 dev、test、prod 共用同一个 Terraform Role。

更推荐：

```text
TerraformExecutionRole-Dev
TerraformExecutionRole-Test
TerraformExecutionRole-Prod
```

这样权限、审计和事故边界更清楚。

---

### 13.2 CI/CD 使用 OIDC，再 Assume Role

现在更现代的做法是：

```text
GitHub Actions / GitLab CI / Azure DevOps
        │
        ▼
OIDC 联邦身份
        │
        ▼
AssumeRoleWithWebIdentity
        │
        ▼
AWS 临时凭证
        │
        ▼
Terraform plan/apply
```

这样可以避免在 CI/CD 平台里保存长期 AWS 密钥。

这是平台工程和 SRE 项目里很值得展示的实践。

---

### 13.3 plan 和 apply 可以用不同权限

成熟团队里，`plan` 和 `apply` 不一定使用同一个 Role。

例如：

```text
TerraformPlanRole   -> 只读 + 生成 plan
TerraformApplyRole  -> 允许变更资源
```

这可以配合审批流，降低误操作风险。

---

### 13.4 Role 权限要最小化

不要为了省事直接给 Terraform Role：

```text
AdministratorAccess
```

学习阶段可以这么玩，但生产里应该按系统边界逐步收敛权限。

比如网络 workspace 使用网络相关权限，应用 workspace 使用应用相关权限，安全账号的权限单独管理。

## 14. 考试容易考的点

### 14.1 `assume_role` 的作用是什么？

答案：让 AWS Provider 调用 STS，获取目标 IAM Role 的临时凭证，再用该临时凭证管理资源。

---

### 14.2 Assume Role 是否等于切换 AWS 账号？

不完全等于。

它可以用于跨账号，也可以用于同账号内切换权限边界。

核心不是“换账号”，而是“临时扮演一个 Role”。

---

### 14.3 Assume Role 失败一定是 Terraform 语法错吗？

不是。

真实 AWS 中大概率是 IAM 权限或 Trust Policy 问题。

---

### 14.4 LocalStack 验证通过是否代表真实 AWS 一定通过？

不代表。

LocalStack 只能验证 Provider 配置和 API 调用路径，真实 AWS 的 IAM 信任策略必须真实验证。

---

## 15. 场景理解

### 场景 1：平台团队管理多个业务账号

平台团队有一个 CI/CD 账号，业务系统分别在 dev、test、prod 账号。

Terraform 的执行链路可以设计为：

```text
CI/CD Account
  └── CICDBaseRole
        ├── assume dev account TerraformExecutionRole
        ├── assume test account TerraformExecutionRole
        └── assume prod account TerraformExecutionRole
```

好处：

- 部署入口统一；
- 每个账号权限隔离；
- CloudTrail 可审计；
- prod 可以增加审批和更严格权限。

---

### 场景 2：GitHub Actions 部署 AWS

GitHub Actions 不保存长期 AWS 密钥，而是通过 OIDC 获取临时身份，然后 Assume Role：

```text
GitHub Actions OIDC
        │
        ▼
AWS IAM OIDC Provider
        │
        ▼
TerraformExecutionRole
        │
        ▼
Terraform apply
```

这比把 AK/SK 写进 GitHub Secrets 更安全。

---

### 场景 3：本地开发者只允许 plan，不允许 apply

开发者本地可以 Assume 一个只读 Role 生成 plan：

```text
DeveloperReadOnlyPlanRole
```

真正 apply 只能在 CI/CD 中由受控 Role 执行：

```text
CICDTerraformApplyRole
```

这样可以减少“本地一把梭 apply prod”的风险。

---

## 16. 本节复习口诀

> Provider 先拿基础身份，STS 再发临时凭证；  
> Trust Policy 决定谁能扮演，Role Policy 决定扮演后能干啥；  
> LocalStack 能测调用路径，真实 AWS 才能测安全边界。

## 17. 你应该真正掌握什么

学完本节，不是只会写：

```hcl
assume_role {
  role_arn = "..."
}
```

而是要能说清楚：

1. Terraform 为什么要 Assume Role；
2. STS 临时凭证解决了什么安全问题；
3. 调用方权限、Trust Policy、目标 Role 权限三者的区别；
4. LocalStack 模拟和真实 AWS IAM 的边界；
5. CI/CD、跨账号、多环境部署中如何使用 Assume Role；
6. 报 `AccessDenied` 时应该先查 IAM，而不是乱改 Terraform 资源代码。

这就是本节真正有生产价值的地方。
