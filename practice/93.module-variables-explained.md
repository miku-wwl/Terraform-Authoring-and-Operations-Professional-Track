> 原始实验：`doc/93.txt`

## 1. 这一节到底在学什么？

这一节的重点是：**模块不要把配置写死，而是通过变量接收外部输入。**

在 Terraform 里，模块本质上是一段可以复用的基础设施代码。  
如果模块内部把所有值都写死，比如名称、环境、文件内容、标签、路径、开关配置，那么这个模块只能服务一个固定场景，复用价值很低。

模块变量的作用就是把这些“可能变化的值”抽出来，让调用方传进去。

一句话理解：

> 模块变量 = 模块对外暴露的参数入口。

这和写代码里的函数参数很像：

```hcl
module "demo" {
  source = "./modules/demo"

  name        = "app-dev"
  environment = "dev"
}
```

这里的 `name`、`environment` 就是传给模块的输入参数。

---

## 2. 为什么模块变量很重要？

前面几节已经开始讲模块了：

- 模块的价值
- 如何选择模块
- 如何创建本地模块
- 如何用本地路径引用模块
- 如何改进模块硬编码

到了这一节，就进入模块真正可复用的关键点：**变量化**。

没有变量的模块，通常会变成这样：

```hcl
resource "local_file" "example" {
  filename = "${path.module}/output/dev.txt"
  content  = "hello dev"
}
```

问题是：

- 文件名写死了
- 内容写死了
- 环境写死了
- 想复用到 `test`、`prod`，只能复制粘贴改代码
- 复制多了以后，每份代码都会慢慢漂移

加上变量以后，模块可以变成：

```hcl
variable "filename" {
  type = string
}

variable "content" {
  type = string
}

resource "local_file" "example" {
  filename = var.filename
  content  = var.content
}
```

这样调用方就可以决定传什么值：

```hcl
module "demo_file" {
  source = "./modules/file"

  filename = "${path.module}/output/dev.txt"
  content  = "hello dev"
}
```

这就是从“写死配置”变成“参数化模块”。

---

## 3. 模块变量解决的核心问题

### 3.1 避免硬编码

模块内部不应该知道所有环境细节。

例如：

```hcl
resource "local_file" "config" {
  filename = "dev-config.txt"
  content  = "dev"
}
```

这类写法只适合一次性实验，不适合平台工程或团队复用。

更好的方式是：

```hcl
variable "environment" {
  type = string
}

resource "local_file" "config" {
  filename = "${var.environment}-config.txt"
  content  = var.environment
}
```

这样模块可以同时服务：

- dev
- test
- staging
- prod

---

### 3.2 明确模块接口

模块变量不是简单地“把值传进去”。  
它更重要的作用是定义模块的**接口边界**。

比如：

```hcl
variable "environment" {
  description = "Deployment environment name, such as dev, test, or prod."
  type        = string
}
```

这个变量告诉调用方：

- 这个模块需要环境名
- 环境名必须是字符串
- 这个值会影响模块内部资源命名或内容生成

所以变量写得好不好，直接决定模块好不好用。

一个成熟模块，通常会认真设计：

- 哪些值必须暴露成变量
- 哪些值应该内部默认
- 哪些值需要类型约束
- 哪些值需要校验
- 哪些值不应该暴露，避免调用方误用

---

### 3.3 支持多环境复用

同一个模块可以被多个环境调用：

```hcl
module "dev_file" {
  source = "./modules/file"

  filename = "${path.module}/output/dev.txt"
  content  = "dev environment"
}

module "prod_file" {
  source = "./modules/file"

  filename = "${path.module}/output/prod.txt"
  content  = "prod environment"
}
```

这就是模块变量最直接的价值：

> 代码结构复用，环境差异参数化。

在真实项目中，这个思想会扩大到：

- VPC / VNet CIDR
- subnet 列表
- AKS / EKS 节点池配置
- VM size
- region
- tags
- backup 开关
- monitoring 开关
- security baseline 参数

## 4. 根模块变量 vs 子模块变量

Terraform 项目里经常有两层：

```text
practice/labs/93/
├── main.tf
├── variables.tf
├── outputs.tf
└── modules/
    └── example/
        ├── main.tf
        ├── variables.tf
        └── outputs.tf
```

这里要分清楚：

### 根模块变量

根模块就是你执行 `terraform init/plan/apply` 的目录。

比如：

```text
practice/labs/93
```

如果这个目录里有：

```hcl
variable "environment" {
  type = string
}
```

那它是根模块变量。

根模块变量可以通过这些方式赋值：

```sh
terraform plan -var="environment=dev"
```

或者：

```hcl
# terraform.tfvars
environment = "dev"
```

---

### 子模块变量

子模块是被 `module` block 调用的目录。

例如：

```hcl
module "example" {
  source = "./modules/example"

  environment = "dev"
}
```

如果 `./modules/example/variables.tf` 里定义了：

```hcl
variable "environment" {
  type = string
}
```

那么调用方必须通过 `module` block 把值传进去。

重点：

> 子模块不能自动读取根模块变量，必须显式传递。

也就是说，下面这种理解是错的：

```hcl
# 根模块定义了 environment
variable "environment" {
  type = string
}

# 子模块里也直接用 var.environment
# 但调用时没传
module "example" {
  source = "./modules/example"
}
```

如果子模块需要 `environment`，就必须写：

```hcl
module "example" {
  source = "./modules/example"

  environment = var.environment
}
```

这个地方很容易出错。  
Terraform 不会“心灵感应式继承变量”。

## 5. 模块变量的常见写法

### 5.1 基础字符串变量

```hcl
variable "name" {
  description = "Base name of the generated resource."
  type        = string
}
```

调用：

```hcl
module "example" {
  source = "./modules/example"

  name = "demo"
}
```

---

### 5.2 带默认值的变量

```hcl
variable "environment" {
  description = "Deployment environment."
  type        = string
  default     = "dev"
}
```

如果调用方不传，默认就是 `dev`。

适合默认值比较安全、比较通用的参数。  
但生产环境里，关键参数不要乱给默认值，否则容易误部署。

例如 `environment = "dev"` 默认值用于实验可以，生产中最好显式传入。

---

### 5.3 数字变量

```hcl
variable "replica_count" {
  description = "Number of replicas."
  type        = number
  default     = 1
}
```

在云资源里常见于：

- 节点数量
- 实例数量
- 保留天数
- 容量大小

---

### 5.4 布尔变量

```hcl
variable "enable_monitoring" {
  description = "Whether monitoring is enabled."
  type        = bool
  default     = true
}
```

常用于开关：

- 是否启用日志
- 是否启用监控
- 是否启用备份
- 是否启用私有网络
- 是否创建额外资源

---

### 5.5 map 变量

```hcl
variable "tags" {
  description = "Common tags."
  type        = map(string)
  default     = {}
}
```

调用：

```hcl
module "example" {
  source = "./modules/example"

  tags = {
    env   = "dev"
    owner = "platform-team"
  }
}
```

这在云上非常常见。  
AWS tags、Azure tags、GCP labels，本质上都经常用 map 表达。

---

### 5.6 object 变量

```hcl
variable "service_config" {
  description = "Service configuration."
  type = object({
    name        = string
    environment = string
    enabled     = bool
  })
}
```

调用：

```hcl
module "example" {
  source = "./modules/example"

  service_config = {
    name        = "api"
    environment = "dev"
    enabled     = true
  }
}
```

当参数之间有强关联时，用 object 会比一堆散变量更清楚。

## 6. 本实验的运行流程

进入实验目录：

```sh
cd /workspace/practice/labs/93
```


```sh
terraform validate
```

```sh

```

```sh
terraform plan -input=false -no-color -out=tfplan
```


```sh
terraform output
```

```sh

```

## 7. 每个命令在验证什么？

### 7.1 `terraform init`

验证模块路径、provider 和工作目录是否能正常初始化。

如果模块 source 写错，比如：

```hcl
source = "./module/example"
```

但真实目录是：

```hcl
source = "./modules/example"
```

那初始化或后续校验就会失败。

---

### 7.2 `terraform fmt -check`

验证代码格式是否符合 Terraform 标准风格。

它不改文件，只检查。  
如果失败，说明代码格式不标准，可以运行：

```sh
terraform fmt -recursive
```

---

### 7.3 `terraform validate`

验证配置是否合法。

常见失败原因：

- 变量定义了但传错类型
- 子模块需要变量，但调用方没传
- output 引用了不存在的资源
- module source 路径错误
- HCL 语法写错

---

### 7.4 `terraform test`

验证实验里的测试断言。

这一节主题是模块变量，所以测试通常会围绕：

- 模块输出是否符合变量输入
- 资源命名是否受变量控制
- 默认值是否生效
- 传入变量后是否生成预期结果

如果 `terraform test` 返回：

```text
1 passed, 0 failed
```

说明当前实验断言通过。

---

### 7.5 `terraform plan`

验证 Terraform 计划要创建、修改或删除什么。

这一步适合观察变量对资源的影响。

例如你修改变量：

```hcl
environment = "test"
```

然后重新 plan，就应该看到相关资源内容或名称发生变化。

---

### 7.6 `terraform apply`

真正执行变更。

本地实验里通常使用轻量 provider 或本地资源，不会产生云费用。  
但在真实云环境里，`apply` 就是会真的创建资源，不能乱跑。

---

### 7.7 `terraform output`

查看模块最终暴露出来的信息。

模块变量负责输入，output 负责输出。  
一个模块好不好用，通常就看这两个边界设计得清不清楚。

---

### 7.8 `terraform destroy`

销毁实验资源，避免残留状态和文件影响下一次实验。

---

## 8. 模块变量和 output 的关系

模块变量是输入：

```hcl
variable "name" {
  type = string
}
```

output 是输出：

```hcl
output "file_name" {
  value = local_file.example.filename
}
```

调用方使用模块时，可以这样读取子模块输出：

```hcl
output "module_file_name" {
  value = module.example.file_name
}
```

所以模块的边界可以理解成：

```text
调用方
  |
  | 输入变量
  v
子模块
  |
  | 输出结果
  v
调用方继续使用
```

成熟 Terraform 模块就是靠这两个接口完成组合：

- variable：我需要你给我什么
- output：我能给你返回什么

## 9. 生产环境中的建模价值

### 9.1 平台团队提供标准模块

平台团队可以写一个标准 AKS 模块：

```hcl
module "aks" {
  source = "git::https://example.com/platform/terraform-aks.git?ref=v1.2.0"

  cluster_name       = "app-dev-aks"
  location           = "australiaeast"
  kubernetes_version = "1.30"
  node_pools         = var.node_pools
  tags               = var.tags
}
```

业务团队只需要填变量，不需要理解底层所有 Azure 资源细节。

这就是平台工程里的“铺路”：

> 平台团队把复杂性封装进模块，把可配置项变成变量，把安全边界写进默认值和校验。

---

### 9.2 减少复制粘贴

没有模块变量时，多环境通常会变成：

```text
dev/main.tf
test/main.tf
prod/main.tf
```

每份都差不多，但又有一点不同。  
时间久了就会出现：

- dev 改了，prod 忘了改
- test 参数不一致
- 安全配置漂移
- tag 不统一
- monitoring 漏开

模块变量的价值就是把公共逻辑收敛到一个模块里，把差异留在输入参数里。

---

### 9.3 支持版本化演进

公共模块通常会配合 Git tag 使用：

```hcl
module "network" {
  source = "git::https://example.com/platform/network.git?ref=v1.0.0"

  environment = "prod"
}
```

升级时从：

```hcl
ref=v1.0.0
```

改到：

```hcl
ref=v1.1.0
```

调用方的变量接口如果保持兼容，升级就会比较平滑。

所以模块变量设计不是小事。  
变量一旦发布，就像 API 一样，后续要考虑兼容性。

## 10. 常见坑

### 坑 1：子模块变量没有传

子模块中：

```hcl
variable "name" {
  type = string
}
```

根模块调用时忘了传：

```hcl
module "example" {
  source = "./modules/example"
}
```

结果会报错，因为 `name` 没有值。

修复：

```hcl
module "example" {
  source = "./modules/example"

  name = "demo"
}
```

---

### 坑 2：变量类型不匹配

变量定义：

```hcl
variable "enabled" {
  type = bool
}
```

错误传法：

```hcl
enabled = "true"
```

更好的传法：

```hcl
enabled = true
```

Terraform 有时会做类型转换，但生产代码不要依赖“看起来能转”。  
类型清楚，后面排障才轻松。

---

### 坑 3：默认值滥用

```hcl
variable "environment" {
  type    = string
  default = "dev"
}
```

实验里这样没问题。  
但生产里如果关键变量默认成 `dev`，可能导致调用方忘记传值，结果资源命名、策略、tag 全都不符合预期。

关键变量建议显式传入，不要乱给默认。

---

### 坑 4：变量太细，模块变成参数怪物

有些模块会暴露几十上百个变量，调用方看着就头大。

例如：

```hcl
variable "enable_a" {}
variable "enable_b" {}
variable "enable_c" {}
variable "name_a" {}
variable "name_b" {}
variable "name_c" {}
```

这种模块虽然“灵活”，但不好用。

更好的做法是按对象聚合：

```hcl
variable "service_config" {
  type = object({
    name    = string
    enabled = bool
    options = map(string)
  })
}
```

模块设计不是变量越多越好。  
变量越多，认知成本越高。

---

### 坑 5：变量校验缺失

如果 environment 只允许：

- dev
- test
- prod

可以加 validation：

```hcl
variable "environment" {
  type = string

  validation {
    condition     = contains(["dev", "test", "prod"], var.environment)
    error_message = "environment must be one of: dev, test, prod."
  }
}
```

这样错误会更早暴露，而不是等到资源命名或策略出问题。

---

## 11. 推荐的模块变量设计习惯

### 11.1 每个变量尽量写 description

不要只写：

```hcl
variable "name" {
  type = string
}
```

更推荐：

```hcl
variable "name" {
  description = "Base name used to generate resource names."
  type        = string
}
```

以后别人看模块时，不用猜。

---

### 11.2 明确类型

不要偷懒：

```hcl
variable "tags" {}
```

更推荐：

```hcl
variable "tags" {
  description = "Common tags applied to resources."
  type        = map(string)
  default     = {}
}
```

---

### 11.3 对关键变量加 validation

例如环境名：

```hcl
variable "environment" {
  description = "Deployment environment."
  type        = string

  validation {
    condition     = contains(["dev", "test", "staging", "prod"], var.environment)
    error_message = "environment must be dev, test, staging, or prod."
  }
}
```

---

### 11.4 默认值只给安全值

适合默认的：

```hcl
enable_monitoring = true
tags              = {}
```

不太适合随便默认的：

```hcl
environment
region
subscription_id
account_id
cluster_name
```

这些值最好由调用方明确传入。

---

### 11.5 输入变量不要泄露实现细节

变量应该描述调用方关心的东西，而不是模块内部怎么实现。

不太好的变量：

```hcl
variable "local_file_resource_name" {
  type = string
}
```

更好的变量：

```hcl
variable "config_name" {
  type = string
}
```

模块内部资源叫什么，不应该强迫调用方知道。

## 12. 和 Terraform Professional 考点的关系

这一节和 Terraform Professional 的关系很直接。

你需要理解：

- module 的输入变量如何定义
- 根模块变量和子模块变量的区别
- `module` block 如何传参
- 默认值如何生效
- 类型约束如何影响 validate/plan
- output 如何把模块结果暴露给调用方
- 模块变量如何帮助复用和抽象
- 为什么公共模块的变量接口要稳定

考试或者面试中，很容易问：

> 为什么要把硬编码改成变量？

好答案不是“因为 Terraform 支持变量”。  
好答案是：

> 因为模块要复用，调用方和模块之间需要明确接口。变量把环境差异、命名差异、容量差异、开关差异从模块实现中抽离出来，让同一套模块可以被多个环境、安全地、可测试地复用。

这个回答就更像工程师，而不是背命令。

## 14. 排障清单

如果实验失败，可以按这个顺序查：

### 14.1 `terraform init` 失败

检查：

- module source 路径是否正确
- 目录是否真实存在
- provider 是否可下载
- `.terraform` 是否损坏

可尝试：

```sh
rm -rf .terraform .terraform.lock.hcl
terraform init -input=false
```

---

### 14.2 `terraform fmt -check` 失败

运行：

```sh
terraform fmt -recursive
```

然后重新检查：

```sh
terraform fmt -check
```

---

### 14.3 `terraform validate` 失败

重点查：

- 子模块需要的变量是否都传了
- 变量类型是否匹配
- output 是否引用了正确名称
- module block 名称是否写错
- 资源地址是否写错

---

### 14.4 `terraform test` 失败

重点查：

- 测试断言引用的 output 是否存在
- 传入变量是否符合测试预期
- 变量默认值是否被覆盖
- run block 是 plan 还是 apply
- 断言条件是否写反

---

### 14.5 `terraform plan/apply` 和预期不一致

重点查：

- 是否改了变量但没重新 plan
- 是否使用了旧的 tfplan
- 是否 output 展示的是旧状态
- 是否有本地生成文件残留
- 是否忘记 destroy 后重新执行

---

## 15. 清理命令

实验结束后执行：

```sh
terraform destroy -auto-approve
rm -f tfplan
rm -rf output logs .terraform .terraform.lock.hcl terraform.tfstate terraform.tfstate.backup
```

如果你只是本地 `local_file` 一类实验，清理的意义主要是避免下一次测试被旧文件或旧状态污染。

---

## 16. 一句话总结

> 模块变量是 Terraform module 的输入接口。  
> 它把环境差异、命名差异和配置差异从模块实现里抽出来，让模块从“一次性脚本”变成“可复用组件”。

这节不要只记住 `variable` 语法。  
真正要记住的是：**模块变量是在设计基础设施代码的 API。**
