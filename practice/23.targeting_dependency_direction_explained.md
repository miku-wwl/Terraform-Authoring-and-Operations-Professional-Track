# Terraform 实操训练 23 讲解版：Targeting 与依赖方向

> 对应原始实验：`Terraform 实操训练 23：Targeting 与依赖方向`  
> 核心主题：`terraform -target`、上游依赖、下游资源、完整收敛  
> 推荐定位：Terraform 自动化 / CI/CD / 故障修复工作流中的高级注意事项

---

## 1. 这一课到底在学什么？

这一课不是单纯让你记住 `-target` 怎么写，而是让你理解一个很容易踩坑的点：

> `-target` 会带上目标资源需要的上游依赖，但不会自动带上依赖目标资源的下游资源。

换句话说：

- 你 target 一个资源时，Terraform 会确保这个资源自己能被创建出来；
- 如果它依赖别的资源，Terraform 可能会顺手创建它需要的上游；
- 但是，其他依赖它的资源，Terraform 不会因为你 target 了它就自动创建。

这就是本实验的重点。

---

## 2. 先理解 Terraform 的依赖方向

假设资源关系是这样：

```text
random_integer.build_number
        ↓
local_file.manifest
```

意思是：

- `random_integer.build_number` 负责生成一个随机 build number；
- `local_file.manifest` 的文件名或内容引用了这个随机数；
- 所以 `local_file.manifest` 依赖 `random_integer.build_number`。

在 Terraform 里，依赖方向可以理解为：

```text
上游资源 → 下游资源
```

这里：

```text
random_integer.build_number 是上游
local_file.manifest 是下游
```

下游需要上游的值才能计算出来。

---

## 3. `-target` 的真实行为

如果你执行：

```sh
terraform plan -target=random_integer.build_number -out=tfplan
terraform apply -auto-approve tfplan
```

Terraform 的意思是：

> 这次只关注 `random_integer.build_number` 这个目标资源，以及它为了创建自己所必须依赖的资源。

但 `local_file.manifest` 是依赖随机数的下游资源，不是随机数创建时必须依赖的资源。

所以结果是：

- `random_integer.build_number` 会被创建；
- `local_file.manifest` 不会自动被创建；
- manifest 文件不会出现；
- Terraform 会提示这是 targeted plan / targeted apply，结果可能不完整。

这不是 bug，而是 Terraform 的设计。

## 4. 本实验的关键流程

实验目录：

```sh
cd /workspace/practice/labs/23
```

先做基础检查：

```sh
terraform init -input=false
terraform fmt -check
terraform validate
terraform test
```

然后执行 targeted workflow：

```sh
terraform plan -input=false -no-color -target=random_integer.build_number -out=tfplan
terraform apply -auto-approve tfplan
```

这一步只让随机数资源进入 state。

接着必须执行完整 plan / apply：

```sh
terraform plan -input=false -no-color -out=tfplan
terraform apply -auto-approve tfplan
```

这一步才会让整个配置收敛，也就是生成依赖随机数的 manifest 文件。

最后查看输出并清理：

```sh
terraform output
terraform destroy -auto-approve
```

## 5. 为什么 targeted apply 后还要跑完整 plan？

这是生产环境里最重要的一点。

`-target` 通常用于救急，例如：

- 某个关键资源坏了，要先恢复；
- 某个 provider bug 导致完整 apply 过不去；
- 某个底层依赖要先单独创建；
- 灾难恢复时要分阶段恢复资源。

但是 targeted apply 不是完整部署。

执行 targeted apply 后，Terraform 的整体配置可能还没有达到最终期望状态。

所以生产里常见的安全流程应该是：

```text
targeted plan
↓
targeted apply
↓
full plan
↓
full apply 或确认 no changes
```

也就是说，`-target` 可以救火，但救完火以后还得检查整栋楼是不是还在冒烟。

---

## 6. 这节课最容易误解的地方

### 误解 1：target 一个上游资源，下游也会自动创建

不会。

Terraform 只会保证目标资源以及目标资源需要的依赖被处理，不会反向处理依赖它的下游资源。

### 误解 2：targeted apply 成功，就说明整个环境成功

不一定。

targeted apply 成功，只代表这次被 target 的那部分成功了，不代表整个配置已经收敛。

### 误解 3：`-target` 是一种正常部署策略

不建议。

长期依赖 `-target` 会让团队失去对完整依赖图的信任。最后会变成：

```text
这次 target A
下次 target B
再下次 target C
然后没人知道完整 apply 会发生什么
```

这在真实生产里非常危险。

---

## 7. 面向考试和面试的理解方式

可以这样记：

> Terraform 默认按完整依赖图工作；`-target` 是人为缩小依赖图。  
> 它会保留目标资源需要的上游依赖，但不会自动包含目标资源的下游消费者。  
> 因此 targeted apply 后，应该再跑完整 plan，确认整体配置收敛。

如果面试官问：

> `terraform -target` 什么时候用？

可以答：

> 我会把它当成异常处理工具，而不是常规发布工具。比如故障恢复、分阶段修复、provider 问题绕过时可以临时使用。但使用后必须重新执行 full plan，确认没有遗漏变更。

这个回答比单纯说“可以只部署某个资源”要成熟很多。

## 9. 常见排障

### 9.1 target 后 manifest 没生成

这是预期结果。

因为你 target 的是：

```sh
random_integer.build_number
```

manifest 是它的下游，不会自动包含。

解决方式：

```sh
terraform plan -input=false -no-color -out=tfplan
terraform apply -auto-approve tfplan
```

跑完整 plan/apply。

---

### 9.2 Terraform 出现 targeting 警告

这是正常现象。

Terraform 在提醒你：

> 这不是完整计划，结果可能不代表整个配置的最终状态。

不要忽略这个警告。它不是报错，但它是一个很重要的风险提示。

---

### 9.3 完整 plan 仍然有变更

说明 targeted apply 后环境还没有收敛。

继续审查完整 plan，如果变更符合预期，再执行：

```sh
terraform apply -auto-approve tfplan
```

---

### 9.4 不确定资源之间的依赖关系

可以从两个角度看：

第一，看 HCL 引用：

```hcl
filename = "manifest-${random_integer.build_number.result}.txt"
```

这种引用会形成隐式依赖。

第二，用 Terraform 图辅助分析：

```sh
terraform graph
```

不过 `terraform graph` 输出比较底层，适合进阶排查。

## 11. 本课总结

这一课的核心不是命令，而是依赖图思维。

你需要记住三句话：

1. Terraform 默认会计算完整依赖图。
2. `-target` 会人为缩小本次 plan/apply 的范围。
3. target 会带上目标资源需要的上游，但不会自动带上依赖目标资源的下游。

生产环境中，`-target` 是急救箱，不是日常饭盒。

偶尔用它救命没问题，天天靠它吃饭，说明模块边界、依赖设计或发布流程已经出问题了。
