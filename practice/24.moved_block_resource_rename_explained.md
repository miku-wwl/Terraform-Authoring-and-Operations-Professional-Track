# Terraform 实操训练 24：`moved` block 与资源重命名讲解版

## 1. 这节 Lab 到底在练什么

这一节练的是 Terraform 里一个非常重要、但很多初学者容易忽略的能力：**资源地址重命名时，如何避免 Terraform 把它当成“删除旧资源 + 创建新资源”。**

Terraform 管理资源时，不只是看资源本身，还非常依赖 **resource address**。例如：

```hcl
terraform_data.database_firewall
```

这个地址就是 Terraform state 里识别资源身份的关键。如果你后来把代码里的名字改成：

```hcl
terraform_data.payment_database_firewall
```

Terraform 默认可能会觉得：

> 哦，旧的 `terraform_data.database_firewall` 不见了，要删除。  
> 新的 `terraform_data.payment_database_firewall` 出现了，要创建。

这在本地练习里只是多创建一个模拟资源，但在真实生产环境里，可能就是数据库、防火墙规则、存储账号、IAM 角色、网络资源被误删重建。这个坑很凶。

所以这节 Lab 的核心就是：用 `moved` block 明确告诉 Terraform：

> 这不是新资源，这是旧资源换了一个地址。

---

## 2. 核心概念：Terraform 资源地址是什么

Terraform 的资源地址通常长这样：

```hcl
<resource_type>.<resource_name>
```

例如：

```hcl
terraform_data.database_firewall
```

如果资源在 module 里，地址还会更长：

```hcl
module.network.azurerm_network_security_group.main
```

这个地址会被 Terraform 写进 state。也就是说，Terraform 不是只靠云平台上的真实资源名称来识别资源，它还会看 state 里的 Terraform 地址。

所以你在代码里重命名资源块，哪怕真实业务含义没有变，Terraform 也可能认为这是两个不同资源。

---

## 3. 为什么直接重命名有风险

假设原来代码是：

```hcl
resource "terraform_data" "database_firewall" {
  input = {
    name = "database-firewall"
  }
}
```

后来你觉得名字太泛了，想改成更清楚的业务命名：

```hcl
resource "terraform_data" "payment_database_firewall" {
  input = {
    name = "database-firewall"
  }
}
```

从人的视角看：

> 我只是把 Terraform 代码里的资源名改得更清楚。

但从 Terraform 的视角看：

> `terraform_data.database_firewall` 消失了。  
> `terraform_data.payment_database_firewall` 新增了。  
> 那就 destroy 一个，再 create 一个。

如果这是真实云资源，后果可能很麻烦：

- 安全组被替换；
- 数据库防火墙规则被重建；
- 存储资源出现短暂不可用；
- 某些不可重建资源触发业务事故；
- 审批里看起来只是“rename”，实际 plan 却出现 destroy/create。

这就是为什么 Terraform 重构不能只看代码，还要看 state 地址迁移。

---

## 4. `moved` block 是干什么的

`moved` block 的作用是声明资源地址迁移。

典型格式：

```hcl
moved {
  from = terraform_data.database_firewall
  to   = terraform_data.payment_database_firewall
}
```

它的意思是：

> state 里原来叫 `terraform_data.database_firewall` 的资源，现在应该被认为是 `terraform_data.payment_database_firewall`。

这样 Terraform 在 plan 时就不会把它当成删除和新建，而是理解为一次地址迁移。

一句话记忆：

> `moved` block 是 Terraform 的“户口迁移证明”。资源没换人，只是地址改了。

---

## 5. 本 Lab 的场景设计

本实验模拟平台团队做资源命名重构。

旧资源地址：

```hcl
terraform_data.database_firewall
```

新资源地址：

```hcl
terraform_data.payment_database_firewall
```

业务含义是：把原来比较泛的 `database_firewall` 改成更具体的 `payment_database_firewall`。

这类重命名在真实项目里很常见，例如：

- `storage` 改成 `audit_log_storage`；
- `vm` 改成 `bastion_vm`；
- `security_group` 改成 `web_security_group`；
- 资源从 root module 移到 child module；
- 单体 Terraform 目录拆成多个 module。

如果没有 `moved` block，这些操作都可能让 Terraform 误以为资源被删除又新建。

## 8. 每条命令在干什么

### `terraform init -input=false`

初始化当前目录，下载 provider，准备 `.terraform` 工作目录。

`-input=false` 表示不要在命令行里交互提问，这个参数很适合 CI/CD。

---

### `terraform fmt -check`

检查 Terraform 文件格式是否符合标准。

它不会自动修改文件，只会告诉你格式是否正确。

CI 里通常使用这个命令阻止格式混乱的代码进入主分支。

---

### `terraform validate`

检查 HCL 配置是否合法，例如：

- 语法是否正确；
- resource block 是否完整；
- provider schema 是否匹配；
- 引用是否基本有效。

它不会真正创建资源。

---

### `terraform test`

运行 `.tftest.hcl` 测试文件。

本 Lab 里，测试用于确认 moved block 的迁移说明被正确记录，例如旧地址、新地址是否符合预期。

---

### `terraform plan -out=tfplan`

生成执行计划，并保存为二进制 plan 文件。

这一步适合生产审批流程：

```sh
terraform plan -input=false -no-color -out=tfplan
```

审批时看这个 plan，执行时也 apply 这个 plan，可以保证审批内容和执行内容一致。

---

### `terraform apply -auto-approve tfplan`

执行之前保存的 plan。

重点是：

```sh
terraform apply tfplan
```

表示执行已经生成好的计划，而不是重新现场计算一个新的计划。

---

### `terraform output`

查看实验输出。

本 Lab 里输出会记录：

- 旧资源地址；
- 新资源地址；
- 新资源名称。

---

### `terraform destroy -auto-approve`

清理本实验创建的资源。

虽然本实验主要是本地模拟，但保持清理习惯很重要。

## 9. 如何判断 `moved` block 是否生效

在真实 Terraform 重构中，你最应该看的是 plan。

如果只是资源地址迁移，理想情况下你应该看到类似语义：

```text
has moved to
```

而不是明显的：

```text
-/+ destroy and then create replacement
```

或者：

```text
- destroy
+ create
```

简单判断：

| Plan 现象 | 意义 |
|---|---|
| 显示 moved | 地址迁移成功 |
| 显示 destroy/create | Terraform 仍认为是删除旧资源、创建新资源 |
| 显示 replace | 可能不只是地址变了，也改了不可更新属性 |
| 显示 no changes | 资源已经收敛 |

生产里不要看到 destroy/create 还硬上。那不是勇敢，那是制造事故。

## 10. `moved` block 适合哪些场景

### 场景 1：资源重命名

例如：

```hcl
from = azurerm_storage_account.main
to   = azurerm_storage_account.audit_logs
```

适合把模糊命名改成更有业务含义的命名。

---

### 场景 2：资源移动到 module

例如：

```hcl
from = azurerm_resource_group.main
to   = module.foundation.azurerm_resource_group.main
```

适合从 root module 重构到 child module。

---

### 场景 3：module 内部资源重构

例如：

```hcl
from = module.network.azurerm_subnet.old
to   = module.network.azurerm_subnet.private
```

适合模块内部整理资源命名。

---

### 场景 4：拆分长期维护的 Terraform 代码

当一个目录越来越大，你想拆模块时，`moved` block 可以帮助你把 state 地址迁移得更安全。

不过这类操作建议分阶段做，不要一次性大重构。

---

## 11. `moved` block 不是什么

`moved` block 不是万能护身符。

它只解决：

> Terraform state 里的地址迁移问题。

它不解决：

- 真实云资源属性变更；
- 不可更新字段导致的 replace；
- provider 行为变化；
- state 已经损坏；
- 资源被云平台手工删除；
- 从一个 Terraform backend 迁移到另一个 backend 的完整流程。

如果你同时改了资源地址和关键属性，即使写了 `moved` block，Terraform 仍然可能 replacement。

比如某些云资源的 name、location、SKU、VPC 绑定是不可原地修改的。地址迁移成功，不代表属性变更安全。

## 12. 常见错误

### 错误 1：`from` 地址写错

```hcl
moved {
  from = terraform_data.database_firewal
  to   = terraform_data.payment_database_firewall
}
```

少写一个字母，Terraform 就找不到旧地址。

解决方式：

```sh
terraform state list
```

用 state 里的真实地址核对。

---

### 错误 2：忘记 module 路径

如果资源在 module 里，地址必须写完整。

错误示例：

```hcl
from = azurerm_subnet.old
```

正确示例可能是：

```hcl
from = module.network.azurerm_subnet.old
```

Terraform 地址不是你脑子里的简称，必须是完整地址。

---

### 错误 3：以为 moved 会改云上的资源名

不会。

`moved` block 改的是 Terraform state 里的地址映射，不是云平台资源上的真实名称。

如果你要改 Azure 资源名、AWS 资源名或 Kubernetes 对象名，那是另一个问题，可能涉及重建。

---

### 错误 4：看到 destroy/create 还继续 apply

这很危险。

一旦 plan 出现 destroy/create，你需要先判断：

- 是不是 moved block 没写对？
- 是不是同时改了不可更新属性？
- 是不是 state 里本来就没有旧地址？
- 是不是资源已经被手工改坏了？

不要把 Terraform 当成“反正它知道我要干嘛”。它不知道。它只知道配置、state 和 provider 返回的信息。

---

## 13. 生产环境推荐做法

生产环境重命名资源时，可以按这个节奏来：

1. 先查看当前 state：

```sh
terraform state list
```

2. 确认旧地址存在。

3. 修改资源块名称。

4. 添加 `moved` block。

5. 执行 plan：

```sh
terraform plan
```

6. 仔细确认 plan 是 moved，而不是 destroy/create。

7. 让同事 review。

8. 保存 plan 并走审批。

9. apply 后再跑一次完整 plan，确认 no changes。

这个流程比“直接改名然后 apply”慢一点，但它能救命。

---

## 14. 和 `terraform state mv` 的区别

Terraform 也有一个命令叫：

```sh
terraform state mv
```

它也能移动 state 地址。

但是两者风格不同：

| 方式 | 特点 |
|---|---|
| `moved` block | 声明式，写在代码里，可 review，可版本控制 |
| `terraform state mv` | 命令式，直接改 state，操作快但审计弱 |

现代 Terraform 项目里，能用 `moved` block 就优先用 `moved` block。

因为它更适合团队协作和代码审查。

`state mv` 更像手术刀，适合少数特殊场景；`moved` block 更像标准变更单，适合常规重构。

## 16. 一句话总结

`moved` block 的价值不是“语法多酷”，而是它能在 Terraform 重构时保护 state，让资源改名、迁移 module、整理结构这些动作不至于被误判成删除重建。

真正上生产时，记住这句话：

> 改 Terraform 资源地址之前，先想 state；看到 destroy/create 之前，别急着 apply。
