## 1. 这节课到底在学什么？

这一节的核心不是单纯记住 `terraform test` 的语法，而是理解：

**Terraform test 文件里的“根级属性”到底适合放什么，以及它们和每个 `run` 测试用例之间是什么关系。**

你可以把 `.tftest.hcl` 文件理解成 Terraform 的测试剧本：

- 根级配置：整份测试文件共享的默认设置；
- `run` 块：一个个具体测试场景；
- `assert`：测试中的判断条件；
- `variables`：测试时传入模块的输入参数；
- `command`：决定这个测试跑 `plan` 还是 `apply`。

所以这一节学的是：

> 哪些配置应该放在测试文件最外层，哪些配置应该放在某个具体 `run` 块里面。

这就是所谓的 **Terraform test 根级属性**。

---

## 2. 为什么 Terraform test 需要“根级属性”？

假设你有多个测试场景：

1. 默认配置能不能通过；
2. 开启某个功能后能不能通过；
3. 输入非法值时会不会失败；
4. 输出值是否符合预期；
5. 命名规则是否满足平台规范。

这些测试可能都需要一组相同变量，例如：

- `environment = "dev"`
- `project = "platform"`
- `region = "local"`
- `tags = { owner = "sre" }`

如果每个 `run` 块里都重复写一遍，就会出现两个问题：

第一，测试文件会越来越啰嗦。

第二，将来默认变量改了，你要改很多地方，很容易漏。

所以 Terraform test 支持把部分共享配置放在根级位置，让多个测试用例复用。

这和正常代码里的“默认配置”很像。

---

## 3. 根级属性和 run 级属性的区别

可以这样理解：

| 位置 | 作用 | 适合放什么 |
|---|---|---|
| 根级 | 整个测试文件共享 | 通用变量、通用 provider 配置、默认测试上下文 |
| `run` 块内部 | 某个测试用例专属 | 特定输入、特定断言、特定测试命令 |

一句话记忆：

> 根级属性管“默认环境”，`run` 块管“具体场景”。

---

## 4. 一个典型结构长什么样？

示意结构如下：

```hcl
variables {
  environment = "dev"
  project     = "platform"
}

run "default_plan_should_pass" {
  command = plan

  assert {
    condition     = output.environment == "dev"
    error_message = "environment output should be dev"
  }
}

run "custom_environment_should_override_default" {
  command = plan

  variables {
    environment = "test"
  }

  assert {
    condition     = output.environment == "test"
    error_message = "run-level variable should override root-level variable"
  }
}
```

这里最关键的点是：

- 根级 `variables` 提供默认值；
- 第二个 `run` 里又写了 `variables`；
- `run` 内部变量可以覆盖根级变量；
- `assert` 用来验证最终输出是否符合预期。

这就是根级属性最常见的用途：**减少重复，同时允许局部覆盖。**

---

## 5. 这在真实工程里有什么价值？

Terraform test 的根级属性看起来只是语法细节，但在团队工程里非常实用。

### 5.1 统一测试上下文

平台团队通常会有统一约定：

- 环境名怎么写；
- 资源命名规则怎么写；
- 标签必须包含哪些字段；
- region/location 默认是什么；
- 模块默认参数是什么。

这些通用设置适合放在根级位置。

这样每个测试用例不用反复声明同样的东西。

### 5.2 降低测试维护成本

如果默认环境从 `dev` 改成 `test`，只需要改根级变量，而不是挨个改每个 `run` 块。

这对后期维护很关键。

Terraform test 文件一多，重复配置会变成灾难。

### 5.3 让测试意图更清晰

如果某个 `run` 块只写了它关心的差异，那么读测试的人能很快看出来：

> 这个测试场景到底在测什么？

这比每个测试都复制一大坨变量清晰很多。

---

## 6. 但是根级属性也不能乱用

根级属性不是越多越好。

如果你把所有东西都塞到根级，测试反而会变得难读。

尤其是某个断言依赖的关键输入，如果藏在根级很远的位置，读测试的人会很痛苦。

建议原则：

| 内容类型 | 推荐位置 |
|---|---|
| 所有测试都一样的默认值 | 根级 |
| 某个场景才需要的变量 | `run` 内部 |
| 会影响断言结果的关键变量 | 优先放在对应 `run` 内部 |
| provider 的通用配置 | 根级 |
| 边界测试输入 | `run` 内部 |

一句话：

> 根级放共性，run 里放个性。

## 7. 和 assert 的关系

上一节 Lab 82 讲的是 Terraform test assertion，这一节 Lab 83 继续往前走，关注 assertion 所处的上下文。

`assert` 不是孤立存在的。

它判断的是当前测试运行时的状态、变量、输出和资源结果。

如果根级变量提供了默认输入，那么 `assert` 判断的就是这些默认输入经过模块处理后的结果。

例如：

```hcl
variables {
  environment = "dev"
}

run "check_environment_output" {
  command = plan

  assert {
    condition     = output.environment == "dev"
    error_message = "environment should be dev"
  }
}
```

这里的断言依赖根级变量。

如果你把根级变量改成：

```hcl
variables {
  environment = "prod"
}
```

那断言也必须同步调整，否则测试就会失败。

所以写 Terraform test 时要注意：

> 根级变量不是背景板，它会直接影响每个测试用例的断言结果。

## 8. plan 测试和 apply 测试的区别

运行手册里包含：

```sh
terraform test
terraform plan -input=false -no-color -out=tfplan
terraform apply -auto-approve tfplan
terraform output
terraform destroy -auto-approve
```

这里要区分两个层次。

### 8.1 `terraform test`

`terraform test` 会读取 `tests/` 目录下的测试文件，并执行其中的 `run` 块。

如果测试里使用：

```hcl
command = plan
```

那它只做计划阶段验证，不会真正创建资源。

如果测试里使用：

```hcl
command = apply
```

那它可能会创建真实资源，并在测试结束后尝试清理。

在本 Lab 这种 Docker 本地训练环境里，通常使用的是轻量资源或本地逻辑，所以风险较低。

### 8.2 `terraform plan/apply/output/destroy`

这几步是正常 Terraform 生命周期：

1. `plan` 生成执行计划；
2. `apply` 按计划创建或修改资源；
3. `output` 查看输出；
4. `destroy` 清理资源。

所以你要把它们理解成两个闭环：

- `terraform test`：测试闭环；
- `plan/apply/output/destroy`：资源生命周期闭环。

Lab 的目标是让这两个闭环都跑通。

## 11. 常见错误与排查方式

### 11.1 `terraform fmt -check` 失败

说明 HCL 格式不符合 Terraform 标准格式。

处理方式：

```sh
terraform fmt -recursive
```

然后再运行：

```sh
terraform fmt -check
```

### 11.2 `terraform test` 找不到变量

常见原因：

- 根级 `variables` 没写；
- 变量名写错；
- 模块里没有声明对应 `variable`；
- `run` 块里覆盖变量时拼写错误。

排查顺序：

1. 看 `variables.tf` 或 `main.tf` 里的变量声明；
2. 看 `.tftest.hcl` 根级变量；
3. 看 `run` 块内部变量；
4. 看断言里引用的名字是否一致。

### 11.3 assert 条件失败

断言失败不一定是坏事。

它说明测试发现了你的假设和实际输出不一致。

优先检查：

- output 名称；
- output 值；
- 根级变量默认值；
- run 内部变量覆盖；
- `condition` 里的比较逻辑。

### 11.4 plan 通过但 test 失败

这通常说明：

- Terraform 配置本身能跑；
- 但测试断言不符合实际结果。

也就是说，代码不是语法错，而是“行为不符合测试期望”。

这是 Terraform test 最有价值的地方。

它不只是帮你查语法，而是帮你查工程约定。

## 12. 面试和工程表达方式

如果面试里聊到这节内容，可以这样说：

> I use Terraform test files to validate module behavior before applying infrastructure changes. Root-level attributes in `.tftest.hcl` are useful for defining shared defaults, such as common variables or provider configuration, while each `run` block represents a specific test scenario. This helps reduce duplication and keeps test cases focused on the behavior being validated.

中文表达：

> 我会用 Terraform test 来验证模块行为。根级属性适合放共享默认值，比如通用变量或 provider 配置；每个 `run` 块负责一个具体测试场景。这样可以减少重复配置，也能让测试用例更聚焦。

这句话比单纯说“我会 terraform test”更像工程师。
