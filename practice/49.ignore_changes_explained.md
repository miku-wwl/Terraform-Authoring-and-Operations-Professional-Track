# Terraform 实操训练 49 讲解：`ignore_changes`

## 1. 这一节到底在学什么？

Lab 49 学的是 Terraform `lifecycle` 里的另一个常用参数：

```hcl
ignore_changes = [...]
```

它的作用是：

> 让 Terraform 忽略某些指定属性的变化，即使远端或本地真实对象和 `.tf` 配置不一致，也不要尝试把它改回去。

这节的关键词是：**漂移**。

Terraform 默认会追求配置、state、真实资源三者一致。  
但是有些字段确实会被外部系统改写，例如标签、注释、运行时状态、自动生成字段。  
这时如果 Terraform 每次都想改回去，反而会和外部系统“打架”。

`ignore_changes` 就是用来处理这种边界的。

一句话理解：

> `ignore_changes` 是告诉 Terraform：这个字段我知道会变，别管它。

---

## 2. 什么是漂移？

Terraform 里的漂移通常指：

```text
真实资源状态 != Terraform state 或 Terraform 配置预期
```

比如 Terraform 创建了一个文件：

```text
output/managed-note.txt
```

内容是：

```text
baseline content
```

后来外部系统把它改成：

```text
外部系统写入的内容
```

这时真实文件内容和 Terraform 配置不一致了。  
这就是漂移。

默认情况下，Terraform plan 会发现：

> 文件内容被改了，我要把它恢复成配置里的内容。

但本实验的要求是：  
这个内容本来就允许外部系统改写，所以 Terraform 不应该恢复它。

---

## 3. `ignore_changes` 的基本写法

典型写法如下：

```hcl
resource "local_file" "managed_note" {
  filename = "${path.module}/output/managed-note.txt"
  content  = "Terraform baseline content"

  lifecycle {
    ignore_changes = [
      content
    ]
  }
}
```

这里的意思是：

> `local_file.managed_note` 这个资源的 `content` 属性如果发生外部变化，Terraform plan 时不要因为这个属性产生更新。

所以外部修改文件内容后，再执行：

```sh
terraform plan
```

Terraform 不应该尝试把 `content` 恢复为基线内容。

---

## 4. 本实验的场景

Lab 49 的背景是：

> 文件内容可能被外部系统改写。

问题是：

> Terraform 不应把该内容恢复为基线。

任务是：

1. 创建 `local_file.managed_note`。
2. 对它的 `content` 属性设置 `ignore_changes`。
3. apply 后手动模拟外部系统改写文件。
4. 再跑 plan，确认 Terraform 不尝试恢复 content。

预期结果：

> 外部修改后，Terraform plan 不尝试恢复 `content`。

这就是一个很标准的 drift 接纳场景。

## 6. 第二次 plan 应该怎么看？

关键是这一句：

```sh
terraform plan -input=false -no-color
```

如果 `ignore_changes = [content]` 生效，Terraform 不应该因为 `content` 被外部修改而计划恢复它。

你要看的不是“文件内容变没变”，而是 Terraform 的态度：

- 没有 `ignore_changes`：Terraform 会尝试把内容改回配置值。
- 有 `ignore_changes`：Terraform 接受这个属性的外部变化。

这就是本实验的核心验证点。

---

## 7. `ignore_changes` 适合什么场景？

适合那些**确实由外部系统管理**的字段。

例如：

### 外部系统管理的标签

某些公司会用统一平台自动写入 tag：

```hcl
lifecycle {
  ignore_changes = [
    tags["LastModifiedBy"],
    tags["CostScanner"]
  ]
}
```

### 控制器管理的注释

Kubernetes 或云平台控制器可能自动写 annotation。

```hcl
lifecycle {
  ignore_changes = [
    metadata[0].annotations
  ]
}
```

### 运行时字段

有些字段由平台运行时自动变化，Terraform 不应该持续纠正。

### 临时内容

像本实验的文件内容，如果明确由外部系统写入，可以忽略。

---

## 8. `ignore_changes` 不适合什么？

最危险的用法是拿它掩盖真实问题。

比如资源漂移了，你不想看见 plan 变更，于是写：

```hcl
lifecycle {
  ignore_changes = all
}
```

这很爽，但也很危险。

因为 Terraform 以后可能不再告诉你关键属性已经偏离配置。

不适合忽略的内容包括：

- 安全组规则
- IAM 权限
- 数据库加密开关
- 公网暴露配置
- 备份策略
- 生产容量配置
- 网络访问控制
- 关键镜像版本
- 关键证书配置

这些字段如果被外部改了，Terraform 应该提醒你，而不是装作没看见。

一句话：

> `ignore_changes` 是边界声明，不是遮羞布。

---

## 9. `ignore_changes` 和漂移治理的关系

Terraform 的价值之一是发现漂移。

如果你滥用 `ignore_changes`，就会削弱 Terraform 的漂移检测能力。

更成熟的做法是：

1. 先确认这个字段到底应该由谁管理。
2. 如果由 Terraform 管，就不要 ignore。
3. 如果由外部系统管，才考虑 ignore。
4. 在代码注释或 README 里写清楚原因。
5. 定期复查 ignore 列表，避免历史遗留越来越多。

例如：

```hcl
lifecycle {
  # This field is managed by the runtime controller.
  # Terraform should not reconcile it.
  ignore_changes = [
    content
  ]
}
```

这种注释很有价值。  
几个月后别人看到，知道你不是随手乱写。

---

## 10. `ignore_changes` 和 `prevent_destroy` 的区别

| 参数 | 作用 |
|---|---|
| `prevent_destroy` | 阻止 Terraform 销毁资源 |
| `ignore_changes` | 忽略指定属性的变化 |

它们都是 `lifecycle` 参数，但用途完全不同。

`prevent_destroy` 是防误删。  
`ignore_changes` 是接受某些属性漂移。

不要混用。

---

## 11. `ignore_changes` 和 `create_before_destroy` 的区别

| 参数 | 作用 |
|---|---|
| `create_before_destroy` | 替换时尽量先创建新资源，再销毁旧资源 |
| `ignore_changes` | 某些属性变了也不要触发更新 |

`create_before_destroy` 关注替换顺序。  
`ignore_changes` 关注是否因为某些属性变化而产生 plan。

---

## 12. `ignore_changes` 的属性路径要写对

本实验忽略的是：

```hcl
content
```

因为 `local_file.managed_note` 的内容属性就叫 `content`。

如果是真实云资源，属性路径可能更复杂，比如：

```hcl
tags
```

或：

```hcl
tags["Owner"]
```

或：

```hcl
metadata[0].annotations
```

如果路径写错，Terraform 仍然会检测到漂移。

所以排障第一条就是：

> 仍出现漂移：确认属性路径写对。

---

## 13. 常见坑

### 坑 1：仍出现漂移

原因通常是：

- 属性路径写错
- 实际变化发生在别的字段
- provider 对这个字段处理方式特殊
- 你以为改的是 content，实际改的是 filename 或权限

解决方式是看 plan 里到底哪个属性变化。

---

### 坑 2：忽略过多

不要为了让 plan 干净，就把很多字段都 ignore。

plan 干净不等于系统健康。  
可能只是你把告警器关了。

---

### 坑 3：文件还没创建就改写

本实验必须先执行 apply，生成：

```text
output/managed-note.txt
```

然后才能模拟外部系统改写。

如果文件不存在，先完成 apply。

---

### 坑 4：以为 ignore 会改 state

`ignore_changes` 的核心不是“把 state 改成外部值”，而是影响 Terraform 如何计划变更。

它告诉 Terraform：  
这个属性的差异不要成为更新动作的原因。

---

### 坑 5：忽略安全字段

如果忽略安全相关字段，可能让危险变更长期存在而没人发现。

例如外部把资源改成公网暴露，如果你 ignore 了相关字段，Terraform 可能不再提示。

这类字段慎重，非常慎重。

---

## 14. 生产环境怎么判断该不该 ignore？

可以问自己几个问题：

1. 这个字段是否确实由外部系统管理？
2. 外部系统是否有明确 owner？
3. 如果这个字段漂移，会不会造成安全或可用性风险？
4. Terraform 是否应该重新收敛这个字段？
5. 有没有更好的建模方式？
6. 是否需要写注释说明？
7. 是否需要监控这个字段的变化？

如果答不清楚，就不要急着 ignore。

---

## 15. 好的使用方式

比较好的使用方式是小范围、明确、带注释：

```hcl
lifecycle {
  # The runtime controller writes this content after deployment.
  # Terraform only creates the file and should not reconcile runtime content.
  ignore_changes = [
    content
  ]
}
```

这样表达了三个信息：

1. 谁管理这个字段。
2. 为什么 Terraform 不管理。
3. 忽略范围很小，只忽略 `content`。

---

## 16. 糟糕的使用方式

糟糕写法：

```hcl
lifecycle {
  ignore_changes = all
}
```

除非你非常确定，否则这基本是在说：

> Terraform 创建完以后就别管这个资源了。

那你就要反问：  
既然 Terraform 不管变化，那这个资源还适合由 Terraform 管吗？

## 18. SRE / Platform Engineering 视角

从 SRE 视角看，`ignore_changes` 的本质是**管理边界声明**。

它应该回答：

> 这个字段到底归谁管？

如果归 Terraform 管，Terraform 就应该发现并修复漂移。  
如果归外部系统管，Terraform 就应该别抢控制权。

这其实是平台工程里的核心问题：  
多个系统同时管理同一个对象时，必须明确 ownership。

否则就会出现：

- Terraform 改回去
- 外部系统又改回来
- Terraform 下次又改回去
- CI/CD 永远有 diff
- 团队互相甩锅

`ignore_changes` 可以解决这种冲突，但前提是你真的想清楚了 ownership。

## 19. 面试/考试记忆版

可以这样记：

- `ignore_changes` 写在 `lifecycle` 里。
- 它让 Terraform 忽略指定属性的变化。
- 适合外部系统管理的字段。
- 不能滥用，否则会掩盖真实漂移。
- 属性路径必须写对。
- 不要轻易 `ignore_changes = all`。
- 安全、权限、网络暴露等关键字段不要随便 ignore。
- 最好写注释说明为什么忽略。

一句话总结：

> `ignore_changes` 是用来声明“这个字段不归 Terraform 管”，不是用来让 plan 看起来干净。

---

## 20. 本 Lab 的价值

Lab 49 的价值在于，它让你理解 Terraform 不是必须管理资源的每一个字段。

真实平台里，经常有多个系统共同作用：

- Terraform 管基础结构
- 控制器管运行时状态
- 安全平台打标签
- 成本平台写 metadata
- 运维脚本写注释
- 应用系统更新某些内容

这时 Terraform 必须知道哪些字段该管，哪些字段不该管。

`ignore_changes` 就是这个边界工具。  
用好了，是工程成熟；用烂了，是漂移黑洞。
