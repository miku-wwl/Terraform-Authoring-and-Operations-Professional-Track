# Terraform AWS Sandbox 35 讲解：使用 `aws_ami` 创建 EC2

## 1. 这一节到底在学什么？

Lab 35 是 Lab 34 的下一步。

Lab 34 重点是：  
> 不要硬编码 AMI ID，而是用 `data "aws_ami"` 动态查询 AMI。

Lab 35 进一步做实战连接：  
> 查询到 AMI 以后，把 `data.aws_ami.latest.id` 传给 `aws_instance`，真正创建一台 EC2。

也就是说，这一节不是只停留在“查 AMI”，而是完整走通：

```text
AMI data source 查询
        ↓
拿到 AMI ID
        ↓
作为 aws_instance.ami 参数
        ↓
创建 EC2
```

这就是 Terraform 里非常典型的资源引用模式。

---

## 2. 为什么不能硬编码 AMI ID？

AMI ID 是 region-specific 的。

同一个操作系统镜像，在不同 AWS region 里的 AMI ID 通常不同。例如：

```text
us-east-1        ami-xxxx
ap-southeast-2   ami-yyyy
ap-south-1       ami-zzzz
```

如果你在 Terraform 里写死：

```hcl
resource "aws_instance" "app" {
  ami           = "ami-1234567890abcdef0"
  instance_type = "t3.micro"
}
```

那么切换 region 以后，这个 AMI ID 很可能失效。

更好的做法是：

```hcl
data "aws_ami" "latest" {
  most_recent = true

  filter {
    name   = "name"
    values = ["tf-lab-app-*"]
  }

  owners = ["self"]
}

resource "aws_instance" "app" {
  ami           = data.aws_ami.latest.id
  instance_type = "t3.micro"
}
```

这样 Terraform 会先查询符合条件的 AMI，再把查到的 ID 传给 EC2。

一句话：

> AMI ID 不要手写死，应该让 Terraform 根据条件动态查。

---

## 3. `data.aws_ami.latest.id` 是什么？

`data.aws_ami.latest` 是一个 data source。

它不会创建 AMI，只是读取已有 AMI。

```hcl
data "aws_ami" "latest" {
  most_recent = true
  ...
}
```

查询完成后，它会暴露很多属性，例如：

```hcl
data.aws_ami.latest.id
data.aws_ami.latest.name
data.aws_ami.latest.architecture
```

其中最常用的是：

```hcl
data.aws_ami.latest.id
```

因为 EC2 资源的 `ami` 参数需要的就是 AMI ID。

所以这行：

```hcl
ami = data.aws_ami.latest.id
```

意思是：

> EC2 使用 data source 查出来的最新 AMI，而不是使用硬编码 AMI。

这就是 Terraform 表达式引用的核心价值。

---

## 4. 本实验的设计思路

这个实验使用 LocalStack，不使用真实 AWS。

整体流程如下：

1. `bootstrap.sh` 在 LocalStack 里注册模拟 AMI。
2. Terraform 使用 `data "aws_ami"` 查询最新的模拟 AMI。
3. Terraform 创建 `aws_instance.app`。
4. `aws_instance.app.ami` 引用 `data.aws_ami.latest.id`。
5. 输出 selected AMI ID 和 instance ID。
6. `verify.sh` 验证实例的 `ImageId` 和 Terraform 选择的 AMI 是否一致。
7. 最后 destroy EC2，并 deregister 模拟 AMI。

这个设计重点验证的是：

```text
data source → resource 引用 → output → verify
```

这套链路在真实工程中非常常见。

---

## 5. LocalStack limited 是什么意思？

这个 lab 的环境分类是：

```text
aws-localstack-limited
```

这说明它可以在 LocalStack 中跑通 Terraform authoring 流程，但不能完全代表真实 AWS 行为。

LocalStack 可以帮你验证：

- provider endpoint 是否正确
- `aws_ami` data source 语法是否正确
- `data.aws_ami.latest.id` 是否能被资源引用
- `aws_instance` 是否能通过 LocalStack EC2 API 创建
- output 和 verify 流程是否完整

但 LocalStack 不能完整验证：

- 真实 AWS 公共 AMI 目录
- Amazon Linux / Ubuntu 官方 AMI 更新逻辑
- AMI architecture 与 instance type 的真实兼容性
- EC2 真实启动过程
- VPC 网络连通性
- metadata service
- 真实计费行为

所以这节课的定位要摆正：

> 它练的是 Terraform 写法和工程链路，不是完整模拟真实 EC2 生产环境。

## 6. 生产环境中要特别注意 architecture

真实 AWS 里，AMI 和实例类型必须匹配。

例如：

- `x86_64` AMI 通常配 `t3.micro`、`t3.small`、`m5.large`
- `arm64` AMI 通常配 `t4g.micro`、`t4g.small`、`m7g.large`

如果你用 `arm64` AMI 去配 `t3.micro`，真实 AWS 里会失败。

因此生产里查询 AMI 时，不能只按 name 查，最好加 architecture：

```hcl
filter {
  name   = "architecture"
  values = ["x86_64"]
}
```

还可以加 virtualization type：

```hcl
filter {
  name   = "virtualization-type"
  values = ["hvm"]
}
```

真实 AWS 中更完整的查询通常类似：

```hcl
data "aws_ami" "amazon_linux" {
  most_recent = true
  owners      = ["amazon"]

  filter {
    name   = "name"
    values = ["al2023-ami-*-x86_64"]
  }

  filter {
    name   = "architecture"
    values = ["x86_64"]
  }

  filter {
    name   = "virtualization-type"
    values = ["hvm"]
  }
}
```

这里的重点是：  
不要只靠一个模糊 name pattern，查询条件要能表达你的真实运行约束。

## 7. 标准运行流程

进入实验目录：

```sh
cd practice/labs/35
```

设置 LocalStack 所需环境变量：

```sh
export AWS_ACCESS_KEY_ID=test
export AWS_SECRET_ACCESS_KEY=test
export AWS_DEFAULT_REGION=us-east-1
export LOCALSTACK_ENDPOINT=http://localhost:4566
```

检查 sandbox：

```sh
bash scripts/check-sandbox.sh
```

注册模拟 AMI：

```sh
bash scripts/bootstrap.sh
```

执行 Terraform：

```sh
terraform init -input=false
terraform fmt -check
terraform validate
terraform plan -input=false -no-color -out=tfplan
terraform apply -auto-approve tfplan
terraform output
```

验证：

```sh
bash scripts/verify.sh
```

清理 Terraform 创建的 EC2：

```sh
terraform destroy -auto-approve
```

清理 bootstrap 创建的模拟 AMI：

```sh
bash scripts/clean.sh
```

清理本地 Terraform 文件：

```sh
rm -rf .terraform .terraform.lock.hcl tfplan terraform.tfstate terraform.tfstate.backup
```

## 8. 运行结果应该怎么看？

关键输出应该包括：

```text
selected_ami_id = "ami-..."
instance_id     = "i-..."
```

你要重点确认两件事：

1. `data.aws_ami.latest` 成功查到了模拟 AMI。
2. `aws_instance.app` 创建出来的 EC2 使用的 ImageId，和 `selected_ami_id` 一致。

这说明链路是正确的：

```text
aws_ami 查到 AMI
        ↓
aws_instance 使用这个 AMI
        ↓
verify 查询 EC2 ImageId
        ↓
二者一致
```

## 9. 常见坑

### 坑 1：`aws_ami` 找不到镜像

通常是因为没有先运行：

```sh
bash scripts/bootstrap.sh
```

因为本实验中的 AMI 是 bootstrap 注册出来的模拟 AMI。

---

### 坑 2：Terraform 打到了真实 AWS

这是最危险的。

如果 provider endpoint 没有指向 LocalStack，Terraform 可能会尝试访问真实 AWS。

实验中应该确认：

```sh
export AWS_ACCESS_KEY_ID=test
export AWS_SECRET_ACCESS_KEY=test
export LOCALSTACK_ENDPOINT=http://localhost:4566
```

并确认 provider 里配置了 LocalStack endpoints。

---

### 坑 3：真实 AWS 中 AMI 和 instance type 不兼容

LocalStack 不一定会替你挡住这个错误。

真实 AWS 中必须检查：

- AMI architecture
- instance type architecture
- region
- owner
- virtualization type

不要以为 LocalStack 通过了，真实 AWS 就一定通过。

---

### 坑 4：`most_recent = true` 不是万能保险

`most_recent = true` 的意思是从匹配结果中选最新的。

但如果 filter 写得太宽，它可能选到你不想要的最新镜像。

所以正确姿势是：

```text
most_recent = true + 精确 filter
```

而不是只写 `most_recent = true`。

## 11. 生产环境推荐写法

真实项目里，不建议把 AMI 查询写得太随意。

更推荐把关键条件变量化：

```hcl
variable "ami_name_pattern" {
  type = string
}

variable "ami_owner" {
  type = string
}

variable "ami_architecture" {
  type    = string
  default = "x86_64"
}

data "aws_ami" "selected" {
  most_recent = true
  owners      = [var.ami_owner]

  filter {
    name   = "name"
    values = [var.ami_name_pattern]
  }

  filter {
    name   = "architecture"
    values = [var.ami_architecture]
  }

  filter {
    name   = "virtualization-type"
    values = ["hvm"]
  }
}

resource "aws_instance" "app" {
  ami           = data.aws_ami.selected.id
  instance_type = var.instance_type
}
```

这样更适合模块化，也方便不同环境复用。

## 12. 面试/考试记忆版

可以这样记：

- AMI ID 是 region-specific，不应该硬编码。
- `data "aws_ami"` 用来动态查询 AMI。
- `most_recent = true` 选择匹配结果中最新的 AMI。
- `data.aws_ami.latest.id` 可以传给 `aws_instance.ami`。
- 真实 AWS 中要检查 architecture 和 instance type 是否匹配。
- LocalStack 可以练 Terraform 写法，但不能完整模拟真实 AMI 生态。

一句话总结：

> 这一节的核心是把 AMI 查询结果接到 EC2 创建流程里，避免硬编码 AMI ID，让配置更适合跨 region 和长期维护。

## 13. 本 Lab 的价值

Lab 35 很适合作为 AWS Terraform data source 的入门实战。

它训练的不是复杂 EC2 架构，而是一个非常基础但很重要的工程习惯：

> 先查询外部事实，再把查询结果传给资源。

这个模式在 Terraform 中到处都是：

- 查 AMI，再创建 EC2
- 查 VPC，再创建 subnet 或 security group
- 查 subnet，再创建 ENI 或 ECS service
- 查 IAM policy，再附加给 role
- 查 Route53 zone，再创建 DNS record

所以不要小看这一节。它是从“会写 Terraform resource”走向“会写真实 Terraform 工程”的关键一步。
