# Terraform 实操训练 94：调用方覆盖模块变量（讲解版）

## 1. 这节课到底在练什么

本节主题是 **调用方覆盖模块变量**。

一句话理解：  
**模块自己可以给变量设置默认值，但真正使用模块时，root module / 调用方可以通过 `module` block 显式传参，把默认值覆盖掉。**

这就是 Terraform module 能复用的关键：

- 模块内部定义“我需要哪些输入”
- 模块内部可以给输入设置默认值
- 调用方决定这次实际传什么值
- 同一个模块可以被不同环境、不同团队、不同场景重复使用

例如同一个 `naming` 模块：

- dev 环境传 `environment = "dev"`
- prod 环境传 `environment = "prod"`
- test 环境传 `environment = "test"`

模块代码不用复制三份，只要调用时传不同变量即可。

---

## 2. 为什么这个点很重要

Terraform 模块不是简单的“复制粘贴代码”。

如果模块里的值全部写死，比如：

```hcl
locals {
  environment = "dev"
  owner       = "platform"
}
```

那这个模块就很难复用。  
换一个环境、换一个团队、换一个命名规范，就得改模块源码。

这不叫模块化，这叫“换了个目录继续硬编码”。

更合理的写法是：

```hcl
variable "environment" {
  type        = string
  description = "Deployment environment"
  default     = "dev"
}
```

然后调用方可以覆盖：

```hcl
module "example" {
  source = "./modules/example"

  environment = "prod"
}
```

这样模块默认可以跑，调用方也能按需覆盖。

---

## 3. 核心概念：默认值 vs 调用方传值

Terraform module variable 通常有两种来源：

### 3.1 模块变量默认值

在子模块中定义：

```hcl
variable "environment" {
  type        = string
  description = "Environment name"
  default     = "dev"
}
```

含义是：  
如果调用方不传 `environment`，模块就使用 `"dev"`。

### 3.2 调用方覆盖变量

在 root module 中调用：

```hcl
module "app" {
  source = "./modules/app"

  environment = "prod"
}
```

含义是：  
这次调用时，`environment` 使用 `"prod"`，不再使用模块里的默认 `"dev"`。

优先级可以记成：

```text
调用方显式传值 > 子模块 variable default
```

这就是“调用方覆盖模块变量”。

---

## 4. 生产环境里的真实意义

这个能力在生产环境非常核心。

因为实际项目里，一个模块往往要同时服务多个环境：

```text
modules/
  storage/
envs/
  dev/
  staging/
  prod/
```

每个环境都调用同一个模块，但传入不同参数：

```hcl
module "storage" {
  source = "../../modules/storage"

  environment = "prod"
  sku         = "premium"
  replica    = 3
}
```

dev 可能是：

```hcl
module "storage" {
  source = "../../modules/storage"

  environment = "dev"
  sku         = "standard"
  replica    = 1
}
```

模块还是同一个模块。  
差异通过输入变量表达，而不是复制模块源码。

这也是平台工程里很重要的一点：  
**平台团队交付模块，业务团队通过变量消费模块。**

## 7. `terraform test` 在这里验证什么

这节的测试重点通常不是“资源有没有真的创建成功”，而是：

> 调用方传入的变量值，是否真的覆盖了模块内部默认值。

例如测试可能会验证 output：

```hcl
assert {
  condition     = output.environment == "prod"
  error_message = "The caller-provided environment value should override the module default."
}
```

这类测试的价值是防止模块重构时出现低级错误：

- 调用方传参了，但模块没接住
- 子模块变量名写错
- output 引用了错误的 local 或 variable
- 默认值被误认为最终值
- 测试只验证了语法，没有验证行为

真正做模块工程时，这种测试非常有用。  
它能保护模块的输入输出契约。

## 8. 常见错误 1：以为 root variable 会自动传进 module

很多新手会写出这种代码。

root module：

```hcl
variable "environment" {
  default = "prod"
}

module "app" {
  source = "./modules/app"
}
```

子模块：

```hcl
variable "environment" {
  default = "dev"
}
```

然后以为子模块会自动拿到 root 里的 `"prod"`。

不会。

Terraform module 的变量作用域是隔离的。  
root module 里的 variable 不会自动进入 child module。

必须显式传：

```hcl
module "app" {
  source = "./modules/app"

  environment = var.environment
}
```

这点非常重要。

记住一句话：

```text
Terraform module 不会读心。你不传，它就不知道。
```

---

## 9. 常见错误 2：变量名对不上

子模块定义：

```hcl
variable "environment" {
  type = string
}
```

调用方传：

```hcl
module "app" {
  source = "./modules/app"

  env = "prod"
}
```

这就错了。

因为子模块要的是 `environment`，调用方传的是 `env`。  
Terraform 不会自动猜测它们是同一个东西。

正确写法：

```hcl
module "app" {
  source = "./modules/app"

  environment = "prod"
}
```

模块接口要稳定、清楚、一致。  
这就是为什么变量命名不要随便飘。

---

## 10. 常见错误 3：模块里继续硬编码

有些人虽然定义了变量，但模块内部不用变量，继续硬编码：

```hcl
variable "environment" {
  type    = string
  default = "dev"
}

locals {
  name = "dev-app"
}
```

这样调用方就算传了 `environment = "prod"`，结果也还是 `dev-app`。

应该改成：

```hcl
locals {
  name = "${var.environment}-app"
}
```

这才是真正让变量参与模块行为。

## 12. 面试/考试表达方式

如果面试官问：

> Terraform module variable default 和调用方传参是什么关系？

可以回答：

> 子模块可以在 `variable` block 中定义默认值。如果调用方在 `module` block 中显式传入同名参数，那么调用方传入的值会覆盖子模块默认值。模块之间变量作用域是隔离的，root module 的变量不会自动传入 child module，必须在 module block 中显式映射，例如 `environment = var.environment`。

更工程化一点可以补一句：

> 这让同一个模块可以服务 dev、staging、prod 等不同环境，差异通过变量表达，而不是复制模块代码。

## 13. 这节课的关键结论

1. 子模块可以用 `variable` 定义输入参数。
2. 子模块可以给变量设置 `default`。
3. 调用方可以在 `module` block 中传入同名参数覆盖默认值。
4. root module 变量不会自动进入 child module，必须显式传递。
5. 模块复用的核心是：公共逻辑放模块里，环境差异通过变量传进去。
6. `terraform test` 可以验证模块输入输出契约，避免重构时把模块行为改坏。
7. 不要把模块写成“换目录的硬编码”，要让变量真正参与资源命名、标签、规格和行为控制。
