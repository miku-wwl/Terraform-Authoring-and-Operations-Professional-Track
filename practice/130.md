## 2. 知识点讲解

Health assessments 用于评估真实基础设施是否仍然符合 Terraform 配置和持续健康条件。课程中把它分成两个关键方向：drift detection 和 continuous validation。前者关注真实资源是否偏离 Terraform 配置，后者关注资源部署后自定义健康条件是否仍然成立。

Drift detection 的典型场景是有人绕过 Terraform 在云控制台手动改了安全组、标签或实例配置。Terraform 配置没有变，但真实基础设施变了，这就是 drift。平台定期检测 drift，可以提醒团队回到 IaC 管理边界。

Continuous validation 更像部署后的持续断言，例如网站是否返回 200、证书是否正常、服务是否仍满足健康条件。课程也提到，Terraform CLI 本身有 `check` block，可以表达类似的健康检查逻辑；HCP Terraform 的价值在于把这些检查平台化、可视化和持续化。

## 3. 考试选择题

1. Drift detection 主要检测什么？
   - A. 真实基础设施是否偏离 Terraform 配置和 state 期望
   - B. Markdown 是否拼写正确
   - C. Docker 镜像是否过期
   - D. Git 用户名是否正确

2. Continuous validation 更关注什么？
   - A. 资源创建后，自定义健康条件是否持续通过
   - B. 删除所有历史 state
   - C. 禁止使用变量
   - D. 强制所有代码写在一个文件

3. Terraform CLI 中哪个语言特性可以表达持续检查类逻辑？
   - A. `check` block
   - B. `terraform fmt`
   - C. `.gitignore`
   - D. provider lock file

## 4. 判断题

1. Drift 可能由人工在云控制台直接修改资源造成。
2. Continuous validation 只在 `terraform init` 之前运行一次。
3. Health assessments 更偏企业治理和持续运维能力。

## 5. 场景题

1. 某安全组通过 Terraform 创建后，被人在控制台手动新增了入站规则。HCP Terraform 的哪类能力可以发现这种偏离？

2. 团队希望确认网站部署后持续返回 HTTP 200。这个需求更接近 drift detection 还是 continuous validation？

## 6. 参考答案与解析

选择题：1-A，2-A，3-A。

判断题：1-对，2-错，3-对。

场景题参考：

1. 这是 drift detection 的典型场景，因为真实基础设施被手动修改，偏离了 Terraform 配置。
2. 更接近 continuous validation，因为它检查资源运行后的健康条件是否持续满足。

## 7. 复习提醒

- Drift detection 看“配置与真实资源是否偏离”。
- Continuous validation 看“部署后的健康条件是否持续成立”。
- `check` block 是 CLI 层面理解 continuous validation 的重要参照。
