# Terraform 实操训练 88：创建自定义模块结构（讲解版）

## 1. 本节核心

本节主题是 **创建自定义 Terraform module 的目录结构**。

这节不是单纯学“文件怎么摆”，而是开始进入 Terraform 工程化的关键阶段：

> 把一堆散落在 root module 里的资源配置，整理成可以复用、可以测试、可以交付给团队使用的模块。

前面学习 Terraform 的时候，很多 lab 都是在一个目录里写 `main.tf`、`variables.tf`、`outputs.tf`。这种方式适合入门，但一旦项目变大，就会出现几个问题：

- 多个环境重复写类似代码。
- 资源命名、标签、默认参数不统一。
- 修改一处逻辑时，需要手动改很多地方。
- 团队协作时，很难判断哪些变量是调用方应该关心的，哪些实现细节应该隐藏。

自定义模块的价值，就是把这些问题收敛起来。

---

## 2. 什么是 Terraform module？

Terraform 里，**每个包含 `.tf` 文件的目录，本质上都可以是一个 module**。

常见有两类：

### 2.1 Root module

你运行 Terraform 命令所在的目录，就是 root module。

例如：

```text
practice/labs/88/
├── main.tf
├── variables.tf
├── outputs.tf
└── tests/
```

如果你在 `practice/labs/88` 目录里执行：

```sh
terraform init
terraform plan
```

那么这个目录就是当前执行入口，也就是 root module。

### 2.2 Child module

被 root module 调用的模块，就是 child module。

例如：

```text
practice/labs/88/
├── main.tf
├── modules/
│   └── naming/
│       ├── main.tf
│       ├── variables.tf
│       └── outputs.tf
└── tests/
```

root module 可以这样调用：

```hcl
module "naming" {
  source = "./modules/naming"

  environment = "dev"
  workload    = "app"
}
```

这里的 `./modules/naming` 就是一个自定义 child module。

---

## 3. 为什么要创建自定义模块结构？

一句话：

> module 是 Terraform 项目从“脚本堆叠”走向“工程资产”的分界线。

没有 module 时，代码更像一次性脚本。

有 module 后，代码开始具备：

- 复用能力
- 输入输出边界
- 版本管理能力
- 测试能力
- 团队交付能力
- 平台化能力

这对你之后做 Platform Engineering、SRE、CAF、AKS/EKS 项目都很关键。

因为真正的云平台团队，不会让每个业务团队从零写 VNet、Subnet、AKS、IAM、Monitor、Policy，而是会提供经过审查的模块。

业务团队只需要填变量：

```hcl
module "aks_cluster" {
  source = "git::https://github.com/org/terraform-azurerm-aks.git?ref=v1.4.2"

  environment = "prod"
  region      = "australiaeast"
  node_count  = 3
}
```

这就是模块化的生产意义。

## 4. 推荐的自定义 module 目录结构

一个基础 module 通常建议这样拆：

```text
modules/
└── example-module/
    ├── main.tf
    ├── variables.tf
    ├── outputs.tf
    ├── versions.tf
    └── README.md
```

各文件职责如下。

### 4.1 main.tf

放模块的核心资源逻辑。

例如：

```hcl
resource "local_file" "example" {
  filename = "${path.module}/output/${var.name}.txt"
  content  = var.content
}
```

它回答的是：

> 这个模块到底创建什么？

---

### 4.2 variables.tf

定义模块对外暴露的输入参数。

例如：

```hcl
variable "name" {
  type        = string
  description = "Name used to generate the file."
}

variable "content" {
  type        = string
  description = "Content written into the file."
}
```

它回答的是：

> 调用方必须或者可以传什么？

变量设计要克制。不要把 provider 的所有底层参数都暴露出去，否则模块就变成了“套壳资源”，没有治理价值。

---

### 4.3 outputs.tf

定义模块对外输出的结果。

例如：

```hcl
output "file_path" {
  value       = local_file.example.filename
  description = "Path of the generated file."
}
```

它回答的是：

> 调用方可以拿到什么结果，继续交给别的模块使用？

比如 VNet module 输出 subnet id，AKS module 再引用它。

---

### 4.4 versions.tf

定义 Terraform 版本和 provider 版本约束。

例如：

```hcl
terraform {
  required_version = ">= 1.11.0"

  required_providers {
    local = {
      source  = "hashicorp/local"
      version = "~> 2.5"
    }
  }
}
```

它回答的是：

> 这个模块在哪些 Terraform / provider 版本下是可预期的？

生产环境里不写版本约束，很容易出现“昨天还能跑，今天 provider 升级后炸了”的情况。

---

### 4.5 README.md

说明模块用途、输入、输出、示例。

一个 module 没有 README，团队里别人很难放心用。

README 至少应该包含：

- 模块用途
- 使用示例
- 输入变量
- 输出值
- 注意事项
- 版本兼容性

---

## 5. Root module 如何调用自定义 module？

假设目录结构如下：

```text
practice/labs/88/
├── main.tf
├── modules/
│   └── naming/
│       ├── main.tf
│       ├── variables.tf
│       └── outputs.tf
└── tests/
    └── naming.tftest.hcl
```

root module 的 `main.tf` 可以调用本地模块：

```hcl
module "naming" {
  source = "./modules/naming"

  environment = "dev"
  workload    = "payment"
}
```

然后输出模块结果：

```hcl
output "resource_name" {
  value = module.naming.resource_name
}
```

这里要注意：

- `source = "./modules/naming"` 是相对路径。
- module 的输出要通过 `module.<模块名>.<output名>` 访问。
- child module 内部资源不要被 root module 直接访问。
- root module 和 child module 的变量空间是隔离的。

## 6. 输入输出边界：module 最重要的设计点

写 module 时，最重要的不是“能不能跑”，而是边界是否清晰。

### 6.1 输入变量应该表达业务意图

不好的设计：

```hcl
variable "vm_size" {
  type = string
}
```

这不一定错，但如果所有底层参数都让调用方决定，平台团队就失去治理能力。

更好的设计可能是：

```hcl
variable "size_tier" {
  type        = string
  description = "Workload size tier: small, medium, large."

  validation {
    condition     = contains(["small", "medium", "large"], var.size_tier)
    error_message = "size_tier must be one of: small, medium, large."
  }
}
```

然后模块内部自己映射：

```hcl
locals {
  vm_size_map = {
    small  = "Standard_B2s"
    medium = "Standard_D2s_v5"
    large  = "Standard_D4s_v5"
  }
}
```

这样调用方表达的是业务需求，模块内部落实平台标准。

---

### 6.2 输出值应该服务组合，而不是暴露所有细节

不好的设计：

```hcl
output "everything" {
  value = local_file.example
}
```

这会把内部实现细节暴露出去。

更好的设计：

```hcl
output "file_path" {
  value       = local_file.example.filename
  description = "Generated file path."
}
```

输出值应该稳定、明确、可组合。

## 7. path.module、path.root、path.cwd 的区别

自定义 module 结构里，经常会遇到路径问题。

Terraform 常用路径变量包括：

| 表达式 | 含义 | 常见用途 |
|---|---|---|
| `path.module` | 当前 module 所在目录 | module 内部引用模板、脚本、文件 |
| `path.root` | root module 所在目录 | root module 级别的路径 |
| `path.cwd` | 执行 Terraform 命令时的工作目录 | 一般少用，容易受执行位置影响 |

在 module 内，优先使用：

```hcl
path.module
```

不要随便依赖 `path.cwd`，因为 CI/CD、Docker、IDE 里执行路径可能不同。

## 8. 测试自定义 module 的意义

本节 lab 里要求执行：

```sh
terraform test
```

这说明 module 不只是“写出来”，还要“可验证”。

Terraform test 可以帮你验证：

- module 输出是否符合预期。
- 输入变量约束是否有效。
- plan/apply 行为是否符合设计。
- 重构后模块行为是否没有被破坏。

例如测试输出命名规则：

```hcl
run "validate_resource_name" {
  command = plan

  assert {
    condition     = output.resource_name == "dev-payment"
    error_message = "resource_name should follow environment-workload naming convention."
  }
}
```

这类测试在生产中很有价值。

因为模块一旦被多个团队引用，小改动也可能影响很多项目。测试就是模块的安全网。

## 9. 本节实验运行流程

进入实验目录：

```sh
cd /workspace/practice/labs/88
```

初始化：

```sh
terraform init -input=false
```

检查格式：

```sh
terraform fmt -check
```

校验语法：

```sh
terraform validate
```

运行测试：

```sh
terraform test
```

生成计划：

```sh
terraform plan -input=false -no-color -out=tfplan
```

应用：

```sh
terraform apply -auto-approve tfplan
```

查看输出：

```sh
terraform output
```

销毁资源：

```sh
terraform destroy -auto-approve
```

这套流程本质上是在训练一个完整闭环：

```text
init -> fmt -> validate -> test -> plan -> apply -> output -> destroy
```

这已经很接近 CI/CD 里的 Terraform module 验证流程。

## 10. 生产环境中的模块设计原则

### 10.1 单一职责

一个 module 不要什么都管。

比如：

- `network` module 管网络。
- `aks` module 管集群。
- `monitoring` module 管监控。
- `iam` module 管权限。
- `policy` module 管治理策略。

不要把 VNet、AKS、Log Analytics、Policy、DNS、RBAC 全塞进一个巨型 module。

巨型 module 的问题是：

- 输入变量爆炸。
- 测试困难。
- 升级风险高。
- 使用者很难理解边界。
- 小变更也可能影响大范围资源。

---

### 10.2 默认值要安全

module 默认值应该偏向安全和可维护，而不是图省事。

例如：

- 默认启用标签。
- 默认启用日志。
- 默认启用最小权限。
- 默认关闭公网入口。
- 默认启用加密。
- 默认设置合理命名规范。

平台团队提供 module，本质上是在提供“默认正确的基础设施”。

---

### 10.3 不要过度抽象

不是所有重复代码都值得抽成 module。

过早抽象会导致：

- module 参数过多。
- 调用方看不懂。
- 资源能力被限制。
- 维护成本比复制代码更高。

一个简单判断标准：

> 如果同一类资源会被多个项目长期复用，并且你希望统一标准，那么它适合抽成 module。

---

### 10.4 module 要版本化

生产环境不要直接引用浮动分支：

```hcl
source = "git::https://github.com/org/module.git"
```

更好的方式是固定版本：

```hcl
source = "git::https://github.com/org/module.git?ref=v1.2.0"
```

否则上游 module 一改，下游项目可能突然 plan 出大量变更。

## 11. 常见坑

### 坑 1：module 里直接写死环境名

不推荐：

```hcl
locals {
  environment = "dev"
}
```

这样模块只能服务一个环境。

应该通过变量传入：

```hcl
variable "environment" {
  type = string
}
```

---

### 坑 2：输出值命名不稳定

如果 output 名字经常变，下游引用就会频繁破坏。

输出值应该当成 API 看待，谨慎修改。

---

### 坑 3：把 provider 配置写死在 child module 里

通常 provider 配置应该由 root module 统一管理。

child module 里可以声明 required provider，但不要随便写死认证、region、subscription 等环境级配置。

---

### 坑 4：module source 不固定版本

公共模块或者 Git 模块不锁版本，很容易引入不可控变更。

---

### 坑 5：module 太薄

如果 module 只是这样：

```hcl
resource "azurerm_resource_group" "this" {
  name     = var.name
  location = var.location
}
```

然后把所有参数原样暴露出去，它可能只是“资源套壳”，治理价值很低。

module 应该封装标准，而不是机械搬运 provider schema。

## 14. 面试表达

如果面试官问：

> 你怎么设计 Terraform module？

你可以这样回答：

```text
我会先明确 module 的职责边界，不会把所有资源都塞进一个大模块。
通常会把 main.tf、variables.tf、outputs.tf、versions.tf 和 README.md 拆清楚。

variables 负责定义调用方可以控制的输入，outputs 负责暴露稳定的组合接口。
对于团队复用的 module，我会加 variable validation、合理默认值、版本约束和 terraform test。
生产环境引用 Git module 时，我会固定 ref 版本，避免上游变更直接影响下游 plan。
```

这比只说“module 可以复用代码”高级很多。

## 15. 本节总结

Terraform 自定义 module 的核心不是目录漂亮，而是把基础设施能力封装成稳定、可测试、可复用的接口。

你可以把 module 理解成 Terraform 世界里的“内部平台产品”：

- 输入变量是用户填写的表单。
- main.tf 是平台内部实现。
- outputs 是交付给下游系统的结果。
- README 是使用说明。
- test 是质量门禁。
- version 是变更边界。

真正成熟的 Terraform 工程，不是靠到处复制 `.tf` 文件，而是靠经过验证的模块体系支撑起来的。
