# Terraform 实操训练 47 讲解：`create_before_destroy`

## 1. 这一节到底在学什么？

Lab 47 学的是 Terraform `lifecycle` 里的一个重要参数：

```hcl
create_before_destroy = true
```

它解决的是资源替换时的顺序问题。

默认情况下，当一个资源必须替换时，Terraform 可能会先销毁旧资源，再创建新资源。  
这在某些场景下会造成服务中断。

`create_before_destroy` 的作用是告诉 Terraform：

> 如果资源必须替换，尽量先创建新资源，再销毁旧资源。

一句话理解：

> 默认可能是“先拆旧房，再盖新房”；`create_before_destroy` 尽量变成“先盖新房，再拆旧房”。

---

## 2. 为什么资源会发生替换？

Terraform 资源变更通常有几种动作：

- create：创建
- update：原地更新
- destroy：销毁
- replace：替换

其中 replace 最需要警惕。

替换通常意味着：

```text
旧资源不能直接改，只能重新创建一个新资源。
```

触发替换的常见原因包括：

- 资源名称变化
- 不可变属性变化
- 镜像版本变化
- 启动模板变化
- 某些网络绑定变化
- provider 标记为 ForceNew 的字段变化
- 显式使用 `replace_triggered_by`
- 在 `terraform_data` 中使用 `triggers_replace`

本实验用的是：

```hcl
triggers_replace
```

它用来模拟“某个值变化后资源必须替换”的场景。

---

## 3. 默认替换顺序有什么风险？

如果 Terraform 替换资源时采用：

```text
destroy old -> create new
```

那么中间会出现空档。

例如：

1. 旧服务实例被删掉。
2. 新服务实例还没创建好。
3. 用户访问失败。
4. 监控开始报警。
5. 部署窗口变成事故窗口。

这就是典型的中断风险。

所以对于服务类资源、运行实例、负载均衡后端、蓝绿发布对象，替换顺序非常关键。

---

## 4. `create_before_destroy` 怎么改变行为？

配置示意：

```hcl
resource "terraform_data" "service" {
  input = {
    image_version = var.image_version
  }

  triggers_replace = [
    var.image_version
  ]

  lifecycle {
    create_before_destroy = true
  }
}
```

这里表达的是：

1. `image_version` 变化会触发替换。
2. 替换时尽量先创建新对象。
3. 新对象创建成功后，再销毁旧对象。

这样可以降低服务中断概率。

---

## 5. 本实验的场景

Lab 47 的背景是：

> 服务镜像版本变化会触发替换。

问题是：

> 如果先销毁旧版本，再创建新版本，可能造成服务中断。

任务是：

1. 用 `triggers_replace` 模拟替换。
2. 启用 `create_before_destroy`。
3. 默认流程跑通。
4. 使用 `image_version=v2` 观察替换行为。

预期结果：

> 测试确认替换策略为 `create_before_destroy`。

---

## 6. `triggers_replace` 是什么？

`triggers_replace` 常见于 `terraform_data` 资源，用来声明：

> 当这些值变化时，这个资源需要替换。

示意：

```hcl
resource "terraform_data" "service_deployment" {
  input = {
    image_version = var.image_version
  }

  triggers_replace = [
    var.image_version
  ]
}
```

当你从：

```hcl
image_version = "v1"
```

改成：

```hcl
image_version = "v2"
```

Terraform 会认为这个资源需要替换。

在真实云资源里，类似替换可能来自 provider 的 ForceNew 属性；在本实验里，用 `triggers_replace` 是为了稳定模拟这个行为，不依赖真实云平台。

## 7. 标准运行流程

进入实验目录：

```sh
cd /workspace/practice/labs/47
```

初始化：

```sh
terraform init -input=false
```

格式检查：

```sh
terraform fmt -check
```

配置校验：

```sh
terraform validate
```

运行测试：

```sh
terraform test
```

第一次生成 plan：

```sh
terraform plan -input=false -no-color -out=tfplan
```

第一次 apply：

```sh
terraform apply -auto-approve tfplan
```

模拟镜像版本升级到 v2：

```sh
terraform plan -input=false -no-color -var 'image_version=v2' -out=tfplan
```

执行替换：

```sh
terraform apply -auto-approve tfplan
```

查看输出：

```sh
terraform output
```

销毁：

```sh
terraform destroy -auto-approve
```

清理本地文件：

```sh
rm -f tfplan
rm -rf .terraform .terraform.lock.hcl
```

## 8. 怎么观察替换行为？

重点看第二次 plan：

```sh
terraform plan -input=false -no-color -var 'image_version=v2' -out=tfplan
```

你应该关注 plan 里是否出现类似替换信号：

```text
must be replaced
```

或者资源动作符号：

```text
+/- create replacement and then destroy
```

如果看到的是：

```text
-/+ destroy and then create replacement
```

那通常表示先销毁再创建。

如果是：

```text
+/- create replacement and then destroy
```

那就更符合 `create_before_destroy` 的意图。

---

## 9. `create_before_destroy` 不等于零中断

这个点非常重要。

`create_before_destroy` 是降低中断风险，不是保证零中断。

真正的零中断还需要配合：

- 负载均衡
- 健康检查
- readiness probe
- 蓝绿发布
- 滚动发布
- 容量预留
- DNS 切换策略
- 数据库迁移策略
- 应用兼容性设计

Terraform 只是改变资源替换顺序。  
它不能保证你的应用已经 ready，也不能保证用户流量已经安全切过去。

所以别把它神化。

---

## 10. 真实平台上的限制：唯一名称冲突

`create_before_destroy` 最大的现实限制之一是：新旧资源可能不能同时存在。

例如某些资源要求名称全局唯一：

```text
my-prod-service
```

如果旧资源还没删除，新资源就不能用同一个名字创建。

这种情况下，即使你写了：

```hcl
create_before_destroy = true
```

Terraform 也可能无法先创建新资源，因为平台 API 会拒绝。

解决思路包括：

1. 给资源名称增加随机后缀。
2. 用版本化名称。
3. 使用蓝绿两套资源。
4. 先创建新资源，用流量切换，再清理旧资源。
5. 拆分资源模型，不要把唯一名称资源直接替换。

一句话：

> `create_before_destroy` 想先建新，但云平台必须允许新旧同时存在。

---

## 11. 真实平台上的限制：配额和容量

另一个常见限制是配额。

例如旧资源正在占用：

- 2 台实例
- 4 个 vCPU
- 100GB 存储
- 1 个公网 IP
- 1 个负载均衡配额

如果先创建新资源，就意味着短时间内资源占用可能翻倍。

你需要确保：

- 账号配额够
- 区域容量够
- 成本可接受
- IP / subnet 容量够
- 负载均衡目标容量够

否则 plan 看起来很好，apply 时还是会失败。

---

## 12. 什么时候适合使用 `create_before_destroy`？

适合：

- 服务实例替换
- 应用版本升级
- 运行时节点替换
- 不想出现空档的资源
- 能允许新旧资源短暂共存的场景
- 蓝绿发布或滚动发布相关资源

不太适合：

- 名称必须唯一的资源
- 新旧不能共存的资源
- 成本非常高、无法双份创建的资源
- 状态型资源，例如数据库主实例
- 有复杂数据迁移要求的资源

对于状态型资源，不要简单相信 `create_before_destroy`。  
数据库替换不是“先建新再删旧”这么简单，还涉及数据同步、切流、回滚和一致性。

---

## 13. 和 `prevent_destroy` 的区别

`create_before_destroy` 和 `prevent_destroy` 都属于 `lifecycle`，但作用完全不同。

| 参数 | 作用 |
|---|---|
| `create_before_destroy` | 替换时尽量先创建新资源，再销毁旧资源 |
| `prevent_destroy` | 阻止 Terraform 销毁资源 |

`create_before_destroy` 不是防误删。  
它只是改变替换顺序。

如果你要保护数据库不被删，应该考虑：

```hcl
prevent_destroy = true
```

如果你要降低替换中断风险，才考虑：

```hcl
create_before_destroy = true
```

---

## 14. 和 `replace_triggered_by` 的区别

Lab 46 里提到了 `replace_triggered_by`。  
它和本节也有关系。

| 参数 | 作用 |
|---|---|
| `create_before_destroy` | 控制替换顺序 |
| `replace_triggered_by` | 控制什么变化会触发替换 |
| `triggers_replace` | 在 `terraform_data` 中模拟或声明替换触发 |

可以这样理解：

- `replace_triggered_by` / `triggers_replace`：决定“要不要替换”
- `create_before_destroy`：决定“替换时先后顺序”

一个管触发，一个管顺序。

## 15. 常见坑

### 坑 1：以为写了就一定能先创建

不一定。

如果平台不允许新旧资源同时存在，或者名称冲突，先创建会失败。

---

### 坑 2：忽略配额

先创建新资源意味着短时间内资源可能翻倍。

生产环境要先确认配额和成本。

---

### 坑 3：以为它能保证零中断

不能。

它只是资源顺序控制，不是流量切换方案。

真正零中断要配合 LB、健康检查、发布策略和应用兼容性。

---

### 坑 4：对数据库乱用

状态型资源替换非常危险。  
数据库替换涉及数据迁移，不是简单 lifecycle 能解决。

---

### 坑 5：没有观察 plan 符号

要看清楚 plan 是：

```text
+/- create replacement and then destroy
```

还是：

```text
-/+ destroy and then create replacement
```

这两个顺序不一样，风险也不一样。

---

## 16. SRE / Platform Engineering 视角

从 SRE 视角看，`create_before_destroy` 体现的是一个核心原则：

> 替换生产服务时，尽量避免出现容量空档。

它不是完整发布系统，但它能帮 Terraform 在资源层面更贴近安全变更。

平台团队设计模块时，可以考虑：

- 哪些资源替换会导致中断？
- 哪些资源允许新旧共存？
- 是否需要版本化命名？
- 是否需要蓝绿部署结构？
- 是否需要提前检查配额？
- 是否需要和健康检查、流量切换配合？
- 是否需要在 README 中明确 lifecycle 设计原因？

成熟做法不是随手加 lifecycle，而是解释为什么加。

## 18. 面试/考试记忆版

可以这样记：

- `create_before_destroy` 写在 `lifecycle` 里。
- 它用于资源替换时尽量先创建新资源，再销毁旧资源。
- 它适合降低服务中断风险。
- 它不能保证零中断。
- 它可能受唯一名称、配额、容量限制影响。
- `triggers_replace` 可以模拟替换触发。
- `replace_triggered_by` 决定替换触发条件，`create_before_destroy` 决定替换顺序。
- 看到 plan 时要区分 `+/-` 和 `-/+`。

一句话总结：

> `create_before_destroy` 是 Terraform 替换顺序控制工具，适合降低中断风险，但前提是新旧资源能同时存在。

## 19. 本 Lab 的价值

Lab 47 的价值在于，它让你开始从“资源能不能创建”转向“资源怎么安全替换”。

真实生产事故里，很多问题不是资源创建失败，而是替换顺序不对：

- 旧服务先没了
- 新服务还没 ready
- 负载均衡目标为空
- DNS 已切但后端没起来
- 配额不足导致新资源创建失败
- 数据型资源被错误替换

`create_before_destroy` 是避免这类问题的一个基础工具。  
它不是万能方案，但它是理解 Terraform 安全变更的必学点。
