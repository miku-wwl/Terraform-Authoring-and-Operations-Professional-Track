# Terraform 实操训练 90：module source 讲解版

## 1. 本节主题

本节讲的是 Terraform module 的 `source` 属性。

一句话理解：

> `source` 决定 Terraform 去哪里找模块代码。

如果前面几节已经理解了“模块就是可复用的 Terraform 代码包”，那这一节就是继续往下走：  
**模块可以放在本地目录，也可以放在 Git 仓库、Terraform Registry、压缩包地址、对象存储等位置。`source` 就是模块代码的定位方式。**

---

## 2. 为什么 module source 很重要

在真实工程里，模块不会永远写在当前目录下面。

平台团队通常会把通用能力沉淀成模块，例如：

- VPC / VNet 模块
- AKS / EKS 集群模块
- IAM / RBAC 模块
- Storage / Database 模块
- Monitoring / Logging 模块

业务团队只需要调用模块，而不是重复写底层资源。

这时候就会出现一个核心问题：

> 调用模块的时候，Terraform 怎么知道模块代码在哪里？

答案就是：

```hcl
module "example" {
  source = "../modules/example"
}
```

这里的 `source = "../modules/example"` 就是在告诉 Terraform：  
去当前目录上一级的 `modules/example` 目录中加载模块。

---

## 3. module source 的常见来源

### 3.1 本地路径

最适合学习、实验、单仓库管理。

```hcl
module "naming" {
  source = "./modules/naming"
}
```

或者：

```hcl
module "network" {
  source = "../modules/network"
}
```

特点：

- 不依赖网络
- 适合本地 lab
- 适合 monorepo
- 修改模块后可以马上测试

缺点：

- 跨仓库复用不方便
- 多项目共享时容易复制粘贴
- 版本管理能力弱

---

### 3.2 Terraform Registry

适合使用社区或官方发布的成熟模块。

```hcl
module "vpc" {
  source  = "terraform-aws-modules/vpc/aws"
  version = "5.0.0"
}
```

特点：

- 使用方便
- 生态成熟
- 通常有文档、版本、示例

生产环境重点：

> Registry 模块必须锁版本，不要裸奔。

推荐写法：

```hcl
module "vpc" {
  source  = "terraform-aws-modules/vpc/aws"
  version = "~> 5.0"
}
```

不推荐：

```hcl
module "vpc" {
  source = "terraform-aws-modules/vpc/aws"
}
```

因为没有版本约束时，后续模块升级可能导致计划变化，甚至引入破坏性变更。

---

### 3.3 Git 仓库

适合企业内部模块库。

```hcl
module "network" {
  source = "git::https://github.com/example/terraform-modules.git//network?ref=v1.2.0"
}
```

这段写法里面有几个关键点：

```text
git::https://github.com/example/terraform-modules.git
```

表示模块来自 Git 仓库。

```text
//network
```

表示使用仓库里的 `network` 子目录。

```text
?ref=v1.2.0
```

表示锁定 Git tag、branch 或 commit。

生产环境建议：

```hcl
module "network" {
  source = "git::https://github.com/example/terraform-modules.git//network?ref=v1.2.0"
}
```

不要长期使用：

```hcl
module "network" {
  source = "git::https://github.com/example/terraform-modules.git//network?ref=main"
}
```

原因很简单：

> `main` 会变，tag 和 commit 才适合做可重复部署。

这就是工程化和玩具脚本之间的区别。

---

### 3.4 压缩包或 HTTP 地址

也可以从压缩包加载模块：

```hcl
module "example" {
  source = "https://example.com/modules/example.zip"
}
```

这种方式在普通团队里不如 Registry 和 Git 常见，但在一些内部发布系统、私有制品仓库中也会见到。

生产环境要注意：

- 地址稳定性
- 下载权限
- 校验与供应链安全
- 是否有版本标识

---

## 4. `terraform init` 和 module source 的关系

只要使用模块，尤其是远程模块，就要理解：

> `terraform init` 会根据 `source` 下载或初始化模块代码。

例如：

```sh
terraform init -input=false
```

Terraform 会做几件事：

1. 读取当前目录的 `.tf` 文件
2. 发现 `module` block
3. 根据 `source` 找模块
4. 下载远程模块，或链接本地模块
5. 放到 `.terraform/modules` 目录中
6. 生成或更新模块清单文件

所以，修改 `source` 后，通常需要重新执行：

```sh
terraform init
```

如果模块 source 变化明显，也可以使用：

```sh
terraform init -upgrade
```

不过生产环境里不要随手 `-upgrade`，因为它可能带来 provider 或 module 版本变化。

## 5. 本地模块 source 的实验理解

本 lab 是 Docker 可运行实验，目标是进入：

```sh
cd /workspace/practice/labs/90
```

然后执行标准 Terraform 流程：

```sh
terraform init -input=false
terraform fmt -check
terraform validate
terraform test
terraform plan -input=false -no-color -out=tfplan
terraform apply -auto-approve tfplan
terraform output
terraform destroy -auto-approve
```

你真正要观察的不是“命令有没有绿”，而是：

1. `main.tf` 里有没有 `module` block
2. `module` block 的 `source` 指向哪里
3. `terraform init` 后是否正确加载模块
4. `terraform test` 的断言是否验证了模块输出
5. `terraform output` 是否能看到模块计算后的结果

## 6. module source 的工程判断

### 6.1 学习阶段

推荐：

```hcl
source = "./modules/example"
```

理由：

- 结构清晰
- 不依赖外部网络
- 适合反复修改
- debug 成本最低

你现在这种 Terraform lab 训练，优先用本地 source 是对的。

---

### 6.2 团队内部模块

推荐：

```hcl
source = "git::https://github.com/org/terraform-modules.git//modules/network?ref=v1.0.0"
```

理由：

- 模块可以独立版本化
- 多项目共享
- 可以 code review
- 可以配合 CI 做测试
- 可以配合 tag 做发布

这才是平台工程团队会用的模式。

---

### 6.3 外部公共模块

推荐：

```hcl
source  = "terraform-aws-modules/vpc/aws"
version = "~> 5.0"
```

但要先做模块评估：

- 是否仍在维护
- stars 不等于质量，但可以参考
- release 是否稳定
- issue 是否大量无人处理
- 是否有破坏性升级记录
- 是否支持当前 provider 版本
- 是否符合公司安全规范

公共模块不是不能用，而是不能无脑用。

## 7. 常见错误

### 错误 1：本地路径写错

```hcl
module "demo" {
  source = "./module/demo"
}
```

但真实目录是：

```text
./modules/demo
```

结果：

```text
Module not found
```

排查方式：

```sh
ls -la
find . -maxdepth 3 -type d
```

---

### 错误 2：远程 Git source 忘记 `git::`

不推荐：

```hcl
source = "https://github.com/example/terraform-modules.git//network"
```

推荐：

```hcl
source = "git::https://github.com/example/terraform-modules.git//network?ref=v1.0.0"
```

`git::` 可以明确告诉 Terraform 这是一个 Git source。

---

### 错误 3：使用 main 分支

```hcl
source = "git::https://github.com/example/modules.git//network?ref=main"
```

问题：

- main 分支随时变化
- 计划结果可能突然变化
- 无法保证可重复部署
- 审计困难

更好的写法：

```hcl
source = "git::https://github.com/example/modules.git//network?ref=v1.2.3"
```

或者极端严格：

```hcl
source = "git::https://github.com/example/modules.git//network?ref=8f3a1c9"
```

---

### 错误 4：改了模块但没重新 init

当你修改了 module source，或者切换了远程模块版本后，应该重新执行：

```sh
terraform init
```

必要时：

```sh
terraform init -upgrade
```

但不要把 `-upgrade` 当成无脑默认选项。

## 8. `source` 和 `version` 的关系

很多初学者会混淆这两个属性。

### Registry 模块

Registry 模块通常这样写：

```hcl
module "vpc" {
  source  = "terraform-aws-modules/vpc/aws"
  version = "5.0.0"
}
```

这里：

- `source` 表示模块在哪里
- `version` 表示使用哪个版本

### Git 模块

Git 模块通常这样写：

```hcl
module "network" {
  source = "git::https://github.com/example/modules.git//network?ref=v1.0.0"
}
```

这里版本信息写在 `source` 的 `?ref=` 后面。

也就是说：

> Registry 用 `version` 管版本，Git source 用 `ref` 管版本。

这个区别要记住，Terraform Professional 考试和真实工作都会碰到。

## 9. `source` 对状态的影响

module source 本身不是资源，不会直接创建云资源。

但是它会间接影响资源地址和资源行为。

例如：

```hcl
module "network" {
  source = "./modules/network"
}
```

里面如果有：

```hcl
resource "terraform_data" "example" {
  input = "hello"
}
```

状态地址会类似：

```text
module.network.terraform_data.example
```

注意这里的 `module.network` 来自 module block 的名称，而不是 source 路径。

也就是说：

```hcl
module "network" {
  source = "./modules/network"
}
```

和：

```hcl
module "vpc" {
  source = "./modules/network"
}
```

状态地址是不一样的。

前者：

```text
module.network....
```

后者：

```text
module.vpc....
```

所以重命名 module block 可能导致 Terraform 认为资源地址变了。  
这时就需要用 `moved` block 做状态迁移，而不是直接让 Terraform 删除重建。

## 10. 本节实验应该掌握什么

完成本 lab 后，你应该能说清楚：

1. `source` 是 module block 里的模块代码来源
2. 本地 source 适合学习和单仓库实验
3. Registry source 适合使用公开模块，但必须锁版本
4. Git source 适合企业内部模块库，必须使用 tag 或 commit
5. 修改 source 后通常需要重新 `terraform init`
6. `source` 不直接决定状态地址，module block 的名称才会进入状态地址
7. 不要在生产里长期引用 `main` 分支
8. `version` 常用于 Registry module，Git module 通常用 `?ref=`

## 13. 复盘问题

### Q1：`source` 是干嘛的？

`source` 用来告诉 Terraform 从哪里加载 module 代码。

---

### Q2：本地模块 source 适合什么场景？

适合学习、实验、单仓库项目和本地快速调试。

---

### Q3：为什么生产环境不建议引用 Git 的 main 分支？

因为 main 分支会变化，无法保证同一份 Terraform 配置在不同时间执行得到相同结果。

---

### Q4：Registry module 和 Git module 怎么锁版本？

Registry module 用：

```hcl
version = "x.y.z"
```

Git module 用：

```hcl
?ref=v1.0.0
```

---

### Q5：修改 module source 后要做什么？

通常需要重新执行：

```sh
terraform init
```

如果要主动升级已缓存的模块版本，可以使用：

```sh
terraform init -upgrade
```

---

## 14. 本节结论

`module source` 看起来只是一个路径或地址，但它背后其实是 Terraform 模块工程化的入口。

学习阶段关注：

> source 指向哪里，Terraform 能不能 init 成功。

生产阶段关注：

> source 是否稳定、是否可审计、是否锁版本、是否可回滚。

真正的重点不是会写：

```hcl
source = "./modules/example"
```

而是知道什么时候该用本地路径，什么时候该用 Git tag，什么时候该用 Registry version，什么时候该拒绝一个不靠谱的公共模块。
