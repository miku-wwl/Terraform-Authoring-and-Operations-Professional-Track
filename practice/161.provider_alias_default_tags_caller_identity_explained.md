# 第 161 节讲解：Provider alias、default tags 与 caller identity

## 1. 本节定位

这一节是综合挑战题的一部分，主题是：

> 在一个 Terraform 配置里同时理解 **provider alias**、**default tags** 和 **caller identity**。

它不是单纯考你背某个 AWS 资源参数，而是训练你在一个真实工程场景里判断：

- 当前 Terraform 到底在用哪个 provider 配置；
- 默认标签是否会自动打到资源上；
- 当前 AWS 调用身份到底是谁；
- 如何通过输出和验证脚本确认配置真的生效；
- 如何在 LocalStack 环境中完成闭环练习，但不误以为它等价于真实 AWS 权限模型。

本节对应原始课程文件 `doc/161.txt`，练习参考实现位于 `practice/labs/161/`。

---

## 2. 一句话讲清楚本节

这节可以理解成：

> 用 Terraform 配置多个 AWS Provider 实例，其中一个是默认 provider，另一个通过 `alias` 命名；通过 `default_tags` 自动给资源打标签；再用 `aws_caller_identity` 查询当前调用身份，确认 provider 调用链路是通的。

考试和实战中，这三个点经常一起出现：

- 多 region / 多账号部署需要 provider alias；
- 企业治理需要 default tags；
- 排查账号、角色、凭证问题需要 caller identity。

---

## 3. Provider alias 是什么

Terraform 里的 provider 负责和云平台 API 通信。默认情况下，一个 provider 类型通常有一个默认配置，例如：

```hcl
provider "aws" {
  region = "us-east-1"
}
```

如果你的配置里需要访问多个区域、多个账号，或者用不同配置调用同一个云平台，就需要 provider alias：

```hcl
provider "aws" {
  region = "us-east-1"
}

provider "aws" {
  alias  = "secondary"
  region = "us-west-2"
}
```

这里有两个 AWS provider 实例：

| Provider 配置 | 含义 |
|---|---|
| `aws` | 默认 AWS provider |
| `aws.secondary` | 带 alias 的 AWS provider |

资源默认使用无 alias 的 provider。如果某个资源要使用 alias provider，就要显式指定：

```hcl
resource "aws_s3_bucket" "secondary_bucket" {
  provider = aws.secondary

  bucket = "example-secondary-bucket"
}
```

记住一句话：

> alias 不是新 provider 类型，而是同一个 provider 的另一个配置实例。

---

## 4. 为什么需要 provider alias

### 4.1 多区域部署

比如你要在两个 region 创建资源：

- 主区域：`us-east-1`
- 灾备区域：`us-west-2`

你就可以用默认 provider 管主区域，用 alias provider 管灾备区域。

```hcl
provider "aws" {
  region = "us-east-1"
}

provider "aws" {
  alias  = "dr"
  region = "us-west-2"
}
```

资源中这样选择：

```hcl
resource "aws_s3_bucket" "primary" {
  bucket = "primary-bucket"
}

resource "aws_s3_bucket" "dr" {
  provider = aws.dr
  bucket   = "dr-bucket"
}
```

### 4.2 多账号部署

真实 AWS 中，你也可能通过不同 assume role 访问不同账号：

```hcl
provider "aws" {
  region = "ap-southeast-2"
}

provider "aws" {
  alias  = "security"
  region = "ap-southeast-2"

  assume_role {
    role_arn = "arn:aws:iam::123456789012:role/SecurityAuditRole"
  }
}
```

考试看到多账号、多 region、shared services、security account、network account，就要立刻想到 provider alias。

## 5. default tags 是什么

`default_tags` 是 AWS Provider 的一个配置能力，可以让通过该 provider 创建的资源自动带上默认标签。

示例：

```hcl
provider "aws" {
  region = "us-east-1"

  default_tags {
    tags = {
      Project     = "terraform-training"
      Environment = "local"
      ManagedBy   = "terraform"
    }
  }
}
```

这样一来，通过这个 provider 创建的支持 tag 的 AWS 资源，通常都会自动带上这些标签。

### 它解决什么问题

真实团队里，资源标签非常重要：

- 成本归属：`CostCenter`
- 负责人：`Owner`
- 环境：`Environment`
- 项目：`Project`
- 自动化管理：`ManagedBy`
- 合规审计：`DataClassification`

不用 `default_tags` 的话，每个资源都手写标签，容易漏、容易不一致。

---

## 6. default tags 和资源 tags 的关系

常见写法是 provider 里定义默认标签，资源里定义个性化标签：

```hcl
provider "aws" {
  region = "us-east-1"

  default_tags {
    tags = {
      Environment = "dev"
      ManagedBy   = "terraform"
      Owner       = "platform-team"
    }
  }
}

resource "aws_s3_bucket" "demo" {
  bucket = "demo-bucket"

  tags = {
    Name = "demo-bucket"
  }
}
```

最终资源可能同时拥有：

```text
Environment = dev
ManagedBy   = terraform
Owner       = platform-team
Name        = demo-bucket
```

### 易错点

如果资源自己的 `tags` 里写了和 `default_tags` 相同的 key，通常资源级别的 tag 会覆盖默认值。

例如：

```hcl
default_tags {
  tags = {
    Environment = "dev"
  }
}

resource "aws_s3_bucket" "demo" {
  tags = {
    Environment = "test"
  }
}
```

最终 `Environment` 更可能是 `test`，因为资源自己的配置更具体。

考试遇到这种题，判断逻辑是：

> provider default tags 是默认值，resource tags 是更具体的资源级设置。

## 7. caller identity 是什么

`aws_caller_identity` 是 AWS Provider 的一个 data source，用来查询当前凭证对应的调用身份。

典型写法：

```hcl
data "aws_caller_identity" "current" {}
```

常见输出：

```hcl
output "account_id" {
  value = data.aws_caller_identity.current.account_id
}

output "caller_arn" {
  value = data.aws_caller_identity.current.arn
}

output "user_id" {
  value = data.aws_caller_identity.current.user_id
}
```

在真实 AWS 中，它可以帮你确认：

- 当前用的是哪个 AWS account；
- 当前调用者 ARN 是 IAM user、role，还是 assumed role；
- 当前凭证有没有连到预期账号；
- assume role 后是否真的切到了目标身份。

### 为什么它在排障中很有用

很多 Terraform 出错，本质不是资源配置错，而是身份错：

- 本来想部署到 dev account，结果连到了 prod account；
- 本来应该 assume role，结果还在用本地默认用户；
- CI/CD 里环境变量覆盖了 profile；
- 多 provider alias 中某个 provider 用错了凭证。

这时 `aws_caller_identity` 就像“照妖镜”：

> 先别急着 apply，先输出当前身份，确认自己到底是谁。

## 8. alias provider 下的 caller identity

如果你有多个 provider，可以用不同 provider 查询不同调用身份。

```hcl
provider "aws" {
  region = "us-east-1"
}

provider "aws" {
  alias  = "secondary"
  region = "us-west-2"
}

data "aws_caller_identity" "default" {}

data "aws_caller_identity" "secondary" {
  provider = aws.secondary
}
```

输出：

```hcl
output "default_account_id" {
  value = data.aws_caller_identity.default.account_id
}

output "secondary_account_id" {
  value = data.aws_caller_identity.secondary.account_id
}
```

真实多账号场景里，如果两个 provider 使用不同 role，那么两个输出可能不同。

在 LocalStack 里，它通常只能模拟调用路径，不能完整还原真实 IAM 权限评估。

## 9. LocalStack 环境下怎么理解本节

本节实验不访问真实 AWS，而是使用 LocalStack：

```text
AWS endpoint: http://localhost:4566
Access key:   test
Secret key:   test
```

这意味着你练的是：

- Terraform AWS Provider 配置写法；
- provider alias 的引用方式；
- default tags 的声明方式；
- data source 的调用路径；
- plan / apply / output / destroy 的完整流程；
- Terraform state 中资源和 data source 的变化。

但你不能把它当成真实 AWS 的完全等价替代。

### LocalStack 可以验证什么

| 能验证 | 说明 |
|---|---|
| Terraform 配置语法 | `terraform validate` 能检查基本合法性 |
| provider alias 引用 | 可以确认 `provider = aws.xxx` 写法是否正确 |
| default tags 配置 | 可以观察资源 tag 是否按预期进入配置和状态 |
| caller identity 调用路径 | 可以确认 STS 类接口调用是否走通 |
| apply/destroy 闭环 | 可以训练完整 Terraform 生命周期 |

### LocalStack 不能完全验证什么

| 不能完全替代 | 原因 |
|---|---|
| 真实 IAM 权限评估 | LocalStack 不等价于 AWS IAM 的全部安全模型 |
| 真实跨账号 assume role | 信任策略、外部 ID、组织 SCP 等需要真实 AWS 验证 |
| 真实 region 差异 | LocalStack 多 region 行为和 AWS 真实服务不完全一致 |
| 真实审计记录 | CloudTrail、IAM Access Analyzer 等需要真实 AWS 环境 |

这一点很重要：

> LocalStack 是练 Terraform 写法和流程的，不是验证生产级 AWS IAM 安全边界的最终依据。

## 10. 推荐实验流程

你可以按下面流程做本节实验。

### 第一步：进入练习目录

```bash
cd practice/labs/161/work
```

具体路径以你的项目实际目录为准。

### 第二步：格式化

```bash
terraform fmt -recursive
```

这一步不是装饰，是基本工程习惯。考试不一定直接考，但实际项目里很重要。

### 第三步：初始化

```bash
terraform init
```

它会下载 provider，并初始化 backend、module 等。

### 第四步：验证配置

```bash
terraform validate
```

主要检查：

- HCL 语法；
- provider 引用；
- resource/data source 配置结构；
- 变量和输出是否能解析。

### 第五步：查看计划

```bash
terraform plan
```

重点看：

- 哪些资源会被创建；
- data source 是否会读取；
- tags 是否符合预期；
- provider alias 是否没有报错。

### 第六步：应用

```bash
terraform apply
```

建议练习时先读 plan，再确认 apply。不要养成盲 apply 的习惯。

### 第七步：查看输出

```bash
terraform output
```

重点看：

- caller identity 输出；
- account id / arn / user id；
- 资源名称；
- tag 相关输出。

### 第八步：销毁

```bash
terraform destroy
```

本地实验也要养成销毁习惯，尤其是以后接入真实 AWS 后，这个习惯能省钱。

## 11. 典型 Terraform 结构示例

下面是一个简化示例，帮助你理解本节结构。真实 lab 以你的 `practice/labs/161/` 为准。

```hcl
terraform {
  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 5.0"
    }
  }
}

provider "aws" {
  region                      = "us-east-1"
  access_key                  = "test"
  secret_key                  = "test"
  s3_use_path_style           = true
  skip_credentials_validation = true
  skip_metadata_api_check     = true
  skip_requesting_account_id  = true

  endpoints {
    s3  = "http://localhost:4566"
    sts = "http://localhost:4566"
  }

  default_tags {
    tags = {
      Project     = "terraform-training"
      Environment = "local"
      ManagedBy   = "terraform"
    }
  }
}

provider "aws" {
  alias                       = "secondary"
  region                      = "us-west-2"
  access_key                  = "test"
  secret_key                  = "test"
  s3_use_path_style           = true
  skip_credentials_validation = true
  skip_metadata_api_check     = true
  skip_requesting_account_id  = true

  endpoints {
    s3  = "http://localhost:4566"
    sts = "http://localhost:4566"
  }

  default_tags {
    tags = {
      Project     = "terraform-training"
      Environment = "local-secondary"
      ManagedBy   = "terraform"
    }
  }
}

data "aws_caller_identity" "default" {}

data "aws_caller_identity" "secondary" {
  provider = aws.secondary
}
```

注意几个细节：

- `provider "aws"` 是默认 provider；
- `provider "aws" { alias = "secondary" }` 是带别名的 provider；
- `data "aws_caller_identity" "secondary"` 显式使用 `provider = aws.secondary`；
- LocalStack 常需要配置 endpoints 和跳过部分 AWS 真实校验。

## 12. 本节最容易错的地方

### 错误 1：以为 alias provider 会自动被资源使用

不会。

资源默认使用默认 provider。要使用 alias provider，必须写：

```hcl
provider = aws.secondary
```

### 错误 2：把 alias 写成字符串

错误写法：

```hcl
provider = "aws.secondary"
```

正确写法：

```hcl
provider = aws.secondary
```

Terraform 的 provider 引用不是普通字符串。

### 错误 3：以为 default tags 会覆盖所有东西

不要这么理解。它更像默认标签。资源级标签更具体，可能覆盖同名 key。

### 错误 4：以为 caller identity 会创建资源

`aws_caller_identity` 是 data source，不是 resource。它查询当前身份，不创建云资源。

### 错误 5：把 LocalStack 结果当成真实 AWS IAM 结论

LocalStack 能证明 Terraform 调用路径和配置结构基本正确，但不能证明真实 AWS 权限、信任策略、SCP、CloudTrail、组织策略全部正确。

## 14. 面试讲法

如果面试官问你：“Terraform 里 provider alias 有什么用？”

你可以这样答：

> Provider alias 用来在同一个 Terraform 配置中声明多个同类型 provider 实例。比如 AWS 场景中，我可以一个 provider 指向主 region，另一个 alias provider 指向灾备 region，或者通过不同 assume role 指向不同账号。资源默认使用 default provider，如果要使用 alias provider，需要在资源或 data source 中显式写 `provider = aws.alias_name`。

如果继续问：“default_tags 有什么价值？”

可以答：

> default_tags 可以在 AWS provider 层统一配置标签，让通过该 provider 创建的资源自动带上组织要求的标签，比如 Environment、Owner、CostCenter、ManagedBy。它适合做成本治理和基础合规，但资源级 tags 仍然可以做更具体的设置。

如果问：“caller identity 用来干嘛？”

可以答：

> `aws_caller_identity` 是 data source，用来查询当前 Terraform 使用的 AWS 调用身份。它常用于排查账号、role、profile、CI/CD 凭证是否正确，尤其在多账号或 assume role 场景里很有用。

## 15. 本节复盘问题

### Q1：provider alias 会创建新的 AWS 账号吗？

不会。alias 只是 Terraform 里的 provider 配置别名。它不会创建 AWS 账号，也不会自动切换权限。是否访问不同账号，取决于 provider 内部配置的凭证、profile 或 assume role。

### Q2：没有写 `provider = aws.secondary` 的资源会用哪个 provider？

默认使用无 alias 的默认 provider。

### Q3：`aws_caller_identity` 是 resource 还是 data source？

是 data source。它读取当前调用身份，不创建资源。

### Q4：default tags 适合解决什么问题？

适合统一标签治理，尤其是成本归属、环境区分、负责人标识和自动化管理。

### Q5：LocalStack 可以证明真实 AWS assume role 一定成功吗？

不能。LocalStack 可以验证 Terraform 配置和调用路径，但真实 assume role 还涉及 IAM role、trust policy、权限策略、SCP、external ID 等真实 AWS 机制。

## 16. 最终记忆版

这一节记住下面这段就够了：

> Provider alias 解决“同一个 Terraform 配置里需要多个 AWS provider 配置”的问题；default tags 解决“资源统一打标签”的问题；caller identity 解决“我现在到底用谁的身份在调用 AWS”的问题。LocalStack 可以帮助本地练完整 Terraform 流程，但不能替代真实 AWS 的 IAM 权限验证。
