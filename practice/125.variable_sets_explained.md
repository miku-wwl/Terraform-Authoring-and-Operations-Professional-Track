# Terraform 考试训练 125：Variable Sets 讲解版

## 1. 这一节到底在学什么？

这一节讲的是 **HCP Terraform / Terraform Cloud 里的 Variable Sets**。

你可以先把它理解成：

> Variable Set = 一组可以被多个 workspace 复用的变量包。

在本地 Terraform 里，我们经常通过 `terraform.tfvars`、环境变量、命令行参数来传变量。但到了 HCP Terraform 这种托管平台里，很多团队会有多个 workspace：

- `app-dev`
- `app-staging`
- `app-prod`
- `network-dev`
- `network-prod`

这些 workspace 可能都需要一些共同变量，比如：

- 云区域：`AWS_REGION=ap-southeast-2`
- 账号 ID：`account_id`
- 通知邮箱：`owner_email`
- 是否开启审批：`enable_approval`
- 云访问密钥或凭据类环境变量

如果每个 workspace 都单独维护一份变量，时间久了就会变成灾难：

- 修改一次，要改很多地方
- 容易漏改
- 不同 workspace 的变量值可能不一致
- 敏感变量散落在各处，不好审计

Variable Set 就是为了解决这个问题。

---

## 2. 核心概念：Variable Set 是“变量复用机制”

Variable Set 的重点不是“变量”本身，而是 **复用**。

普通 workspace variable 只属于一个 workspace。

Variable set 可以集中维护一组变量，然后关联到：

- 整个 organization
- 某个 project
- 某些特定 workspace

所以考试看到下面这种关键词，要立刻想到 variable set：

> multiple workspaces  
> shared variables  
> common environment variables  
> avoid duplicate variable configuration  
> centrally manage variables  
> reusable variable configuration

中文理解就是：

> 多个 workspace 需要同一批变量，不想重复配置。

---

## 3. Workspace Variable vs Variable Set

这是考试很容易考的区别。

| 对比点 | Workspace Variable | Variable Set |
|---|---|---|
| 作用范围 | 单个 workspace | 可关联多个 workspace / project / organization |
| 主要用途 | 单 workspace 的专属配置 | 多 workspace 的公共配置 |
| 是否适合复用 | 不适合 | 适合 |
| 修改影响 | 只影响当前 workspace | 影响所有关联范围内的 workspace |
| 常见场景 | 某个应用独有变量 | 区域、账号、团队邮箱、公共环境变量、凭据 |

一句话记忆：

> Workspace variable 是“单点配置”；Variable set 是“共享配置包”。

---

## 4. Variable Set 可以放哪类变量？

HCP Terraform 里的变量大体可以分两类：

### 4.1 Terraform Variable

这类变量对应 Terraform 配置里的 `variable` block。

例如 Terraform 代码中有：

```hcl
variable "region" {
  type = string
}
```

那你可以在 HCP Terraform 的 variable set 里配置：

```text
region = "ap-southeast-2"
```

这类变量会作为 Terraform input variable 传给运行。

---

### 4.2 Environment Variable

这类变量是运行环境变量，常见于 provider 认证或工具行为控制。

例如 AWS provider 可以读取：

```text
AWS_ACCESS_KEY_ID
AWS_SECRET_ACCESS_KEY
AWS_REGION
```

Azure 也常见：

```text
ARM_CLIENT_ID
ARM_CLIENT_SECRET
ARM_SUBSCRIPTION_ID
ARM_TENANT_ID
```

这类变量不是 Terraform `variable` block，而是运行环境变量。

考试容易挖坑：

> Terraform variable 和 environment variable 都可以放在 variable set 里，但它们不是同一种东西。

---

## 5. Sensitive Variable：敏感变量必须特殊处理

如果变量是密码、token、secret、access key，就应该标记为 **sensitive**。

典型 sensitive 变量包括：

- 数据库密码
- API token
- AWS access key
- Azure client secret
- GitHub token
- webhook secret

标记 sensitive 的目的：

- 避免在 UI 中直接显示
- 降低日志中泄露的风险
- 帮助团队把密钥和普通配置区分开
- 降低误操作和审计风险

但是注意：

> Sensitive 不是魔法盾牌。它能降低暴露风险，但不等于密钥永远不会泄露。

如果你把 secret 明文写进 Git 仓库，那就已经是事故了。正确做法是：

- 不把 secret 提交到 Git
- 在 HCP Terraform 中配置 sensitive variable
- 或使用更高级的动态凭据 / OIDC / Vault 等机制

---

## 6. 作用域：Variable Set 不一定全局生效

另一个考试陷阱是：

> 以为 variable set 创建后，会自动对所有 workspace 生效。

这不一定对。

Variable set 要看它关联到了哪里：

### 6.1 Organization Scope

如果 variable set 关联到整个 organization，那么它可以影响 organization 下符合条件的 workspace。

适合放非常通用的配置，比如：

- 默认区域
- 公司级通知邮箱
- 通用标签策略
- 公共环境变量

### 6.2 Project Scope

如果 variable set 关联到某个 project，只会影响该 project 范围内的 workspace。

适合放某个项目组共用的配置，比如：

- 某个业务线的账号 ID
- 某个项目组的审批开关
- 某个团队的共享变量

### 6.3 Workspace Scope

如果 variable set 只绑定到特定 workspace，它只影响这些 workspace。

适合少量 workspace 共享变量，但又不想扩大影响范围。

---

## 7. 场景理解：为什么不是直接用 workspace variable？

假设有三个 workspace：

- `payment-dev`
- `payment-staging`
- `payment-prod`

它们都需要：

```text
owner_email = "platform-team@example.com"
region = "ap-southeast-2"
cost_center = "payment"
```

如果你用 workspace variable，就要配置三遍。

后来邮箱改了：

```text
owner_email = "cloud-platform@example.com"
```

你又要改三遍。

如果未来有 30 个 workspace，那就是 30 遍。

这时候 variable set 的价值就非常明显：

> 改一处，所有关联的 workspace 都可以复用更新后的变量。

这就是集中管理的价值。

---

## 8. 考试常见问法

### 问法 1：多个 workspace 共享变量

题目说：

> Several workspaces require the same environment variables. What should you use?

答案大概率是：

> Variable Set

不是 `terraform.tfvars`，也不是 `terraform fmt`，更不是 provider mirror。

---

### 问法 2：减少重复变量维护

题目说：

> A team wants to avoid configuring the same variables in every workspace.

答案：

> Create a variable set and attach it to the appropriate scope.

---

### 问法 3：敏感值管理

题目说：

> A developer committed access keys to Git. What is the better approach?

答案方向：

- 不要把 access key 提交到 Git
- 使用 HCP Terraform variables 或 variable sets
- 标记为 sensitive
- 更成熟的方案可以使用动态凭据或外部 secret 管理

---

### 问法 4：作用域判断

题目说：

> A variable set is created. Will it automatically apply to every workspace?

不能直接说一定会。

要看：

- 是否绑定到 organization
- 是否绑定到 project
- 是否绑定到 specific workspace

考试选项里如果有 “depends on the variable set scope / attachment”，通常更准确。

---

## 9. 易错点总结

### 易错点 1：把 variable set 当成 workspace variable

错误理解：

> variable set 就是 workspace variable 的另一个名字。

正确理解：

> workspace variable 是单 workspace 配置，variable set 是跨 workspace 复用的一组变量。

---

### 易错点 2：以为 sensitive variable 可以随便明文提交

错误理解：

> 反正 HCP Terraform 可以标记 sensitive，所以我先把 secret 写进 Git 也没事。

正确理解：

> secret 一旦进 Git，就已经产生泄露风险。Sensitive variable 是为了在平台中安全管理变量，不是给明文提交 Git 找借口。

---

### 易错点 3：以为 variable set 一定全局生效

错误理解：

> 创建 variable set 后，organization 里所有 workspace 都会自动使用。

正确理解：

> variable set 是否生效取决于它的关联范围和作用域。

---

### 易错点 4：分不清 Terraform variable 和 environment variable

错误理解：

> 所有变量都是 Terraform `variable` block。

正确理解：

> HCP Terraform 可以配置 Terraform variable，也可以配置 environment variable。Provider 认证信息常常是 environment variable。

---

## 10. 和真实工程的关系

在真实团队里，variable set 常用于：

- 多环境共享配置
- 多 workspace 统一区域
- 团队级 owner 信息
- 成本中心标签
- 云 provider 凭据
- CI/CD 行为开关
- 审批和通知参数
- policy / run task 相关参数

但是要注意边界：

- 公共变量适合放 variable set
- 每个 workspace 独有变量不要硬塞进公共 variable set
- secret 尽量不要长期静态化，成熟团队会考虑动态凭据
- variable set 变更可能影响多个 workspace，要小心评审

## 11. 本节选择题讲解

### 题 1

Variable set 的主要价值是什么？

正确答案：A. 在多个 workspace 间复用一组变量

解析：

Variable set 的关键词就是复用。它不是用来替代 resource，也不会自动创建 Git 仓库，更不会禁止 sensitive variable。

---

### 题 2

哪类变量更适合标记为 sensitive？

正确答案：A. 数据库密码或云访问密钥

解析：

只要是密码、token、secret、key，都应该按 sensitive 对待。普通环境名 `dev` 通常不需要 sensitive。

---

### 题 3

如果多个 workspace 需要相同的环境变量，最合适的 HCP Terraform 功能是什么？

正确答案：A. variable set

解析：

`terraform fmt` 是格式化代码，`terraform graph` 是生成依赖图，provider mirror 是 provider 分发优化，都不是变量复用机制。

---

## 12. 判断题讲解

### 判断题 1

Variable set 可以减少重复配置变量的维护成本。

答案：对。

因为它可以把共享变量集中维护，然后复用到多个 workspace。

---

### 判断题 2

Sensitive variable 的目的之一是降低敏感值在 UI 或日志中暴露的风险。

答案：对。

Sensitive 的核心价值就是降低密钥暴露风险。

---

### 判断题 3

Variable set 一旦创建，就只能绑定一个 workspace，不能用于复用。

答案：错。

Variable set 的核心就是复用。它可以按 organization、project 或 workspace 范围关联。

---

## 13. 场景题讲解

### 场景题 1

三个应用 workspace 都使用同一组云区域、账号 ID 和通知邮箱。后续邮箱变更时，团队不希望逐个 workspace 修改。你会推荐什么能力？

推荐：

> 使用 variable set。

原因：

- 把共享变量集中维护
- 按 organization、project 或 workspace 范围关联
- 后续变量变更只需要改 variable set
- 降低漏改和不一致风险

---

### 场景题 2

开发者把 access key 作为普通变量明文写在仓库中。请指出风险和更合适的处理方式。

风险：

- access key 可能泄露
- Git 历史记录会长期保存 secret
- 其他人 clone 仓库后也可能拿到密钥
- 审计和轮换成本变高

更合适方式：

- 不要把真实密钥提交到 Git
- 在 HCP Terraform workspace variable 或 variable set 中配置
- 把密钥标记为 sensitive
- 更成熟的场景可以使用 OIDC / dynamic credentials / Vault 等方案
