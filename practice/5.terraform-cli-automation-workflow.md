# Terraform CLI 自动化工作流实操教程

## 1. 教程目标

本教程用于训练 Terraform 在自动化环境中的 CLI 工作流。

重点不是创建云资源，而是掌握 Terraform 进入 CI/CD 流水线后的正确执行方式：

- 禁止交互式输入
- 检查 Terraform 代码格式
- 验证配置合法性
- 使用 Terraform test 做自动化测试
- 保存可审查的执行计划
- 审批后执行同一个 plan 文件
- 清理实验资源

本实验使用 `local_file` 资源在本地生成文件，用来模拟真实流水线里的发布清单和审批记录。

---

## 2. 为什么 Terraform 自动化不能照搬手工命令

手工操作时，常见命令是：

```sh
terraform init
terraform plan
terraform apply
```

这种方式适合人在终端里操作，因为人可以：

- 看到提示
- 输入变量
- 手动确认 `yes`
- 临时调整命令
- 判断输出是否正常

但是 CI/CD runner 没有真正的交互上下文。

如果 Terraform 在流水线里等待输入，任务就会挂起，最后超时失败。

所以自动化环境中的 Terraform 命令必须满足：

- 不等待输入
- 出错快速失败
- 输出适合保存和审查
- plan 和 apply 保持一致
- 所有关键行为都可以重复验证

---

## 3. 自动化环境中的核心命令链

推荐命令链：

```sh
terraform init -input=false
terraform fmt -check
terraform validate
terraform test
terraform plan -input=false -no-color -out=tfplan
terraform apply -auto-approve tfplan
terraform output
terraform destroy -auto-approve
```

这条命令链可以理解为一个最小版 Terraform CI/CD 流水线。

---

## 4. 命令逐个讲解

### 4.1 `terraform init -input=false`

```sh
terraform init -input=false
```

`terraform init` 用于初始化 Terraform 工作目录。

它会完成：

- 下载 provider
- 初始化 `.terraform` 目录
- 准备 backend
- 生成或更新 lock file

`-input=false` 表示禁止交互式输入。

在 CI/CD 中，这个参数很重要。

如果变量、backend、provider 配置缺失，Terraform 应该直接失败，而不是等待用户输入。

正确思路：

```text
CI/CD 中不要让 Terraform 等人输入。
缺配置就快速失败。
```

---

### 4.2 `terraform fmt -check`

```sh
terraform fmt -check
```

`terraform fmt` 用于格式化 Terraform 代码。

加上 `-check` 后，它不会自动修改文件，只会检查格式是否符合规范。

如果格式不正确，命令会失败。

这一步适合作为流水线中的代码质量门禁。

常见用途：

```text
Terraform 代码格式不正确，不允许合并。
```

---

### 4.3 `terraform validate`

```sh
terraform validate
```

`terraform validate` 用于验证 Terraform 配置是否合法。

它可以检查：

- HCL 语法是否正确
- 变量引用是否存在
- resource / data / output 结构是否正确
- provider 配置是否基本合理
- module 调用是否符合定义

需要注意：

`terraform validate` 不会真正创建资源。

它检查的是 Terraform 配置能不能被正确解析和理解。

---

### 4.4 `terraform test`

```sh
terraform test
```

`terraform test` 用于执行 Terraform 测试文件。

测试文件一般放在：

```text
tests/
```

本实验中的测试文件是：

```text
tests/automation_workflow.tftest.hcl
```

它测试的不是 AWS、Azure 或真实云资源，而是测试自动化工作流是否符合规范。

例如检查：

- 命令是否包含 `-input=false`
- `plan` 是否包含 `-no-color`
- `plan` 是否保存为 `tfplan`
- `apply` 是否使用保存后的 `tfplan`

这一步的价值是：

```text
把团队 Terraform 规范变成自动测试，而不是靠人记住。
```

---

### 4.5 `terraform plan -input=false -no-color -out=tfplan`

```sh
terraform plan -input=false -no-color -out=tfplan
```

这是本教程最关键的命令。

它包含三个重要参数。

#### `-input=false`

禁止交互式输入。

CI/CD 中不能等待用户输入变量或确认信息。

#### `-no-color`

去掉终端颜色码。

Terraform 默认输出带颜色，适合人看终端，但不适合保存到日志、审批记录或 artifact。

如果不加 `-no-color`，CI 日志或审查文件里可能出现 ANSI 控制符，影响阅读。

自动化环境中推荐加上：

```sh
-no-color
```

#### `-out=tfplan`

保存本次执行计划。

```sh
-out=tfplan
```

表示把这次 `plan` 的结果保存为一个文件。

这个文件代表 Terraform 准备执行的精确变更内容。

真实生产流程通常是：

```text
生成 plan
    ↓
保存 plan artifact
    ↓
人工审查
    ↓
审批通过
    ↓
apply 同一个 plan 文件
```

这样可以保证：

```text
审批什么，就执行什么。
```

---

### 4.6 `terraform apply -auto-approve tfplan`

```sh
terraform apply -auto-approve tfplan
```

这一步用于执行之前保存好的 plan 文件。

重点是后面的：

```sh
tfplan
```

它表示 Terraform 不重新计算计划，而是执行已经保存并审查过的计划。

推荐写法：

```sh
terraform apply -auto-approve tfplan
```

不推荐生产中直接使用：

```sh
terraform apply -auto-approve
```

区别如下：

| 命令 | 风险 |
|---|---|
| `terraform apply -auto-approve` | 重新计算并直接执行，可能和审批内容不一致 |
| `terraform apply -auto-approve tfplan` | 执行已保存的计划，审批内容和执行内容一致 |

生产环境中应该优先使用保存后的 plan 文件。

---

### 4.7 `terraform output`

```sh
terraform output
```

`terraform output` 用于查看 Terraform 配置中定义的输出。

本实验中可以输出：

- 生成的发布清单路径
- 生成的审批记录路径
- 推荐的流水线命令

在真实项目中，output 常用于给后续步骤提供信息，例如：

- service endpoint
- resource id
- storage account name
- cluster name
- deployment artifact path

---

### 4.8 `terraform destroy -auto-approve`

```sh
terraform destroy -auto-approve
```

用于销毁 Terraform 管理的资源。

本实验中销毁的是本地文件资源。

因为这是练习环境，所以可以直接使用：

```sh
-auto-approve
```

真实生产环境中，销毁操作通常需要更严格的审批。

## 7. Docker 一次性运行方式

从仓库根目录执行：

```powershell
docker run --rm `
  -v "${PWD}:/workspace" `
  -w /workspace/practice/labs/5 `
  --entrypoint sh `
  hashicorp/terraform:1.11 `
  -c "terraform init -input=false && terraform fmt -check && terraform validate && terraform test && terraform plan -input=false -no-color -out=tfplan && terraform apply -auto-approve tfplan && terraform output && terraform destroy -auto-approve"
```

这条命令做了几件事：

| 参数 | 作用 |
|---|---|
| `--rm` | 容器退出后自动删除容器 |
| `-v "${PWD}:/workspace"` | 把当前仓库挂载到容器内 |
| `-w /workspace/practice/labs/5` | 设置容器内工作目录 |
| `--entrypoint sh` | 使用 shell 执行多条命令 |
| `hashicorp/terraform:1.11` | 使用 Terraform 1.11 镜像 |
| `-c "..."` | 执行完整 Terraform 命令链 |

## 9. 正确的自动化工作流模型

可以把本实验理解为下面这个流程：

```text
Terraform 代码提交
        ↓
terraform fmt -check
        ↓
terraform validate
        ↓
terraform test
        ↓
terraform plan -input=false -no-color -out=tfplan
        ↓
保存 tfplan 作为 artifact
        ↓
人工审批
        ↓
terraform apply -auto-approve tfplan
        ↓
terraform output
```

关键点：

```text
plan 阶段生成什么，apply 阶段就执行什么。
```

不要在审批后重新裸跑 `terraform apply`。

## 10. 常见错误写法

### 错误 1：CI 中使用交互式命令

不推荐：

```sh
terraform init
terraform plan
terraform apply
```

问题：

- 可能等待输入
- CI runner 可能挂起
- 缺变量时不会快速失败

推荐：

```sh
terraform init -input=false
terraform plan -input=false -no-color -out=tfplan
terraform apply -auto-approve tfplan
```

---

### 错误 2：plan 没有保存

不推荐：

```sh
terraform plan
terraform apply
```

问题：

- `apply` 时可能重新计算计划
- 审批内容和执行内容可能不一致

推荐：

```sh
terraform plan -out=tfplan
terraform apply tfplan
```

---

### 错误 3：生产中裸用 `apply -auto-approve`

不推荐：

```sh
terraform apply -auto-approve
```

问题：

- 没有人工确认
- 没有绑定已审查的 plan
- 风险较高

推荐：

```sh
terraform apply -auto-approve tfplan
```

---

### 错误 4：CI 日志中保留颜色码

不推荐：

```sh
terraform plan
```

推荐：

```sh
terraform plan -no-color
```

如果需要保存 plan 输出给人审查，应该避免颜色控制符污染日志和 artifact。

## 12. 排障指南

### 12.1 `terraform init` 卡住

检查是否使用：

```sh
terraform init -input=false
```

CI 中缺变量或配置时，应该快速失败，不应该等待用户输入。

---

### 12.2 审批日志出现乱码

检查 `plan` 是否使用：

```sh
-no-color
```

推荐命令：

```sh
terraform plan -input=false -no-color -out=tfplan
```

---

### 12.3 `apply` 内容和审批内容不一致

不要在审批后重新执行：

```sh
terraform apply
```

应该执行：

```sh
terraform apply -auto-approve tfplan
```

---

### 12.4 `terraform test` 失败

检查 `outputs.tf` 中的 `pipeline_commands` 是否仍然包含测试期望的参数：

```text
-input=false
-no-color
-out=tfplan
terraform apply -auto-approve tfplan
```

---

### 12.5 Windows PowerShell 挂载失败

确认当前目录是仓库根目录。

PowerShell 中推荐使用：

```powershell
-v "${PWD}:/workspace"
```

如果路径包含空格，注意引号是否正确。

## 13. 清理方式

正常清理 Terraform 管理的资源：

```sh
terraform destroy -auto-approve
```

如果只想删除本地临时产物，可以在 PowerShell 中执行：

```powershell
Remove-Item -Recurse -Force practice/labs/5/.terraform, practice/labs/5/output -ErrorAction SilentlyContinue
Remove-Item -Force practice/labs/5/*.tfstate*, practice/labs/5/tfplan -ErrorAction SilentlyContinue
```

这些内容属于练习产物，不应该进入版本库：

```text
.terraform/
output/
*.tfstate
*.tfstate.*
tfplan
```

## 15. 核心结论

本实验的核心不是 `local_file`，而是 Terraform 自动化工作流。

真正要掌握的是：

```text
Terraform 在 CI/CD 中必须非交互运行。
Terraform plan 应该保存为 artifact。
审批后 apply 应该执行同一个 plan 文件。
Terraform test 可以把团队规范变成自动验证。
```

最重要的生产级习惯：

```sh
terraform plan -input=false -no-color -out=tfplan
terraform apply -auto-approve tfplan
```

这代表：

```text
审批什么，就执行什么。
```
