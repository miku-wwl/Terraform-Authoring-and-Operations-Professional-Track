# Terraform 考试训练 123：CLI-driven Run Workflow 基础（讲解版）

## 1. 本节要解决什么问题

这一节讲的是 HCP Terraform 里面的 **CLI-driven Run Workflow**。

一句话理解：

> 你还是在本地敲 `terraform plan` / `terraform apply`，但真正的 run、state、历史记录、变量管理等协作能力，由 HCP Terraform 接管。

它不是纯本地 Terraform，也不是完全由 Git 触发的 VCS 工作流。它处在两者中间：

- 入口仍然是本地 Terraform CLI；
- 运行记录进入 HCP Terraform；
- state 由 HCP Terraform 管理；
- 适合保留本地操作习惯，但又想要远端协作和审计能力的团队。

考试里这类题通常不是让你写复杂代码，而是让你判断：某个场景更适合 VCS-driven、CLI-driven，还是 API-driven。

---

## 2. 先建立大图：HCP Terraform 常见三种 workflow

HCP Terraform workspace 常见 workflow 可以粗略分成三类：

| Workflow | 触发方式 | 典型场景 | 考试关键词 |
|---|---|---|---|
| Version Control Workflow | Git 提交 / Pull Request / Webhook | 团队用 Git 审批 Terraform 变更 | Git、PR、VCS、自动 plan、代码审查 |
| CLI-driven Workflow | 本地运行 Terraform CLI | 工程师继续使用本地 CLI，但 state/run 交给 HCP Terraform | local CLI、remote run、cloud block、remote state |
| API-driven Workflow | 通过 API 触发 | 平台系统、自研门户、自动化系统触发 Terraform | API、automation、custom platform |

你可以这样记：

> VCS-driven 看 Git；CLI-driven 看本地命令；API-driven 看系统集成。

---

## 3. CLI-driven Workflow 到底是什么

CLI-driven Workflow 的重点不是“所有东西都在本地执行”。

更准确的理解是：

1. 工程师在本地目录中保留 Terraform 配置；
2. 本地运行 `terraform init`、`terraform plan`、`terraform apply`；
3. Terraform CLI 通过配置连接到 HCP Terraform workspace；
4. HCP Terraform 负责保存 run 记录、state、变量和执行历史；
5. 团队仍然可以在平台上看到这次操作的结果。

所以它适合这种团队：

> 我们暂时不想把 Terraform 工作流完全绑定到 Git PR 上，但又不希望每个人都在自己电脑上各玩各的 state。

这就是 CLI-driven 的价值。

---

## 4. `cloud` block 是本节核心关键词

CLI-driven Workflow 通常需要在 Terraform 配置中加入 `cloud` block，用来告诉 Terraform：这个配置要连接到哪个 HCP Terraform organization 和 workspace。

示例结构如下：

```hcl
terraform {
  cloud {
    organization = "example-org"

    workspaces {
      name = "network-dev"
    }
  }
}
```

这段配置表达的意思是：

- 当前 Terraform 配置要使用 HCP Terraform；
- organization 是 `example-org`；
- workspace 是 `network-dev`；
- state 和 run 会进入这个 workspace 管理。

考试看到这些词要立刻反应过来：

- `cloud` block；
- organization；
- workspace；
- local CLI；
- remote state；
- HCP Terraform run。

这大概率就是 CLI-driven Workflow。

---

## 5. CLI-driven 和 VCS-driven 的区别

这是考试最爱考的点。

### CLI-driven Workflow

特点：

- 用户在本地运行 Terraform CLI；
- 不一定依赖 GitHub / GitLab / Bitbucket 触发；
- 适合临时目录、个人实验、迁移阶段、保留 CLI 习惯的团队；
- 仍然可以把 state 和 run 历史交给 HCP Terraform。

典型题干：

> 平台团队希望工程师继续在本地运行 `terraform plan`，但统一保存 state 和 run 历史。

答案：CLI-driven Workflow。

### Version Control Workflow

特点：

- Terraform 运行由 Git 代码变更触发；
- 适合 PR 审批、分支保护、代码审查；
- 团队治理更强；
- 生产环境常见。

典型题干：

> 公司要求所有 Terraform 变更都必须经过 Git PR 审批，并且提交后自动 plan。

答案：Version Control Workflow。

### 对比记忆

| 对比点 | CLI-driven | VCS-driven |
|---|---|---|
| 谁触发 run | 本地 CLI 命令 | Git / PR / webhook |
| 是否必须接入 Git | 不必须 | 通常需要 |
| 适合什么人 | 习惯本地 CLI 的工程师 | 强调 PR 审批的团队 |
| state 是否可以在 HCP Terraform | 可以 | 可以 |
| run 历史是否进入平台 | 可以 | 可以 |

别被“CLI-driven”这个名字骗了。它不是说 state 一定在本地，也不是说没有 HCP Terraform。

## 6. 一个完整操作流程怎么想

假设你有一个 workspace：`network-dev`。

你的 Terraform 配置里写了：

```hcl
terraform {
  cloud {
    organization = "my-org"

    workspaces {
      name = "network-dev"
    }
  }
}
```

然后你在本地执行：

```bash
terraform init
terraform plan
terraform apply
```

背后的逻辑可以这样理解：

1. `terraform init` 读取 `cloud` block；
2. Terraform CLI 知道自己要连接 HCP Terraform；
3. 当前目录和远端 workspace 关联；
4. `plan` / `apply` 会进入 HCP Terraform 的运行体系；
5. state、run 历史和输出记录会保存在对应 workspace 中。

从工程视角看，这比每个人本地保存 state 要靠谱很多。

---

## 7. 考试高频混淆点

### 混淆点 1：CLI-driven 不等于 local backend

错误理解：

> CLI-driven Workflow 就是所有东西都在本地，所以 state 也在本地。

正确理解：

> CLI-driven 的入口是本地 CLI，但 state 和 run 可以由 HCP Terraform 管理。

考试看到 “local CLI + HCP Terraform state/run” 不要选 local backend，要选 CLI-driven Workflow。

---

### 混淆点 2：CLI-driven 不要求 GitHub 触发

错误理解：

> HCP Terraform 一定要接 GitHub 才能跑。

正确理解：

> VCS-driven 才强调 Git 触发；CLI-driven 可以从本地命令触发。

所以题目说“不想接入 VCS，但希望远端执行和保存 state”，这就是 CLI-driven。

---

### 混淆点 3：VCS-driven 更适合强审批链路

如果题目强调：

- Pull Request；
- branch protection；
- code review；
- merge 后自动 plan；
- 生产变更必须走 Git 审批；

那就优先选 Version Control Workflow。

CLI-driven 不是不能审计，但它不是围绕 Git PR 设计的首选方案。

---

## 8. 真实项目里怎么选

### 场景 A：个人学习 / 临时实验

你还在本地目录里试 Terraform，但想熟悉 HCP Terraform workspace。

适合：CLI-driven。

原因：不用马上接 Git，学习成本低。

---

### 场景 B：小团队迁移到 HCP Terraform

团队之前都在本地跑 Terraform，现在想先把 state 和 run 历史集中起来，但 CI/CD 还没准备好。

适合：CLI-driven。

原因：迁移阻力小，工程师还能继续用熟悉的 CLI。

---

### 场景 C：生产环境基础设施变更

公司要求所有变更必须经过 PR 审批，合并后自动 plan，并且有统一审计。

适合：Version Control Workflow。

原因：它天然围绕 Git 审批链路设计。

---

### 场景 D：自研平台触发 Terraform

公司有一个内部平台，用户点按钮申请环境，后台自动调用 Terraform 创建资源。

适合：API-driven Workflow。

原因：触发源是平台系统，不是本地 CLI，也不是 Git webhook。

---

## 9. 本节原题解析

### 选择题 1

> CLI-driven Workflow 的典型特点是什么？

正确答案：A. 本地运行 Terraform CLI 命令，但 run 与 state 由 HCP Terraform 管理。

解析：

CLI-driven 的关键就是“本地 CLI 入口 + HCP Terraform 管理 run/state”。B 说完全不需要 CLI，错；C 是 VCS-driven；D 和 Terraform Enterprise 离线安装无关。

---

### 选择题 2

> CLI-driven Workflow 通常需要在 Terraform 配置中添加什么？

正确答案：A. `cloud` block，指定 organization 和 workspace。

解析：

`cloud` block 是本节最重要的配置关键词。它把本地 Terraform 配置和 HCP Terraform 的 organization/workspace 关联起来。

---

### 选择题 3

> 哪个场景更适合 Version Control Workflow？

正确答案：A. 所有变更都来自 Git PR，并希望提交后自动 plan。

解析：

Git PR、自动 plan、代码审批，这些都是 VCS-driven Workflow 的典型信号。

---

## 10. 判断题解析

### 判断题 1

> CLI-driven Workflow 仍然要求用户理解 Terraform CLI。

答案：对。

因为用户仍然要在本地运行 `terraform init`、`terraform plan`、`terraform apply` 等命令。

---

### 判断题 2

> CLI-driven Workflow 的代码一定必须托管在 GitHub。

答案：错。

CLI-driven 不要求必须由 GitHub 触发。本地目录也可以通过 `cloud` block 连接到 HCP Terraform workspace。

---

### 判断题 3

> `cloud` block 可以把本地配置与 HCP Terraform organization 和 workspace 关联起来。

答案：对。

这就是 `cloud` block 在本节里的核心作用。

---

## 11. 场景题参考答案

### 场景题 1

> 平台团队希望工程师继续使用本地 `terraform plan`，但统一保存 state 和 run 历史。应该选择哪种 workflow？

应该选择 **CLI-driven Workflow**。

原因：

- 工程师仍然从本地 CLI 发起操作；
- HCP Terraform 负责保存 state；
- run 历史进入平台，便于审计和协作；
- 不强制要求先接入 Git PR 流程。

---

### 场景题 2

> 公司要求任何 Terraform 变更必须经过 Git 分支策略和 PR 审批。CLI-driven Workflow 是否是首选？

通常不是首选。

更适合的是 **Version Control Workflow**。

原因：

- 题干强调 Git 分支策略；
- 题干强调 PR 审批；
- 题干强调代码变更驱动 Terraform 流程；
- 这些都是 VCS-driven Workflow 的典型场景。

CLI-driven 更适合本地命令触发，不是围绕 Git PR 审批链路设计的首选方案。

---

## 12. 考试速记版

记住这几句话就够用：

1. CLI-driven：本地敲 CLI，远端管 run/state。
2. VCS-driven：Git PR / webhook 触发 Terraform run。
3. API-driven：外部系统或自研平台通过 API 触发。
4. `cloud` block：把本地配置连接到 HCP Terraform organization 和 workspace。
5. 看到 Git、PR、自动 plan，优先选 VCS-driven。
6. 看到本地 CLI、remote state、HCP run history，优先选 CLI-driven。

---

## 13. 面试表达版

如果面试官问你：

> CLI-driven Workflow 和 VCS-driven Workflow 有什么区别？

可以这样回答：

> CLI-driven Workflow 保留 Terraform CLI 作为入口，工程师仍然在本地运行 plan/apply，但 state、run history 和协作记录由 HCP Terraform 管理。VCS-driven Workflow 则是把 Git 仓库作为主要触发源，通常通过 PR、merge 或 webhook 触发 plan/apply，更适合强代码审查和生产审批链路。简单说，CLI-driven 强调本地命令入口，VCS-driven 强调 Git 驱动的团队治理。

这个回答很稳，既能体现考试点，也能体现真实工程理解。

---

## 14. 最后复盘

本节最重要的不是背定义，而是能根据题干判断 workflow。

你可以用这个判断法：

- 题干说 Git / PR / webhook / 自动 plan：选 VCS-driven。
- 题干说本地 Terraform CLI / 不想接 VCS / 远端 state：选 CLI-driven。
- 题干说自研平台 / 外部系统 / API 调用：选 API-driven。

这类题不要想复杂。考试就是考你能不能把触发方式和工作流名称对应起来。
