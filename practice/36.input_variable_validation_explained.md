# Terraform 实操训练 36 讲解：输入变量验证基础

## 1. 这一节到底在学什么？

这一节学的是 Terraform 的 **input variable validation**，也就是“输入变量验证”。

它的作用很简单：

> 在 `terraform plan` 阶段就把错误输入拦下来，不要等到 `apply` 调远端 API 时才失败。

生产环境里，这个能力非常重要。因为 CI/CD 流水线一旦跑到 apply 阶段，通常已经经历了初始化、格式检查、静态检查、审批、plan 生成等步骤。如果只是因为一个变量命名不符合规范，最后才失败，那就是白白浪费时间。

本实验用一个工作区名称来演示：  
只有符合命名规范的名称才能通过；非法名称会在 plan 阶段直接失败。

---

## 2. 为什么变量验证很重要？

Terraform 变量经常来自外部：

- CI/CD pipeline 参数
- `.tfvars` 文件
- 环境变量
- Terraform Cloud / HCP Terraform workspace variables
- 人工命令行传参
- 上游模块传入的值

如果不验证，错误值可能一路传到 provider，最后在云平台 API 那里报错。

例如：

```hcl
workspace_name = "Prod_Workspace_01"
```

这个名字人类看起来没问题，但如果团队要求只能使用小写字母、数字和连字符，它就应该被拒绝。

变量验证的价值就是：  
**把隐含规则写进代码，让规则自动执行。**

---

## 3. 基本语法

Terraform 的变量验证通常写在 `variable` 块里：

```hcl
variable "workspace_name" {
  type        = string
  description = "Workspace name must follow platform naming rules."

  validation {
    condition     = can(regex("^[a-z][a-z0-9-]{2,30}$", var.workspace_name))
    error_message = "workspace_name must start with a lowercase letter and contain only lowercase letters, numbers, and hyphens."
  }
}
```

这里有三个关键点：

1. `validation`：声明验证规则。
2. `condition`：表达式返回 `true` 才算通过。
3. `error_message`：失败时告诉用户怎么修。

---

## 4. `regex` 是干什么的？

`regex` 用来判断字符串是否符合某种模式。

例如：

```hcl
regex("^[a-z][a-z0-9-]{2,30}$", var.workspace_name)
```

这个正则大概表达：

- `^`：从字符串开头开始匹配
- `[a-z]`：第一个字符必须是小写字母
- `[a-z0-9-]{2,30}`：后面可以是小写字母、数字、连字符，长度为 2 到 30
- `$`：必须匹配到字符串结尾

所以类似这些值可以通过：

```text
dev-platform
lab36-workspace
prod01
```

这些值会失败：

```text
DevPlatform
dev_platform
36workspace
测试环境
```

失败原因分别是：

- 有大写字母
- 有下划线
- 不是以小写字母开头
- 包含中文字符

---

## 5. 为什么经常搭配 `can()`？

直接使用 `regex` 时，如果匹配失败，Terraform 可能抛出错误。  
而变量验证里的 `condition` 需要的是一个布尔值：`true` 或 `false`。

所以常见写法是：

```hcl
can(regex("^[a-z][a-z0-9-]{2,30}$", var.workspace_name))
```

`can()` 的作用是：

> 尝试执行里面的表达式；如果表达式成功，就返回 `true`；如果表达式报错，就返回 `false`。

这样非法输入不会让 Terraform 表达式本身崩掉，而是优雅地触发变量验证失败，并展示 `error_message`。

---

## 6. 这个实验的设计逻辑

本实验的场景是：

> 平台团队要求工作区名称必须符合统一命名规范。

问题是：

> 如果错误命名到 `apply` 才失败，会浪费流水线时间。

任务是：

> 用 `validation` 和 `regex` 限制名称。

预期结果：

> 合法名称通过，非法名称在 plan 阶段失败。

这正是 CI/CD 里很典型的防御式 Terraform 写法。

## 7. 标准运行流程

进入实验目录：

```sh
cd /workspace/practice/labs/36
```

初始化：

```sh
terraform init -input=false
```

格式检查：

```sh
terraform fmt -check
```

配置语法验证：

```sh
terraform validate
```

运行 Terraform 测试：

```sh
terraform test
```

生成 plan：

```sh
terraform plan -input=false -no-color -out=tfplan
```

执行 apply：

```sh
terraform apply -auto-approve tfplan
```

查看输出：

```sh
terraform output
```

清理资源：

```sh
terraform destroy -auto-approve
```

清理本地文件：

```sh
rm -f tfplan
rm -rf .terraform .terraform.lock.hcl
```

## 8. `terraform validate` 和变量 validation 的区别

这个点很容易混。

### `terraform validate`

主要检查 Terraform 配置本身有没有语法和结构问题，例如：

- HCL 语法错误
- 引用不存在的变量或资源
- provider 配置结构错误
- block 类型写错

它更像是：

> 代码能不能被 Terraform 正确理解？

---

### input variable validation

检查的是变量值是否符合你定义的业务规则，例如：

- 名称只能小写
- 环境只能是 `dev`、`test`、`prod`
- 实例规格必须在白名单内
- 数量必须在 1 到 10 之间
- CIDR 必须符合指定范围

它更像是：

> 输入值是否符合团队规则？

---

## 9. 生产环境常见验证规则

### 环境枚举

```hcl
variable "environment" {
  type = string

  validation {
    condition     = contains(["dev", "test", "stage", "prod"], var.environment)
    error_message = "environment must be one of: dev, test, stage, prod."
  }
}
```

这个可以防止有人传入：

```text
production
prd
Prod
```

看起来都像 prod，但自动化系统最好只接受统一枚举。

---

### 数量范围

```hcl
variable "replica_count" {
  type = number

  validation {
    condition     = var.replica_count >= 1 && var.replica_count <= 10
    error_message = "replica_count must be between 1 and 10."
  }
}
```

这个可以防止有人误传：

```text
0
100
```

---

### 规格白名单

```hcl
variable "instance_type" {
  type = string

  validation {
    condition     = contains(["t3.micro", "t3.small", "t3.medium"], var.instance_type)
    error_message = "instance_type must be one of: t3.micro, t3.small, t3.medium."
  }
}
```

这个适合 FinOps 和平台治理。

不然有人一不小心传个大规格，账单直接起飞。

---

### 命名规范

```hcl
variable "service_name" {
  type = string

  validation {
    condition     = can(regex("^[a-z][a-z0-9-]{2,30}$", var.service_name))
    error_message = "service_name must start with a lowercase letter and contain only lowercase letters, numbers, and hyphens."
  }
}
```

这就是本实验的核心模式。

## 10. 常见坑

### 坑 1：正则写得太复杂

正则越复杂，越容易变成玄学。

建议先从简单规则开始，例如：

```text
只能小写字母、数字、连字符
```

不要一上来就写超复杂的企业级命名宇宙大一统表达式。

---

### 坑 2：错误信息没用

坏的错误信息：

```text
Invalid value.
```

好的错误信息：

```text
workspace_name must start with a lowercase letter and contain only lowercase letters, numbers, and hyphens.
```

错误信息要告诉用户怎么修，而不是只告诉用户“你错了”。

---

### 坑 3：把所有规则都塞进一个 regex

例如你既想限制长度，又想限制前缀，又想限制环境名，又想限制后缀。  
全部塞进一个正则会很难读。

更好的方式是把规则拆开：

```hcl
validation {
  condition     = length(var.name) >= 3 && length(var.name) <= 30
  error_message = "name must be between 3 and 30 characters."
}

validation {
  condition     = can(regex("^[a-z][a-z0-9-]*$", var.name))
  error_message = "name must start with a lowercase letter and contain only lowercase letters, numbers, and hyphens."
}
```

可读性更好，报错也更精准。

---

### 坑 4：本地测试没覆盖非法输入

只测合法输入是不够的。

真正好的测试应该包含：

- 合法输入能通过
- 非法输入会失败
- 失败信息足够清楚

本实验 `terraform test` 返回 `1 passed, 0 failed`，说明验证逻辑被固化进测试了。

---

## 11. CI/CD 里的意义

变量验证放进 Terraform 后，CI/CD 会变得更稳。

典型流水线可以这样：

```sh
terraform init -input=false
terraform fmt -check
terraform validate
terraform test
terraform plan -input=false -no-color -out=tfplan
```

如果变量不合法，最好在 `plan` 阶段就失败。

这比 apply 阶段失败更好，因为：

- 不会触碰远端资源
- 不会制造半成品变更
- 不会浪费审批后的部署窗口
- 日志更容易定位问题
- 平台规则能自动执行

---

## 12. 面试/考试记忆版

可以这样记：

- `variable validation`：让错误输入在 plan 阶段提前失败。
- `condition`：必须返回 `true` 才通过。
- `error_message`：告诉用户如何修复。
- `regex`：用于命名规则、格式规则。
- `can(regex(...))`：把正则匹配失败转换成 `false`。
- 生产常见验证：命名规则、环境枚举、规格白名单、容量范围。

一句话总结：

> 输入变量验证就是把平台规则写进 Terraform，让错误输入在 plan 阶段被拦住，而不是等云平台 API apply 时才炸。

---

## 13. 本 Lab 的价值

Lab 36 看起来很基础，但它是 Terraform 工程化里非常重要的一步。

不会变量验证时，Terraform 配置只是“能跑”。  
会变量验证后，Terraform 模块才开始变得“抗误用”。

这对平台工程尤其重要，因为平台模块通常会被很多人调用。你不能假设每个调用者都知道所有隐藏规则。好的模块应该主动保护使用者，把错误尽早暴露出来。
