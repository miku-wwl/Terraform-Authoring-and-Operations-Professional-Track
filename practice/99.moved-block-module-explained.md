# Terraform 实操训练 99：moved block 迁移到模块 — 讲解版

## 1. 本节要解决什么问题？

这一节讲的是 Terraform 的 `moved` block，重点场景是：**把原来写在 root module 里的资源，迁移到子模块里，同时不让 Terraform 误以为资源要被删除重建。**

一句话理解：

> `moved` block 是 Terraform 的“状态地址迁移声明”。  
> 它告诉 Terraform：这个资源不是没了，也不是新建了，只是从一个地址搬到了另一个地址。

比如以前资源地址可能是：

```hcl
terraform_data.example
```

后来你把它封装进模块，地址变成：

```hcl
module.demo.terraform_data.example
```

如果没有 `moved` block，Terraform 可能会认为：

```text
旧资源 terraform_data.example 要销毁
新资源 module.demo.terraform_data.example 要创建
```

但实际上你只是做了代码结构重构，不应该触发真实资源重建。

---

## 2. 为什么迁移到模块很常见？

生产环境里，一开始 Terraform 代码经常比较简单，所有资源都写在 root module 里：

```text
main.tf
variables.tf
outputs.tf
```

项目变复杂以后，团队会开始拆模块：

```text
modules/
  network/
  compute/
  database/
```

这样做的好处是：

1. 代码职责更清晰。
2. 可以复用模块。
3. 更适合多人协作。
4. 更适合平台团队沉淀标准化能力。
5. 更容易做测试、版本控制和审查。

但是这里有个坑：**Terraform 的 state 记录的是资源地址，不是你心里理解的“同一个资源”。**

所以当资源从 root module 搬到 child module 后，Terraform 看到的是地址变化。

---

## 3. 核心概念：资源地址变了，资源本身没变

Terraform state 里保存的资源地址类似这样：

```text
terraform_data.example
```

当你把资源移动到模块后，地址会变成：

```text
module.migrate_demo.terraform_data.example
```

这时你需要显式告诉 Terraform：

```hcl
moved {
  from = terraform_data.example
  to   = module.migrate_demo.terraform_data.example
}
```

这段的意思是：

> state 里原来叫 `terraform_data.example` 的资源，现在应该被识别为 `module.migrate_demo.terraform_data.example`。

注意，`moved` block **不是创建资源**，也不是导入资源。它只是告诉 Terraform 如何理解 state 地址迁移。

---

## 4. `moved` block 的生产价值

### 4.1 避免误删误建

没有 `moved` block 的情况下，Terraform 可能会给出类似计划：

```text
- destroy terraform_data.example
+ create module.migrate_demo.terraform_data.example
```

这对真实云资源很危险。

比如你迁移的是：

- Azure Storage Account
- AWS RDS
- Kubernetes Namespace
- VNet / Subnet
- IAM Role
- AKS / EKS node group

错误销毁可能直接造成事故。

### 4.2 让重构可审计

使用 `moved` block 以后，代码审查者可以清楚看到：

```text
这个资源不是新增，也不是删除，而是地址迁移。
```

这比直接改代码然后让大家猜 Terraform plan 友好得多。

### 4.3 适合模块化演进

平台工程里，经常会把零散资源逐步沉淀为模块：

```text
root resource -> local module -> shared module -> versioned platform module
```

`moved` block 就是这个演进过程里的安全带。

## 5. 本实验在练什么？

原始课程要求进入：

```sh
cd /workspace/practice/labs/99
```

然后执行完整 Terraform 生命周期：

```sh
terraform init -input=false
terraform fmt -check
terraform validate
terraform test
terraform plan -input=false -no-color -out=tfplan
terraform apply -auto-approve tfplan
terraform output
terraform destroy -auto-approve
```

这套流程不是随便跑命令，而是在训练你形成 Terraform 工程闭环：

| 步骤 | 目的 |
|---|---|
| `terraform init` | 初始化 provider、模块和工作目录 |
| `terraform fmt -check` | 检查 HCL 格式是否标准 |
| `terraform validate` | 检查 Terraform 配置语法和结构 |
| `terraform test` | 执行 Terraform 原生测试 |
| `terraform plan -out=tfplan` | 生成可复用的执行计划 |
| `terraform apply tfplan` | 应用刚才保存的计划 |
| `terraform output` | 查看输出结果 |
| `terraform destroy` | 清理实验资源 |

真正要养成的习惯是：

> 不要只会 `terraform apply`，而要会完整验证、计划、执行、输出和清理。

## 6. 关键命令逐条讲解

### 6.1 `terraform init -input=false`

```sh
terraform init -input=false
```

作用：

- 初始化 Terraform 工作目录。
- 下载 provider。
- 初始化模块。
- 准备 `.terraform/` 目录。

`-input=false` 的含义是：不要交互式提问。

这在 CI/CD 中非常重要，因为流水线不能像人一样回答问题。

---

### 6.2 `terraform fmt -check`

```sh
terraform fmt -check
```

作用：

- 检查 `.tf` 文件格式是否符合 Terraform 标准。
- 不自动修改文件，只检查。

如果失败，通常本地执行：

```sh
terraform fmt -recursive
```

然后再提交代码。

这个命令看起来小，但团队协作里很重要。格式不统一会让 PR diff 很脏，影响代码审查。

---

### 6.3 `terraform validate`

```sh
terraform validate
```

作用：

- 检查配置是否合法。
- 检查变量、资源、模块引用是否基本正确。
- 不访问远程云资源。

它适合在提交前快速检查，也适合放进 CI。

---

### 6.4 `terraform test`

```sh
terraform test
```

作用：

- 执行 Terraform 原生测试文件。
- 检查模块输出、断言条件、计划或应用结果是否符合预期。

本实验验证结果是：

```text
1 passed, 0 failed
```

这说明测试断言通过。

这里你要重点理解：Terraform test 不是代替 plan/apply，而是把一些预期写成可重复执行的测试。

---

### 6.5 `terraform plan -input=false -no-color -out=tfplan`

```sh
terraform plan -input=false -no-color -out=tfplan
```

作用：

- 生成执行计划。
- 保存为 `tfplan` 文件。
- 后续 apply 使用同一个计划。

参数解释：

| 参数 | 作用 |
|---|---|
| `-input=false` | 禁用交互输入 |
| `-no-color` | 去掉彩色输出，方便日志系统读取 |
| `-out=tfplan` | 把计划保存为文件 |

CI/CD 里经常这么做，因为可以实现：

```text
plan 阶段审查 -> approve -> apply 同一个 plan
```

这比重新 plan 一次再 apply 更严谨。

---

### 6.6 `terraform apply -auto-approve tfplan`

```sh
terraform apply -auto-approve tfplan
```

作用：

- 应用之前保存的 `tfplan`。
- 不再重新计算新的计划。

这里有个细节：当你 apply 一个已经保存好的 plan 文件时，Terraform 执行的是那个文件中的计划结果。

这就是生产流水线常见的审批模型：

```text
terraform plan -> 人工/自动审批 -> terraform apply tfplan
```

---

### 6.7 `terraform output`

```sh
terraform output
```

作用：

- 查看 Terraform 输出。
- 验证模块是否暴露了预期结果。

模块化之后，output 很重要。因为 root module 往往需要从 child module 中拿结果：

```hcl
output "demo_id" {
  value = module.migrate_demo.id
}
```

---

### 6.8 `terraform destroy -auto-approve`

```sh
terraform destroy -auto-approve
```

作用：

- 销毁实验资源。
- 清理本地训练环境。

实验环境里可以这么做，但生产环境慎用。真实环境应该有权限控制、审批和保护规则。

## 7. `moved` block 常见写法

### 7.1 从 root resource 移到 module resource

```hcl
moved {
  from = terraform_data.example
  to   = module.demo.terraform_data.example
}
```

这是本节核心场景。

---

### 7.2 资源重命名

```hcl
moved {
  from = terraform_data.old_name
  to   = terraform_data.new_name
}
```

适合资源逻辑没变，只是名字变了。

---

### 7.3 模块重命名

```hcl
moved {
  from = module.old_module
  to   = module.new_module
}
```

适合模块整体改名。

---

## 8. 容易踩的坑

### 坑 1：以为移动代码就等于移动 state

不是。

Terraform 不会自动知道你只是“重构代码”。它只看到地址变了。

你需要用 `moved` block 明确告诉它：

```text
from 旧地址
to 新地址
```

---

### 坑 2：`from` 和 `to` 写错地址

地址必须是真实 Terraform 地址，不是文件路径。

错误理解：

```text
from = ./main.tf
to   = ./modules/demo/main.tf
```

这是错的。

正确关注的是资源地址：

```hcl
from = terraform_data.example
to   = module.demo.terraform_data.example
```

---

### 坑 3：moved block 不是字符串

不要写成这样：

```hcl
moved = {
  from = "terraform_data.example"
  to   = "module.demo.terraform_data.example"
}
```

正确写法是 HCL block：

```hcl
moved {
  from = terraform_data.example
  to   = module.demo.terraform_data.example
}
```

本实验的验证报告里也提到：`moved` block 需要写成真实 HCL block。

---

### 坑 4：以为 moved block 会改变云资源

`moved` block 主要影响 Terraform state 地址映射，不是直接对云资源做操作。

它的目标是让 Terraform 理解：

```text
旧地址和新地址指向同一个受管对象。
```

---

### 坑 5：迁移时不看 plan

迁移类操作一定要认真看 plan。

你希望看到的是“资源已移动”或者没有销毁重建，而不是：

```text
-/+ destroy and create replacement
```

如果 plan 出现销毁重建，要停下来检查：

- `from` 地址是否正确。
- `to` 地址是否正确。
- 模块里的资源参数是否完全一致。
- provider 配置是否变化。
- resource type 是否变化。
- resource name 是否变化。
- 是否误改了 force replacement 参数。

---

## 9. 怎么判断迁移成功？

迁移成功的核心信号是：

1. `terraform validate` 通过。
2. `terraform test` 通过。
3. `terraform plan` 不出现不必要的 destroy/create。
4. 输出结果符合预期。
5. apply 后 state 中资源地址已经变成新地址。
6. destroy 能正常清理。

可以用下面命令辅助观察 state：

```sh
terraform state list
```

迁移前可能看到：

```text
terraform_data.example
```

迁移后应该看到：

```text
module.demo.terraform_data.example
```

这就是 moved block 的价值：**资源还在，但 Terraform state 地址更新了。**

---

## 10. 面试/工作中怎么讲这个知识点？

你可以这样讲：

> 在 Terraform 项目模块化重构时，如果直接把 root module 里的资源移动到 child module，资源地址会变化。Terraform state 是按资源地址追踪对象的，所以可能误判为旧资源销毁、新资源创建。  
> 为了避免这种不必要的重建，可以使用 `moved` block，把旧地址映射到新地址。这样 Terraform 能理解这是一次 state 地址迁移，而不是资源生命周期变化。  
> 实际操作中我会通过 `terraform fmt`、`validate`、`test`、`plan` 和 saved plan apply 来验证迁移安全性，并重点检查 plan 中是否存在意外 destroy/create。

这段话适合面试，也适合你之后写项目文档。

---

## 11. 和 `terraform state mv` 的区别

Terraform 也有命令：

```sh
terraform state mv
```

它也能移动 state 地址。

但是二者风格不同：

| 方式 | 特点 |
|---|---|
| `moved` block | 声明式，写在代码里，可审查，可版本控制 |
| `terraform state mv` | 命令式，直接改 state，依赖人工操作记录 |

现代 Terraform 项目里，如果是代码重构迁移，优先考虑 `moved` block。

原因很简单：

> 平台团队更喜欢可审查、可重复、可追踪的声明式变更。

当然，复杂历史状态修复时，`terraform state mv` 仍然有用，但要谨慎。

## 12. 本节你真正应该记住的东西

### 必背一句话

> `moved` block 用来声明 Terraform state 中资源地址的迁移，常用于资源重命名、模块化重构，避免不必要的销毁和重建。

### 必会判断

看到下面这种迁移：

```text
root resource -> module resource
```

你就应该想到：

```hcl
moved {
  from = old_address
  to   = new_address
}
```

### 必会检查

迁移完成后一定看：

```sh
terraform plan
terraform state list
```

重点确认：

```text
没有误删
没有误建
地址迁移符合预期
```

## 13. 推荐练习方式

建议你按下面顺序练：

1. 先运行原始实验，确认全部通过。
2. 查看 `main.tf` 里的 `moved` block。
3. 注释掉 `moved` block，再跑 `terraform plan`，观察差异。
4. 恢复 `moved` block，再跑 `terraform plan`。
5. 用 `terraform state list` 对比资源地址变化。
6. 自己尝试把资源名字再改一次，并补充新的 `moved` block。

这样你不会只停留在“我知道 moved block 是什么”，而是能真的理解它如何保护 Terraform state。

## 15. 总结

本节的重点不是语法，而是 Terraform 工程化重构能力。

`moved` block 解决的是一个非常实际的问题：

> 代码结构变了，但真实资源不应该被重建。

当你从普通 Terraform 使用者升级到平台工程/SRE 视角时，这类能力非常关键。因为生产环境里最怕的不是你不会创建资源，而是你在重构时误删了资源。

所以这节可以理解为：

> Terraform 模块化重构的安全迁移训练。
