# Terraform 考试训练 120 讲解版：Organization、Project 与 Workspace

> 原始训练主题：HCP Terraform 的组织结构、计费边界和 workspace 作用  
> 训练类型：理论考点 + 选择题 + 判断题 + 场景题  
> 核心目标：把 HCP Terraform 里的 organization / project / workspace 三层关系讲明白，并且避免把 HCP Terraform workspace 和 Terraform CLI workspace 混在一起。

---

## 1. 本节一句话结论

HCP Terraform 的核心层级可以记成：

```text
User
  └── Organization
        └── Project
              └── Workspace
```

其中：

- **Organization**：团队协作和计费边界。
- **Project**：workspace 的分组和管理边界。
- **Workspace**：真正承载 Terraform run、变量、state、VCS 关联、执行设置和历史记录的工作单元。

考试里真正高频的是 **workspace**，因为 HCP Terraform 的 run、state、variables、VCS workflow、policy、team access 等能力，大部分都是围绕 workspace 展开的。

---

## 2. 为什么这节重要？

很多人学 Terraform 时，先接触的是本地 CLI：

```sh
terraform init
terraform plan
terraform apply
terraform workspace list
```

所以看到 HCP Terraform 里的 **workspace**，很容易以为它和本地 CLI workspace 是一个东西。

这就是本节的主要陷阱。

HCP Terraform workspace 不是简单的本地 state 命名空间。它更像一个平台化的 Terraform 运行环境，里面包含：

- 代码来源，比如 VCS repo 和 working directory
- variables / variable sets
- Terraform state
- run 历史
- plan / apply 执行记录
- 执行模式和版本设置
- 团队权限和协作记录
- policy / run task 等治理能力

所以考试看到这些词：

```text
run
state
variables
VCS
execution mode
team access
policy
workspace settings
```

大概率就是在说 **HCP Terraform workspace**。

---

## 3. Organization：组织、团队和账单边界

### 3.1 Organization 是什么？

Organization 可以理解成 HCP Terraform 里的“公司空间”或者“团队空间”。

一个 organization 下面可以有：

- 多个 project
- 多个 workspace
- 多个 team
- 多个用户
- 组织级别的设置
- 计费和订阅计划

课程里明确强调：**organization 是计费边界**。

也就是说，考试题如果问：

> billing is managed at which level?

优先选：

```text
organization
```

不要选 workspace，不要选 project，更不要选 `.terraform` 目录。

---

### 3.2 Organization 的工程类比

你可以这样理解：

```text
Organization = 公司 / 租户 / 账单账户
```

比如一家公司叫 `example-corp`，它在 HCP Terraform 里可能有一个 organization：

```text
example-corp
```

这个 organization 里面再按照团队或业务拆 project 和 workspace。

---

## 4. Project：用来组织 workspace 的分组层

### 4.1 Project 是什么？

Project 是 organization 下面的管理分组。

它的主要作用不是直接部署资源，而是把相关 workspace 放在一起，方便管理。

比如可以按业务域分：

```text
Organization: example-corp

Project: platform
  - network-dev
  - network-prod
  - shared-services-prod

Project: applications
  - app-a-dev
  - app-a-prod
  - app-b-prod

Project: security
  - iam-baseline
  - audit-logging
  - policy-enforcement
```

Project 的价值是让 organization 不至于变成一大坨 workspace 列表。

---

### 4.2 Project 不是什么？

Project 不是：

- provider
- state 文件
- Terraform module
- 自动部署资源的执行单元
- 本地 `.tf` 文件夹

它更像“文件夹”或者“业务域分组”，用于管理多个 workspace。

考试题如果问：

> Project 的主要作用是什么？

选：

```text
organize / group related workspaces
```

---

## 5. Workspace：HCP Terraform 的核心运行单元

### 5.1 Workspace 是什么？

Workspace 是 HCP Terraform 中最关键的对象。

一个 workspace 通常对应一套 Terraform 工作负载，例如：

```text
network-dev
network-prod
app-prod
security-baseline
```

它通常绑定：

- 某个 VCS 仓库
- 仓库中的某个 working directory
- 一组变量
- 一个 state
- 一组 run 历史
- Terraform 版本
- 执行模式
- 权限和协作设置

### 5.2 Workspace 为什么重要？

因为 HCP Terraform 不是简单帮你保存 `.tf` 文件，而是在 workspace 这个层面管理完整的 Terraform 生命周期：

```text
代码变更
  -> 触发 run
  -> plan
  -> policy check
  -> approval
  -> apply
  -> 更新 state
  -> 保存历史记录
```

这一整条链路，都和 workspace 绑定。

所以 workspace 是考试中的高频答案。

---

## 6. HCP Terraform workspace vs Terraform CLI workspace

这是本节最容易考的点。

| 对比项 | HCP Terraform workspace | Terraform CLI workspace |
|---|---|---|
| 所属场景 | HCP Terraform / Terraform Cloud 平台 | 本地 Terraform CLI |
| 本质 | 平台中的协作和运行单元 | 同一配置下的多个 state 命名空间 |
| 是否包含 run 历史 | 是 | 否 |
| 是否管理 VCS 触发 | 是 | 否 |
| 是否包含变量和执行设置 | 是 | 否，本地主要靠变量文件、环境变量、命令参数 |
| 是否用于团队协作 | 是 | 很弱，主要是本地命令能力 |
| 常见考点 | run、state、variables、VCS、policy、team access | `terraform workspace new/select/list` |
| 记忆方式 | “平台上的项目运行环境” | “本地 state 分支/命名空间” |

### 6.1 最重要的判断

这句话是错的：

```text
HCP Terraform workspace 与 Terraform CLI workspace 在语义上完全相同。
```

正确理解是：

```text
它们名字一样，但语义不同。
```

HCP Terraform workspace 是平台级运行单元；CLI workspace 是本地 state 命名空间。

## 8. 场景题讲解

### 场景：公司有网络、应用、安全三类 Terraform 配置，如何组织？

一个合理答案是：

```text
Organization: company-org

Project: platform-network
  Workspace:
    - network-dev
    - network-staging
    - network-prod

Project: application-services
  Workspace:
    - app-a-dev
    - app-a-prod
    - app-b-prod

Project: security-compliance
  Workspace:
    - iam-baseline
    - audit-logging
    - policy-prod
```

### 为什么这样拆？

因为 network、application、security 是不同业务域，放到不同 project 可以更清晰。

每个环境单独建 workspace，则可以让每套环境拥有独立的：

- state
- variables
- run history
- approval flow
- access control

这比把所有环境塞进一个 workspace 更安全，也更容易审计。

## 9. 选择题解析

### 题 1：HCP Terraform 中通常作为计费和团队协作边界的是哪一层？

答案：

```text
A. organization
```

解析：

Organization 是 HCP Terraform 的组织级边界，和团队、计划、计费、组织级设置相关。

---

### 题 2：HCP Terraform workspace 通常包含哪些内容？

答案：

```text
A. run 历史、变量、state、执行配置和代码来源
```

解析：

Workspace 是 HCP Terraform 的核心运行单元，里面包含 Terraform 执行所需的一整套上下文，不只是缓存或本地 shell 历史。

---

### 题 3：关于 project 的作用，哪项更准确？

答案：

```text
A. 用于组织和分组相关 workspace
```

解析：

Project 主要用于管理和分组 workspace。它不是 provider，也不是自动创建云资源的执行器。

错

## 11. 生产环境里的推荐组织方式

### 11.1 不建议的方式

```text
Organization: company
Project: default
Workspace: everything-prod
```

问题：

- state 太大
- blast radius 太大
- 权限不好拆
- plan/apply 风险高
- 审计不清晰
- 网络、应用、安全混在一起

---

### 11.2 更合理的方式

```text
Organization: company

Project: platform
  - network-dev
  - network-prod
  - shared-services-prod

Project: apps
  - app-a-dev
  - app-a-prod
  - app-b-prod

Project: security
  - iam-baseline
  - audit-prod
```

好处：

- state 边界清楚
- 权限可以按团队拆
- 环境隔离更明确
- run history 更容易审计
- 出问题时 blast radius 更小

---

## 12. 高频混淆点总结

| 混淆点 | 正确理解 |
|---|---|
| organization 是不是 workspace？ | 不是。organization 是上层组织和计费边界 |
| project 会不会直接执行 Terraform？ | 通常不会。project 主要是分组管理 |
| workspace 是不是只保存 state？ | 不只是。还包含 run、变量、VCS、执行设置等 |
| HCP workspace 和 CLI workspace 一样吗？ | 不一样。名字一样，语义不同 |
| 看到 billing 选 workspace 吗？ | 不选。billing 通常是 organization |
| 看到 run / state / variables 选什么？ | workspace |

## 15. 最后总结

这一节不难，但很容易因为 “workspace” 这个词翻车。

真正考试时，抓住三句话就够了：

```text
Organization 是团队和计费边界。
Project 用来组织多个 workspace。
Workspace 是 HCP Terraform 中承载 run、variables、state、VCS 和执行设置的核心单元。
```

尤其要记住：

```text
HCP Terraform workspace ≠ Terraform CLI workspace
```

这个区分，比背定义更重要。
