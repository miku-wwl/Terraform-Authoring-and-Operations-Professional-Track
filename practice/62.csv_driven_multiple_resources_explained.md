## 2. 为什么要用 CSV 创建多个资源？

真实工作中，很多资源清单一开始不是写在 `.tf` 文件里，而是在表格里。

例如：

- DNS 记录清单
- 用户清单
- 服务清单
- 端口清单
- 告警规则清单
- 成本中心清单
- team ownership 表
- 应用环境矩阵
- IAM 权限矩阵

如果这些数据已经存在 CSV 里，Terraform 可以用 `csvdecode()` 读取它们，再批量创建资源。

例如 CSV：

```csv
name,content
api,API service config
worker,Worker service config
web,Web service config
```

Terraform 可以根据每一行创建一个本地文件，或者在真实云平台里创建多个资源。

---

## 3. 本节的核心链路

基于 CSV 创建多个资源，通常分 4 步。

### 第一步：读取 CSV

```hcl
locals {
  rows = csvdecode(file("${path.module}/resources.csv"))
}
```

得到的是：

```hcl
list(object)
```

类似：

```hcl
[
  {
    name    = "api"
    content = "API service config"
  },
  {
    name    = "worker"
    content = "Worker service config"
  }
]
```

---

### 第二步：转换成 map

因为 `for_each` 更适合使用 map，所以通常要转换：

```hcl
locals {
  resources_by_name = {
    for row in local.rows : row.name => row
  }
}
```

得到：

```hcl
{
  api = {
    name    = "api"
    content = "API service config"
  }

  worker = {
    name    = "worker"
    content = "Worker service config"
  }
}
```

---

### 第三步：用 `for_each` 创建多个资源

```hcl
resource "local_file" "generated" {
  for_each = local.resources_by_name

  filename = "${path.module}/output/${each.key}.txt"
  content  = each.value.content
}
```

资源地址会是：

```text
local_file.generated["api"]
local_file.generated["worker"]
```

---

### 第四步：输出和测试

```hcl
output "generated_files" {
  value = {
    for name, file in local_file.generated :
    name => file.filename
  }
}
```

测试可以验证：

- CSV 行数正确
- key 正确
- 文件创建数量正确
- output 包含预期资源

---

## 4. 为什么不直接用 `count`？

`csvdecode()` 返回的是 list(object)。

初学者可能会写：

```hcl
resource "local_file" "generated" {
  count = length(local.rows)

  filename = "${path.module}/output/${local.rows[count.index].name}.txt"
  content  = local.rows[count.index].content
}
```

这能跑，但有索引风险。

如果 CSV 原来是：

```csv
name,content
api,API config
worker,Worker config
web,Web config
```

资源地址是：

```text
local_file.generated[0] -> api
local_file.generated[1] -> worker
local_file.generated[2] -> web
```

如果删除中间的 `worker`，`web` 可能从 `[2]` 变成 `[1]`。

这就是 Lab 55 讲过的 `count` 索引挑战。

所以更好的方式是：

```hcl
for_each = local.resources_by_name
```

让资源地址变成：

```text
local_file.generated["api"]
local_file.generated["worker"]
local_file.generated["web"]
```

删除 `worker` 只影响：

```text
local_file.generated["worker"]
```

不会影响 `web`。

---

## 5. CSV 第一列通常应该是稳定 key

如果你要用 CSV 驱动资源创建，必须有一个稳定字段作为 key。

例如：

```csv
name,content
api,API service config
worker,Worker service config
```

这里的 `name` 就是稳定 key。

Terraform 转换时：

```hcl
{
  for row in local.rows : row.name => row
}
```

这意味着：

```hcl
row.name
```

会成为资源地址的一部分。

例如：

```text
local_file.generated["api"]
```

所以 `name` 字段要稳定，不要乱改。

不要把这些放进 key：

- 版本号
- 日期
- 临时状态
- 随机后缀
- 会频繁变化的描述

例如不推荐：

```csv
name,content
api-v1,API config
api-v2,API config
```

版本应该是 value 字段，不应该当资源身份 key。

---

## 6. CSV key 必须唯一

如果你写：

```hcl
locals {
  resources_by_name = {
    for row in local.rows : row.name => row
  }
}
```

那么 CSV 里的 `name` 必须唯一。

错误示例：

```csv
name,content
api,config one
api,config two
```

两个 key 都是 `api`，会冲突。

这不是 Terraform 的 bug，而是数据模型有问题。

如果一个字段要作为资源身份，它就必须唯一。

生产里建议用测试验证唯一性。

---

## 7. CSV 字段默认都是字符串

Lab 60 已经讲过，CSV 是文本格式。

例如：

```csv
name,port,enabled
api,8080,true
worker,9090,false
```

解析后通常是：

```hcl
{
  name    = "api"
  port    = "8080"
  enabled = "true"
}
```

注意：

```hcl
port = "8080"
```

不是 number。

```hcl
enabled = "true"
```

也不是 bool。

需要转换：

```hcl
tonumber(row.port)
```

bool 可以用：

```hcl
row.enabled == "true"
```

如果生产里 CSV 字段要进入资源参数，类型转换必须明确。

---

## 8. 基于 CSV 创建资源的典型结构

一个比较成熟的 Terraform 写法通常是这样：

```hcl
locals {
  raw_rows = csvdecode(file("${path.module}/resources.csv"))

  rows = [
    for row in local.raw_rows : {
      name    = row.name
      content = row.content
      port    = tonumber(row.port)
      enabled = row.enabled == "true"
    }
  ]

  enabled_rows_by_name = {
    for row in local.rows : row.name => row
    if row.enabled
  }
}

resource "local_file" "generated" {
  for_each = local.enabled_rows_by_name

  filename = "${path.module}/output/${each.key}.txt"

  content = jsonencode({
    name    = each.key
    content = each.value.content
    port    = each.value.port
  })
}
```

这里做了几件事：

1. 读取 CSV。
2. 解析 CSV。
3. 显式转换类型。
4. 过滤 enabled 的行。
5. 转成 map。
6. 用 `for_each` 创建多个资源。

这就是很生产化的套路。

## 12. 如何观察 state 地址？

执行 apply 后，可以观察 Terraform 对资源的管理方式。

如果资源使用 `for_each`，地址通常类似：

```text
local_file.generated["api"]
local_file.generated["worker"]
```

这说明资源身份来自 CSV 的 key 字段。

如果你看到：

```text
local_file.generated[0]
local_file.generated[1]
```

那说明用了 `count`，需要警惕索引漂移。

生产里审 plan 时，资源地址可读性非常重要。

## 13. 修改 CSV 后 Terraform 会怎样？

### 新增一行

CSV 原来：

```csv
name,content
api,API config
worker,Worker config
```

新增：

```csv
web,Web config
```

Terraform plan 应该只新增：

```text
local_file.generated["web"]
```

---

### 删除一行

删除：

```csv
worker,Worker config
```

Terraform plan 应该只删除：

```text
local_file.generated["worker"]
```

---

### 修改一行内容

修改：

```csv
api,New API config
```

Terraform plan 应该只更新：

```text
local_file.generated["api"]
```

这就是稳定 key 的好处。

---

## 14. 修改 key 字段会怎样？

如果把 CSV 里的：

```csv
api,API config
```

改成：

```csv
backend-api,API config
```

Terraform 会认为：

```text
local_file.generated["api"] 要删除
local_file.generated["backend-api"] 要创建
```

因为资源身份变了。

所以 key 字段一定要慎重改。

如果生产里必须改 key，又想避免重建，可能需要 `moved` block 做 state 地址迁移。

---

## 15. 和 `moved` block 的关系

如果资源地址从：

```text
local_file.generated["api"]
```

改成：

```text
local_file.generated["backend-api"]
```

Terraform 默认会认为这是删除旧资源、创建新资源。

如果你只是想重命名 state 地址，可以考虑：

```hcl
moved {
  from = local_file.generated["api"]
  to   = local_file.generated["backend-api"]
}
```

这和 Lab 24 的 `moved` block 可以串起来。

生产里不要随便改 key。  
要改，就要规划迁移。

## 16. CSV 数据质量检查

基于 CSV 创建多个资源，最怕 CSV 数据不干净。

常见问题：

- key 重复
- key 为空
- 表头拼错
- 数字字段不是数字
- bool 字段写法不统一
- 有空行
- 字段里有逗号但没加引号
- 删除一行导致关键资源被销毁
- 改 key 导致资源重建

所以 CSV 驱动 Terraform 时，一定要配合测试。

---

## 17. 可以测试哪些东西？

`terraform test` 可以验证：

- CSV 解析行数是否正确
- key 是否包含预期值
- 资源数量是否正确
- output 是否包含预期文件
- 数字字段是否成功转换
- enabled 过滤是否正确
- 生成内容是否符合预期

例如测试思路：

```hcl
assert {
  condition     = length(output.generated_files) == 3
  error_message = "Expected 3 generated files."
}
```

或者：

```hcl
assert {
  condition     = contains(keys(output.generated_files), "api")
  error_message = "Expected api file to be generated."
}
```

测试不是装饰品。  
CSV 是外部输入，必须防止数据悄悄变坏。

## 18. 生产里的强校验思路

Terraform 对 local CSV 数据没有像 variable validation 那样的直接校验块，但可以用这些方式：

### 方法 1：terraform test

写测试断言，验证关键输出。

### 方法 2：check block

用 `check` 检查数据质量。

### 方法 3：precondition

在资源或输出上设置前置条件。

### 方法 4：类型转换主动失败

例如：

```hcl
tonumber(row.port)
```

如果 port 不是数字，plan 会失败。

### 方法 5：CI 中额外检查 CSV

用脚本检查 CSV：

- key 唯一
- 无空字段
- bool 合法
- 数字合法
- 表头正确

生产里可以组合使用。

## 19. CSV 和敏感数据

不要轻易把敏感数据放 CSV。

例如：

```csv
username,password
admin,SuperSecret123
```

这很危险。

原因：

- CSV 容易被提交到 Git
- Terraform state 可能保存内容
- plan/output 可能泄露
- review 时容易暴露
- 权限控制通常比 secret manager 弱

敏感信息应该走 Vault、云 Secret Manager、环境变量或受控 secret workflow。

CSV 更适合非敏感配置清单。

## 20. CSV 适合什么资源？

适合：

- 本地文件生成
- DNS records
- 简单 tag 配置
- 服务清单
- 成本中心映射
- 告警规则
- 用户组映射
- 允许列表
- demo 和训练实验

慎用：

- 数据库账号
- 密码和 token
- 生产 IAM 高权限绑定
- 复杂嵌套网络拓扑
- 强依赖顺序的配置
- 高风险生产资源

CSV 很方便，但不能代替严肃的数据模型。

---

## 21. CSV vs JSON/YAML

CSV 适合扁平表格。

```csv
name,port,env
api,8080,dev
```

JSON/YAML 更适合复杂嵌套结构。

例如：

```json
{
  "services": {
    "api": {
      "ports": [
        {
          "name": "http",
          "port": 8080
        }
      ]
    }
  }
}
```

如果你的数据越来越复杂，CSV 会变得别扭。  
这时要考虑换 JSON/YAML，或者直接用 HCL 变量。

一句话：

> 扁平清单用 CSV，复杂结构别硬塞 CSV。

## 22. 生产建议：CSV 驱动资源的安全流程

如果生产里用 CSV 创建资源，建议流程是：

1. CSV 改动走 PR。
2. CI 跑 `terraform fmt -check`。
3. CI 跑 `terraform validate`。
4. CI 跑 `terraform test`。
5. CI 跑 plan。
6. 人工审查 plan。
7. 检查新增、删除、更新是否符合预期。
8. saved plan apply。
9. 关键资源使用稳定 key。
10. 改 key 时使用 `moved` block 或明确重建审批。

这才是生产级玩法。

- 测试覆盖
- 变更审批

## 25. 面试/考试记忆版

可以这样记：

- `csvdecode()` 返回 `list(object)`。
- 创建多个资源时，不建议直接 `count` 读 CSV list。
- 更推荐用 `for` 表达式把 CSV 行转成 map。
- map key 应该来自稳定唯一字段，例如 `name`。
- 然后用 `for_each` 创建资源。
- CSV 字段通常都是 string，数字要 `tonumber()`。
- key 重复会导致 map 冲突。
- key 改名会导致资源地址变化。
- 生产里 CSV 改动要配合 `terraform test` 和 plan review。
- CSV 适合扁平清单，不适合复杂嵌套或敏感数据。

一句话总结：

> 基于 CSV 创建多个资源的关键，不是会读 CSV，而是把 CSV 数据转换成稳定、唯一、适合 `for_each` 的资源身份模型。
