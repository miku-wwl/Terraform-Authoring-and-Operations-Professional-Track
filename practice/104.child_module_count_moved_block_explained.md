# 第 104 节讲解：为模块内部资源启用 `count`

## 1. 本节要解决什么问题

这一节的核心不是单纯学习 `count`，而是学习一个更真实的 Terraform 重构场景：

> 资源原来已经存在于 state 中，现在你想把它放进 child module，并且在 module 内部启用 `count`，但又不希望 Terraform 误以为这是“删除旧资源 + 创建新资源”。

也就是说，本节真正训练的是 **Terraform 资源地址变化后的 state 迁移能力**。

原始课程说明中提到：本节来自 `doc/104.txt`，主题是在 child module 内部为 AWS 资源启用 `count`，并在 root module 使用 `moved` block 保留既有资源地址。本实验使用 LocalStack 模拟 AWS API，不访问真实 AWS。

---

## 2. 为什么启用 `count` 会影响资源地址

Terraform state 里面保存资源时，不只是保存资源本身，还会保存资源地址。

例如一个普通资源地址可能是：

```hcl
aws_s3_bucket.example
```

如果这个资源被移动到 child module 里，地址会变成：

```hcl
module.storage.aws_s3_bucket.example
```

如果 child module 内部又给资源加了 `count`，地址还会进一步变成：

```hcl
module.storage.aws_s3_bucket.example[0]
```

这就是本节的关键点：**资源实际可能还是同一个，但 Terraform 看到的地址变了。**

如果你不告诉 Terraform 旧地址和新地址之间的对应关系，Terraform 很可能会认为：

- 旧地址对应的资源应该销毁；
- 新地址对应的资源应该创建；
- 最终 plan 中出现 destroy/create，而不是 move。

这在真实生产环境中非常危险，尤其是 S3 bucket、IAM、数据库、网络资源这类不应该随便重建的对象。

---

## 3. `count` 在 child module 内部的典型写法

假设模块内部原来有一个资源：

```hcl
resource "aws_s3_bucket" "this" {
  bucket = var.bucket_name
}
```

如果你希望模块根据变量决定是否创建这个资源，可以加上 `count`：

```hcl
resource "aws_s3_bucket" "this" {
  count  = var.enabled ? 1 : 0
  bucket = var.bucket_name
}
```

这时资源地址就不再是：

```text
module.storage.aws_s3_bucket.this
```

而是：

```text
module.storage.aws_s3_bucket.this[0]
```

只要出现 `count`，Terraform 就会把资源实例化为列表形式。即使 `count = 1`，也必须带 `[0]`。

---

## 4. 为什么需要 `moved` block

`moved` block 的作用是告诉 Terraform：

> 这个资源不是要删除重建，只是 Terraform 配置里的地址变了。

典型写法如下：

```hcl
moved {
  from = aws_s3_bucket.this
  to   = module.storage.aws_s3_bucket.this[0]
}
```

这段配置的含义是：

- 原来 state 中的资源地址是 `aws_s3_bucket.this`；
- 现在资源的新地址是 `module.storage.aws_s3_bucket.this[0]`；
- Terraform 应该把 state 中的记录迁移过去；
- 不要对真实资源执行 destroy/create。

这就是安全重构 Terraform 配置的关键。

---

## 5. 本节实验的推荐理解路径

做这一节时，不要一上来背代码，按下面顺序理解会更稳：

1. 先确认原始资源地址是什么；
2. 再确认资源进入 child module 后的新地址是什么；
3. 再确认启用 `count` 后是否多了 `[0]`；
4. 最后写 `moved` block，把旧地址映射到新地址；
5. 运行 `terraform plan`，确认 plan 里是 move，而不是 destroy/create。

最重要的检查点是 `terraform plan`。

你想看到的是类似“resource has moved”的含义，而不是：

```text
- destroy
+ create
```

如果 plan 里出现销毁和重建，说明地址映射大概率写错了。

## 6. 常见目录结构

本节可能会出现类似这样的结构：

```text
practice/labs/104/
├── main.tf
├── variables.tf
├── outputs.tf
└── modules/
    └── storage/
        ├── main.tf
        ├── variables.tf
        └── outputs.tf
```

root module 负责调用 child module：

```hcl
module "storage" {
  source      = "./modules/storage"
  bucket_name = var.bucket_name
  enabled     = true
}
```

child module 负责真正声明资源：

```hcl
resource "aws_s3_bucket" "this" {
  count  = var.enabled ? 1 : 0
  bucket = var.bucket_name
}
```

root module 中使用 `moved` block 做状态迁移：

```hcl
moved {
  from = aws_s3_bucket.this
  to   = module.storage.aws_s3_bucket.this[0]
}
```

## 7. `count` 和输出值的坑

启用 `count` 后，资源引用方式也会变化。

原来可以这样写：

```hcl
output "bucket_name" {
  value = aws_s3_bucket.this.bucket
}
```

启用 `count` 后应该改成：

```hcl
output "bucket_name" {
  value = aws_s3_bucket.this[0].bucket
}
```

如果 `count` 可能为 0，就不能直接无脑引用 `[0]`，否则会报错。更稳的写法是：

```hcl
output "bucket_name" {
  value = var.enabled ? aws_s3_bucket.this[0].bucket : null
}
```

或者：

```hcl
output "bucket_name" {
  value = try(aws_s3_bucket.this[0].bucket, null)
}
```

考试和实战里都要记住：

> 一旦资源用了 `count`，引用时就要考虑索引和空列表问题。

## 8. `count` 与 `for_each` 的区别

本节重点是 `count`，但考试里经常把 `count` 和 `for_each` 放在一起考。

简单区别：

| 对比点 | count | for_each |
|---|---|---|
| 资源地址 | `resource.name[0]` | `resource.name["key"]` |
| 适合场景 | 是否创建、固定数量、简单列表 | 按 key 管理多个对象 |
| 稳定性 | 删除中间元素可能导致索引漂移 | key 稳定性更好 |
| 常见用途 | `enabled ? 1 : 0` | 多用户、多规则、多 bucket |

本节选择 `count`，更像是在训练“模块内部可选资源”的场景。

例如：

```hcl
count = var.enabled ? 1 : 0
```

这类写法非常常见。

## 10. 推荐操作流程

在实验目录中，推荐按下面流程执行：

```bash
terraform fmt -recursive
terraform init
terraform validate
terraform plan
terraform apply
terraform output
terraform destroy
```

如果你是在验证 moved block，重点看：

```bash
terraform plan
```

观察 plan 是否显示资源移动，而不是销毁重建。

如果要进一步查看 state，可以用：

```bash
terraform state list
```

你应该能看到类似：

```text
module.storage.aws_s3_bucket.this[0]
```

这说明资源已经迁移到新的 module + count 地址下。

## 11. 真实项目里的使用场景

这个能力在真实项目中非常重要。

常见场景包括：

1. 原来所有资源都写在 root module，后来要拆成 network、storage、iam、compute 等 child modules；
2. 原来资源是必选的，后来希望通过变量控制是否创建；
3. 原来一个资源是单例，后来想逐步改成可扩展结构；
4. 团队想重构 Terraform 代码，但不能影响已有生产资源；
5. 想保持 state 连续性，避免人为 `terraform import` 或手动改 state。

在生产环境里，安全重构 Terraform 的核心原则是：

> 先迁移 state 地址，再确认 plan 没有 destroy/create，最后再 apply。

不要看到代码能跑就直接 apply。Terraform 的危险不是代码写错本身，而是 plan 里出现你没看懂的销毁动作。

## 12. 考试易错点

### 易错点 1：以为 `count = 1` 不需要索引

错。

只要用了 `count`，资源地址就一定是带索引的：

```text
aws_s3_bucket.this[0]
```

不是：

```text
aws_s3_bucket.this
```

---

### 易错点 2：moved block 的 `to` 地址忘记 `[0]`

如果目标资源用了 `count`，`to` 必须写到具体实例：

```hcl
to = module.storage.aws_s3_bucket.this[0]
```

而不是：

```hcl
to = module.storage.aws_s3_bucket.this
```

---

### 易错点 3：以为 moved block 会移动真实云资源

`moved` block 移动的是 Terraform state 地址映射，不是移动真实 AWS 资源。

真实资源没有被搬家，只是 Terraform 对它的“登记地址”变了。

---

### 易错点 4：以为 LocalStack 能验证所有 AWS 行为

LocalStack 能验证本地 Terraform 调用链路，但不能完全替代真实 AWS 的权限、配额、服务细节和边界行为。

---

## 13. 一句话总结

本节核心可以总结为：

> 当你把 root module 里的资源迁移到 child module，并且在 child module 内部启用 `count` 时，资源地址会变成 `module.xxx.resource.name[0]`。为了避免 Terraform 销毁重建资源，需要用 `moved` block 把旧地址映射到新地址。

这节学会了，你就掌握了 Terraform 生产级重构里非常关键的一招。
