# Terraform 实操训练 5：自动化环境中的 Terraform CLI 工作流

## 1. 课程元数据

- 课程编号：5
- 来源文件：`doc/5.txt`
- 环境分类：`docker`
- 是否可用 Docker 运行：`yes`
- 核心主题：自动化工作流、非交互式 CLI、保存执行计划、审批门禁、Terraform test
- 前置条件：Docker、`hashicorp/terraform:1.11` 镜像、基础 Terraform CLI 经验

## 2. 知识点讲解

本节的核心是：Terraform 在自动化环境里不能照搬人工命令。人在终端里执行 `terraform plan` 和 `terraform apply` 时，可以读提示、输入变量、确认 `yes`、临时改代码；但 CI/CD runner 没有真正的交互上下文，一旦命令等待输入，流水线就会挂起或超时。

生产环境里常见的改造方向是把“操作员手工工作流”变成“自动化工作流”：代码进入 Git 后，由流水线执行格式检查、验证、计划生成、安全扫描、测试、人工审批和部署。这里最关键的 CLI 习惯是：

- `terraform init -input=false`：禁止交互式输入，缺配置就快速失败。
- `terraform plan -input=false -no-color -out=tfplan`：生成无颜色码、可审查、可保存的执行计划。
- `terraform apply -auto-approve tfplan`：只应用已经被审查过的 plan 文件。
- `terraform test`：把模块或根配置的关键行为变成可重复验证的测试。

本实验不创建 AWS 资源，而是用 `local_file` 模拟流水线产物：发布清单和审批记录。它训练的是自动化工作流本身，而不是云资源 API。

## 3. 实验设计

### 背景

平台团队要把手工 Terraform 操作迁移到 CI/CD。开发团队提交服务名、环境名和审批组后，流水线应生成可审查的发布清单，并在审批后使用同一个 plan 文件执行 apply。

### 问题

旧流程直接运行 `terraform init && terraform plan && terraform apply`，存在三个问题：

- 命令可能等待输入，CI runner 会卡住。
- plan 输出带颜色码，不适合保存为审查 artifact。
- apply 没有绑定已审查的 plan，存在“审批内容”和“实际执行内容”不一致的风险。

### 任务

在 Docker 中运行本地 Terraform 实验，验证自动化命令链：

```sh
terraform init -input=false
terraform fmt -check
terraform validate
terraform test
terraform plan -input=false -no-color -out=tfplan
terraform apply -auto-approve tfplan
terraform output
terraform destroy -auto-approve
```

### 预期结果

Terraform 生成两个本地文件：

- 发布清单：`output/dev-payments-api-release.json`
- 审批记录：`output/dev-payments-api-approval.txt`

测试确认命令链包含非交互式参数、保存 plan 文件，并且 `apply` 使用保存后的 `tfplan`。

## 5. 运行手册

从仓库根目录执行：

```powershell
docker run --rm `
  -v "${PWD}:/workspace" `
  -w /workspace/practice/labs/5 `
  --entrypoint sh `
  hashicorp/terraform:1.11 `
  -c "terraform init -input=false && terraform fmt -check && terraform validate && terraform test && terraform plan -input=false -no-color -out=tfplan && terraform apply -auto-approve tfplan && terraform output && terraform destroy -auto-approve"
```

也可以进入容器后分步执行：

```sh
cd /workspace/practice/labs/5
terraform init -input=false
terraform fmt -check
terraform validate
terraform test
terraform plan -input=false -no-color -out=tfplan
terraform apply -auto-approve tfplan
terraform output
terraform destroy -auto-approve
```

## 7. 排障指南

1. `terraform init` 卡住或等待输入：检查命令是否包含 `-input=false`，CI 中缺变量时应让任务快速失败。
2. 审批人看到乱码：检查 `plan` 是否用了 `-no-color`，否则日志和 artifact 里会混入 ANSI 控制符。
3. `apply` 内容和审批内容不一致：不要重新运行裸 `terraform apply`，必须执行 `terraform apply -auto-approve tfplan`。
4. `terraform test` 失败：检查 `outputs.tf` 里的 `pipeline_commands` 是否仍包含测试期望的自动化参数。
5. Windows PowerShell 挂载失败：确认当前目录是仓库根目录，并使用 `${PWD}:/workspace` 挂载。

## 8. 清理方式

正常清理：

```sh
terraform destroy -auto-approve
```

如果只想删除本地临时产物：

```powershell
Remove-Item -Recurse -Force practice/labs/5/.terraform, practice/labs/5/output -ErrorAction SilentlyContinue
Remove-Item -Force practice/labs/5/*.tfstate*, practice/labs/5/tfplan -ErrorAction SilentlyContinue
```
