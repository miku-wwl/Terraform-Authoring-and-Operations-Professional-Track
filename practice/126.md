## 2. 知识点讲解

Sentinel 是 HashiCorp 生态中的 policy as code 框架，可以和 HCP Terraform 集成，用代码化策略检查 Terraform plan 是否符合组织要求。课程示例中，策略可以阻止没有标签的 EC2，或者限制不允许使用高成本实例类型。考试重点是理解它位于 plan 和 apply 之间，用来在执行前做治理判断。

Sentinel policy 通常被组织成 policy set，再关联到 workspace 或一组 workspace。策略检查结果会影响 run 是否继续。Enforcement mode 决定策略失败后的处理方式：有的模式只是告警，有的可以被授权用户覆盖，有的则会硬性阻断 apply。

这类题通常不会要求你写 Sentinel 代码，而是考你能否识别“策略治理”场景。关键词包括 policy as code、compliance、tag enforcement、allowed instance types、mandatory policy、soft mandatory、advisory、hard mandatory、apply blocked。

## 3. 考试选择题

1. Sentinel 在 HCP Terraform 中主要用于什么？
   - A. 对 Terraform plan 执行策略检查
   - B. 下载 provider 插件
   - C. 压缩本地 state 文件
   - D. 生成 Git commit

2. Policy set 的作用更接近什么？
   - A. 将一组 policy 组织起来并关联到 workspace 范围
   - B. 存放 Docker 镜像
   - C. 替代 Terraform module source
   - D. 管理本地 shell alias

3. 如果策略失败后绝对不允许 apply，最接近哪类 enforcement？
   - A. hard mandatory
   - B. advisory
   - C. 本地 `terraform fmt`
   - D. `terraform output`

## 4. 判断题

1. Sentinel 属于 policy as code 能力。
2. Sentinel policy 一般在 apply 之后才第一次检查。
3. Policy set 可以帮助把策略应用到多个 workspace。

## 5. 场景题

1. 安全团队要求所有生产资源必须包含 `Owner` 和 `CostCenter` 标签，否则不能部署。请说明 Sentinel 可以如何参与这个流程。

2. 某策略失败后只提醒用户，但不阻断 apply。这个行为更接近强制阻断，还是 advisory 类型治理？

## 6. 参考答案与解析

选择题：1-A，2-A，3-A。

判断题：1-对，2-错，3-对。

场景题参考：

1. 可以把标签检查写成 Sentinel policy，放入 policy set，并关联到生产 workspace。run 经过 plan 后进入 policy check，失败时根据 enforcement mode 阻断或提示。
2. 更接近 advisory。它提供提示但不强制阻断；hard mandatory 才是不可覆盖的强制失败。

## 7. 复习提醒

- Sentinel 的位置：plan 之后、apply 之前。
- Policy set 是策略分发和绑定机制。
- Enforcement mode 是考试高频混淆点。
