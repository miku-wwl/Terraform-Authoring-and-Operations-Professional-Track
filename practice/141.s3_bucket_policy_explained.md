# 第 141 节：S3 Bucket Policy 讲解版

## 1. 本节主题

本节主题是 **S3 Bucket Policy**。

你要掌握的是：如何用 Terraform 创建一个 S3 bucket，然后用 `aws_iam_policy_document` 生成 bucket policy，最后通过 `aws_s3_bucket_policy` 把策略绑定到 bucket 上。

本实验使用 **LocalStack** 模拟 AWS API，不访问真实 AWS。LocalStack endpoint 是：

```text
http://localhost:4566
```

实验凭证固定使用：

```text
access_key = test
secret_key = test
```

这套环境适合训练 Terraform 写法、资源依赖、plan/apply/destroy 生命周期和输出验证，但不能完全替代真实 AWS 的权限评估、账号边界、Block Public Access、SCP、IAM Identity Center 等真实治理能力。

---

## 2. 一句话理解 Bucket Policy

**Bucket Policy 就是挂在 S3 bucket 上的资源策略，用来声明谁可以对这个 bucket 或 bucket 内对象做什么操作。**

它和 IAM policy 的视角不同：

- IAM policy：挂在人、角色、用户组上，回答“这个身份能做什么”。
- Bucket policy：挂在 bucket 上，回答“谁能访问这个 bucket”。

所以你可以这样记：

```text
IAM Policy     = 身份视角
Bucket Policy  = 资源视角
```

考试和实战里，bucket policy 经常用于跨账号访问、公开读控制、限制来源 VPC endpoint、强制 TLS、拒绝未加密上传等场景。

---

## 3. 本节实验目标

本节实验目标有三个：

1. 创建一个 S3 bucket。
2. 用 `aws_iam_policy_document` 生成 bucket policy JSON。
3. 用 `aws_s3_bucket_policy` 把策略绑定到 bucket。

核心闭环是：

```text
Terraform 配置
   ↓
aws_s3_bucket
   ↓
aws_iam_policy_document
   ↓
aws_s3_bucket_policy
   ↓
terraform apply
   ↓
验证 bucket policy 是否存在
   ↓
terraform destroy
```

## 4. Terraform 中的核心资源

### 4.1 `aws_s3_bucket`

`aws_s3_bucket` 用来创建 bucket。

示例：

```hcl
resource "aws_s3_bucket" "this" {
  bucket = "tf-lab-141-bucket"
}
```

这里的 `bucket` 是 bucket 名称。真实 AWS 里 bucket 名称需要全局唯一；LocalStack 里通常不用担心全局唯一冲突，但你仍然应该养成使用清晰命名的习惯。

---

### 4.2 `aws_iam_policy_document`

`aws_iam_policy_document` 是 Terraform AWS Provider 提供的 **data source**，用来用 HCL 结构生成 IAM policy JSON。

它的好处是：

- 比手写 JSON 更安全。
- 不容易漏逗号、括号。
- 可以引用 Terraform 资源属性。
- 可读性更好。
- 更适合模块化和变量化。

示例：

```hcl
data "aws_iam_policy_document" "bucket_policy" {
  statement {
    sid    = "AllowReadObjects"
    effect = "Allow"

    principals {
      type        = "AWS"
      identifiers = ["*"]
    }

    actions = [
      "s3:GetObject"
    ]

    resources = [
      "${aws_s3_bucket.this.arn}/*"
    ]
  }
}
```

注意这里的资源 ARN 是：

```hcl
"${aws_s3_bucket.this.arn}/*"
```

这表示 bucket 里面的对象。

如果只写：

```hcl
aws_s3_bucket.this.arn
```

那表示 bucket 本身，不表示 bucket 里的对象。

这就是 S3 policy 最常见的坑之一。

---

### 4.3 `aws_s3_bucket_policy`

`aws_s3_bucket_policy` 负责把 policy JSON 绑定到 bucket。

示例：

```hcl
resource "aws_s3_bucket_policy" "this" {
  bucket = aws_s3_bucket.this.id
  policy = data.aws_iam_policy_document.bucket_policy.json
}
```

这里有两个关键字段：

```hcl
bucket = aws_s3_bucket.this.id
policy = data.aws_iam_policy_document.bucket_policy.json
```

`bucket` 指定策略挂到哪个 bucket。

`policy` 指定策略内容。通常来自 `aws_iam_policy_document` 的 `.json` 输出。

---

## 5. Bucket Policy 的结构拆解

一个 bucket policy 本质上是 JSON。它通常包含这些字段：

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowReadObjects",
      "Effect": "Allow",
      "Principal": "*",
      "Action": "s3:GetObject",
      "Resource": "arn:aws:s3:::example-bucket/*"
    }
  ]
}
```

对应到考试和实战，重点字段是：

| 字段 | 含义 | 易错点 |
|---|---|---|
| `Version` | policy 语言版本 | 通常是 `2012-10-17` |
| `Statement` | 策略语句列表 | 可以有多个 statement |
| `Sid` | 语句 ID | 可选，但建议写清楚 |
| `Effect` | Allow 或 Deny | 显式 Deny 优先级最高 |
| `Principal` | 谁可以访问 | Bucket policy 里经常要写 |
| `Action` | 允许或拒绝的操作 | 例如 `s3:GetObject` |
| `Resource` | 作用到哪个资源 | bucket ARN 和 object ARN 不一样 |
| `Condition` | 条件约束 | 比如 TLS、IP、VPC Endpoint |

---

## 6. Bucket ARN 和 Object ARN 的区别

这是本节必须吃透的点。

S3 bucket 有两类常见 ARN：

### 6.1 Bucket 本身

```text
arn:aws:s3:::my-bucket
```

对应 bucket 级别操作，比如：

```text
s3:ListBucket
```

### 6.2 Bucket 里的对象

```text
arn:aws:s3:::my-bucket/*
```

对应对象级别操作，比如：

```text
s3:GetObject
s3:PutObject
s3:DeleteObject
```

所以如果你允许读取对象，应该写：

```hcl
resources = [
  "${aws_s3_bucket.this.arn}/*"
]
```

如果你允许列出 bucket，应该写：

```hcl
resources = [
  aws_s3_bucket.this.arn
]
```

很多人会把这两个写反。Terraform 不一定能在 plan 阶段帮你识别业务语义错误，所以你自己要清楚资源 ARN 的层级。

## 7. 典型完整示例

下面是一份适合本节理解的示例结构。

### 7.1 Provider 配置

```hcl
terraform {
  required_version = ">= 1.6.0"

  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 5.0"
    }
  }
}

provider "aws" {
  region                      = "us-east-1"
  access_key                  = "test"
  secret_key                  = "test"
  skip_credentials_validation = true
  skip_metadata_api_check     = true
  skip_requesting_account_id  = true

  endpoints {
    s3 = "http://localhost:4566"
  }
}
```

如果实验里还用到了 IAM、STS 等服务，也可以继续在 `endpoints` 里加。

例如：

```hcl
endpoints {
  s3  = "http://localhost:4566"
  iam = "http://localhost:4566"
  sts = "http://localhost:4566"
}
```

---

### 7.2 创建 bucket

```hcl
resource "aws_s3_bucket" "this" {
  bucket = "tf-lab-141-bucket"
}
```

---

### 7.3 生成 bucket policy

```hcl
data "aws_iam_policy_document" "bucket_policy" {
  statement {
    sid    = "AllowReadObjects"
    effect = "Allow"

    principals {
      type        = "AWS"
      identifiers = ["*"]
    }

    actions = [
      "s3:GetObject"
    ]

    resources = [
      "${aws_s3_bucket.this.arn}/*"
    ]
  }
}
```

这段配置的意思是：

```text
允许所有 Principal 对 bucket 中的对象执行 s3:GetObject
```

注意：真实 AWS 中，如果你写 `Principal = "*" + s3:GetObject`，这接近公开读，需要结合 Block Public Access、业务要求和安全审查。实验里可以这么练，但生产环境不能随手这么写。

---

### 7.4 绑定 bucket policy

```hcl
resource "aws_s3_bucket_policy" "this" {
  bucket = aws_s3_bucket.this.id
  policy = data.aws_iam_policy_document.bucket_policy.json
}
```

这一步才是真正把策略挂到 bucket 上。

`aws_iam_policy_document` 只是生成 JSON，不会自己生效。

---

### 7.5 输出 policy

```hcl
output "bucket_name" {
  value = aws_s3_bucket.this.bucket
}

output "bucket_arn" {
  value = aws_s3_bucket.this.arn
}

output "bucket_policy_json" {
  value = data.aws_iam_policy_document.bucket_policy.json
}
```

输出可以帮助你确认 Terraform 生成的 policy 是否符合预期。

## 9. LocalStack 边界

本节虽然可以用 LocalStack 完成核心 Terraform 闭环，但你要清楚边界。

LocalStack 可以帮助你验证：

- Terraform 配置语法是否正确。
- Provider endpoint 是否能连通。
- S3 bucket 是否能创建。
- Bucket policy 是否能绑定。
- Terraform state 是否能记录资源。
- destroy 是否能清理资源。

LocalStack 不能完全替代真实 AWS 验证：

- 真实 IAM 权限评估。
- AWS Organizations SCP。
- S3 Block Public Access 的完整行为。
- 跨账号 Principal 信任关系。
- VPC Endpoint condition 的真实网络路径。
- CloudTrail 审计。
- KMS 加密权限联动。
- 生产级安全基线。

所以本节实验的定位是：

```text
LocalStack 验证 Terraform 声明路径
真实 AWS 验证安全语义和生产行为
```

这个边界要记住，别把本地模拟当成生产等价环境。

## 10. 常见易错点

### 10.1 只写了 bucket ARN，忘了对象 ARN

错误示例：

```hcl
actions = ["s3:GetObject"]

resources = [
  aws_s3_bucket.this.arn
]
```

`GetObject` 是对象级动作，需要：

```hcl
resources = [
  "${aws_s3_bucket.this.arn}/*"
]
```

---

### 10.2 允许 `s3:ListBucket` 时写了对象 ARN

错误示例：

```hcl
actions = ["s3:ListBucket"]

resources = [
  "${aws_s3_bucket.this.arn}/*"
]
```

`ListBucket` 是 bucket 级动作，需要：

```hcl
resources = [
  aws_s3_bucket.this.arn
]
```

---

### 10.3 以为 `aws_iam_policy_document` 会自动创建策略

不会。

`aws_iam_policy_document` 是 data source，只负责生成 JSON。

真正绑定到 bucket 的是：

```hcl
resource "aws_s3_bucket_policy" "this" {
  bucket = aws_s3_bucket.this.id
  policy = data.aws_iam_policy_document.bucket_policy.json
}
```

---

### 10.4 生产环境随便写 `Principal = "*"`

实验可以用 `*` 理解公开访问逻辑，但生产环境要非常谨慎。

尤其是这个组合：

```text
Principal = "*"
Action    = "s3:GetObject"
Resource  = bucket/*
```

这通常意味着公开读取对象。真实项目里必须确认：

- 是否真的需要公开读。
- 是否启用了 Block Public Access。
- 是否有 CloudFront/OAC 更合适。
- 是否需要限制来源 IP、VPC Endpoint、TLS 或账号。
- 是否有审计和审批记录。

---

### 10.5 混淆 IAM policy 和 bucket policy

IAM policy 绑定身份。

Bucket policy 绑定 bucket。

如果题目强调：

```text
grant another AWS account access to this bucket
control access at the bucket resource level
public read for objects
deny unencrypted uploads to this bucket
```

大概率在说 bucket policy。

如果题目强调：

```text
attach permissions to user / role / group
developer role can read bucket
EC2 instance role can put object
```

大概率在说 IAM policy。

## 11. 面试和考试记忆法

你可以这样背：

```text
Bucket Policy = S3 bucket 门口的保安规则
Principal     = 谁要进门
Action        = 进来干什么
Resource      = 进哪扇门 / 哪个房间
Condition     = 满足什么条件才能进
```

其中：

```text
bucket ARN      = bucket 门口
bucket ARN /*   = bucket 里的对象
```

考试中最爱考的就是：

```text
s3:ListBucket  -> bucket ARN
s3:GetObject   -> object ARN，也就是 bucket ARN/*
```

## 12. 和真实项目的关系

真实项目中，bucket policy 常见用途包括：

### 12.1 允许 CloudFront 访问 S3

通过 CloudFront Origin Access Control 或旧的 OAI 限制只有 CloudFront 能读对象。

### 12.2 强制使用 TLS

拒绝非 HTTPS 请求：

```json
"Condition": {
  "Bool": {
    "aws:SecureTransport": "false"
  }
}
```

这类策略通常用 `Deny`，因为显式 Deny 优先级最高。

### 12.3 限制 VPC Endpoint

只允许从指定 VPC Endpoint 访问 S3。

常见条件键：

```text
aws:sourceVpce
```

### 12.4 强制加密上传

拒绝没有指定服务端加密头的 `PutObject`。

常见动作：

```text
s3:PutObject
```

常见条件：

```text
s3:x-amz-server-side-encryption
```

### 12.5 跨账号访问

允许另一个 AWS 账号的 role 访问 bucket。

Principal 通常会写成：

```json
"AWS": "arn:aws:iam::123456789012:role/SomeRole"
```

## 16. 本节结论

第 141 节的核心不是“会创建一个 bucket”，而是理解 **S3 bucket policy 的资源策略模型**。

你真正要拿走的是：

1. Bucket policy 是 S3 的资源级访问控制。
2. Terraform 里推荐用 `aws_iam_policy_document` 生成 policy JSON。
3. 用 `aws_s3_bucket_policy` 把 JSON 绑定到 bucket。
4. bucket ARN 和 object ARN 必须分清。
5. LocalStack 适合练 Terraform 流程，但真实安全语义仍要在真实 AWS 验证。

这节学明白后，你再看 CloudFront + S3、跨账号日志桶、VPC Endpoint 访问 S3、强制 TLS、强制加密上传，就不会觉得乱了。
