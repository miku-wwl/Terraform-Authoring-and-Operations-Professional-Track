# 第 166 节讲解版：挑战五解法一——创建 VPC 与子网基础资源

## 1. 本节定位

这一节是综合挑战五的第一个解法，主题是：**使用 Terraform 创建 VPC 与子网基础资源**。

它不是单纯背 `aws_vpc` 或 `aws_subnet` 的参数，而是训练你在挑战题里快速完成一条闭环：

```text
读题 → 找资源 → 修配置 → plan → apply → 输出 → 验证 → destroy
```

在真实项目里，VPC 和 subnet 是 AWS 网络层的地基。后面的 EC2、ALB、RDS、EKS、NAT Gateway、安全组、路由表，基本都要挂在这个网络基础之上。

本实验使用 LocalStack 模拟 AWS API，不访问真实 AWS 账号。它适合训练 Terraform 写法、资源依赖、state 变化和验证流程，但不能替代真实 AWS 的网络连通性验证。

---

## 2. 这一节到底在练什么

本节核心不是“会不会创建一个 VPC”，而是下面几个能力：

1. **能识别基础网络资源**
   - `aws_vpc`
   - `aws_subnet`
   - 可能还会涉及 tag、CIDR、AZ、输出值

2. **能把题目要求翻译成 Terraform 资源**
   - 题目说“创建 VPC”，你要想到 `aws_vpc`
   - 题目说“创建子网”，你要想到 `aws_subnet`
   - 题目说“输出资源信息”，你要想到 `output`

3. **能完成 Terraform 标准闭环**
   - `terraform fmt`
   - `terraform init`
   - `terraform validate`
   - `terraform plan`
   - `terraform apply`
   - 验证脚本
   - `terraform destroy`

4. **能理解 LocalStack 的边界**
   - LocalStack 可以模拟创建资源的 API 路径
   - 但不能代表真实 AWS 网络真的可用
   - 真实 AWS 里还要考虑路由表、IGW、NAT、安全组、NACL、AZ 可用性等问题

---

## 3. VPC 与子网的关系

可以把 VPC 理解为一块私有网络空间，把 subnet 理解为这块空间里的更小网段。

```text
VPC: 10.0.0.0/16
├── public subnet:  10.0.1.0/24
├── private subnet: 10.0.2.0/24
└── db subnet:      10.0.3.0/24
```

在 Terraform 中，subnet 通常依赖 VPC：

```hcl
resource "aws_vpc" "main" {
  cidr_block = "10.0.0.0/16"
}

resource "aws_subnet" "main" {
  vpc_id     = aws_vpc.main.id
  cidr_block = "10.0.1.0/24"
}
```

关键点是：

- `aws_vpc.main.id` 会形成隐式依赖；
- Terraform 会先创建 VPC，再创建 subnet；
- subnet 的 CIDR 必须落在 VPC 的 CIDR 范围内；
- 后续很多资源会继续引用 subnet id。

## 4. 挑战题里的解题思路

遇到这类挑战题，不要一上来乱写代码。建议按下面顺序拆：

### 第一步：定位资源类型

看到题目是“创建 VPC 与子网基础资源”，优先想到：

```text
aws_vpc
aws_subnet
output
```

如果题目要求 tag，就补 `tags`。

如果题目要求多个 subnet，再考虑：

```text
count
for_each
locals
variables
```

本节是解法一，通常先用最直接的写法完成基础闭环，不必过度抽象。

---

### 第二步：检查 provider 配置

因为本项目使用 LocalStack，所以 AWS Provider 一般会指向本地 endpoint：

```hcl
provider "aws" {
  region                      = "us-east-1"
  access_key                  = "test"
  secret_key                  = "test"
  skip_credentials_validation = true
  skip_metadata_api_check     = true
  skip_requesting_account_id  = true

  endpoints {
    ec2 = "http://localhost:4566"
  }
}
```

这里的重点不是背每个参数，而是理解：

- `access_key` / `secret_key` 是假凭证；
- `endpoint` 指向 LocalStack；
- skip 参数是为了绕过真实 AWS 校验；
- 本实验不会连真实 AWS。

---

### 第三步：声明 VPC

VPC 最核心的是 `cidr_block`。

```hcl
resource "aws_vpc" "main" {
  cidr_block = "10.0.0.0/16"

  tags = {
    Name = "tf-lab-166-vpc"
  }
}
```

考试和真实项目里常见关注点：

- CIDR 是否合理；
- 是否和已有网络冲突；
- 是否预留足够地址空间；
- tag 是否符合组织规范；
- 是否需要 DNS 支持，例如 `enable_dns_support`、`enable_dns_hostnames`。

---

### 第四步：声明 subnet

子网必须绑定到 VPC：

```hcl
resource "aws_subnet" "main" {
  vpc_id     = aws_vpc.main.id
  cidr_block = "10.0.1.0/24"

  tags = {
    Name = "tf-lab-166-subnet"
  }
}
```

这里最重要的是 `vpc_id = aws_vpc.main.id`。

这行代码同时表达了两个含义：

1. subnet 属于这个 VPC；
2. Terraform 创建顺序上，subnet 依赖 VPC。

---

### 第五步：输出资源信息

挑战题经常要求“输出资源信息”，这不是装饰，而是验证入口。

常见 output：

```hcl
output "vpc_id" {
  value = aws_vpc.main.id
}

output "subnet_id" {
  value = aws_subnet.main.id
}

output "vpc_cidr_block" {
  value = aws_vpc.main.cidr_block
}

output "subnet_cidr_block" {
  value = aws_subnet.main.cidr_block
}
```

为什么要输出？

- 方便人眼检查；
- 方便脚本验证；
- 方便后续模块或练习引用；
- 方便观察 state 中资源属性。

## 5. 推荐运行流程

进入实验目录：

```sh
cd /workspace/practice/labs/166
```

格式化检查：

```sh
terraform fmt -check
```

初始化：

```sh
terraform init
```

语法和 provider 配置校验：

```sh
terraform validate
```

生成执行计划：

```sh
terraform plan
```

应用配置：

```sh
terraform apply
```

查看输出：

```sh
terraform output
```

销毁资源：

```sh
terraform destroy
```

如果是 CI 或自动化场景，可以使用：

```sh
terraform plan -input=false -no-color -out=tfplan
terraform apply -auto-approve tfplan
terraform destroy -auto-approve
```

## 6. 你应该重点观察什么

### 观察 1：plan 中的创建顺序

正常情况下，你会看到 Terraform 计划创建：

```text
aws_vpc.main
aws_subnet.main
```

虽然 plan 不一定按严格顺序展示，但依赖图里 subnet 一定依赖 VPC。

---

### 观察 2：apply 后的 state

apply 后，Terraform state 会记录资源地址：

```text
aws_vpc.main
aws_subnet.main
```

这些地址很重要。后续如果改名、加 `count`、迁移到模块，就可能涉及 `moved block` 或 `terraform state mv`。

---

### 观察 3：output 是否稳定

如果 output 名称写错，验证脚本可能失败。

例如验证脚本期待：

```text
vpc_id
subnet_id
```

但你写成：

```text
main_vpc_id
main_subnet_id
```

Terraform 本身可能通过，但实验验证会失败。

---

## 7. LocalStack 能验证什么，不能验证什么

### LocalStack 能验证

- Terraform provider endpoint 配置是否可用；
- `aws_vpc` / `aws_subnet` 资源是否能走通创建流程；
- Terraform state 是否正确记录；
- output 是否能返回资源属性；
- destroy 是否能清理资源；
- 基础依赖关系是否正确。

### LocalStack 不能完全验证

- 真实 AWS VPC 网络连通性；
- 跨 AZ 子网真实可用性；
- 路由表、IGW、NAT Gateway 的真实转发行为；
- 安全组和 NACL 的真实流量效果；
- 真实账号权限、Service Quota、区域限制；
- 生产环境的 IP 地址规划冲突。

所以这节更像是“Terraform 资源建模训练”，不是完整 AWS 网络工程验收。

## 8. 常见错误与排查

### 错误 1：subnet CIDR 不在 VPC CIDR 内

错误示例：

```hcl
resource "aws_vpc" "main" {
  cidr_block = "10.0.0.0/16"
}

resource "aws_subnet" "main" {
  vpc_id     = aws_vpc.main.id
  cidr_block = "192.168.1.0/24"
}
```

问题：`192.168.1.0/24` 不属于 `10.0.0.0/16`。

正确方向：让 subnet CIDR 落在 VPC CIDR 范围内。

---

### 错误 2：忘记绑定 `vpc_id`

`aws_subnet` 必须知道自己属于哪个 VPC。

正确写法：

```hcl
vpc_id = aws_vpc.main.id
```

---

### 错误 3：LocalStack 没启动

如果 endpoint 是 `http://localhost:4566`，但 LocalStack 没启动，`terraform apply` 会连接失败。

排查方向：

```sh
curl http://localhost:4566/_localstack/health
```

或者检查 Docker 容器：

```sh
docker ps
```

---

### 错误 4：provider 仍在访问真实 AWS

如果忘记配置 endpoint，Terraform 可能尝试访问真实 AWS。

排查方向：

- 检查 `provider "aws"` 是否配置了 LocalStack endpoint；
- 检查是否误用了真实 profile；
- 检查环境变量里是否存在真实 AWS 凭证；
- 确认本实验不要使用真实账号。

---

### 错误 5：output 名称和验证脚本不一致

Terraform apply 成功不代表实验通过。

如果验证脚本读取固定 output 名称，你就要按题目要求命名。

---

## 9. 真实 AWS 项目中的延伸

真实项目里，VPC 和 subnet 通常不会只写这么简单。

常见生产配置包括：

```text
VPC
├── public subnets
├── private app subnets
├── private db subnets
├── internet gateway
├── NAT gateway
├── route tables
├── security groups
├── NACLs
├── VPC endpoints
└── flow logs
```

如果是 EKS 项目，还要考虑：

- subnet tags 是否满足 EKS / Load Balancer Controller 发现规则；
- public subnet 和 private subnet 如何划分；
- node group 放在哪些 subnet；
- Pod IP 是否会耗尽；
- NAT Gateway 成本；
- VPC CNI 的 IP 消耗；
- 是否需要 prefix delegation。

所以本节是基础地基，后面可以继续扩展到更完整的 AWS 网络架构。

## 10. 考试与实战记忆点

### 记忆点 1：VPC 是网络边界

VPC 定义一个隔离的网络空间。

### 记忆点 2：Subnet 是 VPC 内的网段

Subnet 必须属于某个 VPC。

### 记忆点 3：Terraform 通过引用建立依赖

```hcl
vpc_id = aws_vpc.main.id
```

这既是资源绑定，也是依赖关系。

### 记忆点 4：output 是验证入口

挑战题里要求输出，通常是为了让验证脚本检查结果。

### 记忆点 5：LocalStack 是模拟，不是生产网络验收

LocalStack 能练 Terraform 闭环，但不能证明真实 AWS 网络行为一定正确。
