# Terraform 实操训练 51 讲解：map 数据类型

## 1. 这一节到底在学什么？

Lab 51 学的是 Terraform 里的 `map` 数据类型。

上一节 Lab 50 讲的是 `list`：一组有顺序的值。  
这一节的 `map` 则是另一种更适合生产建模的数据结构：**用 key 管理 value**。

一句话理解：

> `map` 是 Terraform 里的键值对集合，适合表达“用名字定位对象”的配置。

例如：

```hcl
{
  dev  = "small"
  test = "medium"
  prod = "large"
}
```

这里的 `dev`、`test`、`prod` 是 key，右边的 `"small"`、`"medium"`、`"large"` 是 value。

---

## 2. 什么是 map？

Terraform 中的 map 通常长这样：

```hcl
variable "instance_sizes" {
  type = map(string)

  default = {
    dev  = "small"
    test = "medium"
    prod = "large"
  }
}
```

它表达的是：

```text
dev  环境使用 small
test 环境使用 medium
prod 环境使用 large
```

map 的核心特点是：

1. 每个元素都有 key。
2. key 必须唯一。
3. 通过 key 访问 value。
4. 不靠位置索引。
5. 很适合和 `for_each` 搭配。
6. 适合表达命名对象集合。

---

## 3. map 和 list 的核心区别

上一节的 list 是这样：

```hcl
["dev", "test", "prod"]
```

访问方式是索引：

```hcl
var.environments[0]
```

而 map 是这样：

```hcl
{
  dev  = "small"
  test = "medium"
  prod = "large"
}
```

访问方式是 key：

```hcl
var.instance_sizes["dev"]
```

对比一下：

| 类型 | 访问方式 | 是否有稳定 key | 适合场景 |
|---|---|---|---|
| list | 按索引访问 | 没有 | 顺序有意义的数据 |
| map | 按 key 访问 | 有 | 命名配置、环境配置、服务配置 |

一句话：

> list 适合“第几个”；map 适合“叫什么”。

---

## 4. map 的基本访问方式

示例：

```hcl
locals {
  instance_sizes = {
    dev  = "small"
    test = "medium"
    prod = "large"
  }
}

output "dev_size" {
  value = local.instance_sizes["dev"]
}
```

输出：

```text
dev_size = "small"
```

也可以用点号访问简单 key：

```hcl
local.instance_sizes.dev
```

但生产里更推荐方括号访问：

```hcl
local.instance_sizes["dev"]
```

因为它更明确，也能处理动态 key。

---

## 5. 动态 key 访问

map 很适合根据变量动态读取配置。

例如：

```hcl
variable "environment" {
  type    = string
  default = "dev"
}

locals {
  instance_sizes = {
    dev  = "small"
    test = "medium"
    prod = "large"
  }

  selected_size = local.instance_sizes[var.environment]
}
```

如果：

```hcl
var.environment = "prod"
```

那么：

```hcl
local.selected_size = "large"
```

这就是生产里非常常见的模式：

> 根据环境名，从 map 里取对应配置。

---

## 6. `lookup()`：带默认值的 map 读取

直接用 key 访问时，如果 key 不存在，会报错。

例如：

```hcl
local.instance_sizes["staging"]
```

如果 map 里没有 `staging`，Terraform 会失败。

这时可以用 `lookup()`：

```hcl
lookup(local.instance_sizes, var.environment, "small")
```

意思是：

> 从 `local.instance_sizes` 里查 `var.environment`，如果没有，就返回 `"small"`。

完整示例：

```hcl
locals {
  selected_size = lookup(local.instance_sizes, var.environment, "small")
}
```

这在处理可选环境、默认配置时很有用。

但别滥用默认值。  
如果生产环境 key 写错，静默 fallback 可能掩盖配置错误。

---

## 7. `keys()`：获取所有 key

`keys()` 可以拿到 map 的所有 key。

```hcl
locals {
  instance_sizes = {
    dev  = "small"
    test = "medium"
    prod = "large"
  }

  environments = keys(local.instance_sizes)
}
```

结果类似：

```hcl
["dev", "prod", "test"]
```

注意：map 本身不是按你肉眼书写顺序来表达业务顺序的。  
如果顺序很重要，不要依赖 map，应该用 list 或额外的顺序字段。

`keys()` 常用于：

- 输出支持的环境
- 变量校验
- 生成资源集合
- 遍历 map

---

## 8. `values()`：获取所有 value

`values()` 可以拿到 map 的所有 value。

```hcl
locals {
  sizes = values(local.instance_sizes)
}
```

结果类似：

```hcl
["small", "large", "medium"]
```

同样要注意顺序问题。  
如果你关心顺序，不要盲目依赖 `values()` 的输出顺序。

---

## 9. `merge()`：合并 map

`merge()` 可以合并多个 map。

例如：

```hcl
locals {
  base_tags = {
    ManagedBy = "Terraform"
    Project   = "training"
  }

  env_tags = {
    Environment = "dev"
  }

  tags = merge(local.base_tags, local.env_tags)
}
```

结果是：

```hcl
{
  ManagedBy   = "Terraform"
  Project     = "training"
  Environment = "dev"
}
```

生产里 `merge()` 非常常见，尤其是 tag 管理。

例如：

```hcl
tags = merge(
  local.common_tags,
  {
    Name = "app-server"
  }
)
```

这比到处重复写 tag 更干净。

---

## 10. map 和 `for_each`：生产级重点

map 最适合搭配 `for_each`。

示例：

```hcl
variable "services" {
  type = map(string)

  default = {
    api    = "8080"
    worker = "9090"
    web    = "3000"
  }
}

resource "local_file" "service" {
  for_each = var.services

  filename = "${path.module}/output/${each.key}.txt"
  content  = "port=${each.value}"
}
```

这会创建：

```text
local_file.service["api"]
local_file.service["worker"]
local_file.service["web"]
```

注意这些资源地址是按 key 管理的，不是按数字索引管理。

这就是 map 的生产价值：

> key 稳定，资源地址稳定。

---

## 11. 为什么 map + for_each 通常比 list + count 更稳？

上一节讲过 list + count 的风险：索引漂移。

如果用 list：

```hcl
["api", "worker", "web"]
```

资源地址可能是：

```text
resource.example[0]
resource.example[1]
resource.example[2]
```

如果删除中间的 `"worker"`，后面的索引会变化，可能导致不必要的更新或替换。

而 map + for_each 是：

```hcl
{
  api    = "8080"
  worker = "9090"
  web    = "3000"
}
```

资源地址是：

```text
resource.example["api"]
resource.example["worker"]
resource.example["web"]
```

删除 `worker` 只影响：

```text
resource.example["worker"]
```

不会让 `web` 从 `[2]` 变成 `[1]`。

这就是生产环境里更喜欢 map + for_each 的原因。

---

## 12. map 的类型约束

最简单的 map：

```hcl
type = map(string)
```

表示所有 value 都是 string。

也可以是：

```hcl
type = map(number)
```

例如：

```hcl
variable "service_ports" {
  type = map(number)

  default = {
    api    = 8080
    worker = 9090
    web    = 3000
  }
}
```

更复杂一点，可以是 map object：

```hcl
variable "services" {
  type = map(object({
    port     = number
    replicas = number
  }))

  default = {
    api = {
      port     = 8080
      replicas = 2
    }

    worker = {
      port     = 9090
      replicas = 1
    }
  }
}
```

这在生产模块里非常常见。

---

## 13. map object：更像真实生产配置

很多真实配置不是简单 key-value，而是 key 对应一个结构化对象。

例如：

```hcl
variable "services" {
  type = map(object({
    image = string
    port  = number
    tier  = string
  }))
}
```

调用方可以传：

```hcl
services = {
  api = {
    image = "api:v1"
    port  = 8080
    tier  = "backend"
  }

  web = {
    image = "web:v1"
    port  = 3000
    tier  = "frontend"
  }
}
```

这样你就可以用：

```hcl
each.key
each.value.image
each.value.port
each.value.tier
```

来创建资源。

这就是 Terraform 模块设计里非常重要的建模方式。

## 18. 常见坑

### 坑 1：访问不存在的 key

如果 map 里没有：

```hcl
"staging"
```

你却访问：

```hcl
var.config["staging"]
```

Terraform 会报错。

可以用 `lookup()` 处理默认值，但生产里要小心静默 fallback。

---

### 坑 2：把 map 当 list 用

map 不能这样访问：

```hcl
var.config[0]
```

它应该用 key：

```hcl
var.config["dev"]
```

---

### 坑 3：key 命名不稳定

map 的 key 一旦用于 `for_each`，就会变成资源地址的一部分。

例如：

```text
local_file.service["api"]
```

如果你把 key 从 `api` 改成 `backend-api`，Terraform 会认为旧资源要删，新资源要建。

所以 key 要慎重设计。

---

### 坑 4：把顺序逻辑放进 map

map 适合按 key 管理，不适合表达严格顺序。

如果你需要“第一步、第二步、第三步”，更适合 list。

---

### 坑 5：value 类型不统一

如果声明：

```hcl
type = map(string)
```

就不要混入 number 或 object。

如果 value 是复杂配置，应该声明成：

```hcl
map(object({...}))
```

---

## 19. map key 设计建议

生产里 map key 很重要，因为它常常影响资源地址。

好的 key 应该：

- 稳定
- 简短
- 可读
- 不频繁变化
- 能代表业务身份
- 不包含容易变动的信息

例如服务名：

```hcl
api
worker
frontend
```

通常比下面这种 key 稳定：

```hcl
api-v1
api-2026-07-03
api-temp-new
```

版本、日期、临时状态应该放在 value 里，而不是 key 里。  
否则每次版本变化都可能导致资源地址变化。

## 20. map 和 tag 管理

map 在 tag 管理中非常常见。

例如：

```hcl
locals {
  common_tags = {
    ManagedBy = "Terraform"
    Project   = "terraform-training"
    Owner     = "platform"
  }
}
```

资源里使用：

```hcl
tags = merge(
  local.common_tags,
  {
    Environment = var.environment
    Name        = "app"
  }
)
```

这样可以统一组织标签，减少复制粘贴。

在云平台治理里，tag map 是非常实用的建模方式。

---

## 21. map 和环境配置

map 很适合按环境选择配置：

```hcl
locals {
  env_config = {
    dev = {
      size     = "small"
      replicas = 1
    }

    prod = {
      size     = "large"
      replicas = 3
    }
  }

  current = local.env_config[var.environment]
}
```

然后：

```hcl
local.current.size
local.current.replicas
```

这比写一堆条件表达式更清晰。

不过生产里要配合变量验证，确保：

```hcl
var.environment
```

一定是 map 里存在的 key。

---

## 22. map 和变量验证

可以用 `contains(keys(map), value)` 验证 key 是否存在。

例如：

```hcl
variable "environment" {
  type    = string
  default = "dev"

  validation {
    condition     = contains(["dev", "test", "prod"], var.environment)
    error_message = "environment must be dev, test, or prod."
  }
}
```

如果 allowed list 来自 map：

```hcl
contains(keys(local.env_config), var.environment)
```

不过注意：变量 validation 不能引用 locals。  
如果要做这类校验，通常需要把允许列表放在变量本身、或者用其他结构设计。  
这是 Terraform 语言限制里一个常见细节。

## 23. 生产环境怎么选择 list / set / map？

可以继续用上一节的判断：

| 类型 | 适合场景 |
|---|---|
| list | 顺序有意义 |
| set | 只关心成员，不关心顺序，且去重 |
| map | 按 key 管理命名对象 |

如果你要创建多个长期资源，且每个资源都有名字，优先考虑 map + for_each。

例如：

```hcl
services = {
  api = {
    port = 8080
  }

  web = {
    port = 3000
  }
}
```

比：

```hcl
services = [
  {
    name = "api"
    port = 8080
  },
  {
    name = "web"
    port = 3000
  }
]
```

更适合做 `for_each`，因为 key 更稳定。

## 25. SRE / Platform Engineering 视角

从平台工程角度看，map 非常关键。

因为平台模块经常要表达：

- 多个服务
- 多个环境
- 多个 region
- 多个 subnet
- 多个 policy
- 多个 user
- 多个 alert rule
- 多个成本中心标签

这些对象通常都有稳定名字。  
如果用 map，就可以用 key 稳定管理它们。

例如：

```hcl
for_each = var.services
```

创建出来的资源地址带业务名：

```text
module.app.local_file.service["api"]
module.app.local_file.service["worker"]
```

这比 `[0]`、`[1]` 可读太多，也稳定太多。

这就是生产 Terraform 的味道：  
不是能跑就行，而是资源地址、状态演进、后续维护都要稳。

## 26. 面试/考试记忆版

可以这样记：

- map 是 key-value 集合。
- map 通过 key 访问，例如 `var.config["dev"]`。
- key 必须唯一。
- map 很适合和 `for_each` 搭配。
- map 的 key 会成为资源地址的一部分，所以 key 要稳定。
- `keys(map)` 返回所有 key。
- `values(map)` 返回所有 value。
- `lookup(map, key, default)` 可以带默认值读取。
- `merge(map1, map2)` 可以合并 map。
- list 适合顺序，map 适合命名对象。

一句话总结：

> Terraform map 的核心价值不是存几个键值对，而是用稳定 key 建模生产资源。

## 27. 本 Lab 的价值

Lab 51 的价值在于，它把 Terraform 数据建模推进了一步。

list 让你表达“一组值”。  
map 让你表达“一组有名字的值”。  
而生产 Terraform 更需要后者，因为资源通常都有身份、有 owner、有环境、有服务名。

学好 map 后，你后面理解这些内容会更轻松：

- `for_each`
- map object
- module input
- tag merge
- environment config
- service catalog
- dynamic block
- drift 排查
- state 地址稳定性

别把 map 当成普通字典语法看。  
它是 Terraform 资源建模和状态稳定性的核心工具之一。
