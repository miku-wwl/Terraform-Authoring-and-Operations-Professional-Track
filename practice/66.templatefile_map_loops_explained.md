# Terraform 实操训练 66 讲解：模板处理 map 和循环

## 1. 这一节到底在学什么？

Lab 66 学的是 Terraform 模板里如何处理 **map 和循环**。

上一节 Lab 65 讲的是 `templatefile()` 基础：

```hcl
templatefile(path, vars)
```

也就是把变量传给模板文件，然后生成文本。

Lab 66 继续往前走一步：

> 如果传给模板的不是一个简单字符串，而是 map、list、object，模板里怎么循环输出？

一句话理解：

> Lab 66 学的是在 `.tftpl` 模板文件里用 `%{ for ... }` 遍历 map/list，生成多行配置。

这在生成配置文件时非常常见，比如：

- 生成环境变量文件
- 生成 tag 清单
- 生成端口列表
- 生成服务配置
- 生成用户清单
- 生成 Nginx upstream
- 生成 Prometheus scrape config
- 生成 shell script 片段

---

## 2. 为什么模板需要循环？

如果配置只有一两个变量，简单替换就够了。

模板：

```text
app=${app_name}
env=${environment}
```

Terraform：

```hcl
templatefile("${path.module}/templates/app.tftpl", {
  app_name    = "api"
  environment = "dev"
})
```

但真实配置经常是一组值。

例如 map：

```hcl
tags = {
  Environment = "dev"
  Owner       = "platform"
  ManagedBy   = "Terraform"
}
```

你希望模板输出：

```text
Environment=dev
Owner=platform
ManagedBy=Terraform
```

这就需要模板循环。

---

## 3. 模板循环的基本语法

Terraform 模板里可以写：

```text
%{ for item in items }
${item}
%{ endfor }
```

如果传入：

```hcl
items = ["api", "worker", "web"]
```

模板会输出：

```text
api
worker
web
```

基本结构是：

```text
%{ for 变量名 in 集合 }
使用变量
%{ endfor }
```

注意，这个语法写在模板文件里，不是 `.tf` 资源块里的普通 HCL for 表达式。

---

## 4. 模板里遍历 list

Terraform：

```hcl
locals {
  rendered = templatefile("${path.module}/templates/services.tftpl", {
    services = ["api", "worker", "web"]
  })
}
```

模板 `services.tftpl`：

```text
%{ for service in services }
service=${service}
%{ endfor }
```

渲染结果：

```text
service=api
service=worker
service=web
```

这适合生成简单列表配置。

---

## 5. 模板里遍历 map

如果传入 map：

```hcl
locals {
  rendered = templatefile("${path.module}/templates/tags.tftpl", {
    tags = {
      Environment = "dev"
      Owner       = "platform"
      ManagedBy   = "Terraform"
    }
  })
}
```

模板可以这样写：

```text
%{ for key, value in tags }
${key}=${value}
%{ endfor }
```

输出类似：

```text
Environment=dev
ManagedBy=Terraform
Owner=platform
```

注意：map 的输出顺序不要强依赖业务含义。  
如果你需要稳定顺序，建议在 Terraform locals 里先整理成排序后的 list，再传给模板。

---

## 6. `key, value` 是什么？

模板循环 map 时，常用两个变量：

```text
%{ for key, value in tags }
${key}=${value}
%{ endfor }
```

含义是：

- `key`：map 的键，例如 `Environment`
- `value`：map 的值，例如 `dev`

这和 Terraform `for` 表达式很像：

```hcl
{
  for key, value in var.tags : key => value
}
```

但位置不同：

- HCL `for` 表达式写在 `.tf` 里，用来生成数据结构。
- 模板 `%{ for ... }` 写在 `.tftpl` 里，用来生成文本。

---

## 7. 空白和换行控制

模板循环最容易出现的问题是多余空行。

普通写法：

```text
%{ for key, value in tags }
${key}=${value}
%{ endfor }
```

可能生成多余空白。

Terraform 模板支持 `~` 来修剪空白：

```text
%{ for key, value in tags ~}
${key}=${value}
%{ endfor ~}
```

这里的 `~` 可以减少模板指令附近的换行和空格。

生成配置文件时，空白很重要。  
尤其是：

- YAML
- shell script
- nginx config
- systemd unit
- cloud-init
- Python/INI 配置

如果空白不对，生成文件可能看起来差不多，但实际不能用。

## 8. 模板里遍历 object list

假设 Terraform 传入服务列表：

```hcl
services = [
  {
    name = "api"
    port = 8080
  },
  {
    name = "worker"
    port = 9090
  }
]
```

模板：

```text
%{ for svc in services ~}
service=${svc.name}
port=${svc.port}
%{ endfor ~}
```

输出：

```text
service=api
port=8080
service=worker
port=9090
```

如果想更清楚：

```text
%{ for svc in services ~}
[${svc.name}]
port=${svc.port}

%{ endfor ~}
```

输出：

```text
[api]
port=8080

[worker]
port=9090
```

这在生成配置文件时很实用。

---

## 9. 模板里处理 map(object)

生产里更常见的是 `map(object)`。

Terraform：

```hcl
services = {
  api = {
    port     = 8080
    replicas = 2
  }

  worker = {
    port     = 9090
    replicas = 1
  }
}
```

模板：

```text
%{ for name, svc in services ~}
[${name}]
port=${svc.port}
replicas=${svc.replicas}

%{ endfor ~}
```

输出：

```text
[api]
port=8080
replicas=2

[worker]
port=9090
replicas=1
```

这里：

- `name` 是 map key。
- `svc` 是 object value。
- `svc.port` 和 `svc.replicas` 是对象字段。

这个模式非常实战。

## 13. templatefile 和 HCL for 表达式的区别

很容易混淆：

### HCL for 表达式

写在 `.tf` 文件里，用来生成数据：

```hcl
locals {
  names = [
    for svc in var.services : svc.name
  ]
}
```

结果是 Terraform 数据结构。

### 模板 for 指令

写在 `.tftpl` 文件里，用来生成文本：

```text
%{ for svc in services }
${svc.name}
%{ endfor }
```

结果是字符串文本。

简单记忆：

| 写法 | 位置 | 作用 |
|---|---|---|
| `[for x in xs : ...]` | `.tf` | 生成 list/map |
| `%{ for x in xs }` | `.tftpl` | 生成文本行 |

---

## 14. 建议：复杂逻辑先在 locals 里处理

虽然模板里可以循环，也可以条件判断，但不要把太多业务逻辑塞进模板。

不太好的做法：

```text
%{ for name, svc in services ~}
%{ if svc.enabled && svc.port > 1024 ~}
${name}=${svc.port}
%{ endif ~}
%{ endfor ~}
```

更好的做法：

```hcl
locals {
  enabled_services = {
    for name, svc in var.services :
    name => svc
    if svc.enabled && svc.port > 1024
  }

  template_vars = {
    services = local.enabled_services
  }
}
```

模板只负责输出：

```text
%{ for name, svc in services ~}
${name}=${svc.port}
%{ endfor ~}
```

这样更清楚，也更容易测试。

一句话：

> 数据筛选和转换放在 locals，文本排版放在 template。

---

## 15. 模板里也可以写条件

模板支持条件：

```text
%{ if enabled }
enabled=true
%{ else }
enabled=false
%{ endif }
```

也可以和循环结合：

```text
%{ for name, svc in services ~}
%{ if svc.enabled ~}
${name}=${svc.port}
%{ endif ~}
%{ endfor ~}
```

但还是那句话：  
模板里的逻辑越复杂，维护成本越高。

如果只是简单是否输出一行，可以写在模板里。  
如果是复杂过滤和转换，应该先在 `.tf` 的 locals 里完成。

## 16. 模板循环和排序问题

map 遍历时，输出顺序不要承载业务含义。

如果希望输出稳定，可以在 locals 里先生成排序后的结构。

例如：

```hcl
locals {
  sorted_tag_keys = sort(keys(var.tags))

  sorted_tags = [
    for key in local.sorted_tag_keys : {
      key   = key
      value = var.tags[key]
    }
  ]
}
```

然后传给模板：

```hcl
templatefile("${path.module}/templates/tags.tftpl", {
  tags = local.sorted_tags
})
```

模板：

```text
%{ for tag in tags ~}
${tag.key}=${tag.value}
%{ endfor ~}
```

这样输出顺序更稳定，测试也更容易。

---

## 17. map 模板输出稳定性的建议

如果只是普通配置，人类不关心顺序，可以直接遍历 map。

如果用于：

- golden file 测试
- CI diff
- 生成文档
- 生成稳定配置文件
- 审计输出

建议提前排序。

常见写法：

```hcl
locals {
  tag_lines = [
    for key in sort(keys(var.tags)) : {
      key   = key
      value = var.tags[key]
    }
  ]
}
```

模板再遍历 `tag_lines`。

这比在模板里处理排序更清楚。

## 18. 模板处理 map 的常见输出格式

### key=value 格式

模板：

```text
%{ for key, value in env_vars ~}
${key}=${value}
%{ endfor ~}
```

输出：

```text
APP_ENV=dev
LOG_LEVEL=info
```

适合 `.env` 文件。

---

### INI 格式

模板：

```text
%{ for name, svc in services ~}
[${name}]
port=${svc.port}
replicas=${svc.replicas}

%{ endfor ~}
```

输出：

```text
[api]
port=8080
replicas=2
```

---

### Shell export

模板：

```text
%{ for key, value in env_vars ~}
export ${key}="${value}"
%{ endfor ~}
```

输出：

```sh
export APP_ENV="dev"
export LOG_LEVEL="info"
```

注意 shell 转义风险，复杂值要小心。

## 19. 什么时候不要用模板循环？

如果目标格式是 JSON，很多时候不要用模板循环手写 JSON。

不推荐：

```text
{
%{ for key, value in tags ~}
  "${key}": "${value}",
%{ endfor ~}
}
```

这很容易产生最后一个逗号问题、转义问题、类型问题。

更推荐：

```hcl
jsonencode({
  tags = var.tags
})
```

如果目标格式是 YAML，也可以考虑 `yamlencode()`。  
能用 encode 函数生成结构化格式时，通常比手写模板更安全。

---

## 20. templatefile 和 yamlencode/jsonencode 的取舍

| 目标 | 推荐方式 |
|---|---|
| JSON | 优先 `jsonencode()` |
| YAML | 优先 `yamlencode()`，复杂排版再考虑模板 |
| Shell script | `templatefile()` |
| INI / conf | `templatefile()` |
| Nginx config | `templatefile()` |
| systemd unit | `templatefile()` |
| cloud-init | 视情况，简单可模板，结构化可 yamlencode |

一句话：

> 结构化数据优先 encode，文本配置优先 templatefile。

## 21. 模板和敏感数据

如果模板里输出 secret，比如：

```text
password=${password}
```

渲染结果可能进入：

- Terraform state
- 本地文件
- plan 输出
- CI 日志
- artifact
- 远端资源属性

`sensitive = true` 只能减少 CLI 明文展示，不代表 state 不保存。

所以不要随便用模板生成明文密码文件。  
如果必须生成，要明确保护：

- state 加密
- 文件权限
- CI 日志脱敏
- 不输出敏感内容
- 使用 secret manager
- 限制 artifact 上传

## 22. 常见坑

### 坑 1：模板变量没传

模板里用了：

```text
${tags}
```

但 `templatefile()` 第二个参数里没传 `tags`，会失败。

---

### 坑 2：把 HCL for 写进模板

模板里应该写：

```text
%{ for item in items }
...
%{ endfor }
```

不是：

```hcl
[for item in items : item]
```

两者位置和用途不同。

---

### 坑 3：多余空行

循环模板容易生成多余空白。  
可以用 `~` 控制。

---

### 坑 4：手写 JSON 模板

JSON 更推荐 `jsonencode()`。

---

### 坑 5：模板逻辑太重

复杂过滤、排序、去重、类型转换，最好先在 locals 中完成。

---

### 坑 6：map 输出顺序被误用

map 遍历顺序不要承载业务含义。  
需要稳定输出时先排序。

---

## 23. Terraform test 在这里有什么价值？

Lab 66 的验证报告显示：

```text
terraform test 返回 1 passed, 0 failed
```

对模板循环来说，测试非常有价值。

它可以验证：

- 模板能否成功渲染
- map/list 是否正确传入
- 循环输出是否包含预期行
- 渲染结果是否没有缺字段
- 输出数量是否正确
- 排序是否稳定
- 多余空白是否可接受
- 生成文件是否符合预期

模板问题很多时候不是 Terraform 语法错，而是生成文本不符合预期。  
测试正好补这个短板。

## 24. 生产环境里的典型写法

### Terraform

```hcl
locals {
  env_vars = {
    APP_ENV   = "dev"
    LOG_LEVEL = "info"
  }

  template_vars = {
    env_vars = local.env_vars
  }
}

resource "local_file" "env_file" {
  filename = "${path.module}/output/.env"

  content = templatefile(
    "${path.module}/templates/env.tftpl",
    local.template_vars
  )
}
```

### 模板 `env.tftpl`

```text
%{ for key, value in env_vars ~}
${key}=${value}
%{ endfor ~}
```

### 输出

```text
APP_ENV=dev
LOG_LEVEL=info
```

这个结构清楚：

```text
locals 整理数据
templatefile 渲染文本
local_file 生成文件
terraform test 验证结果
```

## 27. 面试/考试记忆版

可以这样记：

- `templatefile()` 可以把 map/list/object 传给模板。
- 模板里循环语法是 `%{ for item in items } ... %{ endfor }`。
- 遍历 map 可以写 `%{ for key, value in map }`。
- 模板变量必须显式传入。
- object 字段用 `${obj.field}` 访问。
- `~` 可以控制模板空白和换行。
- 复杂过滤、排序、去重最好在 locals 里做。
- JSON 优先用 `jsonencode()`，不要手写 JSON 模板。
- 模板输出可能进入 state，敏感数据要谨慎。
- terraform test 可以验证渲染结果。

一句话总结：

> 模板循环的核心价值，是把 Terraform 中整理好的 map/list/object 渲染成多行文本配置；数据逻辑别塞太多进模板。
