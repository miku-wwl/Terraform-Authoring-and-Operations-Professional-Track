# 第 167 节讲解：挑战五解法二：CSV 驱动安全组规则

## 1. 本节定位

本节是综合挑战题的一部分，主题是 **CSV 驱动安全组规则**。

它不是单纯让你手写几个 `aws_security_group_rule`，而是训练一种更接近真实工程的 Terraform 建模方式：

> 把规则数据放在 CSV 文件里，让 Terraform 读取 CSV，然后用 `for_each` 批量生成安全组规则。

这样做的核心价值是：**配置逻辑和规则数据分离**。

也就是说：

- Terraform 代码负责“怎么创建安全组规则”；
- CSV 文件负责“有哪些规则”；
- 后续新增、删除、修改规则时，优先改 CSV，而不是复制粘贴 Terraform resource 块。

这类题很适合考 Terraform 的表达式能力，比如：

- `csvdecode()`
- `file()`
- `for` expression
- `for_each`
- map key 设计
- 字符串转数字
- 安全组规则的 ingress / egress 建模

---

## 2. 本节要解决的问题

传统写法中，如果你有 5 条安全组规则，可能会写 5 个资源块：

```hcl
resource "aws_security_group_rule" "ssh" {
  type              = "ingress"
  from_port         = 22
  to_port           = 22
  protocol          = "tcp"
  cidr_blocks       = ["10.0.0.0/24"]
  security_group_id = aws_security_group.main.id
}

resource "aws_security_group_rule" "http" {
  type              = "ingress"
  from_port         = 80
  to_port           = 80
  protocol          = "tcp"
  cidr_blocks       = ["0.0.0.0/0"]
  security_group_id = aws_security_group.main.id
}
```

这种写法能跑，但问题很明显：

1. 重复代码多。
2. 新增规则需要改 Terraform 代码。
3. 规则数量多时，review 很痛苦。
4. 容易复制粘贴错端口、协议或 CIDR。
5. 不利于把规则交给安全团队或网络团队维护。

CSV 驱动方式的目标就是把规则变成数据。

例如：

```csv
type,from_port,to_port,protocol,cidr,description
ingress,22,22,tcp,10.0.0.0/24,allow ssh from internal
ingress,80,80,tcp,0.0.0.0/0,allow http
egress,0,0,-1,0.0.0.0/0,allow all outbound
```

Terraform 读取之后，再自动展开成资源。

## 3. 核心函数：file + csvdecode

### 3.1 file()

`file()` 用来读取本地文件内容。

例如：

```hcl
locals {
  raw_rules = file("${path.module}/rules.csv")
}
```

这里 `${path.module}` 表示当前模块目录。这样写比直接写 `rules.csv` 更稳，不容易因为执行目录变化导致找不到文件。

### 3.2 csvdecode()

`csvdecode()` 会把 CSV 文本解析成 list of object。

例如 CSV 内容是：

```csv
type,from_port,to_port,protocol,cidr,description
ingress,22,22,tcp,10.0.0.0/24,allow ssh
```

Terraform 解析后大致相当于：

```hcl
[
  {
    type        = "ingress"
    from_port   = "22"
    to_port     = "22"
    protocol    = "tcp"
    cidr        = "10.0.0.0/24"
    description = "allow ssh"
  }
]
```

注意一个考试和实操高频坑：

> CSV 里读出来的值默认都是字符串。

所以端口号虽然看起来是 `22`，但进入 Terraform 后可能是字符串 `"22"`。

如果资源参数需要 number，最好用 `tonumber()` 转换。

## 4. 推荐建模方式

可以先把 CSV 解析成 local：

```hcl
locals {
  security_group_rules = csvdecode(file("${path.module}/rules.csv"))
}
```

然后用 `for_each` 创建规则：

```hcl
resource "aws_security_group_rule" "this" {
  for_each = {
    for rule in local.security_group_rules :
    "${rule.type}-${rule.protocol}-${rule.from_port}-${rule.to_port}-${rule.cidr}" => rule
  }

  type              = each.value.type
  from_port         = tonumber(each.value.from_port)
  to_port           = tonumber(each.value.to_port)
  protocol          = each.value.protocol
  cidr_blocks       = [each.value.cidr]
  description       = each.value.description
  security_group_id = aws_security_group.main.id
}
```

这里的关键是 `for_each` 需要一个 map。

所以这段代码做了两件事：

1. 遍历 CSV 解析后的每一行规则。
2. 给每一条规则生成一个唯一 key。

例如：

```hcl
"ingress-tcp-22-22-10.0.0.0/24" => rule
```

Terraform 会根据这个 key 追踪资源实例。

---

## 5. for_each key 为什么重要

`for_each` 的 key 是 Terraform state 里定位资源实例的重要标识。

例如：

```hcl
aws_security_group_rule.this["ingress-tcp-22-22-10.0.0.0/24"]
```

如果 key 改了，Terraform 可能认为：

- 旧规则要删除；
- 新规则要创建。

所以 key 设计要尽量稳定。

不建议用 list index 当 key，例如：

```hcl
for_each = {
  for index, rule in local.security_group_rules : index => rule
}
```

这种写法的问题是：如果你在 CSV 中间插入一行，后面所有 index 都可能变化，Terraform 会产生一堆无意义的 diff。

更推荐用规则本身的业务字段组成 key，例如：

- type
- protocol
- from_port
- to_port
- cidr

如果 CSV 里有单独的 `name` 或 `id` 字段，也可以用它做 key，但要保证唯一且稳定。

## 6. 为什么安全组规则适合 CSV 驱动

安全组规则天然就是表格型数据。

每一行通常包含：

| 字段 | 含义 |
|---|---|
| type | ingress 或 egress |
| from_port | 起始端口 |
| to_port | 结束端口 |
| protocol | tcp、udp、icmp、-1 等 |
| cidr | 来源或目标 CIDR |
| description | 规则说明 |

这种结构非常适合 CSV。

和写死在 Terraform 代码里相比，CSV 有几个优势：

- 网络规则更直观。
- 安全团队更容易 review。
- 批量修改更方便。
- Terraform 代码更短。
- 同一套代码可以复用不同环境的规则文件。

比如：

```hcl
variable "rules_file" {
  type    = string
  default = "rules-dev.csv"
}

locals {
  security_group_rules = csvdecode(file("${path.module}/${var.rules_file}"))
}
```

这样 dev、stage、prod 可以传入不同 CSV。

## 7. LocalStack 环境下的理解边界

本节文档明确说明：实验不访问真实 AWS，AWS API 默认指向 LocalStack 的：

```text
http://localhost:4566
```

凭证固定为：

```text
access_key = test
secret_key = test
```

这意味着你练到的是 Terraform 的核心闭环：

1. provider 初始化；
2. 读取 CSV；
3. 生成资源计划；
4. apply 创建模拟资源；
5. 验证资源存在；
6. destroy 清理资源。

但是它不能完全替代真实 AWS 的行为，尤其是：

- 真实安全组规则限制；
- VPC 网络连通性；
- AWS IAM 权限评估；
- 真实 API 的边界条件；
- 企业环境中的变更审批；
- 生产环境安全基线。

所以正确理解是：

> LocalStack 用来训练 Terraform 建模和工作流闭环；真实 AWS 用来验证云平台真实行为。

进入练习目录后，通常可以按这个顺序执行：

```bash
terraform fmt
```

| 步骤 | 目的 |

## 9. 常见错误与排查

### 错误 1：CSV 字段名和 Terraform 代码不一致

如果 CSV 里是：

```csv
from,to,proto,cidr
```

但 Terraform 里写：

```hcl
rule.from_port
rule.protocol
```

就会报属性不存在。

解决方法：统一 CSV header 和 Terraform 字段名。

---

### 错误 2：端口没有转换成数字

CSV 解码出来通常是字符串。

错误写法：

```hcl
from_port = each.value.from_port
```

更稳的写法：

```hcl
from_port = tonumber(each.value.from_port)
```

---

### 错误 3：for_each key 不唯一

如果两行规则生成了同一个 key，Terraform 会报 duplicate key。

例如两行都是：

```csv
ingress,80,80,tcp,0.0.0.0/0,allow http
```

生成 key 后完全相同，就会冲突。

解决方法：

- 删除重复规则；
- 或增加一个唯一字段，例如 `name`；
- 或把 description 也纳入 key，但 description 经常会变，不一定稳定。

---

### 错误 4：直接用 list index 作为 key

这种写法短期能跑，但维护性差：

```hcl
for_each = {
  for idx, rule in local.security_group_rules : idx => rule
}
```

只要 CSV 顺序变化，Terraform 可能产生大量无意义变更。

考试和真实项目里，都更推荐稳定 key。

---

### 错误 5：把 LocalStack 当成真实 AWS

LocalStack 是模拟环境，不等于生产 AWS。

它适合验证：

- Terraform 语法；
- provider endpoint 配置；
- 资源依赖；
- plan/apply/destroy 流程；
- 输出和验证脚本。

但不能保证完全覆盖真实 AWS 的所有限制。

## 10. 考试角度怎么记

这节可以总结成一句话：

> 用 `csvdecode(file(...))` 把 CSV 变成 Terraform 数据结构，再用 `for_each` 批量创建安全组规则。

考试里看到下面关键词，要马上联想到本节：

- CSV driven configuration
- decode CSV file
- dynamic list of security group rules
- avoid repeating resource blocks
- create multiple similar resources
- stable `for_each` keys
- separate data from Terraform logic

常见选择题判断：

| 场景 | 应想到 |
|---|---|
| 多条类似规则，不想复制 resource | `for_each` |
| 规则来自 CSV 文件 | `csvdecode(file(...))` |
| CSV 端口是字符串 | `tonumber()` |
| 避免资源地址乱变 | 稳定 key，不用 index |
| 本地模拟 AWS | LocalStack endpoint + test/test |

---

## 11. 面试角度怎么讲

如果面试官问：“你怎么管理大量安全组规则？”

可以这样回答：

> 我不会为每条规则手写一个 Terraform resource。更好的方式是把规则抽成数据，比如 CSV、YAML 或 map 变量，然后用 `for_each` 生成资源。这样 Terraform 代码只描述资源创建逻辑，规则本身由数据文件维护。对于安全组规则这种表格型配置，CSV 很直观。关键点是要设计稳定的 `for_each` key，避免因为顺序变化导致 Terraform state 地址变化。

这个回答比“我会复制几个 aws_security_group_rule”强很多，听起来更像工程实践。

---

## 12. 本节复盘

你需要真正掌握的是：

1. CSV 是规则数据源，不是 Terraform 逻辑本身。
2. `file()` 负责读取文件内容。
3. `csvdecode()` 负责把 CSV 转成 Terraform 可遍历的数据结构。
4. `for_each` 负责批量创建规则。
5. key 必须稳定、唯一。
6. CSV 中的数字通常需要 `tonumber()`。
7. LocalStack 只验证核心 Terraform 路径，不等同真实 AWS 权限和网络行为。

这一节是非常典型的 Terraform 工程化训练：

> 少写重复代码，多做数据建模。
