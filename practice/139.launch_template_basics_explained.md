# Terraform 实操训练 139：Launch Template 基础配置（讲解版）

## 1. 本节核心：Launch Template 是 EC2 启动参数模板

这一节讲的是 AWS EC2 的 Launch Template，也就是启动模板。

它的作用可以简单理解为：

> 把创建 EC2 实例时需要的一组启动参数保存成模板，后续可以复用。

比如你启动一台 EC2，通常要指定：

```text
AMI
Instance Type
Key Pair
Security Group
Subnet
User Data
IAM Instance Profile
Block Device
Tags
```

如果每次都手写一遍，既重复又容易出错。  
Launch Template 就是把这些参数抽出来，变成一个可复用、可版本化的启动配置。

本节实验重点是基础配置：

```text
1. 创建 launch template
2. 配置 AMI
3. 配置 instance type
4. 配置 tag specifications
5. 输出 launch template id
```

---

## 2. Launch Template 解决什么问题？

Launch Template 解决的是“EC2 启动参数标准化”的问题。

比如平台团队希望所有应用服务器都遵守这些标准：

```text
统一 AMI
统一 instance type
统一 tag
统一 user_data
统一安全组
统一 IAM role
统一磁盘配置
```

如果每个团队自己写 EC2 参数，很容易出现：

```text
有人用错 AMI
有人忘记 tag
有人选错 instance type
有人没挂 IAM role
有人忘记加安全组
```

Launch Template 可以把这些启动参数集中定义，后续交给：

- EC2 instance
- Auto Scaling Group
- Spot Fleet
- EKS node group
- Batch compute environment

等服务复用。

## 3. Terraform 里对应什么资源？

Terraform AWS Provider 中常用资源是：

```hcl
resource "aws_launch_template" "example" {
  name_prefix   = "demo-lt-"
  image_id      = "ami-1234567890abcdef0"
  instance_type = "t3.micro"

```

这个资源会创建一个 Launch Template。

常见参数包括：

```text
name
name_prefix
image_id
instance_type
key_name
vpc_security_group_ids
user_data
iam_instance_profile
block_device_mappings
network_interfaces
tag_specifications
```

本节重点是基础版本：AMI、实例类型和标签规格。

---

## 4. image_id：AMI 是实例从哪里启动

`image_id` 指的是 AMI ID。

AMI 可以理解成 EC2 的系统镜像。

例如：

```hcl
image_id = "ami-1234567890abcdef0"
```

它决定实例启动后是什么系统、预装了什么基础环境。

真实 AWS 中，AMI 可能是：

```text
Amazon Linux
Ubuntu
Windows Server
自定义 Golden Image
Packer 构建的镜像
CIS Hardened Image
EKS Optimized AMI
```

在平台工程里，AMI 很重要。  
因为它会影响：

- OS 版本
- 安全基线
- patch 状态
- 预装 agent
- 启动速度
- 合规要求
- 漏洞风险

所以 Launch Template 里固定 AMI，不只是填一个 ID，而是在定义计算节点的基础镜像标准。

---

## 5. instance_type：实例规格

`instance_type` 指 EC2 实例规格。

例如：

```hcl
instance_type = "t3.micro"
```

它决定：

```text
CPU
内存
网络性能
成本
是否适合突发负载
是否适合生产工作负载
```

常见例子：

```text
t3.micro       小实验、低成本
t3.medium      小型服务
m6i.large      通用型
c7g.large      计算优化
r6i.large      内存优化
g5.xlarge      GPU 场景
```

本地 LocalStack 实验不会真的创建真实 EC2 算力，所以它不能验证真实性能和费用。  
它只能帮助你验证 Terraform 配置结构和 Provider 调用路径。

---

## 6. tag_specifications：给启动出来的资源打标签

Launch Template 里比较容易忽略的是：

```hcl
tag_specifications
```

它不是普通的 `tags`。

普通 `tags` 通常是给 Launch Template 这个模板资源本身打标签。  
而 `tag_specifications` 是定义“从这个模板启动出来的资源应该带什么标签”。

典型写法：

```hcl
resource "aws_launch_template" "example" {
  name_prefix   = "demo-lt-"
  image_id      = "ami-1234567890abcdef0"
  instance_type = "t3.micro"

  tag_specifications {
    resource_type = "instance"

    tags = {
      Name        = "demo-instance"
      Environment = "dev"
      ManagedBy   = "terraform"
    }
  }
}
```

这里表示：

```text
通过这个 Launch Template 启动出来的 EC2 instance，会带上这些 tags。
```

常见 `resource_type` 包括：

```text
instance
volume
network-interface
spot-instances-request
```

如果你只给模板本身打 tag，却忘记给 instance / volume 加 tag，后续成本统计和资源归属可能会乱。

---

## 7. tags 和 tag_specifications 的区别

这是本节特别值得记的点。

| 配置 | 作用对象 |
|---|---|
| `tags` | 给 Launch Template 这个模板本身打标签 |
| `tag_specifications` | 给通过模板启动出来的资源打标签 |

示例：

```hcl
resource "aws_launch_template" "example" {
  name_prefix   = "demo-lt-"
  image_id      = "ami-1234567890abcdef0"
  instance_type = "t3.micro"

  tags = {
    ManagedBy = "terraform"
  }

  tag_specifications {
    resource_type = "instance"

    tags = {
      Name = "demo-instance"
    }
  }
}
```

含义：

```text
Launch Template 本身有 ManagedBy=terraform
从模板启动出的 EC2 instance 有 Name=demo-instance
```

如果要让卷也带标签，可以再加：

```hcl
tag_specifications {
  resource_type = "volume"

  tags = {
    Name        = "demo-volume"
    Environment = "dev"
  }
}
```

生产里这很关键。  
不然 EC2 有 tag，EBS volume 没 tag，FinOps 报表一样会乱。

## 8. Launch Template 的版本化思想

Launch Template 和旧的 Launch Configuration 最大区别之一是：Launch Template 支持版本。

你更新模板参数时，AWS 可以生成新版本：

```text
Launch Template: app-template
├── Version 1: AMI A, t3.micro
├── Version 2: AMI B, t3.micro
└── Version 3: AMI B, t3.medium
```

这带来的好处是：

```text
1. 可以追踪启动配置变化
2. 可以让 Auto Scaling Group 指定具体版本
3. 可以回滚到旧版本
4. 可以逐步替换实例
5. 更适合生产变更管理
```

Terraform 管理 Launch Template 时，改动 AMI、instance type、user_data 等配置，通常会涉及模板版本变化。

所以你要理解：

> Launch Template 不是一次性配置文件，而是可演进、可版本化的启动标准。

## 9. Launch Template 和 Auto Scaling Group 的关系

Launch Template 经常和 Auto Scaling Group 一起用。

关系大概是：

```text
Launch Template = 怎么启动实例
Auto Scaling Group = 启动多少实例、在哪些子网、如何扩缩容
```

例如：

```text
Launch Template:
- 用哪个 AMI
- 用哪个 instance type
- 加什么 tag
- 跑什么 user_data

Auto Scaling Group:
- desired_capacity = 3
- min_size = 2
- max_size = 6
- subnets = private subnets
- health check
- scaling policy
```

一句话：

```text
Launch Template 定义“单台实例长什么样”。
Auto Scaling Group 定义“一组实例怎么运行和扩缩容”。
```

本节只讲 Launch Template 基础配置，还没进入 ASG 扩缩容。  
但你要提前知道它后面通常会被 ASG 使用。

---

## 10. 为什么 AWS 推荐 Launch Template，而不是 Launch Configuration？

Launch Configuration 是老方式。  
Launch Template 是更新、更灵活的方式。

Launch Template 支持更多能力，例如：

```text
版本管理
更多 EC2 参数
Mixed Instances Policy
Spot 配置
更丰富的网络配置
更细粒度的 tag specifications
更适合 Auto Scaling Group 新特性
```

所以真实项目里，新设计通常优先使用 Launch Template。

考试或项目里如果看到 Auto Scaling 相关的新配置，大概率也会更偏向 Launch Template。

## 11. LocalStack 在本节里的作用

本节实验不访问真实 AWS，而是用 LocalStack 模拟 AWS API。

结构是：

```text
Terraform AWS Provider
-> endpoint 指向 http://localhost:4566
-> 使用假凭证 test/test
-> LocalStack 模拟 AWS 服务
```

好处：

```text
不需要真实 AWS 账号
不产生真实费用
不会启动真实 EC2
不会误删云上资源
可以反复练习 Terraform 流程
```

但边界也要清楚：

```text
LocalStack 可以验证 Terraform 配置和 Provider 调用路径
但不能替代真实 AWS 的 IAM 权限、EC2 启动行为、真实扩缩容、网络连通性和实例性能验证
```

特别是 Launch Template 这种资源，在真实 AWS 里还会牵涉：

```text
AMI 是否存在
AMI 是否属于当前 region
实例类型是否在 region 可用
IAM role 是否允许使用
Security Group 是否正确
Subnet 是否可用
User Data 是否能正常执行
EBS 配置是否有效
Tag policy 是否允许
```

LocalStack 通过，只代表你这套 Terraform 结构大体闭环，不代表生产一定没坑。

## 16. 常见错误和排障

## 16.1 AMI ID 写错

真实 AWS 中，如果 AMI ID 不存在，或者不属于当前 region，会失败。

排查：

```text
1. 检查 AMI ID 是否存在
2. 检查 region 是否正确
3. 检查账号是否有权限使用该 AMI
4. 检查 AMI 是否被弃用或私有
```

LocalStack 中通常不会完整模拟这些真实约束。

---

## 16.2 instance_type 在目标 region 不可用

真实 AWS 中，不是所有实例类型都在所有 region / AZ 可用。

例如某些新一代 Graviton 或 GPU 实例，可能在部分区域不可用。

排查：

```text
检查 instance type 是否在目标 region 可用
检查配额是否足够
检查是否需要 capacity reservation
```

---

## 16.3 tag_specifications 的 resource_type 写错

例如：

```hcl
resource_type = "instances"
```

可能是错的。常见正确值是：

```text
instance
volume
network-interface
spot-instances-request
```

如果 tag 没生效，要检查：

```text
1. resource_type 是否正确
2. tags 是否写在 tag_specifications 里面
3. 是否只给模板本身 tags，而忘了给 instance / volume tag
```

---

## 16.4 混淆 tags 和 tag_specifications

错误理解：

```text
我给 Launch Template 写了 tags，所以启动出来的 EC2 也一定带这些 tags。
```

不一定。

`tags` 是给模板资源本身。  
`tag_specifications` 才是给模板启动出的资源。

生产里要特别注意 EBS volume tag。  
很多成本漂移都是因为 instance 有 tag，volume 没 tag。

---

## 16.5 user_data 需要 base64 编码吗？

Launch Template 的 `user_data` 在 AWS API 层通常要求 base64 编码。  
Terraform AWS Provider 的具体资源参数也常用：

```hcl
user_data = base64encode(templatefile("${path.module}/user_data.sh", {}))
```

本节基础实验可能不涉及 user_data，但后续你做真实模板时一定会遇到。

## 17. 生产环境推荐做法

真实项目中，Launch Template 通常不会只写 AMI 和 instance type，还会加入：

```text
IAM instance profile
Security groups
User data
Block device mappings
Metadata options
Monitoring
EBS encryption
Tag specifications
Network interfaces
```

一个更像生产的结构：

```hcl
resource "aws_launch_template" "app" {
  name_prefix   = "${var.name}-"
  image_id      = var.ami_id
  instance_type = var.instance_type

  iam_instance_profile {
    name = aws_iam_instance_profile.app.name
  }

  vpc_security_group_ids = var.security_group_ids

  user_data = base64encode(templatefile("${path.module}/user_data.sh", {
    environment = var.environment
  }))

  metadata_options {
    http_tokens = "required"
  }

  tag_specifications {
    resource_type = "instance"

    tags = {
      Name        = var.name
      Environment = var.environment
      ManagedBy   = "terraform"
    }
  }

  tag_specifications {
    resource_type = "volume"

    tags = {
      Name        = "${var.name}-volume"
      Environment = var.environment
      ManagedBy   = "terraform"
    }
  }
}
```

这里有几个生产级重点：

```text
metadata_options.http_tokens = "required" 开启 IMDSv2
user_data 用 templatefile 管理
volume 也要打 tag
AMI 和 instance_type 用变量传入
```

## 18. Launch Template 和安全基线

生产中 Launch Template 是安全基线入口之一。

你可以通过它统一配置：

```text
IMDSv2
加密 EBS
安全组
IAM role
日志 agent
安全 agent
补丁 agent
启动脚本
标准标签
```

例如强制 IMDSv2：

```hcl
metadata_options {
  http_tokens = "required"
}
```

这可以降低 SSRF 类攻击偷取实例元数据凭据的风险。

所以 Launch Template 不只是“启动 EC2 的参数集合”。  
它也是平台团队把安全标准下发到计算节点的一种方式。

## 19. Launch Template 和 Packer 的关系

真实平台工程里，Launch Template 经常和 Packer 配合。

逻辑是：

```text
Packer 构建 AMI
-> Terraform 把 AMI ID 写入 Launch Template
-> Auto Scaling Group 使用新模板版本
-> 实例滚动替换
```

这就是常见的 immutable infrastructure 流程。

例如：

```text
代码变更 / 镜像基线更新
-> Packer build AMI
-> Terraform update launch template image_id
-> ASG instance refresh
-> 新实例用新 AMI 启动
```

所以 Launch Template 是 AMI 生命周期管理的一部分。

## 23. 最终总结

这一节最重要的理解是：

> Launch Template 是 EC2 启动参数模板，用来标准化 AMI、实例类型、标签、user data、安全组、IAM role 等配置，并经常被 Auto Scaling Group 复用。

本节实验的主线是：

```text
创建 aws_launch_template
-> 配置 image_id
-> 配置 instance_type
-> 配置 tag_specifications
-> 输出 launch_template_id
-> destroy 清理资源
```

一句话收尾：

```text
Launch Template 就像 EC2 的“出厂说明书”：实例按什么镜像、什么规格、什么标签启动，都由它定义。
```
