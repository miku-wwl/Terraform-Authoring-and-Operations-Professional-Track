## 2. 知识点讲解

数据源用于读取 Terraform 配置外部已经存在的信息。它不会创建资源，而是在 plan/apply 过程中通过 provider 读取外部系统，并把读取到的属性放入状态和表达式上下文。

本节把课程中的 AWS EC2 查询改造成 LocalStack 实验：`bootstrap.sh` 先在 LocalStack 中创建两台模拟 EC2，然后 Terraform 使用 `data "aws_instances"` 读取这些实例 ID。生产环境中，同样的模式常用于读取已有 VPC、AMI、subnet、安全组或账号身份。

LocalStack 可以验证 provider endpoint、data source 读取、output 引用这些 Terraform 能力，但它不等价于真实 AWS 的 EC2 网络、计费、metadata 和 AMI 生态。

## 3. 实验设计

背景：平台团队需要读取已有 EC2 实例列表，而不是在当前配置中创建实例。

任务：使用 `data "aws_instances"` 读取 LocalStack 中已有的 `tf-lab-31-*` 实例，并输出实例 ID。

预期结果：Terraform 不创建 EC2，只读取 bootstrap 创建的实例，`verify.sh` 能通过 AWS CLI 查询到至少两台实例。

## 5. 运行手册

```sh
cd practice/labs/31
export AWS_ACCESS_KEY_ID=test
export AWS_SECRET_ACCESS_KEY=test
export AWS_DEFAULT_REGION=us-east-1
export LOCALSTACK_ENDPOINT=http://localhost:4566

bash scripts/check-sandbox.sh
bash scripts/bootstrap.sh
terraform init -input=false
terraform fmt -check
terraform validate
terraform plan -input=false -no-color -out=tfplan
terraform apply -auto-approve tfplan
terraform output
bash scripts/verify.sh
terraform destroy -auto-approve
bash scripts/clean.sh
```

## 7. 排障指南

1. LocalStack endpoint 不通：先运行 `bash scripts/check-sandbox.sh`，确认 `sts get-caller-identity` 成功。
2. Provider 打到真实 AWS：确认 `provider.tf` 中配置了 `endpoints`，并导出了假凭据 `test/test`。
3. 查询结果为空：先运行 `bash scripts/bootstrap.sh` 创建前置 EC2。
4. `terraform init` 失败：确认 Sandbox 可以访问 Terraform Registry。

## 8. 清理方式
