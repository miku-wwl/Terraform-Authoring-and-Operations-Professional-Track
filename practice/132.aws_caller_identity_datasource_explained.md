# Terraform 实操训练 132：`aws_caller_identity` 数据源讲解版

## 1. 本节定位

本节主题是 **`aws_caller_identity` 数据源**。

它看起来很小，但在真实 Terraform 项目里很常见，因为很多模块需要知道“当前 Terraform 正在以哪个 AWS 身份运行”。例如：

- 当前账号 ID 是多少；
- 当前 caller 的 ARN 是什么；
- 当前身份是 IAM User、Assumed Role，还是其他 STS 身份；
- 是否需要把账号 ID 拼进 IAM policy、S3 bucket name、KMS policy 或 ARN 字符串中。

本实验不连接真实 AWS，而是通过 **LocalStack** 模拟 AWS API。这样可以安全练习 Terraform AWS Provider 的写法、数据源查询、输出设计和销毁流程。

---

## 2. 核心概念：什么是 `aws_caller_identity`

`aws_caller_identity` 是 AWS Provider 提供的一个 **data source**。

它对应 AWS STS 的 `GetCallerIdentity` 行为，用来返回当前请求者身份信息。

典型写法如下：

```hcl
data "aws_caller_identity" "current" {}
```

然后可以通过下面这些属性读取结果：

```hcl
data.aws_caller_identity.current.account_id
data.aws_caller_identity.current.user_id
data.aws_caller_identity.current.arn
```

这三个字段分别代表：

| 字段 | 含义 | 常见用途 |
|---|---|---|
| `account_id` | 当前 AWS 账号 ID | 拼接 ARN、区分环境账号、生成命名 |
| `user_id` | 当前调用者的唯一 ID | 排障、审计、身份确认 |
| `arn` | 当前调用者 ARN | IAM policy、日志、调试输出 |

一句话记忆：

> `aws_caller_identity` 不创建资源，它只是问 AWS：“我现在是谁？”

---

## 3. data source 和 resource 的区别

Terraform 里常见两类 block：

```hcl
resource "aws_s3_bucket" "example" {
  bucket = "example"
}
```

以及：

```hcl
data "aws_caller_identity" "current" {}
```

两者区别很关键：

| 类型 | 是否创建资源 | 是否写入 state | 典型作用 |
|---|---:|---:|---|
| `resource` | 是 | 是 | 创建、修改、删除基础设施 |
| `data` | 否 | 通常会记录查询结果 | 查询现有信息，给配置使用 |

`aws_caller_identity` 是 data source，所以它不会创建 AWS 资源，也不会产生额外费用。它只是读取当前 Provider 所代表的身份。

---

## 4. 为什么真实项目里经常用它

### 场景 1：动态拼接 ARN

很多 AWS ARN 里都需要账号 ID，例如：

```hcl
locals {
  account_id = data.aws_caller_identity.current.account_id
}

output "example_role_arn" {
  value = "arn:aws:iam::${local.account_id}:role/example-role"
}
```

这样就不用硬编码账号 ID。

硬编码账号 ID 的问题是：

- dev / staging / prod 账号不同；
- 多账号部署时容易复制错；
- 模块复用性差；
- 安全审计时不清晰。

### 场景 2：模块自适应当前账号

模块中可以这样写：

```hcl
data "aws_caller_identity" "current" {}

locals {
  current_account_id = data.aws_caller_identity.current.account_id
}
```

这样模块被不同 AWS 账号调用时，会自动读取当前账号，而不是依赖调用者手动传入。

### 场景 3：输出当前身份用于排障

在实验、CI/CD、平台工程中，经常需要确认 Terraform 现在到底用了哪个身份：

```hcl
output "caller_arn" {
  value = data.aws_caller_identity.current.arn
}
```

如果你以为自己用了 `prod-deploy-role`，但输出显示是个人 IAM User，那就要警惕了。

这个数据源有点像 Terraform 里的“身份证检查”。

## 5. 本节 LocalStack 实验边界

本实验使用 LocalStack：

```text
http://localhost:4566
```

凭证固定为：

```text
test/test
```

这说明几件事：

1. 本实验不会访问真实 AWS。
2. 不需要真实 access key 和 secret key。
3. LocalStack 返回的是模拟身份，不等价于真实 AWS IAM / STS 身份。
4. 可以练 Terraform 写法，但不能验证真实 IAM 权限边界。

所以这节能训练的是：

- AWS Provider endpoint 覆盖；
- data source 声明方式；
- output 输出方式；
- Terraform plan/apply/destroy 流程；
- LocalStack 模拟 AWS API 的工作方式。

不能训练的是：

- 真实 AWS Organizations 多账号关系；
- 真实 IAM role assume 行为；
- 真实 CloudTrail 审计；
- 真实权限拒绝和 SCP 限制；
- 真实账号 ID 的安全治理逻辑。

## 6. 推荐实验结构

参考目录：

```text
practice/labs/132/
├── main.tf
├── outputs.tf
├── versions.tf
└── README.md
```

一个简化示例可以长这样：

```hcl
terraform {
  required_version = ">= 1.11.0"

  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 5.0"
    }
  }
}

provider "aws" {
  region                      = "us-east-1"
  access_key                  = "test"
  secret_key                  = "test"
  skip_credentials_validation = true
  skip_metadata_api_check     = true
  skip_requesting_account_id  = false
  s3_use_path_style           = true

  endpoints {
    sts = "http://localhost:4566"
  }
}

data "aws_caller_identity" "current" {}

output "account_id" {
  value = data.aws_caller_identity.current.account_id
}

output "user_id" {
  value = data.aws_caller_identity.current.user_id
}

output "arn" {
  value = data.aws_caller_identity.current.arn
}
```

注意：不同 AWS Provider 版本对 `endpoints` 写法、可用 service endpoint、校验参数可能会有细节差异。你的课程实验已经验证通过，所以实际以上传实验目录为准。

## 7. 建议运行流程

进入实验目录：

```sh
cd /workspace/practice/labs/132
```

初始化：

```sh
terraform init -input=false
```

格式检查：

```sh
terraform fmt -check
```

语法与 provider 配置校验：

```sh
terraform validate
```

生成执行计划：

```sh
terraform plan -input=false -no-color -out=tfplan
```

应用：

```sh
terraform apply -auto-approve tfplan
```

查看输出：

```sh
terraform output
```

销毁：

```sh
terraform destroy -auto-approve
```

虽然 data source 本身不创建资源，但 destroy 流程仍然是训练闭环的一部分。生产项目中，一般会把完整流程固定成 CI pipeline 的基础动作。

## 8. 观察重点

运行后重点看三个输出：

```text
account_id
user_id
arn
```

你要能回答：

1. 这些值来自哪里？
2. 它们是 Terraform 自己生成的吗？
3. 它们是真实 AWS 身份吗？
4. 为什么 LocalStack 下的身份不能当作真实权限验证依据？
5. 这些字段在真实模块里可以怎么用？

标准理解是：

> Terraform 通过 AWS Provider 调用 STS API，LocalStack 模拟了这个 API 并返回身份信息。Terraform 把查询结果暴露成 data source 属性，然后 output 输出出来。

## 9. 常见误区

### 误区 1：以为 data source 会创建资源

不会。

`data "aws_caller_identity" "current" {}` 只查询当前身份，不创建 IAM User、Role 或 STS session。

### 误区 2：把 LocalStack 身份当真实 AWS 身份

LocalStack 返回的是模拟身份。它能帮你练 Terraform 行为，但不能证明真实 AWS 账号里 IAM 权限正确。

### 误区 3：把账号 ID 写死在模块里

不推荐。

模块里硬编码账号 ID 会降低复用性。更好的做法是用 `aws_caller_identity` 动态读取，或者由调用方明确传入并做校验。

### 误区 4：忘记 provider alias 场景

如果有多个 provider alias，例如：

```hcl
provider "aws" {
  alias  = "prod"
  region = "us-east-1"
}
```

那么 data source 如果要使用指定 alias，需要写：

```hcl
data "aws_caller_identity" "prod" {
  provider = aws.prod
}
```

否则它默认使用 default provider。

这个点在多账号、多区域、多 provider 实战里非常重要。

## 10. 和多 Provider / Alias 的关系

如果 root module 里定义了多个 AWS Provider：

```hcl
provider "aws" {
  alias  = "dev"
  region = "us-east-1"
}

provider "aws" {
  alias  = "prod"
  region = "us-east-1"
}
```

那么你可以分别查询不同身份：

```hcl
data "aws_caller_identity" "dev" {
  provider = aws.dev
}

data "aws_caller_identity" "prod" {
  provider = aws.prod
}
```

这样可以输出：

```hcl
output "dev_account_id" {
  value = data.aws_caller_identity.dev.account_id
}

output "prod_account_id" {
  value = data.aws_caller_identity.prod.account_id
}
```

真实生产环境中，这个技巧常用于：

- 确认部署账号；
- 防止 dev 配置误打到 prod；
- 在 module 中构造跨账号 IAM policy；
- 给多账号 landing zone 做基础验证。

## 11. 生产环境迁移思路

从 LocalStack 迁移到真实 AWS 时，需要调整重点：

### LocalStack 练习环境

```hcl
access_key = "test"
secret_key = "test"

endpoints {
  sts = "http://localhost:4566"
}
```

### 真实 AWS 环境

通常不要把凭证写进 `.tf` 文件，而是使用：

- AWS CLI profile；
- 环境变量；
- GitHub Actions OIDC；
- Terraform Cloud / HCP Terraform workspace variables；
- IAM Role AssumeRole；
- SSO 登录后的 profile。

真实项目中更推荐：

```hcl
provider "aws" {
  region  = "ap-southeast-2"
  profile = "my-profile"
}
```

或者 CI/CD 中使用 OIDC assume role，不在代码里放长期凭证。

## 12. 排障指南

### 问题 1：`terraform init` 失败

优先检查：

- provider 版本约束；
- 网络是否能下载 provider；
- lock file 是否冲突；
- `.terraform` 目录是否损坏。

可以尝试：

```sh
rm -rf .terraform .terraform.lock.hcl
terraform init -input=false
```

### 问题 2：LocalStack 没启动

如果 endpoint 是：

```text
http://localhost:4566
```

但 LocalStack 没启动，plan/apply 可能会连接失败。

检查：

```sh
curl http://localhost:4566/_localstack/health
```

### 问题 3：凭证错误

LocalStack 里一般使用假凭证：

```text
test/test
```

真实 AWS 中不能这么做。真实环境要依赖 profile、环境变量或 role。

### 问题 4：输出不是你预期的真实账号

在 LocalStack 里，这是正常的。

因为 STS 是模拟的，不代表真实 AWS 账号。你要看的是 Terraform 调用链路是否跑通，而不是账号 ID 是否符合真实组织结构。

## 13. 考试与面试记忆点

可以这样记：

> `aws_caller_identity` 是 AWS Provider 的身份查询 data source，常用于读取当前 account_id、user_id 和 arn。它不创建资源，主要用于动态拼接 ARN、模块自适应账号、多账号排障和 CI/CD 身份确认。

高频判断：

- 看到 “current AWS account ID” → 想到 `aws_caller_identity.current.account_id`
- 看到 “who am I in AWS” → 想到 STS GetCallerIdentity
- 看到 “avoid hardcoding account id” → 想到 data source
- 看到 “multi-provider alias” → 记得 data source 也要指定 `provider = aws.alias`

## 14. 本节小结

本节的价值不是“背一个 data source 名字”，而是理解 Terraform 如何通过 Provider 查询外部系统信息。

`aws_caller_identity` 是很典型的 data source：

- 不创建资源；
- 查询当前调用身份；
- 结果可被 output、local、resource、module 使用；
- 在 LocalStack 中可练语法和流程；
- 在真实 AWS 中常用于账号识别和多账号治理。

这个点很小，但非常工程化。生产 Terraform 代码里，少写死账号 ID，多用数据源或明确变量传递，代码质量会立刻上一个台阶。
