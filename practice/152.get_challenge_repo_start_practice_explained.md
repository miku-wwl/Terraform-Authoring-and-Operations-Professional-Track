## 1. 本节定位

- 做综合挑战时应该按什么流程推进。

一句话总结：**这一节是在教你如何启动 challenge，而不是教你某个单点 Terraform 语法。**

## 3. 这节到底在训练什么？

综合挑战和普通 lab 不一样。

普通 lab 往往是：

> 今天学一个资源，照着写，跑通，结束。

综合挑战更接近考试和真实工作：

> 给你一个不完整或有问题的 Terraform 项目，你需要读题、定位、修正、执行、验证、清理。

所以它训练的是工程闭环能力，而不是单个知识点记忆。

你要练的是这条链路：

```text
读题
  ↓
理解目标
  ↓
定位目录
  ↓
检查 Terraform 配置
  ↓
修正变量 / provider / resource / output
  ↓
terraform fmt / init / validate
  ↓
terraform plan
  ↓
terraform apply
  ↓
验证输出或资源状态
  ↓
terraform destroy
```

这就是 Terraform 考试和真实平台工程里最值钱的能力：**把模糊题面拆成可执行动作。**

## 4. 为什么不用真实 AWS？

原始课程可能会围绕真实 AWS 操作展开，但本训练项目把它改造成了 LocalStack 或本地 provider 可验证的练习。

这样做有几个好处：

1. **不花真实云费用**
   - 不会因为忘记销毁资源而产生账单。
   - 尤其是综合挑战，反复练习很正常，用真实 AWS 不划算。

2. **不会污染真实账号**
   - 不会创建真实 IAM、VPC、S3、Lambda 等资源。
   - 不会把练习环境和生产 / 个人账号混在一起。

3. **适合高频重复训练**
   - 你可以不断 reset、apply、destroy。
   - 失败了也没关系，重新来就行。

4. **更适合考试训练**
   - 考试考的是 Terraform 工作流和判断力。
   - 本地模拟环境足够训练大部分语法、依赖关系、状态变化和输出设计。

但也要记住边界：

> LocalStack 能模拟 Terraform 写法和资源生命周期，但不能完全替代真实 AWS 的 IAM 权限评估、网络连通性、服务限制、区域差异和真实云行为。

## 6. 启动一个 challenge 的推荐流程

### 第一步：进入题目目录

示例：

```sh
cd /workspace/work/challenge-xxx
```

实际目录名称以项目里的 challenge 目录为准。

### 第二步：先看题面和 README

先不要急着 `terraform apply`。

先读：

```sh
ls
cat README.md
```

你要先搞清楚：

- 要创建什么资源；
- 需要修什么错误；
- 最终要输出什么；
- 验证脚本会检查什么；
- 是否需要 LocalStack；
- 是否需要特殊变量。

### 第三步：检查 Terraform 文件

常见文件包括：

```text
main.tf
variables.tf
outputs.tf
providers.tf
versions.tf
terraform.tfvars
```

重点看：

- provider 是否配置到 LocalStack；
- resource 名称是否和输出、验证脚本一致；
- 变量是否有默认值；
- output 是否暴露了题目要求的值；
- 是否存在明显拼写错误；
- 是否存在缺失的依赖关系。

### 第四步：格式化和初始化

```sh
terraform fmt -recursive
terraform init -input=false
```

`fmt` 是最便宜的检查，先做它。  
`init` 负责下载 provider、初始化 backend、准备模块。

### 第五步：静态验证

```sh
terraform validate
```

这一关主要检查 Terraform 配置是否合法，比如：

- resource 类型写错；
- provider 配置不完整；
- 变量引用不存在；
- output 引用了不存在的资源；
- HCL 语法错误。

### 第六步：生成计划

```sh
terraform plan -input=false -no-color -out=tfplan
```

`plan` 是综合挑战里最重要的一步。  
你要看 Terraform 到底准备创建、修改、删除什么。

不要把 `plan` 当成“考试前点一下”的动作。它是你判断配置是否正确的主要证据。

### 第七步：应用计划

```sh
terraform apply -auto-approve tfplan
```

使用保存好的 `tfplan`，可以保证 apply 的内容就是刚才 review 过的计划。

### 第八步：查看输出

```sh
terraform output
```

综合挑战经常会要求你输出某个 ID、ARN、URL、名称或 JSON 结构。  
如果 output 不对，验证脚本大概率会失败。

### 第九步：运行验证脚本

不同 challenge 可能脚本名称不同，常见形式类似：

```sh
./verify.sh
```

或者：

```sh
python verify.py
```

验证脚本是判断你是否完成题目的标准，不要只看 `terraform apply` 成功。

### 第十步：销毁资源

```sh
terraform destroy -auto-approve
```

即使是 LocalStack，也要养成 destroy 的习惯。  
考试和生产工程里，清理能力同样重要。

## 7. 推荐命令清单

如果题目没有特殊要求，可以先按这个顺序跑：

```sh
terraform fmt -recursive
terraform init -input=false
terraform validate
terraform plan -input=false -no-color -out=tfplan
terraform apply -auto-approve tfplan
terraform output
# 按题目要求运行验证脚本
terraform destroy -auto-approve
```

如果你只是在快速检查理论型 lab，也可以使用项目提供的 checklist 或脚本验证方式。

## 8. 综合挑战中的常见坑

### 坑 1：没读题，直接 apply

这是最常见的错误。

综合挑战不是让你机械执行命令。  
很多题目会故意留缺口，例如：

- 变量没传；
- provider alias 没绑定；
- output 名字不对；
- resource 名称不匹配；
- 缺少 required provider；
- region 不一致；
- LocalStack endpoint 没配；
- 验证脚本期望的 tag 没写。

先读题，再执行。

---

### 坑 2：以为 apply 成功就等于做对

`terraform apply` 成功只能说明 Terraform 没报错。  
它不代表满足题目要求。

比如题目要求输出 `bucket_name`，你输出了 `bucket_id`，apply 也可能成功，但验证会失败。

真正的完成标准是：

```text
Terraform 命令通过 + 输出符合题目 + 验证脚本通过 + 可以 destroy
```

---

### 坑 3：混淆参考实现和练习目录

`practice/labs/152/` 是参考实现，不是你第一时间抄的地方。

更好的方式是：

1. 先在 `work/` 目录自己做；
2. 卡住后再对照参考实现；
3. 对照时不要复制粘贴，要找出自己错在哪里；
4. 修完后重新跑完整流程。

这样才是真训练。

---

### 坑 4：忽略 LocalStack 边界

LocalStack 是训练工具，不是真实 AWS。

在真实 AWS 里，你还要考虑：

- IAM 权限；
- 账户级限制；
- region 可用性；
- 网络连通性；
- 服务最终一致性；
- 真实计费；
- 组织 SCP；
- CloudTrail / Config / GuardDuty 等治理工具。

所以本地跑通以后，要能说清楚：  
**这证明了 Terraform 配置逻辑，但不证明真实 AWS 环境一定完全等价。**

## 9. 考试视角怎么记？

这节如果放到 Terraform 考试或面试里，本质上考你三个能力。

### 1. Terraform 工作流

你要熟悉：

```text
fmt → init → validate → plan → apply → output → destroy
```

特别是：

- `init` 不是创建资源；
- `validate` 不访问真实云资源；
- `plan` 是预览变更；
- `apply` 才是执行变更；
- `destroy` 是清理资源；
- 保存 plan 文件可以让 apply 使用同一份计划。

---

### 2. 本地模拟与真实云的边界

你要能解释：

```text
LocalStack 可以模拟 AWS API endpoint，帮助本地验证 Terraform 配置。
但它不能完全替代真实 AWS 的权限、网络、配额和服务行为。
```

这句话很重要。它体现你不是“只会跑 demo”，而是知道模拟环境的边界。

---

### 3. 挑战题拆解能力

综合挑战不是让你背答案，而是看你能不能按步骤排障：

```text
题目要求是什么？
当前配置缺了什么？
Terraform 报错在哪一层？
是语法问题、provider 问题、变量问题、资源问题、输出问题，还是验证脚本问题？
```

这个能力比记某一个参数更接近真实工作。

## 10. 排障入口

### `terraform init` 失败

优先检查：

- provider source 是否正确；
- provider version constraint 是否合理；
- 是否需要网络；
- module source 路径是否存在；
- LocalStack 是否需要提前启动。

### `terraform validate` 失败

优先检查：

- 语法是否正确；
- 变量是否声明；
- resource 名称是否拼错；
- output 是否引用不存在的资源；
- provider alias 是否传对。

### `terraform plan` 失败

优先检查：

- provider 配置；
- endpoint 是否指向 LocalStack；
- credentials 是否是假凭证；
- resource 参数是否满足 provider 要求；
- data source 是否能查询到对象。

### `terraform apply` 失败

优先检查：

- LocalStack 服务是否支持该 AWS API；
- 资源依赖顺序是否正确；
- 名称是否冲突；
- policy JSON 是否合法；
- IAM trust policy 是否写对。

### 验证脚本失败

优先检查：

- output 名称是否符合题目；
- tag 是否完整；
- resource name 是否符合验证脚本预期；
- region / endpoint 是否一致；
- 资源是否真的创建成功。

## 11. 本节你应该形成的肌肉记忆

做 challenge 时，不要慌，也不要乱改。

按这个顺序来：

```text
先读题
再定位目录
再看文件
再 fmt/init/validate
再 plan
再 apply
再 output
再 verify
最后 destroy
```

这套流程练熟以后，你做 Terraform Professional 这种偏实操思维的内容，会明显稳很多。
