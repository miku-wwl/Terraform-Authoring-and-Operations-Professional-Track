# Terraform 考试训练 121：创建 Organization、Project 与 Workspace

## 2. 知识点讲解

本节课程把上一节的结构概念落到实际 UI 操作：创建 organization，进入默认或自定义 project，再创建 workspace。考试不要求记忆按钮顺序，但会考你是否理解创建顺序和对象关系。

创建 organization 后，平台通常会提供默认 project，也允许你创建新的 project。创建 workspace 时，需要选择工作流类型，例如 Version Control Workflow、CLI-driven Workflow 或 API-driven Workflow。不同 workflow 决定代码从哪里来、run 如何触发、谁来驱动执行。

生产环境里，创建 workspace 不是简单建一个空壳。你要考虑代码来源、执行模式、Terraform 版本、变量、凭据、权限、state 管理和审批策略。考试题常会把“创建 workspace”包装成团队协作场景，考你能否选出合适工作流。

## 3. 考试选择题

1. 创建 HCP Terraform workspace 时，通常需要选择什么关键配置？
   - A. workflow 类型
   - B. 本地鼠标速度
   - C. Docker Desktop 主题
   - D. Git commit 作者头像

2. Version Control Workflow 的典型特点是什么？
   - A. 从 Git 仓库获取代码并触发运行
   - B. 不允许保存 run 历史
   - C. 只能用于本地 backend
   - D. 不支持 workspace

3. CLI-driven Workflow 更适合哪类用户？
   - A. 熟悉 Terraform CLI，希望本地命令触发远端运行的用户
   - B. 完全不想使用 Terraform 命令的用户
   - C. 只想使用 `terraform fmt` 修改 YAML 的用户
   - D. 只管理 Docker 镜像仓库的用户

## 4. 判断题

1. 创建 workspace 时选择的 workflow 会影响代码来源和 run 触发方式。
2. 创建 organization 后就一定已经连接了 GitHub 仓库。
3. 默认 project 可以帮助新用户快速开始，但生产中仍应根据组织结构规划 project。

## 5. 场景题

1. 团队要求所有变更必须经过 Git PR，再由平台自动 plan。你会选择哪种 workflow？为什么？

2. 个人学习者想保留本地 Terraform CLI 操作习惯，但希望 run 发生在 HCP Terraform。你会推荐哪种 workflow？

## 6. 参考答案与解析

选择题：1-A，2-A，3-A。

判断题：1-对，2-错，3-对。

场景题参考：

1. 推荐 Version Control Workflow。它适合从 Git 仓库获取代码，结合分支、PR 和自动触发运行，便于审计和团队协作。
2. 推荐 CLI-driven Workflow。它允许本地使用 Terraform CLI，同时让状态和运行进入 HCP Terraform 管理。

## 7. 复习提醒

- Workspace 创建题通常考 workflow 选择。
- VCS 适合 Git 驱动、透明追踪；CLI-driven 适合本地命令习惯。
- Organization、project、workspace 创建只是开始，后续还要配置变量、权限和 state。
