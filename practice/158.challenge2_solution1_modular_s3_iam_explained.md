## 3. 这一节到底在练什么？

这节真正要练的是 **Terraform 模块化思维**。

一个挑战题如果同时出现 S3 和 IAM，很容易写成一个巨大 `main.tf`：

```hcl
resource "aws_s3_bucket" "this" {
  bucket = "..."
}

resource "aws_iam_role" "this" {
  name = "..."
}

resource "aws_iam_policy" "this" {
  name   = "..."
  policy = jsonencode(...)
}
```

这样能跑，但不一定是好工程。

模块化之后，你需要思考：

1. 哪些资源应该放进 module？
2. module 应该暴露哪些变量？
3. module 应该输出哪些结果？
4. root module 负责什么？
5. child module 负责什么？
6. IAM policy 是否应该引用 S3 bucket 的 ARN？
7. 输出是否足够支撑验证脚本？

这才是这节的核心。

---

## 4. 推荐理解模型

可以把本节拆成三层：

```text
root module
├── 调用 s3 module
│   ├── 创建 bucket
│   └── 输出 bucket_name / bucket_arn
│
├── 调用 iam module
│   ├── 创建 role
│   ├── 创建 policy
│   └── 绑定 role 与 policy
│
└── 输出最终验证需要的值
```

root module 像“总装车间”，负责把各个模块拼起来。

child module 像“零件工厂”，负责稳定地产出某一类资源。

## 5. 为什么 S3 和 IAM 很适合练模块化？

因为它们之间天然存在依赖关系。

例如 IAM policy 可能需要允许访问某个 S3 bucket：

```json
{
  "Action": [
    "s3:GetObject",
    "s3:PutObject"
  ],
  "Resource": [
    "arn:aws:s3:::example-bucket",
    "arn:aws:s3:::example-bucket/*"
  ]
}
```

这个 policy 里的 bucket ARN 不应该手写。

更好的方式是：

```hcl
module "storage" {
  source      = "./modules/s3"
  bucket_name = var.bucket_name
}

module "iam" {
  source     = "./modules/iam"
  bucket_arn = module.storage.bucket_arn
}
```

这样做的好处是：

- bucket 名字改了，IAM policy 自动跟着变。
- 依赖关系由 Terraform 图自动推导。
- root module 更像配置编排，不像资源垃圾堆。
- module 可以被其他 challenge 或项目复用。

## 6. 关键 Terraform 概念

### 6.1 Module source

本节大概率使用本地 module：

```hcl
module "s3" {
  source = "./modules/s3"
}
```

本地路径 module 的重点是：

- `source` 是相对 root module 的路径。
- module 内部通常有自己的 `variables.tf`、`main.tf`、`outputs.tf`。
- 修改 module 结构后，必要时重新执行 `terraform init`。

---

### 6.2 Module input variables

模块变量是调用方传入 child module 的参数。

例如：

```hcl
variable "bucket_name" {
  type        = string
  description = "Name of the S3 bucket."
}
```

调用时：

```hcl
module "s3" {
  source      = "./modules/s3"
  bucket_name = var.bucket_name
}
```

这代表 root module 不直接创建 bucket，而是把输入交给 S3 module。

---

### 6.3 Module outputs

outputs 是 module 对外暴露的接口。

例如 S3 module 输出：

```hcl
output "bucket_name" {
  value = aws_s3_bucket.this.bucket
}

output "bucket_arn" {
  value = aws_s3_bucket.this.arn
}
```

IAM module 就可以消费这些输出：

```hcl
bucket_arn = module.s3.bucket_arn
```

这就是 Terraform module 之间的“合同”。

---

### 6.4 IAM policy 与 jsonencode

Terraform 中写 IAM policy，推荐用 `jsonencode()`：

```hcl
policy = jsonencode({
  Version = "2012-10-17"
  Statement = [
    {
      Effect = "Allow"
      Action = [
        "s3:GetObject",
        "s3:PutObject"
      ]
      Resource = [
        var.bucket_arn,
        "${var.bucket_arn}/*"
      ]
    }
  ]
})
```

这样比手写 JSON 字符串更稳定：

- 不容易漏逗号。
- HCL 表达式可以自然插入变量。
- Terraform fmt/validate 更容易检查。

## 8. 一次完整运行流程

进入实验目录：

```sh
cd /workspace/practice/labs/158
```

格式化检查：

```sh
terraform fmt -check -recursive
```

初始化：

```sh
terraform init -input=false
```

语法与 provider 配置检查：

```sh
terraform validate
```

生成执行计划：

```sh
terraform plan -input=false -no-color -out=tfplan
```

应用：

```sh
terraform apply -auto-approve tfplan
```

查看输出：

```sh
terraform output
```

执行验证脚本：

```sh
./verify.sh
```

销毁资源：

```sh
terraform destroy -auto-approve
```

## 9. 观察重点

运行时不要只看 “Apply complete”。

你要重点观察这些东西：

### 9.1 Plan 阶段

看 Terraform 是否准备创建：

- S3 bucket
- IAM role
- IAM policy
- IAM role policy attachment

如果 IAM policy 依赖 bucket ARN，你应该能看到 policy 内容中包含 bucket ARN。

---

### 9.2 Apply 阶段

确认资源是否都被创建成功。

LocalStack 中的 IAM 和 S3 行为是模拟的，所以它适合验证 Terraform 资源声明和依赖关系，不适合验证真实 AWS 中复杂 IAM 权限边界。

---

### 9.3 Output 阶段

常见输出可能包括：

- bucket name
- bucket arn
- IAM role name
- IAM role arn
- IAM policy arn

这些 output 很重要，因为验证脚本通常依赖它们确认资源是否创建成功。

---

### 9.4 Destroy 阶段

综合挑战题一定要练销毁。

如果 destroy 失败，通常说明：

- S3 bucket 里还有对象。
- IAM attachment、policy、role 的依赖顺序有问题。
- LocalStack 状态异常。
- 资源命名冲突或状态文件残留。

## 10. 常见坑

### 坑 1：把所有资源堆在 root module

能跑，但不利于综合挑战训练。

本节重点就是模块化，所以至少应该能解释：

- 哪个 module 管 S3？
- 哪个 module 管 IAM？
- root module 如何传参？
- module output 如何被另一个 module 使用？

---

### 坑 2：IAM policy 里硬编码 bucket ARN

不推荐：

```hcl
Resource = "arn:aws:s3:::my-bucket/*"
```

更推荐：

```hcl
Resource = "${var.bucket_arn}/*"
```

或者：

```hcl
Resource = [
  var.bucket_arn,
  "${var.bucket_arn}/*"
]
```

---

### 坑 3：module 没有 outputs

没有 outputs，root module 就拿不到 child module 创建出的资源信息。

这会导致：

- IAM module 无法引用 S3 module 的 ARN。
- 最终 `terraform output` 信息不足。
- 验证脚本不好写。

---

### 坑 4：以为 LocalStack 等于真实 AWS

LocalStack 是训练环境，不是生产环境。

它可以帮你练：

- Terraform provider 配置
- 资源声明
- 模块拆分
- plan/apply/destroy
- output 和验证脚本

但不能完全替代：

- 真实 IAM 权限评估
- AWS Organizations SCP
- S3 bucket policy 的所有边界条件
- KMS、CloudTrail、VPC endpoint 等真实联动
- 生产级安全审计

---

### 坑 5：忘记 provider 指向 LocalStack

本项目默认 AWS API 指向：

```text
http://localhost:4566
```

如果 provider 没配好，有可能 Terraform 尝试访问真实 AWS。

练习时一定确认：

- access key 是 `test`
- secret key 是 `test`
- endpoint 是 LocalStack
- skip credentials validation 等本地模拟配置正确

## 11. 考试视角怎么记？

这节对应考试里的几个高频点：

### 11.1 模块化

看到“重复资源”“可复用”“标准化 S3/IAM 组件”，优先想到 module。

### 11.2 输入输出

module 的 input 是变量，output 是接口。

调用方通过变量传入配置，通过 output 拿回结果。

### 11.3 依赖关系

不要手写顺序，不要靠脚本 sleep。

Terraform 通过引用关系建立依赖图：

```hcl
bucket_arn = module.s3.bucket_arn
```

这就是依赖。

### 11.4 IAM policy

IAM policy JSON 推荐用 `jsonencode()` 生成。

## 12. 面试表达模板

如果面试官问：你怎么用 Terraform 模块化管理 S3 和 IAM？

可以这样说：

> 我会把 S3 和 IAM 拆成独立 child modules。S3 module 负责创建 bucket 并输出 bucket name 和 ARN；IAM module 通过 input 接收 bucket ARN，生成最小权限 policy，再把 policy attach 到 role。root module 只负责组合这些模块、传入环境参数和统一输出验证信息。这样可以减少硬编码，也能让依赖关系由 Terraform graph 自动管理。

这段话很实用，比单纯说“我会用 module”强很多。
