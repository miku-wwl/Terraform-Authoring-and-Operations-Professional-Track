# Terraform 实操训练 116：TF_LOG 与 TF_LOG_PATH 讲解版

> 基于原始课程文件 `116.md` 整理扩展。  
> 主题：用 Terraform 调试日志定位 `init / plan / apply / provider` 相关问题。

---

## 1. 本节一句话总结

`TF_LOG` 用来打开 Terraform 的调试日志，`TF_LOG_PATH` 用来把日志写入文件。

这两个环境变量的核心价值不是“多打印点东西”，而是在 Terraform 出现复杂问题时，帮你看到：

- Terraform Core 做了什么；
- Provider 做了什么；
- Terraform 和 Provider 之间如何交互；
- 哪一步开始报错；
- 是配置问题、状态问题、Provider 问题，还是远端 API 问题。

一句话记忆：

> `TF_LOG` 决定“打不打日志、打多细”；`TF_LOG_PATH` 决定“日志写到哪里”。

---

## 2. 为什么 Terraform 需要调试日志

平时我们运行 Terraform，大多数时候只看这些命令输出：

```sh
terraform init
terraform validate
terraform plan
terraform apply
terraform destroy
```

这些输出适合日常使用，但不适合深度排障。

比如你遇到：

- provider 下载失败；
- provider 初始化失败；
- backend 连接失败；
- 某个资源 plan 阶段行为异常；
- apply 报远端 API 错误；
- 状态文件和真实资源不一致；
- CI 里失败，本地不失败；
- LocalStack / mock endpoint 行为和真实云不一致。

普通输出通常只会告诉你“失败了”，但不会完整展示 Terraform 内部的调用过程。

这时候就需要 `TF_LOG` 和 `TF_LOG_PATH`。

---

## 3. TF_LOG 是什么

`TF_LOG` 是 Terraform 读取的环境变量。

它控制 Terraform 是否输出内部日志，以及输出什么级别的日志。

常见用法：

```sh
export TF_LOG=TRACE
terraform plan
```

或者：

```sh
TF_LOG=TRACE terraform plan
```

常见日志级别可以理解为：

| 日志级别 | 含义 | 常见用途 |
|---|---|---|
| `ERROR` | 只看错误 | 信息最少，适合粗略定位 |
| `WARN` | 警告和错误 | 适合看潜在风险 |
| `INFO` | 普通运行信息 | 适合看主要流程 |
| `DEBUG` | 调试信息 | 适合排查 provider / core 行为 |
| `TRACE` | 最详细日志 | 深度排障，日志量最大 |

实际排障中最常用的是：

```sh
export TF_LOG=TRACE
```

原因很简单：既然已经开调试日志了，就别半开不开。真要查问题，通常直接上 `TRACE`。

---

## 4. TF_LOG_PATH 是什么

默认情况下，打开 `TF_LOG` 后，日志会直接输出到终端。

问题是：`TRACE` 日志非常多，终端很快就刷屏，很难回看。

所以可以用 `TF_LOG_PATH` 把日志写入文件：

```sh
export TF_LOG=TRACE
export TF_LOG_PATH=logs/terraform-trace.log
terraform plan
```

这样终端不会被大量调试日志污染，排障信息会写到：

```text
logs/terraform-trace.log
```

注意一个关键点：

> `TF_LOG_PATH` 通常需要配合 `TF_LOG` 使用。只设置 `TF_LOG_PATH`，但没有打开 `TF_LOG`，你可能拿不到想要的调试日志。

> Terraform 正常运行时，你会用标准命令；Terraform 异常运行时，你要能打开日志、收集证据、定位问题。

这就是从“会跑命令”进入“会排障”的分水岭。

## 6. 推荐实验跑法

### 6.1 创建日志目录

```sh
cd /workspace/practice/labs/116
mkdir -p logs
```

### 6.2 开启 TRACE 日志并写入文件

```sh
export TF_LOG=TRACE
export TF_LOG_PATH=logs/terraform-trace.log
```

### 6.3 执行 Terraform 命令

```sh
terraform init -input=false
terraform validate
terraform plan -input=false -no-color -out=tfplan
```

如果想观察 apply 阶段日志：

```sh
terraform apply -auto-approve tfplan
```

如果想观察 destroy 阶段日志：

```sh
terraform destroy -auto-approve
```

### 6.4 查看日志文件

```sh
ls -lh logs/
head -n 40 logs/terraform-trace.log
tail -n 80 logs/terraform-trace.log
```

如果日志很大，可以用：

```sh
grep -i "error" logs/terraform-trace.log
grep -i "provider" logs/terraform-trace.log
grep -i "diagnostic" logs/terraform-trace.log
```

## 7. Windows PowerShell 写法

如果你在 Windows PowerShell 里跑 Terraform，环境变量写法不是 `export`。

应该这样写：

```powershell
$env:TF_LOG="TRACE"
$env:TF_LOG_PATH="logs/terraform-trace.log"
terraform plan -input=false -no-color -out=tfplan
```

关闭日志：

```powershell
Remove-Item Env:TF_LOG
Remove-Item Env:TF_LOG_PATH
```

Bash / Linux / macOS 关闭日志：

```sh
unset TF_LOG
unset TF_LOG_PATH
```

## 8. 日志里重点看什么

打开 `TRACE` 后，日志非常多。新手最容易犯的错误是：从第一行看到最后一行，然后直接眼睛瞎掉。

正确做法是带着问题看。

### 8.1 init 失败时

重点看：

- provider source 地址；
- provider version 约束；
- plugin 下载路径；
- `.terraform.lock.hcl` 相关信息；
- 网络、证书、镜像源、缓存路径。

常见关键词：

```sh
grep -i "provider" logs/terraform-trace.log
grep -i "registry" logs/terraform-trace.log
grep -i "install" logs/terraform-trace.log
```

### 8.2 validate 失败时

重点看：

- HCL 解析；
- 变量引用；
- resource / data source schema；
- provider schema 加载。

不过 validate 的错误通常终端输出已经够清楚，日志只是辅助。

### 8.3 plan 失败时

重点看：

- Terraform 如何构建 dependency graph；
- 哪些 resource 进入 refresh；
- provider 是否成功读取远端状态；
- 哪个属性导致 diff；
- 哪个资源在 plan 阶段出现 unknown value。

常见关键词：

```sh
grep -i "diff" logs/terraform-trace.log
grep -i "refresh" logs/terraform-trace.log
grep -i "unknown" logs/terraform-trace.log
```

### 8.4 apply 失败时

重点看：

- 哪个 resource 正在创建、更新或删除；
- provider 调用了哪个 API；
- 远端 API 返回了什么错误；
- 是否出现重试；
- 是否有 timeout；
- 是否存在权限、配额、region、endpoint 问题。

常见关键词：

```sh
grep -i "error" logs/terraform-trace.log
grep -i "timeout" logs/terraform-trace.log
grep -i "retry" logs/terraform-trace.log
```

## 9. TF_LOG 和 TF_LOG_PATH 的真实工程价值

### 9.1 本地排障

本地跑不通 Terraform 时，可以快速打开日志。

典型场景：

```sh
export TF_LOG=TRACE
export TF_LOG_PATH=logs/debug.log
terraform plan -input=false -no-color
```

跑完之后，把日志保留下来，再慢慢分析。

### 9.2 CI/CD 排障

CI 里 Terraform 失败时，终端输出可能被截断。

这时候可以把日志写入文件，再作为 artifact 上传。

例如 GitHub Actions 里可以做：

```sh
export TF_LOG=TRACE
export TF_LOG_PATH=logs/terraform-ci-debug.log
terraform plan -input=false -no-color -out=tfplan
```

然后上传 `logs/terraform-ci-debug.log`。

但是 CI 里要特别小心：

> 调试日志可能包含敏感信息，不要随便公开上传。

### 9.3 Provider 问题定位

Terraform 的很多问题不一定是 Terraform Core 的问题，也可能是 provider 行为。

例如：

- AWS Provider；
- AzureRM Provider；
- Kubernetes Provider；
- Helm Provider；
- Random Provider；
- Local Provider。

调试日志可以帮你区分：

- Terraform Core 还没走到 provider；
- Terraform Core 已经调用 provider；
- provider schema 加载失败；
- provider 调远端 API 失败；
- provider 返回 diagnostic error。

## 10. 生产环境里要避开的坑

### 坑 1：一直开着 TF_LOG

不要把 `TF_LOG=TRACE` 长期开着。

日志会非常大，而且可能拖慢命令执行，还会污染终端输出。

正确做法：

```sh
export TF_LOG=TRACE
export TF_LOG_PATH=logs/debug.log
terraform plan
unset TF_LOG
unset TF_LOG_PATH
```

### 坑 2：把日志提交到 Git

不要把 Terraform 调试日志提交进仓库。

建议 `.gitignore` 加上：

```gitignore
logs/
*.log
```

尤其是云厂商 provider 的日志，可能包含：

- endpoint；
- resource id；
- account id；
- subscription id；
- request metadata；
- 部分敏感字段；
- CI 环境变量痕迹。

### 坑 3：只设置 TF_LOG_PATH

错误示例：

```sh
export TF_LOG_PATH=logs/debug.log
terraform plan
```

这不够。

更稳妥写法：

```sh
export TF_LOG=TRACE
export TF_LOG_PATH=logs/debug.log
terraform plan
```

### 坑 4：日志太大，不会筛选

`TRACE` 日志量巨大，不适合直接肉眼全读。

建议先搜关键词：

```sh
grep -i "error" logs/debug.log
grep -i "diagnostic" logs/debug.log
grep -i "provider" logs/debug.log
grep -i "timeout" logs/debug.log
```

### 坑 5：把日志当成测试结果

日志只是排障证据，不是测试本身。

真正的质量门禁仍然应该是：

```sh
terraform fmt -check
terraform validate
terraform test
terraform plan
```

日志解决的是：

> 出问题以后，我怎么知道它为什么出问题。

## 11. 和 terraform test 的关系

本节原始实验中提到：

```sh
terraform test
```

并且验证结果是：

```text
1 passed, 0 failed
```

这里要理解清楚：

- `terraform test` 是验证 Terraform 配置行为是否符合预期；
- `TF_LOG` 是排查 Terraform 命令内部执行细节；
- 两者不是替代关系。

更真实的工程组合是：

```sh
terraform test
```

如果失败，再打开：

```sh
export TF_LOG=TRACE
export TF_LOG_PATH=logs/test-debug.log
terraform test
```

这样就能在测试失败时留下更详细证据。

## 12. 本节可迁移到真实项目的方式

在真实项目中，你可以把这套流程沉淀成排障手册。

例如：

```sh
mkdir -p logs

export TF_LOG=TRACE
export TF_LOG_PATH=logs/$(date +%Y%m%d-%H%M%S)-terraform-debug.log

terraform init -input=false
terraform plan -input=false -no-color -out=tfplan
```

跑完以后立刻关闭：

```sh
unset TF_LOG
unset TF_LOG_PATH
```

再分析日志：

```sh
grep -i "error" logs/*.log
grep -i "provider" logs/*.log
grep -i "diagnostic" logs/*.log
```

这就是比较专业的 Terraform 排障姿势。

不是上来就“删状态、重 apply、重建资源”。

## 13. 面试表达方式

如果面试官问：

> Terraform 出问题你怎么排查？

你可以这样回答：

```text
我会先区分问题发生在哪个阶段：init、validate、plan、apply 还是 destroy。
常规输出不够时，我会用 TF_LOG 打开 Terraform 调试日志，并用 TF_LOG_PATH 写入文件，避免终端输出被刷掉。
然后根据失败阶段筛选日志，比如 provider、diagnostic、error、timeout、refresh、diff 等关键词。
如果是在 CI/CD 里，我会把日志作为受控 artifact 保留，但会避免泄露敏感信息，也不会把日志提交进 Git。
```

这个回答就比较像工程师，不像只会背命令。

## 16. 最终记忆点

这一节最重要的不是“会设置两个环境变量”，而是建立排障闭环：

```text
命令失败
  ↓
判断失败阶段
  ↓
打开 TF_LOG
  ↓
用 TF_LOG_PATH 保存日志
  ↓
按关键词筛选
  ↓
定位 Terraform Core / Provider / Backend / API / State 问题
  ↓
修复后关闭日志
```

记住一句话：

> `TF_LOG` 是 Terraform 的“黑匣子记录仪”，`TF_LOG_PATH` 是把黑匣子数据保存下来。平时别开，出事再开；开完要关，日志别乱传。
