# Terraform 实操训练 85：引用公共模块前的判断 —— 讲解版

## 1. 本节要解决什么问题

这一节的主题是 **引用公共模块前的判断**。

很多人刚学 Terraform module 时，会觉得公共模块很好用：

> 别人已经写好了 VPC、AKS、EKS、RDS、Storage Account，我直接拿来用不就完了？

生产环境里不能这么简单。公共模块确实能提高效率，但它本质上也是一种 **外部依赖**。只要是外部依赖，就要先判断：

- 它是否可信？
- 它是否维护活跃？
- 它是否符合你们公司的安全、命名、标签、网络、权限标准？
- 它是否会把资源生命周期绑死？
- 它的升级是否会带来破坏性变更？
- 它的抽象是否过度，导致出了问题很难排障？

所以，本节不是让你背“如何引用 module”，而是训练你建立一个工程判断：

> 公共模块可以用，但不能盲用。先评估，再封装，再测试，再上线。

---

## 2. 什么是公共模块

Terraform 公共模块通常指别人已经发布好的 module，例如：

- Terraform Registry 上的官方或社区模块
- GitHub 上开源的 Terraform module
- 云厂商或组织维护的模块
- 公司内部平台团队沉淀出来的共享模块

典型写法类似：

```hcl
module "network" {
  source  = "terraform-aws-modules/vpc/aws"
  version = "5.0.0"

  name = "demo-vpc"
  cidr = "10.0.0.0/16"
}
```

或者从 GitHub 引用：

```hcl
module "example" {
  source = "git::https://github.com/example/terraform-module.git?ref=v1.2.3"
}
```

这里最关键的不是 `source` 怎么写，而是：

> 你一旦引用它，它就会参与你的基础设施生命周期管理。

这意味着它生成的资源会进入 state，之后的 `plan`、`apply`、`destroy` 都会受它影响。

---

## 3. 为什么不能盲目引用公共模块

### 3.1 模块会隐藏复杂度

公共模块通常为了覆盖更多场景，会有大量变量和条件逻辑。

好处是灵活，坏处是你可能并不知道它实际创建了什么资源。

例如你以为只是创建一个 VPC，结果模块可能顺手创建了：

- Route Table
- NAT Gateway
- Internet Gateway
- Subnet
- Security Group
- Flow Log
- IAM Role
- CloudWatch Log Group

这在生产环境里非常危险。因为你以为自己只开了一扇门，实际它帮你修了一整栋楼。

### 3.2 模块升级可能带来破坏性变更

公共模块不是静态的。维护者可能会改变量、默认值、资源结构、输出字段。

如果你没有 pin 版本，而是直接引用最新版本，就可能出现：

```hcl
source = "terraform-aws-modules/vpc/aws"
```

这类写法的问题是：

- 今天 init 下载的是一个版本
- 过几个月 init 可能下载的是另一个版本
- 新版本可能改变资源行为
- `terraform plan` 可能突然出现大量变更

生产里最怕这种：

> 我没改代码，但 plan 变了。

所以公共模块必须固定版本。

### 3.3 模块默认值不一定符合你的安全标准

社区模块为了通用，默认值往往偏“易用”。

但公司内部通常有更严格要求，例如：

- 必须启用加密
- 必须加 tags
- 必须禁用公网访问
- 必须接入日志
- 必须限制 IAM 权限
- 必须符合命名规范
- 必须满足成本控制要求

公共模块的默认行为不一定满足这些要求。

直接拿来用，就是把别人的工程标准塞进你的生产环境。

### 3.4 模块抽象过深会增加排障难度

Terraform 出问题时，常见排障入口是：

```sh
terraform plan
terraform state list
terraform state show
terraform output
terraform console
```

如果模块层级太深，你看到的地址可能是：

```text
module.platform.module.network.module.subnets.aws_subnet.this["private-a"]
```

这时候新人很容易懵：

> 这个资源到底是谁创建的？变量从哪里传进来的？为什么要改这个？

所以公共模块不能只看“能不能跑”，还要看“出了事能不能看懂”。

## 4. 引用公共模块前要检查什么

可以按下面这个清单判断。

### 4.1 来源是否可信

优先级大致是：

1. 官方模块或云厂商维护模块
2. Terraform Registry 上高使用量、高星标、维护活跃的模块
3. 公司内部平台团队维护的模块
4. 普通 GitHub 个人项目
5. 不明来源模块

不是说个人项目一定不能用，而是生产环境要更谨慎。

关键看：

- 是否有清晰 README
- 是否有版本发布记录
- 是否有 issue / PR 活动
- 是否有测试
- 是否说明兼容的 Terraform 和 Provider 版本
- 是否有安全相关配置说明

### 4.2 是否固定版本

推荐：

```hcl
module "network" {
  source  = "terraform-aws-modules/vpc/aws"
  version = "5.8.1"
}
```

不推荐：

```hcl
module "network" {
  source = "terraform-aws-modules/vpc/aws"
}
```

GitHub 模块也要固定 tag：

```hcl
module "network" {
  source = "git::https://github.com/org/network-module.git?ref=v1.2.0"
}
```

不要长期引用分支：

```hcl
source = "git::https://github.com/org/network-module.git?ref=main"
```

这就像生产环境依赖 `latest` 镜像，迟早被背刺。

### 4.3 是否理解它会创建哪些资源

引用前先看：

- `main.tf`
- `variables.tf`
- `outputs.tf`
- `examples/`
- `README.md`
- `CHANGELOG.md`

然后在实验环境运行：

```sh
terraform init -input=false
terraform plan -input=false -no-color
```

重点不是让 plan 通过，而是观察：

- 创建了哪些资源
- 资源名称是否符合预期
- tags 是否齐全
- 是否有公网入口
- IAM 权限是否过大
- 是否有昂贵资源
- 是否有不可接受的默认配置

### 4.4 是否能通过测试约束关键行为

公共模块最好不要直接裸奔上线。

你可以用 `terraform test` 写断言，例如检查：

- 输出是否存在
- 命名是否符合规范
- tags 是否包含必要字段
- 环境变量是否正确传递
- 某些开关是否按预期关闭

测试的价值不是“证明 Terraform 一定没问题”，而是防止模块升级或参数变更时偷偷改变关键行为。

### 4.5 是否适合二次封装

生产中经常不建议业务团队直接引用外部公共模块，而是平台团队做一层内部封装。

例如外部模块：

```hcl
module "raw_vpc" {
  source  = "terraform-aws-modules/vpc/aws"
  version = "5.8.1"
}
```

平台团队封装后给业务团队使用：

```hcl
module "standard_network" {
  source = "../modules/standard-network"

  project = "payment"
  env     = "prod"
}
```

内部封装可以统一：

- 命名规范
- tags
- 安全默认值
- 日志策略
- 子网划分
- provider 版本
- 输出结构
- 测试断言

这才是平台工程的味道。

## 5. 本 lab 的实验理解

原始实验要求进入：

```sh
cd /workspace/practice/labs/85
```

然后执行完整 Terraform 生命周期：

```sh
terraform init -input=false
terraform fmt -check
terraform validate
terraform test
terraform plan -input=false -no-color -out=tfplan
terraform apply -auto-approve tfplan
terraform output
terraform destroy -auto-approve
```

这套流程对应的工程含义如下。

| 命令 | 工程意义 |
|---|---|
| `terraform init -input=false` | 初始化 provider/module，避免交互式卡住 CI |
| `terraform fmt -check` | 检查格式，保证代码风格一致 |
| `terraform validate` | 检查 Terraform 语法和配置基本合法性 |
| `terraform test` | 执行测试断言，验证关键行为 |
| `terraform plan -out=tfplan` | 生成可审查、可复用的执行计划 |
| `terraform apply tfplan` | 按保存的 plan 执行，避免 apply 时重新计算计划 |
| `terraform output` | 查看模块输出，验证对外暴露的接口 |
| `terraform destroy` | 清理资源，保持实验环境干净 |

这一套非常接近 CI/CD 里的 Terraform 工作流。

不是那种本地随手：

```sh
terraform apply
```

然后闭眼输入 `yes`。

生产环境里更推荐：

1. fmt
2. validate
3. test
4. plan
5. 人工或审批系统 review
6. apply saved plan
7. output / drift / policy 检查

## 6. Terraform test 在这里的价值

这一节前后都在讲 Terraform test，所以这里的重点是：

> 引用公共模块前，测试可以帮你守住关键约束。

例如你引用了一个公共模块，但你希望它输出的命名符合公司规范：

```hcl
run "check_name" {
  command = plan

  assert {
    condition     = output.resource_name == "demo-standard-resource"
    error_message = "resource_name output does not match expected naming convention."
  }
}
```

或者你希望模块必须包含环境信息：

```hcl
run "check_environment" {
  command = plan

  assert {
    condition     = output.environment == "dev"
    error_message = "environment output should be dev."
  }
}
```

这样做的意义是：

- 模块升级后，测试可以发现输出变化
- 参数调整后，测试可以发现命名变化
- 默认值被改后，测试可以发现行为变化
- 新人改代码时，不容易悄悄破坏约定

`terraform test` 很适合用来守住 module 的接口契约。

## 7. 状态文件角度理解公共模块

公共模块最终创建的资源也会进入 Terraform state。

例如：

```text
module.network.aws_subnet.private[0]
module.network.aws_route_table.private[0]
module.network.aws_nat_gateway.this[0]
```

这说明资源虽然写在 module 里，但生命周期仍然归当前 root module 管。

所以你要特别注意：

- 更换 module source 可能导致资源地址变化
- 升级 module 可能导致资源被替换
- 修改 `for_each` key 可能导致资源重建
- 删除 module block 可能计划销毁整批资源
- `moved` block 需要真实 HCL block 来迁移 state 地址

公共模块不是“外包给别人就没事了”。

Terraform state 会老老实实记录所有资源，出事故时也是你自己的环境被改。

## 8. 常见错误理解

### 错误 1：公共模块 star 多，所以一定安全

Star 多只能说明受欢迎，不代表适合你的生产标准。

你仍然要检查：

- 默认配置
- 权限边界
- 网络暴露
- 版本兼容
- 维护状态
- 成本影响

### 错误 2：模块越通用越好

不一定。

过度通用的模块往往变量非常多，条件判断复杂，调试难度高。

生产里更好的是：

> 对业务简单，对平台可控。

也就是平台团队可以维护复杂性，但业务团队使用时应该尽量简单。

### 错误 3：直接引用 main 分支没问题

这很危险。

`main` 是移动目标。你今天 init 和下个月 init 可能拿到不同代码。

生产环境要固定版本，升级要走 review。

### 错误 4：只要 terraform validate 通过就行

`validate` 只说明语法和配置结构基本合法。

它不保证：

- 资源符合安全标准
- 模块行为符合预期
- 默认值没有问题
- 输出接口稳定
- 成本可接受

所以还需要 `terraform test`、`plan review`、policy check、代码审查。

## 9. 面试表达方式

如果面试官问：

> Terraform 公共模块可以直接用吗？

可以这样回答：

> 可以用，但我不会直接盲用。公共模块本质上是基础设施代码依赖，引用前要评估来源、维护活跃度、版本锁定、默认安全配置、资源创建范围、升级兼容性和测试覆盖。生产里我更倾向于平台团队对公共模块做一层内部封装，统一命名、标签、安全默认值和输出接口，再通过 terraform test、plan review 和 CI 流程验证关键行为。

这个回答比单纯说“可以用 Terraform Registry”强很多。

它体现的是工程判断，而不是工具记忆。

## 10. 本节记忆口诀

引用公共模块前，记住这句话：

> 先看来源，再锁版本；先看 plan，再写 test；能封装就封装，别让业务直接裸奔。

拆开就是：

1. 来源可信
2. 版本固定
3. 行为可见
4. 安全默认值可控
5. 输出接口稳定
6. 测试断言覆盖关键约束
7. 升级过程可 review
8. state 迁移有计划
