# Terraform 实操训练 114：依赖锁文件讲解版

## 1. 本节你要真正学会什么

这一节的主题是 **Terraform 依赖锁文件**，也就是项目中常见的：

```text
.terraform.lock.hcl
```

这不是一个“可有可无的缓存文件”，而是 Terraform 项目里非常关键的工程文件。它的核心作用是：**锁定 provider 的具体版本和校验信息，让不同机器、不同 CI、不同时间运行 Terraform 时，尽量得到一致的 provider 依赖结果。**

一句话理解：

> `required_providers` 负责说明“我允许用哪些 provider 版本”，而 `.terraform.lock.hcl` 负责记录“这一次实际选中了哪个 provider 版本”。

这就是生产环境里最重要的区别。

---

## 2. 为什么依赖锁文件很重要

Terraform 不是只运行你写的 `.tf` 文件。它还需要下载 provider，例如：

- `hashicorp/local`
- `hashicorp/null`
- `hashicorp/aws`
- `hashicorp/azurerm`
- `hashicorp/kubernetes`

provider 是 Terraform 操作资源的插件。没有 provider，Terraform 就不知道怎么创建 AWS S3、Azure Resource Group、Kubernetes Deployment。

问题在于：provider 会不断发布新版本。如果你的项目只写：

```hcl
terraform {
  required_providers {
    local = {
      source  = "hashicorp/local"
      version = ">= 2.0.0"
    }
  }
}
```

这表示 Terraform 可以选择 `2.0.0` 以上的版本。今天初始化时可能选中 `2.5.1`，几个月后另一个人初始化时可能选中更高版本。

如果新版本 provider 行为变化了，你的 `plan` 可能突然变多、测试可能失败、甚至生产变更风险变高。

所以 Terraform 需要 `.terraform.lock.hcl` 来记录：

- 当前实际使用的 provider 版本
- provider 的来源地址
- provider 的校验哈希
- 不同平台下可验证的 checksum

这让团队协作和 CI/CD 更稳定。

---

## 3. `required_providers` 和 `.terraform.lock.hcl` 的区别

很多初学者会把这两个东西混在一起。直接记这个表：

| 文件 / 配置 | 作用 | 是否应该提交到 Git |
|---|---|---|
| `required_providers` | 声明允许使用哪些 provider，以及版本范围 | 是 |
| `.terraform.lock.hcl` | 锁定当前实际解析出来的 provider 版本和校验信息 | 是 |
| `.terraform/` | 本地下载的 provider、模块缓存目录 | 否 |
| `.terraform.tfstate` / `terraform.tfstate` | Terraform 状态文件 | 看 backend 策略，通常不直接提交 Git |

重点：

```text
required_providers = 版本约束
.terraform.lock.hcl = 版本锁定结果
```

这有点像软件项目中的：

- `package.json` vs `package-lock.json`
- `go.mod` vs `go.sum`
- `requirements.txt` vs lock file

Terraform 的锁文件就是基础设施依赖的“防漂移保险”。

## 7. 每条命令到底在验证什么

### 7.1 `terraform init -input=false`

这是本节最关键的命令。

它会做几件事：

1. 读取 `terraform` block 中的 `required_providers`。
2. 解析 provider 版本约束。
3. 下载符合条件的 provider。
4. 生成或更新 `.terraform.lock.hcl`。

第一次运行时，如果没有锁文件，Terraform 会根据版本约束选择一个合适版本并写入锁文件。

如果已经存在锁文件，Terraform 会优先按照锁文件中的版本来安装 provider，除非你主动升级。

---

### 7.2 `terraform fmt -check`

这个命令不关心 provider 版本本身，主要检查 `.tf` 文件格式。

如果失败，说明代码风格不符合 Terraform 标准格式。

修复方式：

```sh
terraform fmt -recursive
```

然后再执行：

```sh
terraform fmt -check
```

---

### 7.3 `terraform validate`

这个命令会检查配置是否合法，例如：

- provider 配置是否能被识别
- resource 类型是否存在
- variable / output / local 引用是否正确
- HCL 语法是否正确

注意：`validate` 通常不创建真实资源，它更像 Terraform 配置的静态/半静态检查。

---

### 7.4 `terraform test`

`terraform test` 用来运行测试断言。

本节验证结果是：

```text
1 passed, 0 failed
```

这说明当前 lab 的测试路径已经通过。

测试的价值在于：以后你改 provider 版本、改模块结构、改变量默认值，都可以用测试来检查行为有没有被破坏。

---

### 7.5 `terraform plan -out=tfplan`

`plan` 会计算 Terraform 准备做什么变更。

使用：

```sh
terraform plan -input=false -no-color -out=tfplan
```

这里有三个工程化习惯：

- `-input=false`：适合 CI/CD，避免命令卡住等待输入。
- `-no-color`：输出更适合日志系统保存。
- `-out=tfplan`：把计划保存下来，后续 apply 使用同一个计划文件。

生产环境里，这比直接 `terraform apply` 更稳。

---

### 7.6 `terraform apply tfplan`

```sh
terraform apply -auto-approve tfplan
```

这表示应用刚才保存下来的 plan。

这比重新计算 plan 更可控，因为你明确知道 apply 的就是前面审查过的内容。

---

### 7.7 `terraform destroy -auto-approve`

销毁实验资源，避免本地状态残留。

本节是 Docker 本地实验，风险很低；但生产环境里不要随手执行 destroy。

## 8. `.terraform.lock.hcl` 里面一般有什么

典型内容大概长这样：

```hcl
provider "registry.terraform.io/hashicorp/local" {
  version     = "2.5.1"
  constraints = ">= 2.0.0"
  hashes = [
    "h1:...",
    "zh:...",
  ]
}
```

你需要看懂三个东西。

### 8.1 `version`

实际锁定的 provider 版本。

例如：

```hcl
version = "2.5.1"
```

这表示 Terraform 当前实际使用的是 `2.5.1`。

---

### 8.2 `constraints`

来自配置中的版本约束。

例如：

```hcl
constraints = ">= 2.0.0"
```

这表示配置允许 `2.0.0` 以上版本，但锁文件当前实际锁到了某一个具体版本。

---

### 8.3 `hashes`

provider 包的校验信息。

它用于确认下载到的 provider 没有被篡改，也用于跨平台校验。

你不需要手写这些 hash，Terraform 会自动维护。

工程规则很简单：

```text
不要手动乱改 .terraform.lock.hcl 里的 hash。
```

---

## 9. 如何升级锁文件中的 provider

如果你只是运行：

```sh
terraform init
```

Terraform 通常会尊重已有锁文件，不会随便升级 provider。

如果你想主动升级 provider，要用：

```sh
terraform init -upgrade
```

这会让 Terraform 在版本约束允许的范围内重新解析更高版本，并更新 `.terraform.lock.hcl`。

推荐升级流程：

```sh
terraform init -upgrade
terraform fmt -check
terraform validate
terraform test
terraform plan -input=false -no-color -out=tfplan
```

如果测试和 plan 都合理，再考虑合并 PR。

不要在生产主分支上随便升级 provider。provider 升级属于依赖变更，应该像应用依赖升级一样走审查。

## 10. 生产环境里的正确做法

### 10.1 应该提交 `.terraform.lock.hcl`

建议提交到 Git：

```text
.terraform.lock.hcl
```

原因：

1. 保证团队成员用同一 provider 版本。
2. 保证 CI 和本地行为一致。
3. 降低 provider 新版本引入行为变化的风险。
4. 让依赖升级可审查、可回滚。

---

### 10.2 不应该提交 `.terraform/`

不要提交：

```text
.terraform/
```

原因：

1. 它是本地缓存目录。
2. 体积可能很大。
3. 不同操作系统、CPU 架构下内容可能不同。
4. CI/CD 应该自己执行 `terraform init` 拉取依赖。

`.gitignore` 通常应该包含：

```gitignore
.terraform/
*.tfstate
*.tfstate.*
tfplan
crash.log
crash.*.log
override.tf
override.tf.json
*_override.tf
*_override.tf.json
```

但不要把 `.terraform.lock.hcl` 忽略掉。

---

### 10.3 CI 中可以检查锁文件是否被意外修改

在 CI 里可以加检查：

```sh
terraform init -input=false
terraform validate
terraform test
```

如果团队想更严格，可以让 PR 显示 `.terraform.lock.hcl` 的 diff。

只要锁文件变化，就要问一句：

> 这是有意升级 provider，还是本地环境误改？

这就是成熟团队和“乱 apply 团队”的差距。

## 11. 常见坑

### 坑 1：把 `.terraform.lock.hcl` 加进 `.gitignore`

这是初学者常见错误。

结果就是每个人都可能解析出不同 provider 版本，CI 和本地也可能不一致。

正确做法：提交 `.terraform.lock.hcl`。

---

### 坑 2：以为 lock file 会锁住 Terraform CLI 版本

`.terraform.lock.hcl` 锁的是 provider，不是 Terraform CLI 本身。

Terraform CLI 版本通常在这里约束：

```hcl
terraform {
  required_version = ">= 1.11.0"
}
```

所以：

```text
.terraform.lock.hcl 锁 provider
required_version 约束 Terraform CLI
```

不要混。

---

### 坑 3：以为 lock file 可以替代版本约束

不能。

`required_providers` 仍然应该写清楚版本约束，例如：

```hcl
version = "~> 2.5"
```

锁文件记录当前实际版本，但配置文件仍然要表达你的版本策略。

---

### 坑 4：随手 `terraform init -upgrade`

`-upgrade` 会更新依赖解析结果。

这不是普通 init，而是依赖升级动作。

正确做法：

1. 单独开 PR。
2. 看 `.terraform.lock.hcl` diff。
3. 跑 `terraform test`。
4. 跑 `terraform plan`。
5. 确认没有异常变更后再合并。

---

### 坑 5：不同平台 checksum 不完整

如果团队里有人用 macOS，有人用 Linux，CI 用 Linux，可能会遇到跨平台 provider checksum 问题。

这类问题通常可以通过在对应平台执行 `terraform init` 或使用 lock file 平台参数补齐校验信息来解决。

核心思路：让锁文件包含团队和 CI 需要的平台校验信息。

## 12. 排障指南

### 12.1 `terraform init` 失败

优先检查：

1. provider source 是否写错。
2. provider version 约束是否过窄。
3. 网络是否能访问 registry。
4. lock file 是否和当前约束冲突。

可以尝试：

```sh
terraform init -input=false
```

如果是刻意升级，再用：

```sh
terraform init -upgrade
```

---

### 12.2 `terraform fmt -check` 失败

执行：

```sh
terraform fmt -recursive
```

然后重新检查。

---

### 12.3 `terraform test` 失败

优先检查：

1. output 名字是否被改了。
2. assertion 条件是否仍然成立。
3. provider 升级后返回值格式是否变化。
4. 测试是否依赖 apply 后才确定的值。

---

### 12.4 `plan/apply` 不一致

检查：

1. 是否使用了保存的 `tfplan`。
2. apply 前配置是否被改动。
3. provider 是否在 plan 和 apply 之间发生变化。
4. 是否有外部环境变化影响资源结果。

生产环境中，建议用：

```sh
terraform plan -out=tfplan
terraform apply tfplan
```

而不是 plan 一次、apply 时重新算一次。

执行：

```sh

rm -rf output logs .terraform .terraform.lock.hcl terraform.tfstate terraform.tfstate.backup
```

- 如果只是清理本地缓存，不要删 `.terraform.lock.hcl`。
