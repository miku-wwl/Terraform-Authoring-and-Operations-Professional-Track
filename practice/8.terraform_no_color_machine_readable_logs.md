# Terraform 实操训练 8：`-no-color` 与机器可读日志

## 1. 这节课到底在讲什么？

这一节讲的是 Terraform 在 **CI/CD、审批归档、安全审计、机器解析日志** 场景下，为什么要使用：

```sh
-no-color
```

Terraform 默认输出是给人看的，会带颜色。比如新增资源是绿色、删除资源是红色、变更资源是黄色。

人在终端里看很舒服，但 CI 系统、日志文件、文本扫描工具、审批系统并不喜欢这些颜色控制符。

所以这节课的核心是：

> **人在终端看，可以有颜色；机器保存、解析、审计，就应该用纯文本。**

---

## 2. 什么是 ANSI 控制符？

Terraform 的彩色输出本质上不是“真的颜色”，而是在文本里插入一些特殊控制字符。

例如你在终端看到：

```text
+ resource "local_file" "policy" {
    + filename = "output/no-color-policy.txt"
}
```

终端可能显示成绿色。

但如果这个输出被写入文件，文件里可能会出现类似这种东西：

```text
\x1b[32m+\x1b[0m resource "local_file" "policy"
```

这些 `\x1b[32m`、`\x1b[0m` 就是 ANSI 控制符。

它们对终端有意义：

- `\x1b[32m`：开始显示绿色
- `\x1b[0m`：恢复默认颜色

但是对日志文件、审批系统、扫描工具来说，这些就是“脏字符”。

---

## 3. 为什么 CI 里要加 `-no-color`？

在 CI 里，Terraform 输出通常会被保存到：

- GitHub Actions 日志
- GitLab CI 日志
- Jenkins 日志
- build artifact
- 审批材料
- 安全扫描系统
- 变更审计系统

如果不加 `-no-color`，日志里可能混入 ANSI 控制符，带来几个问题：

### 3.1 人看起来很脏

保存后的日志可能不是干净文本，而是各种乱码控制符。

### 3.2 机器解析困难

比如你想用脚本检查日志里是否包含：

```text
Plan: 1 to add, 0 to change, 0 to destroy.
```

如果中间混入控制符，简单的字符串匹配可能失败。

### 3.3 审计材料不专业

平台团队把 Terraform plan 输出作为变更审批材料时，审批人需要看到的是稳定、干净、可复制、可归档的文本，而不是带终端控制符的输出。

所以 CI 环境里推荐这样写：

```sh
terraform plan -input=false -no-color -out=tfplan
```

---

## 4. `-input=false` 是干什么的？

这节实验验证的 CI plan 命令是：

```sh
terraform plan -input=false -no-color -out=tfplan
```

其中：

```sh
-input=false
```

表示不要让 Terraform 在执行过程中等待用户输入。

这对 CI 非常重要。

如果 Terraform 在 CI 里问你：

```text
Do you want to perform these actions?
```

或者等待变量输入，流水线就会卡死。

所以 CI 里常见写法是：

```sh
terraform init -input=false
terraform plan -input=false -no-color -out=tfplan
```

核心原则：

> **CI 不能等人手动输入。**

---

## 5. `-out=tfplan` 是干什么的？

```sh
-out=tfplan
```

表示把这次 plan 的结果保存成一个 plan 文件。

这个文件通常叫：

```text
tfplan
```

它的价值是：

> **先计划，后审批，再按同一个计划 apply。**

也就是说：

```sh
terraform plan -input=false -no-color -out=tfplan
terraform apply -auto-approve tfplan
```

这样可以保证 apply 执行的就是之前审批过的那个 plan。

这比直接：

```sh
terraform apply -auto-approve
```

更适合生产环境。

---

## 6. 重要细节：`tfplan` 本身不是普通文本

这里有一个容易踩坑的点。

```sh
terraform plan -out=tfplan
```

生成的 `tfplan` 文件不是普通文本，而是 Terraform 的二进制 plan 文件。

所以你不能把它当成审批人直接阅读的文件。

如果平台团队要归档“人能读懂的审批材料”，更推荐这样做：

```sh
terraform plan -input=false -no-color -out=tfplan
terraform show -no-color tfplan > output/tfplan.txt
```

这样会得到两个文件：

```text
tfplan              # Terraform apply 使用的二进制计划文件
output/tfplan.txt   # 审批人和审计系统阅读的纯文本计划
```

更专业的 CI 审批流一般是：

```sh
terraform init -input=false
terraform fmt -check
terraform validate
terraform plan -input=false -no-color -out=tfplan
terraform show -no-color tfplan > output/tfplan.txt
```

审批通过后：

```sh
terraform apply -auto-approve tfplan
```

这就是比较像生产环境的做法。

## 7. 这节实验在做什么？

实验背景是：

> 平台团队要把 Terraform plan 输出作为审批材料归档。

所以你要验证：

1. Terraform 输出可以生成干净、无颜色控制符的文本。
2. CI plan 命令里必须包含：
   - `-input=false`
   - `-no-color`
   - `-out=tfplan`
3. 本地会生成一个策略文件：
   - `output/no-color-policy.txt`
4. `terraform test` 能验证配置是否符合要求。

---

## 8. 实验文件说明

本节实验目录是：

```text
practice/labs/8/
├── main.tf
├── outputs.tf
├── tests/
│   └── no_color.tftest.hcl
└── README.md
```

大概职责如下。

### 8.1 `main.tf`

负责定义本地资源，大概率会创建：

```text
output/no-color-policy.txt
```

这个文件用于模拟平台团队生成的“无颜色输出策略”。

### 8.2 `outputs.tf`

负责输出实验结果，例如策略文件路径、CI plan 命令等。

### 8.3 `tests/no_color.tftest.hcl`

这是 Terraform 原生测试文件。

它会检查你的配置是否满足实验要求，尤其是类似：

```hcl
plan_capture_commands.ci_plan
```

里面是否包含：

```text
terraform plan -input=false -no-color -out=tfplan
```

### 8.4 `README.md`

记录实验说明、运行方式和预期结果。

## 9. 运行手册逐行解释

原始运行命令如下：

```sh
cd /workspace/practice/labs/8
terraform init -input=false
terraform fmt -check
terraform validate
terraform test
terraform plan -input=false -no-color -out=tfplan
terraform apply -auto-approve tfplan
terraform output
terraform destroy -auto-approve
```

下面逐行拆开讲。

---

### 9.1 进入实验目录

```sh
cd /workspace/practice/labs/8
```

进入第 8 节实验目录。

---

### 9.2 初始化 Terraform

```sh
terraform init -input=false
```

初始化工作目录，下载 provider，准备 `.terraform` 目录。

`-input=false` 表示不要交互式输入，符合 CI 习惯。

---

### 9.3 检查格式

```sh
terraform fmt -check
```

只检查格式，不自动修改。

如果格式不符合 Terraform 标准，会失败。

CI 里常用这个命令保证代码风格一致。

---

### 9.4 校验配置

```sh
terraform validate
```

检查 Terraform 配置语法、provider schema、引用关系是否合理。

注意：

```sh
terraform validate
```

不会真的创建资源。

---

### 9.5 执行 Terraform 测试

```sh
terraform test
```

运行 `tests/no_color.tftest.hcl`。

这一步是本节实验的关键。

它不是只测试资源能不能创建，而是测试你是否按照“平台团队 CI 规范”写了配置。

比如它可能检查：

```text
ci_plan 命令必须包含 -input=false -no-color -out=tfplan
```

通过结果是：

```text
1 passed, 0 failed
```

---

### 9.6 生成 plan 文件

```sh
terraform plan -input=false -no-color -out=tfplan
```

这一步做三件事：

1. `-input=false`：不等待用户输入。
2. `-no-color`：输出纯文本。
3. `-out=tfplan`：保存计划文件。

看到类似输出：

```text
Plan: 1 to add, 0 to change, 0 to destroy.
```

说明 Terraform 计划创建 1 个资源。

---

### 9.7 按保存的 plan 执行 apply

```sh
terraform apply -auto-approve tfplan
```

这里不是重新计算 plan，而是执行刚才保存的 `tfplan`。

这代表：

> apply 的内容和 plan 审批时看到的内容一致。

这就是生产环境里常见的审批控制思想。

---

### 9.8 查看输出

```sh
terraform output
```

查看 outputs.tf 定义的输出结果。

可能会看到策略文件路径、CI 命令等信息。

---

### 9.9 销毁资源

```sh
terraform destroy -auto-approve
```

销毁实验创建的本地资源。

这节实验创建的是本地文件，所以 destroy 后：

```text
output/no-color-policy.txt
```

会被删除。

## 10. 这节实验的核心命令

最核心的是这句：

```sh
terraform plan -input=false -no-color -out=tfplan
```

你可以这样记：

| 参数 | 作用 | CI 价值 |
|---|---|---|
| `-input=false` | 禁止交互式输入 | 防止流水线卡死 |
| `-no-color` | 禁止颜色控制符 | 日志干净，方便审计和解析 |
| `-out=tfplan` | 保存计划文件 | 确保 apply 使用审批过的 plan |

一句话总结：

> `-input=false` 是为了 CI 不堵车，`-no-color` 是为了日志不乱码，`-out=tfplan` 是为了 apply 不偷换计划。

## 11. 审批归档的推荐写法

如果是比较正式的 CI/CD，可以这样写：

```sh
mkdir -p output

terraform init -input=false
terraform fmt -check
terraform validate
terraform plan -input=false -no-color -out=tfplan
terraform show -no-color tfplan > output/tfplan.txt
```

然后把这些作为 artifact 保存：

```text
tfplan
output/tfplan.txt
```

审批人看：

```text
output/tfplan.txt
```

机器 apply：

```text
tfplan
```

审批通过后：

```sh
terraform apply -auto-approve tfplan
```

## 12. 常见错误

### 12.1 只保存默认彩色输出

错误示例：

```sh
terraform plan > plan.txt
```

问题是输出里可能有 ANSI 控制符。

更好：

```sh
terraform plan -no-color > plan.txt
```

---

### 12.2 以为 `tfplan` 是文本文件

错误理解：

> `tfplan` 可以直接给审批人看。

不对。

`tfplan` 是 Terraform 使用的计划文件，不适合作为人类审批材料。

更好的做法：

```sh
terraform show -no-color tfplan > output/tfplan.txt
```

---

### 12.3 CI 里遗漏 `-input=false`

错误示例：

```sh
terraform plan -no-color -out=tfplan
```

问题是某些情况下 Terraform 可能等待输入，导致 CI 卡住。

更好：

```sh
terraform plan -input=false -no-color -out=tfplan
```

---

### 12.4 apply 时不用保存的 plan

不推荐：

```sh
terraform plan -out=tfplan
terraform apply -auto-approve
```

这里 `apply` 可能重新计算计划。

更好：

```sh
terraform plan -out=tfplan
terraform apply -auto-approve tfplan
```

## 13. 和真实生产环境的关系

这节看起来很小，只是一个 `-no-color` 参数。

但它背后其实是平台工程里的一个重要思想：

> **Terraform 不只是人在本地敲命令，而是要进入标准化、可审计、可追踪的工程流水线。**

生产环境 Terraform 流水线通常会强调：

- 不允许交互式输入
- plan 和 apply 分离
- plan 结果必须归档
- 审批人看到的是干净文本
- apply 必须基于审批过的 plan
- 日志要能被机器扫描
- 输出要适合审计留痕

所以这一节不是“背参数”，而是在练 CI/CD Terraform 的工程习惯。

## 15. 面试表达

如果面试官问：

> Terraform 在 CI 里为什么要用 `-no-color`？

可以回答：

```text
Terraform 默认输出带 ANSI color codes，适合终端阅读，但不适合 CI artifact、审批归档和日志解析。
在 CI 里我会加 -no-color，确保 plan 输出是纯文本，方便审计、grep、扫描和长期归档。
同时我会配合 -input=false 避免流水线卡住，并用 -out=tfplan 保存计划文件，审批通过后用 terraform apply tfplan 执行同一个 plan。
如果需要可读审批材料，我会再执行 terraform show -no-color tfplan > tfplan.txt。
```

这就是比较成熟的工程回答。

## 16. 本节最终结论

这节课你要掌握的不是一个孤立参数，而是一套 CI Terraform 习惯：

```sh
terraform plan -input=false -no-color -out=tfplan
terraform show -no-color tfplan > output/tfplan.txt
terraform apply -auto-approve tfplan
```

其中：

- `-input=false`：防止 CI 等待输入。
- `-no-color`：让日志和审批材料保持纯文本。
- `-out=tfplan`：保存被审批的计划。
- `terraform show -no-color tfplan`：把二进制 plan 转成可读纯文本。
- `terraform apply tfplan`：执行审批过的计划，避免 plan/apply 不一致。

一句话：

> **这是从“本地手敲 Terraform”升级到“CI/CD 可审计 Terraform”的关键一步。**
