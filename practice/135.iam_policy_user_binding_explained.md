# 第 135 节讲解：IAM Policy 与用户绑定

## 1. 本节在讲什么

这一节的核心是：**用 Terraform 创建 IAM Policy，然后把这个 Policy 绑定到指定 IAM User 上**。

你要掌握的不是“真实 AWS 账号里 IAM 到底怎么完全生效”，而是先把 Terraform 资源声明、依赖关系、plan/apply/destroy 路径跑通。

本节实验环境使用 LocalStack：

- AWS API endpoint：`http://localhost:4566`
- 假凭证：`test/test`
- 不访问真实 AWS
- 不产生真实云资源费用
- 重点训练 Terraform AWS Provider 的写法和资源生命周期

可以把它理解成：  
**LocalStack 是 AWS API 的练习靶场，Terraform 是你操作基础设施的声明式工具。**

---

## 2. 为什么 IAM Policy 要单独学

IAM 是 AWS 里最容易“看起来简单，实际很坑”的部分。

一个完整的权限绑定通常至少包含三层：

1. **Principal**：谁拿权限，比如 IAM user、role、group。
2. **Policy**：允许或拒绝什么动作，比如 `s3:ListBucket`。
3. **Attachment / Binding**：把权限策略挂到某个主体上。

这一节聚焦的是：

```text
IAM User  +  IAM Policy  +  Policy Attachment
```

也就是：

```text
创建用户
创建策略
把策略绑定给用户
输出 user name / policy arn
```

考试和项目里经常混淆的点是：  
**Policy 本身只是权限规则，不绑定给任何主体时，它不会让任何人自动获得权限。**

---

## 3. Terraform 里通常会用到哪些资源

本节常见的 Terraform 资源组合如下：

```hcl
resource "aws_iam_user" "demo" {
  name = "demo-user"
}

resource "aws_iam_policy" "demo" {
  name        = "demo-policy"
  description = "Demo IAM policy"

  policy = jsonencode({
    Version = "2012-10-17"
    Statement = [
      {
        Effect = "Allow"
        Action = [
          "s3:ListBucket"
        ]
        Resource = "*"
      }
    ]
  })
}

resource "aws_iam_user_policy_attachment" "demo" {
  user       = aws_iam_user.demo.name
  policy_arn = aws_iam_policy.demo.arn
}
```

这里最关键的是第三个资源：

```hcl
aws_iam_user_policy_attachment
```

它负责把：

```text
aws_iam_policy.demo.arn
```

绑定到：

```text
aws_iam_user.demo.name
```

上。

没有这个 attachment，user 和 policy 就只是两个孤立资源。

---

## 4. 为什么推荐用 `jsonencode`

IAM Policy 本质上是 JSON 文档。你可以直接写 JSON 字符串，但在 Terraform 里更推荐用 `jsonencode`。

### 直接写 JSON 的问题

```hcl
policy = <<EOF
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": ["s3:ListBucket"],
      "Resource": "*"
    }
  ]
}
EOF
```

这种写法能用，但有几个问题：

- 逗号、引号、括号容易写错。
- 变量插值时可读性差。
- Terraform 很难帮你检查结构。
- 长 policy 会变成一坨 JSON 字符串。

### `jsonencode` 的好处

```hcl
policy = jsonencode({
  Version = "2012-10-17"
  Statement = [
    {
      Effect   = "Allow"
      Action   = ["s3:ListBucket"]
      Resource = "*"
    }
  ]
})
```

优点是：

- 用 HCL 写结构，更像 Terraform 原生代码。
- 可以自然引用变量和资源。
- 少写很多双引号。
- 结构更清楚，适合维护。

记忆方式：

```text
Terraform 里写 IAM policy：优先 jsonencode。
```

## 5. IAM Policy 的基本结构

一个最小 IAM Policy 通常长这样：

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": ["s3:ListBucket"],
      "Resource": "*"
    }
  ]
}
```

字段解释：

| 字段 | 含义 |
|---|---|
| `Version` | Policy 语言版本，通常写 `2012-10-17` |
| `Statement` | 权限声明列表 |
| `Effect` | `Allow` 或 `Deny` |
| `Action` | 允许或拒绝的 API 动作 |
| `Resource` | 权限作用的资源 ARN |

这一节重点不是让你设计完美 IAM 策略，而是让你知道：

```text
Policy 是一段权限规则。
Attachment 是把规则挂给用户。
```

## 6. LocalStack 能验证什么，不能验证什么

本节使用 LocalStack 做本地模拟，这是非常适合练 Terraform 的方式。

### 可以验证

LocalStack 可以帮你验证：

- Terraform AWS Provider endpoint 配置是否正确。
- IAM user 是否能被声明和创建。
- IAM policy 是否能被声明和创建。
- user-policy attachment 是否能被 Terraform 管理。
- `terraform plan/apply/destroy` 生命周期是否闭环。
- output 是否能正确输出 user name 和 policy ARN。

### 不能完全替代真实 AWS 的部分

LocalStack 不能完全替代真实 AWS 的 IAM 权限评估。

例如：

- 真实 AWS 的 IAM policy evaluation。
- 显式 deny 与 allow 的复杂优先级。
- SCP、permission boundary、session policy 的组合效果。
- 跨账号访问。
- 真实服务级别权限边界。
- CloudTrail 中的真实审计链路。

所以本节的正确定位是：

```text
LocalStack 验证 Terraform 写法和资源依赖。
真实 AWS 验证 IAM 权限语义和生产行为。
```

别把靶场当战场，这点要记住。

## 7. 本节实验流程

建议按下面顺序理解实验。

### 第一步：准备 provider

典型 LocalStack Provider 配置会把 AWS API 指向本地：

```hcl
provider "aws" {
  region                      = "us-east-1"
  access_key                  = "test"
  secret_key                  = "test"
  skip_credentials_validation = true
  skip_metadata_api_check     = true
  skip_requesting_account_id  = true

  endpoints {
    iam = "http://localhost:4566"
  }
}
```

核心点是：

```text
Terraform 以为自己在调用 AWS，实际上请求发给了 LocalStack。
```

### 第二步：创建 IAM user

```hcl
resource "aws_iam_user" "demo" {
  name = "demo-user"
}
```

这个资源代表一个 IAM 用户。

### 第三步：创建 IAM policy

```hcl
resource "aws_iam_policy" "demo" {
  name = "demo-policy"

  policy = jsonencode({
    Version = "2012-10-17"
    Statement = [
      {
        Effect   = "Allow"
        Action   = ["s3:ListBucket"]
        Resource = "*"
      }
    ]
  })
}
```

这个资源只创建策略，不代表用户已经获得权限。

### 第四步：绑定 policy 到 user

```hcl
resource "aws_iam_user_policy_attachment" "demo" {
  user       = aws_iam_user.demo.name
  policy_arn = aws_iam_policy.demo.arn
}
```

这一步才是绑定动作。

Terraform 通过资源引用自动推导依赖：

```text
attachment 依赖 user 和 policy
```

所以正常情况下不需要手写 `depends_on`。

### 第五步：输出结果

```hcl
output "user_name" {
  value = aws_iam_user.demo.name
}

output "policy_arn" {
  value = aws_iam_policy.demo.arn
}
```

输出的作用是让你快速确认：

- user 是否创建成功。
- policy 是否创建成功。
- policy ARN 是否可用于绑定或排查。

## 8. Terraform 依赖关系怎么理解

这一节非常适合理解 Terraform 的隐式依赖。

比如：

```hcl
user       = aws_iam_user.demo.name
policy_arn = aws_iam_policy.demo.arn
```

这两行不只是拿值，它们还告诉 Terraform：

```text
attachment 必须等 user 和 policy 创建完成后才能创建。
```

销毁时顺序反过来：

```text
先删除 attachment
再删除 policy
再删除 user
```

因为如果 policy 还挂在 user 上，直接删 policy 可能会失败。

这就是 Terraform 图模型的价值：

```text
引用关系 = 依赖关系
```

## 9. `aws_iam_policy` 和 inline policy 的区别

IAM user 绑定 policy 常见有两种方式。

### 方式一：Managed Policy + Attachment

也就是本节方式：

```text
aws_iam_policy
+
aws_iam_user_policy_attachment
```

特点：

- policy 是独立资源。
- 可以复用给多个 user、role、group。
- 适合标准化权限。
- 更接近企业治理方式。

### 方式二：Inline Policy

例如：

```hcl
resource "aws_iam_user_policy" "demo" {
  name = "demo-inline-policy"
  user = aws_iam_user.demo.name

  policy = jsonencode({
    Version = "2012-10-17"
    Statement = []
  })
}
```

特点：

- policy 直接嵌在某个 user 上。
- 不方便复用。
- 生命周期跟 user 更强绑定。
- 小实验可以用，但企业场景里要谨慎。

记忆方式：

```text
要复用、要治理：managed policy。
只给单个主体临时用：inline policy。
```

## 10. 考试容易怎么考

这一节可能不会直接问很复杂的代码，但会考你对概念的判断。

### 高频考点 1：Policy 不等于权限已经生效

题目说：

> 创建了 IAM policy，为什么 user 还是没有权限？

答案方向：

```text
因为 policy 还没有 attach 到 user、group 或 role。
```

### 高频考点 2：jsonencode 的作用

题目问：

> Terraform 中如何更安全地生成 JSON policy？

答案方向：

```text
使用 jsonencode 把 HCL 对象转换成 JSON 字符串。
```

### 高频考点 3：最小权限原则

题目给一个 policy：

```json
"Action": "*",
"Resource": "*"
```

然后问是否适合生产。

答案方向：

```text
不适合。权限太大，违背 least privilege。
```

### 高频考点 4：资源依赖

题目问：

> 为什么 attachment 会在 user 和 policy 后创建？

答案方向：

```text
因为 attachment 引用了 user.name 和 policy.arn，Terraform 自动建立隐式依赖。
```

## 11. 常见错误和排查

### 错误 1：只创建了 policy，忘记 attachment

现象：

```text
Terraform apply 成功，但用户没有获得策略。
```

原因：

```text
aws_iam_policy 只是创建策略，未绑定主体。
```

解决：

```hcl
resource "aws_iam_user_policy_attachment" "demo" {
  user       = aws_iam_user.demo.name
  policy_arn = aws_iam_policy.demo.arn
}
```

### 错误 2：policy JSON 格式错误

现象：

```text
MalformedPolicyDocument
```

解决：

优先使用：

```hcl
jsonencode({...})
```

不要手写大段 JSON 字符串。

### 错误 3：Action 写错

例如：

```hcl
Action = ["s3:ListBuckets"]
```

正确动作通常是：

```hcl
Action = ["s3:ListBucket"]
```

AWS IAM action 名字很严格，多一个字母都不行。

### 错误 4：真实 AWS 里权限仍不生效

可能原因：

- 还有 explicit deny。
- 受 SCP 限制。
- 受 permission boundary 限制。
- Resource ARN 写错。
- 用户实际使用的不是这个 IAM user。
- 临时凭证还没刷新。
- 服务本身还需要额外资源级权限。

这也是为什么 LocalStack 练写法可以，但生产 IAM 还得在真实 AWS 里严肃验证。

## 12. 面试和真实项目怎么讲

你可以这样表达：

> 我用 Terraform 管理 IAM user 和 managed policy，通过 `jsonencode` 生成 policy document，再使用 `aws_iam_user_policy_attachment` 进行绑定。资源之间通过 `user.name` 和 `policy.arn` 建立隐式依赖，Terraform 可以正确处理创建和销毁顺序。LocalStack 可以用于验证 Terraform 声明、provider endpoint、plan/apply/destroy 闭环，但真实 IAM policy evaluation 仍需要在真实 AWS 环境中验证。

这段话比只说“我会写 IAM”强很多。

因为它体现了：

- 你知道 Terraform 资源模型。
- 你知道 IAM policy 和 attachment 的关系。
- 你知道 LocalStack 的边界。
- 你知道生产环境不能乱授权。

## 14. 自测题

### 题 1

只创建 `aws_iam_policy`，是否意味着某个 user 自动获得权限？

答案：否。还需要 attachment，或者通过 group/role 等方式关联。

### 题 2

`jsonencode` 的作用是什么？

答案：把 Terraform HCL 对象转换成 JSON 字符串，适合生成 IAM policy document。

### 题 3

`aws_iam_user_policy_attachment` 需要哪些关键信息？

答案：

```text
user
policy_arn
```

### 题 4

LocalStack 能完全模拟真实 AWS IAM 权限评估吗？

答案：不能。它适合验证 Terraform 资源路径和本地 API 模拟，但真实权限语义需要真实 AWS 验证。

### 题 5

为什么 attachment 通常不需要手写 `depends_on`？

答案：因为它引用了 `aws_iam_user.demo.name` 和 `aws_iam_policy.demo.arn`，Terraform 会自动建立隐式依赖。

## 15. 本节最终掌握标准

学完这一节，你应该能做到：

- 看懂 IAM user、policy、attachment 三者关系。
- 用 `jsonencode` 写基础 IAM policy。
- 明白 managed policy 和 inline policy 的区别。
- 知道 attachment 才是绑定动作。
- 能解释 Terraform 的隐式依赖。
- 能区分 LocalStack 练习边界和真实 AWS IAM 行为。
- 能把这节内容迁移到真实项目里的最小权限设计。

这一节不是难在代码，而是难在别把概念混了。  
**Policy 是“权限说明书”，Attachment 才是“把说明书发给某个人”。**
