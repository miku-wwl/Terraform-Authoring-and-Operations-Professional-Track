# Terraform 实操训练 103：为已有资源启用 for_each（讲解版）

## 1. 这一节到底在讲什么？

本节核心主题是：

> 把已经存在、参数不同但结构相似的 AWS 资源，重构成 `for_each` map 管理，并用 `moved block` 保留 Terraform state 里的地址迁移关系。

换成人话就是：

```text
以前：每个资源单独写一个 resource block
现在：用一个 resource + for_each 管一组资源
风险：Terraform 可能以为旧资源没了、新资源来了
解决：用 moved block 告诉 Terraform：它们是同一批资源，只是地址改了
```

这不是单纯语法题，而是 Terraform 生产重构题。真正的重点是：**代码可以重构，但云上已有资源不能因为地址变化被误删重建。**

---

## 2. 为什么要把已有资源改成 for_each？

假设一开始你有两个 S3 bucket：

```hcl
resource "aws_s3_bucket" "logs" {
  bucket = "demo-logs"
}

resource "aws_s3_bucket" "assets" {
  bucket = "demo-assets"
}
```

资源少的时候这样写很直观。但资源一多，代码就会变成复制粘贴地狱。

例如：

```hcl
resource "aws_s3_bucket" "logs" {}
resource "aws_s3_bucket" "assets" {}
resource "aws_s3_bucket" "backup" {}
resource "aws_s3_bucket" "archive" {}
resource "aws_s3_bucket" "reports" {}
```

这时候更适合改成：

```hcl
locals {
  buckets = {
    logs = {
      bucket = "demo-logs"
    }

    assets = {
      bucket = "demo-assets"
    }
  }
}

resource "aws_s3_bucket" "this" {
  for_each = local.buckets

  bucket = each.value.bucket
}
```

这样资源地址会变成：

```text
aws_s3_bucket.this["logs"]
aws_s3_bucket.this["assets"]
```

`for_each` 的好处是：

- 用 map 表达一组资源
- 用 key 表达每个资源的身份
- 减少重复代码
- 更适合模块化
- 更适合团队维护
- 更接近生产级 Terraform 写法

---

## 3. 最大风险：Terraform 地址变了

Terraform 判断“是不是同一个资源”，不是靠你心里觉得是不是同一个，而是看 state 里的资源地址。

重构前的地址可能是：

```text
aws_s3_bucket.logs
aws_s3_bucket.assets
```

重构后变成：

```text
aws_s3_bucket.this["logs"]
aws_s3_bucket.this["assets"]
```

从 Terraform 的视角看：

```text
旧地址消失了
新地址出现了
```

如果你不告诉 Terraform 它们之间的关系，Terraform 可能计划：

```text
destroy aws_s3_bucket.logs
destroy aws_s3_bucket.assets
create  aws_s3_bucket.this["logs"]
create  aws_s3_bucket.this["assets"]
```

这就是生产环境最危险的地方：你只是想“重构代码”，Terraform 却可能理解成“删了再建”。

## 4. moved block 是什么？

`moved block` 的作用是声明资源地址迁移。

它告诉 Terraform：

> 这个资源不是被删除了，也不是新建了，只是从旧地址搬到了新地址。

示例：

```hcl
moved {
  from = aws_s3_bucket.logs
  to   = aws_s3_bucket.this["logs"]
}

moved {
  from = aws_s3_bucket.assets
  to   = aws_s3_bucket.this["assets"]
}
```

含义是：

```text
aws_s3_bucket.logs   -> aws_s3_bucket.this["logs"]
aws_s3_bucket.assets -> aws_s3_bucket.this["assets"]
```

这样 Terraform 会更新 state 里的地址关系，而不是对真实 AWS 资源执行 destroy/create。

一句话记忆：

```text
moved block 是 Terraform 重构资源地址时的安全带。
```

## 5. for_each 和 count 的区别

`count` 也能创建多个资源：

```hcl
resource "aws_s3_bucket" "this" {
  count  = 2
  bucket = var.bucket_names[count.index]
}
```

它的地址是：

```text
aws_s3_bucket.this[0]
aws_s3_bucket.this[1]
```

问题是：`count` 依赖数字下标。

如果列表顺序变了，Terraform 可能误以为资源身份变了。

而 `for_each` 的地址是：

```text
aws_s3_bucket.this["logs"]
aws_s3_bucket.this["assets"]
```

它依赖稳定 key，不依赖顺序。

所以生产里如果资源有明确业务身份，比如：

- `logs`
- `assets`
- `backup`
- `public_subnet_a`
- `private_subnet_b`
- `app_role`
- `audit_bucket`

通常更适合用 `for_each`。

---

## 6. for_each 的 key 是资源身份

很多人学 `for_each` 只记语法，但生产里真正重要的是 key 设计。

例如：

```hcl
for_each = {
  logs   = {...}
  assets = {...}
}
```

这里的 `logs` 和 `assets` 会成为 Terraform 地址的一部分：

```text
aws_s3_bucket.this["logs"]
aws_s3_bucket.this["assets"]
```

所以 key 不要乱取。

不推荐：

```hcl
for_each = {
  item1 = {...}
  item2 = {...}
}
```

更推荐：

```hcl
for_each = {
  app_logs      = {...}
  static_assets = {...}
  audit_archive = {...}
}
```

好的 key 应该：

- 表达业务身份
- 长期稳定
- 不依赖列表顺序
- 团队成员看得懂
- 不轻易改名

因为 key 一变，Terraform 地址就变；地址一变，就可能又要做 state 迁移。

## 7. 安全重构流程

把已有资源改成 `for_each`，建议按这个流程来：

```text
1. 查看当前 state 地址
2. 设计新的 for_each map
3. 把多个 resource 重构成 for_each
4. 添加 moved block
5. 执行 terraform plan
6. 确认没有 destroy/create
7. apply 更新 state 地址
8. 再次查看 state list
```

常用命令：

```sh
terraform state list
terraform plan -input=false -no-color
terraform apply
terraform state list
```

最重要的是看 plan。

如果你只是做代码重构，但 plan 出现：

```text
Plan: 2 to add, 0 to change, 2 to destroy
```

或者：

```text
-/+ destroy and then create replacement
```

那就不要硬 apply。Terraform 已经在提醒你：这不是单纯地址迁移了。

## 8. moved block 和 terraform state mv 的区别

以前常用这个命令迁移 state 地址：

```sh
terraform state mv aws_s3_bucket.logs 'aws_s3_bucket.this["logs"]'
```

它能用，但它是命令式操作。

缺点是：

- 不容易 code review
- 不容易留在代码历史里
- 团队成员可能漏执行
- CI/CD 场景不够声明式

`moved block` 是声明式写法：

```hcl
moved {
  from = aws_s3_bucket.logs
  to   = aws_s3_bucket.this["logs"]
}
```

优点是：

- 迁移关系写在代码里
- 可以走 PR 审查
- plan 能显示迁移信息
- 团队协作更安全
- 更符合 IaC 的可审计原则

简单记：

```text
state mv：手动搬家
moved block：把搬家路线写进代码，让 Terraform 自己搬
```

## 9. LocalStack 在本实验里的作用

本实验不访问真实 AWS，而是使用 LocalStack 模拟 AWS API。

大致链路是：

```text
Terraform AWS Provider
-> http://localhost:4566
-> LocalStack 模拟 AWS 服务
-> 使用假凭证 test/test
```

这样做的好处是：

- 不需要真实 AWS 账号
- 不产生云费用
- 不会误删真实资源
- 可以反复练习 plan/apply/destroy
- 适合验证 Terraform 行为

但要注意边界：

```text
LocalStack 能验证 Terraform 配置和 Provider 调用路径，
但不等于完全覆盖真实 AWS 的 IAM、配额、区域差异和服务细节。
```

迁移到真实 AWS 时，还要额外检查：

- IAM 权限
- 资源命名唯一性
- AWS 服务限制
- 生产资源保护
- 审批流程
- 变更窗口
- state 备份

> 你能解释为什么 Terraform 没有重建资源，而只是更新了 state 里的资源地址。

## 12. 常见错误

## 12.1 from 地址写错

如果 state 里真实地址是：

```text
aws_s3_bucket.logs
```

但你写成：

```hcl
moved {
  from = aws_s3_bucket.log
  to   = aws_s3_bucket.this["logs"]
}
```

就匹配不上。

排查方式：

```sh
terraform state list
```

以 state 里的地址为准。

---

## 12.2 for_each key 和 moved block 不一致

资源里写的是：

```hcl
for_each = {
  app_logs = {...}
}
```

但 moved block 写的是：

```hcl
moved {
  from = aws_s3_bucket.logs
  to   = aws_s3_bucket.this["logs"]
}
```

这里 `logs` 和 `app_logs` 不一致。

正确写法应该是：

```hcl
moved {
  from = aws_s3_bucket.logs
  to   = aws_s3_bucket.this["app_logs"]
}
```

---

## 12.3 重构地址时顺手改了不可变参数

生产里建议分两步：

```text
第一步：只迁移地址，不改资源参数
第二步：再改资源配置
```

否则你很难判断 replacement 是因为地址迁移失败，还是因为某个参数变化导致资源必须重建。

## 13. 考试重点

本节可以压缩成几条考试结论：

```text
已有资源重构成 for_each，会改变 Terraform 资源地址。
地址变化后，应使用 moved block 声明旧地址到新地址的迁移关系。
moved block 可以避免 Terraform 把代码重构误判为 destroy/create。
for_each 更适合有稳定业务身份的资源集合。
for_each 的 key 会成为资源地址的一部分，所以必须稳定。
```

如果考题说：

> 你有多个已存在资源，现在想改成 map + for_each 管理，同时避免资源重建。

答案基本就是：

```text
使用 moved block 将旧资源地址映射到新的 for_each 实例地址。
```

## 14. 快速自测

### 题目 1

已有资源地址是：

```text
aws_s3_bucket.logs
```

现在重构成：

```hcl
resource "aws_s3_bucket" "this" {
  for_each = {
    logs = {
      bucket = "demo-logs"
    }
  }

  bucket = each.value.bucket
}
```

应该怎么写 moved block？

答案：

```hcl
moved {
  from = aws_s3_bucket.logs
  to   = aws_s3_bucket.this["logs"]
}
```

---

### 题目 2

为什么不能只改代码，不写 moved block？

答案：

```text
因为 Terraform 会看到旧地址消失、新地址出现，可能计划销毁旧资源并创建新资源。
```

---

### 题目 3

`for_each` 的 key 为什么要稳定？

答案：

```text
因为 key 会成为 Terraform 资源地址的一部分。key 变了，地址就变了，Terraform 可能认为这是另一个实例。
```

---

### 题目 4

`moved block` 会修改真实 AWS 资源吗？

答案：

```text
不会。它主要修改 Terraform state 中的地址映射关系，不是重建真实资源。
```

## 15. 总结

这一节的主线是：

```text
已有资源
-> 重构为 for_each map
-> Terraform 地址发生变化
-> 用 moved block 声明旧地址到新地址的迁移
-> plan 确认没有 destroy/create
-> apply 更新 state
```

一句话版：

> `for_each` 解决资源集合建模问题，`moved block` 解决已有资源地址迁移问题。

再狠一点说：

> 不懂 state 地址就重构 Terraform，等于闭眼开铲车进生产环境。
