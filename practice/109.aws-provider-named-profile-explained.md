# Terraform 实操训练 109：AWS Provider 使用 `named profile`

## 1. 本节在讲什么

这一节讲的是 Terraform AWS Provider 读取 AWS 凭证的一种常见方式：**named profile**。

核心点不是单纯背下面这一行：

```hcl
provider "aws" {
  profile = "dev"
}
```

而是要理解：

- Terraform 自己不直接登录 AWS；
- Terraform 通过 AWS Provider 调用 AWS API；
- AWS Provider 需要拿到凭证；
- `profile` 参数可以指定使用本机 AWS 配置文件里的某个 named profile；
- 这样可以在多个账号、多个环境之间切换，而不是把 access key 写死在 Terraform 代码里。

一句话总结：

> `profile` 是 Terraform AWS Provider 选择“用哪套 AWS 身份”的开关。

---

## 2. 什么是 named profile

AWS CLI 通常会把凭证和配置放在两个文件里：

```text
~/.aws/credentials
~/.aws/config
```

一个最简单的 `credentials` 文件可能长这样：

```ini
[default]
aws_access_key_id = default_access_key
aws_secret_access_key = default_secret_key

[dev]
aws_access_key_id = dev_access_key
aws_secret_access_key = dev_secret_key

[prod]
aws_access_key_id = prod_access_key
aws_secret_access_key = prod_secret_key
```

这里的：

- `default`
- `dev`
- `prod`

就是不同的 profile。

如果 Terraform AWS Provider 写：

```hcl
provider "aws" {
  region  = "us-east-1"
  profile = "dev"
}
```

意思就是：

> 这个 Terraform 配置运行时，用 `dev` 这套凭证去调用 AWS API。

---

## 3. 为什么不要把凭证写进 Terraform 代码

最糟糕的写法是这样：

```hcl
provider "aws" {
  region     = "us-east-1"
  access_key = "AKIAxxxxxxxxxxxx"
  secret_key = "xxxxxxxxxxxxxxxx"
}
```

这类写法的问题非常大：

- 凭证容易被提交到 Git；
- 代码一旦泄露，云账号就可能被打穿；
- 不方便区分 dev、test、prod；
- 团队成员无法使用各自身份；
- CI/CD 里也不好管理。

named profile 的思路是：

> Terraform 代码只声明要使用哪个 profile，真实凭证放在本机或运行环境里管理。

这样 Terraform 代码更干净，也更接近真实工程实践。

---

## 4. Terraform AWS Provider 的凭证查找逻辑

AWS Provider 获取凭证时，一般会按照一套 credential chain 查找。

常见来源包括：

1. Provider block 里的显式参数；
2. 环境变量，例如 `AWS_ACCESS_KEY_ID`、`AWS_SECRET_ACCESS_KEY`；
3. `profile` 指定的 named profile；
4. 默认 profile；
5. EC2/ECS/IRSA 等运行环境身份。

本节重点是第三种：

```hcl
provider "aws" {
  region  = "us-east-1"
  profile = "localstack"
}
```

这里的关键是：

- `profile` 只是告诉 Provider 去哪里找凭证；
- 它不是 AWS 账号本身；
- 它不是 IAM Role；
- 它只是本机配置文件里的一个名字。

---

## 5. 本实验为什么使用 LocalStack

本节实验不访问真实 AWS，而是通过 LocalStack 模拟 AWS API。

也就是说，Terraform 里的 AWS Provider 仍然是 AWS Provider，但 endpoint 被指向本地：

```text
http://localhost:4566
```

实验里使用的是假凭证：

```text
test/test
```

这很正常。

因为 LocalStack 不是在校验真实 AWS 账号，而是在模拟 AWS API 行为。你要练的是：

- Provider 如何读取 profile；
- Terraform 如何初始化 AWS Provider；
- plan/apply/destroy 的基本闭环；
- 本地模拟环境和真实 AWS 的边界。

重点不是花钱连真实 AWS。

---

## 6. LocalStack 场景下的 Provider 配置长什么样

在 LocalStack 实验中，Provider 通常会包含 endpoint 覆盖配置。

示例结构类似这样：

```hcl
provider "aws" {
  region  = "us-east-1"
  profile = "localstack"

  access_key = "test"
  secret_key = "test"

  skip_credentials_validation = true
  skip_metadata_api_check     = true
  skip_requesting_account_id  = true

  endpoints {
    s3 = "http://localhost:4566"
  }
}
```

不同实验的资源类型不同，`endpoints` 里面可能不是 `s3`，也可能会配置多个服务 endpoint。

这里最重要的是理解：

- `profile` 用来测试 named profile 读取逻辑；
- `endpoint` 用来把 API 请求导向 LocalStack；
- `skip_*` 参数用于绕过真实 AWS 环境里才需要的校验；
- 这套配置是本地实验友好，不等价于生产最佳实践。

---

## 7. 真实 AWS 项目中怎么用

真实项目里通常不会写 LocalStack endpoint。

更常见的是：

```hcl
provider "aws" {
  region  = var.aws_region
  profile = var.aws_profile
}
```

变量定义：

```hcl
variable "aws_region" {
  type    = string
  default = "ap-southeast-2"
}

variable "aws_profile" {
  type    = string
  default = "dev"
}
```

运行时可以这样切换：

```sh
terraform plan -var='aws_profile=dev'
terraform plan -var='aws_profile=prod'
```

不过生产环境里要谨慎。

如果你让 `profile` 可以随便通过变量传入，就必须确认：

- 当前 workspace 是哪个环境；
- backend state 指向哪里；
- 使用的 AWS 账号是否正确；
- region 是否正确；
- 是否有防止误操作 prod 的 guardrail。

不然你可能以为自己在 dev，实际上打到了 prod。

这就是云工程里的经典事故开场白。

---

## 8. `profile` 和环境变量的关系

很多人会混淆这两种方式：

```sh
export AWS_PROFILE=dev
```

和：

```hcl
provider "aws" {
  profile = "dev"
}
```

它们都可以让 AWS Provider 使用某个 profile，但位置不同：

- `AWS_PROFILE=dev` 是在 shell 环境里指定；
- `provider.profile = "dev"` 是在 Terraform 配置里指定。

如果两边都设置了，就要注意优先级和可读性。

推荐原则：

- 个人本地实验：可以用 `AWS_PROFILE`，简单方便；
- 团队 Terraform 代码：更推荐显式设计 provider 输入，避免靠每个人 shell 里的隐式状态；
- CI/CD：一般不要依赖本机 profile，而是使用 OIDC、AssumeRole、环境注入或云平台原生身份。

---

## 9. named profile 适合什么场景

named profile 很适合：

- 本地开发；
- 多 AWS 账号切换；
- 学习和实验；
- consultant 管理多个客户账号；
- 临时验证某个 IAM 权限；
- 用 SSO 登录后让 Terraform 读取对应 profile。

例如：

```sh
aws sts get-caller-identity --profile dev
terraform plan
```

这类流程可以帮助你先确认身份，再运行 Terraform。

生产环境里，尤其是自动化流水线里，named profile 不一定是最优解。

CI/CD 更常见的是：

- GitHub Actions OIDC AssumeRole；
- AWS IAM Role；
- Terraform Cloud workspace variables；
- workload identity federation；
- pipeline secret store。

所以 named profile 是重要基础，但不是所有场景的最终答案。

## 11. 运行前应该检查什么

本节最容易出问题的不是 Terraform 语法，而是本地凭证环境。

你可以先检查 AWS profile 是否存在：

```sh
cat ~/.aws/credentials
```

或者使用 AWS CLI 验证：

```sh
aws configure list-profiles
```

查看当前身份：

```sh
aws sts get-caller-identity --profile localstack
```

在 LocalStack 场景下，`sts get-caller-identity` 是否可用取决于实验环境和 LocalStack 服务支持情况。不要机械照搬真实 AWS 的判断方式。

真正要看的是：

- profile 名字是否写对；
- credentials 文件是否存在；
- LocalStack 是否启动；
- endpoint 是否指向 `localhost:4566`；
- provider 是否开启了必要的 skip 校验参数。

## 12. 常见错误

### 错误 1：profile 名字写错

例如 Terraform 写的是：

```hcl
profile = "localstack"
```

但 `~/.aws/credentials` 里实际是：

```ini
[local]
aws_access_key_id = test
aws_secret_access_key = test
```

这时 Provider 找不到对应 profile。

解决方式：

- 要么把 Terraform 里的 `profile` 改成 `local`；
- 要么把 credentials 里的 profile 名改成 `localstack`。

---

### 错误 2：把 profile 当成 region

错误理解：

```hcl
profile = "us-east-1"
```

`profile` 是凭证配置名称，不是区域。

区域应该写在：

```hcl
region = "us-east-1"
```

---

### 错误 3：LocalStack 没启动

如果 endpoint 指向：

```text
http://localhost:4566
```

但 LocalStack 没启动，Terraform 就会连接失败。

解决：

```sh
docker ps
```

确认 LocalStack 容器存在，并且端口 4566 已暴露。

---

### 错误 4：真实 AWS 和 LocalStack 配置混在一起

LocalStack 实验里通常会用假凭证和自定义 endpoint。

真实 AWS 项目里不应该把这些东西照搬过去：

```hcl
skip_credentials_validation = true
skip_requesting_account_id  = true
endpoints { ... }
```

这些是本地模拟环境需要的特殊配置，不是生产默认配置。

---

### 错误 5：在团队项目中过度依赖本机 profile

如果 Terraform 代码默认依赖某个本机 profile，比如：

```hcl
profile = "wesley-dev"
```

那别人运行时可能根本没有这个 profile。

解决方式：

- profile 名称参数化；
- README 写清楚本地配置；
- CI/CD 使用更稳定的身份注入方式；
- 不要把个人机器配置变成团队隐式依赖。

---

## 13. 面试角度怎么讲

如果面试官问：

> Terraform AWS Provider 里的 `profile` 是做什么的？

可以这样回答：

> `profile` 用来指定 AWS Provider 从本机 AWS shared credentials/config 文件中读取哪一个 named profile。它适合本地开发、多账号切换和实验环境，能避免把 access key 写死在 Terraform 代码里。真实项目里我会注意它和 `AWS_PROFILE`、环境变量、AssumeRole、CI/CD OIDC 的关系。本地可以用 named profile，流水线里更推荐使用短期凭证或 IAM Role 方式。无论哪种方式，运行前都要确认当前身份、账号、region 和 state backend，避免把 dev 变更打到 prod。

这个回答有几个加分点：

- 不只会背参数；
- 知道凭证来源链；
- 知道本地和 CI/CD 的差异；
- 有生产环境防误操作意识。

## 16. 本节总结

本节训练的是 Terraform AWS Provider 的凭证选择能力。

你需要掌握的不是“`profile` 这个字段怎么拼”，而是：

- named profile 是什么；
- AWS Provider 如何通过 profile 读取凭证；
- 为什么不要把 access key 写进 Terraform 代码；
- LocalStack 里为什么可以用假凭证；
- 本地实验和真实 AWS 项目的差异；
- CI/CD 中为什么通常不依赖本机 named profile；
- 运行 Terraform 前为什么必须确认身份、账号、region 和 backend。

真正的重点是这句话：

> Terraform 执行之前，先确认你是谁、在哪个账号、哪个 region、操作哪份 state。否则命令没错，事故也能很大。
