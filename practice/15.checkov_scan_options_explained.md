# Terraform 实操训练 15：Checkov 扫描选项讲解版

> 目标：把 Checkov 的几个核心扫描参数讲清楚：`-f`、`-d`、`--check`、`--skip-check`、`--soft-fail`。  
> 这个 Lab 的重点不是“修复所有安全问题”，而是学会在 CI / pre-commit 中控制扫描范围和规则范围。

---

## 1. 本 Lab 在练什么？

原始 Lab 15 的主题是 **Checkov 扫描选项**。

Checkov 是一个 IaC 安全扫描工具，可以扫描 Terraform、Kubernetes、CloudFormation 等配置文件，找出潜在的安全风险。

本实验重点练 4 类能力：

| 能力 | 对应参数 | 作用 |
|---|---|---|
| 扫描单个文件 | `-f` | 只扫描指定文件 |
| 扫描整个目录 | `-d` | 扫描目录下的 IaC 文件 |
| 只跑指定规则 | `--check` | 聚焦某几个规则 |
| 跳过指定规则 | `--skip-check` | 暂时忽略某些规则 |

另外还有一个实验中非常实用的参数：

| 参数 | 作用 |
|---|---|
| `--soft-fail` | 即使发现失败项，也让命令返回成功，不阻断实验流程 |

---

## 2. 为什么企业里需要这些扫描选项？

真实团队里，直接上来全量扫描经常会遇到一个问题：

> 扫出来一堆问题，CI 全红，大家直接选择无视。

这就是安全工具落地时最常见的失败方式。

所以平台团队通常不会一开始就要求所有规则全部通过，而是会分阶段推进：

1. 先扫描高风险规则；
2. 对低优先级规则先记录，不阻断；
3. 对暂时不适用的规则使用 `--skip-check`；
4. 后续逐步收敛安全基线。

这也是本实验的工程意义：  
**不是只会跑 Checkov，而是知道怎么把 Checkov 变成可落地的 CI 质量门禁。**

---

## 3. 参数讲解

### 3.1 `-f`：扫描单个文件

`-f` 的意思是 file。

适合场景：

- pre-commit 只扫描本次修改的文件；
- 临时验证某个 Terraform 文件；
- 快速定位某个文件里的安全问题。

示例：

```bash
docker run --rm \
  -v "$PWD":/tf \
  bridgecrew/checkov:latest \
  -f /tf/main.tf \
  --soft-fail
```

PowerShell 写法：

```powershell
docker run --rm `
  -v "${PWD}:/tf" `
  bridgecrew/checkov:latest `
  -f /tf/main.tf `
  --soft-fail
```

这里的重点是：

```bash
-f /tf/main.tf
```

表示只扫描容器内 `/tf/main.tf` 这个文件。

---

### 3.2 `-d`：扫描整个目录

`-d` 的意思是 directory。

适合场景：

- CI 中扫描整个 Terraform module；
- 扫描一个环境目录，例如 `env/dev`、`env/prod`；
- 扫描完整 IaC 仓库。

示例：

```bash
docker run --rm \
  -v "$PWD":/tf \
  bridgecrew/checkov:latest \
  -d /tf \
  --soft-fail
```

PowerShell 写法：

```powershell
docker run --rm `
  -v "${PWD}:/tf" `
  bridgecrew/checkov:latest `
  -d /tf `
  --soft-fail
```

这里的重点是：

```bash
-d /tf
```

表示扫描整个 `/tf` 目录。

---

### 3.3 `--check`：只运行指定规则

`--check` 可以理解为白名单模式：

> 我只关心这些规则，其他规则先不跑。

适合场景：

- 只阻断高危规则；
- 某个阶段只治理一批安全基线；
- CI 里先把最重要的规则跑起来。

示例：

```bash
docker run --rm \
  -v "$PWD":/tf \
  bridgecrew/checkov:latest \
  -d /tf \
  --check CKV_K8S_16,CKV_K8S_19 \
  --soft-fail
```

PowerShell 写法：

```powershell
docker run --rm `
  -v "${PWD}:/tf" `
  bridgecrew/checkov:latest `
  -d /tf `
  --check CKV_K8S_16,CKV_K8S_19 `
  --soft-fail
```

这里的重点是：

```bash
--check CKV_K8S_16,CKV_K8S_19
```

意思是只检查 `CKV_K8S_16` 和 `CKV_K8S_19` 这两个规则。

可以把它理解成：

> 老板今天只看两个 KPI，其他先别吵。

---

### 3.4 `--skip-check`：跳过指定规则

`--skip-check` 可以理解为黑名单模式：

> 大部分规则都跑，但是这些规则先跳过。

适合场景：

- 某些规则暂时不适用于当前环境；
- 某个规则误报较多；
- 团队还没准备好修复某类问题；
- 分阶段治理技术债。

示例：

```bash
docker run --rm \
  -v "$PWD":/tf \
  bridgecrew/checkov:latest \
  -d /tf \
  --skip-check CKV_K8S_14 \
  --soft-fail
```

PowerShell 写法：

```powershell
docker run --rm `
  -v "${PWD}:/tf" `
  bridgecrew/checkov:latest `
  -d /tf `
  --skip-check CKV_K8S_14 `
  --soft-fail
```

这里的重点是：

```bash
--skip-check CKV_K8S_14
```

意思是扫描时跳过 `CKV_K8S_14` 这条规则。

根据原 Lab 的验证结果，跳过 `CKV_K8S_14` 后，失败数从 13 降到了 12。

这说明：

> 不是问题被修好了，而是这条规则被临时排除了。

这个区别很重要。`skip` 不是修复，`skip` 只是暂时不检查。

---

## 4. `--soft-fail` 是干嘛的？

`--soft-fail` 的意思是：

> Checkov 可以报告失败项，但命令本身不要返回失败退出码。

在学习和实验阶段，这个参数很有用。否则只要 Checkov 扫描出问题，命令就可能直接失败，影响你继续观察输出。

在 CI 里，`--soft-fail` 通常用于早期接入阶段：

```bash
checkov -d . --soft-fail
```

含义是：

- 安全问题会被打印出来；
- CI 不会立刻失败；
- 团队可以先收集问题，再决定哪些规则要变成强制门禁。

当团队准备好强制阻断时，就移除 `--soft-fail`：

```bash
checkov -d .
```

这时候如果有失败项，CI 就会失败。

## 5. 本 Lab 的推荐运行顺序

进入实验目录：

```bash
cd /workspace/practice/labs/15
```

### 第一步：扫描单个文件

```bash
docker run --rm \
  -v "$PWD":/tf \
  bridgecrew/checkov:latest \
  -f /tf/main.tf \
  --soft-fail
```

你应该看到 Checkov 输出通过项和失败项。

原 Lab 的验证结果是：

```text
Passed checks: 14
Failed checks: 13
```

这说明 Checkov 成功识别了 Terraform 中定义的 Kubernetes 配置风险。

---

### 第二步：只扫描指定 Kubernetes 规则

```bash
docker run --rm \
  -v "$PWD":/tf \
  bridgecrew/checkov:latest \
  -d /tf \
  --check CKV_K8S_16,CKV_K8S_19 \
  --soft-fail
```

这个命令的意义是：

> 不做全量扫描，只聚焦某几个 Kubernetes Pod 安全规则。

这种方式很适合 CI 的第一阶段落地。

---

### 第三步：跳过某条规则

```bash
docker run --rm \
  -v "$PWD":/tf \
  bridgecrew/checkov:latest \
  -d /tf \
  --skip-check CKV_K8S_14 \
  --soft-fail
```

原 Lab 的验证结果是：

```text
Failed checks: 12
```

对比第一步的 13 个失败项，少了 1 个。  
这说明 `CKV_K8S_14` 被成功跳过了。

## 6. 你应该观察哪些输出？

运行 Checkov 时，不要只看最后的数字。你要重点看这些内容：

### 6.1 Rule ID

例如：

```text
CKV_K8S_16
CKV_K8S_19
CKV_K8S_14
```

Rule ID 是 CI 管控的核心。  
以后你要写 allowlist、blocklist、baseline，全靠这些 ID。

---

### 6.2 Failed check 的文件位置

Checkov 通常会告诉你问题在哪个文件、哪个资源附近。

你要学会把扫描结果映射回 Terraform 代码。

这才是从“会跑工具”进入“会修问题”的关键。

---

### 6.3 失败数量变化

本实验中有一个关键观察点：

| 命令 | 结果 |
|---|---|
| 普通单文件扫描 | Failed checks: 13 |
| 跳过 `CKV_K8S_14` | Failed checks: 12 |

这说明 `--skip-check` 生效了。

## 7. CI 里的落地方式

在 CI 里，通常可以这样分阶段落地。

### 阶段 1：只报告，不阻断

```bash
checkov -d . --soft-fail
```

目的：

- 先让团队看到问题；
- 不要一开始就把所有 PR 打挂；
- 收集当前仓库的安全债务。

---

### 阶段 2：只阻断高风险规则

```bash
checkov -d . --check CKV_K8S_16,CKV_K8S_19
```

目的：

- 先治理最重要的问题；
- 降低噪音；
- 让安全扫描真正进入研发流程。

---

### 阶段 3：全量扫描并阻断

```bash
checkov -d .
```

目的：

- 建立完整 IaC 安全门禁；
- PR 不符合安全基线就不能合并；
- 安全从“建议”变成“工程约束”。

## 8. 常见坑

### 坑 1：把 `--skip-check` 当成修复

跳过规则不是修复。

错误理解：

> 失败项少了，所以安全了。

正确理解：

> 失败项少了，是因为你让 Checkov 不看这一条规则了。

如果要在生产中跳过规则，最好写清楚原因，例如：

```bash
--skip-check CKV_K8S_14
```

同时在 CI 文档中说明：

```text
CKV_K8S_14 暂时跳过，因为当前实验环境不涉及该安全要求。
后续生产基线需要重新评估。
```

---

### 坑 2：长期使用 `latest`

实验中可以使用：

```bash
bridgecrew/checkov:latest
```

但生产 CI 不建议长期使用 `latest`。

原因是：

- 镜像版本会变；
- 规则库会变；
- 同一份代码今天过，明天可能不过；
- CI 结果不可重复。

生产中更推荐固定版本，例如：

```bash
bridgecrew/checkov:3.3.5
```

---

### 坑 3：挂载路径写错

容器里跑 Checkov 时，最容易错的是路径。

这个参数：

```bash
-v "$PWD":/tf
```

表示：

- 宿主机当前目录挂载到容器内 `/tf`
- 所以容器里要扫描 `/tf/main.tf`
- 不是扫描宿主机路径

错误示例：

```bash
checkov -f ./main.tf
```

如果这是在容器里运行，就可能找不到文件。

正确示例：

```bash
checkov -f /tf/main.tf
```

## 9. 本 Lab 的工程理解

这个实验的核心不是 Checkov 语法本身，而是 IaC 安全扫描的落地策略。

你要记住这句话：

> IaC 安全扫描不是越严格越好，而是要能稳定进入团队工作流。

一个成熟的平台团队不会只说：

> 我们用了 Checkov。

更好的说法是：

> 我们通过 Checkov 建立了分阶段 IaC 安全基线：先 soft-fail 收集风险，再用 `--check` 阻断高风险规则，最后逐步扩大到全量规则门禁。

这就是平台工程 / SRE 视角下更值钱的表达。

## 10. 面试表达

如果面试官问：

> 你们怎么把 Checkov 接进 CI？

可以这样回答：

```text
我不会一开始就把 Checkov 全量阻断，因为老项目通常会有大量历史问题，直接阻断会导致团队绕过工具。

我的做法是分阶段：
第一阶段用 --soft-fail 收集风险，不阻断 PR；
第二阶段用 --check 只阻断高风险规则；
对于暂时不适用或误报的规则，用 --skip-check 明确排除，并在文档中记录原因；
最后逐步扩大扫描范围，形成稳定的 IaC 安全基线。
```

这比单纯说“我会用 Checkov 扫 Terraform”高级很多。

## 11. 本 Lab 总结

| 参数 | 记忆方式 | 作用 |
|---|---|---|
| `-f` | file | 扫描单个文件 |

| `--check` | 白名单 | 只跑指定规则 |
