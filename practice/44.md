# Terraform 实操训练 44：Vault Provider

## 1. 课程元数据

- 课程编号：44
- 来源文件：`doc/44.txt`
- 环境分类：`docker`
- 是否可用 Docker 运行：`yes`
- 核心主题：Vault provider、KV v2、读取 secret、敏感输出
- 前置条件：Docker、`hashicorp/terraform:1.11`、`hashicorp/vault:1.18`

## 2. 知识点讲解

Vault provider 允许 Terraform 管理 Vault mount、写入 secret、读取 secret。它很适合训练 provider 集成，但真实生产中要限制 token 权限，并避免把 secret 明文扩散到 state、输出和日志。

## 3. 实验设计

背景：Terraform 需要从 Vault 读取数据库凭据。

问题：直接把密码写进 `.tf` 文件不可接受。

任务：启动本地 Vault dev server，通过 Vault provider 创建 `training` KV v2 mount，写入并读取 `db_creds`。

预期结果：输出 username 为 `app`，并确认 password 字段存在。

## 5. 运行手册

```powershell
docker network create tf-vault-lab
docker run -d --rm --name vault-lab-44 --network tf-vault-lab --network-alias vault --cap-add IPC_LOCK -e VAULT_DEV_ROOT_TOKEN_ID=root hashicorp/vault:1.18
docker run --rm --name tf-practice-44 --network tf-vault-lab -v "${PWD}:/workspace" -w /workspace --entrypoint sh hashicorp/terraform:1.11 practice/labs/44/scripts/run-with-vault.sh
docker stop vault-lab-44
docker network rm tf-vault-lab
```

## 6. 验证报告

状态：通过。首次验证发现输出需显式处理 sensitive，已使用 `nonsensitive()` 输出训练用派生值；随后发现 Vault dev server 默认占用 `secret/`，已改为 `training/` mount；最终完整 Vault provider 流程通过。

## 7. 排障指南

1. 无法解析 `vault`：确认 Vault 容器有 `--network-alias vault`。
2. mount 冲突：不要使用 dev server 默认的 `secret/`。
3. 输出 sensitive 报错：只输出非敏感派生值或显式标记 sensitive。
4. token 失败：确认 `VAULT_DEV_ROOT_TOKEN_ID=root`。

## 8. 清理方式

脚本内执行 `terraform destroy`；容器和网络用：

```powershell
docker stop vault-lab-44
docker network rm tf-vault-lab
```
