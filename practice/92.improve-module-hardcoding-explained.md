> 环境：Docker + `hashicorp/terraform:1.11`

---

## 1. 这一节到底在练什么？

这一节练的是 **Terraform module 设计里的一个关键习惯：不要把可变化的业务参数写死在模块内部**。

前面几节已经开始进入模块化阶段：

- 为什么要用 module；
- 怎么创建本地 module；
- 怎么通过 `source` 引用 module；
- 怎么用本地路径调用 module。

到了这一节，重点就不是“能不能调用模块”，而是进一步思考：

> 这个模块以后能不能复用？能不能给 dev / test / prod 共用？能不能让不同团队安全地传入自己的参数？

如果模块内部全是硬编码，那它表面上是 module，实际上只是把一坨 Terraform 代码换了个文件夹。真正的模块应该通过变量暴露必要输入，通过输出暴露必要结果，把实现细节藏在模块里面。

---

## 2. 什么叫模块硬编码？

所谓硬编码，就是把本来应该由调用方决定的值，直接写死在 module 内部。

例如在 module 里直接写：

```hcl
resource "local_file" "example" {
  filename = "dev-config.txt"
  content  = "environment = dev"
}
```

这段代码能跑，但问题很明显：

- 只能生成固定文件名；
- 只能表示固定环境；
- 想改成 `test` 或 `prod`，必须改 module 源码；
- 多个调用方无法独立传参；
- CI/CD 里很难做环境差异化部署。

更合理的方式是把这些值抽成变量：

```hcl
variable "environment" {
  type        = string
  description = "Deployment environment name, such as dev, test, or prod."
}

variable "filename" {
  type        = string
  description = "Output file name generated by the module."
}

resource "local_file" "example" {
  filename = var.filename
  content  = "environment = ${var.environment}"
}
```

然后由 root module 调用它：

```hcl
module "config" {
  source      = "./modules/config"
  environment = "dev"
  filename    = "dev-config.txt"
}
```

这样，module 才真正开始具备复用价值。

## 3. 为什么硬编码在生产环境里很危险？

硬编码在学习阶段看起来省事，在生产环境里通常会变成坑。

### 3.1 环境无法复用

开发环境、测试环境、预生产环境、生产环境往往只有少量参数不同，例如：

- region；
- instance size；
- subnet CIDR；
- tags；
- naming prefix；
- backup retention；
- enable/disable 某些功能。

如果这些都写死在 module 里面，那每个环境都要复制一份 module。最后会变成：

```text
modules/vpc-dev
modules/vpc-test
modules/vpc-prod
```

这就不是模块化了，这是复制粘贴换皮。

### 3.2 改动风险被放大

如果一个 module 被多个环境引用，但参数写死在内部，那么你为了改 dev 的值，就可能影响 prod。

生产环境里很怕这种情况：

> 我只是想改测试环境，结果 module 源码一改，所有引用这个 module 的地方都受影响。

正确做法是：

- module 提供变量；
- 调用方传入不同值；
- module 本身尽量稳定；
- 环境差异放在 root module 或 tfvars 层。

### 3.3 代码审查不清晰

硬编码会让代码审查变得混乱。

如果所有参数都藏在 module 里面，reviewer 很难快速判断：

- 这个环境到底部署什么规格？
- 有没有打开危险功能？
- 命名规则是否符合团队规范？
- tags 是否完整？
- 是否误用了 prod 配置？

把可变参数暴露出来后，root module 会更清楚：

```hcl
module "app_config" {
  source      = "./modules/app_config"
  environment = "prod"
  owner       = "platform-team"
  cost_center = "cloud-sre"
}
```

这比把所有值塞在 module 内部更适合团队协作。

## 4. Terraform module 的正确分层思路

你可以把 Terraform 项目分成两层来看。

### 4.1 Root module：决定“我要什么”

Root module 是调用层，负责表达当前环境的实际需求。

它应该回答：

- 这是 dev 还是 prod？
- 这个环境部署在哪个区域？
- 资源命名前缀是什么？
- 是否开启备份？
- 规格是多少？
- 谁是 owner？

例如：

```hcl
module "service_config" {
  source      = "./modules/service_config"
  environment = "dev"
  app_name    = "payment"
  owner       = "platform"
}
```

### 4.2 Child module：决定“怎么创建”

Child module 是实现层，负责封装资源结构。

它应该回答：

- 资源怎么组合？
- 命名怎么拼接？
- 输出哪些必要信息？
- 内部依赖关系怎么表达？

例如：

```hcl
resource "local_file" "service_config" {
  filename = "${var.app_name}-${var.environment}.txt"
  content  = "owner=${var.owner}\nenvironment=${var.environment}"
}
```

简单说：

> root module 管业务差异，child module 管资源实现。

## 5. 改进硬编码的一般步骤

可以按这个顺序重构。

### 第一步：识别哪些值不应该写死

常见候选包括：

- 环境名：`dev`、`test`、`prod`；
- 区域：`australiaeast`、`eastus`、`ap-southeast-2`；
- 命名前缀：`demo`、`platform`；
- 标签：`owner`、`project`、`cost_center`；
- 规格：VM size、node count、disk size；
- 开关：是否启用日志、监控、备份；
- 路径或文件名；
- 安全策略参数。

判断标准很简单：

> 如果不同环境、不同团队、不同调用方可能传不同值，就不要硬编码在 module 内部。

### 第二步：在 module 里增加变量

例如：

```hcl
variable "environment" {
  type        = string
  description = "Environment name."
}

variable "app_name" {
  type        = string
  description = "Application name."
}
```

变量要写清楚类型和含义，不要只写一个空壳变量。

### 第三步：把资源里的固定值替换成变量引用

从：

```hcl
filename = "dev-app.txt"
```

改成：

```hcl
filename = "${var.environment}-${var.app_name}.txt"
```

这一步的本质是让 module 从“固定产物”变成“可参数化产物”。

### 第四步：在 root module 调用时传参

```hcl
module "app" {
  source      = "./modules/app"
  environment = "dev"
  app_name    = "api"
}
```

这时候 root module 就能清楚展示当前环境的差异。

### 第五步：用 test assertion 验证重构没有破坏行为

可以通过 `terraform test` 验证：

- 输出值是否符合预期；
- 文件名是否包含环境名；
- 传入变量是否正确影响结果；
- module 输出是否稳定。

例如：

```hcl
run "validate_module_output" {
  command = plan

  assert {
    condition     = output.environment == "dev"
    error_message = "Environment output should match input variable."
  }
}
```

这就是前面 Terraform test 章节和 module 章节开始串起来的地方。

## 8. 每条命令应该关注什么？

### 8.1 `terraform init`

关注点：

- module source 是否能正确解析；
- provider 是否能安装；
- 本地模块路径是否正确；
- `.terraform` 目录是否生成。

模块路径写错时，经常会在这一步失败。

### 8.2 `terraform fmt -check`

关注点：

- HCL 是否符合标准格式；
- module 目录和 root module 是否都需要格式化；
- CI 里是否可以用它挡住低质量提交。

如果失败，可以在本地运行：

```sh
terraform fmt -recursive
```

### 8.3 `terraform validate`

关注点：

- 变量是否声明；
- 输出是否存在；
- module 调用参数是否匹配；
- 资源语法是否正确。

如果你把硬编码改成变量，但忘记声明变量，通常会在这里暴露。

### 8.4 `terraform test`

关注点：

- 测试断言是否验证了 module 输出；
- 测试是否覆盖了参数化后的行为；
- 是否还残留写死值导致断言失败。

本 Lab 的验证结果应为：

```text
1 passed, 0 failed
```

### 8.5 `terraform plan`

关注点：

- Terraform 准备创建、修改或删除什么；
- module 参数变化是否正确反映到资源变化里；
- 是否出现非预期替换。

硬编码改成变量后，要特别注意 plan 有没有显示不该替换的资源。

### 8.6 `terraform apply`

关注点：

- plan 文件是否能被成功应用；
- 实际创建结果是否和 plan 一致；
- 输出是否符合传参后的预期。

### 8.7 `terraform output`

关注点：

- module 是否输出了必要信息；
- root module 是否正确转发 child module 的输出；
- 输出内容是否适合后续自动化流程读取。

## 9. 生产环境里的模块参数化原则

### 9.1 不要暴露所有东西

改进硬编码不等于把所有内部细节都变成变量。

坏例子：

```hcl
variable "internal_resource_name" {}
variable "internal_file_path" {}
variable "internal_content_prefix" {}
variable "internal_content_suffix" {}
```

这样会让 module 失去封装意义。调用方需要理解太多内部细节，后续维护会很痛苦。

更好的方式是只暴露稳定的业务输入：

```hcl
variable "environment" {}
variable "app_name" {}
variable "owner" {}
```

module 内部再根据这些输入生成具体资源名、文件名或配置内容。

### 9.2 变量要有类型约束

不要偷懒写：

```hcl
variable "tags" {}
```

更推荐：

```hcl
variable "tags" {
  type        = map(string)
  description = "Tags applied to all resources."
  default     = {}
}
```

类型约束能提前挡住很多错误。

### 9.3 必要时加 validation

例如环境名只允许几种值：

```hcl
variable "environment" {
  type        = string
  description = "Environment name."

  validation {
    condition     = contains(["dev", "test", "prod"], var.environment)
    error_message = "Environment must be one of: dev, test, prod."
  }
}
```

这比让错误一路跑到云平台 API 再失败要好很多。

### 9.4 默认值要谨慎

默认值适合低风险参数，比如：

```hcl
variable "tags" {
  type    = map(string)
  default = {}
}
```

但对生产关键参数，不一定要给默认值。例如：

```hcl
variable "environment" {
  type = string
}
```

这样调用方必须显式传入，避免误用默认环境。

## 10. 常见错误和排查方式

### 错误 1：变量没有声明

现象：

```text
Reference to undeclared input variable
```

原因：资源中用了 `var.xxx`，但没有在 `variables.tf` 中声明。

处理：补充变量声明。

---

### 错误 2：调用 module 时参数名写错

现象：

```text
Unsupported argument
```

原因：root module 传了 child module 不认识的参数。

处理：检查 module 的 `variable` 名称和调用处是否一致。

---

### 错误 3：本地 module 路径写错

现象：

```text
Unreadable module directory
```

原因：`source = "./modules/xxx"` 路径不存在或相对路径不对。

处理：确认当前 root module 目录和 module 目录的相对关系。

---

### 错误 4：测试断言还在检查旧的硬编码值

现象：`terraform test` 失败。

原因：代码已经参数化了，但测试还在断言旧值。

处理：让测试断言匹配新的变量输入和输出结果。

---

### 错误 5：参数化后导致资源替换

现象：`terraform plan` 显示 destroy/create。

原因：某些参数参与了资源名称或 ForceNew 字段，改动会导致资源重建。

处理：生产环境中要谨慎评估影响，必要时使用迁移策略或 `moved` block。

## 11. 这一节和 Terraform Professional 的关系

Terraform Professional 不只是考“语法会不会写”，更重视你是否理解工程化 Terraform。

这一节对应的能力包括：

- module 设计；
- 输入变量建模；
- 输出结果建模；
- 可复用基础设施代码；
- 测试驱动的模块重构；
- 环境差异管理；
- 避免 copy-paste module。

考试或者面试里如果问：

> 什么时候应该把值抽成变量？什么时候不应该？

你可以这样答：

> 如果一个值代表调用方的业务差异、环境差异或部署策略，就应该暴露成变量；如果它是模块内部实现细节，就应该封装在模块内部。好的 module 不是参数越多越好，而是输入边界清晰、默认值合理、类型约束明确、输出稳定。

这个回答比“硬编码不好，要用 variable”高级很多。

1. 这个 module 里哪些值未来可能因环境而变化？
