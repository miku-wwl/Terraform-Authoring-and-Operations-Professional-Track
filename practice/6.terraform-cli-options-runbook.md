# Terraform 实操训练 6：CLI 命令与参数速查教程

## 1. 这节课到底在学什么

这节课不是教你新资源，也不是教 AWS/Azure。

它教的是一个很基础、但很重要的能力：

> **你不能只会写 `.tf` 文件，还要会查、会用、会解释 Terraform CLI 命令和参数。**

你现在还在从：

```sh
terraform plan
terraform apply
```

这个阶段往工程化阶段过渡，所以这节课非常适合你。

Terraform 真正进入项目、考试、CI/CD 以后，很多问题不是 `.tf` 写不出来，而是命令参数用错了。

比如：

```sh
terraform plan
```

和：

```sh
terraform plan -input=false -no-color -out=tfplan
```

看起来都是 plan，但工程含义完全不一样。

---

## 2. 为什么要学 `terraform -help`

Terraform 的命令和参数很多，而且不同版本可能会有变化。

所以你不能完全依赖记忆，也不能只靠浏览器搜旧文章。

最可靠的方式之一是直接问当前 CLI：

```sh
terraform -help
```

查看 Terraform 支持哪些主命令。

查看某个命令的详细参数：

```sh
terraform plan -help
```

或者：

```sh
terraform apply -help
```

这就是本节课的核心能力：

> **遇到不熟的 Terraform 命令，先用 `terraform <command> -help` 查当前版本支持什么。**

这比你乱搜博客更稳。

---

## 3. 常见 Terraform CLI 命令分类

Terraform 命令可以大致分成几类：

| 类型 | 常见命令 |
|---|---|
| 初始化 | `terraform init` |
| 格式与校验 | `terraform fmt`、`terraform validate` |
| 计划与执行 | `terraform plan`、`terraform apply` |
| 测试 | `terraform test` |
| 输出与清理 | `terraform output`、`terraform destroy` |
| 帮助查询 | `terraform -help`、`terraform <command> -help` |

---

## 4. 初始化类命令

### `terraform init`

```sh
terraform init
```

作用：初始化 Terraform 工作目录。

它会做几件事：

- 下载 provider
- 初始化 backend
- 创建 `.terraform/`
- 生成或更新 `.terraform.lock.hcl`

在自动化环境中推荐写成：

```sh
terraform init -input=false
```

原因是 CI/CD 里不能等待人工输入。

如果缺配置，就应该快速失败。

---

## 5. 格式与校验类命令

### `terraform fmt`

```sh
terraform fmt
```

作用：格式化 Terraform 代码。

本地开发时可以用：

```sh
terraform fmt
```

CI/CD 中更常用：

```sh
terraform fmt -check
```

区别：

| 命令 | 作用 |
|---|---|
| `terraform fmt` | 自动修改格式 |
| `terraform fmt -check` | 只检查，不修改 |

CI/CD 里通常不应该偷偷帮你改代码，而是发现格式不对就失败。

所以流水线中更常见的是：

```sh
terraform fmt -check
```

---

### `terraform validate`

```sh
terraform validate
```

作用：检查 Terraform 配置是否合法。

它会检查：

- HCL 语法
- 变量引用
- resource 写法
- output 写法
- module 调用
- provider 配置结构

但要注意：

> `terraform validate` 不会真正创建云资源。

它只能说明配置在 Terraform 语法层面是合法的。

---

## 6. 计划与执行类命令

### `terraform plan`

```sh
terraform plan
```

作用：生成执行计划。

它会告诉你 Terraform 准备：

- 创建什么
- 修改什么
- 删除什么

比如输出：

```text
Plan: 1 to add, 0 to change, 0 to destroy
```

意思是：

> 准备创建 1 个资源，不修改，不删除。

---

### 自动化环境中的 `terraform plan`

生产或 CI/CD 中推荐：

```sh
terraform plan -input=false -no-color -out=tfplan
```

这三个参数很关键。

### `-input=false`

```sh
-input=false
```

意思是禁止交互输入。

CI/CD 里不能让 Terraform 问：

```text
var.service_name
Enter a value:
```

否则流水线会卡住。

所以自动化中要加：

```sh
terraform plan -input=false
```

### `-no-color`

```sh
-no-color
```

意思是去掉颜色输出。

Terraform 默认输出适合人看，但不一定适合保存成日志或 artifact。

CI/CD、审批系统、日志平台里，最好用纯文本输出。

所以推荐：

```sh
terraform plan -no-color
```

### `-out=tfplan`

```sh
-out=tfplan
```

意思是保存 plan 文件。

这非常重要。

普通写法：

```sh
terraform plan
terraform apply
```

问题是：`apply` 时可能重新计算计划。

更严谨写法：

```sh
terraform plan -out=tfplan
terraform apply tfplan
```

这样做的意义是：

> **审批时看到的 plan，和真正 apply 的 plan，是同一个东西。**

这就是生产环境里很重要的审计能力。

---

## 7. `terraform apply`

普通手工执行：

```sh
terraform apply
```

它会问你：

```text
Do you want to perform these actions?
Only 'yes' will be accepted to approve.
```

你需要输入：

```text
yes
```

本地学习时没问题。

但是 CI/CD 里没人输入 `yes`，所以自动化环境通常会用：

```sh
terraform apply -auto-approve tfplan
```

重点是后面的 `tfplan`。

推荐：

```sh
terraform apply -auto-approve tfplan
```

不推荐生产中直接：

```sh
terraform apply -auto-approve
```

区别如下：

| 命令 | 工程含义 |
|---|---|
| `terraform apply -auto-approve` | 直接执行当前计算出来的变更 |
| `terraform apply -auto-approve tfplan` | 执行之前保存并审查过的计划 |

你现在要重点记住这个习惯：

```sh
terraform plan -out=tfplan
terraform apply tfplan
```

## 8. 测试类命令

### `terraform test`

```sh
terraform test
```

作用：执行 Terraform 测试。

本实验里的测试文件是：

```text
tests/cli_options.tftest.hcl
```

它不是测试云资源，而是测试你的 runbook 或 output 里有没有记录关键命令和参数。

比如检查是否记录了：

```text
terraform init -input=false
terraform plan -input=false -no-color -out=tfplan
terraform apply -auto-approve tfplan
terraform <command> -help
```

这节课的测试目标是：

> 让 Terraform CLI 参数速查表变成可验证的工程产物。

---

## 9. 输出与销毁类命令

### `terraform output`

```sh
terraform output
```

作用：查看 Terraform 配置定义的输出。

本实验中，它会输出类似：

- CLI runbook 文件路径
- command catalog
- automation flags

真实项目中，output 经常用来给后续流程提供信息，比如：

- resource ID
- endpoint
- DNS name
- storage account name
- cluster name

### `terraform destroy`

```sh
terraform destroy
```

作用：销毁 Terraform 管理的资源。

实验中可以用：

```sh
terraform destroy -auto-approve
```

因为你只是删除本地文件。

但真实生产环境里，destroy 很危险，一般需要审批。

## 13. 初学者最容易混淆的点

### 混淆 1：`terraform plan` 和 `terraform plan -out=tfplan`

普通：

```sh
terraform plan
```

只是让你看看计划。

工程化：

```sh
terraform plan -out=tfplan
```

是把计划保存下来，后面可以审查和执行。

### 混淆 2：`terraform apply` 和 `terraform apply tfplan`

普通：

```sh
terraform apply
```

会重新生成执行计划，然后执行。

更严谨：

```sh
terraform apply tfplan
```

执行之前保存的计划。

生产里更看重第二种。

### 混淆 3：`terraform fmt` 和 `terraform fmt -check`

本地开发：

```sh
terraform fmt
```

可以直接改格式。

CI 检查：

```sh
terraform fmt -check
```

只检查，不修改。

### 混淆 4：`-auto-approve` 不是万能安全参数

```sh
terraform apply -auto-approve
```

不是“更高级”，只是“跳过确认”。

在生产里，裸用它反而危险。

更安全的习惯是：

```sh
terraform apply -auto-approve tfplan
```

因为它执行的是已经保存和审查过的 plan。

## 14. 面试或考试里怎么表达

如果别人问：

> 你怎么确认 Terraform CLI 参数？

可以答：

```text
I usually use terraform <command> -help to check the options supported by the current Terraform version. This is especially useful in CI/CD because flags like -input=false, -no-color, and -out=tfplan directly affect whether the workflow is non-interactive, auditable, and reproducible.
```

中文意思：

> 我通常用 `terraform <command> -help` 查看当前 Terraform 版本支持的参数。尤其在 CI/CD 中，`-input=false`、`-no-color`、`-out=tfplan` 这些参数会直接影响工作流是否非交互、可审计、可复现。

这个回答比单纯说“我会用 init/plan/apply”强很多。

## 15. 这节课真正要掌握的结论

这节课最重要的不是背完整 CLI 参数表，而是建立一个习惯：

```text
不确定参数时，先查 terraform <command> -help。
```

然后记住自动化环境里最常用的几个参数：

```text
-input=false
-no-color
-out=tfplan
-auto-approve
```

它们分别解决的问题是：

| 参数 | 解决什么问题 |
|---|---|
| `-input=false` | 防止 CI/CD 等待人工输入 |
| `-no-color` | 让日志和 artifact 更干净 |
| `-out=tfplan` | 保存执行计划，保证 plan/apply 一致 |
| `-auto-approve` | 自动确认执行，避免流水线卡住 |

你现在还在基础阶段，所以这节课的目标不是把所有参数背下来。

真正目标是：

> 从“只会敲 plan/apply”升级到“知道参数为什么影响自动化质量”。
