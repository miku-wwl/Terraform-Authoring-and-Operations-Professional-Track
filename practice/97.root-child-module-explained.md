# Terraform 实操训练 97：root module 与 child module 讲解版

## 1. 这节课在学什么

这一节的核心是理解 Terraform 里的两个模块概念：

- **root module**
- **child module**

一句话说清楚：

> 你当前运行 `terraform init / plan / apply / test` 的目录，就是 root module；被它通过 `module` block 调用的模块，就是 child module。

这不是单纯的名词题。真正重要的是：当项目开始模块化以后，你必须知道变量从哪里进来、资源在哪里创建、输出从哪里返回、测试应该验证哪一层。

---

## 2. 为什么 Terraform 一定会有 root module

Terraform 执行时，必须有一个入口目录。

例如本实验要求进入：

```sh
cd /workspace/practice/labs/97
```

这个目录就是本次运行的 **root module**。

只要你在这个目录里运行：

```sh
terraform init
terraform plan
terraform apply
terraform test
```

Terraform 就会把当前目录下的 `.tf` 文件当成入口配置来加载。

所以 root module 的特点是：

1. 它是 Terraform CLI 执行的入口。
2. 它可以直接声明资源。
3. 它可以调用 child module。
4. 它负责把变量传给 child module。
5. 它可以读取 child module 的 outputs。
6. 它通常代表一个可部署单元，比如一个环境、一个服务、一个平台组件。

你可以把 root module 理解成“项目主程序”。

---

## 3. 什么是 child module

child module 是被 root module 调用的模块。

典型写法类似：

```hcl
module "example" {
  source = "./modules/example"

  name = "demo"
}
```

这里：

```hcl
module "example"
```

表示声明了一个模块调用。

```hcl
source = "./modules/example"
```

表示这个 child module 来自本地路径。

如果 root module 是“主程序”，那 child module 就是“被主程序调用的函数库”或者“可复用组件”。

---

## 4. root module 和 child module 的职责划分

### root module 负责什么

root module 更像编排层，主要负责：

- 接收环境级输入
- 设置 provider
- 调用一个或多个 child module
- 决定模块之间怎么组合
- 暴露最终需要给使用者看的 outputs
- 承担环境级测试入口

例如：

```hcl
module "app_config" {
  source = "./modules/app_config"

  environment = "dev"
  owner       = "platform-team"
}
```

root module 不一定要写很多资源。一个成熟项目里，root module 往往只负责“拼装”。

---

### child module 负责什么

child module 更像封装层，主要负责：

- 隐藏资源实现细节
- 定义输入变量
- 创建资源或构造数据
- 输出可复用结果
- 保持接口稳定
- 让不同环境可以复用同一套逻辑

例如 child module 内部可能有：

```hcl
variable "environment" {
  type = string
}

variable "owner" {
  type = string
}

locals {
  common_tags = {
    Environment = var.environment
    Owner       = var.owner
  }
}

output "tags" {
  value = local.common_tags
}
```

调用方不需要关心模块内部如何拼装 tags，只需要传变量、拿 output。

---

## 5. 变量流向：从 root module 传到 child module

Terraform 模块化以后，最容易混乱的地方是变量作用域。

重点记住：

> child module 不能直接读取 root module 的变量，必须通过 module block 显式传入。

例如 root module 里有：

```hcl
variable "environment" {
  type    = string
  default = "dev"
}

module "naming" {
  source = "./modules/naming"

  environment = var.environment
}
```

child module 里必须自己声明：

```hcl
variable "environment" {
  type = string
}
```

然后 child module 内部才能使用：

```hcl
local.name = "app-${var.environment}"
```

也就是说，模块之间不是“全局变量随便用”。

这是 Terraform 模块设计里非常重要的边界感。

---

## 6. 输出流向：从 child module 返回 root module

child module 产生的结果，如果 root module 想使用，需要 child module 显式声明 `output`。

child module：

```hcl
output "name" {
  value = local.name
}
```

root module：

```hcl
output "app_name" {
  value = module.naming.name
}
```

这里的访问路径是：

```hcl
module.<模块调用名>.<output 名>
```

例如：

```hcl
module.naming.name
```

这说明 root module 不能随便访问 child module 内部的 local、resource 细节。它只能访问 child module 主动暴露出来的 outputs。

这就是模块封装的意义。

---

## 7. 和普通目录拆分的区别

很多初学者会误以为：

> 只要把 `.tf` 文件放进子目录，Terraform 就会自动加载。

这个理解是错的。

Terraform 默认只加载当前工作目录里的 `.tf` 文件。

如果你有这样的结构：

```text
practice/labs/97/
├── main.tf
├── variables.tf
├── outputs.tf
└── modules/
    └── naming/
        ├── main.tf
        ├── variables.tf
        └── outputs.tf
```

Terraform 在 `practice/labs/97` 运行时，只会直接加载 root module 目录下的 `.tf` 文件。

`modules/naming` 里面的文件不会自动执行。

必须在 root module 里写：

```hcl
module "naming" {
  source = "./modules/naming"
}
```

Terraform 才会把这个目录当成 child module 调用。

---

## 8. 状态文件里的模块地址

Terraform state 里会记录资源地址。

root module 里的资源地址通常类似：

```text
local_file.example
```

child module 里的资源地址会带上 module 前缀：

```text
module.naming.local_file.example
```

这个地址很重要，因为：

- `terraform state list` 会看到它
- `terraform plan` 会根据它判断资源是否变化
- 重构模块时可能需要 `moved` block
- 模块调用名变化会影响资源地址
- 从 root 资源迁移到 child module 时，地址会变化

生产环境里，模块化不是随便移动文件。移动资源地址如果没处理好，Terraform 可能以为资源要销毁再创建。

---

## 9. 为什么这节课对生产环境有价值

root module 与 child module 的关系，是 Terraform 工程化的基础。

没有模块时，代码通常会变成这样：

```text
main.tf 越写越大
变量散落各处
不同环境复制粘贴
命名规则不统一
安全规则不统一
测试不好复用
```

有模块以后，可以把稳定逻辑封装起来：

```text
modules/
├── naming/
├── network/
├── storage/
├── aks/
├── eks/
└── monitoring/
```

然后不同环境只负责调用：

```text
envs/
├── dev/
├── test/
├── staging/
└── prod/
```

这就是平台团队常见的 Terraform 代码结构。

root module 负责环境编排，child module 负责可复用能力。

## 10. 本实验应该重点观察什么

根据原始实验要求，本 lab 的完整流程是：

```sh
cd /workspace/practice/labs/97
terraform init -input=false
terraform fmt -check
terraform validate
terraform test
terraform plan -input=false -no-color -out=tfplan
terraform apply -auto-approve tfplan
terraform output
terraform destroy -auto-approve
```

你做这个实验时，别只看命令是否通过。重点观察这些点：

1. 当前执行目录是不是 root module。
2. root module 是否通过 `module` block 调用了 child module。
3. child module 的 `source` 是否是本地路径。
4. 输入变量是否从 root module 显式传入 child module。
5. child module 是否通过 `output` 暴露结果。
6. root module 是否通过 `module.<name>.<output>` 读取结果。
7. `terraform test` 的断言是在验证 root module 输出，还是验证 child module 的行为。
8. `terraform plan` 里是否出现了 `module.<name>` 形式的地址。

## 11. terraform test 在这里测什么

本实验的验证报告里说：

```text
terraform test 返回 1 passed, 0 failed
```

说明这个 lab 至少有一个测试断言通过。

对于 root module 与 child module 主题，测试通常适合验证：

- root module 是否正确调用 child module
- 传入变量是否生效
- child module 输出是否符合预期
- root module 最终 output 是否符合预期
- 模块封装后，行为是否仍然稳定

测试关注的是“模块组合之后的结果”，不是单纯检查文件存在。

例如测试可能长这样：

```hcl
run "validate_module_output" {
  command = plan

  assert {
    condition     = output.app_name == "app-dev"
    error_message = "模块输出的 app_name 不符合预期。"
  }
}
```

这里真正验证的是：

> root module 调用了 child module，并且 child module 通过 output 返回了正确结果。

## 12. 常见错误 1：以为 child module 能直接读 root 变量

错误想法：

```hcl
# child module 内部
locals {
  name = var.environment
}
```

但 child module 里没有声明：

```hcl
variable "environment" {}
```

或者 root module 没有传：

```hcl
environment = var.environment
```

这就会报错。

正确心法：

> 每个 module 都是自己的小世界。需要什么输入，就声明什么变量；调用方要显式传入。

---

## 13. 常见错误 2：以为 root module 能直接读 child local

错误想法：

```hcl
output "name" {
  value = module.naming.local.name
}
```

这是不行的。

root module 只能读取 child module 暴露的 outputs：

```hcl
output "name" {
  value = module.naming.name
}
```

child module 必须有：

```hcl
output "name" {
  value = local.name
}
```

这就是 Terraform 模块的封装边界。

---

## 14. 常见错误 3：修改 module 调用名导致资源地址变化

例如原来是：

```hcl
module "app" {
  source = "./modules/app"
}
```

状态地址可能是：

```text
module.app.local_file.config
```

后来你改成：

```hcl
module "application" {
  source = "./modules/app"
}
```

Terraform 会认为地址变成：

```text
module.application.local_file.config
```

如果没有 `moved` block 或 state 迁移，Terraform 可能计划销毁旧资源、创建新资源。

所以模块调用名不是随便改的。生产环境里，模块名就是资源地址的一部分。

---

## 15. 常见错误 4：把 root module 写成“万能巨石”

有些人虽然用了 child module，但 root module 里仍然塞满资源、locals、复杂逻辑。

这会让模块化失去意义。

比较好的习惯是：

- root module：少写资源，多做编排
- child module：封装具体资源逻辑
- root output：暴露最终使用者需要看的结果
- child output：暴露模块之间需要连接的结果

也就是：

```text
root module = 拼乐高的人
child module = 乐高积木块
```

别让 root module 又造积木、又拼积木、又写说明书。

## 16. 排障入口

### 1. init 失败

重点看：

- `source` 路径是否正确
- 本地模块目录是否存在
- child module 里是否有语法错误
- provider 版本约束是否冲突

命令：

```sh
terraform init -input=false
```

---

### 2. validate 失败

重点看：

- child module 是否缺变量
- root module 是否传错变量名
- output 名称是否写错
- 引用了不存在的 `module.<name>.<output>`

命令：

```sh
terraform validate
```

---

### 3. test 失败

重点看：

- 断言条件是不是写错
- output 名称是否和实际一致
- plan 阶段是否能得到这个值
- apply 后才确定的值是否被提前断言

命令：

```sh
terraform test
```

---

### 4. plan 结果不符合预期

重点看：

- 模块调用名是否变化
- child module source 是否指向了正确目录
- 变量默认值是否被覆盖
- `terraform.tfstate` 是否保留了旧地址

命令：

```sh
terraform plan -input=false -no-color -out=tfplan
```

## 17. 面试 / Terraform Pro 考点理解

这节内容在 Terraform Pro 或实际工程面试里，可能不会直接问：

> 什么是 root module？

更可能换成这些问法：

1. Terraform 当前目录和子目录的 `.tf` 文件都会自动加载吗？
2. module 的 `source` 是干什么的？
3. root module 如何向 child module 传变量？
4. child module 的输出如何被 root module 使用？
5. 为什么改 module block 名字可能导致资源重建？
6. 如何把一堆重复 Terraform 资源抽成模块？
7. module 内部的 local 能不能被外部直接引用？
8. child module 需要自己声明 required providers 吗？
9. 多环境目录结构应该怎么组织？
10. 模块化后怎么写测试断言？

你能回答这些，才说明真的理解了 root module 与 child module。

## 18. 推荐记忆模型

可以用这个模型记：

```text
root module
  = 当前执行目录
  = 入口
  = 编排层
  = 调用 child module
  = 接收最终输入
  = 暴露最终输出

child module
  = 被调用目录
  = 可复用组件
  = 声明自己的 variables
  = 封装资源细节
  = 通过 outputs 暴露结果
```

再压缩成一句：

> root module 负责“用模块”，child module 负责“被复用”。

## 21. 最后总结

本节不是让你记住两个英文名，而是让你建立 Terraform 模块化的基本工程观：

- root module 是入口。
- child module 是复用单元。
- 变量必须显式传入。
- 输出必须显式暴露。
- 子目录不会自动变成模块。
- module 调用名会进入资源地址。
- 模块化会提高复用性，但也带来状态迁移和接口稳定性的要求。

真正写生产级 Terraform 时，root module 与 child module 的边界设计，比语法本身更重要。
