# Terraform 考试训练 124：CLI-driven Run Workflow 实践步骤（讲解版）

## 1. 本节要解决什么问题？

这一课讲的是 **HCP Terraform 的 CLI-driven Run Workflow**。

一句话理解：

> 你还是在本地敲 `terraform plan` / `terraform apply`，但真正的运行记录、状态管理和协作能力会连接到 HCP Terraform workspace。

它不是单纯的本地 Terraform CLI，也不是完全由 Git 触发的 VCS-driven Workflow，而是介于两者之间：

- **命令从本地 CLI 发起**
- **workspace 在 HCP Terraform 中管理**
- **run 记录会出现在 HCP Terraform**
- **state 通常由 HCP Terraform 托管**
- **认证依赖 `terraform login`**
- **绑定关系依赖 `cloud` block**
- **初始化依赖 `terraform init`**

考试中经常考你能不能分清楚：

- `terraform login` 是干什么的
- `cloud` block 是干什么的
- `terraform init` 是干什么的
- plan/apply 是在哪里发起、在哪里记录的

---

## 2. 先记住核心流程

CLI-driven Workflow 的典型顺序是：

```text
1. 在 HCP Terraform 创建 organization / workspace
2. 在本地 Terraform 配置中添加 cloud block
3. 执行 terraform login
4. 执行 terraform init
5. 本地执行 terraform plan / terraform apply
6. 到 HCP Terraform 页面查看 run 记录
```

可以把这个流程理解成三件事：

| 步骤 | 解决的问题 | 关键词 |
|---|---|---|
| `terraform login` | 我是谁 | 身份认证 |
| `cloud` block | 我要连到哪里 | organization / workspace |
| `terraform init` | 当前目录如何接入云端工作区 | 初始化 backend/cloud 集成 |
| `terraform plan/apply` | 发起实际运行 | CLI-driven run |

最容易错的是：**写了 `cloud` block 不代表已经完成连接**。  
你还需要 `terraform init` 让 Terraform 初始化这个配置。

---

## 3. CLI-driven Workflow 是什么？

CLI-driven Workflow 的意思是：

> Terraform 操作仍然由本地 CLI 命令触发，但执行过程会和 HCP Terraform workspace 绑定。

你在本地运行：

```bash
terraform plan
terraform apply
```

但是这些操作会和 HCP Terraform 中对应的 workspace 产生关联。

这和纯本地模式不同：

| 对比项 | 本地 Terraform | CLI-driven Workflow |
|---|---|---|
| 命令在哪里敲 | 本地 | 本地 |
| state 放在哪里 | 本地或自配置 backend | HCP Terraform |
| workspace 管理 | 本地或 backend | HCP Terraform |
| run 记录 | 本地终端为主 | HCP Terraform 页面可见 |
| 团队协作 | 需要自己搭 | 平台提供更多协作能力 |

所以考试中看到：

> user runs Terraform commands locally, but the run appears in HCP Terraform

这基本就是 CLI-driven Workflow。

---

## 4. `terraform login`：解决“我是谁”

`terraform login` 的主要作用是：

> 获取并保存访问 HCP Terraform 的凭据。

你可以把它理解成 Terraform CLI 的网页登录授权。

执行之后，Terraform CLI 会获得一个 token，并把它保存到本地凭据文件中。之后本地 CLI 才能访问 HCP Terraform。

典型命令：

```bash
terraform login
```

如果没有登录，后面执行 `terraform init`、`terraform plan` 时可能会出现认证错误，比如：

```text
Error: Required token could not be found
```

或者提示没有 HCP Terraform 凭据。

考试判断：

| 题目描述 | 应该想到 |
|---|---|
| 没有 credentials |
| token missing |
| authentication failed |
| HCP Terraform login required |

优先检查：

```bash
terraform login
```

---

## 5. `cloud` block：解决“我要连到哪里”

`cloud` block 的作用是告诉 Terraform：

> 当前这个 Terraform 配置应该连接到哪个 HCP Terraform organization 和 workspace。

典型写法：

```hcl
terraform {
  cloud {
    organization = "my-org"

    workspaces {
      name = "my-workspace"
    }
  }
}
```

这里最关键的是两个信息：

- `organization`
- `workspace`

考试中通常会问：

> `cloud` block 中通常需要指定什么？

答案就是：

> organization 和 workspace。

不要被 Docker 镜像、浏览器、Git commit message 这些干扰项带偏。

---

## 6. `terraform init`：解决“初始化连接关系”

很多人会误以为：

> 我写了 `cloud` block，就已经连上 HCP Terraform 了。

不对。

写配置只是声明目标，真正让当前目录初始化这个云端 backend/cloud 集成，需要：

```bash
terraform init
```

`terraform init` 会读取 Terraform 配置，发现 `cloud` block，然后初始化对应的 HCP Terraform 集成。

所以：

```text
cloud block = 配置目标
terraform init = 初始化目标
terraform login = 认证身份
```

考试里如果看到：

> after adding cloud block, what command should be run?

答案通常是：

```bash
terraform init
```

不是：

```bash
terraform fmt
terraform validate
terraform console
terraform force-unlock
```

---

## 7. 本地 plan/apply 和远端 run 记录

CLI-driven Workflow 的特点是：

```bash
terraform plan
terraform apply
```

仍然由本地发起。

但是 HCP Terraform 会记录对应的 run。  
这意味着团队成员可以在 HCP Terraform 页面看到运行历史、状态变化和相关信息。

可以理解为：

```text
本地 CLI = 操作入口
HCP Terraform workspace = 状态与运行记录中心
```

这就是它和 VCS-driven Workflow 的主要区别。

---

## 8. CLI-driven Workflow vs VCS-driven Workflow

Terraform Cloud / HCP Terraform 中常见两种运行方式：

| 工作流 | 触发方式 | 常见场景 |
|---|---|---|
| CLI-driven Workflow | 本地执行 `terraform plan/apply` | 学习、迁移、手动控制、临时操作 |
| VCS-driven Workflow | Git push / PR 触发 run | 团队协作、GitOps、自动化交付 |

CLI-driven 不等于没有 HCP Terraform。  
它只是 **触发入口在本地 CLI**。

VCS-driven 的特点则是：

```text
Git push / Pull Request -> HCP Terraform 自动触发 plan/apply
```

考试中如果题目强调：

> local CLI initiates the run

就是 CLI-driven。

如果题目强调：

> connected to GitHub repository, run triggered by pull request

就是 VCS-driven。

---

## 9. 三个核心组件的职责对比

这是本节最该背熟的一张表：

| 组件 / 命令 | 主要职责 | 考试关键词 |
|---|---|---|
| `terraform login` | 获取并保存 HCP Terraform token | credential、authentication、token |
| `cloud` block | 指定 organization 和 workspace | target、workspace binding、organization |
| `terraform init` | 初始化 cloud/backend 集成 | initialize、configure backend |
| `terraform plan` | 预览变更 | execution plan、proposed changes |
| `terraform apply` | 执行变更 | apply changes、run |

一句话记忆：

> login 管身份，cloud 管目标，init 管初始化，plan/apply 管运行。

## 10. 常见错误与排查

### 错误 1：没有登录

现象：

```text
No credentials found for app.terraform.io
```

原因：

```bash
terraform login
```

没有执行，或者 token 已经过期/无效。

处理：

```bash
terraform login
```

---

### 错误 2：workspace 名称写错

例如配置中写了：

```hcl
workspaces {
  name = "prod"
}
```

但 HCP Terraform 里实际叫：

```text
production
```

这属于什么问题？

> 绑定配置问题。

它不是认证问题，因为你的身份可能是对的，只是你指向了错误的 workspace。

---

### 错误 3：organization 名称写错

例如：

```hcl
organization = "my-company"
```

但实际 organization 是：

```text
my-company-prod
```

这同样属于绑定配置问题。

---

### 错误 4：写了 `cloud` block 但没执行 `terraform init`

现象：

```text
Terraform 没有正确初始化远端配置
```

处理：

```bash
terraform init
```

记住：

> 改了 backend / cloud 配置，通常都要重新 init。

## 11. 考试选择题讲解

### 题 1

> CLI-driven Workflow 中，`terraform login` 的主要作用是什么？

选项：

- A. 获取并保存访问 HCP Terraform 的凭据
- B. 创建本地 Git 分支
- C. 格式化 `.tf` 文件
- D. 删除所有 workspace

答案：**A**

解析：

`terraform login` 是认证动作。  
它和 Git 分支、格式化文件、删除 workspace 都没有关系。

看到：

```text
login
credential
token
authentication
```

直接想到 HCP Terraform 认证。

---

### 题 2

> `cloud` block 中通常需要指定哪些信息？

选项：

- A. organization 和 workspace
- B. Docker 镜像名和端口
- C. 本地浏览器类型
- D. Git commit message 模板

答案：**A**

解析：

`cloud` block 是 Terraform 配置连接 HCP Terraform 的关键配置。  
它至少要让 Terraform 知道连接哪个 organization 和哪个 workspace。

典型结构：

```hcl
terraform {
  cloud {
    organization = "example-org"

    workspaces {
      name = "example-workspace"
    }
  }
}
```

---

### 题 3

> 本地配置添加 `cloud` block 后，通常需要运行哪个命令完成初始化？

选项：

- A. `terraform init`
- B. `terraform fmt`
- C. `terraform force-unlock`
- D. `terraform providers mirror`

答案：**A**

解析：

`terraform init` 用来初始化 Terraform 工作目录，包括 backend、providers、modules 等。  
当你添加了 `cloud` block 后，需要 init 才能完成云端集成初始化。

## 12. 判断题讲解

### 判断题 1

> `terraform login` 与 HCP Terraform 认证有关。

答案：**对**

它就是用来登录 HCP Terraform，并保存访问 token 的。

---

### 判断题 2

> CLI-driven Workflow 中，plan/apply 命令可以从本地 CLI 发起。

答案：**对**

CLI-driven 的重点就是：

```text
本地 CLI 发起
HCP Terraform 记录和管理
```

---

### 判断题 3

> 只要写了 `cloud` block，就不需要执行 `terraform init`。

答案：**错**

写配置不等于初始化完成。  
添加或修改 backend/cloud 相关配置后，通常需要：

```bash
terraform init
```

## 13. 场景题讲解

### 场景题 1

> 用户在本地执行 `terraform plan` 时提示没有 HCP Terraform 凭据。应该优先检查哪个步骤？

答案：

优先检查是否执行过：

```bash
terraform login
```

以及保存的 token 是否仍然有效。

解释：

这是典型认证问题。  
凭据缺失时，Terraform CLI 没有权限访问 HCP Terraform。

---

### 场景题 2

> 用户把 workspace 名称写错，导致本地目录连接不到预期 workspace。这个问题属于认证问题还是绑定配置问题？

答案：

属于 **绑定配置问题**。

解释：

认证问题是：

```text
我是谁？
```

绑定配置问题是：

```text
我要连到哪里？
```

workspace 名称写错，不是身份错，而是目标错。

## 14. 考试高频陷阱

### 陷阱 1：把 `terraform login` 和 `terraform init` 混为一谈

错误理解：

```text
terraform login 会完成所有初始化
```

正确理解：

```text
terraform login 只解决认证
terraform init 负责初始化当前目录
```

---

### 陷阱 2：以为 `cloud` block 会自动创建 workspace

`cloud` block 是声明连接目标，不是万能创建器。  
workspace 通常需要在 HCP Terraform 中存在，或者根据配置和平台行为正确管理。

考试中更稳妥的判断是：

```text
cloud block 负责指定 organization / workspace
```

不要把它理解成自动完成所有平台配置。

---

### 陷阱 3：把 CLI-driven 和 VCS-driven 搞混

CLI-driven：

```text
本地 terraform plan/apply 触发
```

VCS-driven：

```text
Git push / PR 触发
```

题目看触发方式就够了。

---

### 陷阱 4：以为本地 CLI 发起就一定是纯本地运行

不一定。

CLI-driven Workflow 虽然从本地发起命令，但它可以连接 HCP Terraform workspace，并在平台中留下 run 记录。

## 16. 最小实践示例

假设你已经有一个 HCP Terraform organization：

```text
organization: wesley-lab
workspace: cli-driven-demo
```

本地可以写：

```hcl
terraform {
  cloud {
    organization = "wesley-lab"

    workspaces {
      name = "cli-driven-demo"
    }
  }

  required_providers {
    local = {
      source  = "hashicorp/local"
      version = "~> 2.5"
    }
  }
}

resource "local_file" "demo" {
  filename = "${path.module}/demo.txt"
  content  = "hello from cli-driven workflow"
}
```

然后执行：

```bash
terraform login
terraform init
terraform plan
terraform apply
```

复习时不一定非要注册和付费使用 HCP Terraform。  
考试更重要的是理解每一步的职责。

---

## 17. 本节最终结论

CLI-driven Workflow 的核心不是“命令换了”，而是：

> 本地 CLI 仍然是入口，但 Terraform 的 workspace、state 和 run 记录与 HCP Terraform 绑定。

考试时按下面逻辑判断：

| 看到什么 | 选什么 |
|---|---|
| 没有凭据、token、认证失败 | `terraform login` |
| organization / workspace 名称 | `cloud` block |
| 添加 cloud block 后下一步 | `terraform init` |
| 本地执行 plan/apply，云端有 run | CLI-driven Workflow |
| Git push / PR 触发运行 | VCS-driven Workflow |

记住这几组关系，这一类题基本就稳了。
