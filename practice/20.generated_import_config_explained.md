# Terraform 实操训练 20：导入后生成配置讲解版

> 主题：`import block`、`terraform plan -generate-config-out`、导入后资源接管  
> 适合场景：把已经存在的云资源接入 Terraform 管理，尤其是 AWS Security Group 这类历史手工资源。

---

## 1. 这一节到底在学什么？

Lab 20 讲的是 Terraform 1.5+ 之后更现代的资源导入方式：

```hcl
import {
  to = aws_security_group.example
  id = "sg-xxxxxxxx"
}
```

然后用：

```sh
terraform plan -generate-config-out=generated.tf
```

让 Terraform 根据远端真实资源生成一份候选配置。

这一步的意义是：以前导入资源时，你要自己写完整 `.tf` 配置，然后再 `terraform import`。现在可以先写 `import block`，让 Terraform 帮你生成初稿，减少手写资源属性的成本。

但重点来了：**生成配置不是最终生产代码，只是 Terraform 帮你抄了一份“现场记录”。**

你还要人工审查、重构、删除噪音字段、补变量、补命名规范、补模块边界。

---

## 2. import block 的核心概念

`import block` 主要有两个字段：

```hcl
import {
  to = <Terraform 资源地址>
  id = <远端真实资源 ID>
}
```

### `to` 是什么？

`to` 表示资源导入到 Terraform state 里的地址。

例如：

```hcl
to = aws_security_group.web_sg
```

意思是：这个远端安全组以后在 Terraform 里叫 `aws_security_group.web_sg`。

如果你后续想模块化，也可能是：

```hcl
to = module.network.aws_security_group.web_sg
```

### `id` 是什么？

`id` 是云平台里真实存在的资源 ID。

例如 AWS Security Group：

```hcl
id = "sg-0123456789abcdef0"
```

Terraform 不是靠名字导入资源，而是靠平台 ID。  
所以生产里导入前一定要确认：

- 资源 ID 对不对；
- 区域对不对；
- 账号对不对；
- provider 配置对不对；
- 这个资源是不是你真的想接管的那个。

## 3. 为什么要用 `-generate-config-out`？

命令：

```sh
terraform plan -generate-config-out=generated.tf
```

作用是：Terraform 根据 `import block` 和远端资源信息，生成一个候选配置文件。

这能解决一个很现实的问题：

> 历史资源已经存在，但没人知道它到底有哪些属性。  
> 手写 `.tf` 很容易漏字段、写错字段、或者触发意外变更。

使用 `-generate-config-out` 后，你至少能拿到一个 Terraform 视角下的资源快照。

不过它生成出来的东西通常比较“原始”：

- 字段可能很多；
- 命名可能不符合团队规范；
- 默认值可能被显式写出来；
- 变量、locals、模块结构都没有；
- 可能暴露出你不想长期维护的细节。

所以正确姿势不是直接提交 `generated.tf`，而是：

1. 先生成；
2. 再审查；
3. 再重构；
4. 再跑 plan；
5. 确认没有意外变更；
6. 最后才合并进主分支。

## 4. 本 Lab 为什么用 `terraform_data` 模拟？

原课程场景是 AWS 安全组导入，但真实 AWS 导入需要：

- AWS 凭证；
- 正确区域；
- VPC；
- 已存在的安全组 ID；
- 可能还有 IAM 权限。

为了让实验能在 Docker / 本地环境稳定跑通，这个 Lab 用 `terraform_data` 模拟安全组资源和导入流程。

这不是说 `terraform_data` 等于 AWS Security Group，而是借它演示导入工作流里的核心思想：

- import block 需要目标地址；
- import block 需要真实 ID；
- 生成配置只是候选配置；
- 导入后还要继续 plan，确认不会产生漂移或意外修改。

本实验模拟的安全组端口是：

```text
80 / 443 / 22
```

可以理解为：

- 80：HTTP
- 443：HTTPS
- 22：SSH

这里重点不是端口本身，而是理解：**导入后 Terraform 需要知道真实资源当前是什么样子。**

## 5. 推荐执行流程

进入实验目录：

```sh
cd /workspace/practice/labs/20
```

初始化：

```sh
terraform init -input=false
```

格式检查：

```sh
terraform fmt -check
```

配置校验：

```sh
terraform validate
```

运行测试：

```sh
terraform test
```

生成 plan：

```sh
terraform plan -input=false -no-color -out=tfplan
```

执行：

```sh
terraform apply -auto-approve tfplan
```

查看输出：

```sh
terraform output
```

销毁：

```sh
terraform destroy -auto-approve
```

## 6. 真实 AWS 导入时的流程应该怎么做？

假设你要导入一个已经存在的 AWS Security Group。

### 第一步：确认资源 ID

例如：

```text
sg-0123456789abcdef0
```

同时确认：

- 它在哪个 AWS account；
- 它在哪个 region；
- 它属于哪个 VPC；
- 它当前有哪些 inbound / outbound rules。

### 第二步：写 import block

```hcl
import {
  to = aws_security_group.web_sg
  id = "sg-0123456789abcdef0"
}
```

### 第三步：生成候选配置

```sh
terraform plan -generate-config-out=generated.tf
```

### 第四步：审查 generated.tf

重点看：

- ingress 规则有没有被完整识别；
- egress 规则有没有被完整识别；
- tags 是否符合团队规范；
- name / description 是否合理；
- 是否有大量平台默认值；
- 是否包含你不想长期管理的属性。

### 第五步：重构配置

把生成配置整理成更可维护的写法，例如：

```hcl
resource "aws_security_group" "web_sg" {
  name        = var.security_group_name
  description = "Web security group"
  vpc_id      = var.vpc_id

  tags = local.common_tags
}
```

规则也可以拆成独立资源：

```hcl
resource "aws_vpc_security_group_ingress_rule" "https" {
  security_group_id = aws_security_group.web_sg.id
  ip_protocol       = "tcp"
  from_port         = 443
  to_port           = 443
  cidr_ipv4         = "0.0.0.0/0"
}
```

### 第六步：再次 plan

```sh
terraform plan
```

真正安全的状态应该是：

```text
No changes.
```

这表示 Terraform 配置和远端真实资源基本一致，没有准备偷偷改你的生产资源。

## 7. 这节课的生产级重点

### 重点 1：导入不是创建资源

`import` 不是新建资源。  
它只是告诉 Terraform：

> 这个东西已经存在了，从现在开始你把它写进 state，后面由你管理。

所以导入前不要乱 apply。  
导入错对象，相当于 Terraform state 认错亲戚，后续变更会很危险。

---

### 重点 2：生成配置不是最终架构

`generated.tf` 很像一份自动生成的“现场笔录”。

它能帮你减少体力活，但不会替你做架构设计。

生产代码还应该考虑：

- 命名规范；
- 变量抽象；
- 模块拆分；
- tag 标准；
- 环境隔离；
- 变更审批；
- 安全规则最小化；
- 是否要把规则拆成单独资源。

---

### 重点 3：导入后必须继续 plan

导入完成后不要觉得事情结束了。

你要继续跑：

```sh
terraform plan
```

如果出现很多变更，要停下来审查。

最理想结果是：

```text
No changes.
```

或者只有你明确接受的变更。

---

### 重点 4：生成文件已存在会报错

如果你已经有了：

```text
generated.tf
```

再次运行：

```sh
terraform plan -generate-config-out=generated.tf
```

可能会失败，因为 Terraform 不想覆盖已有文件。

解决方式：

```sh
terraform plan -generate-config-out=generated-2026-07-03.tf
```

或者先人工确认旧文件没用了，再删除。

## 8. 常见错误和排障

### 问题 1：`to` 地址不存在

现象：

```text
resource address does not exist
```

原因通常是你写了：

```hcl
to = aws_security_group.web_sg
```

但配置里没有对应资源，或者地址写错了。

检查：

```hcl
resource "aws_security_group" "web_sg" {
  # ...
}
```

---

### 问题 2：`id` 写错

如果写错安全组 ID，Terraform 会找不到资源，或者更糟：导入了不该导入的资源。

生产里一定要通过控制台、CLI 或 CMDB 核对。

例如：

```sh
aws ec2 describe-security-groups --group-ids sg-0123456789abcdef0
```

---

### 问题 3：导入后 plan 要修改资源

这说明 Terraform 配置和真实资源不一致。

处理方式：

1. 不要急着 apply；
2. 对比 generated 配置和远端属性；
3. 判断差异是平台默认值、历史漂移，还是 Terraform 配置写错；
4. 只接受明确理解的变更。

---

### 问题 4：生成配置太丑

正常。  
自动生成的配置追求“还原现场”，不是追求“好维护”。

你的任务是把它重构成团队能长期维护的 IaC。

## 10. 清理命令

```sh
terraform destroy -auto-approve
rm -f tfplan generated.tf
rm -rf .terraform .terraform.lock.hcl
```

如果你在真实 AWS 环境做导入，清理要非常小心：

- `terraform destroy` 可能真的删除资源；
- 不要在生产导入实验后直接 destroy；
- 如果只是想取消 Terraform 管理，应使用 state 操作，而不是 destroy。

例如：

```sh
terraform state rm aws_security_group.web_sg
```

这个命令只是把资源从 Terraform state 移除，不会删除远端真实资源。

## 11. 一句话总结

Lab 20 的核心不是“怎么生成一个 `generated.tf` 文件”，而是学会一套生产级接管流程：

> 先识别真实资源，再用 import block 建立映射，再生成候选配置，再人工审查重构，最后通过 plan 确认 Terraform 不会误改生产资源。
