# Terraform 实操训练 57 讲解：`for_each` 基础

## 1. 这一节到底在学什么？

Lab 57 学的是 Terraform 里的 `for_each` 基础。

前面几节已经铺垫好了：

- Lab 50：`list` 数据类型
- Lab 51：`map` 数据类型
- Lab 55：`count` 的索引挑战
- Lab 56：`set` 数据类型

这一节正式进入 `for_each`。

一句话理解：

> `for_each` 用来根据 map 或 set 创建多个资源实例，并且用稳定 key 管理资源身份。

它和 `count` 最大的区别是：

```text
count    -> 用数字索引管理资源，例如 [0]、[1]
for_each -> 用 key 管理资源，例如 ["api"]、["worker"]
```

这也是为什么生产环境里，很多长期资源更推荐用 `for_each`。

---

## 2. `for_each` 是什么？

`for_each` 是 Terraform 的 meta-argument，用来创建多个资源实例。

示例：

```hcl
variable "services" {
  type = map(string)

  default = {
    api    = "8080"
    worker = "9090"
    web    = "3000"
  }
}

resource "local_file" "service" {
  for_each = var.services

  filename = "${path.module}/output/${each.key}.txt"
  content  = "port=${each.value}"
}
```

这会创建三个资源实例：

```text
local_file.service["api"]
local_file.service["worker"]
local_file.service["web"]
```

注意资源地址不是：

```text
local_file.service[0]
local_file.service[1]
local_file.service[2]
```

而是带 key 的地址。

这就是 `for_each` 的核心价值。

---

## 3. `each.key` 和 `each.value`

使用 `for_each` 时，Terraform 会提供两个特殊对象：

```hcl
each.key
each.value
```

它们的含义取决于你传给 `for_each` 的数据结构。

---

### 3.1 当 `for_each` 使用 map 时

如果有：

```hcl
for_each = {
  api    = "8080"
  worker = "9090"
}
```

那么：

```text
each.key   = api / worker
each.value = 8080 / 9090
```

示例：

```hcl
filename = "${path.module}/output/${each.key}.txt"
content  = "port=${each.value}"
```

生成逻辑是：

```text
api.txt    -> port=8080
worker.txt -> port=9090
```

---

### 3.2 当 `for_each` 使用 set(string) 时

如果有：

```hcl
for_each = toset(["logging", "metrics", "tracing"])
```

那么对于 `set(string)`：

```text
each.key   = logging / metrics / tracing
each.value = logging / metrics / tracing
```

也就是说 key 和 value 通常相同。

示例：

```hcl
resource "local_file" "feature" {
  for_each = toset(["logging", "metrics"])

  filename = "${path.module}/output/${each.key}.txt"
  content  = "feature=${each.value}"
}
```

会创建：

```text
local_file.feature["logging"]
local_file.feature["metrics"]
```

---

## 4. 为什么 `for_each` 比 `count` 更适合长期资源？

Lab 55 讲过 `count` 的索引挑战。

如果使用 `count`：

```hcl
services = ["api", "worker", "web"]
```

资源地址是：

```text
local_file.service[0] -> api
local_file.service[1] -> worker
local_file.service[2] -> web
```

如果删除中间的 `"worker"`：

```hcl
services = ["api", "web"]
```

`web` 可能从 `[2]` 变成 `[1]`，产生索引漂移。

但如果使用 `for_each`：

```hcl
services = {
  api = {}
  worker = {}
  web = {}
}
```

资源地址是：

```text
local_file.service["api"]
local_file.service["worker"]
local_file.service["web"]
```

删除 `worker` 只影响：

```text
local_file.service["worker"]
```

不会影响 `web` 的地址。

这就是生产环境非常看重的：**资源身份稳定**。

---

## 5. `for_each` 的资源地址

使用 `for_each` 后，资源地址长这样：

```text
resource_type.resource_name["key"]
```

例如：

```text
local_file.service["api"]
```

这比 `count` 的：

```text
local_file.service[0]
```

更可读，也更贴近业务身份。

你一眼就知道：

```text
["api"]
```

对应的是 api 服务。

而看到：

```text
[0]
```

你还要去查第 0 个到底是谁。

生产里审 plan 时，这个差别非常大。

## 9. `for_each` 支持哪些类型？

`for_each` 通常支持：

```hcl
map(...)
```

和：

```hcl
set(string)
```

常见写法：

```hcl
for_each = var.services
```

其中 `var.services` 是 map。

或者：

```hcl
for_each = toset(var.enabled_features)
```

其中 `var.enabled_features` 是字符串列表，经过 `toset()` 转成 set。

---

## 10. 为什么不能直接 `for_each = list`？

很多初学者会想写：

```hcl
for_each = ["api", "worker", "web"]
```

这通常不合适。  
`for_each` 需要明确的 key。

如果你有 list(string)，可以转成 set：

```hcl
for_each = toset(["api", "worker", "web"])
```

这样每个字符串本身就是 key：

```text
["api"]
["worker"]
["web"]
```

但要注意：

> set 是无序去重集合。适合唯一字符串成员，不适合表达顺序。

如果你有复杂对象列表，别急着 toset。通常更推荐转换成 map。

---

## 11. list(object) 如何变成 map 给 for_each？

假设你有：

```hcl
variable "services" {
  type = list(object({
    name = string
    port = number
  }))

  default = [
    {
      name = "api"
      port = 8080
    },
    {
      name = "worker"
      port = 9090
    }
  ]
}
```

可以转换成 map：

```hcl
locals {
  services_by_name = {
    for svc in var.services : svc.name => svc
  }
}
```

然后：

```hcl
resource "local_file" "service" {
  for_each = local.services_by_name

  filename = "${path.module}/output/${each.key}.txt"
  content  = "port=${each.value.port}"
}
```

这样资源地址就是：

```text
local_file.service["api"]
local_file.service["worker"]
```

这比 `count.index` 稳定得多。

## 12. `for_each` 和 map(object)

生产里最常见的是 `map(object)`。

示例：

```hcl
variable "services" {
  type = map(object({
    port     = number
    replicas = number
    image    = string
  }))

  default = {
    api = {
      port     = 8080
      replicas = 2
      image    = "api:v1"
    }

    worker = {
      port     = 9090
      replicas = 1
      image    = "worker:v1"
    }
  }
}
```

资源：

```hcl
resource "local_file" "service" {
  for_each = var.services

  filename = "${path.module}/output/${each.key}.txt"

  content = jsonencode({
    name     = each.key
    image    = each.value.image
    port     = each.value.port
    replicas = each.value.replicas
  })
}
```

这就是很典型的平台模块建模方式：

- map key 是稳定资源身份
- object value 是资源配置
- `for_each` 负责创建多个实例

---

## 13. `for_each` 适合什么场景？

适合：

- 多个命名服务
- 多个环境
- 多个 region
- 多个用户
- 多个文件
- 多个告警规则
- 多个 DNS record
- 多个 Kubernetes namespace
- 多个 IAM binding
- 多个安全规则

共同特点是：

> 每个实例都有稳定名字或唯一身份。

只要你能说出“这个资源叫 api / worker / prod / dev”，就很适合考虑 `for_each`。

---

## 14. `for_each` 不适合什么场景？

不适合：

- 只是创建固定数量的临时资源
- 没有稳定身份的资源
- 必须按顺序处理的对象
- 需要严格依赖位置的列表
- key 在 apply 之前无法确定的集合

例如：

```hcl
for_each = some_resource.example[*].id
```

如果这些 ID 要 apply 后才知道，就可能不适合作为 `for_each` key。

Terraform 需要在 plan 阶段知道 resource instance keys。

---

## 15. `for_each` 的 key 必须稳定

这一点非常关键。

`for_each` 的 key 会成为资源地址的一部分。

例如：

```text
local_file.service["api"]
```

如果你把 key 从：

```hcl
api
```

改成：

```hcl
backend-api
```

Terraform 会认为：

```text
local_file.service["api"] 要删除
local_file.service["backend-api"] 要创建
```

即使里面配置几乎一样，key 变了，资源身份就变了。

所以 key 要慎重设计。

好的 key：

```text
api
worker
frontend
prod
dev
us-east
```

不好的 key：

```text
api-v1
api-20260703
api-temp-new
```

版本、日期、临时状态通常不适合作为 key。  
它们应该放在 value 里，而不是 key 里。

---

## 16. `for_each` 和 state 的关系

Terraform state 会记录资源地址。

使用 `for_each` 后，state 里的实例地址带 key：

```text
local_file.service["api"]
local_file.service["worker"]
```

这让 state 更接近业务模型。

当你删除一个 key：

```hcl
worker = {...}
```

Terraform 只会计划删除：

```text
local_file.service["worker"]
```

当你新增一个 key：

```hcl
scheduler = {...}
```

Terraform 只会计划新增：

```text
local_file.service["scheduler"]
```

这比 count 的数字索引稳定很多。

## 17. `for_each` 和输出

如果资源用了 `for_each`，直接引用资源时它是一个 map-like 对象。

例如：

```hcl
resource "local_file" "service" {
  for_each = var.services
}
```

输出所有文件名：

```hcl
output "service_files" {
  value = {
    for name, file in local_file.service : name => file.filename
  }
}
```

输出某个具体实例：

```hcl
output "api_file" {
  value = local_file.service["api"].filename
}
```

注意：

```hcl
local_file.service[0]
```

不适用于 `for_each`。

`for_each` 资源要用 key 访问：

```hcl
local_file.service["api"]
```

这里的 `for_each` 用来生成多个嵌套 block。

所以 Lab 57 是后面 dynamic block 的基础。

## 19. 常见坑

### 坑 1：直接对 list 使用 for_each

如果是 list(string)，通常要：

```hcl
for_each = toset(var.names)
```

如果是 list(object)，通常要转换成 map：

```hcl
for_each = {
  for item in var.items : item.name => item
}
```

---

### 坑 2：key 不稳定

key 一变，资源地址就变。  
Terraform 会认为旧 key 删除，新 key 创建。

---

### 坑 3：用 apply 后才知道的值当 key

`for_each` 的 key 必须在 plan 阶段确定。  
不能依赖 apply 后才生成的 ID。

---

### 坑 4：把 `each.key` 和 `each.value` 搞混

对于 map：

```hcl
each.key   = map key
each.value = map value
```

对于 set(string)：

```hcl
each.key == each.value
```

---

### 坑 5：引用 for_each 资源还用 `[0]`

错误：

```hcl
local_file.service[0].filename
```

正确：

```hcl
local_file.service["api"].filename
```

或者遍历：

```hcl
{
  for k, v in local_file.service : k => v.filename
}
```

---

## 20. `for_each` 和 `count` 怎么选？

可以这样判断：

### 用 count

当你只是关心数量：

```hcl
count = 3
```

或者开关：

```hcl
count = var.enabled ? 1 : 0
```

### 用 for_each

当每个实例有身份：

```hcl
for_each = var.services
```

例如：

```hcl
api
worker
frontend
```

这些明显是业务身份，不应该用 `[0]`、`[1]` 来代表。

一句话：

> count 管数量，for_each 管身份。

---

## 21. 从 count 迁移到 for_each 要小心

如果旧资源地址是：

```text
local_file.service[0]
local_file.service[1]
```

新资源地址改成：

```text
local_file.service["api"]
local_file.service["worker"]
```

Terraform 默认可能认为：

```text
旧资源删除，新资源创建
```

生产中要用 `moved` block 迁移 state 地址：

```hcl
moved {
  from = local_file.service[0]
  to   = local_file.service["api"]
}
```

Lab 24 的 `moved` block 在这里就派上用场了。

```hcl
  filename = "${path.module}/output/${each.key}.txt"

```

这些都是非常实战的模式。

## 25. SRE / Platform Engineering 视角

从平台工程角度看，`for_each` 是 Terraform 生产建模的核心工具之一。

平台模块经常要管理多个对象：

- 多个服务
- 多个 namespace
- 多个环境
- 多个 region
- 多个告警
- 多个策略
- 多个用户或团队配置

这些对象通常都有稳定身份。

用 `for_each` 后，plan 会更可读：

```text
module.service["api"]
module.service["worker"]
```

比：

```text
module.service[0]
module.service[1]
```

更适合审查、排障和长期维护。

这就是工程质量差距。

## 26. 面试/考试记忆版

可以这样记：

- `for_each` 根据 map 或 set 创建多个实例。
- `each.key` 是当前实例的 key。
- `each.value` 是当前实例的 value。
- map 输入时，key 和 value 不同。
- set(string) 输入时，key 和 value 通常相同。
- `for_each` 资源地址是 `["key"]`。
- `count` 资源地址是 `[index]`。
- 有稳定身份的长期资源优先用 `for_each`。
- key 必须在 plan 阶段已知。
- key 一旦变化，资源地址也变化。
- 从 count 迁移到 for_each 时可用 `moved` block。

一句话总结：

> `for_each` 的核心价值，是用稳定 key 管理资源身份，避免 count 数字索引带来的漂移风险。

Lab 57 的价值在于，它正式把你从“按数量创建资源”带到“按身份创建资源”。
