# Terraform 实操训练 140 讲解版：Auto Scaling Group 与 Launch Template

## 1. 本节一句话总结

这一节训练的是：**用 Terraform 建模 AWS Auto Scaling Group（ASG）与 Launch Template 的关系**。

更直白一点：

- **Launch Template** 负责定义“实例怎么启动”。
- **Auto Scaling Group** 负责定义“一组实例怎么维持数量、怎么扩缩容、放在哪些子网里”。
- 在真实 AWS 中，ASG 会根据 Launch Template 批量创建 EC2 实例。
- 在本地 LocalStack Community 中，Launch Template 可以验证，但 ASG 的真实创建 API 不完整，所以本节重点是理解模型，而不是模拟真实扩缩容。

---

## 2. 先把两个核心对象讲明白

### 2.1 Launch Template 是什么？

Launch Template 可以理解为 EC2 的“启动说明书”。

它通常描述：

- 使用哪个 AMI；
- 使用什么 instance type；
- 使用哪些 security group；
- 是否带 user data；
- 是否设置 IAM instance profile；
- 磁盘怎么配置；
- tag 怎么打；
- 网络接口怎么设置。

也就是说，Launch Template 解决的是：

> 如果我要启动一台 EC2，它应该长什么样？

在 Terraform 里，通常对应：

```hcl
resource "aws_launch_template" "app" {
  name_prefix   = "app-"
  image_id      = "ami-123456"
  instance_type = "t3.micro"
}
```

---

### 2.2 Auto Scaling Group 是什么？

Auto Scaling Group 解决的是“实例组”的生命周期管理问题。

它通常描述：

- 最少保留几台实例；
- 最多允许几台实例；
- 当前期望几台实例；
- 放到哪些 subnet；
- 使用哪个 Launch Template；
- 实例健康检查；
- 替换不健康实例；
- 扩缩容策略。

也就是说，ASG 解决的是：

> 我要维持一组 EC2，这组机器应该有多少台，怎么自动恢复，怎么扩容缩容？

在真实 AWS 中，ASG 会读取 Launch Template，然后按 desired capacity 创建实例。

---

## 3. Launch Template 和 ASG 的关系

可以用这个关系理解：

```text
Launch Template
    |
    |  定义单台 EC2 怎么启动
    v
Auto Scaling Group
    |
    |  根据模板维护一组 EC2
    v
EC2 Instances
```

更口语化一点：

- Launch Template 像“菜单配方”；
- ASG 像“后厨调度系统”；
- EC2 实例就是“实际做出来的菜”。

你可以有模板，但没有 ASG，就只是知道机器怎么创建。

你可以有 ASG，但没有模板，ASG 就不知道要创建什么样的机器。

---

## 4. 为什么生产环境通常推荐 Launch Template，而不是 Launch Configuration？

AWS 早期常用 Launch Configuration。现在更推荐 Launch Template。

原因是 Launch Template 更灵活：

- 支持版本管理；
- 支持更多 EC2 新特性；
- 可以在 ASG 中指定版本；
- 更适合蓝绿、灰度、滚动替换；
- 与现代 EC2 功能集成更好。

Terraform 里看到 ASG + Launch Template，是比较现代、常见的写法。

也就是说，本地环境不能完整创建真实 ASG。

所以参考实现的策略是：

- 真实创建 Launch Template；
- 用 `local.asg_spec` 建模 ASG 的关键字段；
- 不强行在 LocalStack Community 里创建 ASG；
- 把真实 ASG 扩缩容行为留到真实 AWS 或支持该 API 的环境中验证。

这不是偷懒，而是正确处理模拟环境边界。

---

## 6. 为什么不能用 LocalStack Community 验证真实 ASG？

因为 ASG 不只是一个简单配置对象。

真实 AWS 中，ASG 涉及很多控制面行为：

- 调用 EC2 创建实例；
- 维持 desired capacity；
- 健康检查；
- 替换不健康实例；
- 跨 AZ 分布；
- 和 Launch Template 版本关联；
- 和 scaling policy、CloudWatch alarm 等组件联动；
- 和 Load Balancer target group 关联；
- 执行 instance refresh。

这些行为属于 AWS 托管控制面的能力。

LocalStack Community 可以模拟一部分 AWS API，但不能完整模拟所有托管控制面逻辑。

因此本节的训练重点应该是：

> 我能不能正确声明 Launch Template，并且准确建模 ASG 应该如何引用它。

---

## 7. Terraform 里通常怎么建模？

一个典型结构可能长这样：

```hcl
resource "aws_launch_template" "app" {
  name_prefix   = "app-"
  image_id      = var.ami_id
  instance_type = var.instance_type

  tag_specifications {
    resource_type = "instance"

    tags = {
      Name = "app-instance"
    }
  }
}

resource "aws_autoscaling_group" "app" {
  name                = "app-asg"
  min_size            = 1
  max_size            = 3
  desired_capacity    = 2
  vpc_zone_identifier = var.subnet_ids

  launch_template {
    id      = aws_launch_template.app.id
    version = "$Latest"
  }
}
```

这段代码的核心依赖关系是：

```text
aws_autoscaling_group.app
depends on
aws_launch_template.app
```

Terraform 会从引用关系中自动推断依赖。

## 8. 本地实验里的 `local.asg_spec` 是什么意义？

因为 LocalStack Community 不能真实创建 ASG，所以实验可以用 local value 把 ASG 的关键配置抽象出来，例如：

```hcl
locals {
  asg_spec = {
    min_size         = 1
    max_size         = 3
    desired_capacity = 2
    launch_template  = aws_launch_template.app.name
  }
}
```

这类写法的意义是：

- 保留真实 ASG 的设计字段；
- 让 Terraform plan/apply 可以通过；
- 让 output 和 test 可以验证模型；
- 避免因为模拟器 API 缺失导致整个实验不可运行；
- 训练你理解真实 AWS 资源之间应该怎么连接。

这很适合作为考试训练和本地上手训练。

## 10. 常见误区

### 误区 1：Launch Template 会自动扩容

不会。

Launch Template 只是模板。它不会自己创建一组实例，也不会自动扩容。

真正负责扩缩容的是 ASG。

---

### 误区 2：ASG 只是“多台 EC2 的集合”

不准确。

ASG 是 AWS 的托管控制面服务，它会持续维持你声明的容量。

比如 desired capacity 是 2，如果一台实例挂了，ASG 会尝试补一台。

---

### 误区 3：LocalStack 跑不过 ASG 就说明代码没意义

不对。

本节的价值在于本地验证 Terraform 建模能力。

真实扩缩容涉及 AWS 托管服务，LocalStack Community 不能完整模拟很正常。

---

### 误区 4：生产中直接用 `$Latest` 一定最好

不一定。

`$Latest` 方便，但生产中有时更推荐固定版本，例如：

```hcl
version = aws_launch_template.app.latest_version
```

或者显式使用某个版本，避免模板变化导致 ASG 行为不可控。

考试和生产设计里要注意这个点。

---

## 11. 真实 AWS 中还要考虑什么？

如果你把这节迁移到真实 AWS，需要额外考虑：

### 11.1 VPC 和 subnet

ASG 必须知道实例要放在哪些子网里。

通常需要：

```hcl
vpc_zone_identifier = var.private_subnet_ids
```

生产中一般优先放 private subnet，而不是 public subnet。

---

### 11.2 Security Group

Launch Template 里通常要绑定 security group。

否则实例即使启动，也可能无法被访问，或者访问边界不清晰。

---

### 11.3 IAM Instance Profile

如果 EC2 需要访问 S3、CloudWatch、SSM、ECR 等服务，就需要 IAM role + instance profile。

Launch Template 里要引用它。

---

### 11.4 User Data

常见用途：

- 安装 agent；
- 拉取启动脚本；
- 注册服务；
- 写入配置；
- 启动应用。

但 user data 不适合放敏感信息。

---

### 11.5 Load Balancer

生产中 ASG 常常会挂到 ALB/NLB 后面。

需要考虑：

- target group；
- health check；
- listener；
- security group；
- deregistration delay；
- rolling update 行为。

---

### 11.6 Instance Refresh

当 Launch Template 更新后，已有实例不会一定自动替换。

如果要滚动替换旧实例，需要设计：

- instance refresh；
- rolling update 策略；
- health check grace period；
- minimum healthy percentage。

## 12. Terraform 考试可能怎么考？

### 考点 1：谁定义 EC2 启动参数？

答案：Launch Template。

---

### 考点 2：谁维护 desired capacity？

答案：Auto Scaling Group。

---

### 考点 3：ASG 如何引用 Launch Template？

通过 `launch_template` block 引用 Launch Template 的 ID/name 和版本。

---

### 考点 4：LocalStack 不能完整模拟 ASG，应该怎么办？

不要把模拟环境当真实 AWS。

可以在本地验证 Terraform 配置模型，把真实扩缩容放到 AWS 环境验证。

---

### 考点 5：Terraform 如何知道 ASG 依赖 Launch Template？

通过资源引用自动推断依赖，例如：

```hcl
id = aws_launch_template.app.id
```

## 14. 排障指南

### 14.1 `terraform init` 失败

优先检查：

- provider 版本约束；
- LocalStack endpoint；
- 网络；
- `.terraform.lock.hcl`；
- AWS provider 配置。

---

### 14.2 `terraform validate` 失败

优先检查：

- HCL 语法；
- 变量是否声明；
- local 是否引用正确；
- output 是否引用不存在的字段。

---

### 14.3 创建 ASG 时报 501

如果看到类似：

```text
CreateAutoScalingGroup returned 501
```

大概率不是你 Terraform 语法错，而是 LocalStack Community 不支持这个 API。

处理方式：

- 本地只建模 ASG 字段；
- 不真实创建 `aws_autoscaling_group`；
- 把真实 ASG 验证放到 AWS 账号里。

---

### 14.4 apply 成功但没有 EC2 实例

在本节 LocalStack limited 设计下，这是合理的。

因为实验不验证真实 ASG 扩缩容，也不模拟真实 EC2 生命周期。

## 15. 这节在生产里的真正价值

这一节不是为了让你背 `aws_launch_template` 或 `aws_autoscaling_group` 的字段。

它真正训练的是：

- Terraform 如何表达资源之间的依赖；
- 如何区分模板资源和控制面资源；
- 如何处理本地模拟环境和真实云环境的边界；
- 如何设计一个可验证但不滥用真实云资源的实验；
- 如何在工程实践中把不可模拟的行为抽象成可测试模型。

这类能力比单纯会写资源块更重要。

## 18. 参考答案

1. Launch Template 定义 EC2 实例启动参数。
2. ASG 维护实例组容量、健康检查和扩缩容。
3. ASG 需要知道按什么模板创建 EC2 实例。
4. 因为 ASG 涉及 AWS 托管控制面行为，LocalStack Community 不完整支持 CreateAutoScalingGroup API。
5. 不一定。通常需要 instance refresh 或其他滚动替换机制。
6. `$Latest` 可能让生产行为随模板变化而变化，版本控制不够稳定。
7. private subnet 更适合后端实例，减少公网暴露面。
8. 通过类似 `aws_launch_template.app.id` 的引用自动建立依赖。
