# Terraform 实操训练 75 讲解：旧式 S3 Backend 与 DynamoDB State Lock

## 1. 这一节到底在学什么？

第 75 节学的是 Terraform 旧式 S3 backend 的 **DynamoDB state locking**。

上一节第 74 节讲的是：

```text
S3 bucket 保存 terraform.tfstate
```

这一节继续补上：

```text
DynamoDB table 保存 state lock
```

一句话理解：

> S3 负责存 state，DynamoDB 负责加锁，防止多人或多个 CI pipeline 同时写 state。

这套方案是 Terraform 在 AWS 上非常经典的远端 state 方案：

```text
S3 backend + DynamoDB lock table
```

虽然课程文件里提到 Terraform 1.14 会提示 `dynamodb_table` 已 deprecated，但大量存量系统仍然在用这套写法，所以你必须会读、会维护、会排障。

---

## 2. 为什么需要 state lock？

Terraform state 是基础设施的账本。

如果两个人同时执行：

```sh
terraform apply
```

就可能出现这种情况：

```text
A 读取 state
B 也读取 state

A 修改资源并写回 state
B 也修改资源并写回 state

最终 B 的 state 覆盖 A 的变化
```

这会导致 state 损坏或状态不一致。

所以 Terraform 在执行可能写入 state 的操作前，需要先获得锁。

典型写入 state 的操作包括：

- `terraform apply`
- `terraform destroy`
- `terraform state mv`
- `terraform state rm`
- `terraform import`
- 某些会刷新或更新 state 的操作

state lock 的目的就是：

> 同一时刻只允许一个 Terraform 进程写 state。

这不是装饰品，这是团队协作和 CI/CD 的安全底线。

---

## 3. 没有 state lock 会怎样？

没有锁时，最危险的是并发写入。

例如：

```text
Developer A 正在 apply 网络资源
GitHub Actions 同时 apply 应用资源
```

如果它们使用同一份 state，又没有 lock，就可能互相覆盖。

结果可能是：

- state 丢资源
- Terraform 以为资源不存在
- 下次 plan 出现大量异常变更
- 已存在资源被重复创建
- 真实云资源和 state 不一致
- 需要手动修 state
- 严重时只能从备份恢复 state

所以远端 state 不只是“把文件放远端”。  
真正生产级还要考虑：

```text
远端存储 + 并发锁 + 权限控制 + 版本恢复
```

---

## 4. 旧式 AWS 生产方案是什么？

课程里讲到的典型旧式方案是：

```text
S3 bucket 保存 terraform.tfstate
DynamoDB table 保存锁信息和 state digest
```

Terraform backend 配置大概类似：

```hcl
terraform {
  backend "s3" {
    bucket         = "terraform-state-demo"
    key            = "env/dev/terraform.tfstate"
    region         = "us-east-1"
    dynamodb_table = "terraform-locks"
  }
}
```

含义：

- `bucket`：state 存放的 S3 bucket
- `key`：state 在 bucket 里的路径
- `region`：AWS region
- `dynamodb_table`：用来保存锁信息的 DynamoDB 表

这就是很多老 AWS Terraform 项目里还能看到的写法。

---

## 5. DynamoDB lock table 的关键结构

课程文件强调：

> DynamoDB 表的分区键必须是 `LockID`，类型为 `String`。

也就是表结构要类似：

```text
Table name: terraform-locks
Partition key: LockID
Type: String
```

这个字段名不是随便起的。

Terraform S3 backend 的旧式 DynamoDB locking 会使用这个 key 来写入锁记录。

如果你把 partition key 写成：

```text
id
lock_id
lockId
LockId
```

都不对。

必须是：

```text
LockID
```

大小写也要对。

这是考试和实战都容易踩的点。

---

## 6. DynamoDB 里保存了什么？

DynamoDB lock table 里会保存 Terraform 锁相关信息。

大致包括：

- `LockID`
- 谁持有锁
- 操作类型
- 创建时间
- Terraform 版本
- state digest 相关信息

你不需要死背每个字段，但要理解：

```text
DynamoDB 表不是保存完整 state
完整 state 在 S3
DynamoDB 主要保存 lock 和 digest
```

也就是说：

```text
S3 = state 文件
DynamoDB = 锁表
```

不要混淆。

---

## 7. 获取锁的流程

当 Terraform 执行会写 state 的操作时，大概流程是：

```text
1. Terraform 读取 backend 配置
2. Terraform 尝试在 DynamoDB 表中写入 lock 记录
3. 如果写入成功，说明拿到锁
4. Terraform 执行 plan/apply/destroy 等操作
5. Terraform 更新 S3 中的 state
6. Terraform 删除 DynamoDB 中的 lock 记录
```

如果第 2 步失败，说明锁已经被别人持有。  
Terraform 不应该继续写 state。

这就是锁保护的核心。

---

## 8. 锁失败时应该怎样？

如果 Terraform 获取锁失败，正确行为是：

```text
停止操作，不继续写 state
```

通常你会看到类似：

```text
Error acquiring the state lock
```

或者锁相关错误信息。

这不是 Terraform 坏了。  
这是 Terraform 在保护 state。

你应该先判断：

1. 是不是另一个人正在 apply？
2. 是不是另一个 CI job 正在跑？
3. 上一次 apply 是否异常中断？
4. DynamoDB lock table 中是否残留旧锁？
5. state backend 是否配置正确？

不要第一反应就强制解锁。

---

## 9. 什么是 stale lock？

stale lock 可以理解成“残留锁”。

例如：

```text
Terraform apply 正在运行
电脑突然断电
进程没来得及释放锁
DynamoDB 表里锁记录还在
```

下次再运行 Terraform，可能会发现锁还存在。

这种情况下需要人工判断：

- 真的没有其他 Terraform 正在运行吗？
- CI pipeline 有没有还在执行？
- 锁记录是不是旧的？
- state 有没有被写坏？

确认安全后，才考虑强制解锁。

---

## 10. `terraform force-unlock`

Terraform 提供：

```sh
terraform force-unlock <LOCK_ID>
```

用于强制释放锁。

但这不是普通排障第一步，而是最后手段。

正确态度：

> 只有确认没有其他 Terraform 操作正在运行时，才可以 force-unlock。

否则你可能释放了别人正在用的锁，让两个 apply 同时写 state，反而造成更大事故。

生产里 force-unlock 最好有记录、有审批、有上下文。

---

## 11. 为什么这叫“旧式”方案？

课程文件里提到，Terraform 1.14 会提示 `dynamodb_table` 已 deprecated。

也就是说，这种 backend 参数在新版本里已经不再是推荐方向。

但是：

```text
deprecated 不等于立刻消失
deprecated 不等于没人用
deprecated 不等于你可以不会
```

大量存量 AWS Terraform 项目，尤其是运行多年的平台项目，仍然可能使用：

```hcl
dynamodb_table = "terraform-locks"
```

所以这一节的重点不是鼓励你新项目永远用旧参数，而是让你能：

- 看懂旧系统
- 维护旧系统
- 排查锁问题
- 做迁移评估
- 不被 deprecated 配置吓到

这就是现实工程。

## 12. 本实验为什么用 LocalStack？

本实验不直接操作真实 AWS，而是使用 Docker 启动 LocalStack 模拟：

- S3
- DynamoDB
- STS

目的不是省略 AWS 知识，而是让你能本地重复练习。

LocalStack 的价值：

1. 不需要真实 AWS 账号。
2. 不产生云费用。
3. 可以在本地创建 S3 bucket。
4. 可以在本地创建 DynamoDB table。
5. 可以验证 S3 backend 初始化。
6. 可以验证 lock table 配置。
7. 适合自动化实验闭环。

可以理解成：

```text
LocalStack = 本地模拟 AWS API 的训练环境
```

但课程文件也提醒：

> LocalStack 不完全等价于真实 AWS 的并发锁竞争。

所以本节主要验证配置和流程，不是做真实生产级并发压测。

## 14. 典型 backend 配置

真实 AWS 里的旧式配置可能类似：

```hcl
terraform {
  backend "s3" {
    bucket         = "my-terraform-state"
    key            = "labs/75/terraform.tfstate"
    region         = "us-east-1"
    dynamodb_table = "terraform-locks"
  }
}
```

LocalStack 里通常还需要 endpoint 和 skip 校验配置，类似：

```hcl
terraform {
  backend "s3" {
    bucket         = "terraform-state"
    key            = "labs/75/terraform.tfstate"
    region         = "us-east-1"
    dynamodb_table = "terraform-locks"

    endpoints = {
      s3       = "http://localhost:4566"
      dynamodb = "http://localhost:4566"
      sts      = "http://localhost:4566"
    }

    skip_credentials_validation = true
    skip_metadata_api_check     = true
    skip_region_validation      = true
    force_path_style            = true
  }
}
```

不同 Terraform / AWS provider 版本可能写法略有差异。  
你要抓住核心：

```text
backend = s3
state = S3 bucket/key
lock = DynamoDB table
LocalStack = endpoint 指向 localhost
```

## 15. DynamoDB 表创建示例

在真实 AWS 或 LocalStack 里，lock table 需要提前创建。

用 AWS CLI 创建的思路类似：

```sh
aws dynamodb create-table   --table-name terraform-locks   --attribute-definitions AttributeName=LockID,AttributeType=S   --key-schema AttributeName=LockID,KeyType=HASH   --billing-mode PAY_PER_REQUEST
```

如果是 LocalStack，要加 endpoint：

```sh
aws --endpoint-url=http://localhost:4566 dynamodb create-table   --table-name terraform-locks   --attribute-definitions AttributeName=LockID,AttributeType=S   --key-schema AttributeName=LockID,KeyType=HASH   --billing-mode PAY_PER_REQUEST
```

重点记住：

```text
AttributeName=LockID
AttributeType=S
KeyType=HASH
```

这就是 DynamoDB lock table 的关键。

## 16. S3 bucket 创建示例

S3 state bucket 也要提前创建。

真实 AWS：

```sh
aws s3 mb s3://terraform-state-demo
```

LocalStack：

```sh
aws --endpoint-url=http://localhost:4566 s3 mb s3://terraform-state-demo
```

然后 Terraform backend 才能使用这个 bucket。

注意：

> backend 初始化发生在 Terraform resource 创建之前，所以不能指望同一个 Terraform 配置先创建 backend bucket 再使用它。

这就是为什么 state bucket 通常要 bootstrap。

## 17. 为什么 backend 资源要提前创建？

很多人会问：

> 我能不能在同一个 Terraform 项目里创建 S3 bucket，然后马上用它当 backend？

正常不行。

因为执行顺序是：

```text
terraform init 初始化 backend
terraform plan 计算资源
terraform apply 创建资源
```

backend 在 plan/apply 之前就要可用。

所以 state bucket 和 lock table 要提前存在。

常见做法：

```text
bootstrap 项目：
  创建 S3 bucket 和 DynamoDB table

业务项目：
  使用 bootstrap 创建好的 backend
```

这是生产里非常常见的分层。

## 21. 如何验证本地没有主 state？

执行 apply 后，在实验目录检查：

```sh
ls -la
```

或者：

```sh
test ! -f terraform.tfstate && echo "local state not found"
```

使用 S3 backend 后，本地没有主 state 是正常现象。

但 `.terraform/` 目录仍然会存在。

`.terraform/` 是 Terraform 初始化缓存和 backend 元数据目录，不等于本地 state 主文件。

不要混淆：

```text
.terraform/        可以存在
terraform.tfstate  不应作为主 state 出现在工作目录
```

## 22. 如何观察 DynamoDB lock？

在 Terraform apply 运行期间，DynamoDB 表里可能出现锁记录。

LocalStack 下可以扫描表：

```sh
aws --endpoint-url=http://localhost:4566 dynamodb scan   --table-name terraform-locks
```

如果 apply 很快结束，锁可能已经释放，所以表里可能为空。

这不一定是错误。

锁记录通常只在操作期间存在。  
Terraform 成功结束后，会释放锁。

## 23. `terraform state list` 验证什么？

执行：

```sh
terraform state list
```

Terraform 会从当前 backend 读取 state，并列出管理的资源地址。

如果 S3 backend 生效，这个命令读取的是 S3 中的 state，而不是本地 `terraform.tfstate`。

所以它可以验证：

```text
Terraform 能从远端 backend 正常读取 state
```

这比单纯看文件更贴近真实使用。

## 24. 锁失败的典型表现

如果 DynamoDB lock 已经被占用，再执行 Terraform 可能看到类似：

```text
Error acquiring the state lock
```

或者锁相关错误。

这时不要乱操作。

排查顺序：

1. 是否有人正在运行 Terraform？
2. 是否有 CI/CD job 正在跑？
3. 是否上一次操作异常中断？
4. DynamoDB 表里是否有残留锁？
5. 锁 ID 是什么？
6. 是否真的需要 `force-unlock`？

正确做法是先确认，不要直接暴力解锁。

## 25. `-lock=false` 是什么？

Terraform 有参数：

```sh
terraform apply -lock=false
```

这会禁用 state lock。

生产环境不建议使用。

除非你非常清楚自己在做什么，否则不要用它绕过锁。

因为它等于告诉 Terraform：

```text
我不需要并发保护，直接写 state
```

在团队环境里，这很危险。

考试或面试里，看到 `-lock=false` 要警觉：

> 它可能解决眼前阻塞，但会扩大 state 损坏风险。

---

## 26. `force-unlock` 和 `-lock=false` 的区别

### `force-unlock`

用于释放已有残留锁：

```sh
terraform force-unlock <LOCK_ID>
```

它处理的是：

```text
锁卡住了，确认安全后释放
```

### `-lock=false`

用于执行命令时不使用锁：

```sh
terraform apply -lock=false
```

它处理的是：

```text
这次操作直接不加锁
```

生产里两者都要谨慎。  
但 `-lock=false` 更容易造成并发写入风险。

## 27. 旧式 DynamoDB locking 的排障清单

如果实验失败，可以按这个顺序查：

### 1. LocalStack 是否启动

```sh
docker ps
```

看 LocalStack 容器是否运行。

### 2. S3 bucket 是否存在

```sh
aws --endpoint-url=http://localhost:4566 s3 ls
```

### 3. DynamoDB table 是否存在

```sh
aws --endpoint-url=http://localhost:4566 dynamodb list-tables
```

### 4. DynamoDB key 是否正确

必须是：

```text
LockID
String
```

### 5. backend endpoint 是否指向 LocalStack

检查是否配置了：

```text
http://localhost:4566
```

### 6. 是否误连真实 AWS

如果没有 skip 校验或 endpoint 配错，可能会尝试连真实 AWS。

### 7. 是否有残留 `.terraform`

可以尝试：

```sh
terraform init -reconfigure
```

### 8. 是否有旧 state 干扰

清理 S3 object 或使用新的 key 再测试。

## 28. `dynamodb_table` deprecated 怎么看？

课程文件提到 Terraform 1.14 会提示 `dynamodb_table` 已 deprecated。

你的理解应该是：

```text
新项目要关注当前推荐方案
旧项目仍可能大量使用 dynamodb_table
```

所以学习这节有两个价值：

1. **维护存量系统**

   很多公司老项目还在用。

2. **理解 state locking 原理**

   即使未来实现方式变了，锁的目的仍然不变：

   ```text
   防止并发写 state
   ```

不要因为 deprecated 就觉得不用学。  
工程里最常见的不是“全新最佳实践”，而是“存量系统维护”。

## 29. 和第 74 节的关系

第 74 节重点是：

```text
S3 中央后端
state 从本地迁移到 S3
```

第 75 节重点是：

```text
S3 backend + DynamoDB lock
防止并发写 state
```

对比：

| 章节 | 主题 | 核心问题 |
|---|---|---|
| 74 | S3 中央后端 | state 放在哪里 |
| 75 | DynamoDB State Lock | 谁能同时写 state |

可以这样记：

```text
S3 解决共享 state
DynamoDB 解决并发写 state
```

两者一起才更接近传统 AWS Terraform 生产方案。

## 30. 和 CI/CD 的关系

在 CI/CD 里，state locking 更关键。

例如：

```text
main 分支 merge 触发 apply
另一个 hotfix merge 也触发 apply
两个 pipeline 同时运行
```

如果没有锁，它们可能同时写 state。

有 DynamoDB lock 后：

```text
第一个 pipeline 拿到锁
第二个 pipeline 获取锁失败或等待
```

这能防止 state 被同时写坏。

所以 Terraform 自动化不是只加：

```sh
terraform apply -auto-approve
```

还要有：

- remote backend
- state locking
- saved plan
- approval gate
- concurrency control
- least privilege IAM
- state backup

这才叫工程化。

## 31. 生产建议

如果维护旧式 S3 + DynamoDB locking，建议：

1. S3 bucket 开启 versioning。
2. S3 bucket 开启 encryption。
3. S3 bucket block public access。
4. DynamoDB table 使用 `LockID` String partition key。
5. 后端 bucket 和 lock table 提前 bootstrap。
6. 每个环境使用独立 state key。
7. 不同项目不要共用同一个 state key。
8. CI/CD 不要使用 `-lock=false`。
9. force-unlock 要有确认和记录。
10. 关注 Terraform 版本升级对 backend locking 的影响。
11. 迁移 deprecated 方案前先备份 state。
12. state 权限要严格控制。

这不是繁琐，这是 state 的级别决定的。

---

## 32. LocalStack 实验的边界

本节验证结论说明：

```text
LocalStack + S3 backend + DynamoDB lock table
```

可以验证旧式 backend 锁配置和 state 写入。

但 LocalStack 不完全等价于真实 AWS 的并发锁竞争。

所以你应该把本实验看成：

```text
配置级验证 + 流程训练
```

而不是：

```text
真实 AWS 并发锁压测
```

真正生产上线前，还要在真实 AWS 环境按当前 Terraform 版本和官方建议验证。

## 33. 面试/考试记忆版

可以这样记：

- S3 backend 把 Terraform state 存到 S3。
- DynamoDB lock table 用来保存 state lock。
- state lock 防止多个 Terraform 同时写 state。
- 获取锁失败时 Terraform 不应该继续写 state。
- 旧式配置使用 `dynamodb_table`。
- DynamoDB table 的 partition key 必须是 `LockID`。
- `LockID` 类型必须是 String。
- S3 bucket 和 DynamoDB table 要提前 bootstrap。
- backend 初始化需要 `terraform init`。
- 锁残留时谨慎使用 `terraform force-unlock`。
- 不要在生产里随便用 `-lock=false`。
- 课程文件提到 `dynamodb_table` 在 Terraform 1.14 会提示 deprecated，但存量系统仍常见。
- LocalStack 可用于本地验证 S3/DynamoDB backend 配置，但不完全等价真实并发竞争。

一句话总结：

> 旧式 S3 backend + DynamoDB locking 的核心，是用 S3 集中保存 state，用 DynamoDB 控制谁能写 state，避免团队或 CI/CD 并发写入导致 state 损坏。

## 34. 本 Lab 的价值

第 75 节的价值在于，它从“state 放远端”继续推进到“state 怎么安全写”。

第 74 节解决的是：

```text
state 不要只放本地
```

第 75 节解决的是：

```text
远端 state 不要被多人同时写坏
```

这就是 Terraform 工程化的关键分水岭。

你以后看到老项目里的：

```hcl
backend "s3" {
  bucket         = "..."
  key            = "..."
  region         = "..."
  dynamodb_table = "..."
}
```

不要慌。  
你要能立刻判断：

```text
这是旧式 S3 backend + DynamoDB lock
S3 存 state
DynamoDB 存 lock
LockID 是关键
deprecated 但存量系统常见
排障重点是 bucket、table、endpoint、lock 残留和 init 配置
```

这就是这一节真正要练出来的能力。
