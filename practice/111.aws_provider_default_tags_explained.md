# Terraform 实操训练 111：AWS Provider Default Tags（讲解版）

## 1. 这一节到底在讲什么？

本节核心主题是：

> 在 AWS Provider 层配置 `default_tags`，让所有支持标签的 AWS 资源自动继承默认标签，并理解资源级 `tags` 如何覆盖 provider 默认标签。

简单说，就是不要每个资源都手写一堆重复标签。

以前你可能这样写：

```hcl
resource "aws_s3_bucket" "logs" {
  bucket = "demo-logs"

  tags = {
    Environment = "dev"
    Owner       = "platform"
    Project     = "terraform-training"
  }
}

resource "aws_s3_bucket" "assets" {
  bucket = "demo-assets"

  tags = {
    Environment = "dev"
    Owner       = "platform"
    Project     = "terraform-training"
  }
}
```

资源一多，标签重复就会爆炸。

更好的方式是放到 AWS Provider 里：

```hcl
provider "aws" {
  region = "us-east-1"

  default_tags {
    tags = {
      Environment = "dev"
      Owner       = "platform"
      Project     = "terraform-training"
    }
  }
}
```

然后支持 tag 的资源会自动带上这些默认标签。

---

## 2. 为什么 default_tags 很重要？

在真实 AWS 项目里，标签不是装饰品。

标签经常用于：

- 成本分摊
- 资源归属
- 环境区分
- 自动化清理
- 安全审计
- 合规检查
- 资源搜索
- FinOps 报表
- 事故排查

例如：

```text
Environment = prod
Owner       = sre-team
CostCenter  = cc-1234
Application = billing
ManagedBy   = terraform
```

如果每个资源都靠工程师手写标签，迟早会出现：

```text
Env = prod
environment = production
Environment = Prod
Owner = wang
owner = platform-team
ManagedBy = Terraform
managed_by = terraform
```

标签一乱，FinOps、审计和自动化都会跟着乱。

`default_tags` 的价值就是：  
把一部分“组织级、项目级、环境级”的通用标签前置到 provider 层，减少重复和遗漏。

---

## 3. provider default_tags 怎么写？

标准写法：

```hcl
provider "aws" {
  region = "us-east-1"

  default_tags {
    tags = {
      Environment = "dev"
      Owner       = "platform"
      ManagedBy   = "terraform"
    }
  }
}
```

这表示：

> 通过这个 AWS provider 创建的、支持 tag 的资源，默认都带上这些标签。

例如：

```hcl
resource "aws_s3_bucket" "example" {
  bucket = "demo-default-tags-example"
}
```

虽然资源块里没有写 `tags`，但最终资源会继承 provider 里的默认标签。

---

## 4. 资源级 tags 会发生什么？

资源自己也可以写 `tags`：

```hcl
resource "aws_s3_bucket" "example" {
  bucket = "demo-default-tags-example"

  tags = {
    Application = "logs"
  }
}
```

最终标签大概会合并成：

```text
Environment = dev
Owner       = platform
ManagedBy   = terraform
Application = logs
```

也就是：

```text
provider default_tags + resource tags = 最终 tags
```

---

## 5. 关键规则：资源级 tags 可以覆盖默认标签

如果 provider 默认标签里写了：

```hcl
default_tags {
  tags = {
    Environment = "dev"
    Owner       = "platform"
  }
}
```

资源级又写了同名 key：

```hcl
resource "aws_s3_bucket" "example" {
  bucket = "demo-default-tags-example"

  tags = {
    Environment = "test"
    Application = "logs"
  }
}
```

最终结果会是：

```text
Environment = test
Owner       = platform
Application = logs
```

也就是说：

> 同名标签 key 冲突时，资源级 `tags` 覆盖 provider 层 `default_tags`。

这很合理。  
Provider 默认标签是“全局默认值”，资源级标签是“这个资源的特殊声明”。

---

## 6. default_tags 和 tags_all 的关系

AWS Provider 里经常会看到两个概念：

```text
tags
tags_all
```

它们容易混。

通常可以这样理解：

| 字段 | 含义 |
|---|---|
| `tags` | 资源块里显式写的标签 |
| `tags_all` | 资源最终合并后的全部标签，包括 provider `default_tags` |

例如：

```hcl
provider "aws" {
  default_tags {
    tags = {
      Environment = "dev"
      Owner       = "platform"
    }
  }
}

resource "aws_s3_bucket" "example" {
  bucket = "demo-example"

  tags = {
    Application = "logs"
  }
}
```

资源级 `tags` 是：

```text
Application = logs
```

最终 `tags_all` 是：

```text
Environment = dev
Owner       = platform
Application = logs
```

考试和排障时，如果你看到 `tags_all`，要知道它代表“最终合并结果”。

---

## 7. 为什么不要每个资源都手写公共 tags？

手写公共 tags 的问题不是“麻烦”这么简单，而是会造成治理风险。

假设你有 100 个资源，每个都手写：

```hcl
tags = {
  Environment = "prod"
  Owner       = "platform"
  CostCenter  = "cloud"
}
```

后来 `CostCenter` 要改成：

```text
finops-001
```

你就要改 100 个地方。

然后很可能出现：

```text
有的改了
有的漏了
有的拼错了
有的大小写不一致
```

用 provider `default_tags` 后，只要改一处：

```hcl
default_tags {
  tags = {
    Environment = "prod"
    Owner       = "platform"
    CostCenter  = "finops-001"
  }
}
```

这就是工程化。  
不是因为少写几行代码，而是减少人为漂移。

## 8. 真实项目里怎么设计标签？

一个比较实用的标签集合可以这样设计：

```hcl
default_tags {
  tags = {
    ManagedBy   = "terraform"
    Environment = var.environment
    Project     = var.project
    Owner       = var.owner
    CostCenter  = var.cost_center
  }
}
```

常见标签含义：

| 标签 | 作用 |
|---|---|
| `ManagedBy` | 标识资源由 Terraform 管理 |
| `Environment` | 区分 dev / test / staging / prod |
| `Project` | 标识项目或系统 |
| `Owner` | 标识责任团队 |
| `CostCenter` | 成本中心 |
| `Application` | 应用名称 |
| `DataClass` | 数据敏感级别 |
| `Criticality` | 业务重要程度 |

注意：不要把敏感信息写进 tag。  
例如密码、token、客户隐私、内部机密，不应该进入标签。

---

## 9. 什么时候资源级 tags 覆盖默认 tags 是好事？

比如 provider 默认环境是：

```hcl
Environment = "dev"
```

但某个资源是共享资源，想特殊标记：

```hcl
tags = {
  Environment = "shared"
}
```

那它覆盖默认值是合理的。

再比如 provider 默认 owner 是：

```hcl
Owner = "platform"
```

但某个资源实际归安全团队负责：

```hcl
tags = {
  Owner = "security"
}
```

这也合理。

所以覆盖能力不是 bug，而是灵活性。

但生产里要小心：  
如果团队不加约束，大家都随便覆盖，标签治理会重新失控。

---

## 10. 如何避免资源级 tags 乱覆盖？

可以结合这些手段：

```text
1. 约定哪些标签允许覆盖
2. 约定哪些标签必须继承默认值
3. 用 code review 检查标签变更
4. 用 policy-as-code 限制关键标签
5. 用 tflint / checkov / OPA 做静态检查
6. 用 AWS Organizations Tag Policies 做组织级治理
```

例如关键标签：

```text
ManagedBy
CostCenter
Environment
Owner
```

一般不建议随便覆盖。

资源特有标签：

```text
Application
Component
Backup
Retention
```

可以按资源需求设置。

## 11. LocalStack 在本节里的作用

本实验不访问真实 AWS，而是使用 LocalStack 模拟 AWS API。

结构大概是：

```text
Terraform AWS Provider
-> endpoint 指向 http://localhost:4566
-> 使用假凭证 test/test
-> LocalStack 模拟 AWS 服务
```

这样做的好处：

- 不需要真实 AWS 账号
- 不产生 AWS 费用
- 不会误删真实资源
- 可以安全反复练习
- 适合验证 Terraform 行为

但要记住边界：

```text
LocalStack 能帮助你验证 Terraform 配置路径
但不等于完整模拟真实 AWS 的所有服务细节、权限边界和组织治理策略
```

迁移到真实 AWS 时，还要考虑：

- IAM 权限
- 真实服务是否支持标签
- AWS Organizations tag policy
- SCP 限制
- 成本分配标签激活
- 资源创建时的 tag-on-create 支持
- 现有资源标签漂移

## 12. 哪些资源会继承 default_tags？

不是所有 Terraform 资源都一定支持 AWS tags。

`default_tags` 只会影响 AWS Provider 中支持 tagging 的资源。

例如常见支持 tag 的资源：

```text
VPC
Subnet
Security Group
EC2
S3 Bucket
RDS
EKS
IAM Role
CloudWatch Log Group
```

但一些特殊资源或数据源可能不适用。

所以不要死记“所有资源都有标签”。  
更准确的说法是：

> 使用 AWS Provider 创建的、支持标签的资源，会合并 provider default_tags。

## 13. 实验里应该观察什么？

运行本节实验时，你应该重点观察：

```text
1. provider 里是否配置了 default_tags
2. 资源是否显式设置 tags
3. 同名 key 是否发生覆盖
4. output 是否能看到最终合并后的标签
5. plan 中是否出现 tags_all
6. LocalStack 是否能完成核心路径验证
```

你要真正理解的是：

```text
默认标签从 provider 来
资源标签从 resource 来
最终结果是合并
冲突时资源级优先
```

## 15. 常见错误和排障

## 15.1 以为 default_tags 会覆盖资源级 tags

错误。

实际规则是：

```text
资源级 tags 覆盖 provider default_tags
```

如果同名 key 结果不符合预期，先检查资源块里是否写了相同 key。

---

## 15.2 忘记看 tags_all

有时候你在资源配置里只看到：

```hcl
tags = {
  Application = "logs"
}
```

以为最终资源只有一个标签。

但实际 provider default_tags 会合并进去。  
这时要看 `tags_all` 或 Terraform 输出。

---

## 15.3 标签大小写不统一

AWS tag key 是大小写敏感的。

下面这些是不同 key：

```text
Environment
environment
ENVIRONMENT
env
```

如果团队不统一，成本和审计报表会很难看。

---

## 15.4 以为所有 AWS 资源都支持 tags

不是。

大部分主流资源支持，但不是所有资源都支持。  
遇到异常时先看 provider 文档和资源 schema。

---

## 15.5 在 tag 里放敏感信息

不要这样做：

```hcl
tags = {
  Password = "super-secret"
  Token    = "abc123"
}
```

Tag 可能出现在控制台、日志、账单、审计系统和第三方工具里。  
敏感信息应该放在安全的 secret 管理系统里。

---

## 16. 生产环境推荐做法

推荐把默认标签抽成变量：

```hcl
variable "environment" {
  type = string
}

variable "project" {
  type = string
}

variable "owner" {
  type = string
}

variable "cost_center" {
  type = string
}
```

Provider 中统一配置：

```hcl
provider "aws" {
  region = var.aws_region

  default_tags {
    tags = {
      ManagedBy   = "terraform"
      Environment = var.environment
      Project     = var.project
      Owner       = var.owner
      CostCenter  = var.cost_center
    }
  }
}
```

资源级只写特殊标签：

```hcl
resource "aws_s3_bucket" "logs" {
  bucket = var.bucket_name

  tags = {
    Application = "logging"
    Retention   = "90-days"
  }
}
```

这样结构最清楚：

```text
通用标签：provider default_tags
资源特有标签：resource tags
最终标签：tags_all
```

## 17. 考试重点怎么记？

本节考试重点：

```text
AWS provider 可以配置 default_tags
default_tags 会自动应用到支持标签的资源
资源级 tags 会和 default_tags 合并
同名 key 冲突时，资源级 tags 覆盖 provider 默认标签
tags_all 表示最终合并后的标签
```

如果考题问：

> 组织希望所有 AWS 资源都默认带上 Environment、Owner、CostCenter，应该怎么做？

答案：

```text
在 AWS provider 中配置 default_tags。
```

如果考题问：

> provider default_tags 和 resource tags 有同名 key，哪个优先？

答案：

```text
resource tags 优先，资源级标签覆盖默认标签。
```

## 19. 和真实项目的关系

在真实平台团队里，`default_tags` 经常是 Terraform AWS Provider 的基础配置之一。

它可以帮助你把资源治理做得更稳：

```text
成本归属：CostCenter
责任团队：Owner
环境分类：Environment
项目归属：Project
自动化识别：ManagedBy
```

对你后面做 FinOps、资源治理、合规扫描、自动清理都很有用。

尤其是多账号、多环境、多团队时，没有统一标签标准，后面会非常痛苦。  
`default_tags` 不是高级技巧，是生产 Terraform 的基本卫生习惯。

---

## 20. 总结

一句话总结：

> AWS Provider `default_tags` 用来统一注入默认标签；资源级 `tags` 用来表达资源特有标签；两者同名冲突时，资源级标签优先。

记忆版：

```text
通用标签放 provider，
特殊标签放 resource，
最终结果看 tags_all，
同名冲突 resource 赢。
```

这节不难，但非常生产级。  
真正的云平台不是资源创建出来就完事，而是要能被归属、计费、审计、搜索和治理。
