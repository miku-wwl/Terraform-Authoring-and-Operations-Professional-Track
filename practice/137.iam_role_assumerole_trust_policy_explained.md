# Terraform 考试训练 137：IAM Role 与 AssumeRole 信任策略讲解版

## 1. 课程定位

本节主题是 **IAM Role 与 AssumeRole 信任策略**。

你要抓住一句话：

> IAM Role 本身不是“谁都能用的权限包”，它必须通过 **trust policy** 明确声明“谁可以来扮演我”。

在 Terraform 里创建 IAM Role 时，最核心的字段之一就是：

```hcl
assume_role_policy = data.aws_iam_policy_document.xxx.json
```

这个字段定义的不是 role 能做什么，而是 **谁可以 assume 这个 role**。

本节练习使用 **Terraform AWS Provider + LocalStack** 完成闭环，不访问真实 AWS 账号。LocalStack 主要帮助你练 Terraform 写法、资源依赖、state 变化和 apply/destroy 流程，但它不能完全模拟真实 AWS IAM 的全部权限评估细节。

---

## 2. 需要先分清的两个 policy

IAM Role 相关题目最容易混的地方是：**信任策略** 和 **权限策略**。

### 2.1 Trust policy：谁能使用这个 Role

Trust policy 回答的问题是：

> 谁可以调用 `sts:AssumeRole` 来获得这个 role 的临时身份？

例如，允许 EC2 服务 assume role：

```json
{
  "Effect": "Allow",
  "Principal": {
    "Service": "ec2.amazonaws.com"
  },
  "Action": "sts:AssumeRole"
}
```

这里的重点是：

- `Principal`：谁可以来扮演这个 role
- `Action`：通常是 `sts:AssumeRole`
- `Effect`：允许还是拒绝

### 2.2 Permission policy：Role 被扮演后能做什么

Permission policy 回答的问题是：

> 这个 role 被成功 assume 之后，可以访问哪些 AWS 资源？

比如允许读取 S3：

```json
{
  "Effect": "Allow",
  "Action": ["s3:GetObject"],
  "Resource": ["arn:aws:s3:::example-bucket/*"]
}
```

这和 trust policy 不是一回事。

考试里看到：

- `assume_role_policy`
- `Principal`
- `sts:AssumeRole`
- EC2 / Lambda / ECS 服务主体

优先想到：**信任策略**。

看到：

- `s3:*`
- `ec2:DescribeInstances`
- `dynamodb:GetItem`
- `Resource`

优先想到：**权限策略**。

## 3. AssumeRole 是什么

`AssumeRole` 可以理解为：

> 一个身份临时“换装”成另一个 role，从而拿到这个 role 对应的临时凭证。

AWS 里很多服务不是直接使用长期 access key 工作，而是通过 role 获取临时权限。

例如 EC2 实例访问 S3 的典型链路是：

```text
EC2 instance
  -> 通过 instance profile 关联 IAM Role
  -> EC2 服务根据 trust policy assume role
  -> 获得临时凭证
  -> 使用 role 的 permission policy 访问 S3
```

所以 role 要能被 EC2 使用，至少需要两类配置：

1. Trust policy 允许 `ec2.amazonaws.com` assume role
2. Role 上绑定 permission policy，定义 assume 之后能做什么

本节重点是第一部分：**role 的 trust policy**。

---

## 4. Terraform 中如何写 trust policy

Terraform 推荐用 `aws_iam_policy_document` 来生成 JSON policy，而不是手写一大段 JSON 字符串。

典型写法：

```hcl
data "aws_iam_policy_document" "ec2_assume_role" {
  statement {
    effect = "Allow"

    principals {
      type        = "Service"
      identifiers = ["ec2.amazonaws.com"]
    }

    actions = ["sts:AssumeRole"]
  }
}
```

然后创建 IAM Role：

```hcl
resource "aws_iam_role" "example" {
  name               = "example-ec2-role"
  assume_role_policy = data.aws_iam_policy_document.ec2_assume_role.json
}
```

这段代码的意思是：

> 创建一个 IAM Role，并允许 EC2 服务扮演它。

注意：这里只是允许 EC2 assume role。  
它并没有自动给 EC2 访问 S3、DynamoDB、CloudWatch 的权限。

---

## 5. `aws_iam_policy_document` 的价值

`aws_iam_policy_document` 是 Terraform AWS Provider 提供的数据源，用来生成 IAM policy JSON。

它的优势是：

- HCL 结构更清晰
- 不容易写错 JSON 引号、逗号和括号
- 可以组合多个 statement
- 可以引用 Terraform 变量和资源属性
- 更适合模块化和复用

手写 JSON 不是绝对不行，但考试和工程实践中，更推荐理解并使用 `aws_iam_policy_document`。

## 6. Principal 怎么理解

`Principal` 表示“谁”。

在 trust policy 中，`Principal` 非常关键。

常见类型包括：

### 6.1 Service Principal

允许 AWS 服务 assume role。

例如 EC2：

```hcl
principals {
  type        = "Service"
  identifiers = ["ec2.amazonaws.com"]
}
```

常见服务主体：

```text
ec2.amazonaws.com
lambda.amazonaws.com
ecs-tasks.amazonaws.com
eks.amazonaws.com
```

### 6.2 AWS Principal

允许某个 AWS 账号、用户或 role assume role。

例如跨账号场景：

```hcl
principals {
  type        = "AWS"
  identifiers = ["arn:aws:iam::123456789012:root"]
}
```

这表示允许账号 `123456789012` 里的主体来 assume role。真实生产环境里通常还会搭配更细的权限边界、条件和审计。

### 6.3 Federated Principal

用于 OIDC / SAML 等联合身份场景。

例如 GitHub Actions OIDC、EKS IRSA 等场景会遇到 `Federated` principal。

本节不需要深入展开，但考试里看到 OIDC provider ARN、`Federated`、`Condition`，要知道这属于更高级的信任策略场景。

## 7. 本节 LocalStack 练习边界

本节使用 LocalStack 的目的不是模拟真实 AWS IAM 的所有安全行为，而是训练 Terraform 基本闭环。

LocalStack 常见 endpoint：

```text
http://localhost:4566
```

本地测试凭证通常是：

```text
AWS_ACCESS_KEY_ID=test
AWS_SECRET_ACCESS_KEY=test
```

这类配置只适合本地实验，不代表真实 AWS 安全实践。

你可以在 LocalStack 中练到：

- provider endpoint 配置
- `aws_iam_policy_document` 写法
- `aws_iam_role` 创建
- role ARN 输出
- `terraform init`
- `terraform validate`
- `terraform plan`
- `terraform apply`
- `terraform destroy`

但不要把 LocalStack 结果误解为：

- 已经验证真实 AWS IAM 权限
- 已经验证真实 EC2 可以成功使用该 role
- 已经验证真实 STS assume role 链路
- 已经完成生产级安全评审

LocalStack 是练 Terraform 路径的，不是替代真实 AWS 安全测试的。

## 10. 考试高频判断点

### 10.1 `assume_role_policy` 不是权限策略

这是最核心的坑。

错误理解：

> `assume_role_policy` 定义 role 可以访问哪些资源。

正确理解：

> `assume_role_policy` 定义谁可以 assume 这个 role。

### 10.2 EC2 要用 role，需要 Service Principal

如果题目说“让 EC2 使用某个 IAM Role”，trust policy 里通常要有：

```text
ec2.amazonaws.com
```

并且 action 是：

```text
sts:AssumeRole
```

### 10.3 Role ARN 是身份标识，不是凭证

输出 `role_arn` 不代表拿到了 access key。  
ARN 只是资源标识。

真正的临时凭证来自 STS assume role 流程。

### 10.4 LocalStack 不等于真实 AWS

LocalStack 可以帮助验证 Terraform 语法和资源生命周期。  
但真实 AWS 的 IAM、STS、服务集成和权限边界仍然需要在真实环境中谨慎验证。

## 11. 场景理解

### 场景 1：EC2 需要访问 S3

你创建了一个 IAM Role，并写了如下 trust policy：

```hcl
principals {
  type        = "Service"
  identifiers = ["ec2.amazonaws.com"]
}

actions = ["sts:AssumeRole"]
```

这只解决了：

> EC2 可以扮演这个 role。

但还没有解决：

> 扮演之后能不能访问 S3。

你还需要给 role attach 权限策略，例如允许 `s3:GetObject` 或 `s3:ListBucket`。

### 场景 2：Lambda 需要使用 Role

如果是 Lambda 使用 role，Principal 就不应该写 EC2，而应该写：

```text
lambda.amazonaws.com
```

所以 trust policy 应该根据“谁来扮演 role”变化。

### 场景 3：跨账号 AssumeRole

如果账号 A 的某个身份要 assume 账号 B 的 role，账号 B 的 role trust policy 要信任账号 A 的主体。

大致结构是：

```hcl
principals {
  type        = "AWS"
  identifiers = ["arn:aws:iam::111122223333:root"]
}
```

真实生产里还要考虑：

- 最小权限
- external ID
- MFA
- condition
- CloudTrail 审计
- 谁能调用 `sts:AssumeRole`

考试不会总是深挖到生产安全细节，但你要知道 trust policy 是跨账号 assume role 的入口之一。

## 14. 场景题练习

### 场景题 1

你创建了一个 IAM Role，并设置 trust policy 允许 `ec2.amazonaws.com` assume role。  
同事说：“这样 EC2 就自动拥有访问所有 S3 bucket 的权限了。”

这个说法哪里不准确？

参考答案：

不准确。Trust policy 只定义 EC2 是否可以 assume 这个 role，不定义 role 被 assume 之后能访问哪些资源。要让 EC2 访问 S3，还需要给这个 role 绑定 permission policy，例如允许 `s3:GetObject`、`s3:ListBucket` 等操作，并且遵守最小权限原则。

---

### 场景题 2

你在本地用 LocalStack 成功创建了 IAM Role，并输出了 role ARN。  
这是否说明真实 AWS 账号中的 EC2 一定可以使用这个 role？

参考答案：

不能说明。LocalStack 只能证明本地 Terraform 配置、provider endpoint、资源创建和 state 流程基本可用。真实 AWS 中还需要验证 IAM Role、Instance Profile、STS、EC2 服务集成、权限策略和账号级安全配置。

---

### 场景题 3

一个 Lambda 函数需要使用 IAM Role，但 Terraform 里的 trust policy 写成了：

```hcl
identifiers = ["ec2.amazonaws.com"]
```

问题是什么？

参考答案：

Principal 写错了。Lambda 要 assume role，service principal 通常应为 `lambda.amazonaws.com`，而不是 `ec2.amazonaws.com`。Trust policy 必须匹配实际要扮演 role 的主体。
