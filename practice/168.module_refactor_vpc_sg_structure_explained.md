> 验证结论：`aws-localstack-pass`

---

## 1. 这一节到底在练什么？

这一节不是单纯让你“会写一个 VPC”或者“会写一个 Security Group”。

它真正训练的是：**当一个 Terraform 项目开始变复杂以后，如何把网络资源从散乱的 root module 配置，重构成更清晰的 module 结构，并且保证 plan / apply / verify / destroy 仍然闭环。**

可以理解为：

> 不是从 0 写资源，而是把原来堆在一起的 Terraform 代码整理成工程化结构。

这类题在 Terraform 考试和真实工作里都很常见，因为企业项目很少永远只停留在一个 `main.tf` 文件里。随着资源越来越多，通常会把 VPC、子网、安全组、IAM、存储、计算等拆成模块。

---

## 2. 关键词解释

### 2.1 Root Module

你执行 `terraform init`、`terraform plan`、`terraform apply` 的目录，就是 root module。

例如：

```text
work/lab-168/
├── main.tf
├── variables.tf
├── outputs.tf
└── modules/
```

这里的 `work/lab-168/` 就是 root module。

Root module 的职责通常是：

- 组合多个子模块；
- 传入变量；
- 读取模块输出；
- 管理整体部署入口。

它不应该无限膨胀成一个“什么都写在里面”的巨大文件。

---

### 2.2 Child Module

`modules/` 下面的目录通常就是 child module。

例如：

```text
modules/
├── vpc/
│   ├── main.tf
│   ├── variables.tf
│   └── outputs.tf
└── security_group/
    ├── main.tf
    ├── variables.tf
    └── outputs.tf
```

这里 `vpc` 和 `security_group` 就是两个模块。

它们的职责是：

- 封装一组相关资源；
- 暴露必要变量；
- 暴露必要输出；
- 让 root module 可以像调用函数一样复用它们。

可以简单记：

> Module 就像 Terraform 里的函数。  
> 变量是参数，output 是返回值。

---

### 2.3 VPC 和 Security Group 的依赖关系

在 AWS 里，Security Group 通常需要绑定到某个 VPC。

也就是说，SG 不是孤立存在的，它需要知道：

```hcl
vpc_id = ...
```

如果 VPC 和 SG 被拆成两个模块，那么常见写法是：

```hcl
module "vpc" {
  source = "./modules/vpc"

  name       = var.name
  cidr_block = var.vpc_cidr
}

module "security_group" {
  source = "./modules/security_group"

  name   = var.name
  vpc_id = module.vpc.vpc_id
}
```

这里的关键是：

```hcl
vpc_id = module.vpc.vpc_id
```

它表达了一个非常重要的依赖关系：

> Security Group 模块依赖 VPC 模块输出的 VPC ID。

Terraform 会根据这个引用关系自动推导资源依赖顺序。

## 3. 为什么要做模块重构？

### 3.1 不重构时的问题

如果所有资源都堆在 root module 里，早期看起来很快：

```text
main.tf 里直接写 VPC、subnet、SG、EC2、S3……
```

但资源一多，就会出现几个问题：

| 问题 | 后果 |
|---|---|
| 文件巨大 | 很难阅读和维护 |
| 资源边界不清楚 | 不知道哪些属于网络，哪些属于计算 |
| 复用困难 | 下一个环境还要复制粘贴 |
| 输出混乱 | 资源之间引用关系越来越乱 |
| 改动风险高 | 改一个 SG 可能误伤 VPC 配置 |

模块化的目标不是“显得高级”，而是降低复杂度。

---

### 3.2 重构后的收益

模块重构之后，结构会更清晰：

```text
root module
├── 调用 vpc module
├── 调用 security_group module
└── 输出最终需要的信息
```

模块分别负责自己的边界：

| 模块 | 负责内容 |
|---|---|
| `vpc` module | VPC、CIDR、网络基础结构 |
| `security_group` module | 安全组、入站规则、出站规则 |
| root module | 组合模块、传参、输出结果 |

这样更接近真实工程实践。

一个合理的练习结构可以是：

```text
work/lab-168/
```

如果是 LocalStack 环境，还需要确保 provider endpoint 指向：

```hcl
endpoints {
  ec2 = "http://localhost:4566"

```

凭证使用固定测试值：

```hcl
access_key = "test"
secret_key = "test"
```

注意：这只是本地实验环境，不是真实 AWS 凭证。

## 5. 关键 Terraform 写法

### 5.1 Root Module 调用 VPC 模块

```hcl
module "vpc" {
  source = "./modules/vpc"

  name       = var.name
  cidr_block = var.vpc_cidr_block
}
```

这里 root module 不直接关心 VPC 里面怎么创建资源，只负责传参。

---

### 5.2 VPC 模块内部

`modules/vpc/main.tf` 示例：

```hcl
resource "aws_vpc" "this" {
  cidr_block = var.cidr_block

  tags = {
    Name = var.name
  }
}
```

`modules/vpc/outputs.tf` 示例：

```hcl
output "vpc_id" {
  value = aws_vpc.this.id
}
```

这一步非常重要。

如果 VPC 模块不输出 `vpc_id`，外面的 SG 模块就无法引用它。

---

### 5.3 Root Module 调用 SG 模块

```hcl
module "security_group" {
  source = "./modules/security_group"

  name   = "${var.name}-sg"
  vpc_id = module.vpc.vpc_id
}
```

重点是：

```hcl
vpc_id = module.vpc.vpc_id
```

这不是普通字符串，而是模块输出引用。

---

### 5.4 Security Group 模块内部

`modules/security_group/main.tf` 示例：

```hcl
resource "aws_security_group" "this" {
  name   = var.name
  vpc_id = var.vpc_id

  ingress {
    description = "Allow HTTP"
    from_port   = 80
    to_port     = 80
    protocol    = "tcp"
    cidr_blocks = ["0.0.0.0/0"]
  }

  egress {
    description = "Allow all outbound"
    from_port   = 0
    to_port     = 0
    protocol    = "-1"
    cidr_blocks = ["0.0.0.0/0"]
  }
}
```

`modules/security_group/variables.tf` 示例：

```hcl
variable "vpc_id" {
  type        = string
  description = "VPC ID where the security group is created."
}
```

这体现了模块之间的边界：

- VPC 模块负责创建 VPC；
- SG 模块不自己创建 VPC；
- SG 模块只接收 `vpc_id`。

## 6. 本节最重要的依赖链

这一节可以记成一条链：

```text
root module
  -> module.vpc
    -> output vpc_id
      -> module.security_group
        -> aws_security_group.this
```

也就是说：

1. root module 调用 VPC 模块；
2. VPC 模块创建 `aws_vpc`；
3. VPC 模块输出 `vpc_id`；
4. root module 把 `module.vpc.vpc_id` 传给 SG 模块；
5. SG 模块用这个 `vpc_id` 创建安全组。

考试看到模块引用题时，重点就看这条链有没有断。

## 7. LocalStack 边界

原始课程强调，本项目使用 LocalStack 模拟 AWS API。

这意味着你可以练：

- Terraform AWS Provider 配置；
- module 调用；
- 变量传递；
- output 输出；
- state 变化；
- plan / apply / destroy 闭环；
- 基础 AWS API 调用路径。

但它不能完全替代真实 AWS：

| 能力 | LocalStack 能不能完全代表真实 AWS？ |
|---|---|
| Terraform 语法验证 | 可以 |
| Provider 调用路径 | 可以练 |
| VPC / SG 基础资源创建 | 可以模拟 |
| 真实网络连通性 | 不能完全代表 |
| 真实 AWS IAM 权限评估 | 不能完全代表 |
| 生产级网络路由行为 | 不能完全代表 |

所以本节重点不是证明“网络真的能通”，而是证明：

> Terraform 代码结构、模块依赖、状态闭环是正确的。

## 8. 实操验证流程

推荐按下面顺序跑：

```bash
terraform fmt -recursive
terraform init
terraform validate
terraform plan
terraform apply
terraform output
terraform destroy
```

每一步关注点不同。

### 8.1 `terraform fmt -recursive`

检查 root module 和 child module 里的 `.tf` 文件格式。

模块化后一定要加 `-recursive`，否则可能只格式化当前目录，漏掉 `modules/` 下面的文件。

```bash
terraform fmt -recursive
```

---

### 8.2 `terraform init`

初始化 provider 和本地 module。

```bash
terraform init
```

如果修改了 module source 或新增了模块，通常也需要重新 init。

---

### 8.3 `terraform validate`

检查语法和引用关系。

```bash
terraform validate
```

常见错误包括：

- 变量没定义；
- output 名称写错；
- module 路径不存在；
- 类型不匹配；
- `module.vpc.vpc_id` 这种引用拼错。

---

### 8.4 `terraform plan`

观察 Terraform 准备创建什么资源。

```bash
terraform plan
```

你应该能看到类似：

- 创建 VPC；
- 创建 Security Group；
- SG 使用 VPC ID；
- 输出相关 ID。

---

### 8.5 `terraform apply`

应用配置。

```bash
terraform apply
```

在 LocalStack 中，这一步会向本地模拟 AWS endpoint 发请求，而不是真实 AWS。

---

### 8.6 `terraform output`

检查输出是否合理。

```bash
terraform output
```

建议输出：

```hcl
output "vpc_id" {
  value = module.vpc.vpc_id
}

output "security_group_id" {
  value = module.security_group.security_group_id
}
```

---

### 8.7 `terraform destroy`

清理资源。

```bash
terraform destroy
```

挑战题一定要有 destroy 闭环，否则容易留下状态残留。

---

## 9. 常见错误

### 错误 1：SG 模块里自己又创建一个 VPC

错误思路：

```text
security_group module 自己创建 VPC
```

这样会让模块边界混乱。

更合理的是：

```text
VPC module 创建 VPC
SG module 接收 vpc_id
```

---

### 错误 2：VPC 模块没有 output

如果 VPC 模块只创建资源，但没有：

```hcl
output "vpc_id" {
  value = aws_vpc.this.id
}
```

root module 就无法写：

```hcl
module.vpc.vpc_id
```

这会直接导致引用错误。

---

### 错误 3：module 路径写错

例如：

```hcl
source = "./module/vpc"
```

但真实目录是：

```text
./modules/vpc
```

这类错误非常常见，考试题里也可能用路径细节卡你。

---

### 错误 4：变量名和 output 名混在一起

模块内部变量这样定义：

```hcl
variable "vpc_id" {}
```

模块输出这样定义：

```hcl
output "vpc_id" {}
```

虽然名字可以一样，但语义完全不同：

| 类型 | 位置 | 含义 |
|---|---|---|
| variable | 模块入口 | 外部传进来的值 |
| output | 模块出口 | 模块暴露出去的值 |

记住：

> variable 是入口，output 是出口。

---

### 错误 5：忘记 `terraform fmt -recursive`

模块化项目如果只执行：

```bash
terraform fmt
```

可能只格式化当前目录。更稳的是：

```bash
terraform fmt -recursive
```

## 10. 考试判断点

### 10.1 看到 module source，判断是不是本地模块

```hcl
source = "./modules/vpc"
```

这是本地模块。

如果是：

```hcl
source = "terraform-aws-modules/vpc/aws"
```

这是 Terraform Registry 模块。

本节重点是本地模块重构。

---

### 10.2 看到 `module.xxx.yyy`，判断 yyy 是 output

例如：

```hcl
vpc_id = module.vpc.vpc_id
```

这里后面的 `vpc_id` 必须是 VPC 模块中定义的 output。

它不是变量名，也不是资源名。

---

### 10.3 看到模块依赖，优先看引用关系

Terraform 不需要你手写：

```hcl
depends_on = [module.vpc]
```

只要你写了：

```hcl
vpc_id = module.vpc.vpc_id
```

Terraform 通常就能自动推导依赖顺序。

只有在没有显式引用但又确实存在顺序要求时，才考虑 `depends_on`。

## 11. 面试解释版本

如果面试官问：

> 你怎么把 VPC 和 Security Group 做模块化？

可以这样答：

```text
我会把 VPC 和 Security Group 拆成两个独立模块。VPC 模块只负责创建网络基础资源，并通过 output 暴露 vpc_id。Security Group 模块不自己创建 VPC，而是通过 variable 接收 vpc_id。Root module 负责组合这两个模块，把 module.vpc.vpc_id 传给 security_group 模块。这样模块边界清晰，依赖关系也能被 Terraform 自动推导。
```

如果想更工程化一点：

```text
这种拆法可以避免网络资源和安全规则混在一起，也方便后续在 dev、staging、prod 中复用。同一套 security_group 模块可以绑定不同环境的 VPC，只需要传入不同的 vpc_id。
```

## 13. 本节复盘问题

### Q1：为什么 SG 模块不应该自己创建 VPC？

因为 SG 模块的职责是管理安全组，而不是管理网络基础设施。它应该通过变量接收 `vpc_id`，这样边界更清晰，也更容易复用。

---

### Q2：`module.vpc.vpc_id` 的最后一个 `vpc_id` 来自哪里？

来自 `modules/vpc/outputs.tf` 中定义的 output。

例如：

```hcl
output "vpc_id" {
  value = aws_vpc.this.id
}
```

---

### Q3：模块化后为什么要用 `terraform fmt -recursive`？

因为 `.tf` 文件分布在 root module 和多个 child module 中。`-recursive` 可以递归格式化所有子目录，避免漏掉 `modules/` 下面的 Terraform 文件。

---

### Q4：LocalStack 中通过了，就等于真实 AWS 完全没问题吗？

不等于。

LocalStack 可以验证 Terraform 语法、Provider 调用路径和基础资源闭环，但不能完全代表真实 AWS 的 IAM 权限、网络连通性和生产级行为。

---

## 14. 最终结论

这一节的核心不是某个 AWS 参数，而是 Terraform 工程结构能力。

你要掌握的是：

- 如何拆分 root module 和 child module；
- 如何让 VPC 模块输出 `vpc_id`；
- 如何让 SG 模块通过变量接收 `vpc_id`；
- 如何用 `module.vpc.vpc_id` 建立模块依赖；
- 如何在 LocalStack 中完成 fmt / init / validate / plan / apply / verify / destroy 闭环。

一句话总结：

> 第 168 节练的是：把散乱的 VPC / SG 配置，重构成可维护、可复用、可验证的 Terraform 模块结构。
