## 2. 知识点讲解

Run trigger 用来解决 workspace 之间存在依赖时的自动触发问题。课程中的典型场景是：网络团队维护公共 IP、子网或网络输出，安全团队的 Terraform 配置通过 remote state 读取这些输出。如果网络 workspace 更新后，安全 workspace 不自动重新运行，就可能继续使用旧数据。

HCP Terraform 的 run trigger 可以让一个 workspace 在源 workspace 成功 apply 后触发另一个 workspace 的 run。它不是替代 remote state data source，而是补足 remote state 的触发问题。Remote state 能让一个配置读取另一个配置的输出，但它不会天然保证下游 workspace 在上游变化后自动重新 plan。

考试容易考这类因果关系：上游 workspace 的 state 输出变化，下游 workspace 依赖这些输出，如何自动让下游重新评估。答案通常指向 run triggers，而不是手动复制输出、把所有配置合并到一个巨大 workspace，或让用户凭记忆执行 apply。

## 3. 考试选择题

1. Run trigger 主要解决什么问题？
   - A. 上游 workspace 变更后，自动触发依赖 workspace 的 run

   - C. 下载 Docker 镜像
   - D. 创建本地 SSH key

2. Remote state data source 的作用是什么？

   - D. 管理浏览器缓存

3. 如果下游配置依赖上游输出，但上游变更后下游没有自动运行，应该考虑什么 HCP Terraform 功能？
   - A. run triggers
   - B. `terraform fmt`
   - C. provider lock file
   - D. `terraform graph`

## 4. 判断题

1. Run trigger 可以帮助处理 workspace 之间的运行依赖。
2. Remote state data source 本身就一定会触发所有依赖 workspace 自动 apply。
3. 上游 workspace apply 成功后触发下游 plan，是 run trigger 的典型用途。

## 5. 场景题

1. 网络 workspace 输出公网 IP 列表，安全 workspace 读取这个列表生成安全规则。网络团队新增 IP 后，安全 workspace 应该自动重新 plan。你会使用什么能力？

2. 有人建议把网络、安全、应用所有 Terraform 配置都放进一个 workspace，只为避免触发顺序问题。请说明这种方案可能带来的问题。

## 6. 参考答案与解析

选择题：1-A，2-A，3-A。

判断题：1-对，2-错，3-对。

场景题参考：

1. 使用 run triggers，让上游网络 workspace apply 成功后触发安全 workspace 的 run，并由 remote state 提供输出数据。
2. 巨大 workspace 会扩大 blast radius，降低职责边界清晰度，增加 plan 噪音和权限管理难度。更合理的是拆分 workspace，并显式管理依赖触发。

## 7. 复习提醒

- Remote state 解决“读数据”，run trigger 解决“何时重新运行”。
- 看到上游/下游 workspace，先判断是否存在触发依赖。
- 不要为了依赖问题盲目合并所有 workspace。
