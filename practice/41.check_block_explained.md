# Terraform 实操训练 41 讲解：Check Block

## 1. 这一节到底在学什么？

Lab 41 学的是 Terraform 里的 `check` block。

前面几节你已经学过：

- `validation`：变量入口校验
- `precondition`：资源评估前检查
- `postcondition`：资源评估后检查
- `fileexists()`：检查本地前置文件是否存在

这一节的 `check` block 又是另一类机制：它不绑定某一个资源生命周期，更像是 Terraform 配置里的**旁路合约检查**。

一句话理解：

> `check` block 适合表达“整体配置必须满足某个规则”，而不是某个变量或某个资源自己的生命周期规则。

---

## 2. `check` block 是什么？

`check` block 是 Terraform 里的独立检查块。  
它可以写在资源之外，用来验证某些条件是否成立。

典型结构如下：

```hcl
check "service_url_must_use_https" {
  assert {
    condition     = startswith(var.service_url, "https://")
    error_message = "Service URL must use HTTPS."
  }
}
```

这里的意思是：

> 服务地址必须以 `https://` 开头。

如果不满足，Terraform 会报告 check 失败。

---

## 3. 本实验的场景

Lab 41 的背景是：

> 服务地址必须使用 HTTPS。

问题是：

> 如果用户传入 HTTP 地址，就不符合生产安全要求。

任务是：

> 用 `check` block 校验 URL 前缀。

预期结果：

> 默认 HTTPS 地址通过检查。

这类规则很像平台团队给模块定义的“安全合约”：

- 服务地址必须是 HTTPS
- 某些关键输出不能为空
- 某些资源组合必须满足组织策略
- 某些外部依赖必须处于可用状态

---

## 4. 为什么不用变量 validation？

你可能会问：  
“检查 URL 前缀，用变量 `validation` 不也能做吗？”

可以。比如：

```hcl
variable "service_url" {
  type = string

  validation {
    condition     = startswith(var.service_url, "https://")
    error_message = "Service URL must use HTTPS."
  }
}
```

这当然没问题。

但是 `check` block 的定位不一样。  
它更适合表达**资源外的整体合约**，尤其是当规则不只属于某一个变量时。

例如：

- 多个变量组合后必须满足条件
- 某个 output 必须满足条件
- 某个 data source 查询结果必须满足条件
- 部署完成后某个外部服务必须可访问
- 某组资源整体要满足某个约束

所以可以这样区分：

> 如果只是单个变量的输入边界，用 `validation`。  
> 如果是整体配置、外部依赖、运行合约，用 `check` block。

---

## 5. `check` block 和 `precondition` 有什么区别？

`precondition` 是资源级检查，写在资源的 `lifecycle` 里。

例如：

```hcl
resource "terraform_data" "service" {
  input = var.service_url

  lifecycle {
    precondition {
      condition     = startswith(var.service_url, "https://")
      error_message = "Service URL must use HTTPS."
    }
  }
}
```

这表示这个检查属于 `terraform_data.service` 这个资源。

而 `check` block 是独立的：

```hcl
check "service_url_contract" {
  assert {
    condition     = startswith(var.service_url, "https://")
    error_message = "Service URL must use HTTPS."
  }
}
```

它不属于某一个具体资源。

所以区别是：

| 机制 | 绑定对象 | 适合场景 |
|---|---|---|
| `validation` | 变量 | 输入值是否合法 |
| `precondition` | 资源或 data source | 资源执行前是否允许继续 |
| `postcondition` | 资源或 data source | 资源评估后结果是否合格 |
| `check` block | 独立检查块 | 全局健康检查、合约检查、外部依赖检查 |

---

## 6. `check` block 的核心结构

一个 `check` block 通常包含一个或多个 `assert`。

示例：

```hcl
check "service_url_contract" {
  assert {
    condition     = startswith(var.service_url, "https://")
    error_message = "service_url must start with https://"
  }
}
```

结构拆开看：

```hcl
check "service_url_contract"
```

定义一个检查块，名字应该表达检查目的。

```hcl
assert
```

定义一条断言。

```hcl
condition
```

判断条件，返回 `true` 或 `false`。

```hcl
error_message
```

失败时给用户看的错误信息。

---

## 7. 为什么 HTTPS 检查很适合讲 `check` block？

因为它不是资源创建本身的能力问题，而是一个**安全合约**。

HTTP 地址也许技术上可以用，但生产规范不允许。

这类规则不一定来自云平台 API，而是来自组织要求：

- 服务入口必须使用 HTTPS
- 镜像地址必须来自受信任 registry
- 环境名必须符合约定
- 外部回调地址必须是公司域名
- 监控 endpoint 必须存在
- 关键开关不能在生产关闭

`check` block 正好适合表达这种“平台约束”。

## 8. 标准运行流程

进入实验目录：

```sh
cd /workspace/practice/labs/41
```

初始化：

```sh
terraform init -input=false
```

格式检查：

```sh
terraform fmt -check
```

配置校验：

```sh
terraform validate
```

运行测试：

```sh
terraform test
```

生成 plan：

```sh
terraform plan -input=false -no-color -out=tfplan
```

执行 apply：

```sh
terraform apply -auto-approve tfplan
```

查看输出：

```sh
terraform output
```

清理：

```sh
terraform destroy -auto-approve
```

清理本地文件：

```sh
rm -f tfplan
rm -rf .terraform .terraform.lock.hcl
```

## 9. 如何手动验证失败场景？

可以故意传入 HTTP 地址：

```sh
terraform plan -var='service_url=http://example.com'
```

预期结果是 check 失败，并提示 URL 必须使用 HTTPS。

然后改成 HTTPS：

```sh
terraform plan -var='service_url=https://example.com'
```

预期结果应该可以通过。

这个验证很直观：

- `http://` 不符合合约
- `https://` 符合合约

## 10. `startswith()` 在这里做了什么？

本实验通常会用到：

```hcl
startswith(var.service_url, "https://")
```

它的作用是判断字符串是否以指定前缀开头。

例如：

```hcl
startswith("https://api.example.com", "https://")
```

返回：

```hcl
true
```

而：

```hcl
startswith("http://api.example.com", "https://")
```

返回：

```hcl
false
```

所以它很适合做 URL scheme 检查。

## 11. 常见坑

### 坑 1：把 `check` 当成资源依赖管理

`check` block 是检查，不是依赖建模。

如果资源 A 必须等资源 B 创建完成，应该通过显式引用或 `depends_on` 建模，而不是靠 `check` block 旁路判断。

错误思路：

> 我写个 check 看 B 是否存在，这样 A 就会依赖 B。

不严谨。

正确思路：

> 真实依赖要写成 Terraform 依赖关系，check 只负责验证合约。

---

### 坑 2：检查外部服务时忽略网络不稳定

`check` block 可以用来表达外部依赖检查，但外部网络可能不稳定。

如果检查逻辑依赖网络访问，就要考虑：

- CI runner 是否能访问外网
- DNS 是否稳定
- 目标服务是否偶发超时
- 是否会造成 flaky pipeline

不要把不稳定的外部探测随便塞进关键部署流程里。

---

### 坑 3：错误信息写得太模糊

不好的错误信息：

```hcl
error_message = "Invalid URL."
```

更好的错误信息：

```hcl
error_message = "service_url must start with https:// for production security requirements."
```

好的错误信息应该告诉用户：

- 哪个值错了
- 为什么错
- 应该怎么改

---

### 坑 4：和 variable validation 重复太多

如果一个规则只是单个变量自己的合法性，优先放在 `validation` 里。  
如果规则是全局合约或跨资源检查，再考虑 `check` block。

不要为了用新语法而乱用新语法。Terraform 写得成熟，不是语法越花越好，而是约束放在正确位置。

## 13. 生产环境怎么用？

`check` block 在生产里可以用于这些场景：

### 检查服务 URL

```hcl
check "service_url_https" {
  assert {
    condition     = startswith(var.service_url, "https://")
    error_message = "Service URL must use HTTPS."
  }
}
```

### 检查输出不为空

```hcl
check "endpoint_is_not_empty" {
  assert {
    condition     = length(trimspace(output.service_endpoint)) > 0
    error_message = "Service endpoint must not be empty."
  }
}
```

### 检查环境合约

```hcl
check "prod_requires_monitoring" {
  assert {
    condition     = var.environment != "prod" || var.monitoring_enabled
    error_message = "Production environment must enable monitoring."
  }
}
```

### 检查命名约定

```hcl
check "service_name_contract" {
  assert {
    condition     = can(regex("^svc-[a-z0-9-]+$", var.service_name))
    error_message = "Service name must match svc-[a-z0-9-]+."
  }
}
```

这些都属于“整体合约”，不一定只属于某一个资源。

## 14. SRE / Platform Engineering 视角

从平台工程角度看，`check` block 的意义很直接：

> 把平台规则写成可执行检查，而不是靠 README 和口头约定。

比如平台团队规定：

- 生产 URL 必须是 HTTPS
- 生产必须开启监控
- 关键资源必须打 owner tag
- 非生产不能使用高成本配置
- 外部依赖必须明确声明

这些规则如果只写在文档里，很容易被忽略。  
写进 Terraform 检查里，才真正进入工程流程。

---

## 15. 面试/考试记忆版

可以这样记：

- `check` block 是资源外检查。
- 它不绑定单个资源生命周期。
- 常用于全局健康检查、外部依赖检查、合约验证。
- `assert` 里写 `condition` 和 `error_message`。
- 单变量合法性优先用 `validation`。
- 资源前后条件优先用 `precondition` / `postcondition`。
- 真实依赖关系不要靠 `check` block 替代。

一句话总结：

> `check` block 是 Terraform 里的旁路合约检查，适合验证整体配置是否符合平台规则。

## 16. 本 Lab 的价值

Lab 41 的价值是让你理解 Terraform 不只是创建资源，也可以表达“部署合约”。

这对 SRE / Platform Engineering 很关键。  
成熟的 Terraform 模块不应该只关心资源能不能创建，还应该关心：

- 配置是否符合安全要求
- 关键输入是否满足组织约束
- 部署结果是否符合预期
- 用户是否违反了平台合约

这就是从“写 Terraform”到“设计 Terraform 平台模块”的区别。
