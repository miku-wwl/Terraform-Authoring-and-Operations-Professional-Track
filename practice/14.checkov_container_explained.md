# Terraform 实操训练 14：Checkov 容器化安装验证讲解

> 目标：把原本“在 Ubuntu 服务器上安装 Checkov”的步骤，改造成**容器化、可重复、适合 CI/CD 的验证方式**。  
> 核心结论：本地或 CI runner 不需要手工安装 Python 包，只需要 Docker，就可以稳定运行 Checkov。

---

## 1. 这个 Lab 在练什么？

本实验练的是一个非常典型的平台工程 / DevOps 场景：

> 团队想在 Terraform 流水线中加入 Checkov 做 IaC 安全扫描，但不希望每台机器都手工安装 Checkov。

传统做法可能是：

```bash
pip install checkov
checkov --version
```

这种方式在个人电脑上还可以，但放到 CI/CD 里就容易出问题：

- runner 上的 Python 版本不一致；
- pip 源、网络、缓存状态不稳定；
- 不同机器安装出的 Checkov 版本可能不同；
- 工具升级后，扫描结果可能突然变化；
- 构建环境被污染，不够干净。

所以这个 Lab 采用了更工程化的方案：

```powershell
docker run --rm bridgecrew/checkov:latest --version
```

也就是直接使用 Checkov 官方容器镜像来运行工具。

---

## 2. Checkov 是什么？

Checkov 是一个 Infrastructure as Code 安全扫描工具，常用于扫描：

- Terraform
- Kubernetes YAML
- Helm chart
- CloudFormation
- Dockerfile
- GitHub Actions workflow
- Terraform plan 文件

在 Terraform 项目里，它常用来检查类似问题：

- S3 bucket 是否公开；
- 安全组是否开放 `0.0.0.0/0`；
- 数据库是否开启加密；
- 日志、备份、版本控制是否启用；
- IAM 权限是否过大；
- Kubernetes 配置是否存在安全风险。

一句话理解：

> Terraform 负责“把基础设施建出来”，Checkov 负责“检查你建出来的东西有没有明显安全坑”。

---

## 3. 为什么要容器化运行 Checkov？

### 3.1 避免污染本机环境

如果直接使用 `pip install checkov`，会依赖本机 Python 环境。  
但是不同机器可能有不同的 Python 版本、pip 版本、依赖缓存，容易出现：

```text
在我机器上可以跑，在 CI runner 上挂了。
```

容器化以后，Checkov 和它的依赖都被封装在镜像里，本机只需要 Docker。

---

### 3.2 提高 CI/CD 可重复性

在 CI/CD 里，最重要的是可重复。

同一个 commit，在今天、明天、不同 runner 上运行，都应该得到尽量一致的结果。

容器化运行可以把工具版本、运行环境、依赖项一起封装起来，减少环境漂移。

---

### 3.3 更容易统一团队标准

平台团队通常希望规定：

```text
所有 Terraform 项目都用同一个版本的 Checkov。
```

如果每个项目自己安装，很难保证版本一致。  
如果统一使用容器镜像，就可以在流水线模板中固定镜像版本。

例如生产环境中更推荐：

```bash
docker run --rm bridgecrew/checkov:3.3.5 --version
```

而不是长期使用：

```bash
docker run --rm bridgecrew/checkov:latest --version
```

---

## 4. `latest` 为什么不适合长期用于生产 CI？

本实验为了学习方便，使用的是：

```powershell
docker run --rm bridgecrew/checkov:latest --version
```

但生产 CI 中不建议长期使用 `latest`。

原因很简单：

> `latest` 不是一个稳定版本承诺，它只是当前镜像仓库里某个默认标签。

今天拉到的 `latest` 可能是 Checkov 3.3.5，过几天可能就是 3.3.6 或更高版本。

这会带来几个风险：

| 风险 | 说明 |
|---|---|
| 扫描结果突然变化 | 新版本可能新增规则，导致原来通过的项目突然失败 |
| 排障困难 | CI 失败时，很难第一时间知道是不是工具版本变了 |
| 审计困难 | 生产流水线需要知道当时到底使用了哪个工具版本 |
| 团队标准不稳定 | 不同时间跑同一个项目，结果可能不同 |

所以生产建议固定版本，例如：

```bash
docker run --rm bridgecrew/checkov:3.3.5 --version
```

进一步严谨的话，可以固定镜像 digest：

```bash
docker run --rm bridgecrew/checkov@sha256:<digest> --version
```

## 6. 命令逐条讲解

### 6.1 验证 Checkov 容器可运行

```powershell
docker run --rm bridgecrew/checkov:latest --version
```

拆开看：

| 部分 | 含义 |
|---|---|
| `docker run` | 启动一个容器 |
| `--rm` | 容器退出后自动删除，不留下垃圾容器 |
| `bridgecrew/checkov:latest` | 使用 Checkov 镜像 |
| `--version` | 输出 Checkov 版本 |

如果命令正常执行，说明：

- Docker 可用；
- 镜像可以拉取；
- Checkov 容器可以启动；
- Checkov 命令可以执行。

本实验中的验证结果是：

```text
Checkov version: 3.3.5
```

---

### 6.2 进入实验目录

```sh
cd /workspace/practice/labs/14
```

这一步是为了进入当前 Lab 的 Terraform 工作目录。

Terraform 命令通常要在包含 `.tf` 文件的目录中运行。

---

### 6.3 初始化 Terraform

```sh
terraform init -input=false
```

`terraform init` 会初始化当前 Terraform 工作目录。

常见动作包括：

- 初始化 backend；
- 下载 provider；
- 初始化 `.terraform` 目录；
- 读取 lock file。

这里加上：

```sh
-input=false
```

表示不要交互式提问。

这在 CI/CD 中很重要，因为流水线不能像人一样手动输入 yes/no。

---

### 6.4 检查格式

```sh
terraform fmt -check
```

这条命令不会自动改文件，只检查格式是否符合 Terraform 标准。

如果格式不对，它会失败。

CI 中通常使用：

```sh
terraform fmt -check -recursive
```

本实验只检查当前目录即可。

---

### 6.5 校验 Terraform 语法

```sh
terraform validate
```

这一步检查 Terraform 配置是否合法，例如：

- 语法是否正确；
- block 是否写错；
- 参数类型是否合理；
- provider schema 是否匹配。

它不会真正创建资源。

---

### 6.6 运行 Terraform Test

```sh
terraform test
```

`terraform test` 用来运行 `.tftest.hcl` 测试文件。

本 Lab 中，它的意义不是测试云资源，而是把“Checkov 安装策略”变成可验证的规则。

也就是说，不只是 README 里口头说一句：

```text
生产 CI 应该固定 Checkov 镜像版本。
```

而是通过测试把这个要求固化下来。

这就是工程化思维：

> 重要规则不要只写在文档里，能自动验证的就尽量自动验证。

## 7. 这个 Lab 的关键学习点

### 7.1 工具也应该被版本管理

很多人只会给应用依赖做版本管理，例如：

- npm package；
- NuGet package；
- Maven dependency；
- Terraform provider。

但 CI/CD 工具本身也要版本管理，例如：

- Terraform CLI；
- Checkov；
- TFLint；
- Trivy；
- tfsec；
- kubectl；
- helm。

否则流水线结果会漂移。

---

### 7.2 容器是统一工具链的好办法

在平台工程里，工具容器化是很常见的做法。

比如：

```bash
docker run --rm hashicorp/terraform:1.11 version
docker run --rm bridgecrew/checkov:3.3.5 --version
docker run --rm aquasec/trivy:0.50.0 --version
```

这比在每台 runner 上安装一堆工具更可控。

---

### 7.3 CI runner 应尽量轻量

理想状态下，CI runner 只需要：

- Git；
- Docker；
- 基础 shell 环境。

其他工具都由容器镜像提供。

这样 runner 更容易维护，也更容易横向扩展。

## 8. 常见错误和排障

### 8.1 镜像拉不下来

可能报错：

```text
Unable to find image 'bridgecrew/checkov:latest' locally
docker: Error response from daemon
```

常见原因：

- 网络访问 Docker Hub 失败；
- 公司网络限制；
- 镜像仓库限流；
- DNS 问题；
- Docker daemon 没启动。

解决思路：

```bash
docker pull bridgecrew/checkov:latest
docker info
docker version
```

公司环境里可以考虑把镜像同步到内部镜像仓库。

---

### 8.2 Terraform test 失败

如果 `terraform test` 失败，重点检查：

- 测试文件路径是否正确；
- `.tftest.hcl` 语法是否正确；
- 测试中断言的内容是否和 `main.tf` 一致；
- 是否遗漏了“固定版本”的说明；
- 是否还在强调长期使用 `latest`。

---

### 8.3 `terraform init` 卡住或等待输入

CI 中不要让 Terraform 等待输入。

应该使用：

```bash
terraform init -input=false
```

后续 `plan` 和 `apply` 在 CI 中也常见类似写法：

```bash
terraform plan -input=false
terraform apply -input=false -auto-approve
```

不过生产环境的 `apply` 一般还要配合审批门禁，不是所有流水线都直接自动 apply。

## 10. 可以如何扩展这个 Lab？

这个 Lab 目前只是验证 Checkov 容器可运行。后续可以扩展成真正的 IaC 安全扫描流水线。

### 10.1 扫描 Terraform 目录

```bash
docker run --rm   -v "$PWD":/tf   bridgecrew/checkov:3.3.5   -d /tf
```

含义：

| 参数 | 作用 |
|---|---|
| `-v "$PWD":/tf` | 把当前目录挂载进容器 |
| `-d /tf` | 让 Checkov 扫描 `/tf` 目录 |

---

### 10.2 输出 JUnit 报告给 CI 使用

```bash
docker run --rm   -v "$PWD":/tf   bridgecrew/checkov:3.3.5   -d /tf   -o junitxml   --output-file-path /tf/checkov-report.xml
```

这样 CI 平台可以读取测试报告。

---

### 10.3 在 GitHub Actions 中使用

示例：

```yaml
name: IaC Security Scan

on:
  pull_request:
    paths:
      - "**/*.tf"

jobs:
  checkov:
    runs-on: ubuntu-latest

    steps:
      - uses: actions/checkout@v4

      - name: Run Checkov
        run: |
          docker run --rm             -v "$PWD":/tf             bridgecrew/checkov:3.3.5             -d /tf
```

这就把本 Lab 的手工命令升级成了 PR 级别的安全门禁。

## 11. 这个 Lab 最终要记住什么？

记住这四句话就够了：

1. Checkov 是 IaC 安全扫描工具，可以检查 Terraform 等配置中的安全问题。
2. 不建议在每个 CI runner 上手工安装 Checkov，容易产生版本漂移。
3. 用 Docker 容器运行 Checkov，可以让工具链更干净、更稳定、更可重复。
4. 生产 CI 不要长期使用 `latest`，应该固定明确版本，必要时固定 digest。
