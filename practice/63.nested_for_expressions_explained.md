# Terraform 实操训练 63 讲解：嵌套 for 表达式

## 1. 这一节到底在学什么？

Lab 63 学的是 Terraform 里的**嵌套 for 表达式**。

前面 Lab 59 和 Lab 61 已经讲过：

- `for` 表达式基础
- `for` 表达式进阶
- 过滤元素
- 转换 list / map
- 为 `for_each` 准备稳定 map
- `flatten()` 展开嵌套列表

这一节继续深入：  
当一个数据结构里面还有一层集合时，怎么用两层 `for` 把它展开、转换、组合成 Terraform 能用的数据。

一句话理解：

> 嵌套 for 表达式就是 for 里面再套 for，用来处理“集合里面还有集合”的数据结构。

典型场景：

```text
多个服务，每个服务有多个端口
多个环境，每个环境有多个 region
多个团队，每个团队有多个 repo
多个用户，每个用户有多个权限
多个 subnet，每个 subnet 有多个 route
```

这类结构都适合用嵌套 for 表达式处理。

---

## 2. 为什么需要嵌套 for？

真实配置经常不是一层 flat list。

例如服务配置：

```hcl
services = [
  {
    name  = "api"
    ports = [8080, 9090]
  },
  {
    name  = "web"
    ports = [3000]
  }
]
```

这不是简单的一组服务。  
它是：

```text
服务列表
  -> 每个服务下面还有端口列表
```

如果你想生成每个服务端口组合：

```hcl
[
  { service = "api", port = 8080 },
  { service = "api", port = 9090 },
  { service = "web", port = 3000 }
]
```

就需要两层遍历：

1. 第一层遍历服务。
2. 第二层遍历每个服务的端口。

这就是嵌套 for 的价值。

---

## 3. 基础结构：for 里面套 for

嵌套 for 常见写法：

```hcl
locals {
  nested = [
    for svc in var.services : [
      for port in svc.ports : {
        service = svc.name
        port    = port
      }
    ]
  ]
}
```

这段逻辑是：

1. 遍历 `var.services`。
2. 对每个 `svc`，再遍历 `svc.ports`。
3. 为每个端口生成一个 object。
4. 最终得到一个 list 里面套 list 的结构。

注意：这一步生成的是嵌套 list，不是一层 list。

---

## 4. 嵌套 for 的直接结果通常是二维 list

假设输入：

```hcl
services = [
  {
    name  = "api"
    ports = [8080, 9090]
  },
  {
    name  = "web"
    ports = [3000]
  }
]
```

表达式：

```hcl
[
  for svc in var.services : [
    for port in svc.ports : {
      service = svc.name
      port    = port
    }
  ]
]
```

结果会是：

```hcl
[
  [
    {
      service = "api"
      port    = 8080
    },
    {
      service = "api"
      port    = 9090
    }
  ],
  [
    {
      service = "web"
      port    = 3000
    }
  ]
]
```

这是二维 list：

```text
list(list(object))
```

如果你要给资源使用，通常需要变成一层 list。

---

## 5. `flatten()`：把二维 list 展平成一层

Terraform 的 `flatten()` 可以把嵌套 list 展开成一层。

写法：

```hcl
locals {
  service_ports = flatten([
    for svc in var.services : [
      for port in svc.ports : {
        service = svc.name
        port    = port
      }
    ]
  ])
}
```

结果：

```hcl
[
  {
    service = "api"
    port    = 8080
  },
  {
    service = "api"
    port    = 9090
  },
  {
    service = "web"
    port    = 3000
  }
]
```

这就是嵌套 for 最常见套路：

```hcl
flatten([
  for outer in outer_collection : [
    for inner in outer.inner_collection : {
      ...
    }
  ]
])
```

一句话：

> 嵌套 for 负责生成组合，`flatten()` 负责把二维结果拍平。

---

## 6. 为什么光 flatten 还不够？

`flatten()` 后得到的是 list(object)。

例如：

```hcl
[
  { service = "api", port = 8080 },
  { service = "api", port = 9090 },
  { service = "web", port = 3000 }
]
```

如果你要用它创建多个长期资源，不建议直接配 `count`。

因为又会回到：

```text
[0]
[1]
[2]
```

的索引问题。

更好的方式是继续转成 map：

```hcl
locals {
  service_ports_by_key = {
    for item in local.service_ports :
    "${item.service}-${item.port}" => item
  }
}
```

这样资源地址可以是：

```text
local_file.service_port["api-8080"]
local_file.service_port["api-9090"]
local_file.service_port["web-3000"]
```

比数字索引稳定得多。

---

## 7. 完整链路：嵌套结构到 for_each

一个完整的生产链路通常是：

```hcl
locals {
  service_ports = flatten([
    for svc in var.services : [
      for port in svc.ports : {
        service = svc.name
        port    = port
      }
    ]
  ])

  service_ports_by_key = {
    for item in local.service_ports :
    "${item.service}-${item.port}" => item
  }
}

resource "local_file" "service_port" {
  for_each = local.service_ports_by_key

  filename = "${path.module}/output/${each.key}.txt"
  content  = "service=${each.value.service}, port=${each.value.port}"
}
```

这条链路很关键：

```text
嵌套输入
-> 嵌套 for 展开组合
-> flatten 展平成一层
-> for 转成 map
-> for_each 创建资源
```

这就是 Terraform 数据建模的生产味儿。

## 15. 嵌套 for 和过滤条件

内层或外层都可以加过滤条件。

例如只展开 enabled 的服务：

```hcl
locals {
  service_ports = flatten([
    for svc in var.services : [
      for port in svc.ports : {
        service = svc.name
        port    = port
      }
    ]
    if svc.enabled
  ])
}
```

或者只展开高端口：

```hcl
locals {
  high_service_ports = flatten([
    for svc in var.services : [
      for port in svc.ports : {
        service = svc.name
        port    = port
      }
      if port > 1024
    ]
  ])
}
```

注意过滤条件放在哪里，含义不同：

- 外层 `if`：过滤整个服务。
- 内层 `if`：过滤服务里面的端口。

这点很重要。

---

## 16. 外层 if 和内层 if 的区别

输入：

```hcl
services = [
  {
    name    = "api"
    enabled = true
    ports   = [80, 8080]
  },
  {
    name    = "worker"
    enabled = false
    ports   = [9090]
  }
]
```

### 外层过滤

```hcl
flatten([
  for svc in var.services : [
    for port in svc.ports : {
      service = svc.name
      port    = port
    }
  ]
  if svc.enabled
])
```

含义：

> 只处理 enabled 的服务。

worker 整个被过滤掉。

### 内层过滤

```hcl
flatten([
  for svc in var.services : [
    for port in svc.ports : {
      service = svc.name
      port    = port
    }
    if port > 1024
  ]
])
```

含义：

> 所有服务都处理，但只保留大于 1024 的端口。

这两个逻辑不一样。

## 17. 嵌套 for 和 key 设计

展开后转 map 时，key 设计非常重要。

例如：

```hcl
"${item.service}-${item.port}"
```

这个 key 由 service 和 port 组成。

它应该满足：

- 唯一
- 稳定
- 可读
- 不容易冲突
- 不包含频繁变化字段

如果你有更复杂组合，可以用更明确的分隔符：

```hcl
"${item.environment}/${item.service}/${item.port}"
```

或者：

```hcl
"${item.environment}-${item.region}-${item.service}"
```

但是要避免 key 中包含会频繁改变的描述、版本、随机值。

---

## 18. key 冲突怎么办？

如果两个 item 生成了同一个 key：

```hcl
"${item.service}-${item.port}"
```

Terraform 会报错，因为 map key 必须唯一。

解决思路：

1. 检查数据是否应该唯一。
2. 使用更多字段组成 key。
3. 修改输入数据模型。
4. 如果本来就是分组需求，使用 `...` 分组语法。
5. 不要随便加随机数糊弄。

例如服务名 + 端口不唯一，可以加协议：

```hcl
"${item.service}-${item.protocol}-${item.port}"
```

key 的设计其实是资源身份设计。

## 19. 嵌套 for 和 `setproduct()`

有些二维组合也可以用 `setproduct()`。

例如：

```hcl
locals {
  environments = ["dev", "prod"]
  regions      = ["au", "us"]

  pairs = setproduct(local.environments, local.regions)
}
```

结果类似：

```hcl
[
  ["dev", "au"],
  ["dev", "us"],
  ["prod", "au"],
  ["prod", "us"]
]
```

然后可以转换：

```hcl
locals {
  env_regions = [
    for pair in local.pairs : {
      environment = pair[0]
      region      = pair[1]
    }
  ]
}
```

但是如果每个环境自己的 region 不同，嵌套 for 更清楚。

简单理解：

- 完全笛卡尔积：可以考虑 `setproduct()`
- 每个对象有自己的子集合：用嵌套 for 更自然

## 20. 嵌套 for 和可读性

嵌套 for 很强，但也容易写得吓人。

不推荐一行写完：

```hcl
flatten([for a in var.a : [for b in a.b : [for c in b.c : {...}]]])
```

这种东西过几天你自己都不想看。

更好的写法是拆 locals：

```hcl
locals {
  service_ports = flatten([
    for svc in var.services : [
      for port in svc.ports : {
        service = svc.name
        port    = port
      }
    ]
  ])

  service_ports_by_key = {
    for item in local.service_ports :
    "${item.service}-${item.port}" => item
  }
}
```

Terraform 工程里，可读性比炫技重要。

## 21. 常见坑

### 坑 1：忘记 flatten

嵌套 for 默认常生成二维 list。

如果资源需要一层 list，却拿到 list(list(object))，就会出错。

---

### 坑 2：过滤条件放错层级

外层 `if` 和内层 `if` 意义不同。

放错位置会导致过滤结果完全不一样。

---

### 坑 3：key 不唯一

展开后转 map 时，key 必须唯一。

否则 Terraform 会报 duplicate key 错误。

---

### 坑 4：key 不稳定

key 一变，资源地址就变。

这会导致 Terraform 认为旧资源删除、新资源创建。

---

### 坑 5：嵌套太深

两层 for 常见。  
三层以上就要考虑拆数据结构、拆 locals、甚至拆模块。

## 22. Terraform test 在这里有什么价值？

Lab 63 的验证报告显示：

```text
terraform test 返回 1 passed, 0 failed
```

对嵌套 for 来说，测试很有价值。

因为嵌套 for 的错误经常是逻辑错误：

- 展开数量不对
- 少 flatten 一层
- 过滤条件错
- key 不符合预期
- 组合字段错
- 输出顺序不稳定
- 资源数量不对

测试可以验证：

- 展开后有几个组合
- 某个 key 是否存在
- 某个组合的字段是否正确
- output 是否符合预期
- for_each map 是否稳定

## 23. 生产环境里的典型模式

### 模式 1：展开嵌套对象

```hcl
locals {
  expanded = flatten([
    for parent in var.parents : [
      for child in parent.children : {
        parent = parent.name
        child  = child
      }
    ]
  ])
}
```

### 模式 2：展开后转 map

```hcl
locals {
  expanded_by_key = {
    for item in local.expanded :
    "${item.parent}-${item.child}" => item
  }
}
```

### 模式 3：给 for_each 使用

```hcl
resource "local_file" "item" {
  for_each = local.expanded_by_key

  filename = "${path.module}/output/${each.key}.txt"
  content  = jsonencode(each.value)
}
```

这三步是嵌套 for 最常见生产模式。

## 26. 什么时候不要用嵌套 for？

不要为了炫技滥用。

如果数据本身可以更简单建模，就先优化输入结构。

例如你已经知道最终就是：

```hcl
service_ports = {
  "api-8080" = {
    service = "api"
    port    = 8080
  }
}
```

那就不一定非要从复杂嵌套结构展开。

嵌套 for 适合处理真实的层级关系。  
如果只是为了把简单事情写复杂，就没必要。

## 27. 面试/考试记忆版

可以这样记：

- 嵌套 for 是 for 里面再套 for。
- 常用于处理 list 里还有 list 的结构。
- 嵌套 for 通常会生成二维 list。
- 常用 `flatten()` 展平成一层。
- 展平后如果要创建资源，常转成 map 给 `for_each`。
- map key 必须唯一且稳定。
- 外层 `if` 过滤父对象，内层 `if` 过滤子对象。
- 两层 for 常见，三层以上要注意可读性。
- `setproduct()` 适合完整笛卡尔积，嵌套 for 适合父子结构。
- terraform test 可以验证展开数量和 key。

一句话总结：

> 嵌套 for 表达式的核心价值，是把层级配置展开成稳定、可测试、可用于 for_each 的资源实例模型。

---

## 28. 本 Lab 的价值

Lab 63 的价值在于，它让你真正开始处理“多维数据”。

单层 for 只是基础转换。  
嵌套 for 才是很多平台工程场景的核心：

- 多服务多端口
- 多环境多区域
- 多团队多仓库
- 多用户多权限
- 多集群多 namespace

学好这节后，你会更容易读懂真实 Terraform 项目里的复杂 locals，也能更稳地把外部数据、服务清单、权限矩阵转换成 Terraform 资源。

这就是 Terraform 从“写资源”走向“做平台自动化”的关键一步。
