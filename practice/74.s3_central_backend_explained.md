# Terraform 实操训练 74 讲解：S3 中央后端

## 1. 这一节到底在学什么？

第 74 节学的是 Terraform 的 **S3 backend**，也就是把 Terraform state 从本地文件迁移到远端 S3 存储。

一句话理解：

> S3 backend 的核心价值，是把 `terraform.tfstate` 从个人电脑挪到一个集中位置，让团队可以共享同一份基础设施状态。

在本地 backend 下，Terraform 默认会在当前目录生成：

```text
terraform.tfstate
terraform.tfstate.backup
```

但使用 S3 backend 后，state 会写入 S3 bucket 中，本地目录不再保存主 state 文件。

这就是本节最重要的观察点：

```text
本地不再生成 terraform.tfstate
state 被写入 S3 backend
```

---

## 2. Terraform state 是什么？

Terraform state 是 Terraform 管理资源的“账本”。

它记录：

- Terraform 创建了哪些资源
- 每个资源的真实 ID
- 资源当前属性
- Terraform 配置和真实基础设施之间的映射关系
- resource address 和云资源之间的对应关系

例如 Terraform 配置里是：

```hcl
resource "aws_s3_bucket" "example" {
  bucket = "demo-bucket"
}
```

真实 AWS 里有一个 bucket。  
Terraform 需要知道：

```text
aws_s3_bucket.example -> demo-bucket
```

这个映射就保存在 state 里。

没有 state，Terraform 就很难判断：

- 资源是否已经存在
- plan 应该新增、修改还是删除
- 哪个配置对应哪个真实资源

所以 state 很重要，不能乱丢。

---

## 3. 本地 state 的问题

默认情况下，Terraform 使用本地 backend。

也就是 state 存在当前目录：

```text
terraform.tfstate
```

这适合个人练习，但不适合团队生产环境。

问题包括：

1. **无法多人共享**

   A 同学 apply 后 state 在 A 的电脑上。  
   B 同学不知道真实资源状态，容易产生冲突。

2. **容易丢失**

   本地文件被删、电脑坏了、目录清理了，state 就没了。

3. **不适合 CI/CD**

   GitHub Actions、GitLab CI、Jenkins 每次运行环境可能都是新的。  
   如果 state 只在本地，CI 就无法持续管理资源。

4. **容易误提交**

   有人可能把 `terraform.tfstate` 提交到 Git。  
   state 里可能包含敏感信息，这是大坑。

5. **没有并发保护**

   两个人同时 apply，可能互相覆盖状态。

所以生产里通常不用本地 state，而是用远端 backend。

---

## 4. 什么是 Terraform backend？

backend 决定 Terraform state 存在哪里，以及 Terraform 如何访问它。

常见 backend：

- local backend
- S3 backend
- AzureRM backend
- GCS backend
- Terraform Cloud / HCP Terraform backend

本节关注的是：

```hcl
backend "s3"
```

它的作用是：

> 把 Terraform state 存到 S3 里。

典型配置：

```hcl
terraform {
  backend "s3" {
    bucket = "terraform-state-demo"
    key    = "labs/74/terraform.tfstate"
    region = "us-east-1"
  }
}
```

这里：

- `bucket`：存 state 的 S3 bucket
- `key`：state 文件在 bucket 里的路径
- `region`：bucket 所在 region

---

## 5. S3 backend 的核心机制

使用 S3 backend 后，Terraform 运行时会：

1. 初始化 backend。
2. 连接 S3。
3. 读取远端 state。
4. 根据配置生成 plan。
5. apply 后把新的 state 写回 S3。

流程可以理解成：

```text
terraform init
  -> 配置 S3 backend

terraform plan
  -> 从 S3 读取 state
  -> 和当前配置对比

terraform apply
  -> 执行变更
  -> 把更新后的 state 写回 S3
```

本地目录里不会再保存主 state：

```text
terraform.tfstate
```

state 的主副本在 S3。

---

## 6. DynamoDB 锁是干什么的？

真实 AWS 生产环境里，S3 backend 通常会配合 DynamoDB 做 state locking。

原因是：

> 防止两个人或两个 pipeline 同时执行 Terraform apply。

如果没有锁，可能出现：

```text
A 正在 apply
B 也开始 apply
A 写 state
B 又写 state
最终 state 混乱
```

DynamoDB lock 的作用是：

```text
谁先执行 Terraform，谁先拿锁
其他人必须等待或失败
```

典型配置类似：

```hcl
terraform {
  backend "s3" {
    bucket         = "terraform-state-demo"
    key            = "labs/74/terraform.tfstate"
    region         = "us-east-1"
    dynamodb_table = "terraform-locks"
  }
}
```

注意：新版本 Terraform 和 AWS backend 对 locking 方式有更新趋势，生产中要按当前官方文档确认最佳做法。  
但概念上你必须理解：

```text
S3 存 state
DynamoDB 或 backend locking 机制负责并发保护
```

本节实验里使用 LocalStack，只做配置级验证，不做真实并发竞态压力测试。

## 9. 本实验最重要的现象

本节最重要的验证点是：

> 配置 S3 backend 后，本地目录不再生成 `terraform.tfstate`，state 会写入 S3 backend。

也就是说，你执行完 Terraform 后，如果在工作目录看不到：

```text
terraform.tfstate
```

这不是失败，反而可能说明 backend 生效了。

但你需要确认 state 已经写入 S3。

可以用 AWS CLI 指向 LocalStack 检查：

```sh
aws --endpoint-url=http://localhost:4566 s3 ls s3://<bucket-name>
```

或者：

```sh
aws --endpoint-url=http://localhost:4566 s3 ls s3://<bucket-name>/<prefix>/
```

如果看到类似：

```text
terraform.tfstate
```

说明 state 已经在 S3 里。

## 10. S3 backend 配置示例

真实 AWS 中可能类似：

```hcl
terraform {
  backend "s3" {
    bucket = "my-terraform-state"
    key    = "env/dev/terraform.tfstate"
    region = "ap-southeast-2"
  }
}
```

LocalStack 中通常要增加一些兼容配置，例如 endpoint、跳过校验等。

示例思路：

```hcl
terraform {
  backend "s3" {
    bucket = "terraform-state"
    key    = "labs/74/terraform.tfstate"
    region = "us-east-1"

    endpoints = {
      s3 = "http://localhost:4566"
    }

    skip_credentials_validation = true
    skip_metadata_api_check     = true
    skip_region_validation      = true
    force_path_style            = true
  }
}
```

不同 Terraform / provider 版本写法可能略有差异。  
本节重点是理解配置意图：

- backend 类型是 S3
- bucket/key 指定 state 位置
- endpoint 指向 LocalStack
- 跳过真实 AWS 校验

## 11. 为什么 backend 需要 `terraform init`？

backend 配置变更后，必须重新运行：

```sh
terraform init
```

因为 `init` 会初始化：

- provider plugin
- modules
- backend
- state 迁移流程

如果你从 local backend 改成 S3 backend，Terraform 会在 init 时询问或处理 state 迁移。

常见命令：

```sh
terraform init
```

自动化环境中可以用：

```sh
terraform init -input=false
```

如果涉及迁移，还可能用：

```sh
terraform init -migrate-state
```

或者重新配置 backend：

```sh
terraform init -reconfigure
```

简单记忆：

> backend 变了，先 init。

---

## 12. `-migrate-state` 和 `-reconfigure` 的区别

### `-migrate-state`

用于把现有 state 迁移到新 backend。

例如：

```text
原来 state 在本地
现在要迁到 S3
```

可以用：

```sh
terraform init -migrate-state
```

它的语义是：

> 我知道 backend 变了，请帮我迁移已有 state。

---

### `-reconfigure`

用于重新配置 backend，但不尝试迁移已有 state。

```sh
terraform init -reconfigure
```

它的语义是：

> 忘掉之前的 backend 配置，按当前配置重新初始化。

在实验环境里，如果你反复清理重跑，`-reconfigure` 经常有用。

生产里不要随便乱用，尤其是已经有真实 state 的时候。

---

## 13. backend 配置不能使用普通变量

一个常见坑是想这样写：

```hcl
terraform {
  backend "s3" {
    bucket = var.state_bucket
  }
}
```

这不行。

backend 初始化发生得很早，早于普通变量求值。  
所以 backend block 里不能直接使用 `var.xxx`、`local.xxx`。

生产里通常有几种方式：

1. 写死 backend 配置。
2. 使用 `-backend-config` 文件。
3. 使用命令行传 backend 参数。
4. 按环境维护不同 backend config。

例如：

```sh
terraform init   -backend-config="bucket=my-terraform-state"   -backend-config="key=env/dev/terraform.tfstate"   -backend-config="region=ap-southeast-2"
```

或者：

```sh
terraform init -backend-config=backend-dev.hcl
```

---

## 14. `backend-dev.hcl` 示例

可以准备一个 backend 配置文件：

```hcl
bucket = "my-terraform-state"
key    = "env/dev/terraform.tfstate"
region = "ap-southeast-2"
```

然后初始化：

```sh
terraform init -backend-config=backend-dev.hcl
```

这种方式适合多环境：

```text
backend-dev.hcl
backend-test.hcl
backend-prod.hcl
```

但要注意：  
backend config 里不要放敏感凭据。

---

## 15. S3 backend 的 key 怎么设计？

`key` 是 state 文件在 S3 bucket 里的路径。

例如：

```hcl
key = "env/dev/network/terraform.tfstate"
```

设计得好，后面很好维护。

常见结构：

```text
<environment>/<system>/<component>/terraform.tfstate
```

例如：

```text
dev/platform/network/terraform.tfstate
prod/platform/eks/terraform.tfstate
prod/app/payment/terraform.tfstate
```

不要所有项目都用同一个 key：

```text
terraform.tfstate
```

否则多个项目会互相覆盖 state，直接炸锅。

## 16. S3 bucket 存 state 的生产建议

真实 AWS 里，state bucket 应该认真配置。

建议：

1. 开启 versioning。
2. 开启 encryption。
3. 限制 bucket public access。
4. 使用严格 IAM policy。
5. 启用访问日志或 CloudTrail。
6. 分环境或分账号隔离 state。
7. 配置 locking。
8. 不允许随便删除 state。
9. state bucket 不和业务数据混用。
10. 做好灾难恢复。

Terraform state 是基础设施控制面的一部分。  
它的重要性很高。

## 17. state 里可能有敏感信息

很多人以为：

```text
我没 output secret，就没事
```

不一定。

Terraform state 可能保存资源属性，包括一些敏感字段。

例如：

- password
- token
- connection string
- private endpoint info
- generated secret
- user data
- provider 返回的敏感属性

所以 state backend 不是普通文件存储。  
它必须按敏感数据保护。

S3 backend 生产环境要配好：

- IAM 权限
- 加密
- versioning
- 审计
- lock
- 删除保护

## 23. `terraform state list` 的意义

运行：

```sh
terraform state list
```

Terraform 会从 backend 读取 state，然后列出当前管理的资源地址。

如果 backend 生效，它读取的是 S3 里的 state，而不是本地 `terraform.tfstate`。

这能验证：

```text
Terraform 仍能访问 state
但 state 不在本地目录
```

这就是远端 backend 的核心效果。

## 24. 常见坑

### 坑 1：忘记先创建 S3 bucket

S3 backend 初始化时，bucket 通常必须已经存在。

Terraform backend 自己不能依赖同一个配置里的 resource 去创建 bucket。  
因为 backend 初始化发生在 resource plan/apply 之前。

解决：

- 手动创建 state bucket
- 用 bootstrap Terraform 单独创建
- 用脚本创建
- 在 LocalStack 实验中提前创建 bucket

---

### 坑 2：backend 配置里使用 var/local

backend block 不能直接用普通变量：

```hcl
bucket = var.bucket
```

要用 `-backend-config` 或写死配置。

---

### 坑 3：LocalStack endpoint 配置错误

如果 endpoint 没指向 LocalStack，Terraform 可能会尝试连接真实 AWS。

要检查：

- S3 endpoint
- STS endpoint
- region
- skip credentials validation
- fake credentials
- path-style 配置

---

### 坑 4：误以为没有 terraform.tfstate 就是失败

使用远端 backend 后，本地没有主 state 是正常现象。

关键是检查 S3 中是否有 state object。

---

### 坑 5：多个项目共用同一个 key

如果两个 Terraform 项目共用：

```hcl
key = "terraform.tfstate"
```

会互相覆盖 state，非常危险。

---

### 坑 6：忘记清理 LocalStack

实验环境里 LocalStack 容器、volume、bucket 数据可能残留。  
下次实验可能被旧 state 干扰。

## 25. backend 和 provider 的区别

这个概念必须分清楚。

### backend

决定 Terraform state 存在哪里。

例如：

```hcl
terraform {
  backend "s3" {}
}
```

### provider

决定 Terraform 管理哪些基础设施 API。

例如：

```hcl
provider "aws" {
  region = "us-east-1"
}
```

backend 和 provider 都可能连接 AWS，但它们不是一回事。

S3 backend 是 Terraform 自己读写 state。  
AWS provider 是 Terraform 创建/修改 AWS 资源。

在 LocalStack 实验中，backend 和 provider 都可能需要指向 LocalStack endpoint。

---

## 26. backend 为什么要单独 bootstrap？

真实生产里，S3 state bucket 本身也可以用 Terraform 创建。

但问题是：

> Terraform 要先有 backend 才能管理资源，而 backend bucket 又是 Terraform 资源。

这就像鸡和蛋。

常见解决办法：

1. 手动创建 state bucket。
2. 用一个 bootstrap 项目创建 backend 基础设施。
3. bootstrap 项目可以使用 local backend。
4. 业务项目使用 bootstrap 创建好的 S3 backend。

结构类似：

```text
bootstrap/
  创建 state bucket、lock table

envs/dev/
  使用 S3 backend 存业务 state
```

生产里这是很常见的分层方式。

## 27. 和 CI/CD 的关系

S3 backend 是 Terraform CI/CD 的基础之一。

在 GitHub Actions 里，如果每次 runner 都是新的：

```text
没有远端 state -> Terraform 不知道之前创建过什么
```

使用 S3 backend 后：

```text
每次 pipeline 都从 S3 读取同一份 state
```

所以 CI 可以持续管理同一套基础设施。

再配合锁机制，可以避免多个 pipeline 同时 apply。

这就是从个人 Terraform 到团队 Terraform 的关键一步。

- Terraform 是否能从 backend 读取 state
- destroy 后流程是否完整
- LocalStack 清理是否干净
- DynamoDB locking 是否至少配置正确

注意：LocalStack 对 DynamoDB 并发锁的真实竞态行为只做配置级验证。  
也就是说，这节不是压测锁竞争，而是验证 backend 配置路径没问题。

## 30. 面试/考试记忆版

可以这样记：

- Terraform state 是资源映射账本。
- backend 决定 state 存在哪里。
- local backend 会在本地生成 `terraform.tfstate`。
- S3 backend 会把 state 存到 S3。
- 使用 S3 backend 后，本地目录不再生成主 state 文件。
- S3 backend 常配合 locking 机制防止并发 apply。
- backend 变更后要运行 `terraform init`。
- 迁移已有 state 可用 `terraform init -migrate-state`。
- 重新配置 backend 可用 `terraform init -reconfigure`。
- backend block 不能直接使用普通变量。
- S3 backend 的 bucket 通常要提前 bootstrap。
- 生产 state bucket 要开启加密、versioning、严格 IAM 和审计。
- CI/CD 使用 remote backend 才能持续管理同一套资源。
- LocalStack 可以模拟 S3/DynamoDB/STS 做本地验证。

一句话总结：

> S3 backend 的核心价值，是把 Terraform state 从个人本地文件迁移到集中、可共享、可保护、适合团队和 CI/CD 的远端位置。

## 31. 本 Lab 的价值

第 74 节的价值很大，因为它从“写 Terraform 配置”进入了“管理 Terraform 运行方式”。

初学 Terraform 时，很多人只关注：

```text
resource 怎么写
variable 怎么写
output 怎么写
```

但真正上生产后，首先要问的是：

```text
state 放在哪里？
谁能访问？
有没有锁？
能不能审计？
CI/CD 怎么读？
多人同时 apply 怎么办？
state 丢了怎么办？
```

这节就是这个分水岭。

如果说前面的 lab 是 Terraform 语言训练，那么第 74 节开始就是 Terraform 工程化训练。

记住一句狠话：

> 不懂 state/backend 的 Terraform，只能算会写脚本；懂 state/backend，才开始像基础设施工程师。
