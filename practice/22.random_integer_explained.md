# Terraform 实操训练 22：`random_integer` 与唯一命名

## 0. 这一课要解决什么问题

这一课讲的是 Terraform 里的随机资源：`random_integer`。

很多云资源、测试产物、临时文件、流水线输出，都不能永远用固定名称。比如：

- `artifact.txt`
- `demo-bucket`
- `test-resource`

这些名字在多人协作、重复执行 CI、或者创建云资源时，很容易冲突。

所以 Terraform 里经常会用随机后缀来生成更不容易冲突的名字，例如：

```text
artifact-4821.txt
artifact-7350.txt
artifact-9182.txt
```

本实验用 `random_integer` 生成一个 1000 到 9999 之间的随机数字，再把它拼进本地 artifact 文件名里。

---

## 1. 核心知识点

### 1.1 `random_integer` 是什么

`random_integer` 是 Terraform random provider 里的资源，用来生成一个指定范围内的随机整数。

典型写法类似：

```hcl
resource "random_integer" "suffix" {
  min = 1000
  max = 9999
}
```

这表示 Terraform 会生成一个 1000 到 9999 之间的整数。

注意，它不是每次运行都重新随机。

这是这节课最重要的一点。

---

### 1.2 随机值会进入 Terraform state

Terraform 的随机资源生成结果后，会把结果保存进 state。

也就是说：

第一次 apply：

```text
random_integer.suffix.result = 4821
```

这个值会被写入 state。

之后你再次运行：

```sh
terraform plan
terraform apply
```

只要资源没有被替换，Terraform 会继续使用 `4821`，而不是重新生成一个新数字。

这很关键。

否则如果随机值每次都变，资源名也会每次变，那 Terraform 就会天天想删旧资源、建新资源，生产环境直接变成“摇号拆迁现场”。

---

## 2. 为什么 Terraform 要这样设计

Terraform 的核心目标不是“每次制造新东西”，而是“让实际状态收敛到配置声明的目标状态”。

所以随机资源虽然名字叫 random，但它在 Terraform 语义里仍然是一个资源。

资源一旦创建，就应该稳定存在。

因此：

```hcl
resource "random_integer" "suffix" {
  min = 1000
  max = 9999
}
```

更准确的理解是：

> 创建这个随机资源时生成一个整数，然后由 Terraform state 管理它的生命周期。

不是：

> 每次执行 Terraform 都重新随机一个整数。

这个区别很重要，Terraform 考试和实际 CI/CD 都会遇到。

---

## 3. 本实验的场景

### 背景

自动化流水线需要创建带唯一后缀的产物名。

例如：

```text
artifact-1234.txt
artifact-5678.txt
```

### 问题

如果所有环境都用固定名称，比如：

```text
artifact.txt
```

那么多个环境、多个开发者、多个 CI job 之间就可能冲突。

### 任务

使用 `random_integer` 生成随机后缀，并把这个后缀用于本地 artifact 文件名。

### 预期结果

随机后缀满足：

```text
1000 <= suffix <= 9999
```

并且文件名中包含这个随机后缀。

## 5. 推荐运行流程

进入实验目录：

```sh
cd /workspace/practice/labs/22
```

初始化 Terraform：

```sh
terraform init -input=false
```

检查格式：

```sh
terraform fmt -check
```

校验配置：

```sh
terraform validate
```

运行 Terraform test：

```sh
terraform test
```

生成 plan 文件：

```sh
terraform plan -input=false -no-color -out=tfplan
```

执行 apply：

```sh
terraform apply -auto-approve tfplan
```

查看输出：

```sh
terraform output
```

最后清理资源：

```sh
terraform destroy -auto-approve
```

## 6. 每个命令在干什么

### 6.1 `terraform init -input=false`

初始化工作目录，下载 provider。

这里通常会下载：

- `hashicorp/random`
- 可能还有 `hashicorp/local`

`-input=false` 表示不要交互式提问，适合 CI/CD。

---

### 6.2 `terraform fmt -check`

检查 `.tf` 文件格式是否符合 Terraform 标准。

如果格式不对，它会失败，但不会自动修改文件。

CI 里常用这个方式阻断格式不规范的提交。

---

### 6.3 `terraform validate`

检查配置语法和基本结构是否正确。

它不会真正创建资源。

可以理解成 Terraform 配置的静态语法检查。

---

### 6.4 `terraform test`

运行 `.tftest.hcl` 测试文件。

本实验里测试重点通常是：

- 随机数是否在 1000 到 9999 之间
- artifact 文件名是否包含随机后缀
- 输出是否符合预期

这一步很适合用来把“实验意图”固化下来。

不是只靠人肉看命令输出，而是让 Terraform 自己帮你验证。

---

### 6.5 `terraform plan -out=tfplan`

生成执行计划，并保存成 `tfplan` 文件。

这一步的意义是：

- 先看 Terraform 准备创建什么
- 把这次计划保存下来
- 后续 apply 就执行这个保存过的计划

---

### 6.6 `terraform apply -auto-approve tfplan`

执行刚才保存的 plan。

注意，这里 apply 的是：

```text
tfplan
```

不是重新现场生成一个新的 plan。

这在 CI/CD 里很重要，可以保证审批内容和执行内容一致。

---

### 6.7 `terraform output`

查看 Terraform 输出。

在这个实验里，你应该能看到类似：

```text
suffix = 4821
artifact_name = "artifact-4821.txt"
```

具体数字会因首次创建时的随机结果而不同。

---

## 7. 关键理解：为什么随机值没变

很多初学者会疑惑：

> 我都叫 random 了，为什么第二次 apply 不换一个？

因为 Terraform 把这个随机值当作资源状态管理。

第一次生成后，state 里会保存结果。

只要下面这些条件没变：

- `random_integer.suffix` 资源还在 state 里
- `min` / `max` 等导致替换的参数没有变化
- 你没有主动 replace 它
- 你没有删除 state

那么结果就会保持稳定。

这不是 bug，这是 Terraform 想要的行为。

---

## 8. 如何强制重新生成随机值

如果你真的想重新生成随机数，可以使用：

```sh
terraform apply -replace=random_integer.suffix
```

这会告诉 Terraform：

> 把 `random_integer.suffix` 这个资源替换掉。

替换后，它会生成一个新的随机整数。

旧命令里也可能看到 `taint`，但现代 Terraform 更推荐使用 `-replace`。

## 9. 生产环境里的注意事项

### 9.1 随机后缀适合降低冲突，但不是万能唯一性保证

`random_integer` 可以降低冲突概率，但范围越小，冲突概率越高。

例如：

```hcl
min = 1
max = 10
```

这个范围太小，多跑几次很容易撞。

本实验使用 1000 到 9999，范围是 9000 个数字，对练习够用。

生产里如果真的需要全局唯一命名，通常要结合：

- 环境名
- 项目前缀
- 区域
- 时间或版本
- 更长的随机字符串

例如：

```text
prod-payments-auckland-8x4k2p
```

比单纯一个随机整数更靠谱。

---

### 9.2 随机值进入 state，state 要保护好

随机值会进入 Terraform state。

这意味着：

- state 丢了，随机值可能重新生成
- state 被别人改了，结果可能漂移
- state 泄露时，里面的名称、ID、部分配置也会暴露

所以生产环境要使用远程 state，并做好权限控制和锁定。

例如：

- Terraform Cloud / HCP Terraform
- AWS S3 + DynamoDB lock
- Azure Storage Account backend
- GCS backend

---

### 9.3 不要把随机命名滥用成架构设计

随机后缀可以解决命名冲突，但它不能替代清晰的命名规范。

糟糕命名：

```text
app-8372
app-2194
app-9001
```

看起来都差不多，排障时非常痛苦。

更好的命名：

```text
dev-terraform-lab22-artifact-8372
stg-payment-api-artifact-2194
prod-platform-runbook-9001
```

随机值应该是后缀，不应该成为名字里唯一有信息量的部分。

## 10. 常见问题排障

### 问题 1：为什么随机值没有变化？

这是正常行为。

因为随机值已经保存在 Terraform state 里了。

想重新生成可以执行：

```sh
terraform apply -replace=random_integer.suffix
```

---

### 问题 2：为什么删除 state 后随机值变了？

因为 state 是 Terraform 记住随机值的地方。

state 没了，Terraform 会认为这个随机资源不存在，于是重新创建并生成新值。

---

### 问题 3：名称还是冲突怎么办？

可以考虑：

- 扩大随机范围
- 使用 `random_string` 生成更长后缀
- 加上环境名、项目名、区域名
- 检查是否多个环境共用同一命名空间

---

### 问题 4：生产中能不能用随机后缀？

可以，但要谨慎。

适合：

- 临时环境
- 测试资源
- preview environment
- 需要避免命名冲突的非核心资源

不适合随便用于：

- 需要长期稳定识别的核心生产资源
- 对外暴露且需要可读性的资源名
- 强依赖固定名称的集成场景

---

## 11. 清理方式

执行：

```sh
terraform destroy -auto-approve
rm -f tfplan
rm -rf output .terraform .terraform.lock.hcl
```

含义：

| 命令 | 作用 |
|---|---|
| `terraform destroy -auto-approve` | 删除 Terraform 管理的资源 |
| `rm -f tfplan` | 删除保存的 plan 文件 |
| `rm -rf output` | 删除本地实验输出目录 |
| `rm -rf .terraform .terraform.lock.hcl` | 删除初始化缓存和 lock file |

---

## 12. 这一课应该记住什么

这节课最重要的不是“怎么生成随机数”，而是下面这几句话：

1. `random_integer` 会生成指定范围内的随机整数。
2. 随机结果会进入 Terraform state。
3. 后续 apply 不会无故改变随机值。
4. 想重新生成，要 replace、taint 或改变触发替换的参数。
5. 生产中随机后缀要配合清晰命名规范，不要拿随机数当架构设计。

一句话总结：

> Terraform 的 random 不是每次都摇骰子，而是创建资源时摇一次，然后把结果记进 state。
