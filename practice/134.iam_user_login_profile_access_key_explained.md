# Terraform 讲解版 134：IAM 用户、登录配置与访问密钥

## 1. 本节主题一句话

本节讲的是：**用 Terraform 管理 AWS IAM User，并为这个用户配置 Console 登录密码与 Programmatic Access Key，同时正确处理敏感输出。**

对应原始课程主题：

> IAM 用户、登录配置与访问密钥

这一节虽然在 LocalStack 里练习，但对应真实 AWS 中非常常见的 IAM 基础操作：创建用户、设置登录配置、生成访问密钥、输出敏感值、最后销毁清理。

---

## 2. 你需要先建立的整体画面

AWS IAM 用户通常有两类访问方式：

| 访问方式 | Terraform 资源 | 用途 |
|---|---|---|
| 控制台登录 | `aws_iam_user_login_profile` | 让用户可以登录 AWS Console |
| API / CLI / SDK 访问 | `aws_iam_access_key` | 让程序、脚本、Terraform、CI/CD 使用访问密钥调用 AWS API |

所以本节的资源关系可以理解成：

```text
aws_iam_user
    ├── aws_iam_user_login_profile
    └── aws_iam_access_key
```

`aws_iam_user` 是主体，`login_profile` 和 `access_key` 都是挂在这个 IAM user 上的能力。

---

## 3. 核心资源 1：`aws_iam_user`

`aws_iam_user` 用来创建 IAM 用户。

常见写法类似：

```hcl
resource "aws_iam_user" "demo" {
  name = "terraform-demo-user"
}
```

这表示 Terraform 会声明一个 IAM 用户，并把它写入 state。

### 重点理解

IAM user 本身只是一个身份对象。它不自动拥有控制台密码，也不自动拥有 access key，更不自动拥有权限。

这点很重要：

```text
IAM User ≠ 能登录控制台
IAM User ≠ 能调用 AWS API
IAM User ≠ 有权限操作资源
```

它只是“人”或者“程序身份”的壳子。

---

## 4. 核心资源 2：`aws_iam_user_login_profile`

`aws_iam_user_login_profile` 用来给 IAM user 创建控制台登录配置。

常见写法类似：

```hcl
resource "aws_iam_user_login_profile" "demo" {
  user                    = aws_iam_user.demo.name
  password_reset_required = true
}
```

### 它解决什么问题？

它解决的是：

> 这个 IAM user 能不能登录 AWS Console？

如果只有 `aws_iam_user`，没有 `aws_iam_user_login_profile`，通常不能直接作为有密码的控制台用户使用。

### `password_reset_required`

`password_reset_required = true` 表示用户第一次登录后需要重置密码。

生产中这是很合理的做法：

```text
平台团队生成初始密码
用户第一次登录
强制用户自己改密码
避免长期使用初始密码
```

---

## 5. 核心资源 3：`aws_iam_access_key`

`aws_iam_access_key` 用来给 IAM user 创建访问密钥。

常见写法类似：

```hcl
resource "aws_iam_access_key" "demo" {
  user = aws_iam_user.demo.name
}
```

它会生成两类关键值：

```text
access key id
secret access key
```

它们通常用于：

- AWS CLI
- SDK
- CI/CD pipeline
- Terraform provider
- 自动化脚本

### 关键点

`secret access key` 只会在创建时返回一次。真实 AWS 里，如果你丢了 secret，不能重新查看，只能重新创建 access key。

所以在 Terraform 中，这类值会进入 state。也就是说：

```text
sensitive output 不等于 state 里没有敏感值
```

`sensitive = true` 只是避免 CLI 输出时直接明文展示，不代表 Terraform state 被自动加密或自动脱敏。

---

## 6. 为什么 output 要加 `sensitive = true`

本节强调把敏感输出标记为 sensitive。

例如：

```hcl
output "access_key_id" {
  value     = aws_iam_access_key.demo.id
  sensitive = true
}

output "secret_access_key" {
  value     = aws_iam_access_key.demo.secret
  sensitive = true
}
```

### 它的作用

`sensitive = true` 的作用是：

```text
避免 terraform output 直接把敏感值明文打印出来
```

例如普通 output 可能会这样：

```text
secret_access_key = "abc123..."
```

而 sensitive output 会显示成类似：

```text
secret_access_key = <sensitive>
```

### 但是要注意

`sensitive = true` 不是保险箱。

它不能解决这些问题：

- state 文件里仍可能保存敏感值
- CI/CD 日志如果手动打印变量，仍可能泄露
- 本地 `.tfstate` 如果被提交到 Git，会造成严重泄露
- 共享 backend 如果权限过大，别人仍可能读取 state

所以生产环境真正要做的是：

```text
remote backend + backend 加密 + state 访问控制 + 最小权限 + 不把 state 提交到 Git
```

## 8. LocalStack 不能替代真实 AWS 的地方

这一节必须分清楚边界。

LocalStack 不能完全替代真实 AWS IAM 的关键行为，例如：

- 真实 IAM policy 权限评估
- 真实用户登录 AWS Console
- 真实 MFA 配置
- 真实 access key 风险
- 真实 AWS Organizations / SCP 限制
- 真实 CloudTrail 审计
- 真实账号级安全基线
- 真实 IAM credential report

所以你在 LocalStack 里跑通，只能说明：

```text
Terraform 配置路径基本正确
```

不能说明：

```text
真实 AWS 安全设计已经合格
```

这个区别很重要，尤其考试和生产实践都容易考。

## 10. 你观察输出时应该看什么

这一节不是机械跑命令，真正要观察的是：

### 1. plan 阶段

看 Terraform 是否计划创建：

```text
aws_iam_user
aws_iam_user_login_profile
aws_iam_access_key
```

如果这三个资源都在 plan 里出现，说明资源关系基本建起来了。

### 2. apply 阶段

看资源是否按依赖顺序创建。

逻辑上应该是：

```text
先创建 IAM user
再创建 login profile / access key
```

Terraform 会根据引用关系自动推导依赖。

### 3. output 阶段

看敏感值是否被隐藏。

如果 secret 明文出现在终端，就说明 output 设计不合格。

### 4. destroy 阶段

看资源是否可以完整删除。

IAM 类资源如果依赖关系没处理好，真实 AWS 里经常出现删除失败，例如用户还挂着 access key、policy、login profile 等。

## 11. 生产环境里的正确姿势

真实 AWS 里，不建议随便长期创建 IAM user + access key。

更推荐的现代做法是：

### 人登录 AWS Console

优先使用：

```text
IAM Identity Center
SSO
MFA
短期会话
```

而不是给每个人长期 IAM user 密码。

### 程序访问 AWS API

优先使用：

```text
IAM Role
STS AssumeRole
OIDC federation
短期临时凭证
```

例如 GitHub Actions 部署 AWS，生产里更推荐 OIDC AssumeRole，而不是把 access key secret 存在 GitHub Secrets 里长期使用。

### IAM User 的适用场景

IAM user 仍可能存在，但应该更谨慎：

- 老系统兼容
- 第三方工具不支持 OIDC / AssumeRole
- 过渡期迁移
- 非交互式长期凭证场景

但要配合：

```text
最小权限
定期轮换
MFA
CloudTrail 审计
Access Analyzer
Credential Report
禁止 root access key
```

## 12. Terraform state 的安全重点

这节最容易被忽略的坑是 state。

当你创建 `aws_iam_access_key` 时，secret 可能会进入 Terraform state。

所以这些事情一定不要做：

```text
不要把 terraform.tfstate 提交到 Git
不要把 state 发给别人
不要在公共 CI 日志里打印 state
不要把 state 放在无权限控制的共享目录
```

生产中更稳妥的是：

```text
S3 backend + DynamoDB lock + KMS encryption
HCP Terraform remote state
严格限制 state 读取权限
```

真正的安全边界不在 `sensitive = true`，而在 state backend 的访问控制。

## 13. 常见错误与排障

### 错误 1：Provider 连到了真实 AWS

本实验应该走 LocalStack。

检查 provider 是否配置了 endpoint：

```text
http://localhost:4566
```

如果误连真实 AWS，可能会真的创建 IAM 用户，风险很大。

---

### 错误 2：output 泄露了 secret

检查 output 有没有：

```hcl
sensitive = true
```

但也别自欺欺人：这只是不在终端直接显示，不代表 state 没有敏感值。

---

### 错误 3：destroy 删除失败

真实 AWS 中 IAM user 如果还有关联资源，可能删除失败。

排查方向：

- access key 是否还存在
- login profile 是否还存在
- inline policy 是否还存在
- managed policy attachment 是否还存在
- group membership 是否还存在

Terraform 管理的资源应该都写进配置，避免“半手工、半 Terraform”的漂移。

---

### 错误 4：Access key 创建后找不到 secret

真实 AWS 中 secret 只在创建时可见。

Terraform 创建后，secret 通常保存在 state 中。丢失后不应试图“找回”，而应该轮换 key。

---

## 14. 考试记忆点

这一节可以压缩成几个考点：

1. `aws_iam_user` 创建 IAM 用户。
2. `aws_iam_user_login_profile` 让用户可以控制台登录。
3. `aws_iam_access_key` 给用户创建 API / CLI 访问密钥。
4. `sensitive = true` 可以隐藏 CLI 输出，但不能从 state 中移除敏感值。
5. LocalStack 可以验证 Terraform 资源路径，但不能验证真实 AWS IAM 权限模型。
6. 生产中更推荐 Role / STS / OIDC / SSO，而不是长期 IAM user access key。

---

## 15. 面试表达版本

如果面试官问你：“你怎么理解 Terraform 管理 IAM user 和 access key？”

可以这样回答：

> Terraform 可以声明 IAM user、login profile 和 access key，并通过资源引用建立依赖关系。login profile 面向 Console 登录，access key 面向 CLI、SDK 或自动化访问。敏感输出应该使用 `sensitive = true`，但这只影响 CLI 展示，不代表 state 中没有敏感值。因此生产环境要重点保护 remote state，并尽量使用 IAM Role、STS AssumeRole、OIDC 或 IAM Identity Center 这类短期凭证方案，减少长期 access key 的风险。

---

## 16. 本节总结

本节不是单纯学习三个 Terraform 资源，而是在训练一个更重要的安全意识：

```text
身份资源可以用 Terraform 管
但凭证安全不能只靠 Terraform output sensitive
```

LocalStack 让你低成本练习资源生命周期。真实 AWS 里，IAM user、login profile、access key 都属于高风险资源，必须结合最小权限、短期凭证、审计和 state 安全一起设计。
