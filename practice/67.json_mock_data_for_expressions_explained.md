# Terraform 实操训练 67 讲解：JSON mock 数据与 for 表达式

## 1. 这一节到底在学什么？

Lab 67 学的是：**JSON mock 数据与 Terraform for 表达式的结合**。

前面你已经学过：

- Lab 29：`jsonencode()` / `jsondecode()`
- Lab 57：`for_each` 基础
- Lab 59：`for` 表达式基础
- Lab 61：`for` 表达式进阶
- Lab 63：嵌套 `for` 表达式
- Lab 64：`flatten()` 与 `distinct()`
- Lab 65-66：`templatefile()` 和模板循环

这一节的重点是把外部 JSON mock 数据读进 Terraform，然后用 `for` 表达式做数据整形。

一句话理解：

> JSON mock 数据负责模拟外部系统输入，for 表达式负责把这些输入转换成 Terraform 更适合使用的数据结构。

---

## 2. 什么是 JSON mock 数据？

mock 数据就是“模拟数据”。

在真实系统里，Terraform 可能会从这些地方获得数据：

- 服务目录
- CMDB
- API 返回结果
- 云资产清单
- 用户权限系统
- 成本中心系统
- 应用配置平台

但在训练或本地实验里，你不一定真的接入这些系统。  
所以可以用一个 JSON 文件模拟外部数据。

例如：

```json
{
  "services": [
    {
      "name": "api",
      "environment": "dev",
      "port": 8080,
      "enabled": true
    },
    {
      "name": "worker",
      "environment": "dev",
      "port": 9090,
      "enabled": false
    },
    {
      "name": "web",
      "environment": "prod",
      "port": 3000,
      "enabled": true
    }
  ]
}
```

这就是一份 mock 服务目录。

Terraform 可以读取它，再转换成资源创建、输出、模板生成所需的数据。

---

## 3. JSON 和 CSV 的区别

Lab 60、62 学过 CSV。  
CSV 适合扁平表格：

```csv
name,port,environment
api,8080,dev
worker,9090,dev
```

JSON 更适合复杂结构：

```json
{
  "services": [
    {
      "name": "api",
      "ports": [8080, 9090],
      "tags": {
        "owner": "platform"
      }
    }
  ]
}
```

简单对比：

| 格式 | 适合场景 |
|---|---|
| CSV | 简单表格、扁平清单 |
| JSON | 嵌套对象、复杂配置、API mock |

如果数据有嵌套结构，JSON 通常比 CSV 更自然。

---

## 4. Terraform 如何读取 JSON？

常见组合是：

```hcl
jsondecode(file("${path.module}/mock/services.json"))
```

拆开看：

```hcl
file(...)
```

读取文件内容，返回字符串。

```hcl
jsondecode(...)
```

把 JSON 字符串解析成 Terraform 数据结构。

完整示例：

```hcl
locals {
  mock_data = jsondecode(file("${path.module}/mock/services.json"))
}
```

如果 JSON 是：

```json
{
  "services": [
    {
      "name": "api",
      "port": 8080
    }
  ]
}
```

那么 Terraform 中可以访问：

```hcl
local.mock_data.services
local.mock_data.services[0].name
local.mock_data.services[0].port
```

---

## 5. JSON 类型会映射成 Terraform 类型

JSON 解析后会映射成 Terraform 的类型。

| JSON | Terraform |
|---|---|
| object | object / map-like object |
| array | tuple / list-like collection |
| string | string |
| number | number |
| boolean | bool |
| null | null |

例如 JSON：

```json
{
  "name": "api",
  "port": 8080,
  "enabled": true,
  "tags": {
    "owner": "platform"
  }
}
```

Terraform 可以像这样访问：

```hcl
local.service.name
local.service.port
local.service.enabled
local.service.tags.owner
```

和 CSV 不同，JSON 里的 number 和 bool 通常能保留为 number / bool。  
CSV 字段通常会变成 string。

## 6. 为什么要配合 for 表达式？

`jsondecode()` 只是把 JSON 读进来。  
但读进来的结构不一定适合直接给资源用。

比如 JSON 是 list：

```hcl
local.mock_data.services
```

类似：

```hcl
[
  {
    name    = "api"
    enabled = true
  },
  {
    name    = "worker"
    enabled = false
  }
]
```

如果要创建多个资源，更推荐转成 map：

```hcl
locals {
  services_by_name = {
    for svc in local.mock_data.services :
    svc.name => svc
  }
}
```

这样可以用于：

```hcl
for_each = local.services_by_name
```

资源地址就会是：

```text
local_file.service["api"]
local_file.service["worker"]
```

比 `count.index` 稳定。

---

## 7. 基础链路：JSON → for → map

这是本节最重要的链路：

```hcl
locals {
  mock_data = jsondecode(file("${path.module}/mock/services.json"))

  services_by_name = {
    for svc in local.mock_data.services :
    svc.name => svc
  }
}
```

含义：

1. 读取 JSON 文件。
2. 解析成 Terraform 对象。
3. 遍历 `services` 数组。
4. 用 `svc.name` 做 key。
5. 每个服务对象作为 value。
6. 得到 `map(object)`。

输出结构类似：

```hcl
{
  api = {
    name    = "api"
    port    = 8080
    enabled = true
  }

  worker = {
    name    = "worker"
    port    = 9090
    enabled = false
  }
}
```

这就是 Terraform 资源建模常用结构。

---

## 8. 过滤 enabled 服务

mock JSON 里经常有开关字段：

```json
{
  "name": "api",
  "enabled": true
}
```

可以用 `for` 表达式过滤：

```hcl
locals {
  enabled_services_by_name = {
    for svc in local.mock_data.services :
    svc.name => svc
    if svc.enabled
  }
}
```

这表示：

> 只保留 enabled = true 的服务。

然后资源：

```hcl
resource "local_file" "service" {
  for_each = local.enabled_services_by_name

  filename = "${path.module}/output/${each.key}.json"
  content  = jsonencode(each.value)
}
```

这样 disabled 的服务不会创建资源。

---

## 9. 提取字段列表

如果只是想提取所有服务名：

```hcl
locals {
  service_names = [
    for svc in local.mock_data.services : svc.name
  ]
}
```

结果：

```hcl
["api", "worker", "web"]
```

提取所有环境：

```hcl
locals {
  environments = distinct([
    for svc in local.mock_data.services : svc.environment
  ])
}
```

结果可能是：

```hcl
["dev", "prod"]
```

如果需要稳定输出，可以：

```hcl
sort(distinct([
  for svc in local.mock_data.services : svc.environment
]))
```

这就把 Lab 64 的 `distinct()` / `sort()` 串起来了。

## 10. 处理嵌套 JSON

JSON 的优势是可以自然表达嵌套结构。

例如：

```json
{
  "services": [
    {
      "name": "api",
      "ports": [8080, 9090]
    },
    {
      "name": "web",
      "ports": [3000]
    }
  ]
}
```

如果想展开所有 service-port 组合：

```hcl
locals {
  service_ports = flatten([
    for svc in local.mock_data.services : [
      for port in svc.ports : {
        service = svc.name
        port    = port
      }
    ]
  ])
}
```

结果：

```hcl
[
  {
    service = "api"
    port    = 8080
  },
  {
    service = "api"
    port    = 9090
  },
  {
    service = "web"
    port    = 3000
  }
]
```

这就是 Lab 63 的嵌套 `for` 表达式在 JSON mock 数据里的应用。

---

## 11. 展开后给 for_each 使用

如果展开后的组合要创建资源，继续转 map：

```hcl
locals {
  service_ports_by_key = {
    for item in local.service_ports :
    "${item.service}-${item.port}" => item
  }
}
```

然后：

```hcl
resource "local_file" "service_port" {
  for_each = local.service_ports_by_key

  filename = "${path.module}/output/${each.key}.json"
  content  = jsonencode(each.value)
}
```

资源地址：

```text
local_file.service_port["api-8080"]
local_file.service_port["api-9090"]
local_file.service_port["web-3000"]
```

这比用 `[0]`、`[1]` 稳定得多。

## 12. JSON mock 数据为什么适合训练？

因为它可以模拟真实 API 响应。

例如未来你做平台项目，可能会接入：

- service catalog API
- cloud inventory API
- cost management API
- GitHub API
- Kubernetes API 输出
- security scanner 输出
- CMDB 数据

这些返回结果大多是 JSON。

在没有真实 API 的情况下，用 JSON mock 文件可以先训练：

- 数据读取
- 字段访问
- 过滤
- 转换
- 展开
- 去重
- 生成 map
- 创建资源
- 输出结果
- 测试断言

这对 Terraform 模块开发很有价值。

## 16. 为什么 key 稳定很重要？

如果你把 JSON 里的 `name` 作为 key：

```hcl
{
  for svc in local.mock_data.services :
  svc.name => svc
}
```

那么资源地址可能是：

```text
local_file.service["api"]
```

如果 JSON 中把：

```json
"name": "api"
```

改成：

```json
"name": "backend-api"
```

Terraform 会认为：

```text
local_file.service["api"] 要删除
local_file.service["backend-api"] 要创建
```

所以 mock 数据里的 key 字段不是随便取的。  
它会影响 Terraform state 地址。

这就是数据建模和状态管理的关系。

## 17. JSON 数据质量检查

mock 数据也要有质量要求，否则你练出来的 Terraform 逻辑不可靠。

常见问题：

- `name` 重复
- `name` 为空
- 缺少必需字段
- 字段类型不一致
- `enabled` 有的行是 bool，有的行是 string
- 嵌套字段缺失
- ports 里混入字符串
- 结构和代码预期不一致

例如：

```json
{
  "name": "api",
  "enabled": "true"
}
```

这里 `"true"` 是字符串，不是 bool。

Terraform 判断：

```hcl
if svc.enabled
```

就可能出问题。

所以 JSON mock 要尽量贴近真实 schema。

---

## 18. 如何验证 key 唯一？

可以先提取 name list：

```hcl
locals {
  service_names = [
    for svc in local.mock_data.services : svc.name
  ]

  unique_service_names = distinct(local.service_names)
}
```

然后判断：

```hcl
length(local.service_names) == length(local.unique_service_names)
```

如果不相等，说明有重复 name。

在 `terraform test` 里可以写断言：

```hcl
assert {
  condition     = length(local.service_names) == length(local.unique_service_names)
  error_message = "service names must be unique."
}
```

这是非常实用的数据质量检查思路。

## 23. JSON vs templatefile

Lab 65-66 学了模板文件。  
这里也要分清楚：

- `jsondecode()`：把 JSON 输入变成 Terraform 数据。
- `jsonencode()`：把 Terraform 数据输出成 JSON。
- `templatefile()`：把 Terraform 数据渲染成文本模板。

例如：

```hcl
content = jsonencode(each.value)
```

比手写 JSON 模板更安全。

如果目标就是 JSON 文件，优先使用 `jsonencode()`。  
如果目标是 shell、conf、INI、Nginx 配置，再考虑 `templatefile()`。

## 24. 常见坑

### 坑 1：JSON 文件路径错误

推荐：

```hcl
file("${path.module}/mock/services.json")
```

不要依赖当前执行目录。

---

### 坑 2：JSON 格式不合法

JSON 不允许：

- 多余逗号
- 单引号字符串
- 注释
- 未转义换行
- 键名不加双引号

如果 JSON 无法解析，`jsondecode()` 会失败。

---

### 坑 3：字段名和代码不一致

JSON 里是：

```json
"service_name": "api"
```

Terraform 却访问：

```hcl
svc.name
```

会失败。

字段名要对齐。

---

### 坑 4：重复 key

把 list 转 map 时：

```hcl
svc.name => svc
```

`svc.name` 必须唯一。

---

### 坑 5：类型不一致

例如有的 `enabled` 是：

```json
true
```

有的是：

```json
"true"
```

这会让表达式很难维护。

---

### 坑 6：过度依赖 mock

mock 数据只是训练和测试输入。  
真实系统接入时，要重新验证 schema、字段、错误情况和边界值。

## 27. 面试/考试记忆版

可以这样记：

- JSON mock 数据用于模拟外部系统输入。
- Terraform 常用 `jsondecode(file(...))` 读取 JSON 文件。
- JSON object/array 会变成 Terraform 可访问的数据结构。
- JSON 的 number/bool 通常比 CSV 更保真。
- 用 `for` 表达式提取字段、过滤、转 map。
- 如果要 `for_each`，通常把 list(object) 转成 map(object)。
- key 字段必须唯一且稳定。
- 嵌套 JSON 可用嵌套 for + flatten 展开。
- 目标是 JSON 输出时优先用 `jsonencode()`。
- mock 数据要配合 terraform test 验证。

一句话总结：

> JSON mock 数据让 Terraform 可以模拟真实外部输入；for 表达式负责把这些输入转换成稳定、可测试、适合资源管理的数据模型。

## 28. 本 Lab 的价值

Lab 67 的价值在于，它把 Terraform 从“手写变量”推进到“外部数据驱动”。

你不再只是写：

```hcl
variable "services" { ... }
```

而是开始处理：

```text
外部 JSON 数据
-> jsondecode
-> for 表达式整形
-> for_each 创建资源
-> terraform test 验证
```

这很接近真实平台工程和 SRE 自动化场景。

真正的重点不是 JSON 文件本身，而是：

> 你能不能把外部结构化数据变成稳定、可审、可测试、不会乱动 state 的 Terraform 模型。
