# Terraform 实操训练 138：IAM Role Policy Attachment 讲解版

## 1. 本节定位

本节主题是 **IAM Role Policy Attachment**，重点训练 Terraform 中三个对象之间的关系：

- `aws_iam_role`：创建一个 IAM Role，也就是“谁可以被假设 / 谁拿到权限”。
- `aws_iam_policy`：创建一份独立 IAM Policy，也就是“允许做什么”。
- `aws_iam_role_policy_attachment`：把 Policy 挂到 Role 上，也就是“把权限交给这个角色”。

一句话理解：

> Role 是身份容器，Policy 是权限说明书，Attachment 是把说明书贴到身份上的动作。

本实验在 **LocalStack** 中完成，不访问真实 AWS。它适合训练 Terraform 资源声明、依赖关系、状态变化和销毁流程，但不能完全代表真实 AWS 的 IAM 权限评估结果。

---

## 2. 为什么需要 Attachment

在 AWS IAM 里，Role 和 Policy 是可以分开管理的。

这样做的好处是：

1. **权限可以复用**  
   一份 Policy 可以挂到多个 Role 上。

2. **职责更清晰**  
   Role 负责“身份”，Policy 负责“权限”，Attachment 负责“绑定关系”。

3. **更适合 Terraform 管理**  
   Terraform 可以分别追踪 Role、Policy、Attachment 的生命周期。

4. **便于最小权限治理**  
   团队可以审查 Policy 内容，而不是把权限散落在多个 Role 的 inline policy 里。

---

## 3. 核心资源关系

本节通常会出现类似下面的结构：

```hcl
resource "aws_iam_role" "example" {
  name = "example-role"

  assume_role_policy = jsonencode({
    Version = "2012-10-17"
    Statement = [
      {
        Effect = "Allow"
        Principal = {
          Service = "ec2.amazonaws.com"
        }
        Action = "sts:AssumeRole"
      }
    ]
  })
}

resource "aws_iam_policy" "example" {
  name        = "example-policy"
  description = "Example managed policy"

  policy = jsonencode({
    Version = "2012-10-17"
    Statement = [
      {
        Effect = "Allow"
        Action = [
          "s3:ListBucket"
        ]
        Resource = "*"
      }
    ]
  })
}

resource "aws_iam_role_policy_attachment" "example" {
  role       = aws_iam_role.example.name
  policy_arn = aws_iam_policy.example.arn
}
```

这段代码里最关键的是最后这个资源：

```hcl
resource "aws_iam_role_policy_attachment" "example" {
  role       = aws_iam_role.example.name
  policy_arn = aws_iam_policy.example.arn
}
```

它表达的是：

> 把 `aws_iam_policy.example` 这份 managed policy，挂载到 `aws_iam_role.example` 这个 role 上。

---

## 4. Role、Policy、Attachment 三者怎么记

可以用一个很土但很稳的比喻：

| Terraform 资源 | 类比 | 作用 |
|---|---|---|
| `aws_iam_role` | 员工工牌 | 表示一个可被使用的身份 |
| `aws_iam_policy` | 权限清单 | 写明这个身份能做什么 |
| `aws_iam_role_policy_attachment` | 贴权限条 | 把权限清单绑定到工牌上 |

不要把 Policy 写好了就当成 Role 已经有权限了。

**Policy 只是存在，Attachment 才是绑定。**

---

## 5. Managed Policy vs Inline Policy

本节重点是 `aws_iam_role_policy_attachment`，它绑定的是 **managed policy**。

常见两种写法：

### 5.1 Managed Policy + Attachment

```hcl
resource "aws_iam_policy" "example" {
  policy = jsonencode({...})
}

resource "aws_iam_role_policy_attachment" "example" {
  role       = aws_iam_role.example.name
  policy_arn = aws_iam_policy.example.arn
}
```

特点：

- Policy 是独立资源。
- 可以复用。
- 可以单独审计。
- 适合平台团队和生产环境治理。

### 5.2 Inline Policy

```hcl
resource "aws_iam_role_policy" "example" {
  role   = aws_iam_role.example.id
  policy = jsonencode({...})
}
```

特点：

- Policy 直接嵌在 Role 下面。
- 复用性差。
- 更适合非常小、非常专用的权限。

考试或面试里，如果看到 `aws_iam_role_policy_attachment`，要立刻想到：

> 这是 managed policy attachment，不是 inline policy。

---

## 6. Terraform 依赖关系

`aws_iam_role_policy_attachment` 同时引用了：

```hcl
role       = aws_iam_role.example.name
policy_arn = aws_iam_policy.example.arn
```

所以 Terraform 会自动建立依赖关系：

1. 先创建 Role。
2. 再创建 Policy。
3. 最后创建 Attachment。

销毁时顺序反过来：

1. 先删除 Attachment。
2. 再删除 Policy。
3. 再删除 Role。

这也是 Terraform 状态管理的价值：它不是简单执行几条命令，而是维护资源之间的图关系。

## 7. LocalStack 实验边界

本节实验通过 LocalStack 模拟 AWS API：

- endpoint：`http://localhost:4566`
- 凭证：`test/test`
- 不访问真实 AWS 账号
- 不需要真实 access key
- 不会产生真实云账单

但是要注意：

LocalStack 可以帮助你验证 Terraform 代码结构、资源依赖和生命周期闭环；但它不能完全验证真实 AWS IAM 的全部行为，例如：

- 真实跨账号 AssumeRole 行为
- IAM policy 最终权限评估细节
- AWS 服务之间真实调用权限
- 组织 SCP、permission boundary 等复杂治理条件

所以这节练的是 Terraform 写法和资源关系，不是完整 AWS IAM 安全审计。

## 9. 观察重点

运行时重点看这几类输出：

### 9.1 Plan 阶段

你应该能看到 Terraform 计划创建：

- IAM Role
- IAM Policy
- IAM Role Policy Attachment

重点不是资源数量，而是确认 Attachment 的 `role` 和 `policy_arn` 引用的是前面创建的对象。

### 9.2 Apply 阶段

Apply 成功说明 LocalStack 接受了这些 IAM API 调用。

这代表 Terraform 资源生命周期跑通，但不等于真实 AWS 权限一定完全符合生产安全要求。

### 9.3 Destroy 阶段

销毁时应该能看到 Attachment 被先删除。否则 Role 或 Policy 可能因为存在绑定关系而无法正常清理。

## 10. 常见错误与排障

### 10.1 Role 名称写错

错误表现：Attachment 找不到对应 Role。

检查：

```hcl
role = aws_iam_role.example.name
```

不要手写字符串，优先引用资源属性。

---

### 10.2 Policy ARN 写错

错误表现：Attachment 找不到对应 Policy。

检查：

```hcl
policy_arn = aws_iam_policy.example.arn
```

不要把 `name` 当成 `arn` 用。

---

### 10.3 Assume Role Policy 写错

Role 的 `assume_role_policy` 是信任策略，不是权限策略。

它回答的问题是：

> 谁可以 assume 这个 role？

而 IAM Policy 回答的问题是：

> assume 之后可以做什么？

这两个不是一回事。

---

### 10.4 把 Attachment 当成 Policy

`aws_iam_role_policy_attachment` 本身不定义权限内容。

它只做绑定：

```hcl
role       = ...
policy_arn = ...
```

真正定义权限的是 `aws_iam_policy` 里的 `policy` JSON。

---

## 11. 生产环境迁移思路

从 LocalStack 迁移到真实 AWS 时，至少要改这些地方：

1. 移除 LocalStack endpoint 配置。
2. 使用真实 AWS 凭证来源，例如 SSO、profile、OIDC、assume role。
3. 明确 region。
4. 收紧 IAM Policy 的 `Action` 和 `Resource`。
5. 避免长期使用 `Resource = "*"`。
6. 给 Policy 和 Role 加上必要标签。
7. 在 CI/CD 中加入 `terraform plan` 审查和 policy check。

生产环境里，IAM 是高风险区域。能跑通只是第一步，权限最小化才是重点。

---

## 12. 考试与面试记忆点

看到 `aws_iam_role_policy_attachment`，记住四句话：

1. 它用于把 managed policy 绑定到 role。
2. 它不负责定义权限内容。
3. `role` 通常引用 role name。
4. `policy_arn` 必须引用 policy ARN。

最容易混淆的是：

- `aws_iam_policy`：定义 managed policy。
- `aws_iam_role_policy`：定义 role inline policy。
- `aws_iam_role_policy_attachment`：把 managed policy 挂到 role。

---

## 13. 本节总结

本节的核心不是“背一个 Terraform 资源名”，而是理解 IAM 权限建模的拆分方式：

```text
Role       = 身份
Policy     = 权限
Attachment = 绑定关系
```

Terraform 的价值在于把这三者声明成可追踪、可审查、可销毁的资源图。

在真实生产环境里，这个模式非常常见。平台团队通常会把标准 Policy 做成模块或公共组件，再通过 Attachment 挂到不同服务 Role 上。这样既能复用权限模型，也方便做合规审查。
