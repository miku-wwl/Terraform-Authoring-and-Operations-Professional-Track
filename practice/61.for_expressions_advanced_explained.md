# Terraform 实操训练 61 讲解：for 表达式进阶

## 1. 这一节到底在学什么？

Lab 61 学的是 Terraform 里的 **for 表达式进阶**。

上一节 Lab 59 已经讲了 for 表达式基础：

- list 转 list
- list 转 map
- map 转 list
- map 转 map
- 使用 `if` 过滤
- 为 `for_each` 准备数据

这一节进一步强调：  
`for` 表达式不只是简单遍历，而是 Terraform 模块里的**数据整形工具**。

一句话理解：

> 进阶 for 表达式的核心价值，是把复杂输入加工成更适合资源、模块和 state 管理的数据结构。

---

## 2. 为什么需要进阶 for 表达式？

真实 Terraform 项目里的输入数据通常不会刚好长成资源想要的样子。

用户可能传：

```hcl
services = [
  {
    name    = "api"
    enabled = true
    ports   = [8080, 9090]
  },
  {
    name    = "worker"
    enabled = false
    ports   = [7070]
  }
]
```

但资源可能需要：

```hcl
{
  api = {
    enabled = true
    ports   = [8080, 9090]
  }
}
```

或者还需要把端口展开成：

```hcl
[
  {
    service = "api"
    port    = 8080
  },
  {
    service = "api"
    port    = 9090
  }
]
```

这些都要靠进阶 for 表达式完成。

---

## 3. 基础复习：for 表达式的两种输出

### 3.1 生成 list

```hcl
locals {
  names = ["api", "worker", "web"]

  upper_names = [
    for name in local.names : upper(name)
  ]
}
```

结果：

```hcl
["API", "WORKER", "WEB"]
```

---

### 3.2 生成 map

```hcl
locals {
  names = ["api", "worker", "web"]

  names_map = {
    for name in local.names : name => upper(name)
  }
}
```

结果：

```hcl
{
  api    = "API"
  worker = "WORKER"
  web    = "WEB"
}
```

生成 map 的关键是：

```hcl
key => value
```

---

## 4. 进阶点一：过滤元素

for 表达式可以用 `if` 过滤元素。

示例：

```hcl
locals {
  services = [
    {
      name    = "api"
      enabled = true
    },
    {
      name    = "worker"
      enabled = false
    },
    {
      name    = "web"
      enabled = true
    }
  ]

  enabled_services = [
    for svc in local.services : svc
    if svc.enabled
  ]
}
```

结果：

```hcl
[
  {
    name    = "api"
    enabled = true
  },
  {
    name    = "web"
    enabled = true
  }
]
```

这在生产里很常见：  
用户传入所有服务，但只对 enabled 的服务创建资源。

---

## 5. 进阶点二：过滤后生成 map

过滤 list 后，通常还会转成 map 给 `for_each`。

```hcl
locals {
  enabled_services_by_name = {
    for svc in local.services : svc.name => svc
    if svc.enabled
  }
}
```

结果：

```hcl
{
  api = {
    name    = "api"
    enabled = true
  }

  web = {
    name    = "web"
    enabled = true
  }
}
```

然后资源里：

```hcl
resource "local_file" "service" {
  for_each = local.enabled_services_by_name

  filename = "${path.module}/output/${each.key}.txt"
  content  = "enabled=${each.value.enabled}"
}
```

这就是生产模块常用套路：

```text
输入 list(object)
-> for 过滤
-> 转成 map(object)
-> for_each 创建资源
```

## 6. 进阶点三：从嵌套结构中提取字段

假设有：

```hcl
locals {
  services = {
    api = {
      ports = [
        {
          name = "http"
          port = 8080
        },
        {
          name = "metrics"
          port = 9090
        }
      ]
    }
  }
}
```

提取 api 的所有端口号：

```hcl
locals {
  api_ports = [
    for p in local.services["api"].ports : p.port
  ]
}
```

结果：

```hcl
[8080, 9090]
```

这就是 Lab 54 的读取嵌套值和 Lab 59 的 for 表达式组合起来。

## 7. 进阶点四：嵌套 for 表达式

有时需要两层遍历。

例如多个服务，每个服务有多个端口：

```hcl
locals {
  services = [
    {
      name  = "api"
      ports = [8080, 9090]
    },
    {
      name  = "web"
      ports = [3000]
    }
  ]
}
```

你想得到：

```hcl
[
  {
    service = "api"
    port    = 8080
  },
  {
    service = "api"
    port    = 9090
  },
  {
    service = "web"
    port    = 3000
  }
]
```

可以写：

```hcl
locals {
  service_ports_nested = [
    for svc in local.services : [
      for port in svc.ports : {
        service = svc.name
        port    = port
      }
    ]
  ]
}
```

但这个结果是嵌套 list：

```hcl
[
  [
    { service = "api", port = 8080 },
    { service = "api", port = 9090 }
  ],
  [
    { service = "web", port = 3000 }
  ]
]
```

还需要 `flatten()`。

---

## 8. 进阶点五：`flatten()` 展平嵌套 list

继续上面的例子：

```hcl
locals {
  service_ports = flatten([
    for svc in local.services : [
      for port in svc.ports : {
        service = svc.name
        port    = port
      }
    ]
  ])
}
```

结果就是一层 list：

```hcl
[
  {
    service = "api"
    port    = 8080
  },
  {
    service = "api"
    port    = 9090
  },
  {
    service = "web"
    port    = 3000
  }
]
```

这个模式非常重要：

```hcl
flatten([
  for outer in collection : [
    for inner in outer.items : {
      ...
    }
  ]
])
```

它经常用于：

- 服务 × 端口
- 环境 × region
- 用户 × 权限
- team × repositories
- subnet × route tables
- alert group × receivers

---

## 9. 进阶点六：展开后再转 map

如果展开后的对象要用于 `for_each`，通常还要转成 map。

例如：

```hcl
locals {
  service_ports_by_key = {
    for item in local.service_ports :
    "${item.service}-${item.port}" => item
  }
}
```

结果：

```hcl
{
  "api-8080" = {
    service = "api"
    port    = 8080
  }

  "api-9090" = {
    service = "api"
    port    = 9090
  }

  "web-3000" = {
    service = "web"
    port    = 3000
  }
}
```

这样就可以：

```hcl
resource "local_file" "service_port" {
  for_each = local.service_ports_by_key

  filename = "${path.module}/output/${each.key}.txt"
  content  = "service=${each.value.service}, port=${each.value.port}"
}
```

资源地址稳定且可读：

```text
local_file.service_port["api-8080"]
local_file.service_port["api-9090"]
local_file.service_port["web-3000"]
```

## 10. 进阶点七：map 分组

有时你需要按某个字段分组。

例如：

```hcl
locals {
  services = [
    {
      name        = "api"
      environment = "dev"
    },
    {
      name        = "worker"
      environment = "dev"
    },
    {
      name        = "web"
      environment = "prod"
    }
  ]
}
```

想得到：

```hcl
{
  dev  = ["api", "worker"]
  prod = ["web"]
}
```

Terraform for 表达式可以通过 group by 的 `...` 语法实现分组：

```hcl
locals {
  services_by_environment = {
    for svc in local.services :
    svc.environment => svc.name...
  }
}
```

结果：

```hcl
{
  dev  = ["api", "worker"]
  prod = ["web"]
}
```

这个 `...` 很容易被忽略，但很有用。

---

## 11. 分组语法什么时候用？

当你生成 map 时，如果多个元素会产生同一个 key，默认会冲突。

例如：

```hcl
{
  for svc in local.services :
  svc.environment => svc.name
}
```

如果两个服务都是 `dev`，key 都是 `dev`，就会冲突。

加上 `...`：

```hcl
svc.environment => svc.name...
```

意思是：

> 如果多个元素属于同一个 key，就把 value 收集成 list。

这就是分组。

---

## 12. 进阶点八：过滤空值

生产里经常要过滤掉 null 或空字符串。

例如 tag：

```hcl
variable "tags" {
  type = map(string)
}
```

过滤空 tag：

```hcl
locals {
  clean_tags = {
    for k, v in var.tags :
    k => trimspace(v)
    if v != null && trimspace(v) != ""
  }
}
```

这样可以避免把空 tag 传给 provider。

这个写法很实用：

```hcl
if v != null && trimspace(v) != ""
```

---

## 13. 进阶点九：结合 `try()` 和 `can()`

复杂对象里有些字段可能不存在，或者类型不一定正确。

例如：

```hcl
locals {
  services = [
    {
      name = "api"
      port = "8080"
    },
    {
      name = "worker"
    }
  ]
}
```

安全提取端口：

```hcl
locals {
  service_ports = {
    for svc in local.services :
    svc.name => try(tonumber(svc.port), null)
  }
}
```

结果可能是：

```hcl
{
  api    = 8080
  worker = null
}
```

也可以过滤掉没有合法 port 的服务：

```hcl
locals {
  services_with_port = {
    for svc in local.services :
    svc.name => svc
    if can(tonumber(svc.port))
  }
}
```

`try()` 和 `can()` 是复杂数据整形里常见工具。

## 14. 进阶点十：控制输出类型

for 表达式输出类型要有意识控制。

生成 list：

```hcl
[for svc in var.services : svc.name]
```

生成 set：

```hcl
toset([for svc in var.services : svc.environment])
```

生成 map：

```hcl
{ for svc in var.services : svc.name => svc }
```

生成排序后的 list：

```hcl
sort([for svc in var.services : svc.name])
```

生产里输出结构要稳定。  
尤其是 test 断言、CI diff、模块 output，都不要让类型和顺序飘来飘去。

## 18. Terraform test 在这里有什么价值？

Lab 61 的验证报告显示：

```text
terraform test 返回 1 passed, 0 failed
```

对进阶 for 表达式来说，测试非常重要。

因为 for 表达式最容易出现的问题不是“语法完全不会”，而是：

- 过滤条件写反
- map key 不唯一
- flatten 少了一层
- 输出类型不是预期
- 嵌套字段路径写错
- 分组结果不符合预期
- 数字字符串没转换
- plan/apply 阶段值不一致

测试能把这些数据转换逻辑固定下来。

## 19. 常见坑

### 坑 1：忘记 `flatten()`

嵌套 for 常常产生 list 里套 list。

如果资源需要一层 list，却得到嵌套 list，就会出问题。

---

### 坑 2：map key 重复

例如：

```hcl
{
  for svc in var.services : svc.environment => svc
}
```

如果多个服务 environment 相同，就会 key 冲突。

要么换唯一 key，要么用分组语法：

```hcl
svc.environment => svc...
```

---

### 坑 3：复杂表达式一行写到底

虽然 Terraform 支持，但人类读起来痛苦。

建议拆 locals：

```hcl
locals {
  enabled_services = ...
  service_ports    = ...
  service_ports_by_key = ...
}
```

---

### 坑 4：把业务不稳定字段当 key

如果 key 是版本号、日期、临时名称，后续变化会导致资源地址变化。

for_each 的 key 要稳定。

---

### 坑 5：过滤条件太隐蔽

例如：

```hcl
if svc.enabled && svc.environment == var.environment && length(svc.ports) > 0
```

这种条件最好拆成清晰的 locals 或写注释。

## 22. SRE / Platform Engineering 视角

从平台工程角度看，进阶 for 表达式是模块内部的数据管道。

一个成熟模块常见流程是：

```text
用户输入
-> locals 中清洗数据
-> 过滤 disabled 项
-> 展开嵌套结构
-> 转成 map
-> for_each 创建资源
-> output 输出摘要
-> terraform test 验证结果
```

这套流程里，for 表达式承担的是“清洗、转换、重组”的角色。

如果你未来做 AKS/EKS、服务 catalog、FinOps、告警平台、IAM 权限矩阵，都会大量用到这种思路。

## 23. 什么时候该停下来重构？

如果 for 表达式已经变得很难读，说明该重构了。

可以考虑：

1. 拆成多个 locals。
2. 给中间变量起清楚名字。
3. 把复杂输入改成更适合的类型。
4. 用 map(object) 替代 list(object)。
5. 把重复逻辑封装到模块。
6. 写 terraform test 固定结果。
7. 在 README 里说明数据结构。

Terraform 不是代码越短越高级。  
可读、可审、可测试，才是生产级。
