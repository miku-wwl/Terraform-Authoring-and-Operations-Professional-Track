# Terraform 实操训练 53 讲解：嵌套类型

## 1. 这一节到底在学什么？

Lab 53 学的是 Terraform 里的**嵌套类型**。

前面三节你已经学过：

- Lab 50：`list`，有顺序的一组值
- Lab 51：`map`，按 key 管理的一组值
- Lab 52：`object`，一个有固定字段的结构化对象

这一节把它们组合起来：

```hcl
list(object({...}))
map(object({...}))
object({
  tags = map(string)
  ports = list(number)
})
```

这就是嵌套类型。

一句话理解：

> 嵌套类型就是把 list、map、object 组合起来，用来表达更真实、更复杂的生产配置。

---

## 2. 为什么需要嵌套类型？

真实生产配置很少只是一个简单字符串。

比如一个服务配置可能包含：

- 服务名
- 镜像
- 副本数
- 端口列表
- 标签
- 环境变量
- 资源限制
- 健康检查配置
- 是否启用监控

如果不用嵌套类型，你可能会写一堆散变量：

```hcl
service_name = "api"
service_image = "api:v1"
service_port_1 = 8080
service_port_2 = 9090
service_tag_owner = "platform"
service_tag_env = "dev"
```

这种写法很快就会失控。

更好的方式是用结构化对象：

```hcl
service = {
  name  = "api"
  image = "api:v1"
  ports = [8080, 9090]
  tags = {
    Owner       = "platform"
    Environment = "dev"
  }
}
```

这就是嵌套类型的价值：  
把复杂配置组织成清晰的数据结构。

---

## 3. 常见嵌套类型一：`list(object)`

`list(object)` 表示：一组有顺序的对象。

示例：

```hcl
variable "services" {
  type = list(object({
    name = string
    port = number
  }))

  default = [
    {
      name = "api"
      port = 8080
    },
    {
      name = "worker"
      port = 9090
    }
  ]
}
```

访问方式：

```hcl
var.services[0].name
var.services[0].port
```

适合场景：

- 一组有顺序的配置项
- 生成配置文件
- 生成说明文档
- 按顺序处理的规则
- 不一定创建长期稳定资源的对象列表

但是要小心：  
如果用 `list(object)` + `count` 创建长期资源，删除中间元素可能造成索引漂移。

---

## 4. 常见嵌套类型二：`map(object)`

`map(object)` 表示：一组按 key 管理的对象。

示例：

```hcl
variable "services" {
  type = map(object({
    image    = string
    port     = number
    replicas = number
  }))

  default = {
    api = {
      image    = "api:v1"
      port     = 8080
      replicas = 2
    }

    worker = {
      image    = "worker:v1"
      port     = 9090
      replicas = 1
    }
  }
}
```

访问方式：

```hcl
var.services["api"].image
var.services["api"].port
var.services["worker"].replicas
```

配合 `for_each`：

```hcl
resource "local_file" "service" {
  for_each = var.services

  filename = "${path.module}/output/${each.key}.txt"
  content  = "image=${each.value.image}, port=${each.value.port}"
}
```

生产里，`map(object)` 非常常用。

一句话：

> 多个有名字的长期对象，优先考虑 `map(object)`。

---

## 5. 常见嵌套类型三：object 里套 list

一个 object 字段也可以是 list。

示例：

```hcl
variable "service" {
  type = object({
    name  = string
    ports = list(number)
  })

  default = {
    name  = "api"
    ports = [80, 443, 8080]
  }
}
```

访问：

```hcl
var.service.name
var.service.ports
var.service.ports[0]
```

适合场景：

- 一个服务有多个端口
- 一个应用有多个域名
- 一个安全规则有多个 CIDR
- 一个告警有多个接收人
- 一个部署有多个 region

这种结构表达能力比散变量强很多。

---

## 6. 常见嵌套类型四：object 里套 map

object 字段也可以是 map。

示例：

```hcl
variable "service" {
  type = object({
    name = string
    tags = map(string)
  })

  default = {
    name = "api"
    tags = {
      Owner       = "platform"
      Environment = "dev"
    }
  }
}
```

访问：

```hcl
var.service.tags["Owner"]
var.service.tags["Environment"]
```

适合场景：

- tag
- label
- annotation
- environment variables
- key-value 配置
- metadata

这类结构在云平台、Kubernetes、CI/CD 配置里非常常见。

---

## 7. 更真实的嵌套 object 示例

生产里常见配置可能长这样：

```hcl
variable "app" {
  type = object({
    name     = string
    image    = string
    replicas = number

    ports = list(object({
      name = string
      port = number
    }))

    tags = map(string)

    resources = object({
      cpu    = string
      memory = string
    })
  })
}
```

调用方传：

```hcl
app = {
  name     = "payments"
  image    = "payments:v1"
  replicas = 2

  ports = [
    {
      name = "http"
      port = 8080
    },
    {
      name = "metrics"
      port = 9090
    }
  ]

  tags = {
    Owner       = "platform"
    Environment = "dev"
  }

  resources = {
    cpu    = "500m"
    memory = "512Mi"
  }
}
```

这种结构已经很接近真实平台模块输入了。

---

## 8. 嵌套类型的核心价值

嵌套类型的价值不是“语法高级”，而是它能让 Terraform 模块接口更像一个稳定 API。

它可以表达：

- 一个服务有多个端口
- 一个环境有多组配置
- 一个应用有标签、资源、健康检查
- 多个服务各自有不同端口和副本数
- 多个 region 各自有不同 CIDR 和 subnet
- 多个告警各自有阈值、严重级别和接收人

这比一堆散变量更清楚，也更适合测试。

## 12. 嵌套字段怎么访问？

假设有：

```hcl
locals {
  app = {
    name = "api"

    resources = {
      cpu    = "500m"
      memory = "512Mi"
    }

    ports = [
      {
        name = "http"
        port = 8080
      },
      {
        name = "metrics"
        port = 9090
      }
    ]
  }
}
```

访问应用名：

```hcl
local.app.name
```

访问 CPU：

```hcl
local.app.resources.cpu
```

访问第一个端口号：

```hcl
local.app.ports[0].port
```

访问第二个端口名称：

```hcl
local.app.ports[1].name
```

嵌套类型本质就是一层一层往下取。

---

## 13. 用 `for` 表达式处理嵌套类型

嵌套类型经常需要转换。

例如提取所有端口号：

```hcl
locals {
  port_numbers = [
    for p in local.app.ports : p.port
  ]
}
```

结果：

```hcl
[8080, 9090]
```

提取所有端口名：

```hcl
locals {
  port_names = [
    for p in local.app.ports : p.name
  ]
}
```

结果：

```hcl
["http", "metrics"]
```

这类写法在生产里很常见。

---

## 14. `map(object)` 配合 `for_each`

假设：

```hcl
variable "services" {
  type = map(object({
    port     = number
    replicas = number
  }))
}
```

使用：

```hcl
resource "local_file" "service" {
  for_each = var.services

  filename = "${path.module}/output/${each.key}.txt"

  content = jsonencode({
    name     = each.key
    port     = each.value.port
    replicas = each.value.replicas
  })
}
```

这里：

```hcl
each.key
```

是服务名，例如 `api`。

```hcl
each.value.port
```

是服务对象里的端口字段。

这种写法就是 Terraform 生产模块的常见套路。

---

## 15. 嵌套类型和 `jsonencode()`

嵌套类型很适合配合 `jsonencode()`。

例如：

```hcl
locals {
  app_config = {
    name = "api"
    ports = [
      {
        name = "http"
        port = 8080
      }
    ]
    tags = {
      Environment = "dev"
    }
  }
}

resource "local_file" "config" {
  filename = "${path.module}/output/app.json"
  content  = jsonencode(local.app_config)
}
```

Terraform 会把嵌套结构转成合法 JSON。

这比手写 JSON 字符串靠谱得多。

Lab 29 学过 `jsonencode` / `jsondecode`，这里就能串起来。

## 16. 嵌套类型和变量校验

嵌套类型也可以做 validation。

例如检查端口范围：

```hcl
variable "app" {
  type = object({
    ports = list(object({
      name = string
      port = number
    }))
  })

  validation {
    condition = alltrue([
      for p in var.app.ports : p.port > 0 && p.port <= 65535
    ])

    error_message = "All ports must be between 1 and 65535."
  }
}
```

这里用了：

```hcl
for p in var.app.ports
```

遍历所有端口对象。

```hcl
alltrue([...])
```

确认所有端口都满足条件。

这就是嵌套类型 + validation 的生产用法。

---

## 17. 嵌套类型和 optional 字段

复杂 object 里经常需要 optional 字段。

示例：

```hcl
variable "service" {
  type = object({
    name     = string
    port     = number
    replicas = optional(number, 1)
    tags     = optional(map(string), {})
  })
}
```

这样调用方可以只传：

```hcl
service = {
  name = "api"
  port = 8080
}
```

Terraform 会使用默认：

```hcl
replicas = 1
tags     = {}
```

这让模块接口更友好。

但 optional 也要谨慎设计。  
默认值应该符合平台团队的真实策略，不要为了省事乱给默认。

## 18. 嵌套类型的常见坑

### 坑 1：访问路径写错

如果结构是：

```hcl
var.app.resources.cpu
```

你写成：

```hcl
var.app.cpu
```

就会失败。

读嵌套类型时一定要看清层级。

---

### 坑 2：list 和 map 混淆

list 用索引：

```hcl
var.ports[0].port
```

map 用 key：

```hcl
var.services["api"].port
```

不要把二者混着用。

---

### 坑 3：类型声明太宽

例如：

```hcl
type = any
```

虽然省事，但会让模块接口变得不清楚。

生产模块里尽量写明确类型：

```hcl
type = map(object({
  port = number
}))
```

而不是直接 `any`。

---

### 坑 4：嵌套太深

嵌套类型很强，但不是越深越好。

如果你写出五六层嵌套：

```hcl
var.a.b.c.d.e.f
```

可读性会很差。

这时应该考虑拆分模块或拆分变量。

---

### 坑 5：key 设计不稳定

如果用 `map(object)` + `for_each`，map key 会成为资源地址的一部分。

例如：

```text
local_file.service["api"]
```

如果你把 key 改成：

```text
backend-api
```

Terraform 会认为旧资源要销毁，新资源要创建。

所以 key 要稳定，不要把版本号、日期、临时状态放进 key。

---

## 19. 嵌套类型设计建议

设计嵌套类型时，可以遵循这些原则：

1. 能表达业务结构，而不是只堆字段。
2. 类型要明确，少用 `any`。
3. key 要稳定。
4. 深度不要过深。
5. 多个命名对象用 `map(object)`。
6. 多个有序对象用 `list(object)`。
7. 单个复杂对象用 `object`。
8. 关键字段加 validation。
9. secret 字段要考虑 sensitive 和 state 风险。
10. 输出时避免暴露整个复杂对象，优先输出必要摘要。

## 21. 生产环境里的典型嵌套类型

### 服务目录

```hcl
variable "services" {
  type = map(object({
    image    = string
    replicas = number
    ports = list(object({
      name = string
      port = number
    }))
    tags = map(string)
  }))

```

### 多环境配置

```hcl
variable "environments" {
  type = map(object({
    region = string
    vpc_cidr = string
    subnet_cidrs = list(string)
    tags = map(string)
  }))
}
```

### 告警规则

```hcl
variable "alerts" {
  type = map(object({
    threshold = number
    severity  = string
    receivers = list(string)
    enabled   = bool
  }))
}
```

这些都比散变量更接近真实平台工程。

## 22. SRE / Platform Engineering 视角

从平台工程角度看，嵌套类型就是模块接口设计的核心。

一个成熟平台模块通常会接收复杂输入：

- 服务 catalog
- 环境配置
- 告警策略
- 网络拓扑
- 成本标签
- 发布策略
- 资源限制
- 安全策略

这些东西很难用简单 string/list 表达清楚。

你需要用嵌套类型把它们建模成稳定、可测试、可扩展的数据结构。

这就是 Terraform 从“写资源”升级到“设计平台模块”的关键。

## 24. 面试/考试记忆版

可以这样记：

- 嵌套类型是 list、map、object 的组合。
- `list(object)`：一组有序对象。
- `map(object)`：一组按 key 管理的对象。
- object 里可以包含 list、map、object。
- 多个长期命名资源通常适合 `map(object)` + `for_each`。
- 顺序有意义才用 `list(object)`。
- 嵌套字段访问要按层级写，例如 `var.app.ports[0].port`。
- 少用 `any`，生产模块应写明确类型。
- 嵌套太深会降低可读性。
- key 稳定性会影响 Terraform state 地址。

一句话总结：

> Terraform 嵌套类型的价值，是把复杂平台配置建模成清晰、可验证、可扩展的数据结构。

Lab 53 是 Terraform 类型系统训练里很关键的一节。

学完它后，你应该能更自然地理解真实项目里的复杂变量：

```hcl
map(object({

    name = string
    port = number
  }))
  tags = map(string)
}))
```

这类结构一开始看起来吓人，但拆开就是：

- map：多个命名对象
- object：每个对象有固定字段
- list(object)：字段里有一组对象
- map(string)：字段里有一组 key-value

这就是 Terraform 工程化建模的基本功。  
后面做 AKS/EKS、平台模块、FinOps 配置、监控告警模块，都会反复用到。
