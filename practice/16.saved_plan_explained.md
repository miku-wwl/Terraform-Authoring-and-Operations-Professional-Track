# Terraform 实操训练 16：保存 Plan 到文件（讲解版）

## 0. 这一节到底在学什么？

这一节学的是 Terraform 生产变更里非常关键的一招：

> **先把 `plan` 固化成一个文件，再用这个文件执行 `apply`。**

也就是：

```sh
terraform plan -out=tfplan
terraform apply tfplan
```

它解决的是一个很真实的生产问题：

> 审批的时候看到的是 A，真正执行的时候不能偷偷变成 B。

如果只是运行：

```sh
terraform plan
terraform apply
```

中间代码、变量、provider、环境状态都有可能变化。这样审批人看到的计划，未必就是最终执行的内容。

而保存 plan 文件后，Terraform 会把“这一次准备执行的动作”固化下来。后续 `apply tfplan` 执行的就是这个已经保存好的计划。

---

## 1. 核心概念：为什么要保存 Plan？

### 1.1 普通流程的问题

普通 Terraform 流程通常是：

```sh
terraform plan
terraform apply
```

问题在于：

```text
plan 看到的内容
        |
        | 中间代码可能被改了
        | 变量可能变了
        | state 可能变了
        v
apply 实际执行的内容
```

如果 `apply` 时重新计算计划，那它可能不是之前审批过的那一版。

这在生产环境里很危险。

---

### 1.2 保存 Plan 的流程

保存 plan 后，流程变成：

```text
terraform plan -out=tfplan
          |
          v
生成一个二进制 plan 文件
          |
          v
审批 / 审查这个 plan
          |
          v
terraform apply tfplan
```

重点是：

> `terraform apply tfplan` 不会重新生成新的计划，而是执行 `tfplan` 文件里已经确定的动作。

这就是“审批内容”和“执行内容”一致的来源。

---

## 2. 本 Lab 的关键命令

### 2.1 初始化

```sh
terraform init -input=false
```

这里的 `-input=false` 表示不允许 Terraform 在命令行里交互式提问。

在 CI/CD 里，这个参数很重要。因为流水线没人坐在屏幕前帮你输入 `yes`。

---

### 2.2 格式检查

```sh
terraform fmt -check
```

这一步只检查格式，不会自动修改文件。

如果格式不对，命令会失败。适合放在 CI 里阻断不规范代码。

---

### 2.3 配置校验

```sh
terraform validate
```

这一步检查 Terraform 配置是否语法和结构有效。

它不会真正创建资源，只是确认配置“能不能被 Terraform 理解”。

---

### 2.4 运行测试

```sh
terraform test
```

本 Lab 用 `terraform test` 固化实验预期，确认实验里的命令链和行为符合要求。

测试通过说明这个 Lab 的关键逻辑没有偏离设计。

## 3. 保存 Plan 到文件

核心命令是：

```sh
terraform plan -input=false -no-color -out=tfplan
```

拆开看：

| 参数 | 含义 |
|---|---|
| `terraform plan` | 生成执行计划 |
| `-input=false` | 禁止交互输入，适合 CI |
| `-no-color` | 去掉终端颜色字符，方便日志阅读 |
| `-out=tfplan` | 把计划保存到 `tfplan` 文件 |

这里生成的 `tfplan` 不是普通文本文件，而是 Terraform 的二进制 plan 文件。

所以你直接用编辑器打开，很可能看到乱码。

这不是坏了，是正常现象。

---

## 4. 查看 Plan 内容

### 4.1 查看人类可读版本

```sh
terraform show tfplan
```

这个命令会把二进制 plan 转成人能看懂的文本。

适合人工审批时阅读。

---

### 4.2 导出 JSON 版本

```sh
terraform show -json tfplan > plan.json
```

这个命令会把 plan 转成 JSON。

这一步在企业环境里很常见，因为 JSON 可以交给工具继续分析，例如：

- 检查是否创建了高风险资源
- 检查是否删除了生产资源
- 检查 tag 是否符合规范
- 检查安全组、网络、IAM 是否有危险变更
- 生成审批报告

可以理解为：

```text
tfplan       给 Terraform 执行
plan.json    给人或工具审查
```

---

## 5. 使用保存好的 Plan 执行 Apply

```sh
terraform apply -auto-approve tfplan
```

这里有两个重点。

第一，`tfplan` 放在命令最后，表示执行保存好的计划。

第二，使用了 `-auto-approve`。

平时直接运行：

```sh
terraform apply
```

Terraform 会让你输入 `yes`。

但如果你已经指定了 plan 文件，说明这个计划理论上已经经过审查。CI/CD 里通常会配合审批 gate，再执行：

```sh
terraform apply tfplan
```

在自动化场景里加 `-auto-approve` 是为了不让流水线卡住。

---

## 6. 为什么 `apply tfplan` 比裸 `apply` 更适合生产？

### 6.1 裸 apply

```sh
terraform apply
```

它会在 apply 时重新生成计划。

风险是：

```text
审批前看到一版
执行时又算一版
```

这就像你点菜的时候点了牛肉面，后厨突然改成了羊肉泡馍，还说“反正都是吃的”。

生产环境不能这么玩。

---

### 6.2 保存 plan 后 apply

```sh
terraform plan -out=tfplan
terraform apply tfplan
```

它的逻辑是：

```text
先锁定菜单
审批菜单
按这份菜单上菜
```

这才符合生产变更的严肃性。

## 8. 你应该观察什么？

运行完成后，重点看这几个点：

### 8.1 `tfplan` 文件是否生成

```sh
ls -lh tfplan
```

应该能看到一个本地文件。

注意，它不是给你手工编辑的。

---

### 8.2 `terraform show tfplan` 是否能正常显示

```sh
terraform show tfplan
```

这一步能看到 Terraform 准备做什么。

---

### 8.3 `plan.json` 是否生成

```sh
ls -lh plan.json
```

可以进一步查看：

```sh
cat plan.json
```

真实项目中，不建议人工硬看 JSON，而是交给策略工具、脚本或 CI 检查。

---

### 8.4 `apply` 是否使用的是同一个 plan

关键命令是：

```sh
terraform apply -auto-approve tfplan
```

不是：

```sh
terraform apply -auto-approve
```

这两个不是一回事。

前者执行保存好的计划；后者会重新生成计划再执行。

## 9. Terraform Plan 文件的几个注意点

### 9.1 Plan 文件是二进制

`tfplan` 不是 Markdown，也不是 JSON。

不要直接编辑它。

想看内容，用：

```sh
terraform show tfplan
```

想给工具解析，用：

```sh
terraform show -json tfplan > plan.json
```

---

### 9.2 Plan 文件不适合长期保存

Plan 文件和当前 Terraform 配置、state、provider 版本、环境上下文都有关系。

它更像“一次变更的执行快照”，不是长期归档资料。

长期归档更适合保存：

- Git commit
- CI 日志
- `terraform show` 文本输出
- `terraform show -json` 输出
- 审批记录

---

### 9.3 Plan 文件可能包含敏感信息

在真实项目中，plan 里可能包含部分敏感值。

所以不要随便把 `tfplan` 或 `plan.json` 上传到公开仓库。

建议在 `.gitignore` 中加入：

```gitignore
tfplan
*.tfplan
plan.json
```

## 10. 和 CI/CD 的关系

这一节其实是在模拟生产流水线中的审批流程。

一个常见流程是：

```text
Pull Request
    |
    v
terraform fmt
    |
    v
terraform validate
    |
    v
terraform plan -out=tfplan
    |
    v
terraform show / show -json
    |
    v
人工审批或策略审批
    |
    v
terraform apply tfplan
```

真正严肃的 Terraform CI/CD，不应该只是“代码一 push 就 apply”。

更合理的是：

```text
代码审查 + plan 审查 + 审批 gate + 固定 plan apply
```

这就是从“会用 Terraform”走向“会在团队里安全落地 Terraform”的区别。

## 11. 常见错误

### 错误 1：直接打开 `tfplan` 看到乱码

原因：

`tfplan` 是二进制文件。

解决：

```sh
terraform show tfplan
```

或者：

```sh
terraform show -json tfplan > plan.json
```

---

### 错误 2：审批后又执行了裸 apply

错误命令：

```sh
terraform apply -auto-approve
```

正确命令：

```sh
terraform apply -auto-approve tfplan
```

前者重新计算计划，后者执行保存好的计划。

---

### 错误 3：以为 `plan.json` 可以直接 apply

不可以。

`plan.json` 是给人或工具看的，不是给 Terraform apply 用的。

Terraform apply 要用的是原始二进制 plan 文件：

```sh
terraform apply tfplan
```

---

### 错误 4：忘记清理本地文件

实验结束后建议执行：

```sh
rm -f tfplan plan.json
```

否则目录里会留下临时文件。

## 12. 面试怎么说？

如果面试官问：

> Terraform 为什么要保存 plan 文件？

可以这样回答：

> 保存 plan 文件是为了保证审批内容和执行内容一致。`terraform plan -out=tfplan` 会把这次计算出的变更计划固化下来，后续 `terraform apply tfplan` 会执行同一个计划，而不是重新计算。这样可以避免代码或变量在审批后发生变化，导致实际 apply 的内容和审批时看到的不一致。在 CI/CD 中，通常会把 `terraform show` 或 `terraform show -json` 的结果用于审查，再经过审批 gate 后执行保存好的 plan。

如果想更像工程师一点，可以补一句：

> 但 plan 文件可能包含敏感信息，也不适合长期保存，所以真实项目里要控制 artifact 权限，并避免把它提交到 Git 仓库。

---

## 13. 本节最终记忆点

记住这三句话就够了：

1. `terraform plan -out=tfplan`：把计划保存成二进制文件。
2. `terraform show -json tfplan > plan.json`：把计划转成 JSON，方便工具审查。
3. `terraform apply tfplan`：执行已经审批过的同一个计划，避免审批和执行不一致。

最核心的一句话：

> **生产环境里，plan 要可审查，apply 要执行同一个 plan。**
