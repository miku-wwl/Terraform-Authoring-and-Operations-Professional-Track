# Terraform 实操训练 27：基础 heredoc 与 indented heredoc 讲解版

## 1. 这节课到底在学什么？

这一节学的是 Terraform 里两种多行字符串写法的区别：

- `<<EOT`：基础 heredoc，会保留文本里的前导空白。
- `<<-EOT`：indented heredoc，会移除所有行共同拥有的最短前导空白。

一句话理解：

> `<<EOT` 更“原样输出”，你写了多少空格，它就尽量保留多少空格；`<<-EOT` 更适合写在缩进良好的 Terraform 代码里，同时让生成出来的文件不要带一堆多余缩进。

这个知识点看起来小，但在生成 YAML、Shell 脚本、配置文件、cloud-init、Kubernetes manifest 时非常重要。缩进错了，Terraform 不一定报错，但下游工具可能直接解析失败。

---

## 2. 为什么 heredoc 很重要？

Terraform 里经常需要生成多行内容，例如：

- 本地脚本：`deploy.sh`
- 配置文件：`app.conf`
- YAML 文件：`deployment.yaml`
- JSON 策略：IAM policy、Azure Policy、OPA policy
- 启动脚本：cloud-init、user data

如果用普通 quoted string 写多行文本，就会变成这样：

```hcl
content = "line1\nline2\nline3\n"
```

这东西能跑，但可读性很差。脚本一长，维护起来就是灾难。

heredoc 的作用就是让你可以在 Terraform 里直接写多行文本：

```hcl
content = <<EOT
line1
line2
line3
EOT
```

这样更接近真实文件内容，适合长期维护。

---

## 3. 基础 heredoc：`<<EOT`

基础 heredoc 的特点是：**保留内容里的前导空白**。

示例：

```hcl
content = <<EOT
    line one
    line two
EOT
```

生成出来的内容大概率还是：

```text
    line one
    line two
```

也就是说，前面的 4 个空格会被保留下来。

这在某些场景是好事，比如你本来就想保留缩进。但在很多配置文件里，这也可能变成坑。

比如你在 Terraform 代码里为了美观，把 heredoc 内容也跟着缩进：

```hcl
resource "local_file" "basic" {
  content = <<EOT
    name: demo
    env: dev
  EOT
}
```

这里生成的文件内容可能会带上多余缩进。如果下游工具对缩进敏感，例如 YAML，就可能出现你肉眼看 Terraform 代码没问题，但生成文件结构不对的问题。

---

## 4. Indented heredoc：`<<-EOT`

Indented heredoc 的写法多了一个减号：

```hcl
content = <<-EOT
  line one
  line two
EOT
```

它的核心特点是：**Terraform 会移除所有行共同拥有的最短前导空白**。

这句话有点绕，可以这样理解：

> 如果每一行开头都有同样数量的缩进，Terraform 可以帮你把这部分共同缩进剥掉。

这样你就可以让 Terraform 代码保持整齐缩进，同时让最终生成的文件更干净。

示例：

```hcl
resource "local_file" "indented" {
  content = <<-EOT
    line one
    line two
  EOT
}
```

生成出来的内容会更接近：

```text
line one
line two
```

这就是 `<<-EOT` 的价值：**代码可以缩进，输出可以干净**。

---

## 5. 两者核心区别对比

| 写法 | 名称 | 是否保留前导空白 | 适合场景 |
|---|---|---|---|
| `<<EOT` | 基础 heredoc | 保留 | 你明确需要原样保留缩进 |
| `<<-EOT` | indented heredoc | 移除共同前导空白 | Terraform 代码需要缩进，但输出文件不想多缩进 |

重点不是背语法，而是记住这个判断：

> 如果你只是想写一段多行配置，大多数时候优先考虑 `<<-EOT`。如果你明确要保留缩进，再用 `<<EOT`。

你要观察它是否保留了文本中的前导空白。

如果文件内容前面仍然带着缩进，这不是 bug，而是 `<<EOT` 的设计行为。

### 观察点 2：`indented.txt`

`indented.txt` 使用 indented heredoc。


- 基础 heredoc 会保留前导空白。
- indented heredoc 会移除共同前导空白。

## 10. 这里最容易踩的坑

### 坑 1：以为 `<<-EOT` 会删除所有缩进

不会。

它删除的是**所有行共同拥有的最短前导空白**。

如果某一行比其他行多缩进，多出来的部分仍然会保留。

例如：

```hcl
content = <<-EOT
    parent:
      child: value
EOT
```

生成结果大概率类似：

```yaml
parent:
  child: value
```

共同的 4 个空格被去掉了，但 `child` 相对 `parent` 的 2 个空格还在。

这正是它适合 YAML 的原因：可以去掉 Terraform 代码缩进，但保留文本内部的相对缩进。

---

### 坑 2：结束标记写错

开始和结束的 delimiter 必须一致。

正确：

```hcl
content = <<-EOT
hello
EOT
```

错误：

```hcl
content = <<-EOT
hello
EOF
```

你开始用的是 `EOT`，结束就也必须是 `EOT`。

---

### 坑 3：YAML 缩进被你自己写坏

`<<-EOT` 只处理共同缩进，不会帮你理解 YAML 结构。

如果 YAML 本身相对缩进写错，Terraform 不会自动救你。

比如：

```yaml
apiVersion: v1
kind: ConfigMap
metadata:
name: demo
```

这类结构错误不是 heredoc 的锅，是 YAML 本身缩进错了。

---

### 坑 4：把 heredoc 当模板系统乱用

heredoc 支持 Terraform 插值：

```hcl
content = <<-EOT
name = ${var.name}
EOT
```

这很好用，但也要小心。

如果你想输出字面量 `${something}`，而不是让 Terraform 插值，就需要额外处理转义。复杂模板建议用 `templatefile()`，不要把所有逻辑都堆进 heredoc。

---

## 11. 什么时候用 `<<EOT`，什么时候用 `<<-EOT`？

我的建议很直接：

- 生成脚本、YAML、配置文件：优先用 `<<-EOT`。
- 你确实需要完全保留缩进：用 `<<EOT`。
- 内容很复杂、变量很多：考虑 `templatefile()`。
- 内容是 JSON：优先考虑 `jsonencode()`，不要手写 JSON 字符串。

尤其是生产环境里，手写 JSON/YAML 字符串非常容易出错。Terraform 能帮你 encode 的，就别硬拼字符串。

## 13. 面试/工作里怎么说？

可以这样表达：

> Terraform heredoc is useful for generating multi-line files such as scripts or configuration snippets. The standard heredoc syntax `<<EOT` preserves leading whitespace, while the indented heredoc syntax `<<-EOT` strips the common leading indentation. In production, I usually prefer `<<-EOT` for readability, especially when generating YAML or shell scripts, but I still validate the generated output because indentation-sensitive formats can easily break.

中文理解就是：

> 我知道 Terraform 的 heredoc 不只是多行字符串，还会影响最终文件缩进。生产里我会优先用 `<<-EOT` 保持代码可读，同时检查生成文件，避免 YAML、脚本这类缩进敏感内容出问题。

这比单纯说“我会 heredoc”更像工程实践。
