# Terraform 实操训练 10：Plugin Cache 实践落地

## 1. 本节核心

本节是对 Lab 9 的实践落地版。Lab 9 讲的是 **Provider Plugin Cache 是什么**，Lab 10 讲的是 **团队里如何标准化配置 Plugin Cache，并处理 `.terraform.lock.hcl` 的关系**。

一句话总结：

> `TF_PLUGIN_CACHE_DIR` / `plugin_cache_dir` 负责让 provider 下载结果可复用，`.terraform.lock.hcl` 负责保证复用的是正确版本和正确校验和。cache 是加速，lock file 是安全。

---

## 2. 为什么只创建缓存目录还不够？

很多人会误以为：

```text
mkdir -p .terraform-plugin-cache
```

这样就已经启用了 provider cache。

但实际上还不够。

你必须告诉 Terraform CLI：

```text
provider plugin cache 目录在哪里
```

否则 Terraform 不知道要使用这个目录。

Terraform 有两种常见方式启用 plugin cache：

1. CLI 配置文件中的 `plugin_cache_dir`
2. 环境变量 `TF_PLUGIN_CACHE_DIR`

---

## 3. 两种启用方式

### 3.1 方式一：使用环境变量 `TF_PLUGIN_CACHE_DIR`

实验里使用的是这种方式：

```sh
mkdir -p .terraform-plugin-cache
export TF_PLUGIN_CACHE_DIR=/workspace/practice/labs/10/.terraform-plugin-cache
```

这表示：

```text
告诉 Terraform：
provider plugin cache 目录在 /workspace/practice/labs/10/.terraform-plugin-cache
```

这种方式特别适合：

```text
CI runner
Docker container
临时 shell
不同项目想指定不同 cache 目录
```

例如 Docker 场景中，可以把宿主机目录挂载到容器内：

```sh
docker run --rm \
  -v "$PWD":/workspace \
  -v /opt/terraform-plugin-cache:/terraform-plugin-cache \
  -w /workspace \
  -e TF_PLUGIN_CACHE_DIR=/terraform-plugin-cache \
  hashicorp/terraform:1.11 \
  init -input=false
```

这里的关系是：

```text
VM 宿主机：
  /opt/terraform-plugin-cache

Docker container 内部：
  /terraform-plugin-cache

Terraform 看到的是：
  TF_PLUGIN_CACHE_DIR=/terraform-plugin-cache
```

---

### 3.2 方式二：使用 Terraform CLI 配置文件

也可以在 Terraform CLI 配置文件中写：

```hcl
plugin_cache_dir = "/home/runner/.terraform.d/plugin-cache"
```

Windows 示例：

```hcl
plugin_cache_dir = "C:/Users/Wesley/AppData/Roaming/terraform.d/plugin-cache"
```

CI container 示例：

```hcl
plugin_cache_dir = "/terraform-plugin-cache"
```

这种方式更适合开发者本机长期使用。

可以这样记：

```text
开发者本机：更适合 CLI config
CI / Docker：更适合 TF_PLUGIN_CACHE_DIR
```

---

## 4. `.terraform.lock.hcl` 的作用

Plugin cache 只负责“下载过的 provider 能不能复用”。

`.terraform.lock.hcl` 负责：

```text
当前项目实际锁定哪个 provider 版本
对应 provider 包的 checksum 是什么
```

Terraform 的 provider 安装逻辑大概是：

```text
required_providers 说明允许哪些版本
        ↓
.terraform.lock.hcl 记录当前锁定版本和 checksum
        ↓
Terraform 在 plugin cache 中查找 provider
        ↓
如果版本和 checksum 匹配：复用 cache
如果不匹配：不复用，甚至报错
```

所以：

```text
cache 不能替代 .terraform.lock.hcl
.terraform.lock.hcl 也不能替代 cache
```

两者职责不同。

---

## 5. 如果 Terraform 代码和 lock file 不一致会怎样？

关键看 `.terraform.lock.hcl` 中的 provider 版本是否满足 Terraform 代码里的 version constraint。

### 5.1 满足约束：无事发生

例如代码中：

```hcl
terraform {
  required_providers {
    random = {
      source  = "hashicorp/random"
      version = ">= 3.5.0"
    }
  }
}
```

lock file 中锁定：

```text
random 3.6.0
```

因为：

```text
3.6.0 满足 >= 3.5.0
```

所以 Terraform 会继续使用 lock file 中的版本。

---

### 5.2 不满足约束：terraform init 报错

例如代码改成：

```hcl
version = ">= 3.7.0"
```

但 lock file 还是：

```text
random 3.6.0
```

因为：

```text
3.6.0 不满足 >= 3.7.0
```

这时候 `terraform init` 会报错。

常见修复方式：

```sh
terraform init -upgrade
```

然后把更新后的 `.terraform.lock.hcl` 一起提交。

可以粗暴记成：

> 版本约束是门槛，lock file 是当前选手；选手过得了门槛就继续用，过不了就报错换人。

---

## 6. 多平台为什么会失败？

团队里可能有不同运行平台：

```text
Windows 开发者
macOS Intel 开发者
macOS Apple Silicon 开发者
Linux CI runner
Docker Linux container
```

同一个 provider 版本，会有不同平台的 provider 包：

```text
linux_amd64
linux_arm64
windows_amd64
darwin_amd64
darwin_arm64
```

如果 `.terraform.lock.hcl` 里没有目标平台的 checksum，其他平台运行时可能失败或需要更新 lock file。

多平台团队可以提前运行：

```sh
terraform providers lock \
  -platform=linux_amd64 \
  -platform=windows_amd64 \
  -platform=darwin_amd64 \
  -platform=darwin_arm64
```

这样可以提前把多平台 checksum 写进 `.terraform.lock.hcl`。

---

## 7. `plugin_cache_may_break_dependency_lock_file` 是什么？

这个配置项名字很吓人，意思大概是：

```text
允许 Terraform 更宽松地使用 plugin cache，
即使这样可能让 dependency lock file 不完整或不可移植。
```

一般不建议使用。

正常团队应该优先：

```text
提交 .terraform.lock.hcl
保证 lock file 包含目标平台 checksum
不要随便绕开 dependency lock file 的校验机制
```

可以这样记：

```text
plugin_cache_may_break_dependency_lock_file 是逃生门，不是最佳实践。
```

## 8. 实验运行命令讲解

实验命令如下：

```sh
cd /workspace/practice/labs/10
mkdir -p .terraform-plugin-cache
export TF_PLUGIN_CACHE_DIR=/workspace/practice/labs/10/.terraform-plugin-cache
terraform init -input=false
terraform fmt -check
terraform validate
terraform test
terraform plan -input=false -no-color -out=tfplan
terraform apply -auto-approve tfplan
terraform output
terraform destroy -auto-approve
```

### 8.1 `mkdir -p .terraform-plugin-cache`

创建 provider plugin cache 目录。

这个目录不应该提交到 Git。

---

### 8.2 `export TF_PLUGIN_CACHE_DIR=...`

告诉 Terraform 使用这个目录作为 provider plugin cache。

CI / Docker 中建议使用绝对路径，避免工作目录变化导致路径错误。

---

### 8.3 `terraform init -input=false`

初始化 Terraform 工作目录。

主要做：

```text
初始化 backend
安装 module
安装 provider
读取或生成 .terraform.lock.hcl
```

Plugin cache 主要影响 provider 安装阶段。

---

### 8.4 `terraform fmt -check`

检查 Terraform 代码格式。

CI 中常用 `-check`，格式不对就让 pipeline 失败，而不是自动帮你改。

---

### 8.5 `terraform validate`

检查 Terraform 配置是否合法。

它检查语法、引用、provider 配置结构等，但不真正创建云资源。

---

### 8.6 `terraform test`

运行 `.tftest.hcl` 测试。

本节测试重点应该是验证生成内容中包含：

```text
plugin_cache_dir
TF_PLUGIN_CACHE_DIR
.terraform.lock.hcl
lock file 说明
多平台说明
CI 绝对路径说明
```

---

### 8.7 `terraform plan -input=false -no-color -out=tfplan`

生成 plan 文件。

参数含义：

```text
-input=false：CI 中禁止交互输入
-no-color：去掉 ANSI 颜色控制符，方便日志归档
-out=tfplan：保存计划，后续 apply 可以使用同一个 plan
```

---

### 8.8 `terraform apply -auto-approve tfplan`

应用刚才生成的计划。

实验环境可以这样做。

生产环境一般会：

```text
plan
人工审批
apply tfplan
```

---

### 8.9 `terraform destroy -auto-approve`

清理实验资源。

---

## 9. 清理命令

完整清理：

```sh
terraform destroy -auto-approve
rm -rf .terraform .terraform-plugin-cache output tfplan terraform.tfstate*
```

含义：

```text
.terraform/：Terraform 当前工作目录初始化结果
.terraform-plugin-cache/：provider cache
output/：实验输出文件
tfplan：plan 文件
terraform.tfstate*：本地 state 文件
```

## 10. 生产级建议

推荐提交：

```text
.terraform.lock.hcl
*.tf
*.tfvars.example
README.md
```

不推荐提交：

```text
.terraform/
.terraform-plugin-cache/
tfplan
*.tfplan
terraform.tfstate
terraform.tfstate.*
```

推荐 `.gitignore`：

```gitignore
.terraform/
.terraform-plugin-cache/
*.tfplan
tfplan
terraform.tfstate
terraform.tfstate.*
output/
```

团队级规则：

```text
1. 提交 .terraform.lock.hcl
2. 不提交 .terraform/
3. 不提交 .terraform-plugin-cache/
4. CI 中使用 TF_PLUGIN_CACHE_DIR 指向容器内绝对路径
5. Docker 场景用 -v 挂载 cache 目录
6. 多平台团队用 terraform providers lock -platform=... 补齐 checksum
7. 默认不要启用 plugin_cache_may_break_dependency_lock_file
8. 记住 cache 只是优化，lock file 才是 provider 版本和校验和控制
```

## 11. 本节的真正价值

这节不是要证明 provider cache 多么核心。

更准确地说：

```text
Provider cache 是局部性能优化，不是生产安全核心机制。
```

它主要适合：

```text
频繁运行 Terraform CI
一次性 runner 每次都是干净环境
provider 比较大
网络到 registry 较慢
多个 Terraform 项目重复使用相同 provider
```

如果只是开发者本机手动运行 Terraform，或者生产 CD apply 频率很低，cache 的价值确实有限。

真正生产级 Terraform 更重要的是：

```text
remote backend
state locking
.terraform.lock.hcl
plan/apply 分离
approval gate
OIDC / short-lived credentials
branch protection
plan artifact 审计
```

## 12. 面试级英文总结

```text
Terraform plugin cache can be enabled either through the CLI configuration file using plugin_cache_dir or through the TF_PLUGIN_CACHE_DIR environment variable. In CI and Docker-based runners, the environment variable is usually more practical because the cache path can be injected per job or mounted as a volume.

However, the cache does not replace .terraform.lock.hcl. The lock file records the selected provider versions and checksums, while the cache only stores downloaded provider packages. Terraform will only safely reuse cached providers when they match the dependency lock file. For multi-platform teams, we should ensure the lock file contains checksums for all target platforms, usually with terraform providers lock -platform=....

Provider caching is useful, but its impact is limited. It is mainly a CI performance optimization for ephemeral runners, not a core production safety mechanism.
```
