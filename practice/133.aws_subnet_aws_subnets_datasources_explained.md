# Terraform 实操训练 133：aws_subnet / aws_subnets 数据源（讲解版）

## 1. 本节核心：data source 是“查已有资源”，不是“创建资源”

这一节讲的是 Terraform AWS Provider 里的两个数据源：

```hcl
data "aws_subnets" "..."
data "aws_subnet" "..."
```

它们的共同点是：

```text
不是创建资源，而是查询资源。
```

简单说：

```text
resource = 我来创建和管理这个资源
data source = 我去查询已经存在的资源
```

本节实验会先创建一个 VPC 和两个 subnet，然后再用数据源查询它们。

所以流程是：

```text
创建 VPC
创建两个 Subnet
用 aws_subnets 查询 subnet ID 列表
用 aws_subnet 查询单个 subnet 的 cidr_block
输出查询结果
```

---

## 2. aws_subnets 和 aws_subnet 有什么区别？

这两个名字只差一个 `s`，但含义不同。

| 数据源 | 查询目标 | 返回结果特点 |
|---|---|---|
| `aws_subnets` | 查询一组子网 | 通常返回多个 subnet IDs |
| `aws_subnet` | 查询一个子网 | 返回单个 subnet 的详细属性 |

一句话记忆：

```text
aws_subnets：复数，查一批。
aws_subnet：单数，查一个。
```

---

## 3. 为什么需要 aws_subnets？

真实项目里，你经常不是自己在当前 Terraform 配置里创建所有网络资源。

常见场景是：

```text
网络团队已经创建好 VPC 和 Subnet
应用团队只需要查询这些 Subnet
然后把应用部署进去
```

例如 EKS、ALB、EC2、RDS 都可能需要 subnet IDs。

你不一定要重新创建 subnet，而是要找到现有 subnet。

这时候就可以用：

```hcl
data "aws_subnets" "selected" {
  filter {
    name   = "vpc-id"
    values = [aws_vpc.main.id]
  }
}
```

它会查询这个 VPC 下面的 subnet 集合。

---

## 4. aws_subnets 的典型写法

假设已经有一个 VPC：

```hcl
resource "aws_vpc" "main" {
  cidr_block = "10.0.0.0/16"
}
```

再创建两个 subnet：

```hcl
resource "aws_subnet" "private_a" {
  vpc_id     = aws_vpc.main.id
  cidr_block = "10.0.1.0/24"
}

resource "aws_subnet" "private_b" {
  vpc_id     = aws_vpc.main.id
  cidr_block = "10.0.2.0/24"
}
```

然后用 `aws_subnets` 按 VPC ID 查询：

```hcl
data "aws_subnets" "selected" {
  filter {
    name   = "vpc-id"
    values = [aws_vpc.main.id]
  }

  depends_on = [
    aws_subnet.private_a,
    aws_subnet.private_b
  ]
}
```

查询结果通常使用：

```hcl
data.aws_subnets.selected.ids
```

它表示查到的一组 subnet ID。

---

## 5. 为什么 data source 有时候也需要 depends_on？

很多人以为只有 resource 才有依赖关系，data source 不需要。

这不完全对。

在本实验里，subnet 是当前配置里刚刚创建出来的：

```text
先创建 subnet
再查询 subnet
```

如果 Terraform 不能从表达式里推断出顺序，data source 可能在 subnet 还没创建时就去查询，导致查不到。

所以可以显式写：

```hcl
depends_on = [
  aws_subnet.private_a,
  aws_subnet.private_b
]
```

含义是：

```text
等这两个 subnet 创建完成后，再执行这个查询。
```

这在实验和某些真实配置里很有用。

---

## 6. aws_subnet 的典型写法

`aws_subnet` 是单数数据源，用来查询一个具体 subnet 的详细信息。

例如从 `aws_subnets` 查到一组 subnet IDs 后，取第一个：

```hcl
data "aws_subnet" "first" {
  id = data.aws_subnets.selected.ids[0]
}
```

然后可以读取这个 subnet 的属性：

```hcl
data.aws_subnet.first.cidr_block
```

输出：

```hcl
output "first_subnet_cidr" {
  value = data.aws_subnet.first.cidr_block
}
```

这就完成了：

```text
先查一组 subnet IDs
再拿其中一个 ID 查询详细信息
最后输出 cidr_block
```

---

## 7. filter 是怎么工作的？

`aws_subnets` 常用 `filter` 来筛选资源。

例如：

```hcl
filter {
  name   = "vpc-id"
  values = [aws_vpc.main.id]
}
```

这表示：

```text
只找属于这个 VPC 的 subnets。
```

在 AWS Provider 里，filter 的写法通常接近 AWS API 的过滤条件。

常见 filter 包括：

```text
vpc-id
availability-zone
tag:Name
tag:Environment
cidr-block
state
```

例如按 tag 查：

```hcl
data "aws_subnets" "private" {
  filter {
    name   = "tag:Tier"
    values = ["private"]
  }
}
```

这表示：

```text
查询 Tag 里 Tier=private 的子网。
```

---

## 8. 查询结果顺序不要乱依赖

一个容易踩坑的点：

```hcl
data.aws_subnets.selected.ids[0]
```

这能取第一个 subnet ID，但你不要在生产里随便假设“第一个就是 private_a”。

因为集合查询的返回顺序不应该成为你的业务逻辑依赖。

更稳的方式是：

```text
用明确 tag 查询
用明确名称查询
用 map 建模
用排序函数处理
```

例如，如果你真的要稳定顺序，可以考虑：

```hcl
sort(data.aws_subnets.selected.ids)[0]
```

但更好的生产设计通常是：  
用 tag 或明确输入变量表达 subnet 的用途，而不是靠数组下标猜。

---

## 9. data source 和 resource 的区别

这是 Terraform 很核心的概念。

| 类型 | 作用 | 是否创建资源 | 是否进入 state |
|---|---|---|---|
| `resource` | 创建和管理资源 | 是 | 是 |
| `data` | 查询已有资源 | 否 | 会记录查询结果相关信息 |

例如：

```hcl
resource "aws_subnet" "example" {
  vpc_id     = aws_vpc.main.id
  cidr_block = "10.0.1.0/24"
}
```

这表示：

```text
Terraform 负责创建这个 subnet。
```

而：

```hcl
data "aws_subnet" "example" {
  id = "subnet-xxxx"
}
```

这表示：

```text
Terraform 去查询这个已经存在的 subnet。
```

不要把 data source 当成资源创建器。  
它只是“查户口”，不是“生孩子”。

---

## 10. 为什么这个实验先创建再查询？

你可能会问：

> 既然 data source 是查已有资源，为什么实验里还要先创建 VPC 和 subnet？

因为这是本地训练，不访问真实 AWS。

为了让查询有目标，实验会先用 Terraform 创建一些模拟资源：

```text
aws_vpc
aws_subnet.private_a
aws_subnet.private_b
```

然后再用 data source 查询它们：

```text
data.aws_subnets.selected
data.aws_subnet.first
```

这样可以在 LocalStack 里完成闭环：

```text
创建 -> 查询 -> 输出 -> 销毁
```

这是为了练 Terraform 行为，不是说真实项目里一定要先创建再查。

真实项目中，data source 更多用于查询其他团队、其他模块、其他 workspace 或手工已有的资源。

---

## 11. LocalStack 在这一节里的作用

本节实验不访问真实 AWS，而是使用 LocalStack 模拟 AWS API。

结构大概是：

```text
Terraform AWS Provider
-> endpoint 指向 http://localhost:4566
-> 使用假凭证 test/test
-> LocalStack 模拟 AWS 服务
```

这样做的好处：

- 不需要真实 AWS 账号
- 不需要真实 access key
- 不产生 AWS 费用
- 不会误删真实资源
- 可以反复练习 Terraform 的 init / plan / apply / destroy
- 适合理解 Provider、资源依赖、data source 查询和 output

但是要记住边界：

```text
LocalStack 可以验证 Terraform 配置逻辑
但不能完全替代真实 AWS 的 IAM 权限、网络连通性、服务配额和真实 API 行为
```

迁移到真实 AWS 时，还要额外检查：

- IAM 权限是否允许 describe subnets
- subnet 是否真的存在
- filter 是否能匹配真实 tag
- region 是否正确
- VPC ID 是否正确
- 多账号场景下 provider alias 是否正确
- 查询结果是否唯一或稳定

## 14. 你运行时应该观察什么？

不要只看“命令通过了”。  
这一节真正要观察的是 data source 的行为。

重点看：

```text
1. aws_vpc 和 aws_subnet 是 resource，负责创建。
2. aws_subnets 是 data source，负责按条件查询 subnet 集合。
3. aws_subnet 是 data source，负责读取单个 subnet 详细信息。
4. data.aws_subnets.selected.ids 是否输出了多个 subnet ID。
5. data.aws_subnet.first.cidr_block 是否读取到了 CIDR。
6. plan 里 data source 的读取时机是否合理。
7. destroy 时只销毁 resource，不是销毁 data source 查询对象。
```

这里有一个关键理解：

```text
data source 本身不会创建云资源，所以 destroy 时也不是“删除 data source 创建的资源”。
```

真正被销毁的是 Terraform 管理的 resource。

## 15. 常见错误和排障

## 15.1 filter 条件写错，查不到 subnet

例如：

```hcl
filter {
  name   = "vpc_id"
  values = [aws_vpc.main.id]
}
```

这里可能写错，正确一般是：

```hcl
name = "vpc-id"
```

排查方向：

```text
检查 filter name 是否符合 AWS API 的过滤字段
检查 values 是否传入正确 VPC ID
检查 region 是否一致
检查 subnet 是否已经创建
```

---

## 15.2 查询结果为空，还去取 ids[0]

如果：

```hcl
data.aws_subnets.selected.ids
```

是空列表，那么：

```hcl
data.aws_subnets.selected.ids[0]
```

就会出错。

所以生产里要避免盲目取第一个。  
更好的做法是用更精确的 filter，或者把 subnet ID 作为明确变量传入。

---

## 15.3 多个 subnet 匹配，但你以为只有一个

如果 filter 太宽：

```hcl
filter {
  name   = "vpc-id"
  values = [var.vpc_id]
}
```

可能会匹配整个 VPC 下所有 subnets。

如果你的目标只是 private subnets，应该加 tag filter：

```hcl
filter {
  name   = "tag:Tier"
  values = ["private"]
}
```

否则你可能把 public subnet 也传给了不该使用 public subnet 的资源。

---

## 15.4 LocalStack 能跑，不代表真实 AWS 一定没问题

LocalStack 验证核心路径，但真实 AWS 还会检查：

```text
IAM 权限
真实资源存在性
真实 region
真实 tag
服务限制
账号边界
网络设计
```

所以不要因为 LocalStack 通过，就直接无脑上生产。

LocalStack 是训练场，不是生产体检报告。

---

## 16. 生产环境推荐写法

更生产一点的查询方式通常依赖 tag。

例如查询 private subnets：

```hcl
data "aws_subnets" "private" {
  filter {
    name   = "vpc-id"
    values = [var.vpc_id]
  }

  filter {
    name   = "tag:Tier"
    values = ["private"]
  }

  filter {
    name   = "tag:Environment"
    values = [var.environment]
  }
}
```

这样比只按 VPC ID 查更安全。

因为 VPC 下可能同时有：

```text
public subnets
private subnets
database subnets
inspection subnets
intra subnets
```

只按 VPC ID 查，范围太大。

## 17. 输出设计建议

输出 subnet IDs 时可以这样：

```hcl
output "subnet_ids" {
  value = data.aws_subnets.private.ids
}
```

输出单个 subnet CIDR：

```hcl
output "first_subnet_cidr" {
  value = data.aws_subnet.first.cidr_block
}
```

但生产里要小心输出内容。

一般可以输出：

```text
subnet_id
vpc_id
cidr_block
availability_zone
```

不要输出敏感信息。  
虽然 subnet 本身不算 secret，但养成“output 最小化”的习惯很重要。

## 18. 和 EKS / RDS / ALB 的关系

这个知识点在真实项目里很常用。

例如 EKS：

```hcl
module "eks" {
  source = "terraform-aws-modules/eks/aws"

  subnet_ids = data.aws_subnets.private.ids
}
```

RDS：

```hcl
resource "aws_db_subnet_group" "main" {
  subnet_ids = data.aws_subnets.database.ids
}
```

ALB：

```hcl
resource "aws_lb" "public" {
  subnets = data.aws_subnets.public.ids
}
```

所以 `aws_subnets` 是 AWS Terraform 里非常常见的“接入已有网络”的方式。

你后面做 EKS / AKS / 平台工程项目，肯定会遇到这种模式。

## 19. 考试重点怎么记？

本节考试重点：

```text
aws_subnets 是复数数据源，用来查询一组 subnet IDs。
aws_subnet 是单数数据源，用来查询一个 subnet 的详细属性。
data source 不创建资源，只查询已有资源。
filter 用来筛选目标资源，例如 vpc-id 或 tag。
LocalStack 可以验证 Terraform 路径，但不能替代真实 AWS 权限和网络验证。
```

如果考题说：

> 需要查询某个 VPC 下所有 subnet IDs。

答案指向：

```hcl
data "aws_subnets"
```

如果考题说：

> 已经有一个 subnet ID，需要读取它的 cidr_block。

答案指向：

```hcl
data "aws_subnet"
```

## 22. 最终总结

这一节最重要的理解是：

> `aws_subnets` 和 `aws_subnet` 都是 Terraform data source，用于查询 AWS 中已有的子网信息；前者查集合，后者查单个资源详情。

完整逻辑：

```text
创建 VPC 和两个 subnet
-> 用 aws_subnets 按 vpc-id 查询 subnet ID 集合
-> 用 aws_subnet 读取单个 subnet 的 cidr_block
-> 输出查询结果
-> destroy 清理 resource
```

这类能力在真实项目中非常常见，尤其是应用团队接入平台团队已经创建好的 VPC、Subnet、EKS、RDS、ALB 时。

一句话收尾：

```text
Terraform 不只是会创建资源，也要会“认路”：data source 就是 Terraform 在云上查资料的能力。
```
