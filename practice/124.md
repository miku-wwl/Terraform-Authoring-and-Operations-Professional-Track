# Terraform 考试训练 124：CLI-driven Run Workflow 实践步骤

## 1. 课程元数据

- 课程编号：124
- 来源文件：`doc/124.txt`
- 环境分类：`theory`
- 是否可用 Docker 运行：`no`
- 核心主题：CLI-driven workflow 的配置、登录、初始化和远端运行
- 前置条件：理解 `cloud` block、workspace、Terraform CLI
- 训练方式：知识点讲解 + 选择题 + 判断题 + 场景题

## 2. 知识点讲解

本节课程演示 CLI-driven Workflow 的实际步骤。典型流程是：在 HCP Terraform 中创建 CLI-driven workspace；在本地 Terraform 配置里添加 `cloud` block，写入 organization 和 workspace；执行 `terraform login` 获取并保存访问令牌；执行 `terraform init` 初始化云端集成；之后使用本地 CLI 发起 plan 和 apply，实际运行记录会出现在 HCP Terraform。

这个流程的核心是身份和绑定关系。`terraform login` 负责认证，`cloud` block 负责告诉 Terraform 要连接哪个 organization 和 workspace，`terraform init` 负责初始化 backend/cloud 集成。缺少其中任何一环，都可能导致本地目录无法和 HCP Terraform workspace 正确关联。

考试中还会考非交互式或自动化场景。比如 state 迁移时可以使用特定 flag 减少人工确认，但对于 CLI-driven run 的基础概念，你更需要掌握登录、初始化、plan/apply 远端执行这些步骤的顺序和职责。

## 3. 考试选择题

1. CLI-driven Workflow 中，`terraform login` 的主要作用是什么？
   - A. 获取并保存访问 HCP Terraform 的凭据
   - B. 创建本地 Git 分支
   - C. 格式化 `.tf` 文件
   - D. 删除所有 workspace

2. `cloud` block 中通常需要指定哪些信息？
   - A. organization 和 workspace
   - B. Docker 镜像名和端口
   - C. 本地浏览器类型
   - D. Git commit message 模板

3. 本地配置添加 `cloud` block 后，通常需要运行哪个命令完成初始化？
   - A. `terraform init`
   - B. `terraform fmt`
   - C. `terraform force-unlock`
   - D. `terraform providers mirror`

## 4. 判断题

1. `terraform login` 与 HCP Terraform 认证有关。
2. CLI-driven Workflow 中，plan/apply 命令可以从本地 CLI 发起。
3. 只要写了 `cloud` block，就不需要执行 `terraform init`。

## 5. 场景题

1. 用户在本地执行 `terraform plan` 时提示没有 HCP Terraform 凭据。应该优先检查哪个步骤？

2. 用户把 workspace 名称写错，导致本地目录连接不到预期 workspace。这个问题属于认证问题还是绑定配置问题？

## 6. 参考答案与解析

选择题：1-A，2-A，3-A。

判断题：1-对，2-对，3-错。

场景题参考：

1. 优先检查是否执行过 `terraform login`，以及凭据是否仍然有效。
2. 属于绑定配置问题。`cloud` block 中 organization 或 workspace 名称错误，会把本地配置指向错误位置或无法初始化。

## 7. 复习提醒

- 步骤顺序：创建 workspace，添加 `cloud` block，`terraform login`，`terraform init`，再 plan/apply。
- `login` 解决身份，`cloud` block 解决目标，`init` 解决初始化。
- 真实练习可选注册 HCP Terraform，但考试主要考流程理解。
