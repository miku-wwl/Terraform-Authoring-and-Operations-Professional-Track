# 第 155 节：挑战一解法二：把输出保存到文件并练习状态操作（讲解版）

## 1. 本节定位

本节是综合挑战题的一种解法练习，核心不是学习某一个新的 AWS 资源，而是训练 Terraform 实战中的完整闭环能力：

1. 读懂题目要求。
2. 找到需要修改的 Terraform 配置。
3. 执行 `fmt`、`init`、`validate`、`plan`、`apply`。
4. 使用 `terraform output` 查看结果。
5. 把输出保存到文件，方便提交、审计或后续脚本读取。
6. 使用 Terraform state 相关命令观察和理解状态文件。
7. 完成验证后执行 `destroy` 清理环境。

这一节的关键词是：**output、输出文件、state list、state show、状态操作、挑战题拆解**。

---

## 2. 一句话理解

Terraform 不只是“写 `.tf` 然后 apply”。真正上手时，你还要会把部署结果导出来、会检查 state 中到底记录了什么、会用 state 命令排查资源和配置之间的关系。

这一节就是让你练：

> 资源部署出来以后，我如何拿到结果？结果如何保存？Terraform state 里到底追踪了哪些对象？

## 3. 实验环境边界

本节实验被改造成 LocalStack 或本地 provider 可验证的练习环境。

- AWS API endpoint 默认指向：`http://localhost:4566`
- AWS 凭证固定为：`test/test`
- 不访问真实 AWS 账号
- 不写入真实 access key、secret key 或 root 账号信息
- 本节重点是 Terraform 操作流程，不是验证真实 AWS IAM 权限或真实云资源行为

也就是说，这个 lab 练的是 Terraform 工作流、输出处理和 state 操作，不是让你花真钱部署真实 AWS。

## 4. 为什么要把 output 保存到文件

在 Terraform 里，`output` 通常用于暴露部署后的关键结果，例如：

- S3 bucket 名称
- IAM user name
- resource ARN
- endpoint 地址
- VPC ID
- subnet ID
- security group ID

平时你在终端里运行：

```bash
terraform output
```

可以直接看到输出结果。但是在真实项目中，只看终端不够。你经常需要把输出保存下来，用于：

1. **提交实验结果**：证明你完成了 lab。
2. **后续脚本读取**：例如 CI/CD 后续阶段需要拿 bucket name 或 endpoint。
3. **排障记录**：保存 apply 后的实际输出，方便之后对比。
4. **审计留痕**：知道某次部署创建了哪些资源标识。

所以本节会练习把 output 保存成普通文本或 JSON 文件。

## 5. Terraform output 的常见用法

### 5.1 查看所有输出

```bash
terraform output
```

适合人工阅读。

示例输出可能类似：

```text
bucket_name = "tf-lab-155-demo"
user_name   = "lab-user"
policy_arn  = "arn:aws:iam::000000000000:policy/example"
```

### 5.2 查看单个输出

```bash
terraform output bucket_name
```

适合只关心一个值的时候。

### 5.3 获取纯字符串值

```bash
terraform output -raw bucket_name
```

`-raw` 会去掉 Terraform 的引号和格式包装，适合 shell 脚本使用。

例如：

```bash
BUCKET_NAME=$(terraform output -raw bucket_name)
echo "$BUCKET_NAME"
```

### 5.4 输出 JSON

```bash
terraform output -json
```

适合给程序读取，例如 `jq`、Python、CI pipeline。

示例：

```bash
terraform output -json > outputs.json
```

然后可以用：

```bash
cat outputs.json | jq .
```

---

## 6. 把输出保存到文件

### 6.1 保存普通文本输出

```bash
terraform output > outputs.txt
```

这会把当前所有 output 写入 `outputs.txt`。

适合人类阅读，也适合作为实验提交截图或记录。

### 6.2 保存 JSON 输出

```bash
terraform output -json > outputs.json
```

这会把输出保存成结构化 JSON。

适合机器读取，比如：

```bash
jq -r '.bucket_name.value' outputs.json
```

### 6.3 保存单个 raw 值

```bash
terraform output -raw bucket_name > bucket_name.txt
```

这种方式最适合后续脚本。

例如：

```bash
aws --endpoint-url=http://localhost:4566 s3 ls "s3://$(cat bucket_name.txt)"
```

## 7. output 文件应该不要提交什么

这是一个很重要的避坑点。

如果 output 里包含：

- access key
- secret key
- token
- password
- private key
- connection string
- database password

那就不要随便提交到 Git。

即使 Terraform 里 output 标记了：

```hcl
sensitive = true
```

也不代表你导出的文件一定安全。尤其是：

```bash
terraform output -json > outputs.json
```

JSON 里可能仍然包含敏感值，只是带有 `sensitive: true` 元信息。

考试和真实工作中都要记住：

> output 是给人或系统消费的接口，但 sensitive output 不能随便落盘、不能随便进 Git。

## 8. Terraform state 是什么

Terraform state 是 Terraform 记账本。

它记录：

- Terraform 管理了哪些资源
- 每个资源的地址是什么
- 每个资源在云上的真实 ID 是什么
- 资源当前属性是什么
- output 当前值是什么
- 依赖关系和 provider 信息

简单说：

> `.tf` 文件描述“我想要什么”，state 记录“Terraform 认为现在已经有什么”。

没有 state，Terraform 就很难知道下一次 plan 应该新增、修改还是删除资源。

## 9. 常见 state 命令

### 9.1 查看 state 里有哪些对象

```bash
terraform state list
```

示例：

```text
aws_s3_bucket.demo
aws_iam_user.lab_user
aws_iam_policy.lab_policy
aws_iam_user_policy_attachment.attach
```

这一步的价值是：你可以确认 Terraform 到底正在追踪哪些资源。

### 9.2 查看某个资源的详细 state

```bash
terraform state show aws_s3_bucket.demo
```

它会显示 Terraform state 中记录的这个资源属性。

注意：

- `state show` 不是直接查询 `.tf` 文件。
- 它展示的是 state 里记录的对象信息。
- 某些属性可能来自 provider 读取云端后的结果。

### 9.3 手动从 state 中移除资源记录

```bash
terraform state rm aws_s3_bucket.demo
```

这个命令非常危险，练习时要理解清楚：

- 它不会删除真实云资源。
- 它只是让 Terraform 不再追踪这个资源。
- 下一次 plan 时，Terraform 可能认为这个资源不存在，然后准备重新创建。

所以它适合处理“资源还在，但我不想让 Terraform 管了”的场景。

### 9.4 移动 state 地址

```bash
terraform state mv aws_s3_bucket.old aws_s3_bucket.new
```

常见场景：

- 资源从 root module 移动到 child module。
- 资源名称重构。
- 避免 Terraform 误以为旧资源要删除、新资源要创建。

现代 Terraform 更推荐在配置里使用 `moved` block 来表达这种迁移关系，但 `state mv` 仍然是重要的排障工具。

### 9.5 导入已有资源

```bash
terraform import aws_s3_bucket.demo my-existing-bucket
```

它的作用是把已经存在的资源纳入 Terraform state。

注意：

- `import` 只导入 state。
- 你仍然需要自己写对应的 `.tf` 配置。
- 导入后要运行 `terraform plan` 检查配置和真实资源是否一致。

## 10. 本节推荐操作流程

可以按下面的顺序做：

```bash
cd practice/labs/155/work
```

### 10.1 格式化配置

```bash
terraform fmt -recursive
```

目的：统一 Terraform 代码格式。

### 10.2 初始化

```bash
terraform init
```

目的：下载 provider、初始化 backend、准备工作目录。

### 10.3 校验配置

```bash
terraform validate
```

目的：检查 Terraform 配置语法和结构是否合理。

### 10.4 查看计划

```bash
terraform plan
```

目的：确认将要创建、修改或删除什么。

### 10.5 应用配置

```bash
terraform apply
```

练习环境里可以根据题目要求决定是否使用：

```bash
terraform apply -auto-approve
```

真实生产环境不建议随手 `-auto-approve`，因为 apply 是会改变基础设施的动作。

### 10.6 查看输出

```bash
terraform output
```

### 10.7 保存输出文件

```bash
terraform output > outputs.txt
terraform output -json > outputs.json
```

### 10.8 查看 state 对象

```bash
terraform state list
```

### 10.9 查看具体资源状态

```bash
terraform state show <资源地址>
```

例如：

```bash
terraform state show aws_s3_bucket.demo
```

### 10.10 执行验证脚本

如果 lab 提供了验证脚本，可以运行：

```bash
./verify.sh
```

或根据项目约定运行 PowerShell / Python 验证脚本。

### 10.11 销毁资源

```bash
terraform destroy
```

或实验环境中：

```bash
terraform destroy -auto-approve
```

## 11. 这节真正想训练你的能力

### 11.1 不是死记命令

挑战题的关键不是“背下来第几行写什么”，而是看到题目以后能拆：

- 要创建什么？
- 要输出什么？
- 输出要保存到哪里？
- 验证依据是什么？
- state 里应该出现哪些资源？
- destroy 后是否清理干净？

### 11.2 会从 state 反查资源

很多 Terraform 新手只看 `.tf` 文件，不看 state。

但真实排障经常要问：

- Terraform 到底有没有管理这个资源？
- 资源地址是什么？
- 资源是在 root module 下面，还是 child module 下面？
- 资源是不是被改名了？
- 为什么 plan 里要 destroy/create？

这些问题都离不开：

```bash
terraform state list
terraform state show <address>
```

### 11.3 会把 output 交给后续流程

Terraform 经常不是单独运行的。

它后面可能接：

- Shell 脚本
- GitHub Actions
- Jenkins
- Python 验证脚本
- Ansible
- Kubernetes 部署
- 测试框架

这些后续流程不喜欢读人类格式文本，更喜欢 JSON。

所以：

```bash
terraform output -json > outputs.json
```

这条命令在工程实践里很有用。

### 12.2 state list 相关

可能问：

> 如何查看 Terraform state 当前管理的所有资源地址？

答案：

```bash

```

### 12.3 state show 相关

可能问：

> 如何查看 state 中某个资源的详细属性？

可能问：

- `moved` block

## 13. 易错点总结

### 易错点 1：把 output 文件当成永远安全

错。

output 可能包含敏感值。尤其是 `terraform output -json`，不要无脑提交。

### 易错点 2：以为 state rm 会删除云资源

错。

`terraform state rm` 只改 Terraform state，不删除真实资源。

### 易错点 3：以为 state show 是实时云查询

不完全对。

它展示的是 Terraform state 中记录的内容。虽然 state 可能来自 provider 刷新后的结果，但它不是普通意义上的云控制台查询。

### 易错点 4：把 outputs.txt 当成 Terraform state

错。

`outputs.txt` 只是你导出的结果文件。真正的状态在 Terraform state 里。

### 易错点 5：忘记 destroy

实验环境里一定养成清理习惯。哪怕是 LocalStack，也要保持工作目录闭环。

## 15. 本节复盘问题

### 问题 1：为什么要保存 Terraform output？

为了把部署后的关键结果保存下来，供提交、审计、后续脚本或验证流程使用。

### 问题 2：什么时候用 `terraform output -json`？

当输出需要被程序、脚本、CI/CD 或 `jq` 读取时，使用 JSON 格式更合适。

### 问题 3：什么时候用 `terraform output -raw`？

当你只需要单个输出值，并希望它作为纯字符串进入 shell 变量或文件时，用 `-raw`。

### 问题 4：`terraform state list` 的作用是什么？

列出当前 state 中 Terraform 正在管理的所有资源地址。

### 问题 5：`terraform state rm` 和 `terraform destroy` 有什么区别？

- `state rm`：只从 state 中移除记录，不删除真实资源。
- `destroy`：根据 state 删除 Terraform 管理的真实资源。

这是非常高频的理解点。

## 17. 本节结论

第 155 节的重点是把 Terraform 从“能 apply”推进到“能交付结果、能检查状态、能排查问题”。

真正的 Terraform 工程能力不是只会写资源块，而是知道资源部署后如何输出、如何验证、如何被 state 追踪，以及如何安全清理。
