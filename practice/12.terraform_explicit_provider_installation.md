# Terraform 实操训练 12：显式 Provider Installation

## 1. 课程定位

本节是上一节 **File System Mirror** 的进阶版。

第 11 节重点是：

> Terraform 可以通过本地 file system mirror 安装 provider，从而适配离线或受限网络环境。

第 12 节重点是：

> Terraform 可以通过 `provider_installation` 显式控制 provider 的安装来源，并且可以配置 mirror 优先、direct 回退。

也就是说，这节不是单纯讲“从 mirror 安装 provider”，而是讲 **Terraform provider 安装源的选择策略**。

---

## 2. 核心概念

Terraform 默认会从官方 registry 下载 provider，例如：

```text
registry.terraform.io/hashicorp/local
registry.terraform.io/hashicorp/aws
registry.terraform.io/hashicorp/azurerm
```

但是在企业环境中，runner 不一定能直接访问公网 registry。

常见原因包括：

```text
1. CI runner 位于内网，不能访问公网
2. 企业安全策略禁止直接访问外部 registry
3. 需要提前审计 provider 包
4. 需要在 air-gapped 环境中运行 Terraform
5. 需要稳定、可控、可复现的 provider 安装来源
```

这时就需要显式配置：

```hcl
provider_installation {
  filesystem_mirror {
    path    = "/workspace/practice/labs/12/mirror"
    include = ["registry.terraform.io/hashicorp/*"]
  }

  direct {
    exclude = ["registry.terraform.io/hashicorp/*"]
  }
}
```

或者在允许联网的开发环境中，使用 mirror 优先、direct 回退：

```hcl
provider_installation {
  filesystem_mirror {
    path    = "/workspace/practice/labs/12/mirror"
    include = ["registry.terraform.io/hashicorp/*"]
  }

  direct {}
}
```

---

## 3. `filesystem_mirror` 是什么？

`filesystem_mirror` 表示：

> Terraform 不直接去公网 registry 下载 provider，而是从一个本地目录中查找 provider 包。

例如本节命令：

```sh
terraform providers mirror /workspace/practice/labs/12/mirror
```

这条命令会把当前配置需要的 provider 下载到：

```text
/workspace/practice/labs/12/mirror
```

之后再通过 CLI 配置文件告诉 Terraform：

```text
初始化时优先从这个 mirror 目录安装 provider。
```

整体流程是：

```text
terraform providers mirror
        ↓
生成本地 mirror 目录
        ↓
TF_CLI_CONFIG_FILE 指向 terraform-cli.rc
        ↓
terraform init
        ↓
Terraform 从 mirror 安装 provider
```

---

## 4. `provider_installation` 是什么？

`provider_installation` 是 Terraform CLI 配置文件中的一个配置块，用来显式控制 provider 的安装来源。

它可以包含多种安装方式：

```text
filesystem_mirror
network_mirror
direct
```

本节重点是：

```text
filesystem_mirror + direct
```

含义如下：

| 配置项 | 作用 |
|---|---|
| `filesystem_mirror` | 从本地文件系统 mirror 安装 provider |
| `direct` | 允许 Terraform 回到 origin registry 直接下载 provider |
| `include` | 指定哪些 provider 使用该安装源 |
| `exclude` | 排除哪些 provider，不使用该安装源 |

---

## 5. `direct` 回退是什么意思？

`direct` 表示：

> 如果 mirror 中没有 provider，Terraform 可以直接访问官方 registry 下载。

这在开发环境中很方便：

```text
开发者本机：
  优先使用 mirror
  mirror 缺包时可以 direct 下载
```

但是在真正的 air-gapped 环境中，不能依赖 direct。

因为 air-gapped 环境本来就不能访问公网：

```text
mirror 缺 provider
        ↓
Terraform 尝试 direct
        ↓
公网不可达
        ↓
terraform init 失败
```

所以生产判断很重要：

```text
允许联网的开发环境：
  可以使用 direct 回退

严格离线 / air-gapped 环境：
  不应该依赖 direct 回退
  mirror 必须提前准备完整
```

这就是本节最核心的边界意识。

```text

```

```text
/workspace/practice/labs/12/mirror
```



### 使用指定 CLI 配置文件初始化

```sh
TF_CLI_CONFIG_FILE=/workspace/practice/labs/12/terraform-cli.rc terraform init -input=false
```

这是本节最关键的一步。

`TF_CLI_CONFIG_FILE` 表示：

```text
不要使用默认 Terraform CLI 配置文件
改用我指定的 terraform-cli.rc
```

也就是说，Terraform 会读取：

```text
/workspace/practice/labs/12/terraform-cli.rc
```

然后根据里面的 `provider_installation` 配置决定 provider 从哪里安装。

`-input=false` 表示：

```text
禁止交互输入，适合 CI 环境。
```

因为 file system mirror 场景下，Terraform 可能提示 lock file 中的 checksum 只适用于当前平台，或者 provider 安装来源不是直接来自官方 registry。

这时需要结合企业的 provider 校验流程处理。

## 10. 为什么会出现 provider 未认证提示？

通过本地 mirror 安装 provider 时，Terraform 可能显示类似：

```text
unauthenticated
```

这不是说 provider 一定有问题，而是表示：

```text
Terraform 没有通过默认 registry 的标准签名验证流程完成认证展示。
```

在企业环境中，常见做法是：

```text
1. mirror 生成前由安全团队审计 provider
2. provider 包进入内部制品库或文件系统 mirror
3. CI runner 只允许从内部 mirror 安装
4. 使用 lock file checksum 校验 provider 内容
```

也就是说，企业把“信任链”从公网 registry 转移到了内部供应链流程。

---

## 11. `filesystem_mirror` 和 plugin cache 的区别

这点很重要。

| 项目 | Plugin Cache | File System Mirror |
|---|---|---|
| 目的 | 减少重复下载 | 控制 provider 安装来源 |
| 场景 | CI 性能优化 | 离线、内网、受控网络 |
| 是否替代 registry | 不替代 | 可以替代 |
| 是否适合 air-gapped | 不够 | 适合 |
| 是否需要提前准备 provider | 通常不需要 | 需要 |
| 生产价值 | 中低 | 较高，尤其受控网络 |

一句话：

```text
plugin cache 是加速下载；
filesystem mirror 是控制下载源。
```

## 12. `direct` 的正确使用边界

### 可以使用 direct 的场景

```text
开发者本机
普通联网 CI
非严格隔离环境
mirror 只是优化项
```

配置可以是：

```hcl
provider_installation {
  filesystem_mirror {
    path = "/path/to/mirror"
  }

  direct {}
}
```

意思是：

```text
先看 mirror
mirror 没有，就去官方 registry
```

---

### 不应该依赖 direct 的场景

```text
air-gapped 环境
涉密环境
严格受控 CI runner
不允许公网访问的企业内网
```

这时应该避免 direct 回退。

更严格的策略是：

```hcl
provider_installation {
  filesystem_mirror {
    path    = "/path/to/mirror"
    include = ["registry.terraform.io/hashicorp/*"]
  }

  direct {
    exclude = ["registry.terraform.io/hashicorp/*"]
  }
}
```

意思是：

```text
hashicorp provider 必须从 mirror 安装
不允许 direct 回到公网下载
```

## 13. 排障指南

### 1. mirror 目录为空

问题：

```text
terraform init 找不到 provider
```

原因：

```text
没有执行 terraform providers mirror
或者 mirror 路径不对
```

解决：

```sh
terraform providers mirror /workspace/practice/labs/12/mirror
```

---

### 2. `TF_CLI_CONFIG_FILE` 路径错误

问题：

```text
Terraform 没有读取 terraform-cli.rc
```

解决：

```sh
TF_CLI_CONFIG_FILE=/workspace/practice/labs/12/terraform-cli.rc terraform init -input=false
```

在 Docker 中要注意：

```text
路径必须是容器内路径，不是宿主机路径。
```

---

### 3. direct 被误用

问题：

```text
以为自己是离线安装
实际 mirror 缺包后 Terraform 又去公网下载
```

解决：

```text
严格 air-gapped 环境中，不要依赖 direct 回退。
使用 direct exclude 明确禁止目标 provider 走公网。
```

---

### 4. provider 未认证提示

问题：

```text
Terraform init 显示 provider 从 mirror 安装，签名状态为 unauthenticated
```

处理方式：

```text
不要简单理解为失败。
需要结合企业内部 provider 校验、lock file checksum 和供应链安全流程判断。
```

---

### 5. 多平台 checksum 警告

问题：

```text
Linux CI 可以用
Windows/macOS 开发者失败
```

解决：

```sh
terraform providers lock   -platform=linux_amd64   -platform=windows_amd64   -platform=darwin_amd64   -platform=darwin_arm64
```

---

## 14. 生产级实践建议

生产中可以这样设计：

```text
开发环境：
  filesystem_mirror 优先
  direct 允许回退
  方便开发者缺包时自动下载

CI 环境：
  尽量使用 mirror
  是否允许 direct 取决于网络和安全要求

air-gapped 环境：
  mirror 必须提前完整准备
  不依赖 direct
  provider 包需要内部审计和校验
  lock file 需要提交并维护

多平台团队：
  使用 terraform providers lock -platform=... 补齐 checksums
```
