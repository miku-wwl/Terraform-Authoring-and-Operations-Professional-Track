# Terraform 考试训练 121：创建 Organization、Project 与 Workspace（讲解版）

## 1. 本节核心：不是背 UI，而是理解对象关系

这一节的主题是 HCP Terraform 的基础对象创建流程：

```text
Organization -> Project -> Workspace
```

考试不会重点考“按钮在哪里点”，而是考你是否理解：

- Organization 是什么
- Project 是什么
- Workspace 是什么
- 创建 Workspace 时为什么要选 workflow
- 不同 workflow 适合什么团队协作场景

你要把它理解成 HCP Terraform 的平台建模题，而不是普通的 UI 操作题。

---

## 2. 三个对象的层级关系

可以这样看：

```text
Organization
└── Project
    └── Workspace
        ├── Terraform configuration
        ├── State
        ├── Variables
        ├── Runs
        ├── Plans
        ├── Applies
        ├── Credentials
        ├── Permissions
        └── Policies
```

一句话记忆：

> Organization 管组织边界，Project 管分组，Workspace 管一套 Terraform 运行单元。

---

## 3. Organization 是什么？

Organization 是 HCP Terraform 里的顶层容器。

它通常对应：

- 一个公司
- 一个团队
- 一个部门
- 一个实验账号
- 一个学习环境

Organization 下面可以有多个 Project。  
Project 下面再创建多个 Workspace。

可以粗暴理解：

```text
Organization = HCP Terraform 里的公司大院
```

团队成员、权限、运行记录、项目分组，都会放在这个组织边界里面管理。

---

## 4. Project 是什么？

Project 是 Organization 下面的分组对象。

它用来组织多个 Workspace。

例如：

```text
Organization: cloud-platform
├── Project: networking
├── Project: kubernetes
├── Project: security
└── Project: application
```

也可以按环境划分：

```text
Organization: cloud-platform
├── Project: dev
├── Project: staging
└── Project: prod
```

创建 Organization 后，HCP Terraform 通常会提供默认 Project。  
学习时可以用默认 Project，但生产环境最好按团队、系统、环境或权限边界规划 Project。

默认 Project 不是不能用，而是不能把所有东西都无脑塞进去。  
不然后面 workspace 一多，就会像垃圾桶一样难管理。

---

## 5. Workspace 是什么？

Workspace 是 HCP Terraform 里真正干活的核心单元。

一个 Workspace 通常对应：

```text
一套 Terraform 配置 + 一份 state + 一组 run 历史
```

例如：

```text
Project: networking
├── Workspace: vpc-dev
├── Workspace: vpc-staging
└── Workspace: vpc-prod
```

再比如：

```text
Project: kubernetes
├── Workspace: aks-dev
├── Workspace: aks-prod
├── Workspace: eks-dev
└── Workspace: eks-prod
```

Workspace 管的不只是 state，还包括：

- Terraform 代码来源
- Terraform 版本
- variables
- cloud credentials
- plan / apply run history
- 审批流程
- policy check
- workspace 权限
- run trigger 方式

所以创建 Workspace 不是“建一个空壳”。  
它是在定义一套基础设施交付流水线。

---

## 6. 创建顺序怎么记？

标准顺序：

```text
1. 创建 Organization
2. 进入默认 Project 或创建自定义 Project
3. 在 Project 里创建 Workspace
4. 为 Workspace 选择 workflow
5. 配置变量、凭据、Terraform 版本、权限和审批策略
```

考试里可以记成：

```text
先 Org，后 Project，最后 Workspace。
```

Workspace 不能脱离 Organization 存在。  
Project 是 Workspace 的分组容器。

---

## 7. Workspace 创建时最关键的配置：Workflow

创建 HCP Terraform Workspace 时，非常重要的一步是选择 workflow 类型。

常见类型有：

```text
Version Control Workflow
CLI-driven Workflow
API-driven Workflow
```

workflow 决定三个问题：

```text
代码从哪里来？
run 怎么触发？
谁来驱动 Terraform 执行？
```

这就是考试最爱考的点。

---

## 8. Version Control Workflow

## 8.1 是什么？

Version Control Workflow 是 Git 驱动的工作流。

通常连接：

- GitHub
- GitLab
- Bitbucket
- Azure DevOps

典型流程：

```text
开发者提交代码
-> 创建 Pull Request / Merge Request
-> HCP Terraform 自动触发 plan
-> 团队 review plan
-> merge 后触发 apply 或人工审批 apply
-> run history 被记录
```

## 8.2 适合什么场景？

适合团队协作和生产环境。

尤其适合这些要求：

- 所有基础设施变更必须走 Git
- 所有变更必须经过 PR
- 希望自动触发 plan
- 希望有审计记录
- 希望团队 review 后再 apply
- 希望基础设施变更和代码历史绑定

考试关键词：

```text
Git
VCS
Pull Request
Merge Request
automatic plan
code review
audit
team collaboration
```

看到这些词，优先选：

```text
Version Control Workflow
```

## 8.3 场景题标准回答

题目：

> 团队要求所有变更必须经过 Git PR，再由平台自动 plan。你会选择哪种 workflow？

答案：

```text
Version Control Workflow
```

原因：

```text
它可以从 Git 仓库获取 Terraform 代码，并基于 PR / merge 等事件触发 run，适合团队协作、代码审查和审计。
```

---

## 9. CLI-driven Workflow

## 9.1 是什么？

CLI-driven Workflow 适合继续使用本地 Terraform CLI，但把 run 和 state 交给 HCP Terraform 管理。

典型流程：

```text
本地写 Terraform
-> 本地执行 terraform plan / terraform apply
-> run 进入 HCP Terraform
-> state 由 HCP Terraform 管理
-> run history 可追踪
```

也就是说，用户仍然保留命令行习惯，但执行结果进入远端平台。

## 9.2 适合什么场景？

适合：

- 个人学习者
- 熟悉 Terraform CLI 的工程师
- 小团队
- 从本地 backend 迁移到 HCP Terraform 的场景
- 暂时不想接 Git 自动化的场景

考试关键词：

```text
local CLI
terraform plan
terraform apply
manual command
remote run
remote state
keep existing CLI workflow
```

看到这些词，优先选：

```text
CLI-driven Workflow
```

## 9.3 场景题标准回答

题目：

> 个人学习者想保留本地 Terraform CLI 操作习惯，但希望 run 发生在 HCP Terraform。你会推荐哪种 workflow？

答案：

```text
CLI-driven Workflow
```

原因：

```text
CLI-driven Workflow 允许用户继续使用本地 Terraform CLI，同时让状态和运行记录进入 HCP Terraform 管理。
```

---

## 10. API-driven Workflow

## 10.1 是什么？

API-driven Workflow 是通过 API 触发 HCP Terraform run。

它更适合自研平台或高度定制化自动化系统。

典型流程：

```text
内部平台 / 自研系统
-> 调用 HCP Terraform API
-> 创建或触发 run
-> 查询 plan / apply 结果
-> 把结果返回给平台用户
```

## 10.2 适合什么场景？

适合：

- 自研 DevOps 平台
- 内部 IaC 自服务平台
- 外部系统统一调度 Terraform
- 不想直接绑定 Git 仓库
- 需要高度定制化触发逻辑

考试关键词：

```text
API
custom platform
internal portal
external orchestrator
automation system
self-service platform
```

看到这些词，可以考虑：

```text
API-driven Workflow
```

---

## 11. 三种 Workflow 对比表

| Workflow | 代码来源 | 谁触发 Run | 典型场景 | 考试关键词 |
|---|---|---|---|---|
| Version Control Workflow | Git 仓库 | Git PR / merge / VCS 事件 | 团队协作、PR 审查、自动 plan | Git、VCS、PR、audit |
| CLI-driven Workflow | 本地目录 | 本地 Terraform CLI | 个人学习、命令行习惯、渐进迁移 | CLI、local command、remote state |
| API-driven Workflow | API 上传或外部系统提供 | 外部系统调用 API | 自研平台、内部自动化系统 | API、custom platform、orchestrator |

一句话判断：

```text
Git 驱动选 VCS。
人手敲命令选 CLI-driven。
系统调系统选 API-driven。
```

---

## 12. 不要混淆：HCP Terraform Workspace vs Terraform CLI Workspace

Terraform CLI 也有 workspace：

```sh
terraform workspace list
terraform workspace new dev
terraform workspace select prod
```

但这个和 HCP Terraform Workspace 不是一回事。

| 概念 | 属于哪里 | 作用 |
|---|---|---|
| Terraform CLI workspace | Terraform CLI / backend | 在同一套配置下切换不同 state |
| HCP Terraform Workspace | HCP Terraform 平台 | 管理配置、state、run、变量、权限、workflow |

本节说的是 HCP Terraform 的 Workspace。  
不是让你执行 `terraform workspace new`。

这点很重要，考试容易用名字混你。

---

## 13. 创建 Workspace 不是只点按钮

生产环境里，创建 Workspace 前应该思考：

```text
1. 这套 workspace 管什么资源？
2. 它属于哪个 Project？
3. 它用什么 workflow？
4. 代码从哪里来？
5. Terraform 版本是否固定？
6. state 是否由 HCP Terraform 管理？
7. variables 怎么配置？
8. AWS/Azure/GCP 凭据怎么注入？
9. 谁能 plan？
10. 谁能 apply？
11. 是否需要审批？
12. 是否需要 policy check？
13. 是否需要和其他 workspace 建依赖？
```

所以 Workspace 是治理单元，不只是执行单元。

## 14. 常见考试陷阱

## 14.1 创建 Organization 后就一定连接了 GitHub？

错。

创建 Organization 只是创建了顶层组织边界。  
连接 GitHub / GitLab 等 VCS 是后续配置，通常发生在 VCS workflow 或 VCS provider 配置过程中。

判断题：

```text
创建 organization 后就一定已经连接了 GitHub 仓库。
```

答案：

```text
错
```

---

## 14.2 创建 Workspace 时选择的 workflow 会影响代码来源和 run 触发方式？

对。

workflow 决定：

```text
代码从哪里来
run 怎么触发
谁来驱动执行
```

判断题：

```text
创建 workspace 时选择的 workflow 会影响代码来源和 run 触发方式。
```

答案：

```text
对
```

---

## 14.3 默认 Project 可以帮助新用户快速开始，但生产中仍应规划 Project？

对。

默认 Project 适合学习和快速开始。  
但生产环境要考虑团队边界、权限边界、系统边界和环境边界。

判断题：

```text
默认 project 可以帮助新用户快速开始，但生产中仍应根据组织结构规划 project。
```

答案：

```text
对
```

## 17. 生产环境中的对象设计示例

一个比较合理的结构可能是：

```text
Organization: company-cloud
├── Project: networking
│   ├── Workspace: vpc-dev
│   ├── Workspace: vpc-staging
│   └── Workspace: vpc-prod
├── Project: kubernetes
│   ├── Workspace: aks-dev
│   ├── Workspace: aks-prod
│   ├── Workspace: eks-dev
│   └── Workspace: eks-prod
└── Project: security
    ├── Workspace: iam-baseline
    └── Workspace: policy-as-code
```

这样设计的好处是：

- Project 对应清楚的治理边界
- Workspace 对应清楚的 state 边界
- 不同环境可以分开 plan/apply
- 权限可以更细粒度控制
- 审计和排障更容易
- blast radius 更小

blast radius 的意思是“爆炸半径”。  
也就是一次错误变更会影响多大范围。

Workspace 边界设计得好，事故影响面就更可控。

## 18. 和你做平台工程有什么关系？

这节虽然看起来像 HCP Terraform 入门，但其实是平台工程基础。

因为平台团队要考虑：

```text
谁能创建 workspace？
谁能触发 plan？
谁能 approve apply？
变量和凭据怎么统一管理？
生产环境和开发环境怎么隔离？
Git PR 怎么接入 Terraform run？
哪些 workspace 要挂 policy？
哪些 workspace 属于哪个团队？
```

这些问题不是按钮问题，是治理问题。

Terraform 到了生产级，重点就不只是：

```sh
terraform apply
```

而是：

```text
如何让团队安全、可审计、可重复地 apply。
```

HCP Terraform 的 Organization / Project / Workspace 就是在解决这个问题。

## 21. 最终总结

本节最重要的三句话：

```text
1. HCP Terraform 的基础层级是 Organization -> Project -> Workspace。
2. Workspace 是管理 Terraform 配置、state、run、变量、权限和 workflow 的核心单元。
3. Workflow 决定代码来源和 run 触发方式：Git 协作用 VCS，本地命令用 CLI，自研平台用 API。
```

考试遇到这类题，不要被 UI 操作干扰。  
先判断场景：

```text
团队 Git PR 自动 plan？选 Version Control Workflow。
个人保留本地 terraform 命令？选 CLI-driven Workflow。
内部平台通过接口触发？选 API-driven Workflow。
```

这就是本节的解题主线。
