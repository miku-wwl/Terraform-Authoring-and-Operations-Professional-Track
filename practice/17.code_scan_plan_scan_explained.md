# Terraform 实操训练 17：Code Scan 与 Plan Scan 讲解版

## 0. 这一节到底在学什么？

这一节的核心不是“怎么跑 Checkov 命令”，而是理解一个非常重要的安全扫描差异：

> **源码扫描看到的是你写在 `.tf` 文件里的内容；Plan 扫描看到的是 Terraform 计算之后准备真正执行的结果。**

也就是说，源码扫描像是“看菜谱”，Plan 扫描像是“看真正要端上桌的菜”。

在真实企业 CI/CD 里，这个差异非常关键。因为很多危险配置并不是直接写死在源码里，而是通过变量、环境参数、pipeline 参数、workspace 参数、tfvars 文件等方式注入进去的。

---

## 1. 背景场景

平台团队已经在代码提交阶段接入了 Checkov，用来扫描 Terraform 源码。

这很好，但还不够。

因为 Terraform 的最终资源配置可能来自多个地方：

- `main.tf` 里的默认值；
- `variables.tf` 里的变量；
- `terraform.tfvars`；
- CI/CD 里的 `-var` 参数；
- workspace 或环境变量；
- module 默认值；
- remote state 或 data source。

所以，一个安全问题可能在源码里看起来没问题，但在 plan 阶段才真正暴露出来。

本 Lab 用 Kubernetes Pod 的 `host_network` 配置来演示这个问题。

---

## 2. 核心知识点：源码扫描和 Plan 扫描的区别

## 2.1 Code Scan：源码扫描

源码扫描通常扫描的是 Terraform HCL 文件，例如：

```bash
checkov -d /tf --framework terraform
```

它看到的是源码结构，比如：

```hcl
host_network = var.host_network_enabled
```

源码扫描能发现很多问题，例如：

- 明文密码；
- 安全组开放过大；
- 加密没有开启；
- Kubernetes 容器权限过高；
- IAM 权限过宽。

但是它有一个天然限制：

> 如果危险值来自变量注入，源码扫描不一定知道最终变量值是多少。

比如源码里只是写了：

```hcl
host_network = var.host_network_enabled
```

源码扫描只能看到这个字段引用了变量。它不一定知道最终运行时这个变量会被传成 `true` 还是 `false`。

---

## 2.2 Plan Scan：Plan 扫描

Plan 扫描扫描的是 Terraform 生成后的 plan JSON，例如：

```bash
terraform plan -out=tfplan
terraform show -json tfplan > plan.json
checkov -f plan.json --framework terraform_plan
```

Plan 阶段 Terraform 已经完成了变量计算、表达式展开、module 解析和资源变更计算。

所以 Checkov 扫描 plan 的时候，看到的更接近真实执行结果。

比如 CI 中传入：

```bash
terraform plan -out=tfplan -var 'host_network_enabled=true'
```

那么 plan JSON 中就会出现最终计算结果：

```json
"host_network": true
```

这个时候 Plan 扫描就能发现风险。

---

## 3. 为什么 `host_network = true` 是风险？

在 Kubernetes 里，Pod 默认使用自己的网络命名空间。

如果开启：

```hcl
host_network = true
```

Pod 就会使用宿主机节点的网络命名空间。

这会带来更高风险：

- Pod 可以直接暴露在节点网络上；
- 容器和宿主机网络隔离变弱；
- 某些端口冲突或监听行为更危险；
- 攻击面扩大；
- 不符合多数 Kubernetes 安全基线。

所以 Checkov 里有对应的 Kubernetes 安全检查，例如本实验聚焦的：

```text
CKV_K8S_19
```

你可以把它理解成：

> 不要随便让 Pod 使用宿主机网络。

---

## 4. 本实验的关键设计

本 Lab 的设计非常聪明：它不是直接把危险配置写死在源码里。

而是这样设计：

```hcl
host_network = var.host_network_enabled
```

然后在 plan 阶段通过参数注入：

```bash
-var 'host_network_enabled=true'
```

这就模拟了真实 CI/CD 里的一个问题：

> 源码本身可能看着正常，但 pipeline 参数可以把它变成危险配置。

这也是为什么成熟团队不会只做 Code Scan，而会在变更审批流程中加入 Plan Scan。

## 6. 实验命令讲解

## 6.1 进入实验目录

```bash
cd /workspace/practice/labs/17
```

这一步进入 Lab 17 的工作目录。

---

## 6.2 初始化 Terraform

```bash
terraform init -input=false
```

作用：

- 下载 provider；
- 初始化 `.terraform` 目录；
- 准备后续 validate、test、plan。

`-input=false` 的意思是不允许 Terraform 交互式提问。

在 CI 里这个参数很重要，因为 CI 不能像人一样输入 yes/no 或变量值。

---

## 6.3 检查格式

```bash
terraform fmt -check
```

作用：

- 检查 `.tf` 文件格式是否符合 Terraform 标准；
- 只检查，不自动修改。

如果格式不符合，命令会失败。

本地开发时可以用：

```bash
terraform fmt
```

自动修复格式。

---

## 6.4 校验语法和配置

```bash
terraform validate
```

作用：

- 检查 Terraform 配置是否合法；
- 检查 provider schema；
- 检查资源字段是否写错；
- 检查变量、引用、表达式是否基本成立。

注意：`validate` 不会真的创建资源，也不会像 plan 一样完整计算所有变更。

---

## 6.5 运行 Terraform test

```bash
terraform test
```

作用：

- 运行 `tests/plan_scan.tftest.hcl`；
- 验证实验配置是否符合预期；
- 把这个 Lab 的关键设计固化成自动化测试。

这一步很像单元测试。

区别是它测试的是 Terraform 配置和模块行为，而不是普通应用代码。

---

## 6.6 生成 plan 文件

```bash
terraform plan -input=false -no-color -out=tfplan -var 'host_network_enabled=true'
```

这条命令是整个实验的关键。

参数解释：

| 参数 | 含义 |
|---|---|
| `-input=false` | 禁止交互输入，适合 CI |
| `-no-color` | 去掉彩色输出，方便日志保存 |
| `-out=tfplan` | 把 plan 保存成二进制文件 |
| `-var 'host_network_enabled=true'` | 在 plan 阶段注入危险变量值 |

注意这里没有执行 `apply`。

也就是说，本实验只是生成计划，不需要真的创建 Kubernetes Pod，也不需要真实 Kubernetes 集群。

---

## 6.7 把二进制 plan 转成 JSON

```bash
terraform show -json tfplan > plan.json
```

`tfplan` 是 Terraform 的二进制 plan 文件，不能直接给普通工具读。

所以要先转成 JSON：

```text
tfplan  →  plan.json
```

Checkov 的 plan 扫描就是基于这个 JSON 文件完成的。

## 7. 源码扫描命令讲解

PowerShell 写法：

```powershell
docker run --rm `
  -v "${PWD}/practice/labs/17:/tf" `
  bridgecrew/checkov:latest `
  -d /tf --framework terraform --check CKV_K8S_19 --soft-fail --compact
```

如果你在 Linux/macOS shell 里运行，可以写成：

```bash
docker run --rm \
  -v "$PWD:/tf" \
  bridgecrew/checkov:latest \
  -d /tf --framework terraform --check CKV_K8S_19 --soft-fail --compact
```

参数解释：

| 参数 | 含义 |
|---|---|
| `docker run --rm` | 临时运行容器，结束后自动删除 |
| `-v "$PWD:/tf"` | 把当前目录挂载进容器的 `/tf` |
| `bridgecrew/checkov:latest` | 使用 Checkov 容器镜像 |
| `-d /tf` | 扫描目录 |
| `--framework terraform` | 按 Terraform 源码扫描 |
| `--check CKV_K8S_19` | 只运行指定规则 |
| `--soft-fail` | 即使发现问题也不让命令失败 |
| `--compact` | 输出更简洁 |

这个命令看的是 Terraform 源码。

因为源码里只是变量引用，所以源码扫描不一定能看到最终危险值。

---

## 8. Plan 扫描命令讲解

PowerShell 写法：

```powershell
docker run --rm `
  -v "${PWD}/practice/labs/17:/tf" `
  bridgecrew/checkov:latest `
  -f /tf/plan.json --framework terraform_plan --check CKV_K8S_19 --soft-fail --compact
```

Linux/macOS shell 写法：

```bash
docker run --rm \
  -v "$PWD:/tf" \
  bridgecrew/checkov:latest \
  -f /tf/plan.json --framework terraform_plan --check CKV_K8S_19 --soft-fail --compact
```

参数解释：

| 参数 | 含义 |
|---|---|
| `-f /tf/plan.json` | 扫描指定 plan JSON 文件 |
| `--framework terraform_plan` | 按 Terraform plan 结果扫描 |
| `--check CKV_K8S_19` | 只检查宿主机网络相关规则 |
| `--soft-fail` | 实验中不阻断流程 |
| `--compact` | 简化输出 |

这个命令看的不是源码，而是 Terraform 计算后的 plan。

由于前面已经注入：

```bash
-var 'host_network_enabled=true'
```

所以 plan 扫描能看到：

```text
host_network = true
```

于是 Checkov 会报告失败。

## 9. 预期实验结果

本实验的预期结果是：

| 扫描方式 | 扫描对象 | 是否看到最终变量值 | 结果 |
|---|---|---:|---|
| Code Scan | `.tf` 源码 | 不一定 | `CKV_K8S_19` 不失败 |
| Plan Scan | `plan.json` | 能看到 | `CKV_K8S_19` 失败 |

更直白地说：

> 源码扫描：没看到刀。  
> Plan 扫描：发现刀已经被塞进包里了。

这就是 Plan Scan 的价值。

## 10. 为什么企业 CI/CD 里需要两个都做？

Code Scan 和 Plan Scan 不是互相替代关系，而是互补关系。

## 10.1 Code Scan 适合早期反馈

适合放在：

- pre-commit；
- pull request；
- 开发者本地检查；
- 代码评审前置检查。

优点：

- 快；
- 不需要完整 plan；
- 不需要真实环境变量；
- 可以尽早发现明显问题。

缺点：

- 对运行时变量、动态表达式、复杂 module 展开不一定准确。

---

## 10.2 Plan Scan 适合变更审批

适合放在：

- PR 合并前；
- 环境部署前；
- change approval gate；
- production apply 前。

优点：

- 看到更真实的资源变更；
- 能发现变量注入后的风险；
- 更适合生产审批。

缺点：

- 要先跑 `terraform plan`；
- 对环境、backend、变量、provider 初始化依赖更多；
- 比源码扫描重。

---

## 11. 推荐的 CI/CD 流水线顺序

一个比较合理的 Terraform 安全扫描流水线可以这样设计：

```text
1. terraform fmt -check
2. terraform validate
3. checkov code scan
4. terraform plan -out=tfplan
5. terraform show -json tfplan > plan.json
6. checkov plan scan
7. 人工审批 plan
8. terraform apply tfplan
```

这里有两个重点：

第一，Code Scan 提供快速反馈。

第二，Plan Scan 保护真正要执行的变更。

如果是生产环境，最好不要审批后重新跑裸 `terraform apply`。

更稳的方式是：

```bash
terraform plan -out=tfplan
terraform apply tfplan
```

这样审批内容和执行内容才是一致的。

## 12. 常见坑

## 12.1 忘记传变量

如果你没有传：

```bash
-var 'host_network_enabled=true'
```

那 plan 里可能不会出现危险值，Plan Scan 也就不会按实验预期失败。

---

## 12.2 忘记生成 plan.json

Checkov 不能直接扫描 `tfplan` 二进制文件。

必须先执行：

```bash
terraform show -json tfplan > plan.json
```

---

## 12.3 把源码扫描和 plan 扫描框架写混

源码扫描用：

```bash
--framework terraform
```

Plan 扫描用：

```bash
--framework terraform_plan
```

这两个别写反。

---

## 12.4 Docker 挂载路径不对

容器里的路径是：

```text
/tf
```

如果你本机路径没挂进去，Checkov 就会报找不到文件。

PowerShell 常见写法是：

```powershell
-v "${PWD}/practice/labs/17:/tf"
```

Linux/macOS 常见写法是：

```bash
-v "$PWD:/tf"
```

具体取决于你是在 repo 根目录还是 lab 目录里运行。

---

## 12.5 `--soft-fail` 的含义别误解

`--soft-fail` 不是说没有问题。

它只是说：

> 即使发现问题，也让命令退出码保持成功。

实验里使用 `--soft-fail` 是为了让流程继续跑完。

生产 CI 里，如果你想让高风险规则阻断部署，就应该去掉 `--soft-fail`。

## 14. 本 Lab 最终你应该记住什么？

这节课可以总结成三句话：

1. **Code Scan 适合早期发现源码里的明显问题。**
2. **Plan Scan 适合发现变量注入、module 展开、表达式计算后的真实风险。**
3. **生产 Terraform 流水线里，安全扫描应该尽量靠近真正 apply 的内容。**

最重要的一句话：

> **不要只相信源码，生产审批要看 plan。**
