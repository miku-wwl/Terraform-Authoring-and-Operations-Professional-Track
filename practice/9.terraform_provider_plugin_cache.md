# Terraform 实操训练 9：Provider Plugin Cache 概念

## 1. 这节课在讲什么

这一节讲的是 **Terraform provider plugin cache**。

简单说：Terraform 每次执行 `terraform init` 时，都需要准备 provider plugin，例如：

- `hashicorp/local`
- `hashicorp/aws`
- `hashicorp/azurerm`
- `hashicorp/random`

如果每个 Terraform 项目都从 Registry 重新下载一遍 provider，就会浪费：

- CI 执行时间；
- 网络带宽；
- Registry 请求次数；
- 开发者本地等待时间。

Provider Plugin Cache 的作用就是：**把下载过的 provider plugin 放到一个共享缓存目录里，下次其他项目需要同版本 provider 时，Terraform 可以复用缓存，而不是重新下载。**

可以把它理解成 Terraform provider 的“本地仓库缓存”。

---

## 2. 为什么 `terraform init` 会慢

Terraform 项目通常会在 `required_providers` 里声明 provider，例如：

```hcl
terraform {
  required_providers {
    local = {
      source  = "hashicorp/local"
      version = "~> 2.5"
    }
  }
}
```

执行：

```sh
terraform init
```

Terraform 会做几件事：

1. 读取当前目录下的 Terraform 配置；
2. 解析需要哪些 provider；
3. 根据 `.terraform.lock.hcl` 锁定的版本和校验和，确定要安装哪个 provider；
4. 从 Terraform Registry 下载 provider plugin；
5. 安装到当前项目的 `.terraform/providers/` 目录中。

如果没有 cache，不同项目即使使用相同版本 provider，也可能重复下载。

这在 CI 里尤其明显，因为 CI runner 经常是一次性的，每次 pipeline 都是干净环境。

---

## 3. Provider Plugin Cache 解决什么问题

Provider Plugin Cache 解决的是：

> 多个 Terraform 项目重复下载相同 provider plugin 的问题。

配置方式是设置环境变量：

```sh
export TF_PLUGIN_CACHE_DIR=/workspace/practice/labs/9/.terraform-plugin-cache
```

然后执行：

```sh
terraform init -input=false
```

Terraform 会把下载到的 provider plugin 缓存在 `TF_PLUGIN_CACHE_DIR` 指定的目录里。

下次再有项目需要同一个 provider 版本时，Terraform 会优先使用缓存。

---

## 4. 这节实验的核心命令

实验运行手册是：

```sh
cd /workspace/practice/labs/9
mkdir -p .terraform-plugin-cache
export TF_PLUGIN_CACHE_DIR=/workspace/practice/labs/9/.terraform-plugin-cache
terraform init -input=false
terraform fmt -check
terraform validate
terraform test
terraform plan -input=false -no-color -out=tfplan
terraform apply -auto-approve tfplan
terraform output
terraform destroy -auto-approve
```

这里最关键的是前两步：

```sh
mkdir -p .terraform-plugin-cache
export TF_PLUGIN_CACHE_DIR=/workspace/practice/labs/9/.terraform-plugin-cache
```

第一行创建缓存目录。

第二行告诉 Terraform：provider plugin 下载后放到这个目录里。

---

## 5. `TF_PLUGIN_CACHE_DIR` 是什么

`TF_PLUGIN_CACHE_DIR` 是 Terraform 支持的环境变量，用来指定 provider plugin 的缓存目录。

例如：

```sh
export TF_PLUGIN_CACHE_DIR=/workspace/practice/labs/9/.terraform-plugin-cache
```

意思是：

> 当前 shell 里执行 Terraform 时，provider plugin cache 使用 `/workspace/practice/labs/9/.terraform-plugin-cache` 这个目录。

注意：这个环境变量只对当前 shell 生效。

如果你开了一个新的终端窗口，或者进入新的 Docker container shell，需要重新设置。

---

## 6. Plugin Cache 和 `.terraform.lock.hcl` 的关系

这是这节课最容易考、也最容易混的地方。

**Plugin Cache 不是 lock file 的替代品。**

它们解决的是两个不同问题。

| 机制 | 解决的问题 |
|---|---|
| `TF_PLUGIN_CACHE_DIR` | provider 下载复用，减少重复下载 |
| `.terraform.lock.hcl` | 锁定 provider 版本和校验和，保证一致性和安全性 |

换句话说：

- cache 负责“下载快一点”；
- lock file 负责“版本对不对、校验和对不对”。

生产环境里，不能因为配置了 plugin cache，就忽略 `.terraform.lock.hcl`。

`.terraform.lock.hcl` 应该提交到 Git 仓库里。

`.terraform-plugin-cache` 通常不应该提交到 Git 仓库里。

---

## 7. 为什么 cache 目录不应该提交到 Git

`.terraform-plugin-cache` 里存放的是 provider 二进制文件。

这些文件通常：

- 体积比较大；
- 和操作系统、CPU 架构有关；
- 可以重新下载；
- 不适合作为源码管理内容。

所以一般 `.gitignore` 里应该包含：

```gitignore
.terraform/
.terraform-plugin-cache/
tfplan
terraform.tfstate
terraform.tfstate.*
```

但是 `.terraform.lock.hcl` 不要忽略。

它应该保留。

## 8. 实验到底在验证什么

这节实验不是让你真的测出“快了多少秒”。

它主要验证三件事：

1. 你知道需要先创建缓存目录；
2. 你知道通过 `TF_PLUGIN_CACHE_DIR` 启用 provider plugin cache；
3. 你知道 cache 只是优化下载，不替代 `.terraform.lock.hcl`。

实验预期生成：

```text
output/plugin-cache-policy.md
```

这个文件相当于是平台团队的规范文档，说明 Terraform 项目在 CI 或本地环境中如何使用 plugin cache。

## 9. `terraform test` 在这里测什么

测试文件是：

```text
practice/labs/9/tests/plugin_cache.tftest.hcl
```

它大概率会检查实验输出中是否包含关键策略内容，例如：

- 是否提到 `TF_PLUGIN_CACHE_DIR`；
- 是否记录了缓存目录；
- 是否记录了 init 命令；
- 是否强调 `.terraform.lock.hcl` 仍然重要；
- 是否生成了 `output/plugin-cache-policy.md`。

所以这节实验不是单纯跑 Terraform resource，而是用 Terraform 生成一份“平台工程规范文档”。

这类实验很贴近真实 Platform Engineering 工作：不是只会写资源，而是要把团队规范固化成代码和文档。

## 10. 为什么 CI 特别需要 Plugin Cache

在 CI/CD 里，Terraform pipeline 通常会频繁执行：

```sh
terraform init -input=false
terraform plan -input=false -no-color -out=tfplan
```

如果团队里有几十个 Terraform 项目，每个项目都下载 provider，就会拖慢 pipeline。

Provider Plugin Cache 可以减少重复下载，尤其适合：

- monorepo 里有多个 Terraform module；
- 多个环境目录，例如 `dev/`、`staging/`、`prod/`；
- CI runner 支持缓存目录持久化；
- provider 很大，例如 `azurerm`、`aws`、`google`。

不过要注意：如果 CI runner 是完全一次性的，而且没有把 cache 目录作为 CI cache artifact 保存，那么 plugin cache 的效果会有限。

---

## 11. CI 里的推荐写法

在 CI 里，可以这样配置：

```sh
mkdir -p "$HOME/.terraform.d/plugin-cache"
export TF_PLUGIN_CACHE_DIR="$HOME/.terraform.d/plugin-cache"
terraform init -input=false
terraform plan -input=false -no-color -out=tfplan
```

然后让 CI 系统缓存这个目录。

例如概念上可以缓存：

```text
~/.terraform.d/plugin-cache
```

不同 CI 平台语法不同，但思想一样：

> 把 provider plugin cache 目录持久化，下次 pipeline 继续复用。

---

## 12. 和上一节 `-no-color` 的关系

第 8 节讲的是：

```sh
terraform plan -input=false -no-color -out=tfplan
```

重点是让 CI 日志和审批材料更适合机器解析。

第 9 节讲的是：

```sh
export TF_PLUGIN_CACHE_DIR=...
terraform init -input=false
```

重点是让 provider 下载更快、更稳定。

它们都是 CI/CD 级 Terraform 工作流的组成部分。

可以这样理解：

| 训练 | 关注点 | 对应能力 |
|---|---|---|
| 第 8 节 | 输出可读、可归档 | CI 日志治理 |
| 第 9 节 | provider 下载复用 | CI 性能优化 |

这两节连起来，就是从“能在本地跑 Terraform”升级到“能写工程化 Terraform pipeline”。

## 13. 常见错误

### 错误 1：忘记创建 cache 目录

错误写法：

```sh
export TF_PLUGIN_CACHE_DIR=/workspace/practice/labs/9/.terraform-plugin-cache
terraform init -input=false
```

如果目录不存在，可能导致 Terraform init 报错或 cache 不生效。

推荐先执行：

```sh
mkdir -p .terraform-plugin-cache
```

---

### 错误 2：以为 cache 能锁版本

错误理解：

> 既然 provider 已经缓存了，就不用 `.terraform.lock.hcl` 了。

这是错的。

cache 只是存文件，不负责定义项目应该使用哪个版本。

版本和校验和仍然由 `.terraform.lock.hcl` 管理。

---

### 错误 3：把 `.terraform-plugin-cache` 提交到 Git

不推荐这样做。

cache 是机器本地状态，不是项目源码。

应该提交：

```text
.terraform.lock.hcl
```

不应该提交：

```text
.terraform/
.terraform-plugin-cache/
```

---

### 错误 4：在新 shell 里忘记重新 export

`export TF_PLUGIN_CACHE_DIR=...` 只对当前 shell 有效。

如果重新进入 Docker container，或者开了新终端，需要重新执行。

可以通过下面命令检查：

```sh
echo $TF_PLUGIN_CACHE_DIR
```

如果没有输出，说明环境变量没有生效。

---

## 14. 清理命令解释

实验清理命令是：

```sh
terraform destroy -auto-approve
rm -rf .terraform .terraform-plugin-cache output tfplan terraform.tfstate*
```

含义如下：

| 命令 | 作用 |
|---|---|
| `terraform destroy -auto-approve` | 删除 Terraform 创建的资源 |
| `rm -rf .terraform` | 删除当前项目本地 Terraform 工作目录 |
| `rm -rf .terraform-plugin-cache` | 删除 provider plugin 缓存目录 |
| `rm -rf output` | 删除实验生成的策略文档 |
| `rm -f tfplan` | 删除 plan 文件 |
| `rm -f terraform.tfstate*` | 删除本地状态文件和备份 |

注意：真实生产环境不要随便删除 state 文件。

本实验使用的是本地练习环境，所以可以清理。

## 15. 面试 / 考试表达方式

如果被问到 Terraform Provider Plugin Cache，可以这样答：

> Terraform provider plugin cache is used to avoid repeatedly downloading the same provider binaries during `terraform init`. It is configured with `TF_PLUGIN_CACHE_DIR`. It improves initialization speed and reduces network usage, especially in CI pipelines. However, it does not replace `.terraform.lock.hcl`, which still controls provider versions and checksums.

中文理解：

> Provider Plugin Cache 是为了减少重复下载 provider，提高 `terraform init` 的速度，尤其适合 CI。它通过 `TF_PLUGIN_CACHE_DIR` 配置。但它不负责锁版本，也不替代 `.terraform.lock.hcl`。

---

## 16. 这一节的核心记忆点

记住这几句话就够了：

1. `terraform init` 会下载 provider plugin。
2. `TF_PLUGIN_CACHE_DIR` 可以让多个项目复用 provider 下载缓存。
3. Plugin cache 优化的是速度和带宽，不是版本治理。
4. `.terraform.lock.hcl` 仍然必须保留，用来锁定 provider 版本和校验和。
5. `.terraform-plugin-cache` 不应该提交到 Git。
6. CI 里要让 cache 目录持久化，否则每次全新 runner 仍然会重新下载。

---

## 17. 一句话总结

这节训练的本质是：

> 从“每个 Terraform 项目各自重复下载 provider”，升级到“平台团队统一复用 provider plugin cache”，让 Terraform init 更快、更稳定、更适合 CI/CD 工程化落地。
