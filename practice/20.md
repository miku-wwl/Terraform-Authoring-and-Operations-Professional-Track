# Terraform 实操训练 20：导入后生成配置

## 1. 课程元数据

- 课程编号：20
- 来源文件：`doc/20.txt`
- 环境分类：`docker`
- 是否可用 Docker 运行：`yes`
- 核心主题：import block、`-generate-config-out`、导入后管理资源
- 前置条件：Docker、`hashicorp/terraform:1.11`

## 2. 知识点讲解

import block 由 `to` 和 `id` 组成：`to` 是 Terraform 资源地址，`id` 是远端真实资源 ID。`terraform plan -generate-config-out=generated.tf` 可以根据 import block 生成候选配置，然后通过 apply 完成 state 导入。

真实导入后还要继续跑 plan，确认 Terraform 不会对已存在资源做意外变更。生成配置只是起点，不是最终架构设计。

## 3. 实验设计

背景：课程用 AWS 安全组演示手工资源导入。

问题：当前阶段不执行 AWS，因此需要本地模拟导入流程。

任务：用 `terraform_data` 模拟安全组，记录 import block 的地址和 ID。

预期结果：输出生成配置命令、目标地址和模拟端口规则。

## 5. 运行手册

```sh
cd /workspace/practice/labs/20
terraform init -input=false
terraform fmt -check
terraform validate
terraform test
terraform plan -input=false -no-color -out=tfplan
terraform apply -auto-approve tfplan
terraform output
terraform destroy -auto-approve
```

## 6. 验证报告

状态：通过。

已执行 `init/fmt/validate/test/plan/apply/output/destroy`。`terraform test` 返回 `1 passed, 0 failed`，输出确认生成配置命令为 `terraform plan -generate-config-out=generated.tf`，模拟安全组端口为 `80/443/22`。首次验证发现端口输出在 plan 阶段未知，已将测试改为 `command = apply`。

真实 AWS 安全组导入暂跳过，因为需要 AWS 凭证、区域、VPC 和安全组 ID。

## 7. 排障指南

1. `to` 地址不存在：先定义目标资源块或确认生成配置目标。
2. 生成文件已存在：换一个输出文件名或先审查旧文件。
3. plan 出现修改：比较生成配置与真实资源属性。
4. 导入后仍漂移：检查是否有平台侧默认值未写入配置。

## 8. 清理方式

```sh
terraform destroy -auto-approve
rm -f tfplan generated.tf
rm -rf .terraform .terraform.lock.hcl
```
