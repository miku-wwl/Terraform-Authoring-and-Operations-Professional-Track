# Terraform 实操训练 7：`-input=false` 与非交互式失败教程

## 1. 这节课到底在学什么

这节课的核心是：

> **Terraform 在 CI/CD 里不能等人输入变量。缺变量就应该快速失败，或者由流水线显式传入变量。**

你现在还处在从本地手工 Terraform 过渡到工程化 Terraform 的阶段，所以这节课非常关键。

在本地手动运行时，你可能会这样执行：

```sh
terraform plan
```

如果 Terraform 发现缺少必填变量，它可能会提示你输入：

```text
var.artifact_name
  Enter a value:
```

人在终端前当然可以输入。

但 CI/CD runner 不会帮你输入。

所以流水线里一旦 Terraform 等待输入，任务就会卡住，最后超时失败。

这就是 `-input=false` 要解决的问题。

---

## 2. 什么是 `-input=false`

`-input=false` 的意思是：

> **禁止 Terraform 进行交互式输入。**

也就是说，Terraform 不能问你：

```text
Enter a value:
```

如果缺少必填变量，它应该直接失败。

例如：

```sh
terraform plan -input=false
```

如果没有提供必填变量，Terraform 会报错，而不是卡住等待。

这在 CI/CD 中是正确行为。

因为自动化环境里更希望：

```text
缺配置 → 快速失败 → 修复流水线参数
```

而不是：

```text
缺配置 → 等待输入 → 流水线挂起 → 最后超时
```

---

## 3. 为什么本地能跑，CI/CD 不一定能跑

本地执行：

```sh
terraform plan
```

如果缺变量，Terraform 可以问你。

CI/CD 执行：

```sh
terraform plan
```

如果缺变量，任务可能会卡住。

所以自动化环境里不要依赖 Terraform 提示输入。

应该显式写成：

```sh
terraform plan -input=false
```

然后通过 `-var` 或 `tfvars` 文件提供变量。

---

## 4. 本实验的背景

本实验中，Terraform 要生成一个构建产物文件。

文件名由变量控制：

```text
artifact_name
```

实验要求生成：

```text
output/ci-artifact.txt
```

也就是说，流水线必须明确传入：

```sh
-var artifact_name=ci-artifact.txt
```

否则 Terraform 不知道要创建哪个文件名。

---

## 5. 必填变量是什么

在 Terraform 中，如果变量没有默认值，它通常就是必填变量。

例如：

```hcl
variable "artifact_name" {
  type        = string
  description = "Name of the artifact file."
}
```

这种变量没有 `default`，所以执行 `plan` 时必须提供值。

可以通过几种方式提供：

### 方式 1：命令行传入

```sh
terraform plan -var artifact_name=ci-artifact.txt
```

### 方式 2：变量文件传入

```sh
terraform plan -var-file=terraform.tfvars
```

### 方式 3：环境变量传入

```sh
export TF_VAR_artifact_name=ci-artifact.txt
terraform plan
```

本实验使用的是第一种：

```sh
-var artifact_name=ci-artifact.txt
```

---

## 6. 正确的 plan 命令

本实验中正确的 plan 命令是：

```sh
terraform plan -input=false -no-color -out=tfplan -var artifact_name=ci-artifact.txt
```

逐个拆开看：

| 参数 | 作用 |
|---|---|
| `-input=false` | 禁止交互式输入，缺变量就快速失败 |
| `-no-color` | 去掉颜色码，方便 CI 日志和 artifact 保存 |
| `-out=tfplan` | 保存执行计划，后续 apply 使用同一个 plan |
| `-var artifact_name=ci-artifact.txt` | 显式传入必填变量 |

这条命令的工程含义是：

> 使用非交互方式生成一个可保存、可审查、变量明确的 Terraform plan。

---

## 7. 正确的 apply 命令

本实验中 apply 命令是：

```sh
terraform apply -auto-approve tfplan
```

这里有两个重点：

### 第一，使用 `tfplan`

```sh
terraform apply tfplan
```

表示执行之前保存好的 plan 文件。

这可以保证：

```text
plan 阶段审查什么，apply 阶段就执行什么。
```

### 第二，使用 `-auto-approve`

```sh
-auto-approve
```

表示不再等待人工输入 `yes`。

CI/CD 中不能让 Terraform 等待确认，所以自动化执行时需要这个参数。

但是要注意：

```sh
terraform apply -auto-approve
```

和：

```sh
terraform apply -auto-approve tfplan
```

不是一个安全等级。

更推荐：

```sh
terraform apply -auto-approve tfplan
```

因为它执行的是已经生成并审查过的 plan。

---

## 8. 为什么 destroy 也需要变量

这节课里最容易忽略的点是：

> **destroy 也可能需要变量。**

很多初学者以为变量只在 `plan` 和 `apply` 里有用。

但实际上，Terraform 在执行 destroy 时也要加载完整配置。

如果配置里依赖必填变量，比如：

```hcl
var.artifact_name
```

那么 destroy 时也需要知道这个变量的值。

所以本实验的 destroy 不能只写：

```sh
terraform destroy -auto-approve
```

而应该写：

```sh
terraform destroy -auto-approve -var artifact_name=ci-artifact.txt
```

否则可能出现：

```text
No value for required variable
```

这就是本节实验修正运行手册的关键点。

## 9. 本实验完整命令链

从实验目录执行：

```sh
cd /workspace/practice/labs/7
```

完整命令链：

```sh
terraform init -input=false
terraform fmt -check
terraform validate
terraform test
terraform plan -input=false -no-color -out=tfplan -var artifact_name=ci-artifact.txt
terraform apply -auto-approve tfplan
terraform output
terraform destroy -auto-approve -var artifact_name=ci-artifact.txt
```

这条命令链可以理解为：

| 步骤 | 作用 |
|---|---|
| `terraform init -input=false` | 初始化，并禁止交互输入 |
| `terraform fmt -check` | 检查代码格式 |
| `terraform validate` | 验证配置是否合法 |
| `terraform test` | 执行测试，确认命令链包含 `-input=false` |
| `terraform plan ... -var artifact_name=ci-artifact.txt` | 非交互生成 plan，并传入必填变量 |
| `terraform apply -auto-approve tfplan` | 执行保存好的 plan |
| `terraform output` | 查看输出 |
| `terraform destroy ... -var artifact_name=ci-artifact.txt` | 带变量销毁资源 |

## 12. 初学者最容易踩的坑

### 坑 1：以为 `terraform plan` 缺变量时让人输入很正常

本地可以。

CI/CD 不行。

自动化中应该使用：

```sh
terraform plan -input=false
```

这样缺变量会直接失败。

---

### 坑 2：只给 plan 传变量，忘了 destroy

错误写法：

```sh
terraform destroy -auto-approve
```

如果配置依赖必填变量，可能失败。

正确写法：

```sh
terraform destroy -auto-approve -var artifact_name=ci-artifact.txt
```

---

### 坑 3：以为 `-input=false` 会自动提供变量

不会。

`-input=false` 只是禁止 Terraform 问你要变量。

它不会替你生成变量值。

所以你还需要：

```sh
-var artifact_name=ci-artifact.txt
```

或者：

```sh
-var-file=terraform.tfvars
```

---

### 坑 4：以为 `apply tfplan` 时还需要重新传 `-var`

本实验中 apply 使用的是保存好的 plan：

```sh
terraform apply -auto-approve tfplan
```

这个 plan 已经包含了 plan 阶段传入的变量值。

所以 apply 阶段通常不需要重新传 `-var`。

但是 destroy 不一样。

destroy 会重新加载配置，所以仍然可能需要变量。

---

## 13. `-var` 和 `-var-file` 怎么选

### 临时单个变量

适合使用：

```sh
-var artifact_name=ci-artifact.txt
```

适合简单实验、临时传参、CI 参数注入。

### 多个变量

更适合使用：

```sh
-var-file=dev.tfvars
```

例如：

```hcl
artifact_name = "ci-artifact.txt"
environment   = "dev"
owner         = "platform-team"
```

然后执行：

```sh
terraform plan -input=false -var-file=dev.tfvars
```

真实项目中，`-var-file` 更常见，因为变量通常不止一个。

## 14. 面试或考试里怎么表达

如果别人问：

> 为什么 CI/CD 里 Terraform 要用 `-input=false`？

可以回答：

```text
In CI/CD, Terraform should run in non-interactive mode. If a required variable is missing, the pipeline should fail fast instead of waiting for user input. We pass required values explicitly through -var, -var-file, or environment variables such as TF_VAR_name.
```

中文意思：

> 在 CI/CD 中，Terraform 应该以非交互模式运行。如果缺少必填变量，流水线应该快速失败，而不是等待人工输入。必填值应该通过 `-var`、`-var-file` 或 `TF_VAR_name` 这类环境变量显式传入。

如果别人追问：

> destroy 为什么也要传变量？

可以回答：

```text
Terraform still loads and evaluates the configuration during destroy. If the configuration references required variables without defaults, destroy also needs those values.
```

中文意思：

> Terraform 在 destroy 时仍然会加载和解析配置。如果配置中引用了没有默认值的必填变量，destroy 同样需要这些变量值。


## 16. 核心结论

这节课真正要掌握的是：

```text
CI/CD 中不要让 Terraform 等待输入。
缺少变量应该快速失败。
必填变量必须通过 -var、-var-file 或环境变量显式提供。
destroy 也可能需要变量。
```

最关键的一组命令是：

```sh
terraform plan -input=false -no-color -out=tfplan -var artifact_name=ci-artifact.txt
terraform apply -auto-approve tfplan
terraform destroy -auto-approve -var artifact_name=ci-artifact.txt
```

一句话总结：

> **`-input=false` 不是帮你填变量，而是禁止 Terraform 等你填变量。CI/CD 中变量必须提前准备好。**
