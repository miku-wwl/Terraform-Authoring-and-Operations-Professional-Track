# Terraform 考试训练 130 讲解版：Health Assessments

> 来源：`130.md`  
> 主题：HCP Terraform Health Assessments、Drift Detection、Continuous Validation、`check` block  
> 目标：能在考试题里快速判断“漂移检测”和“持续验证”分别解决什么问题。

---

## 1. 这一课到底在讲什么？

这一课讲的是 **HCP Terraform 的 Health Assessments**。

你可以把 Health Assessments 理解成：

> Terraform 不只是“部署一次资源”，还要持续关心：  
> **真实云资源有没有被人改歪？资源运行后还健不健康？**

它主要包含两个方向：

| 能力 | 中文理解 | 关注点 |
|---|---|---|
| Drift Detection | 漂移检测 | 真实基础设施是否偏离 Terraform 配置和 state |
| Continuous Validation | 持续验证 | 资源部署后，自定义健康条件是否持续成立 |

这两个概念很像，但考试里一定要分清。

---

## 2. 先建立一个大框架

Terraform 的传统流程通常是：

```bash
terraform init
terraform plan
terraform apply
```

这个流程解决的是：

> 当前这次变更，Terraform 要不要创建、修改、删除资源？

但是企业真实环境里，问题不会在 `apply` 之后结束。

比如：

- 有人手动去云控制台改了安全组。
- 有人改了实例标签。
- 某个服务部署后过一段时间不返回 HTTP 200 了。
- TLS 证书过期了。
- 负载均衡器健康检查失败了。

这些问题，不一定来自 Terraform 代码变更，但它们会影响真实基础设施的状态。

所以 HCP Terraform 引入 Health Assessments，把 Terraform 从“一次性部署工具”往“持续治理平台”方向推进。

---

## 3. Drift Detection：漂移检测

### 3.1 Drift 是什么？

**Drift** 的核心意思是：

> Terraform 配置没有变，但真实基础设施被改了，导致真实资源和 Terraform 期望不一致。

典型例子：

你用 Terraform 创建了一个安全组，只允许 443 入站。

后来有人进入 AWS Console 或 Azure Portal，手动加了一个 0.0.0.0/0 的 SSH 入站规则。

这时候：

- `.tf` 文件没有变。
- Terraform state 可能还记录着旧状态。
- 真实云资源已经变了。
- 基础设施已经偏离 IaC 管理边界。

这就是 drift。

---

### 3.2 Drift Detection 检测什么？

Drift Detection 检测的是：

> 真实基础设施是否偏离 Terraform 配置和 state 期望。

考试题里看到下面这些关键词，基本就往 drift detection 想：

- 手动修改云资源
- 云控制台改配置
- 真实资源和 Terraform 配置不一致
- 资源偏离 state
- 检测 infrastructure drift
- 有人绕过 Terraform 修改资源

---

### 3.3 Drift Detection 的考试判断方式

考试题可能这样问：

> 某安全组通过 Terraform 创建后，被人在控制台手动新增了入站规则。HCP Terraform 的哪类能力可以发现这种偏离？

答案就是：

> Drift Detection。

原因是这个问题不是“服务健不健康”，而是“真实资源配置被改歪了”。

---

## 4. Continuous Validation：持续验证

### 4.1 Continuous Validation 是什么？

Continuous Validation 关注的是：

> 资源创建以后，自定义健康条件是否持续成立。

它不是单纯检查 Terraform 配置和真实资源是否一致，而是检查业务或运维层面的健康条件。

例如：

- 网站是否持续返回 HTTP 200。
- API 是否仍然可访问。
- TLS 证书是否仍然有效。
- 负载均衡器健康检查是否通过。
- DNS 解析是否正常。
- 某个监控端点是否返回预期结果。

---

### 4.2 Continuous Validation 和 Drift Detection 的区别

这是本课最重要的区分。

| 对比点 | Drift Detection | Continuous Validation |
|---|---|---|
| 检查对象 | 资源配置是否偏离 Terraform 期望 | 资源运行后的健康条件是否满足 |
| 常见原因 | 人工手动改云资源 | 服务异常、证书过期、健康检查失败 |
| 典型问题 | 安全组被手动加规则 | 网站不返回 HTTP 200 |
| 考试关键词 | drift、manual change、state mismatch | health check、HTTP 200、validation、post-deployment |
| 本质 | 配置一致性检查 | 健康条件检查 |

一句话记忆：

> Drift Detection 看“资源有没有被改歪”。  
> Continuous Validation 看“资源运行后还健不健康”。

---

## 5. `check` block 的作用

Terraform CLI 里有一个重要语言特性：`check` block。

它可以表达类似“持续健康检查”的逻辑。

示意例子：

```hcl
check "website_health" {
  data "http" "example" {
    url = "https://example.com"
  }

  assert {
    condition     = data.http.example.status_code == 200
    error_message = "Website did not return HTTP 200"
  }
}
```

这个例子表达的是：

> 我希望网站返回 HTTP 200，否则这个健康检查不通过。

注意，这不是在创建资源，而是在表达一个断言。

---

## 6. HCP Terraform 的平台化价值

本地 Terraform CLI 可以运行 `check` block，但 HCP Terraform 的价值在于：

- 把检查放到平台里持续运行。
- 让团队可以看到健康状态。
- 让 drift 和 validation 结果可视化。
- 更适合企业治理、审计和持续运维。
- 让 IaC 不只是部署工具，而是持续管理机制。

所以考试里如果问：

> Health assessments 更偏什么能力？

答案通常是：

> 企业治理和持续运维能力。

---

## 7. 容易混淆的点

### 7.1 Drift Detection 不是格式检查

Drift Detection 不是检查：

- Markdown 拼写
- `.tf` 文件缩进
- `terraform fmt`
- Docker 镜像是否过期
- Git 用户名是否正确

它检查的是：

> 真实基础设施是否偏离 Terraform 期望。

---

### 7.2 Continuous Validation 不是只运行一次

Continuous Validation 的名字里有 **continuous**，重点就是持续。

所以这个判断题是错的：

> Continuous validation 只在 `terraform init` 之前运行一次。

错得很明显。

`terraform init` 是初始化工作目录和 provider/backend/cloud 集成，不是持续健康检查。

---

### 7.3 `check` block 不等于 drift detection

`check` block 更接近 continuous validation。

因为它通常表达的是：

> 某个条件是否成立。

比如 HTTP 200、证书有效、服务可访问。

而 drift detection 更关注 Terraform 管理的资源配置有没有被改动。

## 8. 考试题逐题讲解

### 选择题 1

题目：

> Drift detection 主要检测什么？

正确答案：

> A. 真实基础设施是否偏离 Terraform 配置和 state 期望

解析：

Drift 的核心就是“偏离”。只要看到真实基础设施和 Terraform 期望不一致，就要想到 drift detection。

其他选项都不是 Terraform drift 的关注点。

---

### 选择题 2

题目：

> Continuous validation 更关注什么？

正确答案：

> A. 资源创建后，自定义健康条件是否持续通过

解析：

Continuous validation 重点是“部署后持续验证”。它不是删除 state，也不是限制变量，更不是要求所有代码写在一个文件。

---

### 选择题 3

题目：

> Terraform CLI 中哪个语言特性可以表达持续检查类逻辑？

正确答案：

> A. `check` block

解析：

`check` block 可以定义断言，用来表达资源部署后的健康条件。它是理解 continuous validation 的重要参照。

## 9. 判断题讲解

### 判断题 1

> Drift 可能由人工在云控制台直接修改资源造成。

答案：对。

这是最经典的 drift 来源。Terraform 没改，但云上资源被人改了。

---

### 判断题 2

> Continuous validation 只在 `terraform init` 之前运行一次。

答案：错。

`init` 是初始化，continuous validation 是持续健康验证。两者不是一个层面的事情。

---

### 判断题 3

> Health assessments 更偏企业治理和持续运维能力。

答案：对。

Health assessments 不是基础 CLI 入门功能，而是偏平台化、团队化、持续治理的能力。

题目：

> 团队希望确认网站部署后持续返回 HTTP 200。这个需求更接近 drift detection 还是 continuous validation？

> Continuous Validation。

## 11. 面试/实战理解

在真实公司里，drift detection 和 continuous validation 都很有价值。

### Drift Detection 解决的问题

它解决的是：

> “有没有人绕过 Terraform 偷偷改资源？”

这在企业环境里很关键，因为绕过 Terraform 改资源会带来：

- 配置不可追踪
- 安全风险
- 审计风险
- 下次 `terraform apply` 行为不可预测
- IaC 可信度下降

---

### Continuous Validation 解决的问题

它解决的是：

> “资源创建出来以后，是不是还一直正常？”

这更接近 SRE 和运维视角。

比如你创建了一个网站，不代表它永远可用。Continuous validation 可以帮助团队持续发现健康条件失败的问题。

## 16. 结论

Health Assessments 是 HCP Terraform 从“部署工具”走向“持续治理平台”的重要能力。

它的考试核心不是背 UI 菜单，而是识别两类问题：

1. **资源配置被改歪了**：Drift Detection。
2. **资源运行后健康条件失败了**：Continuous Validation。

把这两个边界分清，第 130 课就不难。
