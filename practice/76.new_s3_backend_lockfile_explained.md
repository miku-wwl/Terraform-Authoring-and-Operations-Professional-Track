# Terraform 实操训练 76 讲解：新式 S3 Backend Lockfile

## 1. 这一节到底在学什么？

第 76 节学的是 Terraform S3 backend 的**新式锁机制**：

```hcl
use_lockfile = true
```

上一节第 75 节讲的是旧式方案：

```hcl
dynamodb_table = "terraform-locks"
```

这一节讲的是新式方案：

```hcl
use_lockfile = true
```

一句话理解：

> 旧式 S3 backend 用 DynamoDB 表做 state lock；新式 S3 backend 可以直接用 S3 lockfile 做锁，不再需要 DynamoDB table。

课程文件里也明确说明：

```text
Terraform 1.14 已经提示 dynamodb_table 参数 deprecated，新项目推荐使用 S3 backend 自带的 use_lockfile。
```

所以这节非常关键：  
你既要会读旧式系统，也要会用新式方案。

---

## 2. 为什么 Terraform 需要 state lock？

Terraform state 是资源管理账本。

它记录：

- Terraform 管理了哪些资源
- 每个资源的真实 ID
- 配置地址和真实资源之间的映射
- 当前资源属性
- 上一次 apply 后的状态

如果两个人或两个 CI pipeline 同时写 state，就可能发生 state 损坏。

例如：

```text
A pipeline 正在 apply
B pipeline 也开始 apply
A 写入 state
B 又写入 state
最终 state 可能丢失 A 的变更
```

所以 Terraform 在执行可能写 state 的操作前，要先获取锁。

state lock 的目标是：

> 同一时间只允许一个 Terraform 进程写入 state。

---

## 3. 第 75 节旧式方案回顾

旧式 AWS S3 backend 常见配置：

```hcl
terraform {
  backend "s3" {
    bucket         = "terraform-state"
    key            = "env/dev/terraform.tfstate"
    region         = "us-east-1"
    dynamodb_table = "terraform-locks"
  }
}
```

它的结构是：

```text
S3 bucket       -> 保存 terraform.tfstate
DynamoDB table  -> 保存 lock 信息
```

DynamoDB 表必须有：

```text
Partition key: LockID
Type: String
```

这套方案很常见，很多公司老项目仍然在用。

但是新项目正在转向更简单的 S3 lockfile 方式。

---

## 4. 第 76 节新式方案是什么？

新式方案：

```hcl
terraform {
  backend "s3" {
    bucket       = "terraform-state"
    key          = "env/dev/terraform.tfstate"
    region       = "us-east-1"
    use_lockfile = true
  }
}
```

它的结构是：

```text
S3 bucket 保存 terraform.tfstate
S3 backend 自己创建 lockfile 做锁
不需要 DynamoDB table
```

也就是说：

```text
旧式：S3 + DynamoDB
新式：S3 + S3 lockfile
```

核心变化：

> locking 从 DynamoDB 表转移到 S3 backend 自带的 lockfile。

---

## 5. `use_lockfile = true` 做了什么？

当你配置：

```hcl
use_lockfile = true
```

Terraform 在需要写 state 时，会尝试在 S3 backend 中创建锁文件。

可以理解为：

```text
执行 apply 前：
  创建 lockfile，表示我正在写 state

apply 成功或失败退出时：
  删除 lockfile，释放锁
```

如果另一个 Terraform 进程发现 lockfile 已存在，就不应该继续写 state。

这和 DynamoDB lock 的目标一样：

```text
防止并发写 state
```

只是锁的存储位置从 DynamoDB 变成了 S3。

---

## 6. 旧式 DynamoDB lock 和新式 lockfile 对比

| 对比项 | 旧式 DynamoDB lock | 新式 S3 lockfile |
|---|---|---|
| state 存储 | S3 | S3 |
| lock 存储 | DynamoDB table | S3 lockfile |
| backend 参数 | `dynamodb_table` | `use_lockfile = true` |
| 是否需要 DynamoDB | 需要 | 不需要 |
| 新项目推荐度 | 旧系统常见 | 新项目更推荐 |
| 维护成本 | 需要维护 S3 + DynamoDB | 只需要 S3 |
| 迁移关注点 | DynamoDB 表、LockID | lockfile 行为、S3 权限 |
| 课程位置 | 第 75 节 | 第 76 节 |

一句话：

> 旧式方案多一个 DynamoDB 组件，新式方案把锁也放回 S3 backend 内部处理。

---

## 7. 为什么 `dynamodb_table` deprecated 还要学？

因为现实工作不是只有新项目。

你以后看到公司老项目，很可能还有：

```hcl
dynamodb_table = "terraform-locks"
```

这时你必须知道：

- 它是旧式 state locking。
- DynamoDB 表必须有 `LockID`。
- 它不是业务数据库。
- 它是 Terraform backend lock table。
- deprecated 不代表马上不能用。
- 迁移前要备份 state，并验证 backend 行为。

第 75 节练旧式，是为了你能维护存量系统。  
第 76 节练新式，是为了你能做新项目和迁移设计。

这就是工程现实。

---

## 8. 新式 lockfile 的价值

`use_lockfile = true` 的价值主要是降低复杂度。

旧式方案需要准备：

```text
S3 bucket
DynamoDB table
DynamoDB LockID key
IAM 对 S3 的权限
IAM 对 DynamoDB 的权限
```

新式方案主要需要：

```text
S3 bucket
S3 lockfile 权限
IAM 对 S3 的权限
```

少维护一个 DynamoDB table，bootstrap 更简单。

对于新项目来说，这更干净。

但你仍然要理解：

```text
lock 不是可有可无
只是实现方式变了
```

## 11. 典型 backend 配置

真实 AWS 中，新式配置可能类似：

```hcl
terraform {
  backend "s3" {
    bucket       = "my-terraform-state"
    key          = "env/dev/terraform.tfstate"
    region       = "ap-southeast-2"
    use_lockfile = true
  }
}
```

LocalStack 中通常还要加 endpoint 和跳过真实 AWS 校验的配置。

示例思路：

```hcl
terraform {
  backend "s3" {
    bucket       = "terraform-state"

      s3  = "http://localhost:4566"

    skip_metadata_api_check     = true
    skip_region_validation      = true
    force_path_style            = true

}
```

不同 Terraform 版本、AWS backend 版本可能细节略有差异。

```text
backend "s3"

use_lockfile = true

```

## 13. S3 bucket 仍然要提前创建

新式 lockfile 不需要 DynamoDB，但仍然需要 S3 bucket。

也就是说，backend 初始化前，bucket 必须已经存在。

因为执行顺序是：

```text
terraform init 初始化 backend
terraform plan 计算资源
terraform apply 创建资源
```

backend 初始化发生在资源创建之前。  
所以不能指望同一个 Terraform 项目先创建 backend bucket，再立刻使用它。

生产里仍然需要 bootstrap：

```text
bootstrap 项目创建 state bucket
业务项目使用 state bucket 作为 backend
```

执行 apply 后，在实验目录检查：

```sh

```


## 19. 如何观察 lockfile？

`use_lockfile = true` 的锁文件通常只在 Terraform 操作期间存在。

也就是说，apply 正常结束后，lockfile 会被释放/删除。

所以你 apply 后再查 S3，可能只看到：

```text
terraform.tfstate
```

看不到 lockfile，这不一定是错误。

如果想观察 lockfile，通常要在 Terraform 操作进行中检查。  
但实验中 apply 很快，未必能抓到。

本节核心验证不是“锁文件一直存在”，而是：

```text
use_lockfile 配置能初始化
state 能正常写入 S3
流程不依赖 DynamoDB
```

## 20. `terraform state list` 的意义

运行：

```sh
terraform state list
```

Terraform 会从当前 backend 读取 state，并列出资源地址。

如果 S3 backend 生效，它读的是 S3 state，而不是本地 `terraform.tfstate`。

这能验证：

```text
Terraform 可以通过 S3 backend 正常读取远端 state
```

这一步比只查文件更接近真实 Terraform 使用方式。

## 21. 锁失败时会怎样？

无论旧式 DynamoDB lock 还是新式 S3 lockfile，目标都一样：

```text
防止并发写 state
```

如果 Terraform 获取锁失败，正确行为是：

```text
停止执行，不继续写 state
```

你可能看到锁相关错误。

排查时要问：

1. 是否另一个 Terraform 正在运行？
2. 是否 CI pipeline 正在 apply？
3. 是否上一次操作异常中断？
4. lockfile 是否残留？
5. 是否需要人工解锁？
6. 是否有人用了 `-lock=false` 绕过锁？

不要一上来就强制处理。  
锁失败通常是在保护你。

## 22. 新式 lockfile 是否还需要 DynamoDB？

不需要。

这是本节重点。

如果使用：

```hcl
use_lockfile = true
```

就不需要：

```hcl
dynamodb_table = "terraform-locks"
```

也不需要创建 DynamoDB lock table。

不要把两个方案混着理解。

当然，在迁移期可能会看到旧配置残留。  
生产迁移要按 Terraform 当前版本和官方建议谨慎操作。

---

## 23. 能不能同时写 `dynamodb_table` 和 `use_lockfile`？

新项目不建议混用。

你要把它们看作两种锁方案：

```text
旧式：dynamodb_table
新式：use_lockfile
```

如果看到两者同时出现，要特别警惕：

- 是不是迁移中间状态？
- Terraform 版本是否支持？
- backend 行为是否明确？
- 团队是否已经统一方案？
- 是否有 migration 文档？

对于学习和新项目，保持清晰：

```text
第 75 节：练 dynamodb_table
第 76 节：练 use_lockfile
```

## 24. 新式 lockfile 的 IAM 权限注意点

真实 AWS 中，使用 S3 lockfile 意味着 Terraform 需要对 S3 lockfile 对象有相应权限。

除了 state object 的读写权限，还要确保可以创建、读取、删除 lockfile。

大致需要关注：

- `s3:GetObject`
- `s3:PutObject`
- `s3:DeleteObject`
- `s3:ListBucket`
- 对 state key prefix 的权限
- 对 lockfile 对象的权限

具体 IAM policy 要按真实 backend 行为和组织规范设计。

LocalStack 实验不会严格模拟所有 IAM 细节。  
生产中不能忽略权限边界。

## 25. S3 state bucket 的生产建议

无论旧式还是新式，state bucket 都很重要。

建议：

1. 开启 versioning。
2. 开启 encryption。
3. block public access。
4. 使用最小权限 IAM。
5. 不和业务文件混用。
6. 开启审计。
7. 限制删除 state。
8. 按环境设计 key prefix。
9. 给 CI/CD 单独角色。
10. 定期验证恢复流程。

`use_lockfile` 简化了锁组件，但没有降低 state 本身的重要性。

## 26. key 设计仍然重要

S3 backend 的 `key` 是 state 在 bucket 里的路径。

示例：

```hcl
key = "env/dev/network/terraform.tfstate"
```

不要多个项目共用同一个 key。

推荐结构：

```text
<environment>/<system>/<component>/terraform.tfstate
```

例如：

```text
dev/platform/network/terraform.tfstate
prod/platform/eks/terraform.tfstate
prod/app/payment/terraform.tfstate
```

key 设计错了，多个 Terraform 项目可能互相覆盖 state。  
这比普通配置错误严重得多。

---

## 27. backend block 仍然不能直接用 var/local

和旧式方案一样，backend 初始化发生得很早。

所以你不能写：

```hcl
terraform {
  backend "s3" {
    bucket = var.state_bucket
  }
}
```

backend block 里不能直接引用普通变量。

常见做法：

```sh
terraform init -backend-config=backend-dev.hcl
```

或者：

```sh
terraform init   -backend-config="bucket=my-state-bucket"   -backend-config="key=env/dev/terraform.tfstate"   -backend-config="region=ap-southeast-2"
```

`use_lockfile` 也可以放在 backend config 中：

```hcl
use_lockfile = true
```

---

## 28. backend-dev.hcl 示例

```hcl
bucket       = "my-terraform-state"
key          = "env/dev/terraform.tfstate"
region       = "ap-southeast-2"
use_lockfile = true
```

初始化：

```sh
terraform init -backend-config=backend-dev.hcl
```

这种方式适合：

- 多环境
- CI/CD
- backend 参数不写死在模块中
- 同一套模块复用不同 state 位置

但 backend config 不要放敏感凭据。

## 29. `terraform init -reconfigure` 什么时候用？

如果你之前初始化过旧 backend，现在修改成新式 lockfile：

```hcl
use_lockfile = true
```

可能需要：

```sh
terraform init -reconfigure
```

它的意思是：

> 按当前 backend 配置重新初始化。

如果涉及已有 state 迁移，要更谨慎，可能需要：

```sh
terraform init -migrate-state
```

实验环境可以清理重跑。  
生产环境不要随便 reconfigure 或 migrate，必须先备份 state。

## 30. 从旧式 DynamoDB lock 迁移到新式 lockfile 要注意什么？

如果你维护老项目，想从：

```hcl
dynamodb_table = "terraform-locks"
```

迁移到：

```hcl
use_lockfile = true
```

建议至少考虑：

1. 当前 Terraform 版本是否支持。
2. 团队是否停止所有 apply。
3. state 是否已备份。
4. backend config 如何变更。
5. IAM 是否支持 lockfile 所需 S3 操作。
6. CI/CD runner 是否使用新配置。
7. 是否需要 `terraform init -reconfigure` 或迁移流程。
8. 旧 DynamoDB table 是否保留一段时间。
9. 是否有回滚方案。
10. 是否记录变更窗口。

不要把 backend 迁移当普通变量修改。  
它影响的是 Terraform 控制面。

## 31. 本实验常见坑

### 坑 1：仍然创建 DynamoDB table

第 76 节练的是新式 lockfile，不需要 DynamoDB。

如果你还在纠结 `LockID`，那是第 75 节内容。

---

### 坑 2：忘记 `use_lockfile = true`

如果没写，可能只是 S3 backend，但没有启用新式 lockfile。

---

### 坑 3：S3 bucket 未创建

backend 初始化前 bucket 必须存在。

---

### 坑 4：LocalStack endpoint 配错

endpoint 没指向 `localhost:4566`，可能尝试连接真实 AWS。

---

### 坑 5：误以为看不到 lockfile 就失败

lockfile 只在操作期间存在，apply 结束后可能已删除。

---

### 坑 6：把 `.terraform/` 当作 state

`.terraform/` 可以存在，不代表本地 backend。  
要看有没有主 `terraform.tfstate`。

---

### 坑 7：backend 配置用 var/local

backend block 不能直接引用普通变量。

## 33. 和第 74、75 节的关系

|---|---|---|
| 74 | S3 中央后端 | state 从本地迁移到 S3 |
| 75 | 旧式 S3 + DynamoDB lock | DynamoDB table 做 state lock |

可以这样记：

```text

```

这三节连起来，就是 Terraform AWS backend 的工程化路线。

## 34. 和 CI/CD 的关系

CI/CD 里最怕多个 pipeline 同时写 state。

新式 lockfile 方案也要解决这个问题：

```text
pipeline A 拿到 lockfile
pipeline B 发现锁存在，不能同时写 state
```

所以 S3 backend + lockfile 可以支持团队自动化。

但 CI/CD 还要注意：

- pipeline concurrency 限制
- saved plan
- approval gate
- backend config 注入
- AWS credentials 权限
- state bucket 权限
- plan/apply 分离
- apply 失败后的锁处理

不要以为配置了 backend 就等于完整 CI/CD 成熟。

## 35. 生产判断：旧式还是新式？

简单建议：

### 新项目

优先考虑：

```hcl
use_lockfile = true
```

理由：

- 不需要 DynamoDB table
- backend 组件更少
- 更符合新版本方向

### 存量项目

如果还在用：

```hcl
dynamodb_table = "..."
```

不要盲目改。

应该：

- 先理解现状
- 查 Terraform 版本
- 备份 state
- 评估迁移窗口
- 更新 CI/CD
- 验证 IAM
- 做回滚计划

一句话：

> 新项目往新式走，老项目先稳住，再计划迁移。

## 36. 面试/考试记忆版

可以这样记：

- S3 backend 用 S3 保存 Terraform state。
- 旧式 state lock 使用 `dynamodb_table`。
- 新式 S3 backend lock 使用 `use_lockfile = true`。
- Terraform 1.14 已提示 `dynamodb_table` deprecated。
- `use_lockfile` 不需要 DynamoDB table。
- 新式方案仍然需要提前创建 S3 bucket。
- state 主文件仍然写入 S3，而不是本地目录。
- lockfile 通常只在 Terraform 操作期间存在。
- backend block 不能直接使用普通变量。
- 修改 backend 后需要重新 `terraform init`。
- 生产中 state bucket 仍需加密、versioning、严格 IAM。
- LocalStack 可以验证配置和流程，但不等于完整生产并发环境。

一句话总结：

> 新式 S3 backend lockfile 的核心，是用 `use_lockfile = true` 在 S3 backend 内完成 state locking，减少 DynamoDB 依赖，同时继续保护远端 state 的并发写入安全。

## 37. 本 Lab 的价值

第 76 节的价值在于，它让你跟上 Terraform S3 backend locking 的新方向。

第 75 节让你能维护旧系统。  
第 76 节让你能设计新系统。

你以后看到两种配置：

```hcl
dynamodb_table = "terraform-locks"
```

和：

```hcl
use_lockfile = true
```

要能马上判断：

```text
前者是旧式 DynamoDB locking
后者是新式 S3 lockfile locking
两者目标都是防止并发写 state
新项目更适合 use_lockfile
存量项目迁移要谨慎
```

这就是这一节真正要练出来的能力。

不要只把它当一个参数变化。  
它背后是 Terraform state 管理方式的工程演进。
