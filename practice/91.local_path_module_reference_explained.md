# Terraform 实操训练 91：本地路径引用模块（讲解版）

## 1. 本节定位

这一节的主题是 **本地路径引用模块**。

它属于 Terraform module 训练中的基础但非常关键的一环：  
先不急着引用 Terraform Registry 或 Git 远程模块，而是先学会在同一个代码仓库里，用相对路径引用自己写的 child module。

原始训练文件强调：

- 课程编号：91
- 核心主题：本地路径引用模块
- 实验目录：`practice/labs/91`
- 运行环境：Docker + `hashicorp/terraform:1.11`
- 目标流程：`init / fmt / validate / test / plan / apply / output / destroy`

这节的重点不是“能不能写出一行 `source = "../modules/xxx"`”，而是理解：

> Terraform root module 如何找到 child module，以及本地模块路径变化为什么会影响 `terraform init`、代码组织和团队协作。

---

## 2. 什么是本地路径引用模块？

Terraform module 有两类常见角色：

- **root module**：你执行 `terraform init / plan / apply` 的目录。
- **child module**：被 root module 通过 `module` block 调用的模块。

本地路径引用，就是 root module 通过文件系统路径调用 child module。

典型写法如下：

```hcl
module "example" {
  source = "./modules/example"

  name = "demo"
}
```

或者：

```hcl
module "example" {
  source = "../modules/example"

  name = "demo"
}
```

这里的 `source` 不是云资源地址，而是告诉 Terraform：

> 这个 child module 的代码在本地哪个目录。

---

## 3. 为什么要先学本地模块？

因为真实工程里，大部分团队不会一上来就把所有模块发布到公共 Registry。

更常见的演进顺序是：

1. 先在一个 root module 里把资源写通。
2. 发现重复代码后，把一部分资源抽成本地 child module。
3. 在同一个仓库内多处调用。
4. 等模块稳定后，再考虑拆成独立仓库、私有 Registry 或公共 Registry 模块。

所以，本地模块是 Terraform 工程化的第一步。

它解决的是：

- 代码复用；
- 目录分层；
- 复杂配置拆分；
- 多环境复用；
- 后续模块发布的基础。

---

## 4. 目录结构怎么理解？

一个常见结构如下：

```text
practice/labs/91/
├── main.tf
├── variables.tf
├── outputs.tf
├── tests/
│   └── basic.tftest.hcl
└── modules/
    └── naming/
        ├── main.tf
        ├── variables.tf
        └── outputs.tf
```

这里可以这样理解：

```text
practice/labs/91/        # root module，执行 Terraform 命令的地方
modules/naming/          # child module，被 root module 调用
tests/                   # terraform test 的测试文件
```

root module 通过类似下面的方式调用 child module：

```hcl
module "naming" {
  source = "./modules/naming"

  prefix = "lab91"
}
```

注意：`source = "./modules/naming"` 是相对于当前 root module 目录解析的。

也就是说，如果你在 `practice/labs/91` 里执行 Terraform，那么 `./modules/naming` 指向：

```text
practice/labs/91/modules/naming
```

---

## 5. `terraform init` 在本节中做了什么？

很多人以为 `terraform init` 只是下载 provider。  
其实它还会初始化 module。

当 Terraform 看到：

```hcl
module "naming" {
  source = "./modules/naming"
}
```

它会检查这个 module source，并准备后续 plan/apply/test 所需的模块代码。

本地路径模块通常不需要从网络下载，但仍然需要 init 来建立 Terraform 的工作目录元数据。

所以本节里，下面这一步不能跳过：

```sh
terraform init -input=false
```

如果你改了 module 的 source 路径，也应该重新执行 `terraform init`。

---

## 6. 本地路径引用的关键细节

### 6.1 相对路径是相对 root module 目录

假设你当前执行目录是：

```text
/workspace/practice/labs/91
```

那么：

```hcl
source = "./modules/naming"
```

表示：

```text
/workspace/practice/labs/91/modules/naming
```

而：

```hcl
source = "../modules/naming"
```

表示：

```text
/workspace/practice/modules/naming
```

路径写错，Terraform 就找不到模块。

---

### 6.2 本地模块不应该依赖调用方的内部变量

child module 应该通过 `variable` 接收输入，而不是偷偷依赖 root module 里的 local 或 variable。

不推荐：

```hcl
# child module 内部假设调用方一定有 local.project_name
name = local.project_name
```

推荐：

```hcl
variable "project_name" {
  type = string
}

resource "local_file" "example" {
  filename = "${var.project_name}.txt"
  content  = "demo"
}
```

然后调用方显式传入：

```hcl
module "example" {
  source = "./modules/example"

  project_name = "lab91"
}
```

这样模块才是可复用的。

---

### 6.3 模块输出要通过 `module.<name>.<output>` 读取

child module 内部可以定义 output：

```hcl
output "resource_name" {
  value = local.name
}
```

root module 读取时，不能直接访问 child module 内部资源，应该通过：

```hcl
output "name_from_module" {
  value = module.naming.resource_name
}
```

这就是模块边界。

模块像一个小黑盒：

- 输入：variables
- 内部实现：resources / locals / data sources
- 输出：outputs

调用方不应该乱摸模块内部资源。

## 7. 实验命令逐步讲解

原始实验要求进入：

```sh
cd /workspace/practice/labs/91
```

然后依次执行：

```sh
terraform init -input=false
terraform fmt -check
terraform validate
terraform test
terraform plan -input=false -no-color -out=tfplan
terraform apply -auto-approve tfplan
terraform output
terraform destroy -auto-approve
```

下面逐步解释每一步的意义。

---

### 7.1 初始化

```sh
terraform init -input=false
```

作用：

- 初始化 Terraform 工作目录；
- 解析本地 module source；
- 下载或准备 provider；
- 生成 `.terraform` 目录；
- 生成或更新 `.terraform.lock.hcl`。

本节要重点观察：  
本地模块虽然不需要从远程下载，但仍然属于 Terraform 初始化的一部分。

---

### 7.2 格式检查

```sh
terraform fmt -check
```

作用：

- 检查 `.tf` 文件是否符合 Terraform 标准格式；
- 不会自动修改文件；
- CI 里常用它来阻止不规范代码合并。

如果失败，可以本地运行：

```sh
terraform fmt -recursive
```

再重新检查。

---

### 7.3 静态校验

```sh
terraform validate
```

作用：

- 检查 Terraform 语法；
- 检查变量、输出、模块引用是否合理；
- 检查 provider 配置是否基本有效。

如果本地 module 路径写错，或者 module 输入变量没传，通常会在这里或 init 阶段暴露。

---

### 7.4 测试

```sh
terraform test
```

作用：

- 执行 `tests/` 目录下的 `.tftest.hcl` 文件；
- 验证 module 输出是否符合预期；
- 帮你把“我觉得写对了”变成“测试确实通过了”。

原始文件里给出的验证结果是：

```text
1 passed, 0 failed
```

这说明本节实验闭环已经验证通过。

---

### 7.5 生成计划

```sh
terraform plan -input=false -no-color -out=tfplan
```

作用：

- 计算 Terraform 准备创建、修改或删除什么；
- 把计划保存成 `tfplan`；
- 避免 apply 时重新计算计划。

这一步在 CI/CD 里很重要，因为它能把“审批的计划”和“真正执行的计划”绑定在一起。

---

### 7.6 应用计划

```sh
terraform apply -auto-approve tfplan
```

作用：

- 执行前面保存的计划；
- 不再交互式确认；
- 适合训练环境和自动化流水线。

注意：生产环境通常不会随便 `-auto-approve`，而是会加审批流程。

---

### 7.7 查看输出

```sh
terraform output
```

作用：

- 查看 root module 暴露出来的输出；
- 如果 root module 输出了 child module 的 output，你可以在这里看到模块调用结果。

本节可以重点观察：

```hcl
module.<module_name>.<output_name>
```

是如何被 root module output 再次暴露出来的。

---

### 7.8 销毁资源

```sh
terraform destroy -auto-approve
```

作用：

- 清理实验创建的资源；
- 避免残留 state 和本地文件影响下次实验。

虽然本节多半是本地模拟资源，但养成销毁习惯非常重要。

---

## 8. 本节最容易踩的坑

### 坑 1：路径写错

错误示例：

```hcl
source = "./module/naming"
```

但真实目录叫：

```text
modules/naming
```

这会导致 Terraform 找不到 child module。

排查方式：

```sh
ls -R
```

确认目录结构，再检查 `source`。

---

### 坑 2：改了模块路径但没重新 init

修改前：

```hcl
source = "./modules/naming"
```

修改后：

```hcl
source = "./modules/name"
```

这时应该重新执行：

```sh
terraform init
```

否则 Terraform 可能还停留在旧初始化状态。

---

### 坑 3：child module 输入变量没传

child module 定义：

```hcl
variable "prefix" {
  type = string
}
```

调用时如果没传：

```hcl
module "naming" {
  source = "./modules/naming"
}
```

就会报错。

正确写法：

```hcl
module "naming" {
  source = "./modules/naming"

  prefix = "lab91"
}
```

---

### 坑 4：调用方直接访问 child module 内部资源

不推荐：

```hcl
module.naming.local_file.example.filename
```

正确方式是 child module 先输出：

```hcl
output "filename" {
  value = local_file.example.filename
}
```

root module 再读取：

```hcl
module.naming.filename
```

模块边界清楚，后续重构才不会爆炸。

---

## 9. 生产环境中怎么用？

本地路径模块适合这些场景：

1. **单仓库多模块**
   - 网络模块；
   - 命名模块；
   - 标签模块；
   - IAM 模块；
   - 监控模块。

2. **团队内部早期封装**
   - 模块还没稳定；
   - 先在 monorepo 内使用；
   - 等接口稳定后再拆出去。

3. **训练和 PoC**
   - 不依赖网络；
   - 不需要发布模块；
   - 适合快速验证 module 设计。

4. **CI 流水线测试**
   - 用 `terraform test` 验证模块输出；
   - 用 `terraform validate` 保证引用路径正确；
   - 用 `fmt` 保证格式一致。

## 10. 和远程模块的区别

本地模块：

```hcl
module "example" {
  source = "./modules/example"
}
```

优点：

- 简单；
- 不依赖网络；
- 适合开发中模块；
- 修改后立即可测。

缺点：

- 只能在当前仓库或当前文件系统结构中复用；
- 不适合跨团队大规模共享；
- 路径结构变化会影响调用方。

远程 Git 模块：

```hcl
module "example" {
  source = "git::https://github.com/example/terraform-module.git?ref=v1.0.0"
}
```

优点：

- 可跨仓库复用；
- 可以用 tag 固定版本；
- 适合团队或组织级共享。

缺点：

- 需要网络；
- 版本管理更复杂；
- 调试周期稍长。

所以训练顺序很合理：

> 先本地模块，再远程模块，再 Registry 模块。

## 11. 考试和面试怎么考？

### 可能考点 1：root module 和 child module 区别

你应该能回答：

- root module 是执行 Terraform 命令的目录；
- child module 是被 `module` block 调用的目录；
- child module 通过 variables 接收输入，通过 outputs 暴露结果。

---

### 可能考点 2：本地 module source 路径

看到：

```hcl
source = "./modules/vpc"
```

要知道它是本地相对路径，不是 Registry 模块。

---

### 可能考点 3：什么时候需要 `terraform init`

当你新增或修改 module block，特别是改了 `source` 时，应该重新执行：

```sh
terraform init
```

---

### 可能考点 4：模块边界

调用方应该通过：

```hcl
module.name.output_name
```

读取模块输出，而不是直接依赖模块内部资源细节。

## 13. 清理命令

实验结束后可以执行：

```sh
terraform destroy -auto-approve
rm -f tfplan
rm -rf output logs .terraform .terraform.lock.hcl terraform.tfstate terraform.tfstate.backup
```

这一步的目的不是为了“仪式感”，而是避免：

- state 残留；
- plan 文件残留；
- `.terraform` 初始化缓存影响下一次实验；
- 本地输出文件污染后续测试。

---

## 14. 一句话总结

本节要记住一句话：

> 本地路径引用模块，是 Terraform 模块化的第一步；核心不是路径本身，而是学会用清晰的输入、输出和目录边界，把一坨 Terraform 配置拆成可复用的工程单元。
