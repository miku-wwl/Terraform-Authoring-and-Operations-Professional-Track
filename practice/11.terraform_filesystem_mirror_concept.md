# Terraform 实操训练 11：File System Mirror 概念

## 1. 本节核心

这一节讲的是 **File System Mirror**，也就是把 Terraform Provider 预先下载到一个本地目录，然后让 `terraform init` 从这个本地目录安装 provider，而不是直接访问 `registry.terraform.io`。

一句话总结：

> **Plugin Cache 是为了减少重复下载；File System Mirror 是为了在离线、内网、受控网络环境中安装 provider。**

这节比前面的 plugin cache 更偏企业生产场景，尤其适合：

- CI runner 不能访问公网；
- 公司要求所有 provider 先经过安全扫描；
- 涉密、金融、政府、内网环境；
- 想把 provider 安装来源固定到内部镜像目录；
- 想减少对公网 Terraform Registry 的直接依赖。

---

## 2. Plugin Cache 和 File System Mirror 的区别

很多人会把 plugin cache 和 file system mirror 混在一起，但它们解决的问题不一样。

| 项目 | Plugin Cache | File System Mirror |
|---|---|---|
| 主要目标 | 加速重复下载 | 离线或内网安装 provider |
| 典型场景 | CI 高频执行 `terraform init` | runner 不能访问公网 registry |
| 是否主动作为安装来源 | 不是主要安装源，只是缓存 | 是明确的 provider 安装来源 |
| 配置方式 | `TF_PLUGIN_CACHE_DIR` / `plugin_cache_dir` | `provider_installation` + `filesystem_mirror` |
| 是否适合受控企业环境 | 一般 | 很适合 |
| 是否能替代 lock file | 不能 | 也不能 |

简单理解：

```text
Plugin Cache:
  Terraform 还是正常找 registry，只是下载过的 provider 可以复用。

File System Mirror:
  Terraform 被配置为从指定本地 mirror 目录找 provider。
```

所以 Lab 11 的工程价值比 Lab 9/10 更强一些。

## 3. 为什么企业需要 File System Mirror？

默认情况下，Terraform 初始化 provider 时会访问公网 registry：

```text
registry.terraform.io
```

但生产企业里经常有这些限制：

```text
CI runner 不能出公网
生产网络禁止直接访问第三方下载源
所有二进制包必须先经过安全扫描
需要固定 provider 供应链来源
需要可审计、可复现的安装流程
```

这时候如果你直接执行：

```sh
terraform init -input=false
```

可能会失败，因为 runner 根本访问不了 Terraform Registry。

File System Mirror 的作用就是：

```text
先在一个能访问公网的环境中下载 provider
        ↓
把 provider 放进 mirror 目录
        ↓
把 mirror 目录复制到内网 runner 或共享文件系统
        ↓
terraform init 从本地 mirror 安装 provider
```


## 5. `terraform providers mirror` 在干嘛？

命令：

```sh
terraform providers mirror /workspace/practice/labs/11/mirror
```

它的作用是根据当前 Terraform 配置需要的 provider，把 provider 包下载到指定目录。

比如本节使用的是：

```text
hashicorp/local v2.9.0
```

执行后，mirror 目录里会出现类似结构：

```text
mirror/
  registry.terraform.io/
    hashicorp/
      local/
        terraform-provider-local_2.9.0_linux_amd64.zip
        index.json
        2.9.0.json
```

这个目录就是本地 provider 镜像。

注意：

```text
terraform providers mirror
```

是“准备 provider mirror”的动作。它需要在能访问 provider 源的环境中执行。

如果 runner 完全离线，你不能指望它自己执行 mirror 下载。真实企业里通常是：

```text
联网环境下载 provider mirror
        ↓
安全扫描
        ↓
发布到内网文件服务器 / artifact registry / 镜像目录
        ↓
内网 CI runner 使用 mirror
```

## 6. 为什么要删除 `.terraform` 和 `.terraform.lock.hcl`？

命令：

```sh
rm -rf .terraform .terraform.lock.hcl
```

这是为了模拟一个干净环境。

如果不删除 `.terraform`，Terraform 可能已经有初始化结果，你就看不出它是不是从 mirror 安装 provider。

如果不删除 `.terraform.lock.hcl`，Terraform 可能直接沿用已有 provider 选择。实验里删除它，是为了更明显地验证：

```text
terraform init 可以在指定 CLI 配置下，从 file system mirror 完成 provider 安装。
```

但是生产中不要随便删 `.terraform.lock.hcl`。

生产推荐：

```text
提交 .terraform.lock.hcl
不要提交 .terraform/
不要提交 mirror/
```

mirror 通常作为外部供应链或 artifact 管理，不直接放进业务仓库。

## 7. `TF_CLI_CONFIG_FILE` 是什么？

命令：

```sh
TF_CLI_CONFIG_FILE=/workspace/practice/labs/11/terraform-cli.rc terraform init -input=false
```

`TF_CLI_CONFIG_FILE` 的作用是指定 Terraform CLI 配置文件路径。

默认情况下，Terraform 会读取用户目录里的 CLI 配置文件，例如：

```text
~/.terraformrc
~/.terraform.d/terraform.rc
```

但在 CI / Docker / 实验环境中，我们不想依赖用户主目录里的配置，所以直接用环境变量指定：

```text
请 Terraform 使用这个配置文件：
/workspace/practice/labs/11/terraform-cli.rc
```

这非常适合 CI，因为路径明确、行为可控。

## 8. `terraform-cli.rc` 里通常写什么？

本节的 `terraform-cli.rc` 大概率包含类似配置：

```hcl
provider_installation {
  filesystem_mirror {
    path    = "/workspace/practice/labs/11/mirror"
    include = ["registry.terraform.io/hashicorp/local"]
  }

  direct {
    exclude = ["registry.terraform.io/hashicorp/local"]
  }
}
```

意思是：

```text
对于 registry.terraform.io/hashicorp/local：
  从 /workspace/practice/labs/11/mirror 安装

direct 中排除 hashicorp/local：
  不允许它再直接去公网 registry 下载
```

这就是 file system mirror 的核心配置。

### `include` 的作用

```hcl
include = ["registry.terraform.io/hashicorp/local"]
```

表示这个 mirror 负责哪些 provider。

### `direct` 的作用

```hcl
direct {
  exclude = ["registry.terraform.io/hashicorp/local"]
}
```

表示除了被排除的 provider，其他 provider 仍然可以走默认直接下载。

如果企业完全禁止公网下载，可以更严格地配置，让所有 provider 都必须来自 mirror。

## 9. 为什么 init 显示 `unauthenticated`？

验证报告里说：

```text
初始化时显示 provider 从 mirror 安装，签名状态为 unauthenticated，
这是 file system mirror 的常见提示。
```

这个容易吓人，但不要立刻误判为失败。

因为从 file system mirror 安装 provider 时，Terraform 看到的是本地 mirror 目录中的 provider 包，而不是直接从 Terraform Registry 走完整的在线签名验证流程。

所以它可能显示：

```text
unauthenticated
```

这不是说 provider 一定不安全，而是说：

```text
Terraform 当前没有通过在线 registry 的签名链路完成认证显示。
```

企业里真正要解决这个问题，不是靠临时下载，而是靠供应链流程：

```text
provider 来源控制
checksum 校验
lock file 审查
mirror 发布流程
artifact 扫描
访问权限控制
```

所以要记住：

```text
unauthenticated 是 file system mirror 场景中常见提示，
但生产里必须用流程保证 mirror 内容可信。
```

## 10. File System Mirror 和 `.terraform.lock.hcl` 的关系

File system mirror 也不能替代 lock file。

关系是：

```text
filesystem mirror:
  provider 从哪里安装

.terraform.lock.hcl:
  provider 应该是什么版本、什么 checksum
```

也就是说：

```text
mirror 决定来源
lock file 决定版本和完整性
```

真实生产里应该是：

```text
1. required_providers 约束允许版本范围
2. .terraform.lock.hcl 锁定具体版本和 checksum
3. filesystem mirror 提供对应 provider 包
4. terraform init 从 mirror 安装并校验
```

## 11. 多平台 checksum 警告是什么？

排障指南里写：

```text
多平台校验和警告：按提示用 terraform providers lock -platform=... 补齐。
```

原因是同一个 provider 版本，不同平台的包不一样：

```text
linux_amd64
linux_arm64
windows_amd64
darwin_amd64
darwin_arm64
```

如果 lock file 里只有 Linux 的 checksum，Windows 或 macOS 开发者可能会遇到校验和问题。

团队可以提前执行：

```sh
terraform providers lock \
  -platform=linux_amd64 \
  -platform=linux_arm64 \
  -platform=windows_amd64 \
  -platform=darwin_amd64 \
  -platform=darwin_arm64
```

这样 lock file 会包含多个目标平台的 checksums。

## 13. 排障指南解释

### 1. mirror 目录为空

问题：

```text
terraform init 找不到 provider
```

原因可能是没有先执行：

```sh
terraform providers mirror /workspace/practice/labs/11/mirror
```

解决：

```sh
rm -rf mirror
terraform providers mirror /workspace/practice/labs/11/mirror
```

---

### 2. `terraform.rc` 路径无效

问题：

```text
Terraform 没有使用你的 mirror 配置
```

原因可能是路径写成了宿主机路径，而容器里看不到。

Docker / CI 里要用容器内路径：

```text
/workspace/practice/labs/11/terraform-cli.rc
```

而不是你本机的：

```text
C:\Users\xxx\...
/Users/xxx/...
```

---

### 3. 多平台校验和警告

问题：

```text
当前 lock file 不包含目标平台 checksum
```

解决：

```sh
terraform providers lock \
  -platform=linux_amd64 \
  -platform=windows_amd64 \
  -platform=darwin_arm64
```

然后提交更新后的：

```text
.terraform.lock.hcl
```

## 15. 生产级理解

File System Mirror 的生产定位是：

```text
Provider 供应链控制
离线环境安装
内网 CI runner 支持
减少公网依赖
提升可审计性
```

它比 plugin cache 更像企业能力。

### Plugin Cache 的定位

```text
提高 terraform init 速度
减少重复下载
主要是性能优化
```

### File System Mirror 的定位

```text
控制 provider 安装来源
支持离线或受限网络
主要是供应链和网络治理能力
```

所以这节比 cache 更值得放进生产工程化知识体系里。

## 16. 面试级总结

英文表达：

> File system mirror is used when Terraform runners cannot directly access the public provider registry, such as in private, restricted, or air-gapped environments. We can use `terraform providers mirror` to pre-download provider packages into a local mirror directory, and then configure Terraform CLI with `provider_installation` and `filesystem_mirror`, usually through `TF_CLI_CONFIG_FILE` in CI. Unlike plugin cache, which is mainly a performance optimization, file system mirror controls the provider installation source. It still does not replace `.terraform.lock.hcl`, which locks provider versions and checksums.

中文表达：

> File System Mirror 主要解决离线或受限网络环境下 provider 安装问题。通过 `terraform providers mirror` 预先下载 provider，再通过 CLI 配置文件中的 `provider_installation` 指向本地 mirror，让 `terraform init` 从 mirror 安装 provider。它不同于 plugin cache，cache 主要是加速，mirror 主要是控制安装来源和支持离线环境。mirror 不能替代 `.terraform.lock.hcl`，lock file 仍然负责 provider 版本和 checksum 的一致性。
