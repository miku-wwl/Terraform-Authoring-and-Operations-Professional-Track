# Terraform 实操训练 102：为已有资源启用 `count`

## 1. 本节在讲什么

这一节的核心不是单纯学习 `count` 语法，而是学习一个非常常见的生产级重构场景：

> 原来 Terraform 里写了两个独立资源，现在希望把它们重构成一个带 `count` 的资源，但又不想让 Terraform 误以为资源被删除再新建。

也就是说，本节真正要解决的问题是：

- 代码结构要变得更整洁；
- Terraform state 里的资源地址也要跟着变；
- 但真实资源不能被销毁；
- 迁移过程必须可控、可审计、可验证。

这就是 `moved` block 的价值。

---

## 2. 为什么这个场景很重要

在真实项目中，Terraform 代码通常不是一开始就写得很完美。

一开始你可能这样写：

```hcl
resource "aws_s3_bucket" "logs_a" {
  bucket = "example-logs-a"
}

resource "aws_s3_bucket" "logs_b" {
  bucket = "example-logs-b"
}
```

后来你发现这两个资源结构几乎一样，于是想重构成：

```hcl
resource "aws_s3_bucket" "logs" {
  count  = 2
  bucket = var.bucket_names[count.index]
}
```

从代码角度看，这样更统一。

但是 Terraform 看资源不是只看“资源长得像不像”，而是看 **state 地址**。

原来的地址可能是：

```text
aws_s3_bucket.logs_a
aws_s3_bucket.logs_b
```

重构后的地址变成：

```text
aws_s3_bucket.logs[0]
aws_s3_bucket.logs[1]
```

如果你不告诉 Terraform 这是一场“搬家”，Terraform 可能会认为：

- 旧资源没了，要删除；
- 新资源出现了，要创建。

这在生产环境里就很危险。你只是想整理代码，结果 Terraform 可能准备删真实资源。

---

## 3. `moved` block 的作用

`moved` block 是 Terraform 用来声明资源地址迁移关系的机制。

它的语义非常直接：

> 这个旧地址，不是真的删除了，而是搬到了这个新地址。

例如：

```hcl
moved {
  from = aws_s3_bucket.logs_a
  to   = aws_s3_bucket.logs[0]
}

moved {
  from = aws_s3_bucket.logs_b
  to   = aws_s3_bucket.logs[1]
}
```

这段配置告诉 Terraform：

- `aws_s3_bucket.logs_a` 迁移到 `aws_s3_bucket.logs[0]`；
- `aws_s3_bucket.logs_b` 迁移到 `aws_s3_bucket.logs[1]`；
- 不要把它们当成删除和新建；
- 应该只更新 state 里的资源地址映射。

一句话总结：

> `moved` block 是 Terraform 重构代码时的“户口迁移证明”。

没有它，Terraform 可能觉得旧户口注销、新户口新建。

有了它，Terraform 知道只是搬家。

---

## 4. `count` 改造前后的核心差异

### 改造前：多个资源独立声明

```hcl
resource "aws_s3_bucket" "bucket_a" {
  bucket = "demo-a"
}

resource "aws_s3_bucket" "bucket_b" {
  bucket = "demo-b"
}
```

优点：

- 简单直观；
- 初学者容易看懂；
- 每个资源有独立名字。

缺点：

- 重复代码多；
- 资源数量变多后维护困难；
- 不方便用变量统一管理。

---

### 改造后：使用 `count`

```hcl
variable "bucket_names" {
  type = list(string)
}

resource "aws_s3_bucket" "bucket" {
  count  = length(var.bucket_names)
  bucket = var.bucket_names[count.index]
}
```

优点：

- 代码更简洁；
- 资源数量由变量控制；
- 适合批量创建结构相似的资源。

缺点：

- 地址变成带索引的形式；
- `count.index` 顺序很重要；
- 如果列表顺序变化，可能引发资源错位；
- 对已有资源改造时必须处理 state 迁移。

---

## 5. 为什么不能直接改代码

这是本节最容易踩坑的地方。

很多人会觉得：

> 反正资源名字、参数、bucket 名都一样，我直接把两个 resource 改成一个 count resource 不就行了吗？

不行。

Terraform 判断资源身份时，核心依据不是云上的名字，而是 state 里的资源地址。

例如云上同一个 bucket，在 Terraform 眼里：

```text
aws_s3_bucket.bucket_a
```

和：

```text
aws_s3_bucket.bucket[0]
```

是两个不同地址。

所以直接改代码可能导致 plan 里出现：

```text
- destroy aws_s3_bucket.bucket_a
+ create  aws_s3_bucket.bucket[0]
```

这不是你想要的。

你想要的是：

```text
# aws_s3_bucket.bucket_a has moved to aws_s3_bucket.bucket[0]
```

这才说明 Terraform 理解了你的重构意图。

## 6. 本实验为什么使用 LocalStack

本节实验不访问真实 AWS，而是通过 LocalStack 模拟 AWS API。

实验中的 AWS Provider 会被配置到类似这样的 endpoint：

```text
http://localhost:4566
```

并使用假凭证：

```text
test/test
```

这样做的好处是：

- 不需要真实 AWS 账号；
- 不会产生真实云资源费用；
- 可以本地重复练习；
- 适合验证 Terraform 行为；
- 适合训练 state 迁移、plan 观察、apply 验证这些流程。

但是要注意：

> LocalStack 是模拟环境，不等于真实 AWS 生产环境。

它适合练 Terraform 行为和迁移流程，但真实项目上线前仍然要在真实 AWS 环境中做更严格验证。

## 8. 观察 plan 时应该看什么

本节最关键的不是命令跑通，而是观察 `terraform plan` 的输出。

你要重点看：

### 1. 是否出现 moved 提示

理想情况下，plan 中应该能体现类似含义：

```text
resource has moved
```

或者显示旧地址移动到了新地址。

这说明 Terraform 识别了迁移关系。

---

### 2. 是否避免了 destroy/create

你需要警惕这种输出：

```text
- destroy
+ create
```

如果只是为了启用 `count`，却出现资源销毁和重建，通常说明迁移没有写对。

正确目标是：

```text
0 to add, 0 to change, 0 to destroy
```

或者只有 state 地址迁移，而不是实际资源替换。

---

### 3. `from` 和 `to` 是否一一对应

例如：

```hcl
moved {
  from = aws_s3_bucket.bucket_a
  to   = aws_s3_bucket.bucket[0]
}
```

这里的 `[0]` 必须和你的变量列表顺序对应。

如果顺序错了，Terraform 可能不会销毁资源，但你的逻辑映射会错。

这类错误更阴险，因为 plan 可能看起来没炸，但资源语义已经错位。

---

## 9. `count` 的真实项目风险

`count` 很好用，但它有一个经典风险：**索引依赖顺序**。

例如：

```hcl
bucket_names = ["a", "b", "c"]
```

对应地址是：

```text
aws_s3_bucket.bucket[0] -> a
aws_s3_bucket.bucket[1] -> b
aws_s3_bucket.bucket[2] -> c
```

如果后来你把列表改成：

```hcl
bucket_names = ["b", "c"]
```

Terraform 可能会认为：

```text
bucket[0] 从 a 变成 b
bucket[1] 从 b 变成 c
bucket[2] 删除
```

这可能造成资源错位。

所以生产环境里，如果资源有稳定业务身份，很多时候 `for_each` 比 `count` 更安全。

例如：

```hcl
resource "aws_s3_bucket" "bucket" {
  for_each = toset(var.bucket_names)
  bucket   = each.value
}
```

地址会变成：

```text
aws_s3_bucket.bucket["a"]
aws_s3_bucket.bucket["b"]
```

这种地址更稳定，不容易因为列表顺序变化而错位。

但是本节重点是练习 **已有资源迁移到 count 地址**，所以重点仍然是 `count + moved`。

## 10. 常见错误

### 错误 1：只改 resource，不写 moved block

表现：

```text
Terraform wants to destroy old resources and create new indexed resources.
```

原因：

Terraform 不知道你是在重构。

解决：

为每个旧地址写一个明确的 `moved` block。

---

### 错误 2：`to` 地址索引写错

例如把 A 迁移到 `[1]`，把 B 迁移到 `[0]`。

这种错误可能不会立刻炸，但会造成资源语义混乱。

解决：

检查变量列表顺序，确保：

```text
旧资源 A -> 新资源 [0]
旧资源 B -> 新资源 [1]
```

---

### 错误 3：以为 `moved` 会改变云资源

`moved` block 主要影响 Terraform state 地址迁移。

它不是用来改云资源属性的。

如果你同时修改了资源参数，Terraform 仍然可能产生真实变更。

解决：

迁移步骤尽量拆开：

1. 先只做地址迁移；
2. 确认没有 destroy/create；
3. 再做属性变更。

---

### 错误 4：迁移后马上删除 moved block

`moved` block 可以在团队协作中保留一段时间。

原因是：

- 不是所有人都在同一时间 apply；
- CI/CD 环境可能还有旧 state；
- 分支合并顺序可能不同。

过早删除 `moved` block，会让后续环境失去迁移信息。

---

## 11. 面试角度怎么讲

如果面试官问：

> Terraform 里怎么把已有资源迁移成 `count` 管理，避免重建？

可以这样回答：

> 我不会直接改资源地址，因为 Terraform state 以资源地址识别对象。把独立资源改成 `count` 后，地址会从 `aws_xxx.a` 变成 `aws_xxx.name[0]` 这种索引形式。如果直接改代码，Terraform 可能计划 destroy/create。正确做法是用 `moved` block 显式声明旧地址到新地址的迁移关系，然后通过 plan 检查是否只发生 state 地址迁移，而不是资源重建。生产环境里我会先单独做迁移 PR，只做地址变更，不混入属性变更，并且保留 moved block 一段时间，确保不同环境都完成迁移。

这个回答比只说“用 moved block”更像真实工程师。

## 14. 本节总结

本节训练的是 Terraform 重构能力。

你需要掌握的不是“`count` 怎么写”，而是：

- 已有资源为什么不能直接改成 `count`；
- Terraform 为什么会把旧地址和新地址当成不同资源；
- `moved` block 如何声明地址迁移；
- 如何通过 plan 判断是否避免销毁重建；
- LocalStack 能验证什么，不能替代什么；
- 生产环境里为什么要把 state 迁移和属性变更拆开做。

真正的重点是这句话：

> Terraform 重构不是改代码这么简单，本质上是在安全地迁移 state 里的资源身份。
