# Terraform 实操训练 50 讲解：list 数据类型

## 1. 这一节到底在学什么？

Lab 50 学的是 Terraform 里的 `list` 数据类型。

前面很多 lab 已经开始进入 Terraform 工程化主题，比如 lifecycle、Vault、sensitive、check block。  
从 Lab 50 开始，重点回到 Terraform 语言本身的数据建模能力。

`list` 是 Terraform 最常用的数据类型之一。它可以表示一组**有顺序**的值。

一句话理解：

> `list` 是 Terraform 里的有序列表，适合表达一组同类型、顺序有意义的数据。

---

## 2. 什么是 list？

Terraform 中的 list 通常长这样：

```hcl
variable "subnet_names" {
  type = list(string)

  default = [
    "subnet-a",
    "subnet-b",
    "subnet-c"
  ]
}
```

这个变量表示一组字符串。

它有几个特点：

1. 有顺序。
2. 可以通过索引访问。
3. 通常要求元素类型一致。
4. 适合传给模块或资源循环使用。
5. 可以和 `length()`、`for`、`join()`、`contains()` 等函数配合。

---

## 3. list 和 string 有什么区别？

普通字符串是一个值：

```hcl
variable "environment" {
  type    = string
  default = "dev"
}
```

list 是多个值：

```hcl
variable "environments" {
  type    = list(string)
  default = ["dev", "test", "prod"]
}
```

如果你需要表达“多个同类对象”，就应该考虑 list。

例如：

- 多个 subnet 名称
- 多个 availability zone
- 多个允许端口
- 多个 tag key
- 多个服务名称
- 多个 IP 地址
- 多个告警接收人

---

## 4. list 的核心特征：有序

list 最重要的特征是**有序**。

例如：

```hcl
locals {
  zones = ["az-a", "az-b", "az-c"]
}
```

它的顺序是固定的：

```text
zones[0] = az-a
zones[1] = az-b
zones[2] = az-c
```

所以可以通过索引访问：

```hcl
local.zones[0]
```

返回：

```text
az-a
```

注意 Terraform 索引从 0 开始，不是从 1 开始。

---

## 5. list 的索引访问

示例：

```hcl
locals {
  services = ["api", "worker", "frontend"]
}

output "first_service" {
  value = local.services[0]
}
```

输出：

```text
first_service = "api"
```

如果访问：

```hcl
local.services[3]
```

就会越界，因为只有 0、1、2 三个索引。

所以生产里访问 list 索引时，要小心长度。

---

## 6. 常用函数：`length()`

`length()` 可以计算 list 里有多少个元素。

```hcl
locals {
  services = ["api", "worker", "frontend"]
}

output "service_count" {
  value = length(local.services)
}
```

输出：

```text
service_count = 3
```

这个函数很常见，适合：

- 校验列表非空
- 输出资源数量
- 判断是否启用某些逻辑
- 配合 `count` 创建资源

---

## 7. 常用函数：`contains()`

`contains()` 可以判断 list 是否包含某个值。

```hcl
locals {
  allowed_envs = ["dev", "test", "prod"]
}

output "is_valid_env" {
  value = contains(local.allowed_envs, "dev")
}
```

返回：

```text
true
```

Lab 38 的白名单验证就大量依赖这个思路。

例如：

```hcl
validation {
  condition     = contains(["dev", "test", "prod"], var.environment)
  error_message = "environment must be one of: dev, test, prod."
}
```

这里 list 就是白名单。

---

## 8. 常用函数：`join()`

`join()` 可以把 list 拼成字符串。

```hcl
locals {
  services = ["api", "worker", "frontend"]
}

output "service_csv" {
  value = join(",", local.services)
}
```

输出：

```text
api,worker,frontend
```

生产里常用于：

- 拼接说明文本
- 生成配置文件
- 生成标签内容
- 输出摘要信息
- 把端口列表或服务列表变成字符串

---

## 9. list 和 `for` 表达式

list 经常和 `for` 表达式配合使用。

例如把服务名统一加前缀：

```hcl
locals {
  services = ["api", "worker", "frontend"]

  service_names = [
    for service in local.services : "svc-${service}"
  ]
}
```

结果是：

```hcl
[
  "svc-api",
  "svc-worker",
  "svc-frontend"
]
```

这就是 Terraform 数据转换的常见写法。

---

## 10. list 和 `count`

`count` 经常和 list 配合使用。

示例：

```hcl
variable "file_names" {
  type    = list(string)
  default = ["a.txt", "b.txt", "c.txt"]
}

resource "local_file" "example" {
  count = length(var.file_names)

  filename = "${path.module}/output/${var.file_names[count.index]}"
  content  = "generated by Terraform"
}
```

这里：

```hcl
count.index
```

会依次是：

```text
0
1
2
```

然后分别读取：

```hcl
var.file_names[0]
var.file_names[1]
var.file_names[2]
```

这种模式适合简单列表。

---

## 11. list + count 的风险：索引漂移

虽然 list + count 很方便，但生产里要小心索引漂移。

比如原来：

```hcl
["api", "worker", "frontend"]
```

对应资源地址：

```text
local_file.example[0] -> api
local_file.example[1] -> worker
local_file.example[2] -> frontend
```

如果你删除中间的 `"worker"`：

```hcl
["api", "frontend"]
```

Terraform 可能会认为：

```text
example[1] 从 worker 变成 frontend
example[2] 要删除
```

这可能导致不必要的更新或替换。

如果元素有稳定名字，生产里通常更推荐 `for_each` 搭配 map 或 set。

一句话：

> list 适合顺序有意义的场景；如果你真正关心的是名字稳定，优先考虑 map + for_each。

---

## 12. list 和 set 的区别

list 和 set 都能表达多个值，但差别很大。

| 类型 | 是否有序 | 是否允许重复 | 适合场景 |
|---|---|---|---|
| list | 有序 | 允许重复 | 顺序有意义的数据 |
| set | 无序 | 不允许重复 | 去重集合、成员关系 |
| map | 按 key 访问 | key 唯一 | 命名对象集合 |

例如：

```hcl
list(string)
```

适合：

```hcl
["subnet-a", "subnet-b", "subnet-c"]
```

如果你只是关心“是否包含某个值”，不关心顺序，可以考虑 set。

如果你要按名字稳定管理资源，更适合 map。

---

## 13. list 的类型约束

Terraform 里建议明确写类型：

```hcl
type = list(string)
```

这表示 list 里的每个元素都应该是字符串。

还可以是：

```hcl
type = list(number)
```

例如端口：

```hcl
variable "ports" {
  type    = list(number)
  default = [80, 443, 8080]
}
```

也可以是对象列表：

```hcl
variable "services" {
  type = list(object({
    name = string
    port = number
  }))

  default = [
    {
      name = "api"
      port = 8080
    },
    {
      name = "worker"
      port = 9090
    }
  ]
}
```

对象列表更适合表达结构化数据。

## 18. 常见坑

### 坑 1：索引越界

如果 list 只有三个元素：

```hcl
["a", "b", "c"]
```

合法索引是：

```text
0, 1, 2
```

访问：

```hcl
var.items[3]
```

会失败。

---

### 坑 2：把 list 当 map 用

list 用数字索引访问：

```hcl
var.items[0]
```

map 用 key 访问：

```hcl
var.items["api"]
```

不要混。

---

### 坑 3：list 顺序变化导致资源变化

如果 list 和 `count` 绑定，改顺序可能影响资源地址和计划结果。

生产中要格外小心。

---

### 坑 4：元素类型不一致

如果声明：

```hcl
type = list(string)
```

就不要混入数字或对象。

错误示例：

```hcl
["api", 8080, true]
```

这种数据应该重新建模。

---

### 坑 5：该用 map 却用了 list

如果你管理的是一组有名字的对象，比如服务：

```text
api
worker
frontend
```

并且希望资源地址稳定，map + for_each 往往比 list + count 更好。

---

## 19. 生产环境怎么选择 list / set / map？

可以按这个判断：

### 用 list

当你关心顺序时：

- subnet 顺序
- priority 顺序
- 配置文件行顺序
- 展示顺序
- 有序端口列表

### 用 set

当你只关心成员关系，不关心顺序，而且不允许重复时：

- allowed regions
- allowed environments
- enabled features
- security group cidr 去重

### 用 map

当每个元素有稳定名字时：

- services by name
- environments by name
- users by username
- subnets by AZ name
- policies by policy name

一句话：

> 顺序重要用 list，去重集合用 set，稳定 key 管理对象用 map。

## 20. SRE / Platform Engineering 视角

从平台工程角度看，list 不是小语法，而是建模工具。

例如你设计一个模块：

```hcl
variable "allowed_ports" {
  type = list(number)
}
```

这表达的是：

> 用户可以传一组端口，模块会按顺序处理它们。

如果你设计：

```hcl
variable "services" {
  type = list(object({
    name = string
    port = number
  }))
}
```

这表达的是：

> 用户可以传一组服务定义，每个服务都有 name 和 port。

但是如果你要用这些服务创建长期资源，可能要问：

> 这些服务是不是应该用 map，以服务名作为稳定 key？

这就是 Terraform 建模能力。

## 22. 面试/考试记忆版

可以这样记：

- `list` 是有序集合。
- list 可以通过 `[index]` 访问。
- Terraform 索引从 0 开始。
- `length(list)` 返回元素数量。
- `contains(list, value)` 判断是否包含。
- `join(separator, list)` 可以拼接字符串列表。
- list + count 简单好用，但可能有索引漂移风险。
- 如果元素有稳定名字，生产里常用 map + for_each。
- 顺序重要用 list；去重用 set；按 key 管对象用 map。

一句话总结：

> Terraform list 适合表达有顺序的一组值，但长期资源建模时要警惕 list + count 带来的索引漂移。
