# Terraform 实操训练 25 校正版：Quoted String 与转义序列

## 1. 这一课到底在学什么？

这一课学习 Terraform 里的 **quoted string**，也就是用双引号包起来的字符串。

Terraform 配置里经常要写字符串：

- 文件内容
- 文件路径
- 资源名称
- JSON / YAML 片段
- shell 命令
- policy 文本
- Windows 路径
- 带引号的说明文字

一旦字符串内容本身包含双引号、反斜杠、换行符，就不能随便直接写。你需要使用**转义序列**，告诉 Terraform：这里的字符是字符串内容，不是 Terraform 语法的一部分。

这一课的核心是三个常用转义：

```hcl
\"   # 在字符串内容里保留一个双引号
\\   # 在字符串内容里保留一个反斜杠
\n   # 在字符串内容里插入一个换行
```

一句话总结：

> quoted string 是 Terraform 最普通的字符串；转义序列是为了在字符串里安全表达特殊字符。

## 2. 原教程为什么会出错？

原教程的问题不是思路错，而是**转译过程中把反斜杠吃掉了**。

比如原文里应该显示：

```hcl
content = "service_name = \"payment-api\""
```

但转译后变成了：

```hcl
content = "service_name = "payment-api""
```

这就不是“讲解内容”了，而是**错误 Terraform 代码**。Terraform 会把第二个双引号当成字符串结束，后面的 `payment-api` 就变成无法解析的语法。

再比如 Windows 路径应该写成：

```hcl
content = "C:\\tools\\terraform"
```

但如果转译后变成：

```hcl
content = "C:\tools\terraform"
```

这里的 `\t` 会被当成 tab 转义，最终内容就不是你想要的 `C:\tools\terraform`。

所以这一版的重点是把所有 `\"`、`\\`、`\n` 都恢复正确。

## 3. Terraform quoted string 的基本规则

Terraform 中最常见的字符串写法是：

```hcl
"hello terraform"
```

quoted string 适合短字符串。如果字符串内部出现特殊字符，就要使用转义序列。

---

## 4. 保留双引号：`\"`

你希望最终生成的文本是：

```text
service_name = "payment-api"
```

Terraform 代码里要写：

```hcl
content = "service_name = \"payment-api\""
```

分层理解：

```text
Terraform 代码里的写法：content = "service_name = \"payment-api\""
Terraform 解析后的内容：service_name = "payment-api"
```

外层 `"` 是 Terraform 字符串边界；里面的 `\"` 才表示字符串内容中的普通双引号。

错误写法：

```hcl
content = "service_name = "payment-api""
```

这个会导致字符串提前结束，通常在 `terraform validate` 阶段就会炸。

---

## 5. 保留反斜杠：`\\`

你希望最终生成的文本是：

```text
C:\tools\terraform
```

Terraform 代码里要写：

```hcl
content = "C:\\tools\\terraform"
```

分层理解：

```text
Terraform 代码里的写法：C:\\tools\\terraform
Terraform 解析后的内容：C:\tools\terraform
```

错误写法：

```hcl
content = "C:\tools\terraform"
```

这里的 `\t` 会被 Terraform 解释为 tab。这个错误更阴险：有时候语法不一定炸，但最终文件内容已经错了。

生产里还有一个实用建议：如果目标系统接受，Windows 路径可以尽量写成正斜杠：

```hcl
content = "C:/tools/terraform"
```

这样可以绕开反斜杠转义问题。

---

## 6. 插入换行：`\n`

你希望最终生成：

```text
line1
line2
line3
```

Terraform quoted string 里应该写：

```hcl
content = "line1\nline2\nline3\n"
```

注意，不要在 quoted string 里直接把字符串拆成这样：

```hcl
content = "line1
line2
line3"
```

这种写法不适合作为 Terraform quoted string 的标准练习方式。多行文本更推荐使用 heredoc。

正确的 quoted string 写法：

```hcl
content = "line1\nline2\nline3\n"
```

更适合多行文本的 heredoc 写法：

```hcl
content = <<EOT
line1
line2
line3
EOT
```

---

## 7. 常用转义序列速查表

| 你想要的最终内容 | Terraform quoted string 里写法 | 说明 |
|---|---|---|
| `"` | `\"` | 保留双引号 |
| `\` | `\\` | 保留反斜杠 |
| 换行 | `\n` | newline |
| 回车 | `\r` | carriage return |
| tab | `\t` | 制表符 |
| Unicode 字符 | `\uNNNN` | 4 位十六进制 Unicode |
| Unicode 字符 | `\UNNNNNNNN` | 8 位十六进制 Unicode |

还有两个 Terraform 模板相关的特殊写法：

| 你想要的最终内容 | Terraform 里写法 | 用途 |
|---|---|---|
| `${` | `$${` | 避免被当成插值表达式开始 |
| `%{` | `%%{` | 避免被当成模板 directive 开始 |

这节实验模拟一个很常见的场景：平台团队用 Terraform 生成一个本地文件，文件内容包含：

- 带双引号的配置行

## 10. 推荐 `main.tf`

```hcl
terraform {
  required_version = ">= 1.5.0"

  required_providers {
    local = {
      source  = "hashicorp/local"
      version = "~> 2.5"
    }
  }
}

locals {
  quoted_line   = "service_name = \"payment-api\""
  windows_path  = "C:\\tools\\terraform"
  multiline     = "line1\nline2\nline3\n"

  file_content = "${local.quoted_line}\npath = ${local.windows_path}\n${local.multiline}"
}

resource "local_file" "escaped_strings" {
  filename = "${path.module}/output/escaped-strings.txt"
  content  = local.file_content
}

output "quoted_line" {
  value = local.quoted_line
}

output "windows_path" {
  value = local.windows_path
}

output "multiline" {
  value = local.multiline
}

output "file_path" {
  value = local_file.escaped_strings.filename
}
```

运行后，最终文件内容应该类似：

```text
service_name = "payment-api"
path = C:\tools\terraform
line1
line2
line3
```

## 11. 推荐 `tests/escape_sequences.tftest.hcl`

```hcl
run "escape_sequences_are_rendered_correctly" {
  command = apply

  assert {
    condition     = output.quoted_line == "service_name = \"payment-api\""
    error_message = "quoted_line should preserve inner double quotes."
  }

  assert {
    condition     = output.windows_path == "C:\\tools\\terraform"
    error_message = "windows_path should preserve literal backslashes."
  }

  assert {
    condition     = output.multiline == "line1\nline2\nline3\n"
    error_message = "multiline should contain newline characters."
  }
}
```

这里测试的不是云资源，而是 Terraform 字符串解析结果。

这个思路很好：**不要靠肉眼猜字符串最终长什么样，用 test 固化预期。**

## 13. 每条命令在干嘛？

### 13.1 `terraform init -input=false`

初始化当前目录，下载 provider，准备 `.terraform` 工作目录。

`-input=false` 表示不要在命令行里交互式提问，适合 CI/CD。

### 13.2 `terraform fmt -check`

检查 `.tf` 文件格式是否符合 Terraform 标准。

它只是检查格式，不会自动修改。CI 里常用这个命令阻止格式不统一的代码进入主分支。

### 13.3 `terraform validate`

检查 Terraform 配置语法和基本结构是否正确。

如果 quoted string 写崩了，比如内部双引号没有写成 `\"`，通常这里就会报错。

### 13.4 `terraform test`

运行 `.tftest.hcl` 测试文件。

这一课里，测试重点是验证字符串最终解析结果是否符合预期。

### 13.5 `terraform plan -out=tfplan`

生成执行计划并保存成二进制 plan 文件。

这一步可以让后续 apply 执行的内容和当前 plan 保持一致。

### 13.6 `terraform apply -auto-approve tfplan`

根据保存好的 plan 执行变更。

这里会生成实验里的本地输出文件，例如 `output/escaped-strings.txt`。

### 13.7 `terraform output`

查看 Terraform 输出值，用来确认转义后的内容是否符合预期。

### 13.8 `terraform destroy -auto-approve`

删除本实验创建的本地资源，恢复环境。

## 14. 常见错误 1：字符串提前结束

错误写法：

```hcl
content = "name = "demo""
```

Terraform 会认为字符串在 `name = ` 后就结束了，后面的 `demo` 变成无法理解的语法。

正确写法：

```hcl
content = "name = \"demo\""
```

---

## 15. 常见错误 2：Windows 路径被转义成 tab

你想要：

```text
C:\tools\terraform
```

错误写法：

```hcl
content = "C:\tools\terraform"
```

正确写法：

```hcl
content = "C:\\tools\\terraform"
```

更省心的替代写法：

```hcl
content = "C:/tools/terraform"
```

---

## 16. 常见错误 3：把 `\n` 写成真实换行

你想要多行内容，quoted string 里应该写：

```hcl
content = "line1\nline2\nline3\n"
```

如果文本很多，直接换 heredoc：

```hcl
content = <<EOT
line1
line2
line3
EOT
```

生产里别硬在一长串 quoted string 里堆 `\n`。可读性会很差。

## 17. quoted string 什么时候够用？什么时候不够用？

### 17.1 适合 quoted string 的场景

quoted string 适合短文本：

```hcl
name    = "demo-app"
path    = "C:\\tools\\terraform"
message = "hello\nworld\n"
```

### 17.2 不适合 quoted string 的场景

如果你要生成大段配置，例如 JSON、YAML、shell script、nginx config，quoted string 会变得很难读：

```hcl
content = "line1\nline2\nline3\nline4\n..."
```

这种时候更推荐：

- heredoc
- `templatefile()`
- `jsonencode()`
- `yamlencode()`

例如 heredoc：

```hcl
content = <<EOT
line1
line2
line3
EOT
```

如果是 JSON，优先考虑：

```hcl
content = jsonencode({
  name = "demo"
  env  = "dev"
})
```

这样比手写一堆 `\"` 稳得多。

---

## 18. 生产环境里的判断原则

| 场景 | 推荐方式 |
|---|---|
| 短字符串 | quoted string |
| 字符串里有少量引号或路径 | quoted string + 转义 |
| 多行说明文本 | heredoc |
| 大段模板文件 | `templatefile()` |
| JSON 内容 | `jsonencode()` |
| YAML 内容 | `yamlencode()` 或模板 |
| shell script | heredoc 或单独脚本文件 |

真正工程化的做法不是“全靠转义”，而是选对表达方式。

## 20. 考试记忆点

这一课可以记成四句话：

1. quoted string 用双引号包裹字符串。
2. 字符串内部要保留双引号，用 `\"`。
3. 字符串内部要保留反斜杠，用 `\\`。
4. 字符串内部要换行，用 `\n`；复杂多行文本优先用 heredoc。

再补一句工程判断：

> 短文本可以用 quoted string；复杂文本别硬转义，优先 heredoc、templatefile 或 encode 函数。

## 21. 这一课的核心 takeaway

Terraform 代码里的字符串写法，不一定等于最终文件里的真实内容。

你要分清三层：

```text
Terraform 源码写法
        ↓
Terraform 解析后的字符串值
        ↓
最终生成到文件 / 传给 provider 的内容
```

比如：

```hcl
"C:\\tools\\terraform"
```

最终内容是：

```text
C:\tools\terraform
```

比如：

```hcl
"service_name = \"payment-api\""
```

最终内容是：

```text
service_name = "payment-api"
```

比如：

```hcl
"line1\nline2\n"
```

最终内容是：

```text
line1
line2
```

把这三层想清楚，quoted string 和转义序列就不会再坑你。
