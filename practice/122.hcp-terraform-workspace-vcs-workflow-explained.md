# Terraform 考试训练 122 讲解版：Workspace 与 Version Control Workflow

> 来源：`doc/122.txt`  
> 主题：HCP Terraform workspace、VCS 集成、working directory、运行触发、state 边界  
> 训练类型：理论题 / 考试题 / 场景题

---

## 1. 这一节到底在讲什么？

这一节讲的是 **HCP Terraform 里的 workspace 怎么和 Git 仓库绑定，并通过 Version Control Workflow 自动触发 Terraform run**。

你可以把 HCP Terraform workspace 理解成：

> 一个集中管理 Terraform 运行、变量、状态、审批和历史记录的执行单元。

它不是简单的“一个文件夹”，也不是单纯的“一个 Terraform 配置目录”。  
在生产环境里，一个 workspace 通常对应一套清晰的基础设施边界，例如：

- `network-dev`
- `network-prod`
- `app-dev`
- `app-prod`
- `security-shared`
- `platform-observability`

真正要记住的是：

> Git 仓库保存代码，HCP Terraform workspace 负责运行代码、保存 state、管理变量、记录 run 历史和控制审批。

---

## 2. 核心概念：Workspace 是什么？

HCP Terraform workspace 是 HCP Terraform 平台里的核心运行单元。

一个 workspace 通常会管理这些东西：

| 能力 | 说明 |
|---|---|
| Terraform 配置来源 | 例如 GitHub、GitLab、Bitbucket、CLI-driven、API-driven |
| Variables | Terraform variables，例如 `region`、`instance_type` |
| Environment Variables | 环境变量，例如云厂商凭证、代理配置 |
| Sensitive Values | 敏感变量，例如 token、secret、access key |
| State | 当前基础设施状态 |
| Run History | 每一次 plan / apply 的执行历史 |
| Terraform Version | 当前 workspace 使用的 Terraform 版本 |
| Working Directory | 在仓库中实际运行 Terraform 的子目录 |
| Execution Mode | 远程执行、本地执行等 |
| Run Trigger | 代码变更、手动触发、API 触发等 |

所以考试里看到 workspace，千万别只想到“文件夹”。

更准确的理解是：

> Workspace = 代码来源 + 变量 + state + 执行配置 + 运行历史 + 审批控制。

---

## 3. Version Control Workflow 是什么？

Version Control Workflow 指的是：

> HCP Terraform workspace 连接到一个 VCS 仓库，然后根据仓库中的代码变更自动触发 Terraform run。

VCS 可以是：

- GitHub
- GitLab
- Bitbucket
- Azure DevOps Repos

这一节重点不是让你记住所有平台名字，而是要理解流程。

典型流程如下：

```text
开发者提交 Terraform 代码
        ↓
代码进入 Git 仓库
        ↓
HCP Terraform workspace 监听到代码变化
        ↓
workspace 拉取指定目录中的 Terraform 配置
        ↓
执行 terraform init
        ↓
执行 terraform plan
        ↓
可选：policy check / cost estimation
        ↓
等待审批或自动 apply
        ↓
保存 state 和 run history
```

这就是考试里的关键词：

> VCS 集成、自动 plan、集中执行、审批、state 管理、run 历史。

---

## 4. Git 仓库、目录、Workspace 的关系

考试最容易混淆的点就在这里。

一个 Git 仓库里可以有很多 Terraform 配置目录，例如：

```text
infra-repo/
├── network/
│   ├── main.tf
│   ├── variables.tf
│   └── outputs.tf
├── app/
│   ├── main.tf
│   ├── variables.tf
│   └── outputs.tf
└── security/
    ├── main.tf
    ├── variables.tf
    └── outputs.tf
```

这时候不应该把整个仓库直接当成一个 workspace 粗暴管理。

更合理的做法是：

| Workspace | Repository | Working Directory | State |
|---|---|---|---|
| `network-prod` | `infra-repo` | `network/` | 独立 |
| `app-prod` | `infra-repo` | `app/` | 独立 |
| `security-shared` | `infra-repo` | `security/` | 独立 |

这里的关键配置就是：

> working directory

也就是说：

> 同一个 Git 仓库可以被多个 workspace 使用，每个 workspace 指向不同的子目录，并维护自己的 state、变量和 run 历史。

---

## 5. 为什么要拆成多个 Workspace？

因为不同基础设施边界通常有不同的生命周期、权限、变量和审批要求。

例如 `network/` 和 `app/`：

| 维度 | network workspace | app workspace |
|---|---|---|
| 生命周期 | 很稳定，变更少 | 变更多 |
| 审批要求 | 更严格 | 相对灵活 |
| 变量 | CIDR、VPC、Subnet | Image、replica、service config |
| State | 网络资源 state | 应用资源 state |
| 影响范围 | 影响大 | 影响相对小 |
| 团队 | 平台/网络团队 | 应用团队 |

如果你把它们塞进同一个 workspace，问题就来了：

- state 变得很大
- plan 噪音增加
- app 变更可能牵扯 network
- 审批边界不清晰
- 权限控制困难
- 回滚和排障都更麻烦

生产里拆 workspace，不是为了显得高级，而是为了减少爆炸半径。

---

## 6. Workspace 和 Terraform CLI Workspace 不是一回事

这里要特别注意。

Terraform CLI 里也有 `terraform workspace` 命令，比如：

```sh
terraform workspace new dev
terraform workspace select prod
```

但这个 CLI workspace 和 HCP Terraform workspace 不是同一个概念。

| 对比项 | Terraform CLI workspace | HCP Terraform workspace |
|---|---|---|
| 使用场景 | 本地 CLI 的多个 state 命名空间 | HCP Terraform 平台的运行单元 |
| 管理内容 | 主要是 state 隔离 | state、变量、运行、审批、版本、VCS |
| 适用范围 | 小型或简单环境隔离 | 团队协作和生产 IaC 平台 |
| 是否连接 VCS | 不一定 | 可以连接 |
| 是否有 run history | 本地没有平台级历史 | 有完整 run 历史 |

考试如果说的是 HCP Terraform workspace，就不要脑补成本地 `terraform workspace` 命令。

## 7. Version Control Workflow 的考试信号词

看到下面这些词，基本就要想到 HCP Terraform workspace + VCS workflow：

尤其是题目说：

> 开发者提交代码后，HCP Terraform 自动生成 plan。

这就是非常典型的 Version Control Workflow。

## 8. 自动 Plan 和手动 Apply 怎么理解？

很多生产环境会采用这种模式：

```text
代码提交 / PR 创建
        ↓
自动生成 plan
        ↓
团队 review plan
        ↓
人工确认 apply
        ↓
HCP Terraform 执行 apply
```

这背后的价值是：

1. **代码变更可追踪**  
   每次基础设施变更都来自 Git commit。

2. **Plan 可审查**  
   apply 之前先看到会创建、修改、删除什么。

3. **审批更集中**  
   不靠某个人本地电脑执行生产变更。

4. **State 更安全**  
   state 放在集中平台，而不是散落在个人机器上。

5. **运行历史可追溯**  
   出事故时能查到谁改了什么、什么时候改的、plan 是什么。

一句话：

> VCS Workflow 把 Terraform 从“我本地跑一下”升级成“团队可审计的基础设施变更流水线”。

---

## 9. Policy Check / Cost Estimation 在这里是什么角色？

课程提到 workspace 可以执行：

- plan
- policy check
- cost estimation
- apply

这里不要机械理解成每次 run 都一定有所有步骤。

更准确的说法是：

> HCP Terraform 的 run 流程可以包含 policy check 和 cost estimation，具体是否出现取决于组织配置、版本能力和是否启用相关功能。

考试通常不会考你产品套餐细节，而是考你是否理解它们的位置：

```text
plan 之后，apply 之前
```

它们的作用是：

| 步骤 | 作用 |
|---|---|
| Policy Check | 检查是否违反组织策略，例如禁止公网 S3、禁止过大实例 |
| Cost Estimation | 预估本次变更可能带来的成本变化 |
| Manual Approval | 人工确认是否允许 apply |

这就是生产治理能力。

## 10. 场景题模板：一个仓库多个目录怎么办？

题目：

> 一个 Git 仓库下有 `network/` 和 `app/` 两套 Terraform 配置。团队希望分别管理它们的 state 和审批。应该如何设计 workspace？

标准回答可以这样写：

```text
应该创建两个 HCP Terraform workspace，分别连接到同一个 Git 仓库，但配置不同的 working directory。

例如：
- network workspace 指向 network/ 目录
- app workspace 指向 app/ 目录

这样两套配置拥有独立的 state、变量、run history 和审批流程。network 和 app 的变更可以分别 plan、review 和 apply，避免共享 state 带来的耦合和风险。
```

考试里这类题要抓住 4 个关键词：

- two workspaces
- same repository
- different working directory
- independent state and approval

## 11. 场景题模板：提交代码后自动 Plan，Apply 需要审批

题目：

> 开发者提交代码后，HCP Terraform 自动生成 plan，但 apply 需要人工确认。这个流程体现了哪些 workspace 能力？

标准回答可以这样写：

```text
这体现了 HCP Terraform workspace 的 VCS 集成能力、自动 run 触发能力、远程 plan 能力、集中 run history、审批控制和 state 管理能力。

代码变更来自 Git 仓库，workspace 检测到变更后拉取 Terraform 配置并生成 plan。apply 阶段需要人工确认，说明 workspace 可以把基础设施变更纳入集中审查流程，避免未经审批的生产变更。
```

考试别只写“自动化”。  
要写出它自动化了什么、控制了什么、保存了什么。

## 12. 本节选择题解析

### 题 1

> Version Control Workflow 中，HCP Terraform workspace 通常从哪里获取 Terraform 代码？

答案：A. 已连接的 Git 仓库

解析：

Version Control Workflow 的核心就是 workspace 连接 VCS 仓库，从仓库中拉取 Terraform 配置，然后触发 run。

其他选项为什么错：

- 本地浏览器缓存：不是 Terraform 配置来源。
- Docker Hub 镜像标签：这是容器镜像来源，不是 Terraform 代码来源。
- Terraform Registry provider 二进制文件：这是 provider 下载来源，不是业务 Terraform 配置来源。

---

### 题 2

> Workspace 可以管理哪些内容？

答案：A. 变量、state、run 历史、Terraform 版本和代码来源

解析：

HCP Terraform workspace 是平台运行单元，它管理的不只是代码，还包括变量、环境变量、state、运行历史、版本、工作目录等。

其他选项都太窄：

- `.ssh` 目录不是 workspace 的核心管理对象。
- Markdown 文档不是重点。
- 操作系统用户也不是 Terraform workspace 管理对象。

---

### 题 3

> 如果一个仓库中有多个 Terraform 子目录，workspace 通常需要配置什么？

答案：A. working directory

解析：

一个仓库可能有多个 Terraform root module。  
workspace 要知道到底在哪个目录执行 Terraform，所以需要配置 working directory。

例如：

```text
repo/
├── network/
└── app/
```

如果 workspace 管 `network/`，working directory 就配置成 `network/`。  
如果 workspace 管 `app/`，working directory 就配置成 `app/`。

## 13. 判断题解析

### 判断题 1

> Workspace 可以与 VCS 仓库关联，并基于代码变更触发运行。

答案：对。

解析：

这正是 Version Control Workflow 的核心。

---

### 判断题 2

> Workspace 只保存 Terraform 代码，不保存任何 run 或 state 信息。

答案：错。

解析：

代码通常保存在 Git 仓库。  
HCP Terraform workspace 管理的是代码来源、变量、state、run history、执行配置等。

这道题就是故意把“代码仓库”和“workspace”混在一起。

---

### 判断题 3

> 生产中常用多个 workspace 区分环境、系统或团队边界。

答案：对。

解析：

生产环境常见设计是：

- 按环境拆：`dev` / `staging` / `prod`
- 按系统拆：`network` / `app` / `database`
- 按团队拆：`platform` / `security` / `data`

核心目标是隔离 state、变量、权限、审批和变更影响范围。

---

## 14. 高频误区

### 误区 1：Workspace 就是 Git 目录

错。

Git 目录只是代码的位置。  
workspace 是 HCP Terraform 里的运行和管理单元。

---

### 误区 2：一个仓库只能对应一个 workspace

错。

一个仓库可以对应多个 workspace。  
每个 workspace 可以配置不同的 working directory。

---

### 误区 3：一个 workspace 可以随便管所有东西

理论上可以，生产上不建议。

如果一个 workspace 同时管理网络、数据库、应用、安全策略，风险会很高：

- plan 巨大
- state 巨大
- apply 爆炸半径大
- 审批边界模糊
- 团队责任不清

工程上要按边界拆。

---

### 误区 4：自动 plan 就等于自动 apply

错。

自动 plan 只是生成变更预览。  
apply 可以配置为人工审批后再执行。

这是很多企业使用 HCP Terraform 的核心价值之一。

---

## 15. 用一句话记忆

> HCP Terraform workspace 是远程运行和状态管理单元；VCS workflow 让它从 Git 仓库拉代码，自动 plan，并通过审批控制 apply。

再压缩一点：

> Git 管代码，workspace 管运行和 state。

## 16. 和真实项目怎么对应？

假设你做的是平台工程项目，可以这样设计：

```text
platform-infra/
├── environments/
│   ├── dev/
│   ├── staging/
│   └── prod/
├── modules/
│   ├── network/
│   ├── aks/
│   └── monitoring/
└── README.md
```

可能对应这些 workspace：

| Workspace | Working Directory | 用途 |
|---|---|---|
| `platform-dev` | `environments/dev` | 开发环境 |
| `platform-staging` | `environments/staging` | 预生产环境 |
| `platform-prod` | `environments/prod` | 生产环境 |

或者按系统拆：

| Workspace | Working Directory | 用途 |
|---|---|---|
| `network-prod` | `environments/prod/network` | 生产网络 |
| `aks-prod` | `environments/prod/aks` | 生产 AKS |
| `monitoring-prod` | `environments/prod/monitoring` | 生产监控 |

拆法没有唯一答案，但要说得出边界理由：

- state 隔离
- 权限隔离
- 审批隔离
- 变更频率不同
- 爆炸半径不同
- 团队职责不同

## 17. 考试答题套路

遇到 HCP Terraform workspace 题，先问自己 5 个问题：

1. 代码从哪里来？  
   通常是 Git 仓库 / VCS。

2. Terraform 在哪个目录运行？  
   看 working directory。

3. state 放在哪里？  
   在 HCP Terraform workspace 管理。

4. 谁触发 run？  
   可能是 VCS commit、手动触发、CLI 或 API。

5. apply 是否需要审批？  
   生产环境常见是 plan 自动、apply 人工确认。

这 5 个问题能覆盖大部分考试题。
