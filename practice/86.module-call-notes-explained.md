# Terraform 实操训练 86：模块调用注意事项（讲解版）

## 1. 这节课在练什么

本 Lab 的主题是 **Terraform module 调用时的注意事项**。

前面你已经开始接触 Terraform module 的价值：复用、封装、统一标准、降低重复代码。但真正到了工程实践里，module 不是“看到就直接套”。module 调用写错，轻则 `terraform validate` 报错，重则造成生产环境资源命名冲突、状态漂移、模块升级困难，甚至让团队不知道某个资源到底是哪个 module 创建的。

所以这一节的重点不是背 `module` block 的语法，而是理解：

- module 调用方和 module 内部代码之间的边界；
- `source`、输入变量、输出值的使用方式；
- module 是否应该被当成黑盒；
- module 版本、路径、命名和状态地址的风险；
- 为什么生产环境不能随便改 module 名称和 source 地址；
- 如何用 `terraform test` 给 module 调用加一层基础验证。

---

## 2. 先把 module 调用看成“函数调用”

Terraform module 很像编程语言里的函数。

调用一个函数时，你通常会关心：

1. 函数名是什么；
2. 参数传什么；
3. 返回值是什么；
4. 函数内部不要随便被外部代码乱碰；
5. 函数版本升级会不会影响调用方。

Terraform module 也是类似的。

典型 module 调用长这样：

```hcl
module "network" {
  source = "./modules/network"

  name_prefix = "dev"
  cidr_block  = "10.0.0.0/16"
}
```

这里可以把它理解成：

```text
调用 ./modules/network 这个模块，
给它传入 name_prefix 和 cidr_block，
然后通过 module.network.xxx 读取它暴露出来的输出。
```

最关键的一点是：**调用方应该通过变量和 output 跟 module 交互，而不是假装自己知道 module 里面所有资源的内部实现。**

---

## 3. module block 里的几个关键点

### 3.1 `module "xxx"` 里的名字不是随便改的

```hcl
module "app_config" {
  source = "./modules/app_config"
}
```

这里的 `app_config` 是 Terraform 状态地址的一部分。

例如 module 里有一个资源：

```hcl
resource "local_file" "config" {
  filename = "config.json"
  content  = "hello"
}
```

在状态中，它可能会变成：

```text
module.app_config.local_file.config
```

如果你把调用名改成：

```hcl
module "config_app" {
  source = "./modules/app_config"
}
```

Terraform 可能会认为：

```text
旧资源 module.app_config.local_file.config 不见了
新资源 module.config_app.local_file.config 出现了
```

这就有可能触发资源销毁和重建。

**工程判断：**

- 新项目可以统一命名；
- 已经 apply 过的 module 名称不要随便改；
- 真要改，要考虑 `moved` block 或 `terraform state mv`；
- 生产环境里，module 名称就是状态地址的一部分，不只是一个漂亮名字。

---

### 3.2 `source` 决定 module 从哪里来

常见 source 类型有几种：

```hcl
# 本地相对路径
source = "./modules/network"

# Git 仓库
source = "git::https://github.com/example/terraform-modules.git//network?ref=v1.2.0"

# Terraform Registry
source  = "terraform-aws-modules/vpc/aws"
version = "5.0.0"
```

#### 本地路径适合什么？

本地路径适合训练、单仓库项目、内部模块快速开发。

例如：

```hcl
module "naming" {
  source = "./modules/naming"
}
```

优点是简单、直观、方便调试。

缺点是多个仓库复用不方便，版本边界也不清晰。

#### Git source 适合什么？

Git source 适合企业内部 module 共享。

例如：

```hcl
module "network" {
  source = "git::https://github.com/company/terraform-modules.git//azure/network?ref=v1.4.2"
}
```

注意这里最好带 `ref`。

如果不固定版本：

```hcl
source = "git::https://github.com/company/terraform-modules.git//azure/network"
```

你以后每次 init 拉到的内容可能变化，团队复现环境会变得很痛苦。

#### Registry source 适合什么？

Registry source 适合使用成熟公共模块。

例如 AWS VPC module、Azure naming module 这类。

但公共模块不是“拿来主义”。引用前至少要看：

- 是否活跃维护；
- 是否支持当前 Terraform 版本；
- 是否有清晰 examples；
- 是否锁定 provider 版本；
- 是否有破坏性升级记录；
- 是否创建了你不想要的默认资源；
- 是否隐藏了过多复杂逻辑。

## 4. 输入变量：调用方只应该传 module 需要的东西

module 的输入变量通常在 module 内部的 `variables.tf` 中定义：

```hcl
variable "name_prefix" {
  type        = string
  description = "Prefix used for resource names"
}

variable "environment" {
  type        = string
  description = "Deployment environment"
}
```

调用方传参：

```hcl
module "app" {
  source = "./modules/app"

  name_prefix = "demo"
  environment = "dev"
}
```

### 好的 module 输入应该是什么样？

好的输入变量应该：

- 名字清楚；
- 类型明确；
- 默认值谨慎；
- 必填项和可选项分开；
- 不把调用方暴露给过多内部细节；
- 不让一个变量承担多个意思。

例如：

```hcl
variable "replica_count" {
  type        = number
  description = "Number of application replicas"
  default     = 2
}
```

比下面这种更好：

```hcl
variable "settings" {
  type = any
}
```

`type = any` 很自由，但自由过头了。短期省事，长期排障像翻垃圾袋。

## 5. 输出值：module 对外暴露的正式接口

module 内部可以定义 output：

```hcl
output "config_file_path" {
  value = local_file.config.filename
}
```

调用方读取：

```hcl
output "app_config_path" {
  value = module.app.config_file_path
}
```

这说明调用方不需要知道 module 里面资源叫 `local_file.config`，只需要知道 module 对外承诺了一个输出：

```text
config_file_path
```

这就是接口边界。

### 为什么不要依赖 module 内部资源？

假设你写：

```hcl
module.app.local_file.config.filename
```

这通常不是正确的 module 调用方式。module 对外暴露什么，就只能稳定依赖什么。

module 作者以后可能把内部资源从：

```hcl
local_file.config
```

改成：

```hcl
local_sensitive_file.config
```

只要 output 没变，调用方就不应该受影响。

这就是封装。

## 6. module 调用最容易踩的坑

### 坑 1：source 写错路径

例如：

```hcl
source = "modules/app"
```

和：

```hcl
source = "./modules/app"
```

看起来差不多，但 Terraform 解析方式不同。

本地路径建议明确写：

```hcl
source = "./modules/app"
```

或者：

```hcl
source = "../modules/app"
```

否则 Terraform 可能把它当成 Registry module 地址解析。

---

### 坑 2：改 module 名称导致状态地址变化

这个前面说过，是生产环境高危操作。

```hcl
module "app" {}
```

改成：

```hcl
module "application" {}
```

不是单纯“重命名变量”，而是改了状态地址。

正确做法是：

```hcl
moved {
  from = module.app
  to   = module.application
}
```

或者在特殊情况下使用：

```sh
terraform state mv module.app module.application
```

不过对初学阶段来说，记住一句话就够了：

> 已经 apply 过的 module 名称，不要手贱随便改。

---

### 坑 3：没有固定 module 版本

公共 module 或 Git module 如果不锁版本，今天能跑，不代表下个月还一样。

推荐：

```hcl
module "vpc" {
  source  = "terraform-aws-modules/vpc/aws"
  version = "5.1.0"
}
```

Git source 推荐：

```hcl
source = "git::https://github.com/company/modules.git//vpc?ref=v1.1.0"
```

不要长期依赖默认分支：

```hcl
source = "git::https://github.com/company/modules.git//vpc"
```

这就像生产依赖直接跟着 `main` 跑，迟早有一天会被更新背刺。

---

### 坑 4：module 输入太多，变成“伪封装”

有些 module 表面上封装了，实际上调用方要传 50 个变量。

这类 module 经常变成：

```text
main.tf 里少写了资源，
但是多写了 50 行变量。
```

这种封装意义不大。

好的 module 应该抽象稳定能力，比如：

- 标准命名；
- 标准 tag；
- 标准网络结构；
- 标准安全基线；
- 标准监控配置；
- 标准日志设置。

而不是把 provider 资源的每个参数都原样搬成 variable。

---

### 坑 5：module 输出太少，调用方没法组合

反过来也有问题。

如果 module 创建了关键资源，但没有输出必要信息，调用方就很难和其他模块组合。

例如 network module 创建了 subnet，但是不输出 subnet id。后面的 app module 就没法使用。

比较合理的是：

```hcl
output "subnet_ids" {
  value = {
    app = azurerm_subnet.app.id
    db  = azurerm_subnet.db.id
  }
}
```

调用方就可以写：

```hcl
subnet_id = module.network.subnet_ids.app
```

## 7. 为什么这节课和 `terraform test` 有关系

你的 Lab 要跑：

```sh
terraform test
```

这说明本节不是只验证语法，而是要验证 module 调用是否符合预期。

测试可以检查：

- module 是否能被正常 init；
- module 输入变量是否被正确传入；
- module 输出是否符合预期；
- 命名规则是否生效；
- 环境标识是否正确；
- 某些默认值是否符合团队标准。

例如测试文件可以类似这样：

```hcl
run "module_output_should_match_expected_value" {
  command = plan

  assert {
    condition     = output.config_name == "dev-app-config"
    error_message = "Module output config_name should follow the expected naming rule."
  }
}
```

这里的价值是：**你不只是写了一个 module 调用，而是给调用结果加了一个可重复执行的断言。**

这就是 IaC 工程质量提升的入口。

## 8. 推荐实验执行顺序

进入目录：

```sh
cd /workspace/practice/labs/86
```

初始化：

```sh
terraform init -input=false
```

检查格式：

```sh
terraform fmt -check
```

如果失败，说明 HCL 格式不标准，可以本地修复：

```sh
terraform fmt -recursive
```

验证语法和 provider/module 结构：

```sh
terraform validate
```

运行测试：

```sh
terraform test
```

生成计划：

```sh
terraform plan -input=false -no-color -out=tfplan
```

应用计划：

```sh
terraform apply -auto-approve tfplan
```

查看输出：

```sh
terraform output
```

清理资源：

```sh
terraform destroy -auto-approve
```

---

## 9. 每条命令到底在帮你检查什么

| 命令 | 检查重点 |
|---|---|
| `terraform init` | module source 能不能解析，provider 能不能下载 |
| `terraform fmt -check` | HCL 格式是否统一 |
| `terraform validate` | 变量、输出、资源引用是否基本正确 |
| `terraform test` | 测试断言是否通过 |
| `terraform plan` | Terraform 准备创建/修改/删除什么 |
| `terraform apply` | 实际执行计划 |
| `terraform output` | 检查 module 对外暴露的结果 |
| `terraform destroy` | 清理实验资源，避免污染环境 |

这套顺序很接近真实团队里的 IaC 检查流水线。

## 10. 生产环境里怎么理解 module 调用

### 初学者视角

> module 是复用代码。

这个理解没错，但不够。

### 工程师视角

> module 是基础设施能力的接口封装。

比如团队定义一个 AKS module，它不是简单创建一个 AKS cluster，而是统一封装：

- 命名规范；
- 默认 node pool；
- network profile；
- managed identity；
- private cluster 策略；
- Azure Monitor；
- tag；
- diagnostic settings；
- upgrade channel；
- policy baseline。

调用方只需要关心：

```hcl
module "aks" {
  source = "git::https://github.com/company/platform-modules.git//aks?ref=v2.3.1"

  environment = "prod"
  region      = "australiaeast"
  node_count  = 3
}
```

这才是 module 的真正价值。

不是少写几行代码，而是把组织标准沉淀成可以复用、可以测试、可以审查、可以升级的基础设施产品。

## 11. 判断一个 module 调用是否靠谱

你可以用下面这张清单检查：

| 检查项 | 判断问题 |
|---|---|
| source 是否清楚 | 是本地、Git 还是 Registry？ |
| 版本是否锁定 | 公共/Git module 是否固定版本？ |
| module 名称是否稳定 | 是否已经 apply 过？是否会影响 state 地址？ |
| 输入变量是否明确 | 有没有乱用 `any`？有没有传太多内部细节？ |
| 输出是否够用 | 调用方是否能组合其他 module？ |
| 是否有测试 | 有没有 `terraform test` 断言关键输出？ |
| 是否容易升级 | changelog、version、examples 是否清楚？ |
| 是否符合团队规范 | 命名、tag、安全、监控是否统一？ |

## 12. 和 Terraform Professional 考试的关系

Terraform Professional 不会只考“module block 怎么写”。它更喜欢考你对工程行为的判断。

你需要能分清：

- local module、Registry module、Git module 的区别；
- 为什么要锁 module 版本；
- 改 module 名称为什么会影响 state；
- output 为什么是 module 的稳定接口；
- module 什么时候应该拆，什么时候不该拆；
- module 升级为什么要看 plan；
- 为什么测试可以验证 module 输出和规则；
- 为什么公共 module 引入前要评估维护状态和安全影响。

一句话：

> 考试考语法，但更考你是不是像平台工程师一样理解 Terraform module 的生命周期。

## 13. 常见报错与排障思路

### 13.1 `Module not found`

可能原因：

- `source` 路径写错；
- 本地 module 目录不存在；
- Git 地址错误；
- Registry module 名称错误；
- 网络无法访问 module source。

处理方式：

```sh
terraform init -upgrade
```

或者检查：

```sh
ls ./modules
```

确认路径是否真实存在。

---

### 13.2 `Unsupported argument`

例如：

```text
An argument named "environment" is not expected here.
```

说明调用方传了 module 没有声明的变量。

检查 module 内部是否有：

```hcl
variable "environment" {}
```

如果没有，要么补变量，要么删除调用方参数。

---

### 13.3 `Reference to undeclared output value`

例如：

```text
module.app.config_name
```

但 module 内部没有：

```hcl
output "config_name" {}
```

解决方式：在 module 内部补 output，或者改调用方引用。

---

### 13.4 `terraform test` 断言失败

优先看三件事：

1. output 名称是否写对；
2. 断言条件是否真的符合当前代码；
3. 测试运行阶段是 `plan` 还是 `apply`。

有些值在 plan 阶段未知，必须用 apply 阶段测试。这个点在 Terraform test 里很常见。

---

## 14. 本 Lab 的学习目标

做完这个 Lab，你应该能说清楚：

1. Terraform module 调用不是简单复制粘贴；
2. `module "name"` 会影响状态地址；
3. `source` 要明确，公共/Git module 要锁版本；
4. 输入变量是调用方传给 module 的正式参数；
5. output 是 module 对外暴露的稳定接口；
6. 不要让调用方依赖 module 内部实现；
7. `terraform test` 可以验证 module 调用结果；
8. module 是平台工程里沉淀标准能力的重要手段。

## 16. 一句话总结

Terraform module 调用的核心不是“少写代码”，而是**用稳定接口封装基础设施能力，并通过版本、输出、测试和状态管理保证它能长期安全复用**。
