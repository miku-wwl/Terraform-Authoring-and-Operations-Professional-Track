# Terraform 实操训练 89：创建第一个本地模块｜讲解版

## 1. 本节要学什么

本节的核心主题是 **创建第一个本地模块**。

你可以把 Terraform module 理解成 Terraform 世界里的“函数”或者“组件”：

- root module：当前直接运行 `terraform init/plan/apply` 的目录。
- child module：被 root module 通过 `module` block 调用的子目录。
- local module：模块代码就在本仓库、本地目录里，例如 `./modules/storage_label`。
- public module：从 Terraform Registry、GitHub 或其他远程地址引用的模块。

这一节的重点不是“会写一个 module block”就完事，而是要理解：  
**什么时候该把 Terraform 代码拆成模块，以及拆成模块后，变量、输出、状态和测试如何配合。**

---

## 2. 为什么需要本地模块

如果一个 Terraform 项目一直把所有资源都写在一个 `main.tf` 里，刚开始很爽，后面会变成一锅粥。

本地模块的价值主要有四个：

| 价值 | 说明 |
|---|---|
| 复用 | 同一类资源逻辑可以被多个环境、多个 stack 调用 |
| 封装 | 把复杂实现藏在模块内部，对外只暴露变量和输出 |
| 降低重复 | 避免 dev/staging/prod 到处复制粘贴同一坨 HCL |
| 更容易测试 | module 可以单独写 test，验证输入输出是否符合预期 |

本地模块是学习 Terraform module 最好的入口。  
因为它不依赖远程 registry，也不需要先处理版本发布、私有模块源、凭证等额外问题。

## 3. 目录结构怎么看

原始 lab 要求进入：

```sh
cd /workspace/practice/labs/89
```

典型结构可以理解成这样：

```text
practice/labs/89/
├── main.tf
├── variables.tf
├── outputs.tf
├── tests/
│   └── module.tftest.hcl
└── modules/
    └── example/
        ├── main.tf
        ├── variables.tf
        └── outputs.tf
```

其中：

- `practice/labs/89` 是 root module。
- `modules/example` 是本地 child module。
- root module 通过 `module "xxx"` 调用本地模块。
- test 文件用于验证模块调用后的结果是否符合预期。

真实工程中，本地模块常见位置有两种：

```text
repo/
├── environments/
│   ├── dev/
│   └── prod/
└── modules/
    ├── network/
    ├── compute/
    └── monitoring/
```

或者：

```text
repo/
├── stacks/
│   ├── aks/
│   └── eks/
└── modules/
    ├── naming/
    ├── tags/
    └── node_pool/
```

前者偏平台工程，后者偏产品化基础设施。

## 4. module block 的核心写法

本地模块最关键的是 `source`。

示例：

```hcl
module "example" {
  source = "./modules/example"

  name        = "demo"
  environment = "dev"
}
```

这里要注意：

```hcl
source = "./modules/example"
```

表示模块来源是当前目录下的相对路径。

不要写成：

```hcl
source = "modules/example"
```

虽然有些场景下也可能解析成功，但本地路径最好明确使用 `./` 或 `../`，可读性更强，也更不容易和 registry module 搞混。

---

## 5. root module 和 child module 的职责区别

很多初学者写 module 时会犯一个错误：  
**把 root module 和 child module 混在一起。**

更清晰的分工应该是：

### root module 负责什么

root module 更像“装配层”：

- 指定 provider。
- 指定 backend。
- 调用模块。
- 传入环境参数。
- 汇总输出结果。
- 作为 CI/CD 的执行入口。

### child module 负责什么

child module 更像“组件实现层”：

- 定义资源。
- 定义模块变量。
- 定义模块输出。
- 封装资源命名、标签、依赖关系。
- 不直接关心自己被 dev、prod 还是 test 调用。

一句话：  
**root module 管环境，child module 管能力。**

---

## 6. 变量如何传入模块

模块内部通常会有 `variables.tf`：

```hcl
variable "name" {
  type        = string
  description = "Base name used by the module"
}

variable "environment" {
  type        = string
  description = "Deployment environment"
}
```

root module 调用时传值：

```hcl
module "example" {
  source = "./modules/example"

  name        = "app"
  environment = "dev"
}
```

模块内部使用：

```hcl
locals {
  full_name = "${var.name}-${var.environment}"
}
```

这里最重要的理解是：

- root module 不能直接访问 child module 里面的 `var.xxx`。
- child module 也不能直接访问 root module 的 `var.xxx`。
- 两者之间必须通过 module input/output 明确传递。

Terraform module 不是“文件 include”，不是把所有 HCL 粘在一起。  
它更像一个有边界的组件。

---

## 7. 输出如何从模块返回

child module 里可以定义：

```hcl
output "full_name" {
  value = local.full_name
}
```

root module 读取时：

```hcl
output "module_full_name" {
  value = module.example.full_name
}
```

这个 `module.example.full_name` 的含义是：

```text
module.<模块调用名>.<模块输出名>
```

注意，不是访问模块内部任意 local 或 resource。  
root module 只能读取 child module 显式暴露出来的 outputs。

这就是封装。

---

## 8. terraform init 为什么很关键

本节运行手册第一步是：

```sh
terraform init -input=false
```

对于本地模块，`terraform init` 也很重要。

原因是 Terraform 会在 init 阶段解析 module source，把模块依赖准备好。  
如果你新增、删除、移动了 module source，通常需要重新运行 init。

常见报错包括：

```text
Module not installed
```

或者：

```text
Unreadable module directory
```

这类问题通常不是资源逻辑错了，而是模块路径、目录结构或 init 没做对。

## 9. terraform test 在这里测什么

原始 lab 里要求运行：

```sh
terraform test
```

并且验证结果是：

```text
1 passed, 0 failed
```

在这个主题里，测试通常会验证几类内容：

| 测试对象 | 例子 |
|---|---|
| 模块能否正常被调用 | `module.example` 是否能初始化 |
| 输入参数是否生效 | name/environment 是否组合成预期结果 |
| 输出是否正确 | `output.module_full_name` 是否等于预期值 |
| 断言是否能保护模块契约 | 输入输出变化时测试是否能失败 |

一个典型测试断言可能类似：

```hcl
run "validate_module_output" {
  command = plan

  assert {
    condition     = output.module_full_name == "app-dev"
    error_message = "Module output should match expected naming convention."
  }
}
```

这里测试的不是云资源是否真的存在，而是模块的行为契约是否稳定。

## 10. 完整运行流程讲解

原始命令如下：

```sh
cd /workspace/practice/labs/89
terraform init -input=false
terraform fmt -check
terraform validate
terraform test
terraform plan -input=false -no-color -out=tfplan
terraform apply -auto-approve tfplan
terraform output
terraform destroy -auto-approve
```

逐条解释：

### 10.1 进入实验目录

```sh
cd /workspace/practice/labs/89
```

这一步让当前目录变成 root module。

---

### 10.2 初始化

```sh
terraform init -input=false
```

作用：

- 初始化 Terraform 工作目录。
- 解析本地 module。
- 下载 provider。
- 创建 `.terraform/` 目录。
- 生成或更新 `.terraform.lock.hcl`。

`-input=false` 的意思是不要交互式询问。  
这在 CI/CD 里很重要，因为流水线不能卡在那里等你输入。

---

### 10.3 检查格式

```sh
terraform fmt -check
```

作用是检查 HCL 格式是否标准。

如果失败，可以运行：

```sh
terraform fmt -recursive
```

然后再重新检查。

---

### 10.4 静态校验

```sh
terraform validate
```

作用是检查 Terraform 配置是否合法，例如：

- 变量引用是否存在。
- module source 是否能解析。
- output 表达式是否有效。
- provider/resource block 基本结构是否正确。

它不会真正创建资源。

---

### 10.5 运行测试

```sh
terraform test
```

作用是执行 `tests/` 下的 `.tftest.hcl` 文件。

这一节的重点是：  
**模块不是写完能 plan 就够了，还要用测试保护模块契约。**

---

### 10.6 生成计划文件

```sh
terraform plan -input=false -no-color -out=tfplan
```

重点参数：

| 参数 | 作用 |
|---|---|
| `-input=false` | 禁止交互输入 |
| `-no-color` | 输出不带颜色，适合日志保存 |
| `-out=tfplan` | 保存 plan 结果，供 apply 使用 |

保存 plan 的好处是：  
你 apply 的就是刚刚审核过的那份 plan，而不是重新计算出来的新 plan。

---

### 10.7 应用计划

```sh
terraform apply -auto-approve tfplan
```

这里使用的是前面保存的 `tfplan`。

注意：  
如果你传了 plan 文件，就不需要再手动确认 yes。

---

### 10.8 查看输出

```sh
terraform output
```

用于查看 root module 暴露出来的 outputs。

如果 root module 把 child module 的输出再次暴露出来，你就能在这里看到模块返回值。

---

### 10.9 清理资源

```sh
terraform destroy -auto-approve
```

销毁实验创建的资源，避免状态和资源残留。

## 11. 本地模块常见坑

### 坑 1：路径写错

错误示例：

```hcl
module "example" {
  source = "./module/example"
}
```

但真实目录是：

```text
modules/example
```

就会出现模块目录找不到。

---

### 坑 2：改了 module source 但没重新 init

如果你移动了模块目录，或者新建了模块目录，记得重新运行：

```sh
terraform init
```

否则可能出现 module not installed 之类的问题。

---

### 坑 3：root module 直接访问 child module 内部细节

错误思路：

```hcl
output "name" {
  value = module.example.local.full_name
}
```

模块内部 local 不能被外部直接访问。

正确做法是 child module 定义 output，然后 root module 通过 `module.example.xxx` 读取。

---

### 坑 4：模块变量没有类型约束

不推荐：

```hcl
variable "tags" {}
```

推荐：

```hcl
variable "tags" {
  type        = map(string)
  description = "Tags applied to resources"
  default     = {}
}
```

模块被复用后，类型约束就是防呆护栏。  
没有类型约束，错误会拖到更晚才爆。

---

### 坑 5：模块输出过多

不要把模块内部所有资源 ID、name、对象都暴露出去。  
只暴露调用方真正需要的内容。

模块输出越多，外部依赖越重，后续重构越难。

---

## 12. 生产环境怎么用这个知识点

真实平台工程里，本地模块常用于这些场景：

| 模块类型 | 用途 |
|---|---|
| naming module | 统一资源命名规则 |
| tagging module | 统一标签、owner、cost center |
| network module | 封装 VPC/VNet/subnet/security group |
| aks/eks module | 封装集群、node pool、identity、addon |
| monitoring module | 封装日志、指标、告警 |
| policy module | 封装 Azure Policy / AWS Config 规则 |

对你后续做 AKS/EKS 项目很关键。  
比如你完全可以把节点池、命名规则、标签、监控告警做成本地模块，再由不同环境调用。

这样项目看起来就不是“堆命令”，而是有平台工程味道的 IaC 设计。

---

## 13. 模块设计的判断标准

一个模块设计得好不好，可以用这几个问题判断：

1. 这个模块是否解决了一个明确问题？
2. 输入变量是否足够清晰？
3. 输出是否只暴露必要信息？
4. 调用方是否不需要理解模块内部细节？
5. 模块是否能被 dev/staging/prod 复用？
6. 模块是否有 test 保护核心行为？
7. 模块是否避免硬编码环境信息？

如果答案大多是 yes，这个模块就比较健康。

## 14. 面试表达方式

如果面试官问：

> 你怎么理解 Terraform module？

可以这样回答：

> Terraform module 是 Terraform 配置的复用和封装单元。root module 负责组合环境和调用模块，child module 负责封装具体资源逻辑。好的模块应该通过 input variables 暴露可配置项，通过 outputs 暴露必要结果，同时配合 terraform test 保护模块契约，避免后续重构或版本升级破坏调用方。

如果问：

> 本地模块和远程模块有什么区别？

可以这样回答：

> 本地模块通过相对路径引用，适合项目内部复用和早期开发；远程模块通过 registry、GitHub 或私有源引用，适合跨项目、跨团队共享。生产环境里通常会先用本地模块沉淀模式，稳定后再抽成版本化远程模块。

> root 管环境，module 管能力；  
> 输入靠 variables，输出靠 outputs；  
> 路径变了要 init，契约变了要 test；  
> 模块不是复制粘贴，是有边界的组件封装。

## 17. 本节小结

本节你需要真正带走的是：

- 本地模块通过 `source = "./xxx"` 引用。
- root module 和 child module 有清晰边界。

- 外部不能直接访问模块内部 local/resource。
