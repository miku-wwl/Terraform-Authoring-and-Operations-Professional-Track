# Terraform 实操训练 29 讲解版：`jsonencode` 与 `jsondecode`

## 1. 这一课要学什么

本 Lab 讲的是 Terraform 里两个非常常用的数据转换函数：

- `jsonencode`
- `jsondecode`

一句话理解：

> `jsonencode`：把 Terraform 里的 object / map / list 转成 JSON 字符串。  
> `jsondecode`：把 JSON 字符串解析回 Terraform 可以操作的数据结构。

这个能力在真实工程里非常重要，因为 Terraform 经常要和外部系统交换 JSON，例如：

- IAM Policy
- Kubernetes manifest
- 服务目录配置
- 云平台 API 返回结果
- IP 白名单
- 安全规则
- CI/CD 传入的配置文件

不要手写复杂 JSON。手写 JSON 最容易死在引号、逗号、转义符上。Terraform 已经提供了 `jsonencode`，能自动帮你把结构化数据安全地转成 JSON。

---

## 2. 为什么需要 `jsonencode`

假设你要生成一个服务描述文件，例如：

```json
{
  "name": "payments",
  "owner": "platform-team",
  "skills": ["terraform", "json", "automation"]
}
```

最笨的方法是手写字符串：

```hcl
content = "{ \"name\": \"payments\", \"owner\": \"platform-team\" }"
```

这玩意儿看起来就烦，而且一复杂就容易炸。

更工程化的写法是先用 Terraform 原生对象表达数据：

```hcl
locals {
  service = {
    name  = "payments"
    owner = "platform-team"
    skills = [
      "terraform",
      "json",
      "automation"
    ]
  }
}
```

然后用：

```hcl
jsonencode(local.service)
```

Terraform 会自动生成合法 JSON。

核心逻辑是：

> 人写结构，机器生成 JSON。

这比人肉拼字符串靠谱太多。

---

## 3. `jsonencode` 的作用

`jsonencode` 会把 Terraform 数据类型转换成 JSON 字符串。

常见映射关系大概是：

| Terraform 类型 | JSON 类型 |
|---|---|
| string | string |
| number | number |
| bool | boolean |
| list / tuple | array |
| map / object | object |
| null | null |

例如：

```hcl
locals {
  app_config = {
    name    = "payments"
    enabled = true
    replicas = 3
    tags = {
      env  = "dev"
      team = "platform"
    }
  }
}
```

执行：

```hcl
jsonencode(local.app_config)
```

会得到类似：

```json
{"enabled":true,"name":"payments","replicas":3,"tags":{"env":"dev","team":"platform"}}
```

注意：输出可能是一行 JSON，不一定自动美化缩进。  
这不影响机器读取。如果人要读，可以再用 `jq` 或编辑器格式化。

---

## 4. 为什么需要 `jsondecode`

`jsondecode` 反过来做事。

它把 JSON 字符串变成 Terraform 可以读取、索引、筛选的数据结构。

比如外部文件 `data/service.json` 内容是：

```json
{
  "name": "payments",
  "skills": ["terraform", "json", "automation"]
}
```

Terraform 可以这样读取：

```hcl
locals {
  service_from_file = jsondecode(file("${path.module}/data/service.json"))
}
```

然后就能像普通对象一样访问：

```hcl
local.service_from_file.name
length(local.service_from_file.skills)
```

也就是说，JSON 不再只是一个字符串，而是变成了 Terraform 可操作的数据。

## 8. 每条命令在干什么

### `terraform init -input=false`

初始化工作目录，下载 provider，准备 `.terraform` 目录。

`-input=false` 的意思是禁止交互输入。  
这在 CI/CD 里很重要，因为流水线不能卡住等你输入 yes/no。

---

### `terraform fmt -check`

检查 Terraform 文件格式。

如果格式不符合标准，它不会自动修改，而是直接报错。  
这适合 CI，因为 CI 的职责是阻断不合格代码，不是偷偷帮你改代码。

---

### `terraform validate`

检查语法和配置结构是否有效。

比如：

- 函数名写错
- 引用不存在
- 资源配置不合法
- 类型不匹配

这些通常会在 validate 阶段暴露。

---

### `terraform test`

运行 `.tftest.hcl` 测试文件。

本 Lab 里测试的核心是：

- `jsonencode` 是否生成预期 JSON。
- `jsondecode` 是否能读取外部 JSON。
- 解码后的服务名是否正确。
- 技能数量是否正确。

测试通过说明实验逻辑被固化了，不只是“手工跑了一次”。

---

### `terraform plan -out=tfplan`

生成执行计划，并保存到 `tfplan` 文件。

这个习惯很好，适合生产流程：

```sh
terraform plan -out=tfplan
terraform apply tfplan
```

这样 apply 执行的就是刚才审批过的 plan，而不是重新计算出来的新 plan。

---

### `terraform output`

查看 Terraform 输出结果。

本 Lab 重点关注类似：

- 解码后的服务名
- 技能数量
- 生成的 JSON 文件路径

## 9. 重点理解：不要手拼 JSON

这是这节课最重要的点。

不推荐：

```hcl
content = "{\"name\":\"payments\",\"skills\":[\"terraform\",\"json\"]}"
```

推荐：

```hcl
content = jsonencode({
  name = "payments"
  skills = [
    "terraform",
    "json"
  ]
})
```

前者像在手搓炸弹。  
后者是让 Terraform 干它该干的活。

---

## 10. `jsonencode` 常见使用场景

### 场景 1：生成策略文件

例如 IAM policy、Azure policy、Kubernetes policy。

```hcl
locals {
  policy = {
    Version = "2012-10-17"
    Statement = [
      {
        Effect   = "Allow"
        Action   = ["s3:GetObject"]
        Resource = "*"
      }
    ]
  }
}
```

```hcl
jsonencode(local.policy)
```

这比手写 JSON policy 安全得多。

---

### 场景 2：生成应用配置

例如生成 `appsettings.json`、`config.json`。

```hcl
resource "local_file" "config" {
  filename = "${path.module}/output/config.json"

  content = jsonencode({
    service = "payments"
    env     = "dev"
    debug   = false
  })
}
```

---

### 场景 3：给外部工具传参数

很多 CI/CD 工具、扫描工具、部署工具都吃 JSON。

Terraform 可以先组织结构化数据，再用 `jsonencode` 输出给外部工具。

---

## 11. `jsondecode` 常见使用场景

### 场景 1：读取服务清单

```hcl
locals {
  service_catalog = jsondecode(file("${path.module}/data/service.json"))
}
```

然后：

```hcl
local.service_catalog.name
local.service_catalog.skills
```

---

### 场景 2：读取外部平台导出的 JSON

比如某个平台导出了资源列表：

```json
{
  "environments": ["dev", "test", "prod"]
}
```

Terraform 可以解码后循环使用。

---

### 场景 3：读取团队维护的配置文件

团队可以把配置写在 JSON 文件中，Terraform 读取后生成资源。

这适合服务目录、环境矩阵、命名规则、标签规则等配置。

---

## 12. 常见坑

### 坑 1：JSON 文件不是合法 JSON

比如最后多了一个逗号：

```json
{
  "name": "payments",
}
```

这是非法 JSON。  
`jsondecode` 会直接失败。

---

### 坑 2：字段路径写错

JSON 是：

```json
{
  "service": {
    "name": "payments"
  }
}
```

访问时应该是：

```hcl
local.config.service.name
```

不是：

```hcl
local.config.name
```

---

### 坑 3：把 JSON 字符串当对象用

如果你只是：

```hcl
locals {
  raw = file("${path.module}/data/service.json")
}
```

这时 `local.raw` 只是字符串。

要变成对象，必须：

```hcl
jsondecode(local.raw)
```

---

### 坑 4：数字和字符串类型混淆

JSON 里：

```json
{
  "port": 8080
}
```

这是 number。

```json
{
  "port": "8080"
}
```

这是 string。

Terraform 后续比较或传参时，类型可能影响结果。

## 14. 生产环境建议

### 建议 1：复杂 JSON 尽量使用 `jsonencode`

尤其是 policy、manifest、配置文件。  
不要靠手写字符串硬拼。

---

### 建议 2：外部 JSON 进入 Terraform 后先校验结构

比如读取 JSON 后，可以用 test 或 validation 确认关键字段存在。

---

### 建议 3：不要把 JSON 文件当成“无类型魔法盒”

JSON 进入 Terraform 后还是有类型的。  
对象、数组、字符串、数字、布尔值都要分清。

---

### 建议 4：生成给人看的 JSON，可以额外格式化

`jsonencode` 生成的 JSON 可能是一行。  
机器读没问题，人读可以用：

```sh
jq . output/service.json
```

> 读进来，用 `jsondecode`。  
> 别手拼 JSON，容易把自己拼没。
