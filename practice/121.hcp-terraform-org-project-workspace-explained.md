# Terraform 考试训练 121：创建 Organization、Project 与 Workspace（讲解版）

## 1. 这一节到底在考什么？

这一节考的不是“按钮在哪里点”，而是 HCP Terraform 里面几个核心对象的层级关系：

```text
Organization
└── Project
    └── Workspace
        ├── Terraform configuration
        ├── State
        ├── Runs / Plans / Applies
        ├── Variables
        ├── Credentials
        └── Policies / Permissions
```

你要记住一句话：

> Organization 管团队边界，Project 管分组，Workspace 管一套 Terraform 运行单元。

考试题通常不会让你背 UI 操作顺序，而是会包装成团队协作、Git 工作流、本地 CLI、远端运行、状态管理这些场景，让你判断应该怎么建 workspace、选哪种 workflow。

---

## 2. 三个对象怎么理解？

## 2.1 Organization：最高层组织边界

Organization 是 HCP Terraform 里的顶层容器。

它通常对应：

- 一个公司
- 一个团队
- 一个部门
- 一个学习账号
- 一个实验环境

Organization 下面可以有多个 Project，Project 下面再有多个 Workspace。

可以把 Organization 理解成“平台的大院子”。  
所有成员、权限、计费、团队协作边界，基本都围绕 Organization 展开。

---

## 2.2 Project：工作区分组

Project 是 Organization 下面的分组对象。

它的作用是把 workspace 按用途、团队、环境或系统进行归类。

常见 project 划分方式：

```text
Organization: acme-cloud
├── Project: platform
├── Project: application
├── Project: security
├── Project: networking
└── Project: learning
```

也可以按环境分：

```text
Organization: acme-cloud
├── Project: dev
├── Project: staging
└── Project: prod
```

考试重点是：创建 Organization 后，平台可能提供默认 Project，但生产环境里不要只靠默认 Project 瞎堆 workspace。  
真正做平台工程时，Project 应该服务于团队边界、权限边界和资源治理边界。

---

## 2.3 Workspace：真正执行 Terraform 的单元

Workspace 是 HCP Terraform 里最重要的运行单元。

一个 Workspace 通常对应一套 Terraform 配置和一份 state。

例如：

```text
Project: platform
├── Workspace: network-prod
├── Workspace: aks-prod
├── Workspace: monitoring-prod
└── Workspace: dns-prod
```

Workspace 里面会管理：

- Terraform 代码来源
- Terraform 版本
- Variables
- Cloud credentials
- State
- Plan / Apply 记录
- Run trigger
- Policy check
- 审批流程
- 权限控制

所以创建 workspace 不是“建个空文件夹”，而是在定义一套基础设施交付流水线。

---

## 3. 创建顺序怎么记？

简单记：

```text
先有 Organization
再有 Project
最后有 Workspace
```

也就是：

```text
Organization -> Project -> Workspace
```

不要反过来理解。

Workspace 不能脱离 Organization 存在。  
Project 是 Workspace 的分组容器。

---

## 4. Workspace 创建时最重要的选择：Workflow

创建 Workspace 时，最关键的配置之一是 workflow 类型。

这也是本节最容易出考试题的地方。

常见 workflow 有三类：

```text
Version Control Workflow
CLI-driven Workflow
API-driven Workflow
```

---

## 5. Version Control Workflow

## 5.1 是什么？

Version Control Workflow 也就是 VCS 驱动的工作流。

它通常连接 GitHub、GitLab、Bitbucket、Azure DevOps 等代码仓库。

典型流程是：

```text
开发者提交代码
-> Git PR / Merge Request
-> HCP Terraform 自动触发 plan
-> 审核 plan
-> merge / apply
-> 记录 run history
```

## 5.2 适合什么场景？

适合团队协作场景。

尤其是这种要求：

- 所有变更必须经过 Git PR
- 代码必须可审计
- plan 自动触发
- apply 有审批
- 团队希望统一入口管理基础设施变更

考试里看到这些关键词，基本优先考虑 Version Control Workflow：

```text
Git
VCS
Pull Request
Merge Request
automatic plan
team collaboration
audit
code review
```

## 5.3 典型判断

题目说：

> 团队要求所有基础设施变更都必须经过 Git PR，然后自动生成 plan。

答案应该选：

```text
Version Control Workflow
```

原因是它直接把 Git 仓库和 HCP Terraform run 绑定起来。

---

## 6. CLI-driven Workflow

## 6.1 是什么？

CLI-driven Workflow 适合继续使用本地 Terraform CLI，但把运行和状态交给 HCP Terraform 管理。

典型流程是：

```text
本地写 Terraform 代码
-> 本地执行 terraform plan / apply
-> 实际 run 进入 HCP Terraform
-> state 存在 HCP Terraform
-> run history 可追踪
```

它保留了本地命令习惯，但不是完全本地执行。

## 6.2 适合什么场景？

适合：

- 个人学习者
- 小团队
- 习惯命令行的人
- 还没接 Git 自动化流程的人
- 想从 local backend 迁移到 HCP Terraform 的人

考试里看到这些关键词，可以考虑 CLI-driven Workflow：

```text
local Terraform CLI
command line
developer runs commands manually
remote state in HCP Terraform
keep local workflow
```

## 6.3 典型判断

题目说：

> 个人学习者想保留本地 Terraform CLI 操作习惯，但希望 run 和 state 由 HCP Terraform 管理。

答案应该选：

```text
CLI-driven Workflow
```

---

## 7. API-driven Workflow

## 7.1 是什么？

API-driven Workflow 是通过 API 创建和触发 HCP Terraform run。

它更适合平台系统、内部工具、自研流水线或自动化平台。

典型流程：

```text
内部平台 / CI 系统
-> 调用 HCP Terraform API
-> 上传配置或触发 run
-> 查询 plan/apply 状态
-> 返回结果给平台用户
```

## 7.2 适合什么场景？

适合：

- 自研 DevOps 平台
- 内部 IaC 平台
- 高度定制化流水线
- 不想直接绑定 Git 仓库
- 需要由外部系统统一调度 Terraform runs

考试里如果看到：

```text
custom platform
API
automation system
external orchestrator
self-service portal
```

可以考虑 API-driven Workflow。

---

## 8. 三种 Workflow 对比

| Workflow | 谁触发 Run | 代码来源 | 适合场景 | 考试关键词 |
|---|---|---|---|---|
| Version Control Workflow | Git 事件 / PR / merge | Git 仓库 | 团队协作、代码审查、自动 plan | VCS、Git、PR、audit |
| CLI-driven Workflow | 本地 Terraform CLI | 本地目录 | 个人学习、命令行习惯、远端 state | local CLI、manual command |
| API-driven Workflow | 外部系统调用 API | API 上传或外部系统提供 | 自研平台、自动化系统 | API、custom platform、orchestrator |

---

## 9. Local Terraform Workspace 和 HCP Terraform Workspace 不要混淆

这是 Terraform 学习里很容易绕晕的点。

Terraform CLI 里也有 workspace，例如：

```sh
terraform workspace list
terraform workspace new dev
terraform workspace select prod
```

这个是 Terraform CLI 的 workspace 概念。

HCP Terraform 里的 Workspace 是平台级运行单元。

两者不是一个层级的东西。

简单区分：

| 名称 | 属于哪里 | 主要作用 |
|---|---|---|
| Terraform CLI workspace | 本地 CLI / backend 机制 | 在同一套配置下切换不同 state |
| HCP Terraform workspace | HCP Terraform 平台对象 | 管理配置、state、run、变量、权限、策略 |

考试如果说的是 HCP Terraform 创建 Organization、Project、Workspace，那就是平台对象，不是 `terraform workspace new` 那个命令。

---

## 10. 生产环境创建 Workspace 时要考虑什么？

考试题可能只问 workflow，但真实项目里创建 workspace 要考虑更多。

至少要想清楚这些问题：

```text
1. 这套 Terraform 管什么资源？
2. 代码从哪里来？
3. 谁可以 plan？
4. 谁可以 apply？
5. state 放在哪里？
6. 变量怎么管理？
7. 云凭据怎么注入？
8. Terraform 版本固定吗？
9. 是否需要 policy check？
10. 是否需要审批？
11. 是否需要和其他 workspace 建依赖？
```

这就是平台工程思维：  
Workspace 不是一个 UI 表单，而是一套基础设施交付边界。

---

## 11. 常见考试陷阱

## 11.1 陷阱一：创建 Organization 后就自动连接 GitHub

错误。

创建 Organization 不代表已经连接 GitHub。  
GitHub / VCS 连接通常是在创建 workspace 或配置 VCS provider 时处理。

所以判断题：

> 创建 organization 后就一定已经连接了 GitHub 仓库。

答案是：

```text
错
```

---

## 11.2 陷阱二：Workspace 只是保存 state 的地方

不完整。

Workspace 会管理 state，但它不只是 state storage。

它还管理：

- runs
- variables
- credentials
- permissions
- policies
- Terraform version
- workflow
- run history

所以更准确地说：

> Workspace 是 HCP Terraform 中一套 Terraform 配置的远端执行与治理单元。

---

## 11.3 陷阱三：所有团队都应该用 CLI-driven Workflow

不对。

CLI-driven 适合保留本地命令习惯。  
但团队协作、Git 审计、PR 自动 plan 场景，更适合 Version Control Workflow。

---

## 11.4 陷阱四：Project 不重要

也不对。

学习环境里你可以全放默认 Project。  
但生产环境里，Project 会影响资源分组、权限管理和团队边界。

别把生产环境做成“所有东西都塞 default project”的垃圾桶。

---

## 12. 场景题怎么答？

## 场景 1：团队要求所有变更必须经过 Git PR，再由平台自动 plan

推荐：

```text
Version Control Workflow
```

原因：

- 代码来自 Git 仓库
- PR 可以触发 plan
- 团队可以审查变更
- run history 可追踪
- 适合审计和协作

可以这样回答：

> 我会选择 Version Control Workflow。因为团队要求所有变更经过 Git PR，并希望平台自动 plan。VCS workflow 能把 Git 仓库和 HCP Terraform run 绑定起来，使基础设施变更具备代码审查、自动计划、审计记录和团队协作能力。

---

## 场景 2：个人学习者想保留本地 CLI 操作习惯，但希望 run 发生在 HCP Terraform

推荐：

```text
CLI-driven Workflow
```

原因：

- 本地继续使用 Terraform CLI
- 不强制依赖 Git 自动触发
- run 和 state 可以进入 HCP Terraform
- 适合学习和渐进式迁移

可以这样回答：

> 我会推荐 CLI-driven Workflow。因为用户希望继续使用本地 Terraform CLI，但又希望 run 和 state 由 HCP Terraform 管理。CLI-driven workflow 正好保留本地命令行体验，同时获得远端状态、运行记录和平台治理能力。

## 16. 和真实项目的关系

在真实平台团队里，HCP Terraform 对象设计通常长这样：

```text
Organization: company-cloud
├── Project: networking
│   ├── Workspace: vpc-dev
│   ├── Workspace: vpc-staging
│   └── Workspace: vpc-prod
├── Project: kubernetes
│   ├── Workspace: eks-dev
│   ├── Workspace: eks-prod
│   ├── Workspace: aks-dev
│   └── Workspace: aks-prod
└── Project: security
    ├── Workspace: iam-baseline
    └── Workspace: policy-as-code
```

这时候 workspace 的设计会直接影响：

- blast radius
- state 边界
- 审批边界
- 权限边界
- 团队协作方式
- CI/CD 流程
- 故障排查范围

所以这节虽然看起来是“创建对象”，本质上是在训练你做 Terraform 平台建模。

---

## 17. 总结

本节核心可以压缩成三句话：

1. HCP Terraform 的层级是 Organization -> Project -> Workspace。
2. Workspace 是真正管理 Terraform 配置、state、run、变量、权限和 workflow 的单元。
3. Workflow 类型决定代码来源和 run 触发方式：Git 协作用 VCS，本地命令用 CLI，自研平台用 API。

考试遇到这类题，先不要急着看按钮。  
先判断场景里是谁驱动 Terraform：

```text
Git 驱动？选 Version Control Workflow。
人手命令驱动？选 CLI-driven Workflow。
外部系统驱动？选 API-driven Workflow。
```
