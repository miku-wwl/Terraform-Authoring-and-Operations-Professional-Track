## 2. 知识点讲解

`aws_instance` 和 `aws_instances` 都是 EC2 相关 data source，但语义不同。`aws_instance` 期望匹配一个实例，如果过滤条件匹配多个实例，真实 AWS provider 会报错；`aws_instances` 用于返回多个实例 ID 和 IP 列表。

生产环境里，data source 必须写清楚过滤条件。过宽会读到错误资源，过窄会读不到资源。常见做法是通过 tag、state、VPC、subnet 等条件约束查询范围。

本实验用 LocalStack 创建两台实例，其中一台标记 `Team=production`。用户需要用 `aws_instance` 精确读取生产实例，再用 `aws_instances` 读取所有 lab 实例。

## 3. 实验设计

背景：团队中已有多台 EC2，Terraform 配置需要读取其中一台生产实例和一组 lab 实例。

任务：使用 tag filter 精确匹配单个实例，并用多实例 data source 获取列表。

预期结果：输出中包含一个生产实例 ID 和多个 lab 实例 ID。

## 5. 运行手册

```sh
cd practice/labs/33
export AWS_ACCESS_KEY_ID=test
export AWS_SECRET_ACCESS_KEY=test
export AWS_DEFAULT_REGION=us-east-1
export LOCALSTACK_ENDPOINT=http://localhost:4566

bash scripts/check-sandbox.sh
bash scripts/bootstrap.sh
terraform init -input=false
terraform fmt -check
terraform validate
terraform plan -input=false -no-color -out=tfplan
terraform apply -auto-approve tfplan
terraform output
bash scripts/verify.sh
terraform destroy -auto-approve
bash scripts/clean.sh
```

## 7. 排障指南

1. `aws_instance` 匹配多个结果：增加 tag 或 instance-state 过滤条件。
2. 查询为空：确认 `bootstrap.sh` 已创建实例。
3. LocalStack 查询正常但 Terraform 为空：检查 provider endpoint 和 region 是否一致。
4. 输出类型不对：区分单个对象属性和列表属性。

## 8. 清理方式
