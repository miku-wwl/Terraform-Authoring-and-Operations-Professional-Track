# Terraform 实操训练 101 讲解：重命名带 `count` 的资源

## 1. 本节在学什么

这一节的核心主题是：

> 当 Terraform 资源用了 `count` 创建多个实例后，如果你只是想给资源块改名，不想让 Terraform 删除旧资源、再创建新资源，就应该使用 `moved` block 显式告诉 Terraform：旧地址已经搬到了新地址。

原始课程文件的主题是“重命名带 count 的资源”，并且说明本实验通过 LocalStack 模拟 AWS API，不访问真实 AWS。

---

## 2. 先理解 Terraform 的“资源地址”

Terraform 判断一个资源是不是“同一个资源”，主要不是看真实云上的名字，而是看它在 Terraform 状态文件里的地址。

例如：

```hcl
resource "aws_s3_bucket" "old" {
  count  = 2
  bucket = "demo-${count.index}"
}
```

这不是一个资源，而是两个实例：

```text
aws_s3_bucket.old[0]
aws_s3_bucket.old[1]
```

这里的完整资源地址包括三部分：

```text
资源类型.资源名[count 下标]
```

也就是：

```text
aws_s3_bucket.old[0]
aws_s3_bucket.old[1]
```

如果你把资源块从 `old` 改名成 `new`：

```hcl
resource "aws_s3_bucket" "new" {
  count  = 2
  bucket = "demo-${count.index}"
}
```

Terraform 默认会认为：

```text
aws_s3_bucket.old[0] 不存在了，需要 destroy
aws_s3_bucket.old[1] 不存在了，需要 destroy

aws_s3_bucket.new[0] 是新资源，需要 create
aws_s3_bucket.new[1] 是新资源，需要 create
```

这就是“只是改了代码里的名字，却触发删除重建”的根源。

---

## 3. 为什么需要 `moved` block

`moved` block 的作用是给 Terraform 状态迁移提供一张“搬家登记表”。

它告诉 Terraform：

> 状态里的旧地址不是被删除了，而是搬到了新地址。

对于带 `count` 的资源，不能只写一个笼统的移动。你需要针对每一个实例下标写清楚。

例如：

```hcl
moved {
  from = aws_s3_bucket.old[0]
  to   = aws_s3_bucket.new[0]
}

moved {
  from = aws_s3_bucket.old[1]
  to   = aws_s3_bucket.new[1]
}
```

这样 Terraform 在 plan 阶段会理解：

```text
aws_s3_bucket.old[0] -> aws_s3_bucket.new[0]
aws_s3_bucket.old[1] -> aws_s3_bucket.new[1]
```

这不是资源删除重建，而是状态地址迁移。

---

## 4. 一句话理解这个实验

这个实验不是为了教你“怎么改变量名”，而是为了训练你在生产环境里安全重构 Terraform 代码。

生产环境里最怕这种情况：

```text
你只是想把 resource 名字改得更清晰
Terraform 却计划 destroy 一批真实资源
```

`moved` block 就是用来避免这种事故的。

## 5. `count` 场景下最容易踩的坑

### 坑 1：只移动资源块，不移动实例

错误理解：

```hcl
moved {
  from = aws_s3_bucket.old
  to   = aws_s3_bucket.new
}
```

在 `count` 资源上，这种写法通常不是你真正想要的。因为状态里实际存在的是：

```text
aws_s3_bucket.old[0]
aws_s3_bucket.old[1]
```

不是一个单独的 `aws_s3_bucket.old`。

更稳的做法是逐个实例写清楚：

```hcl
moved {
  from = aws_s3_bucket.old[0]
  to   = aws_s3_bucket.new[0]
}

moved {
  from = aws_s3_bucket.old[1]
  to   = aws_s3_bucket.new[1]
}
```

### 坑 2：下标对错了

例如：

```hcl
moved {
  from = aws_s3_bucket.old[0]
  to   = aws_s3_bucket.new[1]
}
```

这会把旧的第 0 个实例迁移到新的第 1 个实例。

如果你的业务含义依赖下标，比如：

```text
[0] = dev
[1] = prod
```

那下标对错就不是“小问题”，而是生产事故预备役。

### 坑 3：把重命名和结构重构混在一起做

比较危险的做法：

```text
同时改 resource 名字
同时改 count 数量
同时改 bucket 命名规则
同时改 tags
同时改模块路径
```

这样 plan 一旦变复杂，你很难判断 Terraform 到底是在：

```text
移动状态？
修改资源？
删除重建？
```

更安全的做法是分步提交：

```text
第一步：只做 moved block + resource 重命名
第二步：确认 plan 只有 moved，没有 destroy/create
第三步：再做其他字段修改
```

## 6. 实验环境为什么用 LocalStack

原始文件说明，本实验不访问真实 AWS，而是使用 LocalStack 的 `http://localhost:4566` endpoint 模拟 AWS API，并使用假凭证。

这很适合练习 Terraform 行为，因为本节真正要验证的是：

```text
Terraform state address 的迁移逻辑
```

而不是 AWS S3 / IAM / EC2 本身。

也就是说：

```text
LocalStack 负责模拟 AWS API
Terraform 负责演示状态迁移
moved block 负责保护资源不被误删重建
```

## 7. 推荐实验观察点

你在 `practice/labs/101/` 里跑实验时，不要只看命令是否成功，要重点观察 plan 输出。

### 观察点 1：有没有 destroy/create

安全重命名时，你最希望看到的是类似：

```text
has moved to
```

而不是：

```text
-/+ destroy and then create replacement
```

或者：

```text
- destroy
+ create
```

如果 Terraform 出现 destroy/create，那说明它没有正确理解这是状态地址移动。

### 观察点 2：状态地址是否变化

可以在重命名前后观察：

```sh
terraform state list
```

重命名前可能是：

```text
aws_s3_bucket.old[0]
aws_s3_bucket.old[1]
```

重命名后应该变成：

```text
aws_s3_bucket.new[0]
aws_s3_bucket.new[1]
```

重点是：真实资源没有被重建，只是 Terraform state 里的地址变了。

### 观察点 3：`terraform plan` 是否干净

重命名迁移完成后，再跑一次：

```sh
terraform plan -input=false -no-color
```

理想结果应该接近：

```text
No changes. Your infrastructure matches the configuration.
```

这说明 Terraform 状态和代码已经重新对齐。

根据原始文件，本节需要在 LocalStack 环境下完成初始化、计划、应用、验证和销毁。可以按下面流程理解：


## 9. 真实项目里的迁移方式

生产项目里遇到这种需求时，建议按这个顺序来：

### 第一步：先看当前状态

```sh
terraform state list
```

找到旧资源地址：

```text
aws_xxx.old_name[0]
aws_xxx.old_name[1]
```

### 第二步：只改资源块名称

例如从：

```hcl
resource "aws_s3_bucket" "old_name" {
  count = 2
}
```

改成：

```hcl
resource "aws_s3_bucket" "new_name" {
  count = 2
}
```

### 第三步：补上 `moved` block

```hcl
moved {
  from = aws_s3_bucket.old_name[0]
  to   = aws_s3_bucket.new_name[0]
}

moved {
  from = aws_s3_bucket.old_name[1]
  to   = aws_s3_bucket.new_name[1]
}
```

### 第四步：只看 plan，不急着 apply

```sh
terraform plan -input=false -no-color
```

确认没有误删重建。

### 第五步：apply 状态迁移

```sh
terraform apply
```

### 第六步：迁移稳定后，再决定是否保留 `moved` block

团队项目里通常建议保留一段时间，尤其是多人协作、多个环境不是同时 apply 的时候。不要刚迁移完就立刻删掉，否则别的环境还没迁移时会踩坑。

## 10. `moved` block 和 `terraform state mv` 的区别

两者都能做状态地址迁移，但适用场景不同。

### `moved` block

写在代码里：

```hcl
moved {
  from = aws_s3_bucket.old[0]
  to   = aws_s3_bucket.new[0]
}
```

优点：

```text
可审计
可 code review
适合团队协作
适合多环境重复执行
适合 GitOps / CI/CD
```

### `terraform state mv`

命令式操作：

```sh
terraform state mv 'aws_s3_bucket.old[0]' 'aws_s3_bucket.new[0]'
```

优点：

```text
适合一次性手工修复
适合紧急状态修正
```

缺点：

```text
不容易审计
不容易在多环境复用
容易出现“我本地修好了，别人环境没修”的问题
```

在现代 Terraform 项目里，如果是代码重构导致的资源地址变化，优先考虑 `moved` block。

## 11. 这节课真正的生产价值

这一节的价值不在于 `count` 本身，而在于 Terraform 重构能力。

你以后做平台工程、SRE、云基础设施项目时，经常会遇到：

```text
资源命名不规范，需要重命名
模块结构混乱，需要拆分
早期 demo 代码要整理成生产模块
单资源要迁移成模块
模块路径要调整
```

这些都可能改变 Terraform resource address。

如果不懂 `moved` block，你会把“代码重构”变成“基础设施删除重建”。

这就是 Terraform 里很典型的：

```text
代码层的小改动
状态层的大后果
```

## 12. 面试表达方式

如果面试官问你：

> Terraform 里资源重命名怎么避免重建？

可以这样回答：

```text
Terraform 通过 state 中的 resource address 跟踪资源。
如果直接修改 resource block 的名字，Terraform 会认为旧地址消失、新地址出现，从而计划 destroy/create。
为了避免这种情况，可以使用 moved block 声明旧地址到新地址的迁移。
如果资源使用了 count，需要对每个实例下标分别声明迁移，例如 old[0] 到 new[0]、old[1] 到 new[1]。
我会先用 terraform state list 确认旧地址，再添加 moved block，然后通过 terraform plan 确认没有 destroy/create，最后再 apply。
```

这段回答比单纯说“用 moved block”更像真实干过活。

## 14. 总结

本节你要掌握的是：

1. Terraform 用资源地址追踪状态，不是只看真实云资源名称。
2. `count` 会让一个资源块变成多个带下标的实例。
3. 资源重命名会改变 Terraform resource address。
4. 直接重命名可能导致 destroy/create。
5. `moved` block 可以把旧地址迁移到新地址。
6. 带 `count` 的资源要特别注意实例下标。
7. 生产环境中，重构 Terraform 代码必须先看 plan，确认没有误删重建。

这一节非常实用。它不是“语法课”，而是 Terraform 工程化能力的一部分。
