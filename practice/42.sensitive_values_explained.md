# Terraform 实操训练 42 讲解：Sensitive 参数

## 1. 这一节到底在学什么？

Lab 42 学的是 Terraform 里的敏感值处理：`sensitive = true`。

这节很重要，因为很多初学者会误解：

> 只要加了 `sensitive = true`，秘密就安全了。

不对。

`sensitive = true` 的主要作用是：**减少 CLI 输出里的明文泄露**。  
它不是加密机制，也不会自动保护 Terraform state。

一句话记住：

> `sensitive = true` 是“遮住屏幕输出”，不是“加密保存秘密”。

---

## 2. 为什么需要 sensitive？

Terraform 经常要处理敏感输入，例如：

- API token
- 数据库密码
- SSH private key
- webhook secret
- cloud access key
- third-party service credential

如果这些值直接出现在 `terraform plan`、`terraform apply`、`terraform output` 或 CI 日志里，就会有泄露风险。

所以 Terraform 提供了 `sensitive` 标记，用来告诉 Terraform：

> 这个值不要随便在 CLI 输出中明文展示。

---

## 3. 输入变量如何标记 sensitive？

典型写法：

```hcl
variable "api_token" {
  type        = string
  sensitive   = true
  description = "API token used by automation."
}
```

这样做之后，Terraform 在很多 CLI 输出里会隐藏这个变量值。

例如 plan/apply 输出里可能显示：

```text
(sensitive value)
```

这比直接打印 token 安全得多。

---

## 4. 输出也可以标记 sensitive

如果你把敏感值作为 output 输出，也必须显式标记：

```hcl
output "api_token" {
  value     = var.api_token
  sensitive = true
}
```

否则 Terraform 会拒绝或提醒你：敏感值不能作为普通 output 暴露。

这点很关键：

> 输入 sensitive，不代表所有输出自动安全。敏感 output 也要标记。

---

## 5. 本实验的场景

Lab 42 的背景是：

> 自动化中需要传入 API token。

问题是：

> token 不应该在 plan/apply 输出中明文展示。

任务是：

1. 将输入变量标记为 sensitive。
2. 将敏感输出标记为 sensitive。
3. 输出一个非敏感派生状态，例如 `token_is_configured`。
4. 验证 token 被隐藏，但能确认 token 已配置。

预期结果：

```text
token_is_configured = true
```

而 token 本身不会在普通输出里明文展示。

---

## 6. 为什么要输出非敏感派生状态？

很多时候，我们不需要看到 token 本身，只需要知道它有没有配置。

所以可以输出这种安全派生值：

```hcl
output "token_is_configured" {
  value = length(trimspace(var.api_token)) > 0
}
```

这样输出的是：

```text
true
```

而不是 token 原文。

这是一种很好的工程习惯：

> 不输出秘密本身，只输出是否配置、是否有效、是否启用这类非敏感状态。

---

## 7. `sensitive = true` 到底保护了什么？

它主要保护的是 CLI 显示层面。

例如：

```sh
terraform output
```

可能显示：

```text
api_token = <sensitive>
token_is_configured = true
```

这可以降低日志泄露风险。

但是注意：

> 它不保证敏感值不进入 state。

很多情况下，敏感值仍然会被写入 Terraform state。  
所以你不能因为用了 `sensitive = true`，就把 state 随便提交到 Git。

---

## 8. 最容易踩的坑：state 仍然可能有敏感值

Terraform state 是 Terraform 的事实来源。  
它记录资源属性、变量影响结果、输出值等信息。

即使 CLI 把值显示为：

```text
(sensitive value)
```

state 里仍可能保存真实值。

所以生产里必须把 state 当成敏感材料管理。

这意味着：

- 不要把 `terraform.tfstate` 提交到 Git。
- 远程 state 后端要启用访问控制。
- state backend 要开启加密。
- CI 日志不要打印 state。
- 不要把 plan 文件随便上传到公开位置。
- 限制谁可以读取 state。

一句话：

> Terraform state 的权限，应该按“能看到秘密”的级别处理。

## 9. 标准运行流程

进入实验目录：

```sh
cd /workspace/practice/labs/42
```

初始化：

```sh
terraform init -input=false
```

格式检查：

```sh
terraform fmt -check
```

配置校验：

```sh
terraform validate
```

运行测试：

```sh
terraform test
```

生成 plan：

```sh
terraform plan -input=false -no-color -out=tfplan
```

执行 apply：

```sh
terraform apply -auto-approve tfplan
```

查看输出：

```sh
terraform output
```

查看敏感输出：

```sh
terraform output api_token
```

清理：

```sh
terraform destroy -auto-approve
```

清理本地文件：

```sh
rm -f tfplan
rm -rf .terraform .terraform.lock.hcl
```

## 10. 为什么 `terraform output api_token` 可以单独查看？

普通 `terraform output` 会尽量隐藏 sensitive 输出。

但是如果你显式执行：

```sh
terraform output api_token
```

Terraform 会认为你是有意读取这个输出，所以可能展示该值。

这说明 sensitive 不是强访问控制。  
它更像是 CLI 的防误打印机制。

真正的访问控制应该靠：

- state backend 权限
- CI/CD secret 权限
- 云平台 IAM
- Git 仓库权限
- 审计和日志策略

不要把 `sensitive = true` 当成权限系统。

---

## 11. Plan 文件也要小心

本实验会生成：

```sh
terraform plan -out=tfplan
```

要注意，保存下来的 plan 文件也可能包含敏感信息或资源变更细节。

所以：

```text
tfplan
```

也不应该随便提交到 Git 或发给不该看的人。

清理时删除：

```sh
rm -f tfplan
```

这是好习惯。

---

## 12. 常见坑

### 坑 1：以为 sensitive 会加密

不会。

`sensitive = true` 只是让 Terraform 在部分 CLI 输出中隐藏值。  
它不是加密，不是密钥管理系统，也不是访问控制。

---

### 坑 2：把 state 提交到 Git

这是严重错误。

不要提交：

```text
terraform.tfstate
terraform.tfstate.backup
```

`.gitignore` 里应该至少包含：

```gitignore
*.tfstate
*.tfstate.*
.terraform/
tfplan
```

---

### 坑 3：只给变量加 sensitive，忘记 output

如果输出了敏感值，也要写：

```hcl
sensitive = true
```

否则 Terraform 会提醒你，或者导致敏感值泄露。

---

### 坑 4：在 CI 里 echo token

就算 Terraform 里标了 sensitive，如果你在 shell 里写：

```sh
echo $API_TOKEN
```

照样会泄露。

`sensitive` 只能管 Terraform 自己的显示逻辑，管不了你在脚本里主动打印秘密。

---

## 13. 生产环境应该怎么处理 secret？

Lab 42 是为了学习 Terraform sensitive 机制，不是说生产里应该把 token 直接写在 `.tfvars` 里。

更成熟的做法是使用：

- AWS Secrets Manager
- Azure Key Vault
- HashiCorp Vault
- GitHub Actions Secrets
- GitLab CI/CD Variables
- Kubernetes Secret 管理流程
- SOPS + KMS
- Terraform Cloud/Enterprise workspace variables

Terraform 只负责引用或注入必要值，而不是把秘密散落在本地文件和 Git 仓库里。

---

## 14. 安全处理 Terraform 敏感值的基本原则

可以记这几条：

1. 能不输出就不输出。
2. 必须输出时标记 `sensitive = true`。
3. 能输出派生状态，就不要输出原文。
4. state 按秘密材料处理。
5. plan 文件也要按敏感文件处理。
6. `.tfvars` 不要随便提交。
7. CI 日志不要打印秘密。
8. 使用正式 secret manager，而不是靠本地文件裸存。

这些才是生产级 Terraform 的基本安全纪律。

## 17. 本 Lab 的价值

Lab 42 的价值在于，它纠正了一个很危险的误解：

> “Terraform 输出隐藏了，所以秘密安全了。”

工程上不能这么想。

正确理解应该是：

- CLI 隐藏只是第一层防护。
- state 仍然可能保存秘密。
- plan 文件也可能包含敏感信息。
- 真正安全要靠后端权限、加密、secret manager 和日志治理。

这节学好后，你写 Terraform 会更像生产工程师，而不是只会“把 token 塞进去让它能跑”的脚本玩家。
